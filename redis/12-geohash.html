第 12 章 · Geohash 空间索引

Z-order 位交错 · 52 位整数 · ZSet score 复用 · 9 邻居矩形覆盖 · Haversine 二次过滤 · 墨卡托纬度边界

地理坐标 (lon, lat) 是二维数据,但 B-tree、跳跃表、哈希表均针对一维有序键设计。Redis 通过 Geohash 编码将二维坐标映射到一维 52 位整数,以 ZSet 的 score 字段存储,使一维有序结构具备空间范围查询能力。GEOADD / GEOSEARCH / GEORADIUS 等命令都构建在该映射之上。

Geohash 的核心是 Z-order(Morton)曲线:把经度 26 位与纬度 26 位位交错(interleave64),让前缀相同的 hash 值对应几何邻近的点。这一性质把"求圆形范围内的点"转化为"查 9 个 Geohash 前缀对应的 score 区间",再用 Haversine 公式精确过滤候选点的真实距离。

本章描述 Geohash 的位交错构造、Z-order 曲线性质、52 位精度边界、墨卡托投影限制、9 邻居单元搜索策略,以及 Haversine 距离的最终过滤。把整个 GEO 命令族放进 ZSet 的视角下看,你会发现 Redis "用现有数据结构封装新语义" 的工程哲学贯彻得多么彻底 — GEO 没有新建任何底层结构,完全是 ZSet 的 score/member 字段被解释为另一种含义。

12.1 二维到一维的映射问题

"给定一个圆,找圆内所有的点" — 这是地理服务的核心查询。要让 Redis 这种纯键值 / 一维有序的存储支持它,必须先回答一个数学问题:怎么把二维坐标映射成一维数,同时尽量保留邻近性?这一节先排除两个直观但错误的方案,再引入 Z-order 曲线。

12.1.2 平面填充曲线对比

图 12-1-2 · 三种平面填充曲线的局部性比较

Z-order 曲线(也称 Morton 编码)的核心性质:二维坐标位交错后,前缀相同的 hash 值对应几何上邻近的点。Hilbert 曲线在局部性上略优(连续点间的距离方差更小),但编码 / 解码需要更多位运算。Redis 选择 Z-order 在工程上更简洁 — interleave64 用 4 次 mask + 4 次 shift 即可完成,而 Hilbert 需要 26 次状态机迭代。

Redis 的工程取舍:Geohash + Z-order 让 GEO 命令零额外索引开销 — 完全复用 ZSet 的 skiplist + dict,新增的只是命令实现里的位运算函数。如果选 Hilbert 或 R-tree,会引入新的数据结构,违反 Redis "用最少结构覆盖最多场景" 的原则。

12.2 位交错构造 Geohash

位交错(bit interleaving)是 Geohash 的核心。把经度的 26 位与纬度的 26 位交替排布成一个 52 位整数,这个整数就是 Z-order 曲线在该坐标点的"位置"。这一节先讲清"递归二分"产生每一位的过程,再讲interleave64 的位运算实现,最后用 4×4 网格直观展示 Z-order。

12.2.2 interleave64 位运算技巧

图 12-2-2 · 位交错的高效实现 — 5 次 mask 扩散完成 O(1) 交错

/* src/util.c — 位交错的紧凑实现 */
uint64_t interleave64(uint32_t xlo, uint32_t ylo) {
    static const uint64_t B[] = {
        0x5555555555555555ULL, 0x3333333333333333ULL,
        0x0F0F0F0F0F0F0F0FULL, 0x00FF00FF00FF00FFULL,
        0x0000FFFF0000FFFFULL
    };
    static const unsigned int S[] = {1, 2, 4, 8, 16};
    uint64_t x = xlo, y = ylo;
    x = (x | (x << S[4])) & B[4];   x = (x | (x << S[3])) & B[3];
    x = (x | (x << S[2])) & B[2];   x = (x | (x << S[1])) & B[1];
    x = (x | (x << S[0])) & B[0];
    y = (y | (y << S[4])) & B[4];   y = (y | (y << S[3])) & B[3];
    y = (y | (y << S[2])) & B[2];   y = (y | (y << S[1])) & B[1];
    y = (y | (y << S[0])) & B[0];
    return x | (y << 1);
}

12.2.3 Z-order 曲线的几何可视化

图 12-2-3 · 4×4 网格上的 Z-order 编码顺序

Z-order 的"角落跳跃":cell 3 (右上角) 到 cell 4 (左下角下一行) 物理上相邻,但 Z-order 编码 3 → 4 跨越很大。这就是为什么 GEOSEARCH 必须查 3×3 = 9 个邻居单元,而不是只查目标点所在单元 — 圆可能跨过 Z-order 的"跳跃边界"落入相邻单元。

12.3 52 位精度边界

为什么 Redis 选 26+26 = 52 位,不是 32+32 = 64 位或 16+16 = 32 位?这一节回答精度选型的工程逻辑:52 位刚好对齐 IEEE 754 double 的尾数,且 0.6 m 精度恰好高于 GPS 自身误差 — 再高就是浪费。

12.3.1 位数与精度对照

表 12-3-1 · Geohash 位数与赤道单元尺寸
位数(经/纬各 N 位)单元尺寸(赤道附近)应用场景

10 / 10 = 20156 km × 156 km省 / 州

12.3.2 选择 52 位的三个理由

Redis 选 26+26 = 52 位的工程逻辑:
① 52 位完美装入 ZSet score 的 double 尾数:IEEE 754 double 有 52 位尾数 + 11 位指数 + 1 位符号,52 位整数能完整无损地塞进 double 表示;
② 0.6 m 精度对消费级 LBS 充足:消费 GPS 自身误差 5-10 m,工业 GPS 1-3 m,提高 Geohash 精度无收益;
③ 不开放配置,避免合并问题:若位数可调,两个不同位数的 GEO key 无法合并(类似 PFMERGE 的精度兼容问题),工程上不划算。

12.3.3 精度随位数指数衰减

图 12-3-1 · 单元尺寸随位数指数减小(对数纵轴,赤道附近)

规律:每加 2 位(每方向各 1 位),单元尺寸减半 — 几何级数衰减。Redis 默认 52 位刚好落在 0.6 m,这是工程边界(对齐 IEEE 754 double 尾数)。超出此精度需引入更多位 + 不同 score 表示方案。
高纬度提醒:经度 1° 实际距离在高纬度变小(地球收缩),cell 形状由正方形变为细长矩形 — 这是墨卡托投影固有的性质。

12.6 Haversine 距离过滤

9 邻居单元给出的是矩形覆盖,而用户要的是圆形范围。这中间的差异由 Haversine 公式的二次过滤消除。这一节讲清 Haversine 的几何含义、为什么 Redis 选它而不是更精确的 Vincenty,以及该选择的误差预算。

12.6.1 Haversine 公式几何含义

图 12-6-1 · 球面两点之间的最短弧长 — Haversine 公式

12.6.2 矩形覆盖与圆形目标的差异

图 12-6-2 · GEOSEARCH 的两阶段过滤(粗 → 精)

矩形覆盖与圆形目标之间的差异由 Haversine 过滤消除。该差异决定了 Geohash 范围查询是"先粗后细"的两阶段过程,Geohash 仅用于裁剪候选集,不直接给出距离。如果跳过 Haversine 直接返回 9 矩形内的点,你会得到正方形的近似结果(很多业务场景实际可接受,但 Redis 默认严格)。

12.11.3 反例:用 GEODIST 代替路径规划

错误:把 GEODIST 当导航距离

dist = redis.geodist("stores", "user_loc", "shop_A", "km")
display(f"距离您 {dist} km,预计配送 8 分钟")
  • 症状:UI 显示 2.3 km / 配送 8 分钟,实际驾车 4.5 km / 配送 20 分钟
  • 根因:Haversine 是地球表面的"鸟飞路线" — 不考虑道路、河流、建筑、单行线、限行
  • 绕道系数:市内配送 1.3-1.8 倍, 山地 / 老城区可达 2.5 倍
  • 结果:GEODIST 严重低估真实出行距离 → 用户投诉、配送超时

修复:GEO 做粗筛,导航做精算

# 阶段 1 — Redis GEO 粗筛 O(log N + M)
candidates = redis.geosearch(
    "stores",
    fromlonlat=(lon, lat),
    radius=5, unit="km",
    withcoord=True, count=50,
)

# 阶段 2 — 路径规划服务精算 (≤ 50 次)
for shop in candidates:
    route = navi.route(user_loc, shop.coord, mode="driving")
    if route.distance < 3000:
        valid.append((shop, route))
  • 阶段 1:Redis 完成 O(log N) 粗筛, 不依赖外部 API
  • 阶段 2:导航 API 只算 ≤ 50 次, 而非全表
  • 排序:按真实驾车距离排序, 用户体验贴近现实
  • 降级:导航 API 故障时退回 Haversine, 在 UI 提示"估算"
    • - 真实距离展示准确 - 用户体验提升

12.11.4 反例:不在同 slot 的 GEOSEARCHSTORE

错误:cluster 下默认 key 不同 slot

# Redis Cluster 6 节点环境
127.0.0.1:7001> GEOADD stores 116.4 39.9 "shop_A"
(integer) 1

127.0.0.1:7001> GEOSEARCHSTORE result stores \
    FROMLONLAT 116.4 39.9 BYRADIUS 5 km
(error) CROSSSLOT Keys in request don't hash to the
        same slot

# 失败原因:
# - "result" 计算 slot = 5398 (节点 7001)
# - "stores" 计算 slot = 12539 (节点 7003)
# 两 key 不在同节点,store 操作无法跨节点

修复:hash tag 强制同 slot

# 用 {geo} 标签让 slot 计算只看 {} 内的部分
127.0.0.1:7001> GEOADD '{geo}:stores' 116.4 39.9 "shop_A"
(integer) 1

127.0.0.1:7001> GEOSEARCHSTORE '{geo}:result' \
    '{geo}:stores' \
    FROMLONLAT 116.4 39.9 BYRADIUS 5 km
(integer) 1   # ✓ 成功

# 原理:
# - "{geo}:stores" → slot 算 "geo" → 同 slot
# - "{geo}:result" → slot 算 "geo" → 同 slot
# - 两 key 必在同节点,STORE 操作可执行

# 业务约定:同业务的 GEO key 共用 hash tag
# {geo:cn}:beijing, {geo:cn}:shanghai → 同 slot
# {geo:us}:nyc, {geo:us}:la → 另一 slot

12.12.3 geohashEncode 关键代码

/* src/geohash.c — 简化版 */

#define GEO_LAT_MIN  -85.05112878
#define GEO_LAT_MAX   85.05112878
#define GEO_LONG_MIN -180.0
#define GEO_LONG_MAX  180.0

int geohashEncode(const GeoHashRange *long_range,
                  const GeoHashRange *lat_range,
                  double longitude, double latitude,
                  uint8_t step, GeoHashBits *hash) {
    /* 1. 越界检查 */
    if (longitude > 180.0 || longitude < -180.0 ||
        latitude > 85.05112878 || latitude < -85.05112878) {
        return 0;
    }

    hash->bits = 0;
    hash->step = step;

    /* 2. 把经纬度归一化到 [0, 1) */
    double lat_offset =
        (latitude - lat_range->min) /
        (lat_range->max - lat_range->min);
    double long_offset =
        (longitude - long_range->min) /
        (long_range->max - long_range->min);

    /* 3. 缩放到 [0, 2^step) 整数 */
    lat_offset  *= (1ULL << step);
    long_offset *= (1ULL << step);

    /* 4. 位交错: lat 占奇数位,lon 占偶数位 */
    hash->bits = interleave64(lat_offset, long_offset);
    return 1;
}

/* Haversine 实现 (src/geohash_helper.c) */
double geohashGetDistance(double lon1d, double lat1d,
                          double lon2d, double lat2d) {
    double lat1r = deg_rad(lat1d);
    double lat2r = deg_rad(lat2d);
    double u = sin((lat2r - lat1r) / 2);
    double v = sin((deg_rad(lon2d - lon1d)) / 2);
    double a = u * u + cos(lat1r) * cos(lat2r) * v * v;
    return 2.0 * EARTH_RADIUS_IN_METERS *
           asin(sqrt(a));
}

/* ★ 关键不变量:
 *   1. interleave64 的输入必须是已经缩放到 [0, 2^step) 的整数
 *   2. step ≤ 26,否则结果超出 52 位无法装 double
 *   3. Haversine 的 EARTH_RADIUS = 6372797.560856 m (WGS84 平均)
 */

12.12.4 设计取舍

图 12-12-2 · GEO 数据结构选型决策树

取舍 1:9 邻居代替 R-tree

R-tree 提供更紧凑的范围查询(只需 O(log N) 棵子树),但需要额外 O(N) 内存维护索引节点。9 邻居方案完全复用 ZSet 的 skiplist,无额外内存,代价是查询时多扫 ~9× 候选。Redis 选择前者契合"轻量、零额外索引"原则 — 一个 GEO key 与一个普通 ZSet key 在内存占用上完全相同。

取舍 2:Haversine 而非 Vincenty

Haversine 假设球面,Vincenty 用 WGS84 椭球面更精确。Redis 选择 Haversine:1 km 范围内误差 < 1 m,超过 GPS 精度,且 Vincenty 的迭代计算开销(每次 GEODIST 增加 ~100×)在高频 LBS 场景下显著拖累 P99。对绝大多数业务,Haversine 的"足够准 + 100× 快"是更好的选择。

取舍 3:固定 26+26 = 52 位精度

Redis 不开放 Geohash 位数配置:52 位完美装入 double 尾数 52 位,无精度丢失;同时 0.6 m 精度对消费级 LBS 充足。可配置反而增加 PFMERGE 类问题(两 GEO key 合并需位数一致),工程上不划算。这与第 14 章 HyperLogLog 不开放精度配置的逻辑一致 — 把可调参数转化为算法不变量,降低使用复杂度。
