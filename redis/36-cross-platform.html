第 36 章 · 跨平台适配

ae.c 编译期后端选择 · Linux/FreeBSD/macOS/Windows 差异 · fork 与 fsync 语义 · 容器化与 Kubernetes · 跨平台压测护栏

Redis 源码可在 Linux、FreeBSD、macOS、WSL2、Docker 容器内编译运行,但"可编译"与"适合作为生产基线"是两个不同问题。后者由事件循环后端、fork() 行为、文件描述符上限、文件系统 fsync 语义、服务管理器模型共同决定。任一项不匹配,QPS 压测数字均不具备外推价值。

Redis 通过 src/ae.c 在编译期选择事件循环后端(epoll / kqueue / evport / select),热路径不引入运行时多态。该决定与 Linux 的 epoll + fork() + jemalloc + sysctl 组合共同构成生产首选路径。跨平台部署需独立评估每条系统调用的语义差异。

  • "运行时指纹"采集方法:用 INFO server + CONFIG GET dir + df -T + /proc/<pid>/limits 而非平台名称定生产基线
  • ae.c 编译期后端选择机制:HAVE_EPOLL / HAVE_KQUEUE / HAVE_EVPORT 三个宏的优先级
  • 四种后端能力对比:epoll / kqueue / evport / select 的 fd 扩展性与触发模式
  • Linux 生产基线检查清单:内核版本、jemalloc、THP、overcommit、fd 上限、数据盘、systemd
  • macOS 不可作为生产的四条证据:fd 默认 256、无 overcommit、fork 不稳定、kqueue ≠ epoll
  • FreeBSD 作为非 Linux 生产平台的可行路径:kqueue + jemalloc + jail
  • Windows 三条路径:WSL2 / Docker Desktop / Memurai 商业兼容,以及为何停维 Win32 端口不应使用
  • 虚拟化文件系统的 fsync 延迟对比:9p / fuse.osxfs / virtiofs / VM 内 ext4 差距可达 100 倍
  • 跨平台压测最低字段表与"数字可外推"判定规则
  • 容器化 Redis 的 cgroup memory.max 与 maxmemory 关系,以及为何需要 1.5 倍裕量防 fork OOM-kill
  • Kubernetes StatefulSet 部署模式与 Sentinel quorum 的配合
  • 四个被拒绝的跨平台方案:libevent 抽象层、IOCP 后端、运维适配器、线程池隐藏 fd

36.0 章首速览

核心结论: Redis 跨平台不是"一份源码,多平台运行"的均质问题,而是"编译期后端选择 + 平台语义差异"的合成问题。Linux + epoll + jemalloc + sysctl 是唯一被充分验证的生产基线;FreeBSD 是可选生产路径;macOS 仅适合开发;Windows 必须经 WSL2/Docker/商业兼容产品。任何跨平台压测必须记录后端、文件系统、虚拟化层,否则数字无效。
真实事故: 某团队在 macOS 上做了 100K QPS 的 SET 压测,据此评估"3 台 c5.xlarge 足够支撑生产"。上线后第二天 BGSAVE 触发 OOM-kill,RDB 永久失败。根因:macOS Mach 内核的 fork 行为与 Linux vm.overcommit_memory=1 差异显著,峰值内存外推错误。教训:跨平台数字必须标注"源平台 → 目标平台"是否可外推。

36.1 平台风险定位

跨平台适配的第一步不是问"机器是什么系统",而是采集 Redis 实际运行时指纹:事件后端、持久化目录、文件描述符上限、fork 能力、服务管理器。平台名称可能误导(同样是"Linux"可能是裸金属、KVM、WSL2、容器),运行时证据不会。

36.2 ae.c:编译期事件后端选择

Redis 不在运行时引入跨平台事件循环抽象层(如 libevent、libuv),而是在编译期通过预处理宏将单个后端的实现 include 进 ae.c。该设计使热路径仅一层函数指针调用,无运行时多态开销。

/* src/ae.c — 简化版后端选择 */
#ifdef HAVE_EVPORT
    #include "ae_evport.c"     /* Solaris / illumos */
#else
    #ifdef HAVE_EPOLL
        #include "ae_epoll.c"  /* Linux 2.6+ */
    #else
        #ifdef HAVE_KQUEUE
            #include "ae_kqueue.c"  /* FreeBSD / macOS */
        #else
            #include "ae_select.c"  /* POSIX fallback */
        #endif
    #endif
#endif

/* 编译结果:
 * Linux  → ae_epoll.c
 * macOS  → ae_kqueue.c
 * FreeBSD → ae_kqueue.c
 * Solaris → ae_evport.c
 * 其他 POSIX → ae_select.c (受 FD_SETSIZE 限制)
 */

36.3 Linux 生产首选

36.4 macOS:仅适合开发

macOS 默认 maxfiles 限制远低于 Linux,且无 vm.overcommit_memory 等价旋钮。Darwin 内核的 fork 行为在大数据集下表现不稳定,BGSAVE 峰值内存难以预测。macOS 是开发与功能验证平台,不是生产容量评估平台。

36.4.1 macOS 与 Linux 关键差异

36.5 FreeBSD:kqueue 与 jemalloc 的另一条生产路径

FreeBSD 是 Redis 支持较好的非 Linux 平台。kqueue 与 epoll 在 Redis 常见场景的复杂度接近,jemalloc 与 FreeBSD 系统 malloc 共享技术血缘。该平台适合已具备 FreeBSD 运维体系的团队,不建议为单一 Redis 节点临时引入 FreeBSD。

36.7 文件系统陷阱

Windows / macOS 容器与 WSL2 的常见性能问题不在 CPU,而在文件系统路径。Redis 的 RDB/AOF 是高频 fsync 工作负载,数据目录置于宿主机共享路径会显著降低性能。

#!/usr/bin/env bash
# ci/check-redis-fs.sh
set -euo pipefail
dir="$(redis-cli --raw CONFIG GET dir | tail -n 1)"
fs_type="$(df -T "$dir" | awk 'NR==2 {print $2}')"
case "$dir:$fs_type" in
    /mnt/*:*|*:9p|*:fuse.osxfs|*:virtiofs)
        echo "FAIL redis dir=$dir fs=$fs_type is not a production-like data path"
        exit 1
        ;;
    *)
        echo "PASS redis dir=$dir fs=$fs_type"
        ;;
esac

36.10.3 容器内的 fork() 行为

图 36-10-2 · cgroup memory.max 同时限制父子进程总和
容器内 fork OOM 规则: cgroup v2 的 memory.max 同时限制父进程与子进程总内存。Redis fork 期间若父进程 used + 子进程 CoW > cgroup 上限,子进程被 OOM-killer 杀死,RDB 失败。生产应:cgroup memory.max ≥ Redis maxmemory × 1.5。

36.11 源码深潜与设计取舍

跨平台差异不止 ae_* 文件。生产部署时,平台差异会同时穿过网络事件、后台子进程、文件系统同步、进程资源限制四条路径。

36.11.1 跨平台代码的四条路径

图 36-11-1 · 四条代码路径分别封装平台差异
编译期单后端 + 平台条件编译",而非"运行时抽象层"。每条路径都是独立适配,代价是源码维护负担,收益是热路径无运行时开销 + 可针对每个平台做最优优化 ★ 该设计已稳定 15+ 年,接口未变;但代价是 Windows 原生支持事实上被放弃 ★ 对生产用户的启示:Linux 是唯一被四条路径同时充分验证的平台,其他平台总有至少一条路径处于 best-effort 状态

36.11.2 被拒绝的跨平台方案

方案表面优势拒绝原因
libevent / libuv 统一事件层一套 API 覆盖所有平台热路径多一层抽象,Windows IOCP 与 readiness 模型仍不匹配
原生 Windows IOCP 后端Windows 服务直接部署IOCP 是 completion model,与 aeFileProc(fd, mask) 模型差异显著
每平台一套运维适配器安装体验统一systemd / launchd / rc.d / SCM 语义差异由平台包维护
用线程池隐藏 fd / IO 差异表面更"现代"命令执行需串行保护数据结构,线程切换复杂化简单模型

36.11.3 设计取舍

取舍 1:编译期后端选择 vs 运行时插件
ae.c 在编译期 include 单个后端实现,热路径无运行时多态。该选择牺牲了"二进制跨平台",换取事件循环路径极短。Linux 生产二进制必为 epoll,macOS 二进制必为 kqueue,无配置项可切换。
取舍 2:不适配 Windows IOCP
IOCP 模型要求先投递异步 I/O 请求,再通过完成端口收取结果。ae 接口为 readiness 模型(aeFileProc(fd, mask) 在 fd 可读 / 可写时回调)。两者语义差异需要在 IOCP 层包装出虚拟的 readiness 回调,引入额外延迟与状态。Redis 选择不适配,Windows 用户使用 WSL2 / Docker / 商业兼容产品。
取舍 3:不引入跨平台运维抽象
systemd、launchd、rc.d、Windows SCM 在服务依赖、启动顺序、日志、信号处理上语义差异显著。Redis 核心不适配每个 SCM,服务脚本由各平台包维护者(apt、brew、pkg)提供。这一取舍把运维责任精确切割到了平台发行版边界,核心代码保持 POSIX 纯净。
