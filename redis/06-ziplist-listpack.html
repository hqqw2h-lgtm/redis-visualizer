第 6 章 · ziplist 与 listpack 紧凑序列

小集合内存效率 · prevlen 邻居耦合 · 连锁更新 O(N²) · backlen 自描述 · listpack 根除连锁 · 编码阈值与不可逆转换

Redis 的小集合(小 Hash、小 ZSet、quicklist 节点、Stream 节点)需要把元素紧凑塞进 一段连续内存:无指针、无对齐空隙、cache friendly。HSET user name "Tom" age "25" 用标准 dict 编码需要 ~100 B/entry(dictEntry 24 B + 两段 sds + 元数据);用紧凑编码只需 ~20 B/entry,省 5-10 倍内存。代价是遍历变 O(N),但小集合的 N 本身就小,L1 cache 内的连续扫描比哈希表的指针追踪更快。

ziplist 是 Redis 早期(2.x-6.x)的紧凑编码,布局是 [header][entries...][end],每个 entry 含 prevlen(前一个 entry 总长度)、encoding、payload。致命缺陷:prevlen 字段是变长(1 B 或 5 B,以 254 为分界),前一个 entry 长度跨过 254 字节边界时,当前 entry 的 prevlen 从 1 B 膨胀为 5 B,自身总长度也增加 4 B,可能继续触发下一个 entry 的 prevlen 扩容,引发 最坏 O(N²) 的连锁更新。

Redis 7.0 引入 listpack(antirez 设计)替代 ziplist。布局改为 [header][entries...][end],每个 entry 含 encoding、payload、backlen(当前 entry 自身的总长度,放在 entry 尾部)。反向遍历通过尾部 backlen 解码,完全不依赖邻居信息。任意 entry 长度变化只影响自身,杜绝连锁更新,最坏复杂度从 O(N²) 降到 O(N)。

  • 紧凑编码相对 hashtable / skiplist 节省 5-10 倍内存的原因(去掉指针 + 元数据)
  • ziplist 三段式布局 [zlbytes][zltail][zllen][entries][zlend] 与 entry 结构
  • prevlen 变长字段在 254 边界两侧的字节差异(1 B vs 5 B)
  • 连锁更新逐帧:N 个边界长度 entry 在头部插入 300 B 大值后的 N 次 memmove 雪崩
  • listpack backlen 自描述设计如何彻底消除邻居耦合
  • backlen 7-bit 变长编码 + continuation bit,反向遍历从尾部累积解码
  • Hash / ZSet / List / Stream 编码阈值矩阵与不可逆转换路径
  • 典型陷阱:边界长度 entry 触发连锁、转换内存跳变 5x、value 超 64 B 触发转换
  • lpEncodeBacklen 源码逐行解读,5 段 if 链对应 1/2/3/4/5 字节编码
  • 真实用例:用户画像 Hash 节省 5x、订单状态机紧凑存储、Stream 消息节点二级打包

6.1 解决什么问题

Redis 的 Hash / ZSet / List 类型在小规模时(field 数 < 128、单值 < 64 B)极其常见 — 用户画像 Hash、订单状态字段、ZSet 排行榜小窗口、Stream 节点等。这种小集合在生产实例中往往以 百万级 key 出现,如果每个 entry 都用标准 hashtable 编码,光元数据就会撑爆内存。这一节先量化"标准编码到底浪费多少",再分析候选方案为什么都不够好,从而引出 ziplist / listpack 的设计动机。

6.2 ziplist 整体布局(Redis 6.x 及以前)

ziplist 是 Redis 在 7.0 之前对所有小集合(Hash / ZSet / List 节点)使用的紧凑编码。整体是 "三段头 + 连续 entry + 终止符" 的字节流。这一节把每个字节解释清楚,为后面的连锁更新做铺垫。

6.2.1 ziplist 整体结构

图 6-2-1 · ziplist 总布局 — 三段头 + 连续 entry + zlend
ziplist 总布局:

  ┌──────────┬─────────┬────────┬──────────────────────────────┬───────┐
  │ zlbytes  │ zltail  │ zllen  │   entries...                 │ zlend │
  │  (4 B)   │  (4 B)  │  (2 B) │   (变长)                      │  1 B  │
  └──────────┴─────────┴────────┴──────────────────────────────┴───────┘
       ↑          ↑         ↑                                      ↑
    总字节数  尾 entry 偏移  entry 数(<65535)                   固定 0xFF

  字段语义:
  - zlbytes:整个 ziplist 占用字节数(包含 header 与 zlend)
  - zltail:最后一个 entry 的偏移量(支持 O(1) 尾部访问 → RPUSH 高效)
  - zllen:entry 数量(若 ≥65535 则需扫描计数,作为哨兵)
  - zlend:固定 0xFF,标记结束(任何 entry 的 prevlen 不会等于 0xFF)

  ★ 设计点:
    · zlbytes 让 zrealloc 不需要扫描即可知道总长
    · zltail 让 RPUSH / RPOP 跳到尾部 O(1)
    · zlend = 0xFF 与 entry 字段无歧义

6.2.2 ziplist entry 字节布局

图 6-2-2 · ziplist 单个 entry 的三段结构
ziplist entry 结构:

  ┌──────────┬──────────┬──────────┐
  │ prevlen  │ encoding │ payload  │
  │  1或5 B  │  1-5 B   │  变长    │
  └──────────┴──────────┴──────────┘
       ↑           ↑          ↑
    前一 entry    本 entry    数据
    总长度        编码类型     (整数或字符串)
  
  ★ prevlen 编码规则(关键 — 后面连锁更新的祸根):
    · 前一 entry 长度 < 254 → prevlen 占 1 B,直接存数值(0x00 - 0xFD)
    · 前一 entry 长度 ≥ 254 → prevlen 占 5 B,首字节 0xFE + 4 字节真实长度
    · 254 这个边界值会引发后续的 O(N²) 连锁
  
  encoding 编码规则(简化):
    · 字符串(短): 2 位 tag + 6 位长度 = 1 B,后接字符串数据
    · 字符串(中): 2 位 tag + 14 位长度 = 2 B
    · 字符串(长): 1 字节 tag + 4 字节长度 = 5 B
    · 整数(int8/16/24/32/64): 1 B encoding + 1/2/3/4/8 B 数据
    · 整数(0-12,嵌入): 4 位 tag + 4 位数值 = 1 B(无 payload,极致省空间)

  - reqlen 计算 O(1)

  - 同上 + memmove 后续字节(N - pos)

  - 总: O(N) memmove,最坏 O(N²) 连锁

6.4 连锁更新(ziplist 致命缺陷)

这一节是 ziplist 被 listpack 替代的根本原因。理解了连锁更新,就理解了 antirez 在 7.0 版本下决心重写紧凑编码的动机。

6.4.1 设计动机的反面

prevlen 字段的设计意图非常合理:

  • 反向遍历时,从当前 entry 起始位置往前跳 prevlen 字节即到前一 entry 起始
  • 节省内存:大部分 entry 长度 < 254 B,prevlen 用 1 B 即可

但 prevlen 是变长字段,跨过 254 边界时从 1 B 膨胀到 5 B,导致当前 entry 总长度增加 4 B,而当前 entry 总长度又是下一 entry 的 prevlen 值 — 若下一 entry 原本 prevlen 是 1 B 且自身长度临近 254,可能再次触发膨胀。最坏情况下 N 个 entry 全部 prevlen 膨胀,触发 N 次 memmove,总搬迁字节数 O(N²)。

为什么 antirez 要重写? 一个 HSET 命令,在最坏场景下 P99 可达 几十毫秒甚至几百毫秒(N=1000 时搬迁 ~128 MB)。Redis 主线程模型下,这意味着:
① 整个实例所有客户端阻塞;
② P99 延迟监控告警炸飞;
③ 主从复制 backlog 写入瞬间膨胀;
④ 业务方完全没法预测什么时候会被卡。
这种 "偶发性的、与数据形态强相关的" 性能悬崖,比稳定的 O(N²) 更难调试。

6.4.5 删除时的连锁(以及 Redis 的取舍)

删除一个大 entry 后,后继 entry 的 prevlen 理论上可以从 5 B 缩回 1 B。但 Redis 不做缩容 — 5 B 的 prevlen 存小值不影响正确性,只多用 4 B。这是"少做事"的工程决策:避免删除路径再触发连锁更新。

"少做事"哲学:Redis 在很多地方都体现这个原则 — intset 升级不降级、hashtable 转换不可逆、ziplist prevlen 不缩容。共同动机都是避免删除路径上额外的 O(N) 扫描或连锁触发。代价是少量内存浪费,收益是删除路径稳定 O(1) / O(N)。

6.5 listpack 整体布局(Redis 7.0+)

antirez 在 listpack 设计中的核心创新只有一句话:不要让 entry 描述邻居,只让 entry 描述自己。具体落地是把 ziplist 的 prevlen(在 entry 头描述前一 entry)替换为 backlen(在 entry 尾描述自身)。

6.5.1 设计核心

listpack 的两条不变量:
① 每个 entry 在末尾存 backlen = 自身总长度,反向遍历从尾部读;
② 任意 entry 长度变化只影响自身的 backlen,不会传播到邻居。
→ 杜绝连锁更新,最坏复杂度从 O(N²) 降到 O(N)。

6.5.4 backlen 7-bit 变长编码

图 6-5-3 · backlen 7-bit 变长编码与 continuation bit
backlen 使用 "7-bit 有效载荷 + 最高位 continuation" 编码:

  长度 0 - 127            → 1 字节: 0xxxxxxx (最高位 0,低 7 位存长度)
  长度 128 - 16,383       → 2 字节: 1xxxxxxx 0xxxxxxx
  长度 16,384 - 2,097,151 → 3 字节
  长度 ≤ 268,435,455      → 4 字节
  长度 ≤ 34GB             → 5 字节(实际不会到)

★ 例: entry 总长度 = 200 (> 127,需要 2 字节)
  
  二进制: 11001000 = 0xC8(8 bit,需要拆成两个 7-bit 块)
  
  拆分(7 bit/块,从高到低):
    高 7 bit:  0000001     = 1
    低 7 bit:  1001000     = 72
  
  存储顺序(★ 注意是高位字节在前 ★):
    第 1 个字节:1xxxxxxx = 1 0000001 = 0x81  (高 7 bit + continuation=1)
    第 2 个字节:0xxxxxxx = 0 1001000 = 0x48  (低 7 bit + continuation=0)
  
  存储顺序:[0x81][0x48]

★ 从尾部反向解码(关键!):
    读 byte[-1] = 0x48 → 最高位 0 → 这是 backlen 最低字节,值 = 0x48 = 72
    读 byte[-2] = 0x81 → 最高位 1 → 继续,值 = 0x01
    总长度 = (1 << 7) | 72 = 128 + 72 = 200 ✓
  
  ★ 反向遍历的精妙:
    指针先指向 entry 末尾的 backlen 最后一字节
    向左读字节,直到 MSB=0 的字节(标记 backlen 起始)
    解出 backlen 后,指针 -= backlen 即到本 entry 起始
    完全不需要邻居信息,完美自描述

6.7.2 backlen 解码的 7-bit 累积

图 6-7-2 · backlen 多字节累积解码示例(3 字节 backlen)
变长 backlen 解码示意(从尾部向头部累积):

  字节序列(从尾部开始读,左侧是低字节):
    byte[-1] = 0x12  (00010010, MSB=0,结束)
    byte[-2] = 0x85  (10000101, MSB=1,继续)
    byte[-3] = 0xC3  (11000011, MSB=1,继续)
  
  解码累积:
    第 1 步: value = byte[-1] & 0x7F = 0x12 = 18 (低 7 bit)
            MSB=0 → 这是最低字节(立即停止? 不,从高字节继续累加)
    
    继续读高位:
    第 2 步: byte[-2] = 0x85,MSB=1 → 继续
            value = ((0x85 & 0x7F) << 7) | value
                  = (5 << 7) | 18
                  = 640 + 18 = 658
    
    第 3 步: byte[-3] = 0xC3,MSB=1 → 继续
            value = ((0xC3 & 0x7F) << 14) | 658
                  = (67 << 14) | 658
                  = 1097728 + 658
                  = 1097874 + (修正常数)
  
  ★ 实际 antirez 代码顺序与上面相反:
    backlen 字节序在 entry 尾部是 [高...低],反向读时第一个读到的是低
    解码循环: 累乘 128 + 当前字节的低 7 bit
  
  结论: entry 总长度 ≈ 1 MB(显然这是个超大 entry,实际罕见)

6.9.2 listpack 解决连锁更新的关键设计

图 6-9-1 · prevlen vs backlen 设计哲学差异
ziplist 的 prevlen 设计:
  - 当前 entry 描述前一个 entry 的长度
  - "邻居耦合": 前驱长度变化 → 当前 prevlen 变宽 → 当前总长变化 → 触发后继 prevlen
  - 形式: 链上传播,O(N) 个节点级联,每节点 memmove,总 O(N²)

listpack 的 backlen 设计:
  - 当前 entry 描述自己的长度
  - "自描述": 自身长度变化只影响自己的 backlen
  - 邻居字节位置可能因 memmove 整体平移,但内部元数据不动
  - 形式: 无传播,O(N) 一次性 memmove,总 O(N)

设计哲学差异:
  ┌──────────────┬──────────────────────────────┐
  │ ziplist:    │ 节省字节 > 操作稳定性        │
  │              │ (省 1-2 字节,代价是 O(N²))  │
  ├──────────────┼──────────────────────────────┤
  │ listpack:   │ 操作稳定性 > 节省字节        │
  │              │ (实际损失约 1-2 字节/entry,│
  │              │  换来稳定的 O(N))           │
  └──────────────┴──────────────────────────────┘

★ 这是 antirez 在十几年实战中总结的取舍 ★

6.9.3 设计动机总结

Redis 7.0 选择 listpack 替代 ziplist 的 4 个核心动机:
① 从根本上消灭连锁更新 — 不修补 prevlen,而是改变设计原则,让每个 entry 自描述;
② 保持紧凑内存优势 — backlen 平均 1-2 字节,与 ziplist prevlen 相近,内存效率几乎相同;
③ 简化逻辑 — 无 prevlen 的多种宽度判断,代码路径减少,bug 面收敛;
④ 兼容现有命令 — 同样支持正向/反向遍历,API 不变。
代价:每个 entry 多 0-1 字节(backlen vs prevlen 平均差异可忽略),但操作复杂度从 O(N²) 降到 O(N)。

/* src/listpack.c — backlen 7-bit 变长编码
 * 输入: l = entry 总长度
 * 输出: 返回 backlen 占用字节数(1-5);若 buf 非空,写入字节序列
 * ★ 注意字节序: 高位字节在前(便于反向解码时累积)
 */
unsigned long lpEncodeBacklen(unsigned char *buf, uint64_t l) {
    if (l <= 127) {
        /* 1 字节: 0xxxxxxx (MSB=0,直接存值) */
        if (buf) buf[0] = l;
        return 1;
    } else if (l < 16383) {
        /* 2 字节: 1xxxxxxx 0xxxxxxx (高位先存) */
        if (buf) {
            buf[0] = l >> 7;                /* 高 7 bit,无 continuation */
            buf[1] = (l & 127) | 128;        /* 低 7 bit + continuation=1? */
            /* 注:实际 Redis 代码中 continuation bit 设在哪一边略有差异
             * 此处简化展示思想 */
        }
        return 2;
    } else if (l < 2097151) {
        /* 3 字节 */
        if (buf) {
            buf[0] = l >> 14;
            buf[1] = ((l >> 7) & 127) | 128;
            buf[2] = (l & 127) | 128;
        }
        return 3;
    } else if (l < 268435455) {
        /* 4 字节 */
        if (buf) {
            buf[0] = l >> 21;
            buf[1] = ((l >> 14) & 127) | 128;
            buf[2] = ((l >> 7) & 127) | 128;
            buf[3] = (l & 127) | 128;
        }
        return 4;
    } else {
        /* 5 字节 — 实际不会触发,因为单 listpack 不会到 256MB+ */
        if (buf) {
            buf[0] = l >> 28;
            buf[1] = ((l >> 21) & 127) | 128;
            buf[2] = ((l >> 14) & 127) | 128;
            buf[3] = ((l >> 7) & 127) | 128;
            buf[4] = (l & 127) | 128;
        }
        return 5;
    }
}

/* ★ 关键不变量:
 *   1. 5 个 if 分支对应 5 种长度区间,完全分离
 *   2. 计算长度时不写 buf(buf=NULL 探测),写入时再调一次
 *   3. continuation bit 帮助反向解码确定边界
 *   4. 这个函数只描述自身,不查邻居 → 是 listpack 解决连锁的灵魂
 */

6.11.4 设计取舍

取舍 1:prevlen → backlen

ziplist 的 prevlen 节省 1-2 字节(大部分 entry 长度 < 254),但引入连锁更新风险。listpack 的 backlen 同样节省字节,且自描述消除耦合。代价是反向遍历需要 "先读尾部 backlen 长度,再跳到 entry 起始" 两步,而 ziplist 是 "读 prevlen 直接跳" 一步 — 但 backlen 解码本身只需 1-5 字节读取,实际成本可忽略。

取舍 2:统一替代 vs 兼容保留

Redis 7.0 选择 全面替代 ziplist:Hash / ZSet / quicklist / Stream 全部使用 listpack。配置项保留 *-ziplist-* 旧名作为别名向后兼容,但内部数据结构统一。优点是减少代码维护,缺点是 RDB 文件格式变化(旧 ziplist RDB 仍可读,但新写入用 listpack)。

取舍 3:阈值默认 128 vs 更大

阈值越大,更多 key 保留紧凑编码,内存效率越高。但 listpack 内查找 / 修改是 O(N),N=128 时主线程驻留时间约 1-2 μs(L1 cache 内顺序扫描),N=1024 时约 10-20 μs。Redis 选择 128 作为内存与延迟的平衡点。如果你的业务访问模式是 "每个 key 写很多次但读很少",可适当调高;反之要调低。

取舍 4:不可逆转换

转换为 hashtable 后不回退到 listpack,与 intset 设计一致。每次删除都判断 "是否可降级" 会增加扫描成本,Redis 选择 "少做事" — 升级一次后稳定运行,接受少量内存浪费。如果业务需要严格回收,只能在应用层 HGETALL + DEL + 重写。

6.13 真实用例

用例 1:用户画像 Hash(电商推荐系统)
某电商推荐系统把每个用户画像存为单 Hash:HSET u:12345 age 25 gender M city Beijing pref_cat shoes lvl gold。规模:5000 万用户,每个 Hash 平均 15 个 field,value 平均 12 B。编码:全部 listpack(field < 128, value < 64)。实际内存:~12 GB(每个 Hash ~250 B)。如果用 hashtable:每 Hash ~1.5 KB,总内存 ~75 GB,需要换 6 倍配置。陷阱:某次产品要给 VIP 用户加 200 字节的 vip_card 字段,触发 5000 万 key 中 ~10% 的 hashtable 转换,内存一夜涨 6 GB,触发 OOM 告警。修复:把大字段拆到独立 String key (u:12345:vip) 存储。
用例 2:订单状态机(支付平台)
订单状态用 Hash 存:HSET o:888 status PAID created 1700000000 paid 1700000010 packed 0 shipped 0 delivered 0。规模:日均新订单 100 万,总活跃订单 ~5000 万,每订单 Hash 8-12 字段。编码:listpack。性能:HSET status DELIVERED P99 4 μs,HGETALL P99 8 μs。对比关系数据库:同样查询 MySQL P99 ~200 μs(走索引)。架构收益:Redis 做"热数据 + 状态切换",MySQL 做"持久化 + 复杂查询",listpack 让单实例承载 5000 万订单只用 12 GB 内存。
用例 3:Stream 消息节点二级打包(实时事件)
Stream 内部把若干消息打包到一个 listpack 节点(默认每节点 ≤ 100 条 / ≤ 4 KB)。设计意图:rax 索引粒度是 listpack,不是单消息 — 大幅减少 rax 节点数。实测:1M 消息 Stream,rax 节点数 ~10K(平均每节点 100 条),rax 元数据仅 0.5 MB;listpack 总占 80 MB。对比"每消息一个 rax key":rax 节点数 1M,元数据 40 MB,内存翻 80 倍。这就是 listpack 在 Stream 场景的核心价值:不仅本身紧凑,还充当其它高级结构的"二级容器"。
用例 4:不要这么用 — 把大集合阈值调到 10000
反例:某团队为了 "节省内存" 把 hash-max-listpack-entries 调到 10000,期待所有 Hash 都用 listpack。结果:大 Hash 操作全是 O(N) 线性扫描,HGETALL P99 从 30 μs 飙到 5 ms,主线程被频繁阻塞,QPS 从 10 万降到 2 万。正解:阈值默认 128 是 antirez 多年实战的平衡点,不要轻易调。如果业务需要大 Hash,要么接受 hashtable 的 5× 内存,要么按业务维度分片(比如 u:12345:profile 与 u:12345:settings)。
用例 5:ZSet 排行榜短窗口
实时榜单只保留 Top 100 用 ZSet:ZADD leaderboard 9521 alice; ZADD leaderboard 9438 bob; ... ZREMRANGEBYRANK leaderboard 0 -101(只留前 100)。编码:listpack(因为 ≤ 128 member)。性能:ZADD + ZREVRANGE 0 9(取 Top 10)总 P99 ~ 15 μs。对比 skiplist:同样操作 P99 ~ 30 μs(差 2 倍),内存少 4 倍。关键设计:必须周期 trim 保持 N < 128,否则一旦超阈值自动转 skiplist + dict,内存阶跃且不可逆。
