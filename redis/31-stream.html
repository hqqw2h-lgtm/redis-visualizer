第 31 章 · Stream 消费者组与可靠消息

XREADGROUP 协作消费 · 双层 PEL 共享指针 · XCLAIM/XAUTOCLAIM 死亡恢复 · at-least-once 语义 · XTRIM 与孤儿 NACK · 部署边界 · 性能基线

Redis Stream 在数据结构层面是 rax 树索引的 listpack 追加日志(详见第 8 章 rax-stream),其工程价值核心在消费者组(Consumer Group)与 PEL(Pending Entries List)机制。XREADGROUP + XACK 构成 at-least-once 投递语义:消费者从组级 last_id 之后读取新消息,服务端为每条投递创建 streamNACK 节点登记到组 PEL 与消费者 PEL 两层 rax,业务确认后通过 XACK 从两层 PEL 删除;崩溃消费者留下的 NACK 通过 XCLAIM/XAUTOCLAIM 转移所有权,实现死亡恢复。

Redis Stream 与 Kafka 的核心差异不在持久化(均支持),而在消费者组治理:Kafka 自动 rebalance,Redis 拒绝自动重分配并将该责任推给应用层 XAUTOCLAIM 巡检。该取舍换取了更简单的内部实现与零停顿延迟,代价是动态扩缩容场景需应用层补偿。Stream 不能等同于 Kafka 的小型替代品,而是单 key/单节点为核心的轻量可靠日志。

  • XREAD 广播 vs XREADGROUP 协作两种语义的差异,以及消费者组提供的四项能力(水平扩展、可靠投递、死亡恢复、进度追踪)
  • stream / streamCG / streamConsumer / streamNACK 四层结构以及双层 rax 共享指针的内存布局

31.0 章首速览

核心结论: 消费者组是 Stream 的真正"杀手锏"——把"消息存储"与"消费状态追踪"在服务端原生合一。双层 PEL 共享指针让"组内总未确认"与"某消费者未确认"都能 O(log N) 查询;XACK 只删 PEL 不删消息体使多组共享同一日志;Redis 拒绝自动 rebalance换来零停顿,代价是 XAUTOCLAIM 巡检责任在应用层。投递语义最强只能达到 at-least-once,业务必须用 message id 作幂等键。
真实事故: 某团队部署订单消费者组,峰值 5 万 msg/s。一台消费者实例 OOM 后未重启,持有的 12 万 NACK 滞留在 PEL。一周后内存告警,排查发现 XPENDING 显示该消费者 idle 7 天。根因:未部署 XAUTOCLAIM 巡检,崩溃消费者的 NACK 永久滞留。教训:消费者组的死亡恢复责任 100% 在应用层,部署后必须配套巡检任务,同时设置 delivery_count 阈值与 DLQ 防止毒消息循环。

31.1 消费者组解决的问题

Redis Stream 提供两种消费模式:XREAD 是广播语义,所有客户端独立读取所有消息;XREADGROUP 是协作语义,每条消息仅投递给组内一个消费者。消费者组是 Stream 用于"任务队列"场景的唯一正确接口。

31.2 核心数据结构

Stream 消费者组由四层结构组合而成:stream 持有消息日志与 cgroups rax;streamCG 持有组级 last_id 与 group PEL;streamConsumer 持有消费者级 PEL;streamNACK 是每条未确认消息的元数据对象,被两层 PEL 通过共享指针索引。

/* stream.h */
typedef struct stream {
    rax *rax;                   /* 消息存储: id → listpack */
    uint64_t length;            /* 消息总数 */
    streamID last_id;           /* 最新消息 ID */
    streamID first_id;          /* 最早消息 ID */
    streamID max_deleted_entry_id;
    uint64_t entries_added;
    rax *cgroups;               /* 消费者组: name → streamCG */
} stream;

typedef struct streamCG {
    streamID last_id;           /* 该组已派发的最大 ID */
    long long entries_read;     /* 已读消息数 */
    rax *pel;                   /* 组 PEL: id → streamNACK */
    rax *consumers;             /* name → streamConsumer */
} streamCG;

typedef struct streamConsumer {
    mstime_t seen_time;         /* 最后 XREADGROUP 时间 */
    mstime_t active_time;       /* 最后活跃 (取消息或 XACK) */
    sds name;
    rax *pel;                   /* 消费者 PEL: id → 同 NACK 指针 */
} streamConsumer;

typedef struct streamNACK {
    mstime_t delivery_time;     /* 最后投递时间 */
    uint64_t delivery_count;    /* 累计投递次数 */
    streamConsumer *consumer;   /* 当前所有者 */
} streamNACK;

31.4 PEL 机制深入

streamNACK 是 PEL 的最小单元,固定 24 字节:8 字节 delivery_time、8 字节 delivery_count、8 字节 consumer* 指针。每次重投递更新 delivery_time 与 delivery_count,XCLAIM 修改 consumer 指针。

图 31-4-1 · streamNACK 字段详解

PEL 的 rax 索引

group.pel 与 consumer.pel 都是 rax 树,key 都是 12 字节 streamID(8 字节 ms + 4 字节 seq),value 是 streamNACK* 指针。两层指针指向同一对象,XACK 仅释放对象一次。

XACK 实现

/* t_stream.c - 简化 */
void xackCommand(client *c) {
    int acknowledged = 0;
    streamCG *group = streamLookupCG(stream, group_name);

    for (int j = 3; j < c->argc; j++) {
        streamID id;
        streamParseStrictIDOrReply(c, c->argv[j], &id, 0, NULL);
        unsigned char buf[sizeof(streamID)];
        streamEncodeID(buf, &id);

        /* 查找组 PEL */
        streamNACK *nack = raxFind(group->pel, buf, sizeof(buf));
        if (nack == raxNotFound) continue;

        /* 从消费者 PEL 删除 */
        raxRemove(nack->consumer->pel, buf, sizeof(buf), NULL);
        /* 从组 PEL 删除 */
        raxRemove(group->pel, buf, sizeof(buf), NULL);
        /* 释放 NACK 对象 */
        streamFreeNACK(nack);
        acknowledged++;
    }
    addReplyLongLong(c, acknowledged);
}
XACK 语义陷阱: XACK 不是"删除消息",而是"删除消息的未确认债务"。消息体在 stream.rax 中保留(由 XTRIM 控制),group.last_id 也不被 XACK 推进(由 XREADGROUP > 推进)。即使消息体已被 trim,XACK 仍能成功——因为 PEL 中有索引。

31.5 死亡恢复:XCLAIM 与 XAUTOCLAIM

消费者崩溃后,其 PEL 中的 NACK 没有任何自然路径被释放——这是 Redis Stream 与 Kafka 的最关键差异。Redis 把这个责任完全推给应用层,通过 XCLAIM 与 XAUTOCLAIM 主动转移所有权。

图 31-5-1 · 崩溃消费者的 PEL 滞留

XCLAIM 的 NACK 转移

# 接管 idle 超 1 小时的消息给 consumer C
XCLAIM mystream order-grp C 3600000 1700000000000-4 1700000000000-5

# 完整语法
XCLAIM key group consumer min-idle-time id [id ...]
  [IDLE ms]              # 设置新的 idle (默认重置为 0)
  [TIME ms-unix-time]    # 设置新的 delivery_time
  [RETRYCOUNT count]     # 设置 delivery_count
  [FORCE]                # ID 不在 PEL 时也创建 NACK
  [JUSTID]               # 仅返回 ID,不返回消息体

图 31-5-2 · XCLAIM 转移的 6 步操作

XCLAIM 的 6 步实现

步骤操作复杂度

# 推荐巡检模板
def autoclaim_loop(redis_cli, group, recovery_consumer):
    while True:
        cursor = "0-0"
        while True:
            cursor, claimed, deleted = redis_cli.xautoclaim(
                "order:stream", group, recovery_consumer,
                min_idle_time=300000,  # 5 分钟
                start=cursor, count=100)
            for msg_id, fields in claimed:
                process_with_idempotency(msg_id, fields)
                redis_cli.xack("order:stream", group, msg_id)
            # 清理孤儿 NACK (消息体已 trim)
            for msg_id in deleted:
                redis_cli.xack("order:stream", group, msg_id)
            if cursor == "0-0":
                break
        time.sleep(30)  # 每 30 秒一轮

31.6 投递语义

消息中间件的三种投递语义在 Redis Stream 中的实现路径完全不同:at-least-once 是默认且推荐;at-most-once 易丢消息不推荐;exactly-once 在不引入 2PC 的前提下不可实现。

图 31-6-1 · 三种语义的实现路径

定理 31.6.1 · 不可实现性证明

命题: 在不引入跨组件原子提交(2PC)的前提下,Redis Stream 不能保证 exactly-once 投递。

证明: 设崩溃事件 X 可发生在 D, P, A 之间任意时点。
情形 1:D 完成,P 未开始时崩溃 → 重启后消费者重读 PEL → P 重复执行
情形 2:D, P 完成,A 未开始时崩溃 → 重启后重读 PEL → P 重复执行
情形 3:D, P, A 全完成 → 无重复
情形 1 和 2 中,P 至少执行一次但可能两次 ⇒ at-least-once
反证:若希望 exactly-once,必须 P 与 A 原子提交 → 需 2PC → Redis 无原生支持。 ∎

推论: 业务层必须用 message id 作幂等键来"模拟"exactly-once。 
# 推荐模板: 用 message id 作幂等键
def consume(redis_cli, db):
    while True:
        msgs = redis_cli.xreadgroup(
            "order-grp", "worker-a",
            {"order:stream": ">"},
            count=10, block=5000)

        for stream, entries in msgs:
            for msg_id, fields in entries:
                try:
                    # 1. 检查幂等键
                    if not redis_cli.set(
                            f"processed:{msg_id}", 1,
                            ex=86400, nx=True):
                        redis_cli.xack("order:stream", "order-grp", msg_id)
                        continue

                    # 2. 业务处理 (INSERT ON CONFLICT 双保险)
                    db.execute(
                        "INSERT INTO orders (id, ...) VALUES (?, ...)"
                        " ON CONFLICT (id) DO NOTHING",
                        (fields["orderId"], ...))

                    # 3. 确认
                    redis_cli.xack("order:stream", "order-grp", msg_id)

                except TemporaryError:
                    pass  # 不 XACK, 等待重投递
                except PermanentError:
                    # 进入 DLQ
                    redis_cli.xadd("order:dlq", {"orig_id": msg_id, ...})
                    redis_cli.xack("order:stream", "order-grp", msg_id)

31.7 XTRIM 与内存控制

Stream 是只追加日志,XADD 不断推进 last_id 但不主动释放老消息。不裁剪即 OOM。XTRIM 提供 MAXLEN / MINID 两种维度,各有精确 / 近似(~)两档,共四种策略。

✓ 高吞吐:顺序磁盘 I/O,集群水平扩展
✓ 严格分区有序
✓ 长期保留(天/周/月)

✗ 重启慢:大数据量加载
✗ 复杂运维
Redis Stream 选择
✓ 简单:无 rebalance,部署即用
✓ 低延迟:内存操作,无磁盘 I/O
✓ 与 Redis 其他能力共用(Pub/Sub, Lua)
✓ 灵活的消费者管理
✗ 单节点容量受限
✗ 无原生跨 key 消费者组
✗ 应用层负责巡检与扩缩
适用边界: Redis Stream 适合中小规模(< 10 万 msg/s,< 数十 GB 数据);Kafka 适合大规模(百万级 msg/s,持久日志,跨数据中心)。把 Stream 当作 Kafka 替代品是常见误解,两者架构假设完全不同。

/* t_stream.c · XACK 双层 PEL 删除 */
int streamDelConsumerGroupPEL(streamCG *group, streamID *id) {
    unsigned char buf[sizeof(streamID)];
    streamEncodeID(buf, id);

    streamNACK *nack = raxFind(group->pel, buf, sizeof(buf));
    if (nack == raxNotFound) return 0;

    raxRemove(nack->consumer->pel, buf, sizeof(buf), NULL);
    raxRemove(group->pel, buf, sizeof(buf), NULL);
    streamFreeNACK(nack);
    return 1;
}

/* t_stream.c · XCLAIM NACK 转移核心 */
void claimNACK(streamNACK *nack, streamConsumer *new_consumer,
               streamID *id, mstime_t now) {
    unsigned char buf[sizeof(streamID)];
    streamEncodeID(buf, id);

    if (nack->consumer != new_consumer) {
        raxRemove(nack->consumer->pel, buf, sizeof(buf), NULL);
        raxInsert(new_consumer->pel, buf, sizeof(buf), nack, NULL);
    }

    nack->consumer = new_consumer;
    nack->delivery_time = now;
    nack->delivery_count++;
    /* group.pel 索引保持不变 (key 未变) */
}

设计取舍

Redis Stream 的双层 PEL 是空间换时间的精确权衡。若仅维护 group.pel,XPENDING 按消费者过滤需 O(group_pel_size) 遍历;若仅维护 consumer.pel 集合,XPENDING summary 与 NACK 总量查询需累加所有消费者。共享指针的双层 rax 让两类查询都为 O(log N),代价是每条 NACK 多一个 rax 节点(~64 字节)。该设计与数据库覆盖索引的多列检索同构,选择最优查询路径而非压缩存储。

Redis 拒绝实现 rebalance 是与 Kafka 形成的关键架构差异。Kafka rebalance 在消费者增减时自动重新分配 partition,代价是 stop-the-world 停顿(合作 rebalance 改善但未消除)。Redis 选择"消费者各自处理 PEL,新消费者只处理新消息,崩溃消费者 PEL 由 XAUTOCLAIM 接管",代码复杂度极低,无任何停顿,但负载均衡责任推给应用层。该取舍在中小规模(< 10 万 msg/s)是务实选择;大规模高吞吐场景应选择 Kafka 等专用日志系统。

XACK 不删除消息体而仅删除 PEL 索引,是 Stream 与传统消息队列的核心差异。传统 MQ 的 ACK 表示"消息可从队列移除";Stream 的 XACK 仅表示"消费组对此消息的债务清算"。消息体由 XTRIM 独立控制,允许多个消费者组共享同一日志,各自有独立的 PEL 与 last_id。该设计将"日志保留"与"消费状态"解耦,但引入孤儿 NACK 风险(XTRIM 后 PEL 残留),需要 XAUTOCLAIM 第三返回值或应用层主动清理。

XAUTOCLAIM 在 Redis 6.2 引入是对 XCLAIM 工程实践的合并。原始 XCLAIM 需要先 XPENDING 查询 idle 超阈值的 ID,再传给 XCLAIM 转移所有权,两次往返且需客户端追踪状态。XAUTOCLAIM 内置游标式扫描,单次往返完成"扫描 + 转移",并在 7.0 增加孤儿 NACK 报告。该 API 演进体现了 Redis 对生产可观测性的持续投入:从提供原语到提供成品工具。
