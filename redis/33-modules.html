第 33 章 · Modules API

.so 共享库 · ~400 个 RedisModule_* · 自定义命令/类型/阻塞 · in-process 同体 · RediSearch / RedisJSON / Vector Sets

Redis 4.0 引入的 Modules API 允许通过 C 编写共享库(.so),在不修改 Redis 源码的前提下扩展自定义命令、数据类型与生命周期钩子。RediSearch、RedisJSON、RedisTimeSeries、RedisBloom 等大型功能模块均基于此实现,这一机制把 Redis 从"数据结构服务器"升级为可编程数据平台。

Modules API 是经典的"稳定接口 + 不稳定实现"设计模式,与 Linux 系统调用、PostgreSQL Extensions、Nginx 模块同构。共同收益是低延迟与原生数据访问(in-process,无 IPC),共同代价是崩溃同体:模块崩溃即宿主崩溃。约 400 个 RedisModule_* 前缀函数构成稳定 ABI 边界,模块只能通过这一边界访问 keyspace,不应直接依赖内部 dict/sds/robj 等结构。

  • Modules 解决"无模块时只能 Fork 源码 / Lua / 客户端实现"三条路径的局限
  • API 边界设计:不透明句柄(RedisModuleKey/String/CallReply)隔离内部 dict/sds/robj
  • 四阶段生命周期 Loaded → Registered → Serving → Unloading 与 OnLoad/OnUnload 模板
  • CreateCommand 的 9 个常用 flag:write/readonly/fast/deny-oom/allow-loading 等
  • 命令复制三策略:ReplicateVerbatim / Replicate 多条等价 / 不复制
  • CreateDataType 的 9 字节类型名 + encver 版本管理,RDB 序列化/反序列化模板
  • BlockClient + 后台线程 + UnblockClient 的阻塞命令时序,以及"后台线程红线"
  • 键空间事件(SET/DEL/EXPIRED/EVICTED)订阅 + Cluster 消息(scatter-gather)
  • 主流模块巡礼:RediSearch / RedisJSON / RedisTimeSeries / RedisBloom / Vector Sets
  • API 边界开销公式 η = C_api / (C_native + C_api) 与"何时该用模块"的判定
  • Cluster 模式下模块的版本一致性约束 + 滚动升级 / 蓝绿部署的三种方案
  • 模块崩溃的工程缓解:ASAN/Valgrind/Fuzz + 部署隔离 + Sentinel failover + 资源限制

前置知识:C 语言与 dlopen 动态链接;Redis 命令执行流程(参见第 6 章);第 17/18 章 RDB/AOF 序列化;第 25-27 章主从/Sentinel/Cluster;第 29 章 Lua(对比模块);pthread 多线程编程基础。

33.0 章首速览

一句话总览。客户端发出 FT.SEARCH/JSON.GET/MYMOD.* 类的命令,经 Redis 命令分发器分派到模块的回调函数;回调通过 RedisModule_* API 边界访问 keyspace、event loop、RDB/AOF、ACL,模块负责复制策略与自定义类型的序列化。整套机制是 in-process 同进程加载,零 IPC,崩溃同体。

33.2 架构与 API 边界

Modules 的核心设计是用不透明句柄隔离内部结构。模块只看到 RedisModuleKey/RedisModuleString/RedisModuleCallReply,看不到 dict/sds/robj/rax。

API 边界的同构。

  • Modules API ↔ Linux syscall:用户态程序 ↔ 模块 .so;syscall ↔ RedisModule_*;内核 dict ↔ Redis 内部 dict;EFAULT ↔ REDISMODULE_ERR。共同意图:用户代码不直接操作内核数据,通过稳定接口隔离实现细节。
  • Modules API ↔ PostgreSQL Extensions:CREATE EXTENSION ↔ MODULE LOAD;PG_FUNCTION_INFO_V1 ↔ CreateCommand;CREATE TYPE ↔ CreateDataType;SPI_execute ↔ RedisModule_Call。共同特点:in-process 同进程、崩溃互相影响、版本管理用户负责。

33.3 模块生命周期

图 33-3-1 · 四阶段生命周期 — Load → Register → Serve → Unload

#include "redismodule.h"

int RedisModule_OnLoad(RedisModuleCtx *ctx,
                       RedisModuleString **argv, int argc) {
    /* 1. 模块身份声明 */
    if (RedisModule_Init(ctx, "mymodule", 1, REDISMODULE_APIVER_1)
        == REDISMODULE_ERR) return REDISMODULE_ERR;

    /* 2. 注册自定义命令 */
    if (RedisModule_CreateCommand(ctx, "mymodule.set",
            MySetCommand, "write deny-oom",
            1, 1, 1) == REDISMODULE_ERR) return REDISMODULE_ERR;

    /* 3. 注册自定义数据类型 (可选) */
    static RedisModuleType *MyDataType;
    RedisModuleTypeMethods tm = {
        .version     = REDISMODULE_TYPE_METHOD_VERSION,
        .rdb_load    = MyDataTypeRdbLoad,
        .rdb_save    = MyDataTypeRdbSave,
        .aof_rewrite = MyDataTypeAofRewrite,
        .free        = MyDataTypeFree,
        .mem_usage   = MyDataTypeMemUsage,
    };
    MyDataType = RedisModule_CreateDataType(ctx, "mydata-ty", 0, &tm);
    if (MyDataType == NULL) return REDISMODULE_ERR;

    /* 4. 订阅事件 (可选) */
    RedisModule_SubscribeToKeyspaceEvents(ctx,
        REDISMODULE_NOTIFY_SET | REDISMODULE_NOTIFY_DEL, OnKeyChange);

    /* 5. 注册定时器 (可选) */
    RedisModule_CreateTimer(ctx, 1000, OnPeriodicTask, NULL);
    return REDISMODULE_OK;
}

int RedisModule_OnUnload(RedisModuleCtx *ctx) {
    /* 清理全局资源, 关闭外部连接 */
    return REDISMODULE_OK;
}
加载方式命令/配置时机参数支持
启动时loadmodule /path/mod.so(redis.conf)服务启动前基本参数
运行时MODULE LOAD /path/mod.so服务运行中仅 argv
高级(7.0+)MODULE LOADEX /path/mod.so CONFIG k v ARGS arg1服务运行中配置 + 参数

UNLOAD 的两条硬约束:① 已注册数据类型有 key 时不能 MODULE UNLOAD,需先删除所有该类型的 key;② 卸载时正在执行的命令不会中断,需等其结束。生产环境推荐 loadmodule(写入配置,启动顺序确定);调试用 MODULE LOADEX(7.0+ 支持运行时配置)。

33.4 自定义命令

CreateCommand 签名解释了 Redis 如何识别命令的 key 参数位置,从而支持 Cluster slot 路由。

int RedisModule_CreateCommand(
    RedisModuleCtx *ctx,
    const char *name,             /* 命令名 (如 "mymod.incr") */
    RedisModuleCmdFunc cmdfunc,   /* 处理函数 */
    const char *strflags,         /* 标志字符串 */
    int firstkey,                 /* 第一个 key 的位置 (从 1) */
    int lastkey,                  /* 最后 key 位置, -1=末尾 */
    int keystep                   /* key 间隔步长 */
);
标志含义影响
write写命令副本拒绝;写入 AOF
readonly读命令只读副本可执行;不写 AOF
fastO(1) 或 O(log N)不计入慢日志
deny-oom内存不足拒绝maxmemory 超限返回 OOM
allow-loading加载 RDB/AOF 期间可用管理类命令
no-monitor不在 MONITOR 输出敏感命令
getkeys-api通过回调声明 key不规则 key 参数
random命令结果非确定复制改用效果复制
pubsubPub/Sub 类不参与事务
 
int MySetCommand(RedisModuleCtx *ctx,
                 RedisModuleString **argv, int argc) {
    if (argc != 3) return RedisModule_WrongArity(ctx);
    RedisModule_AutoMemory(ctx);  /* RAII 风格 */

    RedisModuleKey *key = RedisModule_OpenKey(ctx, argv[1],
        REDISMODULE_READ | REDISMODULE_WRITE);

    int type = RedisModule_KeyType(key);
    if (type != REDISMODULE_KEYTYPE_EMPTY &&
        type != REDISMODULE_KEYTYPE_STRING) {
        RedisModule_CloseKey(key);
        return RedisModule_ReplyWithError(ctx, "WRONGTYPE");
    }

    RedisModule_StringSet(key, argv[2]);
    RedisModule_ReplicateVerbatim(ctx);     /* 复制到副本 */
    RedisModule_ReplyWithSimpleString(ctx, "OK");
    return REDISMODULE_OK;  /* AutoMemory 自动 close key */
}

33.5 自定义数据类型

RedisModuleType *RedisModule_CreateDataType(
    RedisModuleCtx *ctx,
    const char *name,                  /* 必须恰好 9 字符 */
    int encver,                        /* 编码版本号 */
    RedisModuleTypeMethods *methods    /* 回调函数集 */
);

typedef struct RedisModuleTypeMethods {
    uint64_t version;
    RedisModuleTypeLoadFunc    rdb_load;     /* RDB 反序列化 */
    RedisModuleTypeSaveFunc    rdb_save;     /* RDB 序列化 */
    RedisModuleTypeRewriteFunc aof_rewrite;  /* AOF 重写 */
    RedisModuleTypeMemUsageFunc mem_usage;   /* MEMORY USAGE */
    RedisModuleTypeDigestFunc  digest;       /* DEBUG DIGEST */
    RedisModuleTypeFreeFunc    free;         /* key 删除 */
    /* Redis 7+ 扩展 */
    RedisModuleTypeAuxLoadFunc aux_load;
    RedisModuleTypeAuxSaveFunc aux_save;
} RedisModuleTypeMethods;

9 字节类型名约束。类型名编码到 RDB 文件的 8 字节字段 + 1 字节 encver,共 9 字节固定标识。读 RDB 时按 9 字节查找已注册类型,冲突即加载失败。

  • ✓ "mytree-ab"(9 字符)
  • ✓ "RJSON-RL"(RedisJSON 实际名)
  • ✗ "mytype"(6 字符,拒绝)
  • ✗ "mytype-name"(11 字符,拒绝)

一旦持久化,类型名不可变更;升级版本用 encver 区分,不改名。

图 33-5-1 · RDB 序列化时序 — type_id 写入 + 模块回调

void MyTreeRdbSave(RedisModuleIO *rdb, void *value) {
    MyTree *tree = value;
    RedisModule_SaveUnsigned(rdb, tree->node_count);
    for (size_t i = 0; i < tree->node_count; i++) {
        MyNode *n = &tree->nodes[i];
        RedisModule_SaveSigned(rdb, n->key);
        RedisModule_SaveStringBuffer(rdb, n->val, n->val_len);
    }
}

void *MyTreeRdbLoad(RedisModuleIO *rdb, int encver) {
    if (encver != 0) return NULL;       /* 不支持的版本 */
    uint64_t count = RedisModule_LoadUnsigned(rdb);
    MyTree *tree = MyTreeNew();
    for (uint64_t i = 0; i < count; i++) {
        int64_t key = RedisModule_LoadSigned(rdb);
        size_t len;
        char *val = RedisModule_LoadStringBuffer(rdb, &len);
        MyTreeInsert(tree, key, val, len);
        RedisModule_Free(val);
    }
    return tree;
}

void MyTreeFree(void *value)       { MyTreeDestroy((MyTree*)value); }
size_t MyTreeMemUsage(const void *v){
    const MyTree *t = v;
    size_t sum = sizeof(MyTree) + t->node_count * sizeof(MyNode);
    for (size_t i = 0; i < t->node_count; i++) sum += t->nodes[i].val_len;
    return sum;
}

encver 向后兼容原则:模块新版本必须能读旧版 RDB(encver < current),旧模块不需读新版 RDB(拒绝即可)。滚动升级硬规则:先升级所有节点的模块,再触发 BGSAVE;绝对禁止 slot 迁移期间升级模块。

33.6 阻塞命令

阻塞命令让模块在后台线程执行耗时操作(HTTP / 磁盘 I/O / 大计算),不阻塞 Redis 主线程。

图 33-6-1 · 阻塞命令时序 — BlockClient → 后台 → UnblockClient → reply_cb

struct FetchArgs { RedisModuleBlockedClient *bc; char *url; };

void *fetch_thread(void *arg) {
    struct FetchArgs *args = arg;
    char *result = http_get(args->url);        /* 后台 I/O */
    RedisModule_UnblockClient(args->bc, result); /* 唤醒, 携带 privdata */
    free(args->url); free(args);
    return NULL;
}

int reply_callback(RedisModuleCtx *ctx,
                  RedisModuleString **argv, int argc) {
    /* 主线程调用, 可安全使用 ctx */
    char *result = RedisModule_GetBlockedClientPrivateData(ctx);
    if (result) {
        RedisModule_ReplyWithStringBuffer(ctx, result, strlen(result));
        free(result);
    } else {
        RedisModule_ReplyWithError(ctx, "FETCH_FAILED");
    }
    return REDISMODULE_OK;
}

int MyFetchCommand(RedisModuleCtx *ctx,
                  RedisModuleString **argv, int argc) {
    if (argc != 2) return RedisModule_WrongArity(ctx);
    size_t url_len;
    const char *url = RedisModule_StringPtrLen(argv[1], &url_len);

    RedisModuleBlockedClient *bc = RedisModule_BlockClient(
        ctx, reply_callback, timeout_callback, free_privdata, 5000);

    struct FetchArgs *args = malloc(sizeof(*args));
    args->bc = bc;  args->url = strndup(url, url_len);
    pthread_t tid;
    pthread_create(&tid, NULL, fetch_thread, args);
    pthread_detach(tid);
    return REDISMODULE_OK;
}

线程安全红线。后台线程禁止使用主线程的 ctx 调用 Redis API、禁止直接 OpenKey、禁止 ReplyWith*。允许:计算 / 外部 I/O / 调用 UnblockClient(线程安全) / pthread_mutex 保护自身状态。
进阶:RedisModule_GetThreadSafeContext 允许后台操作 keyspace,但 ThreadSafeContextLock 期间主线程被阻塞——破坏单线程模型,仅特殊场景使用。

定理 33-11-1 · API 边界开销的渐近行为。设原生命令开销 C_native,模块 API 边界开销 C_api ≈ 1-3 µs(取决于参数数量):

  • 模块命令总开销 C_module = C_native + C_api
  • API 开销占比 η = C_api / C_module

简单 O(1) 命令(C_native ≈ 0.5-1 µs): η ≈ 50-75% → 模块开销显著 → 常用 GET/SET 不应用模块实现。
复杂命令(C_native ≈ 10 µs - ms): η ≈ 5%-0.01% → 模块开销可忽略 → 适合 FT.SEARCH / JSON.SET 等复杂操作。

/* module.c - moduleLoad 简化 */
int moduleLoad(const char *path, void **module_argv, int module_argc) {
    void *handle = dlopen(path, RTLD_NOW | RTLD_LOCAL);
    if (handle == NULL) {
        serverLog(LL_WARNING, "Module load failure: %s", dlerror());
        return C_ERR;
    }
    int (*onload)(void*, void**, int);
    onload = (int (*)(void*, void**, int))(unsigned long)
        dlsym(handle, "RedisModule_OnLoad");
    if (onload == NULL) { dlclose(handle); return C_ERR; }

    RedisModuleCtx ctx = REDISMODULE_CTX_INIT;
    RedisModule *module = zmalloc(sizeof(*module));
    module->handle = handle;
    if (onload((void*)&ctx, module_argv, module_argc) == REDISMODULE_ERR) {
        moduleFreeContext(&ctx); dlclose(handle); zfree(module);
        return C_ERR;
    }
    dictAdd(modules, module->name, module);
    return C_OK;
}

/* module.c - BlockClient 简化 */
RedisModuleBlockedClient *RM_BlockClient(RedisModuleCtx *ctx,
    RedisModuleCmdFunc reply_callback,
    RedisModuleCmdFunc timeout_callback,
    void (*free_privdata)(RedisModuleCtx*, void*),
    long long timeout_ms) {
    client *c = ctx->client;
    RedisModuleBlockedClient *bc = zmalloc(sizeof(*bc));
    bc->client          = c;
    bc->module          = ctx->module;
    bc->reply_callback  = reply_callback;
    bc->timeout_callback= timeout_callback;
    bc->free_privdata   = free_privdata;
    bc->timeout         = mstime() + timeout_ms;
    blockClient(c, BLOCKED_MODULE);
    c->bpop.module_blocked_handle = bc;
    return bc;
}

设计取舍

Redis 选择 in-process 模块而非 out-of-process 插件(独立进程 + IPC)的根本原因是延迟。Redis 的核心价值是亚毫秒级响应,任何 IPC 都会引入额外的进程间通信延迟(本地 socket ~10 µs、共享内存 ~1 µs),与 Redis 自身命令执行(~1 µs)同量级,严重影响性能。Modules 选择 in-process 让模块直接访问 Redis 内存数据,开销限于函数调用与 API 边界封装(亚微秒级)。代价是崩溃风险:模块的任何 SIGSEGV 都会带走整个 Redis 进程。该取舍与 Linux 内核模块、PostgreSQL Extensions 完全一致,反映"性能 vs 隔离"的根本矛盾。

API 边界设计为 ~400 个 RedisModule_* 函数是稳定接口与可演进实现的精确平衡。若直接暴露 dict/sds/robj 等内部结构,模块会绑定 Redis 内部实现细节,核心重构(例如 sds 结构调整)必然破坏所有模块。Redis 选择不透明句柄隔离实现,核心可自由优化内部数据结构,只要 API 行为不变,模块即可继续工作。该模式与 Linux syscall 层、Java JNI、Win32 API 完全同构。

CreateDataType 的 9 字节类型名约束是 RDB 紧凑性的工程权衡。若类型名可变长,RDB 需额外存长度,且解析复杂;固定 9 字节让类型查找简化为单次比较。代价是命名空间紧张,模块开发者必须谨慎选择名字避免冲突。这与 ext4 文件系统的 inode 类型字段(固定 16 位)设计一致:固定长度换取简洁的存储与解析。

模块的复制策略(ReplicateVerbatim vs Replicate)选择了 "显式胜于隐式"。若 Redis 自动检测模块命令是否需复制,会引入复杂启发式且容易出错。Redis 选择让模块开发者明确声明复制方式,代价是开发者必须理解复制语义,收益是行为可预测、性能可控。该哲学贯穿 Modules API 整体:Redis 提供能力与原语,模块开发者负责正确使用。

33.14 本章要点

  1. Modules API 提供约 400 个 RedisModule_* 函数,作为模块与 Redis 核心的稳定 ABI 边界
  2. 模块以 .so 形式通过 dlopen 加载,生命周期 Load → 注册 → Serve → Unload
  3. CreateCommand 需声明 firstkey/lastkey/keystep 与 flags,用于路由与权限
  4. CreateDataType 注册自定义类型,9 字节类型名 + encver,需提供 rdb_load/save/free/mem_usage 回调
  5. 命令复制三策略:ReplicateVerbatim(原样)/ Replicate(多条)/ 不复制;写命令忘复制是灾难
  6. BlockClient + 后台线程实现异步操作,后台线程禁止使用主线程 ctx
  7. 键空间事件订阅 + 集群消息支持二级索引、scatter-gather 跨节点查询
  8. API 边界开销 C_api ≈ 1-3 µs;简单命令 η ≈ 50-75% 不该用模块;复杂命令 η ≈ 5% 适合模块
  9. Cluster 模式下模块必须所有节点同版本,推荐全停 / 滚动 / 蓝绿三种升级方案
  10. 崩溃隔离五层防护:ASAN/Valgrind + 部署隔离 + Sentinel + 资源限制 + 代码审查
