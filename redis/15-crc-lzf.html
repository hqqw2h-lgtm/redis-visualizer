15.1 CRC 多项式除法基础

CRC(Cyclic Redundancy Check,循环冗余校验)的核心思想是把消息字节串当作 GF(2) 上的多项式系数,再用一个固定的"生成多项式"做长除法,余数就是校验码。GF(2) 是只有 0 和 1 两个元素的有限域,加减都退化成 XOR,所以硬件实现极快。这一节先讲清楚代数本质,再分析为什么 k 位 CRC 漏检概率刚好是 1 / 2k。

15.1.1 CRC 的代数本质

图 15-1-1 · 把字节串看作多项式 — GF(2) 上的长除法

GF(2) 的工程意义:加减乘除都可以用纯 XOR + 移位实现 — 一个时钟周期内硬件就能完成 1 bit 的 CRC 推进。软件上把 8 bit 一次预先打表,就得到 5 ns/byte 的实战吞吐。

15.2 CRC16-CCITT(Cluster slot)

Redis Cluster 把 key 空间切成 16384 个 slot,每个 slot 由一个 master 节点负责。客户端发命令前要先决定 key 落在哪个 slot,公式是 slot = crc16(key) % 16384。这一节先看 CRC16 的计算流程,再分析 hash tag {...} 怎么强制把多个 key 路由到同一个 slot — 这是 Redis Cluster 中实现"事务 / 跨 key 操作"的唯一手段。

15.2.3 CRC16 的实际 C 实现

/* src/crc16.c — Redis 实现 (实际版) */
static const uint16_t crc16tab[256] = {
    0x0000, 0x1021, 0x2042, 0x3063, 0x4084, 0x50A5, 0x60C6, 0x70E7,
    0x8108, 0x9129, 0xA14A, 0xB16B, 0xC18C, 0xD1AD, 0xE1CE, 0xF1EF,
    0x1231, 0x0210, 0x3273, 0x2252, 0x52B5, 0x4294, 0x72F7, 0x62D6,
    0x9339, 0x8318, 0xB37B, 0xA35A, 0xD3BD, 0xC39C, 0xF3FF, 0xE3DE,
    /* ... 224 个常量,共 256 项 ... */
};

uint16_t crc16(const char *buf, int len) {
    int counter;
    uint16_t crc = 0;
    for (counter = 0; counter < len; counter++)
        crc = (crc << 8) ^ crc16tab[((crc >> 8) ^ *buf++) & 0xFF];
    return crc;
}

每字节执行:1 次表查 + 1 次 XOR + 1 次左移。现代 x86-64 CPU 流水线在 5 个时钟周期内完成,2.5 GHz 时钟下约 2 ns/byte。即使百万 QPS 的 Cluster 集群,CRC16 的 CPU 占比也 < 0.1%。

15.3 CRC64-Jones(RDB 校验)

RDB 文件的最后 8 字节是 CRC64-Jones 校验码,覆盖从 "REDIS" magic 起到 EOF 标记前的所有字节。加载时边读边累计 CRC64,读完与文件末 8 字节比对,不等就拒绝加载。这一节先看 RDB 末尾布局,再讲 CRC64-Jones 与 CRC16-CCITT 在"反射"上的差异。

15.3.1 RDB 文件尾部 CRC64 布局

图 15-3-1 · RDB 文件结构 — 末尾 8 字节 CRC64

关闭校验 rdbchecksum no(不推荐生产):加载时把末 8 字节当 0 不比对;保存时也不计算(末 8 字节为 0)。节省 ~5% BGSAVE CPU,但代价是损坏 RDB 会被静默加载, 错乱数据无法察觉。

15.3.2 CRC64-Jones 与 CRC16 的差异

图 15-3-2 · 反射(LSB-first)vs 非反射(MSB-first)

CRC64-Jones 的"反射"含义:每字节按 bit 反序(LSB-first)处理,对应硬件位串行电路从低位往高位送数据。CRC16-CCITT 用非反射(MSB-first)。两种风格在标准 CRC 表中各占一半,选择由具体协议规定 — Redis 选择对应 RDB 协议的 CRC64-Jones,与 Bitcoin、PostgreSQL WAL 等同源。

15.4 8 位表驱动加速

所有 CRC 软件实现都在"内存 vs 性能"之间选点。Redis 选择 8 位表 — 内存极小、性能足够。这一节量化各方案的吞吐与内存,再看 CRC16/CRC64 在 Redis 源码中的紧凑实现。

15.4.1 不同表驱动方式的吞吐对比

图 15-4-1 · 4 种 CRC 实现方案的吞吐 vs 内存对比

Redis 选择 8 位表(B 方案):内存开销小(L1 cache 完全装下)、实现简单、性能足够。极端场景(如 16 GB RDB 文件)仍可接受 5 ns/byte ≈ 80 秒的全文校验时间;而且 CRC 计算与 RDB 写入并行进行,实际并不延长 BGSAVE 时间。

15.4.2 CRC16 表驱动的紧凑实现

/* src/crc16.c — Redis 实现的核心循环 */
static const uint16_t crc16tab[256] = {
    0x0000, 0x1021, 0x2042, 0x3063, 0x4084, 0x50A5, 0x60C6, 0x70E7,
    0x8108, 0x9129, 0xA14A, 0xB16B, 0xC18C, 0xD1AD, 0xE1CE, 0xF1EF,
    /* ... 240 个常量,共 256 项 ... */
};

uint16_t crc16(const char *buf, int len) {
    int counter;
    uint16_t crc = 0;
    for (counter = 0; counter < len; counter++)
        crc = (crc << 8) ^ crc16tab[((crc >> 8) ^ *buf++) & 0xFF];
    return crc;
}

/* 性能特征:
 *   每字节 1 次表查、1 次 XOR、1 次左移、1 次右移、1 次 AND
 *   现代 x86-64 流水线 5 周期 / 字节
 *   2.5 GHz CPU → 2 ns/byte (实测,L1 命中)
 */

15.4.3 CRC64 实现的对称结构

/* src/crc64.c — 简化片段 (实际是反射版) */
static const uint64_t crc64_tab[256] = {
    UINT64_C(0x0000000000000000), UINT64_C(0x7ad870c830358979),
    UINT64_C(0xf5b0e190606b12f2), UINT64_C(0x8f689158505e9b8b),
    UINT64_C(0xc038e5739841b68f), UINT64_C(0xbae095bba8743ff6),
    /* ... 共 256 项 64-bit 常量 ... */
};

uint64_t crc64(uint64_t crc, const unsigned char *s, uint64_t l) {
    while (l--) {
        crc = crc64_tab[(uint8_t)(crc ^ *s++)] ^ (crc >> 8);
    }
    return crc;
}

/* ★ 关键差异:
 *   - CRC16 用 (crc >> 8) 取高 8 位查表,(crc << 8) 推进
 *   - CRC64 用 (uint8_t)crc 取低 8 位查表,(crc >> 8) 推进
 *   高位查 + 左移 (CRC16) vs 低位查 + 右移 (CRC64)
 *   是 "非反射 vs 反射" 的代码体现。
 */
CRC64 流式累计:第二个参数 crc 是上一次返回值,所以 crc64(0, data, len) 是首次调用,后续可以 crc64(prev, more_data, more_len) 继续累计。RDB 写入用 rio.c → rioGenericUpdateChecksum 在每次写出字节流时调用,实现真正的"边写边算"。

15.7 hash 链查重复

LZF 压缩的核心问题是"在 8KB 窗口里快速找到当前 3 字节的最长匹配"。LZF 用一个固定大小的 hash 表(htab,默认 64K 项)记录"上一次出现这个 hash 的位置",实现 O(1) 摊销查找。这一节看 hash 链的逻辑与 HLOG 调参的影响。

/* LZF 压缩主循环伪码 — htab 栈分配, 编码完即释放 */
htab = alloca((1 << HLOG) * sizeof(uint32_t));   /* 栈分配 256 KB */
uint32_t cur = 0;
while (cur + 3 <= len) {
    uint32_t h   = hash3(in[cur], in[cur+1], in[cur+2]);
    uint32_t idx = h & ((1 << HLOG) - 1);
    uint32_t ref = htab[idx];
    htab[idx] = cur;                                /* 记录当前位置 */
    if (ref && cur - ref <= 8192) {                 /* 窗口内? */
        size_t match_len = extend_match(in, ref, cur, len);
        if (match_len >= 3) {
            emit_backref(cur - ref - 1, match_len);
            cur += match_len;
            continue;
        }
    }
    emit_literal(in[cur]);
    cur += 1;
}

15.8 LZF 解压流程

LZF 解压是纯顺序流式解码 — 没有 hash 表、没有匹配查找,只是按 token 头字节分发到三个分支。这也是为什么解压速度 (~600 MB/s) 远快于压缩 (~200 MB/s)。但 BACKREF 的循环复制有个微妙的坑:必须逐字节 / memmove,不能 memcpy。

解压复杂度严格线性,无 hash 表,故解压远快于压缩。这一不对称性影响 quicklist list-compress-depth 的取舍:写入(可能压缩)比读取(必解压)更慢 ~3 倍,但热路径 LPUSH/RPUSH 都落在头尾不压缩节点,实际无差。

15.11.3 反例 1:用 CRC64 做完整性认证

错误:把 CRC64 当作认证码

场景: 跨机房传 RDB, 担心中间人篡改

攻击步骤:

  1. 截获 RDB 文件(含末尾 CRC64)
  2. 修改 key admin_password 的 value
  3. 因为 CRC 是线性的, 攻击者可以反向构造一段 padding 让总 CRC64 不变
  4. 把改过的 RDB 发回去, Redis 加载完全通过, 不察觉

后果: 攻击者控制 Redis 数据 · 校验"通过"给运维错觉 · CRC 在密码学上完全可解

正确:用 HMAC-SHA256 做认证

# 发送方 (压缩 + 认证)
openssl dgst -sha256 -hmac "shared_secret" \
  -binary dump.rdb > dump.rdb.hmac

# 接收方 (验证)
calc=$(openssl dgst -sha256 -hmac "shared_secret" \
       -binary dump.rdb)
expected=$(cat dump.rdb.hmac)
if [ "$calc" = "$expected" ]; then
  redis-server --dbfilename dump.rdb
else
  echo "RDB 被篡改" >&2
  exit 1
fi

# 收益:
# - HMAC-SHA256 抗主动篡改
# - 攻击者不知 secret,无法构造合法 hmac
# - CRC 留给检测随机损坏 (磁盘 bit-rot)

15.11.4 反例 2:list-compress-depth 设到 5

误以为 depth 越大越省内存:实际上 depth ≥ 3 收益极小 — quicklist 头尾几个 listpack 节点本身就占总内存大头(每节点 4-8 KB),把它们也压缩反而拖慢 LPUSH/RPUSH 热路径。正确做法:大 list 用 list-compress-depth = 1,中等 list 保持默认 0。

15.12.4 设计取舍

取舍 1:CRC 表驱动 8 位 vs 16 位

8 位表 256 项,内存占用极小(CRC16 512 B,CRC64 2 KB),性能 5 ns/byte。16 位表性能提升至 3 ns/byte,但内存增加 256 倍(CRC64 16 位表要 512 KB,L1 cache 完全装不下,反而因 cache miss 变慢)。Redis 选择 8 位表,匹配"代码简洁优先"的设计原则。

取舍 2:LZF 而非 LZ4 / Snappy

LZ4 解压速度快 2× 于 LZF,Snappy 接近 LZF。Redis 选择 LZF 的关键:实现极简(< 1000 行 C 代码)、无 SIMD 依赖、跨平台一致。LZ4 库虽有性能优势但引入额外依赖。Redis 7.4 后社区有提案支持 LZ4,主线尚未合并。

取舍 3:21 字节最小压缩阈值

LZF 头部开销 ~5 字节,小于 21 字节字符串压缩后通常不能净节省。21 是经验阈值,在 src/rdb.c 中硬编码 — 没必要变成配置项,因为对所有数据都一致。

取舍 4:list-compress-depth 默认 0

默认不压缩:多数 list 用例对延迟敏感(队列、最新消息列表),解压代价不可接受。需要节省内存的 list 业务可手动设 1 或 2。Redis 团队选择"默认 = 安全 = 不压缩",符合"配置项默认值应该让 99% 用户开箱即用"原则。

设计取舍总览

Redis 在 CRC + LZF 上的 4 个设计选择:
① CRC16 + CRC64 而非 CRC32C:历史选择,CRC 不是热点,无强动机切换;
② 8 位表驱动:512 B / 2 KB 内存,L1 命中,5 ns/byte 已经足够;

④ list-compress-depth 默认 0:延迟优先,需要内存优化时再开。
