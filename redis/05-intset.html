第 5 章 · intset 整数集合

全整数 Set 的紧凑编码 · 8B header + 连续数组 · INT16/INT32/INT64 三档自适应 · 升级 O(N) 不可降级 · 二分查找 · memmove 维序 · 阈值 512 元素

Redis Set 类型在元素全为整数且数量较少时,使用 intset 编码——一段连续内存按升序存储 int16/int32/int64,8 字节 header 后紧跟整数数组。相比 hashtable 编码每元素 24-48 字节(dictEntry + key sds + value 包装),intset 把 512 个 16-bit 整数压到约 1 KiB,内存效率约提升 8-10 倍。

代价是插入与删除需要 memmove 维护有序性,最坏 O(N)。intset 仅在以下条件全部满足时使用:所有元素都是整数 + 元素数量不超过 set-max-intset-entries(默认 512)。任意条件失败立即转为 hashtable,且不可逆。

这一章是 Redis "编码自适应"思想最干净的样本:数据结构本身极简(只有 3 个字段、150 行 C 代码),却把"内存极致压缩 + 支持任意 64-bit 整数 + 有序范围扫描"三个矛盾目标同时解决。理解 intset 是理解后续 ziplist/listpack/quicklist/rax 等"紧凑编码"族的入门钥匙。

  • intset 三字段(encoding / length / contents[])与 8B header 紧跟柔性数组的字节布局
  • 三种 encoding(INT16 / INT32 / INT64)的数值范围、触发条件与字节宽度
  • intsetSearch 二分查找 + 头/尾速判,顺序插入退化为 O(1)
  • intsetAdd 不升级路径:zrealloc 扩容 + memmove 维序 + _intsetSet 写值
  • intsetUpgradeAndAdd 升级路径(本章核心):从后往前重排避免覆盖,prepend 标志决定头尾
  • intsetRemove 不降级的设计动机:"少做事"原则
  • 六种候选数据结构对比:为什么只有 intset 适配"全整数小集合"
  • 八大典型陷阱:类型污染、阈值穿越、不可逆转换、SPOP 抖动等
  • 核心源码 intset.c + t_set.c 的调用链与位字节翻转(intrev32ifbe)细节
  • "少做事"哲学:升级一次后不再降级,转换后不回头,把判断逻辑剪到最简

前置知识:第 1 章 SDS(理解 sdshdr 与紧凑头部);第 3 章 dict(理解 dictEntry 24-48B 元数据放大);C99 柔性数组成员(flexible array member);jemalloc size class 对齐(8/16/32/48/64 字节)。读者需要熟悉 redis-cli 与 OBJECT ENCODING / MEMORY USAGE 调试命令。

5.1 解决什么问题

在讨论数据结构之前,先看清楚 intset 要解决的 具体业务场景。Redis 设计时的一个反复出现的问题是:Set 类型的元素经常 全是整数 且 数量较少,却被通用 hashtable 编码用元数据撑大十倍。这一节先做需求拆解,再给候选方案矩阵,最后用编码决策图把 intset 的位置定下来。

5.1.1 业务需求拆解

Redis Set 类型的常见应用场景中,大量场景的元素是整数。下表统计了主流业务里 Set 的元素特征:

表 5-1-1 · Set 业务场景元素特征采样

用户标签 ID 集合标签 ID(int)5-5095%+
文章 ID 关联集合文章 ID10-100095%+
SKU 库存SKU 编号50-50080%(混合)
黑名单 IP(整数化)IPv4 转 uint32100-10k100%
在线设备 ID设备 ID10-100090%+

核心观察:实际生产中 全整数小集合占 Set 用例的 60-80%。如果统一使用 hashtable,这部分 Set 浪费大量内存;如果只保留 hashtable,无法在小整数场景达到内存极致。Redis 选择"双轨制"——小整数走 intset,大集合或非整数走 hashtable,由写入路径自动切换。

5.1.2 候选方案的不足

如果不用 intset,有六种数据结构可以承载"整数集合"语义。逐一分析它们在小整数 Set 场景的劣势:

表 5-1-2 · 六种候选数据结构的对比
方案内存(每元素)查找插入在小整数 Set 场景的劣势
hashtable (dict)24-48 BO(1)O(1)小整数场景内存浪费 5-10 倍
单链表16-24 BO(N)O(1)(头插)查找 O(N),无序

intset 选择 三种 encoding 自适应 解决"既要省内存又要支持大数"的矛盾:全是 16-bit 范围内的整数时使用 INT16,出现 32-bit 时升到 INT32,出现 64-bit 时升到 INT64。和有序数组比,intset 多了 encoding 字段允许动态宽度;和 hashtable 比,intset 没有 dictEntry 元数据;和位图比,intset 不要求元素密集。

5.1.3 编码决策图

图 5-1-1 · Set 编码选择决策树(SADD 路径)

三个决策点:① 元素是不是整数;② 元素数是否超阈值;③ 数值是否在当前 encoding 范围内。任一答案"否"或"超"会触发不同分支。其中 "非整数 → hashtable" 这条边是不可逆的——一旦转换,即使再删除非整数元素也不会回到 intset。

5 个 INT16 元素 + 8B header + jemalloc 对齐 = 80 字节。等价 hashtable 编码(5 个 dictEntry + dict 元数据)约 350-450 字节,内存差距 4-5 倍。

用例提前预告:这 80 字节的 Set 如果转为 hashtable,实际占用会跳到 ~440 字节(dictEntry 24B × 5 + sds key 16B × 5 + dict 容器 96B + bucket 数组 64B + redisObject 16B × 5)。生产中常见的"权限位 Set" 或"标签 ID Set" 通常 5-50 个元素,intset 编码下只占 100 byte 量级,hashtable 下可达 KB 量级,集群级别累计差距非常显著。

5.2 整体布局

intset 的字节布局是 Redis 紧凑编码的"教科书样本":一个 8 字节 header(encoding 4B + length 4B)紧跟一段连续的整数数组,没有任何额外元数据。这一节用三种 encoding 的对比图把字节宽度差异画清楚,然后给 C 结构定义与字段语义对照。

5.2.4 C 结构定义

/* src/intset.h */
typedef struct intset {
    uint32_t encoding;
    uint32_t length;
    int8_t contents[];
} intset;

#define INTSET_ENC_INT16 (sizeof(int16_t))   /* = 2 */
#define INTSET_ENC_INT32 (sizeof(int32_t))   /* = 4 */
#define INTSET_ENC_INT64 (sizeof(int64_t))   /* = 8 */

/* 关键宏:由 encoding 反查每元素字节宽度
 * 用 sizeof(int16_t/int32_t/int64_t) 而非常量字面量,
 * 让 encoding 字段与 sizeof 直接相等,代码极简
 */
关于柔性数组成员(flexible array member):contents 声明为 int8_t[] 是因为 C99 柔性数组只支持单一字节类型,实际访问时根据 encoding 强转为 int16_t* / int32_t* / int64_t*。结构体本身 sizeof(intset) = 8(只算 header,不算 contents),所以分配时要显式写 sizeof(intset) + length × encoding。

5.3 intsetSearch 二分查找

intset 的 contents 是有序数组,所有读路径(SISMEMBER / SADD 去重 / SREM)都从 intsetSearch 开始。利用二分查找把比较次数压到 ⌈log₂(N)⌉,加上 head/tail 速判,实际场景下大多数操作只需 1-3 次比较。

5.4 intsetAdd 不升级插入路径

intsetAdd 是 SADD 命令在 intset 编码下的最常见路径。当新值的字节宽度 ≤ 当前 encoding 时,走"不升级路径":二分定位插入点,zrealloc 扩容一格,memmove 把后续元素右移一格,写入新值。整个过程是经典的"有序数组插入"。

顺序插入是最优场景:每次都尾部追加 → O(1) 摊销时间。逆序或随机插入退化为 O(N)。生产中如果 SADD 元素是递增 ID(雪花 ID、自增 PK 等),intset 表现接近 hashtable;如果是随机散列(UUID 整数化、哈希值),内存优势仍在但写入慢 5-10 倍。

5.5 intsetUpgradeAndAdd 升级路径(本章核心)

升级路径是 intset 实现里最精巧的部分。新值超出当前 encoding 时,所有现有元素必须按新宽度重新存储——这是 intset 自适应的代价。这一节讲清楚"从后往前重排"的设计动机、prepend 标志的来由,以及完整 5 帧动画。

5.5.1 设计动机

新值超出当前 encoding 时,所有现有元素必须按新宽度重新存储。例如 5 个 INT16 元素 + 插入 65535(超出 INT16 上限):

  • 原 contents 内存:5 × 2 = 10 B
  • 新 contents 内存:6 × 4 = 24 B
  • 必须把每个旧 INT16 值搬到新的 INT32 位置
关键约束:zrealloc 后内存块可能在原位扩展,旧元素与新位置重叠。INT16 偏移 < INT32 偏移,如果从前往后搬,前面元素覆盖后面元素的旧值,数据丢失。必须从后往前搬迁——先搬最后一个旧元素到最后一个新位置,再倒数第二个,以此类推。这样写新位置时,旧位置已经被读过,不会丢数据。

5.5.4 升级方向决策(prepend 标志)

图 5-5-3 · prepend 标志的来由
为什么触发升级的值必为极值:能引发 INT16→INT32 升级的值,必须超过 INT16 范围(< -32768 或 > 32767)。同时所有现有元素都在 INT16 范围内(≤ 32767 且 ≥ -32768)。所以新值要么 < 所有现存值(负数,放头),要么 > 所有现存值(正数,放尾)。不存在"插入中间"的情况——这就是 prepend 只需 1 bit 的原因。

5.5.5 升级路径复杂度

表 5-5-1 · 升级路径复杂度
场景时间空间触发条件
INT16 → INT32 升级O(N)每元素 +2 B插入 [-2^31, -2^15) ∪ (2^15, 2^31)
INT32 → INT64 升级O(N)每元素 +4 B插入 [-2^63, -2^31) ∪ (2^31, 2^63)
INT16 → INT64 升级(直接)O(N)每元素 +6 B插入超 INT32 范围

升级在 intset 生命周期内 最多发生 2 次(INT16 → INT32 → INT64)。每次 O(N) 单位代价,均摊到所有元素仍是 O(1)。这是 intset 的优雅之处:虽然单次升级可能导致 P99 抖动(N=512 时拷贝 512 个 8B 写入,~1-3 μs),但全生命周期总成本可控。

5.6 intsetRemove 删除操作

删除是插入的镜像:二分定位 → memmove 把后续元素左移一格 → zrealloc 缩容。但 intset 不会因删除大数而降级 encoding——这是 Redis "少做事"哲学的体现,本节会展开讨论。

表 5-6-1 · intsetRemove 复杂度

为什么不降级:降级需要每次删除都扫描所有元素判断"剩余值的最大宽度"。例如要从 INT64 降到 INT32,必须遍历 contents 确认所有值都在 [-2^31, 2^31) 内,O(N) 成本叠加在每次 SREM 上。Redis 选择"少做事"——升级一次后不再降级,接受少量内存浪费。生产中如内存紧张可手动 DEL key 后重建。

5.7.3 设计动机

intset 在"全整数小集合"这一狭窄但高频的场景下,实现了:

  • 比 hashtable 节省 5-10 倍内存(没有 dictEntry 元数据)
  • 比通用有序数组省一半空间(自适应宽度,小数不占大宽度)
  • 支持 64-bit 大数(自动升级)
  • 有序性(支持范围扫描的雏形,虽然 Set 不暴露)

代价是插入/删除最坏 O(N),但小集合(< 512 元素)的 N 本身就小,memmove 的实际成本在 microsecond 量级。

"少做事"是核心哲学:不可逆转换的动机是简化命令路径——每次 SREM 都判断"是否能降级"会增加扫描成本。Redis 选择"少做事"——只在必要时升级,转换后不回头,把判断逻辑剪到最简。这一思想在后续的 ziplist→listpack、quicklist 节点压缩等设计里反复出现。

/* src/intset.c - 升级 + 插入 (本章核心) */
static intset *intsetUpgradeAndAdd(intset *is, int64_t value) {
    uint8_t curenc = intrev32ifbe(is->encoding);
    uint8_t newenc = _intsetValueEncoding(value);
    int length = intrev32ifbe(is->length);
    int prepend = value < 0 ? 1 : 0;

    /* 立即更新 encoding,以便 _intsetSet 使用新 encoding */
    is->encoding = intrev32ifbe(newenc);
    is = intsetResize(is, intrev32ifbe(is->length) + 1);

    /* 从后往前重排:必须从 length-1 走到 0,
     * 因为新 encoding 偏移 > 旧 encoding 偏移,
     * 写靠前会破坏未读的旧值 */
    while (length--)
        _intsetSet(is, length + prepend,
                   _intsetGetEncoded(is, length, curenc));

    /* 在头或尾插入新值 */
    if (prepend)
        _intsetSet(is, 0, value);
    else
        _intsetSet(is, intrev32ifbe(is->length), value);

    is->length = intrev32ifbe(intrev32ifbe(is->length) + 1);
    return is;
}
/* src/intset.c - 不升级路径 */
intset *intsetAdd(intset *is, int64_t value, uint8_t *success) {
    uint8_t valenc = _intsetValueEncoding(value);
    uint32_t pos;
    if (success) *success = 1;

    /* valenc > 当前 encoding → 走升级路径 */
    if (valenc > intrev32ifbe(is->encoding)) {
        return intsetUpgradeAndAdd(is, value);
    } else {
        /* 已在 encoding 范围内 → 二分定位 */
        if (intsetSearch(is, value, &pos)) {
            if (success) *success = 0;  /* 已存在 */
            return is;
        }

        /* 扩容 + memmove + 写入 */
        is = intsetResize(is, intrev32ifbe(is->length) + 1);
        if (pos < intrev32ifbe(is->length))
            intsetMoveTail(is, pos, pos+1);
    }

    _intsetSet(is, pos, value);
    is->length = intrev32ifbe(intrev32ifbe(is->length) + 1);
    return is;
}
/* src/intset.c - 二分查找 */
static uint8_t intsetSearch(intset *is, int64_t value, uint32_t *pos) {
    int min = 0, max = intrev32ifbe(is->length) - 1, mid = -1;
    int64_t cur = -1;

    /* 空集合 */
    if (intrev32ifbe(is->length) == 0) {
        if (pos) *pos = 0;
        return 0;
    } else {
        /* 头/尾速判:顺序插入退化为 O(1) */
        if (value > _intsetGet(is, max)) {
            if (pos) *pos = intrev32ifbe(is->length);
            return 0;
        } else if (value < _intsetGet(is, 0)) {
            if (pos) *pos = 0;
            return 0;
        }
    }

    /* 标准二分 */
    while (max >= min) {
        mid = ((unsigned int)min + (unsigned int)max) >> 1;
        cur = _intsetGet(is, mid);
        if (value > cur) min = mid + 1;
        else if (value < cur) max = mid - 1;
        else break;
    }

    if (value == cur) {
        if (pos) *pos = mid;
        return 1;
    } else {
        if (pos) *pos = min;
        return 0;
    }
}
关于 intrev32ifbe:Redis 在内存中始终用小端序存 encoding 与 length 字段,以保证 RDB 在不同字节序机器之间可移植。intrev32ifbe(integer reverse 32 if big-endian)在大端机器上做字节翻转,在小端机器(主流 x86/ARM)上是 no-op。读源码时可以把它当成"访问 length 的语法糖"。

5.9.4 设计取舍

取舍 1:三种 encoding vs 单一 INT64

单一 INT64 简化代码,但小整数场景内存浪费 4 倍(2 B → 8 B)。三种 encoding 自适应在 INT16 起步、按需升级,内存效率最优。代价是升级时一次 O(N) 重排——但生命周期内最多 2 次,均摊 O(1)。

取舍 2:连续数组 vs 链表

连续数组提供 cache locality 与 O(log N) 二分查找,但插入/删除需 memmove。链表插入 O(1) 但查找 O(N)、cache miss 多。小集合场景下连续数组的整体成本(查找 + 写)优于链表。Redis 在 quicklist 设计里把这两种思路结合(链表 + 内部 listpack 块),后续章节会讨论。

取舍 3:不降级 vs 自动降级

降级需要每次删除都扫描所有元素判断"剩余值的最大宽度"。Redis 选择"少做事"——升级一次后不再降级,接受少量内存浪费。生产中如内存紧张可手动 DEL key 后重建。

取舍 4:不可逆 intset → hashtable

转换为 hashtable 后,即使删除非整数元素回到全整数,也不转回 intset。原因同上——避免每次操作都判断"是否可降级"。代价是被污染的 key 内存永久放大,需应用层重建。

5.11 真实用例

用例 1:用户标签系统(电商个性化)
某电商用户标签存储:每用户一个 Set,key 形如 tags:user:<uid>,member 是预定义的标签 ID(0-100k 范围)。典型规模:每用户 5-50 个标签,P99 不超过 200。编码状态:OBJECT ENCODING 显示 95% 是 intset INT32,5% 因混入新标签 ID 升级到 INT64。实际收益:1 亿用户 × 平均 30 标签,使用 intset 编码总内存 ~12 GB;若用 hashtable 估算 ~150 GB。关键防护:① 业务层只允许整数 ID 写入(防止字符串污染);② 标签 ID 集中在 INT32 范围(避免不必要的 INT64 升级);③ 定期监控 set_polluted_count。
用例 2:在线设备 ID(物联网平台)
IoT 平台用 Set 维护"过去 5 分钟在线设备 ID":SADD online:5min:<ts> <dev_id>。设备 ID:雪花 ID(64-bit),自然走 intset INT64 编码。规模:每分钟新建一个 key,保留 60 个 key。每 key 5000-50000 设备 → 接近阈值上限,部分 key 超阈值转 hashtable。调优:通过 CONFIG SET set-max-intset-entries 1024 把 80% 的 key 留在 intset,内存节省约 40%;同时 SADD P99 从 30 μs 降到 12 μs(因 hashtable 路径有 dictAdd 开销)。
用例 3:权限位集合(后台管理系统)
每角色一个 Set,member 是权限编号(1-128)。规模:每角色 10-80 个权限,完美匹配 intset INT16。实际表现:1000 个角色 × 平均 30 权限,总内存 ~80 KB(intset 编码);hashtable 估算 ~5 MB,差距 60 倍。关键操作:① SISMEMBER 频繁(每次请求都检查权限),intset 二分查找 O(log 80) ≈ 6 次比较,P99 < 1 μs;② SADD/SREM 极少(只在权限变更时),O(N) memmove 不是瓶颈。结论:读多写少场景是 intset 的甜点。
用例 4:不要这么用 — 时间戳 Set
反例:某团队用 Set 存"过去 1 小时所有请求时间戳":SADD reqs:<uid> <timestamp_ms>。问题:① 时间戳是毫秒精度 INT64,直接走 INT64 编码,内存 8B/元素;② 元素数随访问频率线性增长,易超 512 阈值;③ 转 hashtable 后单 key 数 KB,1 万用户 × 100 次请求 = 数 MB。正解:用 有过期的 ZSet + 滑动窗口(以时间戳为 score),或用 HyperLogLog 估算 UV;时间戳本身不应该作为 Set 成员。
用例 5:文章 ID 关联(新闻推荐)
推荐系统维护"用户已浏览文章 ID":SADD viewed:<uid> <article_id>。规模:每用户平均 200-500 篇,P99 1000+ 篇。策略:① 阈值前(< 512)走 intset,内存最省;② 超阈值后转 hashtable,SISMEMBER 性能不退化;③ 定期 trim:用 LRU 思想,每月凌晨 SREM 一年前浏览的文章,把 hashtable 的 key 重建为 intset(SMEMBERS → DEL → SADD)。收益:重建后 70% 的 key 回到 intset,集群内存下降 35%。

# 期望: 96 (升级后)

# 3. 类型污染 → hashtable 不可逆
redis-cli SADD s "abc"
redis-cli OBJECT ENCODING s
# 期望: hashtable

redis-cli SREM s "abc"
redis-cli OBJECT ENCODING s
# 期望: hashtable      ← 转换不可逆

# 4. SISMEMBER (intset 之上的二分)
redis-cli DEL s
redis-cli SADD s 3 7 15 42 100
redis-cli SISMEMBER s 42
# 期望: 1 (命中)
redis-cli SISMEMBER s 99
# 期望: 0 (未命中)

# 5. SREM (memmove + 不降级)
redis-cli SREM s 15
redis-cli SMEMBERS s
# 期望: 3 7 42 100
redis-cli OBJECT ENCODING s
# 期望: intset (encoding 保持)

# 6. 阈值穿越 → 转 hashtable
for i in $(seq 1 600); do
  redis-cli SADD s2 $i > /dev/null
done
redis-cli OBJECT ENCODING s2
# 期望: hashtable (超 512)
