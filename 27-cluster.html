第 27 章 · Cluster 集群

16384 hash slot · CRC16 路由 · Gossip 协议 · PFAIL/FAIL 多数派 · MOVED/ASK 重定向 · 在线 reshard · 去中心化 failover

单节点 Redis 的容量与吞吐受限于单机硬件:内存上限取决于物理 RAM,写吞吐上限取决于单核 CPU。主从复制与 Sentinel 提供高可用,但不解决水平扩展问题:所有节点存储相同数据,内存与写吞吐上限不变。Redis Cluster 通过 16384 个 hash slot 将 keyspace 分散到多个 master 节点,每个节点独立处理一部分 slot 的读写,实现存储与吞吐的水平扩展。

Cluster 的核心设计选择有四:其一,固定 16384 个 slot 替代一致性 hash,bitmap 仅 2 KB 可塞入 Gossip 包;其二,Gossip 协议替代中心化协调器,无单点故障且消息复杂度 O(N);其三,PFAIL → FAIL 多数派投票替代外部 Sentinel,故障检测内置;其四,ASK + MIGRATE 支持在线 reshard,客户端按重定向处理迁移中 slot。这些选择共同形成"足够强大且足够简洁"的分布式方案,代价是多 key 命令、Lua 脚本、事务必须确保所有 key 在同一 slot。

本章按"分片 → Gossip → 故障检测 → 重定向 → 在线迁移 → 部署适配"组织,聚焦 Cluster 的工程语义边界与生产可观测性。读完后,你应能解释 MOVED 与 ASK 的差异、cluster-node-timeout 的取舍、为何不能把 Cluster 节点同时交给 Sentinel 监控。

  • 16384 slot 与 CRC16 路由:slot = CRC16(key) & 0x3FFF,bitmap 大小如何决定集群规模
  • hash tag {tag} 强制相关 key 落同 slot,粒度过粗导致热点、过细失去意义
  • Gossip 协议 PING/PONG 周期性传播;O(log N) 与 1000 节点约 14 秒是理想传播模型,不是生产 SLO
  • PFAIL 单点判定 vs FAIL 多数派(⌊N/2⌋+1)确认,网络分区下的安全性证明
  • MOVED(永久迁移,更新缓存)vs ASK(临时跳转,需先发 ASKING)的语义差异
  • SETSLOT MIGRATING/IMPORTING + MIGRATE 三步骤完成在线 reshard
  • MIGRATE 正常路径(DUMP+RESTORE+DEL)、超时校验边界与大 key 阻塞主线程的工程后果
  • Failover 流程:PFAIL → FAIL → 投票 → replica 提升,configEpoch 递增防脑裂
  • cluster-node-timeout 的取舍:5-10 秒平衡故障检测速度与误判风险
  • 10 大陷阱:CROSSSLOT、hash tag 过粗、cluster-require-full-coverage、failover 过慢、大 key 迁移、Cluster 与 Sentinel 共用

前置知识:第 25 章主从复制(replica 提升机制);第 26 章 Sentinel(对比 Cluster 的内置 failover);CRC16 算法基础;TCP/网络分区基础概念。读者需要熟悉 redis-cli --cluster 命令族。

27.1 为什么需要 Cluster

主从 + Sentinel 解决"故障切换",但所有 master 拥有完整 keyspace,容量上限不变。Cluster 通过分片解决"容量与吞吐"问题,代价是多 key 操作的 slot 约束。这一节先把三种部署模式的能力差异铺平,再讲清楚 Cluster 实际解决的四个核心问题。

27.1.2 Cluster 解决的四个核心问题

图 27-1-1 · 问题 → 设计决策的映射
四个问题对应四项设计决策。每项决策都引入新的工程复杂度,但相比"单机扩展到 1 TB 内存 + 100 核 CPU"的硬件成本,分布式方案的复杂度是更优解。Cluster 的存在不是为了"通用最优",而是为了"当单机不够用时,用最少的额外认知负担拿到水平扩展能力"。

27.2 16384 个 Hash Slot

Cluster 不直接 hash key 到节点,而是先把 key hash 到 16384 个 slot,再把 slot 分配给节点。这一层间接性是 reshard、扩容、迁移都能"按 slot 粒度"工作的关键。这一节拆开 slot 分配算法、为什么是 16384、以及 clusterState 中的 slot 映射结构。

27.2.1 Slot 分配算法 keyHashSlot

/* src/cluster.c */
unsigned int keyHashSlot(char *key, int keylen) {
    int s, e;

    /* 检查 hash tag {...} */
    for (s = 0; s < keylen; s++)
        if (key[s] == '{') break;

    if (s == keylen) return crc16(key, keylen) & 0x3FFF;

    for (e = s + 1; e < keylen; e++)
        if (key[e] == '}') break;

    if (e == keylen || e == s + 1)
        return crc16(key, keylen) & 0x3FFF;

    /* hash tag 存在,仅对 {...} 内容计算 */
    return crc16(key + s + 1, e - s - 1) & 0x3FFF;
}
表 27-2-2 · slot 计算与 hash tag 语义示例
KEYCRC16 输入slot说明
order:O1001order:O1001217无 hash tag → 整 key
order:O1002order:O100212474另一个 slot
payment:{O1001}O10012190用 hash tag
shipment:{O1001}O10012190共享 → 同 slot
user:{O1001}:profileO10012190共享 → 同 slot
hash tag 语义:
  • {} 内的字符串作为 CRC16 输入
  • 多个 key 共享同一个 hash tag → 落入同一 slot(MULTI / Lua / mget 可跨 key 操作)
  • {} 缺失或空 → 整个 key 作为 CRC16 输入
  • slot 范围:0 – 16383(214 个)
  • 分配方式:每个 master 持有一段或多段连续 slot

& 0x3FFF 是 % 16384 的位运算等价。CRC16 是 16-bit 输出,与 0x3FFF 与运算后保留低 14 位,即得到 0-16383 范围内的 slot 编号。

27.2.2 为什么是 16384 而非 65536

表 27-2-1 · 16384 vs 65536 slot 的工程对比
维度16384 slot65536 slot (假设)
bitmap 大小16384/8 = 2 KB65536/8 = 8 KB
Gossip PING 大小~3 KB (含其他字段)~9 KB
100 节点集群每秒带宽~3 MB/s~9 MB/s
1000 节点集群每秒带宽~30 MB/s~90 MB/s

关键约束:Gossip 包大小。每个 PING 携带发送者的 myslots bitmap,bitmap 越大 → 单包越大 → 集群总带宽越大。16384 是"足够细粒度"与"Gossip 带宽可控"的平衡点。reshard 时迁移单位是 slot,100 万 keys 时每 slot ~61 keys,迁移粒度合理;扩容到 65536 后,集群规模反而下降。Antirez 在 GitHub issue 中明确说明该决策依据。

27.2.3 clusterState 中的 slot 映射

/* src/cluster.h */
typedef struct clusterState {
    clusterNode *myself;
    uint64_t currentEpoch;
    int state;                                  /* CLUSTER_OK / FAIL */
    int size;                                   /* master 节点数 */
    clusterNode *slots[CLUSTER_SLOTS];          /* slot → 主节点指针 */
    rax *slots_to_keys;                         /* slot → keys 索引 */
    clusterNode *migrating_slots_to[CLUSTER_SLOTS];
    clusterNode *importing_slots_from[CLUSTER_SLOTS];
    /* ... */
} clusterState;

#define CLUSTER_SLOTS 16384

clusterState.slots[] 是 16384 长度的指针数组,每个元素指向负责该 slot 的 master 节点。命令路由时通过 slot = keyHashSlot(key) 计算 slot,再 clusterState.slots[slot] 查找节点,O(1) 完成路由决策。migrating_slots_to[] 与 importing_slots_from[] 标记正在迁移的 slot,迁移期间用于决定返回 ASK 重定向。

27.3 Gossip 协议

Cluster 节点之间没有 leader,也没有中心化协调器。它们通过 Gossip 协议(每节点定期把自己以及部分其他节点的状态摘要,随机播给一个对端)实现"最终一致的全局视图"。这一节先列出消息类型,再讲端口模型与 clusterMsg 结构,最后看 clusterCron 如何驱动整个心跳调度。

27.3.4 clusterCron 心跳调度

/* src/cluster.c (简化) */
void clusterCron(void) {
    /* 每 100ms 一轮,根据 cluster-node-timeout 选择目标 */

    /* 1. 随机选 1 个节点发 PING (保证活跃度) */
    static long long iteration = 0;
    iteration++;
    if (!(iteration % 10)) {  /* 每秒一次 */
        clusterNode *target = pickRandomNode();
        clusterSendPing(target, CLUSTERMSG_TYPE_PING);
    }

    /* 2. 检查所有节点,超时未收到 PONG 则立即 PING */
    di = dictGetSafeIterator(server.cluster->nodes);
    while ((de = dictNext(di))) {
        clusterNode *node = dictGetVal(de);
        if (node->flags & (CLUSTER_NODE_MYSELF | CLUSTER_NODE_NOADDR))
            continue;

        mstime_t since_last_pong = mstime() - node->pong_received;
        if (since_last_pong > server.cluster_node_timeout / 2) {
            clusterSendPing(node->link, CLUSTERMSG_TYPE_PING);
        }

        /* 3. 超过 cluster-node-timeout 标记 PFAIL */
        if (since_last_pong > server.cluster_node_timeout) {
            if (!(node->flags & CLUSTER_NODE_PFAIL)) {
                node->flags |= CLUSTER_NODE_PFAIL;
                serverLog(LL_WARNING, "Marking node %.40s as PFAIL",
                          node->name);
            }
        }
    }
}

clusterCron 是 Cluster 的核心定时器,频率 100 ms,主要任务包括:周期性 PING、超时检测、PFAIL 标记、failover 仲裁、连接重建。所有 Cluster 状态变更都源自 clusterCron 的触发或 PING/PONG 消息处理。

27.4 PFAIL 与 FAIL 故障检测

"是否有节点宕机"在分布式系统里没有简单答案 — 网络抖动、GC 卡顿、负载尖刺都会让节点短暂"失联"但实际仍活着。Cluster 用 两阶段判定(主观 PFAIL → 多数派 FAIL)来兼顾"快"与"准":单节点的怀疑只在本地视图里传播,只有 ⌊N/2⌋+1 个 master 都怀疑时才升级为 FAIL,触发 failover。

27.4.1 两阶段故障判定

图 27-4-1 · PFAIL ↔ FAIL 状态机

PFAIL · Probable FAIL · 主观下线

  • 触发:单个节点判定某节点 cluster-node-timeout 未响应
  • 作用:标记本地视图, 通过 Gossip 传播给其他节点
  • 含义:"我认为该节点可能故障"
  • 特点:单方面判定, 不触发 failover

FAIL · Confirmed FAIL · 客观下线

  • 触发:半数以上 master 在 2 × cluster-node-timeout 内都报同一节点 PFAIL
  • 作用:广播 FAIL 给全网, 触发 replica 提升
  • 含义:"集群多数派认定该节点故障"
  • 特点:多数派确认, 触发 failover
为何需要两阶段:PFAIL 单点判定避免误报扩散;FAIL 多数派投票防止网络分区下的脑裂。两阶段共同保证 安全性优于活性。

27.4.2 多数派投票的安全性证明

FAIL 判定形式化 · 设 N = master 总数, T = cluster-node-timeout

FAIL(B) ⟺ |{ M ∈ Masters : M.pfail_report(B, t) = true, t ∈ [now − 2T, now] }| ≥ ⌊N/2⌋ + 1

分区 B · 少数派(|B| < N/2)

  • B 内节点会把所有 A 内节点标记 PFAIL
  • 但 B 内 master 数 |B|m < ⌊N/2⌋ + 1
  • 无法升级为 FAIL → 不触发 failover
  • 少数派分区"沉默",对外不可用

分区 A · 多数派(|A| > N/2)

  • A 内 master 数 |A|m > N/2
  • 可正确将 B 内节点标记 FAIL
  • A 内 replica 提升为新 master, 继续服务
  • 客户端只连得到 A → 服务持续
结论:网络分区时, 少数派分区无法错误标记多数派为故障, 多数派分区可正确隔离少数派, 保证 failover 不会同时在两个分区发生(脑裂)。

27.5 MOVED 重定向

客户端不会被强制要求"先查路由表再发命令"。它可以盲发,节点收到不属于自己的 slot 时返回 MOVED。这种"惰性路由"让客户端实现可以从最简单的"无脑重试"逐步演进到"Smart Client 缓存完整拓扑"。这一节看 MOVED 的字段语义、Smart Client 的缓存策略,以及 CLUSTER SLOTS 命令的输出格式。

27.5.1 MOVED 错误的语义

$ redis-cli -p 7000 SET order:O1001 "PAID"
(error) MOVED 217 192.168.1.102:7002

$ redis-cli -p 7002 SET order:O1001 "PAID"
OK
图 27-5-1 · MOVED 字段语义

✓ 正确处理

  1. 解析 slot 与目标节点
  2. 更新本地 slot 缓存:slot_cache[slot] = (ip, port)
  3. 重新连接目标节点重发命令
  4. 后续相同 slot 的命令直接发往新节点

✗ 错误处理(反例)

  • 仅重试不刷新缓存 → 每次请求两跳
  • 整体清空 slot 缓存 → 后续所有请求需要重新探测
  • 阻塞等待 cluster topology 收敛 → 客户端假死

27.5.2 Smart Client 的 slot 缓存

27.6 ASK 重定向

reshard 期间,某个 slot 的 keys 分散在源/目标两个节点上 — 既不能"全在源"也不能"全在目标"。如果客户端收到 MOVED 直接更新缓存,后续未迁出的 key 会被错误路由到目标节点;如果不更新缓存,后续已迁出的 key 又会被错误路由回源节点。ASK 重定向是这个矛盾的解法:它表示"这次去那边问问,缓存别动"。

目标节点前置命令无ASKING
持久性永久一次性

27.6.2 ASK 的完整交互流程

图 27-6-1 · 迁移期间的 ASK 跳转
ASK 的工程必要性 · reshard 期间 slot 12001 在 A 与 B 之间, 部分 keys 在两侧:
  • A 持有未迁出的 keys
  • B 持有已迁入的 keys
  • A 与 B 都不能完整服务 slot 12001

✗ 错误方式 1:阻塞至 reshard 完成

服务受影响, 违背"在线迁移"目标

✗ 错误方式 2:A 处理所有, B 缓冲

迁移期间 A 不能 DEL, 迁移无法推进

✓ Redis 选择:按 key 粒度判断

  • A 收到请求, 检查本地是否还有该 key
  • 有 → 直接处理
  • 无 → 返回 ASK, 告知客户端去 B 找
  • B 收到 ASKING + 命令, 接受处理

27.6.3 为什么需要 ASKING

/* src/cluster.c: getNodeByQuery 简化 */
clusterNode *getNodeByQuery(client *c, ...) {
    int slot = keyHashSlot(c->cmd, c->argv);
    clusterNode *node = server.cluster->slots[slot];

    /* slot 不属于本节点 */
    if (node != myself) {
        if (server.cluster->importing_slots_from[slot]) {
            /* 本节点正在 importing 该 slot */
            if (c->flags & CLIENT_ASKING) {
                /* 客户端发了 ASKING,接受 */
                return myself;
            }
        }
        /* 否则返回 MOVED */
        return node;
    }

    if (server.cluster->migrating_slots_to[slot]) {
        /* slot 正在 migrating 出去 */
        if (lookupKey(c->db, c->argv[1]) == NULL) {
            /* key 不在本节点,返回 ASK */
            return server.cluster->migrating_slots_to[slot];
        }
    }
    return myself;
}
ASKING 是一次性令牌:告诉 B "这次请求是从 A 发来的,你接受它",设置 client flag CLIENT_ASKING,一次命令后自动清除。没有 ASKING 时 B 会检查 slot 归属(仍登记在 A),返回 MOVED 12001 A,客户端反复在 A 与 B 之间跳转(死循环)。ASK 是 Redis Cluster 在 reshard 期间维持服务可用的关键机制,核心思想是"区分临时跳转与永久路由"。

27.7 Slot 迁移机制

Slot 迁移由 5 条命令编排完成:源节点先标记 MIGRATING,目标节点先标记 IMPORTING,然后用 GETKEYSINSLOT 拿 key 列表,MIGRATE 按 key/批次搬迁(正常返回时目标已创建且源端删除),最后 SETSLOT NODE 通知集群更新归属。这一节看每条命令的作用,以及超时/网络错误下的确认与重试边界。

27.7.1 迁移涉及的命令

表 27-7-1 · Slot 迁移命令族
命令目标节点作用
CLUSTER SETSLOT <slot> MIGRATING <target-id>源节点slot 标记为"正在迁出"
CLUSTER SETSLOT <slot> IMPORTING <source-id>目标节点slot 标记为"正在导入"
CLUSTER GETKEYSINSLOT <slot> <count>源节点列出该 slot 的 keys
MIGRATE <host> <port> "" 0 5000 KEYS k1 k2 ...源节点逐 key 搬迁;批次不等于分布式事务
CLUSTER SETSLOT <slot> NODE <target-id>所有节点迁移完成,更新 slot 归属

27.7.2 MIGRATE 的原子性

图 27-7-1 · MIGRATE 单 key 的原子三步
MIGRATE 的边界 · 对单个 key 的正常路径:
  • +OK 表示目标写入成功且源端删除完成
  • 明确失败且未发送/未恢复时源端保留 key
  • 网络超时或 IOERR 后不能仅凭客户端结果判断两端状态,需要扫描校验

实现机制

  • 源端 DUMP + RESTORE + DEL 在主线程顺序执行
  • 任意步骤失败, 不执行后续步骤
  • DEL 是 RESTORE 成功后的最后一步

⚠ 注意:MIGRATE 不是分布式事务

  • 多 key 一次 MIGRATE 时, 部分成功部分失败可能存在
  • 每 key 的正常迁移路径有顺序保证, 但批次不保证
  • 失败、超时需要应用层幂等重试或扫描清理

27.7.3 大 key 迁移的阻塞

表 27-7-2 · MIGRATE 单 key 执行成本与阻塞窗口

27.9 数学分析:Gossip 收敛速度

Gossip 常用"流行病传播"模型帮助理解 — 已知节点每轮再感染若干未知节点,感染数近似指数增长直到覆盖大部分集群。本节的 O(log N) 与 1000 节点约 14 秒是理想化推导,用于建立量级直觉;生产收敛还受 cluster-node-timeout、丢包、节点负载与实现采样策略影响。

27.9.1 Push-Based Gossip 的收敛

图 27-9-1 · Gossip 收敛公式推导(理想模型)
设 N 个节点,初始 1 个节点持有信息 (sender)
每轮每个已知节点随机选一个其他节点发送

  I(t) = 第 t 轮已知节点数
  E[I(t+1)] = I(t) + I(t) × (N - I(t)) / N

连续时间近似 (logistic 方程):
  s = I/N  (已知比例)
  ds/dt = s(1 - s)
  s(t) = 1 / (1 + (N-1)e^(-t))

求解全集群收敛:
  令 s(T) = 1 - 1/N (仅剩 1 节点未知)
  解得 T ≈ 2 ln(N - 1) = O(log N)

更精确的高概率收敛:
  T = ln N + c · ln(ln N)
  其中 c 是常数
  对所有节点以概率 (1 - 1/N) 收敛

27.9.2 实际收敛时间估算

表 27-9-1 · 不同集群规模的收敛时间
集群规模 N理论轮数 ⌈2 ln N⌉假设心跳周期模型估算时间
6 节点4 轮1 秒/轮~4 秒
50 节点8 轮1 秒/轮~8 秒

图 27-9-2 · 对数增长的工程意义
对数增长的工程意义:

  集群规模从 6 扩到 1000 (167 倍增长)
  理想模型下收敛时间从 4 秒到 14 秒 (3.5 倍增长)

  log N 增长意味着:
    扩大集群对收敛速度的影响极小
    Gossip 协议天然支持大规模集群
    生产环境 ~1000 节点上限主要受 Gossip 带宽、运维复杂度与故障域影响
    而非收敛速度限制

对比中心化协调:
  中心化协调器(单点模型):
    所有信息流经协调器
    协调器带宽 = O(N²)
    协调器宕机 = 系统不可用

  Gossip (Cluster):
    每节点带宽 = O(N)
    总带宽 = O(N²) 但分散到所有节点
    任意节点宕机不影响系统

27.10.2 Cluster 与 Sentinel 不可共用

不要把同一组 Cluster 数据节点再交给 Sentinel 自动故障转移:Cluster 通过 PFAIL/FAIL 自行处理 failover,Sentinel 通过 SDOWN/ODOWN 处理 failover。两套自动 failover 权威同时操作同一批节点,会导致角色与 slot 归属状态互相覆盖,恢复成本很高。

正确选择:需要分片 + 高可用 → Cluster(内置 failover);仅需高可用,无分片需求 → 主从 + Sentinel。不要混用。

27.12.3 hash tag 设计准则

图 27-12-1 · hash tag 粒度的取舍
hash tag 设计准则:

  ✓ tag 粒度 = 业务关联粒度
    订单与其支付/物流: tag = 订单号
    用户与其购物车/订单: tag = 用户 ID (但要谨慎)

  ✗ tag 过粗 (热点风险)
    所有用户购物车 → 同一 tag → 单 slot 热点
    解决: cart:{user-id} 而非 {cart}:user-id

  ✗ tag 过细 (失去意义)
    单一 key 用 hash tag → 与无 tag 等价

  ✓ 设计验证
    redis-cli CLUSTER KEYSLOT key1
    redis-cli CLUSTER KEYSLOT key2
    确认相关 key 落同 slot
cluster-require-full-coverage 是缓存型业务的隐形杀手:默认 yes 时,只要有任何一个 master 故障(且无 replica 接管),整个集群拒绝所有写入与读取 — 包括其他健康 master 上的 slot。缓存型业务通常不能接受"局部故障 → 全局不可用",生产配置 cluster-require-full-coverage no,让健康 slot 继续服务。

/* src/cluster.c: 路由决策 */
clusterNode *getNodeByQuery(client *c, struct redisCommand *cmd,
                             robj **argv, int argc, int *hashslot,
                             int *error_code) {
    clusterNode *n = NULL;
    int slot = 0;
    int slot_initialized = 0;

    /* 计算 slot (考虑多 key 命令的一致性) */
    for (int i = 0; i < argc; i++) {
        if (cmd->key_specs[i].flags & CMD_KEY_NONE) continue;
        int s = keyHashSlot(argv[i]->ptr, sdslen(argv[i]->ptr));
        if (!slot_initialized) {
            slot = s;
            slot_initialized = 1;
        } else if (s != slot) {
            *error_code = CLUSTER_REDIR_CROSS_SLOT;
            return NULL;
        }
    }
    *hashslot = slot;

    n = server.cluster->slots[slot];
    if (n == NULL) {
        *error_code = CLUSTER_REDIR_DOWN_UNBOUND;
        return NULL;
    }

    /* 不属于本节点 */
    if (n != myself) {
        if (server.cluster->importing_slots_from[slot] != NULL) {
            if (c->flags & CLIENT_ASKING) return myself;
        }
        *error_code = CLUSTER_REDIR_MOVED;
        return n;
    }

    /* slot 正在迁出 */
    if (server.cluster->migrating_slots_to[slot] != NULL) {
        if (lookupKey(c->db, argv[1], LOOKUP_NOTOUCH) == NULL) {
            *error_code = CLUSTER_REDIR_ASK;
            return server.cluster->migrating_slots_to[slot];
        }
    }

    return myself;
}

27.13.3 PFAIL → FAIL 升级

/* src/cluster.c: 多数派达成时升级为 FAIL */
void markNodeAsFailingIfNeeded(clusterNode *node) {
    int failures;
    int needed_quorum = (server.cluster->size / 2) + 1;

    if (!nodeTimedOut(node)) return;
    if (nodeFailed(node)) return;

    failures = clusterNodeFailureReportsCount(node);
    if (nodeIsMaster(myself)) failures++;

    if (failures < needed_quorum) return;

    serverLog(LL_NOTICE, "Marking node %.40s as FAIL (quorum reached).",
              node->name);

    node->flags &= ~CLUSTER_NODE_PFAIL;
    node->flags |= CLUSTER_NODE_FAIL;
    node->fail_time = mstime();

    /* 广播 FAIL 消息给全网 */
    if (nodeIsMaster(myself))
        clusterSendFail(node->name);
}

27.13.4 设计取舍

取舍 1:固定 16384 slot vs 一致性 hash

Cluster 选择固定 16384 slot 而非一致性 hash 的根本原因是 Gossip 包大小约束。一致性 hash 通常通过 virtual nodes 实现细粒度分片,每个 vnode 需要在节点间传播位置信息。Redis Cluster 的 16384 slot 可表示为 2 KB 固定大小 bitmap,直接塞入 PING 消息,集群规模可达 ~1000 节点。一致性 hash 方案下,每节点 256 vnodes 时 1000 节点集群需传播 256,000 vnode 信息,带宽不可控。

取舍 2:Gossip vs Raft / Paxos

Cluster 选择 Gossip 而非 Raft / Paxos 的根本原因是 性能与简单性。Raft 需要 leader 选举与日志复制,所有写操作经过 leader 串行化。Cluster 的 slot 路由可以完全去中心化:每个客户端缓存完整映射,直接路由到 master,无需 leader 协调。Gossip 仅用于"成员状态最终一致",不要求强一致,这与 Cluster 的实际需求(failover 在秒级完成即可)契合。

取舍 3:PFAIL → FAIL 两阶段判定

PFAIL → FAIL 两阶段判定的核心是 "安全性优于活性"。如果允许任何单节点判定故障即触发 failover,网络抖动会导致大量误 failover。两阶段确保 failover 仅在多数派确认时触发,网络分区下少数派分区无法触发 failover,自动降级。代价是故障检测窗口约为 cluster-node-timeout × 2,可接受。

取舍 4:ASK vs MOVED

ASK 重定向是 reshard 期间维持服务可用的关键。如果不区分 ASK 与 MOVED,客户端会反复在源/目标节点间跳转直至 reshard 完成。ASK 表示"临时跳转,不更新缓存",MOVED 表示"永久迁移,更新缓存"。该区分使 reshard 对客户端透明,几乎不影响业务。

用例 1:大型电商订单缓存
某电商订单系统用 Cluster 6 master + 6 replica 架构,16384 slot 均匀分布。关键设计:订单与其支付/物流用 {order_id} hash tag 强制同 slot,MGET 一次拉完;cluster-node-timeout 配置 5000 ms,failover 总时长 6-8 秒;cluster-require-full-coverage 设为 no,单 master 故障期间其他 slot 继续服务。实际运行:日均 10 亿写,稳态 P99 0.9 ms,reshard 期间 P99 短暂飙到 2 ms 后回落。
用例 2:不要这么用 — 把 Cluster 当 Kafka
反例:某团队把所有事件写入一个 Stream,放在 Cluster 单 slot 上。结果:该 slot 所在 master CPU 100%,其他 5 个 master 几乎空闲,扩容到 12 节点也没用 — 单 key 是 slot 上限。正解:按 key 前缀(events:user:{uid})分散到多 slot,或直接换 Kafka / Pulsar 这类原生分区的消息系统。Cluster 的水平扩展粒度是 slot,不是 key。
