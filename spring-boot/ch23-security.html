第 23 章 · Spring Security 过滤器链

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

安全机制不是"功能完成后再附加"的边角组件,而是每个 HTTP 请求必须穿越的一条管线。在 Spring Boot 3.x 默认 HttpSecurity 配置下,这条管线由约 14 个 Servlet 过滤器组成,实际数量随启用的安全特性(OAuth2、Remember-Me、Session 管理、CSRF 模式等)而变化。Spring Security 的核心价值在于:即使应用一行配置都不写,默认就开启了 CSRF 防护、Session 固定攻击防护、安全响应头与基础认证。

本章按"Servlet 桥接 → 过滤器逐层 → 认证体系 → OAuth2 流程 → JWT 验签 → CSRF 防护 → SecurityContext 传播 → 方法级安全 → 陷阱与限制"的顺序展开。前置知识:第 5 章 AOP 帮助理解责任链模式,第 14 章 Spring MVC 帮助理解 Servlet 过滤器执行位置。

23.1 Servlet Filter 桥接到 Spring Security

Servlet 规范定义了 jakarta.servlet.Filter 接口,用于在请求到达 Servlet 之前进行拦截处理。Spring Security 没有直接向 Servlet 容器注册十几个独立的 Filter 实例,而是通过一个"桥接"设计将所有安全过滤器封装在自己的过滤器链中。

23.1.1 DelegatingFilterProxy 桥接架构

DelegatingFilterProxy 是注册在 Servlet 容器中的标准 Filter,它的唯一职责是把请求转发给 Spring ApplicationContext 中名为 springSecurityFilterChain 的 Bean(即 FilterChainProxy 实例)。这种间接性使 Spring Security 的所有过滤器都能享受依赖注入、生命周期管理与 Bean 后置处理。

图 23-1 · DelegatingFilterProxy 三层桥接架构

FilterChainProxy 内部维护一个 List<SecurityFilterChain>,每个 SecurityFilterChain 绑定一个 RequestMatcher。当请求匹配时,该链中的所有过滤器依次执行。只有第一个匹配的链会被执行——这就是"优先匹配"策略,这个特性带来的最大风险就是顺序错误。

23.1.2 FilterChainProxy 匹配决策

图 23-2 · FilterChainProxy 匹配决策流程

定义多个 SecurityFilterChain 时,更具体的路径规则(如 /api/**)必须排在通配规则(/**)之前,否则所有请求都会被通配链捕获,导致 API 安全策略失效。@Order 注解用于控制 Bean 的优先级。

23.1.3 多链配置案例

下面这段配置把 API 请求与后台页面拆成两条安全链。读代码时不能只看 authorizeHttpRequests,还要看 securityMatcher 与 @Order——它们决定请求先落到哪条链。

@Bean
@Order(1)
SecurityFilterChain apiChain(HttpSecurity http) throws Exception {
    return http
        .securityMatcher("/api/**")
        .sessionManagement(session ->
            session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
        .csrf(AbstractHttpConfigurer::disable)
        .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()))
        .authorizeHttpRequests(auth -> auth
            .requestMatchers(HttpMethod.GET, "/api/public/**").permitAll()
            .anyRequest().authenticated()
        )
        .build();
}

@Bean
@Order(2)
SecurityFilterChain webChain(HttpSecurity http) throws Exception {
    return http
        .securityMatcher("/**")
        .formLogin(Customizer.withDefaults())
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/", "/login", "/assets/**").permitAll()
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated()
        )
        .build();
}

开启 logging.level.org.springframework.security=DEBUG 后,访问 /api/orders 应只命中第一条链,不会再落入 /** 兜底链:

DEBUG o.s.security.web.FilterChainProxy : Securing GET /api/orders
DEBUG o.s.security.web.FilterChainProxy : Invoking BearerTokenAuthenticationFilter (7 12)
DEBUG o.s.security.web.FilterChainProxy : Invoking AuthorizationFilter (12 12)
DEBUG o.s.security.web.FilterChainProxy : Secured GET /api/orders

如请求未携带 Bearer Token,这条无状态 API 链会返回认证挑战,而非跳转登录页:

GET /api/orders HTTP/1.1
Host: localhost:8080

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer
Cache-Control: no-cache, no-store, max-age=0, must-revalidate

23.2.4 ExceptionTranslationFilter 的两路分流

ExceptionTranslationFilter 是过滤器链中最容易被忽视但最重要的一层。它捕获下游过滤器(主要是 AuthorizationFilter)抛出的两类异常,并将其转化为不同的 HTTP 响应。

异常类型 含义 默认处理 HTTP 状态
AuthenticationException 当前请求未认证或认证已失效 调用 AuthenticationEntryPoint 401 或登录页
AccessDeniedException 已认证但权限不足 调用 AccessDeniedHandler 403

不同认证模式的 EntryPoint 不同:表单登录链使用 LoginUrlAuthenticationEntryPoint(返回 302 跳转登录页);JWT 资源服务器链使用 BearerTokenAuthenticationEntryPoint(返回 401 加 WWW-Authenticate: Bearer)。

23.3 认证架构

Spring Security 的认证核心由三个组件组成:AuthenticationManager 是入口接口,ProviderManager 是默认实现,AuthenticationProvider 是实际执行认证的策略。

23.3.1 AuthenticationManager 与 ProviderManager

图 23-4 · AuthenticationManager 三层认证架构

ProviderManager 遍历所有 AuthenticationProvider,调用其 supports(Class) 方法判断是否能处理当前 Token 类型。第一个认证成功的 Provider 会返回结果;如某个 Provider 不支持当前 Token,则继续尝试下一个。认证失败通常会记录为最后一个异常,某些账户状态类异常会立即中止链路。实际项目中通常让一种 Token 类型只对应一个明确的 Provider,避免诊断困难。

23.3.2 用户名密码认证的最小可运行示例

最小可运行认证示例通常由 UserDetailsService、PasswordEncoder 与默认表单登录组成。认证成功后,服务端会建立 SecurityContext 并把会话标识写回浏览器。

@Bean
UserDetailsService users(PasswordEncoder encoder) {
    UserDetails admin = User.withUsername("admin")
        .password(encoder.encode("secret"))
        .roles("ADMIN")
        .build();
    return new InMemoryUserDetailsManager(admin);
}

@Bean
PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder();
}

表单登录成功时,默认行为是 302 跳转并下发 JSESSIONID:

Content-Type: application/x-www-form-urlencoded

username=admin and password=secret

HTTP/1.1 302 Found

Location: /

密码错误时,DaoAuthenticationProvider 会抛出 BadCredentialsException,随后由失败处理器转成登录失败响应:

23.3.3 PasswordEncoder 选型

算法 安全等级 计算开销 适用场景
BCrypt 高 中(可调强度) 默认推荐,Spring Security 5+ 内置
Argon2 极高 高(内存密集) 新项目首选,防 GPU 攻击
PBKDF2 中 中 FIPS 合规场景
SCrypt 高 高 内存密集型场景
NoOpPasswordEncoder 无 极低 仅用于测试,生产环境严禁

DelegatingPasswordEncoder 是 Spring Security 5+ 的默认实现,它通过密码前缀(如 {bcrypt}$2a$10$...)识别算法,支持密码算法的平滑升级——旧密码继续用旧算法验证,新密码用新算法加密。

23.3.4 URL 授权决策

授权发生在认证之后。下面的规则允许普通用户访问 /account/me,但只有 ADMIN 能进入 /admin/**:

@Bean
SecurityFilterChain authorizationChain(HttpSecurity http) throws Exception {
    return http
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/account/me").hasAnyRole("USER", "ADMIN")
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated()
        )
        .formLogin(Customizer.withDefaults())
        .build();
}

用户 alice 只有 ROLE_USER 时,访问个人接口成功:

GET /account/me HTTP/1.1
Cookie: JSESSIONID=7C1A3E8F0F1C0E5E9A1D4C8B8B7F1A20

HTTP/1.1 200 OK
Content-Type: application/json

{"status":200, "principal":"alice", "authorities":["ROLE_USER"]}

同一个用户访问后台接口会被 AuthorizationFilter 拦截,最终由 ExceptionTranslationFilter 输出 403:

GET /admin/reports HTTP/1.1
Cookie: JSESSIONID=7C1A3E8F0F1C0E5E9A1D4C8B8B7F1A20

HTTP/1.1 403 Forbidden
Content-Type: application/json

{"status":403, "error":"Forbidden", "path":"/admin/reports"}

23.4 OAuth2 Authorization Code Flow

OAuth2 授权码模式是最安全的 OAuth2 授权流程,适用于有后端服务器的 Web 应用。整个流程涉及浏览器、客户端应用、授权服务器与资源服务器四方交互。

23.4.1 授权码完整流程

图 23-5 · OAuth2 授权码完整流程(浏览器/客户端/授权服务器/资源服务器 四方时序)

state 参数是防 CSRF 的关键。客户端生成随机值,授权服务器原封不动地在步骤 6 中返回。客户端验证 state 是否一致——如不一致,说明请求被篡改,必须拒绝。

23.4.2 PKCE 扩展:公开客户端

对于原生应用(移动端、SPA)等无法安全存储 client_secret 的公开客户端,PKCE(Proof Key for Code Exchange,RFC 7636)提供了额外的安全保障。

图 23-6 · PKCE 扩展流程(无 client_secret 的公开客户端如何防止授权码被劫持)

PKCE 安全原理:即使攻击者截获了授权码 code,由于不知道 code_verifier,也无法换取 token。code_challenge 是 SHA256 哈希值,无法逆向推导出 code_verifier。

23.4.3 Token 存储策略对比

存储方式 安全性 XSS 风险 CSRF 风险 适用场景
HttpOnly Cookie 极高 低(JS 无法读取) 需 CSRF 防护 传统 Web 应用
内存变量 高 低(页面刷新丢失) 无 SPA 短生命周期
SessionStorage 中 中(同标签页可访问) 无 SPA 单标签页
LocalStorage 低 高(任何 JS 可读) 无 不推荐用于敏感 Token

23.4.4 Spring Authorization Server 自建授权服务器

Spring 官方的 Spring Authorization Server 项目(从 Spring Security 5.x 独立出来)实现了完整的 OAuth2 授权码流程,包括客户端注册、授权端点、令牌端点与 JWKS 端点。它不再依赖第三方授权服务器(如 Keycloak),适合需要完全掌控认证逻辑的企业内部系统。Token 签发使用 RSA 非对称密钥,资源服务器通过 JWKS 端点获取公钥进行本地验签。

@Bean
public RegisteredClientRepository registeredClientRepository() {
    RegisteredClient client = RegisteredClient.withId(UUID.randomUUID().toString())
        .clientId("my-spa-client")
        .clientAuthenticationMethod(ClientAuthenticationMethod.NONE)
        .authorizationGrantType(AuthorizationGrantType.AUTHORIZATION_CODE)
        .redirectUri("http://localhost:3000/callback")
        .scope(OidcScopes.OPENID)
        .scope("read")
        .clientSettings(ClientSettings.builder()
            .requireProofKey(true)
            .build())
        .build();
    return new InMemoryRegisteredClientRepository(client);
}

23.4.5 Keycloak 与 OIDC 集成

Spring Boot 3.x 与 Spring Security 6 中,不再推荐使用老的 Keycloak Spring Security 适配器。更稳妥的方式是把 Keycloak 当作标准 OIDC Provider:登录应用使用 oauth2Login(),资源服务器使用 oauth2ResourceServer().jwt(),通过 issuer 与 JWKS 验证 Bearer Token,然后把 JWT claims 转成 Spring Security authorities。

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http
        .oauth2ResourceServer(oauth2 -> oauth2
            .jwt(jwt -> jwt
                .jwkSetUri("https://keycloak.example.com/realms/myrealm/protocol/openid-connect/certs")
                .jwtAuthenticationConverter(keycloakJwtConverter())
            )
        )
        .authorizeHttpRequests(auth -> auth
            .requestMatchers("/admin/**").hasRole("ADMIN")
            .anyRequest().authenticated()
        );
    return http.build();
}

23.5 JWT 深度解析

JWT(JSON Web Token,RFC 7519)由三部分组成,以点号分隔:Header.Payload.Signature。每一部分都是 Base64URL 编码的 JSON。JWT 默认只是签名,不是加密——任何拿到 token 的人都能解码 header 与 payload,因此不应把密码、身份证号、内部密钥等敏感数据放进 claims。

23.5.1 JWT 三段式结构

图 23-7 · JWT 三段式结构(Header . Payload . Signature)

Payload 中的标准 claim 名称遵循 RFC 7519:iss(签发者)、sub(主体)、aud(受众)、exp(过期时间)、nbf(生效时间)、iat(签发时间)、jti(JWT ID,防重放)。自定义 claim 应避免与标准 claim 冲突。

23.5.2 Token 生命周期

图 23-8 · Access Token 与 Refresh Token 双 Token 生命周期

Access Token 通常短期有效(15 分钟至 1 小时),Refresh Token 长期有效(数天至数月)。这种分离设计的目的是:即使 Access Token 泄露,影响窗口也很短;Refresh Token 可以撤销且不暴露在每次业务请求中。

23.5.3 HS256 与 RS256 对比

特性 HS256 HMAC-SHA256 RS256 RSA-SHA256
算法类型 对称签名 非对称签名
密钥 共享密钥 secret 私钥签名 公钥验证
性能 签名与验证都较快 签名更慢,验证可在资源服务器本地完成
密钥分发 需安全传输共享密钥 公钥可公开分发(JWKS)
微服务适用性 中(所有服务需持有密钥) 高(资源服务器只需公钥)
安全风险 密钥泄露即可伪造签名 私钥泄露才危险,公钥泄露无影响

23.5.4 JWKS 端点发现流程

图 23-9 · JWKS 端点发现与公钥缓存流程

kid(Key ID)字段允许授权服务器同时维护多个签名密钥。这在密钥轮换(Key Rotation)场景下至关重要:旧 Token 用旧 kid 验证,新 Token 用新 kid 验证,平滑过渡而无需所有 Token 立即失效。

23.5.5 Spring Cloud Gateway 边缘验签

在微服务架构中,Spring Cloud Gateway 作为边缘网关可以在入口处验证 JWT,而非每个微服务独立验证。Gateway 启动时从授权服务器的 JWKS 端点一次性拉取公钥并缓存(定期刷新),后续所有请求的 JWT 验签都在本地完成,无网络开销。验签通过后,将用户信息作为请求头转发给下游微服务。由于 Gateway 是响应式架构,安全配置使用的是 WebFlux 的 ServerHttpSecurity。

@Bean
public SecurityWebFilterChain springSecurityFilterChain(ServerHttpSecurity http) {
    return http
        .csrf(ServerHttpSecurity.CsrfSpec::disable)
        .authorizeExchange(exchanges -> exchanges
            .pathMatchers("/auth/**", "/public/**").permitAll()
            .anyExchange().authenticated()
        )
        .oauth2ResourceServer(oauth2 -> oauth2
            .jwt(jwt -> jwt
                .jwkSetUri("https://auth.example.com/.well-known/jwks.json")
            )
        )
        .build();
}

23.5.6 无状态与有状态对比输出

JWT 资源服务器链通常是无状态的:认证信息来自每次请求的 Authorization 头,响应里不会创建 JSESSIONID。

GET /api/orders/O1001 HTTP/1.1
Authorization: Bearer eyJraWQiOiJrZXktMDEiLCJhbGciOiJSUzI1NiJ9...

HTTP/1.1 200 OK
Content-Type: application/json
X-Authenticated-Customer: C1001

{"orderId":"O1001", "customerId":"C1001", "status":"PAID"}

无 Token 或 Token 过期时,资源服务器返回 Bearer 错误,不依赖 Session 恢复身份:

GET /api/orders/O1001 HTTP/1.1
Authorization: Bearer expired.jwt.value

HTTP/1.1 401 Unauthorized
WWW-Authenticate: Bearer error="invalid_token", error_description="Jwt expired at 2026-05-02T07:20:00Z"

传统表单登录则是有状态的:登录成功后身份保存在服务端 Session 中,后续请求靠 Cookie 找回 SecurityContext。

GET /account/me HTTP/1.1
Cookie: JSESSIONID=7C1A3E8F0F1C0E5E9A1D4C8B8B7F1A20

HTTP/1.1 200 OK
Content-Type: application/json

{"username":"admin", "authorities":["ROLE_ADMIN"], "contextSource":"HttpSessionSecurityContextRepository"}

CSRF(Cross-Site Request Forgery,跨站请求伪造)攻击利用浏览器自动携带 Cookie 的特性,诱导已登录用户在不知情的情况下发送恶意请求。Spring Security 默认开启 CSRF 防护。

23.6.4 SPA 与 Spring Security 配合

当 React 或 Vue 单页应用与 Spring Boot API 后端分离部署时,CSRF 防护通常使用 CookieCsrfTokenRepository.withHttpOnlyFalse()。原因是:前端 JavaScript 需要读取 Cookie 中的 CSRF Token 值,然后在每个非 GET 请求的 X-XSRF-TOKEN 请求头中携带。如 Cookie 设置了 HttpOnly,JavaScript 无法按这个约定回传请求头,请求会被 CsrfFilter 拒绝。Axios 默认支持自动读取 XSRF-TOKEN Cookie 并设置请求头。

@Bean
public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    http.csrf(csrf -> csrf
        .csrfTokenRepository(CookieCsrfTokenRepository.withHttpOnlyFalse())
        .csrfTokenRequestHandler(new CsrfTokenRequestAttributeHandler())
    );
    return http.build();
}

{"status":403, "error":"Forbidden", "message":"Invalid CSRF token found"}

前端把 Cookie 中的 XSRF-TOKEN 同步放到 X-XSRF-TOKEN 请求头后,同一个请求才会通过:

POST /api/profile HTTP/1.1
Cookie: JSESSIONID=7C1A3E8F0F1C0E5E9A1D4C8B8B7F1A20; XSRF-TOKEN=abc123
X-XSRF-TOKEN: abc123
Content-Type: application/json

{"displayName":"Customer C1001"}

HTTP/1.1 200 OK
Content-Type: application/json

{"username":"alice", "displayName":"Customer C1001", "updated":true}

23.7 SecurityContext 传播

SecurityContextHolder 默认使用 MODE_THREADLOCAL 策略,将认证信息存储在当前线程的 ThreadLocal 中。这在同步 Servlet 模型中运行良好,但在异步与响应式场景下面临挑战。

23.7.1 ThreadLocal 存储模型

图 23-13 · SecurityContext 的 ThreadLocal 存储与异步线程的"丢失"

SecurityContextHolderFilter 在请求开始时从 SecurityContextRepository(如 HttpSessionSecurityContextRepository)加载 SecurityContext 到 ThreadLocal,在请求结束时清理 ThreadLocal。这保证了同一线程在请求生命周期内可以通过静态方法 SecurityContextHolder.getContext() 获取当前用户。

23.7.2 异步场景的传播策略

图 23-14 · SecurityContext 的四种异步传播方案对比

使用 CompletableFuture.supplyAsync() 或直接 new Thread() 创建的线程不会自动继承 SecurityContext。必须使用 DelegatingSecurityContextRunnable 包装 Runnable,或使用被 DelegatingSecurityContextExecutorService 包装过的线程池。

23.7.3 WebFlux 与 ReactiveSecurityContextHolder

在响应式 Spring Security 中,SecurityContext 不存储在 ThreadLocal 中,而是通过 Reactor 的 Context(类似协程的上下文)传播。这是因为 WebFlux 的请求可能在多个线程间切换(EventLoop 线程、boundedElastic 线程等),ThreadLocal 无法跨线程传递。

获取当前用户的方式也完全不同:MVC 用 SecurityContextHolder.getContext()(同步),WebFlux 用 ReactiveSecurityContextHolder.getContext()(返回 Mono):

@GetMapping("/me")
public UserInfo currentUser() {
    Authentication auth = SecurityContextHolder.getContext().getAuthentication();
    return new UserInfo(auth.getName(), auth.getAuthorities());
}

@GetMapping("/me-reactive")
public Mono<UserInfo> currentUserReactive() {
    return ReactiveSecurityContextHolder.getContext()
        .map(SecurityContext::getAuthentication)
        .map(auth -> new UserInfo(auth.getName(), auth.getAuthorities()));
}

23.7.4 异步线程的修复方案

@Async
public CompletableFuture<Report> generateReport() {
    Authentication auth = SecurityContextHolder.getContext().getAuthentication();
    return CompletableFuture.completedFuture(new Report(auth.getName()));
}

@Bean
public TaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    executor.setMaxPoolSize(20);
    executor.setThreadNamePrefix("async-secure-");
    executor.initialize();
    return new DelegatingSecurityContextAsyncTaskExecutor(executor);
}

包装后的线程池在每次提交任务时,自动捕获提交线程的 SecurityContext 并在执行任务的线程中恢复,任务结束后清理。这种"装饰器模式"对业务代码完全透明。

23.8 方法级安全

URL 授权适合保护入口,方法级安全适合保护业务能力。实际项目里二者通常同时存在:Controller 先挡住明显不该访问的路径,Service 再用业务参数做细粒度检查。

23.8.1 注解驱动的方法级安全

@Configuration
@EnableMethodSecurity
class MethodSecurityConfig {
}

@Service
class InvoiceService {
    @PreAuthorize("hasRole('ADMIN') or #owner == authentication.name")
    public InvoiceDto findInvoice(String owner, String invoiceId) {
        return repository.findByOwnerAndId(owner, invoiceId);
    }

    @PreAuthorize("hasAuthority('payment:approve')")
    public void approve(String invoiceId) {
        repository.markApproved(invoiceId);
    }

    @PostAuthorize("returnObject.owner == authentication.name")
    public InvoiceDto getMyInvoice(String invoiceId) {
        return repository.findById(invoiceId);
    }

    @PreFilter("filterObject.amount < 10000")
    public void batchApprove(List<InvoiceDto> invoices) {
        invoices.forEach(this::approveOne);
    }
}

23.8.2 SpEL 表达式与方法参数

注解 执行时机 表达式可用变量 典型用途
@PreAuthorize 方法执行前 authentication,方法参数(#paramName) 基于参数的访问控制
@PostAuthorize 方法执行后 returnObject,authentication 基于返回值的过滤
@PreFilter 方法执行前 filterObject(集合元素) 过滤入参集合
@PostFilter 方法执行后 filterObject(返回集合元素) 过滤返回集合
@Secured 方法执行前 仅角色字符串 简单角色检查(已弃用)

23.8.3 方法级安全的运行输出

当 alice 访问自己的发票时,SpEL 表达式 #owner == authentication.name 为真,方法正常返回:

INFO o.s.s.a.m.AuthorizationManagerBeforeMethodInterceptor : Authorized invocation InvoiceService.findInvoice
INFO c.example.InvoiceController : returned invoice INV-1001 to alice amount=1280.00

同一个用户审批支付时,因为缺少 payment:approve 权限,拦截发生在方法体执行之前:

DEBUG o.s.s.a.m.AuthorizationManagerBeforeMethodInterceptor : Authorizing method invocation ReflectiveMethodInvocation: void InvoiceService.approve String
DEBUG o.s.s.a.m.AuthorizationManagerBeforeMethodInterceptor : Failed to authorize ReflectiveMethodInvocation
ERROR c.example.ApiExceptionHandler : AccessDeniedException: Access Denied

对外的 HTTP 响应仍然是 403,避免把内部方法名与 SpEL 细节暴露给客户端:

POST /api/invoices/INV-1001/approve HTTP/1.1
Cookie: JSESSIONID=9A9E0D6D8C4B4F0A8F1E3C2D1A0B5C77

HTTP/1.1 403 Forbidden
Content-Type: application/json

{"status":403, "error":"Forbidden", "message":"Access is denied"}

23.9 陷阱与限制

23.9.1 陷阱 1:JWT 的 alg none 攻击

JWT 标准允许 alg: "none"(无签名)。如验证库没有显式拒绝 none 算法,攻击者可以伪造任意 JWT 绕过认证。

修复:验证 JWT 时必须显式指定允许的算法列表。JwtDecoders.fromIssuerLocation(issuerUri) 默认安全;但自定义 NimbusJwtDecoder 时需确认 .jwsAlgorithm(SignatureAlgorithm.RS256) 已设置。绝不允许客户端控制验证算法。

NimbusJwtDecoder decoder = NimbusJwtDecoder.withJwkSetUri(jwksUri)
    .jwsAlgorithm(SignatureAlgorithm.RS256)
    .build();
decoder.setJwtValidator(JwtValidators.createDefaultWithIssuer(issuer));

23.9.2 陷阱 2:SecurityContext 在异步线程中丢失

SecurityContextHolder 默认使用 ThreadLocal 策略。当使用 @Async、CompletableFuture 或手动创建线程时,新线程没有 SecurityContext,Authentication 为 null,权限检查失败。

修复方案 A 适合手动创建的短生命周期子线程,线程池中不推荐:

SecurityContextHolder.setStrategyName(
    SecurityContextHolder.MODE_INHERITABLETHREADLOCAL);

修复方案 B 是生产推荐,使用 Spring 提供的装饰器:

@Bean
public TaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    return new DelegatingSecurityContextAsyncTaskExecutor(executor);
}

23.9.3 陷阱 3:CSRF 保护与前后端分离冲突

Spring Security 默认开启 CSRF 保护。SPA 前端通过 AJAX 发送 POST 或 PUT 或 DELETE 请求时,如未正确传递 CSRF Token,会返回 403 Forbidden。许多开发者直接 csrf().disable() 关闭 CSRF——这是错误的做法,等于把所有写操作暴露给跨站攻击。

修复:使用 CookieCsrfTokenRepository.withHttpOnlyFalse(),将 CSRF Token 放在 Cookie 中(XSRF-TOKEN),前端框架(如 Axios、Angular HttpClient)会自动读取并放入请求头 X-XSRF-TOKEN。

23.9.4 陷阱 4:多个 SecurityFilterChain 的匹配顺序

如有多条 SecurityFilterChain,必须确保具体路径的链排在前面,/** 通配的链排在最后,因为 FilterChainProxy 只执行第一个匹配的链。

错误做法:

@Order(1)
http.securityMatcher("/**")
    .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())

@Order(2)
http.securityMatcher("/api/**")
    .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()))

正确做法:

@Order(1)
http.securityMatcher("/api/**")
    .oauth2ResourceServer(oauth2 -> oauth2.jwt(Customizer.withDefaults()))

@Order(2)
http.securityMatcher("/**")
    .authorizeHttpRequests(auth -> auth.anyRequest().authenticated())

23.9.5 陷阱 5:JWT 时钟偏移导致 Token 验证失败

签发服务器与验证服务器之间时钟不同步(在容器与云环境中很常见),会导致刚签发的 Token 因 nbf(Not Before)检查失败而被拒绝。

修复:配置时钟偏移容忍 jwtDecoder.setClockSkew(Duration.ofSeconds(60))。同时确保所有服务器使用 NTP 同步时钟。

NimbusJwtDecoder decoder = NimbusJwtDecoder.withJwkSetUri(jwksUri).build();
OAuth2TokenValidator<Jwt> withClockSkew = new DelegatingOAuth2TokenValidator<>(
    new JwtTimestampValidator(Duration.ofSeconds(60)),
    new JwtIssuerValidator(issuer));
decoder.setJwtValidator(withClockSkew);

23.9.6 陷阱 6:HttpOnly Cookie 与 SPA 读取需求

CookieCsrfTokenRepository 默认 httpOnly=true,JavaScript 无法读取。SPA 场景必须使用 withHttpOnlyFalse() 工厂方法,但要注意:这意味着 XSS 漏洞可以读取 CSRF Token——前端的 XSS 防护(CSP、Sanitizer)同样重要。

23.9.7 陷阱 7:HttpSession 序列化与集群

HttpSessionSecurityContextRepository 默认把 SecurityContext 存储在 HttpSession 中。如启用了 Spring Session(Redis 或 JDBC 后端),SecurityContext、Authentication 与 Principal 必须实现 Serializable。自定义 Authentication 实现忘记实现 Serializable 会导致集群环境下登录态无法跨节点复用。

23.10 源码入口与扩展边界

Spring Security 的源码入口在 Servlet 层:容器先调用 DelegatingFilterProxy,它再委托 Spring Bean FilterChainProxy。FilterChainProxy 只做两件事:按请求匹配第一条 SecurityFilterChain,再按顺序执行链内安全过滤器。

23.11 本章小结

Servlet 桥接:DelegatingFilterProxy 把 Spring 管理的 FilterChainProxy 接入 Servlet 容器,使安全过滤器能享受依赖注入与生命周期管理。

过滤器链:Spring Boot 3.x 默认配置约 14 个过滤器,从 DisableEncodeUrlFilter 到 AuthorizationFilter,覆盖 Session 加载、CSRF、认证、异常翻译与最终授权。多 SecurityFilterChain 配置时,具体路径链必须排在通配链之前。

认证体系:AuthenticationManager → ProviderManager → AuthenticationProvider 三层架构;UserDetailsService 与 PasswordEncoder 是用户名密码认证的核心;DelegatingPasswordEncoder 支持算法平滑升级。

OAuth2 流程:授权码模式四方(浏览器、客户端、授权服务器、资源服务器)交互;state 防 CSRF;PKCE 为公开客户端提供额外保障;Spring Authorization Server 支持自建授权服务。

JWT 验签:Header.Payload.Signature 三段式;HS256 对称 与 RS256 非对称对比;JWKS 端点配合 kid 实现密钥轮换;Spring Cloud Gateway 边缘验签减少下游服务负担。

CSRF 防护:Synchronizer Token 与 Double Submit Cookie 两种模式;SPA 必须使用 CookieCsrfTokenRepository.withHttpOnlyFalse();Bearer Token 与纯 API 服务可禁用 CSRF。

SecurityContext 传播:ThreadLocal 默认策略;异步场景使用 DelegatingSecurityContextAsyncTaskExecutor;WebFlux 用 ReactiveSecurityContextHolder 与 Reactor Context。

方法级安全:@PreAuthorize 与 @PostAuthorize 与 @PreFilter 与 @PostFilter 配合 SpEL 表达式;基于参数与返回值的细粒度授权。

七大陷阱:JWT alg none 攻击;异步线程 SecurityContext 丢失;CSRF 与 SPA 冲突;多链匹配顺序;时钟偏移导致 nbf 失败;HttpOnly 与 SPA 读取冲突;Session 序列化与集群。
