第 16 章 · 事务管理

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

数据库里资金减少而订单未创建成功——这种部分成功是最危险的 bug。事务管理确保一组操作要么全部成功,要么全部回滚,是所有业务系统的基石。Spring 事务抽象通过一个 @Transactional 注解管理 JDBC、JPA、R2DBC 甚至分布式事务,但魔鬼藏在细节里。

本章从 Spring 事务抽象层架构设计起,深入 @Transactional 背后的 AOP 代理机制,逐一解析七种传播行为与四种隔离级别,剖析回滚规则与异常处理关系,对比声明式与编程式事务适用场景,最后探讨分布式事务方案。

16.1 事务抽象层架构

Spring 事务管理的核心设计理念是抽象——不管底层是 JDBC、Hibernate、JPA 还是 JTA,上层代码使用统一的 API。该抽象由三个核心接口构成:PlatformTransactionManager(事务管理器)、TransactionDefinition(事务定义)、TransactionStatus(事务状态)。

16.2 @Transactional 代理机制

@Transactional 不直接修改代码。Spring 通过 AOP 代理在目标方法前后织入事务管理逻辑。核心执行者是 TransactionInterceptor。

图 16-2-1 · 代理创建流程(扫描 → Advisor → JDK/CGLIB → 注入 Interceptor)
图 16-2-2 · 方法调用完整时序(getTransaction → invoke → commit/rollback)
图 16-2-3 · ThreadLocal 资源绑定(同线程共享 Connection — @Async 必丢失)

一个线程对应一个事务资源上下文。这就是 @Async 导致原事务上下文丢失的原因——异步方法在另一个线程执行,该线程没有绑定原事务的 Connection。

protected Object invokeWithinTransaction(Method method, Class<?> targetClass,
        InvocationCallback invocation) throws Throwable {
    TransactionAttributeSource tas = getTransactionAttributeSource();
    TransactionAttribute txAttr = tas.getTransactionAttribute(method, targetClass);
    PlatformTransactionManager ptm = determineTransactionManager(txAttr);
    TransactionInfo txInfo = createTransactionIfNecessary(ptm, txAttr, methodId);

    Object retVal;
    try {
        retVal = invocation.proceedWithInvocation();
    } catch (Throwable ex) {
        completeTransactionAfterThrowing(txInfo, ex);
        throw ex;
    } finally {
        cleanupTransactionInfo(txInfo);
    }
    commitTransactionAfterReturning(txInfo);
    return retVal;
}

16.3 七种传播行为

传播行为(Propagation)回答核心问题:一个事务方法调用另一个事务方法时,如何处理事务边界。Spring 定义 7 种传播行为。

图 16-3-1 · REQUIRED 传播行为(默认 — 加入或创建新事务)
图 16-3-2 · REQUIRES_NEW 传播行为(挂起外层 + 新事务独立提交)
图 16-3-3 · NESTED 传播行为(同事务 + Savepoint — 部分回滚不影响外层)

表 16-3-1:七种传播行为对比

传播行为外层有事务外层无事务典型场景
REQUIRED加入创建新事务默认选择,绝大多数业务方法
REQUIRES_NEW挂起外层,创建新事务创建新事务审计日志、独立记账
NESTED创建 Savepoint创建新事务批量处理中单条失败不影响整批

MANDATORY加入抛异常DAO 层方法
NEVER抛异常非事务运行明确禁止在事务中调用

16.4 隔离级别与锁策略

隔离级别(Isolation Level)解决并发事务之间的数据可见性问题。SQL 标准定义四个级别,从低到高隔离强度递增,并发性能递减。

16.5 回滚规则与异常处理

Spring 事务回滚规则直接决定什么情况下数据被撤销。这是极易出错的区域——默认行为与许多开发者直觉不一致。

图 16-5-1 · 默认回滚规则(RuntimeException 回滚 / Checked 提交 — 最大陷阱)

默认行为设计理由:Spring 沿用 EJB 传统。Checked Exception 视为可恢复业务异常,应由调用者处理而非自动回滚;RuntimeException 视为不可恢复编程错误,应当回滚。实际项目中很多自定义业务异常继承 Exception,不会触发回滚——这是 Bug 温床。

@Transactional(rollbackFor = Exception.class)
public void createOrder(OrderDTO dto) throws BusinessException {
    orderRepository.save(new Order(dto.id(), "CREATED"));
    creditService.check(dto.customerId());
}

@Transactional(rollbackFor = Exception.class, noRollbackFor = DuplicateOrderException.class)
public void processOrder(OrderDTO dto) throws Exception { }

16.6 编程式事务

声明式事务(@Transactional)适用 80% 场景,但有些场景需要更精细控制——一个方法中提交多次事务、动态决定隔离级别、批处理中细粒度管理每批事务边界。

public ImportResult importOrders(List<OrderDTO> orders) {
    ImportResult result = new ImportResult();
    Lists.partition(orders, 50).forEach(batch -> {
        txTemplate.executeWithoutResult(status -> {
            try {
                for (OrderDTO dto : batch) orderRepository.save(toEntity(dto));
                result.addSuccess(batch.size());
            } catch (Exception e) {
                status.setRollbackOnly();
                result.addFailure(batch.size(), e.getMessage());
            }
        });
    });
    return result;
}

16.7 分布式事务

业务操作涉及多个数据库、消息队列或微服务时,单机事务不再足够——需要分布式事务保证跨资源数据一致性。

16.9 源码深潜

事务源码入口不是 @Transactional 注解本身,而是代理链中的 TransactionInterceptor#invoke。它读取事务属性,委托 PlatformTransactionManager 开启事务,把连接绑定到 TransactionSynchronizationManager,根据目标方法的返回或异常决定提交还是回滚。

图 16-9-1 · 源码调用链(完整事务生命周期 — 6 层调用栈)

读源码抓住三层:TransactionInterceptor 是 AOP 切入点,TransactionAspectSupport 是模板算法,PlatformTransactionManager 是资源适配层。JDBC、JPA、JTA、R2DBC 的差异都被压到具体 manager 的 doBegin / doCommit / doRollback 中。

表 16-9-1:架构设计取舍

设计点取舍生产影响
声明式事务走 AOP 代理自调用 / private / final 会绕过业务代码干净,但必须理解代理边界
PlatformTransactionManager 抽象资源每种资源管理器都要适配JDBC / JPA / JTA / R2DBC 共享 @Transactional 语义
事务资源放入 ThreadLocal跨线程天然丢失上下文同一请求线程内 Repository 复用同一连接
回滚规则由异常类型裁决checked exception 默认提交业务异常必须显式配置 rollbackFor
传播行为在 manager 中实现连接挂起 / 恢复成本高REQUIRES_NEW 能独立提交,放大连接池压力

16.末 本章要点

  • 统一抽象是核心设计。PlatformTransactionManager 抽象屏蔽 JDBC / JPA / JTA / R2DBC 差异,TransactionDefinition 定义事务需求,TransactionStatus 暴露运行时状态。
  • @Transactional 本质是 AOP 代理。TransactionInterceptor 在目标方法前后织入 begin / commit / rollback 逻辑。连接通过 TransactionSynchronizationManager 绑定到 ThreadLocal。
  • 传播行为决定事务边界。7 种传播行为覆盖加入 / 创建 / 挂起 / 拒绝所有组合。REQUIRED 是 80% 场景的正确选择,REQUIRES_NEW 用于必须独立提交的审计日志,NESTED 用于批处理中部分回滚。
  • 隔离级别权衡一致性与性能。从 READ_UNCOMMITTED 到 SERIALIZABLE,隔离强度递增,并发性能递减。
  • 回滚规则必须显式建模。默认只对 RuntimeException / Error 回滚,Checked Exception 不回滚。写事务通常要显式配置 rollbackFor。
  • 编程式事务用于精细控制。声明式事务方法级粒度不够时,使用 TransactionTemplate 或直接操作 PlatformTransactionManager。
  • 分布式事务优先考虑 Saga。XA / 2PC 提供强一致性但性能差且脆弱,Saga 模式通过补偿机制实现最终一致性。
  • 牢记七大陷阱:自调用绕过代理、private / final 方法无效、checked exception 不回滚、REQUIRES_NEW 连接池耗尽、@Async 不共享调用者事务、readOnly 非强制、timeout 不一定能打断慢 SQL。
