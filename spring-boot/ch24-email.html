订单创建后的确认邮件、支付授权后的通知、每月账单的 PDF 附件推送——邮件是几乎所有业务系统绕不开的基础能力。然而直接使用 JavaMail API 发送一封带 HTML 正文、嵌入式图片与 PDF 附件的多部分邮件,需要手动管理 Session、Transport、MimeMultipart、BodyPart 的层层嵌套,代码冗长且极易出错。

Spring 从 1.x 时代就提供了 MailSender 与 JavaMailSender 抽象层,将 JavaMail 的底层复杂性封装到极致;Spring Boot 更进一步,通过 MailSenderAutoConfiguration 读取 spring.mail.* 配置即可自动装配一个生产就绪的 JavaMailSenderImpl。本章按"JavaMail 与 Spring 抽象 → Boot 自动配置 → HTML 模板邮件 → MIME 多部分附件 → 异步与重试 → 源码与陷阱"的顺序展开。前置知识:第 20 章 异步处理与 TaskExecutor 线程池体系,第 2 章 自动配置原理。

24.1 JavaMail 与 Spring 抽象

在理解 Spring 的邮件抽象之前,必须先搞清楚它所封装的底层——JavaMail API。这套 Jakarta(原 javax)标准 API 从 JDK 1.2 时代沿用至今,是 Java 世界发送与接收邮件的事实标准。

24.1.1 JavaMail 核心三件套

JavaMail 的世界围绕三个核心类运转:Session 负责配置与认证、Transport 负责连接 SMTP 服务器并投递邮件、MimeMessage 承载邮件内容。理解它们之间的关系是掌握 Spring 邮件抽象的基础。

Properties props = new Properties();
props.put("mail.smtp.host", "smtp.company.com");
props.put("mail.smtp.port", "587");
props.put("mail.smtp.auth", "true");
props.put("mail.smtp.starttls.enable", "true");

Session session = Session.getInstance(props, new Authenticator() {
    @Override
    protected PasswordAuthentication getPasswordAuthentication() {
        return new PasswordAuthentication("order@company.com", "secret");
    }
});

MimeMessage message = new MimeMessage(session);
message.setFrom(new InternetAddress("noreply@company.com"));
message.setRecipient(Message.RecipientType.TO,
    new InternetAddress("customer@example.com"));
message.setSubject("订单确认", "UTF-8");

MimeMultipart multipart = new MimeMultipart("mixed");
MimeBodyPart htmlPart = new MimeBodyPart();
htmlPart.setContent("<h1>订单已确认</h1>", "text/html; charset=UTF-8");
multipart.addBodyPart(htmlPart);

MimeBodyPart attachPart = new MimeBodyPart();
attachPart.attachFile(new File("/reports/payment-receipt-O1001.pdf"));
multipart.addBodyPart(attachPart);

message.setContent(multipart);
Transport.send(message);

运行时可以打开 JavaMail debug 日志,确认它走到了 SMTP 协议的哪一步:

DEBUG SMTP: connected to host smtp.company.com port 587

DATA

这段代码虽然能工作,但存在几个明显问题:SMTP 配置散落在 Properties 中缺乏类型安全;Session 的创建方式(getInstance 与 getDefaultInstance)容易混淆;MimeMultipart 的手动嵌套极易出错;异常处理与资源清理全靠开发者自己管理。

24.1.2 Spring MailSender 抽象体系

Spring 从 1.x 开始就提供了 org.springframework.mail 包来简化邮件发送。这套抽象的核心设计理念是:将"发什么"与"怎么发"彻底分离。

public interface MailSender {
    void send(SimpleMailMessage simpleMessage) throws MailException;
    void send(SimpleMailMessage... simpleMessages) throws MailException;
}

public interface JavaMailSender extends MailSender {
    MimeMessage createMimeMessage();
    MimeMessage createMimeMessage(InputStream contentStream);
    void send(MimeMessage mimeMessage);
    void send(MimeMessage... mimeMessages);
    void send(MimeMessagePreparator mimeMessagePreparator);
    void send(MimeMessagePreparator... mimeMessagePreparators);
}

@FunctionalInterface
public interface MimeMessagePreparator {
    void prepare(MimeMessage mimeMessage) throws Exception;
}

MailSender 是最顶层的抽象——只支持纯文本邮件,接口极简。JavaMailSender 扩展了它,增加了 MimeMessage 的创建与发送能力,并引入了 MimeMessagePreparator 回调模式来简化 MimeMessage 的构建。JavaMailSenderImpl 是 Spring 提供的标准实现,内部持有 JavaMail 的 Session 并管理 Transport 连接。

24.1.3 SimpleMailMessage 纯文本极简方案

SimpleMailMessage 是 Spring 为最常见的纯文本邮件场景设计的值对象。它不依赖 JavaMail API,因此可以在测试中轻松创建与断言。

@Service
public class WelcomeEmailService {
    private final MailSender mailSender;

    public WelcomeEmailService(MailSender mailSender) {
        this.mailSender = mailSender;
    }

    public void sendWelcome(String to, String username) {
        SimpleMailMessage msg = new SimpleMailMessage();
        msg.setFrom("noreply@myapp.com");
        msg.setTo(to);
        msg.setSubject("注册成功通知");
        msg.setText(String.format(
            "尊敬的 %s\n\n感谢注册 MyApp\n账号已激活\n如有问题请联系 support@myapp.com",
            username));
        msg.setSentDate(new Date());
        mailSender.send(msg);
    }
}

纯文本邮件成功发送时,业务日志一般只需要记录收件人、主题与消息类型,避免把正文全文打进日志:

2026-05-08 10:15:21 INFO WelcomeEmailService : mail.sent type=simple to=customer-c1001@example.com subject=注册成功通知

24.1.4 MimeMessageHelper 富邮件瑞士军刀

MimeMessageHelper 是 Spring 邮件模块中最重要的工具类。它将 JavaMail 的 MimeMessage 加 MimeMultipart 加 MimeBodyPart 的复杂嵌套封装为直观的方法调用,是构建 HTML 邮件、内嵌图片、多附件邮件的首选。

@Service
public class OrderConfirmationService {
    private final JavaMailSender mailSender;

    public void sendConfirmation(Order order) throws MessagingException {
        MimeMessage message = mailSender.createMimeMessage();
        MimeMessageHelper helper = new MimeMessageHelper(
            message,
            MimeMessageHelper.MULTIPART_MODE_MIXED_RELATED,
            StandardCharsets.UTF_8.name()
        );

        helper.setFrom("orders@shop.com", "MyShop 订单中心");
        helper.setTo(order.getCustomerEmail());
        helper.setSubject("订单 " + order.getId() + " 确认");
        helper.setPriority(1);

        String htmlContent = buildOrderHtml(order);
        helper.setText(htmlContent, true);

        helper.addInline("logo",
            new ClassPathResource("static/images/logo.png"));

        helper.addAttachment("payment-receipt-O1001.pdf",
            new ByteArrayResource(generateInvoicePdf(order)),
            "application/pdf");

        mailSender.send(message);
    }
}

这段代码生成的 MIME 结构可以用 MailHog 或 GreenMail 查看,重点是正文、内嵌图片与附件被放在正确层级:

Content-Type: multipart/mixed
  multipart/related
    text/html charset UTF-8
    image/png name logo.png Content-ID logo
  application/pdf name payment-receipt-O1001.pdf

Subject: 订单 O1001 确认
X-Priority: 1

对比原生 JavaMail 代码,使用 MimeMessageHelper 后代码量减少了近 60%,而且不需要手动管理 MimeMultipart 的嵌套结构。Helper 内部会根据 multipartMode 自动创建正确的 MIME 层次。

24.1.5 MailException 异常体系

Spring 把所有 JavaMail 的 checked exception 统一翻译为 MailException 的 unchecked 子类,便于业务代码处理。

异常类型 触发场景 处理建议
MailAuthenticationException SMTP 用户名或密码错误 不要重试,检查配置
MailSendException SMTP 投递失败(网络、被拒等) 可重试,看 SMTP 返回码决定
MailParseException 消息构建失败(MIME 格式错误) 不要重试,代码 bug
MailPreparationException MimeMessagePreparator 内部异常 不要重试,代码 bug

24.2 Spring Boot 邮件自动配置

Spring Boot 对邮件的自动配置堪称"零代码集成"的典范——只需在 application.yml 中填入 SMTP 服务器信息,就能获得一个生产就绪的 JavaMailSender Bean。

24.2.1 spring.mail 属性全景

Spring Boot 通过 MailProperties 类将所有邮件相关配置绑定到 spring.mail.* 命名空间。理解每个属性的含义与默认值是正确配置的前提。

spring:
  mail:
    host: smtp.company.com
    port: 587
    username: sender@company.com
    password: ${MAIL_PASSWORD}
    protocol: smtp
    default-encoding: UTF-8
    test-connection: true
    properties:
      mail.smtp.auth: true
      mail.smtp.starttls.enable: true
      mail.smtp.starttls.required: true
      mail.smtp.connectiontimeout: 5000
      mail.smtp.timeout: 5000
      mail.smtp.writetimeout: 5000
      mail.smtp.ssl.trust: smtp.company.com

启用 spring.mail.test-connection=true 后,应用启动阶段会立即验证 SMTP 可用性;这能把错误提前暴露在启动日志里,而非等第一封业务邮件才失败:

2026-05-08 10:18:03 INFO MailSenderValidatorAutoConfiguration : Mail server connection validated

24.2.2 MailSenderAutoConfiguration 决策树

MailSenderAutoConfiguration 的条件判断逻辑清晰而严谨。它首先检查 classpath 上是否存在 JavaMail 相关类,然后根据是否配置了 spring.mail.host 或 JNDI 名称来决定创建哪种类型的 JavaMailSender。

图 24-2 · MailSenderAutoConfiguration 装配决策树

24.2.3 自动配置源码

@AutoConfiguration
@ConditionalOnClass({ MimeMessage.class, MimeType.class })
@ConditionalOnMissingBean(MailSender.class)
@EnableConfigurationProperties(MailProperties.class)
@Import({ MailSenderJndiConfiguration.class,
           MailSenderPropertiesConfiguration.class })
public class MailSenderAutoConfiguration {
}

@Configuration
@ConditionalOnProperty(prefix = "spring.mail", name = "host")
class MailSenderPropertiesConfiguration {

    @Bean
    JavaMailSenderImpl mailSender(MailProperties properties) {
        JavaMailSenderImpl sender = new JavaMailSenderImpl();
        sender.setHost(properties.getHost());
        sender.setPort(properties.getPort());
        sender.setProtocol(properties.getProtocol());
        sender.setDefaultEncoding(properties.getDefaultEncoding().name());

        if (StringUtils.hasText(properties.getUsername())) {
            sender.setUsername(properties.getUsername());
            sender.setPassword(properties.getPassword());
        }

        if (properties.getProperties().isEmpty() == false) {
            sender.setJavaMailProperties(asProperties(properties.getProperties()));
        }
        return sender;
    }
}

24.2.5 通知服务案例

将上述配置与业务逻辑结合,一个典型的通知服务只需注入 JavaMailSender 即可。Spring Boot 的自动配置让代码完全不需要关心 Session 的创建与 Transport 的管理。

@Service
@Slf4j
public class NotificationService {

    private final JavaMailSender mailSender;

    @Value("${app.mail.from:noreply@myapp.com}")
    private String fromAddress;

    public NotificationService(JavaMailSender mailSender) {
        this.mailSender = mailSender;
    }

    public void sendPasswordReset(String to, String resetToken) {
        try {
            MimeMessage message = mailSender.createMimeMessage();
            MimeMessageHelper helper = new MimeMessageHelper(message, false, "UTF-8");

            helper.setFrom(fromAddress, "MyApp 安全中心");
            helper.setTo(to);
            helper.setSubject("密码重置请求");
            helper.setText(
                "<p>系统收到密码重置请求</p>" +
                "<p>请点击链接重置密码 <a href='https://myapp.com/reset?token=" +
                resetToken + "'>重置密码</a></p>" +
                "<p>链接 24 小时内有效</p>",
                true);

            mailSender.send(message);
            log.info("password reset mail sent to={}", to);
        } catch (MailException | MessagingException e) {
            log.error("send password reset mail failed to={}", to, e);
            throw new EmailSendException("password reset mail send failed", e);
        }
    }
}

成功路径应该只有一条业务日志,能把邮件投递动作与客户请求关联起来:

2026-05-08 10:22:44 INFO NotificationService : password reset mail sent to=bob@example.com

认证失败、地址非法、服务器拒绝都会被 Spring 包装成 MailException 的不同子类;异常块要保留原始 cause,方便定位 SMTP 返回码:

2026-05-08 10:23:12 ERROR NotificationService : send password reset mail failed to=bob@example.com

org.springframework.mail.MailAuthenticationException: Authentication failed
Caused by: jakarta.mail.AuthenticationFailedException
  535 5.7.8 Username and Password not accepted

24.3 HTML 邮件与模板引擎

在真实的业务系统中,纯文本邮件只能覆盖最简单的通知场景。用户期望看到的是带有品牌 Logo、格式化表格、行动按钮的富文本邮件。但构建 HTML 邮件比构建 Web 页面要复杂得多——邮件客户端对 HTML 与 CSS 的支持参差不齐。

24.3.1 Thymeleaf 邮件模板引擎

Spring Boot 项目中最自然的选择是使用 Thymeleaf 作为邮件模板引擎。关键在于:为邮件创建一个独立的 TemplateEngine,而非复用 Web MVC 的引擎——邮件模板的解析上下文与缓存策略与 Web 页面不同。

@Configuration
public class EmailTemplateConfig {

    @Bean("emailTemplateEngine")
    public TemplateEngine emailTemplateEngine() {
        SpringTemplateEngine engine = new SpringTemplateEngine();

        ClassLoaderTemplateResolver resolver = new ClassLoaderTemplateResolver();
        resolver.setPrefix("mail-templates/");
        resolver.setSuffix(".html");
        resolver.setTemplateMode(TemplateMode.HTML);
        resolver.setCharacterEncoding("UTF-8");
        resolver.setCacheable(true);

        engine.setTemplateResolver(resolver);
        return engine;
    }
}

模板引擎单独注册后,可以在启动日志或 Bean 列表里看到它与 Web MVC 的模板引擎分离:

2026-05-08 10:27:01 INFO EmailTemplateConfig : registered bean=emailTemplateEngine prefix=mail-templates suffix=.html cacheable=true

24.3.3 FreeMarker 模板替代方案

如项目已经在使用 FreeMarker,也可以用它来渲染邮件模板。FreeMarker 的模板语法比 Thymeleaf 更传统(类似 JSP 的标签式),在纯字符串模板场景下性能略优。

@Configuration
public class FreeMarkerEmailConfig {

    @Bean("emailFreeMarkerConfig")
    public freemarker.template.Configuration freeMarkerConfiguration() throws IOException {
        freemarker.template.Configuration config = new freemarker.template.Configuration(
            freemarker.template.Configuration.VERSION_2_3_32);
        config.setClassLoaderForTemplateLoading(getClass().getClassLoader(), "mail-templates");
        config.setDefaultEncoding("UTF-8");
        config.setTemplateExceptionHandler(TemplateExceptionHandler.RETHROW_HANDLER);
        return config;
    }
}

@Service
public class FreeMarkerEmailService {

    @Qualifier("emailFreeMarkerConfig")
    private final freemarker.template.Configuration fmConfig;
    private final JavaMailSender mailSender;

    public void sendOrderConfirmation(Order order) throws Exception {
        Template template = fmConfig.getTemplate("order-confirmation.ftl");

        Map<String, Object> model = Map.of(
            "customerName", order.getCustomerName(),
            "orderId", order.getId(),
            "items", order.getItems(),
            "totalAmount", order.getTotalAmount());

        String html = FreeMarkerTemplateUtils.processTemplateIntoString(template, model);

        MimeMessage message = mailSender.createMimeMessage();
        MimeMessageHelper helper = new MimeMessageHelper(message, true, "UTF-8");
        helper.setTo(order.getCustomerEmail());
        helper.setSubject("订单 " + order.getId() + " 确认");
        helper.setText(html, true);
        mailSender.send(message);
    }
}

FreeMarker 模板缺少变量或语法错误时会在渲染阶段失败,邮件还没有进入 SMTP 发送阶段:

24.3.4 内联 CSS 与邮件客户端兼容性

邮件 HTML 的最大痛点是 CSS 支持。多数邮件客户端会剥离 <style> 标签与外部样式表,只保留行内 style 属性。因此在发送前需要将 CSS 规则"内联化"。

方案 时机 性能 适用场景
jsoup 加 CSSInliner Java 库 运行时 中 模板动态变化
Premailer Ruby 或 juice JS 构建时 高 模板固定,Maven 构建阶段处理
<style> 加 行内混合 编写时 低 快速 demo,生产不推荐

24.3.5 嵌入内联图片

邮件中的图片有两种方式:外链 URL 与内嵌 CID(Content-ID)。外链受制于邮件客户端的图片拦截策略,而 CID 内嵌图片会直接包含在邮件 MIME 结构中,显示更可靠。

public void sendWithInlineImages(String to, String html) throws MessagingException {
    MimeMessage message = mailSender.createMimeMessage();
    MimeMessageHelper helper = new MimeMessageHelper(message,
        MimeMessageHelper.MULTIPART_MODE_MIXED_RELATED, "UTF-8");

    helper.setTo(to);
    helper.setSubject("带内嵌图片的邮件");
    helper.setText(html, true);

    helper.addInline("logo",
        new ClassPathResource("images/logo.png"), "image/png");
    helper.addInline("banner",
        new ClassPathResource("images/banner.jpg"), "image/jpeg");
    helper.addInline("product-1",
        new UrlResource("https://cdn.shop.com/product/1.jpg"), "image/jpeg");

    mailSender.send(message);
}

内联图片的关键验证点是 Content-ID 与 HTML 中的 cid: 完全一致;大小写或尖括号处理错误都会导致客户端显示红叉:

HTML body
  img src cid logo
  img src cid banner

MIME parts
  Content-Type image/png name logo.png
  Content-ID logo
  Content-Disposition inline

  Content-Type image/jpeg name banner.jpg
  Content-ID banner
  Content-Disposition inline

24.4 附件与多部分邮件

当邮件需要同时携带 HTML 正文、内嵌图片与文件附件时,底层的 MIME 结构变得复杂——它是一个多层嵌套的"信封套信封"结构。理解 MIME multipart 的三种模式是构建复杂邮件的关键。

24.4.1 MIME 多部分结构解析

MIME(Multipurpose Internet Mail Extensions)定义了三种常用的 multipart 子类型:

子类型 语义 典型用途
multipart/mixed 正文与附件混合在一个层级 正文 加 附件
multipart/related HTML 正文与内嵌资源关联,共同组成整体 HTML 加 CID 内嵌图片
multipart/alternative 同一内容的多种格式,客户端选择最优 纯文本 加 HTML(降级显示)
图 24-5 · MIME 多部分消息嵌套结构(信封套信封套信封)

嵌套规则:mixed 包含 related,related 包含 alternative。附件挂最外,内嵌图片挂中间,多格式文本挂最内。

24.4.2 MimeMessageHelper 的四种 multipartMode

MimeMessageHelper 提供四种模式来自动管理上述嵌套结构,只需在构造时指定模式,Helper 会自动创建正确的 MimeMultipart 层次。

public static final int MULTIPART_MODE_NO = 0;
public static final int MULTIPART_MODE_MIXED = 1;
public static final int MULTIPART_MODE_RELATED = 2;
public static final int MULTIPART_MODE_MIXED_RELATED = 3;
模式 数值 创建的 MIME 结构 适用场景
NO 0 无 multipart 纯文本或纯 HTML
MIXED 1 mixed 正文 加 附件
RELATED 2 related HTML 加 内嵌图片(无附件)
MIXED_RELATED 3 mixed 包含 related 包含 alternative 全功能,支持所有场景

24.4.3 文件附件的多种来源

生产环境中附件来源多样——本地文件、classpath 资源、内存字节流、远程 URL。MimeMessageHelper.addAttachment() 统一接受 InputStreamSource,而 Spring 的 Resource 体系天然实现了这个接口。

@Service
public class ReportEmailService {

    private final JavaMailSender mailSender;
    private final ReportGenerator reportGenerator;

    public void sendMonthlyReport(String to, YearMonth month) throws MessagingException {
        MimeMessage message = mailSender.createMimeMessage();

            MimeMessageHelper.MULTIPART_MODE_MIXED_RELATED, "UTF-8");

        helper.setTo(to);

            new FileSystemResource(csvFile));

    }
}

24.5 异步发送与重试

SMTP 邮件发送是一个典型的 I O 密集型操作——从 DNS 解析、TCP 三次握手、TLS 协商到数据传输,一封邮件的发送通常需要 200ms 到 2s。如在 Web 请求的线程中同步发送,会严重拖慢接口响应。

24.5.1 @Async 非阻塞发送

最简单的异步方案是使用 Spring 的 @Async 注解。它将邮件发送操作从调用线程转移到专用的 TaskExecutor 线程池,让业务逻辑能立即返回。

@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean("emailExecutor")
    public TaskExecutor emailTaskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(2);
        executor.setMaxPoolSize(5);
        executor.setQueueCapacity(100);
        executor.setThreadNamePrefix("email-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.initialize();
        return executor;
    }
}

@Service
@Slf4j
public class AsyncEmailService {

    private final JavaMailSender mailSender;
    private final TemplateEngine emailTemplateEngine;

    @Async("emailExecutor")
    public CompletableFuture<Boolean> sendAsync(String to, String subject,
            String templateName, Map<String, Object> variables) {
        try {
            Context ctx = new Context();
            ctx.setVariables(variables);
            String html = emailTemplateEngine.process(templateName, ctx);

            MimeMessage message = mailSender.createMimeMessage();
            MimeMessageHelper helper = new MimeMessageHelper(message, true, "UTF-8");
            helper.setTo(to);
            helper.setSubject(subject);
            helper.setText(html, true);

            mailSender.send(message);
            log.info("async mail sent to={}", to);
            return CompletableFuture.completedFuture(true);
        } catch (Exception e) {
            log.error("async mail failed to={}", to, e);
            return CompletableFuture.completedFuture(false);
        }
    }
}

异步发送的业务接口会先返回,邮件线程稍后写入自己的日志;这正是 @Async 的可观测特征:

2026-05-08 10:45:11 INFO UserRegistrationService : registered id=42 email=customer-c1001@example.com cost=38ms

2026-05-08 10:45:11 INFO AsyncEmailService : async mail sent to=customer-c1001@example.com

24.5.2 Spring Retry 自动重试

SMTP 连接偶发性失败(网络抖动、服务器繁忙返回 421)是常态。使用 Spring Retry 可以自动在失败后按退避策略重试。

@Configuration
@EnableRetry
public class RetryConfig { }

@Service
@Slf4j
public class ReliableEmailService {

    private final JavaMailSender mailSender;

    @Retryable(
        retryFor = { MailSendException.class },
        noRetryFor = { MailAuthenticationException.class, MailParseException.class },
        maxAttempts = 3,
        backoff = @Backoff(delay = 1000, multiplier = 2.0, maxDelay = 10000)
    )
    public void sendWithRetry(MimeMessage message) {
        log.info("attempting mail send");
        mailSender.send(message);
    }

    @Recover
    public void handleSendFailure(MailException ex, MimeMessage message) {
        log.error("mail send failed exhausted retries reason={}", ex.getMessage());
        saveToFailedQueue(message, ex);
    }
}

瞬时 SMTP 失败时,Spring Retry 会按退避策略重新调用同一个方法;日志中应能看到多次尝试与最终恢复逻辑:

2026-05-08 10:48:21 INFO ReliableEmailService : attempting mail send
2026-05-08 10:48:21 WARN RetryTemplate : Retry count=2

24.5.3 数据库队列加调度器模式

对于高可靠性要求的场景(如交易通知、法规邮件),仅靠 @Async 与 Retry 还不够——应用重启会丢失内存中等待发送的邮件。更健壮的方案是使用数据库作为持久化队列。

@Entity
@Table(name = "email_queue")
public class EmailTask {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String toAddress;

    @Column(nullable = false)
    private String subject;

    @Lob
    private String htmlBody;

    @Lob
    private byte[] attachmentData;
    private String attachmentName;
    private String attachmentType;

    @Enumerated(EnumType.STRING)
    private EmailStatus status = EmailStatus.PENDING;

    private int retryCount = 0;
    private int maxRetries = 3;
    private String lastError;

    private LocalDateTime createdAt = LocalDateTime.now();
    private LocalDateTime nextRetryAt = LocalDateTime.now();
    private LocalDateTime sentAt;
}

public enum EmailStatus {
    PENDING, SENDING, SENT, FAILED, DEAD_LETTER
}

@Service
@Slf4j
public class EmailQueueProcessor {

    private final EmailTaskRepository repository;
    private final JavaMailSender mailSender;

    @Scheduled(fixedDelay = 5000)
    @Transactional
    public void processQueue() {
        List<EmailTask> tasks = repository
            .findTop20ByStatusAndNextRetryAtBeforeOrderByCreatedAt(
                EmailStatus.PENDING, LocalDateTime.now());

        for (EmailTask task : tasks) {
            try {
                task.setStatus(EmailStatus.SENDING);
                repository.save(task);

                sendEmail(task);

                task.setStatus(EmailStatus.SENT);
                task.setSentAt(LocalDateTime.now());
            } catch (Exception e) {
                task.setRetryCount(task.getRetryCount() + 1);
                task.setLastError(e.getMessage());

                if (task.getRetryCount() >= task.getMaxRetries()) {
                    task.setStatus(EmailStatus.DEAD_LETTER);
                    log.error("email entered dead letter id={}", task.getId());
                } else {
                    task.setStatus(EmailStatus.PENDING);
                    task.setNextRetryAt(LocalDateTime.now()
                        .plusSeconds((long) Math.pow(2, task.getRetryCount()) * 30));
                }
            }
            repository.save(task);
        }
    }

    private void sendEmail(EmailTask task) throws MessagingException {
        MimeMessage message = mailSender.createMimeMessage();
        MimeMessageHelper helper = new MimeMessageHelper(message,
            task.getAttachmentData() != null, "UTF-8");
        helper.setTo(task.getToAddress());
        helper.setSubject(task.getSubject());
        helper.setText(task.getHtmlBody(), true);

        if (task.getAttachmentData() != null) {
            helper.addAttachment(task.getAttachmentName(),
                new ByteArrayResource(task.getAttachmentData()),
                task.getAttachmentType());
        }

        mailSender.send(message);
    }
}

数据库队列的输出不是控制台返回值,而是任务状态从 PENDING 到 SENT 或 DEAD_LETTER 的可追踪变化:

-----+------------------+--------+-------------+---------------------+--------------------
9001 | finance@corp.com | SENT   | 0           | 2026-05-08 10:50:00 | 2026-05-08 10:50:06

失败并进入死信时,lastError 要保留 SMTP 返回码,方便后续人工重放或修正配置:

24.5.5 速率限制与发送节流

SMTP 服务商通常有速率限制——Gmail 每天 500 封、Amazon SES 每秒 14 封。超限会导致 421 Too many connections 或 454 Too many messages 错误。需要在发送层加入速率控制。

@Service
public class RateLimitedEmailSender {

    private final JavaMailSender mailSender;
    private final RateLimiter rateLimiter;

    public RateLimitedEmailSender(JavaMailSender mailSender) {
        this.mailSender = mailSender;
        this.rateLimiter = RateLimiter.create(14.0);
    }

    public void send(MimeMessage message) {
        rateLimiter.acquire();
        mailSender.send(message);
    }

    public void sendBatch(List<MimeMessage> messages) {
        for (MimeMessage msg : messages) {
            rateLimiter.acquire();
            try {
                mailSender.send(msg);
            } catch (MailException e) {
                handleFailure(msg, e);
            }
        }
    }
}

速率限制的可观测结果是发送吞吐稳定在服务商限额以内,而非出现一串 421 或 454 拒绝:

24.6.3 设计取舍与被拒绝方案

Spring Mail 的抽象很薄,这是刻意设计:它不试图重新实现 SMTP,也不隐藏 Jakarta Mail 的 MIME 模型,而是把 JavaMail 最容易写错的配置、异常与 multipart 结构包起来。

设计选择 得到什么 代价 适用边界
JavaMailSenderImpl 直接封装 Jakarta Mail 透传所有 SMTP 与 MIME 细节 仍需理解底层属性名 标准 SMTP、企业邮件服务
MimeMessageHelper 管 multipart 减少嵌套错误,统一编码 极复杂邮件仍需检查原始 MIME HTML、CID、附件
Boot 自动配置仅在 host 存在时生效 不误创建无效 sender 本地测试需显式提供 Fake SMTP 生产配置清晰
邮件发送默认同步 调用语义简单,异常直接返回 Web 请求线程可能被 SMTP 阻塞 低频管理邮件
异步与队列由应用层选择 可按可靠性定制 需要额外状态机与幂等 交易通知、批量报表

被拒绝的方案:不应在业务代码里手写 Transport.send() 旁路 Spring 配置;不应把大附件硬塞进 MIME;不应把"SMTP 返回 250"当成"用户收到了邮件";不应把模板渲染、附件生成与 SMTP 投递放在同一个数据库事务里长时间执行。

24.6.4 扩展边界与验证护栏

邮件系统的扩展边界建议按阶段划分:模板渲染服务只输出 HTML,MIME 构建服务只负责 MimeMessage,投递服务只负责 SMTP 与 队列状态,投递结果再异步回写。

24.7 陷阱与限制

邮件发送看似简单——mailSender.send(message) 一行代码的事。但生产环境中,从 SMTP 协议层到邮件客户端渲染层,处处暗藏陷阱。以下是六大高频问题。

24.7.1 陷阱 1 SMTP 连接超时与资源泄漏

JavaMail 底层使用 java.net.Socket,其默认超时为 0(无限等待)。当 SMTP 服务器因负载过高或网络分区导致无响应时,发送线程会永久阻塞,最终耗尽线程池。

修复:必须显式设置三种超时。

spring:
  mail:
    properties:
      mail.smtp.connectiontimeout: 5000
      mail.smtp.timeout: 5000
      mail.smtp.writetimeout: 5000
属性 控制阶段 推荐值
mail.smtp.connectiontimeout TCP 三次握手 5 秒
mail.smtp.timeout Socket 读(等待 SMTP 响应) 10 秒
mail.smtp.writetimeout Socket 写(发送邮件数据) 10 秒

缺少超时配置时,线程会长时间卡在 SMTPTransport.protocolConnect() 或 socket 读写上;配置超时后,异常会在可控时间内返回:

2026-05-08 10:58:12 ERROR NotificationService : send mail timeout to=ops@example.com

org.springframework.mail.MailSendException Mail server connection failed
Caused by com.sun.mail.util.MailConnectException
  Couldn't connect to host port smtp.company.com 587 timeout 5000
Caused by java.net.SocketTimeoutException Connect timed out

24.7.2 陷阱 2 HTML 邮件渲染兼容性

邮件 HTML 与网页 HTML 是两个完全不同的语境。邮件客户端的渲染引擎差异远大于浏览器:

客户端 渲染引擎 常见不兼容 稳妥做法
Outlook Windows Word 渲染引擎 max-width、float、复杂背景图 TABLE 布局、固定宽度、必要时 VML
Gmail 自定义清洗器 外部 CSS、部分 <style> 媒体查询 发送前 CSS 内联化
Apple Mail WebKit 仍受图片拦截影响 指定图片尺寸与替代文本
Yahoo 与 企业网关 自定义改写器 class 改名、脚本与表单被剥离 避免依赖 class 选择器与交互脚本

邮件 HTML 的生存法则:用 TABLE 布局,不用 div 加 CSS;所有样式内联(行内 style 属性);不用 CSS3 动画、transition、transform;不用 <video>、<audio>、<iframe>;图片必须指定 width 与 height;背景图片用 VML 兜底。

24.7.3 陷阱 3 附件大小超限

SMTP 服务商 附件大小限制 备注
Gmail 25 MB
Outlook 20 MB
Amazon SES 10 MB raw message 上限 40 MB
企业 Exchange 10 至 50 MB 管理员配置

需要注意:附件经过 Base64 编码后体积增大约 33%,一个 7.5 MB 的文件编码后约 10 MB。大文件应上传到云存储,邮件中放下载链接:

public void sendWithLargeAttachment(String to, File file) throws MessagingException {
    if (file.length() > 10 * 1024 * 1024) {
        String downloadUrl = cloudStorage.upload(file);
        String html = String.format(
            "<p>附件过大,请通过链接下载</p>" +
            "<p><a href='%s'>%s %s</a></p>" +
            "<p>链接 7 天内有效</p>",
            downloadUrl, file.getName(), humanReadableSize(file.length()));
        sendHtmlEmail(to, "文件分享", html);
    } else {
        sendWithAttachment(to, "文件分享", file);
    }
}

如仍然把大文件塞进 MIME,很多 SMTP 服务器会在 DATA 阶段直接拒绝整封邮件:

SMTP reply 552 5.3.4 Message size exceeds fixed maximum message size

24.7.4 陷阱 4 SPF DKIM DMARC 域名验证

使用未经验证的发件人地址,邮件会被标记为垃圾或直接被拒收。完整的可送达性需要三个 DNS 配置:

协议 全称 作用 DNS 记录类型
SPF Sender Policy Framework 声明哪些 IP 可代表域名发信 TXT 在主域名
DKIM DomainKeys Identified Mail 数字签名防篡改,接收方用 DNS 公钥验证 TXT 在 selector._domainkey 子域
DMARC Domain-based Message Authentication 基于 SPF 加 DKIM 的策略层 TXT 在 _dmarc 子域

完整可送达性清单:配置 SPF 记录;配置 DKIM 签名;配置 DMARC 策略;设置 List-Unsubscribe 头(Gmail 要求);避免垃圾词汇;维护发件人信誉(控制退信率小于 2 percent)。

域名认证失败时,邮件可能并非"发送失败",而是在接收方被拒收或进入垃圾箱;这类问题要看退信与 DMARC 报告:

Status 5.7.26

24.7.5 陷阱 5 默认 Session 全局共享

错误认知:Session.getDefaultInstance() 适合多租户动态配置。真相:getDefaultInstance() 返回 JVM 级默认 Session,第一次调用的 Properties 会被锁定,后续调用忽略新 Properties。

Properties tenant1Props = new Properties();
tenant1Props.put("mail.smtp.host", "smtp.tenant1.com");
Session s1 = Session.getDefaultInstance(tenant1Props);

Properties tenant2Props = new Properties();
tenant2Props.put("mail.smtp.host", "smtp.tenant2.com");
Session s2 = Session.getDefaultInstance(tenant2Props);

Session s1Fixed = Session.getInstance(tenant1Props, auth1);
Session s2Fixed = Session.getInstance(tenant2Props, auth2);

JavaMailSenderImpl 内部使用 Session.getInstance(),所以默认不会串租户。但如自定义 Session Bean,务必使用 getInstance() 而非 getDefaultInstance()。

24.7.6 陷阱 6 非 ASCII 字符编码

JavaMail 默认使用平台编码,不同 OS 默认编码不同:Windows 是 GBK 或 GB2312,Linux 与 macOS 是 UTF-8。直接设置中文主题会导致乱码。

修复:始终显式指定 UTF-8 编码。

message.setSubject("订单确认", "UTF-8");

MimeMessageHelper helper = new MimeMessageHelper(message, true, "UTF-8");
helper.setSubject("订单确认");

String encodedName = MimeUtility.encodeText("月度报告.pdf", "UTF-8", "B");
helper.addAttachment(encodedName, resource);

Spring Boot 全局配置:spring.mail.default-encoding=UTF-8。

24.8 本章小结

JavaMail 抽象:Session 加 Transport 加 MimeMessage 三件套;Spring 通过 MailSender 加 JavaMailSender 接口将其简化,JavaMailSenderImpl 是标准实现。

工具类:SimpleMailMessage 适合纯文本场景;MimeMessageHelper 是富邮件构建的瑞士军刀,封装了 multipart 嵌套、附件、内嵌图片与编码。

Boot 自动配置:spring.mail.host 触发 MailSenderAutoConfiguration 创建 JavaMailSenderImpl;test-connection=true 在启动期验证 SMTP 可用性。

模板引擎:为邮件配置独立的 TemplateEngine(Thymeleaf 或 FreeMarker);CSS 必须内联化;内嵌图片用 CID 协议;MULTIPART_MODE_MIXED_RELATED 支持完整结构。

MIME 多部分:mixed 包含 related 包含 alternative 的嵌套规则;附件来源覆盖 ClassPath、FileSystem、Url、ByteArray 与自定义 InputStreamSource。

异步与可靠性:@Async 加专用线程池避免 Web 线程阻塞;Spring Retry 处理 SMTP 瞬时失败;数据库队列加 @Scheduled 调度器实现持久化重试与死信;RateLimiter 控制发送速率以适应服务商限制。

六大陷阱:SMTP 超时未配置导致线程泄漏;HTML 在 Outlook 中布局错乱;附件超过 SMTP 服务商限制;SPF 与 DKIM 与 DMARC 未配置导致进垃圾箱;getDefaultInstance 多租户串配置;非 UTF-8 编码导致中文乱码。
