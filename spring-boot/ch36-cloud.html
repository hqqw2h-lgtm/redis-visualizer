第 36 章 · Spring Cloud 集成概览

本书前 35 章聚焦单个 Spring Boot 应用的内部机制。生产环境的真实形态是「N 个 Spring Boot 服务 + 注册中心 + 配置中心 + 网关 + 调用链追踪」组成的分布式系统。Spring Cloud 是把这些 Spring Boot 服务组装为微服务集群的胶水工程。本章按 Spring Cloud 2025.0(Northfields) / 2024.0(Moorgate)Release Train 与 Spring Boot 3.4 / 3.5 LTS 为基线展开。

本章不是 Spring Cloud 的完整教程,而是「定位 + 集成钩子 + 主流子项目 + 陷阱」四位一体的概览。九个主线节依次回答:Spring Cloud 八大组件如何利用 Spring Boot 的 ConfigData / AutoConfiguration / Actuator 三个扩展点挂入启动流程;Eureka / Nacos / Consul 三大注册中心的 CAP 取舍;Spring Cloud Config 与 Nacos Config 的 ConfigData 模型;OpenFeign 与 LoadBalancer 的代理与负载均衡链路;Resilience4j 的熔断 / 重试 / 限流 / 隔离 / 超时五大模块;Spring Cloud Gateway 的 Predicate / GatewayFilter / GlobalFilter 三层模型;Micrometer Tracing + OpenTelemetry 的全链路追踪;六类常见陷阱与对应排查路径;关键源码入口与扩展边界。

36.1 微服务演进与 Spring Cloud 的定位

表 36-1-1:从单体到微服务的三阶段

36.1.1 Spring Cloud Release Train

Spring Cloud 不是单一 jar,而是「伞项目」(Umbrella Project)。每个 Release Train 锁定一组子项目版本,与特定 Spring Boot 主线版本配合。版本命名按伦敦地铁站字母顺序:Aylesford → Brixton → Camden → ... → Moorgate(2024) → Northfields(2025)。

36.2.1 ConfigData 与 Bootstrap 路径

Spring Cloud 2020 之前使用 bootstrap.yml + BootstrapApplicationListener 在主上下文之前创建一个 Bootstrap 上下文加载远程配置。该机制在 Spring Cloud 2020 起被 ConfigData API 取代,新项目使用 spring.config.import=configserver: / spring.config.import=nacos: 把远程配置融入标准 Spring Boot 配置加载流程。

spring:
  application:
    name: order-service
  config:
    import:
      - optional:configserver:http://config-server:8888
      - optional:nacos:order-service.yaml

optional: 前缀表示远程不可用时不阻塞启动。生产环境通常去掉 optional:,让远程不可用直接 fail-fast。

36.2.2 AutoServiceRegistration 触发时机

AbstractAutoServiceRegistration 实现 ApplicationListener<WebServerInitializedEvent>,在 Web 服务器(Tomcat / Netty / Jetty)绑定端口后才触发 register(Registration)。该顺序确保被注册的 ip:port 已可接收流量,避免「注册成功但服务未就绪」的窗口。

启动顺序:
1. Bean 创建完成
2. ApplicationStartedEvent 触发
3. ApplicationRunner / CommandLineRunner 执行
4. WebServer 绑定 ip:port(WebServerInitializedEvent)
5. AbstractAutoServiceRegistration 监听到事件 → register
6. ApplicationReadyEvent 触发 → 健康检查任务启动

36.3 服务发现:Eureka / Nacos / Consul

服务发现解决「服务 A 如何找到服务 B 的当前 IP:Port 列表」的核心问题。该问题不能用静态配置解决,因为 B 可能有 N 个实例,实例随时上线下线。三大主流实现在一致性模型、健康检查机制、配置中心耦合度上存在显著差异。

36.3.1 Eureka 客户端配置

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-netflix-eureka-client</artifactId>
</dependency>

spring:
  application:
    name: order-service
eureka:
  client:
    serviceUrl:
      defaultZone: http://eureka-1:8761/eureka/,http://eureka-2:8761/eureka/
    registry-fetch-interval-seconds: 30
  instance:
    prefer-ip-address: true
    lease-renewal-interval-in-seconds: 30
    lease-expiration-duration-in-seconds: 90

无需在主类增加 @EnableEurekaClient。Spring Cloud 2020 起 starter 引入即自动配置。prefer-ip-address: true 让客户端注册时使用 IP 而非 hostname,避免 DNS 解析依赖。

36.3.2 Nacos 客户端配置

<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
    <version>2025.0.0.0</version>
</dependency>

spring:
  application:
    name: order-service
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848
        namespace: dev
        group: DEFAULT_GROUP
        cluster-name: HZ
        metadata:

namespace 用于环境隔离(dev / test / staging / prod),group 用于业务域分组,cluster-name 用于跨可用区路由。metadata 字段透传到注册中心,可在客户端按 metadata 过滤实例。

36.3.3 Consul 客户端配置

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-consul-discovery</artifactId>
</dependency>

spring:
  application:
    name: order-service
  cloud:
    consul:
      host: consul.example.com
      port: 8500
      discovery:
        prefer-ip-address: true
        health-check-path: /actuator/health
        health-check-interval: 15s
        health-check-timeout: 10s
        tags:
          - version=v1
          - profile=production

Consul 健康检查由 Consul 服务端主动发起 HTTP 请求,因此 health-check-path 必须在客户端可访问。Spring Boot Actuator 暴露 /actuator/health 后即可被 Consul 直接复用。

36.4 配置中心:Spring Cloud Config 与 Nacos Config

本地 application.yml 在多服务 / 多环境下管理成本指数增长。配置中心解决「集中管理 + 版本化 + 动态刷新 + 加密存储」四个核心问题。Spring Cloud 2025.0 默认推荐通过 spring.config.import 接入,不再需要 bootstrap.yml。

36.4.1 Spring Cloud Config Server

Spring Cloud Config 由 Server 与 Client 组成。Server 从 Git / SVN / Vault / 本地文件读取配置,通过 REST API 暴露给 Client。Client 通过 spring.config.import=configserver: 在启动时拉取远程配置。

spring:
  application:
    name: config-server
  cloud:
    config:
      server:
        git:
          uri: https://github.com/example/config-repo
          search-paths: '{application}'
          default-label: main
          username: ${GIT_USERNAME}
          password: ${GIT_TOKEN}
        encrypt:
          enabled: true
encrypt:
  key-store:
    location: classpath:keystore.jks
    password: ${KEYSTORE_PASSWORD}
    alias: configkey

spring:
  application:
    name: order-service
  config:
    import: configserver:http://config-server:8888
  cloud:
    config:
      label: main
      profile: production
      fail-fast: true

Client 启动时按约定 <application>-<profile>.yml 拉取(例如 order-service-production.yml)。fail-fast: true 确保配置不可用时启动失败,避免使用本地默认值掩盖问题。

36.4.2 Nacos Config

spring:
  application:
    name: order-service
  config:
    import: optional:nacos:order-service.yaml?refresh=true
  cloud:
    nacos:
      config:
        server-addr: 127.0.0.1:8848
        file-extension: yaml
        namespace: production
        group: DEFAULT_GROUP

?refresh=true 启用动态刷新。Nacos Config 通过长轮询监听服务端变更,平均延迟 1 秒内推送到客户端。

36.4.3 @RefreshScope 动态刷新

@RefreshScope 通过 AOP 把 Bean 包装为代理。配置变更时 RefreshEvent 触发清除该 scope 的 Bean 缓存,下次访问时重新创建 Bean,新值通过 @Value 注入。

@RestController
@RefreshScope
public class FeatureController {

    @Value("${feature.dark-launch:false}")
    private boolean darkLaunch;

    @Value("${feature.flag.timeout-ms:1000}")
    private long timeoutMs;

    @GetMapping("/feature/state")
    public Map<String, Object> state() {
        return Map.of(
            "darkLaunch", darkLaunch,
            "timeoutMs", timeoutMs
        );
    }
}

修改 Nacos 控制台中 feature.dark-launch=true 后,下次请求 /feature/state 返回新值,无需重启。配合 @RefreshScope 的常见误区:把数据库连接池 / Hibernate SessionFactory 这类重量级 Bean 加 @RefreshScope 会导致每次配置变更都重建连接池,代价过高。

图 36-4-1 · Spring Cloud Config 启动时序(ConfigData 加载链 + RefreshScope 动态刷新链)

配置中心的边界是「配置」,不是「业务状态」。把库存数 / 订单状态 / 黑名单这类高频业务状态写进配置中心会绕过缓存一致性、审计与事务模型。这类数据应存放在数据库或缓存,通过业务 API 访问。

36.4.4 加密与 Vault 集成

Spring Cloud Config Server 内置 {cipher}... 占位符,客户端拉取后由 Server 解密。HashiCorp Vault 提供更完整的密钥管理能力(动态秘钥、租约、审计),通过 spring-cloud-starter-vault-config 集成:

spring:
  config:
    import: vault://
  cloud:
    vault:
      uri: https://vault.example.com:8200
      authentication: KUBERNETES
      kubernetes:
        role: order-service
      kv:
        enabled: true
        backend: secret
        application-name: order-service

Vault 与 Kubernetes 集成后,Pod 通过 ServiceAccount Token 认证,获取临时数据库密钥(默认租约 1 小时,可续期)。客户端通过 @Value("${db.password}") 透明使用,无需知道密钥来自 Vault 还是本地配置。

36.5 服务间调用:Spring Cloud OpenFeign

OpenFeign 是 Netflix 开源的声明式 HTTP 客户端,Spring Cloud 在其基础上深度集成 Spring MVC 注解(@RequestMapping @GetMapping @PostMapping)、Spring Cloud LoadBalancer、Resilience4j。开发者只需声明接口,运行时由 FeignClientFactoryBean 创建 JDK 动态代理,自动完成「方法签名 → HTTP 请求」「响应 → POJO 反序列化」转换。

36.5.1 @FeignClient 注解与代理模型

<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
</dependency>

@SpringBootApplication
@EnableFeignClients
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}

@FeignClient(
    name = "order-service",
    contextId = "orderClient",
    configuration = OrderClientConfig.class
)
public interface OrderClient {

    @GetMapping("/api/orders/{id}")
    OrderDto findById(@PathVariable("id") Long id);

    @GetMapping("/api/orders")
    List<OrderDto> findAll(@RequestParam("page") int page,
                            @RequestParam("size") int size);

    @PostMapping("/api/orders")
    OrderDto create(@RequestBody CreateOrderRequest request);
}

name = "order-service" 是注册中心中的 serviceId。contextId 区分同一 serviceId 下的多个 Feign 客户端(防止 Bean 名称冲突)。configuration 指定该客户端独立的 Encoder / Decoder / Contract / Logger 配置(不与全局共享)。

36.5.2 LoadBalancer 选实例

Spring Cloud LoadBalancer(scl)替代了 Netflix Ribbon。默认实现 RoundRobinLoadBalancer 提供轮询,可替换为 RandomLoadBalancer WeightedServiceInstanceListSupplier 等。

@LoadBalancerClient(name = "order-service", configuration = OrderLoadBalancerConfig.class)
public class OrderLoadBalancerConfig {

    @Bean
    public ServiceInstanceListSupplier serviceInstanceListSupplier(
            ConfigurableApplicationContext context) {
        return ServiceInstanceListSupplier.builder()
            .withDiscoveryClient()
            .withCaching()
            .withZonePreference()
            .withHealthChecks()
            .build(context);
    }
}

withCaching() 启用本地缓存(默认 35 秒),withZonePreference() 优先选择同 zone 实例,withHealthChecks() 启用客户端主动健康检查。

36.5.4 Feign 与 Resilience4j 集成

spring:
  cloud:
    openfeign:
      circuitbreaker:
        enabled: true
        group:

@FeignClient(
    name = "order-service",
    fallbackFactory = OrderClientFallbackFactory.class

@Component
class OrderClientFallbackFactory implements FallbackFactory<OrderClient> {

    public OrderClient create(Throwable cause) {

            @Override

36.6 熔断与限流:Resilience4j

Spring Cloud 2025.0 推荐 Resilience4j 作为熔断 / 重试 / 限流 / 隔离 / 超时的统一实现。Resilience4j 由前 Hystrix 维护者重写,无 RxJava 依赖,与 Reactor 与 CompletableFuture 原生集成,适配 Java 17+ 与 Spring Boot 3.x。

36.6.1 五大模块全景

模块解决的问题配置前缀关键参数
CircuitBreaker快速失败 + 自动恢复resilience4j.circuitbreaker.instances.<name>failureRateThreshold / slidingWindowSize / waitDurationInOpenState
Retry瞬时失败重试resilience4j.retry.instances.<name>maxAttempts / waitDuration / retryExceptions
RateLimiter限流(令牌桶)resilience4j.ratelimiter.instances.<name>limitForPeriod / limitRefreshPeriod / timeoutDuration
Bulkhead并发隔离resilience4j.bulkhead.instances.<name>maxConcurrentCalls / maxWaitDuration
TimeLimiter异步超时resilience4j.timelimiter.instances.<name>timeoutDuration / cancelRunningFuture

36.6.2 CircuitBreaker 三态机

resilience4j:
  circuitbreaker:
    instances:
      orderService:
        failure-rate-threshold: 50
        slow-call-rate-threshold: 80
        slow-call-duration-threshold: 2s
        sliding-window-type: COUNT_BASED
        sliding-window-size: 100
        minimum-number-of-calls: 50
        wait-duration-in-open-state: 30s
        permitted-number-of-calls-in-half-open-state: 10
        automatic-transition-from-open-to-half-open-enabled: true
        record-exceptions:
          - java.io.IOException
          - org.springframework.web.client.HttpServerErrorException
        ignore-exceptions:
          - com.example.BusinessRuleException

熔断器三态机源自 Nygard《Release It》Stability Patterns 章节:CLOSED(正常,所有请求通过)→ 失败率超阈值 → OPEN(拒绝所有请求快速失败)→ wait-duration-in-open-state 到期 → HALF_OPEN(放 permitted-number-of-calls-in-half-open-state 个探针请求)→ 探针成功率 ≥ 阈值 → CLOSED;探针失败率超阈值 → OPEN。

图 36-6-1 · CircuitBreaker 三态机(CLOSED → OPEN → HALF_OPEN 迁移)

36.6.3 Retry 与幂等性

resilience4j:
  retry:
    instances:
      orderService:
        max-attempts: 3
        wait-duration: 200ms
        exponential-backoff-multiplier: 2.0
        retry-exceptions:
          - java.io.IOException
          - java.util.concurrent.TimeoutException
        ignore-exceptions:
          - com.example.BusinessRuleException

Retry 必须仅用于幂等接口。GET 与 PUT 默认幂等,POST 默认非幂等。非幂等 POST 启用 Retry 会导致重复下单 / 重复支付等业务事故。修复路径:对支付 / 下单类接口在客户端生成幂等键(UUID),服务端按幂等键拒绝重复请求;或在 Resilience4j 配置中显式禁用该接口的 Retry。

36.6.4 RateLimiter 与 Bulkhead

resilience4j:
  ratelimiter:
    instances:
      orderApi:
        limit-for-period: 100
        limit-refresh-period: 1s
        timeout-duration: 100ms
  bulkhead:
    instances:
      orderApi:
        max-concurrent-calls: 20
        max-wait-duration: 50ms

RateLimiter 实现令牌桶(Token Bucket):每 limit-refresh-period 补充 limit-for-period 个令牌,请求需获取一个令牌,获取超过 timeout-duration 抛 RequestNotPermitted。Bulkhead 实现并发隔离:同时执行的请求不超过 max-concurrent-calls,超过时等待 max-wait-duration,超时抛 BulkheadFullException。

36.6.5 TimeLimiter 与异步

@Service
public class OrderQueryService {

    @TimeLimiter(name = "orderService", fallbackMethod = "fallbackOrders")
    @CircuitBreaker(name = "orderService", fallbackMethod = "fallbackOrders")
    public CompletableFuture<List<OrderDto>> findOrders(Long customerId) {
        return CompletableFuture.supplyAsync(() ->
            orderRepository.findByCustomerId(customerId)
                .stream()
                .map(OrderDto::from)
                .toList()
        );
    }

    public CompletableFuture<List<OrderDto>> fallbackOrders(Long customerId, Throwable t) {
        log.warn("findOrders degraded for customer={} cause={}", customerId, t.getMessage());
        return CompletableFuture.completedFuture(List.of());
    }
}

@TimeLimiter 仅对 CompletableFuture 与 Reactor Mono Flux 生效。同步阻塞方法用 @TimeLimiter 不会被取消,需改写为异步形态。

36.7 API 网关:Spring Cloud Gateway

Spring Cloud Gateway 是基于 Spring WebFlux + Project Reactor + Reactor Netty 的反应式 API 网关,Spring 官方在 2018 年推出以替代 Netflix Zuul 1.x。其核心模型:RouteDefinition(路由定义)、Predicate(路由匹配条件)、GatewayFilter(路由级过滤器)、GlobalFilter(全局过滤器)。

36.7.1 路由定义

spring:
  cloud:
    gateway:
      routes:
        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**
            - Method=GET,POST
          filters:
            - StripPrefix=1
            - AddRequestHeader=X-Gateway-Source, gateway-v1
            - name: RequestRateLimiter
              args:
                redis-rate-limiter.replenishRate: 100
                redis-rate-limiter.burstCapacity: 200
                key-resolver: '#{@principalKeyResolver}'

        - id: order-service
          uri: lb://order-service
          predicates:
            - Path=/api/orders/**

            - StripPrefix=1

                name: orderServiceCb

uri: lb://order-service 表示走 Spring Cloud LoadBalancer 选择 order-service 的某个实例。Path=/api/orders/** 是 Predicate,匹配以 /api/orders/ 开头的请求。StripPrefix=1 是 GatewayFilter,转发前去掉路径第一段(此处去掉 /api)。

36.7.4 自定义 GlobalFilter

@Component
public class TenantHeaderGlobalFilter implements GlobalFilter, Ordered {

    private static final Logger log = LoggerFactory.getLogger(TenantHeaderGlobalFilter.class);

    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        String tenantId = exchange.getRequest().getHeaders().getFirst("X-Tenant-Id");
        if (tenantId == null || tenantId.isBlank()) {
            exchange.getResponse().setStatusCode(HttpStatus.BAD_REQUEST);
            return exchange.getResponse().setComplete();
        }
        ServerHttpRequest mutated = exchange.getRequest().mutate()
            .header("X-Internal-Tenant", tenantId.trim().toLowerCase())
            .build();
        return chain.filter(exchange.mutate().request(mutated).build())
            .doOnSuccess(v -> log.info("tenant={} path={}", tenantId, mutated.getPath()));
    }

    @Override
    public int getOrder() {
        return -100;
    }
}

getOrder() 返回值越小越先执行。ReactiveLoadBalancerClientFilter 的 order 为 10150,NettyRoutingFilter 的 order 为 Integer.MAX_VALUE,自定义全局过滤器通常使用负数确保在路由解析前执行。

36.7.5 Gateway 与 Web Starter 冲突

Spring Cloud Gateway 基于 WebFlux,不能引入 spring-boot-starter-web(基于 Servlet)。两者同时存在会触发启动失败:Spring MVC found on classpath, which is incompatible with Spring Cloud Gateway。Gateway 模块只能用 WebFlux 栈,业务服务可继续用 MVC 或 WebFlux 栈。

36.8.3 Observation API

Spring Framework 6 引入 io.micrometer.observation.Observation 作为统一的「指标 + 追踪 + 日志」观测点,把 Micrometer 指标、Tracing span、自定义事件三者合一:

@Service
public class CheckoutService {

    private final ObservationRegistry observationRegistry;
    private final OrderRepository orderRepository;

    public Order checkout(CheckoutRequest request) {
        return Observation.createNotStarted("order.checkout", observationRegistry)
            .lowCardinalityKeyValue("region", request.region())
            .highCardinalityKeyValue("orderId", String.valueOf(request.orderId()))
            .observe(() -> doCheckout(request));
    }

    private Order doCheckout(CheckoutRequest request) {
        return orderRepository.create(request);
    }
}

lowCardinalityKeyValue 适合 region / status 等取值有限的维度,会成为指标 tag。highCardinalityKeyValue 适合 orderId 等高基维度,只进 trace 不进指标(避免基数爆炸)。

36.9.1 非幂等 POST 重试导致重复事务

下单接口 POST /api/orders 默认非幂等。Resilience4j Retry 与 Feign 内置重试在网络抖动时会触发再次提交。修复路径:客户端在请求体或 header 中携带幂等键(Idempotency-Key: <uuid>),服务端用幂等键作为唯一约束(数据库唯一索引或 Redis SetNX)。Stripe / GitHub API 均采用此模式。

36.9.2 LoadBalancer 缓存陈旧

Spring Cloud LoadBalancer 默认在 ServiceInstanceListSupplier.builder().withCaching() 时缓存实例列表 35 秒。实例下线后,缓存中的过期实例仍可能被选中,导致连接被拒。

修复路径:缩短缓存 TTL 至 5-10 秒;启用客户端 Retry(spring.cloud.loadbalancer.retry.enabled=true);对返回 5xx 的实例触发立即标记为 down(HealthCheckServiceInstanceListSupplier)。

36.9.3 K8s 双重服务发现

在 Kubernetes 环境中同时使用 Eureka / Nacos / Consul 与 K8s Service 会导致两套发现机制并存:K8s Service 通过 ClusterIP 路由,客户端通过 Eureka 获取 Pod IP 直连。两条路径的健康检查、熔断、重试策略不一致,导致流量行为复杂。

修复路径:K8s 环境下统一用 spring-cloud-kubernetes-discovery,让 K8s Service 充当注册中心;或继续用外部注册中心但禁用 K8s Service 路由(改用 Headless Service);或迁移到 Service Mesh(Istio / Linkerd)统一治理。

36.9.4 跨边界 trace 断链

Trace 在以下边界容易断链:@Async 异步方法(线程切换丢失 MDC)、自定义 ExecutorService(同上)、Kafka 消息(producer / consumer 默认不传播)、Redis Lettuce(连接复用不传播)、Reactive flatMap 切换 Scheduler(Reactor Context 未自动绑定 MDC)。

修复路径:@Async 用 ContextSnapshotFactory 包装;自定义线程池用 ContextExecutorService.wrap;Kafka 加 micrometer-tracing-kafka 或 OTel Kafka instrumentation;Redis 加 OTel Lettuce instrumentation;Reactor 用 Hooks.enableAutomaticContextPropagation()。

36.9.5 Spring Cloud Sleuth 与 Tracing 共存

升级 Spring Boot 3.x 时,旧的 spring-cloud-starter-sleuth 必须移除,改用 micrometer-tracing-bridge-brave 或 micrometer-tracing-bridge-otel。两者同时存在会导致 Span 被双重创建,trace 不完整。

修复路径:mvn dependency:tree 确认无 spring-cloud-sleuth 依赖;移除所有 @NewSpan 注解,改用 @Observed 或 Observation API;配置文件中的 spring.sleuth.* 全部改为 management.tracing.*。

36.9.6 配置中心当业务状态使用

把库存数 / 订单状态 / 黑名单这类高频业务状态写入配置中心会绕过缓存一致性、审计与事务模型,导致更新延迟、数据不一致、无法回滚。

修复路径:配置中心仅存「应用启动 + 运行参数」(数据库连接、特性开关、超时阈值);业务状态进数据库 + 缓存,通过业务 API 访问;特性开关用专用平台(如 Unleash / LaunchDarkly)。
