第 2 章 · 自动配置引擎

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

@SpringBootApplication 注解的核心扩展点是 @EnableAutoConfiguration。该注解通过 @Import(AutoConfigurationImportSelector.class) 引入一个 DeferredImportSelector 实现,负责加载 classpath 中所有 META-INF/spring/...AutoConfiguration.imports 文件中列出的自动配置类。

候选清单加载完成后,自动配置引擎执行三层处理:第一层是 AutoConfigurationImportFilter 预过滤,基于元数据进行 OnClassCondition 快速过滤;第二层是 AutoConfigurationSorter 按 @AutoConfigureBefore / @AutoConfigureAfter / @AutoConfigureOrder 注解执行 Kahn 拓扑排序;第三层是 ConditionEvaluator 在配置类解析期执行 13 类 @Conditional 注解的短路求值。

本章按"宏观流程 → 条件求值 → 评估报告 → 排序算法 → 注册机制 → 自定义实战 → 调试技巧 → 源码深潜"八个层次组织。所有设计决策的目标是建立一套可覆盖、可排序、可诊断的自动装配系统:用户自定义 Bean 优先级高于框架默认实现,基础设施依赖顺序由声明决定,失败原因可通过条件评估报告完整追溯。

2.1 宏观流程

图 2-1-1 · @SpringBootApplication 注解层级(3 → 5 关联)

@SpringBootApplication 由三类注解组合而成:@SpringBootConfiguration(继承 @Configuration)、@EnableAutoConfiguration 与 @ComponentScan。@EnableAutoConfiguration 通过 @Import 引入 AutoConfigurationImportSelector,该类实现 DeferredImportSelector,在所有常规 @Configuration 处理完毕后再执行,确保用户自定义配置具备更高优先级。

public class AutoConfigurationImportSelector
    implements DeferredImportSelector, Ordered {

  @Override
  public String[] selectImports(AnnotationMetadata metadata) {
    List<String> configs = getCandidateConfigurations(metadata, attributes);
    configs = removeDuplicates(configs);
    configs = removeExclusions(configs, exclusions);
    configs = applyFilters(configs, autoConfigurationMetadata);
    return configs.toArray(new String[0]);
  }

  protected List<String> getCandidateConfigurations(...) {
    return ImportCandidates.load(
      AutoConfiguration.class, getBeanClassLoader()
    ).getCandidates();
  }
}

2.2 @Conditional 条件求值

图 2-2-1 · 两阶段短路求值(类级 + 方法级)

SpringBootCondition.matches() 一旦返回 ConditionOutcome.noMatch(...),ConditionEvaluator 立即返回,后续条件不再求值。类级 OnClassCondition 优先通过 ASM MetadataReader 读取元数据,避免直接 Class.forName 加载候选类,实现快速跳过。

图 2-2-2 · 条件优先级金字塔(成本由低到高)

成本由低到高排列:@ConditionalOnClass 仅查 ClassLoader,几近零开销;@ConditionalOnBean 需扫描已注册 BeanDefinition;@ConditionalOnExpression 需解析 SpEL 表达式,开销最高。优先级设计原则是成本最低的检查最先执行。

@ConditionalOnJava JDK 版本范围匹配 Parse

拓展案例:MyBatis Starter 条件组合

MybatisAutoConfiguration 同时使用 @ConditionalOnClass 与 @ConditionalOnSingleCandidate 双重条件:仅当 SqlSessionFactory 存在 classpath 且容器中恰好一个 DataSource 候选时激活。该模式在多数据源场景下避免歧义注入。

@Configuration
@ConditionalOnClass({ SqlSessionFactory.class, SqlSessionFactoryBean.class })
@ConditionalOnSingleCandidate(DataSource.class)
@EnableConfigurationProperties(MybatisProperties.class)
@AutoConfigureAfter(DataSourceAutoConfiguration.class)
public class MybatisAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public SqlSessionFactory sqlSessionFactory(DataSource dataSource) throws Exception {
        SqlSessionFactoryBean factory = new SqlSessionFactoryBean();
        factory.setDataSource(dataSource);
        return factory.getObject();
    }

    @Bean
    @ConditionalOnMissingBean
    public SqlSessionTemplate sqlSessionTemplate(SqlSessionFactory sqlSessionFactory) {
        return new SqlSessionTemplate(sqlSessionFactory);
    }
}

拓展案例:SpringDoc 默认启用模式

SpringDoc 使用 @ConditionalOnProperty(matchIfMissing = true) 实现默认启用、按需关闭的设计模式。开发环境无需额外配置即获得 API 文档界面,生产环境通过单行属性禁用。

@Configuration
@ConditionalOnProperty(
    name = "springdoc.api-docs.enabled",
    matchIfMissing = true
)
@ConditionalOnClass(OpenAPI.class)
public class SpringDocAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean
    public OpenAPI openAPI() {
        return new OpenAPI()
            .info(new Info().title("API Documentation"));
    }
}

2.4 拓扑排序

图 2-4-1 · AutoConfiguration 依赖 DAG(@AutoConfigureAfter)

箭头 A → B 表示 B 声明 @AutoConfigureAfter(A.class),即 A 必须在 B 之前处理。这些有向边构成有向无环图(DAG),需经拓扑排序确定最终处理顺序。

表 2-4-1:Kahn 算法步骤

步骤 处理节点 入度变化
t0 入度 0:DataSource -
t1 移除 DataSource Hikari / Jdbc / Flyway 入度 0
t2 移除 Hikari -
t3 移除 Jdbc Tx 入度 1
t4 移除 Flyway Jpa 入度 0
t5 移除 Jpa Tx 入度 0
t6 移除 Tx 排序完成
图 2-4-2 · Kahn 算法时间轴(7 步移除节点)
function topologicalSort(Graph dag):
    Map<Node, Integer> inDegree = computeInDegrees(dag)
    Queue<Node> queue = inDegree.entrySet()
        .stream().filter(e -> e.getValue() == 0)
        .map(Map.Entry::getKey)
        .collect(toQueue())

    List<Node> result = new ArrayList<>()
    while (queue not empty):
        Node node = queue.poll()
        result.add(node)
        for (Node neighbor : dag.neighbors(node)):
            inDegree.merge(neighbor, -1, Integer::sum)
            if (inDegree.get(neighbor) == 0):
                queue.add(neighbor)

    if (result.size() not equals dag.nodeCount()):
        throw new CycleDetectedException
    return result

参考 Kahn 算法原文:Kahn, A. B. (1962) "Topological sorting of large networks", Communications of the ACM, 5(11), 558-562。Kahn 算法时间复杂度 O(V + E),空间复杂度 O(V)。

图 2-4-3 · 循环依赖检测(Kahn 完成时 result.size < V)

Kahn 算法结束后 result.size() < V 表明图中存在环。Spring Boot 抛出异常并报告参与循环的配置类。生产环境的循环依赖通常由两个 starter 的 @AutoConfigureBefore / @AutoConfigureAfter 相互矛盾引起,合并第三方 starter 时需特别校验。

拓展案例:Spring Security JDBC 排序约束

JdbcUserDetailsManagerAutoConfiguration 必须在 DataSourceAutoConfiguration 与 HibernateJpaAutoConfiguration 之后处理。该约束保证 @ConditionalOnBean(DataSource.class) 求值时 DataSource 已注册,认证模块得以正确初始化。

@AutoConfiguration(
    after = {
        DataSourceAutoConfiguration.class,
        HibernateJpaAutoConfiguration.class
    }
)
@ConditionalOnClass(JdbcUserDetailsManager.class)
@ConditionalOnBean(DataSource.class)
public class JdbcUserDetailsManagerAutoConfiguration {

    @Bean
    @ConditionalOnMissingBean(UserDetailsService.class)
    public JdbcUserDetailsManager jdbcUserDetailsManager(DataSource dataSource) {
        return new JdbcUserDetailsManager(dataSource);
    }
}

@ConfigurationProperties(prefix = "app.notification")
public class NotificationProperties {

@AutoConfiguration
@ConditionalOnClass(NotificationService.class)
@ConditionalOnProperty(
    prefix = "app.notification",
    name = "enabled",
    havingValue = "true",
    matchIfMissing = true
)
@EnableConfigurationProperties(NotificationProperties.class)

    @Bean
    @ConditionalOnBean(NotificationService.class)

    public NotificationHealthIndicator notificationHealthIndicator(
            NotificationService service) {
        return new NotificationHealthIndicator(service);
    }
}

陷阱编号 名称 现象 修复策略
1 @ConditionalOnMissingBean 幽灵 Bean 同类型 Bean 重复注册 自动配置类用 @AutoConfiguration + .imports 注册
2 @AutoConfigureAfter 仅对自动配置类生效 普通 @Configuration 注解被忽略 用 @DependsOn 或 @Bean 参数声明依赖
3 @ConditionalOnProperty 三大坑 新环境功能不生效 matchIfMissing + 对齐 havingValue 大小写
4 类级别 @ConditionalOnBean 不可靠 求值时机受配置类处理顺序影响 改用 @Bean 方法级别 @ConditionalOnBean
5 @AutoConfigureOrder 与 @Order 混淆 用 @Order 控制配置类顺序无效 @AutoConfigureOrder 用于配置类,@Order 用于 Bean

7 多 starter 排序冲突 StarterA 依赖 StarterB Bean 失败 通过 @AutoConfigureAfter 显式声明依赖
8 OnMissingBean 类型匹配陷阱 @Bean 返回 Object 导致条件误判 @Bean 方法明确返回业务接口类型

拓展案例:Druid 替换默认连接池

阿里巴巴 Druid Starter 通过 @AutoConfiguration(before = DataSourceAutoConfiguration.class) 与 @ConditionalOnMissingBean(DataSource.class) 实现"有条件替换"。引入依赖即取代默认 HikariCP,无需修改业务代码。

@AutoConfiguration(before = DataSourceAutoConfiguration.class)
@ConditionalOnClass(DruidDataSource.class)
@EnableConfigurationProperties({
    DruidStatProperties.class,
    DataSourceProperties.class
})
public class DruidDataSourceAutoConfigure {

    @Bean
    @ConditionalOnMissingBean(DataSource.class)

    public DataSource dataSource() {

    }

    @Bean
    @ConditionalOnProperty(name = "spring.datasource.druid.stat-view-servlet.enabled")
    public ServletRegistrationBean<StatViewServlet> statViewServlet() {
        return new ServletRegistrationBean<>(new StatViewServlet(), "/druid/*");
    }
}

@Component
public class AutoConfigDiagnostics implements ApplicationRunner {

    private final ConditionEvaluationReport report;

    public AutoConfigDiagnostics(ConfigurableApplicationContext context) {
        this.report = ConditionEvaluationReport.get(context.getBeanFactory());
    }

    @Override
    public void run(ApplicationArguments args) {
        ConditionEvaluationReport.ConditionAndOutcomes outcomes =
            report.getConditionAndOutcomesBySource()
                .get("com.example.MyAutoConfiguration");

        if (outcomes != null) {
            outcomes.forEach(co -> {
                System.out.printf("condition=%s, match=%s%n",
                    co.getCondition().getClass().getSimpleName(),
                    co.getOutcome().isMatch());
            });
        }
    }
}

2.末 本章要点

  • @SpringBootApplication 由 @SpringBootConfiguration、@EnableAutoConfiguration 与 @ComponentScan 组合而成。@EnableAutoConfiguration 通过 @Import(AutoConfigurationImportSelector.class) 引入 DeferredImportSelector,在用户 @Configuration 处理完毕后批量执行,确保用户 Bean 优先级。

  • 自动配置引擎包含三层处理:AutoConfigurationImportFilter 元数据预过滤、AutoConfigurationSorter 拓扑排序、ConditionEvaluator 13 类 @Conditional 注解的短路求值。短路求值显著降低典型 Web 应用的实际条件评估成本。

  • ConditionEvaluationReport 记录全部候选类的匹配证据,包含 positive matches、negative matches、exclusions 与 unconditional classes 四类。诊断时优先查看 negative matches,通过 Actuator /actuator/conditions 端点获得结构化 JSON。

  • AutoConfigurationSorter 使用 Kahn 算法对 @AutoConfigureBefore / @AutoConfigureAfter / @AutoConfigureOrder 形成的 DAG 排序,时间复杂度 O(V + E)。环检测在排序结果不完整时抛出异常并报告参与循环的类。

  • Boot 3.0 起自动配置注册迁移到 META-INF/spring/...AutoConfiguration.imports。spring.factories 中 EnableAutoConfiguration key 不再被读取,但其他 SPI 仍走旧文件。

  • 自定义 starter 标准结构包含 @ConfigurationProperties、@AutoConfiguration + @ConditionalOnXxx、META-INF/spring/...AutoConfiguration.imports 与 spring-configuration-metadata.json 四部分。

  • 调试入口包含命令行 --debug、Actuator /actuator/conditions 与程序化 ConditionEvaluationReport.get。三者用途互补:--debug 用于人工排障,Actuator 用于自动化脚本,程序化访问用于集成测试断言。

  • 自动配置陷阱清单覆盖 @ConditionalOnMissingBean 幽灵 Bean、@AutoConfigureAfter 适用范围、@ConditionalOnProperty 三大坑、类级 @ConditionalOnBean 时机问题、@AutoConfigureOrder 与 @Order 混淆、@ConditionalOnClass 字符串与 Class 形式、starter 间排序冲突、OnMissingBean 类型匹配陷阱八类。
