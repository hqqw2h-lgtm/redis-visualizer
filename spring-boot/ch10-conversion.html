第 10 章 · 类型转换与格式化

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

Spring 类型转换体系经历了三代演进:PropertyEditor(Java Beans 时代)、Converter 与 ConversionService(Spring 3.0)、Formatter 与 FormatterRegistry(Spring 3.0)。三代机制在同一应用进程内共存,各自负责不同维度:PropertyEditor 兼容历史绑定,Converter 提供线程安全的通用类型转换,Formatter 提供 Locale 感知的双向文本转换。

本章接续第 9 章的 Binder 机制。@ConfigurationProperties 字符串到强类型字段的映射、@Value 占位符的类型注入、Spring MVC @RequestParam 与 @PathVariable 的参数绑定、SpEL 表达式的类型强制,均通过同一个 ConversionService 抽象完成。该统一性使得自定义 Converter 一处注册即可在所有绑定入口生效,前提是注册到正确的 ConversionService 实例。

10.1 三代转换机制演进

图 10-1-1 · 三代演进时间轴(2004 → 2018,5 个里程碑)

每一代解决前一代的痛点:Converter 通过无状态接口设计解决 PropertyEditor 的线程安全与状态污染问题;Formatter 通过引入 Locale 参数解决 Converter 在 Web 表单本地化场景的不足;ApplicationConversionService 与 WebConversionService 在 Boot 层面预注册大量常用 Converter,降低自定义负担。

10.2 Converter SPI

图 10-2-1 · Converter 三层接口结构(4 种接口 → 1 个统一注册表)

四种接口呈包含关系:Converter 与 ConverterFactory 在内部由 Converters 注册表适配为 GenericConverter,统一以 ConvertiblePair(sourceType, targetType) 作为索引键。ConditionalGenericConverter 通过 matches() 方法在缓存命中后追加运行时判定,支持注解驱动的精细匹配。

图 10-2-2:Converter<S,T> 单一职责

@FunctionalInterface
public interface Converter<S, T> {
    @Nullable
    T convert(S source);
}

Converter 是函数式接口,实现类应保持无状态。所有依赖通过构造器注入,所有可变状态使用线程安全容器。该约束是其作为全局单例可被并发调用的前提。

图 10-2-3:三种实现示例

@Component
public class StringToLocalDateConverter implements Converter<String, LocalDate> {

    private static final DateTimeFormatter FORMATTER =
            DateTimeFormatter.ofPattern("yyyy-MM-dd");

    @Override
    public LocalDate convert(String source) {
        if (source == null || source.isBlank()) {
            return null;
        }
        return LocalDate.parse(source.trim(), FORMATTER);
    }
}

@Component
public class StringToMoneyConverter implements Converter<String, Money> {

    @Override
    public Money convert(String source) {
        String[] parts = source.trim().split("\\s+", 2);
        if (parts.length != 2) {
            throw new IllegalArgumentException("Expected: <currency> <amount>");
        }
        return new Money(Currency.getInstance(parts[0]), new BigDecimal(parts[1]));
    }
}

StringToLocalDateConverter 持有 DateTimeFormatter 作为 static final 字段,DateTimeFormatter 自身不可变且线程安全。StringToMoneyConverter 不持有任何字段,完全无状态。两者均可作为 Spring 单例 Bean 安全注册。

表 10-2-1:ConverterFactory 适用场景

String → 单一领域类型 1 不适用 Converter

图 10-2-4:ConverterFactory 实现示例

public interface ConverterFactory<S, R> {
    <T extends R> Converter<S, T> getConverter(Class<T> targetType);
}

@Component
public class StringToEnumConverterFactory implements ConverterFactory<String, Enum> {

    @Override
    public <T extends Enum> Converter<String, T> getConverter(Class<T> targetType) {
        return new StringToEnumConverter<>(targetType);
    }

    private static class StringToEnumConverter<T extends Enum>
            implements Converter<String, T> {

        private final Class<T> enumType;

        StringToEnumConverter(Class<T> enumType) {
            this.enumType = enumType;
        }

        @Override
        public T convert(String source) {
            if (source == null || source.isBlank()) return null;
            return Enum.valueOf(enumType, source.trim().toUpperCase());
        }
    }
}

工厂模式将"目标类型已知"的细节延迟到运行时。Boot 内置的 LenientStringToEnumConverterFactory 在此基础上额外支持忽略大小写、连字符与下划线互换,使得 yml 中可写 log-level: debug 直接绑定到 LogLevel.DEBUG。

图 10-2-5 · GenericConverter 时序(候选列表 + matches 判定 + 父类型回退)

GenericConverter 与 Converter 的关键差异在于 convert() 方法多两个参数:sourceType 与 targetType。这两个 TypeDescriptor 实例携带泛型参数(如 List<Integer> 中的 Integer)与字段注解(如 @DateTimeFormat),使得同一个转换器可根据上下文产生不同行为。

图 10-2-6:ConditionalGenericConverter 示例

@Component
public class JsonStringToObjectConverter implements ConditionalGenericConverter {

    private final ObjectMapper objectMapper;

    public JsonStringToObjectConverter(ObjectMapper objectMapper) {
        this.objectMapper = objectMapper;
    }

    @Override
    public Set<ConvertiblePair> getConvertibleTypes() {
        return Set.of(new ConvertiblePair(String.class, Object.class));
    }

    @Override
    public boolean matches(TypeDescriptor sourceType, TypeDescriptor targetType) {
        return targetType.hasAnnotation(JsonDeserialize.class);
    }

    @Override
    public Object convert(Object source, TypeDescriptor sourceType,
                          TypeDescriptor targetType) {
        try {
            return objectMapper.readValue((String) source, targetType.getType());
        } catch (JsonProcessingException e) {
            throw new ConversionFailedException(sourceType, targetType, source, e);
        }
    }
}

getConvertibleTypes() 返回的 ConvertiblePair(String, Object) 几乎匹配所有 String → 任意对象 的转换。matches() 在缓存命中后再次检查目标字段是否带 @JsonDeserialize,只有显式标注的字段才走 JSON 反序列化路径。该模式避免了过宽匹配污染普通转换路径。

表 10-2-2:三种注册方式

注册方式 生效范围 适用场景
@Component 自动扫描 由 Boot 自动注册到 mvcConversionService Web 层简单转换
@ConfigurationPropertiesBinding 注册到 @ConfigurationProperties 的 Binder 配置绑定专用
WebMvcConfigurer.addFormatters() 仅 Web 层 mvcConversionService Web 表单与请求参数
手动注入 ConfigurableConversionService 取决于注入实例 测试 / 批处理 / 自定义编排

@Component 注解的 Converter 默认仅注册到 Web 层,@ConfigurationProperties 绑定路径不可见。需在两处均生效时,应同时标注 @Component 与 @ConfigurationPropertiesBinding,或在 WebMvcConfigurer 与 BinderFactory 中分别注册。

10.3 Formatter SPI

图 10-3-1 · Formatter 接口结构(Printer + Parser → Formatter,AnnotationFormatterFactory 旁路)

Formatter 同时继承 Printer 与 Parser,接口意图是"对同一类型 T 提供双向、Locale 感知的字符串与对象转换"。AnnotationFormatterFactory 提供"根据注解动态构造 Printer / Parser"的机制,是 @DateTimeFormat 与 @NumberFormat 的实现基础。

图 10-3-2:Formatter 实现示例

@Component
public class MoneyFormatter implements Formatter<Money> {

    @Override
    public String print(Money money, Locale locale) {
        NumberFormat nf = NumberFormat.getCurrencyInstance(locale);
        nf.setCurrency(money.getCurrency());
        return nf.format(money.getAmount());
    }

    @Override
    public Money parse(String text, Locale locale) throws ParseException {
        NumberFormat nf = NumberFormat.getCurrencyInstance(locale);
        ParsePosition position = new ParsePosition(0);
        Number number = nf.parse(text, position);
        if (number == null || position.getIndex() != text.length()) {
            throw new ParseException("Unparseable money: " + text,
                    position.getErrorIndex());
        }
        return new Money(nf.getCurrency(), new BigDecimal(number.toString()));
    }
}
Locale print(Money(CNY, 1234.56)) parse() 接受输入
Locale.US $1,234.56 $1,234.56
Locale.GERMANY 1.234,56 € 1.234,56 €
Locale.CHINA ¥1,234.56 ¥1,234.56
Locale.JAPAN ￥1,235 ￥1,235

不同 Locale 下千分位、小数符与货币符各异,NumberFormat.getCurrencyInstance(locale) 是唯一能正确处理这些差异的标准 API。Formatter 将该 API 封装为统一接口,使得业务代码不感知 Locale 差异。

图 10-3-3 · Formatter 在 ConversionService 中的适配(Formatter → 2 个 Converter)

FormattingConversionService 在 addFormatter() 内部将 Formatter 适配为两个 Converter:PrinterConverter 处理 T → String,ParserConverter 处理 String → T。该适配使得调用方通过 ConversionService 统一接口即可访问所有 Formatter,无需关心是否经过适配。

字段类型 引擎实现 备注

java.util.Calendar DateFormatter 旧 API
Long DateFormatter (毫秒时间戳) 数字时间戳

@DateTimeFormat 由 DateTimeFormatAnnotationFormatterFactory 统一处理。注解可指定 pattern(自定义模式)、iso(ISO 8601 标准)、style(本地化样式)三种格式来源,三者互斥。

图 10-3-4:@DateTimeFormat 与 @NumberFormat 用法

public class OrderForm {

    @DateTimeFormat(pattern = "yyyy-MM-dd")
    private LocalDate orderDate;

    @DateTimeFormat(iso = DateTimeFormat.ISO.DATE_TIME)
    private LocalDateTime createdAt;

    @DateTimeFormat(pattern = "HH:mm")
    private LocalTime deliveryTime;

    @NumberFormat(style = NumberFormat.Style.CURRENCY)
    private BigDecimal price;

    @NumberFormat(style = NumberFormat.Style.PERCENT)
    private Double discount;

    @NumberFormat(pattern = "#,##0.00")
    private BigDecimal weight;
}

POST /orders
Content-Type: application/x-www-form-urlencoded

orderDate=2026-04-20
createdAt=2026-04-20T10:15:30
deliveryTime=09:30
price=128.00

@DateTimeFormat(pattern = "yyyy-MM-dd") 走 DateTimeFormatter.STRICT 模式,要求月份与日期均为两位数字。"2024-1-5" 会被拒绝,需改用 "yyyy-M-d" 才能接受单位数。该严格性是为了避免格式歧义(如 "01-02-03")。

图 10-3-5 · Formatter 在 Spring MVC 中的双向时序(GET 渲染 + POST 解析)

请求 Locale 由 LocaleResolver 决定(详见第 13 章)。Formatter.print() 与 Formatter.parse() 均在请求线程上下文中获取当前 Locale,业务代码看到的始终是 Money 对象,不感知字符串与本地化细节。

10.4 ConversionService 内部实现

图 10-4-1:GenericConversionService 内部结构

GenericConversionService 字段布局:

  ┌───────────────────────────────────────────────────────┐
  │ Converters converters                                 │
  │   ├── globalConverters: List<GenericConverter>        │
  │   └── converters: Map<ConvertiblePair, ConvertersForPair> │
  │            key  = (sourceType, targetType)            │
  │            value = LinkedList<GenericConverter>       │
  ├───────────────────────────────────────────────────────┤
  │ Map<ConverterCacheKey, GenericConverter> converterCache│
  │   实现: ConcurrentReferenceHashMap (软引用)            │
  │   键: (sourceTypeDescriptor, targetTypeDescriptor)    │
  │   值: 命中的 GenericConverter 或 NO_MATCH 哨兵          │
  └───────────────────────────────────────────────────────┘

Converters 注册表按 ConvertiblePair 索引,同一类型对可挂多个 GenericConverter(实际由 Converter 与 ConverterFactory 适配而来)。converterCache 缓存 (sourceType, targetType) 到具体匹配结果的映射,失败匹配以 NO_MATCH 哨兵存入,避免后续重复遍历。

图 10-4-2 · convert() 解析算法(4 层匹配 + NO_MATCH 哨兵优化)

匹配算法分四层:① 同类型快速通道 ② 缓存查找 ③ 注册表精确匹配 ④ 父类型回退。第 ④ 步遵循 Class.isAssignableFrom() 规则,使得注册的 Object → String 转换器能匹配 Integer → String 请求(若无更精确转换器)。

表 10-4-1:TypeDescriptor 携带的元数据

元数据 用途 示例
Class<?> objectType 目标基础类型 List.class
ResolvableType type 完整泛型签名 List<Integer>
Annotation[] annotations 字段或参数上的所有注解 @DateTimeFormat
TypeDescriptor elementTypeDescriptor 集合元素类型 Integer 在 List<Integer>
TypeDescriptor mapKeyTypeDescriptor Map 的键类型 String 在 Map<String, Long>
TypeDescriptor mapValueTypeDescriptor Map 的值类型 Long 在 Map<String, Long>

TypeDescriptor 是 GenericConverter 与普通 Converter 的关键差异。普通 Converter 只知道 Class<T>,对集合元素类型与字段注解无感;GenericConverter 通过 TypeDescriptor 可访问完整类型上下文,实现对 List<Integer> 与 List<String> 的差异化处理。

图 10-4-3 · converterCache 命中率优化(冷启动 → 热点 → NO_MATCH)

ConcurrentReferenceHashMap 使用软引用键,在内存压力下可被回收。NO_MATCH 哨兵的存在使得"已确认无匹配"的查询也只需一次缓存命中,无需重复遍历注册表。生产实测显示首次转换约 50-200 µs,缓存命中后降至 < 5 µs。

表 10-4-2:ApplicationConversionService 增强转换器

转换器 功能 yml 示例
StringToDurationConverter "5s" / "500ms" / "PT30S" → Duration timeout: 5s
StringToDataSizeConverter "10MB" / "1GB" / "512KB" → DataSize max-size: 10MB

10.5 与 SpEL 的协作

图 10-5-1 · SpEL 求值时的类型强制(SpEL 引擎委托 ConversionService)

SpEL 引擎不内置类型转换,而是通过 TypeConverter SPI 委托外部实现。StandardTypeConverter 是默认实现,内部封装 ConversionService。该解耦设计使得 SpEL 与 @Value、@ConfigurationProperties、Binder 共享同一套类型系统,避免出现"占位符注入能识别 5s 但 SpEL 不能"的不一致。

表 10-5-1:@Value 注入的类型转换路径

注解写法 解析阶段 转换器
@Value("${app.port}") int port 占位符 → ConversionService StringToInteger
@Value("${app.timeout}") Duration t 占位符 → ConversionService StringToDurationConverter
@Value("#{2 * 1024}") long size SpEL → TypeConverter NumberToLong (Integer → Long)
@Value("#{T(LocalDate).now()}") LocalDate today SpEL → 直接返回 无需转换
@Value("#{${app.base} + 1}") int port 占位符嵌套 SpEL StringToInteger 再加法
@Value("${app.tags}") List<String> 占位符 → ConversionService DelimitedStringToCollection
@Value("${app.intervals}") int[] arr 占位符 → ConversionService DelimitedStringToArray + StringToInteger

${...} 是属性占位符,在 Environment 中查值后通过 ConversionService 转目标类型。#{...} 是 SpEL 表达式,在 SpEL 引擎内求值后通过 TypeConverter 转目标类型。两者可嵌套,先解析占位符再求值表达式。

    private DataSize maxFileSize;
    private Period retentionPeriod;
    private List<String> allowedOrigins;

}

myapp:

Binder 通过 BindConverter(内部委托 ApplicationConversionService)将 yml 中的字符串转换为字段目标类型。StringToDurationConverter、StringToDataSizeConverter、StringToPeriodConverter 均由 ApplicationConversionService 在启动时自动注册,无需开发者额外配置。

表 10-5-2:三处 ConversionService 实例对比

实例 注册路径 作用范围
ApplicationConversionService.sharedInstance 静态单例 @ConfigurationProperties 默认

mvcConversionService WebMvcAutoConfiguration Web 层 @RequestParam/@PathVariable

Boot 应用中典型实例数量为 2:mvcConversionService(Web 层)与 Binder 内部使用的 ApplicationConversionService.sharedInstance(配置绑定)。自定义 Converter 需在两处分别注册才能在两个路径下均生效。

10.6 陷阱与限制

图 10-6-1 · 类型转换陷阱分类(4 大类 8 个高发场景)

表 10-6-1:陷阱明细与诊断

# 陷阱 症状 根因 修复
1 @Component Converter 未生效 转换抛 ConverterNotFound 类不在 ComponentScan 路径 调整包结构或显式 addConverter
2 Web 注册转换器配置绑定不生效 yml 绑定失败,Controller 正常 mvcConversionService 与 Binder 不共享 加 @ConfigurationPropertiesBinding
3 @InitBinder 局部覆盖全局 同名字段在不同 Controller 行为不一致 PropertyEditor 优先级高于 ConversionService 删除 @InitBinder,改用全局 Converter
4 GenericConverter 过宽 String → Integer 走自定义 JSON ConvertiblePair (String, Object) 命中率高 改 ConditionalGenericConverter
5 yyyy-MM-dd 不接受 1-5 DateTimeParseException DateTimeFormatter STRICT 模式 改 yyyy-M-d 或自定义多格式 Converter
6 "yes" 注入到 boolean 字段意外为 true 业务字符串被误转布尔 StringToBooleanConverter 接受多别名 yml 加引号 "yes"
7 并发转换返回污染数据 偶发 NPE 或错值 Converter 持有实例字段 改无状态实现
8 SimpleDateFormat NPE 高并发下偶发 SDF 非线程安全 ThreadLocal 包装或 DateTimeFormatter 替代

图 10-6-2:@DateTimeFormat 严格匹配示例

模式 yyyy-MM-dd 的解析行为:

  输入字符串              结果
  ───────────────────────────────────────────
  "2026-04-20"          ✓ LocalDate.of(2026, 4, 20)
  "2026-04-5"           ✗ DateTimeParseException(dd 要求两位)
  "2026-4-20"           ✗ DateTimeParseException(MM 要求两位)
  "2026/04/20"          ✗ DateTimeParseException(分隔符不匹配)
  "26-04-20"            ✗ DateTimeParseException(yyyy 要求四位)

模式 yyyy-M-d 的解析行为:

  "2026-04-20"          ✓ LocalDate.of(2026, 4, 20)
  "2026-4-20"           ✓ LocalDate.of(2026, 4, 20)
  "2026-4-5"            ✓ LocalDate.of(2026, 4, 5)
  "2026-04-5"           ✓ LocalDate.of(2026, 4, 5)

yyyy-M-d 模式接受 1 至 2 位月份与日期,适合人工输入场景。yyyy-MM-dd 适合机器生成场景(如 HTML5 <input type="date">,RFC 3339 强制两位)。模式选择应匹配数据来源约定。

反例与正例对比

反模式(有状态 Converter):

  @Component
  class StatefulConverter implements Converter<String, Order> {
      private Order lastConverted;

      @Override
      public Order convert(String source) {
          Order order = parse(source);
          this.lastConverted = order;
          return order;
      }
  }

  并发后果:
    Thread-1 写 lastConverted = A
    Thread-2 写 lastConverted = B
    Thread-1 读 lastConverted → B (数据竞争)

正模式(无状态 Converter):

  @Component
  class StatelessConverter implements Converter<String, Order> {
      private final OrderParser parser;

      StatelessConverter(OrderParser parser) {
          this.parser = parser;
      }

      @Override
      public Order convert(String source) {
          return parser.parse(source);
      }
  }

  并发安全:每次调用独立,parser 自身不可变

Converter 与 Formatter 作为单例 Bean 被并发访问。所有可变状态必须使用 ThreadLocal、原子类或不可变对象封装。DateTimeFormatter、Pattern、ObjectMapper 均为线程安全,可作为 static final 字段;SimpleDateFormat、Calendar 非线程安全,需特别处理。

图 10-7-2:GenericConversionService.getConverter 摘要

protected GenericConverter getConverter(TypeDescriptor sourceType,
                                         TypeDescriptor targetType) {
    ConverterCacheKey key = new ConverterCacheKey(sourceType, targetType);
    GenericConverter converter = this.converterCache.get(key);
    if (converter != null) {
        return (converter != NO_MATCH) ? converter : null;
    }
    converter = this.converters.find(sourceType, targetType);
    if (converter == null) {
        converter = getDefaultConverter(sourceType, targetType);
    }
    if (converter != null) {
        this.converterCache.put(key, converter);
        return converter;
    }
    this.converterCache.put(key, NO_MATCH);
    return null;
}

设计取舍

取舍 1:Converter 与 Formatter 双 SPI

Converter 与 Formatter 在能力上有重叠(均能完成 String ↔ T),但分别面向不同关注点。Converter 强调"通用类型转换,无 Locale 依赖";Formatter 强调"Web 表单本地化展示与解析"。两者在 ConversionService 内部统一为 GenericConverter,对调用方透明。该双轨设计避免了"通用 API 被本地化复杂度污染"的问题。

取舍 2:converterCache 软引用

ConcurrentReferenceHashMap 使用软引用键,允许在内存压力下回收。代价是缓存命中后可能再次查找,生产实测无可观察影响。该设计避免了"缓存无限增长导致 OOM"的退化场景,适合长生命周期的单例 ConversionService。

取舍 3:NO_MATCH 哨兵

否定缓存使得"无匹配"也成为 O(1) 查询。代价是缓存条目数量加倍,但 ConvertiblePair 数量受限(典型应用 < 200 对),内存影响可忽略。该模式来自经典编译器优化技术,在 Spring 框架内也广泛应用于注解处理与 BeanFactory 查找。

10.8 本章要点

要点 源码依据 易错点
三代机制共存:PropertyEditor / Converter / Formatter TypeConverterDelegate 局部 PropertyEditor 优先于全局 Converter
Converter 三层接口:Converter / Factory / Generic Converters 注册表 GenericConverter 过宽影响其他匹配
