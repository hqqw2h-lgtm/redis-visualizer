第 15 章 · Reactive 与 WebFlux

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

Reactive 编程是一套解决"少量线程处理大量并发请求"问题的标准。理解背压、EventLoop 与 Scheduler 的协作机制,是判断何时使用 WebFlux、何时坚持传统 MVC 的关键。

传统阻塞模型(如 HikariCP 连接池,见第 18 章)中,线程在等待数据库、远程 HTTP 或文件 I/O 时被占用。Reactive 模型将等待交给非阻塞 I/O 与回调信号,数据就绪后由 EventLoop 或 Scheduler 推进后续处理。本章从 Reactive Streams 规范起,经过 Project Reactor 的 Mono / Flux,到 Spring WebFlux 实际应用与性能对比。

背压(Backpressure)是 Reactive Streams 的核心机制:下游通过 request(n) 告诉上游准备接收 n 个元素。

R2DBC 反应式数据库

public interface OrderRepository extends ReactiveCrudRepository<Order, Long> {
    Flux<Order> findByAgeGreaterThan(int age);
}

@RestController
class OrderController {
    @GetMapping(value = "/api/orders/older-than/{age}",
        produces = MediaType.APPLICATION_NDJSON_VALUE)

        return orders.findByAgeGreaterThan(age)
            .map(OrderDto::from);

15.3 Project Reactor 核心

Spring WebFlux 使用 Project Reactor 作为 Reactive Streams 实现。核心类型只有两个:Mono<T>(0..1 元素)与 Flux<T>(0..N 元素)。

图 15-3-1 · Mono vs Flux(0..1 vs 0..N — 共享 Publisher 协议 + 背压 + 惰性)
图 15-3-2 · 操作符链执行(Assembly 时构建 + Subscription 时反向 request 信号)

Assembly Time vs Subscription Time

Reactor 操作符链是完全惰性的。调用 subscribe() 之前,.map()、.filter() 等操作符仅构建内部 Publisher 装饰器链(Assembly),不会触发任何数据处理。

  • 同一个 Flux 可被多次 subscribe,每次产生独立数据流
  • 操作符顺序在 Assembly 阶段固定,但 Subscription 阶段信号反向传播
  • 常见 Bug:忘记 subscribe(),响应式链从未执行
Flux<Integer> chain = Flux.just(1, 2, 3, 4, 5)
    .doOnSubscribe(s -> log.info("subscribed"))
    .map(v -> { log.info("map {}", v); return v * 2; })
    .filter(v -> { log.info("filter {}", v); return v > 5; });

log.info("assembly finished");
chain.subscribe(v -> log.info("result={}", v));

15.4 Scheduler 调度策略

Project Reactor 提供四种内置 Scheduler,用于控制操作符在哪个线程上执行。

图 15-4-1 · 四种 Scheduler(immediate / single / parallel / boundedElastic)

表 15-4-1:Scheduler 选型

Scheduler线程数适用不适用
immediate当前线程轻量回调跨线程切换
single1串行守护、定时调度高并发
parallelCPU 核数数据转换、CPU 密集阻塞 I/O
boundedElastic默认 10×CPU 核数阻塞 I/O 隔离CPU 密集计算
图 15-4-2 · publishOn vs subscribeOn(关键差异 — 影响范围 + 链中位置)

subscribeOn(Scheduler) 影响订阅信号与数据源执行线程。一条链通常只放一个;多个 subscribeOn 不叠加切换效果。

publishOn(Scheduler) 影响其下游操作符执行线程。可在链中多次使用,实现线程弹跳。

Flux.just("A")
    .doOnSubscribe(s -> log.info("subscribed on {}", Thread.currentThread().getName()))
    .doOnNext(v -> log.info("source {} on {}", v, Thread.currentThread().getName()))
    .subscribeOn(Schedulers.boundedElastic())
    .publishOn(Schedulers.parallel())
    .map(v -> {
        log.info("map {} on {}", v, Thread.currentThread().getName());
        return v.toLowerCase();
    })
    .blockLast();

WebFlux + 阻塞下游(真实生产事故)

某团队在 Spring Cloud Gateway 自定义 Filter 中直接调用 JDBC 查询 IP 黑名单,EventLoop 线程被阻塞。Gateway 默认只有少量 EventLoop 线程,几个慢查询就占满所有线程,网关彻底冻结。

Mono.fromCallable(() -> jdbcTemplate.queryForObject("SELECT ...", String.class))
    .subscribeOn(Schedulers.boundedElastic());

return r2dbcTemplate.selectOne(query(where("id").is(id)), Order.class);

15.5 WebFlux vs Spring MVC 性能权衡

WebFlux 并非在所有场景下都优于 Spring MVC。选择取决于应用特性。

表 15-5-1:MVC 与 WebFlux 选型

维度Spring MVCSpring WebFlux
编程模型命令式、同步阻塞声明式、异步非阻塞
线程模型Thread-per-Request 200 线程Event Loop CPU 核心数线程
低并发延迟通常略优(调用链更短)接近,有操作符 / 调度开销
高并发 I/O线程池耗尽后排队明显非阻塞链路下更稳定
内存占用工作线程栈是主要开销线程栈少,有连接 / 缓冲区状态
调试难度堆栈清晰堆栈碎片化
生态兼容JDBC、JPA、模板引擎需 R2DBC、WebClient
学习曲线低高(Reactive 思维转变)

选择指南

  • 选 MVC:团队不熟悉 Reactive、下游大量 JDBC / JPA 调用、不需要高并发长连接
  • 选 WebFlux:需要处理大量并发连接(SSE、WebSocket、API Gateway)、全链路非阻塞
  • 混合使用:同一 Spring Boot Web 应用通常只选择 MVC 或 WebFlux 作为主 Web 栈

15.末 本章要点

  • Reactive 不等于更快。低并发、CPU 密集型场景下,MVC 直线调用栈比 Reactive 操作符链更快更易调试。Reactive 优势在高并发 I/O 密集型场景。
  • 背压是 Reactive 核心价值。没有背压的异步只是更快地把下游压垮。request(n) 机制让消费者按自己的速率拉取数据。
  • EventLoop 线程绝不能阻塞。一旦在 EventLoop 上做 JDBC 查询或 Thread.sleep(),整个应用吞吐量断崖式下降。必须用 Schedulers.boundedElastic() 隔离阻塞操作。
  • Mono 与 Flux 是 Project Reactor 核心类型。Mono 表示 0..1 元素,Flux 表示 0..N 元素。两者共享 Publisher 协议、背压机制与惰性求值。
  • Scheduler 类型对应不同场景。immediate 用于轻量回调,single 用于串行守护,parallel 用于 CPU 密集计算,boundedElastic 用于阻塞 I/O 隔离。
  • WebFlux 与 MVC 选型基于业务特性。下游全部非阻塞、需要处理大量并发连接时选 WebFlux;下游存在 JDBC / JPA 等阻塞调用、团队不熟悉 Reactive 时选 MVC。
  • Spring Cloud Gateway 是 WebFlux 最佳实践。Gateway Filter 链天然适配 Reactive 模型。
