第 13 章 · i18n 国际化:MessageSource 与 Locale 解析

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

应用走出单一语言市场后,国际化(Internationalization,缩写 i18n)从可选项变为必选项。Spring 的 i18n 架构围绕三个核心问题展开:消息来源(MessageSource)、用户语言识别(LocaleResolver)、消息格式化(MessageFormat)。三者协同工作,使得一次 getMessage("order.created", args, locale) 调用即可返回 "Order #1024 created" 或 "订单 #1024 已创建"。

本章按"接口体系 → 自动配置 → Locale 解析 → 消息格式化 → 视图集成 → 边界陷阱"组织。第一节解析 MessageSource 接口层次与三种实现;第二节剖析 Spring Boot 自动配置流程与 MessageSourceAutoConfiguration;第三节梳理消息解析算法与 Locale 回退链;第四节对比四种 LocaleResolver 的适用场景;第五节深入 MessageFormat 与 ICU 复数规则;第六节展示 Thymeleaf 与 REST API 的集成模式;末节列举五类高频生产陷阱。

阅读本章前建议掌握第 8 章的 Resource 抽象、第 12 章的校验消息模板与第 14 章 MVC 拦截器基础。

13.1 MessageSource 接口体系

13.1.1 接口层次结构

Spring 的国际化消息体系围绕 MessageSource 接口构建。该接口定义核心方法:给定消息 code、参数数组与 Locale,返回格式化后的本地化字符串。Spring 提供三种开箱即用的实现,分别面向不同的使用场景。

图 13-1 · MessageSource 接口体系(三种实现 + HierarchicalMessageSource 父子链)

ResourceBundleMessageSource 基于 JDK ResourceBundle,是 Spring Boot 默认实现;ReloadableResourceBundleMessageSource 使用 Spring Resource 抽象,适合外置文件热重载;StaticMessageSource 主要用于单元测试与原型验证。

13.1.2 三种实现的对比

实现类资源加载方式热重载编码控制典型场景
ResourceBundleMessageSourceJDK ResourceBundle(classpath)不支持Spring Boot 默认 UTF-8;手动声明应显式设置Spring Boot 默认、大多数应用
ReloadableResourceBundleMessageSourceSpring Resource 抽象file: 协议最可靠;JAR 内 classpath 不适合原生 UTF-8 支持需运行时修改消息的运营场景
StaticMessageSource代码硬编码 addMessage()不适用不适用单元测试、原型验证

13.1.3 三种实现的基本用法

@Bean
public MessageSource messageSource() {
    ResourceBundleMessageSource source = new ResourceBundleMessageSource();
    source.setBasenames("messages", "validation", "error");
    source.setDefaultEncoding("UTF-8");
    source.setUseCodeAsDefaultMessage(true);
    return source;
}

@Bean
public MessageSource reloadableMessageSource() {
    ReloadableResourceBundleMessageSource source =
        new ReloadableResourceBundleMessageSource();
    source.setBasenames("classpath:messages", "file:/config/messages");
    source.setDefaultEncoding("UTF-8");
    source.setCacheSeconds(60);
    return source;
}

@Bean
public MessageSource staticMessageSource() {
    StaticMessageSource source = new StaticMessageSource();
    source.addMessage("greeting", Locale.CHINESE, "欢迎,{0}");
    source.addMessage("greeting", Locale.ENGLISH, "Welcome, {0}");
    return source;
}

13.1.4 HierarchicalMessageSource 父子链

HierarchicalMessageSource 接口允许 MessageSource 形成父子关系。当子 MessageSource 找不到指定 code 时,自动委托给父 MessageSource。AbstractApplicationContext 也实现了 MessageSource 接口,内部委托给名为 messageSource 的 Bean。父子 ApplicationContext 之间自动建立 MessageSource 的父子链——子容器的 messageSource 找不到 code 时,回退到父容器。

这一机制对模块化应用尤其重要:基础模块的通用消息(错误码、按钮文案)定义在父容器,业务模块的特有消息定义在子容器,业务模块查找时优先匹配自身,缺失时回退到基础模块。

13.2 Spring Boot 自动配置

13.2.1 MessageSourceAutoConfiguration 流程

Spring Boot 通过 MessageSourceAutoConfiguration 自动创建 ResourceBundleMessageSource Bean。读取 spring.messages.* 属性配置行为,无需手动声明 @Bean——除非有特殊需求。

spring:
  messages:
    basename: messages,validation,errors
    encoding: UTF-8
    cache-duration: 3600s
    fallback-to-system-locale: false
    use-code-as-default-message: false

启动检查日志可明确看到 basename、编码与缓存策略已进入 MessageSource:

13.2.2 自动配置的关键约束

约束项默认值/规则违约后果
basename默认 messages资源缺失时 ResourceBundleCondition 不创建 Bean
Bean 名称必须为 messageSource名称不匹配则容器不委托
编码Spring Boot 默认 UTF-8旧版 JDK 或手写配置不显式设置时可能乱码
缓存默认永久缓存(cacheMillis = -1)测试环境调试需显式设置 cache-duration

AbstractApplicationContext.initMessageSource() 在容器初始化阶段查找名为 messageSource 的 Bean。如未找到,创建 DelegatingMessageSource——它将所有调用委托给父 MessageSource,无父则返回 null 或抛 NoSuchMessageException。这意味着 Bean 名称错误会导致框架内部无法访问消息源,即便业务代码可通过其他名称注入。

13.2.3 源码入口与运行时调用链

Spring i18n 源码主线分两段:启动期创建 messageSource Bean,请求期解析 Locale 并调用 getMessage(...)。前者属于 Spring Boot 自动配置,后者属于 Spring Framework MVC 与 context 模块。

13.2.4 资源解析与缓存

ResourceBundleMessageSource 的运行时成本主要分两层缓存:第一层缓存 ResourceBundle,避免每次都从 classpath 读取 properties;第二层缓存 MessageFormat,避免每次有参数消息都重新解析模式。理解这两层缓存,可解释"为何修改 classpath 下的 messages 文件不会立刻生效"以及"为何带参数的消息会触发单引号转义规则"。

图 13-4 · ResourceBundleMessageSource 解析与缓存路径(双层缓存)
源码入口真实职责
MessageSourceAutoConfiguration.ResourceBundleCondition检查 basename 对应资源是否存在
AbstractApplicationContext.initMessageSource()查找名为 messageSource 的 Bean
AbstractMessageSource.getMessageInternal(...)处理参数解析、默认消息、父 fallback
ResourceBundleMessageSource.resolveCode(...)从 bundle 查 code,按 Locale 创建 MessageFormat
ReloadableResourceBundleMessageSource.refreshProperties(...)根据 lastModified 决定重新加载

13.3.1 Locale 回退链算法

MessageSource 的消息解析遵循一套精确的 Locale 回退链:从最具体的 Locale(如 zh_CN)逐步回退到更宽泛的 Locale(如 zh),最终到默认文件(无 Locale 后缀)。

@Service
public class OrderNotificationService {

    private final MessageSource messageSource;

    public OrderNotificationService(MessageSource messageSource) {
        this.messageSource = messageSource;
    }

    public String getCreatedMessage(Long orderId, BigDecimal total, Locale locale) {
        String created = messageSource.getMessage(
            "order.created",
            new Object[]{orderId},
            locale
        );

        String totalMsg = messageSource.getMessage(
            "order.total",
            new Object[]{total},
            locale
        );

        return created + " " + totalMsg;
    }
}

13.3.3 MessageSourceAccessor 辅助类

MessageSourceAccessor 封装了 MessageSource,提供便捷方法——无需每次传入 Locale(默认使用 LocaleContextHolder.getLocale()),也无需处理 NoSuchMessageException。

@Service
public class NotificationService {

    private final MessageSourceAccessor accessor;

    public NotificationService(MessageSource messageSource) {
        this.accessor = new MessageSourceAccessor(messageSource);
    }

    public String getWelcomeMessage(String username) {
        return accessor.getMessage("welcome", new Object[]{username});
    }

    public String getSafeMessage(String code) {
        return accessor.getMessage(code, "Unknown Order message");
    }
}

13.3.4 Spring Security 的 MessageSource 集成

Spring Security 使用 MessageSource 本地化所有认证/授权错误消息。SpringSecurityMessageSource 是预配置的 ResourceBundleMessageSource,basename 为 org.springframework.security.messages。它内置 20+ 种语言翻译。

public class SpringSecurityMessageSource extends ResourceBundleMessageSource {
    public SpringSecurityMessageSource() {
        setBasename("org.springframework.security.messages");
    }
}

public abstract class AbstractUserDetailsAuthenticationProvider
        implements AuthenticationProvider, MessageSourceAware {

    protected MessageSourceAccessor messages =
        SpringSecurityMessageSource.getAccessor();

    @Override
    public Authentication authenticate(Authentication auth) {
        if (order == null) {
            throw new BadCredentialsException(
                messages.getMessage(
                    "AbstractUserDetailsAuthenticationProvider.badCredentials",
                    "Bad credentials"));
        }
    }
}

业务侧只需在 messages_zh_CN.properties 中定义 AbstractUserDetailsAuthenticationProvider.badCredentials=用户名或密码错误,即可覆盖默认英文文案。

13.4 LocaleResolver 解析策略

13.4.1 四种 LocaleResolver 实现

MessageSource 解决了"消息从哪里来"的问题,而 LocaleResolver 解决"用户说什么语言"。Spring MVC 提供四种 LocaleResolver 实现,覆盖从 HTTP 头自动检测到 URL 参数手动切换的全部场景。

图 13-6 · LocaleResolver 实现类全景(4 种实现 — 数据来源决定可切换性)

13.4.2 四种实现的对比

实现Locale 来源可切换持久性适用场景
AcceptHeaderLocaleResolverHTTP Accept-Language 头只读每次请求由浏览器决定Spring Boot 默认
SessionLocaleResolverHttpSession 属性可写Session 生命周期内需用户手动切换、无 Cookie 场景
CookieLocaleResolverCookie可写Cookie 过期前(可跨 Session)电商、SaaS 多语言站点
FixedLocaleResolver硬编码只读永久单语言内部系统

13.4.3 四种实现的配置示例

@Bean
public LocaleResolver sessionLocaleResolver() {
    SessionLocaleResolver resolver = new SessionLocaleResolver();
    resolver.setDefaultLocale(Locale.SIMPLIFIED_CHINESE);
    return resolver;
}

@Bean
public LocaleResolver cookieLocaleResolver() {
    CookieLocaleResolver resolver = new CookieLocaleResolver("app-locale");
    resolver.setDefaultLocale(Locale.ENGLISH);
    resolver.setCookieMaxAge(Duration.ofDays(365));
    resolver.setCookiePath("/");
    return resolver;
}

@Bean
public LocaleResolver fixedLocaleResolver() {
    return new FixedLocaleResolver(Locale.JAPAN);
}

13.4.4 LocaleChangeInterceptor 切换语言

@Configuration
public class WebConfig implements WebMvcConfigurer {

    @Bean
    public LocaleResolver localeResolver() {
        CookieLocaleResolver resolver = new CookieLocaleResolver("lang");
        resolver.setDefaultLocale(Locale.ENGLISH);
        resolver.setCookieMaxAge(Duration.ofDays(30));
        return resolver;
    }

    @Bean
    public LocaleChangeInterceptor localeChangeInterceptor() {
        LocaleChangeInterceptor interceptor = new LocaleChangeInterceptor();
        interceptor.setParamName("lang");
        return interceptor;
    }

    @Override
    public void addInterceptors(InterceptorRegistry registry) {
        registry.addInterceptor(localeChangeInterceptor());
    }
}

13.4.8 URL 路径策略

public class PathLocaleResolver implements LocaleResolver {

    private static final Pattern LOCALE_PATTERN =
        Pattern.compile("^/(en|zh|ja|ko|de|fr)/");

    @Override
    public Locale resolveLocale(HttpServletRequest request) {
        String path = request.getServletPath();
        Matcher matcher = LOCALE_PATTERN.matcher(path);
        if (matcher.find()) {
            return Locale.forLanguageTag(matcher.group(1));
        }
        return Locale.ENGLISH;
    }

    @Override
    public void setLocale(HttpServletRequest request,
                          HttpServletResponse response,
                          Locale locale) {
    }
}

Spring 默认不使用 ICU——ResourceBundleMessageSource 底层使用 JDK java.text.MessageFormat,不支持 ICU 的 plural / select / selectordinal 语法。要使用 ICU,需自定义 MessageSource 实现,或在前端处理复数逻辑。

13.5.4 日期与数字按 Locale 格式化

@RestController
public class LocaleFormattingController {

    @GetMapping("/format-demo")
    public Map<String, String> formatDemo(Locale locale) {
        BigDecimal price = new BigDecimal("12345.67");
        LocalDate date = LocalDate.of(2026, 1, 15);
        double percent = 0.856;

        NumberFormat currencyFmt = NumberFormat.getCurrencyInstance(locale);
        NumberFormat percentFmt = NumberFormat.getPercentInstance(locale);
        DateTimeFormatter dateFmt = DateTimeFormatter
            .ofLocalizedDate(FormatStyle.LONG)
            .withLocale(locale);

        return Map.of(
            "currency", currencyFmt.format(price),
            "percent", percentFmt.format(percent),
            "date", date.format(dateFmt)
        );
    }
}

    }

    @ExceptionHandler(OrderNotFoundException.class)
    public ProblemDetail handleOrderNotFound(
            OrderNotFoundException ex,
            Locale locale) {

        String title = messageSource.getMessage(
            "error.order.notfound.title", null, locale);
        String detail = messageSource.getMessage(
            "error.order.notfound.detail",
            new Object[]{ex.getOrderId()},
            locale);

        ProblemDetail problem = ProblemDetail.forStatus(
            HttpStatus.NOT_FOUND);
        problem.setTitle(title);
        problem.setDetail(detail);
        problem.setType(URI.create(
            "https://api.example.com/errors/order-not-found"));
        problem.setProperty("orderId", ex.getOrderId());

        return problem;
    }
}

13.6.3 Content-Language 响应头

@RestControllerAdvice
public class ContentLanguageAdvice implements ResponseBodyAdvice<Object> {
    @Override
    public boolean supports(MethodParameter returnType,
                            Class<? extends HttpMessageConverter<?>> type) {
        return true;
    }

    @Override
    public Object beforeBodyWrite(Object body, MethodParameter returnType,
                                  MediaType contentType,
                                  Class<? extends HttpMessageConverter<?>> type,
                                  ServerHttpRequest request,
                                  ServerHttpResponse response) {
        Locale locale = LocaleContextHolder.getLocale();
        response.getHeaders().setContentLanguage(locale);
        return body;
    }
}

注意不要在普通 Servlet Filter 中读取 LocaleContextHolder。DispatcherServlet 在进入 MVC 后才根据 LocaleResolver 绑定 Locale。

13.7.1 陷阱一:旧 JDK 或手写 MessageSource 编码不一致

JDK 8 及之前版本的 java.util.ResourceBundle 默认使用 ISO-8859-1 编码读取 .properties 文件——即便文件本身是 UTF-8 保存。

# messages_zh_CN.properties (UTF-8 保存)
greeting=欢迎使用本系统

# 运行结果(未设置 UTF-8)
getMessage("greeting", ...) -> "ä½ å¥½ï¼Œä¸ç"

# 旧版 workaround: 使用 native2ascii 转义
greeting=\u4f60\u597d\uff0c\u4e16\u754c

spring:
  messages:
    encoding: UTF-8

13.7.2 陷阱二:消息 key 缺失时的行为

messageSource.getMessage("nonexistent.key", null, locale);
// NoSuchMessageException

spring.messages.use-code-as-default-message=true
messageSource.getMessage("nonexistent.key", null, locale);
// 返回 "nonexistent.key" 静默失败

messageSource.getMessage("nonexistent.key", null,
    "Default fallback message", locale);
// 返回 "Default fallback message"

13.7.3 陷阱三:ReloadableResourceBundleMessageSource 的热重载限制

@Bean
public MessageSource messageSource() {
    ReloadableResourceBundleMessageSource source =
        new ReloadableResourceBundleMessageSource();

    source.setBasenames(
        "classpath:messages",
        "file:/opt/app/config/messages"
    );
    source.setCacheSeconds(30);
    source.setDefaultEncoding("UTF-8");

    return source;
}

JAR 是一个 ZIP 压缩包,运行时不能像普通文件一样被运营人员直接修改;即便能拿到时间戳,也通常只是 JAR 文件本身的修改时间。ReloadableResourceBundleMessageSource 通过比较资源的 lastModified 时间戳判断是否需要重新加载,因此外置 file: 资源才是生产热重载的可靠形态。

13.7.5 陷阱五:MessageFormat 的单引号转义

# 错误: 单引号导致参数 {0} 被吞掉
welcome.msg=It's {0}'s birthday
# getMessage("welcome.msg", ["Customer C1001"], en)
# -> "Its {0}s birthday"

# 正确: 双单引号转义
welcome.msg=It''s {0}''s birthday
# getMessage("welcome.msg", ["Customer C1001"], en)
# -> "It's Customer C1001's birthday"

# 另一个常见错误: 中文消息中的英文名
error.sql=数据库执行 '{0}' 时发生错误
# -> "数据库执行 {0} 时发生错误" (参数未替换)

# 正确
error.sql=数据库执行 ''{0}'' 时发生错误
# -> "数据库执行 'SELECT * FROM orders' 时发生错误"

alwaysUseMessageFormat 的影响:AbstractMessageSource 有一个 alwaysUseMessageFormat 属性(默认 false)。设为 false 时,如消息不含占位符,Spring 跳过 MessageFormat 解析,直接返回原始字符串——此时单引号无需转义。但一旦消息包含占位符,MessageFormat 解析被激活,单引号规则立即生效。

13.8 本章小结

本章系统剖析了 Spring 的国际化体系,核心要点归纳如下:

MessageSource 三种实现各有定位:ResourceBundleMessageSource 是 Spring Boot 默认选择,基于 JDK ResourceBundle;ReloadableResourceBundleMessageSource 支持 file: 协议热重载,适合运行时修改消息;StaticMessageSource 用于测试。HierarchicalMessageSource 支持父子链委托。

Spring Boot 通过 MessageSourceAutoConfiguration 自动创建 messageSource Bean。Bean 名称硬编码于 AbstractApplicationContext,自定义时务必沿用此名。spring.messages.* 属性配置 basename、编码与缓存策略。

消息解析遵循 Locale 回退链:zh_CN → zh → default。层次文件须完整,否则中间层缺失会导致意外回退到默认语言。fallbackToSystemLocale 影响 ResourceBundle 候选列表,生产环境应设为 false。

四种 LocaleResolver 覆盖全部场景:AcceptHeaderLocaleResolver 适合纯自动检测;CookieLocaleResolver 适合跨 Session 持久化;SessionLocaleResolver 适合 Session 内切换;FixedLocaleResolver 用于单语言系统。LocaleChangeInterceptor 实现 URL 参数切换,但不兼容只读的 AcceptHeaderLocaleResolver。

MessageFormat 模式支持位置参数、日期/时间/数字/货币格式化与选择格式;ICU4J 扩展提供完整的复数规则支持。Spring Validation 错误消息深度依赖 MessageSource 实现本地化,FieldError 携带 MessageSourceResolvable 接口的多个候选 code。

Thymeleaf 通过 #{...} 语法无缝集成 MessageSource;REST API 通过 @ExceptionHandler 加 ProblemDetail 实现本地化错误响应。Content-Language 响应头需手动设置,不要在 Servlet Filter 中读取 LocaleContextHolder。

五大陷阱:旧 JDK 或手写 MessageSource 须显式设为 UTF-8;消息缺失行为取决于 useCodeAsDefaultMessage 与 defaultMessage;热重载只对 file: 资源生效;Locale 回退链中间文件不可缺失;MessageFormat 的单引号必须用 '' 转义。
