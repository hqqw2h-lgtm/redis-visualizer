第 18 章 · HikariCP 连接池算法

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

数据库连接是 Web 应用最昂贵资源之一——创建一个 TCP 连接 + TLS 握手 + 数据库认证动辄数十毫秒。连接池的职责是复用这些连接,而 HikariCP 通过少做事、快借还、低开销维护,在常见微基准里长期保持领先。

上一章探讨 JDBC、JPA、R2DBC 如何进入数据库。当业务代码需要查询关系型数据库时,第一步通常是从连接池借一个连接。本章深入 HikariCP 的核心数据结构 ConcurrentBag(无锁并发容器)与 FastList(减少显式检查的高速列表),以及连接生命周期状态机与配置调优的科学方法。

18.1 为什么 HikariCP 最快

HikariCP 是 Spring Boot 2.x 以来默认数据库连接池。常见获取 / 归还连接微基准中,通常明显快于传统锁实现连接池。下面是教学化对比,具体数字会随 JDK、硬件、驱动版本与压测方法变化。

设计点 对应实现 性能收益

FastList 去掉部分边界检查、尾部扫描、引用比较 降低 Statement 关闭路径局部开销

18.2 ConcurrentBag 无锁借还算法

ConcurrentBag 是 HikariCP 的核心数据结构,也是性能远超竞品的关键。借鉴 .NET ConcurrentBag<T> 思想,针对 JDBC 连接池场景做深度定制。

图 18-2-1 · 三级连接获取流程(ThreadLocal 命中 → sharedList 扫描 → handoffQueue 等待)

应用线程调用 getConnection() 时,ConcurrentBag 依次尝试三层级获取策略。绝大多数情况下第一级 ThreadLocal 命中,避免任何跨线程竞争。

表 18-2-1:三级获取性能对比

层级 数据结构 命中场景 延迟
1 ThreadLocal 同一线程反复借还连接 微秒级,无锁
2 sharedList 跨线程或首次借连接 微秒级,CAS
3 handoffQueue 池内全部连接被占用 受 connectionTimeout 限制

ThreadLocal 命中率通常很高。传统 Servlet 线程池模型中,同一工作线程反复借还连接,ThreadLocal 命中率高;具体比例取决于请求调度、线程池大小、连接持有时间。命中时路径很短,通常只需本地列表读取与一次 CAS,避免全局扫描。

Checkout 诊断探针

@RestController
class CheckoutProbeController {
    private final DataSource dataSource;
    private final HikariDataSource hikari;

    CheckoutProbeController(DataSource dataSource) {
        this.dataSource = dataSource;
        this.hikari = dataSource.unwrap(HikariDataSource.class);
    }

    @GetMapping("/probe/checkout")
    Map<String, Object> checkout() throws SQLException {
        long start = System.nanoTime();
        try (Connection ignored = dataSource.getConnection()) {
            HikariPoolMXBean pool = hikari.getHikariPoolMXBean();
            long waitedMicros = TimeUnit.NANOSECONDS.toMicros(System.nanoTime() - start);
            return Map.of(
                "waitedMicros", waitedMicros,
                "active", pool.getActiveConnections(),
                "idle", pool.getIdleConnections(),
                "pending", pool.getThreadsAwaitingConnection()
            );
        }
    }
}

图 18-2-2 · CAS 状态翻转(NOT_IN_USE ↔ IN_USE — AtomicInteger 原子操作)

ConcurrentBag 中每个 PoolEntry 维护 AtomicInteger state 字段。状态翻转通过 compareAndSet 原子操作完成,无需加锁。

图 18-2-3 · 连接归还流程(状态翻转 → handoffQueue 唤醒等待线程)

连接归还比获取更简单。调用 connection.close()(实际上是代理拦截)后,HikariCP 将连接状态翻转回 NOT_IN_USE,尝试唤醒等待中的线程。

图 18-2-4 · SynchronousQueue 手递手交接(零缓冲容量 0 — 直接传递引用)

第三级 handoffQueue 使用 SynchronousQueue(公平模式)——容量为 0 的阻塞队列。不存储任何元素,生产者与消费者直接配对交接。

表 18-2-2:为什么选择 SynchronousQueue

特性 含义
零缓冲 不长期存储元素,避免归还连接堆在中间队列
直接交接 归还者 offer 与借用者 poll 直接配对,延迟最小
公平模式 HikariCP 使用公平 SynchronousQueue,等待最久线程优先获得
避免饥饿 公平模式保证线程顺序,避免某线程长期排队

论文引用:Michael & Scott, Simple, Fast, and Practical Non-Blocking and Blocking Concurrent Queue Algorithms, PODC 1996,ConcurrentBag 的 handoffQueue 基于 SynchronousQueue,实现受此论文启发。Lea, D., A Java Fork / Join Framework, ACM Java Grande 2000,Work-Stealing 思想影响 ThreadLocal 亲和设计。

18.3 FastList vs ArrayList

HikariCP 没有使用 JDK 的 ArrayList,而是自定义 FastList。两个看似微小的优化,在每秒数万次调用场景下产生显著差异。

源码关键差异

public E get(int index) {
    rangeCheck(index);
    return elementData[index];
}
private void rangeCheck(int index) {
    if (index >= size)
        throw new IndexOutOfBoundsException(/* msg */);
}

public T get(int index) {
    return elementData[index];
}

public boolean remove(Object o) {
    for (int i = 0; i < size; i++)
        if (o.equals(elementData[i])) {
        }
}

public boolean removeLast(T element) {
    for (int i = size - 1; i >= 0; i--)
        if (element == elementData[i]) {
            final int numMoved = size - i - 1;
            if (numMoved > 0)
                System.arraycopy(elementData, i + 1, elementData, i, numMoved);
            elementData[--size] = null;
            return true;
        }
    return false;
}

FastList 使用 == 引用比较替代 equals()。HikariCP 内部 Statement 对象引用是唯一的,避免虚方法调用开销。

图 18-3-1 · LIFO 关闭顺序(后创建先关闭 — FastList 尾部命中)

JDBC 编程中,Statement 通常按 LIFO 顺序关闭(后创建的先关闭)。FastList 的 removeLast() 从尾部开始扫描,绝大多数情况下第一次迭代命中目标。加上去除 get() 的边界检查,每次调用节省约 10ns,高并发下累积效果显著。

典型 Web 请求中,getConnection() 与 close() 各调用一次,但每次数据库调用可能涉及 3 至 10 个 Statement。每个 Statement 的 close() 触发 FastList 的 removeLast()。按 1000 QPS 计算,每秒产生 3,000 至 10,000 次 remove 操作。表中教学数据估算,单次差值约 15.5ns,每秒可节省约 46 至 155μs CPU 时间。

18.4 连接生命周期状态机

HikariCP 中每个连接(PoolEntry)都有严格状态管理,通过 AtomicInteger 实现无锁状态转换。连接正在被业务线程使用时,maxLifetime 到期通常先打上 evicted 标记,等归还时再关闭,不强行中断正在执行的 SQL。

图 18-4-1 · PoolEntry 连接状态机(NOT_IN_USE / IN_USE / RESERVED / REMOVED — 4 态 + 软驱逐)

表 18-4-1:状态机转换规则

当前状态 触发事件 新状态
创建 addConnectionExecutor NOT_IN_USE
NOT_IN_USE borrow CAS 成功 IN_USE
IN_USE requite 归还 NOT_IN_USE
NOT_IN_USE softEvictConnection RESERVED
IN_USE MaxLifetimeTask 到期 IN_USE 标记 evicted
RESERVED closeConnection REMOVED
NOT_IN_USE idleTimeout 检查 REMOVED

HouseKeeper 与 MaxLifetimeTask

HikariCP 使用 ScheduledExecutorService 运行后台维护任务。HouseKeeper 默认每 30 秒执行一次,负责池状态检查、空闲连接回收、补足 minimumIdle。单条连接的 maxLifetime 由创建连接时安排的 MaxLifetimeTask 负责。

表 18-4-2:三个超时参数

参数 作用范围 计时起点
connectionTimeout 应用线程等待从池中借到连接的最大时间 borrow 调用
idleTimeout 连接被归还到池中后保持空闲的最大时间 归还入池
maxLifetime 连接从物理创建到软驱逐的最大生命周期 物理创建

三个参数不是连续时间线上三个阶段,是三个独立计时器,各自关注连接生命周期不同方面。

MySQL wait_timeout 经典问题

MySQL 默认 wait_timeout=28800(8 小时),超时后 MySQL 主动关闭连接。HikariCP 默认 maxLifetime 约 30 分钟,通常已小于 MySQL 默认值。风险主要出现在 maxLifetime 调得过大,或云数据库 / 代理层空闲超时比 MySQL 默认值短。

图 18-5-1 · 池大小决策树(CPU 密集 / I/O 密集 / 混合 → 单源 / 多源 → 容器 / 物理)

连接池大小黄金公式源自 PostgreSQL 团队经验:连接数 = (CPU 核心数 x 2) + 有效磁盘数。据 PostgreSQL Wiki Number Of Database Connections 与 Markus Winand 分析,特定基准测试中,600 连接的数据库当连接池从 50 降到 10 时,吞吐量反而提升约 10,000 倍。

表 18-5-2:为什么不是越多越好

原因 说明
数据库上下文切换激增 每个连接占用内核线程
CPU 缓存失效率上升 多线程频繁切换破坏 L1 / L2 缓存
磁盘 I/O 调度竞争 多并发请求同时争抢磁盘队列
锁竞争概率按 N 平方增长 行锁、表锁、间隙锁竞争

final class PoolSizeAdvisor {
    static int recommendedPoolSize(int spindleCount, int upperBound) {
        int effectiveCpu = Runtime.getRuntime().availableProcessors();
        int formula = (effectiveCpu * 2) + spindleCount;
        return Math.min(formula, upperBound);
    }

    public static void main(String[] args) {
        int size = recommendedPoolSize(0, 10);
        System.out.printf("effectiveCpu=%d, spindle=0, recommendedMaximumPoolSize=%d%n",
            Runtime.getRuntime().availableProcessors(), size);
    }

18.6 陷阱与限制

图 18-6-1 · 五大陷阱(连接数越多越慢 / maxLifetime / LB / 泄漏 / 多池隔离)

表 18-6-1:五大陷阱速查

序号 陷阱 现象 修复
1 连接数越多吞吐越低 池调大反而 P95 上升、DB CPU 飙高 按公式重新评估,从 8 至 10 起步
2 maxLifetime 大于 wait_timeout Communications link failure maxLifetime 小于最短外部超时
3 LB 空闲超时未考虑 长时间空闲后第一次请求失败 启用 keepaliveTime 保活
4 泄漏检测被忽略 active 长时间不回落,日志被淡化 区分真泄漏与慢查询
5 多数据源池隔离失败 一个池耗尽拖垮其他池 总连接预算拆分,独立监控

陷阱 1:连接数越多吞吐越低

很多团队遇到数据库慢查询,第一反应是加大连接池。PostgreSQL 团队实测发现:600 连接的数据库,将连接池从 50 降到 10,吞吐量反而大幅提升。原因:每个数据库连接占用内存与 CPU 调度资源,过多连接导致数据库内部锁竞争加剧。

公式:maximumPoolSize = (CPU核心数 x 2) + 有效磁盘数(PostgreSQL Wiki 推荐)。对 4 核 SSD 服务器,可从 8 至 10 起步压测。不要未经验证就设为 50 或 100。

陷阱 2:maxLifetime 大于 wait_timeout

MySQL 默认 wait_timeout=28800(8 小时),超时后 MySQL 主动关闭连接。HikariCP 不知道连接已被 MySQL 或代理层断开——下次从池中借到这个死连接时,执行 SQL 报 Communications link failure。

修复:maxLifetime 设为比数据库、代理或负载均衡器最短连接存活限制更短。Hikari 默认约 30 分钟通常小于 MySQL 默认 8 小时,但不一定小于云数据库代理、NAT 或 LB 超时。

陷阱 3:负载均衡器空闲超时

AWS ALB / NLB、阿里云 SLB、NAT Gateway、数据库代理等中间层可能有自己的空闲连接超时。如果连接长期空闲,中间层可能在 HikariCP 不知情情况下关闭 TCP 连接,造成与 wait_timeout 同样的死连接问题。

修复:优先让 maxLifetime 小于最短外部超时。固定池大小导致 idleTimeout 不生效时,可配置 keepaliveTime 做轻量保活。Kubernetes 中还要注意 Service / Ingress / NAT 的空闲超时配置。

陷阱 4:leakDetectionThreshold 误报

leakDetectionThreshold 触发告警时,日志中打印获取连接的堆栈。很多团队看到 apparent connection leak detected 后选择忽略,认为是慢查询导致的误报。真正的泄漏(忘记关闭 Connection)与慢查询表现一样——都是连接被长时间持有。

区分方法:active connections 持续增长且不回落 — 真泄漏(代码中有未关闭的 DataSource.getConnection());active 数在高峰后回落 — 慢查询(优化 SQL 或调大阈值)。使用 HikariPoolMXBean 监控 getActiveConnections() 趋势。

陷阱 5:多数据源池隔离

Spring Boot 自动配置默认只围绕一个主 DataSource 工作。配置多数据源时,每个数据源需要独立 HikariCP 配置,要统一计算数据库端总连接预算。常见错误是每个池都照抄同一个很大的 maximumPoolSize:单个慢数据源会耗尽自己的池,多个池叠加压垮数据库总连接数。

18.末 本章要点

  • HikariCP 快在少做事。默认不做连接测试查询(用 JDBC4 isValid())、推荐固定池大小减少伸缩抖动、尽量减少高频路径上的额外检查——每一项省掉的操作都是性能来源。

  • ConcurrentBag 的 ThreadLocal 亲和是关键。同一线程倾向于拿回上次用过的连接,避免跨线程竞争。三级降级策略(ThreadLocal、sharedList、SynchronousQueue)在不同负载下都能稳定借连接。

  • FastList 的尾部扫描适配 Statement LIFO 关闭顺序。结合 == 引用比较,降低虚方法调用开销。每秒数万次调用累积出可观节省。

  • 连接生命周期由状态机严格管理。PoolEntry.state 通过 AtomicInteger 与 CAS 实现无锁状态转换。maxLifetime 到期采用软驱逐:正在使用的连接标记 evicted,归还后才关闭,不打断业务 SQL。

  • 连接数不是越多越好。据 PostgreSQL Wiki:特定基准测试中,600 连接的数据库把池从 50 降到 10,吞吐量提升约 10,000 倍(实际因场景而异)。maximumPoolSize = (CPU x 2) + 磁盘数 是经过验证的起点。

  • maxLifetime 必须小于最短外部连接超时。数据库 wait_timeout、代理、NAT、负载均衡器都可能提前关闭空闲连接,必要时配合 keepaliveTime 保活。

  • 五大陷阱:连接数越多吞吐越低、maxLifetime 大于 wait_timeout、负载均衡器空闲超时未考虑、泄漏检测被忽略、多数据源池隔离失败。每个陷阱都有对应监控信号与修复方案。
