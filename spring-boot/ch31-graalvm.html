第 31 章 · GraalVM Native Image

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

传统 Spring Boot 应用启动 3 至 8 秒、占用 200 至 500MB 内存,在 Serverless 与容器化时代代价过高。GraalVM Native Image 用 AOT(Ahead-of-Time)编译把启动压到几十至几百毫秒,空闲内存降到几十 MB,代价是放弃 JIT 长期吞吐优势与反射的灵活性。

本章按"JIT 分层编译 → AOT 流水线 → 封闭世界假设 → Spring AOT 引擎 → 三大 OS 编译差异 → 性能对比 → 陷阱"七层展开,目标是让读者在评估 Spring Boot Native 时,既能用 RuntimeHints 与 Tracing Agent 精确补齐反射 / 资源 / 代理 / 序列化 hints,又能在 CI 中跑分层测试(JVM 单测验 hints + Native smoke test 验真二进制),并在 Serverless / FaaS 与长跑服务之间做出正确选型。

31.1 JIT 分层编译

HotSpot JVM 采用分层编译(Tiered Compilation),代码从解释执行逐步提升到高度优化机器码。共 5 层(Tier 0 至 4),由 C1(Client Compiler 快速编译 轻量优化)与 C2(Server Compiler 激进优化 编译慢但码质量高)协作。阈值由 -XX:CompileThreshold 与自适应 profiling 决定,随负载动态调整。

图 31-1-1 · JIT 分层编译流水线(Tier 0 → 1-3 → 4 + Deoptimization 回退)

31.1.1 去优化(Deoptimization)

C2 基于 profile 做激进假设(如"虚方法只一个实现")。当假设被打破(加载新子类),JVM 必须丢弃优化后的机器码,回退到解释执行,重新收集 profile 并编译。这就是 JVM 需要"预热"才能达到峰值性能的根因。

31.1.2 JMH 基准:Jackson 序列化预热

长跑 Spring Boot 服务中,C2 对热点方法(如 Jackson 序列化)深度优化,约 10000 次调用预热后吞吐可提升 5 至 10 倍。

@BenchmarkMode(Mode.Throughput)
@Warmup(iterations = 3, time = 1)
@Measurement(iterations = 5, time = 1)
@Fork(1)
public class JacksonBenchmark {

    private final ObjectMapper mapper = new ObjectMapper();
    private final OrderDto dto = new OrderDto("Customer C1001 Wang", 28, "shanghai");

    @Benchmark
    public byte[] serializeToJson() throws Exception {
        return mapper.writeValueAsBytes(dto);
    }
}

31.2 AOT 编译流程

GraalVM native-image 在构建时把 Java 应用编译为平台原生可执行文件,消除运行期 JIT 编译与类加载开销。

31.2.1 直接编译

native-image \
  -jar target/demo-0.0.1-SNAPSHOT.jar \
  -H:Name=demo \
  --no-fallback

--report-unsupported-elements-at-runtime 把部分构建期错误推迟到运行期,教学排障可临时使用,生产构建不建议默认开启。

31.3 封闭世界假设

Native Image 核心前提是封闭世界假设(Closed-World Assumption):构建时必须知道所有类、方法、字段。任何运行期动态行为必须在构建时声明。

图 31-3-1 · Reachability Metadata 文件体系(6 类配置 + Tracing Agent 自动收集)

31.3.1 Tracing Agent

java \
  -agentlib:native-image-agent=config-output-dir=src/main/resources/META-INF/native-image \
  -jar target/demo-0.0.1-SNAPSHOT.jar

跑覆盖关键路径的测试流量后,Agent 自动生成 metadata。但只能覆盖测试运行时实际执行到的路径,未覆盖路径仍需手动配置:

31.3.2 设计取舍

表 31-3-1:Native Image 核心取舍

取舍点 Native 选择 被拒绝 / 受限 工程后果
类发现 构建期闭包可达性 运行期任意 classpath scanning 新增反射路径必须补 hints 或 agent 覆盖
Bean 注册 Spring AOT 生成代码 启动时动态解析 BeanDefinition 启动快 构建产物与源码关系更重要
类初始化 可 build-time init 所有 static initializer 都在运行期 时机错误会把时间 / 随机数 / 环境变量固化进二进制
代理 预声明 JDK proxy / AOT 代理类 运行期 CGLIB / Byte Buddy 动态框架需要 AOT 集成或替代方案
性能优化 构建期静态优化 C2 运行期 profile 投机优化 冷启动强 长跑吞吐未必赢
部署 平台原生二进制 一次构建跨平台运行 必须按 OS / CPU 矩阵构建

Native Image 的正确定位不是"更快的 JVM",而是"更受约束的部署形态"。严重依赖运行期插件 / 脚本 / 动态类加载 / Java Agent / 复杂字节码生成的应用,JIT 模式通常更稳。

31.4 Spring AOT 引擎

Spring Boot 3.x 通过 AOT 引擎在构建时预计算 Bean 定义、处理条件注解、生成反射 metadata,大幅减少手动配置。这层是 GraalVM 之前的"Spring 元模型预计算"。

表 31-4-1:Spring AOT 关键源码入口

入口 角色 构建期职责
spring-boot-maven-plugin spring-boot-gradle-plugin process-aot goal / task 启动 AOT 处理流程
SpringApplicationAotProcessor AOT 主入口 运行 ApplicationContext 但不启 Web server 收集 BeanFactory 状态
ApplicationContextAotGenerator 上下文生成器 遍历 BeanDefinition 生成 ApplicationContextInitializer 源码
BeanFactoryInitializationAotProcessor BeanFactory 级处理器 框架在 BeanFactory 初始化阶段贡献代码与 hints
BeanRegistrationAotProcessor 单 Bean 级处理器 为特定 Bean 生成实例化 / 注入 / 生命周期回调代码
RuntimeHints RuntimeHintsRegistrar hints API 声明反射 / 资源 / 代理 / 序列化 / JNI 运行期需求
GeneratedFiles 输出管理 写 target/spring-aot/main/sources 与 META-INF/native-image
图 31-4-1 · Spring AOT → Native Image 全链路(process-aot 生成代码 + RuntimeHints 提示)

运行期最大变化:Spring 不再用反射 + classpath scanning 发现 Bean,而是执行构建期生成的 initializer。Native Image 的快不是只靠 GraalVM 编译器,Spring AOT 把容器启动从"解释元数据"改成"执行生成代码"也贡献了大半。

31.4.2 RuntimeHints API

@Configuration
@ImportRuntimeHints(MyRuntimeHints.class)
class NativeHintsConfiguration { }

class MyRuntimeHints implements RuntimeHintsRegistrar {

    @Override
    public void registerHints(RuntimeHints hints, ClassLoader classLoader) {
        hints.reflection().registerType(MyEntity.class,
            MemberCategory.DECLARED_FIELDS,
            MemberCategory.INVOKE_DECLARED_CONSTRUCTORS);
        hints.resources().registerPattern("templates/*.html");
        hints.proxies().registerJdkProxy(MyService.class);
    }
}

生成的 resource-config.json proxy-config.json 与代码逐项对照。RuntimeHintsPredicates 可在单测中防止重构误删:

@Test
void registersOrderDtoForJacksonBinding() {
    RuntimeHints hints = new RuntimeHints();
    new MyRuntimeHints().registerHints(hints, getClass().getClassLoader());
    assertThat(RuntimeHintsPredicates.reflection()
        .onType(OrderDTO.class)
        .withMemberCategory(MemberCategory.INVOKE_DECLARED_CONSTRUCTORS))
        .accepts(hints);
    assertThat(RuntimeHintsPredicates.resource()
        .forResource("templates/order.html"))
        .accepts(hints);
}

31.4.3 @RegisterReflectionForBinding

适合声明会被 Jackson / Web Binder / Data Binder 反射绑定的 DTO,避免手写 reflect-config.json。Hibernate 管理的 JPA 实体通常由 Spring Boot / Hibernate 的 AOT 集成生成必要 hints,无需手动注册。

public record OrderDTO(Long id, String orderNo, CustomerDTO customer) { }
public record CustomerDTO(Long id, String name) { }

@Configuration
@RegisterReflectionForBinding({
    OrderDTO.class,
    CustomerDTO.class
})
public class NativeConfig { }

等效 reflect-config.json 自动生成:

[
  {
    "name": "com.example.order.OrderDTO",
    "allDeclaredFields": true,
    "allDeclaredMethods": true,
    "allDeclaredConstructors": true
  },
  {
    "name": "com.example.order.CustomerDTO",
    "allDeclaredFields": true,
    "allDeclaredConstructors": true
  }
]

反射缺失时,通常不是构建失败,而是运行期第一次走到该路径时:

ERROR ... org.graalvm.nativeimage.MissingReflectionRegistrationError
The program tried to reflectively invoke method
  com.example.order.OrderDTO.<init>()
without it being registered for runtime reflection.
Action:
- add @RegisterReflectionForBinding(OrderDTO.class), or
- register the type in a RuntimeHintsRegistrar, or
- regenerate reflect-config.json with tracing coverage for this endpoint.

31.4.4 扩展边界

表 31-4-2:Hints 类型适用边界

类型 API 适合 不适合
Reflection hints.reflection().registerType Jackson DTO 配置绑定 框架回调构造器 全包 allDeclaredMethods 粗放开
Resources hints.resources().registerPattern templates/*.html db/migration/*.sql 证书 / schema 运行期用户上传文件
Proxies hints.proxies().registerJdkProxy JDK 动态代理接口组合 CGLIB 子类代理

31.7 陷阱与限制

图 31-7-1 · GraalVM Native 四大陷阱速查(metadata 不全 / 编译慢 / 吞吐低 / 库不兼容)

31.7.1 陷阱 1:反射 metadata 不完整(致命)

最常见失败:应用编译成功,运行期 NoClassDefFoundError / MissingReflectionRegistrationError。Points-to Analysis 无法追踪反射 / Class.forName / 配置文件动态加载的类。修复:① 跑 Tracing Agent 自动收集;② Spring 用 @RegisterReflectionForBinding 或 RuntimeHintsRegistrar;③ 测试覆盖所有代码路径(Agent 只能记录实际执行到的反射)。

31.7.2 陷阱 2:编译时间过长(高)

Native Image 编译 3 至 10 分钟(取决应用规模与机器),需 8 至 16GB 内存。"改代码 → 编译 → 测试"循环极慢。应对:日常 JIT 模式(mvn spring-boot:run),仅 CI / CD 构建 Native;-Pnative Profile 隔离避免每次 build 触发 native 编译。

31.7.3 陷阱 3:运行时吞吐低于 JIT(中)

Native 启动快 / 内存低,但长跑吞吐可能低于 JIT 30 至 50%。原因:无运行时 profiling,无法投机优化(profile-guided devirtualization)。适用 Serverless / CLI / 短生命周期;7×24 长跑服务 JIT 通常更好。不要无脑追求 Native。

31.7.4 陷阱 4:第三方库不兼容(中)

许多库在 Native 中无法工作或需额外 hints:Mockito 这类测试期字节码生成工具不应进生产镜像;Logback Groovy 配置依赖运行时动态加载;复杂 CGLIB / Byte Buddy 代理可能受限。升级 Native 前必须验证所有依赖的兼容性。

表 31-7-1:陷阱速查
