第 8 章 · Resource 资源抽象

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

Java 标准库的 java.net.URL 类仅支持 HTTP/FTP 等少数标准协议,对类路径资源、ServletContext 资源、相对文件路径的支持要么缺失,要么使用笨拙。Spring 通过 Resource 接口统一了所有资源访问方式——无论资源位于类路径、文件系统、远程 URL 还是 JAR 包内部,均可通过同一套 API 读取。该抽象的核心价值不仅在于 API 统一,更在于将配置加载、静态资源服务、模板加载、Range 响应等多种场景接入同一条可观察的链路。

本章按"问题 → 机制 → 实现 → 边界/陷阱"组织。首节剖析 Resource 接口体系与六大实现类的语义差异;第二节深入 ResourceLoader 与 ResourcePatternResolver 的分层设计与前缀解析机制;第三节讲解 PathMatchingResourcePatternResolver 的通配符扫描算法;第四节梳理 Spring Boot 静态资源处理的完整链路;第五节解析 ResourceRegion 与 HTTP Range 协议的实现;第六节给出源码入口表与扩展边界;末节列举五类典型陷阱与修复方案。

阅读本章前,建议先掌握第 3 章的 ApplicationContext 接口体系——ApplicationContext 本身实现了 ResourcePatternResolver,这意味着任何 Spring 上下文都具备资源加载能力。

8.1 Resource 接口体系与实现类

8.1.1 Resource 接口的设计原则

Resource 接口位于 org.springframework.core.io 包,继承自 InputStreamSource。其设计遵循三个原则:其一,只定义读取语义,写入由子接口 WritableResource 提供;其二,提供统一的元数据查询接口(存在性、可读性、长度、修改时间);其三,通过 getInputStream() 作为最低公约数,所有实现都必须支持流式读取,而 getFile() 仅在文件系统场景下可用。

该设计的核心权衡是"通用性优先于性能"。InputStream 是所有资源类型的最低公约数——文件、JAR 内条目、HTTP 响应、内存字节都可暴露为 InputStream。getFile() 仅作为优化路径存在,调用方需通过 isFile() 检查。

8.1.3 六大核心实现类的对比

下表汇总六大实现类的关键属性。getFile() 支持列标记为"仅文件系统时"的实现,在 fat JAR 部署场景下会抛出 FileNotFoundException,这是生产环境最常见的资源加载故障。

实现类 资源来源 getFile() 支持 可写 典型用途
ClassPathResource 类路径(ClassLoader 或 Class 相对) 仅文件系统时,JAR 内抛异常 否 配置文件、模板、SQL 脚本
FileSystemResource 文件系统绝对/相对路径 始终支持 是 日志文件、用户上传、外部配置

ServletContextResource ServletContext 相对路径 取决于容器实现 否 WAR 包内 Web 资源
ByteArrayResource 内存字节数组 不支持 否 单元测试 mock、动态生成内容
InputStreamResource 已有 InputStream 的包装 不支持 否 一次性流(网络响应、管道)

8.1.4 InputStreamResource 的一次性语义

InputStreamResource 是六大实现中唯一具有一次性语义的实现。其 isOpen() 返回 true,表示底层流已打开且只能读取一次。多次调用 getInputStream() 会抛出 IllegalStateException,因为 InputStream 一旦读尽即耗尽。

Resource cp = new ClassPathResource("application.yml");
InputStream is1 = cp.getInputStream();   // 始终可用
InputStream is2 = cp.getInputStream();   // 可重复创建新流

Resource fs = new FileSystemResource("/var/log/app/access.log");
File file = fs.getFile();
OutputStream os = ((WritableResource) fs).getOutputStream();

Resource remote = new UrlResource("https://config.example.com/app.yml");
InputStream is = remote.getInputStream();

byte[] data = "{\"key\":\"value\"}".getBytes(StandardCharsets.UTF_8);
Resource mock = new ByteArrayResource(data, "test-config.json");
mock.getInputStream();  // 第一次
mock.getInputStream();  // 第二次,ByteArrayResource 可重复读取

需要多次读取场景下,应使用 ByteArrayResource 将流内容缓冲到字节数组,避免一次性资源耗尽问题。

8.2 ResourceLoader 与前缀解析机制

8.2.1 ResourceLoader 与 ResourcePatternResolver 的分层

Spring 资源加载体系由两个接口构成:ResourceLoader 提供单资源加载能力,通过 getResource(String location) 返回单个 Resource;ResourcePatternResolver 继承自 ResourceLoader,扩展了批量扫描能力,通过 getResources(String pattern) 返回 Resource[]。ApplicationContext 实现了 ResourcePatternResolver,因此具备完整的资源加载与扫描能力。

图 8-2 · ResourceLoader 接口层次与 ApplicationContext 的关系

8.2.2 前缀分发机制

Spring 通过资源路径的前缀决定使用哪种 Resource 实现。该分发机制的核心实现在 DefaultResourceLoader.getResource() 方法中,按 ProtocolResolver → / 开头 → classpath: → URL → 默认路径的顺序判断。前缀的语义差异如下表所列:

前缀 示例 返回类型 说明
classpath: classpath:db/schema.sql ClassPathResource 从当前 ClassLoader 加载,只返回第一个匹配
classpath*: classpath*:META-INF/spring.factories Resource[] 仅用于 getResources(),扫描所有 ClassLoader 与 JAR
file: file:/etc/app/config.yml FileSystemResource 绝对文件系统路径
http:/https: https://cdn.example.com/data.json UrlResource 远程 HTTP 资源
(无前缀) data/config.yml 取决于 ApplicationContext 类型 由上下文决定默认行为

8.2.4 DefaultResourceLoader 的解析算法

DefaultResourceLoader.getResource() 是所有 ApplicationContext 的基础资源加载实现。其源码逻辑按下列顺序判断资源类型:

public Resource getResource(String location) {
    Assert.notNull(location, "Location must not be null");

    for (ProtocolResolver resolver : this.protocolResolvers) {
        Resource resource = resolver.resolve(location, this);
        if (resource != null) return resource;
    }

    if (location.startsWith("/")) {
        return getResourceByPath(location);
    }

    if (location.startsWith(CLASSPATH_URL_PREFIX)) {
        return new ClassPathResource(
            location.substring(CLASSPATH_URL_PREFIX.length()),
            getClassLoader());
    }

    try {
        URL url = ResourceUtils.toURL(location);
        return (ResourceUtils.isFileURL(url)
            ? new FileUrlResource(url)
            : new UrlResource(url));
    } catch (MalformedURLException ex) {
        return getResourceByPath(location);
    }
}

getResourceByPath() 是模板方法,由具体 ApplicationContext 子类覆写:ClassPathXmlApplicationContext 返回 ClassPathContextResource,FileSystemXmlApplicationContext 返回 FileSystemContextResource,WebApplicationContext 返回 ServletContextResource。

8.2.5 ProtocolResolver 扩展点

ProtocolResolver 是 Spring 4.3 引入的 SPI,允许注册自定义协议前缀。例如注册 s3:// 前缀的解析器后,resourceLoader.getResource("s3://bucket/key") 返回从 AWS S3 读取的 Resource。Spring Cloud AWS 通过该机制实现了 S3 资源透明访问。

public class S3ProtocolResolver implements ProtocolResolver {
    private final AmazonS3 s3Client;

    @Override
    public Resource resolve(String location, ResourceLoader resourceLoader) {
        if (location.startsWith("s3://")) {
            String path = location.substring("s3://".length());
            String bucket = path.substring(0, path.indexOf('/'));
            String key = path.substring(path.indexOf('/') + 1);
            return new S3Resource(s3Client, bucket, key);
        }
        return null;
    }
}

@Configuration
public class S3Config {
    @Bean
    ApplicationRunner registerS3Protocol(
            ConfigurableApplicationContext ctx, AmazonS3 s3) {
        return args -> ctx.addProtocolResolver(new S3ProtocolResolver(s3));
    }
}

注册后,ResourceLoader 优先尝试自定义协议解析器;命中 s3:// 时不进入默认的 URL/classpath 分支。ProtocolResolver 必须在不能处理时返回 null,让默认链路继续工作——这是扩展点的关键约定。

8.2.6 Spring Boot 配置文件加载

Spring Boot 启动时,ConfigDataEnvironmentPostProcessor 使用 ResourceLoader 按预定义位置搜索配置文件。下列位置按"低优先级 → 高优先级"排列,后加载的位置覆盖前面同名 key:

优先级 位置 说明
1 classpath:/ 类路径根目录
2 classpath:/config/ 类路径下的 config/ 目录
3 file:./ 项目根目录
4 file:./config/ 项目根目录的 config/ 子目录
5 file:./config/*/ config 下的任意子目录(2.4+ 起)

该搜索顺序解释了为何外部化的 config/application.yml 能覆盖 JAR 内的默认配置——文件系统路径优先级高于类路径。底层每次搜索都是一个 ResourceLoader.getResource() 调用。

8.3 PathMatchingResourcePatternResolver 通配符扫描

8.3.1 通配符扫描的复杂性

PathMatchingResourcePatternResolver 是 ResourcePatternResolver 的核心实现,支持 Ant 风格通配符与 classpath*: 前缀。其扫描算法的复杂性远超直觉——需要处理文件系统目录、JAR 包内资源、嵌套 JAR(Spring Boot fat JAR)、多 ClassLoader 等多种场景。

扫描流程的核心步骤是:解析路径模式提取根路径与通配符部分;通过 ClassLoader.getResources() 收集所有匹配根路径的 URL;遍历每个 URL,根据协议(file 或 jar)使用不同策略扫描;最后用 AntPathMatcher 过滤通配符部分。

8.3.3 Ant 风格路径匹配规则

Spring 使用 AntPathMatcher 实现路径模式匹配,语法与 Apache Ant 构建工具一致。下表列出四种核心模式:

模式 含义 匹配示例 不匹配示例
? 匹配一个字符 config?.yml 匹配 config1.yml config12.yml
* 匹配一层路径内零或多个字符 *.xml 匹配 beans.xml sub/beans.xml
** 匹配零或多层路径 com/**/Mapper.xml 匹配 com/a/b/Mapper.xml (无)
{name} 路径变量(URI 模板) /api/orders/{id} 匹配 /api/orders/1001 (无)

    "classpath*:META-INF/spring/*.factories");

8.3.4 完整资源解析管线

图 8-5 · 从 @Value 注解到实际资源读取的完整管线(注入是惰性,I/O 在 getInputStream)

8.3.5 @Value 注入资源

Spring 支持通过 @Value 注解将资源路径直接解析为 Resource 对象注入。这是加载配置文件、模板、静态数据的最简洁方式。注入时不会读取内容,真正的 I/O 发生在调用 getInputStream() 时。

@Service
public class DataImportService {
    @Value("classpath:data/initial-data.json")
    private Resource initialData;

    @Value("${import.template-path:classpath:templates/default.xlsx}")
    private Resource templateFile;

    @Value("classpath*:plugins/**/*.json")
    private Resource[] pluginConfigs;

    public void importData() throws IOException {
        try (InputStream is = initialData.getInputStream()) {
            List<DataRecord> records = objectMapper.readValue(
                is, new TypeReference<List<DataRecord>>() {});
            repository.saveAll(records);
        }
    }

    public void loadAllPlugins() throws IOException {
        for (Resource config : pluginConfigs) {
            try (InputStream is = config.getInputStream()) {
                PluginDescriptor desc = objectMapper.readValue(is, PluginDescriptor.class);
                pluginRegistry.register(desc);
            }
        }
    }
}

8.3.6 ResourceLoaderAware 与构造器注入

需要在运行时动态加载资源(路径在编译期未知)的 Bean,可实现 ResourceLoaderAware 接口或直接构造器注入 ResourceLoader。后者更符合依赖最小化原则,推荐使用。

@Component
public class TemplateEngine {
    private final ResourceLoader resourceLoader;

    public TemplateEngine(ResourceLoader resourceLoader) {
        this.resourceLoader = resourceLoader;
    }

    public String render(String templateName, Map<String, Object> model) {
        Resource template = resourceLoader.getResource(
            "classpath:templates/" + templateName + ".html");
        return doRender(template, model);
    }
}

由于 ApplicationContext 继承了 ResourcePatternResolver,直接注入 ApplicationContext 也能获得资源加载能力。但按依赖最小化原则,如果只需加载资源应声明 ResourceLoader 类型而非 ApplicationContext。

8.3.7 MyBatis Mapper XML 加载范例

MyBatis 与 Spring 集成时,Mapper XML 文件的加载是 classpath*: 通配符的经典使用场景。MybatisAutoConfiguration 从配置属性 mybatis.mapper-locations 读取路径模式,使用 PathMatchingResourcePatternResolver 扫描所有匹配的 XML 文件。

mybatis:
  mapper-locations: classpath*:mappers/**/*.xml

@Bean
public SqlSessionFactory sqlSessionFactory(DataSource dataSource) throws Exception {
    SqlSessionFactoryBean factory = new SqlSessionFactoryBean();
    factory.setDataSource(dataSource);

    ResourcePatternResolver resolver = new PathMatchingResourcePatternResolver();
    Resource[] mapperResources = resolver.getResources("classpath*:mappers/**/*.xml");

    factory.setMapperLocations(mapperResources);
    return factory.getObject();
}

若使用 classpath: (无星号),只会从主 ClassLoader 返回第一个匹配——多模块项目中其他 JAR 包的 Mapper XML 会被遗漏,导致运行时 BindingException: Invalid bound statement。

8.4.2 ResourceHandlerRegistry 自定义映射

当默认配置不满足需求时,通过实现 WebMvcConfigurer.addResourceHandlers() 注册自定义资源映射。该机制支持多 location、缓存控制、资源链(版本化、转换)等高级特性。

@Configuration
public class WebConfig implements WebMvcConfigurer {
    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        registry.addResourceHandler("/uploads/**")
                .addResourceLocations("file:/var/app/uploads/")
                .setCacheControl(CacheControl.maxAge(30, TimeUnit.DAYS));

        registry.addResourceHandler("/docs/**")
                .addResourceLocations("classpath:/documentation/")
                .setCacheControl(CacheControl.noCache());

        registry.addResourceHandler("/assets/**")
                .addResourceLocations(
                    "classpath:/custom-assets/",
                    "classpath:/default-assets/")
                .setCacheControl(CacheControl.maxAge(365, TimeUnit.DAYS)
                    .cachePublic()
                    .immutable())
                .resourceChain(true)
                .addResolver(new VersionResourceResolver()
                    .addContentVersionStrategy("/**"));
    }
}

三个 handler 的可观察行为分别是:上传目录带 30 天缓存;文档目录每次需重新校验;资产目录经过资源链生成内容哈希 URL 并配合 immutable 缓存。

8.4.3 HTTP 缓存控制策略

正确的缓存策略对性能至关重要。Spring Boot 通过 CacheControl 构建器与 HTTP 响应头(Cache-Control、ETag、Last-Modified)实现精细的缓存控制。

图 8-7 · HTTP 缓存决策流程(max-age / no-cache / no-store + 条件请求)

CacheControl 构建器提供了流式 API 构造各种缓存策略:

CacheControl immutable = CacheControl.maxAge(365, TimeUnit.DAYS)
    .cachePublic()
    .immutable();

CacheControl shortLived = CacheControl.maxAge(5, TimeUnit.MINUTES)
    .mustRevalidate();

CacheControl noCache = CacheControl.noCache();
CacheControl noStore = CacheControl.noStore();

8.4.4 资源版本化策略

资源版本化通过在 URL 中嵌入版本标识实现缓存破坏(cache busting)。当资源内容变化时,URL 随之变化,浏览器被迫获取新版本。Spring 提供两种策略:ContentVersionStrategy 基于内容 MD5 哈希生成版本号;FixedVersionStrategy 使用固定版本号(通常来自构建号或 Git commit)。

策略 版本来源 优势 劣势
ContentVersionStrategy 内容 MD5 哈希 内容不变 URL 不变,精确失效 启动时计算所有哈希,有开销
FixedVersionStrategy 构建号 / Git commit 简单,无运行时计算 任何资源变化全部缓存失效 
registry.addResourceHandler("/assets/**")
    .addResourceLocations("classpath:/static/")
    .resourceChain(true)
    .addResolver(new VersionResourceResolver()
        .addContentVersionStrategy("/**"));

模板渲染需配合 ResourceUrlProvider 自动转换 URL。Thymeleaf 的 @{} 语法默认触发该转换:

<link rel="stylesheet" th:href="@{/assets/style.css}">
<!-- 渲染结果: /assets/style-a1b2c3d4e5.css -->
<script th:src="@{/assets/app.js}"></script>
<!-- 渲染结果: /assets/app-f6e7d8c9b0.js -->

8.4.5 WebJars 集成

WebJars 将前端库(jQuery、Bootstrap、Vue.js 等)打包为标准 Maven/Gradle 依赖,通过 META-INF/resources/webjars/ 提供服务。Spring Boot 自动配置了 /webjars/** 的资源映射,无需额外配置。

<dependency>
    <groupId>org.webjars</groupId>
    <artifactId>bootstrap</artifactId>
    <version>5.3.0</version>
</dependency>
<dependency>
    <groupId>org.webjars</groupId>
    <artifactId>webjars-locator-core</artifactId>
</dependency>

<link rel="stylesheet" href="/webjars/bootstrap/css/bootstrap.min.css">
<script src="/webjars/bootstrap/js/bootstrap.bundle.min.js"></script>

webjars-locator-core 自动处理版本号,使 HTML 中无需硬编码版本字符串。物理路径解析为 classpath:/META-INF/resources/webjars/bootstrap/5.3.0/css/bootstrap.min.css。

8.5 ResourceRegion 与 HTTP Range 协议

8.5.1 大文件传输的需求与 Range 协议

当客户端请求大文件(视频、大型文档)时,一次性传输整个文件浪费带宽且降低用户体验。HTTP/1.1 的 Range 协议允许客户端只请求资源的一部分。Spring 通过 ResourceRegion 与 ResourceHttpRequestHandler 提供完整的 Range 请求支持。

ResourceRegion 表示一个 Resource 的连续区间,包含底层资源、起始位置、读取字节数三个属性。其设计使得视频流式传输、断点续传等场景可统一处理。

8.5.2 ResourceRegion 的核心结构

public class ResourceRegion {
    private final Resource resource;
    private final long position;
    private final long count;

    public ResourceRegion(Resource resource, long position, long count) {
        Assert.notNull(resource, "Resource must not be null");
        Assert.isTrue(position >= 0, "'position' must be >= 0");
        Assert.isTrue(count >= 0, "'count' must be >= 0");
        this.resource = resource;
        this.position = position;
        this.count = count;
    }
}

手动创建 ResourceRegion 可实现自定义 Range 响应处理:

@GetMapping("/videos/{filename}")
public ResponseEntity<?> streamVideo(
        @PathVariable String filename,
        @RequestHeader HttpHeaders headers) throws IOException {
    Resource video = new ClassPathResource("videos/" + filename);
    long contentLength = video.contentLength();
    List<HttpRange> ranges = headers.getRange();

    if (ranges.isEmpty()) {
        return ResponseEntity.ok()
            .contentType(MediaType.parseMediaType("video/mp4"))
            .contentLength(contentLength)
            .body(video);
    }

    HttpRange range = ranges.get(0);
    long start = range.getRangeStart(contentLength);
    long end = range.getRangeEnd(contentLength);
    long rangeLength = Math.min(end - start + 1, contentLength - start);

    ResourceRegion region = new ResourceRegion(video, start, rangeLength);
    return ResponseEntity.status(HttpStatus.PARTIAL_CONTENT)
        .contentType(MediaType.parseMediaType("video/mp4"))
        .header(HttpHeaders.CONTENT_RANGE,
            "bytes " + start + "-" + (start + rangeLength - 1) + "/" + contentLength)
        .body(region);
}

8.6.1 关键源码入口表

8.6.3 设计取舍与被拒绝方案

Spring 没有将所有资源都强行抽象为 File,这是最重要的设计取舍。Resource 暴露 getFile() 是为了兼容文件系统场景,但主能力始终是 InputStreamSource。一旦部署到 fat JAR、远程 URL 或内存资源,File 语义即不成立。

设计选择 设计动机 被拒绝方案 扩展边界
前缀分发而非注册表查找 字符串配置简单,适合 XML、注解、YAML 每个资源类型写 Java Bean 新协议用 ProtocolResolver
Resource 优先流式读取 兼容 JAR、URL、内存与 ServletContext 强制所有资源落地为 File 第三方只收 File 时复制到临时文件
classpath*: 仅在 resolver 层支持 单资源与批量扫描职责分离 getResource() 返回数组 批量扫描必须注入 ResourcePatternResolver
静态资源链用 resolver/transformer 版本化、压缩、WebJars 可组合 Handler 内写死规则 自定义 ResourceResolver 保留默认顺序
HTTP 缓存遵循协议头 让浏览器/CDN 参与缓存 每次请求都返回 200 哈希 URL + 长缓存必须成对出现

扩展时有两条红线:其一,不要在库代码中调用 resource.getFile() 作为通用读取方式;其二,自定义 ResourceResolver / ProtocolResolver 必须在不能处理时返回 null,让默认链路继续工作。

8.6.4 验证护栏与测试策略

资源问题最适合用小探针与 HTTP 断言做守护。下列三组测试覆盖 fat JAR 安全读取、通配符扫描、静态资源缓存三条高风险路径。

@Test
void classpathResourceIsReadByStreamNotFile() throws Exception {
    Resource resource = new ClassPathResource("data/initial-data.json");
    assertThat(resource.exists()).isTrue();
    assertThatCode(() -> StreamUtils.copyToByteArray(resource.getInputStream()))
        .doesNotThrowAnyException();
}

@Test
void mapperPatternFindsResourcesFromAllClasspathRoots() throws Exception {
    ResourcePatternResolver resolver = new PathMatchingResourcePatternResolver();
    Resource[] resources = resolver.getResources("classpath*:mappers/**/*.xml");
    assertThat(resources)
        .extracting(Resource::getFilename)
        .contains("UserMapper.xml", "OrderMapper.xml");
}

@Test
void staticResourceSupportsConditionalRequest() throws Exception {
    MvcResult first = mockMvc.perform(get("/assets/app.js"))
        .andExpect(status().isOk())
        .andExpect(header().exists(HttpHeaders.ETAG))
        .andReturn();

    mockMvc.perform(get("/assets/app.js")
            .header(HttpHeaders.IF_NONE_MATCH,
                first.getResponse().getHeader(HttpHeaders.ETAG)))
        .andExpect(status().isNotModified());
}

建议在开发环境打开 org.springframework.core.io 与 org.springframework.web.servlet.resource 的 DEBUG 日志。健康的静态资源请求应能观测到"请求路径 → 资源位置 → 缓存/版本化决策 → 响应状态"的完整链路。

8.7 典型陷阱与边界

8.7.1 陷阱一:classpath: 与 classpath*: 的语义差异

这是 Spring 资源加载中最常见的困惑点。classpath: 仅从当前 ClassLoader 返回第一个匹配,而 classpath*: 扫描所有 ClassLoader 与 JAR 返回全部匹配。两者行为差异巨大,前者用于通配符场景时同样只返回单一路径下的匹配。

Resource r1 = resourceLoader.getResource("classpath:META-INF/spring.factories");
Resource[] rs1 = resolver.getResources("classpath:mappers/**/*.xml");

Resource[] rs2 = resolver.getResources("classpath*:META-INF/spring.factories");
Resource[] rs3 = resolver.getResources("classpath*:mappers/**/*.xml");

多模块项目中,若模块 A 与模块 B 都有 mappers/ 目录,使用 classpath: 的 MyBatis 配置只会加载其中一个模块的 Mapper XML,另一个模块的 Mapper 在运行时抛出 BindingException: Invalid bound statement。修复方式:始终使用 classpath*: 确保扫描所有源。

8.7.2 陷阱二:FileSystemResource 相对路径基于 JVM 工作目录

FileSystemResource 使用 java.nio.file.Path 解析路径。当路径为相对路径时,它相对于 JVM 的工作目录(user.dir 系统属性),而非项目根目录或应用入口类所在目录。同一个相对路径在不同环境(IDE、生产、Docker)下解析结果不同,导致难以预测的资源加载错误。

环境 相对路径 data/config.yml 解析结果
IDE 中 /home/dev/project/data/config.yml
生产部署 /opt/app/data/config.yml
Docker 容器 /data/config.yml

修复方案有两种:其一,使用绝对路径并通过配置注入;其二,使用 classpath: 加载,实现环境无关性。

@Value("${app.config.path}")
private String configPath;
Resource cfg = new FileSystemResource(configPath);

Resource template = new ClassPathResource("data/config.yml");

8.7.3 陷阱三:fat JAR 中 Resource.getFile() 抛异常

这是从开发到生产部署时最常见的资源加载故障。在 IDE 中运行时,类路径资源位于文件系统的 target/classes/ 目录,getFile() 正常工作。但打包为 fat JAR(java -jar app.jar)后,资源嵌套在 JAR 文件内部——JAR 中的条目不是文件系统上的独立文件,getFile() 抛出 FileNotFoundException。

图 8-11 · IDE 与 fat JAR 部署的资源路径差异(getFile() 在 JAR 中失效)

修复方案是始终使用 getInputStream() 替代 getFile():

@Service
public class ReportService {
    @Value("classpath:templates/report.html")
    private Resource template;

    public String generateReport() throws IOException {
        try (InputStream is = template.getInputStream()) {
            return new String(is.readAllBytes(), StandardCharsets.UTF_8);
        }
    }
}

若不确定某个 Resource 是否支持 getFile(),先检查 resource.isFile()(Spring 5.0 引入)。该方法返回 true 时表示 getFile() 可安全调用。

8.7.4 陷阱四:必须使用 File 时的临时文件方案

某些第三方库只接受 File 参数(如某些 PDF 处理库、Excel 库)。在 fat JAR 部署场景下,需将 InputStream 复制到临时文件,再传递给第三方 API。

public File toTempFile(Resource resource) throws IOException {
    File temp = File.createTempFile("spring-resource-", ".tmp");
    temp.deleteOnExit();
    try (InputStream is = resource.getInputStream();
         OutputStream os = new FileOutputStream(temp)) {
        is.transferTo(os);
    }
    return temp;
}

File.createTempFile() 在 JVM 临时目录创建新文件,deleteOnExit() 注册 JVM 退出时自动清理。该模式是兼容 fat JAR 与第三方 File API 的标准做法。

8.7.5 陷阱五:静态资源缺少版本哈希导致缓存失效

若静态资源 URL 不包含内容版本标识,浏览器根据 Cache-Control 头缓存资源。部署新版本应用时,资源 URL 不变(/static/app.js),浏览器继续使用旧缓存——用户看到的是旧版界面,这是前端发布最隐蔽的故障模式。

配置方式 行为 适用场景
无版本化 + 长缓存 部署后浏览器仍用旧缓存,需强制刷新 不推荐
无版本化 + 短缓存 频繁回源,丧失缓存收益 开发环境
内容哈希版本化 + 长缓存 内容变化触发新 URL,精确失效 推荐(生产环境)
固定版本化 + 长缓存 整体版本号变化,所有资源同时失效 简单场景

启用内容哈希版本化需三步:其一,在 WebMvcConfigurer 配置 VersionResourceResolver + ContentVersionStrategy;其二,在模板中使用 URL 转换机制(Thymeleaf 的 @{} 或 ResourceUrlProvider);其三,设置长期 Cache-Control(max-age=31536000, immutable)。遗漏任何一步都会导致版本化失效。

8.8 本章小结

本章系统剖析了 Spring 的 Resource 资源抽象与静态资源处理机制,核心要点归纳如下:

Resource 接口体系统一了类路径、文件系统、URL、内存字节等所有资源的访问方式。六大实现类各有适用场景:ClassPathResource 最常用于配置与模板;InputStreamResource 是一次性的;ByteArrayResource 适合单元测试 mock;UrlResource 用于远程资源;FileSystemResource 是唯一支持写入的实现。

ResourceLoader 通过前缀分发机制将 classpath:、file:、http: 等前缀路由到不同的 Resource 实现。ProtocolResolver SPI 允许注册自定义协议(如 s3://)。ResourcePatternResolver 扩展了批量扫描能力,支持 Ant 风格通配符与 classpath*: 全局扫描。

PathMatchingResourcePatternResolver 的扫描算法处理了文件系统、JAR 包内资源、嵌套 JAR、多 ClassLoader 等多种场景。MyBatis Mapper XML 加载是其经典使用场景。

Spring Boot 静态资源按 META-INF/resources/ → resources/ → static/ → public/ 优先级搜索。ResourceHandlerRegistry 支持自定义映射,CacheControl 与 VersionResourceResolver 实现精确缓存控制。WebJars 自动映射为 /webjars/**。

ResourceRegion 与 ResourceHttpRequestHandler 实现了 HTTP Range 请求(206 Partial Content),支持视频流式传输与断点续传。ResourceHttpRequestHandler 自动处理 ETag、Last-Modified、Range 等协议细节。

五类典型陷阱:classpath: 与 classpath*: 的语义差异、相对路径基于 JVM 工作目录的不可预测性、fat JAR 中 getFile() 抛异常、第三方 API 只接受 File 的临时文件方案、静态资源缺少版本哈希导致的缓存失效。修复方案均围绕"流式优先"与"显式版本化"两条原则展开。
