第 3 章 · IoC 容器与 Bean 生命周期

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

IoC 容器的核心抽象是 BeanDefinition(Bean 蓝图)与 BeanFactory(实例工厂)。前者是注解、XML 与 @Bean 方法等多种声明形式被解析后的统一元数据,后者负责依赖解析、实例化、生命周期回调与扩展点调度。

容器通过四阶段处理 BeanDefinition:第一阶段由 BeanFactoryPostProcessor 修改注册表(自动配置在此阶段写入候选);第二阶段 BeanFactory.preInstantiateSingletons 按依赖 DAG 顺序触发非懒加载单例创建;第三阶段对每个 Bean 执行 Aware 接口回调、初始化方法与 BeanPostProcessor 链;第四阶段在容器关闭时按反向顺序执行销毁回调。

循环依赖是单例创建过程中的关键挑战。Spring 通过三级缓存(singletonObjects、earlySingletonObjects、singletonFactories)提供单例的早期引用,使 Setter 注入与字段注入的循环依赖可在不破坏 AOP 代理一致性的前提下被解决。本章按"BeanDefinition → 依赖 DAG → 三级缓存 → 生命周期 → Scope → BeanPostProcessor → FactoryBean → 源码深潜"八层组织。

3.2 依赖 DAG

图 3-2-1 · 典型依赖 DAG(4 层 × 10 节点 · 箭头=被依赖→依赖者)

箭头方向表示"依赖谁":被依赖者先创建,依赖者后创建。DefaultListableBeanFactory.preInstantiateSingletons 遍历所有非懒加载单例 BeanDefinition,对每个 Bean 调用 getBean,递归创建依赖。

图 3-2-2 · getBean() 的 DFS 后序遍历(下行递归 + 上行回溯 8 步)

Spring 的 getBean() 实质执行 DFS 后序遍历的拓扑排序。与 Kahn 算法的 BFS 方式不同,但保证被依赖者先于依赖者创建。检测到环时(isCurrentlyInCreation)触发循环依赖处理(详见 §3.3)。

表 3-2-1:依赖声明四种方式对比

方式 显式程度 容器解析 推荐度
构造器参数注入 最显式 创建实例时解析 推荐(强制不可变)
@Autowired 字段 较隐式 实例化后 BeanPostProcessor 解析 仅测试或脚手架
@Autowired Setter 显式 populateBean 阶段解析 适用于循环依赖
@DependsOn 隐式依赖 显式声明字符串 preInstantiateSingletons 排序 隐式资源依赖(Topic)

3.3 三级缓存解决循环依赖

表 3-3-1:三级缓存数据结构

级别 缓存名 Map 类型 存放内容 进入时机 退出时机
L1 singletonObjects ConcurrentHashMap 完整初始化的 Bean 完成所有初始化后 容器销毁
L2 earlySingletonObjects ConcurrentHashMap 半成品 Bean(可能为代理) L3 ObjectFactory 被调用后 升入 L1 时
L3 singletonFactories HashMap<String, ObjectFactory<?>> Bean 创建工厂函数 构造器执行完毕后 ObjectFactory 调用后
图 3-3-1 · 为什么需要三级缓存(L1/L1+L2/L1+L2+L3 三方案对比)
图 3-3-2 · 循环依赖完整时序(5 swimlane × 16 step · A↔B 注入)

表 3-3-2:getSingleton 查找路径

步骤 操作 结果
1 singletonObjects.get(beanName) L1 miss / hit
2 isSingletonCurrentlyInCreation(beanName) 判断是否在创建链中
3 earlySingletonObjects.get(beanName) L2 miss / hit
4 singletonFactories.get(beanName) L3 miss / hit
5 factory.getObject 触发 getEarlyBeanReference 生成早期引用(可能为代理)
6 earlySingletonObjects.put(beanName, ref) 升入 L2
7 singletonFactories.remove(beanName) 离开 L3 
protected Object getSingleton(String beanName, boolean allowEarlyReference) {
    Object singletonObject = this.singletonObjects.get(beanName);
    if (singletonObject == null && isSingletonCurrentlyInCreation(beanName)) {
        singletonObject = this.earlySingletonObjects.get(beanName);
        if (singletonObject == null && allowEarlyReference) {
            synchronized (this.singletonObjects) {
                singletonObject = this.singletonObjects.get(beanName);
                if (singletonObject == null) {
                    singletonObject = this.earlySingletonObjects.get(beanName);
                    if (singletonObject == null) {
                        ObjectFactory<?> singletonFactory =
                            this.singletonFactories.get(beanName);
                        if (singletonFactory != null) {
                            singletonObject = singletonFactory.getObject();
                            this.earlySingletonObjects.put(beanName, singletonObject);
                            this.singletonFactories.remove(beanName);
                        }
                    }
                }
            }
        }
    }
    return singletonObject;
}

L3 中的 ObjectFactory 本质是 Thunk(Ingerman, 1961, "Thunks: a way of compiling procedure statements", Communications of the ACM)。Thunk 将代理创建延迟到真正被引用时执行。无循环依赖时 Thunk 不会被调用,代理仍由 postProcessAfterInitialization 创建,无额外开销。

图 3-3-3 · 无法解决的循环依赖决策树(3 致命场景 + 1 可解场景)

Spring Boot 2.6 起默认禁止循环依赖(spring.main.allow-circular-references=false)。官方推荐通过重构代码消除循环。必须使用时,在 application.properties 中显式开启或在注入点添加 @Lazy。

拓展案例:@Transactional + @Async 失败

OrderService 带 @Transactional 与 InventoryService 带 @Async 互相注入,会抛出 BeanCurrentlyInCreationException。原因:AsyncAnnotationBeanPostProcessor 在 postProcessAfterInitialization 创建全新代理对象,L3 暴露给对方的是原始对象,两者地址不一致。修复:在任一注入点加 @Lazy。

@Service
public class OrderService {
    @Lazy
    @Autowired
    private InventoryService inventoryService;
}

3.6 BeanPostProcessor 链

多个 BPP 按 Order 排序形成处理链,每个 BPP 在 before 与 after 阶段对 Bean 实施变换,最终结果可能是被多层包装的代理。

表 3-6-1:核心 BPP 职责

图 3-6-2 · BPP 优先级金字塔(3 级 · PriorityOrdered → Ordered → Normal)

PriorityOrdered 实现的 BPP 最先排序与执行,其次 Ordered 实现,最后普通 BPP。同级别内 getOrder() 返回值越小优先级越高。AbstractAutoProxyCreator 通常排在最后,确保包装的是已完成所有注入的 Bean。

@Component
public class TraceBeanPostProcessor implements BeanPostProcessor, Ordered {

    @Override
    public Object postProcessBeforeInitialization(Object bean, String beanName) {
        if (beanName.equals("paymentService")) {
            System.out.println("[BPP before] " + beanName);
        }
        return bean;
    }

    @Override
    public Object postProcessAfterInitialization(Object bean, String beanName) {
        if (beanName.equals("paymentService")) {
            System.out.println("[BPP after] " + beanName + " class=" + bean.getClass().getName());
        }
        return bean;
    }

    @Override
    public int getOrder() {
        return Ordered.LOWEST_PRECEDENCE;
    }
}

3.末 本章要点

  • BeanDefinition 是 Bean 的元数据蓝图,与 Java 中 Class 之于 Object 的关系一致。容器先收集所有蓝图,再按依赖关系决定实例化顺序。10+ 核心属性涵盖类型、作用域、懒加载、依赖、构造参数、初始化与销毁方法、工厂方法等维度。

  • 依赖 DAG 由 @Autowired、构造参数、@DependsOn 与自动配置的 @AutoConfigureAfter 构成。容器通过 DFS 后序遍历(等价于拓扑排序)确定创建顺序,被依赖者先于依赖者创建。

  • 三级缓存通过 L1 完整对象、L2 早期引用、L3 ObjectFactory 解决单例的循环依赖。L3 中的 ObjectFactory 是 Thunk,延迟代理创建到真正被引用时执行。仅 setter / 字段注入循环可被解决,构造器循环、prototype 循环、@Async 终态代理循环无解。

  • Bean 生命周期由实例化、属性注入、Aware 回调、初始化前 BPP、@PostConstruct、InitializingBean、自定义 init、初始化后 BPP、就绪、@PreDestroy、DisposableBean、自定义 destroy 共 12 个阶段构成。@PostConstruct 与 @PreDestroy 是 JSR-250 标准,推荐使用。

  • 六类 Scope 中,prototype 不调用销毁回调,持有资源时需自行管理。@RefreshScope 通过 CGLIB 代理与 RefreshScope.refreshAll 实现配置中心热刷新。

  • BeanPostProcessor 链按 PriorityOrdered → Ordered → 普通三级排序。AbstractAutoProxyCreator 通常排最后,确保 AOP 代理包装的是已完成注入的 Bean。

  • FactoryBean 与 @Bean 方法均可自定义实例创建,前者适用于框架级集成(SqlSessionFactoryBean、JndiObjectFactoryBean),后者为日常首选。@Configuration 类经 CGLIB 代理后,内部 @Bean 方法间的调用返回容器单例。

  • 源码核心入口:AbstractApplicationContext.refresh → DefaultListableBeanFactory.preInstantiateSingletons → AbstractAutowireCapableBeanFactory.doCreateBean。DefaultSingletonBeanRegistry.getSingleton 是三级缓存查找的核心方法。
