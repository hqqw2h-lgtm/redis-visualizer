第 17 章 · 数据访问与 Spring Data

几乎每个 Spring Boot 应用都需要与数据库交互。从最底层的 JDBC 到 ORM 框架 Hibernate,从关系型数据库到 MongoDB、Redis、Elasticsearch——Spring 构建了统一的数据访问抽象,无论使用哪种存储技术,都能获得一致的异常体系、声明式事务与 Repository 编程模型。

本章从 Spring 数据访问哲学(异常翻译 + 模板方法)讲起,深入 JdbcTemplate 执行流程、Spring Data Repository 代理魔法、JPA / Hibernate 会话管理、R2DBC 非阻塞模型,最后探讨多数据源路由架构设计。

17.1 Spring 数据访问抽象层

Spring 之前,Java 程序员直接使用 JDBC 时面临两个痛点:样板代码爆炸与异常体系混乱。Spring 数据访问抽象层用两个核心设计解决:Template 模式消灭样板代码,DataAccessException 层级统一异常语义。

17.2 JdbcTemplate 与 NamedParameterJdbcTemplate

JdbcTemplate 是 Spring 最古老也最稳定的数据访问工具——从 Spring 1.0 沿用至今未被废弃。它直接操作 JDBC API,提供最精细的 SQL 控制,同时消灭 80% 样板代码。

表 17-2-1:核心方法族

方法用途返回值注意事项
query(sql, RowMapper)查询多行List T空结果返回空 List
queryForObject(sql, RowMapper)查询单行T0 行抛 EmptyResultDataAccessException
queryForList(sql, elementType)查询单列多行List TelementType 必须是单列可转换类型
update(sql, args)INSERT/UPDATE/DELETEint 影响行数
batchUpdate(sql, List)批量操作int 数组底层使用 JDBC batch
execute(sql)DDL 或存储过程void不返回结果
 
@Repository
@RequiredArgsConstructor
public class UserJdbcDao {
    private final JdbcTemplate jdbcTemplate;

    public Order findOne(long id) {
        return jdbcTemplate.queryForObject(
            "select id, name, email, status from orders where id = ?",
            (rs, rowNum) -> new Order(
                rs.getLong("id"),
                rs.getString("name"),
                rs.getString("email"),
                rs.getString("status")
            ),
            id
        );
    }

    public int deactivate(long id) {
        return jdbcTemplate.update(
            "update orders set status = ? where id = ?",
            "INACTIVE", id
        );
    }
}

连接管理关键:JdbcTemplate 不直接调用 DataSource.getConnection(),而是通过 DataSourceUtils.getConnection()。该工具类检查 TransactionSynchronizationManager 中是否有当前事务绑定连接——有则复用,无则从连接池获取新连接。这是 JdbcTemplate 无缝参与 @Transactional 事务的关键。

17.4 JPA / Hibernate 集成

JPA(Java Persistence API)是 Java 官方 ORM 标准,Hibernate 是最流行 JPA 实现。Spring Boot spring-boot-starter-data-jpa 自动配置 EntityManagerFactory、TransactionManager、连接池,开箱即用。

17.5 R2DBC 响应式数据访问

传统 JDBC 的 executeQuery() 是阻塞的——调用线程一直等待数据库返回结果。R2DBC(Reactive Relational Database Connectivity)是 JDBC 响应式替代品。

表 17-5-1:JDBC vs R2DBC

指标JDBC 阻塞模型R2DBC 非阻塞模型
典型线程数量默认线程池约 200EventLoop 约 2-4
等待 I/O 时请求线程被阻塞占用线程释放,结果回来后继续
并发连接效率线程数容易成为上限少量线程支撑更多并发
 
@Service
public class OrderService {
    private final DatabaseClient client;

    public Flux<Order> findActiveOrders() {
        return client.sql("SELECT id, name, email FROM orders WHERE status = :status")
            .bind("status", "ACTIVE")
            .map((row, meta) -> new Order(row.get("id", Long.class), row.get("name", String.class)))
            .all();
    }
}

public interface OrderRepository extends ReactiveCrudRepository<Order, Long> {
    Flux<Order> findByStatus(String status);
}

R2DBC 不是 JDBC 替代品而是补充。不支持存储过程、没有 ORM(无懒加载、无 dirty checking、无二级缓存),只有应用全响应式栈时才能发挥最大价值。

17.6 多数据源配置

现实应用往往需要连接多个数据库:读写分离主从库、不同业务模块独立库。Spring Boot 自动配置默认只创建一个 DataSource,多数据源需手动配置。

图 17-6-1 · AbstractRoutingDataSource 流程(AOP 设置 ThreadLocal → 路由 → 主从库)
public class DataSourceContextHolder {
    private static final ThreadLocal<String> CONTEXT = new ThreadLocal<>();
    public static void setDataSource(String key) { CONTEXT.set(key); }
    public static String getDataSource() { return CONTEXT.get(); }
    public static void clear() { CONTEXT.remove(); }
}

@Aspect
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class DataSourceAspect {
    @Before("execution(* com.example.service..*.find*(..))")
    public void autoSwitchToSlave() {
        DataSourceContextHolder.setDataSource("SLAVE");
    }

    @After("execution(* com.example.service..*(..))")
    public void clear() { DataSourceContextHolder.clear(); }
}

17.末 本章要点

  • DataAccessException 是基石。Spring 将所有数据库厂商异常翻译为统一、与数据库无关的 unchecked 异常树。
  • Template 模式消灭样板代码。所有 Template 都遵循获取资源到执行回调到翻译异常到释放资源统一流程。
  • JdbcTemplate 仍是最佳 SQL 控制工具。NamedParameterJdbcTemplate 用命名参数。通过 DataSourceUtils 无缝参与 Spring 事务。
  • Spring Data Repository 实现只写接口。方法名推导自动生成 JPQL,@Query 处理复杂查询,Specification API 构建动态查询。
  • JPA 核心是 EntityManager 生命周期。一级缓存保证事务内对象一致性与 Dirty Checking。Open-in-View 虽方便但长时间占用连接,生产建议关闭。
  • R2DBC 是 JDBC 响应式补充。不支持 ORM 特性,只适合全响应式技术栈。
  • 多数据源路由用 AbstractRoutingDataSource。数据源切换必须发生在事务开始之前。
  • 牢记七大陷阱:N+1 用 FETCH JOIN;Open-in-View 关闭;@Modifying 加 clearAutomatically;Lazy 用 DTO;queryForObject 空抛异常;Entity equals 不用自增 ID;长方法名用 @Query。
