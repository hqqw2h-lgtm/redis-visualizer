第 0 章 · 快速起步

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

Spring Boot 的核心抽象由三层构成:starter 负责声明能力依赖,自动配置负责按 classpath 与配置项条件注册 Bean,嵌入式 Web 服务器负责承接外部流量。Quick Start 的目标是建立一个最小可运行项目,使后续章节中的源码切片、条件判定与运行时观测都能在同一项目上复现。

本章选取 Spring Boot 3.3.5 作为基线版本,Java 17、Maven Wrapper 与 curl 作为命令工具,starter-web、starter-actuator 与 spring-boot-configuration-processor 作为初始依赖。所有命令、日志与端点输出均基于该组合验证,后续章节的扩展实验在此基础上推进。

Quick Start 的范畴限定为八条可观察路径:依赖装配、自动配置、容器刷新、请求处理、配置绑定、健康检查、排障决策与回归测试。每条路径在本章对应一个独立的二级节,提供原理图、源码入口与可执行命令,使读者完成本章后能够独立复现并解释每条路径的运行结果。

图 0-1-2 · 主类所在包决定组件扫描根

@SpringBootApplication 缺省以主类所在包为扫描根,递归扫描所有子包中的 @Component、@Service、@Repository 与 @Controller。主类放在最外层包是保证业务包均被扫描的前提。

package com.example.demo;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;

@SpringBootApplication
public class DemoApplication {
    public static void main(String[] args) {
        SpringApplication.run(DemoApplication.class, args);
    }
}

主类构成三层抽象边界:Boot 层负责装配策略,Spring Framework 层负责容器生命周期,Servlet 容器层负责网络入口。后续排障章节均以这三层划分作为定位起点。

0.3 自动配置与条件判定

图 0-3-1 · 自动配置导入与条件判定 10 步管线

自动配置由 AutoConfigurationImportSelector 加载 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 中列出的候选类。每个候选类经过 @ConditionalOnClass、@ConditionalOnMissingBean、@ConditionalOnProperty 等条件评估,仅命中者注册到容器。

表 0-3-1:常见条件注解语义

注解 触发条件 典型应用
@ConditionalOnClass classpath 存在指定类 DataSourceAutoConfiguration 依赖 javax.sql.DataSource
@ConditionalOnMissingClass classpath 不存在指定类 互斥能力检测
@ConditionalOnBean 容器已注册指定类型 Bean 依赖前置 Bean 才生效
@ConditionalOnMissingBean 容器未注册指定类型 Bean 用户自定义 Bean 时退让
@ConditionalOnProperty 配置项存在且匹配指定值 通过开关控制启用
@ConditionalOnWebApplication 当前为 Servlet/Reactive Web 应用 WebMvcAutoConfiguration
@ConditionalOnResource classpath 存在指定资源 模板引擎加载默认模板目录

启动命令附加 --debug 参数,Boot 会输出条件评估报告到控制台。

./mvnw spring-boot:run \
  -Dspring-boot.run.arguments="--debug"

图 0-3-2 · @ConditionalOnMissingBean 的 back-off 决策路径

@ConditionalOnMissingBean 是 back-off 机制的核心。框架默认实现仅在容器未持有同类型 Bean 时注册,声明用户级 @Bean 即可全权接管该能力,避免冲突而无需额外排除配置。

@Configuration
class WebTuning {
    @Bean
    WebMvcConfigurer corsConfigurer() {
        return new WebMvcConfigurer() {};
    }
}

表 0-3-2:自动配置设计取舍对比

候选方案 局限 Boot 选择
项目生成时写死配置类 模板进入用户仓库后框架失去演进通道 starter 仅声明依赖,自动配置在运行时按条件装配
classpath 全部启用 与用户自定义 Bean 易冲突,副作用难以预测 @ConditionalOnClass + @ConditionalOnMissingBean
用户手写所有 Bean 入门成本高,样板代码冗余 框架默认装配,用户声明同类 Bean 时自动 back-off
运行时扫描所有 jar 启动慢且行为不可预测 预生成 AutoConfiguration.imports,按条件筛选

设计目标是默认值可随版本演进。复制框架自动配置类到用户项目并就地修改,会破坏条件、排序与默认值的协同关系,升级时需逐行合并模板,违背 back-off 设计。

package com.example.demo;

import org.springframework.boot.SpringApplication;
import org.springframework.boot.autoconfigure.SpringBootApplication;
import org.springframework.boot.context.metrics.buffering.BufferingApplicationStartup;

@SpringBootApplication
public class DemoApplication {
    public static void main(String[] args) {
        SpringApplication app = new SpringApplication(DemoApplication.class);
        app.setApplicationStartup(new BufferingApplicationStartup(2048));
        app.run(args);
    }
}

0.5 第一个 Controller 请求

图 0-5-1 · DispatcherServlet 请求分发 10 步时序

请求处理由 DispatcherServlet.doDispatch() 主导。HandlerMapping 解析请求路径与 @RequestMapping 元数据匹配,HandlerAdapter 通过反射调用控制器方法,HttpMessageConverter 将返回值序列化为响应体。

package com.example.demo;

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RequestParam;
import org.springframework.web.bind.annotation.RestController;

import java.util.Map;

@RestController
class HelloController {
    @GetMapping("/hello")
    Map<String, String> hello(@RequestParam(defaultValue = "Boot") String name) {
        return Map.of("message", "Hello, " + name);
    }
}

package com.example.demo;

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.autoconfigure.web.servlet.WebMvcTest;
import org.springframework.test.web.servlet.MockMvc;

import static org.springframework.test.web.servlet.request.MockMvcRequestBuilders.get;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.jsonPath;
import static org.springframework.test.web.servlet.result.MockMvcResultMatchers.status;

@WebMvcTest(HelloController.class)
class HelloControllerTest {
    @Autowired MockMvc mvc;

    @Test
    void helloReturnsJson() throws Exception {
        mvc.perform(get("/hello").param("name", "Spring"))
           .andExpect(status().isOk())
           .andExpect(jsonPath("$.message").value("Hello, Spring"));
    }
}

@WebMvcTest 仅装配 MVC 切片,跳过完整 ApplicationContext 创建,典型测试耗时小于 800 ms,适合 Controller 层快速回归。

package com.example.demo;

import org.springframework.boot.context.properties.ConfigurationProperties;

@ConfigurationProperties(prefix = "app.greeting")
public record GreetingProperties(
    String prefix,
    String suffix,
    String audience
) {}

package com.example.demo;

import org.springframework.boot.context.properties.EnableConfigurationProperties;
import org.springframework.context.annotation.Configuration;

@Configuration
@EnableConfigurationProperties(GreetingProperties.class)
class AppConfig {
}

@RestController
class HelloController {
    private final GreetingProperties props;

    HelloController(GreetingProperties props) {
        this.props = props;
    }

    @GetMapping("/hello")
    Map<String, String> hello(@RequestParam(defaultValue = "Boot") String name) {
        return Map.of("message", props.prefix() + ", " + name + props.suffix());
    }
}

package com.example.demo;

import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.boot.test.web.client.TestRestTemplate;
import org.springframework.boot.test.web.server.LocalServerPort;

import static org.assertj.core.api.Assertions.assertThat;

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class QuickStartRuntimeTest {
    @LocalServerPort int port;
    @Autowired TestRestTemplate rest;

    @Test
    void healthAndControllerAreReachable() {
        String base = "http://localhost:" + port;

        String health = rest.getForObject(base + "/actuator/health", String.class);
        String hello = rest.getForObject(base + "/hello?name=Boot", String.class);

        assertThat(health).contains("\"status\":\"UP\"");
        assertThat(hello).contains("Hello, Boot");
    }
}

0.末 本章要点

  • 项目骨架由 starter 依赖与组件扫描根包共同决定。spring-boot-starter-web 引入 MVC、JSON、Tomcat 与日志四类能力,@SpringBootApplication 以主类所在包为扫描根递归注册组件。

  • 自动配置由 AutoConfigurationImportSelector 加载 META-INF/spring/AutoConfiguration.imports 中的候选类。每个候选类由 ConditionEvaluator 按 @ConditionalOnClass、@ConditionalOnMissingBean 等条件筛选,命中者注册到容器,结果记录于 ConditionEvaluationReport。

  • 启动主体耗时集中在 AbstractApplicationContext.refresh(),涵盖 BeanFactoryPostProcessor 执行、Bean 实例化与嵌入式容器启动三个阶段。BufferingApplicationStartup 与 /actuator/startup 端点提供步骤级时间线。

  • 请求处理由 DispatcherServlet.doDispatch() 主导,经 RequestMappingHandlerMapping 路径匹配、RequestMappingHandlerAdapter 反射调用、HttpMessageConverter 序列化三步完成。DEBUG 日志按该顺序输出可观察事件。

  • 配置绑定通过 Binder 与 Relaxed Binding 规则将 PropertySources 中的外部键转为 Java 属性。/actuator/configprops 暴露绑定结果,/actuator/env 暴露 PropertySource 顺序。

  • Actuator 端点是排障的标准取证工具。生产环境暴露面应限定为 health,info,prometheus,并通过 Spring Security 与内网 management 端口控制访问范围。

  • 排障决策按现象分类:端口冲突看启动日志、404 看 mappings、Bean 缺失看 conditions、配置未生效看 env 与 configprops、启动慢看 startup。证据应同时包含日志、端点输出与修复命令。

  • 回归测试应覆盖 MVC 切片(@WebMvcTest)与端到端冒烟(@SpringBootTest + RANDOM_PORT)两层。新增功能优先以新切片测试叠加,不修改既有冒烟用例。
