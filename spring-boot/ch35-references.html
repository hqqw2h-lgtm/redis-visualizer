35.1 学习路线分阶段

Spring Boot 学习不应当从「官方文档第一页」线性开始。下面这条路线按「能力可被验证」分阶段,适合本书读者继续进阶,也适合用来诊断团队中「会用 Boot 但不知道为什么」的能力断层。

35.1.1 阶段一:启动与配置

阶段目标:能解释 SpringApplication.run(MyApp.class, args) 从主线程进入到 ApplicationReadyEvent 触发之间的全部步骤,包括 Environment 构建、ApplicationContext 创建与刷新、SpringApplicationRunListeners 事件广播。

必读官方文档:Spring Boot Reference 的 Core Features → Externalized Configuration、Profiles、Logging、Internationalization 五个章节。

可运行证据:在本地启动应用时增加 --debug 与 logging.level.org.springframework.boot=DEBUG,观察事件序列。

ApplicationContextInitializedEvent  -> ApplicationContext 初始化前
ApplicationPreparedEvent            -> ApplicationContext 准备后,refresh 前
ApplicationStartedEvent             -> refresh 完成,ApplicationRunner 执行前
ApplicationReadyEvent               -> ApplicationRunner CommandLineRunner 完成后
ApplicationFailedEvent              -> 启动任意阶段失败

35.1.2 阶段二:自动配置与条件装配

阶段目标:看到一个第三方 starter 时,能预测它会在 classpath 上注册哪些自动配置类,以及条件成立 / 不成立的根本原因。

必读官方文档:Spring Boot Reference 的 Auto-configuration、Condition Annotations、Creating Your Own Auto-configuration 三个章节。

可运行证据:暴露 conditions 端点并查询特定自动配置类。

management:
  endpoints:
    web:
      exposure:
        include: health,info,conditions,beans,env,mappings

curl -s http://localhost:8080/actuator/conditions \
    | jq '.contexts.application.positiveMatches.DataSourceAutoConfiguration[0]'

{
  "condition": "OnClassCondition",
  "message": "@ConditionalOnClass found required classes 'javax.sql.DataSource', 'org.springframework.jdbc.datasource.embedded.EmbeddedDatabaseType'"
}

curl -s http://localhost:8080/actuator/conditions \
    | jq '.contexts.application.negativeMatches.DataSourceAutoConfiguration'

{
  "notMatched": [
    {
      "condition": "OnBeanCondition",
      "message": "@ConditionalOnMissingBean (types: javax.sql.DataSource; SearchStrategy: all) found beans of type 'javax.sql.DataSource' myDataSource"
    }
  ],
  "matched": []
}

读法:positiveMatches 回答「为什么有资格生效」,negativeMatches 回答「为什么没生效」。negativeMatches 中 OnBeanCondition 表示自定义 Bean 已存在,OnClassCondition 表示 classpath 缺少必要类,OnPropertyCondition 表示配置开关未启用。

35.1.3 阶段三:Web / Data / Transaction 主路径

阶段目标:对一个 HTTP 请求能描述其从 Servlet 容器进入到业务方法返回的全部经过;对一个 @Transactional 方法能描述其代理边界与回滚规则。

必读官方文档:Spring Framework Reference 的 Web Servlet、Data Access、Transaction Management 三个章节;Spring Boot Reference 的 Web、Data、Web Reactive 三个章节。

可运行证据:对一个 GET 请求开启 logging.level.org.springframework.web=TRACE,可看到完整的 handler mapping → handler adapter → exception resolver 链路;对一个 @Transactional 方法开启 logging.level.org.springframework.transaction=TRACE,可看到 Creating new transaction Initiating transaction commit Initiating transaction rollback 等关键日志。

35.1.4 阶段四:Actuator / Metrics / Tracing

阶段目标:线上出问题时能在 30 秒内拉出关键证据,而不是猜测或重启。

必读官方文档:Spring Boot Reference 的 Production-ready Features 章节;Micrometer 官方文档(https://micrometer.io/);OpenTelemetry Specification(https://opentelemetry.io/docs/specs/otel/)。

可运行证据:对 HikariCP 连接池查询 hikaricp.connections.active 与 hikaricp.connections.pending;对 HTTP 请求查询 http.server.requests 按 uri 与 status 维度;对追踪查询 /actuator/traces 或 OpenTelemetry collector 的 traces endpoint。

35.1.5 阶段五:升级 / AOT / Native / Cloud

阶段目标:能完成 Spring Boot 小版本升级(2.7 → 3.0、3.x → 3.x+1)与 Native Image 构建;能在 Spring Cloud 2024.x → 2025.x BOM 升级时正确处理 breaking changes。

必读官方文档:Spring Boot Migration Guide(GitHub Wiki);Spring Boot Release Notes;GraalVM Native Image Reference;Spring Native Image Reference;Spring Cloud Release Notes。

可运行证据:升级前用 mvn dependency:tree 与 gradle dependencyInsight 锁定关键依赖版本;构建 Native Image 时启用 -H:+PrintAnalysisCallTree -H:+ReportExceptionStackTraces 收集详细诊断;Spring Cloud 升级后用 conditions 端点确认 starter 替换。

35.1.6 阶段六:源码与贡献

阶段目标:能在 IntelliJ 中导入 spring-projects/spring-boot 与 spring-projects/spring-framework,本地跑通核心模块测试,定位 Issue 中提到的源码位置并提交 PR。

必读官方文档:CONTRIBUTING.md(spring-boot 仓库根目录);Triage Guide(GitHub Wiki);Code of Conduct;Coding Conventions。

可运行证据:本地 ./gradlew :spring-boot-project:spring-boot:test 跑通 spring-boot 核心模块单元测试;./gradlew :spring-boot-project:spring-boot-autoconfigure:test --tests "*JpaProperties*" 单独跑指定测试。

35.2.1 Spring Framework Reference

URL: https://docs.spring.io/spring-framework/reference/

主线分章:Core(IoC / Resources / Validation / Data Binding / Type Conversion / SpEL / AOP)、Testing、Data Access(Transaction / DAO / JDBC / O R M / XML Marshalling)、Web Servlet、Web Reactive、Integration、Languages(Kotlin / Groovy / Dynamic)。

阅读策略:不通读,按章节标题精确查找。Core 章节是其他章节的前置,任何对 Bean 生命周期 / 代理 / SpEL 不熟悉的疑问都先回 Core。Web Servlet 与 Web Reactive 完全独立,选定一个为项目主线。

35.2.2 Spring Boot Reference

URL: https://docs.spring.io/spring-boot/reference/

主线分章:Documentation Overview / Getting Started / Upgrading / Using Spring Boot / Core Features / Web / Data / IO / Messaging / Container Images / Production-ready Features / Deploying / GraalVM Native Image / CLI / Build Tool Plugins / Developer Tools / Spring Boot Application Properties / Auto-configuration Classes / Test Auto-configuration Annotations。

阅读策略:Core Features 与 Production-ready Features 是日常引用最多的两章,值得通读。Auto-configuration Classes 是反向索引,从 starter 出发回查激活了哪些类。

35.2.3 Spring Initializr 与起步项目

URL: https://start.spring.io/

Initializr 不仅是 Web 界面,还提供 REST API:

curl https://start.spring.io/starter.zip \
    -d type=maven-project \
    -d language=java \
    -d bootVersion=3.5.0 \
    -d baseDir=order-service \
    -d groupId=com.example \
    -d artifactId=order-service \
    -d packageName=com.example.orderservice \
    -d packaging=jar \
    -d javaVersion=21 \
    -d dependencies=web,data-jpa,actuator,prometheus \
    -o order-service.zip

CLI 命令 spring init --dependencies=web,actuator order-service 内部即调用此 API。Initializr 输出的 BOM 与版本组合保证可编译,是新项目的版本基线。

35.2.4 Common Application Properties

URL: https://docs.spring.io/spring-boot/appendix/application-properties/

按命名空间分组:spring.*(核心)、management.*(Actuator)、server.*(Servlet / Reactive 容器)、logging.*、info.*、debug / trace、第三方 starter 自带前缀。

每个属性条目包含:名称、默认值、类型、描述、since 版本。IDE 中通过 spring-boot-configuration-processor 在编译期生成 META-INF/spring-configuration-metadata.json,IntelliJ / VS Code 自动补全直接读取。自定义 @ConfigurationProperties Bean 可加入 metadata,享受同等补全体验。

35.2.5 Javadoc 与版本检索

URL: https://docs.spring.io/spring-boot/api/java/

Javadoc 是 API 精确语义的唯一可信来源。常用查询:@ConfigurationProperties 的 prefix 默认值、@RequestMapping consume / produce 的解析规则、HikariConfig 的字段类型与单位、MeterRegistry 各方法的线程安全约定。

历史版本通过 URL 路径切换:https://docs.spring.io/spring-boot/3.4/api/java/ 对应 3.4 LTS,https://docs.spring.io/spring-boot/3.5-SNAPSHOT/api/java/ 对应主分支预览。

35.2.6 Migration Guides 与 Release Notes

URL: https://github.com/spring-projects/spring-boot/wiki

Wiki 中按版本组织,例如「Spring Boot 3.0 Migration Guide」「Spring Boot 3.5 Release Notes」「Spring Boot 3.4.0 M1 Release Notes」。

阅读策略:升级前必读对应版本的 Migration Guide;升级中遇到 breaking change 时按 Release Notes 中的 deprecation timeline 回溯到首次提示版本;Spring Cloud 升级独立查 spring-cloud-release Wiki(https://github.com/spring-cloud/spring-cloud-release/wiki)。

图 35-2-2 · Spring Boot 与 Spring Cloud Release Train 对应关系时间线(2022–2025)

时间线读法:Spring Cloud Release Train 名按伦敦地铁站字母顺序命名(Aylesford → Brixton → Camden → ...),每年发布一个 train 对应当年发布的 Boot 主线版本。LTS 版本在 Boot 3.4 起每两年发布,期间小版本仅维护补丁。

35.3.1 入门与体系类

Spring 实战 第 6 版与 Pro Spring 6 第 6 版形成互补:前者面向 Boot 应用层,案例基于 Spring Boot 3.0;后者覆盖 Spring Framework 6 全模块,适合作为参考手册。中文读者的前置可读 Spring 揭秘,对 IoC 容器的设计动机讲解清晰,虽然版本停留在 Spring 3.0,但概念至今未变。

35.3.2 源码深潜类

Spring 源码深度解析 第 2 版基于 Spring 5.x,对 BeanFactory / ApplicationContext / AOP / 事务 / MVC 五大核心模块的代码逐行注释。Pro Spring 6 不直接读源码但每章末尾给出关键源码入口,适合配合 IDE 跳转。Spring Boot 3.x 源码本身在 spring-projects/spring-boot 仓库的 spring-boot-project/spring-boot-autoconfigure/src/main/java/org/springframework/boot/autoconfigure/ 下按子领域组织,直接阅读优于二手书。

35.3.3 设计与架构类

Patterns of Enterprise Application Architecture 把 DAO、Service Layer、Domain Model、Transaction Script、Active Record 等模式正式化,Spring Data 的 JpaRepository CrudRepository 设计直接借鉴该书。Domain-Driven Design 与 Implementing Domain-Driven Design 解释 Spring 应用为什么常被组织为「Application / Domain / Infrastructure」三层。POSA 1 中的 Reactor 模式是 Spring WebFlux / Reactor Core 的设计源头。

35.3.4 经典基础与性能

Effective Java 第 3 版的「Item 17 Minimize Mutability」「Item 18 Favor Composition over Inheritance」「Item 49 Check Parameters for Validity」直接对应 Spring 推荐的「构造函数注入 + 不可变 Bean + Bean Validation」模式。Java Concurrency in Practice 解释 TaskExecutor ThreadPoolTaskExecutor 与 WebFlux Schedulers 的并发模型选择。Java Performance 第 2 版覆盖 G1 / ZGC / Shenandoah 与容器化 JVM 调优,对 Spring Boot 生产部署直接相关。

35.5.1 IoC / DI 起源

Fowler 2004 年的文章首次系统区分 Setter Injection、Constructor Injection、Interface Injection 三种依赖注入形态,并把「依赖注入」与「服务定位器」明确切割。Spring 2.x 默认 Setter,3.x 起官方推荐 Constructor Injection,与文中「构造函数最能表达不可变依赖契约」论点一致。

35.5.2 AOP 与代理

Kiczales 1997 年 ECOOP 论文定义 Aspect / Advice / Pointcut / Weaving 概念,AspectJ 是其参考实现。Spring AOP 选择「JDK 动态代理 + CGLIB 子类代理」而非 AspectJ 字节码 weaving 的设计动机:无需修改构建工具链、运行时按需织入、与 Spring Bean 生命周期天然对齐。

35.5.3 并发与连接池

Doug Lea 的 ForkJoinPool 论文是 Java 7 加入 java.util.concurrent.ForkJoinPool 的设计依据,Spring ParallelTaskExecutor 基于该池。HikariCP 的并发控制不直接来自论文,但其 ConcurrentBag 设计借鉴了 java.util.concurrent.SynchronousQueue 与 ThreadLocal 优化思路,Brett Wooldridge 在 wiki 中给出详细推导。

35.5.4 缓存

Caffeine 用 TinyLFU 作为 admission policy,W-TinyLFU 作为 eviction policy。该设计相比传统 LRU 在 Zipf 分布下命中率提升 8 至 25 个百分点。Spring @Cacheable 默认实现可替换为 Caffeine,只需引入 caffeine 依赖并配置 CacheManager Bean。

35.5.5 I/O 与 Reactor

Schmidt 1995 年的 Reactor 模式论文定义了「单线程事件循环 + 非阻塞 I/O 多路复用」的标准结构,Tomcat NIO Connector、Netty EventLoop、Reactor-Netty 均为其实现。理解 Reactor 模式是理解为什么 WebFlux 不能在 EventLoop 线程上做阻塞 JDBC 调用的根本依据。

35.5.6 追踪与可观测性

Google Dapper 论文(2010)定义 Trace / Span / Context Propagation 三大概念,所有现代追踪系统(Zipkin / Jaeger / OpenTelemetry / SkyWalking)均沿用。W3C Trace Context 规范(2020)统一了 traceparent 与 tracestate HTTP header 格式,Spring Boot 3.x 通过 Micrometer Tracing 默认遵循该规范。

35.5.7 AOT 与 Native Image

GraalVM Native Image 基于 Substrate VM,核心论文给出「Closed-World Assumption + Points-To Analysis + 编译期初始化」三者如何协作把 Java 应用编译为静态二进制。Spring AOT 在该基础上引入 RuntimeHints 机制,把 Spring Bean 创建流程中的反射 / 代理 / 资源加载提前到编译期注册。

35.8.2 自动配置

入口:AutoConfigurationImportSelector#selectImports(AnnotationMetadata)。关键步骤:从 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 读取候选;通过 OnClassCondition OnBeanCondition OnPropertyCondition 等 SpringBootCondition 子类筛选;按 @AutoConfigureOrder @AutoConfigureBefore @AutoConfigureAfter 排序。

35.8.3 MVC / WebFlux 请求路径

MVC 入口:DispatcherServlet#doDispatch(HttpServletRequest, HttpServletResponse)。关键步骤:getHandler(request) 通过 RequestMappingHandlerMapping 返回 HandlerExecutionChain;getHandlerAdapter(handler) 返回 RequestMappingHandlerAdapter;applyPreHandle 与 applyPostHandle 触发拦截器;processDispatchResult 通过 HandlerExceptionResolver 处理异常。

WebFlux 入口:DispatcherHandler#handle(ServerWebExchange)。关键步骤:Flux.fromIterable(handlerMappings).concatMap(...) 异步匹配;HandlerAdapter#handle(...) 通过 RequestMappingHandlerAdapter 调用 controller 方法;HandlerResultHandler#handleResult(...) 处理返回值。

35.8.4 数据访问与事务

事务入口:TransactionInterceptor#invoke(MethodInvocation)。关键步骤:createTransactionIfNecessary(...) 通过 PlatformTransactionManager 开启事务;invokeWithinTransaction(...) 调用业务方法;completeTransactionAfterThrowing(...) 或 commitTransactionAfterReturning(...) 收尾。回滚规则定义在 RuleBasedTransactionAttribute#rollbackOn(Throwable),默认仅 RuntimeException 与 Error 触发回滚。

JdbcTemplate 入口:JdbcTemplate#execute(StatementCallback)。R2DBC 入口:R2dbcEntityTemplate#select(Query, Class)。JPA 入口:SimpleJpaRepository#findById(ID)。

35.8.5 安全过滤链

Spring Security 入口:FilterChainProxy#doFilter(ServletRequest, ServletResponse, FilterChain)。关键步骤:getFilters(request) 返回匹配的 SecurityFilterChain;依次执行 SecurityContextHolderFilter LogoutFilter UsernamePasswordAuthenticationFilter BearerTokenAuthenticationFilter AuthorizationFilter 等。AuthorizationManager#check(...) 是授权决策入口,实现包括 AuthenticatedAuthorizationManager AuthorityAuthorizationManager WebExpressionAuthorizationManager。

35.9.1 文档版本错位

Spring Boot 2.x 与 3.x 在配置项、自动配置类、starter 名称上存在多处不兼容(例如 spring.datasource.driver-class-name 在 3.x 仍可用但 spring.datasource.tomcat.* 已移除;spring-boot-starter-actuator 默认仅暴露 health 与 info,2.x 默认全暴露)。任何博客 / 教程在 URL 中未显式标注「3.x」「3.5」时,默认按 2.x 处理,需要逐项核对当前版本文档。

35.9.2 二手源码注释滞后

Spring 源码深度解析 第 2 版基于 Spring 5.x,涉及 ApplicationContext refresh 12 步、BeanFactory 三级缓存、AOP 代理创建等核心流程的源码注释大致仍适用 6.x;但涉及 Reactive、Native、AOT、Observation 等 6.x 后引入的能力时,二手书无法替代直接阅读 6.x 源码。

35.9.3 国内镜像延迟

Maven 公司内部 Nexus 镜像可能与 Maven Central 存在数小时至数天的同步延迟,Spring Boot Milestone(M)与 Release Candidate(RC)版本尤其受影响。升级到最新 RC 时,直接用 https://repo.spring.io/milestone 仓库,避免依赖企业镜像。

<repositories>
    <repository>
        <id>spring-milestones</id>
        <url>https://repo.spring.io/milestone</url>
    </repository>
</repositories>

35.9.4 收藏夹替代 Wiki

把链接列表存在浏览器收藏夹会随时间稀释为「曾经看过的页面」。团队 Wiki 应当包含:每个文档的 URL、版本号、所属类别(Reference / How-to / Properties / Javadoc / Migration / Issue)、最后核对日期、由哪个项目使用。

35.9.5 论文与生产时机错配

论文回答「为什么这样设计」,不回答「线上为什么挂了」。事故响应阶段的工具链是 Actuator endpoint、应用日志、APM 系统、慢查询日志;论文阅读应当排进事故复盘后的「设计回顾」环节,与产品规划、技术债务清理同周期。

35.9.6 跨语言文档错漏

中文文档可读性高,但翻译延迟与术语漂移会导致与官方文档存在差异。涉及精确语义(配置项默认值、注解 since 版本、API 行为约定)时以英文 Reference 与 Javadoc 为准。中文文档作为「概念入口」与「中文社区习惯术语映射」使用。

35.9.7 GitHub Issue 误读

spring-projects/spring-boot/issues 包含三类信息:bug 报告(Spring 团队负责修复)、enhancement 请求(可能 backlog 中或被拒绝)、question(通常重定向到 StackOverflow)。引用 Issue 作为「行为依据」前需确认其 label(type: bug / type: enhancement / for: stackoverflow)与状态(status: declined / status: superseded / 已 closed 是否 fixed)。
