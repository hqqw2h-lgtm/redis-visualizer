Spring 1.x 引入的 RmiServiceExporter、HttpInvokerServiceExporter、HessianServiceExporter、BurlapServiceExporter 通过代理透明把远程调用伪装成本地方法调用。Spring 5.3 起这些组件被标记 @Deprecated,Spring Framework 6.0(Spring Boot 3.0)正式移除整个 spring-remoting 子模块。

替代方案分四代:RestTemplate(2009 同步)、WebClient(2017 响应式)、@HttpExchange(Spring 6 声明式)、RestClient(Spring 6.1 同步流畅)。跨进程二进制通信走 gRPC + Protobuf,精准字段查询走 Spring for GraphQL,全双工流式走 RSocket。

本章按「演进与废弃 → REST 四代 → gRPC → GraphQL → RSocket → 陷阱 → 源码深潜」展开,聚焦每种方案在 Spring Boot 3.x / Spring 6.x 下的物理形态、调用链路与失败语义。

25.1.1 经典 RMI 代码(已废弃)

@Bean
public RmiServiceExporter rmiExporter(OrderService orderService) {
    RmiServiceExporter exporter = new RmiServiceExporter();
    exporter.setServiceName("OrderService");
    exporter.setService(orderService);
    exporter.setServiceInterface(OrderService.class);
    exporter.setRegistryPort(1099);
    return exporter;
}

@Bean
public RmiProxyFactoryBean orderService() {
    RmiProxyFactoryBean proxy = new RmiProxyFactoryBean();
    proxy.setServiceUrl("rmi://server:1099/OrderService");
    proxy.setServiceInterface(OrderService.class);
    return proxy;
}

代理透明的代价是把"网络不可靠 + 延迟 + 序列化版本敏感"包装成普通方法调用。Order order = orderService.findById(1L) 看起来像本地调用,实际经历了序列化、TCP、反序列化、执行、再次序列化、TCP、反序列化全过程。

25.1.2 遗留系统迁移路径

绞杀者模式(Strangler Fig)在迁移期同时保留旧 RMI 与新 REST。@RestController 包装 RMI 调用,新流量走 REST,旧流量逐步切流。

@RestController
@RequestMapping("/api/orders")
public class OrderServiceAdapter {

    @Autowired
    @Qualifier("legacyOrderService")
    private OrderService legacyOrderService;

    @GetMapping("/{id}")
    public ResponseEntity<OrderDTO> findById(@PathVariable Long id) {
        try {
            Order order = legacyOrderService.findById(id);
            return ResponseEntity.ok(OrderDTO.from(order));
        } catch (RemoteAccessException e) {
            return ResponseEntity.status(503).header("Retry-After", "5").build();
        }
    }
}

refreshStubOnConnectFailure(true) 与 lookupStubOnStartup(false) 是兼容期常用配置,前者在连接断开时重建 stub,后者推迟到首次调用时才查找远程对象。

25.2.1 RestTemplate

@Configuration
public class RestTemplateConfig {
    @Bean
    public RestTemplate restTemplate(RestTemplateBuilder builder) {
        return builder
            .rootUri("https://api.example.com")
            .setConnectTimeout(Duration.ofSeconds(5))
            .setReadTimeout(Duration.ofSeconds(10))
            .defaultHeader("Accept", MediaType.APPLICATION_JSON_VALUE)
            .additionalInterceptors(new LoggingInterceptor())
            .errorHandler(new CustomErrorHandler())
            .build();
    }
}

@Service
@RequiredArgsConstructor
public class OrderClient {
    private final RestTemplate restTemplate;

    public Order findById(Long id) {
        return restTemplate.getForObject("/api/orders/{id}", Order.class, id);
    }

    public List<Order> findAll() {
        ResponseEntity<List<Order>> response = restTemplate.exchange(
            "/api/orders", HttpMethod.GET, null,
            new ParameterizedTypeReference<List<Order>>() {});
        return response.getBody();
    }

    public Order create(CreateOrderRequest request) {
        return restTemplate.postForObject("/api/orders", request, Order.class);
    }
}

getForObject 等高级方法隐藏了 ResponseEntity,只在不需要响应头与状态码时使用。生产排障必须保留 URL、Header、状态码与响应体的完整链路日志。

25.2.2 WebClient

@Configuration
public class WebClientConfig {
    @Bean
    public WebClient webClient() {
        HttpClient httpClient = HttpClient.create()
            .option(ChannelOption.CONNECT_TIMEOUT_MILLIS, 5000)
            .responseTimeout(Duration.ofSeconds(10))
            .doOnConnected(conn -> conn
                .addHandlerLast(new ReadTimeoutHandler(10))
                .addHandlerLast(new WriteTimeoutHandler(5)));

        return WebClient.builder()
            .baseUrl("https://api.example.com")
            .clientConnector(new ReactorClientHttpConnector(httpClient))
            .defaultHeader("Accept", MediaType.APPLICATION_JSON_VALUE)
            .codecs(cfg -> cfg.defaultCodecs().maxInMemorySize(16 * 1024 * 1024))
            .build();
    }
}

@Service
@RequiredArgsConstructor
public class OrderReactiveClient {
    private final WebClient webClient;

    public Mono<Order> findById(Long id) {
        return webClient.get()
            .uri("/api/orders/{id}", id)
            .retrieve()
            .onStatus(HttpStatusCode::is4xxClientError, response ->
                response.bodyToMono(ErrorResponse.class)
                    .flatMap(err -> Mono.error(new OrderNotFoundException(err.message()))))
            .bodyToMono(Order.class);
    }

    public Mono<OrderDetails> getOrderDetails(Long orderId) {
        Mono<Order> order = webClient.get()
            .uri("/api/orders/{id}", orderId)
            .retrieve().bodyToMono(Order.class);
        Mono<List<Product>> products = order.flatMap(o ->
            webClient.get()
                .uri(b -> b.path("/products").queryParam("ids", o.getProductIds()).build())
                .retrieve().bodyToFlux(Product.class).collectList());
        return Mono.zip(order, products).map(t -> new OrderDetails(t.getT1(), t.getT2()));
    }

    public Flux<StockPrice> streamPrices(String symbol) {
        return webClient.get()
            .uri("/stream/prices/{symbol}", symbol)
            .accept(MediaType.TEXT_EVENT_STREAM)
            .retrieve()
            .bodyToFlux(StockPrice.class);
    }
}

Mono.zip 把多条远程调用合并为并发执行,而非串行等待。SSE(text/event-stream)流的关键观测点是事件持续输出,中断需排查代理缓冲、超时与心跳。

@Service
@RequiredArgsConstructor
public class OrderModernClient {
    private final RestClient restClient;

    public Order findById(Long id) {
        return restClient.get()
            .uri("/api/orders/{id}", id)
            .retrieve()
            .body(Order.class);
    }

    public List<Order> findAll() {
        return restClient.get()
            .uri("/api/orders")
            .retrieve()
            .body(new ParameterizedTypeReference<List<Order>>() {});
    }

    public ResponseEntity<Order> create(CreateOrderRequest request) {
        return restClient.post()
            .uri("/api/orders")
            .contentType(MediaType.APPLICATION_JSON)
            .body(request)
            .retrieve()
            .toEntity(Order.class);
    }

    public Order findByIdSafe(Long id) {
        return restClient.get()
            .uri("/api/orders/{id}", id)
            .retrieve()
            .onStatus(HttpStatusCode::is4xxClientError, (req, resp) -> {
                if (resp.getStatusCode() == HttpStatus.NOT_FOUND) {
                    throw new OrderNotFoundException("not found " + id);
                }
                throw new ClientException("client error " + resp.getStatusCode());
            })
            .onStatus(HttpStatusCode::is5xxServerError, (req, resp) -> {
                throw new ServerException("server error " + resp.getStatusCode());
            })
            .body(Order.class);
    }
}

RestClient 复用 RestTemplate 的 HttpMessageConverter 与 ClientHttpRequestFactory,但提供了 WebClient 风格的链式 API。新项目的同步 HTTP 调用首选该客户端。

25.2.4 @HttpExchange 声明式

@HttpExchange(url = "/api/orders", accept = "application/json")
public interface OrderApi {

    @GetExchange("/{id}")
    Order findById(@PathVariable Long id);

    @GetExchange
    List<Order> findAll(@RequestParam(required = false) String status);

    @PostExchange
    @ResponseStatus(HttpStatus.CREATED)
    Order create(@RequestBody CreateOrderRequest request);

    @PutExchange("/{id}")
    Order update(@PathVariable Long id, @RequestBody UpdateOrderRequest request);

    @DeleteExchange("/{id}")
    void delete(@PathVariable Long id);

    @GetExchange("/{id}")
    Mono<Order> findByIdReactive(@PathVariable Long id);
}

@Configuration
public class HttpInterfaceConfig {

    @Bean
    public OrderApi orderApiSync(RestClient.Builder builder) {
        RestClient restClient = builder.baseUrl("https://api.example.com").build();
        RestClientAdapter adapter = RestClientAdapter.create(restClient);
        HttpServiceProxyFactory factory = HttpServiceProxyFactory.builderFor(adapter).build();
        return factory.createClient(OrderApi.class);
    }

    @Bean
    public OrderApi orderApiReactive(WebClient.Builder builder) {
        WebClient webClient = builder.baseUrl("https://api.example.com").build();
        WebClientAdapter adapter = WebClientAdapter.create(webClient);
        HttpServiceProxyFactory factory = HttpServiceProxyFactory.builderFor(adapter).build();
        return factory.createClient(OrderApi.class);
    }
}

HttpServiceProxyFactory 通过 JDK 动态代理生成接口实现。注解信息编译期解析为 HttpRequestValues,运行时由 RestClientAdapter 或 WebClientAdapter 映射为底层调用。该方式取代 Spring Cloud OpenFeign 在原生 Spring 中的位置。

25.3.1 Protocol Buffers 定义

syntax = "proto3";
package com.example.order;
option java_multiple_files = true;
option java_package = "com.example.order.grpc";

service OrderService {
    rpc GetOrder (GetOrderRequest) returns (OrderResponse);
    rpc ListOrders (ListOrdersRequest) returns (stream OrderResponse);
    rpc BatchCreateOrders (stream CreateOrderRequest) returns (BatchCreateResponse);
    rpc OrderChat (stream ChatMessage) returns (stream ChatMessage);
}

message GetOrderRequest { int64 id = 1; }
message OrderResponse {
    int64 id = 1;
    string customer_id = 2;
    int32 amount = 3;
    OrderStatus status = 4;
    google.protobuf.Timestamp created_at = 5;
}

enum OrderStatus { UNKNOWN = 0; CREATED = 1; PAID = 2; SHIPPED = 3; }

proto3 默认 nullable 行为为零值,新增字段不会破坏兼容性。字段编号一旦发布不能修改,删除字段需保留编号占位(reserved 5;)。

25.3.2 服务端实现

grpc:
  server:
    port: 9090
    security:
      enabled: true
    max-inbound-message-size: 4MB
    max-inbound-metadata-size: 8KB

@GrpcService
public class OrderGrpcService extends OrderServiceGrpc.OrderServiceImplBase {

    @Autowired
    private OrderRepository orderRepository;

    @Override
    public void getOrder(GetOrderRequest request, StreamObserver<OrderResponse> responseObserver) {
        Order order = orderRepository.findById(request.getId()).orElse(null);
        if (order == null) {
            responseObserver.onError(Status.NOT_FOUND
                .withDescription("Order not found: " + request.getId())
                .asRuntimeException());
            return;
        }
        responseObserver.onNext(toResponse(order));
        responseObserver.onCompleted();
    }

    @Override
    public void listOrders(ListOrdersRequest request, StreamObserver<OrderResponse> responseObserver) {
        int pageSize = request.getPageSize() > 0 ? request.getPageSize() : 100;
        orderRepository.streamAll().limit(pageSize)
            .map(this::toResponse)
            .forEach(responseObserver::onNext);
        responseObserver.onCompleted();
    }
}

StreamObserver 是 gRPC 异步回调接口。Unary 模式下调用 onNext + onCompleted 一次,Server Streaming 调用 onNext 多次,异常用 onError 终止。

@Service
public class OrderGrpcClient {

    @GrpcClient("order-service")
    private OrderServiceGrpc.OrderServiceBlockingStub blockingStub;

    @GrpcClient("order-service")
    private OrderServiceGrpc.OrderServiceStub asyncStub;

    public OrderResponse getOrder(Long id) {
        try {
            return blockingStub
                .withDeadlineAfter(5, TimeUnit.SECONDS)
                .getOrder(GetOrderRequest.newBuilder().setId(id).build());
        } catch (StatusRuntimeException e) {
            if (e.getStatus().getCode() == Status.Code.NOT_FOUND) {
                throw new OrderNotFoundException("Order " + id + " not found");
            }
            throw new RemoteServiceException("gRPC call failed", e);
        }
    }
}

withDeadlineAfter 是 gRPC 超时机制,通过 grpc-timeout HTTP/2 header 传递。Status code 16 个标准值与 HTTP 状态码不同,NOT_FOUND = 5,DEADLINE_EXCEEDED = 4,UNAVAILABLE = 14。

25.3.4 拦截器与异常映射

@GrpcGlobalServerInterceptor
public class LoggingInterceptor implements ServerInterceptor {
    @Override
    public <ReqT, RespT> ServerCall.Listener<ReqT> interceptCall(
            ServerCall<ReqT, RespT> call, Metadata headers,
            ServerCallHandler<ReqT, RespT> next) {
        String method = call.getMethodDescriptor().getFullMethodName();
        long start = System.nanoTime();
        log.info("grpc start method={}", method);
        ServerCall<ReqT, RespT> wrapped = new ForwardingServerCall.SimpleForwardingServerCall<>(call) {
            @Override
            public void close(Status status, Metadata trailers) {
                long elapsedMs = (System.nanoTime() - start) / 1_000_000;
                log.info("grpc end method={} status={} elapsedMs={}", method, status.getCode(), elapsedMs);
                super.close(status, trailers);
            }
        };
        return next.startCall(wrapped, headers);
    }
}

@GrpcAdvice
public class GrpcExceptionAdvice {
    @GrpcExceptionHandler(OrderNotFoundException.class)
    public StatusRuntimeException handleNotFound(OrderNotFoundException e) {
        return Status.NOT_FOUND.withDescription(e.getMessage()).asRuntimeException();
    }

    @GrpcExceptionHandler(IllegalArgumentException.class)
    public StatusRuntimeException handleBadRequest(IllegalArgumentException e) {
        return Status.INVALID_ARGUMENT.withDescription(e.getMessage()).asRuntimeException();
    }
}

25.4.2 Controller 实现

@Controller
public class CustomerGraphQLController {

    @Autowired
    private CustomerService customerService;

    @QueryMapping
    public Customer customerById(@Argument Long id) {
        return customerService.findById(id)
            .orElseThrow(() -> new CustomerNotFoundException(id));
    }

    @QueryMapping
    public CustomerConnection customers(
            @Argument int page,
            @Argument int size,
            @Argument CustomerStatus status) {
        Page<Customer> result = customerService.findAll(page, size, status);
        return new CustomerConnection(
            result.getContent(),
            result.getTotalElements(),
            result.getTotalPages(),
            result.hasNext());
    }

    @MutationMapping
    public Customer createCustomer(@Argument CreateCustomerInput input) {
        return customerService.create(input);
    }

    @SchemaMapping(typeName = "Customer", field = "orders")
    public List<Order> customerOrders(Customer customer) {
        return orderService.findByCustomerId(customer.getId());
    }
}

@SchemaMapping(typeName="Customer", field="orders") 只在客户端 query 中包含 orders 字段时才执行,实现 GraphQL 的"按需解析"语义。

25.4.3 DataLoader 解决 N+1

@Controller
public class CustomerGraphQLController {

    @BatchMapping(typeName = "Customer", field = "orders")
    public Map<Customer, List<Order>> customerOrders(List<Customer> customers) {
        List<Long> customerIds = customers.stream().map(Customer::getId).toList();
        Map<Long, List<Order>> ordersByCustomerId =
            orderService.findByCustomerIds(customerIds).stream()
                .collect(Collectors.groupingBy(Order::getCustomerId));
        return customers.stream().collect(Collectors.toMap(
            customer -> customer,
            customer -> ordersByCustomerId.getOrDefault(customer.getId(), List.of())));
    }
}

@BatchMapping 自动注册 DataLoader,把同一请求中所有 Customer 对象的 orders 字段合并为单次批量调用。SELECT ... WHERE customer_id IN (1,2,3) 取代 N 次 SELECT ... WHERE customer_id = ?。

@Bean
public WebGraphQlInterceptor queryComplexityInterceptor() {
    return (request, chain) -> {

    };
}

MaxQueryDepthInstrumentation 与 MaxQueryComplexityInstrumentation 是 graphql-java 内置防护。深度限制阻止递归引用,复杂度限制阻止笛卡尔积爆炸。

@Controller
public class OrderRSocketController {

    @Autowired
    private OrderService orderService;

    @MessageMapping("order.findById")
    public Mono<Order> findById(Long id) {
        return Mono.fromCallable(() -> orderService.findById(id))
            .subscribeOn(Schedulers.boundedElastic());
    }

    @MessageMapping("order.audit")
    public Mono<Void> auditLog(AuditEvent event) {
        return Mono.fromRunnable(() -> auditService.record(event))
            .subscribeOn(Schedulers.boundedElastic())
            .then();
    }

    @MessageMapping("order.stream")
    public Flux<Order> streamOrders(OrderFilter filter) {
        return Flux.interval(Duration.ofSeconds(1))
            .flatMap(tick -> Mono.fromCallable(() -> orderService.findRecentlyUpdated(filter)))
            .flatMapMany(Flux::fromIterable);
    }

    @MessageMapping("order.chat")
    public Flux<ChatResponse> chat(Flux<ChatRequest> requests) {
        return requests
            .doOnNext(req -> log.info("received {}", req.getMessage()))
            .map(req -> new ChatResponse("echo " + req.getMessage(), Instant.now()));
    }
}

@MessageMapping("order.findById") 中 order.findById 是 RSocket route,客户端通过 requester.route("order.findById") 寻址。route 与 HTTP path 不同,使用 . 作为分层符。

25.5.2 客户端

@Configuration
public class RSocketClientConfig {
    @Bean
    public RSocketRequester rSocketRequester(RSocketRequester.Builder builder) {
        return builder
            .rsocketConnector(connector -> connector
                .reconnect(Retry.backoff(10, Duration.ofSeconds(1))
                    .maxBackoff(Duration.ofSeconds(30))))
            .tcp("localhost", 7000);
    }
}

@Service
@RequiredArgsConstructor
public class OrderRSocketClient {
    private final RSocketRequester requester;

    public Mono<Order> findById(Long id) {
        return requester.route("order.findById").data(id).retrieveMono(Order.class);
    }

    public Mono<Void> auditLog(AuditEvent event) {
        return requester.route("order.audit").data(event).send();
    }

    public Flux<Order> streamOrders(OrderFilter filter) {
        return requester.route("order.stream").data(filter).retrieveFlux(Order.class);
    }

    public Flux<ChatResponse> chat(Flux<ChatRequest> requests) {
        return requester.route("order.chat").data(requests).retrieveFlux(ChatResponse.class);
    }
}

Retry.backoff(10, Duration.ofSeconds(1)) 配置最多 10 次重连,初始间隔 1000 ms,指数退避到 30000 ms。RSocket 协议支持连接恢复,但客户端仍需显式配置重连策略。

25.6.1 陷阱 1:RestTemplate 阻塞 EventLoop

// 反例:WebFlux Controller 内同步调用
@GetMapping("/orders/{id}")
public Mono<OrderDetails> getOrder(@PathVariable Long id) {
    Order order = restTemplate.getForObject("/api/orders/{id}", Order.class, id);
    return Mono.just(new OrderDetails(order));
}

// 正例:WebClient 非阻塞
@GetMapping("/orders/{id}")
public Mono<OrderDetails> getOrder(@PathVariable Long id) {
    return webClient.get()
        .uri("/api/orders/{id}", id)
        .retrieve()
        .bodyToMono(Order.class)
        .map(OrderDetails::new);
}

// 折中:必须用阻塞客户端时切 boundedElastic
@GetMapping("/orders/{id}")
public Mono<OrderDetails> getOrder(@PathVariable Long id) {
    return Mono.fromCallable(() ->
            restTemplate.getForObject("/api/orders/{id}", Order.class, id))
        .subscribeOn(Schedulers.boundedElastic())
        .map(OrderDetails::new);
}

WebFlux 的 EventLoop 默认数量为 CPU 核数。阻塞调用占用 EventLoop 时,后续请求只能排队。Schedulers.boundedElastic() 提供专用阻塞线程池(默认 10 × CPU),适合包装无法替换的同步 IO。

25.6.2 陷阱 2:连接池隔离

@Bean
public WebClient customerServiceClient() {
    ConnectionProvider provider = ConnectionProvider.builder("customer-pool")
        .maxConnections(50)
        .maxIdleTime(Duration.ofSeconds(20))
        .maxLifeTime(Duration.ofMinutes(5))
        .pendingAcquireTimeout(Duration.ofSeconds(5))
        .pendingAcquireMaxCount(100)
        .build();
    HttpClient httpClient = HttpClient.create(provider).responseTimeout(Duration.ofSeconds(5));
    return WebClient.builder()
        .baseUrl("https://customer-service")
        .clientConnector(new ReactorClientHttpConnector(httpClient))
        .build();
}

每个下游服务一个独立 ConnectionProvider。maxConnections=50 限制单服务最大连接数,pendingAcquireTimeout=5000 ms 在池满时阻塞 5 秒,超过抛 PoolAcquireTimeoutException。监控指标 reactor.netty.connection.provider.{active,pending}.connections 是必采点。

25.6.3 陷阱 3:gRPC Deadline 传播

@GrpcGlobalClientInterceptor
public class DeadlinePropagationInterceptor implements ClientInterceptor {
    @Override
    public <ReqT, RespT> ClientCall<ReqT, RespT> interceptCall(
            MethodDescriptor<ReqT, RespT> method,
            CallOptions options,
            Channel next) {
        Deadline deadline = Context.current().getDeadline();
        if (deadline != null && options.getDeadline() == null) {
            options = options.withDeadline(deadline);
        }
        return next.newCall(method, options);
    }
}

Context.current().getDeadline() 返回入站请求的剩余截止时间。拦截器将其透传给下游 stub,避免上游已超时下游仍在执行。此机制是 gRPC 链路超时收敛的基础。

25.6.4 陷阱 4:GraphQL 复杂度

@Bean
public Instrumentation queryComplexityInstrumentation() {
    return new MaxQueryComplexityInstrumentation(200,
        (env, childComplexity) -> {
            if (env.getField().getName().equals("orders")) {
                return childComplexity * 50;
            }
            if (env.getField().getName().equals("reviews")) {
                return childComplexity * 100;
            }
            return 1 + childComplexity;
        });
}

复杂度计算考虑列表字段的预估返回数量。深嵌套查询(如 customers.orders.customer.orders.customer.orders)会指数级放大数据库负载。Persisted Queries(查询白名单)是更严格的防护:服务端只接受预先注册的查询哈希。

25.6.5 陷阱 5:@HttpExchange 传输异常

@Service
public class CustomerService {
    private final CustomerApi customerApi;

    public Customer findByIdSafe(Long id) {
        try {
            return customerApi.findById(id);
        } catch (ResourceAccessException e) {
            log.error("transport error customerId={}", id, e);
            throw new ServiceUnavailableException("Customer service unavailable", e);
        } catch (HttpClientErrorException e) {
            if (e.getStatusCode() == HttpStatus.NOT_FOUND) {
                throw new CustomerNotFoundException(id);
            }
            throw e;
        }
    }
}

onStatus 处理器只覆盖 HTTP 状态码语义。连接超时、DNS 解析失败、TLS 握手失败抛 ResourceAccessException,需在调用侧或 requestInterceptor 中处理。

25.6.6 陷阱 6:序列化版本

public class Order implements Serializable {
    private static final long serialVersionUID = 1L;
    private Long id;
    private String customerId;
    private String emailNotice;

    private void readObject(ObjectInputStream in) throws IOException, ClassNotFoundException {
        ObjectInputStream.GetField fields = in.readFields();
        this.id = (Long) fields.get("id", null);
        this.customerId = (String) fields.get("customerId", null);
        this.emailNotice = (String) fields.get("emailNotice", null);
    }
}

serialVersionUID 显式声明后,新增字段不影响二进制兼容性。readObject 自定义实现允许在反序列化时为新字段提供默认值。根本解决方案是迁移到 JSON / Protobuf,二者天然支持字段增删的前后兼容。

25.7.1 设计取舍

取舍 1:同步与响应式并存

Spring 6 同时维护 RestClient(同步)与 WebClient(响应式),而非强推统一抽象。代价是两套 API 文档与运维知识,收益是 Spring MVC 应用可平滑接入虚拟线程,而 WebFlux 应用保持非阻塞 IO。

取舍 2:@HttpExchange 仅声明契约,不内置服务发现

@HttpExchange 是 Spring Framework 原生 API,不绑定 Spring Cloud。代价是服务发现、负载均衡、熔断需自行装配,收益是接口可在不依赖 Spring Cloud 的应用中使用,运行时开销低于 OpenFeign。

取舍 3:GraphQL 的字段级解析换来 N+1 风险

GraphQL 把"取哪些字段"的决策权交给客户端,代价是字段嵌套时容易触发 N+1 查询,收益是一次请求完成多源数据聚合。@BatchMapping 与 DataLoader 是必备护栏,而非可选优化。
