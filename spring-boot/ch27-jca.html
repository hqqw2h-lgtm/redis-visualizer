第 27 章 · JCA 连接器架构

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

JCA(Jakarta Connector Architecture,旧称 Java Connector Architecture)由 JSR-322 定义,核心目标是把 Java 应用接入异构 EIS(Enterprise Information System)系统时必须处理的四件事标准化:连接生命周期、XA 事务登记、安全主体传播、入站消息分发。

资源适配器(ResourceAdapter,RAR 包)由应用服务器部署。ManagedConnectionFactory 创建受管连接,ConnectionManager 与连接池协作分配连接,XAResource 把 EIS 操作登记进全局事务,MessageEndpoint 接收 EIS 主动推送的消息。

Spring Boot 应用通常不复刻完整 JCA 容器。JndiObjectFactoryBean 从应用服务器查找已部署的 ConnectionFactory,CciTemplate 封装 CCI(Common Client Interface)交互,JtaTransactionManager 把 JCA 资源接入 JTA 事务,GenericMessageEndpointManager 注册入站端点。

本章按「JCA 合约 → JNDI 消费 → CCI 出站 → JTA 事务 → 入站端点 → 陷阱 → 源码深潜」展开,聚焦 SAP、CICS、AS/400、大型机等遗留系统的集成边界。

27.2.2 Spring Bean 装配

@Configuration
@EnableTransactionManagement
public class JcaClientConfiguration {

    @Bean
    JndiObjectFactoryBean customerConnectionFactory(CustomerJcaProperties properties) {
        JndiObjectFactoryBean bean = new JndiObjectFactoryBean();
        bean.setJndiName(properties.connectionFactoryJndiName());
        bean.setProxyInterface(jakarta.resource.cci.ConnectionFactory.class);
        bean.setLookupOnStartup(true);
        return bean;
    }

    @Bean
    CciTemplate customerCciTemplate(ConnectionFactory customerConnectionFactory) {
        CciTemplate template = new CciTemplate(customerConnectionFactory);
        template.setOutputRecordCreator(inputRecord -> {
            RecordFactory factory = customerConnectionFactory.getRecordFactory();
            return factory.createMappedRecord("customer-output");
        });
        return template;
    }

    @Bean
    PlatformTransactionManager transactionManager() {
        return new JtaTransactionManager();
    }
}

setLookupOnStartup(true) 强制启动期 JNDI 查找,避免运行时首次调用才暴露 JNDI 名错误。setProxyInterface 让注入点接收的是接口代理而非 Object,提升类型安全。

27.3 CCI 出站调用

图 27-3-1 · CciTemplate 执行链时序(模板方法 8 步 — 取连接 → 创建 Interaction → 执行 → 关闭)

27.3.1 InteractionSpec 与 RecordFactory

@Bean
InteractionSpec readCustomerSpec() {
    CustomerInteractionSpec spec = new CustomerInteractionSpec();
    spec.setFunctionName("CUSTOMER_READ");
    spec.setInteractionVerb(InteractionSpec.SYNC_SEND_RECEIVE);
    spec.setExecutionTimeout(3000);
    return spec;
}

InteractionSpec 是厂商特定接口(CustomerInteractionSpec 由 RA 提供),描述本次交互的元数据(函数名、超时、同步/异步语义)。RecordFactory 创建 MappedRecord(键值对)、IndexedRecord(列表)等标准 CCI 数据结构。

27.3.2 Gateway 实现

@Component
public class CustomerGateway {

    private final CciTemplate cciTemplate;
    private final InteractionSpec readCustomerSpec;

    public CustomerGateway(CciTemplate cciTemplate, InteractionSpec readCustomerSpec) {
        this.cciTemplate = cciTemplate;
        this.readCustomerSpec = readCustomerSpec;
    }

    public CustomerSnapshot findCustomer(String customerId) {
        return (CustomerSnapshot) cciTemplate.execute(readCustomerSpec,
            recordFactory -> {
                MappedRecord record = recordFactory.createMappedRecord("CUSTOMER_READ");
                record.put("CUSTOMER_ID", customerId);
                record.put("INCLUDE_BLOCK_STATUS", "Y");
                return record;
            },
            output -> {
                MappedRecord record = (MappedRecord) output;
                return new CustomerSnapshot(
                    (String) record.get("CUSTOMER_ID"),
                    (String) record.get("NAME"),
                    (String) record.get("BLOCK_STATUS"));
            });
    }
}

CciTemplate.execute() 接收 RecordCreator 与 RecordExtractor 两个回调:前者构造输入记录,后者把输出记录映射成 DTO。模板内部完成 getConnection() / createInteraction() / close() 全过程,异常被转换为 DataAccessException 体系。

27.3.3 连接池指标

@Component
public class JcaPoolMetrics implements MeterBinder {

    private final ManagedConnectionPoolMXBean pool;

    public JcaPoolMetrics(ManagedConnectionPoolMXBean pool) {
        this.pool = pool;
    }

    @Override
    public void bindTo(MeterRegistry registry) {
        Gauge.builder("jca.connections.active", pool, ManagedConnectionPoolMXBean::getActiveCount)
            .description("Active managed JCA connections")
            .register(registry);
        Gauge.builder("jca.connections.idle", pool, ManagedConnectionPoolMXBean::getIdleCount)
            .description("Idle managed JCA connections")
            .register(registry);
        Gauge.builder("jca.connections.waiting", pool, ManagedConnectionPoolMXBean::getWaitingThreadCount)
            .description("Threads waiting for JCA connection")
            .register(registry);
    }
}

JCA 池指标必须通过 JMX MBean(ManagedConnectionPoolMXBean)从应用服务器暴露,Spring 进程内不维护池状态。waitingThreadCount 持续大于 0 是池容量不足或连接泄漏的早期信号。

27.4 XA 事务

图 27-4-1 · XA 两阶段提交时序(begin → 业务操作 → prepare(全员投票) → commit(协调器决议))

27.4.1 强一致业务场景

@Service
public class OrderReservationService {

    private final CustomerGateway customerGateway;
    private final ReservationGateway reservationGateway;
    private final OrderAuditRepository auditRepository;

    @Transactional
    public ReservationResult reserve(OrderCommand command) {
        CustomerSnapshot customer = customerGateway.findCustomer(command.customerId());
        ReservationResult result = reservationGateway.reserveStock(
            command.orderId(), command.items());
        auditRepository.save(OrderAudit.success(
            command.orderId(),
            customer.customerId(),
            result.reservationId()));
        return result;
    }
}

@Transactional 触发 JtaTransactionManager 开启全局事务,JDBC 连接与 JCA 连接的 XAResource 都会被 enlist 进同一个 XID。任意一方 prepare 失败,JTA 协调器对所有资源调用 rollback,数据库与 EIS 状态同步回滚。

27.5 入站消息端点

图 27-5-1 · Message Inflow 时序(EIS 推消息 → RA 用 WorkManager 调度 → ME 包事务 → Service 处理)

27.5.1 ActivationSpec 与 EndpointManager

@Configuration
public class JcaInboundConfiguration {

    @Bean
    ActivationSpec orderStatusActivationSpec() {
        OrderStatusActivationSpec spec = new OrderStatusActivationSpec();
        spec.setDestination("ORDER.STATUS.CHANGED");
        spec.setMaxSessions(8);
        spec.setRedeliveryAttempts(5);
        return spec;
    }

    @Bean
    GenericMessageEndpointManager orderStatusEndpointManager(
            ResourceAdapter resourceAdapter,
            ActivationSpec orderStatusActivationSpec,
            MessageEndpointFactory orderStatusEndpointFactory) {
        GenericMessageEndpointManager manager = new GenericMessageEndpointManager();
        manager.setResourceAdapter(resourceAdapter);
        manager.setActivationSpec(orderStatusActivationSpec);
        manager.setMessageEndpointFactory(orderStatusEndpointFactory);
        return manager;
    }
}

GenericMessageEndpointManager 在启动时调用 resourceAdapter.endpointActivation(factory, spec),把 endpoint 工厂注册到 RA。RA 收到 EIS 消息后,通过 WorkManager 调度 Work 实例,在工作线程上调用 factory.createEndpoint().beforeDelivery() / handle() / afterDelivery()。

27.5.2 业务处理器

@Component
public class OrderStatusEndpoint implements OrderStatusListener {

    private final ProcessedEventRepository processedEvents;
    private final OrderStatusService orderStatusService;

    @Transactional
    @Override
    public void onStatusChanged(OrderStatusMessage message) {
        if (processedEvents.existsByEventId(message.eventId())) {
            log.info("skip duplicate jca message eventId={}", message.eventId());
            return;
        }
        orderStatusService.applyStatus(
            message.orderId(),
            message.status(),
            message.changedAt());
        processedEvents.save(new ProcessedEvent(message.eventId()));
    }
}

入站端点必须实现幂等。RA 在事务回滚或 ACK 失败时会重投消息,业务侧通过 eventId 唯一约束阻止重复处理。processedEvents 表与业务表共享同一个事务,保证「处理 + 标记」原子化。

27.6.1 陷阱 1:ClassLoader 隔离

+---------------------------+      +---------------------------+
| RAR ClassLoader           |      | WAR ClassLoader           |
|   sapjco3.jar             |      |   sapjco3.jar (重复)      |
|   JCoTable v3.0.18        |      |   JCoTable v3.0.18        |
+---------------------------+      +---------------------------+
              |                                    |
              v                                    v
   class JCoTable@RAR  !=  class JCoTable@WAR
   即使字节码相同, JVM 视为不同类 -> ClassCastException

修复策略:

1. EIS 客户端库放共享 ClassLoader
   WebLogic    : $DOMAIN_HOME/lib/
   WebSphere   : Shared Library
   WildFly     : JBoss Module(modules.xml)
2. 业务边界只暴露标准类型
   MappedRecord output = (MappedRecord) interaction.execute(spec, input);
   String userName = (String) output.get("USERNAME");
3. 厂商私有类型留在 Gateway 内部

27.6.2 陷阱 2:XA in-doubt 监控

@Scheduled(fixedRate = 300_000)
public void monitorInDoubtTransactions() {
    int count = jtaTransactionManager.getInDoubtTransactionCount();
    if (count > 0) {
        log.warn("found {} in-doubt XA transactions, manual intervention may be needed", count);
        alertService.sendAlert(AlertLevel.HIGH, "XA in-doubt transactions: " + count);
    }
}

prepare 阶段后协调器崩溃是 in-doubt 事务的典型来源。资源端在重启前一直锁住相关行,业务影响放大。XAResource.recover() 必须正确实现 TMSTARTRSCAN / TMENDRSCAN,否则事务管理器拿不到 in-doubt 列表。

27.6.3 陷阱 3:云原生支持

JCA RA 通常依赖以下应用服务器特性,这些在轻量容器(Docker / Kubernetes)中需要额外配置:

+-------------------+----------------+--------------------------+
| 依赖项             | 应用服务器     | 容器化处理                |
+-------------------+----------------+--------------------------+
| native 共享库     | LD_LIBRARY_PATH| 镜像内安装 + ldconfig    |
| ClassLoader 隔离  | 容器自动管理   | 手动管理 module 路径     |
| WorkManager       | 容器自动管理   | 自定义 Executor + 监控   |
| JTA 事务日志      | 持久化目录     | PersistentVolume 挂载    |
| preStop 优雅停止  | 容器钩子       | SmartLifecycle + grace   |
| 安全域            | LDAP 集成      | 显式 SecretManager       |
+-------------------+----------------+--------------------------+

27.6.4 陷阱 4:服务器绑定

应用服务器 RA 部署方式 JNDI 命名空间 迁移成本
WebSphere wsadmin 脚本 / Admin Console eis/CustomerCF 高
WebLogic JCA Resource Adapter Module eis/CustomerCF 高
WildFly RAR + standalone.xml java:/eis/CustomerCF 中
Spring Boot 嵌入式 ResourceAdapterFactoryBean 应用内 仅测试用

跨应用服务器迁移时,RA 配置、JNDI 命名空间、安全域映射、连接池参数都需要重新调整。架构层面应优先评估 EIS 是否提供 REST / OData,REST + Outbox 模式可避免 JCA 的容器绑定。

27.6.5 陷阱 5:异常透明度

@Component
public class InstrumentedCciTemplate {

    public <T> T execute(InteractionSpec spec, RecordCreator creator, RecordExtractor<T> extractor) {
        long start = System.nanoTime();
        try {
            T result = (T) cciTemplate.execute(spec, creator, extractor);
            log.info("interaction={} elapsedMs={} status=OK",
                describeSpec(spec), (System.nanoTime() - start) / 1_000_000);
            return result;
        } catch (DataAccessException e) {
            String eisCode = extractEisCode(e);
            log.error("interaction={} elapsedMs={} status=FAIL eisCode={} cause={}",
                describeSpec(spec),
                (System.nanoTime() - start) / 1_000_000,
                eisCode,
                rootCause(e));
            throw e;
        }
    }
}

ResourceException 的子类(EISSystemException、SecurityException、LocalTransactionException)携带 EIS 错误码,但 RA 默认仅在 message 中拼接。包装层需要显式提取并记录,排障时能直接定位到 EIS 端业务错误而非 Java 调用栈。

27.7.1 设计取舍

取舍 1:Spring 不应复刻完整 JCA 容器

JCA 设计前提是完整应用服务器:JNDI、安全域、WorkManager、连接池、事务日志、XA recovery、ClassLoader 隔离。Spring Boot 进程临时拼装这些组件通常仅覆盖 happy path,无法处理 in-doubt recover、重投、安全域映射等场景。生产环境的强一致 JCA 集成应部署在 WebSphere / WebLogic / WildFly 中,Spring 仅作为业务消费者。

取舍 2:CCI 标准化与厂商私有 API

CCI 提供 Connection、Interaction、MappedRecord 等标准接口,业务代码只依赖标准类型时易于切换 RA。但厂商私有 API(如 SAP JCo 的 JCoFunction、CICS 的 COMMAREA)能力更强、性能更好。折中方案:Gateway 层使用厂商 API 实现,对外只暴露 DTO;业务层只见标准类型。

取舍 3:JCA 与 REST/Outbox 的选择

场景特征 推荐方案 理由
EIS 已提供 REST / OData REST + Outbox 避免 JCA 容器绑定,云原生友好
强一致(库存、账务) JCA + JTA/XA REST 难以实现跨系统 2PC
仅查询、最终一致 REST JCA 重型基础设施成本不合理
EIS 仅支持私有协议 JCA RA 厂商通常仅提供 JCA 接入
受监管环境合规审计 JCA XA 事务日志符合审计要求

27.8 本章要点

要点 源码 / 配置依据 易错点
JCA 标准化连接 / 事务 / 安全 / 入站 / 生命周期五大合约 JSR-322 / Jakarta Connectors Spring 仅消费,不复刻容器
ResourceAdapter(RAR)由应用服务器部署 ra.xml + 厂商描述符 Spring Boot 进程不打包 RAR
JndiObjectFactoryBean 启动期查找 setLookupOnStartup(true) 运行时首次失败延迟暴露
CciTemplate 封装 CCI 出站 getConnection / createInteraction / close 异常包装为 DataAccessException
JtaTransactionManager 把 JCA + JDBC enlist 同一 XID XAResource.prepare/commit in-doubt 事务必须监控
GenericMessageEndpointManager 注册入站 endpointActivation() 端点必须幂等
连接池指标走 JMX ManagedConnectionPoolMXBean MeterBinder 桥接 Micrometer Spring 不维护池状态
ClassLoader 隔离:EIS 客户端库放共享 ClassLoader WebSphere Shared Library / WildFly Module 同名类不同 CL → ClassCastException
XA in-doubt 事务必须监控 + 人工干预 getInDoubtTransactionCount() 资源永久锁定风险
业务边界只暴露标准类型(MappedRecord / DTO) Gateway 模式 厂商私有类型穿透业务边界
EIS 已提供 REST 时优先 REST + Outbox 对比矩阵 JCA 重型基础设施成本不合理
