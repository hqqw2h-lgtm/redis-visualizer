第 34 章 · Groovy 与动态语言集成

Groovy 在 Spring 生态中处于「曾经的核心抽象、当下的边缘运行时、特定领域不可替代」三者并存的状态。其字节码与 Java 完全互操作,可保留动态分发或按 @CompileStatic 切换为静态调度;其元编程与 AST 变换支撑了 Spock、Spring Cloud Contract、Gradle DSL 三个仍然活跃的子生态。Spring Framework 的 org.springframework.scripting 包(<lang:groovy> <lang:bsh> <lang:jruby> 命名空间)、Spring Boot CLI、@Grab 依赖注解则属于历史路线,自 Spring Boot 3.x 起新项目应避免引入。

本章按「生态定位 → 三大抽象接口 → Bean DSL → 脚本动态加载 → CLI 与 @Grab → 测试 DSL → 与 Kotlin 对比 → 陷阱 → 源码入口」九层展开,目标是让读者准确判断 Groovy 在 Spring Boot 3.x 项目中的边界:既能继续使用 Spock 与 Spring Cloud Contract,又能在需要规则热更新时正确选择 Refreshable Bean 或外部规则引擎,并避免在 Native Image 与高频远程调用路径中误用 Groovy 动态特性。

34.1 Groovy 在 Spring 生态中的定位

Groovy 1.0 发布于 2007 年,2008 年 Spring 2.5 引入 <lang:groovy> 命名空间将其纳入容器一等公民。2014 年 Spring Boot 1.0 发布的 Groovy CLI 把单文件 @Grab Web 应用推到了顶峰。2020 年后 Apache 基金会接手 Groovy(包名从 org.codehaus.groovy 迁至 org.apache.groovy)、Spring Boot 推进 GraalVM Native Image、Kotlin 取代 Groovy 成为 Spring 官方一级支持的 JVM 扩展语言,Groovy 在 Spring Boot 中的角色被收敛到「测试 DSL + 规则脚本 + 构建脚本」三个明确边界。

34.1.1 仍然活跃的三大支柱

第一支柱是 Spock 测试框架。Spock 用 Groovy AST 变换重写测试代码,实现 given: when: then: where: 块、原生 Mock / Stub / Spy、数据驱动表格、字符串方法名。Spring Boot 测试模板默认未引入 Spock,但 spock-spring 与 @SpringBootTest 完全兼容,在 Java 与 Kotlin 主代码项目中按 src/test/groovy/ 引入 Spock 是常见做法。

第二支柱是 Spring Cloud Contract。该项目把消费者驱动契约用 Groovy DSL 表达(Contract.make { request { ... } response { ... } }),由 spring-cloud-contract-converter 把 DSL 转成 WireMock stub 与 Pact 文件。其 DSL 可读性、闭包嵌套、动态值占位 ($(consumer(...), producer(...))) 是 Java 注解模型难以匹敌的。Spring 官方 2024 仍在维护并发布新版本。

第三支柱是 Gradle Groovy DSL。build.gradle 是 Groovy 脚本,plugins { } dependencies { } tasks.withType { } 全部由 Groovy 闭包实现。Gradle 同时提供 Kotlin DSL(build.gradle.kts),团队按偏好二选一。Spring Boot 官方 start.spring.io 二者都支持。

34.1.2 已 deprecated 或事实弃用的路线

<lang:groovy> <lang:bsh> <lang:jruby> 三个 XML 命名空间从 Spring 2.5 起一直保留,但 Spring Framework 团队自 5.x 起未再扩展功能。BeanShell 2.0b6 自 2016 年后无新版本,JRuby 与 Spring 集成的活跃用户数极少。

Spring Boot CLI 在 Spring Boot 3.x 中仍在 spring-boot-cli 模块提供,但官方文档在「附录」位置标注其作为「快速原型工具」而非生产路径,推荐使用 start.spring.io 与 spring init 替代。@Grab 注解依赖 Apache Ivy,与现代 Maven Central 镜像、企业 Nexus 配置存在版本协调成本。

org.springframework.scripting 包提供的 ScriptFactoryPostProcessor 与 RefreshableScriptTargetSource 仍然可用,Spring Framework 6.x 未标记 @Deprecated,但官方文档未推荐其作为主流模式。运行时规则热更新场景在现代项目中通常由「Java 评估器 + JSON / YAML 配置」「外部规则引擎(Drools / Easy Rules)」「Sidecar Lambda 函数」替代。

34.1.3 Groovy 与 Kotlin 的角色分配

Spring Framework 5.0 (2017) 把 Kotlin 升为一级支持语言,内置 RouterFunctionDsl BeanDefinitionDsl MockMvc Kotlin 扩展;Spring Framework 6.0 (2022) 与 Spring Boot 3.0 全面拥抱 Kotlin 协程、Java 17 record、Jakarta EE 9。在该背景下,Groovy 的「动态特性」与 Kotlin 的「编译期安全 + 协程」职责出现明确分工:Kotlin 接管业务代码,Groovy 守住测试 DSL 与构建脚本两块阵地。

34.2 Spring 动态语言三大抽象

org.springframework.scripting 包定义了三个核心接口,把「脚本来源 → 编译为对象 → 求值为结果」拆为三层职责。理解这三层是阅读 ScriptFactoryPostProcessor 与 RefreshableScriptTargetSource 源码的前置条件。

34.2.1 ScriptFactory:语言适配器

ScriptFactory 是每种脚本语言的适配器,负责把 ScriptSource 编译为目标对象(对 Groovy 是 Class,对 BeanShell 是 Interpreter 解释执行的 proxy)。

public interface ScriptFactory {
    String getScriptSourceLocator();
    Class<?>[] getScriptInterfaces();
    boolean requiresConfigInterface();

    Object getScriptedObject(ScriptSource source, Class<?>... actualInterfaces)
            throws IOException, ScriptCompilationException;

    Class<?> getScriptedObjectType(ScriptSource source)
            throws IOException, ScriptCompilationException;

    boolean requiresScriptedObjectRefresh(ScriptSource source);
}

三种内置实现各自对应一种语言:GroovyScriptFactory 编译 .groovy 为字节码,BshScriptFactory 创建 BeanShell Interpreter 包装,JRubyScriptFactory 创建 JRuby RubyContainer。其中只有 GroovyScriptFactory 在 Spring Boot 3.x 项目中仍偶尔出现。

34.2.2 ScriptSource:脚本物理来源

ScriptSource 抽象脚本的物理来源,提供「读取内容 + 检测变更 + 推荐类名」三个职责。

public interface ScriptSource {
    String getScriptAsString() throws IOException;
    boolean isModified();
    String suggestedClassName();
}

两个内置实现:ResourceScriptSource 包装 Spring Resource(支持 classpath: file: url: 协议);StaticScriptSource 包装内联字符串。isModified() 是热更新机制的核心钩子:ResourceScriptSource 通过比对 Resource.lastModified() 时间戳判断,StaticScriptSource 通过 setScript() 调用累计版本号判断。

34.2.3 ScriptEvaluator:表达式求值

ScriptEvaluator 与 ScriptFactory 的差异在于:ScriptFactory 创建持久的「脚本化对象」(可被注入为 Bean),ScriptEvaluator 执行一次性「脚本表达式」并返回结果。

public interface ScriptEvaluator {
    Object evaluate(ScriptSource script) throws ScriptCompilationException;
    Object evaluate(ScriptSource script, Map<String, Object> arguments)
            throws ScriptCompilationException;
}

GroovyScriptEvaluator 直接调用 GroovyShell.evaluate();StandardScriptEvaluator 通过 JSR-223 ScriptEngineManager.getEngineByName(...) 适配任意符合规范的引擎。在 Spring Boot 项目中,前者用于「Groovy 闭包式规则求值」,后者已被 GraalVM Polyglot API 在多语言场景中替代。

34.3 GroovyBeanDefinitionReader 与 Bean DSL

GroovyBeanDefinitionReader 是 Spring 3.0 引入的 BeanDefinitionReader 实现,把 Groovy 闭包语法翻译为 BeanDefinition 注册到 BeanDefinitionRegistry。其与 XmlBeanDefinitionReader 同族,产物等价,语法风格不同。

34.3.1 Bean DSL 基本语法

beans {
    dataSource(HikariDataSource) {
        jdbcUrl = 'jdbc:postgresql://localhost:5432/orders'
        username = 'app'
        password = 'secret'
        maximumPoolSize = 20
    }

    orderRepository(JdbcOrderRepository) {
        dataSource = ref('dataSource')
    }

    orderService(OrderServiceImpl) {
        orderRepository = ref('orderRepository')
        maxRetryAttempts = 3
    }

    if (System.getenv('PROFILE') == 'dev') {
        emailService(MockEmailService)
    } else {
        emailService(SmtpEmailService) {
            host = 'smtp.example.com'
            port = 587
        }
    }
}

三条核心规则:name(Class) { ... } 等价于 <bean id="name" class="...">;property = value 等价于 <property name="property" value="value"/>;ref('beanName') 等价于 <ref bean="beanName"/>。

34.3.2 加载入口与运行时

Spring Boot 应用加载 Groovy Bean DSL 通常通过 ApplicationContextInitializer:

@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        new SpringApplicationBuilder()
                .sources(Application.class)
                .initializers((ApplicationContextInitializer<GenericApplicationContext>) ctx -> {
                    GroovyBeanDefinitionReader reader = new GroovyBeanDefinitionReader(ctx);
                    reader.loadBeanDefinitions("classpath:beans.groovy");
                })
                .run(args);
    }
}

加载完成后 BeanDefinitionRegistry 中出现的定义结构与 XmlBeanDefinitionReader 完全一致,后续的 Bean 实例化、依赖注入、生命周期回调走标准 Spring 路径,与定义来源无关。

34.3.4 与 Java Config 共存

GroovyBeanDefinitionReader 注册的定义与 @Configuration 类注册的定义并存于同一 BeanFactory,无优先级差异。混用时常见做法是把核心生产代码放 @Configuration,把租户 / 环境相关的可变配置放 .groovy 脚本,通过 Initializer 加载。

34.4 Groovy 脚本动态加载

Spring 提供两条独立的脚本加载路径:GroovyShell / GroovyClassLoader 用于命令式编译执行;ScriptFactoryPostProcessor + RefreshableScriptTargetSource 用于声明式 Bean 化与热更新。

34.4.1 GroovyShell 与 GroovyClassLoader

GroovyShell shell = new GroovyShell();
Script script = shell.parse("return amount * (1 - discount)");
Binding binding = new Binding();
binding.setVariable("amount", new BigDecimal("1000"));
binding.setVariable("discount", new BigDecimal("0.15"));
script.setBinding(binding);
Object result = script.run();

GroovyShell.parse(String) 把脚本编译为 Script 子类的字节码,在 GroovyClassLoader 中加载。Script 实例可缓存,重复 run() 仅花费方法调用成本。每次新建 GroovyShell 都会创建独立 GroovyClassLoader,跨实例的脚本类型互不兼容(参见 34.8.2)。

GroovyClassLoader.parseClass(String) 提供更底层的入口,直接返回 Class<?>:

GroovyClassLoader gcl = new GroovyClassLoader();
Class<?> ruleClass = gcl.parseClass(scriptText);
PricingRule rule = (PricingRule) ruleClass.getDeclaredConstructor().newInstance();
BigDecimal price = rule.evaluate(order);

34.4.2 Refreshable Bean 机制

ScriptFactoryPostProcessor 在 BeanFactory 中扫描所有 ScriptFactory 类型的 Bean 定义,把它们替换为指向「脚本化对象」的 JDK 动态代理。当 refreshCheckDelay > 0 时,代理内部持有 RefreshableScriptTargetSource,周期性调用 ScriptSource.isModified(),若返回 true 则触发重编译并替换目标对象。

@Configuration
public class ScriptingConfig {

    @Bean
    public ScriptFactoryPostProcessor scriptFactoryPostProcessor() {
        ScriptFactoryPostProcessor processor = new ScriptFactoryPostProcessor();
        processor.setDefaultRefreshCheckDelay(5_000);
        return processor;
    }

    @Bean
    public GroovyScriptFactory pricingEngineFactory() {
        return new GroovyScriptFactory("file:/opt/rules/pricing.groovy");
    }
}

业务侧通过 Java 接口注入,完全隔离脚本细节:

@Service
public class CheckoutService {

    private final PricingEngine pricingEngine;

    public CheckoutService(@Qualifier("pricingEngineFactory") PricingEngine pricingEngine) {
        this.pricingEngine = pricingEngine;
    }

    public BigDecimal checkout(Cart cart, Customer customer) {
        return pricingEngine.calculatePrice(cart, customer);
    }
}

34.4.3 编译缓存与 ClassLoader 隔离

每次脚本变更都会创建新的 GroovyClassLoader,旧的 Class 失去强引用后由 GC 回收。这种隔离避免了「同名类的字节码定义冲突」问题(JVM 不允许同一 ClassLoader 加载两个 defineClass 同名的类),代价是同名类在不同版本之间不能通过 instanceof 判定。该限制即陷阱 34.8.2 的根源。

ASCII 帧:脚本热更新过程

帧 0(t=0s):
    Proxy ──> [v1 ScriptObject]@CL-A    GroovyClassLoader-A
    Source.lastModified = 1714600000

帧 1(t=5s, 文件未变):
    Timer 调用 isModified
    Source.lastModified == 1714600000  → false
    Proxy 仍指向 v1@CL-A

帧 2(t=10s, 文件被修改, lastModified = 1714600010):
    Timer 调用 isModified
    Source.lastModified != 1714600000  → true
    Factory.getScriptedObject(source)
        Loader-B = new GroovyClassLoader
        Loader-B.parseClass(scriptText)  → v2 Class
    Proxy ──> [v2 ScriptObject]@CL-B    GroovyClassLoader-B
    [v1 ScriptObject]@CL-A 失去引用

帧 3(t=11s):
    GC 回收 GroovyClassLoader-A 与 v1 Class
    业务调用看到 v2 行为
    新接收的入参用 v2 类型,旧入参引用仍存活

34.4.4 何时不应使用 Refreshable Bean

热更新带来的代价集中在四方面:跨版本 ClassCastException(陷阱 34.8.2)、动态分发性能损耗(陷阱 34.8.3)、堆栈追踪噪声(陷阱 34.8.1)、Native Image 不可用(陷阱 34.8.6)。在以下场景应避免:核心业务路径的高频规则评估、需要 Native Image 编译的服务、跨服务边界的分布式规则、多团队并发修改的规则集。这类场景应改用「Java 评估器 + JSON 规则」「外部规则引擎(Drools / Easy Rules)」或「Sidecar 微服务」。

34.5 @Grab 与 Spring Boot CLI

@Grab 是 Groovy Grape 模块提供的注解,在脚本顶部声明 Maven 依赖,Grape 在编译期解析 Apache Ivy XML 并下载 jar 到本地仓库。Spring Boot CLI 在此基础上预设了 spring-boot-starter-* 的隐式依赖,允许单文件 Web 应用。

34.5.1 @Grab 与 Grape

@Grab(group = 'org.apache.commons', module = 'commons-lang3', version = '3.14.0')
import org.apache.commons.lang3.StringUtils

println StringUtils.repeat('=', 60)
println 'Grape resolved commons-lang3 from local Ivy cache'

Grape 默认仓库为 Maven Central,缓存目录为 ~/.groovy/grapes/。企业环境通过 ~/.groovy/grapeConfig.xml 切换为内部 Nexus / Artifactory:

<ivysettings>
    <settings defaultResolver="downloadGrapes"/>
    <resolvers>
        <chain name="downloadGrapes">
            <ibiblio name="enterprise-nexus"
                     root="https://nexus.example.com/repository/maven-public/"
                     m2compatible="true"/>
        </chain>
    </resolvers>
</ivysettings>

34.5.2 Spring Boot CLI 状态评估

Spring Boot CLI 在 Spring Boot 3.x 中仍可通过 SDKMAN 或手动下载安装,支持 spring init 项目脚手架与 spring run 单文件运行。官方文档把它定位为「快速原型 + 学习工具」,主要替代品包括:start.spring.io Web 界面、spring-boot-maven-plugin run、IDE 内嵌 Spring Initializr。

34.5.3 @Grab 的现代替代

@Grab 单文件场景的现代替代是 JBang(https://www.jbang.dev/)。JBang 支持 Java / Groovy / Kotlin 单文件,语法 //DEPS group:artifact:version 声明依赖,与 Spring Boot 集成的 catalog 提供 jbang spring-cli@spring-projects-experimental ... 命令。JBang 的活跃度高于 Spring Boot CLI,被 Quarkus 与多个 Spring 实验项目采用。

34.6 Spock 与 Spring Cloud Contract

Spock 与 Spring Cloud Contract 是 Groovy 在 Spring Boot 3.x 项目中最稳固的两块阵地。Spock 用 Groovy AST 变换重写测试代码,Spring Cloud Contract 用 Groovy 闭包表达 HTTP 契约。两者的共同点是「Groovy 闭包 + 元编程」与「Java 注解 + 反射」相比在 DSL 表达力上具有结构性优势。

34.6.1 Spock 基础语法

class OrderServiceSpec extends Specification {

    OrderService orderService = new OrderService()

    def 'should calculate order total with 8 percent tax'() {
        given: 'an order with two items totaling 55 USD'
        Order order = new Order(items: [
            new OrderItem(name: 'Widget', price: 30.00, quantity: 1),
            new OrderItem(name: 'Gadget', price: 25.00, quantity: 1)
        ])

        when: 'service computes the total'
        BigDecimal total = orderService.calculateTotal(order)

        then: 'total includes the configured tax rate'
        total == 59.40
    }

    def 'should reject creation when items list is empty'() {
        given: 'an empty order'
        Order order = new Order(items: [])

        when: 'service attempts creation'
        orderService.createOrder(order)

        then: 'validation exception is raised with explanatory message'
        ValidationException ex = thrown()
        ex.message.contains('empty')
    }
}

given: when: then: 三段不是注释,而是 Spock AST 变换识别的代码块标记。Spock 编译期把每个块翻译为带有上下文信息的字节码,失败时报错可直接指向块名与表达式。thrown() 与 notThrown() 替代 assertThrows,可读性更高。字符串方法名通过 def 'description'() 语法,无需额外 @DisplayName 注解。

34.6.2 Mock / Stub / Spy 三态

class CheckoutServiceSpec extends Specification {

    OrderRepository orderRepository = Stub()
    EmailService emailService = Mock()
    AuditLogger auditLogger = Spy(new AuditLogger())

    CheckoutService service = new CheckoutService(orderRepository, emailService, auditLogger)

    def 'should send confirmation email after successful checkout'() {
        given: 'repository returns a saved order'
        orderRepository.save(_) >> new Order(id: 1L, name: 'Customer C1001', email: 'c1001@example.com')

        when: 'checkout is invoked with a new cart'
        Order order = service.checkout(new Cart(customerId: 1L))

        then: 'a single confirmation email is dispatched to the customer'
        1 * emailService.sendConfirmation('c1001@example.com', _ as Order)

        and: 'audit log records the order creation event'
        1 * auditLogger.log({ String entry -> entry.contains('ORDER_CREATED') })

        and: 'returned order carries the persisted identifier'
        order.id == 1L
    }
}

三个角色的语义在 Spock 中严格区分:Stub 只回答返回值,不参与交互校验;Mock 同时回答返回值与校验调用次数 / 参数;Spy 包装真实对象,可选择性覆盖部分方法。1 * service.method(args) 是交互校验语法,数字 0 表示「禁止调用」。

34.6.3 数据驱动:where 块

class PricingTierSpec extends Specification {

    PricingService pricingService = new PricingService()

    def 'discount calculation: #scenario'() {
        expect: 'discount matches the configured tier rule'
        pricingService.calculateDiscount(orderAmount, customerTier) == expectedDiscount

        where: 'cross product of order amount and customer tier'
        scenario               | orderAmount | customerTier | expectedDiscount
        'small basic'          | 50.0        | 'BASIC'      | 0.0
        'medium basic'         | 200.0       | 'BASIC'      | 0.05
        'large basic'          | 1000.0      | 'BASIC'      | 0.10
        'small silver'         | 50.0        | 'SILVER'     | 0.05
        'medium silver'        | 200.0       | 'SILVER'     | 0.10
        'large silver'         | 1000.0      | 'SILVER'     | 0.15
        'small gold'           | 50.0        | 'GOLD'       | 0.10
        'medium gold'          | 200.0       | 'GOLD'       | 0.15
        'large gold'           | 1000.0      | 'GOLD'       | 0.20
    }
}

where: 表的每行展开为独立用例,Spock 报告中显示 9 条结果,失败时能精确定位到 #scenario 字符串。@ParameterizedTest 的 JUnit 5 等价写法需要 @CsvSource 或 @MethodSource,行数与可读性均落后。

34.6.4 与 @SpringBootTest 集成

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class OrderApiIntegrationSpec extends Specification {

    @Autowired
    TestRestTemplate restTemplate

    @Autowired
    OrderRepository orderRepository

    @SpringBean
    EmailService emailService = Mock()

    def setup() {
        orderRepository.deleteAll()
    }

    def 'POST orders should create record and dispatch email'() {
        given: 'a valid order request payload'
        Map request = [name: 'Customer C1001', email: 'c1001@example.com']

        when: 'request is posted to the order API'
        ResponseEntity<Map> response = restTemplate.postForEntity('/api/orders', request, Map)

        then: 'response indicates resource creation with 201 status'
        response.statusCode.value() == 201
        response.body.name == 'Customer C1001'

        and: 'a single confirmation email is sent to the captured address'
        1 * emailService.sendConfirmation('c1001@example.com', 'Customer C1001')

        and: 'order is persisted in the repository'
        orderRepository.count() == 1L
    }
}

@SpringBean 是 spock-spring 提供的注解,把 Mock 对象注入 ApplicationContext 替换原 Bean,等价于 Mockito 的 @MockBean 但语义更精确。

34.6.5 Spring Cloud Contract Groovy DSL

Contract.make {
    description 'should return order details by identifier'

    request {
        method GET()
        url '/api/orders/O1001'
        headers {
            accept applicationJson()
        }
    }

    response {
        status OK()
        headers {
            contentType applicationJson()
        }
        body([
            id    : 1001,
            name  : $(consumer('Customer C1001'), producer(regex('[A-Z][a-z]+ [A-Z][0-9]+'))),
            email : $(consumer('c1001@example.com'), producer(regex('.+@.+\\..+'))),
            status: 'ACTIVE'
        ])
    }
}

$(consumer(...), producer(...)) 是 Spring Cloud Contract 的核心 DSL 元素:消费者侧使用固定值生成 stub,生产者侧使用正则保证多种数据组合都满足契约。该模式无法用 YAML 或 Java DSL 等价表达。

34.8 陷阱与限制

图 34-8-1 · 六大 Groovy 陷阱速查思维导图(动态/ClassLoader/反射/MissingMethod/迁移/Native)

34.8.1 陷阱一:动态分发堆栈噪声

Groovy 默认动态分发,每次方法调用经过 MetaClassImpl.invokeMethod CachedSAMClass CallSiteArray 多层中间件。生产环境抛出的异常堆栈包含 30 行以上 Groovy 内部帧,真正的业务行通常埋在中间。

修复路径:对生产规则脚本使用 @CompileStatic 减少分发层数;在脚本顶部加入结构化日志,在抛异常前打印脚本名 / 版本 / 输入摘要;为 Refreshable Bean 注入 BeanPostProcessor 拦截脚本边界异常并补充上下文。

34.8.2 陷阱二:ClassLoader 隔离的同名异类 ClassCastException

每次脚本变更触发新的 GroovyClassLoader。旧版本对象传给新版本方法时,JVM 视为「同名但不同 ClassLoader」,抛出 ClassCastException: com.example.Order cannot be cast to com.example.Order。

修复路径:让脚本对象只暴露父 ClassLoader 加载的接口(放在 src/main/java),业务侧只引用接口类型;脚本热更新事件触发后清理所有持有脚本类型的缓存(Spring ApplicationCacheManager Caffeine Map<Order, ?>);跨服务边界传输时使用 DTO(JSON / Protobuf 序列化反序列化绕过 ClassLoader 身份)。

34.8.3 陷阱三:反射密集脚本的性能损耗

Groovy 动态模式的方法调用经过 MetaClass.invokeMethod 与 CallSite 缓存,JMH 基准显示遍历 1000 元素集合时:Java 0.8 ms / op、Groovy 动态 6.2 ms / op、Groovy @CompileStatic 1.1 ms / op、GraalVM Polyglot JS 3.4 ms / op。在 10000 TPS 高频路径上,动态 Groovy 的尾延迟相比 Java 恶化 5 至 8 倍。

修复路径:核心路径用 Java 或 @CompileStatic Groovy;Groovy 仅做「决策」(返回应用什么策略)而非「执行」(批量遍历);编译产物纳入热点缓存,避免每次请求都重编译。

34.8.4 陷阱四:类型擦除导致的运行时 MissingMethodException

class OrderProcessor {
    def process(order) {
        def items = order.getItems()
        items.each { /* ... */ }
    }
}

Java 端把 getItems() 重命名为 getLineItems() 后,Groovy 编译期不报错,运行时才抛 MissingMethodException: No signature of method: com.example.Order.getItems()。该问题在 Java 编译期被静态类型阻挡,在动态 Groovy 中只能被运行时流量或单元测试发现。

修复路径:对生产脚本默认 @CompileStatic;CI 流水线把 Groovy 脚本作为 compileTestGroovy 的一部分编译;Java 端 API 变更通过 Spring Cloud Contract 或脚本契约测试同步验证。

34.8.5 陷阱五:Groovy 4 包名迁移与 Spock 兼容

Groovy 4.0 (Apache 接管) 把包名从 org.codehaus.groovy 迁移至 org.apache.groovy。Spring Boot 3.x BOM 锁定 Groovy 4.0+,旧 Spock 2.3-groovy-3.0 引用 org.codehaus.groovy.ast.ClassHelper.makeCached(Class) 在新版本不存在,启动测试时抛 NoSuchMethodError。

依赖矩阵速查:
Spring Boot 2.x  →  Groovy 2.5 / 3.0  →  Spock 2.x-groovy-3.0
Spring Boot 3.x  →  Groovy 4.0+        →  Spock 2.4-M4-groovy-4.0 起

修复路径:升级 Spring Boot 3.x 时同步升级 Spock 至 spock-core:2.4-M4-groovy-4.0 与 spock-spring:2.4-M4-groovy-4.0;groovy:4.0.21 显式声明在 testImplementation;CI 加入「无 org.codehaus.groovy 依赖」断言。

34.8.6 陷阱六:GraalVM Native Image 与 Groovy 动态特性不兼容

GraalVM Native Image 基于「Closed-World Assumption」,在 AOT 阶段确定所有可达类。Groovy 的动态特性 — GroovyClassLoader.parseClass(运行时编译)、MetaClass.invokeMethod(动态分发)、ProxyGenerator(运行时代理)、ExpandoMetaClass(运行时方法注入)、Eval.me / GroovyShell.evaluate(字符串求值)— 全部依赖运行时类加载。Native Image 构建时直接抛 Unsupported method groovy.lang.GroovyClassLoader.parseClass(String) is reachable。

修复路径:Native Image 优先项目用 @CompileStatic Groovy(产物等价 Java 字节码);运行时规则迁移到 JSON / YAML 数据驱动 + Java 评估器;不可避免的脚本运行时迁移到独立 JVM 模式 Sidecar 服务。
