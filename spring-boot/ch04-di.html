第 4 章 · 依赖注入机制

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

依赖注入(Dependency Injection,DI)是 IoC(Inversion of Control,控制反转)在"对象获取依赖"维度上的具体实现。Martin Fowler 在 2004 年的论文 "Inversion of Control Containers and the Dependency Injection pattern" 中首次明确这两个概念的区分,并归纳出 Constructor Injection、Setter Injection、Interface Injection 三类形式。

Spring 的 DI 解析由 AutowiredAnnotationBeanPostProcessor(AABPP)与 DefaultListableBeanFactory.resolveDependency() 协作完成。AABPP 在 Bean 实例化阶段扫描注入点构建 InjectionMetadata,在属性填充阶段调用 BeanFactory 解析每个依赖。BeanFactory 的解析算法按"类型 → @Qualifier → @Primary → @Priority → 名称"五层优先级裁决候选 Bean。

本章按"DI 哲学 → 三种注入方式 → 解析算法 → JSR-330 兼容 → 高级注入(@Bean / FactoryBean / @Lookup)→ 条件注入 → 陷阱清单"七层组织。每一层均给出 Mermaid 图、源码入口、异常输出与修复路径,使依赖注入从"能用"上升到"可调试、可重构、可演进"。

4.1 IoC 与 DI 的关系

图 4-1-1 · IoC 与 DI 的层次关系(IoC = 总原则,DI 是其一种实现)

IoC 是设计原则,涵盖依赖管理、回调、事件驱动等多种模式。DI 是 IoC 在"对象获取依赖"这一维度上的具体实现。Field Injection 不在 Fowler 原始分类中,是 Spring 通过反射扩展的形式。

表 4-1-1:Fowler 三分类

注入类型 机制 Spring 支持 现状
Constructor Injection 构造器参数传入依赖 完全支持 主流选择,推荐使用
Setter Injection setter 方法传入依赖 完全支持 适用于可选依赖
Interface Injection 实现框架接口接受依赖 不直接支持 已被淘汰

Interface Injection 要求被注入类实现框架定义的接口(如 Avalon 的 Serviceable),造成框架强耦合,违背 DI 初衷。Spring 从未采用此模式。

图 4-1-2 · 好莱坞原则:new() 耦合 vs DI 解耦

好莱坞原则:"Don't call us, we'll call you"。应用代码声明所需依赖,框架在正确时机交付,而非应用代码主动获取。

表 4-1-2:三大 DI 框架对比

维度 Spring DI Google Guice CDI (Jakarta)
配置 注解 + JavaConfig + XML Module 类 + 注解 注解 + beans.xml
默认 Scope Singleton 无(每次 new) @Dependent(每次 new)
AOP 内建(CGLIB / JDK Proxy) guice-aop 内建(拦截器)
条件注入 @Conditional 家族 无 @Alternative + @Priority
延迟 @Lazy / ObjectProvider Provider Instance
JSR-330 完全支持 参考实现 JSR-330 + JSR-365
启动速度 中(反射 + 代理) 快 中
生态规模 最大 中(Google 内部广泛使用) Jakarta EE 服务器标配

拓展案例:Spring Framework 自身迁移

Spring Framework 5.0 起系统性将核心组件从 setter 注入迁移到构造器注入。DispatcherServlet、AbstractApplicationContext 的多数内部依赖在 5.x 改为构造器传入。Juergen Hoeller 在多个场合明确构造器注入是 Spring 5+ 的推荐方式。Spring Boot 所有官方 starter 源码统一采用构造器注入。

4.3 @Autowired 解析算法

图 4-3-1 · AABPP 处理管线(收集元数据 + populateBean 注入 · 12 阶段)

表 4-3-1:AABPP 两阶段

@Configuration
public class DataSourceConfig {

    @Bean
    @Primary
    public DataSource primaryDataSource() {
        return DataSourceBuilder.create()
            .url("jdbc:mysql://primary:3306/app").build();
    }

    @Bean
    public DataSource replicaDataSource() {
        return DataSourceBuilder.create()
            .url("jdbc:mysql://replica:3306/app").build();
    }
}

@Service
public class OrderService {
    private final DataSource ds;

    public OrderService(DataSource ds) {
        this.ds = ds;
    }
}

@Service
public class ReportService {
    private final DataSource replicaDs;

    public ReportService(@Qualifier("replicaDataSource") DataSource replicaDs) {
        this.replicaDs = replicaDs;
    }
}

@Qualifier 优先级高于 @Primary。两者均无时按字段名 / 参数名匹配 Bean 名作为 fallback。

图 4-3-4 · 集合注入类型映射(List 按 @Order · Map 用 BeanName)

List<T> 注入按 @Order / @Priority 排序,Map<String, T> 的 key 为 Bean 名,value 为 Bean 实例。

表 4-3-3:可选依赖三种处理方式

方式 缺失时行为 推荐度
@Autowired(required = false) 注入 null 一般
Optional 构造器参数 注入 Optional.empty 适用于明确建模可选性
ObjectProvider 提供 ifAvailable 等丰富 API 推荐(Spring 推荐) 
private final ObjectProvider<MetricsCollector> metricsProvider;

public MyService(ObjectProvider<MetricsCollector> metricsProvider) {
    this.metricsProvider = metricsProvider;
}

public void doWork() {
    metricsProvider.ifAvailableOrElse(
        m -> m.record("work.done"),
        () -> System.out.println("metrics absent"));
}

拓展案例:Spring Cloud @LoadBalanced

@LoadBalanced 是基于 @Qualifier 的自定义限定符注解。RestTemplate Bean 添加 @LoadBalanced 后,Spring Cloud LoadBalancer 拦截器在该实例上启用客户端负载均衡。

@Target({ElementType.FIELD, ElementType.PARAMETER, ElementType.METHOD})
@Retention(RetentionPolicy.RUNTIME)
@Qualifier
public @interface LoadBalanced { }

@Bean
@LoadBalanced
public RestTemplate restTemplate() {
    return new RestTemplate();
}

拓展案例:CDI 迁移到 Spring Boot

Jakarta EE 应用使用 CDI 作为 DI 容器。迁移到 Spring Boot 时,使用 @Inject / @Named 的领域层可无缝运行。需要替换的是 CDI 特有的 @Alternative(改用 @Primary 或 @Profile)、Instance<T>(改用 ObjectProvider<T>)、@Produces(改用 @Bean)。

4.5 高级注入

图 4-5-1 · @Bean 方法参数解析时序(等同构造器注入语义)
@Configuration
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(
            HttpSecurity http,
            JwtDecoder jwtDecoder,
            CustomAuthEntryPoint entryPoint) throws Exception {

        return http
            .oauth2ResourceServer(o -> o.jwt(j -> j.decoder(jwtDecoder)))
            .exceptionHandling(e -> e.authenticationEntryPoint(entryPoint))
            .build();
    }
}

@Bean 方法的参数自动从容器解析,等同构造器注入语义。该模式是组装多依赖复杂 Bean 的常用方式。

图 4-5-2 · FactoryBean 工作模式(getBean 默认拿 getObject() 返回值)
public class HttpClientFactoryBean implements FactoryBean<HttpClient> {

    private Duration connectTimeout = Duration.ofSeconds(5);

    @Override
    public HttpClient getObject() {
        return HttpClient.newBuilder()
            .connectTimeout(connectTimeout)
            .followRedirects(HttpClient.Redirect.NORMAL)
            .build();
    }

    @Override
    public Class<?> getObjectType() { return HttpClient.class; }

    @Override
    public boolean isSingleton() { return true; }
}

FactoryBean 适用于框架级集成。Spring Data 的 RepositoryFactoryBean、MyBatis-Spring 的 MapperFactoryBean 是典型案例。& 前缀获取 FactoryBean 自身(getBean("&myFactory"))。

图 4-5-3 · @Lookup 方法注入(每次调用都从容器拿新 prototype)
@Service
public abstract class CheckoutService {

    @Lookup
    protected abstract ShoppingCart createCart();

    public Order checkout(Order order) {
        ShoppingCart cart = createCart();
        return new Order(cart);
    }
}

checkout order=u-1001, cart=7f2a91c0
checkout order=u-1001, cart=44b0d8e3

@Lookup 通过 CGLIB 在运行时重写方法体。每次调用从容器获取新 prototype 实例。替代方案是 ObjectProvider<T>,后者推荐度更高(无需抽象类)。

表 4-5-1:三种自定义注入对比

方式 适用场景 优点 局限
@Bean 方法 组装复杂 Bean,需要其他 Bean 作为参数 直观,等同构造器 仅在 @Configuration 类中
FactoryBean 框架级集成,需要复杂创建逻辑或代理 可获取工厂自身,可控制 isSingleton 略显繁琐,日常 Bean 不必使用
@Lookup Singleton 内部每次需要新 prototype CGLIB 重写,保证每次新实例 需抽象类或可被代理的非 final 类,推荐改用 ObjectProvider

4.7 陷阱与限制

表 4-7-1:七大高频陷阱

编号 陷阱 现象 修复策略
1 字段注入难以单元测试 测试需反射或启动容器 改用构造器注入
2 构造器循环依赖 BeanCurrentlyInCreationException 重构 / @Lazy / 事件驱动
3 @Autowired 在 static 字段无效 字段为 null 构造器桥接 / 避免 static 工具类
4 多候选 Bean 无 @Qualifier NoUniqueBeanDefinitionException @Primary / @Qualifier / List / 参数名
5 @Value 字段在构造器中为 null NullPointerException @Value 放构造器参数上
6 Prototype 注入 Singleton 永远同一实例 prototype 变成事实上的 singleton ObjectProvider / @Lookup
7 @PostConstruct 父类先于子类执行 父类 init 依赖子类状态导致 NPE 模板方法模式 / abstract 钩子

拓展案例:ApplicationContextRunner 验证注入链

ApplicationContextRunner 不启动完整 Web 容器,创建最小 ApplicationContext 验证 @Primary 与 @Qualifier 行为。该工具适合作为团队约定的注入回归测试。

class AutowireResolutionTests {

    private final ApplicationContextRunner runner =
        new ApplicationContextRunner()
            .withBean("emailSender", NotificationSender.class, EmailSender::new)
            .withBean("smsSender", NotificationSender.class, SmsSender::new);

    @Test
    void qualifierWinsOverPrimary() {
        runner.withBean(DefaultSenderConfig.class)
            .withBean(AlertService.class)
            .run(context -> {
                AlertService service = context.getBean(AlertService.class);
                assertThat(service.sender()).isInstanceOf(SmsSender.class);
            });
    }
}

4.末 本章要点

  • IoC 是设计原则,DI 是其在"对象获取依赖"维度的具体实现。Martin Fowler 在 2004 年的论文中明确两者关系并归纳出 Constructor、Setter、Interface 三类注入形式。

  • 三种注入方式的优劣明确:构造器注入支持 final、完全初始化、纯 POJO 测试;Setter 注入适用于可选依赖;字段注入隐藏依赖数量,Spring 团队不推荐。

  • AutowiredAnnotationBeanPostProcessor 处理 @Autowired 注入分两阶段:postProcessMergedBeanDefinition 扫描注入点构建 InjectionMetadata,postProcessProperties 反射赋值。DefaultListableBeanFactory.resolveDependency 按"类型 → @Qualifier → @Primary → @Priority → 名称"五层裁决。

  • JSR-330 注解(@Inject、@Named、Provider、@Singleton)由 Spring 完全支持,适用于框架无关的领域层与跨 DI 容器的公共库。

  • @Bean 方法参数自动注入是组装多依赖复杂 Bean 的常用方式。FactoryBean<T> 适用于框架级集成与代理生成。@Lookup 通过 CGLIB 重写方法体解决 Singleton 注入 Prototype 失效问题,推荐替代为 ObjectProvider<T>。

  • 条件注入由 @Profile 与 @Conditional 家族实现。Bean 选择按 @Profile → @Conditional → @ConditionalOnClass → @ConditionalOnMissingBean → @ConditionalOnProperty 五层判定。

  • 七大陷阱:字段注入难测试、构造器循环依赖、static 字段无效、多候选无 @Qualifier、@Value 字段在构造器中为 null、Prototype 注入 Singleton 失效、@PostConstruct 父类先于子类执行。每类陷阱有明确的修复路径。

  • Spring Boot 2.6 起默认禁止循环依赖,生产代码应优先通过重构提取 SharedLogic 或事件驱动消除循环,而非依赖 spring.main.allow-circular-references=true。
