第 29 章 · Resource 资源抽象与 IO

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

org.springframework.core.io.Resource 是 Spring 对低级资源访问的核心抽象,把 classpath、文件系统、URL、Servlet 上下文、字节数组、字节流等多种资源源统一为一组接口。ResourceLoader 负责按字符串前缀(classpath: / file: / http: / 无前缀)加载资源,ResourcePatternResolver 在此基础上支持 Ant 风格通配符匹配多个资源。

ApplicationContext 自身实现 ResourceLoader 与 ResourcePatternResolver 接口。PathMatchingResourcePatternResolver 是默认实现,支持 classpath*:**/*.xml 这类跨 JAR 包的批量扫描。AntPathMatcher 提供 **(任意目录层级)、*(任意单层)、?(单字符)三类通配符。

本章按「Resource 接口 → ResourceLoader → ApplicationContext 集成 → ResourcePatternResolver → IO 实战 → 嵌入式服务器视角 → 陷阱 → 源码深潜」展开,聚焦配置加载、模板引擎、批处理、文件上传下载等高频场景。

29.1.1 InputStreamSource 与 Resource

package org.springframework.core.io;

public interface InputStreamSource {
    InputStream getInputStream() throws IOException;
}

public interface Resource extends InputStreamSource {
    boolean exists();
    default boolean isReadable() { return exists(); }
    default boolean isOpen() { return false; }
    default boolean isFile() { return false; }
    URL getURL() throws IOException;
    URI getURI() throws IOException;
    File getFile() throws IOException;
    long contentLength() throws IOException;
    long lastModified() throws IOException;
    Resource createRelative(String relativePath) throws IOException;
    String getFilename();
    String getDescription();
}

public interface WritableResource extends Resource {
    default boolean isWritable() { return true; }
    OutputStream getOutputStream() throws IOException;
}

isOpen() 返回 true 的资源(如 InputStreamResource)只能读取一次,重复调用 getInputStream() 抛异常。isFile() 区分了文件系统资源与其他形态(JAR 内部资源、字节数组、网络资源都不是文件)。

29.2.1 ClassPathResource

ClassPathResource resource = new ClassPathResource("config/app.properties");
try (InputStream in = resource.getInputStream()) {
    Properties props = new Properties();
    props.load(in);
    System.out.println(props.getProperty("server.port"));
}

ClassPathResource 通过 ClassLoader.getResourceAsStream() 加载,优先使用调用方的 ClassLoader,缺省时退回 Thread.currentThread().getContextClassLoader()。在 fat JAR 中,资源位于 BOOT-INF/classes/ 或 lib JAR 内部,getFile() 抛 FileNotFoundException,但 getInputStream() 仍可用。

29.2.2 FileSystemResource

FileSystemResource resource = new FileSystemResource("/var/data/uploads/file.bin");
log.info("exists={} length={} file={}",
    resource.exists(), resource.contentLength(), resource.isFile());

try (OutputStream out = resource.getOutputStream()) {
    out.write(payload);
}

FileSystemResource 同时实现 WritableResource,可通过 getOutputStream() 写入。Path 必须在 JVM 进程可访问的文件系统(本地、NFS、SMB 挂载),网络协议地址应使用 UrlResource。

29.2.3 UrlResource

UrlResource http = new UrlResource("https://example.com/data.json");
try (InputStream in = http.getInputStream()) {
    byte[] bytes = in.readAllBytes();
    log.info("downloaded {} bytes", bytes.length);
}

UrlResource file = new UrlResource("file:/etc/hosts");
log.info("file scheme isFile={}", file.isFile());

UrlResource 支持 http: / https: / ftp: / file: / jar: 等任意 URL scheme。生产环境下载远程资源应使用 RestClient 或 WebClient 替代,以获得超时、重试、连接池等控制。

29.2.4 ByteArrayResource 与 InputStreamResource

ByteArrayResource memory = new ByteArrayResource("hello".getBytes(StandardCharsets.UTF_8));
log.info("memory length={}", memory.contentLength());

ByteArrayInputStream in = new ByteArrayInputStream("world".getBytes(StandardCharsets.UTF_8));
InputStreamResource stream = new InputStreamResource(in);
log.info("stream isOpen={}", stream.isOpen());

ByteArrayResource 适合内存中已有字节数据,可重复读取。InputStreamResource 仅适合「已打开的一次性流」,isOpen() 返回 true,重复调用 getInputStream() 抛 IllegalStateException。

29.3 ResourceLoader

图 29-3-1 · ResourceLoader 加载策略(前缀决定 Resource 类型 + Loader 子类决定无前缀的默认行为)

29.3.1 接口签名

public interface ResourceLoader {
    String CLASSPATH_URL_PREFIX = ResourceUtils.CLASSPATH_URL_PREFIX;
    Resource getResource(String location);
    ClassLoader getClassLoader();
}

public interface ResourcePatternResolver extends ResourceLoader {
    String CLASSPATH_ALL_URL_PREFIX = "classpath*:";
    Resource[] getResources(String locationPattern) throws IOException;
}

29.3.2 前缀解析规则

前缀 行为 示例
classpath: 单 classpath 资源,使用 ClassLoader.getResource() classpath:application.yml
classpath*: 跨所有 classpath 条目(含 JAR)匹配 classpath*:META-INF/spring.factories
file: 文件系统绝对路径 file:/etc/app/config.yml
http: / https: HTTP URL https://example.com/data.json
ftp: FTP URL ftp://server/file.bin
jar: JAR 内部资源 jar:file:/lib/foo.jar!/META-INF/MANIFEST.MF
(无前缀) 由具体 Loader 决定 DefaultResourceLoader 走 classpath

29.3.3 ApplicationContext 作为 ResourceLoader

@Service
public class ConfigService {

    private final ResourceLoader resourceLoader;

    public ConfigService(ResourceLoader resourceLoader) {
        this.resourceLoader = resourceLoader;
    }

    public Properties loadConfig(String location) throws IOException {
        Resource resource = resourceLoader.getResource(location);
        if (!resource.exists()) {
            throw new IllegalStateException("config not found: " + location);
        }
        Properties props = new Properties();
        try (InputStream in = resource.getInputStream()) {
            props.load(in);
        }
        return props;
    }
}

注入 ResourceLoader 时实际收到的是 ApplicationContext 实例(同时实现 ResourceLoader + ResourcePatternResolver)。前缀策略由具体 ApplicationContext 子类决定:

ApplicationContext 无前缀策略
ClassPathXmlApplicationContext classpath
FileSystemXmlApplicationContext 文件系统
GenericWebApplicationContext Servlet 上下文
AnnotationConfigApplicationContext classpath
AnnotationConfigServletWebServerApplicationContext(Spring Boot Web) Servlet 上下文

29.3.4 ResourceLoaderAware

@Component
public class TemplateRegistry implements ResourceLoaderAware {

    private ResourceLoader resourceLoader;

    @Override
    public void setResourceLoader(ResourceLoader loader) {
        this.resourceLoader = loader;
    }

    public Resource resolveTemplate(String name) {
        return resourceLoader.getResource("classpath:templates/" + name);
    }
}

ResourceLoaderAware 是 Spring 的「Bean 感知接口」之一,容器在 Bean 初始化阶段调用 setResourceLoader() 注入。现代代码优先通过构造器注入 ResourceLoader,接口方式仅在无法修改构造器签名时使用。

29.4.1 Ant 通配符

29.4.2 典型模式

@Component
public class ResourceScanner {

    private final ResourcePatternResolver resolver;

    public ResourceScanner(ApplicationContext ctx) {
        this.resolver = ctx;
    }

    public List<Resource> scanSqlScripts() throws IOException {
        Resource[] resources = resolver.getResources("classpath*:db/migration/V*__*.sql");
        return List.of(resources);
    }

    public List<Resource> scanI18nProperties() throws IOException {
        Resource[] resources = resolver.getResources("classpath*:i18n/**/*.properties");
        return List.of(resources);
    }

    public List<Resource> scanProtoFiles() throws IOException {
        Resource[] resources = resolver.getResources("classpath:proto/**/*.proto");
        return List.of(resources);
    }
}

表 29-4-1:classpath: vs classpath*: 差异

维度 classpath: classpath*:
来源 单个 ClassLoader 资源 所有 ClassLoader 条目(含 JAR)
性能 单次 getResource() 调用 多次 getResources() + 合并
典型场景 应用自身配置(classpath:application.yml) 跨 JAR 扫描(classpath*:META-INF/spring.factories)
Spring Boot 自动配置 application.yml META-INF/spring.factories / META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports

classpath*: 是 Spring 自动配置机制的基础。SpringFactoriesLoader 通过 classpath*:META-INF/spring.factories 跨所有 starter JAR 加载自动配置类列表。

29.4.3 AntPathMatcher 直接使用

PathMatcher matcher = new AntPathMatcher();

assert matcher.match("/api/orders/*", "/api/orders/1001");
assert matcher.match("/api/orders/**", "/api/orders/1001/items/2");
assert matcher.match("/api/?rder", "/api/order");
assert !matcher.match("/api/orders/*", "/api/orders/1001/items");

Map<String, String> vars = matcher.extractUriTemplateVariables(
    "/api/orders/{orderId}/items/{itemId}",
    "/api/orders/1001/items/2");
log.info("orderId={} itemId={}", vars.get("orderId"), vars.get("itemId"));

AntPathMatcher 同时是 Spring MVC @RequestMapping 路径匹配的底层实现,支持路径变量提取与模式比较(更具体的模式优先)。

29.5 Spring Boot IO 实战

29.5.1 静态资源加载

spring:
  web:
    resources:
      static-locations:
        - classpath:/static/
        - classpath:/public/
        - file:/opt/app/static/
      cache:
        period: 3600

Spring Boot 默认从以下位置加载静态资源:classpath:/META-INF/resources/、classpath:/resources/、classpath:/static/、classpath:/public/。spring.web.resources.static-locations 覆盖默认列表,可混合 classpath 与 file 协议。

29.5.2 Resource 返回 classpath 文件

@RestController
@RequestMapping("/io")
public class ResourceController {

    private final ResourceLoader resourceLoader;

    public ResourceController(ResourceLoader resourceLoader) {
        this.resourceLoader = resourceLoader;
    }

    @GetMapping("/manual")
    public ResponseEntity<Resource> manual() throws IOException {
        Resource resource = resourceLoader.getResource("classpath:docs/io-manual.txt");
        long length = resource.contentLength();
        log.info("resource=classpath:docs/io-manual.txt length={}", length);
        return ResponseEntity.ok()
            .contentType(MediaType.TEXT_PLAIN)
            .contentLength(length)
            .header(HttpHeaders.CONTENT_DISPOSITION, "inline; filename=\"io-manual.txt\"")
            .body(resource);
    }
}

返回 Resource 类型时,Spring MVC 通过 ResourceHttpMessageConverter 自动写入响应体,支持 Range 请求(断点续传)与 If-Modified-Since 条件 GET。

29.5.3 InputStream 字节级读取

@GetMapping("/manual/stats")
public Map<String, Object> manualStats() throws IOException {
    Resource resource = resourceLoader.getResource("classpath:docs/io-manual.txt");
    long bytes = 0;
    byte[] buffer = new byte[8192];
    try (InputStream in = new BufferedInputStream(resource.getInputStream())) {
        for (int read; (read = in.read(buffer)) != -1; ) {
            bytes += read;
        }
    }
    return Map.of(
        "resource", resource.getFilename(),
        "bytes", bytes,
        "readMode", "buffered-input-stream");
}

InputStream 适合「消费内容」场景(摘要计算、统计、逐块解析)。必须用 try-with-resources 关闭,避免 fat JAR 中 ZIP 流持有 JAR 文件描述符。

29.5.4 文件上传

@PostMapping(value = "/upload", consumes = MediaType.MULTIPART_FORM_DATA_VALUE)
public UploadResult upload(@RequestPart("file") MultipartFile file) throws IOException {
    String original = Optional.ofNullable(file.getOriginalFilename()).orElse("upload.bin");
    String safeName = Paths.get(original).getFileName().toString();
    Path target = Paths.get("build/uploads").resolve(safeName).normalize();

    if (!target.startsWith(Paths.get("build/uploads").toAbsolutePath().normalize())) {
        throw new SecurityException("path traversal attempt: " + original);
    }

    Files.createDirectories(target.getParent());
    try (InputStream in = file.getInputStream()) {
        Files.copy(in, target, StandardCopyOption.REPLACE_EXISTING);
    }

    log.info("uploaded filename={} size={} savedTo={}", safeName, file.getSize(), target);
    return new UploadResult(safeName, file.getSize(), target.toString());
}

public record UploadResult(String filename, long bytes, String savedTo) {}

上传安全要点: - 使用 Paths.get(original).getFileName() 去除路径分隔符,阻止 ../../etc/passwd 攻击 - target.startsWith(uploadRoot) 二次校验绝对路径,确保不逃逸到上传目录之外 - 大文件配置 spring.servlet.multipart.max-file-size 与 max-request-size,防止内存耗尽

29.5.5 文件下载

@GetMapping("/download/{name}")
public ResponseEntity<Resource> download(@PathVariable String name) throws IOException {
    String safeName = Paths.get(name).getFileName().toString();
    Path file = Paths.get("build/uploads").resolve(safeName).normalize();
    Resource resource = new FileSystemResource(file);
    if (!resource.exists()) {
        return ResponseEntity.notFound().build();
    }
    long length = resource.contentLength();
    return ResponseEntity.ok()
        .contentType(MediaType.APPLICATION_OCTET_STREAM)
        .contentLength(length)
        .header(HttpHeaders.CONTENT_DISPOSITION,
            "attachment; filename=\"" + safeName + "\"")
        .body(resource);
}

Content-Type、Content-Length、Content-Disposition 三个响应头是下载接口的契约。缺失任意一个都会导致客户端行为不一致(浏览器、网关、SDK 各自猜测)。attachment 强制下载弹窗,inline 内嵌显示。

29.5.6 StreamingResponseBody 大文件

@GetMapping(value = "/events.csv", produces = "text/csv")
public StreamingResponseBody streamCsv() {
    log.info("accepted streaming csv request");
    return outputStream -> {
        Writer writer = new OutputStreamWriter(outputStream, StandardCharsets.UTF_8);
        writer.write("id,message\n");
        for (int i = 1; i <= 1_000_000; i++) {
            writer.write(i + ",event-" + i + "\n");
            if (i % 10_000 == 0) {
                writer.flush();
            }
        }
        writer.flush();
    };
}

StreamingResponseBody 在异步任务中执行,Controller 线程立即返回。适合不能一次性放进内存的响应:报表导出、日志下载、CSV 大数据集。响应使用 Transfer-Encoding: chunked,客户端边接收边消费。

29.5.7 NIO FileChannel 零拷贝

@PostMapping("/nio/copy")
public Map<String, Object> copyWithNio(@RequestParam String source,
                                        @RequestParam String target) throws IOException {
    Path sourcePath = Paths.get("build/uploads")
        .resolve(Paths.get(source).getFileName()).normalize();
    Path targetPath = Paths.get("build/uploads")
        .resolve(Paths.get(target).getFileName()).normalize();

    Files.createDirectories(targetPath.getParent());

    try (FileChannel in = FileChannel.open(sourcePath, StandardOpenOption.READ);
         FileChannel out = FileChannel.open(targetPath,
             StandardOpenOption.CREATE,
             StandardOpenOption.TRUNCATE_EXISTING,
             StandardOpenOption.WRITE)) {
        long position = 0;
        long size = in.size();
        while (position < size) {
            long transferred = in.transferTo(position, size - position, out);
            if (transferred <= 0) {
                throw new EOFException("no bytes transferred from " + sourcePath);
            }
            position += transferred;
        }
        return Map.of("source", source, "target", target, "bytesCopied", position);
    }
}

FileChannel.transferTo() 在 Linux 上调用 sendfile(2) 系统调用,数据在内核态直接从源 fd 拷贝到目标 fd,无需经过用户态缓冲区。文件 → 文件、文件 → socket 场景下吞吐显著优于 read() + write() 循环。

29.6.2 OS I/O 多路复用对比

+----------------+-----------------------+----------------------+----------------------+
| OS              | Linux                  | macOS / BSD           | Windows               |
+----------------+-----------------------+----------------------+----------------------+
| 多路复用机制    | epoll                  | kqueue                | IOCP                  |
| API             | epoll_create / ctl /   | kqueue / kevent       | CreateIoCompletionPort|
|                 | wait                   |                       | / ReadFile / GQCS     |
| 模型            | Reactor(就绪通知)     | Reactor(就绪通知)    | Proactor(完成通知)   |
| 内核结构        | 红黑树 + 就绪链表      | 哈希表 + knote 链表   | 完成包 FIFO 队列      |
| 注册复杂度      | O(log n)               | O(1)                  | O(1)                  |
| 等待复杂度      | O(就绪数)              | O(就绪数)             | O(1)(取队首)         |
| 触发模式        | LT / ET                | LT / ET (EV_CLEAR)    | N/A(完成通知)        |
| 事件类型        | 仅 fd                  | fd + 文件 + 信号 + 定时器| socket / file / pipe |

+----------------+-----------------------+----------------------+----------------------+

Java NIO 的 Selector 是 Reactor 模型的统一抽象,JDK 在不同平台选择不同实现:Linux 使用 EPollSelectorImpl,macOS 使用 KQueueSelectorImpl,Windows 使用 WEPollSelectorImpl(基于 wepoll,模拟 epoll 语义)。Windows IOCP 是 Proactor 模型,JDK NIO Selector API 不直接使用。

29.6.3 Reactor vs Proactor

Reactor(epoll / kqueue):
  内核通知:fd 就绪,可以读写
  应用动作:read() / write() 完成 IO
  代表:Java NIO Selector、Netty NioEventLoop

Proactor(IOCP):
  内核通知:IO 已完成,缓冲区已有结果
  应用动作:处理完成事件
  代表:Windows IOCP、Linux io_uring(部分)

表 29-6-2:Servlet vs Reactive 选型

场景 推荐 原因
传统 CRUD Servlet (Tomcat) 阻塞编程直观,JDBC / JPA 生态成熟
高并发网关 Reactive (WebFlux + Netty) EventLoop 资源占用低
微服务密集调用 Reactive 大量 IO 等待,非阻塞并行
CPU 密集计算 Servlet EventLoop 不能被长时间占用
SSE / WebSocket Reactive 天然支持背压与长连接
团队经验有限 Servlet Reactive 调试难度大
JDK 21 + 阻塞 IO Servlet + Virtual Threads 兼顾阻塞写法与高并发

29.6.4 Virtual Threads

spring:
  threads:
    virtual:
      enabled: true

Spring Boot 3.2+ 支持虚拟线程。Tomcat 在虚拟线程模式下,每个请求由一个虚拟线程承载,载体线程(Carrier Thread)由 ForkJoinPool 提供。阻塞调用时虚拟线程让出载体线程,不再占用平台线程资源。

虚拟线程的 pinning 限制:synchronized 块、Object.wait()、native 方法在执行期间会"钉住"载体线程。检测方法:-Djdk.tracePinnedThreads=short 启动参数。修复:synchronized 改为 ReentrantLock,JDBC 驱动升级到支持虚拟线程的版本(MySQL Connector/J 8.2+)。

29.7.1 陷阱 1:InputStreamResource

// 反例
InputStreamResource resource = new InputStreamResource(
    new FileInputStream("data.bin"));
try (InputStream in1 = resource.getInputStream()) {
    in1.readAllBytes();
}
try (InputStream in2 = resource.getInputStream()) {
    in2.readAllBytes();  // IllegalStateException
}

// 正例:用 ByteArrayResource 缓存内容
byte[] bytes = Files.readAllBytes(Paths.get("data.bin"));
ByteArrayResource resource = new ByteArrayResource(bytes);
try (InputStream in1 = resource.getInputStream()) { in1.readAllBytes(); }
try (InputStream in2 = resource.getInputStream()) { in2.readAllBytes(); }  // OK

29.7.2 陷阱 2:路径穿越

// 反例:直接信任 originalFilename
String name = file.getOriginalFilename();
Path target = Paths.get("/var/uploads", name);
Files.copy(file.getInputStream(), target);
// 攻击者上传 "../../etc/passwd" → 写入 /etc/passwd

// 正例:剥离路径 + 二次校验
String safeName = Paths.get(file.getOriginalFilename()).getFileName().toString();
Path uploadRoot = Paths.get("/var/uploads").toAbsolutePath().normalize();
Path target = uploadRoot.resolve(safeName).normalize();
if (!target.startsWith(uploadRoot)) {
    throw new SecurityException("path traversal: " + safeName);
}
Files.copy(file.getInputStream(), target);

startsWith 校验必须在 normalize() 之后执行,且 uploadRoot 与 target 都需要绝对路径,避免相对路径绕过。

29.7.3 陷阱 3:fat JAR 内 getFile()

// 反例
Resource resource = resourceLoader.getResource("classpath:templates/email.html");
File file = resource.getFile();  // fat JAR 中抛 FileNotFoundException
String content = Files.readString(file.toPath());

// 正例:统一用 InputStream
Resource resource = resourceLoader.getResource("classpath:templates/email.html");
String content;
try (InputStream in = resource.getInputStream()) {
    content = new String(in.readAllBytes(), StandardCharsets.UTF_8);
}

fat JAR 中的 classpath 资源位于 BOOT-INF/classes/ 或 lib JAR 内部,URL 形态为 jar:file:...!/BOOT-INF/classes/templates/email.html,不对应文件系统中的真实文件。isFile() 返回 false 时不要调 getFile()。

29.7.4 陷阱 4:同步 IO 占满 Worker

// 反例:Tomcat Worker 线程被同步调用占满
@RestController
public class OrderController {
    @GetMapping("/orders/{id}")
    public Order get(@PathVariable Long id) {
        return restTemplate.getForObject(  // 同步阻塞 5 秒
            "https://slow-service/orders/{id}", Order.class, id);
    }
}
// 200 个 Worker 线程被 5 秒慢调用占满 → 新请求排队 → 超时

// 正例 1:@Async 异步
@Async
public CompletableFuture<Order> getAsync(Long id) {
    return CompletableFuture.completedFuture(
        restTemplate.getForObject(
            "https://slow-service/orders/{id}", Order.class, id));
}

// 正例 2:WebFlux 非阻塞
@GetMapping("/orders/{id}")
public Mono<Order> get(@PathVariable Long id) {
    return webClient.get()
        .uri("/orders/{id}", id)
        .retrieve()
        .bodyToMono(Order.class);
}

// 正例 3:Virtual Threads
// spring.threads.virtual.enabled=true
@GetMapping("/orders/{id}")
public Order get(@PathVariable Long id) {
    return restTemplate.getForObject(
        "https://slow-service/orders/{id}", Order.class, id);
}
// 阻塞期间虚拟线程让出载体线程,不占用平台线程

29.7.5 陷阱 5:大文件 OOM

// 反例
@GetMapping("/export")
public ResponseEntity<byte[]> export() {
    byte[] data = generateLargeReport();  // 1 GB → OOM
    return ResponseEntity.ok().body(data);
}

// 正例
@GetMapping("/export")
public StreamingResponseBody export() {
    return outputStream -> {
        try (Writer writer = new OutputStreamWriter(outputStream)) {
            for (Record record : queryAllRecords()) {
                writer.write(record.toCsvLine());
                writer.write('\n');
                writer.flush();
            }
        }
    };
}

29.8.1 PathMatchingResourcePatternResolver 流程

// 简化逻辑,基于 Spring Framework 6.x 源码
public Resource[] getResources(String locationPattern) throws IOException {
    if (locationPattern.startsWith(CLASSPATH_ALL_URL_PREFIX)) {
        // classpath*: 跨所有 ClassLoader 条目
        if (getPathMatcher().isPattern(locationPattern.substring(CLASSPATH_ALL_URL_PREFIX.length()))) {
            return findPathMatchingResources(locationPattern);
        } else {
            return findAllClassPathResources(
                locationPattern.substring(CLASSPATH_ALL_URL_PREFIX.length()));
        }
    } else {
        if (getPathMatcher().isPattern(locationPattern)) {
            return findPathMatchingResources(locationPattern);
        } else {
            return new Resource[]{getResourceLoader().getResource(locationPattern)};
        }
    }
}

findPathMatchingResources() 内部: 1. 拆分根路径(无通配符前缀)与模式(含通配符) 2. 调用 getResources(rootPath) 获取根目录候选 3. 对每个根目录递归扫描,使用 AntPathMatcher.match() 过滤

29.8.2 设计取舍

取舍 1:统一抽象 vs 性能开销

Resource 接口屏蔽了 classpath、文件、URL 等差异,代价是多一层方法调用与对象包装。收益是同一套业务代码可处理多种资源源,fat JAR 部署与文件系统部署无差异。性能敏感场景(批处理大量文件)可直接使用 Files.newInputStream()。

取舍 2:classpath*: 跨 JAR 扫描的代价

classpath*:META-INF/spring.factories 在大型 fat JAR 中需扫描数百个 JAR 文件,启动时间显著影响。Spring Boot 3.x 的 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 文件格式更扁平,扫描成本更低。

取舍 3:getFile() 与 getInputStream() 的兼容性

getFile() 仅文件系统资源支持,fat JAR 中失败。统一使用 getInputStream() 是最兼容的方式,但牺牲了 Files.size() / Files.lastModified() 等文件 API 能力。isFile() 检查是分支选择的依据。
