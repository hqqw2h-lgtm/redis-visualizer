第 20 章 · 异步处理与任务调度

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

异步处理是高性能系统的基石——它将耗时操作从主线程剥离,让请求快速返回,同时在后台完成繁重工作。Spring 提供了从简单的 @Async 注解到完整的线程池管理与定时调度的全套方案。

但异步并非银弹——它引入了线程安全、事务边界、上下文传播、资源管理等全新维度的复杂性。一个配置不当的 SimpleAsyncTaskExecutor 可以在几分钟内创建数万个线程导致 OOM;一个 @Scheduled 任务默认只有一个线程,一个慢任务会阻塞所有后续调度。Java 21 的 Virtual Thread 彻底改变了"一请求一线程"的成本模型,Spring Boot 3.2 加 对此提供了一等公民支持。

本章按"@Async 代理机制 → TaskExecutor 体系 → @Scheduled 调度 → Virtual Thread 集成 → CompletableFuture 编排 → 陷阱与限制"的顺序,系统讲解 Spring 异步编程的全景。前置知识:第 3 章 IoC 容器的 Bean 生命周期与 BeanPostProcessor 机制,第 5 章 AOP 代理原理。了解 java.util.concurrent 基础有助于理解执行器模型。

20.1 @Async 异步方法执行

Spring 的 @Async 是最简单的异步编程入口:只需一个注解,方法调用就从同步变为异步。但"简单"的背后隐藏着一整套代理机制——理解它,才能在生产中避开各种诡异问题。

20.1.1 @EnableAsync 激活异步支持

一切的起点是 @EnableAsync。这个注解通过 @Import(AsyncConfigurationSelector.class) 导入异步配置,激活 AsyncAnnotationBeanPostProcessor——后者负责扫描所有标注了 @Async 的 Bean,为它们创建代理。

@Configuration
@EnableAsync
public class AsyncConfig {
}

把 org.springframework.aop 与 org.springframework.scheduling 日志调到 DEBUG 后,启动时能看到异步代理被装配:

INFO  o.s.s.a.AsyncAnnotationBeanPostProcessor : Found @Async annotation on bean 'emailService'
DEBUG o.s.a.f.autoproxy.AbstractAutoProxyCreator : Creating implicit proxy for bean 'emailService'
INFO  c.example.AsyncConfig : @EnableAsync is active, async proxy mode=PROXY

Spring Boot 不会自动启用 @Async——与 @EnableScheduling 类似,必须显式在配置类上添加 @EnableAsync。这是一个有意的设计决策:异步执行改变了方法调用的线程模型,不应在开发者不知情的情况下隐式激活。

20.1.2 @Async 代理创建与调用链路

AsyncAnnotationBeanPostProcessor 继承自 AbstractAdvisingBeanPostProcessor,在 Bean 初始化完成后(postProcessAfterInitialization 阶段)介入。它检查 Bean 的方法上是否存在 @Async 注解,如存在则通过 AOP 为该 Bean 创建代理。代理的核心拦截器是 AsyncExecutionInterceptor,负责将方法调用提交到 TaskExecutor 线程池。

20.1.3 源码入口:@Async 从注解到线程切换

@Async 的核心不是"新开线程"四个字,而是 Spring AOP、执行器选择与返回值适配三者的组合。源码阅读可从 @EnableAsync 的导入链开始,最后落到 AsyncExecutionInterceptor.invoke()。

源码入口 关键方法 运行期职责

ProxyAsyncConfiguration asyncAdvisor() 注册 AsyncAnnotationBeanPostProcessor

AsyncAnnotationAdvisor buildAdvice() 绑定 AnnotationAsyncExecutionInterceptor

AsyncExecutionAspectSupport determineAsyncExecutor() 按 @Async("beanName")、AsyncConfigurer、默认 executor 顺序选择

图 20-2 · Spring Async 源码到运行时线程模型(@EnableAsync → BPP → Advisor → Interceptor → Executor)

这里有一个关键边界:@Async 代理只包住"从代理对象进入的方法调用"。目标方法内部的 this.sendEmail()、private 方法、构造器逻辑、同类内部调用都不会经过 AnnotationAsyncExecutionInterceptor,因此不会切线程。

20.1.4 默认 SimpleAsyncTaskExecutor 的危险

在纯 Spring Framework 中,如找不到可用的 Executor 或 TaskExecutor,@Async 会回退到 SimpleAsyncTaskExecutor。它的行为非常简单粗暴:每次调用都创建一个新线程,没有线程复用,没有队列缓冲,没有最大线程数限制。Spring Boot 应用通常会自动提供 applicationTaskExecutor,但显式覆盖异步配置时仍要确认最终使用的是哪个执行器。

SimpleAsyncTaskExecutor 在高并发场景下极其危险。假设每秒有 1000 个 @Async 调用,它会在 1 秒内创建 1000 个线程。每个线程默认占用约 1 MB 栈空间,10 秒后就是 10 GB 级别的栈空间压力,加上操作系统对线程数的限制,系统很快会抛出 OutOfMemoryError unable to create native thread 或被 OOM Killer 杀死。生产环境必须显式确认异步执行器有边界。

public class SimpleAsyncTaskExecutor implements AsyncTaskExecutor {
    @Override
    public void execute(Runnable task) {
        Thread t = new Thread(task);
        t.setDaemon(false);
        t.start();
    }
}

20.1.5 配置自定义 TaskExecutor

生产环境必须配置 ThreadPoolTaskExecutor,控制线程数量、队列容量与拒绝策略:

@Configuration
@EnableAsync
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        executor.setQueueCapacity(200);
        executor.setThreadNamePrefix("async-");
        executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
        executor.setWaitForTasksToCompleteOnShutdown(true);
        executor.setAwaitTerminationSeconds(60);
        executor.initialize();
        return executor;
    }

    @Override
    public AsyncUncaughtExceptionHandler getAsyncUncaughtExceptionHandler() {
        return (ex, method, params) ->
            log.error("Async method {} threw exception", method.getName(), ex);
    }
}

调用方线程会立即返回,真正的业务日志出现在 async- 线程中;这也是排查 @Async 是否生效的第一眼证据:

INFO  [http-nio-8080-exec-7] c.example.OrderController : accepted orderId=O1001 elapsed=4ms
INFO  [async-1] c.example.EmailService : start sending welcome mail orderId=O1001
INFO  [async-1] c.example.EmailService : mail sent orderId=O1001 elapsed=476ms

20.1.6 @Async 的四种返回类型

@Async 方法支持四种返回类型,选择取决于调用方是否需要获取异步结果。

void 适合发射即忘场景。调用方不关心结果,异常只能通过 AsyncUncaughtExceptionHandler 捕获:

@Async
public void sendEmail(String to) {
    mailSender.send(to, "Welcome");
}

Future<T> 是传统阻塞获取方式。调用方使用 future.get() 阻塞等待:

@Async
public Future<Report> generateReport(Long id) {
    Report report = doHeavyWork(id);
    return new AsyncResult<>(report);
}

CompletableFuture<T> 是推荐方式。从 Spring 4.2 起,@Async 方法返回的 CompletableFuture 由框架自动适配,无需 AsyncResult 包装。CompletableFuture 支持函数式编排(thenApply、thenCompose、allOf),远比传统 Future.get() 阻塞获取灵活:

@Async
public CompletableFuture<Report> generateReport(Long id) {
    Report report = doHeavyWork(id);
    return CompletableFuture.completedFuture(report);
}

ListenableFuture<T> 已废弃。Spring 6.0 将其标记为 @Deprecated,迁移到 CompletableFuture 即可。

20.1.7 上下文传播案例:Spring Cloud Sleuth 与 Micrometer Tracing

在分布式追踪系统中,每个请求会携带 traceId 与 spanId。当使用 @Async 切换线程时,这些追踪信息默认会丢失。Spring Cloud Sleuth 通过 LazyTraceExecutor、TraceableExecutorService 包装 TaskExecutor,在任务提交时捕获当前 Span,在异步线程启动时恢复。Spring Boot 3.x 迁移到 Micrometer Tracing 后,通常需要配合 Micrometer Context Propagation 或框架提供的 executor 包装能力,不能假设任意自定义线程池都会自动传播 traceId。

20.1.8 设计取舍:为什么选择代理而非改写字节码

@Async 复用了 Spring AOP 代理体系,这让它和事务、缓存、安全等注解保持一致,但也继承了代理模型的限制。Spring 同时提供 AspectJ 模式,但默认仍是 proxy mode。

方案 优点 代价或被拒绝原因
Proxy mode 默认 零额外工具链;与 @Transactional 一致;启动时创建代理即可 自调用不生效;final 与 private 方法受限;只拦截 Spring Bean
AspectJ compile-time weaving 可拦截自调用与非 public 方法;语义彻底 需要 AJ 编译器或织入插件,构建复杂
Load-time weaving 不改编译流程,运行期织入更强 类加载期复杂,容器与 Agent 兼容成本高
显式 CompletableFuture.supplyAsync 行为透明,调用点可见 样板多,执行器与上下文传播容易散落各处

生产项目通常选择 proxy mode 加 明确线程池 加 清晰服务边界。如果一个类里大量需要异步自调用,往往说明边界划分错了:把异步职责拆到独立 Bean,比引入 AspectJ 更易维护。

20.2 TaskExecutor 体系

Spring 的 TaskExecutor 是 JDK java.util.concurrent.Executor 的语义子接口,为 Spring 生态提供统一的任务执行抽象。从"每次新建线程"的 SimpleAsyncTaskExecutor 到"虚拟线程"的 VirtualThreadTaskExecutor,Spring 提供了丰富的实现。

20.2.2 ThreadPoolTaskExecutor 核心参数

ThreadPoolTaskExecutor 是生产环境最常用的实现。它内部包装了 JDK 的 ThreadPoolExecutor,但提供了 Spring 风格的配置(支持属性注入、生命周期管理、优雅关闭)。理解其参数之间的关系是调优的关键。

图 20-4 · 线程池任务提交执行流程(core → queue → max → reject 4 阶段决策)
@Bean("orderExecutor")
public ThreadPoolTaskExecutor orderExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    executor.setMaxPoolSize(50);
    executor.setQueueCapacity(200);
    executor.setKeepAliveSeconds(60);
    executor.setThreadNamePrefix("order-async-");
    executor.setRejectedExecutionHandler(new ThreadPoolExecutor.CallerRunsPolicy());
    return executor;
}

启用 Actuator 后,ThreadPoolTaskExecutor 会暴露活跃线程、队列与完成任务数:

executor_active_threads{name="orderExecutor"} 10.0
executor_queued_tasks{name="orderExecutor"} 37.0
executor_pool_size_threads{name="orderExecutor"} 10.0
executor_completed_tasks_total{name="orderExecutor"} 1842.0

ThreadPoolExecutor 的扩容逻辑常让人意外:它先填满核心线程,然后把任务堆到队列里,只有队列满了才开始创建非核心线程。这意味着如 queueCapacity = Integer.MAX_VALUE(LinkedBlockingQueue 默认值),maxPoolSize 永远不会生效。Spring 的 ThreadPoolTaskExecutor 默认 queueCapacity = Integer.MAX_VALUE,必须显式设置合理值。

20.2.3 三种执行器的对比

执行器 调度模型 适用场景 Spring 集成度
ThreadPoolTaskExecutor 固定大小线程池加 任务队列 I/O 密集型异步任务 最完善,Actuator 暴露指标
ForkJoinPool 工作窃取算法 CPU 密集型递归或分治计算 CompletableFuture 默认 commonPool
VirtualThreadTaskExecutor Java 21 加 虚拟线程 极高并发 I/O 阻塞代码 Spring Boot 3.2 加 内置支持

ForkJoinPool 默认线程数等于 CPU 核心数减 1,不适合包含阻塞 I/O 的任务;VirtualThreadTaskExecutor 极低开销可达百万级并发,但 synchronized 会导致 pinning。

20.2.4 @Async("specificExecutor") 指定执行器

当系统中有多个 TaskExecutor Bean 时,可以通过 @Async("beanName") 指定使用哪个执行器,实现任务隔离:

@Service
public class NotificationService {

    @Async("emailExecutor")
    public CompletableFuture<Void> sendEmail(String to, String body) {
        mailSender.send(to, body);
        return CompletableFuture.completedFuture(null);
    }

    @Async("smsExecutor")
    public CompletableFuture<Void> sendSms(String phone, String text) {
        smsGateway.send(phone, text);
        return CompletableFuture.completedFuture(null);
    }
}

两个下游通道使用不同线程名前缀,说明邮件与短信任务被隔离到不同执行器:

INFO  [sms-1] c.example.NotificationService : send sms to +86138 0000 0000
INFO  [sms-1] c.example.NotificationService : sms accepted messageId=sms-7788
INFO  [email-1] c.example.NotificationService : email accepted messageId=mail-9911

20.2.5 优雅关闭

应用关闭时,正在执行的异步任务可能被粗暴中断。通过配置优雅关闭,可让已提交的任务执行完毕后再关闭线程池:

@Bean
public ThreadPoolTaskExecutor taskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    executor.setMaxPoolSize(50);
    executor.setQueueCapacity(200);
    executor.setWaitForTasksToCompleteOnShutdown(true);
    executor.setAwaitTerminationSeconds(60);
    return executor;
}

收到关闭信号后,线程池先停止接收新任务,再等待已提交任务完成;超时才会强制中断:

INFO  [SpringApplicationShutdownHook] o.s.s.c.ThreadPoolTaskExecutor : Shutting down ExecutorService 'taskExecutor'
INFO  [SpringApplicationShutdownHook] c.example.AsyncConfig : waiting up to 60s for async tasks to finish
INFO  [order-async-8] c.example.PaymentReceiptJob : payment receipt generated orderId=O1001
INFO  [SpringApplicationShutdownHook] o.s.s.c.ThreadPoolTaskExecutor : ExecutorService 'taskExecutor' shut down successfully

20.2.6 Spring Kafka 并发消费模型案例

Spring Kafka 的 ConcurrentMessageListenerContainer 通过 concurrency 参数控制消费者线程数。每个消费线程对应一个 Kafka 分区(或多个)。底层并不使用 ThreadPoolTaskExecutor,而是直接管理 KafkaMessageListenerContainer 实例数组。每个容器内部使用 SimpleAsyncTaskExecutor 创建一个长期运行的轮询线程。设置 spring.kafka.listener.concurrency=3 就会启动 3 个消费线程。优雅关闭同样重要:ContainerProperties.setShutdownTimeout() 控制消费者关闭前最长等待时间。

20.2.7 扩展边界:执行器、装饰器与上下文传播

异步扩展最容易出错的地方,是把"执行任务"、"传播上下文"、"限流隔离"混在一个自定义 executor 里。Spring 已经给出了分层扩展点。

扩展点 适合做什么 边界
AsyncConfigurer.getAsyncExecutor() 定义全局默认 @Async 执行器 不要写业务路由,多业务池用 @Async("beanName")
@Async("executorName") 按任务类型隔离线程池 名称是运行期契约,重命名 Bean 要同步测试
TaskDecorator 捕获并恢复 MDC、traceId、租户 ID 不要传播事务连接、EntityManager、Request 对象本体
DelegatingSecurityContextAsyncTaskExecutor 传播 Spring Security SecurityContext 线程池复用场景不要用 MODE_INHERITABLETHREADLOCAL 替代
RejectedExecutionHandler 定义满载行为:抛错、调用方执行、丢弃 DiscardPolicy 极少适合业务任务,会隐藏数据丢失
Micrometer executor metrics 观察活跃线程、队列、完成任务、拒绝数 没有指标的线程池无法调优

上下文传播的核心原则:只传播"值",不要传播"资源句柄"。TraceId、用户 ID、租户 ID 可以复制;数据库事务、Hibernate Session、Servlet request 输入流都不能跨线程复用。

20.3.2 源码入口:@Scheduled 如何注册成定时任务

@Scheduled 不是每次请求触发,而是在容器启动阶段扫描 Bean 方法并注册任务。核心处理器是 ScheduledAnnotationBeanPostProcessor,它在单例 Bean 初始化之后解析注解,再把任务交给 ScheduledTaskRegistrar。

源码入口 关键方法 运行期职责
@EnableScheduling @Import(SchedulingConfiguration.class) 注册 ScheduledAnnotationBeanPostProcessor
ScheduledAnnotationBeanPostProcessor postProcessAfterInitialization() 扫描 Bean 方法上的 @Scheduled 与 @Schedules
ScheduledAnnotationBeanPostProcessor processScheduled() 解析 fixedRate、fixedDelay、cron、initialDelay
ScheduledTaskRegistrar scheduleTasks() 把任务注册到 TaskScheduler 或 ScheduledExecutorService
ThreadPoolTaskScheduler scheduleAtFixedRate() 管理调度线程池、错误处理与优雅关闭
SimpleAsyncTaskScheduler setVirtualThreads(true) Boot 虚拟线程场景下,用调度线程触发、虚拟线程执行任务

20.3.3 三种调度模式

@Component
public class DataCleanupTask {

    @Scheduled(fixedRate = 5000)
    public void heartbeat() {
        healthChecker.ping();
    }

    @Scheduled(fixedDelay = 5000)
    public void cleanExpiredSessions() {
        sessionRepo.deleteExpired();
    }

    @Scheduled(cron = "0 30 2 * * ?")
    public void generateDailyReport() {
        reportService.generateDaily(LocalDate.now().minusDays(1));
    }

    @Scheduled(cron = "${app.cleanup.cron:0 0 3 * * ?}")
    public void configurableTask() {
    }
}

fixedRate 按计划时间触发,fixedDelay 按"上次结束后再延迟"触发,cron 则按表达式命中:

INFO  [scheduling-1] c.example.DataCleanupTask : heartbeat tick mode=fixedRate

INFO  [scheduling-1] c.example.DataCleanupTask : cleanExpiredSessions started after previous completion + 5s

20.3.5 fixedRate 与 fixedDelay 的执行模式

图 20-6 · fixedRate=5s(任务执行 7s 会迟到 — 立即补下一轮挤占调度线程)
图 20-7 · fixedDelay=5s(节奏不追赶 — 上次结束 + 5s 才开始下一轮)

当 fixedRate 等于 5000 但任务执行用了 7 秒时,下一次执行会在上一轮结束后尽快开始(因为已经迟到 2 秒),而非再等待 5 秒。同一个 @Scheduled 方法的周期性执行通常不会并发重入,也不会把每个错过的时间点都补跑一遍;它只是让下一次已到期执行尽快开始,并挤占调度线程。使用 fixedDelay 可以避免这种节奏问题——它保证每次执行结束后至少等待 N 毫秒。

20.3.6 单线程默认调度器的陷阱

@EnableScheduling 默认只创建一个单线程的 TaskScheduler(底层是 ScheduledExecutorService with poolSize 等于 1)。这意味着如有 5 个 @Scheduled 任务,它们共享同一个线程——一个慢任务会阻塞所有其他任务的调度。

假设有 3 个 @Scheduled 任务,都设置 fixedRate 等于 1000(每秒执行一次)。如任务 A 意外耗时 10 秒,任务 B 与 C 在这 10 秒内完全不会被调度。任务 A 完成后,B 与 C 已经到期的下一轮执行会在同一个线程中串行启动。这种行为在生产环境中极为隐蔽——因为每个任务单独测试都正常,但合在一起就出问题。

@Configuration
@EnableScheduling
public class SchedulerConfig implements SchedulingConfigurer {

    @Override
    public void configureTasks(ScheduledTaskRegistrar taskRegistrar) {
        ThreadPoolTaskScheduler scheduler = new ThreadPoolTaskScheduler();
        scheduler.setPoolSize(5);
        scheduler.setThreadNamePrefix("sched-");
        scheduler.setErrorHandler(t -> log.error("Scheduled task error", t));
        scheduler.setWaitForTasksToCompleteOnShutdown(true);
        scheduler.setAwaitTerminationSeconds(30);
        scheduler.initialize();
        taskRegistrar.setTaskScheduler(scheduler);
    }
}

20.4 Virtual Thread 集成 Java 21 加

Java 21 正式发布的 Virtual Thread(虚拟线程,Project Loom)是 Java 并发模型的一次革命。传统平台线程由操作系统管理,每个线程约 1 MB 栈空间,创建与上下文切换成本高昂。虚拟线程由 JVM 管理,一个 JVM 可以轻松运行数百万个虚拟线程——而 Spring Boot 3.2 加 只需一行配置就能启用。

20.4.1 平台线程与虚拟线程对比

    return new VirtualThreadTaskExecutor("async-vt-");
}

@Bean

public TaskScheduler taskScheduler() {
    SimpleAsyncTaskScheduler scheduler = new SimpleAsyncTaskScheduler();
    scheduler.setVirtualThreads(true);

}

@Bean

    );
}

20.4.3 虚拟线程调度机制

20.4.4 Pinning 问题:synchronized 是虚拟线程的天敌

虚拟线程的核心优势在于遇到阻塞 I/O 时自动从载体线程"卸载"(unmount),让载体线程去执行其他虚拟线程。但当虚拟线程进入 synchronized 块时,它会被"钉住"(pinned)在载体线程上,阻塞 I/O 不再触发卸载——导致载体线程被占用,本质上退化为平台线程模型。

许多 JDBC 驱动内部大量使用 synchronized。在 Spring Boot 3.2 发布初期,MySQL Connector/J 的 synchronized 块导致虚拟线程在数据库操作时被钉住,严重时载体线程全部被占用,吞吐量反而低于平台线程。MySQL Connector/J 8.2.0 加 已将关键路径改为 ReentrantLock。启用虚拟线程前,务必检查所有依赖的 synchronized 使用情况。

public class LegacyConnectionPool {
    private final Object lock = new Object();
    public Connection getConnection() {
        synchronized (lock) {
            return pool.borrowObject();
        }
    }
}

public class ModernConnectionPool {
    private final ReentrantLock lock = new ReentrantLock();
    public Connection getConnection() {
        lock.lock();
        try {
            return pool.borrowObject();
        } finally {
            lock.unlock();
        }
    }
}

开启 -Djdk.tracePinnedThreads=short 后,真正的 pinning 会由 JVM 打印出来:

java -Djdk.tracePinnedThreads=short -jar target/app.jar

    java.base/java.lang.VirtualThread$VThreadContinuation.onPinned VirtualThread.java:180
    java.base/java.lang.VirtualThread.parkOnCarrierThread VirtualThread.java:642
    com.example.LegacyConnectionPool.getConnection LegacyConnectionPool.java:716

20.4.5 Spring Boot 3.2 加 虚拟线程基准解读

虚拟线程常见收益来自"让阻塞 I/O 不再长期占住昂贵的平台线程"。在包含 JDBC 与 HTTP 调用的 REST API 中,平台线程模型通常会受限于 Tomcat 工作线程与下游连接池;启用虚拟线程后,请求线程本身不再是主要瓶颈,但吞吐仍会被数据库连接池、下游限流、CPU 与 pinning 限制。结论不是"并发一定提升到 10000",而是:I/O 密集型阻塞代码更容易扩展,CPU 密集型计算几乎不会因此变快。

20.5 异步编排模式

单个 @Async 调用只是起点。在真实的微服务架构中,一个业务操作往往需要调用多个下游服务,并将结果汇总——这就是 fan-out/fan-in(扇出与扇入)模式。CompletableFuture 提供了强大的异步编排能力。

20.5.1 CompletableFuture 核心组合操作

@Service
public class ProductAggregationService {

    @Autowired private ProductService productService;
    @Autowired private PriceService priceService;
    @Autowired private ReviewService reviewService;
    @Autowired private InventoryService inventoryService;

    public CompletableFuture<ProductDetail> getProductDetail(Long productId) {
        CompletableFuture<Product> productFuture =
            productService.findById(productId);
        CompletableFuture<Price> priceFuture =
            priceService.getPrice(productId);
        CompletableFuture<List<Review>> reviewsFuture =
            reviewService.getReviews(productId);
        CompletableFuture<Inventory> inventoryFuture =
            inventoryService.checkStock(productId);

        return CompletableFuture.allOf(
                productFuture, priceFuture, reviewsFuture, inventoryFuture)
            .thenApply(v -> ProductDetail.builder()
                .product(productFuture.join())
                .price(priceFuture.join())
                .reviews(reviewsFuture.join())
                .inventory(inventoryFuture.join())
                .build());
    }
}

fan-out/fan-in 后,总耗时接近最慢的下游,而非四个下游耗时相加:

INFO  [price-async-2] c.example.PriceService : price=42 done in 59ms
INFO  [review-async-3] c.example.ReviewService : reviews=42 done in 83ms
INFO  [inventory-async-4] c.example.InventoryService : inventory=42 done in 70ms
INFO  [http-nio-8080-exec-9] c.example.ProductAggregationService : product detail aggregated id=42 elapsed=86ms

20.5.2 CompletableFuture 链式操作速查

转换操作:

future.thenApply(order -> order.getId())

future.thenCompose(order ->
    shipmentService.createShipment(order.getId()))

customerFuture.thenCombine(orderFuture,
    (customer, order) -> new CustomerOrder(customer, order))

聚合操作:

CompletableFuture.allOf(f1, f2, f3)
    .thenApply(v -> List.of(f1.join(), f2.join(), f3.join()))

CompletableFuture.anyOf(primary, fallback)
    .thenApply(result -> (Response) result)

异常处理:

future.exceptionally(ex -> {
    log.warn("Fallback {}", ex.getMessage());
    return DEFAULT_VALUE;
})

future.handle((result, ex) -> {
    if (ex != null) return fallback;
    return result;
})

future.whenComplete((r, ex) -> metrics.record(ex == null))

20.5.3 超时控制

CompletableFuture<Response> result = callRemoteService()
    .orTimeout(3, TimeUnit.SECONDS);

CompletableFuture<Response> result2 = callRemoteService()
    .completeOnTimeout(Response.empty(), 3, TimeUnit.SECONDS);

public CompletableFuture<SearchResult> search(String query) {
    CompletableFuture<List<Product>> dbSearch =
        productRepo.searchAsync(query)
            .completeOnTimeout(List.of(), 2, TimeUnit.SECONDS);

    CompletableFuture<List<Product>> esSearch =
        elasticService.searchAsync(query)
            .completeOnTimeout(List.of(), 2, TimeUnit.SECONDS);

    return dbSearch.thenCombine(esSearch, (db, es) -> {
        List<Product> merged = new ArrayList<>(db);
        merged.addAll(es);
        return new SearchResult(merged);
    });
}

completeOnTimeout 会把慢服务替换成默认值;orTimeout 则会让 Future 以异常完成:

20.5.4 电商商品详情页聚合服务案例

在一个典型的电商平台中,商品详情页需要聚合来自 5 至 8 个微服务的数据:商品基本信息、价格(可能有动态促销价)、库存、评论摘要、推荐商品、优惠券、配送信息等。如串行调用,每个服务 50 毫秒,总耗时 400 毫秒;使用 CompletableFuture.allOf() 并行调用后,总耗时约等于最慢的那个服务(约 80 毫秒)。

注意两点:使用自定义 ThreadPoolTaskExecutor 而非 ForkJoinPool.commonPool(),避免 CPU 密集型任务饿死 I/O 任务;对每个下游调用设置 completeOnTimeout(),防止一个慢服务拖垮整个页面。

20.6 陷阱与限制

异步编程的每一个便利背后,都有一个等着踩的坑。以下是 Spring 异步与调度中最常见的七个陷阱。

20.6.1 陷阱一:@Async 加 @Transactional 同一方法导致事务丢失

@Async 将方法执行切换到新线程,而 Spring 事务绑定在 ThreadLocal 上。新线程没有原始线程的事务上下文,因此 @Transactional 在 @Async 方法中会开启一个全新的事务(而非参与调用方的事务)。如调用方回滚,异步方法的事务不会受影响——数据不一致。

图 20-11 · @Async + @Transactional 事务隔离问题(主线程 TX-A 回滚 / 异步线程 TX-B 提交 → 数据不一致!)

正确做法是分离关注点:

@Async
public void processOrderAsync(Long orderId) {
    orderTransactionService.processInTransaction(orderId);
}

@Service
public class OrderTransactionService {
    @Transactional
    public void processInTransaction(Long orderId) {
    }
}

20.6.2 陷阱二:@Async 自调用不走代理

与所有基于 AOP 代理的注解一样(@Transactional、@Cacheable),同一个类内部的方法调用(this.asyncMethod())不会经过代理,@Async 注解被完全忽略——方法在调用方线程同步执行。

@Service
public class OrderService {
    public void createOrder(Order order) {
        orderRepo.save(order);
        this.sendNotification(order);
    }

    @Async
    public void sendNotification(Order order) {
        notificationClient.send(order);
    }
}

修复方案一:注入自身代理(用 @Lazy 避免循环):

@Service
public class OrderService {
    @Autowired @Lazy
    private OrderService self;

    public void createOrder(Order order) {
        orderRepo.save(order);
        self.sendNotification(order);
    }

    @Async
    public void sendNotification(Order order) {
    }
}

修复方案二(推荐):拆分到独立 Bean:

@Service
public class OrderService {
    @Autowired
    private NotificationService notificationService;

    public void createOrder(Order order) {
        orderRepo.save(order);
        notificationService.sendAsync(order);
    }
}

@Service
public class NotificationService {
    @Async
    public void sendAsync(Order order) {
    }
}

20.6.3 陷阱三:默认 SimpleAsyncTaskExecutor 导致 OOM

在纯 Spring Framework 中,或 Spring Boot 的 applicationTaskExecutor 自动配置被禁用、绕开或无法被唯一解析时,@Async 才可能回退到 SimpleAsyncTaskExecutor。它为每个任务创建新线程,无上限,高并发下分分钟 OOM。

spring:
  task:
    execution:
      pool:
        core-size: 10
        max-size: 50
        queue-capacity: 200
      thread-name-prefix: "app-async-"
      shutdown:
        await-termination: true
        await-termination-period: 60s

从 Spring Boot 2.1 起,如没有自定义 TaskExecutor Bean,Spring Boot 会自动创建一个 ThreadPoolTaskExecutor,可通过 spring.task.execution.* 属性配置。只有当没有可解析的 Executor 或 TaskExecutor(例如禁用了相关自动配置、定义了多个候选但没有指定 @Async value、或自定义异步配置没有暴露执行器)时,才会走框架级回退路径。

20.6.4 陷阱四:@Scheduled 单线程默认,一个慢任务阻塞全部

@EnableScheduling 默认创建 poolSize 等于 1 的 TaskScheduler。一个执行 30 秒的 @Scheduled 方法会阻塞所有其他定时任务 30 秒。

spring:
  task:
    scheduling:
      pool:
        size: 5
      thread-name-prefix: "sched-"
      shutdown:
        await-termination: true
        await-termination-period: 30s

20.6.5 陷阱五:SecurityContext 不传播到 @Async 线程

Spring Security 的 SecurityContext 默认存储在 ThreadLocal 中。@Async 切换线程后,异步方法中 SecurityContextHolder.getContext().getAuthentication() 返回 null——所有基于权限的检查(@PreAuthorize)都会失败。

推荐方案:使用 DelegatingSecurityContextExecutor 包装线程池:

@Bean("secureExecutor")
public Executor secureExecutor() {
    ThreadPoolTaskExecutor delegate = new ThreadPoolTaskExecutor();
    delegate.setCorePoolSize(10);
    delegate.setMaxPoolSize(50);
    delegate.initialize();
    return new DelegatingSecurityContextAsyncTaskExecutor(delegate);
}

@Async("secureExecutor")
public void auditAction(String action) {
    Authentication auth = SecurityContextHolder.getContext().getAuthentication();
    auditLog.record(auth.getName(), action);
}

避免使用 SecurityContextHolder.MODE_INHERITABLETHREADLOCAL——这种方式对线程池不安全:线程被复用时会携带上一个请求的 SecurityContext。

20.6.6 陷阱六:Virtual Thread 与 synchronized 的 Pinning 性能退化

虚拟线程在 synchronized 块中执行阻塞操作时会被钉住在载体线程上,失去虚拟线程的核心优势。ForkJoinPool 默认载体线程数等于 CPU 核心数——如所有载体线程都被 pinned,系统吞吐量骤降。

java -Djdk.tracePinnedThreads=short -jar target/app.jar
java -Djdk.tracePinnedThreads=full -jar target/app.jar

Thread[#42,ForkJoinPool-1-worker-3,5,main]
    java.base/java.lang.VirtualThread$VThreadContinuation.onPinned VirtualThread.java:180
    com.example.LegacyPool.getConnection LegacyPool.java:55

20.6.7 陷阱七:CompletableFuture.get() 阻塞事件循环线程

在 WebFlux(Reactive)或 Netty EventLoop 线程中调用 CompletableFuture.get() 或 .join() 会阻塞事件循环线程,导致所有请求处理停滞——这是经典的 Reactive 死锁。

@GetMapping("/product/{id}")
public Mono<ProductDetail> getProduct(@PathVariable Long id) {
    CompletableFuture<ProductDetail> future = aggregationService.getDetail(id);
    return Mono.fromFuture(future);
}
