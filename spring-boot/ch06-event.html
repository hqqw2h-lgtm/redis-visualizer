第 6 章 · 事件机制

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

Spring Event 是进程内的观察者模式实现,作用范围限定于同一个 ApplicationContext。其核心组件包括 ApplicationEventPublisher(发布入口)、ApplicationEventMulticaster(多播器)、ApplicationListener(监听器接口)与 @EventListener(注解形式)。事件机制的设计目标是组件解耦,而非消息持久化或跨进程投递。

发布者只表达事实(订单已创建、用户已登录、应用已就绪),不依赖监听器实现。ApplicationEventMulticaster 负责类型匹配、监听器排序、同步或异步调用、错误处理。@EventListener 注解的方法在容器启动后由 EventListenerMethodProcessor 包装为 ApplicationListenerMethodAdapter 注册到 Multicaster。

@TransactionalEventListener 不在事件发布时立即执行,而是注册 TransactionSynchronization,等待事务管理器在 BEFORE_COMMIT、AFTER_COMMIT、AFTER_ROLLBACK、AFTER_COMPLETION 阶段触发。本章按"发布管线 → Multicaster → 同步异步 → 注解适配 → 事务事件 → 顺序错误处理 → 排查流"七层组织。

public record OrderCreatedEvent(String orderId, BigDecimal amount) {}

@Service
class OrderService {
    private final ApplicationEventPublisher publisher;

    OrderService(ApplicationEventPublisher publisher) {
        this.publisher = publisher;
    }

    public Long createOrder(BigDecimal amount) {
        publisher.publishEvent(new OrderCreatedEvent("O1001", amount));
        return 1001L;
    }
}

@Component
class OrderAuditListener {
    @EventListener
    void audit(OrderCreatedEvent event) {
        log.info("audit orderId={}, amount={}", event.orderId(), event.amount());
    }
}

@Bean(name = "applicationEventMulticaster")
ApplicationEventMulticaster applicationEventMulticaster() {
    SimpleApplicationEventMulticaster multicaster = new SimpleApplicationEventMulticaster();
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setThreadNamePrefix("event-mc-");
    executor.setCorePoolSize(4);
    executor.setMaxPoolSize(16);
    executor.setQueueCapacity(200);
    executor.initialize();
    multicaster.setTaskExecutor(executor);
    multicaster.setErrorHandler(error ->
        log.error("event listener failed: {}", error.toString(), error));
    return multicaster;
}

6.4 @EventListener 注解适配

图 6-4-1 · @EventListener 注解适配时序(每个 bean 扫描 → Adapter → Multicaster)

EventListenerMethodProcessor 在容器启动后期(afterSingletonsInstantiated)扫描所有 Bean 的 @EventListener 方法,为每个方法创建 ApplicationListenerMethodAdapter 并注册到 Multicaster。该扫描只发生一次。

表 6-4-1:@EventListener 关键能力

能力 配置 适用场景
condition (SpEL) condition = "#event.amount() > 1000" 业务规则过滤
classes 多事件类型 classes = 一个方法处理多种事件
返回值再发布 方法返回值非 null 时自动发布 事件链反应式编排
异步 @Async + @EventListener 不阻塞发布线程
事务阶段 @TransactionalEventListener 提交后或回滚后执行
泛型支持 @EventListener void f(EntityChangedEvent) 仅匹配特定泛型实参 
@EventListener(condition = "#event.amount().compareTo(T(java.math.BigDecimal).valueOf(1000)) > 0")
LargeOrderEvent detectLargeOrder(OrderCreatedEvent event) {
    log.info("large order detected: {}", event.orderId());
    return new LargeOrderEvent(event.orderId());
}

@EventListener
void notifyManager(LargeOrderEvent event) {
    log.info("manager notified for orderId={}", event.orderId());
}
图 6-4-2 · 返回值事件链(@EventListener 返回值自动作为新事件发布)

返回值发布机制使得简单的事件编排无需手动调用 publishEvent。返回值为 null 时不发布;返回 Collection 或数组时逐个发布。

图 6-7-2 · 泛型事件类型解析(ResolvableTypeProvider 解决泛型擦除)
class EntityChangedEvent<T> extends ApplicationEvent implements ResolvableTypeProvider {
    private final T entity;

    EntityChangedEvent(Object source, T entity) {
        super(source);
        this.entity = entity;
    }

    @Override
    public ResolvableType getResolvableType() {
        return ResolvableType.forClassWithGenerics(
            getClass(),
            ResolvableType.forInstance(entity));
    }
}

ResolvableTypeProvider 让 Multicaster 可在运行时获得精确的泛型实参,使 EntityChangedEvent<Order> 与 EntityChangedEvent<Product> 的监听器能被正确区分。

6.末 本章要点

  • Spring Event 是进程内观察者模式,作用范围限定于同一个 ApplicationContext。ApplicationEventPublisher 接口由 ApplicationContext 实现,事件被交给 ApplicationEventMulticaster 进行多播。

  • POJO 事件被 AbstractApplicationContext.publishEvent 包装为 PayloadApplicationEvent。SimpleApplicationEventMulticaster 是默认实现,执行类型匹配、监听器排序、同步或异步调用、错误处理。

  • @EventListener 在容器启动后期由 EventListenerMethodProcessor.afterSingletonsInstantiated 扫描并注册为 ApplicationListenerMethodAdapter。注解支持 SpEL condition、多事件类型、返回值再发布、泛型类型匹配。

  • 默认事件交付为同步。异步实现有两条路径:Multicaster 配置全局 TaskExecutor 或单个监听器加 @Async。异步监听器异常默认被吞,需配置 AsyncUncaughtExceptionHandler。

  • @TransactionalEventListener 通过注册 TransactionSynchronization 使监听器在事务的 BEFORE_COMMIT、AFTER_COMMIT、AFTER_ROLLBACK、AFTER_COMPLETION 阶段执行。默认阶段为 AFTER_COMMIT。无活跃事务时默认静默跳过,可通过 fallbackExecution=true 改为立即执行。

  • @Order 控制同步监听器顺序与异步监听器的提交顺序,但不保证异步完成顺序。同步链中前一个监听器抛异常会中断后续,需配置 ErrorHandler 才能继续。

  • 生产排查的六类常见失败:无事务的事务监听静默跳过、异步异常被吞、同步异常中断、泛型事件未实现 ResolvableTypeProvider 导致误匹配、父子 Context 监听器无法匹配、@EventListener 方法可见性问题。

  • Spring Event 不是消息队列,设计边界明确:无持久化、无消费者组、无跨进程投递、无重试协议。需要这些能力时应改用 Outbox Pattern + Kafka / RabbitMQ / Spring Cloud Stream。
