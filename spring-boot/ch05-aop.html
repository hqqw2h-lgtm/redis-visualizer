第 5 章 · AOP 代理机制

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

面向切面编程(Aspect-Oriented Programming,AOP)将横切关注点(事务、日志、安全、缓存、追踪)从业务逻辑中分离,以声明式方式织入目标对象。Spring AOP 采用运行时代理实现,在 Bean 初始化完成后由 AnnotationAwareAspectJAutoProxyCreator 决定是否生成代理对象。

代理生成有两种路径:JDK 动态代理(java.lang.reflect.Proxy)要求目标实现接口,代理类与目标实现同接口为兄弟关系;CGLIB(Code Generation Library)通过 ASM 字节码框架在运行时生成目标类的子类,代理类继承目标类。Spring Boot 2.x 起默认设置 spring.aop.proxy-target-class=true,优先使用 CGLIB 以保证策略一致性。

本章按"AOP 概念模型 → JDK 动态代理 → CGLIB → 代理类型决策 → 拦截器链 → 自调用与代理陷阱 → AspectJ 编译时织入 → 源码入口与扩展边界"八层组织,系统讲解 Spring AOP 的工作机制与生产环境中最高频的代理失效问题。

5.2 JDK 动态代理

图 5-2-1 · JDK 动态代理架构(接口实现 + InvocationHandler)

JDK 动态代理由 java.lang.reflect.Proxy 在运行时生成,要求目标对象至少实现一个接口。代理类通过反射调用目标方法。

图 5-2-2 · JDK 代理调用时序(@Before / @AfterReturning 包裹反射调用)
public final class $Proxy0 extends Proxy implements Service {
    private static Method m3;

    public $Proxy0(InvocationHandler h) {
        super(h);
    }

    @Override
    public Object doWork(String param) {
        return h.invoke(this, m3, new Object[]{param});
    }

    static {
        m3 = Service.class.getMethod("doWork", String.class);
    }
}

表 5-2-1:反射调用性能演化

调用次数 实现方式 开销特征
1-15 NativeMethodAccessorImpl(JNI 反射) 较大
16+ GeneratedMethodAccessor(动态字节码) 接近直接调用
JIT C2 编译后 内联优化 纳秒级

HotSpot JVM 的反射膨胀阈值由 -Dsun.reflect.inflationThreshold=15 控制。在热点路径上,JDK 动态代理经膨胀与 JIT 编译后,代理开销可忽略。瓶颈通常在拦截器链业务逻辑,而非代理机制本身。

拓展案例:Spring Data JPA Repository

JpaRepository 接口由 Spring Data 在运行时通过 JDK 动态代理生成 $Proxy 实例,内部委托给 SimpleJpaRepository。TransactionalRepositoryProxyPostProcessor 为代理织入 TransactionInterceptor,使所有默认 CRUD 方法自动运行在事务中。

5.3 CGLIB 字节码生成

图 5-3-1 · CGLIB 代理架构(Enhancer 子类 + FastClass 直调)

CGLIB 通过 ASM 字节码框架在运行时生成目标类的子类。Spring Boot 2.x 起默认配置 spring.aop.proxy-target-class=true,即使类实现接口也优先使用 CGLIB。

图 5-3-2 · CGLIB 代理调用时序(MethodProxy.invokeSuper 经 FastClass 直调)

表 5-3-1:FastClass 索引机制

步骤 操作 性能特征
1 方法签名 → hashCode → switch O(1) 哈希查找
2 switch(index) → 直接跳转 完全绕过反射
3 调用 target.method(args) 与直接调用几乎相同 
public class OrderService$$FastClass extends FastClass {

    public int getIndex(String signature) {
        switch (signature.hashCode()) {
            case 3522941:    return 0;  // save
            case -1422950858: return 1;  // findById(Long)
            case 791524107: return 2;  // delete(Long)
        }
        return -1;
    }

    public Object invoke(int index, Object obj, Object[] args) {
        OrderService target = (OrderService) obj;
        switch (index) {
            case 0: return target.save((Order) args[0]);
            case 1: return target.findById((Long) args[0]);
            case 2: target.delete((Long) args[0]); return null;
        }
        throw new IllegalArgumentException("invalid index");
    }
}
图 5-3-3 · ASM 字节码生成流程(Enhancer → defineClass → 实例化)

CGLIB 为每个被代理类生成 3 个 Class 对象(子类 + 2 个 FastClass),驻留 Metaspace。代理大量类的微服务场景需关注 -XX:MaxMetaspaceSize 配置以防 Metaspace OOM。JDK 8 之前对应 PermGen 空间。

拓展案例:@Transactional CGLIB 代理

未实现接口的 Service 类标注 @Transactional 时,Spring 必须使用 CGLIB 代理。开启事务 TRACE 日志可观察代理在目标方法前后包住事务边界。

5.5 拦截器链执行模型

图 5-5-1 · Advice 正常返回执行顺序(@Around 包裹 @Before / Returning / After)
图 5-5-2 · Advice 异常路径(@AfterThrowing 拦截 → @After finally → 异常向上)
图 5-5-3 · proceed() 递归调用栈(ReflectiveMethodInvocation)

ReflectiveMethodInvocation 维护 currentInterceptorIndex 计数器。每次 proceed() 递增索引并执行下一个拦截器;索引超过链长度时调用目标方法。该过程是责任链模式(Gamma 等 1994 年 GoF Design Patterns)的递归实现。

表 5-5-1:Advice 类型对比

Advice 类型 执行时机 必须 proceed 异常拦截
@Before 目标方法前 否 否
@AfterReturning 目标方法正常返回后 否 否
@AfterThrowing 目标方法抛异常时 否 是
@After 目标方法之后(类似 finally) 否 是
@Around 包围整个方法调用 是(否则跳过) 是 
@Aspect
@Component
@Order(10)
public class AuditAspect {
    @Pointcut("execution(* com.example.order..*(..))")
    void orderOperation() {}

    @Before("orderOperation()")
    public void before(JoinPoint jp) {
        System.out.println("[before] " + jp.getSignature().toShortString());
    }

    @AfterReturning(pointcut = "orderOperation()", returning = "result")
    public void afterReturning(Object result) {
        System.out.println("[return] " + result);
    }

    @AfterThrowing(pointcut = "orderOperation()", throwing = "ex")
    public void afterThrowing(RuntimeException ex) {
        System.out.println("[throw] " + ex.getClass().getSimpleName());
    }
}

@Around("execution(* com.example.order.OrderService.pay(..))")
public Object measure(ProceedingJoinPoint pjp) throws Throwable {
    long start = System.nanoTime();
    try {
        Object result = pjp.proceed();
        return result;
    } finally {
        System.out.println("cost=" + (System.nanoTime() - start) + "ns");
    }
}

拓展案例:Micrometer Tracing 拦截器链

Micrometer Tracing(Spring Cloud Sleuth 后继)利用 AOP 拦截器链注入 trace-id 与 span-id。TraceAspect 拦截 @Scheduled、@Async 与 RestTemplate 调用,将每次方法执行包装在 Span 中。

@Aspect
public class TraceAspect {
    @Around("@annotation(org.springframework.scheduling.annotation.Scheduled)")
    public Object traceScheduled(ProceedingJoinPoint pjp) throws Throwable {
        Span span = tracer.nextSpan().name(pjp.getSignature().getName());
        try (Tracer.SpanInScope ws = tracer.withSpan(span.start())) {
            return pjp.proceed();
        } finally {
            span.end();
        }
    }
}

实际拦截器链执行顺序:ExposeInvocationInterceptor → TraceAspect(@Around) → TransactionInterceptor(@Transactional) → 目标方法。

5.6 代理陷阱

图 5-6-1 · Self-Invocation 绕过代理(this 指向目标对象,不经过 Proxy)

表 5-6-1:六类常见代理失效场景

编号 场景 失效原因
1 this 调用同类方法 this 指向原始对象,绕过代理
2 private 方法标 @Transactional CGLIB 无法覆写 private
3 final 方法标切面注解 CGLIB 无法覆写 final
4 非 Spring 管理对象(new 出来) 没有代理

@Service
public class OrderService {
    public void createOrder(String orderNo) {
        System.out.println("[create] tx active = " +
                TransactionSynchronizationManager.isActualTransactionActive());
        this.reserveStock(orderNo);
    }

    @Transactional
    public void reserveStock(String orderNo) {
        System.out.println("[reserve] tx active = " +
                TransactionSynchronizationManager.isActualTransactionActive());
        throw new IllegalStateException("stock not enough");
    }
}

@Service
public class OrderService {
    @Autowired
    private OrderService self;

    }

    }

@EnableAspectJAutoProxy(exposeProxy = true)
@SpringBootApplication
public class App { }

@Service
public class OrderService {
    public void createOrder(Order order) {
        ((OrderService) AopContext.currentProxy()).validate(order);
    }

    @Transactional
    public void validate(Order order) { }
}

拓展案例:@Cacheable 自调用陷阱

Service 类的 getOrder() 内部调用 this.getCachedOrder() 时,即使后者标注 @Cacheable,缓存永远不命中。每次调用都直接打到数据库。修复方式与 @Transactional 一致:AopContext 或注入自身引用,推荐改为职责拆分。

# 修复 proxy.getCachedOrder

5.末 本章要点

  • AOP 是横切关注点(事务、日志、安全、缓存、追踪)分离的实现手段。Spring AOP 采用运行时代理,代理生成在 BeanPostProcessor 阶段由 AbstractAutoProxyCreator 完成。

  • JDK 动态代理由 java.lang.reflect.Proxy 生成,要求目标实现接口,通过反射调用目标方法。HotSpot JVM 反射膨胀阈值为 15 次后生成字节码,JIT 编译后开销可忽略。

  • CGLIB 通过 ASM 生成目标类的子类,使用 MethodInterceptor 拦截调用,通过 FastClass 索引实现对目标方法的零反射调用。每个被代理类生成 3 个 Class 对象,需关注 Metaspace 容量。

  • Spring Boot 2.x 默认 proxyTargetClass=true,即使类实现接口也优先使用 CGLIB,以保证策略一致性。final 类无法被 CGLIB 代理。

  • 拦截器链是责任链模式的递归实现。ReflectiveMethodInvocation.proceed() 按 currentInterceptorIndex 递归调用下一个拦截器,索引超过链长度时调用目标方法。

  • 自调用(self-invocation)是最常见的代理失效。this.method() 直接调用目标对象,绕过代理,导致 @Transactional、@Cacheable、@Async 失效。修复方式按推荐顺序为:职责拆分 > 注入自身引用 > AopContext.currentProxy。

  • AspectJ 编译时织入支持字段、构造器、private 方法等更广泛的 JoinPoint,且无自调用问题,代价是构建复杂度与调试难度。仅在代理限制确实阻塞时才引入。

  • 源码核心入口:AnnotationAwareAspectJAutoProxyCreator → ProxyFactory → JdkDynamicAopProxy / CglibAopProxy → ReflectiveMethodInvocation.proceed。验证代理生效的标准做法是打印 bean.getClass() 与 AopUtils.isAopProxy(bean)。
