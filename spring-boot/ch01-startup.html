第 1 章 · 启动流程

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

SpringApplication.run() 是 Spring Boot 应用的统一入口。该方法表面上仅一行调用,内部由 12 个阶段串联组成完整启动流水线。每个阶段对应一组确定的源码入口、可观察事件与扩展点。

启动流程的工程价值体现在三个维度:其一是诊断维度,启动失败必然落在某一具体阶段,精确分阶段是定位故障的前提;其二是性能维度,冷启动总耗时由各阶段叠加,微服务与 Serverless 场景下需对单阶段进行针对性优化;其三是扩展维度,自定义 Starter、配置中心集成、服务注册都依赖启动事件提供的扩展点。

本章在第 0 章 Quick Start 项目基础上,沿时间轴逐阶段展开 SpringApplication.run() 的内部状态机:SpringApplication 构造期推断 WebApplicationType、SPI 机制按 spring.factories 与 AutoConfiguration.imports 加载扩展、Environment 准备阶段建立 PropertySource 优先级栈、ApplicationContext.refresh() 完成 13 个内部子步骤、嵌入式 Web 容器在指定子步骤绑定端口、监听器按 7 类启动事件接收回调。

1.2 SpringApplication 构造与 WebApplicationType

图 1-2-1 · SpringApplication 构造器内部 7 步流程
图 1-2-2 · WebApplicationType 三级推断决策树
static WebApplicationType deduceFromClasspath() {
    if (ClassUtils.isPresent("org.springframework.web.reactive.DispatcherHandler", null)
        && !ClassUtils.isPresent("org.springframework.web.servlet.DispatcherServlet", null)
        && !ClassUtils.isPresent("org.glassfish.jersey.servlet.ServletContainer", null)) {
        return WebApplicationType.REACTIVE;
    }
    for (String className : SERVLET_INDICATOR_CLASSES) {
        if (!ClassUtils.isPresent(className, null)) {
            return WebApplicationType.NONE;
        }
    }
    return WebApplicationType.SERVLET;
}

表 1-2-1:WebApplicationType 三类

类型 推断条件 选用 Context 默认嵌入式容器
SERVLET classpath 包含 Servlet API 与 ConfigurableWebApplicationContext AnnotationConfigServletWebServerApplicationContext Tomcat
REACTIVE classpath 仅含 DispatcherHandler 而无 DispatcherServlet AnnotationConfigReactiveWebServerApplicationContext Netty
NONE 其他情形 AnnotationConfigApplicationContext 不创建嵌入式容器

spring-webmvc 与 spring-webflux 同时存在 classpath 时,推断结果为 SERVLET。该规则在引入 starter 时易引发推断偏差,排查路径包括 mvn dependency:tree 与 spring.main.web-application-type=reactive 强制覆盖。

图 1-3-1 · spring.factories 加载流程(从 jar 到 SPI 缓存)

SpringFactoriesLoader.loadFactories() 通过 ClassLoader.getResources() 枚举所有 jar 包中的 META-INF/spring.factories,解析后按接口分组合并,缓存供 SpringApplication 构造期使用。

ClassLoader 双亲委派模型决定了 spring.factories 的可见范围。同名 key 的多个 value 在合并时去重保留,顺序依赖 ClassLoader.getResources() 的枚举顺序。

拓展案例:Nacos Config 高优先级注入

Nacos 通过实现 PropertySourceLocator 接口,在 Environment 准备阶段从 Nacos Server 拉取配置并作为高优先级 PropertySource 注入。该 PropertySource 优先级高于 application.yml,确保远程配置可覆盖本地默认值,保证 @Value 与 @ConfigurationProperties 绑定到远程值。

public class NacosPropertySourceLocator
        implements PropertySourceLocator {
    @Override
    public PropertySource<?> locate(Environment env) {
        ConfigService configService = NacosFactory.createConfigService(properties);
        String config = configService.getConfig(dataId, group, timeout);
        return new NacosPropertySource(dataId, mapFromConfig(config));
    }
}

拓展案例:Gateway 选择 Netty

Spring Cloud Gateway Server WebFlux 引入 spring-boot-starter-webflux,触发 REACTIVE 推断,默认选用 Netty。Netty 事件循环模型与 Project Reactor 背压机制配合,支持 Gateway 过滤器链的异步处理。误以 Servlet 模式启动会使背压、路由过滤器与 EventLoop 假设全部失效。

@Bean
public RouteLocator customRoutes(RouteLocatorBuilder builder) {
    return builder.routes()
        .route("order-service", r -> r.path("/api/orders/**")
            .filters(f -> f
                .circuitBreaker(c -> c.setName("orderCB"))
                .retry(3))
            .uri("lb://order-service"))
        .build();
}

1.末 本章要点

  • SpringApplication.run() 由 12 个阶段构成完整启动流水线。阶段 10 (refresh) 占用启动总耗时的 70 % 至 90 %,是性能优化的重点。

  • WebApplicationType.deduceFromClasspath() 通过 classpath 上 DispatcherHandler、DispatcherServlet、ServletContainer 三类指示类的存在性推断应用类型。spring-webmvc 与 spring-webflux 同存时优先选择 SERVLET。

  • SPI 机制由 SpringFactoriesLoader 加载 META-INF/spring.factories 中的 SPI 接口实现。Boot 3 将自动配置类迁移到 META-INF/spring/...AutoConfiguration.imports,其他 SPI 仍走 spring.factories。

  • Environment 由有序 PropertySource 列表组成。优先级遵循"越晚绑定越优先":命令行参数 > SPRING_APPLICATION_JSON > JVM 系统属性 > 环境变量 > application.yml。

  • AbstractApplicationContext.refresh() 由 13 个子步骤组成。子步骤 5 (invokeBeanFactoryPostProcessors) 完成 BeanDefinition 注册,子步骤 9 (onRefresh) 创建嵌入式 WebServer,子步骤 11 (finishBeanFactoryInitialization) 实例化所有非懒加载单例,占 refresh 总耗时的 50 % 以上。

  • 嵌入式容器选择由 WebApplicationType 与 classpath 共同决定。SERVLET 默认 Tomcat,REACTIVE 默认 Netty。容器底层均使用 NIO Selector,Linux 映射 epoll、macOS 映射 kqueue、Windows 映射 IOCP。

  • 启动事件包含 7 类:starting、environmentPrepared、contextPrepared、contextLoaded、started、ready、failed。@EventListener 仅可见 Context 创建后的事件,完整事件链需通过 SPI 或 addListeners 注册。

  • 启动性能优化包含 lazy-init、AOT、CDS、Native Image、Context Indexer 五类技术。Native Image 将冷启动从秒级降至 50-200 ms,适用于 Serverless 与 CLI 场景。
