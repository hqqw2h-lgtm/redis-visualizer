应用上线后用户反馈"慢了",如果没有可观测体系,只能猜。Metrics + Traces + Logs 三支柱协同,可在 30 秒内定位到具体的数据库查询。本章按"三支柱模型 → Micrometer 门面 → 分布式追踪 → Actuator 端点 → Exemplar 关联 → 陷阱"展开,目标是把可观测性当成一等公民设计进 Spring Boot 应用,而不是事后补一层日志聚合。

读完本章应能:用 Observation 一次埋点同时产出指标 / Span / 带 traceId 的日志,用 W3C TraceContext 做跨服务追踪,把 Actuator 端点合理拆成 liveness / readiness / startup,并用 Exemplar 把 Grafana 异常点直达 Tempo Trace,从而让告警 → Trace → Log → 根因的闭环在 30 秒内走完。

30.1 可观测性三支柱

可观测性(Observability)指从系统外部输出推断内部状态的能力。现代分布式系统建立在三支柱之上。

图 30-1-1:三支柱协同

flowchart LR M[Metrics 数值趋势 / 告警] --> TR[定位入口 哪里异常] L[Logs 业务事件 / 错误上下文] --> TR T[Traces 跨服务因果链] --> TR TR --> TID[traceId / spanId 串起三类信号] TID --> R[排障闭环 告警 → Trace → Log → 根因]

30.1.1 协同链路

单独使用任何一个支柱都不够。实际排障流程:告警(Metrics)→ 错误日志(Logs)→ 找到 traceId → 跨服务追踪(Traces)→ 定位根因。三者通过 traceId 串联,这正是 Exemplar 与 MDC 的价值。

30.1.2 SkyWalking vs Micrometer

Apache SkyWalking 用 Java Agent(字节码注入)实现零侵入追踪;Micrometer Tracing 是 Spring Boot 3 的观测门面:HTTP 服务端、常见客户端自动接入,业务级 span 用 Observation API 补充。SkyWalking 开箱即用 + 拓扑 + UI;Micrometer 更贴近 Spring 生态,后端可换 Zipkin / Tempo / OTLP。

表 30-1-1:两种方案对比

维度 SkyWalking Micrometer Tracing
接入 Java Agent 零代码 自动 + Observation 手动
内存 Agent 约 30MB 进程内低开销
UI 自带拓扑 搭 Grafana / Tempo
后端 私有 OAP 可换 Zipkin / Tempo / OTLP
灵活度 改 Agent 较重 业务可控 
@RestController
public class OrderController {
    private final ObservationRegistry registry;

    @GetMapping("/orders/{id}")
    public Order getOrder(@PathVariable Long id) {
        return Observation.createNotStarted("order.query", registry)
            .lowCardinalityKeyValue("order.type", "standard")
            .observe(() -> orderService.findById(id));
    }
}

30.2 Micrometer 门面架构

Micrometer 是 Spring Boot 的指标门面(类 SLF4J 之于日志):应用代码只依赖 Micrometer API,底层可对接 Prometheus / Datadog / CloudWatch / Graphite / InfluxDB / NewRelic 等。

图 30-2-1:门面架构

flowchart TD APP["应用代码 @Timed @Counted MeterRegistry"] APP --> MR[MeterRegistry 核心接口] MR --> CR[CompositeMeterRegistry 同时推多后端] CR --> P[PrometheusMeterRegistry] CR --> D[DatadogMeterRegistry] CR --> CW[CloudWatchMeterRegistry] CR --> G[GraphiteMeterRegistry] CR --> I[InfluxMeterRegistry] CR --> NR[NewRelicMeterRegistry]

表 30-2-1:四大 Meter 类型

Meter 记录什么 业务示例 Prometheus 形态
Counter 只增不减 订单总数 / 登录失败 *_total
Gauge 当前瞬时 队列深度 / 在线连接 单条当前值
Timer 次数 + 总耗时 + 分布 HTTP 请求 / 支付 *_seconds_count/sum/bucket
DistributionSummary 非时间分布 订单金额 / 消息大小 *_count/sum/bucket

30.2.1 自定义业务指标

@Component
public class OrderMetrics {

    private final Counter orderCounter;
    private final AtomicInteger pendingOrders = new AtomicInteger();
    private final Timer paymentTimer;
    private final DistributionSummary orderAmount;

    public OrderMetrics(MeterRegistry registry) {
        this.orderCounter = Counter.builder("orders.created")
            .tag("channel", "web")
            .description("已创建订单总数")
            .register(registry);

        Gauge.builder("orders.pending", pendingOrders, AtomicInteger::get)
            .description("当前待处理订单数")
            .register(registry);

        this.paymentTimer = Timer.builder("payment.duration")
            .publishPercentiles(0.5, 0.95, 0.99)
            .serviceLevelObjectives(Duration.ofMillis(200), Duration.ofSeconds(1))
            .register(registry);

        this.orderAmount = DistributionSummary.builder("order.amount")
            .baseUnit("CNY")
            .publishPercentiles(0.5, 0.9)
            .register(registry);
    }

    public void onOrderCreated(Order order) {
        orderCounter.increment();
        pendingOrders.incrementAndGet();
        orderAmount.record(order.getTotalAmount());
    }

    public void onPaymentCompleted(Runnable payment) {
        paymentTimer.record(payment);
        pendingOrders.decrementAndGet();
    }
}

30.3 分布式追踪

Dapper(Google, 2010)提出 Trace-Span 模型,定义分布式追踪基本概念,成为后续 Zipkin / Jaeger / OpenTelemetry 的理论基础。

30.3.1 Trace 与 Span 层级

图 30-3-1:Trace → Span 树

flowchart TD TRACE[Trace 一次外部请求完整调用树] --> ROOT[Root Span Gateway /orders] ROOT --> ORD[Child Span OrderController.create] ORD --> DB[Child Span JPA findById] ORD --> PAY[Child Span WebClient → PaymentService] PAY --> STR[Child Span Stripe API call]

30.3.2 W3C TraceContext 跨服务传播

图 30-3-2:W3C TraceContext 传播

sequenceDiagram participant A as Order Service participant B as Order Service participant C as Inventory Service Note over A: 生成 traceId 4bf92f3577b34da6a3ce929d0e0e4736 root span 00f067aa0ba902b7 A->>B: GET /api/orders/O1001 traceparent 00-traceId-spanId-01 Note over B: 解析 traceparent 创建 child span a6f4d2c9b7e01842 B->>A: 返回订单 A->>C: POST /inventory/check 同 traceparent Note over C: 创建 child span d3c91a6679ef1020 C->>A: 返回库存 Note over A: 所有 span 共享同一 traceId

traceparent 格式:00-{traceId}-{spanId}-{flags}。00 版本号,traceId 32 位十六进制(128 bit),spanId 16 位十六进制(64 bit),flags=01 表示采样。W3C TraceContext 是跨厂商标准,取代各自为政的 X-B3-*(Zipkin)与 uber-trace-id(Jaeger)。

30.4 Spring Boot Actuator

Actuator 把健康检查、指标暴露、配置查看等运维能力标准化。

图 30-4-1:Actuator 端点架构

flowchart TD REQ[HTTP / JMX 请求] --> M[Endpoint Handler Mapping] M --> EP[Actuator 端点 /actuator/*] EP --> H[/health 健康检查/] EP --> ME[/metrics 指标 JSON/] EP --> PR[/prometheus 文本/] EP --> IN[/info 应用信息/] EP --> MG[/env /beans /loggers 敏感/] EP --> HX[/httpexchanges HTTP 交换/] H --> IND[HealthIndicator HealthContributor] IND --> SA[StatusAggregator UP / DOWN / OUT_OF_SERVICE]

Spring Boot 3 用 /actuator/httpexchanges,旧 /httptrace 已不再推荐。/env /beans /threaddump /heapdump /loggers 必须严格保护,/shutdown 生产通常禁用。

表 30-4-1:健康状态优先级

状态 含义 HTTP K8s 行为
DOWN 组件不可用 503 liveness 失败重启;readiness 失败摘流量
OUT_OF_SERVICE 已下线 503 readiness 失败移除负载均衡
UP 正常 200 接收流量
UNKNOWN 未知 200 保持现状

图 30-4-2:Health 聚合决策

flowchart TD AGG[StatusAggregator 汇总所有 HealthIndicator] --> D1[DiskSpace UP 剩余 50GB] AGG --> D2[DataSource UP 连接正常] AGG --> D3[Redis DOWN 连接超时] AGG --> D4[Rabbit UP 通道正常] D1 --> R[聚合规则] D2 --> R D3 --> R D4 --> R R --> F[最终 DOWN 因 Redis DOWN]

聚合规则:任一 DOWN → 整体 DOWN;任一 OUT_OF_SERVICE → 整体 OUT_OF_SERVICE;全部 UP → 整体 UP。

30.4.1 自定义 HealthIndicator

@Component
public class PaymentGatewayHealthIndicator implements HealthIndicator {

    private final PaymentClient paymentClient;

    @Override
    public Health health() {
        try {
            Duration latency = paymentClient.ping();
            if (latency.toMillis() > 5000) {
                return Health.down()
                    .withDetail("latency_ms", latency.toMillis())
                    .withDetail("reason", "网关响应过慢")
                    .build();
            }
            return Health.up().withDetail("latency_ms", latency.toMillis()).build();
        } catch (Exception e) {
            return Health.down(e).build();
        }
    }
}

支付网关 128ms 时输出 UP;超过 5 秒输出 DOWN,Kubernetes readiness 应把 Pod 摘流量。

30.5 Exemplar:指标与追踪桥梁

传统监控中指标和追踪是两个独立系统:Grafana 看到 P99 突增 → 手动到 Jaeger 搜对应时段 Trace,耗时易遗漏。Exemplar 把"异常指标点 → Trace"一键关联。

30.5.2 自定义 Observation 一次产出三类信号

@RestController
class OrderController {

    private final ObservationRegistry observationRegistry;
    private final OrderService orderService;

    @PostMapping("/orders")
    OrderReceipt create(@RequestBody CreateOrderRequest request) {
        Observation observation = Observation.createNotStarted("order.checkout", observationRegistry)
            .contextualName("checkout order")
            .lowCardinalityKeyValue("channel", request.channel())
            .lowCardinalityKeyValue("payment", request.paymentMethod())
            .highCardinalityKeyValue("order.id", request.orderId());

        return observation.observe(() -> {
            observation.event(Observation.Event.of("checkout started"));
            log.info("checkout started orderId={} amount={}", request.orderId(), request.amount());
            OrderReceipt receipt = orderService.create(request);
            observation.event(Observation.Event.of("checkout completed"));
            log.info("checkout completed orderId={} receipt={}", request.orderId(), receipt.id());
            return receipt;
        });
    }
}

ApplicationStartup 仅覆盖启动 启动慢与运行慢可分别定位 不记录业务请求
Micrometer 做门面 后端可换业务少改 适配多后端复杂度
Observation 一次产指标 + Span 一次埋点统一信号 概念抽象

30.6.1 扩展边界

可观测体系应是"分层订阅":业务模块只暴露 Observation,基础设施由自动配置接入 Tracing / Metrics / Logging。禁止业务层直接持有 MeterRegistry 写复杂逻辑;高基数维度只入 trace attribute / log,不入 metric tag;@Async / CompletableFuture 必须用 ContextPropagatingTaskDecorator 传播 TraceContext。

public static void main(String[] args) {
    SpringApplication app = new SpringApplication(ObservabilityApp.class);
    app.setApplicationStartup(new BufferingApplicationStartup(2048));
    app.run(args);
}

启动验证:curl /actuator/startup 返回 step 列表;curl /actuator/prometheus | grep http_server_requests_seconds_count 给出请求计数。

30.7 陷阱与限制

图 30-7-1:五大陷阱全景

flowchart LR ROOT((五大陷阱)) ROOT --> T1[指标基数爆炸] ROOT --> T2[TraceContext 在异步边界丢失] ROOT --> T3[Actuator 暴露敏感端点] ROOT --> T4[健康检查误配 Pod 被错重启] ROOT --> T5[日志只打印不结构化]

30.7.1 陷阱 1:指标基数爆炸

把 customerId orderId URL 路径参数 等高基数值放进 metric tag,每个唯一值都创建新时间序列。100 万用户 × 10 个指标 = 1000 万时序 → Prometheus 内存爆 + 查询超时。修复:metric tag 只用低基数(HTTP method / status code / service name);高基数放 trace span attribute。

30.7.2 陷阱 2:TraceContext 异步边界丢失

CompletableFuture @Async new Thread() 时 TraceContext 不会自动传播 → 下游 span 与上游 traceId 不同 → 调用链断裂。

@Bean
public ThreadPoolTaskExecutor applicationTaskExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);
    executor.setTaskDecorator(new ContextPropagatingTaskDecorator());
    executor.initialize();
    return executor;
}

30.7.3 陷阱 3:Actuator 暴露敏感端点

management.endpoints.web.exposure.include=* 会把 /env(数据库密码) /configprops(API Key) /heapdump(整个堆下载)暴露。修复:① 生产仅暴露 health,info,metrics,prometheus;② 绑定独立 management 端口;③ 网络策略限制访问。

30.7.4 陷阱 4:健康检查误配致 Pod 死循环重启

livenessProbe 指 /actuator/health(含所有组件),Redis 临时不可达 → 整体 DOWN → K8s 重启 → 重启后 Redis 仍不可达 → 再次重启 → 死循环。修复:分离 liveness / readiness:liveness 指 /actuator/health/liveness(仅 livenessState);readiness 指 /actuator/health/readiness(可包含外部依赖)。

30.7.5 陷阱 5:日志只打印不结构化

纯文本日志在分布式环境难以聚合检索。修复:JSON / Logstash 格式 + MDC 自动注入 traceId,Loki / Elasticsearch 可按 traceId / serviceId 等结构字段查询。

异步丢 traceId 调用链断裂 未传播上下文 ContextPropagatingTaskDecorator
