在分布式系统中,服务之间的直接调用会带来强耦合与级联故障风险。消息队列(Message Queue)通过引入中间代理(Broker),让生产者与消费者在时间与空间上解耦——生产者不必知道消费者是谁,消费者不必在生产者发送时在线。JMS(Java Message Service)是 Java 平台对消息中间件的标准抽象,而 Spring 通过 JmsTemplate 与 @JmsListener 将这套 API 简化到极致。

本章按"JMS 规范 → JmsTemplate 发送 → @JmsListener 消费 → 事务可靠性 → Spring Boot 自动配置 → 陷阱与限制"的顺序,逐层深入 Spring Boot 对 JMS 的封装。前置知识:第 21 章 消息驱动架构(Spring Messaging 抽象层),第 16 章 事务管理(有助于理解 JMS 事务管理)。

22.1 JMS 规范与架构

JMS(Jakarta 迁移后称 Jakarta Messaging)定义了 Java 应用程序与消息中间件交互的标准 API。它类似于 JDBC 之于数据库——提供统一的编程接口,底层由各消息中间件厂商实现。JMS 1.1(JSR 914,2002 年)统一了 P2P 与 Pub 或 Sub 两种模型的 API;JMS 2.0(JSR 343,2013 年)引入了简化 API(JMSContext)、共享订阅与延迟发送等特性。Spring Boot 3.x 使用 jakarta.jms.* 包名,老项目中的 javax.jms.* 示例需要整体迁移。

22.1.1 核心 API 类层次

图 22-1 · JMS 核心 API 类层次结构(ConnectionFactory → Connection → Session → Producer/Consumer)

JMS API 遵循工厂模式的级联创建:ConnectionFactory → Connection → Session → MessageProducer 或 MessageConsumer。每一层都管理着特定的资源与生命周期。Connection 代表与 Broker 的 TCP 连接(重量级资源,应当池化),Session 是轻量级的单线程上下文(事务与确认的边界),MessageProducer 与 MessageConsumer 是绑定到特定 Destination 的发送与接收端点。

22.1.2 JMS 2.0 简化 API 与 JmsTemplate

JMS 2.0 引入了 JMSContext,它将 Connection 与 Session 合二为一,并支持 try-with-resources 自动关闭。但在 Spring 生态中,几乎不会直接使用 JMSContext——JmsTemplate 早在 JMS 2.0 之前就已提供了比 JMSContext 更优雅的简化方案,且它内部仍使用 JMS 1.1 API 以保持最大兼容性。

22.1.3 Point-to-Point 模型 Queue

在 P2P 模型中,消息被发送到一个队列(Queue)。每条消息只被一个消费者接收与处理——即使有多个消费者监听同一个队列,Broker 也会确保消息仅投递给其中一个(竞争消费)。这非常适合任务分发场景:多个 Worker 从同一队列取任务,每个任务只需处理一次。

图 22-2 · Point-to-Point Queue 消息模型(竞争消费 — 1 条消息仅 1 消费者)

规则:每条消息只被一个 Consumer 消费;消息持久化到 Broker 直到被消费;多 Consumer 等于 竞争消费(负载均衡)。

22.1.4 Publish-Subscribe 模型 Topic

在 Pub 或 Sub 模型中,消息被发布到一个主题(Topic)。所有当前活跃的订阅者都会收到消息副本——这是一对多的广播模式。典型场景包括事件通知、配置变更广播、日志分发等。注意:如订阅者在消息发布时不在线(且没有创建持久订阅),消息会丢失。

图 22-3 · Publish-Subscribe Topic 消息模型(广播 — 1 条消息所有订阅者各收 1 份)
维度 Queue P2P Topic Pub Sub
消费者关系 竞争消费,一条消息只投递给一个 广播消费,每个订阅者都收到副本
消息持久化 始终持久化直到被消费或过期 仅持久订阅者的消息才持久化
典型场景 任务分发、订单处理、工作队列 事件通知、配置广播、日志分发
消费者在线要求 不要求 非持久订阅要求在线;持久订阅不要求
Spring Boot 默认 spring.jms.pub-sub-domain=false spring.jms.pub-sub-domain=true

22.1.5 JMS 五种消息类型

JMS 定义了五种标准消息体类型,每一种封装不同格式的载荷。在 Spring Boot 中,通常不需要手动构造这些消息对象——JmsTemplate.convertAndSend() 配合 MessageConverter 会自动完成对象到消息的转换。

消息类型 载荷格式 典型用途 Spring 推荐度
TextMessage String JSON XML 最常用:JSON 序列化的业务对象 强烈推荐
MapMessage 键值对(基本类型) 简单结构化数据 一般
BytesMessage 原始字节流 二进制数据、文件传输 一般
ObjectMessage Java Serializable Java 对象序列化(有安全风险) 不推荐
StreamMessage 顺序读写的基本类型流 流式处理,极少使用 不推荐

ObjectMessage 使用 Java 原生序列化,存在反序列化攻击(Deserialization Attack)的安全隐患。攻击者可以构造恶意的序列化数据,在反序列化时执行任意代码。ActiveMQ Artemis 默认禁止反序列化未白名单的类。生产环境中应始终使用 TextMessage 加 JSON,而非 ObjectMessage。

22.2 JmsTemplate 发送消息

JmsTemplate 是 Spring 对 JMS API 的核心封装,遵循 Spring 一贯的 Template 模式(类似 JdbcTemplate、RestTemplate)。它处理了 Connection、Session、Producer 的创建与关闭、异常转换(checked 转 unchecked)、以及消息转换,让发送一条消息从原生 JMS 的 20 行加 代码缩减到一行。

22.2.1 三种发送方式

@Autowired
private JmsTemplate jmsTemplate;

public void sendRaw() {
    jmsTemplate.send("order.queue", session -> {
        TextMessage msg = session.createTextMessage();
        msg.setText("{\"orderId\":\"ORD-001\",\"amount\":128.00}");
        msg.setStringProperty("orderType", "STANDARD");
        msg.setJMSCorrelationID("corr-12345");
        return msg;
    });
}

public void sendOrder(Order order) {
    jmsTemplate.convertAndSend("order.queue", order);

    jmsTemplate.convertAndSend("order.queue", order, message -> {
        message.setStringProperty("priority",
            order.isPriority() ? "HIGH" : "NORMAL");
        message.setLongProperty("requestedTtlMs", 60000);
        return message;
    });
}

public OrderConfirmation placeOrder(Order order) {
    Message reply = jmsTemplate.sendAndReceive("order.request", session -> {
        TextMessage msg = session.createTextMessage();
        msg.setText(objectMapper.writeValueAsString(order));
        return msg;
    });

    if (reply instanceof TextMessage textReply) {
        return objectMapper.readValue(textReply.getText(), OrderConfirmation.class);
    }
    throw new IllegalStateException("Unexpected reply type");
}

同一个 JmsTemplate 可以覆盖"只发送"、"对象转换发送"、"请求与应答"三种路径,区别会体现在消息属性与返回体上:

INFO OrderSender : sendRaw destination=order.queue correlationId=corr-12345 orderType=STANDARD
INFO OrderSender : convertAndSend destination=order.queue orderId=ORD-001 priority=HIGH requestedTtlMs=60000
INFO OrderSender : sendAndReceive destination=order.request replyType=TextMessage confirmationId=CNF-001 status=ACCEPTED

22.2.2 MessageConverter 策略

MessageConverter 决定了 convertAndSend() 如何将 Java 对象转换为 JMS Message,以及 @JmsListener 方法如何将 Message 转回 Java 对象。Spring 提供了几种内置实现,其中 MappingJackson2MessageConverter(JSON)是生产环境的最佳选择。

图 22-4 · MessageConverter 策略体系(3 种内置实现 + 推荐路径)
@Configuration
public class JmsConfig {

    @Bean
    public MessageConverter jacksonJmsMessageConverter() {
        MappingJackson2MessageConverter converter =
            new MappingJackson2MessageConverter();
        converter.setTargetType(MessageType.TEXT);
        converter.setTypeIdPropertyName("_type");

        Map<String, Class<?>> typeIdMappings = new HashMap<>();
        typeIdMappings.put("order", Order.class);
        typeIdMappings.put("payment", PaymentEvent.class);
        typeIdMappings.put("shipment", ShipmentNotification.class);
        converter.setTypeIdMappings(typeIdMappings);

        return converter;
    }
}

转换后的 JMS 消息不是 Java 序列化对象,而是带 _type 属性的 TextMessage。消费者端正是依赖这个属性把 JSON 还原为目标类型:

JMSDestination = queue://order.queue
JMSDeliveryMode = PERSISTENT
JMSCorrelationID = corr-12345
_type = order
body = TextMessage(JSON)

如不设置 typeIdPropertyName,消费者端只能接收 Message 或 String 参数,无法自动反序列化为业务对象。

22.2.3 JmsMessagingTemplate Spring Messaging 抽象

JmsMessagingTemplate 是 JmsTemplate 之上的更高层封装,它使用 Spring Messaging 的 Message<T> 抽象(而非 JMS 原生的 jakarta.jms.Message),让 JMS 与 Kafka、AMQP 等消息系统共享统一的编程模型。

@Autowired
private JmsMessagingTemplate messagingTemplate;

public void sendWithHeaders(Order order) {
    Map<String, Object> headers = Map.of(
        "orderType", "EXPRESS",
        "region", "APAC");

    messagingTemplate.convertAndSend("order.queue", order, headers);
}

public void sendSpringMessage(Order order) {
    org.springframework.messaging.Message<Order> message =
        MessageBuilder.withPayload(order)
            .setHeader("orderType", "EXPRESS")
            .setHeader("traceId", UUID.randomUUID().toString())
            .build();

    messagingTemplate.send("order.queue", message);
}

JmsMessagingTemplate 会把 Spring Messaging header 映射为 JMS property,适合跨中间件统一追踪字段。

22.2.4 Destination 解析策略

JmsTemplate 通过 DestinationResolver 将字符串名称(如 "order.queue")解析为 JMS Destination 对象。Spring 提供两种策略:

22.2.6 订单处理系统案例

在一个电商系统中,订单服务在用户下单后,将订单信息发送到 fulfillment.queue。多个仓库的履约工作者(Fulfillment Worker)竞争消费这些订单,根据库存与距离分配仓库。通过 JMS Queue 的竞争消费机制,系统天然实现了负载均衡——即使某个 Worker 处理较慢,其他 Worker 会自动接管更多订单。

@Service
public class OrderService {

    private final JmsTemplate jmsTemplate;

    public OrderService(JmsTemplate jmsTemplate) {
        this.jmsTemplate = jmsTemplate;
    }

    @Transactional
    public Order createOrder(CreateOrderRequest request) {
        Order order = orderRepository.save(buildOrder(request));

        jmsTemplate.convertAndSend("fulfillment.queue", new FulfillmentCommand(
            order.getId(),
            order.getItems(),
            order.getShippingAddress()),
            message -> {
                message.setStringProperty("region",
                    order.getShippingAddress().getRegion());
                return message;
            });

        return order;
    }
}

22.3 @JmsListener 消息消费

与 JmsTemplate 的"拉取"模式不同,@JmsListener 采用"推送"模式——它注册一个监听器容器,当消息到达时由容器回调相应的方法。这种声明式风格让消费者代码极其简洁,同时容器在幕后管理着连接、Session、线程池与错误处理。

22.3.1 基础用法

@Component
public class FulfillmentListener {

    @JmsListener(destination = "fulfillment.queue")
    public void handleFulfillment(FulfillmentCommand command) {
        log.info("Processing fulfillment for order {}", command.getOrderId());
        warehouseService.allocateAndShip(command);
    }

    @JmsListener(destination = "payment.queue")
    public void handlePayment(Message message) throws JMSException {
        if (message instanceof TextMessage textMessage) {
            String json = textMessage.getText();
            String traceId = message.getStringProperty("traceId");
            log.info("[{}] Payment received {}", traceId, json);
        }
    }

    @JmsListener(destination = "notification.queue")
    public void handleNotification(
            @Payload NotificationEvent event,
            @Header("eventType") String eventType,
            @Header(name = "priority", defaultValue = "NORMAL") String priority) {
        log.info("Event [{}] priority={} {}", eventType, priority, event);
    }

    @JmsListener(destination = "price.request")
    @SendTo("price.response")
    public PriceQuote handlePriceRequest(PriceRequest request) {
        BigDecimal price = pricingService.calculate(request);
        return new PriceQuote(request.getProductId(), price);
    }
}

监听器方法没有显式 receive(),但容器会在线程池里调用它,并把反序列化后的对象、header 与原始消息交给方法参数:

INFO FulfillmentListener : Processing fulfillment for order O1001
INFO FulfillmentListener : [trace-7f8a] Payment received {"paymentId":"PAY-O1001"}
INFO FulfillmentListener : Event [order-created] priority=HIGH NotificationEvent orderId=O1001
INFO PriceListener : replied destination=price.response productId=SKU-REDIS-0011 price=129.00

22.3.2 DefaultJmsListenerContainerFactory 配置

@JmsListener 的背后是 DefaultMessageListenerContainer,它由 DefaultJmsListenerContainerFactory 创建。工厂的配置决定了并发度、事务模式、确认方式等关键行为。

@Configuration
@EnableJms
public class JmsListenerConfig {

    @Bean
    public DefaultJmsListenerContainerFactory jmsListenerContainerFactory(
            ConnectionFactory connectionFactory,
            MessageConverter messageConverter) {

        DefaultJmsListenerContainerFactory factory =
            new DefaultJmsListenerContainerFactory();
        factory.setConnectionFactory(connectionFactory);
        factory.setMessageConverter(messageConverter);
        factory.setConcurrency("2-10");
        factory.setSessionTransacted(true);
        factory.setSessionAcknowledgeMode(Session.CLIENT_ACKNOWLEDGE);
        factory.setErrorHandler(t -> log.error("JMS listener error", t));
        factory.setReceiveTimeout(1000L);

        return factory;
    }

    @Bean
    public DefaultJmsListenerContainerFactory topicListenerFactory(
            ConnectionFactory connectionFactory) {

        DefaultJmsListenerContainerFactory factory =
            new DefaultJmsListenerContainerFactory();
        factory.setConnectionFactory(connectionFactory);
        factory.setPubSubDomain(true);
        factory.setSubscriptionDurable(true);
        factory.setClientId("notification-service");
        return factory;
    }
}

启动输出可以直接验证并发度、事务 Session 与持久订阅是否真正生效:

INFO DefaultMessageListenerContainer : Started listener container for destination 'fulfillment.queue' concurrency=2-10 sessionTransacted=true

22.3.3 三种确认模式

模式 确认时机 消息丢失风险 适用场景
AUTO_ACKNOWLEDGE 监听方法正常返回后由容器自动确认 中(吞掉异常会误确认) 简单业务、可接受重复的消息

22.3.4 错误处理与死信队列

@Configuration
public class JmsErrorConfig {

    @Bean
    public DefaultJmsListenerContainerFactory jmsListenerContainerFactory(
            ConnectionFactory connectionFactory) {
        DefaultJmsListenerContainerFactory factory =
            new DefaultJmsListenerContainerFactory();
        factory.setConnectionFactory(connectionFactory);
        factory.setSessionTransacted(true);
        factory.setErrorHandler(new JmsErrorHandler());
        return factory;
    }

    static class JmsErrorHandler implements ErrorHandler {
        @Override
        public void handleError(Throwable t) {
            log.error("JMS processing failed {}", t.getMessage());
            metricsRegistry.counter("jms.errors").increment();
        }
    }
}

@JmsListener(destination = "order.queue")
public void handleOrder(FulfillmentCommand command) {
    try {
        fulfillmentService.process(command);
    } catch (TemporaryException e) {
        throw new JmsException("Temporary failure will retry", e);
    } catch (PermanentException e) {
        log.error("Permanent failure discarding message {}", command, e);
        deadLetterService.record(command, e);
    }
}

22.3.6 运维控制:运行时暂停与恢复

在生产环境中,有时需要临时暂停消息消费(如下游系统维护)。Spring 的 JmsListenerEndpointRegistry 提供了运行时控制能力:

@RestController
@RequestMapping("/admin/jms")
public class JmsAdminController {

    @Autowired
    private JmsListenerEndpointRegistry registry;

    @PostMapping("/pause/{listenerId}")
    public String pause(@PathVariable String listenerId) {
        MessageListenerContainer container =
            registry.getListenerContainer(listenerId);
        if (container != null) {
            container.stop();
            return "Listener " + listenerId + " paused";
        }
        return "Listener not found";
    }

    @PostMapping("/resume/{listenerId}")
    public String resume(@PathVariable String listenerId) {
        MessageListenerContainer container =
            registry.getListenerContainer(listenerId);
        if (container != null) {
            container.start();
            return "Listener " + listenerId + " resumed";
        }
        return "Listener not found";
    }
}

22.4 JMS 事务与可靠性

消息中间件的核心价值之一是可靠投递——确保消息不丢失、不重复、不乱序。JMS 通过三个层次的事务机制来保障这一点,从轻量级的 Session 本地事务到重量级的 JTA 与 XA 分布式事务,每一层都在"可靠性"与"性能"之间做出不同的取舍。

22.4.1 本地 JMS 事务

本地 JMS 事务的边界是单个 Session。在事务 Session 中,send() 的消息不会立即对其他消费者可见——只有调用 session.commit() 后,Broker 才会正式投递这些消息。如调用 session.rollback(),所有在该事务中发送的消息都会被丢弃,所有接收的消息都会被重新投递。

@Bean
public DefaultJmsListenerContainerFactory jmsListenerContainerFactory(
        ConnectionFactory connectionFactory) {
    DefaultJmsListenerContainerFactory factory =
        new DefaultJmsListenerContainerFactory();
    factory.setConnectionFactory(connectionFactory);
    factory.setSessionTransacted(true);
    return factory;
}

@Bean
public JmsTemplate jmsTemplate(ConnectionFactory connectionFactory) {
    JmsTemplate template = new JmsTemplate(connectionFactory);
    template.setSessionTransacted(true);
    return template;
}

本地事务最重要的可观测信号是 commit 与 rollback:成功时消息被确认,失败时 Broker 会重新投递:

DEBUG JmsTransactionManager : Creating transacted JMS Session
INFO  PaymentReceiptListener : processed payment=PAY-O1001 messageId=ID:pay-O1001

WARN  PaymentReceiptListener : failed payment=PAY-O1002 messageId=ID:pay-O1002 rolling back session

22.4.2 JTA 分布式事务:JMS 加 数据库

当一个业务操作既需要写数据库又需要发送消息时,如何保证两者要么都成功,要么都失败。这就是分布式事务(Distributed Transaction)的经典场景。JTA(Java Transaction API)通过 Two-Phase Commit(2PC)协议协调 JMS Broker 与数据库的事务。

图 22-8 · JTA + XA 两阶段提交时序(JMS + DB 资源协调)

Spring Boot 3.x 项目通常选择 Narayana 这类 Jakarta 兼容的事务管理器:

<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-jta-narayana</artifactId>
</dependency>

spring:
  jta:
    transaction-manager-id: order-service

@Service
public class OrderService {

    @Transactional
    public void placeOrder(Order order) {
        orderRepository.save(order);
        jmsTemplate.convertAndSend("fulfillment.queue",
            new FulfillmentCommand(order));
    }
}

XA 成功提交时应看到两个资源都经历 prepare 与 commit;如任一资源 prepare 失败,Transaction Manager 会协调全部回滚:

INFO NarayanaTransactionManager : begin global transaction xid=order-ORD-001
INFO NarayanaTransactionManager : XAResource db-xa prepare XA_OK
INFO NarayanaTransactionManager : XAResource jms-xa prepare XA_OK
INFO NarayanaTransactionManager : XAResource db-xa commit COMMITTED
INFO NarayanaTransactionManager : XAResource jms-xa commit COMMITTED
INFO OrderService               : placed order=ORD-001 db=committed jms=committed

XA 事务的代价:2PC 协议引入了额外的网络往返(每个参与者两次通信),且在 Prepare 阶段所有参与者都持有锁。这使得 XA 事务比本地事务慢 5 至 10 倍。更糟糕的是,如 Transaction Manager 在 Phase 1 与 Phase 2 之间崩溃,事务处于 in-doubt 状态,需要人工干预。生产环境中越来越多的团队选择事务性 Outbox 模式替代 XA。

22.4.3 Exactly-Once 交付的工程挑战

消息系统的三种投递语义:At-Most-Once(最多一次,可能丢失)、At-Least-Once(至少一次,可能重复)、Exactly-Once(恰好一次)。JMS 的 SESSION_TRANSACTED 模式保证的是 At-Least-Once——如消费者在 commit 之前崩溃,消息会被重投递。真正的 Exactly-Once 需要消费者端实现幂等性。

图 22-9 · 幂等消费者模式(用 messageId 去重表实现 Exactly-Once)
@Component
public class IdempotentOrderListener {

    @Autowired private ProcessedMessageRepository processedRepo;
    @Autowired private OrderService orderService;

    @JmsListener(destination = "order.queue")
    @Transactional
    public void handleOrder(Order order, @Header("JMSMessageID") String messageId) {
        if (processedRepo.existsById(messageId)) {
            log.warn("Duplicate message ignored {}", messageId);
            return;
        }

        orderService.processOrder(order);
        processedRepo.save(new ProcessedMessage(messageId, Instant.now()));
    }
}

@Entity
@Table(name = "processed_messages")
public class ProcessedMessage {
    @Id
    private String messageId;
    private Instant processedAt;
}

幂等消费者的输出要同时证明"首次处理成功"与"重复消息被跳过"。重复消息不是错误,它是 At-Least-Once 语义下的正常情况:

INFO IdempotentOrderListener : Processing messageId=ID:msg-42 orderId=ORD-042
INFO IdempotentOrderListener : Saved processed_messages messageId=ID:msg-42
WARN IdempotentOrderListener : Duplicate message ignored ID:msg-42

22.4.4 银行转账案例

在银行系统中,一笔转账操作需要同时完成三件事:扣减源账户余额(数据库操作);增加目标账户余额(可能在另一个数据库);发送交易通知消息(JMS 操作)。这三件事必须作为一个原子操作——任何一步失败都必须全部回滚。这就是 XA 事务在金融领域不可替代的原因。虽然 XA 有性能代价,但在"数据一致性"高于"吞吐量"的场景中,它仍是最直接的解决方案。然而现代架构更倾向于使用 Saga 模式或事务性 Outbox 模式来避免 XA 的复杂性。

22.5.2 嵌入式 Broker 测试利器

Spring Boot 支持嵌入式 Artemis Broker,无需安装外部消息中间件即可运行集成测试。只需将 spring.artemis.mode 设为 embedded(或仅添加 artemis-server 依赖且不配置 broker-url),Boot 会自动启动一个内存中的 Broker。

spring:
  artemis:
    mode: embedded
    embedded:
      enabled: true
      queues: order.queue,fulfillment.queue
      topics: order.events

@SpringBootTest
@ActiveProfiles("test")
class OrderMessagingTest {

    @Autowired JmsTemplate jmsTemplate;
    @Autowired JmsListenerEndpointRegistry registry;

    @Test
    void shouldSendAndReceiveOrder() {
        Order order = new Order("ORD-001", BigDecimal.valueOf(128.00));
        jmsTemplate.convertAndSend("order.queue", order);

        await().atMost(5, SECONDS).untilAsserted(() -> {
            Order processed = orderRepository.findById("ORD-001").orElse(null);
            assertThat(processed).isNotNull();
            assertThat(processed.getStatus()).isEqualTo(OrderStatus.PROCESSING);
        });
    }
}

集成测试输出应同时看到嵌入式 Broker 启动、消息发送、监听器消费与 Awaitility 断言通过:

INFO EmbeddedActiveMQ    : AMQ221007 Server is now live
INFO OrderMessagingTest  : sent test order ORD-001 to order.queue
INFO OrderListener       : received order ORD-001 status PROCESSING
INFO OrderMessagingTest  : assertion passed order ORD-001 status=PROCESSING

22.5.3 连接池 PooledConnectionFactory

JMS Connection 是重量级的 TCP 连接资源。在没有连接池的情况下,JmsTemplate 每次发送都会创建新的 Connection 转 Session 转 Producer 转 发送 转 关闭。这在高并发场景下会导致严重的性能瓶颈。Spring Boot 通过 spring.artemis.pool.enabled=true 启用连接池。

spring:
  artemis:
    pool:
      enabled: true
      max-connections: 10
      idle-timeout: 30s
      max-sessions-per-connection: 500
      block-if-full: true
      block-if-full-timeout: 5s

22.5.4 自动配置决策树

图 22-10 · Spring Boot JMS 自动配置决策树(classpath + 属性 + 用户 Bean 三维判断)

Spring Boot 的 JMS 自动配置类(JmsAutoConfiguration、ArtemisAutoConfiguration)使用了多层 @ConditionalOn* 注解:@ConditionalOnClass(ConnectionFactory.class) 确保 JMS API 在 classpath 上;@ConditionalOnMissingBean(ConnectionFactory.class) 允许用户自定义 Bean 覆盖自动配置;@ConditionalOnProperty 根据属性值决定嵌入式与外部模式。

22.6 陷阱与限制

JMS 在 Spring Boot 中使用虽然简单,但生产环境中的陷阱往往隐藏在配置细节与运行时行为中。

22.6.1 陷阱一:ObjectMessage 反序列化漏洞

ObjectMessage 使用 Java 原生序列化(java.io.Serializable),这是 Java 生态中最危险的安全漏洞之一。攻击者可以构造包含恶意对象的序列化数据(如 Apache Commons Collections 的 Gadget Chain),在反序列化时触发任意代码执行。

错误用法:

jmsTemplate.send("queue", session -> {
    ObjectMessage msg = session.createObjectMessage();
    msg.setObject(order);
    return msg;
});

@JmsListener(destination = "queue")
public void handle(ObjectMessage msg) throws JMSException {
    Order order = (Order) msg.getObject();
}

正确做法是使用 TextMessage 加 JSON。

22.6.2 陷阱二:JMS 事务与数据库事务边界不一致

这是最常见也最危险的陷阱:开发者以为 @Transactional 同时覆盖了数据库与 JMS 操作,但实际上 Spring 的 DataSourceTransactionManager 只管理数据库事务,不管理 JMS 事务。

错误用法:

@Transactional
public void placeOrder(Order order) {
    orderRepository.save(order);
    jmsTemplate.convertAndSend("fulfillment.queue", order);
    validateInventory(order);
}

如 validateInventory 抛异常,DB 回滚但消息已发送(无法撤回)。结果数据库中没有订单,但履约队列中有一条孤儿消息。

事务性 Outbox 模式:

@Transactional
public void placeOrder(Order order) {
    orderRepository.save(order);
    validateInventory(order);
    outboxRepository.save(new Outbox(
        "fulfillment.queue", toJson(order), OutboxStatus.PENDING));
}

@Scheduled(fixedDelay = 1000)
public void publishOutbox() {
    List<Outbox> pending = outboxRepo.findByStatus(PENDING);
    for (Outbox msg : pending) {
        jmsTemplate.convertAndSend(msg.getDestination(), msg.getPayload());
        msg.setStatus(SENT);
        outboxRepo.save(msg);
    }
}

22.6.3 陷阱三:连接池未启用导致连接泄漏

Spring Boot 默认不启用 JMS 连接池(spring.artemis.pool.enabled 默认 false)。在没有连接池的情况下,JmsTemplate 每次发送都会创建新的 TCP 连接。如发送频率很高,会迅速耗尽 Broker 的连接限制(Artemis 默认最大连接数约 1000)。

ActiveMQConnectionTimedOutException AMQ219013 Timed out waiting to receive cluster topology

解决方案:

spring:
  artemis:
    pool:
      enabled: true
      max-connections: 10
      idle-timeout: 30s

CachingConnectionFactory 在低并发场景下工作良好,但在高并发场景下,单个 Connection 成为瓶颈。生产环境应使用 spring.artemis.pool.enabled=true(基于 Apache PooledJMS 的 JmsPoolConnectionFactory)。

22.6.4 陷阱四:毒消息无限重投递循环

当 @JmsListener 处理某条消息时始终抛出异常(如消息格式损坏、业务规则不可满足),Session 回滚后 Broker 会重新投递这条消息。如没有配置最大重投递次数,这条毒消息(Poison Message)会无限循环。

解决方案 1:Broker 端配置 DLQ(推荐):

<address-setting match="#">
    <dead-letter-address>DLQ</dead-letter-address>
    <max-delivery-attempts>5</max-delivery-attempts>
    <redelivery-delay>5000</redelivery-delay>
</address-setting>

解决方案 2:应用层区分临时错误与永久错误:

@JmsListener(destination = "order.queue")
public void handleOrder(Order order) {
    try {
        processOrder(order);
    } catch (TemporaryException e) {
        throw e;
    } catch (PermanentException e) {
        log.error("Poison message sending to DLQ manually {}", order, e);
        jmsTemplate.convertAndSend("order.dlq", order);
    }
}

22.6.5 陷阱五:Topic 持久订阅 clientId 冲突

JMS 持久订阅(Durable Subscription)要求每个订阅者有唯一的 clientId。问题在于:如服务部署了多个实例(水平扩展),它们共享相同的配置,会导致 clientId 冲突——只有第一个连接的实例能成功,后续实例启动失败。

spring:
  jms:
    listener:
      subscription-durable: true
      client-id: notification-service

部署 3 个实例时,Instance 1 成功,Instance 2 与 3 启动失败抛 InvalidClientIDException。

JMS 2.0 共享持久订阅:

@Bean
public DefaultJmsListenerContainerFactory topicFactory(ConnectionFactory cf) {
    DefaultJmsListenerContainerFactory f = new DefaultJmsListenerContainerFactory();
    f.setConnectionFactory(cf);
    f.setPubSubDomain(true);
    f.setSubscriptionDurable(true);
    f.setSubscriptionShared(true);
    return f;
}

@JmsListener(
    destination = "order.events",
    containerFactory = "topicFactory",
    subscription = "notification-shared-sub")
public void handle(OrderEvent event) {
}

22.6.6 陷阱六:并发消费者破坏消息顺序

当 concurrency 设置大于 1 时,DefaultMessageListenerContainer 会创建多个消费者线程并行处理消息。这意味着同一个 Queue 中的消息不再保证处理顺序——消息 A 可能比消息 B 先到达队列,但消息 B 被线程 2 先处理完成。

解决方案:Artemis Message Group——同一 orderId 的消息总是路由到同一个消费者:

jmsTemplate.convertAndSend("order.queue", statusChange, message -> {
    message.setStringProperty("JMSXGroupID", statusChange.getOrderId());
    return message;
});

Artemis 的 Message Group 通过 JMSXGroupID 属性将消息分组,同一组的消息始终投递给同一消费者(通过一致性哈希绑定)。这在保持组内有序的同时,不同组之间仍并行处理。这是在"全局有序"(单消费者、低吞吐)与"完全无序"(多消费者、高吞吐)之间的最佳折中方案。

22.7 源码入口与扩展边界

JMS 在 Spring Boot 中看起来只是一行 convertAndSend() 与一个 @JmsListener,但运行时实际跨越自动配置、模板、消息转换、监听器容器、事务同步与 Broker 投递策略。定位问题时,要先判断它发生在发送端、容器端、事务端还是 Broker 端。

22.8 本章小结

JMS 规范核心:ConnectionFactory 转 Connection 转 Session 转 Producer 与 Consumer 的级联创建模型;Queue(P2P 竞争消费)与 Topic(Pub Sub 广播)两种消息模型,覆盖了任务分发与事件通知两大场景。

JmsTemplate 发送:send()(原始控制)、convertAndSend()(自动序列化加 MessagePostProcessor)、sendAndReceive()(同步请求与应答);配合 MappingJackson2MessageConverter 实现安全的 JSON 序列化。

@JmsListener 消费:声明式监听,通过 DefaultJmsListenerContainerFactory 配置并发度、事务模式与确认方式;JmsListenerEndpointRegistry 提供运行时暂停与恢复能力。

确认模式:AUTO_ACKNOWLEDGE(正常返回即确认)转 CLIENT_ACKNOWLEDGE(手动控制)转 SESSION_TRANSACTED(commit 或 rollback,生产推荐);配合 Artemis DLQ 处理不可恢复的失败消息。

事务可靠性:Session 本地事务 转 JmsTransactionManager 转 JTA 与 XA 分布式事务;Exactly-Once 需要消费者端幂等(messageId 去重表);事务性 Outbox 模式是 XA 的轻量替代。

Boot 自动配置:spring-boot-starter-artemis 提供 ConnectionFactory、JmsTemplate、ContainerFactory 的开箱即用配置;嵌入式 Broker 用于测试;spring.artemis.pool.enabled=true 启用连接池。

六大陷阱:ObjectMessage 反序列化攻击转 用 JSON TextMessage;JMS 与 DB 事务边界不一致转 Outbox 模式;连接池未启用转 启用 pool;毒消息循环转 DLQ 加 max-delivery-attempts;Topic clientId 冲突转 JMS 2.0 共享订阅;并发消费者破坏顺序转 Message Group。
