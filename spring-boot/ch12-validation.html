第 12 章 · Bean Validation

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

Jakarta Bean Validation 3.x(原 JSR-380)是 Java 平台的标准化数据校验规范,定义了基于注解的约束声明模型与 Validator 执行模型。Hibernate Validator 是该规范的参考实现,Spring Boot 通过 ValidationAutoConfiguration 与 MethodValidationAutoConfiguration 完成集成。

第 11 章介绍了数据绑定。绑定完成后,数据在结构上已落入目标对象,但语义合法性仍需独立校验。本章接续讨论 Bean Validation 的执行模型:@NotNull、@Size、@Pattern 等约束注解的运行时机制;@Valid 与 @Validated 的语义差异;Validation Groups 的分组激活规则;ConstraintValidator 的自定义实现;方法级校验的 AOP 代理模型;Spring MVC 与 WebFlux 的异常分支(MethodArgumentNotValidException / HandlerMethodValidationException / ConstraintViolationException)。

图 12-2-2:典型约束组合示例

public class CreateOrderRequest {

    @NotBlank(message = "用户名不能为空")
    @Size(min = 2, max = 50, message = "用户名长度 {min}-{max}")
    private String username;

    @NotBlank
    @Email(message = "邮箱格式不正确")
    private String email;

    @NotNull
    @Min(value = 18, message = "年龄不能小于 {value}")
    @Max(value = 150, message = "年龄不能大于 {value}")
    private Integer age;

    @NotBlank
    @Size(min = 8, max = 128)
    @Pattern(regexp = "^(?=.*[A-Z])(?=.*[a-z])(?=.*\\d).*$",
             message = "密码必须包含大小写字母和数字")
    private String password;

    @Past
    private LocalDate birthday;

    @Positive
    @Digits(integer = 10, fraction = 2)
    private BigDecimal balance;

    @Size(min = 1, max = 10)
    private List<@NotBlank String> tags;
}

message 属性支持 {value} 与 {min} 等模板占位符,在校验失败时由 MessageInterpolator 插值。List<@NotBlank String> 利用 Java 8 的 ElementType.TYPE_USE 注解,使 @NotBlank 作用于集合元素而非集合本身。

图 12-2-3:约束组合(Constraint Composition)

@NotNull
@Size(min = 2, max = 50)
@Pattern(regexp = "^[a-zA-Z0-9_]+$")
@Target({ ElementType.FIELD, ElementType.PARAMETER })
@Retention(RetentionPolicy.RUNTIME)
@Documented
@Constraint(validatedBy = {})
@ReportAsSingleViolation
public @interface ValidUsername {
    String message() default "用户名必须是 2-50 位字母数字下划线";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

@Constraint(validatedBy = {}) 表示无独立 Validator,所有校验由组合内的注解执行。@ReportAsSingleViolation 将多个内部违规合并为一条对外的违规消息,提升错误可读性。

@RestControllerAdvice
public class ValidationExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ProblemDetail handleValidation(MethodArgumentNotValidException ex) {
        ProblemDetail problem = ProblemDetail.forStatus(HttpStatus.BAD_REQUEST);
        problem.setTitle("Validation Failed");
        problem.setType(URI.create("https://api.example.com/errors/validation"));

        List<Map<String, Object>> fieldErrors = ex.getFieldErrors().stream()
                .map(fe -> Map.<String, Object>of(
                        "field", fe.getField(),
                        "rejected", String.valueOf(fe.getRejectedValue()),
                        "message", Objects.toString(fe.getDefaultMessage(), "invalid")))
                .toList();
        problem.setProperty("errors", fieldErrors);
        return problem;
    }

    @ExceptionHandler(ConstraintViolationException.class)
    public ProblemDetail handleConstraintViolation(ConstraintViolationException ex) {
        ProblemDetail problem = ProblemDetail.forStatus(HttpStatus.BAD_REQUEST);
        problem.setTitle("Constraint Violation");
        List<Map<String, String>> violations = ex.getConstraintViolations().stream()
                .map(cv -> Map.of(
                        "path", cv.getPropertyPath().toString(),
                        "message", cv.getMessage()))
                .toList();
        problem.setProperty("violations", violations);
        return problem;
    }
}

ProblemDetail 是 Spring Framework 6 新增的 RFC 7807 标准类型。响应 Content-Type: application/problem+json 由 ProblemDetail 自动设置。setProperty() 添加非标准扩展字段,type / title / status 是 RFC 7807 强制字段。

12.4 分组校验

图 12-4-1 · 分组激活规则(Default 分组在指定 group 时被跳过 — 高频陷阱)

Default 是 jakarta.validation.groups.Default 接口的引用。约束注解未指定 groups 时,默认归为 Default 分组。@Validated(OnCreate.class) 仅激活 OnCreate 分组,Default 分组的约束被跳过,该行为是分组校验最常见的陷阱。

表 12-4-1:分组使用模式

模式使用方式适用场景
单分组@Validated(OnCreate.class)创建场景
多分组并集@Validated({OnCreate.class, Default.class})创建场景同时校验默认约束
分组继承 Defaultinterface OnCreate extends Default自动包含 Default 约束
@GroupSequence 有序@GroupSequence({A.class, B.class})阶段性校验
GroupSequenceProvider程序化决定分组运行时动态分组

图 12-4-2:分组示例

public interface OnCreate {}
public interface OnUpdate {}

public class OrderRequest {

    @Null(groups = OnCreate.class, message = "创建时不能指定 ID")
    @NotNull(groups = OnUpdate.class, message = "更新时必须指定 ID")
    private Long id;

    @NotBlank(groups = OnCreate.class, message = "创建时用户名不能为空")
    @Size(min = 2, max = 50)
    private String username;

    @NotBlank(groups = OnCreate.class)
    @Email
    private String email;

    @NotBlank(groups = OnCreate.class)
    @Size(min = 8, max = 128, groups = { OnCreate.class, OnUpdate.class })
    private String password;

    @NotNull
    private Boolean active;
}

@PostMapping("/api/orders")
public ResponseEntity<Order> createOrder(
        @Validated(OnCreate.class) @RequestBody OrderRequest request) {
    return ResponseEntity.status(HttpStatus.CREATED).body(orderService.create(request));
}

@PutMapping("/api/orders/{id}")
public ResponseEntity<Order> updateOrder(
        @PathVariable Long id,
        @Validated(OnUpdate.class) @RequestBody OrderRequest request) {
    return ResponseEntity.ok(orderService.update(id, request));
}
图 12-4-3 · @GroupSequence 有序校验(快速失败 — 前阶段全过才执行后阶段)
public interface BasicCheck {}
public interface AdvancedCheck {}
public interface DatabaseCheck {}

@GroupSequence({ BasicCheck.class, AdvancedCheck.class, DatabaseCheck.class })
public interface OrderedChecks {}

public class PaymentRequest {

    @NotBlank(groups = BasicCheck.class)
    @CreditCardNumber(groups = AdvancedCheck.class)
    @CardNotExpired(groups = DatabaseCheck.class)
    private String cardNumber;

    @NotNull(groups = BasicCheck.class)
    @Positive(groups = BasicCheck.class)
    @DecimalMax(value = "999999.99", groups = AdvancedCheck.class)
    private BigDecimal amount;
}

@GroupSequence 实现"快速失败"语义:前一阶段全部通过才执行下一阶段。该模式适合"先廉价格式校验、再昂贵数据库查询"的优化场景,避免在格式错误时浪费数据库资源。

12.5 自定义 ConstraintValidator

图 12-5-1 · 自定义约束三件套(注解 + Validator + 使用)

图 12-5-2:自定义注解与实现

@Documented
@Constraint(validatedBy = PhoneNumberValidator.class)
@Target({ ElementType.FIELD, ElementType.PARAMETER })
@Retention(RetentionPolicy.RUNTIME)
public @interface PhoneNumber {
    String message() default "手机号格式不正确";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
    String region() default "CN";
}

public class PhoneNumberValidator implements ConstraintValidator<PhoneNumber, String> {

    private static final Map<String, Pattern> PATTERNS = Map.of(
            "CN", Pattern.compile("^1[3-9]\\d{9}$"),
            "US", Pattern.compile("^\\+1[2-9]\\d{9}$"),
            "JP", Pattern.compile("^0[789]0-?\\d{4}-?\\d{4}$"));

    private String region;

    @Override
    public void initialize(PhoneNumber annotation) {
        this.region = annotation.region();
    }

    @Override
    public boolean isValid(String value, ConstraintValidatorContext context) {
        if (value == null) return true;
        Pattern pattern = PATTERNS.get(region);
        return pattern != null && pattern.matcher(value).matches();
    }
}

isValid() 对 null 返回 true 是 Bean Validation 的约定:null 检查由 @NotNull 负责,自定义校验器只关注非空场景的格式校验。该约定使得 @NotNull + @PhoneNumber 的组合可独立组合,符合关注点分离原则。

表 12-5-1:ConstraintValidator 生命周期

图 12-5-3:跨字段约束(类级别)

@Documented
@Constraint(validatedBy = PasswordMatchValidator.class)
@Target(ElementType.TYPE)
@Retention(RetentionPolicy.RUNTIME)
public @interface PasswordMatch {
    String message() default "密码与确认密码不一致";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

public class PasswordMatchValidator
        implements ConstraintValidator<PasswordMatch, RegistrationForm> {

    @Override
    public boolean isValid(RegistrationForm form, ConstraintValidatorContext context) {
        if (form == null || form.getPassword() == null
                || form.getConfirmPassword() == null) {
            return true;
        }
        boolean matches = form.getPassword().equals(form.getConfirmPassword());
        if (!matches) {
            context.disableDefaultConstraintViolation();
            context.buildConstraintViolationWithTemplate("密码与确认密码不一致")
                    .addPropertyNode("confirmPassword")
                    .addConstraintViolation();
        }
        return matches;
    }
}

@PasswordMatch
public class RegistrationForm {
    @NotBlank @Size(min = 8) private String password;
    @NotBlank private String confirmPassword;
}

类级别约束的 Validator 接收整个对象作为参数,可访问任意字段。addPropertyNode("confirmPassword") 将错误绑定到具体字段,使得前端响应中错误显示在正确的输入框旁,而非笼统的对象级。

图 12-5-4:依赖注入校验器

@Documented
@Constraint(validatedBy = UniqueEmailValidator.class)
@Target(ElementType.FIELD)
@Retention(RetentionPolicy.RUNTIME)
public @interface UniqueEmail {
    String message() default "该邮箱已被注册";
    Class<?>[] groups() default {};
    Class<? extends Payload>[] payload() default {};
}

public class UniqueEmailValidator implements ConstraintValidator<UniqueEmail, String> {

    private final OrderRepository orderRepository;

    public UniqueEmailValidator(OrderRepository orderRepository) {
        this.orderRepository = orderRepository;
    }

    @Override
    public boolean isValid(String email, ConstraintValidatorContext context) {
        if (email == null) return true;
        return !orderRepository.existsByEmail(email);
    }
}

SpringConstraintValidatorFactory 通过 Spring BeanFactory 创建 Validator 实例,支持构造器注入。该机制使得 Validator 可访问 Repository、Service 等依赖,但 Validator 自身不应持有可变状态,以保证并发安全。

图 12-6-1 · 方法级校验 AOP 时序(参数校验 + 返回值校验,失败抛 ConstraintViolationException)

表 12-6-1:方法级校验注解位置

图 12-6-2:Service 方法级校验示例

@Service
@Validated
public class OrderService {

    public Order createOrder(
            @NotNull(message = "客户 ID 不能为空") Long customerId,
            @Valid @NotNull OrderRequest request) {
        return orderRepository.save(toEntity(customerId, request));
    }

    @NotNull
    public Order findById(@Positive Long orderId) {
        return orderRepository.findById(orderId)
                .orElseThrow(() -> new OrderNotFoundException(orderId));
    }

    @Valid
    public List<@NotNull Order> findByCustomer(
            @NotNull Long customerId,
            @Min(1) int page,
            @Min(1) @Max(100) int size) {
        return orderRepository.findByCustomerId(customerId,
                PageRequest.of(page - 1, size));
    }
}

@Validated 标注在类上是激活方法级校验的必要条件。仅在方法或参数上标注无效。方法级校验通过 MethodValidationInterceptor AOP 代理实现,因此存在 Spring AOP 的固有限制:final 方法、private 方法、self-invocation 不被代理。

表 12-6-2:方法级校验异常类型

入口异常类型HTTP 映射
Service 层 @Validated 类ConstraintViolationException400
Spring 6.1+ MVC @RequestParamHandlerMethodValidationException400
Spring < 6.1 MVC + 类级 @ValidatedConstraintViolationException400
@ConfigurationProperties 启动期BindValidationException启动失败

异常类型差异要求全局异常处理器同时覆盖以上类型,否则 Spring 6.0 升级到 6.1+ 时部分校验失败会从 ConstraintViolationException 切换为 HandlerMethodValidationException,原处理器返回 500 而非 400。

反例与正例对比

反模式(原始类型 + 缺级联):

  public class OrderRequest {
      @NotNull
      @Min(1)
      private int quantity;          // primitives, @NotNull 永远通过

      @NotEmpty
      private List<String> tags;     // 元素无约束

      @NotNull
      private Address address;       // 缺 @Valid,内部不校验
  }

  问题:
    - quantity=0 通过 @NotNull,Min 错误信息误导
    - tags=["", " "] 通过校验
    - address.street=null 通过校验

正模式(包装类 + 类型注解 + @Valid):

  public class OrderRequest {
      @NotNull
      @Min(1)
      private Integer quantity;

      @Size(min = 1)
      private List<@NotBlank String> tags;

      @NotNull
      @Valid
      private Address address;
  }

图 12-8-2:MethodValidationInterceptor.invoke 摘要

@Override
public Object invoke(MethodInvocation invocation) throws Throwable {
    Method method = invocation.getMethod();
    Object[] args = invocation.getArguments();
    Object target = invocation.getThis();
    Class<?>[] groups = determineValidationGroups(invocation);

    ExecutableValidator execVal = this.validator.forExecutables();
    Set<ConstraintViolation<Object>> result = execVal.validateParameters(
            target, method, args, groups);
    if (!result.isEmpty()) {
        throw new ConstraintViolationException(result);
    }

    Object returnValue = invocation.proceed();

    result = execVal.validateReturnValue(target, method, returnValue, groups);
    if (!result.isEmpty()) {
        throw new ConstraintViolationException(result);
    }
    return returnValue;
}

设计取舍

取舍 1:LocalValidatorFactoryBean 双接口桥接

LocalValidatorFactoryBean 同时实现 Spring Validator 与 Jakarta Validator,使得 Spring MVC 字段错误体系与 Jakarta 校验体系共享同一引擎。代价是类层次复杂,接口方法可能存在歧义。该设计避免了"Spring 维护一套校验器、Jakarta 维护另一套"的双重负担,是 Spring 生态长期演进的关键妥协。

取舍 2:方法级校验走 AOP 而非编译期

AOP 代理在运行期拦截方法调用,无需修改源码或字节码。代价是 self-invocation、final 方法、private 方法不被代理。编译期织入(如 AspectJ)可解除该限制,但引入额外的构建步骤与字节码处理复杂性,与 Spring 一贯的"运行时优先"哲学冲突。Spring 选择 AOP 路径,通过文档明确告知 self-invocation 限制。

取舍 3:默认不保证约束执行顺序

Bean Validation 规范不保证同一对象内多个约束的执行顺序,允许实现者自由优化(并行执行、缓存)。代价是开发者无法依赖隐式顺序。需要顺序时使用 @GroupSequence 显式表达,该机制使得"快速失败"成为有意识的设计决策而非偶然行为。

12.9 本章要点

要点源码依据易错点
Jakarta Validation 3.x 由 Hibernate Validator 实现LocalValidatorFactoryBean缺 spring-boot-starter-validation
@Valid 标准注解,不支持分组jakarta.validation.Valid分组场景误用 @Valid
@Validated Spring 扩展,支持分组与方法级 AOPorg.springframework.validation.annotation.Validated类上才能激活方法级校验
@RequestBody + @Valid 失败抛 MethodArgumentNotValidExceptionRequestResponseBodyMethodProcessor与 ConstraintViolation 异常区分
简单参数校验 6.1+ 抛 HandlerMethodValidationExceptionHandlerMethodValidator旧版本兼容需类级 @Validated
嵌套对象与集合元素校验需 @ValidValidatorImpl 级联遍历字段缺 @Valid 内部不校验
原始类型 @NotNull 永远通过Jackson 反序列化默认值DTO 字段统一 Integer / Long
@GroupSequence 实现快速失败jakarta.validation.GroupSequence默认无序
