32.1 TestContext 引擎

Spring Boot 测试基础设施建立在 Spring Framework TestContext 框架上,再叠加 Boot 自动配置层。理解这条链是后续所有切片与缓存优化的前提。

32.1.1 @SpringBootTest 完整语义

@SpringBootTest
class OrderServiceIntegrationTest {

    @Autowired
    private OrderService orderService;

    @Autowired
    private ApplicationContext context;

    @Test
    void contextLoads() {
        assertThat(context).isNotNull();
        assertThat(orderService).isNotNull();
    }
}

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
class FullStackTest {

    @LocalServerPort
    private int port;

    @Autowired
    private TestRestTemplate restTemplate;

    @Test
    void healthEndpoint() {
        ResponseEntity<String> resp = restTemplate
            .getForEntity("http://localhost:" + port + "/actuator/health", String.class);
        assertThat(resp.getStatusCode()).isEqualTo(HttpStatus.OK);
    }
}

webEnvironment 四种模式:MOCK(默认 模拟 Servlet 不启真实端口)、RANDOM_PORT(真实嵌入式 + 随机端口)、DEFINED_PORT(server.port)、NONE(无 Web)。生产测试推荐 RANDOM_PORT,既避端口冲突又测真实 HTTP 栈。

32.1.3 上下文缓存与 @DirtiesContext

ApplicationContext 创建昂贵(扫描 classpath / 实例化 Bean / 初始化数据源,几秒至十几秒)。Spring 测试通过缓存解决:相同配置共享同一个 ApplicationContext。缓存 key 是 MergedContextConfiguration,组成包括 @ContextConfiguration 配置类 / XML、@ActiveProfiles、@TestPropertySource、@MockBean / @SpyBean 声明的 mock(关键)、@DynamicPropertySource 动态属性。

 hitCount = 1, missCount = 1, failureCount = 0]

@DirtiesContext 标注后会销毁并重建 ApplicationContext,是性能杀手。100 个测试类中 10 个用 @DirtiesContext,可能多花 10 倍上下文启动时间。仅在确实修改上下文状态(动态注册 Bean / 修改 DataSource)时使用,优先 classMode = AFTER_CLASS 而非默认 AFTER_EACH_TEST_METHOD。

调试缓存问题时,不要只数测试类数量。两个测试类都 @SpringBootTest,但只要 @ActiveProfiles @TestPropertySource @DynamicPropertySource 或 mock 组合不同,就是不同的 MergedContextConfiguration。Spring Framework 自身 25000+ 测试,CI 中上下文缓存把上下文创建次数从数千次降到约 200 次,CI 时间缩短 70%+。

32.2 测试切片

32.2.1 设计取舍:为什么 Boot 做切片

切片目标不是"更像单元测试",而是保留某一层框架契约:MVC 参数绑定与异常处理、JPA 实体映射与 Repository 查询、Jackson 模块注册、RestClient 序列化与错误处理。纯 Mockito 单测快但看不到这些契约;完整 @SpringBootTest 真实但贵。

表 32-2-1:测试方案选型

方案 被拒绝为默认建议的原因 Boot 给出的边界
全部 @SpringBootTest 慢 缓存 key 易碎 失败定位范围大 只放跨层关键流程与烟雾测试
全部纯 Mockito 快但不验证 MVC / JPA / Jackson 契约 业务算法 规则计算 状态机
手动 new Controller + Repository 少了 Filter Advice Converter 事务 用切片
全部 E2E 维护成本高 反馈慢 难覆盖异常分支 只保核心用户路径

切片缺协作者就用 mock 或 @Import 精确补,不要为一个 Service 缺失就升级回 @SpringBootTest。如果切片需要大量 @Import 才能跑,说明边界已经不再是单层测试,应升级集成测试。

32.2.2 @WebMvcTest 只加载 Web 层

@WebMvcTest(OrderController.class)
class OrderControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private OrderService orderService;

    @Test
    void shouldReturnOrder() throws Exception {
        Order order = new Order(1L, "LAPTOP", BigDecimal.valueOf(999.99));
        given(orderService.findById(1L)).willReturn(Optional.of(order));

        mockMvc.perform(get("/api/orders/1"))
            .andExpect(status().isOk())
            .andExpect(jsonPath("$.productName").value("LAPTOP"))
            .andExpect(jsonPath("$.amount").value(999.99));
    }

    @Test
    void shouldReturn404WhenNotFound() throws Exception {
        given(orderService.findById(99L)).willReturn(Optional.empty());

        mockMvc.perform(get("/api/orders/99"))
            .andExpect(status().isNotFound());
    }
}

输出体现"Web 层通过 Service 由 mock 接管",不会出现数据库连接或 Repository 初始化日志。

32.2.3 @DataJpaTest 只加载 JPA 层

@DataJpaTest
class OrderRepositoryTest {

    @Autowired
    private TestEntityManager em;

    @Autowired
    private OrderRepository orderRepository;

    @Test
    void shouldFindByCustomerId() {
        Order order = new Order("O1001", "C1001", BigDecimal.valueOf(128.00));
        em.persistAndFlush(order);

        Optional<Order> found = orderRepository.findByCustomerId("C1001");
        assertThat(found).isPresent();
        assertThat(found.get().getId()).isEqualTo("O1001");
    }

    @Test
    void shouldEnforceUniqueOrderNo() {
        em.persistAndFlush(new Order("O1001", "C1001", BigDecimal.valueOf(128.00)));

        assertThatThrownBy(() ->
            em.persistAndFlush(new Order("O1001", "C1002", BigDecimal.valueOf(98.00))))
            .isInstanceOf(PersistenceException.class);
    }
}

@DataJpaTest 默认嵌入式数据库 + 每方法 @Transactional 自动回滚。重复约束在 persistAndFlush() 时被数据库拦截。

32.3 MockMvc 与 WebTestClient

MockMvc 模拟 HTTP 完整处理流(DispatcherServlet → HandlerMapping → Controller → ViewResolver),但不启真实 HTTP 服务器。速度快,验证完整请求处理管线。

32.3.1 两种 setup 模式

// 模式 1 standaloneSetup 手动指定 Controller 不加载上下文
MockMvc mockMvc = MockMvcBuilders
    .standaloneSetup(new OrderController(mockOrderService))
    .setControllerAdvice(new GlobalExceptionHandler())
    .addFilters(new CharacterEncodingFilter("UTF-8", true))
    .build();

// 模式 2 webAppContextSetup 完整 WebApplicationContext
@SpringBootTest
@AutoConfigureMockMvc
class OrderControllerIntegrationTest {

    @Autowired
    private MockMvc mockMvc;
}

standaloneSetup 更快(无需上下文),适合纯 Controller 单测;webAppContextSetup 加载完整 Web 配置(拦截器 / Filter / 异常处理器),适合集成测试。多数场景推荐 @WebMvcTest + 自动注入的 MockMvc(本质是 webAppContextSetup 切片版)。

class AdminControllerSecurityTest {

32.4 Mock 与 Stub

测试中常需隔离被测组件与外部依赖(数据库 / 第三方 API / 消息队列)。Spring Boot 长期通过 @MockBean @SpyBean 提供与 Mockito 集成;Boot 3.4+ 优先迁移 @MockitoBean @MockitoSpyBean。核心语义相同:在 ApplicationContext 中替换真实 Bean。

32.4.1 Mock Bean 源码路径:BeanDefinition 替换

@MockBean 关键是"上下文级替换"。框架在 ApplicationContext 刷新前收集 mock 定义,把同类型或同名称的真实 BeanDefinition 替换成 Mockito mock;被测 Bean 构造时拿到的已经是 mock,而非先创建真实 Bean 再反射改字段。

图 32-4-1:Mockito Bean 替换进入缓存 key

flowchart LR Field[@MockBean PaymentGateway] Parser[Mockito annotation parser collect definitions] Customizer[MockitoContextCustomizer] Key[MergedContextConfiguration cache key includes mocks] Registry[BeanDefinitionRegistry replace or add bean] Context[ApplicationContext refresh] Service[OrderService constructor gets mock] TestCode[then paymentGateway should] Field --> Parser --> Customizer --> Key Customizer --> Registry --> Context --> Service --> TestCode

表 32-4-1:Mock Bean 机制

机制 运行时结果 常见误解
按类型替换 容器中已有单个同类型 Bean 时被 mock 替代 不是 Mockito @Mock 字段再手动注入
按名称替换 多 Bean 场景用名称或 qualifier 精确替换 类型相同的多个 Bean 不能靠运气匹配
参与 cache key 不同 mock 组合会生成不同上下文 多 mock 一个字段也会拖慢测试套件
refresh 前生效 真实 Bean 不会先完成初始化 可避免真实外部客户端启动
reset 策略 测试方法间重置 mock 交互 不要把 mock 状态当跨测试共享数据

只测纯业务算法用 Mockito @Mock + 手动构造,不要启动 Spring;要验证 Spring 注入与 AOP 代理后协作用 @MockBean @MockitoBean;要验证真实 HTTP 序列化与超时重试用 WireMock 或 Testcontainers,不要用 mock 替代整个网络层。

32.4.2 @MockBean 完全替换

@SpringBootTest
class OrderServiceTest {

    @MockBean
    private PaymentGateway paymentGateway;

    @MockBean
    private InventoryClient inventoryClient;

    @Autowired
    private OrderService orderService;

    @Test
    void shouldCreateOrder() {
        given(paymentGateway.charge(any(PaymentRequest.class)))
            .willReturn(new PaymentResult("PAY-O1001", true));
        given(inventoryClient.reserve("SKU-REDIS-001", 2))
            .willReturn(true);

        Order order = orderService.createOrder(
            "SKU-REDIS-001", 2, BigDecimal.valueOf(199.98));

        assertThat(order.getStatus()).isEqualTo(OrderStatus.CONFIRMED);
        assertThat(order.getPaymentId()).isEqualTo("PAY-O1001");

        then(paymentGateway).should().charge(any(PaymentRequest.class));
        then(inventoryClient).should().reserve("SKU-REDIS-001", 2);
    }
}

输出不应出现真实支付网关 URL 或库存服务地址,Mockito 只记录交互。

32.4.5 WireMock 集成测试

@MockBean 替换的是自己的 Client Bean,测试不到 HTTP 序列化、重试逻辑、超时处理。WireMock 启本地 HTTP 服务器,模拟真实响应(含延迟 / 错误码 / 网络故障)。

@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@WireMockTest(httpPort = 8089)
class PaymentIntegrationTest {

    @Autowired
    private OrderService orderService;

    @Test
    void shouldHandlePaymentGatewayTimeout() {
        stubFor(post(urlEqualTo("/api/payments/authorize"))
            .willReturn(aResponse()
                .withStatus(200)
                .withFixedDelay(5000)
                .withBody("{\"paymentId\":\"payment:O1001\"}")));

        assertThatThrownBy(() -> orderService.createOrder("SKU-REDIS-001", 1))
            .isInstanceOf(PaymentTimeoutException.class);
    }

    @Test
    void shouldRetryOnServerError() {
        stubFor(post(urlEqualTo("/api/payments/authorize"))
            .inScenario("retry")
            .whenScenarioStateIs(STARTED)
            .willReturn(aResponse().withStatus(503))
            .willSetStateTo("recovered"));

        stubFor(post(urlEqualTo("/api/payments/authorize"))
            .inScenario("retry")
            .whenScenarioStateIs("recovered")
            .willReturn(aResponse()
                .withStatus(200)
                .withBody("{\"paymentId\":\"PAY-O1001\",\"status\":\"AUTHORIZED\",\"amount\":128.00}")));

        Order order = orderService.createOrder("SKU-1", 1);
        assertThat(order.getPaymentId()).isEqualTo("PAY-O1001");
    }
}

32.5.3 单例容器模式

默认每个测试类启停自己的容器。50 个测试类需 PostgreSQL 就启停 50 次,每次 3-5 秒。单例模式让所有测试共享同一容器实例。

public abstract class AbstractIntegrationTest {

    static final PostgreSQLContainer<?> POSTGRES;
    static final GenericContainer<?> REDIS;

    static {
        POSTGRES = new PostgreSQLContainer<>("postgres:16-alpine")
            .withDatabaseName("testdb");
        POSTGRES.start();

        REDIS = new GenericContainer<>("redis:7-alpine")
            .withExposedPorts(6379);
        REDIS.start();
    }

    @DynamicPropertySource
    static void registerProperties(DynamicPropertyRegistry registry) {
        registry.add("spring.datasource.url", POSTGRES::getJdbcUrl);
        registry.add("spring.datasource.username", POSTGRES::getUsername);
        registry.add("spring.datasource.password", POSTGRES::getPassword);
        registry.add("spring.data.redis.host", REDIS::getHost);
        registry.add("spring.data.redis.port", () -> REDIS.getMappedPort(6379));
    }
}

class OrderServiceTest extends AbstractIntegrationTest {

32.5.4 @ServiceConnection 扩展边界

@ServiceConnection 不是简单地把容器端口写进 Environment。它通过 ContainerConnectionDetailsFactory 为容器生成类型化 ConnectionDetails,再被对应自动配置消费。PostgreSQL 容器生成 JdbcConnectionDetails,Redis 容器生成 RedisConnectionDetails。测试配置绕开字符串属性拼接,避免多服务地址写错 key。

PostgreSQLContainer + ServiceConnection
  -> PostgreSQLContainerConnectionDetailsFactory
  -> JdbcConnectionDetails
  -> DataSourceAutoConfiguration
  -> HikariDataSource jdbc:postgresql://localhost:32789/test

自定义容器可接入,但边界是"连接信息",不是"容器生命周期"。容器启动 / 等待策略 / 镜像版本仍由 Testcontainers 管,Spring Boot 只把已启动服务暴露成自动配置能理解的连接详情。

32.7.1 陷阱 1:@MockBean 导致缓存失效

@SpringBootTest
class TestA {
    @MockBean PaymentService paymentService;
}

@SpringBootTest
class TestB {
    @MockBean ShippingService shippingService;
}

@SpringBootTest
class TestC {
    @MockBean PaymentService paymentService;
    @MockBean NotificationService notificationService;
}

@SpringBootTest
abstract class BaseTest {
    @MockBean protected PaymentService paymentService;
    @MockBean protected ShippingService shippingService;
    @MockBean protected NotificationService notificationService;
}

class TestA extends BaseTest { }
class TestB extends BaseTest { }
class TestC extends BaseTest { }

每个独特 @MockBean 组合生成新缓存 key。50 个测试类各异组合 → 50 个 ApplicationContext,每个 3-5 秒,总多 2-4 分钟。修正:统一到基类,所有测试共享一个上下文。

32.7.4 陷阱 4:@Transactional 回滚隐藏 commit-time 错误

@DataJpaTest
class OrderRepositoryTest {

    @Autowired
    private OrderRepository orderRepository;

    @Test
    void shouldRejectDuplicateOrderNo() {
        orderRepository.save(new Order("O1001", "C1001", BigDecimal.valueOf(128.00)));
        orderRepository.save(new Order("O1001", "C1002", BigDecimal.valueOf(98.00)));
    }
}

@Test
void shouldRejectDuplicateOrderNo() {
    em.persistAndFlush(new Order("O1001", "C1001", BigDecimal.valueOf(128.00)));
    assertThatThrownBy(() ->
        em.persistAndFlush(new Order("O1001", "C1002", BigDecimal.valueOf(98.00))))
        .isInstanceOf(PersistenceException.class);
}

@Test
void shouldRejectDuplicateOrderNoWithRepository() {
    orderRepository.saveAndFlush(new Order("O1001", "C1001", BigDecimal.valueOf(128.00)));
    assertThatThrownBy(() ->
        orderRepository.saveAndFlush(new Order("O1001", "C1002", BigDecimal.valueOf(98.00))))
        .isInstanceOf(DataIntegrityViolationException.class);
}

@DataJpaTest 默认每方法 @Transactional 自动回滚。约束在 flush() 或 commit() 才检查的错误,在回滚测试中永不暴露。修正:用 persistAndFlush() 或 saveAndFlush() 强制刷新。

    registry.add("spring.datasource.url", postgres::getJdbcUrl);

Testcontainers 把容器内部端口映射到宿主机随机端口(PG 容器 5432 → 宿主 32789)。硬编码 localhost:5432 在 CI 必失败。修正:@DynamicPropertySource 或 @ServiceConnection。

32.7.6 陷阱 6:MockMvc standaloneSetup 不执行 Servlet Filter

MockMvc mockMvc = MockMvcBuilders
    .standaloneSetup(new AdminController(adminService))
    .build();

mockMvc.perform(get("/admin/orders"))
    .andExpect(status().isOk());

@Autowired

MockMvc mockMvc = MockMvcBuilders

Flush @DataJpaTest 的约束测试必须 persistAndFlush
Filter 安全测试用 @WebMvcTest 或显式 addFilters
并行 共享资源用 SAME_THREAD 串行化
