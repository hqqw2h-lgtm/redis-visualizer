第 21 章 · 消息驱动架构

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

订单服务完成支付后,通常需要触发库存扣减、物流发货、积分累加、邮件推送等下游操作。如全部使用同步 HTTP 调用,任何一个下游故障都可能拖垮整个链路。消息驱动架构通过引入消息中间件(Broker),将生产者与消费者在时间、空间上彻底解耦:生产者只管发送、消费者只管接收,中间件负责可靠投递、流量削峰与持久化存储。

Spring 从最底层的 Message<T> 抽象,到 Spring Kafka 与 Spring AMQP 的中间件专用集成,再到 Spring Cloud Stream 的跨 Broker 统一编程模型,构建了一套完整的消息驱动生态。但异步解耦并非银弹——消息丢失、重复消费、顺序保证、序列化兼容、事务一致性,每一个都是生产环境中的深水问题。

本章按"Spring 消息抽象 → Kafka 集成 → RabbitMQ 集成 → Spring Cloud Stream → 可靠性保障与 Outbox → 陷阱与限制"的顺序,系统讲解 Spring 消息驱动架构。前置知识:第 20 章 异步处理与 TaskExecutor 线程池体系,第 6 章 Spring 事件机制。了解消息中间件(Kafka 与 RabbitMQ)基础概念会更顺手。

21.1 Spring 消息抽象

在深入具体的 Kafka 或 RabbitMQ 集成之前,需要理解 Spring 的消息抽象层。spring-messaging 模块定义了一套与中间件无关的核心接口——这套抽象不仅被 Spring Integration 使用,也是 Spring Cloud Stream、Spring WebSocket STOMP 等模块的基石。

21.1.1 核心三件套:Message、MessageChannel、MessageHandler

Message<T> 是 Spring 消息体系中最基础的抽象。它由两部分组成:payload(消息负载,业务数据)与 headers(消息头,元数据)。这个设计将"内容"与"路由 与 控制信息"分离。

public interface Message<T> {
    T getPayload();
    MessageHeaders getHeaders();
}

public class MessageHeaders implements Map<String, Object> {
    public static final String ID = "id";
    public static final String TIMESTAMP = "timestamp";
    public static final String CONTENT_TYPE = "contentType";
    public static final String REPLY_CHANNEL = "replyChannel";
    public static final String ERROR_CHANNEL = "errorChannel";
}

Message<Order> msg = MessageBuilder
    .withPayload(order)
    .setHeader("orderType", "EXPRESS")
    .setHeader("priority", 1)
    .build();

把 Message 打印出来时,最重要的是确认 payload 与 headers 没有混在一起:业务字段在 payload,路由、追踪、内容类型等控制字段在 headers。

{
  "payload": {
    "orderId": "O1001",
    "amount": 128.50,
    "status": "CREATED"
  },
  "headers": {
    "id": "5b38fbf9-f6f0-48e7-8bc8-1f7e9ed6b5ac",
    "timestamp": 1717056000123,
    "contentType": "application/json",
    "orderType": "EXPRESS",
    "priority": 1
  }
}

MessageChannel 是消息传输的管道抽象。它有两种子类型:SubscribableChannel(推模式——消息到达时主动推送给订阅者)与 PollableChannel(拉模式——消费者主动轮询获取消息)。这对应了消息中间件中两种经典的消费模型。

21.1.3 SubscribableChannel 推模式

SubscribableChannel channel = new DirectChannel();
channel.subscribe(msg -> log.info("received {}", msg.getPayload()));
channel.send(MessageBuilder.withPayload("Hello").build());

一次同步发送的日志非常直接:调用 send() 的线程就是执行 handler 的线程:

INFO [main] c.example.DirectChannelDemo : received Hello
INFO [main] c.example.DirectChannelDemo : sendResult=true

典型实现:DirectChannel 同步单播(默认轮询选择一个 handler);PublishSubscribeChannel 广播给所有 handler,配置 executor 后才异步。

21.1.4 PollableChannel 拉模式

PollableChannel channel = new QueueChannel(100);
channel.send(MessageBuilder.withPayload("Task-1").build());

Message<?> msg = channel.receive(5000);
if (msg != null) {
    process(msg.getPayload());
}

拉模式的输出重点是"先入队、后轮询",消费者可以控制批量、超时与节奏:

INFO [worker-1] c.example.QueueChannelDemo : queued payload=Task-1 capacity=100
INFO [worker-1] c.example.QueueChannelDemo : polled payload=Task-1 elapsedMs=2

典型实现:QueueChannel 内存队列(BlockingQueue);PriorityChannel 优先级队列。

21.1.5 ErrorChannel 异步异常归宿

Spring Messaging 的另一个关键约定:异常不要只停留在堆栈里,也可以被包装成 ErrorMessage 投递到 errorChannel,由统一的错误处理器记录、告警或转发到死信队列。

QueueChannel errorChannel = new QueueChannel();

Message<String> command = MessageBuilder
    .withPayload("pay:O1001")
    .setHeader("traceId", "trc-7f91")
    .setErrorChannel(errorChannel)
    .build();

try {
    paymentHandler.handleMessage(command);
} catch (Exception ex) {
    errorChannel.send(new ErrorMessage(ex, command.getHeaders()));
}

21.1.6 Spring Boot 消息基础设施自动配置

Spring Boot 对消息中间件的自动配置遵循"约定优于配置"原则:检测 classpath 上的依赖,然后创建对应的 Template、ConnectionFactory、ListenerContainerFactory 等核心 Bean。

依赖 自动配置 Bean 主要职责
spring-kafka KafkaProperties、ProducerFactory、KafkaTemplate、ConsumerFactory、ConcurrentKafkaListenerContainerFactory、KafkaAdmin 绑定 spring.kafka.*、生产者与消费者模板、监听容器
spring-amqp RabbitProperties、CachingConnectionFactory、RabbitTemplate、RabbitAdmin、SimpleRabbitListenerContainerFactory 绑定 spring.rabbitmq.*、连接池、模板、Exchange 与 Queue 自动声明

启动日志可以直接验证 Boot 是否真的创建了对应基础设施 Bean,而非只看配置文件:

KafkaAutoConfiguration matched:
  - kafkaTemplate
  - kafkaProducerFactory
  - kafkaConsumerFactory
  - kafkaListenerContainerFactory

RabbitAutoConfiguration matched:
  - rabbitTemplate
  - rabbitConnectionFactory
  - rabbitListenerContainerFactory
  - rabbitAdmin

Message<T>、MessageChannel、MessageHandler 看似简单,但它们统一了 Spring 生态中所有消息传递场景的编程模型:Spring Integration 的企业集成模式、Spring Cloud Stream 的微服务消息总线、WebSocket STOMP 的实时推送——底层都用同一套 Message 抽象。

21.1.7 源码入口与运行时调用路径

Spring Boot 的消息集成不是一个黑盒 starter,而是一组自动配置类把底层客户端、监听容器与注解处理器接到应用上下文里。

领域 源码入口 关键函数或 Bean 运行时职责
通用消息抽象 Message、MessageHeaders、MessageChannel GenericMessage、MessageBuilder 统一 payload、headers、errorChannel、replyChannel 约定

Kafka 注解驱动 KafkaAnnotationDrivenConfiguration kafkaListenerAnnotationProcessor() 注册 @KafkaListener 扫描器
Kafka Listener 扫描 KafkaListenerAnnotationBeanPostProcessor processKafkaListener() 把业务方法解析成 MethodKafkaListenerEndpoint
Kafka 容器执行 ConcurrentMessageListenerContainer pollAndInvoke() 创建 consumer 线程,循环 poll、转换、调用 listener、提交 offset
Kafka 发送 KafkaTemplate doSend() 构造 ProducerRecord,封装 CompletableFuture<SendResult>
Kafka 错误处理 DefaultErrorHandler、DeadLetterPublishingRecoverer handleRemaining()、accept() 退避重试、seek 回滚、发布到 DLT
Rabbit 自动配置 RabbitAutoConfiguration rabbitTemplate() 创建连接工厂、模板、监听容器工厂
Rabbit 容器执行 SimpleMessageListenerContainer receiveAndExecute() 从 queue 拉取消息,执行 listener,按 ack mode 确认
Rabbit 发送 RabbitTemplate convertAndSend()、doSend() 消息转换、设置属性、调用 channel basicPublish

21.2 Spring Kafka 集成

Apache Kafka 是分布式消息系统的事实标准——高吞吐、持久化、水平扩展、消费者组。Spring Kafka 在原生 Kafka 客户端之上提供了 KafkaTemplate(生产者)与 @KafkaListener(消费者)两个核心抽象,以及完整的序列化、错误处理与事务支持。

21.2.1 KafkaTemplate 生产者 API

spring:
  kafka:
    bootstrap-servers: localhost:9092
    producer:
      key-serializer: org.apache.kafka.common.serialization.StringSerializer
      value-serializer: org.springframework.kafka.support.serializer.JsonSerializer
      acks: all
      retries: 3
      properties:
        enable.idempotence: true
        max.in.flight.requests.per.connection: 5

@Service
public class OrderEventProducer {

    private final KafkaTemplate<String, OrderEvent> kafkaTemplate;

    public OrderEventProducer(KafkaTemplate<String, OrderEvent> kafkaTemplate) {
        this.kafkaTemplate = kafkaTemplate;
    }

    public void sendOrderCreated(OrderEvent event) {
        kafkaTemplate.send("order-created", event.getOrderId(), event);
    }

    public void sendWithCallback(OrderEvent event) {
        CompletableFuture<SendResult<String, OrderEvent>> future =
            kafkaTemplate.send("order-created", event.getOrderId(), event);

        future.whenComplete((result, ex) -> {
            if (ex == null) {
                RecordMetadata meta = result.getRecordMetadata();
                log.info("send success topic={} partition={} offset={}",
                    meta.topic(), meta.partition(), meta.offset());
            } else {
                log.error("send failed orderId={}", event.getOrderId(), ex);
            }
        });
    }

    public void sendWithHeaders(OrderEvent event) {
        ProducerRecord<String, OrderEvent> record = new ProducerRecord<>(
            "order-created", null, System.currentTimeMillis(),
            event.getOrderId(), event);
        record.headers().add("source", "order-service".getBytes());
        record.headers().add("eventType", "order-created".getBytes());
        kafkaTemplate.send(record);
    }
}

生产者示例不要只看 send() 调用是否返回,还要看 Broker 给出的 topic、partition、offset:

INFO [kafka-producer-network-thread | producer-1] c.example.OrderEventProducer : send success topic=order-created partition=0 offset=42
INFO [main] c.example.OrderEventProducer : key=O1001 eventType=order-created source=order-service

21.2.2 @KafkaListener 消费者 API

spring:
  kafka:
    consumer:
      group-id: order-processing-group
      auto-offset-reset: earliest
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      properties:
        spring.json.trusted.packages: "com.example.event"
      enable-auto-commit: false
    listener:
      ack-mode: manual_immediate
      concurrency: 3

@Component
public class OrderEventConsumer {

    @KafkaListener(topics = "order-created", groupId = "order-processing-group")
    public void handleOrderEvent(OrderEvent event) {
        log.info("processing order event {}", event.getOrderId());
        orderService.processOrder(event);
    }

    @KafkaListener(topics = "order-created", groupId = "inventory-group")
    public void handleWithAck(
            ConsumerRecord<String, OrderEvent> record,
            Acknowledgment ack) {
        try {
            log.info("consume partition={} offset={} key={}",
                record.partition(), record.offset(), record.key());
            inventoryService.deductStock(record.value());
            ack.acknowledge();
        } catch (Exception e) {
            log.error("inventory deduction failed", e);
            throw new IllegalStateException("inventory deduction failed", e);
        }
    }

    @KafkaListener(
        topics = "order-created",
        groupId = "analytics-batch-group",
        containerFactory = "batchListenerFactory")
    public void handleBatch(List<OrderEvent> events, Acknowledgment ack) {
        log.info("batch processing {} messages", events.size());
        analyticsService.batchInsert(events);
        ack.acknowledge();
    }
}

消费者日志要与生产者输出对上:同一个 key、同一个 partition、同一个 offset,处理成功之后才提交 offset:

INFO [container#0-0-C-1] c.example.OrderEventConsumer : processing order event O1001
INFO [container#1-0-C-1] c.example.OrderEventConsumer : consume partition=0 offset=42 key=O1001
INFO [container#1-0-C-1] o.s.k.l.KafkaMessageListenerContainer : Committing offset order-created-0@43

21.2.3 Consumer Offset 管理策略

ack-mode 提交时机 适用场景
RECORD 每条消息处理后自动确认 单条处理逻辑
BATCH 每批 poll 的消息处理后确认 默认值,大多数场景
MANUAL 业务代码显式调用 ack.acknowledge() 关键业务,处理失败不能丢消息
MANUAL_IMMEDIATE 同 MANUAL,但立即同步提交 严格 At-Least-Once
COUNT 每处理 N 条确认 高吞吐场景
TIME 每隔 N 毫秒确认 低延迟周期性提交

自动提交的风险:enable.auto.commit=true 默认每 5 秒提交一次。如处理期间应用崩溃,offset 已提交但消息未处理完,将导致消息丢失。生产环境推荐 enable-auto-commit: false 加 ack-mode: manual 加 业务幂等。

21.2.6 错误处理与死信队列

消费者处理消息时抛出异常,Spring Kafka 提供了多层错误处理机制:从简单重试到死信队列(DLT,Dead Letter Topic)。

@Configuration
public class KafkaErrorHandlingConfig {

    @Bean
    public ConcurrentKafkaListenerContainerFactory<String, Object>
            kafkaListenerContainerFactory(
                ConsumerFactory<String, Object> consumerFactory,
                KafkaTemplate<String, Object> kafkaTemplate) {

        ConcurrentKafkaListenerContainerFactory<String, Object> factory =
            new ConcurrentKafkaListenerContainerFactory<>();
        factory.setConsumerFactory(consumerFactory);

        DeadLetterPublishingRecoverer recoverer =
            new DeadLetterPublishingRecoverer(kafkaTemplate,
                (record, ex) -> new TopicPartition(
                    record.topic() + ".DLT", record.partition()));

        DefaultErrorHandler errorHandler = new DefaultErrorHandler(
            recoverer, new FixedBackOff(1000L, 3L));

        errorHandler.addNotRetryableExceptions(
            DeserializationException.class,
            ClassCastException.class);

        factory.setCommonErrorHandler(errorHandler);
        return factory;
    }
}

@Component
public class RetryableOrderConsumer {

    @RetryableTopic(
        attempts = "4",
        backoff = @Backoff(delay = 1000, multiplier = 2.0, maxDelay = 10000),
        dltStrategy = DltStrategy.FAIL_ON_ERROR,
        topicSuffixingStrategy = TopicSuffixingStrategy.SUFFIX_WITH_INDEX_VALUE)
    @KafkaListener(topics = "order-created", groupId = "retryable-group")
    public void handleWithRetry(OrderEvent event) {
        orderProcessor.process(event);
    }

    @DltHandler
    public void handleDlt(OrderEvent event) {
        log.error("message finally entered DLT {}", event);
        alertService.notifyDeadLetter("order-created", event);
    }
}

21.2.7 Exactly-Once 语义

Kafka 的 Exactly-Once Semantics(EOS)通过两个机制组合实现:幂等生产者(防止生产端重复)加 Kafka 事务(让消费 offset 与下游 Kafka 生产结果在同一个事务中提交)。它主要覆盖 Kafka 内部的 read-process-write 链路,不等于对数据库、HTTP 下游也天然恰好一次。

spring:
  kafka:
    producer:
      transaction-id-prefix: tx-order-
      properties:
        enable.idempotence: true
    consumer:
      isolation-level: read_committed

EOS 的代价:Exactly-Once 语义主要限定在 Kafka 的 read-process-write 链路内。一旦处理结果落到外部系统(如数据库),Kafka 事务不能替数据库提交或回滚。例如"消费 Kafka → 写入数据库"场景,数据库写入成功但 offset 未提交会导致重复消费;Kafka 事务提交成功但数据库提交失败会造成不一致。解决方案是使用幂等消费、事务协调或 Outbox 模式(见 21.5 节)。

21.3 Spring AMQP 与 RabbitMQ 集成

如说 Kafka 是分布式日志思维,那么 RabbitMQ 是经典的消息队列思维。RabbitMQ 实现了 AMQP(Advanced Message Queuing Protocol),其核心在于灵活的 Exchange 转 Routing 转 Queue 路由模型。Spring AMQP 提供了 RabbitTemplate(生产者)与 @RabbitListener(消费者)的高级抽象。

21.3.1 RabbitTemplate 生产者 API

spring:
  rabbitmq:
    host: localhost
    port: 5672
    username: guest
    password: guest
    virtual-host: /
    publisher-confirm-type: correlated
    publisher-returns: true

@Service
public class NotificationProducer {

    private final RabbitTemplate rabbitTemplate;

    public NotificationProducer(RabbitTemplate rabbitTemplate) {
        this.rabbitTemplate = rabbitTemplate;
    }

    public void sendNotification(NotificationEvent event) {
        rabbitTemplate.convertAndSend(
            "notification-exchange",
            "notification.email",
            event);
    }

    public void sendWithProperties(NotificationEvent event) {
        rabbitTemplate.convertAndSend(
            "notification-exchange",
            "notification.sms",
            event,
            message -> {
                MessageProperties props = message.getMessageProperties();
                props.setDeliveryMode(MessageDeliveryMode.PERSISTENT);
                props.setPriority(5);
                props.setExpiration("60000");
                props.setHeader("source", "order-service");
                return message;
            });
    }

    public PaymentResult requestPayment(PaymentRequest request) {
        return (PaymentResult) rabbitTemplate.convertSendAndReceive(
            "payment-exchange",
            "payment.process",
            request);
    }
}

RabbitMQ 生产端更关心 exchange、routing key、headers 与消息属性是否按预期进入 AMQP 消息:

21.3.2 @RabbitListener 消费者 API

@Component
public class NotificationConsumer {

    @RabbitListener(bindings = @QueueBinding(
        value = @Queue(value = "email-queue", durable = "true"),
        exchange = @Exchange(value = "notification-exchange", type = "topic"),
        key = "notification.email.*"))
    public void handleEmailNotification(NotificationEvent event) {
        emailService.send(event.getTo(), event.getSubject(), event.getBody());
    }

    @RabbitListener(queues = "sms-queue", ackMode = "MANUAL")
    public void handleSmsWithAck(
            NotificationEvent event,
            Channel channel,
            @Header(AmqpHeaders.DELIVERY_TAG) long tag) throws IOException {
        try {
            smsGateway.send(event.getPhone(), event.getText());
            channel.basicAck(tag, false);
        } catch (RecoverableException e) {
            channel.basicNack(tag, false, true);
        } catch (FatalException e) {
            channel.basicNack(tag, false, false);
        }
    }
}

手动确认模式下,业务成功应输出 basicAck;可恢复异常输出 basicNack(requeue=true);不可恢复异常输出 basicNack(requeue=false),随后消息会进入 DLX:

21.3.3 Exchange 四种类型

RabbitMQ 的核心路由能力由 Exchange 提供。生产者将消息发送到 Exchange,Exchange 根据其类型与 Routing Key 将消息路由到一个或多个 Queue。

Exchange 类型 路由依据 命中示例 典型场景
Direct Routing Key 精确匹配 key=order-created 转 绑定 order-created 的 Queue 命令、点对点任务
Topic Routing Key 通配符 shipment-created.cn.express 转 shipment-created.cn.* 事件分类、区域 与 业务线拆分
Fanout 忽略 Routing Key 所有绑定队列都收到 广播、配置刷新、日志分发
Headers Header 匹配 format=pdf,type=report 转 x-match=all 完全命中 少量复杂属性路由

Direct Exchange Routing Key 精确匹配,适合命令模式;默认 Exchange("")就是 Direct 类型。Topic Exchange Routing Key 通配符匹配:* 匹配一个单词,# 匹配零或多个;最灵活,也最常用。Fanout Exchange 忽略 Routing Key,广播到所有绑定队列;性能最高(无路由计算)。Headers Exchange 根据消息 Headers 路由(非 Routing Key);使用较少,适合复杂分类逻辑。

21.3.4 消息确认模式

模式 语义 适用场景
AUTO listener 正常返回后由容器确认;抛异常则 reject 或 nack 普通业务消息,配合幂等处理
MANUAL 消费者显式调用 basicAck 或 basicNack 关键业务消息(订单、支付)
NONE RabbitMQ autoAck=true,Broker 投递后即认为已确认 指标采样、非关键日志

21.3.5 死信交换机 DLX 模式

RabbitMQ 的 Dead Letter Exchange(DLX)是一个强大的模式:当消息被拒绝(basicNack 加 requeue=false)、TTL 过期或队列已满时,消息自动转发到 DLX 绑定的队列——而不是被丢弃。

@Configuration
public class DeadLetterConfig {

    @Bean
    public Queue orderQueue() {
        return QueueBuilder.durable("order-queue")
            .withArgument("x-dead-letter-exchange", "order-dlx")
            .withArgument("x-dead-letter-routing-key", "order.dead")
            .withArgument("x-message-ttl", 30000)
            .build();
    }

    @Bean
    public DirectExchange orderExchange() {
        return new DirectExchange("order-exchange");
    }

    @Bean
    public Binding orderBinding() {
        return BindingBuilder.bind(orderQueue())
            .to(orderExchange()).with("order.process");
    }

    @Bean
    public DirectExchange deadLetterExchange() {
        return new DirectExchange("order-dlx");
    }

    @Bean
    public Queue deadLetterQueue() {
        return QueueBuilder.durable("order-dead-letter-queue").build();
    }

    @Bean
    public Binding deadLetterBinding() {
        return BindingBuilder.bind(deadLetterQueue())
            .to(deadLetterExchange()).with("order.dead");
    }
}

如消费者拒绝且不重入队列,死信消息会带着原始交换机、路由键与死亡原因进入 DLQ:

  "headers": {
    "x-first-death-exchange": "order-exchange",
    "x-first-death-queue": "order-queue",
    "x-first-death-reason": "rejected",

21.4 Spring Cloud Stream

当系统同时使用 Kafka 与 RabbitMQ,或未来可能从一个迁移到另一个时,为每种中间件维护一套完全不同的生产者与消费者代码会带来巨大成本。Spring Cloud Stream 通过 Binder 抽象层将应用代码与底层中间件解耦,用统一的函数式编程模型(Supplier、Function、Consumer)编写消息驱动逻辑。

21.4.2 函数式编程模型

Spring Cloud Stream 4.x 完全拥抱函数式编程:只需定义标准的 java.util.function Bean,框架自动将它们绑定到消息通道。

@Configuration
public class OrderStreamConfig {

    @Bean
    public Supplier<OrderEvent> orderSource() {
        return () -> orderQueue.poll();
    }

    @Bean
    public Function<OrderEvent, EnrichedOrder> enrichOrder() {
        return order -> {
            CustomerInfo customer = customerService.findById(order.getCustomerId());
            return new EnrichedOrder(order, customer);
        };
    }

    @Bean
    public Consumer<EnrichedOrder> processOrder() {
        return enrichedOrder -> {
            log.info("processing enriched order {}", enrichedOrder.getOrderId());
            fulfillmentService.fulfill(enrichedOrder);
        };
    }
}

spring:
  cloud:
    stream:
      function:
        definition: orderSource;enrichOrder;processOrder
      bindings:
        orderSource-out-0:
          destination: order-created
          content-type: application/json
        enrichOrder-in-0:
          destination: order-created
          group: enrichment-group
        enrichOrder-out-0:
          destination: inventory-reserved
        processOrder-in-0:
          destination: inventory-reserved
          group: processing-group
      kafka:
        binder:
          brokers: localhost:9092
          auto-create-topics: true

启动后,三个函数会被 Binder 绑定成三条消息链路。业务代码没有出现 Kafka 或 RabbitMQ API,但日志会显示具体 binder 与 destination:

Bean 名称即绑定名:Spring Cloud Stream 函数式模型的核心约定——Bean 方法名决定了绑定名的前缀。enrichOrder 方法自动生成 enrichOrder-in-0(输入)与 enrichOrder-out-0(输出)两个绑定。如把方法命名为 enrich_order(带下划线),绑定名就变成了 enrich_order-in-0——拼写错误会导致绑定静默失败,消息无法路由。

21.5 消息可靠性保障

消息驱动架构的最大挑战不是发消息,而是保证消息不丢、不重、按序、与数据库操作原子一致。本节从生产者、消费者、事务三个维度解构可靠性,并介绍解决"数据库加消息"原子性的终极方案:Outbox 模式。

21.5.1 生产者可靠性

Kafka 生产者可靠性配置:

spring:
  kafka:
    producer:
      acks: all
      retries: 2147483647
      properties:
        enable.idempotence: true
        max.in.flight.requests.per.connection: 5
        delivery.timeout.ms: 120000
        request.timeout.ms: 30000
        linger.ms: 5
        batch.size: 16384

RabbitMQ 生产者可靠性配置:

spring:
  rabbitmq:
    publisher-confirm-type: correlated
    publisher-returns: true
    template:
      mandatory: true

@Configuration
public class RabbitConfirmConfig {

    @Bean
    public RabbitTemplate rabbitTemplate(ConnectionFactory connectionFactory) {
        RabbitTemplate template = new RabbitTemplate(connectionFactory);
        template.setMandatory(true);

        template.setConfirmCallback((correlationData, ack, cause) -> {
            if (!ack) {
                log.error("message not confirmed by broker {} cause {}",
                    correlationData, cause);
            }
        });

        template.setReturnsCallback(returned ->
            log.warn("message cannot be routed exchange={} routingKey={} replyCode={}",
                returned.getExchange(),
                returned.getRoutingKey(),
                returned.getReplyCode()));

        return template;
    }
}

21.5.2 消费者三种语义保证

语义 实现方式 适用场景
At-Most-Once 先提交 offset 或 ack,再处理消息 指标采样、日志(允许丢失)
At-Least-Once 先处理消息,再提交 offset 或 ack 大多数业务场景,配合幂等处理
Exactly-Once 处理与提交在同一事务边界,需幂等 金融、支付内部流水

At-Most-Once 配置:Kafka enable.auto.commit=true 或提前提交;RabbitMQ acknowledge-mode=none。处理失败时消息已确认会导致消息丢失。

At-Least-Once 配置:Kafka enable.auto.commit=false 加 ack-mode=manual;RabbitMQ acknowledge-mode=manual。处理成功但提交前崩溃会导致消息被重新消费(重复)。

Exactly-Once 实现:Kafka 事务加 幂等生产者覆盖 Kafka 内部链路;通用方案是幂等消费(业务去重)。处理与提交全部在一个事务中完成,性能代价最高。

21.5.3 ChainedTransactionManager 与局限性

当"消费 Kafka 消息 → 写入数据库"需要原子一致时,需要跨资源事务协调。Spring 提供了 ChainedTransactionManager,按顺序提交多个事务管理器——虽然不是真正的 XA 两阶段提交,但对部分场景可用。

@Configuration
public class ChainedTxConfig {

    @Bean
    public KafkaTransactionManager<String, Object> kafkaTxManager(
            ProducerFactory<String, Object> producerFactory) {
        return new KafkaTransactionManager<>(producerFactory);
    }

    @Bean
    public ChainedKafkaTransactionManager<String, Object> chainedTxManager(
            KafkaTransactionManager<String, Object> kafkaTxManager,
            DataSourceTransactionManager dbTxManager) {
        return new ChainedKafkaTransactionManager<>(kafkaTxManager, dbTxManager);
    }
}

链式事务管理器按顺序协调多个本地事务,但它不是真正的 XA 两阶段提交。如后提交的资源成功、先提交的资源已无法补偿,就可能出现不一致。ChainedKafkaTransactionManager 属于历史方案,Spring Kafka 已不推荐依赖它解决跨资源原子性;对"数据库 加 消息"这类长期演化的业务,更稳妥的方案是 Outbox 模式。

21.5.4 Outbox 模式

Outbox 模式将"发送消息"转化为"数据库写入"——既然数据库事务是可靠的,就利用它来保证"业务数据写入"与"消息发布意图"的原子性。核心思路:业务操作与消息写入同一个数据库事务中(写入 outbox 表),独立进程(Poller 或 CDC)负责从 outbox 表读取消息并发布到 Broker。

@Service
public class OrderService {

    private final OrderRepository orderRepository;
    private final OutboxRepository outboxRepository;
    private final ObjectMapper objectMapper;

    public OrderService(OrderRepository orderRepository,
                        OutboxRepository outboxRepository,
                        ObjectMapper objectMapper) {
        this.orderRepository = orderRepository;
        this.outboxRepository = outboxRepository;
        this.objectMapper = objectMapper;
    }

    @Transactional
    public Order createOrder(CreateOrderRequest request) throws Exception {
        Order order = orderRepository.save(request.toOrder());

        OutboxEvent event = OutboxEvent.builder()
            .aggregateType("Order")
            .aggregateId(order.getId())
            .eventType("order-created")
            .payload(objectMapper.writeValueAsString(new OrderCreatedEvent(order)))
            .build();
        outboxRepository.save(event);

        return order;
    }
}

@Component
public class OutboxPoller {

    private final OutboxRepository outboxRepository;
    private final KafkaTemplate<String, String> kafkaTemplate;

    public OutboxPoller(OutboxRepository outboxRepository,
                        KafkaTemplate<String, String> kafkaTemplate) {
        this.outboxRepository = outboxRepository;
        this.kafkaTemplate = kafkaTemplate;
    }

    @Scheduled(fixedDelay = 1000)
    @Transactional
    public void pollAndPublish() {
        List<OutboxEvent> events =
            outboxRepository.findByPublishedFalseOrderByCreatedAtAsc();

        for (OutboxEvent event : events) {
            String topic = event.getAggregateType().toLowerCase() + "-events";
            try {
                kafkaTemplate.send(topic, event.getAggregateId(), event.getPayload())
                    .get(5, TimeUnit.SECONDS);
                event.setPublished(true);
                event.setPublishedAt(Instant.now());
                outboxRepository.save(event);
            } catch (Exception ex) {
                log.warn("outbox publish failed id={}", event.getId(), ex);
            }
        }
    }
}
图 21-8 · Outbox 模式与 Debezium CDC 流程(单事务原子写 + Poller/CDC 异步发布)

轮询与 CDC:Outbox 模式有两种实现方式。轮询(Polling)简单直接但有延迟(取决于轮询间隔),且在高吞吐下可能成为瓶颈。CDC(Change Data Capture)通过监听数据库 binlog 与 WAL 实时捕获 outbox 表的变更,延迟更低(毫秒级)、吞吐更高。Debezium 是最流行的 CDC 工具,它与 Kafka Connect 集成。

21.5.5 设计取舍与被拒绝方案

方案 看起来的优点 生产中的代价 推荐
同步 HTTP 串联所有下游 代码直观,事务边界清楚 下游慢反压上游;重试风暴放大故障 仅用于强一致查询或短链路命令
@Transactional 里直接 send 看起来原子 数据库回滚时消息可能已发;消息失败时数据库可能已提交 用 Outbox 或事务同步回调
XA 或 2PC 跨数据库与 Broker 理论上强一致 Broker、驱动支持复杂;锁持有时间长 金融核心少数场景评估
Kafka EOS 覆盖所有副作用 名字诱人 仅覆盖 Kafka read-process-write 外部副作用仍要幂等
RabbitMQ 无限 nack 重入 期望故障自动恢复 毒丸消息热循环,拖垮队列 有限次重试,不可恢复入 DLX

21.6 陷阱与限制

消息驱动架构的异步本质让问题变得隐蔽——bug 不会立即崩溃,而是悄悄丢消息、默默重复消费、缓慢积压直到系统雪崩。

21.6.1 陷阱一:@KafkaListener 默认每分区单线程

Kafka 保证分区内有序,因此每个分区只能被消费者组中的一个消费者线程处理。设置 concurrency=3 意味着启动 3 个消费者线程,但如 topic 只有 2 个分区,多余的线程会空闲。反过来,如一个分区内消息处理耗时很长,该分区的所有后续消息都会排队等待——吞吐量受限于最慢的分区。

@KafkaListener(topics = "events", concurrency = "10")
public void wrongConcurrency(Event e) {
}

@KafkaListener(topics = "events", concurrency = "6")
public void correctConcurrency(Event e) {
}

21.6.2 陷阱二:RabbitMQ 消息 TTL 与队列 TTL 混淆

RabbitMQ 有两种 TTL:消息级 TTL(每条消息单独设置过期时间)与队列级 TTL(队列中所有消息统一过期时间)。两者的过期判定机制完全不同:队列级 TTL 的消息在队头过期时被丢弃(高效),消息级 TTL 需要逐条检查(可能延迟)。

@Bean
public Queue ttlQueue() {
    return QueueBuilder.durable("ttl-queue")
        .withArgument("x-message-ttl", 60000)
        .build();
}

如队列中前面一条消息 TTL 较长(60 秒),后面一条 TTL 较短(5 秒),后面的短 TTL 消息已到期但前面的长 TTL 消息未到期,后面的消息不会被立即移除,直到前面的被消费或过期。

21.6.3 陷阱三:Kafka 消费者 Rebalance 重复消费

当消费者组发生 Rebalance(新成员加入、旧成员退出、心跳超时)时,分区会被重新分配。如消费者正在处理消息但尚未提交 offset,Rebalance 后新的消费者会从上次提交的 offset 开始消费——导致已处理但未提交的消息被重复消费。

防御措施:

spring:
  kafka:
    consumer:
      properties:
        max.poll.interval.ms: 600000
        max.poll.records: 50
        session.timeout.ms: 30000
        heartbeat.interval.ms: 10000
        partition.assignment.strategy: org.apache.kafka.clients.consumer.CooperativeStickyAssignor

最根本的解决方案是消费者端幂等处理:

@KafkaListener(topics = "order-created")
public void handleIdempotent(OrderEvent event, Acknowledgment ack) {
    if (processedEventRepository.existsByEventId(event.getEventId())) {
        log.info("event already processed skipping {}", event.getEventId());
        ack.acknowledge();
        return;
    }
    orderService.process(event);
    processedEventRepository.save(event.getEventId());
    ack.acknowledge();
}

21.6.4 陷阱四:Spring Cloud Stream Bean 名称即绑定名

Spring Cloud Stream 函数式编程模型有一个容易踩坑的隐式约定:@Bean 方法的名称直接决定了绑定名的前缀(methodName-in-0、methodName-out-0)。如方法名拼写错误,绑定会静默失败——应用启动正常但消息不流通。

@Bean
public Consumer<OrderEvent> handleOrderEvent() {
    return event -> process(event);
}

如配置写的是 processOrder-in-0,但实际 Bean 名是 handleOrderEvent,消息永远不会被消费且没有报错。修复:确保 Bean 名与绑定名前缀一致,或显式指定 function.definition。

21.6.5 陷阱五:Kafka Producer acks=0 消息静默丢失

acks=0 意味着生产者发送消息后不等待任何确认——消息可能还没到达 Broker 就成功返回。网络抖动、Broker 宕机、分区 Leader 切换,任何一种情况都会导致消息静默丢失。

acks 参数 语义 吞吐 可靠性
0 不等待确认 最高 可能丢消息
1 等待 Leader 确认 中等 Leader 宕机可能丢
all 等待所有 ISR 确认 最低 最高,推荐生产配置

生产环境推荐配置 acks: all 加 enable.idempotence: true,并在 Broker 或 topic 侧配置 min.insync.replicas=2,保证至少 2 个 ISR 副本写入成功。

21.6.6 陷阱六:JSON 反序列化类型不匹配

当生产者与消费者使用 JSON 序列化,且消费者的 OrderEvent 类与生产者不同版本(新增字段、删除字段、类型变更)时,反序列化会抛出 DeserializationException。如没有配置错误处理,消息会被无限重试,消费者永远卡在这条毒丸消息上。

防御措施:

spring:
  jackson:
    deserialization:
      fail-on-unknown-properties: false
      fail-on-null-for-primitives: false
  kafka:
    consumer:
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.ErrorHandlingDeserializer
      properties:
        spring.deserializer.value.delegate.class: org.springframework.kafka.support.serializer.JsonDeserializer

ErrorHandlingDeserializer 在反序列化失败时不抛异常,而是将错误信息包装在 Header 中传递给 ErrorHandler。根本方案是使用 Avro 加 Schema Registry,从 Schema 注册阶段就阻止不兼容变更。

21.7.4 回归护栏

新增 listener 必须有一条 poison-message 测试,确认异常不会被吞掉,最终进入 DLT 或 DLQ。任何非幂等业务写入必须带业务唯一键或已处理事件表,不能只依赖 broker 的恰好一次语义。调整 concurrency、分区数、max.poll.records、prefetch 时,同时更新 lag、P95 与重试次数基线。修改消息 DTO 时必须跑旧版本样例消息反序列化测试;JSON 至少验证未知字段,Avro 至少验证 Schema Registry 兼容性。

21.8 本章小结

Spring Messaging 提供中间件无关的消息抽象——Message<T>(payload 加 headers)、MessageChannel(传输管道)、MessageHandler(消息处理器)构成了统一的编程模型。

Spring Kafka 覆盖完整的生产消费链路——KafkaTemplate 发送(支持回调、事务、幂等),@KafkaListener 消费(支持消费者组、分区分配、并发控制)。手动 offset 管理 加 DefaultErrorHandler 加 DeadLetterPublishingRecoverer 保障消费可靠性。

Spring AMQP 利用 RabbitMQ 灵活的路由模型——Direct、Topic、Fanout、Headers 四种 Exchange 覆盖从精确匹配到广播的所有路由需求。消息确认模式(AUTO、MANUAL、NONE)控制可靠性与吞吐的平衡。DLX 模式处理无法消费的毒丸消息。

Spring Cloud Stream 统一了跨中间件编程——Binder 抽象层让业务代码与 Kafka 或 RabbitMQ 解耦。函数式编程模型(Supplier、Function、Consumer)极简但强大。注意 Bean 名称就是绑定名前缀这一隐式约定。

消息可靠性需要全链路设计——生产者(acks=all 加 幂等 加 重试)、消费者(手动 ACK 加 幂等处理 加 DLQ)、事务(ChainedTransactionManager 或 Outbox 模式)。At-Least-Once 加 幂等消费是大多数场景的最优选择。

Outbox 模式是数据库加消息原子一致性的主方案——业务数据与消息发布意图写入同一个数据库事务,独立进程(Poller 或 Debezium CDC)负责将 outbox 表的消息发布到 Broker。CDC 方案延迟更低、吞吐更高。
