第 19 章 · 缓存策略与 W-TinyLFU

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

缓存是性能优化的第一选择——一次数据库查询通常 5 毫秒,本地内存命中只需约 50 纳秒,差距约 100,000 倍。但缓存的核心难题不是"怎么存",而是"内存满了之后淘汰谁"。

本章从三种经典淘汰策略(LRU、LFU、FIFO)的数据结构入手,引出 Spring Boot 默认本地缓存 Caffeine 使用的 W-TinyLFU 算法——一个将 LRU 的近期性与 LFU 的频率优势结合的混合策略。然后从算法层面上升到架构层面,讨论 Spring Cache 抽象如何统一多种缓存实现,以及生产环境中多级缓存(Caffeine 加 Redis)的设计模式。

前置知识:了解 HashMap 与双向链表即可。无需前置章节,但第 18 章 HikariCP 中"空间换时间"的思想在本章会再次出现。

19.1.3 心智模型的边界

Spring Cache 的注解模型只解决"是否命中、是否写入、何时失效"。下列能力不在通用模型内,需要在业务侧或 provider 侧单独封装:Caffeine 的 refreshAfterWrite 与异步 loader、Redis 的分布式锁与 pub/sub、跨节点的 L1 一致性、强一致更新场景下的事务边界与外部 outbox。

19.2 经典缓存淘汰策略

缓存的核心挑战是:内存有限时,淘汰哪些数据。三种经典策略各有优劣,理解它们是理解 W-TinyLFU 的前提。

19.2.1 LRU 最近最少使用

LRU 的核心思想:最近没被访问的数据,未来也不太可能被访问。实现基于双向链表加 HashMap。

图 19-2 · LRU 操作步骤(HashMap 定位 → 摘除 → 移到头部 → 淘汰尾部)

LRU 实现简单、操作 O(1),但有两个明显短板:扫描污染——一次大表全表扫描会把整个 LRU 队列冲刷一遍,挤掉真正的热点数据;不考虑频率——只关心"最近"二字,如果某个 key 被访问过 1000 次但最近一分钟没访问,会被一次性单点访问的新 key 挤出去。

19.2.2 LFU 最不经常使用

LFU 走向另一个极端——按访问频率排序,淘汰访问次数最少的条目。O'Neil 等人在 1993 年的论文 "The LRU-K Page Replacement Algorithm"(SIGMOD)首次系统分析了频率在页面置换中的价值。纯 LFU 的数据结构通常是频率桶链表——每个桶对应一个频率值,桶内维护双向链表。Shah 等人 2010 年的论文给出了 O(1) 时间复杂度的 LFU 实现。

图 19-3 · LFU 频率桶结构(HashMap → 频率桶链表 → 桶内 LRU)

LFU 的问题:历史热点难淘汰——某个 key 在过去一小时被访问 10000 次,即使现在已不再被访问,频率值仍远高于其他 key,长期占据缓存空间;新数据不友好——刚进入缓存的新 key 频率为 1,容易被立即淘汰。

19.2.3 FIFO 先进先出

FIFO 是最简单的淘汰策略——完全不考虑访问频率与时间新旧,只看谁先进来。它的实现就是一个队列:新数据从队尾入队,淘汰时从队头移除。

图 19-4 · FIFO 队列淘汰(新数据队尾入 / 容量满时队头出)

FIFO 的优势在于实现简单、无额外元数据开销(不需要维护访问时间或频率计数器),适合缓存命中率不敏感的场景(如 DNS 本地缓存、日志去重窗口)。在绝大多数业务场景中,FIFO 的命中率远不如 LRU 与 LFU。

19.2.4 三种策略的综合对比

策略 核心思想 数据结构 get/put 优点 缺点
LRU 最近未访问 等于 最该淘汰 HashMap 加 双向链表 O(1) 实现简单、性能稳定 不考虑频率,易被扫描污染
LFU 访问最少 等于 最该淘汰 HashMap 加 频率桶链表 O(1) 频率敏感、抗扫描 历史热点难淘汰、空间开销大
FIFO 最先进入 等于 最该淘汰 Queue O(1) 极简实现 完全不考虑访问模式

19.2.5 Redis 作为近似 LRU 缓存的实践

Redis 通过 maxmemory-policy: allkeys-lru 可以变成一个巨大的近似 LRU 缓存:当内存达到上限时,它会基于采样结果淘汰相对更久未使用的 key,而不是维护一条全局精确 LRU 链表。Spring Boot 通过 Lettuce 客户端连接 Redis,适合作为 L2 分布式缓存层。生产环境中通常设置 2 至 8 GB 内存限制,配合 LRU 淘汰策略,让 Redis 自动管理缓存容量。

spring:
  data:
    redis:
      host: redis-cluster.internal
      port: 6379
      lettuce:
        pool:
          max-active: 32
          max-idle: 8

maxmemory 2gb
maxmemory-policy allkeys-lru

19.3 Caffeine W-TinyLFU 算法

Caffeine 是 Spring Boot 推荐的本地缓存库,其核心算法 W-TinyLFU(Window TinyLFU)综合了 LRU 的近期性与 LFU 的频率优势,在大量读热点与混合访问负载下能取得很高的命中率。

19.3.1 三区架构

W-TinyLFU 把缓存拆成三个分区:Window Cache 收容新进入的数据,无准入门槛;Main Cache 内部分为 Probation 试用区与 Protected 保护区;TinyLFU 准入过滤器位于 Window 与 Main 之间,负责频率比较。

图 19-5 · W-TinyLFU 三区架构(Window 1% / Probation 20% / Protected 80% + TinyLFU 过滤)

Window 区的存在是关键:它给新进入的数据一个"试用窗口",避免新数据因为初始频率为 0 而立即被 LFU 淘汰。同时它的容量极小(约 1%),保证扫描污染的破坏力被限制在很小范围内。

19.3.2 Spring Cloud LoadBalancer 案例

Spring Cloud LoadBalancer 使用 Caffeine 缓存从 Eureka 或 Nacos 获取的服务实例列表,默认 TTL 为 35 秒。这避免了每次 RPC 调用都向注册中心查询——高频微服务调用场景下,没有这层 Caffeine 缓存会把注册中心打垮。@LoadBalancerClient 注解内部为每个服务名创建独立的 Caffeine Cache 实例,利用 W-TinyLFU 算法保证热点服务的实例列表常驻内存。

spring:
  cloud:
    loadbalancer:
      cache:
        ttl: 35s
        capacity: 256

@LoadBalancerClient(name = "order-service",
    configuration = CustomLoadBalancerConfig.class)
public class OrderServiceClient { }

19.3.3 Count-Min Sketch 频率估算

TinyLFU 使用 Count-Min Sketch 估算每个 key 的访问频率,而非为每个 key 维护精确计数器。这是一种概率数据结构,用极少内存实现近似频率统计。

图 19-6 · Count-Min Sketch 工作原理(4 hash → 4 行计数器 → 取最小值估算)

Count-Min Sketch 的精妙之处:多个 hash 函数能容忍单个 hash 冲突;查询时取最小值,因为最小值最不容易被冲突高估;空间复杂度 O(d 乘 w),与实际 key 数量无关。

维度 精确计数 Map Count-Min Sketch
内存 随 key 数增长 固定大小
误差 精确 近似,可能高估
适合场景 小 key 集合 海量 key 的热点判断

19.3.4 Alibaba Sentinel 滑动窗口对照

Sentinel 的 QPS 限流器在概念上与 Count-Min Sketch 类似——都用近似计数实现高效统计。Sentinel 使用 LeapArray(环形滑动窗口)数据结构:将一秒钟划分为多个小窗口(默认 2 个 500 毫秒窗口),每个窗口维护独立的请求计数器。窗口过期后自动复用,内存占用恒定。

public class LeapArray<T> {
    protected int windowLengthInMs;
    protected int sampleCount;
    protected final AtomicReferenceArray<WindowWrap<T>> array;

    public WindowWrap<T> currentWindow(long timeMillis) {
        long timeId = timeMillis / windowLengthInMs;
        int idx = (int) (timeId % array.length());
        return array.get(idx);
    }
}

19.3.5 频率衰减 Aging

Count-Min Sketch 的计数器会不断增长,如不衰减,历史热点的频率会永远高于新数据。Caffeine 使用一种简洁的衰减策略——所有计数器右移 1 位。

key 衰减前估算频率 右移 1 位后 含义
A 旧热点 1000 500 历史热度被削弱
B 新热点 12 6 新近访问仍保留相对优势
C 冷数据 1 0 很快被淘汰出候选

衰减触发时机:Caffeine 在写入次数达到 sampleSize(约等于缓存容量的 10 倍)时执行一次 reset。结果是历史热点的频率被削弱,新热点有机会反超。

19.3.6 Admission Filter 准入决策

当 Window 区淘汰出一个候选者(candidate),而 Probation 区也有一个受害者(victim)需要被淘汰时,TinyLFU 过滤器进行频率 PK。

图 19-7 · TinyLFU 准入决策流程(频率 PK + 低频拒绝 + 平局随机准入)

这里的"随机准入"不是业务随机缓存,而是算法层面对接近频率的候选者做轻微扰动,避免某些访问模式下长期僵持。工程上只需要记住:新对象先进入 Window,只有经过 TinyLFU 频率比较后,才有资格长期留在 Main Cache。

19.3.7 Caffeine 源码入口地图

Caffeine 的难点不在 Cache.getIfPresent() API,而在"高并发访问时如何既维护访问顺序,又不让每次 get 都抢全局锁"。读源码时抓住下面几类对象,比从 Builder 一路追到底更快。

19.3.8 W-TinyLFU 落到运行时的设计取舍

设计取舍 Caffeine 的选择 被拒绝的替代方案 结果

19.4 Spring Cache 抽象

Spring 提供了统一的缓存抽象层,通过注解声明式地管理缓存,底层可以切换不同的缓存实现(Caffeine、Redis、JCache/Ehcache 3 等)。

19.4.2 @Cacheable 第一次 miss 第二次 hit

@Cacheable 的语义是:先查缓存,命中就短路返回,未命中才执行方法体。最推荐显式写 key,避免多参数默认 SimpleKey 让排查变困难。

@Service
public class InventoryService {
    private final InventoryRepository repository;

    public InventoryService(InventoryRepository repository) {
        this.repository = repository;
    }

    @Cacheable(cacheNames = "inventory", key = "'inventory:' + #sku",
               unless = "#result == null")
    public InventoryDto findBySku(String sku) {
        log.info("DB query inventory sku={}", sku);
        return repository.findBySku(sku)
                .map(InventoryDto::from)
                .orElse(null);
    }
}

第一次请求是 miss,Spring 执行方法体,日志里能看到 key 生成与 DB 查询。

TRACE o.s.cache.interceptor.CacheInterceptor : Computed cache key 'inventory:SKU-REDIS-001' for operation Builder findBySku caches=[inventory]
TRACE o.s.cache.interceptor.CacheInterceptor : No cache entry for key 'inventory:SKU-REDIS-001' in cache(s) [inventory]
INFO  c.example.cache.InventoryService       : DB query inventory sku=SKU-REDIS-001
TRACE o.s.cache.interceptor.CacheInterceptor : Creating cache entry for key 'inventory:SKU-REDIS-001' in cache(s) [inventory]

第二次请求命中缓存,响应相同,但方法体不执行,日志里不会再出现 DB query inventory。

TRACE o.s.cache.interceptor.CacheInterceptor : Computed cache key 'inventory:SKU-REDIS-001' for operation Builder findBySku caches=[inventory]

19.4.3 @CachePut 更新数据库后覆盖缓存

@CachePut 与 @Cacheable 最大的不同是:它永远执行方法体,然后把返回值写回缓存。适合"更新 DB 后让缓存立刻变成新值"的路径。

@CachePut(cacheNames = "inventory", key = "'inventory:' + #sku")
public InventoryDto updatePrice(String sku, BigDecimal price) {
    Inventory inventory = repository.findBySkuOrThrow(sku);
    inventory.changePrice(price);
    Inventory saved = repository.save(inventory);
    log.info("DB update inventory sku={} price={} version={}",
        sku, price, saved.version());
    return InventoryDto.from(saved);
}

接口返回的是更新后的对象,下一次 GET 会直接命中新值,不需要等 TTL。

INFO  c.example.cache.InventoryService       : DB update inventory sku=SKU-REDIS-001 price=128.00 version=8
TRACE o.s.cache.interceptor.CacheInterceptor : Creating cache entry for key 'inventory:SKU-REDIS-001' in cache(s) [inventory]
INFO  c.example.cache.HttpAccessLog          : PUT /api/inventory/SKU-REDIS-001 status=200 duration=18ms source=db

19.4.4 @CacheEvict 删除或失效后清缓存

@CacheEvict 适合删除、状态迁移、批量导入后让读缓存失效。默认是方法成功返回后才删除缓存;如业务方法抛异常,缓存仍保留。

@CacheEvict(cacheNames = "inventory", key = "'inventory:' + #sku")
public void discontinue(String sku) {
    repository.markDiscontinued(sku);
    log.info("DB mark inventory discontinued sku={}", sku);
}

@CacheEvict(cacheNames = "inventory", allEntries = true)
public ImportReport importCatalog(Path csv) {
    ImportReport report = importer.importFrom(csv);
    log.warn("inventory catalog imported, cache cleared rows={}", report.rows());
    return report;
}

删除成功后缓存条目被清掉;下一次读取会重新 miss,并按当前 DB 状态返回。allEntries = true 会清整个 cache,但务必限制在具体 cacheName,避免误伤其他缓存。

TRACE o.s.cache.interceptor.CacheInterceptor : Invalidating cache key 'inventory:SKU-REDIS-001' in cache(s) [inventory]
WARN  c.example.cache.CatalogImportService   : inventory catalog imported, cache cleared rows=1208
TRACE o.s.cache.interceptor.CacheInterceptor : Invalidating entire cache 'inventory'

19.4.5 注解处理流程的真实分岔

@Cacheable、@CachePut、@CacheEvict 可以组合在同一方法上,但源码不是"按注解书写顺序执行"。CacheAspectSupport 会先处理 beforeInvocation=true 的 evict,再寻找 @Cacheable 命中;只有没有命中或存在强制执行的 put 时才调用业务方法;返回后再处理 put 与 after-invocation evict。

19.5 源码架构:从注解到 Cache.put

如只把 Spring Cache 理解成三个注解,很容易在生产上踩到"为什么没命中""为什么 DB 还是被打爆""为什么事务回滚后缓存已经删了"这些坑。源码阅读时抓住三个入口即可:CacheInterceptor 负责进入 AOP 环绕调用;CacheAspectSupport 负责组织注解语义;CacheOperationSource 负责把方法上的注解解析成可执行的缓存操作。

19.5.1 源码入口地图

图 19-9a · Spring Cache 源码入口地图(Interceptor → Aspect → Operation → Resolve → Invoke)

这张图背后的设计取舍是"注解语义与底层缓存实现解耦"。Spring 不让 CacheInterceptor 直接依赖 Caffeine 或 Redis,而是把缓存能力压缩到 Cache 与 CacheManager 接口;代价是它只能表达通用语义,像 Caffeine 的 refresh、Redis 的分布式锁、批量 pipeline 这类 provider 特性都不在 @Cacheable 的核心模型里。

19.5.2 关键源码入口职责

源码入口 主要职责 阅读时要看什么
CacheInterceptor.invoke 把 AOP 方法调用包装成 CacheOperationInvoker 自调用为什么绕过;异常如何向外抛
CacheAspectSupport.execute 处理 evict、cacheable、put 的执行顺序 @CachePut 为什么一定执行方法体
CacheOperationSource 从方法和类上找缓存注解 接口或实现类注解放在哪里会被识别
CacheOperationExpressionEvaluator 计算 SpEL 的 key、condition、unless #result 只在返回后可用
SimpleKeyGenerator 默认 key 生成策略 多参数默认 key 可读性差但不是简单字符串拼接
AbstractCacheInvoker 对 Cache.get/put/evict 包一层错误处理 缓存故障是否吞掉取决于 CacheErrorHandler

19.5.3 condition 与 unless 的执行时机

condition 在调用方法前判断,适合"这个请求要不要走缓存";unless 在方法返回后判断,适合"这个结果要不要写缓存"。这不是语法差异,而是源码执行时机差异:condition 在构建 CacheOperationContext 后即可计算;unless 必须等 CacheOperationInvoker 拿到返回值。

@Cacheable(
        cacheNames = "inventory",
        key = "#tenantId + ':' + #sku",
        condition = "#sku.length() <= 32",
        unless = "#result == null || #result.discontinued()")
public InventoryDto findInventory(String tenantId, String sku) {
    log.info("load inventory tenant={} sku={}", tenantId, sku);
    return repository.findAvailableInventory(tenantId, sku)
            .map(InventoryDto::from)
            .orElse(null);
}

短 SKU 第一次 miss 后写缓存;下架商品或 null 结果不会写缓存;超长 SKU 从一开始就跳过缓存路径。

19.5.4 把 key 设计显式化

生产系统不要把 key schema 藏在 SpEL 字符串里到处复制。跨服务、跨语言、跨 Redis 工具排查时,推荐用一个可测试的 KeyGenerator 固化版本号、租户、方法名与参数规范。

@Configuration
class CacheKeyConfig {
    @Bean("versionedKeyGenerator")
    KeyGenerator versionedKeyGenerator() {
        return (target, method, params) -> {
            String joined = Arrays.stream(params)
                    .map(String::valueOf)
                    .collect(Collectors.joining(":"));
            return "v2:" + target.getClass().getSimpleName()
                + ":" + method.getName() + ":" + joined;
        };
    }
}

@Cacheable(cacheNames = "inventory", keyGenerator = "versionedKeyGenerator")
public InventoryDto findInventory(String tenantId, String sku) {
    return repository.findAvailableInventory(tenantId, sku)
            .map(InventoryDto::from)
            .orElse(null);
}

这个设计的取舍是:key 更长、Redis 内存占用略高,但换来可灰度的版本前缀和可审计的业务语义。对百万级 key 的缓存,key 长度要纳入容量估算;对低频后台缓存,可读性通常比节省几十字节更重要。

19.5.5 缓存错误处理边界

Spring 的默认立场偏"缓存是业务的一部分":底层 Cache.get 抛异常会向上传播。很多读多写少系统希望 Redis 短暂抖动时降级到 DB,这时要显式配置 CacheErrorHandler,并把降级日志打到可告警的级别。

@Bean
CacheErrorHandler cacheErrorHandler() {
    return new CacheErrorHandler() {
        @Override
        public void handleCacheGetError(RuntimeException ex, Cache cache, Object key) {
            log.warn("cache get failed cache={} key={} fallback to method invocation",
                cache.getName(), key, ex);
        }

        @Override
        public void handleCachePutError(RuntimeException ex, Cache cache, Object key, Object value) {
            throw ex;
        }

        @Override
        public void handleCacheEvictError(RuntimeException ex, Cache cache, Object key) {
            throw ex;
        }

        @Override
        public void handleCacheClearError(RuntimeException ex, Cache cache) {
            throw ex;
        }
    };
}

边界规范:读缓存失败可以按业务等级降级,写缓存与删缓存失败通常不能吞。否则 @CachePut 成功返回但缓存未更新,下一次读仍然命中旧值;@CacheEvict 失败则可能把已经删除或降权的数据继续暴露给用户。

19.5.6 生产边界 checklist

边界 必问问题 推荐做法
代理边界 调用是否经过 Spring AOP 代理 避免 this.xxx();缓存方法放到独立 Bean
Key 边界 key 是否含租户、版本、业务 ID 显式 SpEL 或统一 KeyGenerator
结果边界 null、空集合、下架对象是否可缓存 用 unless 明确写入规则

观测边界 能否从日志、metrics、Redis 反推一次命中 TRACE 日志加 cache.gets 加 provider stats

19.6 Spring Boot 自动配置入口与扩展边界

Spring Cache 的源码要分两层读:Spring Framework 提供注解拦截与 Cache 抽象;Spring Boot 负责根据 classpath 与 spring.cache.* 创建具体 CacheManager。出问题时先判断卡在哪一层,否则很容易把自动配置问题误判成 Caffeine 或 Redis 算法问题。

19.6.3 设计取舍与被拒绝方案

方案 优点 问题或不作为默认模型的原因

19.7 多级缓存架构

生产环境中通常需要多级缓存,兼顾延迟与容量。

19.7.1 L1/L2/L3 三级架构

图 19-10 · 三级缓存架构(L1 Caffeine 0.1us / L2 Redis 0.5-2ms / L3 DB 5-50ms)
层级 位置 典型延迟 容量 淘汰与职责
L1 Caffeine 进程内内存 约 0.1 微秒 1 万至 10 万条 W-TinyLFU,挡住热点读
L2 Redis 分布式集群 约 0.5 至 2 毫秒 百万至亿级 allkeys-lru 或 TTL,跨实例共享
L3 数据库 真实数据源 约 5 至 50 毫秒 业务容量 权威数据,缓存 miss 后兜底

19.7.2 缓存失效策略对比

多级缓存的最大挑战是缓存一致性。当数据更新时,如何确保所有层级的缓存都能及时失效。

策略 一致性延迟 实现复杂度 多实例支持 适用场景
TTL 过期 0 至 TTL 低 天然支持 静态数据、报表
本地事件 准实时 中 单实例最佳 单体应用内的库存、价格
Pub/Sub 准实时 高 专为多实例设计 微服务集群
TTL 加事件 准实时加兜底 中高 需配合 Pub/Sub 生产推荐方案

缓存一致性是分布式系统中最难的问题之一。先删缓存再更新 DB存在并发读重建脏缓存的风险;先更新 DB 再删缓存是推荐方案,需配合消息队列做重试保证删除成功;延迟双删先删缓存 → 更新 DB → 延迟 N 秒再删一次,兜底清理并发脏数据;Canal 订阅 Binlog 是终极方案,监听 MySQL binlog 自动同步缓存,但引入额外组件。

19.7.3 设计缓存的三个根本问题

Phil Karlton 有一句名言:"计算机科学只有两个难题——缓存失效与命名"。在设计多级缓存时,永远要回答三个问题:缓存什么——热点数据、计算开销大的结果、不频繁变更的配置;何时失效——TTL 多长、事件触发还是主动轮询;失效后怎么办——缓存击穿防护(互斥锁、布隆过滤器)、降级策略。

19.7.4 JetCache 案例

JetCache 是阿里巴巴开源的缓存框架,通过 @Cached 注解即可实现 L1 Caffeine 加 L2 Redis 的两级缓存。当 L1 本地缓存失效时,自动从 L2 Redis 加载并回填;当数据更新时,JetCache 通过 Redis Pub/Sub 广播失效消息,通知集群中所有节点清除对应的 L1 缓存条目。这解决了多实例部署下本地缓存不一致的经典问题。

@Cached(name = "userCache:",
        cacheType = CacheType.BOTH,
        localLimit = 1000,
        expire = 3600,
        localExpire = 60)
@CacheRefresh(refresh = 300, stopRefreshAfterLastAccess = 600)
public Order getOrderById(long customerId) {
    return orderMapper.selectById(customerId);
}

19.9 陷阱与限制

19.9.1 陷阱一:@Cacheable 的自调用失效

@Cacheable 基于 AOP 代理实现。this.method() 调用绕过代理,缓存注解无效。

@Service
public class OrderService {
    public Order getOrder(Long id) {
        return this.getOrderById(id);
    }

    @Cacheable("orders")
    public Order getOrderById(Long id) { return null; }
}

修复方式是把缓存方法拆到独立 Bean,让调用经过代理:

@Service
public class OrderFacade {
    private final OrderLookupService lookup;

    public OrderFacade(OrderLookupService lookup) {
        this.lookup = lookup;
    }

    public Order getOrder(Long id) {
        return lookup.getOrderById(id);
    }
}

@Service
public class OrderLookupService {
    @Cacheable("orders")
    public Order getOrderById(Long id) { return null; }
}

19.9.2 陷阱二:@Cacheable 默认缓存 null 值

数据库中不存在的 ID 查询返回 null,null 被缓存,后续插入该 ID 的数据后缓存命中的仍然是 null。这是"缓存穿透"的典型变体。修复:使用 unless 排除 null 结果——@Cacheable(value = "orders", unless = "#result == null")。

19.9.3 陷阱三:缓存 key 的 SpEL 表达式错误

多参数方法默认使用 SimpleKeyGenerator,key 是 SimpleKey(arg1, arg2)。它不会因为普通 hashCode 冲突就串 key,但日志或 Redis 中很难人工排查;如参数对象的 equals/hashCode 写错,风险会被放大。修复方式是显式指定 key:

@Cacheable(value = "orders", key = "#customerId + ':' + #orderId")
public Order getOrder(Long customerId, Long orderId) { return null; }

19.9.4 陷阱四:Redis 缓存的序列化问题

Spring Cache 默认使用 JDK 序列化,对象类新增字段后旧缓存反序列化失败,出现 SerializationException,应用局部不可用。修复方案:使用 JSON 序列化(GenericJackson2JsonRedisSerializer);制定缓存版本策略(key 中包含版本号);部署时清除缓存或设置合理的 TTL。

19.9.5 陷阱五:多级缓存的一致性窗口

L1(Caffeine)加 L2(Redis)架构中,更新数据时先更新 DB → 再删 Redis → 再清 Caffeine。但多实例部署下,Node-A 清了自己的 Caffeine,Node-B 的 Caffeine 仍有旧数据,造成不一致。修复:通过 Redis Pub/Sub 或消息队列广播缓存失效事件到所有节点。注意 Redis Pub/Sub 是 best-effort 的——如某个节点断线期间错过了消息,它的本地缓存会一直是旧数据,直到 TTL 过期。

19.9.6 陷阱六:@CacheEvict 与事务回滚

@CacheEvict 默认在方法成功返回后执行,但如方法所在事务在更晚的 commit 阶段回滚,缓存已被清除而数据库回到旧值——下次读 miss 时从 DB 重建的仍是旧值,看起来"刚改的数据丢了"。强一致路径应把缓存失效放到事务提交后事件(@TransactionalEventListener)或 outbox,而不是只靠注解顺序。

19.9.7 陷阱七:缓存击穿与雪崩

缓存击穿指某个热点 key 突然失效,瞬间大量请求穿透到 DB;缓存雪崩指大量 key 在同一时刻集中失效。防护方式:为相同业务的 key 设置带随机抖动的 TTL,避免对齐失效;使用单机锁(synchronized)或分布式锁(Redis Setnx)保证同一 key 仅一次重建;关键路径配合限流与降级,Sentinel 的 DegradeRule 可在 DB 异常时返回兜底值。

19.10 本章小结

没有完美的淘汰策略——LRU 怕扫描污染,LFU 对新数据不友好,FIFO 完全不考虑访问模式。W-TinyLFU 用混合策略在绝大多数场景下取得最优命中率。

Count-Min Sketch 是"用精度换内存"——它不存储精确计数,而是用多个哈希函数取最小值估算频率;估算值可能高于真实频率,但高估误差可控。

Spring Cache 是门面模式——应用代码只依赖 @Cacheable 注解,底层可以随时从 ConcurrentHashMap 切换到 Caffeine 或 Redis,无需改代码。

多级缓存的一致性是难点——L1 缓存快但容量小且不跨节点同步,L2 缓存慢但全局一致。Redis Pub/Sub 或 MQ 广播是同步 L1 的常见方案。
