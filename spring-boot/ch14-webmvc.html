每一个到达 Spring Boot 应用的 HTTP 请求,从 Servlet 容器进入 DispatcherServlet,经过路由匹配、参数解析、业务执行、异常处理、返回值转换,最终以 JSON 或 HTML 形式回到客户端。这条管线是 Spring Web MVC 的核心——一个运行 20 年仍在持续演进的请求处理引擎。

理解这条管线对生产问题排查至关重要。"参数绑定失败""JSON 序列化异常""拦截器执行顺序混乱"这些问题的诊断,均依赖对管线的精确认知。本章从 DispatcherServlet 的 9 大策略组件起,逐层拆解每个环节的设计与实现,以 7 个高频陷阱收尾。

14.1 DispatcherServlet 架构全景

Spring Web MVC 采用经典 Front Controller 模式——所有 HTTP 请求先到达中央调度器 DispatcherServlet,再由它委托给各策略组件完成路由、参数绑定、业务执行、视图渲染。该设计将"请求调度"与"业务逻辑"彻底解耦。

protected void doDispatch(HttpServletRequest request, HttpServletResponse response) throws Exception {
    HttpServletRequest processedRequest = request;
    HandlerExecutionChain mappedHandler = null;
    boolean multipartRequestParsed = false;

    try {
        ModelAndView mv = null;
        Exception dispatchException = null;

        try {
            processedRequest = checkMultipart(request);
            multipartRequestParsed = (processedRequest != request);

            mappedHandler = getHandler(processedRequest);
            if (mappedHandler == null) {
                noHandlerFound(processedRequest, response);
                return;
            }

            HandlerAdapter ha = getHandlerAdapter(mappedHandler.getHandler());

            if (mappedHandler.applyPreHandle(processedRequest, response) == false) {
                return;
            }

            mv = ha.handle(processedRequest, response, mappedHandler.getHandler());

            mappedHandler.applyPostHandle(processedRequest, response, mv);

        } catch (Exception ex) {
            dispatchException = ex;
        }

        processDispatchResult(processedRequest, response,
                              mappedHandler, mv, dispatchException);

    } finally {
        if (multipartRequestParsed) {
            cleanupMultipart(processedRequest);
        }
    }
}

Spring Cloud Gateway 的 DispatcherHandler

WebFlux 中 DispatcherHandler 是 DispatcherServlet 的等价物。维护 HandlerMapping、HandlerAdapter、HandlerResultHandler 三类策略组件,所有方法返回 Mono / Flux。

public Mono<Void> handle(ServerWebExchange exchange) {
    return Flux.fromIterable(this.handlerMappings)
        .concatMap(mapping -> mapping.getHandler(exchange))
        .next()
        .flatMap(handler -> invokeHandler(exchange, handler))
        .flatMap(result -> handleResult(exchange, result));
}

14.4 异常处理机制

Web 应用中异常不可避免——参数校验失败、数据库连接超时、业务规则冲突、资源不存在。Spring MVC 提供三层异常处理机制,从局部到全局逐层兜底。

表 14-4-1:HandlerExceptionResolver 链

优先级Resolver职责
1 最高ExceptionHandlerExceptionResolver调用 @ExceptionHandler 方法
2ResponseStatusExceptionResolver处理 @ResponseStatus 注解的异常

@RestControllerAdvice
public class GlobalExceptionHandler {

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<ProblemDetail> handleValidation(
            MethodArgumentNotValidException ex) {
        ProblemDetail pd = ProblemDetail.forStatus(HttpStatus.BAD_REQUEST);
        pd.setTitle("Validation Failed");
        Map<String, String> errors = ex.getBindingResult()
            .getFieldErrors().stream()
            .collect(Collectors.toMap(
                FieldError::getField,
                fe -> fe.getDefaultMessage() != null ? fe.getDefaultMessage() : "",
                (first, ignored) -> first
            ));
        pd.setProperty("fieldErrors", errors);
        return ResponseEntity.badRequest().body(pd);
    }

    @ExceptionHandler(Exception.class)
    public ProblemDetail handleGeneric(Exception ex) {
        ProblemDetail pd = ProblemDetail.forStatus(HttpStatus.INTERNAL_SERVER_ERROR);
        pd.setTitle("Internal Server Error");
        pd.setDetail("An unexpected error occurred.");
        return pd;
    }
}

14.6 内容协商与序列化

同一 API 端点,不同客户端要求不同格式(JSON、XML、CSV)——这是内容协商(Content Negotiation)解决的问题。

图 14-6-1 · 内容协商决策树(URL 参数 → Accept 头 → 默认 → Converter 匹配)
@Configuration
public class ContentNegotiationConfig implements WebMvcConfigurer {

    @Override
    public void configureContentNegotiation(ContentNegotiationConfigurer configurer) {
        configurer
            .favorParameter(true)
            .parameterName("format")
            .defaultContentType(MediaType.APPLICATION_JSON)
            .mediaType("json", MediaType.APPLICATION_JSON)
            .mediaType("xml", MediaType.APPLICATION_XML)
            .mediaType("csv", MediaType.valueOf("text/csv"));
    }
}

    return builder -> builder
        .simpleDateFormat("yyyy-MM-dd HH:mm:ss")
        .timeZone(TimeZone.getTimeZone("Asia/Shanghai"))
        .serializationInclusion(JsonInclude.Include.NON_NULL)
        .featuresToDisable(

            DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES
        )
        .modulesToInstall(JavaTimeModule.class);
}

自定义 ObjectMapper Bean 覆盖 Spring Boot 全部默认配置。推荐使用 Jackson2ObjectMapperBuilderCustomizer,增量定制不影响 Boot 默认配置。

14.末 本章要点

  • DispatcherServlet 是 Spring MVC 的中枢。通过 initStrategies 初始化 9 大策略组件,所有请求由 doDispatch 统一调度。WebFlux 的 DispatcherHandler 是其响应式等价物。
  • HandlerMapping 决定路由归属。RequestMappingHandlerMapping 启动时扫描所有 @RequestMapping,运行时按最精确匹配原则选择 Handler。Spring Boot 2.6+ 默认使用性能更优的 PathPatternParser。
  • 参数解析与返回值处理是可扩展的链式架构。30+ 个 ArgumentResolver 与 HttpMessageConverter 按顺序尝试匹配。
  • 异常处理有三层防线。Controller 内 @ExceptionHandler → @ControllerAdvice 全局 → HandlerExceptionResolver 链 → BasicErrorController。Spring Boot 3.x 推荐 ProblemDetail(RFC 7807)。
  • Filter / Interceptor / ControllerAdvice 各有其位。Filter 在 Servlet 层(可包装 Request/Response),Interceptor 在 Spring MVC 层,ControllerAdvice 在方法级别。
  • 内容协商让同一 API 服务多种格式。Jackson ObjectMapper 定制推荐使用 Jackson2ObjectMapperBuilderCustomizer。
  • 7 大陷阱需警惕:InputStream 一次性、@ModelAttribute vs @RequestBody、循环视图、void 返回 200、preHandle false 不触发 afterCompletion、后缀匹配默认禁用、@ControllerAdvice 无限定范围。
