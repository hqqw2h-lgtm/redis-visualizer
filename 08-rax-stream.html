第 8 章 · rax 基数树与 Stream

路径压缩前缀树 · raxNode 柔性数组 · iscompr 双形态 · listpack 二级打包 · 多 rax 设计 · Consumer Group · PEL 至少一次

Stream 要解决的是一个很朴素的问题:消息按 ID 排序写入,还要能按范围读取、让消费组确认进度。哈希表很快,但没有顺序;跳表能排序,但每条消息都要额外节点;Redis 最后用 rax 做内存里的有序索引。

rax(Radix Tree,基数树)是一棵路径压缩的前缀树:相同前缀只存一次,分叉处再拆节点。Stream ID 是 8 字节毫秒时间 + 8 字节序号,同一毫秒内的消息至少共享前 8 字节,序号较小时还会共享更多前缀。rax 利用这个特点减少索引元数据,但不要把它理解成“固定倍数更快”的魔法常数。

Stream 内部把若干相邻消息打包到一个 listpack(master entry + 增量字段),rax 索引 "每个 listpack 的首消息 ID"。XADD 时根据 ID 找到 listpack 节点追加;XRANGE 时按 rax 范围扫描;消费组(Consumer Group)用另一棵 rax 维护 PEL(Pending Entries List)。一个 Stream 容器内同时存在 3-N 棵 rax:主索引 / 消费组索引 / 每个组内的 PEL / 每个消费者内的 PEL。

  • rax 的核心思想:路径压缩 = 前缀共享 + 节点合并,把 Trie 节点数从 N×K 降到 N
  • raxNode 9 字段(iskey / isnull / iscompr / size:29 / data[])与柔性数组排布
  • 压缩节点(iscompr=1)与普通节点(iscompr=0)的字节布局差异
  • raxInsert 三种情况:压缩节点中部分裂 / 普通节点添加分支 / 标记中间节点
  • raxFind / raxSeek / raxIterator 上下兄弟跳转的 O(K) 实现
  • Stream ID = ms(8B) + seq(8B) 大端编码,字典序 = 时间序的关键设计
  • Stream 容器的多 rax 关系:主 rax(索引 listpack)+ 消费组 rax + PEL rax
  • XADD / XREADGROUP / XACK / XAUTOCLAIM 的 rax 操作链路
  • Consumer Group + PEL 的"至少一次"保证机制与消息认领流程
  • 典型陷阱:不限 MAXLEN 导致 OOM、忘 XACK 导致 PEL 暴涨、消费者死掉消息悬挂

前置知识:第 6 章 ziplist/listpack(理解 Stream 内部 listpack 节点的紧凑追加);第 4 章 skiplist(用于性能对比);Stream 命令基础(XADD / XREAD / XGROUP / XACK)。读者需要熟悉位字段 bit-field 与柔性数组(C99 flexible array member)。

8.1 解决什么问题

Stream 是 Redis 5.0 引入的"日志式"数据结构,设计目标是替代 List + Pub/Sub 在消息队列场景下的不足:既要 List 的有序追加 + 持久化,又要 Pub/Sub 的多消费者并行 + 消费进度跟踪。这一节先讲清楚 Stream 的访问模式,再分析为什么没有现成的数据结构能直接用。

8.2 raxNode 整体布局

rax 的核心创新是 把"分叉"与"无分叉"当作两种节点。一段连续无分叉的字符走压缩节点(iscompr=1),把字节序列原样存进 data[];到了分叉点切换为普通节点(iscompr=0),data[] 前半段是分叉字符,后半段是子节点指针。这一节用字节布局图把两种形态彻底拆开。

8.2.2 raxNode C 结构定义

/* src/rax.h */
typedef struct raxNode {
    uint32_t iskey:1;     /* 该节点是否对应一个完整 key (终止节点) */
    uint32_t isnull:1;    /* value 是否 NULL */
    uint32_t iscompr:1;   /* 0=普通节点(分叉) / 1=压缩节点(无分叉) */
    uint32_t size:29;     /* 普通节点=子数;压缩节点=共享字符串长度 */
    /* C99 柔性数组 (flexible array member):
     * 普通节点 data 排布: [chars][padding][child_ptrs][value_ptr (iskey=1)]
     * 压缩节点 data 排布: [chars][padding][single_child_ptr][value_ptr (iskey=1)]
     */
    unsigned char data[];
} raxNode;

typedef struct rax {
    raxNode *head;        /* 根节点 */
    uint64_t numele;      /* 全部 key 数量 */
    uint64_t numnodes;    /* 全部节点数量 */
} rax;

/* 头部 4 byte 解读:
 *   bits 0    : iskey
 *   bits 1    : isnull
 *   bits 2    : iscompr
 *   bits 3-31 : size  (29 bit, 最大 ~5.4 亿)
 */
3 bit 位字段 + 29 bit size + 柔性数组:头部固定 4 byte,data[] 紧跟其后。Redis 用 8 byte 对齐手动 padding,让指针读写不跨缓存行。这种"一个 struct 对应一段紧凑内存"的写法是 Redis 极致内存优化的标志,后面的 listpack / quicklist 节点都是同款思路。

8.4.3 raxIterator + raxSeek 上下兄弟跳转

/* src/rax.h - 迭代器结构 */
typedef struct raxIterator {
    int flags;
    rax *rt;                    /* 关联的 rax */
    unsigned char *key;         /* 当前 key 缓冲区 */
    void *data;                 /* 当前 value */
    size_t key_len;             /* 当前 key 实际长度 */
    size_t key_max;             /* 缓冲区容量 */
    unsigned char key_static_string[RAX_ITER_STATIC_LEN];  /* 栈上小缓冲 */
    raxNode *node;              /* 当前节点 */
    raxStack stack;             /* 父节点栈 (用于回溯) */
    raxNodeCallback node_cb;
} raxIterator;

/* 典型用法 - 范围扫描 (XRANGE 内核) */
void scan_range(rax *t, unsigned char *start, size_t slen,
                unsigned char *end,   size_t elen) {
    raxIterator iter;
    raxStart(&iter, t);

    /* raxSeek operator:
     *   ">="  : 大于等于
     *   ">"   : 严格大于
     *   "<="  : 小于等于
     *   "<"   : 严格小于
     *   "^"   : 第一个 key (rax 头)
     *   "$"   : 最后一个 key (rax 尾)
     */
    if (!raxSeek(&iter, ">=", start, slen)) {
        raxStop(&iter);
        return;
    }

    /* raxNext: 返回下一个 key,自动维护 iter.key 与 iter.data */
    while (raxNext(&iter)) {
        if (memcmp(iter.key, end, elen) > 0) break;
        process(iter.key, iter.key_len, iter.data);
    }

    raxStop(&iter);
}

/* raxSeek 关键路径 (简化):
 *   1. 从 root 开始走 target key
 *   2. 走到不匹配点时,根据 operator 决定向左/右兄弟跳
 *   3. 兄弟跳转通过 raxStack 回溯 + 父节点的 chars 排序
 *   4. 时间复杂度: O(K),K = key 长度
 */

8.5 stream 整体结构

Stream 不是单纯的"rax + 消息"。一个 Stream 容器要同时回答 4 个问题:

  • 消息存在哪里?
  • 消费组都在哪?
  • 每个组消费到哪了?
  • 哪些消息已分配但未确认?

/* Stream 顶层容器 */
typedef struct stream {
    rax *rax;                    /* 主索引: key=listpack 首 ID, value=listpack ptr */
    uint64_t length;             /* 总消息数 (含未删除) */
    streamID last_id;            /* 最后一条消息 ID */
    streamID first_id;           /* 第一条消息 ID */
    streamID max_deleted_entry_id;  /* 删除游标 */
    uint64_t entries_added;      /* 历史累计 (含已删除) */
    rax *cgroups;                /* 消费组索引: key=组名, value=streamCG */
} stream;

/* 消费组 */
typedef struct streamCG {
    streamID last_id;            /* 下次发哪条 (group 进度) */
    long long entries_read;      /* 已分配消息总数 */
    rax *pel;                    /* 组级 PEL: key=ID, value=streamNACK */
    rax *consumers;              /* 组内消费者: key=name, value=streamConsumer */
} streamCG;

/* 单消费者 */
typedef struct streamConsumer {
    mstime_t seen_time;          /* 最后访问时间 */
    sds name;                    /* 消费者名 */
    rax *pel;                    /* 消费者级 PEL: 共享 streamNACK */
    long long active_time;
} streamConsumer;

/* 未确认消息描述符 (Not Acknowledged) */
typedef struct streamNACK {
    mstime_t delivery_time;      /* 最后投递时间 */
    uint64_t delivery_count;     /* 投递次数 (含 retry) */
    streamConsumer *consumer;    /* 当前持有者 */
} streamNACK;

/* ★ NACK 同时被两棵 rax 引用:
 *   - cg.pel        (组级,所有未 ACK 的)
 *   - consumer.pel  (持有者,worker 视角)
 *   两棵 rax 共享同一个 streamNACK 实例 (引用),不复制
 *   XACK 时同步从两棵 rax 删除

8.6.2 streamAppendItem 关键代码

/* src/t_stream.c - 简化版 */
int streamAppendItem(stream *s, robj **argv, int64_t numfields,
                     streamID *added_id, streamID *use_id) {
    streamID id;

    /* 1. 生成 ID (如果用户指定了 use_id 就用,否则用当前时间) */
    if (use_id) id = *use_id;
    else streamNextID(&s->last_id, &id);

    /* 2. ID 必须严格递增 */
    if (streamCompareID(&id, &s->last_id) <= 0) {
        errno = EDOM;
        return C_ERR;
    }

    /* 3. 找尾 listpack */
    raxIterator ri;
    raxStart(&ri, s->rax);
    raxSeek(&ri, "$", NULL, 0);  /* 走到最右 */

    unsigned char *lp = NULL;
    size_t lp_bytes = 0;
    int lp_count_now = 0;
    streamID master_id;

    if (raxNext(&ri)) {
        lp = ri.data;
        lp_bytes = lpBytes(lp);
        /* 从 master entry 读 count */
        unsigned char *p = lpFirst(lp);
        lp_count_now = lpGetInteger(p);
        /* master ID 编码在 rax key 里 */
        streamDecodeID(ri.key, &master_id);
    }

    /* 4. 决定是追加现有 listpack 还是新建 */
    int new_node = 0;
    if (lp == NULL ||
        lp_bytes >= server.stream_node_max_bytes ||
        lp_count_now >= server.stream_node_max_entries) {
        new_node = 1;
    }

    if (new_node) {
        /* 新建 listpack: 写 master entry + 第一条 msg */
        master_id = id;
        lp = createListpackForStreamMaster(argv, numfields, &id);
        unsigned char rax_key[16];
        streamEncodeID(rax_key, &id);
        raxInsert(s->rax, rax_key, sizeof(rax_key), lp, NULL);
    } else {
        /* 追加 msg 到现有 listpack */
        lpAppendStreamEntry(lp, argv, numfields, &id, &master_id);
        /* 更新 master entry 的 count */
        lpReplaceCount(lp, lp_count_now + 1);
        /* rax 节点的 value 可能因 lpAppend realloc 而改变 */
        raxFind(s->rax, rax_key, 16);  /* 重新挂指针 */
    }

    raxStop(&ri);

    s->last_id = id;
    s->length++;
    if (added_id) *added_id = id;
    return C_OK;
}

8.7 消费组与 PEL 流转

8.7.1 设计动机

Stream 消费组解决"多消费者并行处理 + 消息至少一次保证"的问题:

  • 每个消息只分给组内一个消费者 — 通过 group.last_id 推进保证
  • 消费者处理失败可由其他消费者认领 — 通过 XCLAIM / XAUTOCLAIM
  • 消息处理完后 XACK 删除 PEL — 否则一直挂着,可被认领

XACKO(1)/ID从 group PEL + consumer PEL 删除批量 ACK 按 ID 数增长
XPENDINGsummary O(1);range O(count)PEL 范围扫描扩展形式别无限 COUNT
XAUTOCLAIM与扫描候选数/COUNT 相关PEL 扫描 + 转移COUNT 太大直接拉高延迟
XLENO(1)读 stream->length< 1 μs
XDELO(1)/ID定位 listpack 后标记 deleted不会立刻压缩所有历史洞
XGROUP CREATEO(1) 级别按 group 名写 cgroups rax组数量多时管理成本上升

N = stream 内 listpack 节点数(消息数 / 100);P = PEL 大小;G = 消费组数;K = 返回元素数。

8.8 为什么不选其他算法

这一节是 Antirez 在 src/rax.c 顶部注释里反复强调过的话题。把候选方案的对比再展开一次,让读者明白为什么"路径压缩前缀树"是 Stream 场景的必然选择。

8.8.2 设计动机总结

Redis 选择 rax + listpack 节点 + 多 rax 设计的 5 个理由:
① 路径压缩适配 ID 特征:同毫秒 ID 前 8 字节相同,小序号还会共享更多前缀;
② 单次 rax 查找受 ID 字节长约束:key 固定 16 字节,但命令延迟仍会受返回数量、PEL 大小和内存分配影响;
③ listpack 二级打包:rax 索引粒度是 "listpack 节点"(50-100 条消息),不是单条;
④ 消费组用同一棵 rax:数据结构统一,代码复用度高;
⑤ PEL 同样用 rax:按消息 ID 范围查未确认消息高效,但 XPENDING range 仍要限制 COUNT。

8.9.4 反例 2:多消费者用同一 consumer name

错误:多 client 共享同 consumer name

client A:
  XREADGROUP GROUP g c1 COUNT 10 STREAMS events '>'
client B:
  XREADGROUP GROUP g c1 COUNT 10 STREAMS events '>'
                       ^^ 同一个 consumer name

# 现象:
# - 两个 client 都拿到同样的 PEL
# - 消息被重复处理 → 业务逻辑错乱
# - worker-1.pel 显示 100 条,但实际 client A
#   只处理了 50 条,client B 也处理了相同 50 条
# - XAUTOCLAIM 无法判断真正的责任人

# 后果:
# - 订单重复扣款
# - 计数器多次累加
# - 数据不一致

修复:每 client 独立 consumer name

client A (host: app-1, pid: 1234):
  consumer_name = "app-1.1234"
  XREADGROUP GROUP g app-1.1234 COUNT 10
             STREAMS events '>'

client B (host: app-2, pid: 5678):
  consumer_name = "app-2.5678"
  XREADGROUP GROUP g app-2.5678 COUNT 10
             STREAMS events '>'

# 收益:
# - 每 client 独立 PEL
# - 消息归属清晰
# - worker 死掉后 XAUTOCLAIM 转移
#   到其他活跃 consumer
# - XINFO CONSUMERS events g 看每个
#   consumer 的 idle / pending

8.9.5 反例 3:用 Stream 当大型消息队列

Stream 不是 Kafka:Redis Stream 适合单实例 < 1M 消息的场景。超过这个量级:
① 单 key 几 GB 影响 RDB / replication
② consumer group 数 > 100 时 PEL 总占用爆炸
③ 没有分区 / 副本机制,横向扩展困难
→ 10 GB+ 流量请直接走 Kafka / RocketMQ / Pulsar。Redis Stream 的甜点场景是 "实时事件 + 短保留 + 业务快速分发"。

8.10.2 raxAddChild 关键代码

/* src/rax.c - 给普通节点添加一个 child (情况 B 核心) */
raxNode *raxAddChild(raxNode *n, unsigned char c, raxNode **childptr,
                     raxNode ***parentlink) {
    /* n 必须是普通节点 */
    assert(n->iscompr == 0);

    /* 1. 在 sorted chars 中找插入位置 (二分) */
    int pos;
    unsigned char *p = n->data;
    for (pos = 0; pos < n->size; pos++) {
        if (p[pos] >= c) break;
    }
    /* pos 是新字符插入位置 */

    /* 2. 计算扩容后大小 */
    size_t curlen = raxNodeCurrentLength(n);
    size_t newlen = curlen
                  + 1                  /* 新字符 */
                  + sizeof(raxNode *)  /* 新 child 指针 */
                  + raxPaddingSize(n->size + 1) - raxPaddingSize(n->size);

    /* 3. realloc 节点本身 */
    raxNode *newn = rax_realloc(n, newlen);
    if (newn == NULL) return NULL;
    n = newn;

    /* 4. 在 chars 数组中给新字符腾位 (移动后续 chars + ptrs) */
    unsigned char *src_chars = n->data + pos;
    unsigned char *dst_chars = src_chars + 1;
    memmove(dst_chars, src_chars, n->size - pos);

    /* 5. 移动 child 指针数组 (考虑 padding 变化) */
    raxNode **child_array = (raxNode **)(n->data + n->size + 1
                                         + raxPaddingSize(n->size + 1));
    /* (省略指针数组重排细节) */

    /* 6. 写入新字符 + 新 child 指针 */
    n->data[pos] = c;
    raxNode *child = raxNewNode(0, 0);  /* 创建新空压缩节点 */
    child_array[pos] = child;

    /* 7. 更新 size */
    n->size++;

    /* 8. 通过 parentlink 通知调用方:节点地址可能改变了 */
    if (parentlink) **parentlink = n;
    *childptr = child;
    return n;
}

/* ★ 关键不变量:
 *   1. data 中字符必须 sorted (二分查找前提)
 *   2. realloc 后节点地址可能改变,父节点指针必须同步更新
 *   3. padding 字节数因 size 改变而改变,要重算 child_array 偏移
 */

8.10.3 设计取舍

图 8-10-1 · Stream 数据结构选型决策树

取舍 1:rax vs B+ 树

B+ 树是磁盘存储的标准,但内存场景下:

  • B+ 树节点固定大小,有大量未使用空间
  • 节点内是数组比较,常数大
  • 不能利用 key 的公共前缀

rax 在内存场景下:

  • 节点大小动态(size bit-field 灵活),无空间浪费
  • 路径压缩复用前缀,Stream ID 节省 30-100 倍
  • 复杂度只与 key 长度相关,与 N 无关

取舍 2:每 listpack 一个 rax key vs 每消息一个 rax key

如果每消息一个 rax key:

  • rax 节点数 = 消息数,百万消息 = 百万 rax 节点
  • 每节点 ~40 B 元数据,内存 40 MB 仅元数据
  • 每次插入都要走 raxInsert,可能触发 split

每 listpack 一个 rax key:

  • rax 节点数 = 消息数 / ~80(默认每 listpack 80 条均值)
  • 内存元数据降为 0.5 MB
  • listpack 内紧凑追加,大部分 XADD 是 lpAppend(O(1))

取舍 3:多棵 rax(主索引 + 消费组 + PEL)

代码统一性优先于代码复用:每个角色单独一棵 rax,语义清晰。代价是 rax 实现需要支持多种 value 类型(主 rax 的 value 是 listpack 指针,PEL 的 value 是 streamNACK 指针),但 value 在 raxNode 里就是 void*,不影响实现。

8.12 真实用例

用例 1:实时事件总线(电商订单系统)
某电商订单服务用 Stream 做事件总线:订单创建 → XADD orders.created * order_id ... amount ...;库存服务、积分服务、风控服务都是 consumer group。设置:MAXLEN ~ 1M(保留约 1 小时),每 group 独立 PEL,worker 处理后必 XACK,定时 XAUTOCLAIM 30s 回收死消费者。实际运行:日均 2 亿事件,Stream 长度稳定 ~1M,内存占 80 MB,XADD P99 20 μs,XREADGROUP P99 50 μs。故障演练:杀掉一个库存 worker,30s 内 XAUTOCLAIM 转给其他 worker,业务无感知。
用例 2:CDC 数据同步(MySQL → ES)
把 binlog 转成 Stream 事件 (XADD cdc.users * op insert id 123 name alice),Elasticsearch 同步 worker 用 consumer group 消费。关键设计:用 XADD * op ... seq <binlog_pos> 加上 binlog 位置作字段(不是 ID),让重启时按 XRANGE last_acked_pos + 续传;PEL 长度作为同步延迟指标 (XPENDING cdc.users sync_g)。陷阱:早期版本用了 XADD MAXLEN 100000(精确),写入 P99 抖到 200 μs;改成 MAXLEN ~ 100000 后稳定在 30 μs。
用例 3:轻量任务队列(异步发邮件)
Web 应用把"发邮件"任务塞进 Stream:XADD email_queue * to user@x.com tpl welcome;邮件 worker 用 consumer group 拉取并发送。设计:每 worker 用 hostname+pid 做 consumer name(worker.app-1.1234),处理完 XACK,失败不 ACK,等下次扫到 idle > 5min 时由其他 worker 接手。规模:5 worker,日均 50 万邮件,Stream 长度稳定 ~3K,PEL 平均 < 50。对比简单 Redis 队列:Stream 的 PEL 可以显式追踪未确认任务,提供 "至少一次" 语义;若用 Celery 等框架,仍需按其 broker/ack/visibility timeout 配置单独评估。
用例 4:不要这么用 — 大文件分发
反例:某团队用 XADD logs * payload <100KB JSON> 把每条访问日志(含完整 request body)塞进 Stream。结果:listpack 节点很快被撑大(单 entry > 4KB),实际每个 listpack 只能放 1-2 条消息,rax 节点数 = 消息数,内存爆炸 + XADD 因频繁 raxInsert 而降速到 200 μs。正解:大 payload 单独存 (S3 / 文件系统),Stream 只放引用 (URL + metadata)。
