第 18 章 · SQL 生成与执行计划

JPQL → SQM → SQL AST → Dialect → JDBC · IN padding · EXPLAIN · 5 个性能场景

应用代码描述的是 JPQL、Criteria 或 Repository 方法。数据库收到的是带 ? 占位符的方言 SQL。两者之间存在一条多阶段翻译管线:HQL 解析、SQM 构建、SQM-to-SQL-AST 转换、Dialect 渲染、JDBC 绑定与执行。

本章按管线顺序拆解 Hibernate 6 的实现,给出每一阶段的可观察输入输出、源码入口、典型陷阱与修复对照。重点放在三个层面:① SQM 与 SQL AST 的语义边界 ② Dialect 对分页、锁、函数、布尔、引号的渲染分叉 ③ JDBC 绑定与执行计划的性能权衡。

  • SQL 生成 5 阶段管线:HQL Parser → SQM → SQL AST → Dialect → JDBC bind,每阶段的输入输出
  • SQM(Semantic Query Model)的实体级语义树 + QueryInterpretationCache 缓存机制
  • SQL AST 的关系级表现:TableGroup / ColumnReference / Predicate
  • 4 大数据库方言差异:分页 / 锁 / 字符串拼接 / 自增 / 布尔 / 引号
  • JDBC bind 日志配置(jdbc.bind=TRACE)+ datasource-proxy 显示真实参数
  • IN padding 让 SQL 形状从 N 收敛到 log2(N),稳定 plan cache
  • 5 个生产性能场景反例 + 修复 + 量化:N+1 / fetch+分页退化 / 外键无索引 / cascade ALL / OSIV
  • SQL 快照测试 + SQL 计数测试 + bind 数量测试 + 方言差异测试
  • 10 大陷阱(IN 撑爆 / fetch+分页 / show_sql / dialect 错检 / 枚举映射 / N+1 / 计划缓存 / 隐式 join / Inspector 改写 SQL)
  • 核心源码 9 大入口:StandardHqlTranslator / SqmToSqlAstConverter / SqlAstTranslator / JdbcSelectExecutor

18.1 公共示例 DDL

CREATE TABLE customers (
    id          BIGSERIAL    NOT NULL,
    email       VARCHAR(180) NOT NULL,
    name        VARCHAR(120) NOT NULL,
    region      VARCHAR(40)  NOT NULL,
    created_at  TIMESTAMPTZ  NOT NULL DEFAULT now(),
    CONSTRAINT pk_customers PRIMARY KEY (id),
    CONSTRAINT uk_customers_email UNIQUE (email)
);
CREATE INDEX idx_customers_region ON customers (region);

CREATE TABLE orders (
    id            BIGSERIAL      NOT NULL,
    customer_id   BIGINT         NOT NULL,
    status        VARCHAR(20)    NOT NULL,
    total_amount  NUMERIC(14,2)  NOT NULL,
    created_at    TIMESTAMPTZ    NOT NULL DEFAULT now(),
    version       BIGINT         NOT NULL DEFAULT 0,
    CONSTRAINT pk_orders PRIMARY KEY (id),
    CONSTRAINT fk_orders_customer FOREIGN KEY (customer_id)
        REFERENCES customers (id) ON DELETE RESTRICT,
    CONSTRAINT ck_orders_status CHECK (status IN ('NEW','PAID','SHIPPED','CANCELLED'))
);
CREATE INDEX idx_orders_status_created ON orders (status, created_at DESC);
CREATE INDEX idx_orders_customer ON orders (customer_id);

CREATE TABLE order_items (
    id          BIGSERIAL      NOT NULL,
    order_id    BIGINT         NOT NULL,
    product_id  BIGINT         NOT NULL,
    quantity    INTEGER        NOT NULL,
    unit_price  NUMERIC(14,2)  NOT NULL,
    CONSTRAINT pk_order_items PRIMARY KEY (id),
    CONSTRAINT fk_items_order FOREIGN KEY (order_id) REFERENCES orders (id) ON DELETE CASCADE,
    CONSTRAINT fk_items_product FOREIGN KEY (product_id) REFERENCES products (id) ON DELETE RESTRICT,
    CONSTRAINT ck_items_qty CHECK (quantity > 0)
);
CREATE INDEX idx_items_order ON order_items (order_id);
CREATE INDEX idx_items_product ON order_items (product_id);

idx_orders_status_created 同时覆盖 status 等值与 created_at 排序,是后续性能场景中"分页 + 状态过滤"的核心。

SqmSelectStatement
├── SqmQuerySpec
│   ├── SqmFromClause
│   │   └── SqmRoot(Order as o)
│   │       └── SqmAttributeJoin(o.customer as c, fetch=true)
│   ├── SqmWhereClause
│   │   ├── SqmComparisonPredicate(o.status = ?status)
│   │   └── SqmComparisonPredicate(c.region = ?region)
│   └── SqmOrderByClause

└── limit = 20

SQM 仍是实体世界。它知道 Order.customer.email,但不知道物理列名是 customer_id 还是 email_address。物理表/列映射要等到 SQM 转 SQL AST 时才会发生。

18.8 监控与回归护栏

@Test
void shouldExecuteOnlyOneSqlForJoinFetch() {
    long before = stats.getPrepareStatementCount();

    em.createQuery("""
        select o from Order o
        join fetch o.customer
        where o.status = :status
        """, Order.class)
        .setParameter("status", OrderStatus.PAID)
        .setMaxResults(20)
        .getResultList();

    long after = stats.getPrepareStatementCount();
    assertThat(after - before).isEqualTo(1L);
}

@Test
void inQueryWithPaddingShouldBindPowerOfTwoParameters() {
    sqlCapture.clear();

    repository.findAllByIdIn(List.of(10L, 11L, 12L));

    SqlEvent event = sqlCapture.single();
    assertThat(event.sql()).contains("in (?, ?, ?, ?)");
    assertThat(event.binds()).hasSize(4);
    assertThat(event.binds().get(3).value()).isEqualTo(12L);
}

@ParameterizedTest
@ValueSource(strings = {"postgresql", "mysql", "sqlserver"})
void pagingSqlShouldMatchDialect(String database) {
    SqlEvent event = runWithDatabase(database, () ->
        repository.findWindow(PageRequest.of(2, 10, Sort.by("id")))
    );

    switch (database) {
        case "postgresql", "sqlserver" ->
            assertThat(event.normalizedSql()).contains("offset ? rows fetch");
        case "mysql" ->
            assertThat(event.normalizedSql()).contains("limit ?, ?");
    }
}
