第 9 章 · Criteria API 与类型安全查询

CriteriaBuilder · CriteriaQuery · Root · Path · Predicate · Selection · Metamodel · Customer_.field · Subquery · Fetch · construct · Data/Count 拆分 · SQM 共享后端

JPQL 用字符串表达查询有两个先天缺陷:一是字段重命名只能靠 IDE 字符串替换(漏改 = 运行期失败);二是动态条件只能靠字符串拼接(危险 + 缓存淹没)。Criteria API 是 JPA 给这两个问题准备的官方答案——把查询从字符串变成对象树:Root 表示 FROM 根实体、Path 表示属性导航、Predicate 表示条件节点、Selection 表示返回形状,所有节点由 CriteriaBuilder 工厂创建。配合编译期生成的元模型 Customer_(由 hibernate-jpamodelgen 注解处理器生成),你写 customer.get(Customer_.email) 而不是 customer.get("email")——字段重命名时编译失败,根本不会运行到生产。这对动态条件场景(搜索表单、可选过滤、复合查询)是质的提升:用户输入哪些字段就 predicates.add(...) 哪些 Predicate,最后 cb.and(predicates.toArray(...)) 一次组合,生成的 SQL 仍然是参数化的、命中查询缓存的、跨数据库一致的。

但 Criteria 不是免费的——它的代码体积通常是等价 JPQL 的 2-3 倍,可读性下降,初次接触有陡峭的学习曲线。所以选型规则简单:固定查询用 JPQL,动态查询用 Criteria。本章按照"对象分工 → Provider 内部 → 实战场景"三层展开。第 1-2 节讲清楚四大对象(CriteriaBuilder/CriteriaQuery/Root/Path)的职责边界,以及 Metamodel 是怎样保证编译期类型安全的;第 3-7 节分别覆盖 Predicate 组合、Join/Fetch、Subquery、DTO 投影、分页查询(特别强调 Data Query 和 Count Query 必须分开 Root,不能复用)五个生产高频场景;第 8 节进入 Hibernate 6 的 SQM 翻译链路,展示 Criteria 与 JPQL 共享后端的设计;最后两节把 8 个生产陷阱和一份 Repository 评审清单整理出来,可以直接放到团队 PR 模板里。

  • Criteria 四大对象分工:CriteriaBuilder(节点工厂)/ CriteriaQuery(查询树)/ Root(FROM 根)/ Path(属性导航)
  • Metamodel(Customer_.email)如何在编译期校验字段名 — 字段重命名时编译失败而非运行期失败
  • 启用 hibernate-jpamodelgen 注解处理器:Maven / Gradle 配置
  • 动态 Predicate 组合模式:List<Predicate> + Optional.ifPresent + cb.and(...)
  • 嵌套 OR 组合:cb.and(active, cb.or(nameHit, emailHit)) 与 SQL 括号结构对应
  • join()(过滤/聚合)vs fetch()(加载关联)的区别 — 与第 8 章 JPQL 的 JOIN/FETCH 语义一致
  • Subquery 关联子查询:existsOrder.from(Order.class) + cb.equal(order.get(...), customer)
  • 聚合子查询比较:子查询 cb.avg(...) 与外层 cb.greaterThan(...)
  • DTO 构造投影 cb.construct(CustomerOrderListItem.class, ...) 与 group by 列匹配规则
  • Data Query 与 Count Query 必须用独立 Root — count 复用 fetch 会污染结果,count 复用 order 浪费
  • 分页稳定排序:必须末尾追加主键 ID tie-breaker,否则跨页抖动

  • "陷阱 / 现象 / 修复"8 项清单 + Repository 评审 8 项 Checklist

前置知识:第 6 章关联映射(理解 @OneToMany 与 Criteria 的 customer.join(Customer_.orders) 对应)、第 8 章 JPQL(本章很多 SQL 行为与 JPQL 一致,特别是 JOIN/FETCH 语义、集合 fetch + 分页冲突、auto flush 时机)、第 5 章 Spring Data JPA 基础(本章 9.7 节分页查询返回 PageImpl 涉及 Spring Data 的 Page 抽象)。本章是把第 8 章的"实体语言"从字符串切换到对象树,核心 SQL 行为不变,变化的是 Java 端的表达方式。后续第 10 章 Native SQL 是当 Criteria 也表达不了时(CTE、窗口函数、厂商 hint)的最后逃生通道。

Criteria 不是 JPQL 的"高级版",是动态查询的专用工具 — 我见过团队把所有查询都改写成 Criteria 来"防字符串风险",代码膨胀 3 倍,可读性灾难,review 时间翻倍。这是过度设计。Criteria 的真正价值在"条件结构本身随用户输入变化":搜索表单(用户填了 name 就加 like 条件,填了 status 就加 equal 条件)、动态过滤(根据角色显示不同范围的数据)、组合复用(把"加权限过滤"和"加软删除过滤"做成可注入的 Predicate 工厂)——这些场景拼字符串非常痛苦,Criteria 大幅减负。固定查询(每次执行 SQL 形态一样、只参数变)仍然应该用 JPQL:可读性高、代码短、Metamodel 改动 IDE 也能跟住(IDE 会重命名 JPQL 字符串里的属性名)。两个接口共存,各司其职。

9.1 三大对象分工

Criteria 看起来对象多,但本质就 4 个:CriteriaBuilder(工厂)、CriteriaQuery(查询树根)、Root(FROM 根)、Path(属性导航)。Predicate / Selection / Subquery 都是 CriteriaBuilder 的产物。

对象关系

图 9-1-1 · 四大对象关系:Builder 创建节点,Query 持有树,Root 指向 FROM

9.1.1 最小可执行查询

CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Customer> query = cb.createQuery(Customer.class);
Root<Customer> customer = query.from(Customer.class);

query.select(customer)
     .where(cb.equal(customer.get(Customer_.status), CustomerStatus.ACTIVE))
     .orderBy(cb.asc(customer.get(Customer_.id)));

List<Customer> result = em.createQuery(query).getResultList();
select c1_0.id, c1_0.name, c1_0.status, c1_0.age
from customers c1_0
where c1_0.status = ?
order by c1_0.id asc;

对比等价 JPQL:

em.createQuery(
    "select c from Customer c where c.status = :status order by c.id",
    Customer.class)
    .setParameter("status", CustomerStatus.ACTIVE)
    .getResultList();

JPQL 三行,Criteria 七行——这就是"代码长度差"的具体观感。但请注意 Criteria 版的两个优势:(1) Customer_.status 编译期校验;(2) 后续如果要根据某个 boolean flag 加额外条件,Criteria 只需要 if (flag) where = cb.and(where, ...);,JPQL 要拼字符串。

9.1.2 Metamodel 编译期校验

Customer_.status 是注解处理器 hibernate-jpamodelgen 在编译期生成的元模型。Java 字段重命名时,元模型同步变化,Criteria 代码会编译失败而非运行期失败。

Path<Integer> age = customer.get(Customer_.age);

query.select(customer)
     .where(cb.between(age, 18, 35))
     .orderBy(cb.desc(customer.get(Customer_.createdAt)));

这里 age 的类型是 Path<Integer>——元模型保留了字段的具体类型。所以 cb.between(age, 18, 35) 编译期就检查参数类型,你不能传 String 给 between。如果用字符串字段名 customer.get("age"),得到的是 Path<Object>,任何类型都能传,运行期才报错。

启用 Metamodel 生成(Maven)

<plugin>
    <groupId>org.apache.maven.plugins</groupId>
    <artifactId>maven-compiler-plugin</artifactId>
    <configuration>
        <annotationProcessorPaths>
            <path>
                <groupId>org.hibernate.orm</groupId>
                <artifactId>hibernate-jpamodelgen</artifactId>
            </path>
        </annotationProcessorPaths>
    </configuration>
</plugin>

注解处理器在编译时扫描 @Entity 类,为每个实体生成 {Entity}_ 元模型类(如 Customer → Customer_)。生成的类放在 target/generated-sources/annotations/ 目录,IDE 自动识别为源根。

9.3 Predicate 组合

Predicate 组合是 Criteria 的核心价值——动态条件按需累加,空条件不出现在 SQL 里,最后 cb.and(...) 一次性组合。

动态条件组合流程

图 9-3-1 · 搜索表单按字段填充情况累加 Predicate

9.3.1 可选条件组合

public List<Customer> searchCustomers(CustomerSearch form) {
    CriteriaBuilder cb = em.getCriteriaBuilder();
    CriteriaQuery<Customer> query = cb.createQuery(Customer.class);
    Root<Customer> customer = query.from(Customer.class);

    List<Predicate> predicates = new ArrayList<>();

    Optional.ofNullable(form.name())
        .filter(s -> s.length() > 0)
        .map(s -> "%" + escapeLike(s.toLowerCase(Locale.ROOT)) + "%")
        .map(p -> cb.like(cb.lower(customer.get(Customer_.name)), p, '\\'))
        .ifPresent(predicates::add);

    Optional.ofNullable(form.minAge())
        .map(min -> cb.greaterThanOrEqualTo(customer.get(Customer_.age), min))
        .ifPresent(predicates::add);

    Optional.ofNullable(form.maxAge())
        .map(max -> cb.lessThanOrEqualTo(customer.get(Customer_.age), max))
        .ifPresent(predicates::add);

    query.select(customer);
    if (predicates.size() > 0) {
        query.where(cb.and(predicates.toArray(Predicate[]::new)));
    }

    return em.createQuery(query).getResultList();
}

private static String escapeLike(String value) {
    return value
        .replace("\\", "\\\\")
        .replace("%", "\\%")
        .replace("_", "\\_");
}
select c1_0.id, c1_0.name, c1_0.status, c1_0.age
from customers c1_0
where lower(c1_0.name) like ? escape '\'
  and c1_0.age >= ?
  and c1_0.age <= ?;

注意 escapeLike 这个细节:用户输入可能包含 % 或 _ 这两个 LIKE 通配符。如果用户搜索 "50% off",直接拼 %50% off% 会让 % 在 SQL 里继续作通配符,匹配出"50任意字符 off"——和用户期望不符。escape '\' 配合 \\% 转义让 % 当字面量。这是动态查询里常被忽略的安全细节。

9.3.2 嵌套 OR 组合

Predicate active = cb.equal(customer.get(Customer_.status), CustomerStatus.ACTIVE);
Predicate nameHit = cb.like(cb.lower(customer.get(Customer_.name)), "%ops%");
Predicate emailHit = cb.like(cb.lower(customer.get(Customer_.email)), "%ops%");

query.select(customer)
     .where(cb.and(active, cb.or(nameHit, emailHit)));
select c1_0.id, c1_0.name, c1_0.email_address, c1_0.status
from customers c1_0
where c1_0.status = ?
  and (lower(c1_0.name) like ?
       or lower(c1_0.email_address) like ?);

SQL 的 and (or) 括号结构由 Criteria 树的嵌套精确反映。这种结构如果用 JPQL 字符串拼,出错率非常高(漏一个括号,意思全变)。Criteria 的对象树天然保证嵌套正确。

9.4 Join 与 Fetch Join

Criteria 区分 join() 和 fetch()——和第 8 章 JPQL 的 JOIN/FETCH 语义完全对应。

Join vs Fetch

图 9-4-1 · join 用于过滤/聚合,fetch 用于加载关联

9.4.1 普通 JOIN 用于聚合

public record CustomerOrderTotal(String name, BigDecimal total) {}

CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<CustomerOrderTotal> query = cb.createQuery(CustomerOrderTotal.class);
Root<Customer> customer = query.from(Customer.class);
Join<Customer, Order> order = customer.join(Customer_.orders, JoinType.INNER);

Expression<BigDecimal> total = cb.sum(order.get(Order_.amount));

query.select(cb.construct(
        CustomerOrderTotal.class,
        customer.get(Customer_.name),
        total))
     .groupBy(customer.get(Customer_.id), customer.get(Customer_.name))
     .having(cb.greaterThan(total, new BigDecimal("1000.00")))
     .orderBy(cb.desc(total));

List<CustomerOrderTotal> rows = em.createQuery(query).getResultList();
select c1_0.name, sum(o1_0.amount)
from customers c1_0
inner join orders o1_0 on o1_0.customer_id = c1_0.id
group by c1_0.id, c1_0.name
having sum(o1_0.amount) > ?
order by sum(o1_0.amount) desc;

CriteriaQuery<Customer> query = cb.createQuery(Customer.class);

9.5 Subquery

Criteria 的 Subquery 由外层 query.subquery(Type.class) 创建,有自己的 Root 但与外层 Root 处于同一作用域——可以引用外层 Root 实现关联子查询。

子查询作用域

图 9-5-1 · 关联子查询:外层 Root 与子查询 Root 共享作用域

9.5.1 EXISTS 关联子查询

CriteriaQuery<Customer> query = cb.createQuery(Customer.class);
Root<Customer> customer = query.from(Customer.class);

Subquery<Long> existsOrder = query.subquery(Long.class);
Root<Order> order = existsOrder.from(Order.class);

existsOrder.select(order.get(Order_.id))
    .where(
        cb.equal(order.get(Order_.customer), customer),
        cb.equal(order.get(Order_.status), OrderStatus.PAID));

query.select(customer).where(cb.exists(existsOrder));
select c1_0.id, c1_0.name, c1_0.status
from customers c1_0
where exists (
    select o1_0.id from orders o1_0
    where o1_0.customer_id = c1_0.id
      and o1_0.status = ?
);

注意 cb.equal(order.get(Order_.customer), customer)——子查询的 order.customer 直接与外层的 customer Root 比较,Hibernate 自动生成关联条件 o1_0.customer_id = c1_0.id。这是 Criteria 子查询最优雅的部分。

9.5.2 聚合子查询比较

Subquery<Double> avgAge = query.subquery(Double.class);
Root<Customer> colleague = avgAge.from(Customer.class);

avgAge.select(cb.avg(colleague.get(Customer_.age)))
      .where(cb.equal(
          colleague.get(Customer_.fulfillmentCenter),
          customer.get(Customer_.fulfillmentCenter)));

query.select(customer)
     .where(cb.greaterThan(customer.get(Customer_.age).as(Double.class), avgAge));
select c1_0.id, c1_0.name, c1_0.age
from customers c1_0
where cast(c1_0.age as double) > (
    select avg(c2_0.age) from customers c2_0
    where c2_0.fulfillment_center_id = c1_0.fulfillment_center_id
);

这是"找出年龄高于本部门均值的员工"的经典查询。子查询返回 Double(因为 AVG 返回 Double),外层用 customer.get(Customer_.age).as(Double.class) 显式转型再比较,避免 Integer 与 Double 比较的方言差异。

9.6 投影与 DTO 构造

Criteria 的投影方式比 JPQL 更丰富:除了 construct(DTO, ...),还有 tuple()(命名访问)和 array()(原始 Object[])。

投影方式对比

9.6.1 DTO 构造投影

public record CustomerOrderListItem(
    Long id,
    String name,
    String fulfillmentCenterName,
    Long orderCount) {}

CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<CustomerOrderListItem> query = cb.createQuery(CustomerOrderListItem.class);
Root<Customer> customer = query.from(Customer.class);
Join<Customer, FulfillmentCenter> center = customer.join(Customer_.fulfillmentCenter, JoinType.LEFT);
Join<Customer, Order> order = customer.join(Customer_.orders, JoinType.LEFT);

query.select(cb.construct(
        CustomerOrderListItem.class,
        customer.get(Customer_.id),
        customer.get(Customer_.name),
        center.get(FulfillmentCenter_.name),
        cb.count(order.get(Order_.id))))
     .groupBy(
        customer.get(Customer_.id),
        customer.get(Customer_.name),
        center.get(FulfillmentCenter_.name))
     .orderBy(cb.asc(customer.get(Customer_.id)));
select c1_0.id, c1_0.name, d1_0.name, count(o1_0.id)
from customers c1_0
left join fulfillment_centers d1_0 on d1_0.id = c1_0.fulfillment_center_id
left join orders o1_0 on o1_0.customer_id = c1_0.id
group by c1_0.id, c1_0.name, d1_0.name
order by c1_0.id asc;

注意 group by 必须包含所有非聚合的 SELECT 列——否则数据库报"列不在 GROUP BY 中"。这是 Criteria 与 JPQL 共有的约束,不是 Criteria 特有。

9.7 分页查询

分页查询是 Criteria 最容易踩坑的场景——必须把 Data Query 和 Count Query 分别构建,各自用独立的 Root,只复用 Predicate 构建逻辑。

9.7.1 Data Query 与 Count Query 拆分

public Page<Customer> findPage(CustomerSearch search, int page, int size) {
    CriteriaBuilder cb = em.getCriteriaBuilder();

    CriteriaQuery<Customer> dataQuery = cb.createQuery(Customer.class);
    Root<Customer> dataRoot = dataQuery.from(Customer.class);
    dataRoot.fetch(Customer_.fulfillmentCenter, JoinType.LEFT);

    List<Predicate> dataPredicates = userPredicates(cb, dataRoot, search);

    dataQuery.select(dataRoot)
        .distinct(true)
        .where(dataPredicates.toArray(Predicate[]::new))
        .orderBy(
            cb.desc(dataRoot.get(Customer_.createdAt)),
            cb.asc(dataRoot.get(Customer_.id)));

    List<Customer> content = em.createQuery(dataQuery)
        .setFirstResult(page * size)
        .setMaxResults(size)
        .getResultList();

    CriteriaQuery<Long> countQuery = cb.createQuery(Long.class);
    Root<Customer> countRoot = countQuery.from(Customer.class);
    List<Predicate> countPredicates = userPredicates(cb, countRoot, search);

    countQuery.select(cb.count(countRoot))
        .where(countPredicates.toArray(Predicate[]::new));

    Long total = em.createQuery(countQuery).getSingleResult();
    return new PageImpl<>(content, PageRequest.of(page, size), total);
}

分页setFirstResult / setMaxResults否

count 查询不能复用 data query 的 root,因为 fetch / order 会污染 count 结果。复用 userPredicates(cb, root, search) 仅复用条件构建逻辑——传入 cb 和 root,返回 Predicate 列表,这种"工厂方法"模式让动态条件可重用。

稳定排序必须在末尾追加主键 ID — 这是分页查询的另一个隐形陷阱。如果你只 orderBy(cb.desc(createdAt)),而 createdAt 在某些行有重复值(批量导入产生大量同时间戳),数据库返回顺序不保证稳定——同一条 SQL 跑两次,返回顺序可能不同。结果:用户翻页时看到第 1 页和第 2 页都有同一条记录,或者某条记录"凭空消失"。修复:orderBy(cb.desc(createdAt), cb.asc(id))——id 必然唯一,保证排序稳定。这条规则适用于所有分页查询(Criteria/JPQL/原生 SQL),不限于 Criteria。

9.8.1 设计取舍

取舍 1:对象树 vs 字符串

JPQL 短文本可读性高,但字段重命名后字符串无法编译期校验。Criteria 通过 Metamodel + 对象树将字段绑定到 Java 类型,代价是代码冗长。这就是为什么 Criteria 不是 JPQL 的替代品而是补充——固定查询用 JPQL,动态查询用 Criteria。

取舍 2:SQM 共享后端

Criteria 与 JPQL 共享 SQM,避免双份解析与翻译实现。SQM 是中间表示,后续 SQL AST 转换、Dialect 渲染、JDBC 执行三方共用。这意味着 Hibernate 6 任何 SQM 层的优化(查询计划缓存、新函数支持、分页优化)Criteria 自动受益。

取舍 3:动态构建延迟执行

em.createQuery(criteria) 之前仅构建语法树,不与数据库交互。getResultList() 触发 SQM → SQL AST → 执行。延迟执行允许动态条件累积——你可以在 createQuery 之前任意修改 CriteriaQuery 的结构,重置 where、添加新 join、改 selection。

取舍 4:可移植 SQL

Criteria 不暴露分页 / 函数 / 锁的具体 SQL 语法,Dialect 在最后阶段渲染。同一 Criteria 在 PostgreSQL / MySQL / Oracle / SQL Server 上生成不同 SQL。这是跨厂商兼容性的力量,但也意味着你无法直接利用某厂商的特殊语法(比如 PostgreSQL 的 DISTINCT ON、MySQL 的 STRAIGHT_JOIN)——这些场景必须用 Native SQL。

@Test
void searchKeepsStableCriteriaContract() {

"空条件策略"是 Criteria 在生产里的隐形地雷 — 搜索表单常常允许"全部不填",此时 Criteria 生成的 SQL 是 select * from customers(没有 where)。在小表上没事,在 1000 万行的客户表上就是几分钟的全表扫描 + OOM。生产 Criteria 必须有空条件策略:要么直接拒绝(返回 400 "请至少提供一个条件"),要么默认加范围限制(where created_at > now() - interval '30 days'),要么强制分页(默认 size 20,无 size 参数也强制 size 20)。我见过的最严重事故就是搜索表单没空条件保护,某天用户全部清空点查询,3 个数据库连接被全表扫挂死,整个产品 5 分钟不可用。这不是 bug,这是缺设计。
