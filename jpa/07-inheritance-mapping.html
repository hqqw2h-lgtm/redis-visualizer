第 7 章 · 继承映射策略

SINGLE_TABLE · JOINED · TABLE_PER_CLASS · @MappedSuperclass · Discriminator · Persister · Proxy · UNION ALL · LEFT JOIN · 决策树 · 守护测试

面向对象的"继承"在关系数据库里没有原生对应物——表是扁平的二维结构,而类层级是树。把"CreditCardPayment extends Payment"放进 schema 必须做出取舍:是用一张大表加 discriminator 列区分子类,还是父表 + 子表用主键 JOIN 起来,还是每个子类一张独立完整表用 UNION ALL 合并?JPA 把这三种取舍固定为 InheritanceType.SINGLE_TABLE / JOINED / TABLE_PER_CLASS,再补一个 @MappedSuperclass 处理"只复用字段不要继承表"的场景。本章把这四种形态挨个拆开:DDL 长什么样、INSERT / SELECT 的 SQL 形态、约束能不能强、ID 策略受什么限制、多态查询代价多大、什么时候选哪一个。

选错策略在系统初期影响有限——子类只有 2 个、记录量 10 万以下时,三种策略的查询延迟差异都在毫秒级,业务感知不到。但策略一旦上线就极难迁移:迁 SINGLE_TABLE → JOINED 需要拆表 + 改外键 + 改所有 JPQL 多态查询的预期 SQL;迁 JOINED → TABLE_PER_CLASS 需要换 ID 生成器(IDENTITY 在 TPC 下被禁);迁 TPC → JOINED 需要把所有子类表的共享列抽到父表里。这些迁移在记录量到亿级、活跃 QPS 上千时几乎无法在线完成,只能停服几个小时跑数据迁移脚本——所以选型这一步必须当场想清楚未来 3-5 年的子类增长、查询模式、字段差异。

  • 三种 @Inheritance 策略的 DDL / INSERT / 多态 SELECT 的真实形态对比,以及它们生成的 SQL 差异(单表扫描 vs LEFT JOIN vs UNION ALL)
  • SINGLE_TABLE 为什么"子类专属列不能 not null"——单表共享一行结构,新插入的另一类型行必然 NULL
  • JOINED 子表的主键同时是父表外键,真正的"is-a"关系在 schema 层强制成立;LEFT JOIN 数随子类深度线性增加
  • TABLE_PER_CLASS 必须共享 SEQUENCE / UUID 而不能 IDENTITY——跨表 ID 唯一性不能由单表自增保证
  • @MappedSuperclass 不是继承策略,只是"字段复用工具"——父类不进运行期实体集合,无法做 JPQL 查询根
  • Discriminator 的三种形态:@DiscriminatorColumn / @DiscriminatorValue / @DiscriminatorFormula,以及"不要用类名做 discriminator"的硬规矩
  • 三种 Persister 的源码入口:SingleTableEntityPersister / JoinedSubclassEntityPersister / UnionSubclassEntityPersister,以及它们各自的 SQL AST 编译路径
  • 父类引用 LAZY 时返回的代理类继承自父类,instanceof CreditCardPayment 判断为 false 的根因与 Hibernate.unproxy() 修复
  • 策略选择决策树:多态查询是否高频 / 子类字段差异多大 / 数据库强约束需求 / 子类未来还会扩展多少
  • 三种策略各自的回归测试套路——Persister 类型断言 / Discriminator 业务码守护 / JOIN 数上限守护

前置知识:第 4 章 @Entity 与基本字段映射;第 5 章 @Id / @GeneratedValue 与 ID 生成器;第 6 章 @ManyToOne / @OneToMany 关联映射的外键模型。本章在这些基础上把"类继承"也压平到关系模型,生成的 SQL 形态会反复用到前几章的 JOIN / 外键 / ID 概念。

7.0 章首速览

三种 @Inheritance 策略 + 一种 @MappedSuperclass 字段复用形态,生成出截然不同的 schema 与查询 SQL。先用一张总览图把四种形态的物理形态摆在一起,再用对照矩阵把"父类是不是实体 / 父表存不存在 / 多态查询的 SQL 模板"这三件事讲清楚。

四种映射形态的物理形态对比

图 7-0 · SINGLE_TABLE / JOINED / TABLE_PER_CLASS / @MappedSuperclass 的 schema 形态

四种形态的对照矩阵

形态父类是实体父表多态查询子类强约束重点观测
SINGLE_TABLE是单表单表扫描否(子类列须 nullable)discriminator 稳定
JOINED是有LEFT JOIN是JOIN 数
TABLE_PER_CLASS通常抽象无UNION ALL是父类排序昂贵
@MappedSuperclass否无不支持子类自定义无父类查询
三种策略的本质差异在"父类是否落表"。SINGLE_TABLE 把父类与所有子类压到一张表里,父类查询是单表 scan;JOINED 给父类一张独立表、给每个子类一张子表,父类查询是 LEFT JOIN N 个子表才能补齐子类专属列;TABLE_PER_CLASS 父类完全没表,所有列复制到每个子类表里,父类查询只能 UNION ALL 所有子类表。三者的查询代价、约束能力、子类增长成本都源自这个"父表存在性"的差异——这是选型时第一个要回答的问题:"我以后会不会以父类为查询根?"。

7.1 业务模型

本章贯穿用一个统一的业务模型——支付域的 Payment 抽象类与两个具体子类 CreditCardPayment / BankTransfer。父类持有所有子类共享的 id 与 amount;两个子类各自持有自己的支付方式专属字段(卡号 / 过期日期、银行名 / 账号)。这种"共享身份 + 各自细节"是继承映射的标准触发场景。

抽象父类 Payment

public abstract class Payment {
    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "payment_seq")
    @SequenceGenerator(name = "payment_seq", sequenceName = "payment_seq", allocationSize = 50)
    private Long id;

    @Column(nullable = false, precision = 19, scale = 2)
    private BigDecimal amount;

    protected Payment() {}
    protected Payment(BigDecimal amount) { this.amount = amount; }
}

具体子类 CreditCardPayment / BankTransfer

public class CreditCardPayment extends Payment {
    @Column(name = "card_number")
    private String cardNumber;

    @Column(name = "exp_date")
    private String expDate;

    public CreditCardPayment(BigDecimal amount, String cardNumber, String expDate) {
        super(amount);
        this.cardNumber = cardNumber;
        this.expDate = expDate;
    }
}

public class BankTransfer extends Payment {
    @Column(name = "bank_name")
    private String bankName;

    @Column(name = "account_no")
    private String accountNo;

    public BankTransfer(BigDecimal amount, String bankName, String accountNo) {
        super(amount);
        this.bankName = bankName;
        this.accountNo = accountNo;
    }
}

测试数据

em.persist(new CreditCardPayment(new BigDecimal("99.00"), "4111111111111111", "2030-12"));
em.persist(new BankTransfer(new BigDecimal("250.00"), "Bank A", "62220001"));

下面三节(7.2 / 7.3 / 7.4)分别给这同一组业务模型套上三种 @Inheritance 策略,看 schema、INSERT、SELECT 的真实差异。

7.2 SINGLE_TABLE

所有子类共享一张表,行级靠 discriminator 列区分类型——这是 JPA 的默认策略(不写 @Inheritance 注解时即为 SINGLE_TABLE)。父类与所有子类的列累加在同一张表里,子类专属列对其他类型的行必然为 NULL。

形态:单表 + 类型列

图 7-2 · SINGLE_TABLE 的物理形态

注解配置

@Entity
@Table(name = "payment")
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorColumn(name = "payment_type", length = 8)
public abstract class Payment { /* ... */ }

@Entity
@DiscriminatorValue("CC")
public class CreditCardPayment extends Payment { /* ... */ }

@Entity
@DiscriminatorValue("BT")
public class BankTransfer extends Payment { /* ... */ }

生成的 DDL

create table payment (
    id            bigint not null,
    payment_type  varchar(8) not null,
    amount        numeric(19, 2) not null,
    card_number   varchar(255),
    exp_date      varchar(255),
    bank_name     varchar(255),
    account_no    varchar(255),
    primary key (id)
);

create index idx_payment_type on payment (payment_type);

持久化后的真实数据形态

| id | payment_type | amount | card_number      | exp_date | bank_name | account_no |
|----|--------------|--------|------------------|----------|-----------|------------|
|  1 | CC           |  99.00 | 4111111111111111 | 2030-12  | NULL      | NULL       |
|  2 | BT           | 250.00 | NULL             | NULL     | Bank A    | 62220001   |

第 1 行是信用卡支付,bank_name / account_no 必须为 NULL;第 2 行是银行转账,card_number / exp_date 必须为 NULL。这就是 SINGLE_TABLE "子类专属列必须可空" 的根因:同一张表既要装 CC 又要装 BT,谁的字段对方就是 NULL。

多态查询 SQL — 单表扫描

select p.id, p.payment_type, p.amount,
       p.card_number, p.exp_date, p.bank_name, p.account_no
from payment p
order by p.id;

子类查询 SQL — 自动加 discriminator filter

select p.id, p.payment_type, p.amount, p.card_number, p.exp_date
from payment p
where p.payment_type = 'CC';

JPQL 写 select c from CreditCardPayment c 时,Hibernate 自动追加 where payment_type = 'CC'——这就是 discriminator 的核心价值。父类查询 select p from Payment p 不带 where,扫整张表。

SINGLE_TABLE 优劣矩阵

维度评价说明
多态查询性能最优单表扫描,无 JOIN 无 UNION
子类查询性能最优discriminator 上有索引时 = 索引扫描
子类字段约束受限子类专属列不能 not null
表列数累积每加一个子类,表多几列
写入性能最优单表 INSERT,无跨表事务
NULL 占用空间有稀疏列在很多 DB 里压缩存储得很好
适用场景子类少 + 列差异小共享字段为主、专属字段少
SINGLE_TABLE 的子类专属列别加 nullable=false — 这是踩坑率最高的"自损一千"配置:你写 @Column(name="card_number", nullable=false),DDL 就生成 card_number varchar(255) not null。然后插入 BankTransfer 时(card_number 必然为 null)直接抛 ConstraintViolationException——Bank Transfer 的字段从来就跟 card_number 没关系。要"数据库强约束"必须改 JOINED;在 SINGLE_TABLE 下只能用 Bean Validation 在应用层强约束。

7.3 JOINED

JOINED 是把"is-a"关系真正映射到 schema 上的策略——父表存共享列,每个子类一张独立子表,子表的主键同时是父表的外键(id PK + id FK references payment(id))。子类专属列在子表内可以放心地 not null,因为子表里所有行都是这个具体子类。

形态:父表 + N 个子表 + 共享主键

图 7-3 · JOINED 的物理形态

注解配置

@Entity
@Table(name = "payment")
@Inheritance(strategy = InheritanceType.JOINED)
@DiscriminatorColumn(name = "payment_type", length = 8)
public abstract class Payment { /* ... */ }

@Entity
@Table(name = "credit_card_payment")
@DiscriminatorValue("CC")
public class CreditCardPayment extends Payment {
    @Column(name = "card_number", nullable = false)
    private String cardNumber;

    @Column(name = "exp_date", nullable = false)
    private String expDate;
}

@Entity
@Table(name = "bank_transfer")
@DiscriminatorValue("BT")
public class BankTransfer extends Payment {
    @Column(name = "bank_name", nullable = false)
    private String bankName;

    @Column(name = "account_no", nullable = false)
    private String accountNo;
}

生成的 DDL — 三张表 + 两个外键

create table payment (
    id            bigint not null,
    payment_type  varchar(8) not null,
    amount        numeric(19, 2) not null,
    primary key (id)
);

create table credit_card_payment (
    id           bigint not null,
    card_number  varchar(255) not null,
    exp_date     varchar(255) not null,
    primary key (id),
    foreign key (id) references payment (id)
);

create table bank_transfer (
    id          bigint not null,
    bank_name   varchar(255) not null,
    account_no  varchar(255) not null,
    primary key (id),
    foreign key (id) references payment (id)
);

INSERT — 双表写入

insert into payment (amount, payment_type, id)
values (99.00, 'CC', 1);

insert into credit_card_payment (card_number, exp_date, id)
values ('4111111111111111', '2030-12', 1);

每持久化一条 CreditCardPayment,Hibernate 触发两条 INSERT——先写父表 payment 拿到 id,再写子表 credit_card_payment 用同一个 id。两条 INSERT 在同一个事务里,任一失败都回滚。

多态查询 SQL — LEFT JOIN 所有子表

select p.id, p.payment_type, p.amount,
       cc.card_number, cc.exp_date,
       bt.bank_name, bt.account_no
from payment p
left join credit_card_payment cc on p.id = cc.id
left join bank_transfer bt on p.id = bt.id
order by p.id;

查父类 = LEFT JOIN 所有子表(每个子类一个 LEFT JOIN);查具体子类 = INNER JOIN 父表与该子表(只取这一种)。JOIN 数 = 子类数,所以 JOINED 在子类多、子类深时代价线性增长。

JOINED 优劣矩阵

维度评价说明
多态查询中等LEFT JOIN 子表数 = 子类数
子类字段约束强子表内可放心 not null
schema 规范度高3NF 友好,去 NULL
表数量多1 父 + N 子
写入性能中等双表 INSERT;批量略慢于单表
子类深度限制推荐 ≤ 2 层三层继承 = 三层 JOIN
适用场景需 DB 强约束 / 子类字段差异大财务、法规相关数据
电商订单的 Payment 选 JOINED 而不是 SINGLE_TABLE 的真实原因 — 我们在订单系统里把 Payment 改成 JOINED,核心理由不是"性能"也不是"美学",而是合规审计。审计员要求"信用卡支付的卡号必须 not null,银行转账的银行名必须 not null"——SINGLE_TABLE 在 schema 层做不到(card_number 可以为 NULL,只能在 application 层校验)。改 JOINED 后,DBA 直接看 credit_card_payment 表 DDL 就能确认每张卡都有卡号,Bank Transfer 表里每行都有银行名——这种"审计员不需要看代码就能确认数据约束"的能力,比单表扫描省的几毫秒重要 100 倍。代价是多态查询多两个 LEFT JOIN,在 P99 200ms 的预算里完全可以承担。

7.4 TABLE_PER_CLASS

每个子类一张完整独立表——把父类的所有列复制进去,父类本身没有物理表。子类查询是单表,父类查询是 UNION ALL 所有子类表。这是三种策略里"父类抽象度最低"的一种,也是最不常用的一种。

形态:每子类一张完整表 + UNION ALL

图 7-4 · TABLE_PER_CLASS 的物理形态

注解配置

@Entity
@Inheritance(strategy = InheritanceType.TABLE_PER_CLASS)
public abstract class Payment {
    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "payment_seq")
    @SequenceGenerator(name = "payment_seq", sequenceName = "payment_seq", allocationSize = 50)
    private Long id;

    @Column(nullable = false, precision = 19, scale = 2)
    private BigDecimal amount;
}

@Entity
@Table(name = "credit_card_payment")
public class CreditCardPayment extends Payment { /* ... */ }

@Entity
@Table(name = "bank_transfer")
public class BankTransfer extends Payment { /* ... */ }

多态查询 SQL — UNION ALL 拼接所有子类表

select p.id, p.amount,
       p.card_number, p.exp_date,
       p.bank_name, p.account_no, p.clazz_
from (
    select id, amount,
           card_number, exp_date,
           cast(null as varchar(255)) as bank_name,
           cast(null as varchar(255)) as account_no,
           1 as clazz_
    from credit_card_payment

    union all

    select id, amount,
           cast(null as varchar(255)) as card_number,
           cast(null as varchar(255)) as exp_date,
           bank_name, account_no,
           2 as clazz_
    from bank_transfer
) p
order by p.id;

clazz_ 是 Hibernate 注入的"虚拟 discriminator"——TPC 没有 discriminator 列,Hibernate 在 UNION ALL 里给每个子类硬编码一个数字,用来还原行属于哪个子类、应该实例化成哪个 Java 类。

TABLE_PER_CLASS 不能使用 IDENTITY — 配 GenerationType.IDENTITY 时 Hibernate 启动直接抛 MappingException: Cannot use identity column key generation with <union-subclass>。原因:跨子类表的 ID 必须全局唯一(credit_card_payment.id=1 与 bank_transfer.id=1 在父类视角是同一个 Payment——会导致 UNION ALL 后两行 id 重复)。IDENTITY 是单表自增,无法跨表保证唯一性。解决方案:共享 SEQUENCE 或 UUID——所有子类表共用一个 payment_seq,或者用 @GeneratedValue(strategy=AUTO) + UUID 生成器。

TABLE_PER_CLASS 优劣矩阵

维度评价说明
子类查询最优单表扫描,字段紧凑
多态查询最差UNION ALL 后排序 / 分页
父类分页昂贵UNION 全取后内存 / 子查询排序
数据库约束强子类表内 not null OK
ID 策略限制不能 IDENTITY,必须共享 SEQUENCE / UUID
子类增加成本高每加一个子类 = 加一张完整表 + UNION ALL 多一段
表数量多N 个子类 = N 张表
适用场景几乎只查子类父类查询极少 / 子类完全独立的业务

7.5 @MappedSuperclass

@MappedSuperclass 严格来说不是"继承策略"——它根本不在 @Inheritance 三选一的范围内。它的语义是"把父类的字段复用进所有子类的表里,父类自己不是实体、不能查询、不能关联"。最典型用途:BaseEntity 提供 id / createdAt / updatedAt 三件套,让所有业务实体直接 extend 进来用。

形态:仅字段复用,父类无表无查询

图 7-5 · @MappedSuperclass 的字段复用模型

注解配置

@MappedSuperclass
public abstract class BaseEntity {
    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE)
    private Long id;

    @Column(name = "created_at", nullable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at", nullable = false)
    private LocalDateTime updatedAt;
}

@Entity
@Table(name = "orders")
public class Order extends BaseEntity {
    @Column(name = "order_no", nullable = false)
    private String orderNo;
}

@Entity
@Table(name = "products")
public class Product extends BaseEntity {
    @Column(name = "product_name", nullable = false)
    private String productName;
}

生成的 DDL — 字段复制进每个子类表

create table orders (
    id          bigint not null,
    order_no    varchar(255) not null,
    created_at  timestamp not null,
    updated_at  timestamp not null,
    primary key (id)
);

create table products (
    id            bigint not null,
    product_name  varchar(255) not null,
    created_at    timestamp not null,
    updated_at    timestamp not null,
    primary key (id)
);

父类查询限制 — 直接抛异常

em.createQuery("select b from BaseEntity b", BaseEntity.class).getResultList();
[Error] org.hibernate.query.sqm.UnknownEntityException:
Could not resolve root entity 'BaseEntity'

@MappedSuperclass 不进入 Hibernate 运行时元模型的实体集合,所以 JPQL 无法以它为查询根,@OneToMany BaseEntity / @ManyToOne BaseEntity 这种关联也是非法的。需要"查父类"的能力请改用 @Inheritance 的三种策略之一。

@MappedSuperclass vs SINGLE_TABLE 的边界

需求选 @MappedSuperclass选 SINGLE_TABLE / JOINED / TPC
共享审计字段(createdAt 等)★ 首选过度设计
父类作为 JPQL 查询根不支持必须
父类作为关联目标(@ManyToOne)不支持必须
子类是同一业务概念的不同形态语义不对正确选项
子类是完全不同的业务实体★ 首选语义不对

7.6 Discriminator 设计

在 SINGLE_TABLE 与 JOINED 下,Hibernate 需要一种方式从单行数据判断"这一行属于哪个 Java 子类"——这就是 discriminator(辨别列)。它有三种声明形态:列方式、值方式、SQL 表达式方式。生产里有一个铁律:discriminator 值不要绑定 Java 类名。

三种 Discriminator 注解

注解作用默认值推荐配置
@DiscriminatorColumn定义类型列DTYPE / STRING / 长度 31显式命名 payment_type
@DiscriminatorValue子类的类型值实体类名(如 "CreditCardPayment")短稳业务码 (CC, BT)
@DiscriminatorFormulaSQL 表达式生成类型无仅用于兼容遗留 schema

推荐配置

@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorColumn(
    name = "payment_type",
    discriminatorType = DiscriminatorType.STRING,
    length = 8)
public abstract class Payment { }

@Entity @DiscriminatorValue("CC") public class CreditCardPayment extends Payment {}
@Entity @DiscriminatorValue("BT") public class BankTransfer extends Payment {}

正反对照:为什么不要让 discriminator 绑定类名

反例:省略 @DiscriminatorValue

@Entity
public class CreditCardPayment
    extends Payment {
    // 默认 discriminator 值 = 类名
    // → "CreditCardPayment"
}

问题 1:列长度要 17+ 字符,索引膨胀。
问题 2:重命名类(IDE rename)→ DDL 期待新值,但历史数据是旧值。
问题 3:跨语言迁移(Java → Kotlin / 包名调整)立刻不一致。

正例:显式短业务码

@Entity
@DiscriminatorValue("CC")
public class CreditCardPayment
    extends Payment {
    // discriminator = "CC"
    // 与 Java 类名解耦
}

短业务码,索引紧凑。
类重命名不影响数据库历史。
支持非 Java 客户端直接读 DB。

@DiscriminatorFormula 的极端用法

@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorFormula(
    "case when card_number is not null then 'CC' " +
    "     when bank_name   is not null then 'BT' " +
    "end")
public abstract class Payment { }

仅在接管遗留 schema 没有 discriminator 列时使用。该方式脆弱:每次查询都执行 case 表达式,无索引可用,且引入"类型判断逻辑"耦合到 SQL 表达式里。新项目永远不要这么干。

discriminator 是数据库层的"业务身份码",而不是"Java 实现细节" — 把它当成长存的合同。一旦上线,discriminator 列里的值就成了所有读取这张表的客户端(Java 应用、报表系统、ETL、DBA SQL 脚本、其他语言的微服务)的共识——它们都靠这个值判断"这一行是 CC 还是 BT"。Java 那边的类重命名是 IDE 一键操作,但数据库里几十亿行的历史 discriminator 值不会跟着变。把 discriminator 值定义成 8 个字符以内的稳定业务码(CC / BT / WX_PAY / ALI_PAY),写进《领域对象命名规范》文档,谁都不准改——这是把"类继承映射"做成"数据库长寿合同"的核心动作。

7.7 策略选择决策树

没有"通用最优策略"——选型由四个业务问题决定:(1) 是否会以父类作为查询根 / 关联目标?(2) 多态列表 / 统计是不是高频?(3) 子类专属字段差异多大?(4) 数据库强约束需求?用决策树把这四个问题串起来,落到三选一的具体策略上。

策略选择决策树

选型四步走

1
问"父类是不是实体"。如果业务上从来不会"按父类查询"或"以父类做关联目标",直接 @MappedSuperclass——零多态成本、不占用实体集合命名空间。
2
问"多态查询频率"。如果父类列表 / 父类分页 / 父类统计是高频接口(每秒 100+),倾向 SINGLE_TABLE——单表 scan 几乎一定是最快的。如果父类查询每天只跑几次报表,选 JOINED / TPC 都行。
3
问"子类字段差异 + 约束需求"。子类专属字段多 + 必须在 DB 层 not null → JOINED。子类专属字段少 + 接受 NULL → SINGLE_TABLE。子类几乎完全独立 / 父类查询极少 → TABLE_PER_CLASS。
4
问"未来 3 年子类增长"。预计还会加 5+ 子类:SINGLE_TABLE 列会膨胀到几十列,改 JOINED;TABLE_PER_CLASS 每加一个子类要新建表 + 改 UNION ALL,改 JOINED 或组合。组合(策略模式 + 普通字段)在子类爆炸时是真正的解,不要硬上继承。

7.8 源码深潜

三种策略在 Hibernate 内部对应三个不同的 EntityPersister 实现——它们各自负责把"select p from Payment p"翻译成自己策略下的 SQL 形态。源码入口是 EntityManager.createQuery → SessionImpl → QueryInterpretationCache → 选 Persister → 生成 SQL AST → 输出最终 SQL。

四个核心设计取舍

01
每策略独立 Persister

三种策略的 SQL 形态差异显著:单表 + filter / 多表 LEFT JOIN / UNION ALL。统一 Persister 实现会引入大量分支判断,代码可读性差、性能不可控。Hibernate 选择三个独立 Persister,各自专注于自己策略的 SQL 优化(如 JOIN 顺序、filter 下推、字段裁剪)。代价是源码结构复杂,但运行时性能稳定。

02
@MappedSuperclass 不注册实体

@MappedSuperclass 仅做字段复用,不允许查询或关联。该限制使运行期元模型保持清晰——实体集合中仅包含真正的 @Entity 类。允许 @MappedSuperclass 查询将引入虚拟 union(把所有子类表 union 起来),语义和性能都不透明,且会与 TABLE_PER_CLASS 在 DDL 上无差别,造成认知冲突。

03
SINGLE_TABLE 强依赖 discriminator

单表实现必须能从行数据快速判断 Java 类型。discriminator 列是该判断的唯一来源,Hibernate 在 ResultSet 处理阶段先读它,再决定实例化哪个子类。生产中应避免使用 nullable 列推断类型(例如根据 card_number 是否为 null 判断是不是 CC),该方案脆弱、无法索引、且可能与未来新子类冲突。

04
TABLE_PER_CLASS 用 UNION ALL

由于无父表,父类查询必须合并所有子类表。Hibernate 用 UNION ALL 实现合并,代价是父类排序 / 分页性能较差(必须先 union 出全部行再排序)。强行建虚拟父表将退化为 JOINED,失去 TABLE_PER_CLASS 的"子类完全独立"特性——所以这个取舍是有意为之。

7.9 代理与 instanceof

Hibernate 的 LAZY 代理基于 CGLIB / ByteBuddy 在运行期生成,代理类继承自父类(Payment)而非具体子类(CreditCardPayment)。这导致一个常见的"判断不对"问题——instanceof CreditCardPayment 对代理对象返回 false,即使数据库里这一行确实是 CC。

反例:代理 instanceof 失败

Payment proxy = em.getReference(Payment.class, 1L);

if (proxy instanceof CreditCardPayment cc) {
    String num = cc.getCardNumber();
} else {
    String type = proxy.getClass().getName();
    // 输出: com.example.Payment$HibernateProxy$Xa3F4...
}

proxy.getClass() 返回 com.example.Payment$HibernateProxy$xxx——这是一个动态生成的代理类,继承自 Payment 而非 CreditCardPayment。instanceof CreditCardPayment 自然返回 false,即使你查的 id=1 在数据库里确实是 CC 行。

修复:Hibernate.unproxy 强制初始化

Payment real = Hibernate.unproxy(proxy, Payment.class);
if (real instanceof CreditCardPayment cc) {
    String num = cc.getCardNumber();
}

Hibernate.unproxy() 触发代理初始化(执行真实 SELECT 拉数据),并返回真实子类实例(CreditCardPayment 而非代理)。之后 instanceof 判断恢复正常。

三种修复路径对比

方案触发懒加载API 侵入推荐场景
Hibernate.unproxy(proxy)是低★ 标准修复
切换为 EAGER 加载查询时加载高(影响性能)极少场景
JOIN FETCH 在 JPQL 里取真实子类查询时加载中(改 JPQL)明确知道要哪个子类时
用 visitor 模式 / instanceof 替代为 type 字段判断不触发需重构业务代码不想触发懒加载时
不要在 hashCode / equals 里用 instanceof 判断子类 — 实体的 equals / hashCode 必须对代理与真实对象一致,否则同一个 id 在 Set / Map 里会成两条。如果你写 if (other instanceof CreditCardPayment cc) ... 在 equals 里,代理对象的 equals 会走 else 分支,导致代理与真实子类判等不成立。修复:equals / hashCode 只基于 id 字段,绝不基于子类专属字段或 instanceof 判断。这是所有 JPA 实体的通用最佳实践,在继承场景下尤其关键。

7.11 回归护栏

继承策略一旦上线,任何"重构 / 框架升级 / 调整注解"的改动都可能改变最终生成的 SQL 形态——单表退化为多表 JOIN、JOIN 顺序变化、discriminator 值漂移,这些变化在测试环境可能毫无表现,生产高 QPS 下立刻爆炸。回归护栏的目的就是在 CI 里断言关键不变量。

护栏 1:Persister 类型断言

@Test
void paymentShouldUseExpectedPersister() {
    SessionFactoryImplementor sfi = emf.unwrap(SessionFactoryImplementor.class);
    EntityPersister persister = sfi.getRuntimeMetamodels()
        .getMappingMetamodel()
        .getEntityDescriptor(Payment.class);

    assertThat(persister.getClass().getSimpleName())
        .isEqualTo("SingleTableEntityPersister");
}

这个测试守护"策略不会被无意改掉"——比如同事把 @Inheritance(SINGLE_TABLE) 改成 JOINED 时,生成 SQL 立刻从单表 scan 变成 LEFT JOIN N 子表,P99 翻 5 倍。这个测试在 CI 里失败,改动就出不去。

护栏 2:Discriminator 业务码守护

@Test
void discriminatorShouldUseStableBusinessCodes() {
    em.persist(new CreditCardPayment(new BigDecimal("99.00"), "4111", "2030-12"));
    em.persist(new BankTransfer(new BigDecimal("250.00"), "Bank A", "62220001"));
    em.flush();

    List<String> types = em.createNativeQuery(
        "select payment_type from payment order by id", String.class)
        .getResultList();

    assertThat(types).containsExactly("CC", "BT");
}

这个测试守护"discriminator 的实际写入值是稳定业务码"——同事不小心删了 @DiscriminatorValue("CC"),新写入的值会变成 "CreditCardPayment"(类名),与历史数据不一致,所有按 type 过滤的 SQL 立刻失效。这个测试直接断言写入值,失败立刻可见。

护栏 3:JOIN 数上限守护

@Test
void joinedQueryShouldKeepExpectedJoinCount() {
    SqlRecorder recorder = sqlRecorder.clear();

    em.createQuery("select p from Payment p", Payment.class).getResultList();

    String sql = recorder.singleSelectSql();
    long joinCount = sql.toLowerCase().split(" join ").length - 1L;
    assertThat(joinCount).isLessThanOrEqualTo(2L);
}

这个测试守护"JOIN 数不会因为新增子类悄悄涨"——JOINED 策略下每加一个子类多一个 LEFT JOIN,深度从 2 到 5 时 P99 跳 5 倍但单测可能完全过。CI 里设上限(≤ 2)能在 PR 阶段拦下"新加子类"动作,触发架构讨论而不是默默上线。
