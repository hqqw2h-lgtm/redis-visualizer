第 1 章 · JPA 全局架构

对象状态机 · SQL 生成器 · 事务协调器 · 调用链 · PersistenceContext · 配置 · 可观测

JPA 是一组运行在 Java 对象世界与关系数据库之间的中间层规范。它由三类核心责任构成:对实体对象进行状态跟踪,在合适时机将状态变化翻译为 SQL,并在事务边界内将这些 SQL 通过 JDBC 发送到数据库。理解 JPA 的关键不在于背 EntityManager 的方法清单,而在于掌握三条主线:实体在何时进入哪种状态、Provider 在何时生成何种 SQL、SQL 在何时通过事务真正落库。

JPA 规范本身只定义 API 与语义,具体执行由 Provider 完成。Hibernate 是当前主流实现,EclipseLink 是参考实现。本书后续章节默认以 Hibernate 6.x 为参照,JPA 规范以 Jakarta Persistence 3.1 为基准。本章建立的"状态决定 SQL"抽象贯穿全书,后续每一章都是这个模型在某个具体场景下的展开:第 2 章把状态机展开为完整生命周期,第 3 章把 PersistenceContext 拆开看缓存与脏检查,第 8 章解读 JPQL 编译,第 17 章潜入 Hibernate 源码看真实实现。

  • JPA 的三类运行时职责:对象状态机 / SQL 生成器 / 事务协调器 — 它们之间的耦合而非孤立理解
  • 从 Service → EntityManager → PersistenceContext → Provider → JDBC → DB 的完整六层调用链
  • EntityManagerFactory / EntityManager / EntityTransaction 的线程安全语义与生命周期归属
  • PersistenceContext 内部的四类数据:Identity Map / Snapshot / ActionQueue / Collection 跟踪表
  • JPQL → AST → Dialect → SQL 的三入口模型,以及 PostgreSQL / MySQL / Oracle / H2 的方言差异
  • persistence.xml 与 Spring Boot application.yml 两条配置路径的等价映射
  • 如何用 Hibernate Statistics 把 SQL 数量、事务时长、缓存命中率变成可观测信号与回归护栏
  • 七类入口章典型陷阱:EMF 重复创建 / EM 跨线程 / flush 当 commit / IDENTITY 误解 / JPQL 等同 SQL 等

真实事故:某团队的报表服务在容器化部署后 RT P99 从 80ms 飙升到 1.2s。链路追踪显示每次请求都在 Hibernate 启动阶段耗时 180-300ms。根因:开发者在 @RestController 方法内直接调用 Persistence.createEntityManagerFactory("demoPU"),EMF 是应用级单例(创建包含元模型扫描、方言探测、连接池初始化),每请求新建一次相当于每请求重做应用启动。修复:把 EMF 抽为 Spring Bean 或静态字段,单例化后 P99 回落到 95ms。教训:本章 §1.3 与 §1.8 强调 "EMF 单例 / EM 请求级 / Tx 跟随 EM" 不是教条,而是性能护栏。

1.1 三大核心职责

JPA 在运行时同时扮演三类角色,任何一个角色被孤立理解,都会导致对运行时行为的误判。这三类角色之间的耦合关系是:对象状态决定 SQL 类型,SQL 类型决定事务的提交语义。

图 1-1-1 · JPA 的三类运行时角色及它们之间的关系

1.1.1 对象状态机

JPA 实体在运行时必然处于以下四种状态之一:Transient(临时,未与 PC 关联)、Managed(受管,被 PC 跟踪)、Detached(游离,曾受管现已脱离)、Removed(已移除,等待 flush 删除)。状态切换由 EntityManager 上的 API 触发,并不直接对应 SQL 执行——这一点是新手最常见的误区。

import jakarta.persistence.*;

@Entity
@Table(name = "customers")
public class Customer {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String name;

    @Column(nullable = false, unique = true)
    private String email;

    protected Customer() {}

    public Customer(String name, String email) {
        this.name = name;
        this.email = email;
    }

    public Long getId() { return id; }
    public String getEmail() { return email; }
    public void changeEmail(String email) { this.email = email; }
}

对应的表结构:

create table customers (
    id     bigint generated by default as identity primary key,
    name   varchar(100) not null,
    email  varchar(200) not null unique
);

最小生命周期演示:

EntityManager em = emf.createEntityManager();
EntityTransaction tx = em.getTransaction();

tx.begin();
Customer c = new Customer("C1001", "alice@example.com"); // Transient
em.persist(c);                                           // Transient → Managed
tx.commit();                                             // flush + commit
em.close();                                              // Managed → Detached

persist() 的语义是"将 Transient 对象纳入当前 PersistenceContext 管理",并不直接发出 INSERT。SQL 何时发送由 ID 生成策略、flush 模式与事务提交点共同决定。IDENTITY 策略下 Provider 必须执行 INSERT 才能拿到自增主键,因此该策略下 INSERT 在 persist() 调用内即时发出;SEQUENCE 与 TABLE 策略允许 Provider 推迟 INSERT 到 flush 时机。

1.1.2 SQL 生成器

JPA 的 UPDATE 不依赖显式调用。当 Managed 实体的字段被修改后,Provider 在 flush 阶段比对当前对象与加载快照,将差异翻译为 UPDATE SQL。该机制称为脏检查 (Dirty Checking)。

tx.begin();
Customer c = em.find(Customer.class, 1L);   // SELECT + 加入 PC
c.changeEmail("alice.new@example.com");      // 仅修改 Java 对象
tx.commit();                                  // flush 阶段检测差异 + UPDATE

Hibernate 生成的 SQL:

select c1_0.id, c1_0.email, c1_0.name
from customers c1_0
where c1_0.id = ?

update customers
set email = ?, name = ?
where id = ?

业务代码只表达对象意图,Provider 负责差异计算与 SQL 生成。该抽象层带来的代价是:开发者必须能阅读底层 SQL,因为唯一键冲突、锁等待、慢查询等问题最终都发生在 SQL 层,而不是对象层。这条原则贯穿全书——任何"看起来是对象问题"的故障,根源都在它最终翻译出来的 SQL 上。

1.1.3 事务协调器

flush 与 commit 是两个不同概念。flush() 将待写入的 SQL 发送给 JDBC,commit() 才真正提交事务。一个未提交的事务即使已 flush,数据库仍可在 rollback 时撤销。

tx.begin();
Customer c = em.find(Customer.class, 1L);
c.changeEmail("rollback@example.com");
em.flush();      // SQL 已发往数据库,事务尚未提交
tx.rollback();   // 数据库回滚,修改不生效

em.clear();
tx.begin();
Customer reloaded = em.find(Customer.class, 1L);
String emailNow = reloaded.getEmail();   // 仍为旧值
tx.commit();

预期 SQL 与状态:

[SQL] select c1_0.id, c1_0.email, c1_0.name from customers c1_0 where c1_0.id=?
[SQL] update customers set email=?, name=? where id=?
[ROLLBACK]
[SQL] select c1_0.id, c1_0.email, c1_0.name from customers c1_0 where c1_0.id=?
[email] alice.new@example.com

1.2 运行时调用链

从应用代码的一行 em.persist(customer) 到数据库的一行 INSERT,中间经过六层抽象。理解这条调用链,才能在生产事故里准确定位"故障发生在哪一层"——是业务代码、ORM、JDBC 还是数据库。

图 1-2-1 · 从应用代码到数据库的完整路径(时序图)

表 1-2-1 · 六层职责对应

层次应用代码所见运行时责任典型故障模式
ApplicationService / Repository / 实体对象表达业务意图事务边界过大或过小
EntityManagerpersist / find / query / remove管理 PC 与事务EM 跨线程使用
PersistenceContextManaged 实体与字段快照一级缓存与脏检查内存膨胀
ProviderHibernate / EclipseLinkSQL 生成与加载计划N+1 与方言差异
JDBCPreparedStatement参数绑定与网络传输连接泄漏与慢 SQL
DatabaseTables / Indexes / Constraints持久化与并发控制死锁与扫描过多

层与层之间的接口是固定的,具体实现可替换。该分层使得"切换 Provider 不修改业务代码"在理论上成立,但缓存配置、批处理策略、SQL 形状仍依赖 Provider 的具体行为。

1.2.1 控制权传递时序

tx.commit() 触发的时序较为隐式:EntityManager 先调用 PersistenceContext 的 flush 方法,后者扫描所有 Managed 实体的脏字段,生成 INSERT/UPDATE/DELETE 语句的执行队列,再委托给 Provider 转换为目标方言下的 SQL,最后由 JDBC 发送到数据库。

图 1-2-2 · 单事务内的应用线程时间轴

1.2.2 异常路径

事务异常路径中,任一步失败均触发回滚。Provider 在 flush 阶段检测到唯一键冲突时,会抛出 PersistenceException 并将事务标记为仅可回滚 (rollback-only)。后续 commit() 调用将转为隐式 rollback,并抛出 RollbackException。

常见误区:看到 RollbackException 就以为"我没显式 rollback,框架自己回滚了"——实际是 commit 阶段触发隐式回滚。正确的诊断顺序:① 看完整 SQL 日志(开 show_sql),② 找到第一个抛异常的 SQL,③ 检查它涉及的唯一约束 / 外键 / 非空字段。一旦确认是脏检查阶段的 SQL 报错,就不要再去看上层业务代码——上层只是触发器,真正的原因在 SQL 层。

1.3 核心对象的生命周期

JPA 核心类不多,但每一个都有明确的生命周期归属:谁创建、谁销毁、是否线程安全、是否可共享。把这张图熟记于心,可以避免 80% 的入门陷阱。

图 1-3-1 · JPA 核心对象关系树

Persistence 是静态启动入口,EntityManagerFactory 是应用级单例,EntityManager 与 EntityTransaction 跟随单次请求或事务存活,Managed 实体只在所属 PC 内有效。

1.3.1 RESOURCE_LOCAL 启动样例

EntityManagerFactory emf =
    Persistence.createEntityManagerFactory("demoPU");

EntityManager em = emf.createEntityManager();
EntityTransaction tx = em.getTransaction();

try {
    tx.begin();
    Customer found = em.find(Customer.class, 1L);
    tx.commit();
} catch (RuntimeException ex) {
    if (tx.isActive()) tx.rollback();
    throw ex;
} finally {
    em.close();
}

应用结束时 emf.close()。容器化环境中由 Spring 或 Jakarta EE 容器接管该生命周期。

表 1-3-1 · 核心对象生命周期速查

对象线程安全创建成本推荐生命周期反模式
EntityManagerFactory是高,需解析元数据应用级单例每请求创建

1.3.2 EntityManagerFactory 的内部结构

EMF 内部持有以下组件:实体元模型 (Metamodel)、SQL 生成器、二级缓存 (L2Cache,可选)、查询计划缓存、连接池配置、Dialect 实例。EMF 的初始化包含扫描注解、构建元模型、创建会话工厂等步骤,在生产环境中典型耗时为 200-2000 ms,具体值取决于实体数量与方言探测开销。

1.4 PersistenceContext

PersistenceContext 是 EntityManager 内部维护的内存区域,包含四类数据:Identity Map (主键到对象的映射)、Snapshot (实体加载时的字段值副本)、ActionQueue (待 flush 的 SQL 操作)、Collection 跟踪表 (受管集合代理及其状态)。理解 PC 是理解一级缓存、脏检查、级联、批处理的基础。

图 1-4-1 · PersistenceContext 内部 4 子结构 + 4 可观察行为

1.4.1 Identity Map 的可观察行为

tx.begin();
Customer a = em.find(Customer.class, 1L);
Customer b = em.find(Customer.class, 1L);
boolean same = (a == b);                   // 必为 true
tx.commit();

仅第一次 find() 触发 SELECT。第二次直接命中 Identity Map,返回同一 Java 对象引用。该机制保证同一 PC 内对同一主键的所有访问都获得一致视图。

-- 第一次 find:
select c1_0.id, c1_0.email, c1_0.name from customers c1_0 where c1_0.id=?
-- 第二次 find:无 SQL

1.4.2 PersistenceContext 清空与跨上下文比较

tx.begin();
Customer a = em.find(Customer.class, 1L);
em.clear();                                  // PC 清空,所有 Managed → Detached
Customer b = em.find(Customer.class, 1L);    // 重新 SELECT
boolean same = (a == b);                     // false
tx.commit();
select c1_0.id, c1_0.email, c1_0.name from customers c1_0 where c1_0.id=?
select c1_0.id, c1_0.email, c1_0.name from customers c1_0 where c1_0.id=?

em.clear() 后 PC 中所有实体进入 Detached 状态,持久化代理 (集合 / 关联) 被切断。后续 find() 重新触发 SELECT 并构建新的 Managed 实体。

加入 PCpersist / merge / find / query进入 Identity Map

1.4.3 PC 范围:transaction 范围 vs extended 范围

JPA 规范定义两种 PC 范围。PersistenceContextType.TRANSACTION (默认) 中 PC 跟随事务边界,事务结束即清空;PersistenceContextType.EXTENDED 中 PC 跨多个事务存活,常用于 Stateful Session Bean 与 Spring 的 OpenEntityManagerInView 模式。

1.5 SQL 生成路径

Provider 将三类输入 (JPQL 文本、Criteria 程序对象、PersistenceContext 中的脏实体) 统一转换为 SQL AST,再由 Dialect 渲染为目标数据库特定的 SQL 文本。AST 是中间表示,使得不同输入可共享后续的 SQL 优化与方言渲染逻辑。

图 1-5-1 · 三条入口(JPQL / Criteria / Dirty) 汇合于 AST

1.5.1 JPQL 到 SQL 的转换样例

List<Customer> result = em.createQuery("""
    select c
    from Customer c
    where c.email like :domain
    order by c.id
    """, Customer.class)
    .setParameter("domain", "%@example.com")
    .setMaxResults(10)
    .getResultList();

PostgreSQL Dialect 输出:

select c1_0.id, c1_0.email, c1_0.name
from customers c1_0
where c1_0.email like ? escape ''
order by c1_0.id
fetch first ? rows only

MySQL Dialect 输出:

select c1_0.id, c1_0.email, c1_0.name
from customers c1_0
where c1_0.email like ? escape ''
order by c1_0.id
limit ?

JPQL 语法对开发者一致,但分页关键字 (fetch first 与 limit)、字符串函数、日期函数、布尔类型等在不同方言下的渲染存在差异。生产部署前需对关键查询进行集成测试,避免方言切换导致行为漂移。

1.6 配置入口

JPA 标准启动方式通过 META-INF/persistence.xml 声明 Persistence Unit。Spring Boot 项目通过 application.yml + 自动配置生成等价的 EMF。两条路径汇合于同一类对象,后续运行时无差异。

图 1-6-1 · 两种配置路径汇合于 EMF

1.6.1 标准 JPA 配置

<persistence xmlns="https://jakarta.ee/xml/ns/persistence"
             version="3.1">
  <persistence-unit name="demoPU" transaction-type="RESOURCE_LOCAL">
    <provider>org.hibernate.jpa.HibernatePersistenceProvider</provider>
    <class>com.example.Customer</class>
    <properties>
      <property name="jakarta.persistence.jdbc.url"
                value="jdbc:h2:mem:demo"/>
      <property name="jakarta.persistence.jdbc.user" value="sa"/>
      <property name="jakarta.persistence.jdbc.password" value=""/>
      <property name="hibernate.hbm2ddl.auto" value="create-drop"/>
      <property name="hibernate.show_sql" value="true"/>
      <property name="hibernate.format_sql" value="true"/>
    </properties>
  </persistence-unit>
</persistence>

启动验证:

EntityManagerFactory emf =
    Persistence.createEntityManagerFactory("demoPU");
boolean alive = emf.isOpen();
emf.close();

1.6.2 Spring Boot 等价配置

spring:
  datasource:
    url: jdbc:h2:mem:demo
    username: sa
    password:
  jpa:
    hibernate:

      hibernate.format_sql: true

1.7 观测与生产边界

JPA 的运行时行为对生产可观测性提出明确要求。每一层职责均应映射到具体指标,避免线上问题仅依赖代码审视。Hibernate 的 Statistics 接口提供 SQL 计数、查询耗时、缓存命中、二级缓存大小等核心指标,可通过 Micrometer 桥接到 Prometheus / Grafana。

1.7.1 SQL 数量护栏

@Test
void findCustomerById_issuesOneSelect() {
    Statistics statistics = sessionFactory.getStatistics();
    statistics.clear();

    txTemplate.executeWithoutResult(status -> {
        Customer found = em.find(Customer.class, 1L);
        assertThat(found.getId()).isEqualTo(1L);

1.9.1 设计取舍

取舍 1:状态机分层

JPA 将"对象状态"与"SQL 执行"显式分离。对象状态由 PC 维护,SQL 执行由 Provider 控制。该分离使得脏检查与批处理成为可能,代价是开发者必须理解状态切换与 SQL 时序之间的间接关系。直接调用 INSERT / UPDATE 的传统 DAO 模式简化了心智模型,但失去了批处理优化、一级缓存、级联与脏检查的运行时支持。

取舍 2:JPQL 中间层

JPQL 引入 AST 中间表示,使得不同方言可共享语法解析与查询优化逻辑。代价是 JPQL 与底层 SQL 语义并非完全对应:fetch join 在 JPQL 中是显式语法,在 SQL 中需结合具体方言渲染;子查询、窗口函数等高级特性在 JPQL 中支持有限。Native Query 与 Criteria API 是补充入口,本书第 9-10 章详细讨论。

取舍 3:RESOURCE_LOCAL 与 JTA

RESOURCE_LOCAL 由 EntityManager 自身管理事务,适用于单数据源场景;JTA 由容器管理,支持多数据源 XA 事务。容器化部署中默认采用 JTA,但 XA 协议引入两阶段提交开销。Spring Boot 默认采用单数据源 + RESOURCE_LOCAL 模拟,保留事务管理器抽象,该方案在多数业务场景下性能与一致性兼得。
