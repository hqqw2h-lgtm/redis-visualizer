第 12 章 · N+1 问题全解

LAZY 触发 · EAGER 提前 · OSIV 越界 · JOIN FETCH · EntityGraph · @BatchSize · DTO · BatchFetchQueue · CollectionPersister · Hibernate Statistics · SQL 数量回归测试

N+1 是 ORM 世界里最古老、最隐蔽、最容易复发的性能反模式。一段看上去人畜无害的业务循环 —— for (Customer c : customers) c.getOrders().size(); —— 在第一次跑通时只看见"逻辑没问题",在 staging 数据量小时只看见"接口够快",直到生产环境的真实流量打过来,日志里突然出现一千行 select ... from orders where customer_id = ?,接口 P99 飙到秒级,数据库 CPU 占用 90% 上不去也下不来。这就是 N+1。它的本质是把"一次业务读取"在数据库层放大成了"1 + N 次远程往返",而 N 完全等于父行数 —— 数据涨 10 倍,SQL 涨 10 倍,延迟涨 10 倍。

修复 N+1 的核心思路要从一开始就摆正:改的是查询入口的"加载形状",而不是业务循环的写法。把 for 写成 stream、把 size() 改成 count(*)、把 forEach 拆成 parallel —— 这些都治标不治本,真正的修复必须发生在第一条 SELECT 出去之前:让那条 SELECT 同时把 N 条子行也一并捞回来,或者拼成一条 IN 批量,或者根本绕开实体直接走 DTO。本章把 N+1 的成因、四种主修复(JOIN FETCH / EntityGraph / @BatchSize / DTO)、检测与回归测试讲到能在生产事故复盘会上守住每一个细节。

  • N+1 的"1 + N" SQL 数学规律 —— 为什么父行数线性放大 SQL 数量,以及它在连接池 / DB CPU / P99 上的乘数效应
  • N+1 的三大触发源:循环访问 LAZY 集合 / EAGER 默认值 / OSIV 序列化越界 —— 每个的特征 SQL 形状不同,定位策略也不同
  • Hibernate 内部的集合初始化路径:AbstractPersistentCollection → InitializeCollectionEvent → CollectionPersister → BatchFetchQueue → CollectionLoaderSingleKey / CollectionBatchLoader
  • 修复 1 · JOIN FETCH:JPQL 里把关联折叠到一条 SQL,以及它在分页 / 多 List 集合下的硬性限制(MultipleBagFetchException / HHH90003004)
  • 修复 2 · @EntityGraph:让 Spring Data 派生方法保留语义但显式声明加载形状,和 LOAD vs FETCH graph 的差异
  • 修复 3 · @BatchSize:把 N 次单 SELECT 折叠为 ceil(N/size) 次 IN 批量,适合"不确定何时访问"的兜底场景
  • 修复 4 · DTO 投影:绕开实体图、绕开 PersistenceContext,只读 API 的最简形状
  • 四种修复的选型决策树 —— 详情查询走 JOIN FETCH,Repository 复用走 EntityGraph,不确定访问走 BatchSize,只读 API 走 DTO
  • 检测与回归:Hibernate Statistics、SQL 形状捕获、SQL 数量断言、OSIV 关闭测试 —— 把 N+1 永久挡在 CI 关卡之外
  • 陷阱清单:@ManyToOne 默认 EAGER、双向 toString 死循环、LOAD graph 漏 EAGER、多 List fetch、BatchSize 过小

前置知识:第 2 章实体生命周期与 PersistenceContext;第 11 章抓取策略(FetchType.LAZY/EAGER、代理、PersistentCollection、BatchFetchQueue 已在那里铺好底层概念)。本章是把第 11 章学到的"加载机制"对准"N+1 这个具体反模式"做一次完整剿灭,后续第 13 章进入 JPA 缓存层次,会借助本章的 SQL 计数能力去验证缓存命中率。

N+1 的乘数效应 —— 一张速查面板

图 12-0-2 · 父行数 / SQL 数 / 串行延迟的乘数关系
N+1 不是"SQL 太慢",而是"SQL 太多" —— 单条 SQL 都很快才是它最迷惑人的地方。慢 SQL 在数据库慢查询日志里一抓一个准,DBA 一秒就能定位。但 N+1 的每一条 SQL 都在 0.5-2 ms 内返回 —— 慢查询日志(默认阈值 1s)里干干净净一片祥和,DBA 看不到任何"慢"的迹象。它的特征是 QPS 异常 + 接口慢 + DB CPU 高 的奇怪组合 —— 把数据库的 SHOW PROCESSLIST 抓 20 次截图叠在一起,你会看到几十条几乎一样的 select ... where customer_id = ? 在并行执行,这才是真正的烟枪。生产里"慢查询日志没东西但接口慢"九成是 N+1。

12.1 业务模型

本章贯穿一个最经典的电商业务模型:Customer 1 - N Order。这种"父-多子"结构是 N+1 出现频率最高的形状 —— 列表页要展示客户名称,点开要看订单明细,Jackson 序列化又要把 orders 字段一并写出去,任何一环没拦住,N+1 就会冒出来。

@Entity
@Table(name = "customers")
public class Customer {
    @Id
    private Long id;

    private String name;

    @Enumerated(EnumType.STRING)
    private CustomerStatus status;

    @OneToMany(mappedBy = "customer", fetch = FetchType.LAZY)
    private List<Order> orders = new ArrayList<>();
}

@Entity
@Table(name = "orders")
public class Order {
    @Id
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "customer_id")
    private Customer customer;

    private BigDecimal amount;
}

public enum CustomerStatus { ACTIVE, INACTIVE, BLOCKED }

关键的两个声明:@OneToMany 显式 LAZY(默认就是 LAZY,但写出来表达意图);@ManyToOne 也显式 LAZY(这是反默认 —— JPA 规范里 ManyToOne 默认是 EAGER,这是 N+1 最常见的隐藏种子)。这两个 LAZY 是后续 12.2 节的"实验台" —— 不显式 LAZY 的话连"复现 N+1"都做不出来,因为 EAGER 会在父查询那一步就把 N+1 直接发出去。

12.2.1 最小复现 —— 10 条 SQL 现场抓现行

List<Customer> customers = em.createQuery("""
    select c from Customer c order by c.id
    """, Customer.class)
    .setMaxResults(10)
    .getResultList();

for (Customer c : customers) {
    int size = c.getOrders().size();
}
-- [SELECT 1] 父查询
select id, name, status
from   customers
order  by id
fetch  first 10 rows only;

-- [SELECT 2..11] 每个父行触发一次集合初始化
select id, customer_id, amount
from   orders
where  customer_id = ?;
-- ... 重复 10 次,每次换一个 customer_id ...
[SQL 总数]      11
[网络往返]      11
[串行延迟]      11 × RTT
[规律]          1 + N (N = 父行数)
[父查询占比]    1/11  ≈  9%
[N+1 占比]      10/11 ≈  91%

这 11 条 SQL 是 N+1 的"经典心电图"。识别它的窍门是 查 SQL 日志,看 where 外键 = ? 是否出现 N 次。一旦看见这个特征,根因 100% 是循环访问 LAZY 集合,无需再做更多排查。

不要用 customers.parallelStream() 包一下就以为修好了 N+1。常见误判:把 for 循环改成 parallelStream,SQL 数量没变(还是 N+1),只是把 N 次串行变成 N 次并行 —— DB CPU 占用更猛、连接池被一次性吃满、Hibernate Session 不是线程安全的(parallelStream 跨线程操作 PersistentCollection 会触发 ConcurrentModificationException 或更隐蔽的状态错乱)。修 N+1 永远是改"加载形状",不是改"循环并发"。

12.3 EAGER 与 OSIV

很多人以为 N+1 只在循环里出现 —— 其实 EAGER 默认值和 OSIV 是更隐蔽的两个变体:它们让 N+1 在你根本没写循环的情况下也能发出去。这一节把这两个"暗箱触发器"摊开。

12.3.1 EAGER 提前触发 —— 你没循环,SQL 自己出去了

@OneToMany(mappedBy = "customer", fetch = FetchType.EAGER)
private List<Order> orders = new ArrayList<>();

// 业务代码:只是查 5 个客户,根本没访问 orders
List<Customer> customers = em.createQuery("""
    select c from Customer c where c.status = :s
    """, Customer.class)
    .setParameter("s", CustomerStatus.ACTIVE)
    .setMaxResults(5)
    .getResultList();

// 此时 SQL 已经发了 6 条 —— 父 1 条 + 子 5 条 secondary select
-- [SELECT 1] 父查询
select id, name, status from customers where status = ? fetch first 5 rows only;

-- [SELECT 2..6] EAGER 在父查询完成后立即触发,不需要业务代码访问
select id, customer_id, amount from orders where customer_id = ?;
-- ... 重复 5 次 ...
[现象]    EAGER 在父查询完成后立即触发 secondary select
[特征]    业务代码里看不到任何 customer.getOrders() 调用
[结论]    EAGER 不是修复,只是把 N+1 的触发点从"业务循环"提前到"父查询完成"
[反直觉]  Hibernate 默认 @ManyToOne EAGER —— JPA 规范如此,但绝大多数生产场景应改 LAZY

12.3.2 OSIV 序列化触发 —— Controller 已经 return 了,SQL 还在跑

@RestController
class CustomerController {
    @GetMapping("/customers")
    List<Customer> list() {
        // 返回 entity,看上去 Service 已经 return 了
        return customerService.listCustomers();
    }
}

@Service
class CustomerService {
    @Transactional(readOnly = true)
    List<Customer> listCustomers() {
        return em.createQuery("select c from Customer c", Customer.class)
            .setMaxResults(3)
            .getResultList();
    }
}
-- [SELECT 1] Service 内的父查询
select id, name from customers fetch first 3 rows only;

-- [SELECT 2..4] Service 已经 return,Controller 已经 return,
-- Spring 调 Jackson 序列化时才发出
select id, customer_id, amount from orders where customer_id = ?;
[根因]   Jackson 反射调用 customer.getOrders() 触发 LAZY 初始化
         OSIV(Open Session In View)默认开,Session 一直撑到响应写出
[症状]   接口慢、SQL 数飙高,但代码里完全看不到"哪行触发了 N+1"
[修复]   spring.jpa.open-in-view=false  +  Service 层返回 DTO
         (光关 OSIV 不返回 DTO,会在序列化时抛 LazyInitializationException)

OSIV 配置矩阵

配置序列化时结果推荐
spring.jpa.open-in-view=true(默认)Session 还活着,Jackson 自动触发 LAZY静默 N+1,问题被掩盖关闭
spring.jpa.open-in-view=falseSession 已关,Jackson 触发抛 LIE问题立刻暴露,被迫改 DTO★ 推荐
Service 加载 Entity 直接返回OSIV 默默掩盖看似没事,实则随时退化始终返回 DTO
启动参数 + DTO 双重保险编译期 / 运行期双层防线N+1 入不了 Controller★ 生产标准
OSIV 在 Spring Boot 默认开 —— 这是 Spring 团队反复辩论后保留的"开发友好"默认值。Spring Boot 启动时会打印一行 WARN 提醒你 OSIV 是开的(spring.jpa.open-in-view is enabled by default. Therefore, database queries may be performed during view rendering.),原因是关掉它会让"刚学 JPA、还没掌握 DTO 投影"的开发者立刻被 LazyInitializationException 劝退。这是面向初学者的体验取舍,不是面向生产的最佳实践。任何上线的项目都应该明确 spring.jpa.open-in-view=false,然后在 Service 层强制 DTO 边界 —— 这一行配置 + 一次性把所有 Controller 的返回类型改成 DTO,是最值得做的"一次性投资"。我们团队曾经因为没改这一行,在双 11 当天 Jackson 序列化 8000 个客户列表触发了 8000 + 80000 = 8.8 万条 SQL,DB 直接崩 20 分钟。

12.4 修复 1 · JOIN FETCH

JOIN FETCH 是 N+1 的"第一直觉修复" —— 既然 N+1 的本质是"集合没和父行一起加载",那就在 JPQL 里写一个 join fetch,Hibernate 把父子关联折叠到一条 SQL 里执行。这是修复中最直观、SQL 最少的方案,但也最有"踩雷"潜力 —— 集合 + 分页、多 List 集合都会让它原地翻车。

图 12-4-1 · JOIN FETCH 的执行路径

12.4.1 实现 —— 一条 SQL 搞定父 + 子

List<Customer> customers = em.createQuery("""
    select distinct c from Customer c
    join fetch c.orders
    where c.status = :status
    order by c.id
    """, Customer.class)
    .setParameter("status", CustomerStatus.ACTIVE)
    .getResultList();

// 这次循环不再发任何 SQL —— orders 已经在第一条 SQL 里被装配好了
customers.forEach(c -> c.getOrders().size());
select distinct c.id, c.name, c.status,
                o.id, o.customer_id, o.amount
from    customers c
join    orders    o on o.customer_id = c.id
where   c.status = ?
order   by c.id;
[SQL 数]            1                 (从 11 → 1)
[customers 行]      10                (去重后)
[joined 物理行]     50                (10 × 5 平均)
[orders 已初始化]   true              (无需额外 SQL)
[网络往返]          1
[内存代价]          50 行物理结果集 → 10 个 Customer + 50 个 Order 对象

12.4.2 JOIN FETCH 的边界 —— 它的"不适用"清单更值得记

维度适用不适用 / 翻车场景
集合数1 个 OneToMany 集合2 个 List bag → MultipleBagFetchException
分页否(本质冲突)setMaxResults + JOIN FETCH → HHH90003004 内存分页警告
详情查询单实体详情(订单详情含订单行)列表页 + 大集合(10 客户 × 1000 订单 = 1 万行)
父 × 子行积可控(< 数千行)行积爆炸 —— 内存 / 网络都吃不消
distinct 必加OneToMany 必须 select distinct不加 distinct,父行被笛卡尔放大

12.4.3 集合 + 分页冲突 —— 必须明白的"为什么"

反例:JOIN FETCH + 分页

em.createQuery("""
    select distinct c from Customer c
    join fetch c.orders
    """, Customer.class)
    .setFirstResult(0)
    .setMaxResults(20)
    .getResultList();

Hibernate 抛 WARN:HHH90003004 firstResult/maxResults specified with collection fetch; applying in memory.
实际行为:先把 所有匹配父子笛卡尔行全捞回 JVM,再在内存里翻页。
100 万客户 × 10 订单 = 1000 万行进 JVM —— OOM。

正例:两步查 —— 分页 + 二次抓取

// 第 1 步:只分页父表,不 fetch
List<Long> ids = em.createQuery("""
    select c.id from Customer c
    where c.status = :s order by c.id
    """, Long.class)
    .setParameter("s", CustomerStatus.ACTIVE)
    .setFirstResult(0).setMaxResults(20)
    .getResultList();

// 第 2 步:用 id IN 查父+子,无分页
List<Customer> cs = em.createQuery("""
    select distinct c from Customer c
    join fetch c.orders
    where c.id in :ids
    """, Customer.class)
    .setParameter("ids", ids)
    .getResultList();

SQL 数:2 条 —— 第 1 条数据库分页,第 2 条 IN 抓取。
无内存分页警告,无 OOM 风险。

看到 HHH90003004 警告日志,把它当成 ERROR 看待。这条警告的真实含义是 "Hibernate 已经在内存里给你做了分页,SQL 没分页",当父子行积小(几百行)时你看不出问题,等到生产数据量上来一次性千万行进 JVM 直接 OOM。生产环境必须把 HHH90003004 配成日志告警,或者更狠 —— 在测试里直接断言这条日志不出现(用 ListAppender 抓 logger output 断言)。

12.5 修复 2 · @EntityGraph

JOIN FETCH 写在 JPQL 字符串里,但 Spring Data Repository 的"派生方法"(findByStatusOrderByIdAsc)是没有 JPQL 字符串可写的 —— 它的 SQL 是从方法名自动派生的。这时 @EntityGraph 是唯一的"在派生方法上声明加载形状"的方式。

图 12-5-1 · @EntityGraph 修复路径

12.5.1 Repository 方法 + @EntityGraph 注解

public interface CustomerRepository extends JpaRepository<Customer, Long> {

    @EntityGraph(attributePaths = "orders")
    List<Customer> findDistinctByStatusOrderByIdAsc(CustomerStatus status);
}

// 调用方
List<Customer> customers = repository
    .findDistinctByStatusOrderByIdAsc(CustomerStatus.ACTIVE);
customers.forEach(c -> c.getOrders().size());
select distinct c.id, c.name, c.status,
                o.id, o.customer_id, o.amount
from    customers c
left    join orders o on o.customer_id = c.id
where   c.status = ?
order   by c.id;
[SQL 数]      1
[fetch plan]  jakarta.persistence.fetchgraph: orders
[空集客户]    保留 (LEFT JOIN,没有 orders 的客户也返回)
[与 JOIN FETCH 的差异]
              · JOIN FETCH 是 INNER JOIN(默认),无订单的客户被过滤掉
              · @EntityGraph 是 LEFT OUTER JOIN,无订单的客户也保留
              · 业务上前者更常错杀,后者更安全

12.5.2 LOAD vs FETCH graph —— 易错点

类型hint key语义陷阱
FETCH graphjakarta.persistence.fetchgraph只加载图里列出的属性,其他全 LAZY★ 推荐 —— 严格控制
LOAD graphjakarta.persistence.loadgraph图里的强制 EAGER,图外的按 mapping 默认走EAGER 默认值会漏网,N+1 重新出现
Spring Data 默认FETCH graph同上—
显式声明type = EntityGraph.EntityGraphType.LOAD切换到 LOAD 语义除非确定要,否则不切
EntityGraph 和 JOIN FETCH 不是"二选一",而是"分工不同"。同一个 Repository 接口里,详情查询用 @Query + JPQL JOIN FETCH(SQL 完全可控、可读、能加复杂 where),列表查询用派生方法 + @EntityGraph(保留方法名语义、避免一堆 @Query 字符串)。两者底层都走 LoadQueryInfluencers 的同一条路径,生成的 SQL 形状几乎一致 —— 选哪个看的是代码可维护性,不是"哪个更快"。

12.6 修复 3 · @BatchSize

JOIN FETCH 和 EntityGraph 都需要"在查询时就知道要不要加载关联"。但生产里很多场景不知道 —— Service 把实体往上传,有的接口要展开 orders,有的不要,提前 fetch 是浪费,不 fetch 又会 N+1。@BatchSize 是这个矛盾的兜底解:让首次访问 LAZY 集合时,Hibernate 把"所有同类待初始化的 owner"打包成 IN 一次拉回。

图 12-6-1 · @BatchSize 的批量补取路径

12.6.1 注解配置 —— 一行注解,SQL 数从 N 降到 ceil(N/size)

@OneToMany(mappedBy = "customer", fetch = FetchType.LAZY)
@BatchSize(size = 32)
private List<Order> orders = new ArrayList<>();

// 业务代码不变,N+1 自动变 1 + ceil(N/32)
List<Customer> customers = repository
    .findTop100ByStatusOrderByIdAsc(CustomerStatus.ACTIVE);

for (Customer c : customers) {
    int size = c.getOrders().size();
}
-- [SELECT 1] 父查询
select id, name, status from customers
where  status = ? order by id fetch first 100 rows only;

-- [SELECT 2..4] 每批 32 个 customer_id 拉一次
select id, customer_id, amount from orders
where  customer_id in (?, ?, ..., ?);   -- 32 个占位符

-- [SELECT 5] 剩余 4 个 customer_id
select id, customer_id, amount from orders
where  customer_id in (?, ?, ?, ?);
[父行]      100
[batch]     32
[SQL 总数]  1 + ceil(100 / 32) = 1 + 4 = 5
[相比 N+1]  101 → 5,降 95%
[相比 JOIN FETCH]  虽然多 4 条 SQL,但每条 SQL 走 PK 索引极快
                   且无父子笛卡尔行积,内存压力更小

BatchSize 不是越大越好 —— 16-32 是甜蜜区。BatchSize 太大(如 1000)有三个隐藏代价:(1) IN 子句变长,某些数据库(Oracle 11g)有 IN 列表 1000 条上限,直接报错;(2) 单条 SQL 解析时间增长,prepared statement plan cache 命中率下降;(3) 一次拉回大量数据进 JVM,GC 压力陡升。生产经验:OneToMany 集合 16-32,ManyToOne to-one 关联 32-50,几乎覆盖所有场景。再大就没收益了。

12.7 修复 4 · DTO 投影

前三种修复都是在"返回实体"的前提下优化加载。但很多 API 根本不需要实体 —— 它们只要"客户名 + 订单数 + 总金额"这种聚合视图,managed entity 的脏检查、Proxy、PersistenceContext 全是浪费。DTO 投影是 N+1 修复里最干净、最不会复发、性能最好的方案 —— 缺点只有一个:DTO 不能修改回 DB。

图 12-7-1 · DTO 投影路径

12.7.1 record 投影 —— Java 17+ 最推荐写法

public record CustomerSummaryDto(
    Long       customerId,
    String     name,
    Long       orderCount,
    BigDecimal totalAmount
) {}

List<CustomerSummaryDto> summaries = em.createQuery("""
    select new com.acme.CustomerSummaryDto(
        c.id, c.name, count(o.id), coalesce(sum(o.amount), 0)
    )
    from   Customer c
    left   join c.orders o
    where  c.status = :status
    group  by c.id, c.name
    order  by c.id
    """, CustomerSummaryDto.class)
    .setParameter("status", CustomerStatus.ACTIVE)
    .getResultList();
select c.id, c.name,
       count(o.id), coalesce(sum(o.amount), 0)
from   customers c
left   join orders o on o.customer_id = c.id
where  c.status = ?
group  by c.id, c.name
order  by c.id;

[返回]     List<CustomerSummaryDto>
[特点]     无 Proxy / 无 PersistenceContext / 不可修改 / 序列化稳定
[内存]     每条只持有 4 个字段,远小于 Entity + 关联

12.9 检测与回归

修一次 N+1 不难,难的是让它不再回来。N+1 是典型的"写代码时看不见 bug 但跑起来才出 SQL"的反模式 —— 没有回归测试守护,半年后总会有同事改一段看似无关的代码就让它复活。本节给出三层守护:Hibernate Statistics、SQL 形状断言、OSIV 关闭测试。

图 12-9-1 · N+1 检测与回归链路

12.9.1 启用 Hibernate Statistics

logging.level.org.hibernate.SQL=DEBUG
logging.level.org.hibernate.orm.jdbc.bind=TRACE
@Autowired
EntityManagerFactory emf;

private Statistics stats() {
    return emf.unwrap(SessionFactory.class).getStatistics();
}

@BeforeEach
void resetStatistics() {
    stats().clear();
}

12.9.2 SQL 数量断言 —— 最简单也最有效的回归

@Test
void customerListShouldNotRegressToNPlusOne() {
    long before = stats().getPrepareStatementCount();

    List<Customer> customers = repository.findTop100ByStatusOrderByIdAsc(
        CustomerStatus.ACTIVE);
    customers.forEach(c -> c.getOrders().size());

    long delta = stats().getPrepareStatementCount() - before;

    // 100 个父 + BatchSize=32 → 1 + ceil(100/32) = 5
    assertThat(delta)
        .as("SQL count must stay bounded; if this fails, N+1 has regressed")
        .isLessThanOrEqualTo(8L);
}

12.9.3 SQL 形状断言 —— 不只看数量,还看形状

@Test
void detailQueryShouldUseJoinFetch() {
    sqlRecorder.clear();

    repository.findDetailById(10L).orElseThrow();

    String sql = sqlRecorder.singleSelectSql();

    assertThat(sql).contains("join orders");
    assertThat(sql).doesNotContain("where customer_id = ?");
}

    String json = restTemplate.getForObject("/customers/10", String.class);

12.10.1 设计取舍

取舍 1:LAZY 默认 (集合) / EAGER 默认 (to-one) —— 规范的历史包袱

JPA 规范规定 @OneToMany / @ManyToMany 默认 LAZY,@ManyToOne / @OneToOne 默认 EAGER。前者源于"集合通常很大,提前加载浪费",后者源于"单关联通常都要用,延迟加载麻烦"。但生产里 @ManyToOne 默认 EAGER 是 N+1 最常见的隐藏种子 —— 列表页查 100 个 Order,每个 Order 自动 EAGER 加载 Customer,1 + 100 条 SQL 就这么悄悄发出去。生产标准是 所有关联都显式声明 LAZY,需要时再用 JOIN FETCH / EntityGraph 显式拉。

取舍 2:BatchFetchQueue 批量收集 —— 效率与简单的折中

Hibernate 在 PersistenceContext 中收集"同类未初始化代理",首次访问时按固定 batch size 查询。代价是批次大小是静态配置,不能动态适应数据形状(比如有时父行 5 个、有时 5000 个,batch=32 在两边都不是最优)。但动态批次的复杂度收益不大,Hibernate 选了"配置简单"这一条。生产里 16-32 是经验最优值。

取舍 3:Hibernate 不自动改写 LAZY 为 EAGER —— "加载形状是业务责任"

Hibernate 检测到 N+1 时不会自动加 JOIN —— 这是有意为之。理由:决定加载形状的责任属于业务层(详情查询要拉所有关联,列表查询不要),框架擅自加 JOIN 可能触发笛卡尔行积、内存爆炸。Hibernate 的策略是"日志警告 + 让你测试断言",而不是"自动修"。这是务实的取舍。

12.11.2 补救范式 —— 四个生产里反复用到的修复套路

反例:Lombok @Data 直接放在 Entity 上

@Entity
@Data  // 自动生成 toString / equals / hashCode
public class Customer {
    @OneToMany(mappedBy = "customer")
    private List<Order> orders;
}

toString() 会遍历 orders,触发 LAZY 初始化 → N+1。
equals/hashCode 也含集合 → HashSet 操作触发 N+1。
更糟:双向关联时无限递归 StackOverflow。

正例:精确控制 Lombok 范围

@Entity
@Getter @Setter
@ToString(exclude = "orders")
@EqualsAndHashCode(of = "id")
public class Customer {
    @OneToMany(mappedBy = "customer")
    @ToString.Exclude
    private List<Order> orders;
}

toString 不包含集合 → 调试 / 日志安全。
equals/hashCode 只用 id → HashSet 安全 + 无 N+1。
Entity 上禁用 @Data 是团队规范级红线。

排查 N+1 时,先看异常栈里有没有 toString —— 不少"诡异 N+1"的根因都是它。生产里见过最离谱的一次:某个监控埋点写了 log.debug("loaded customer: {}", customer),DEBUG 级别在生产是关的,代码里看上去没问题。后来某次升级把 logging 级别全开成 DEBUG 排障,这一行的 toString 触发了 customer.orders 的初始化,1 万客户的接口直接发了 1 万 + 1 条 SQL,DB 当场崩。Entity 的 toString 必须排除所有集合字段,这是团队规范级别的硬性要求。

@Test
void listShouldUseFetchPlan() {
    long before = stats().getPrepareStatementCount();

    List<Customer> result = repository.findDistinctByStatusOrderByIdAsc(
        CustomerStatus.ACTIVE);

        .mapToInt(c -> c.getOrders().size())
