第 5 章 · 主键生成策略

IDENTITY · SEQUENCE · TABLE · UUID v7 · Snowflake · @EmbeddedId · Optimizer · allocationSize · ID gap · ActionQueue · JDBC batch · 时钟回拨 · worker id

主键生成不是"随便挑一个 @GeneratedValue"的小事——它是写入吞吐、批处理能力、索引形态、跨服务可扩展性的共同入口。一行 strategy = GenerationType.IDENTITY 可以把一个本来能跑 5 万 TPS 的批量导入压缩到 5000 TPS,因为 IDENTITY 在 JDBC 层根本无法 batch;一行 allocationSize = 50 写错成 allocationSize = 1,会让 SEQUENCE 的"号段预分配"退化成"每条 INSERT 一次 nextval";一个 char(36) 的 UUID 列会让 B-tree 索引比 binary(16) 大上 56%,在亿行表里直接吃掉几十 GB 磁盘和缓存。本章把五种策略 (IDENTITY / SEQUENCE / TABLE / UUID v7 / Snowflake) 与复合主键 @EmbeddedId 拆开,把每一种策略的"生成时机 / 是否能 batch / Optimizer 行为 / 跨数据库支持 / 失败模式 / 治理要点"讲清楚,目标是让你下一次 PR review 时,看到 @GeneratedValue 就能预测它会生成多少 SQL、能不能进 ActionQueue、会不会留 ID gap。

主键策略的选择不仅影响 SQL,还决定实体进入 PersistenceContext 的时机——这是很多人没意识到的底层耦合。IDENTITY 因为 ID 来自 INSERT 后的 getGeneratedKeys(),Hibernate 必须立即执行 INSERT 才能拿到 ID,因此不能延迟到 flush,也就不能合并多条 INSERT 进 JDBC batch;SEQUENCE 与 UUID 在 INSERT 之前就拿到 ID,可加入 ActionQueue 排队,等到 flush 时再统一发出 batch。同样的 persist() 调用,因为 ID 策略不同,产生的 SQL 数量、提交延迟、数据库压力可以差一个数量级。这种"注解一字之差,生产表现天壤之别"的情况在 JPA 里反复出现,本章是其中代价最大的一节——主键改动几乎一定要伴随数据迁移,选错就是几个月起步的债务。

  • 四种 JPA 标准策略 (IDENTITY / SEQUENCE / TABLE / AUTO) 的生成时机、可 batch 性、跨方言支持矩阵
  • IDENTITY 为什么"配了 hibernate.jdbc.batch_size=50 也没用"——Identity Map 与 EntityKey 注册的根本约束

  • TABLE 策略真正的代价是全局行锁——为什么"兼容性最好"换来的是"扩展性最差"

5.0 章首速览

主键策略的核心维度是"ID 在 INSERT 之前就已知,还是 INSERT 之后才知道"——这一条决定了 Hibernate 能不能把多个 persist() 缓存到 ActionQueue,等到 flush 时统一发 JDBC batch。其他差异 (跨方言支持、Optimizer 行为、索引局部性、ID 大小) 都是次生的工程取舍。

"ID 何时已知"是主键策略的第一区分维度——这不是性能微调,而是 Hibernate 内部能不能把 INSERT 进 ActionQueue 排队的根本约束。IDENTITY 必须 INSERT 后才能拿到 ID,因此 persist() 必须立即触发 INSERT,否则后续的 EntityKey 注册、级联引用、equals/hashCode 全都没法做;SEQUENCE 与 UUID 在 INSERT 之前就能拿到 ID,可以延迟到 flush 时统一执行,从而合并 JDBC batch。同样的循环 for (int i; i<1000; i++) em.persist(...),IDENTITY 会发 1000 条独立 INSERT,SEQUENCE 在 allocationSize=50 下只发 20 个 batch——后者的吞吐量在生产中通常是前者的 5-10 倍。选 IDENTITY 等于"主动放弃 batch 能力",这一点必须在选型时就清楚。

5.1 IDENTITY:配置最简单,代价最隐蔽

IDENTITY 是大多数教程默认的策略——"数据库自增,完事了"。它的价值在于 DDL 简单 (bigint AUTO_INCREMENT 或 bigserial),开发者不需要关心 ID 从哪里来,数据库直接保证唯一性和单调性。但这种"什么都不用配"的便利,代价是把整条写入路径锁死在"一条 SQL 一次往返"上——这是隐藏在配置便利背后的真实成本,而且不读源码的话根本意识不到。

IDENTITY 时序

图 5-1-1 · IDENTITY 策略下 persist 与 INSERT 的时序

5.1.1 IDENTITY 禁用 JDBC Batch

@Entity
@Table(name = "products")
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;
}

for (int i = 0; i < 3; i++) {
    em.persist(new Product("P-" + i));
}

观察日志:

[SQL] insert into products (name) values (?)
[SQL] insert into products (name) values (?)
[SQL] insert into products (name) values (?)
[batch effective] 0

即使配置 hibernate.jdbc.batch_size=50,IDENTITY 策略下每次 persist() 必须立即 INSERT 才能拿到自增 ID,JDBC 层无法合并多条 INSERT。原因在于 Hibernate 必须在 persist() 返回前为实体分配 EntityKey,否则后续级联、双向引用、equals/hashCode 行为均无定义。这不是 Hibernate "做得不够好",而是 Identity Map 模式与"ID 来自 INSERT 后"两个约束的逻辑必然。

5.1.2 IDENTITY 适用场景

场景适用理由
单库 OLTP CRUD是配置简单,QPS 不高时差异不明显
报表数据导入否无法 batch,几十万行 INSERT 几十分钟
事件流落库否写入热路径,batch 收益巨大
多服务共享表否应用侧无法预知 ID,关联映射困难
单库批量同步否改 SEQUENCE 收益显著
"我们用了 hibernate.jdbc.batch_size=50,为什么 batch 没生效"——99% 都是 IDENTITY。这是 JPA 群里出现频率最高的问题之一,几乎所有人都先怀疑 batch 配置写错了、driver 不支持、事务边界有问题,最后才发现根因是主键策略。Hibernate 6 在启动时会发出 HHH000404: Disabling contextual LOB creation 之外没有显式警告说"你这是 IDENTITY,batch 不会生效"。生产规则:任何"批量写入 > 1000 行 / 单事务"的实体,都不允许用 IDENTITY;PR review 时遇到"批量导入 + IDENTITY"的组合直接打回。

5.2 SEQUENCE:单库高吞吐首选

SEQUENCE 是单库高吞吐场景下的标准答案——它通过"预分配号段"把数据库往返从"每条 INSERT 一次"压缩到"每 N 条 INSERT 一次 nextval",同时保留 JDBC batch 能力。Hibernate 6 在 PostgreSQL / Oracle / DB2 / H2 上默认就走这条路。代价是 SEQUENCE 在 MySQL 5.x 上不被原生支持,需要走 SequenceStyleGenerator 用辅助表模拟(本质退化为 TABLE 策略)。

SEQUENCE 预分配机制

图 5-2-1 · SEQUENCE 预分配号段后的 INSERT batch

5.2.1 SEQUENCE 配置示例

@Entity
@Table(name = "sales_orders")
public class SalesOrder {

    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "order_seq")
    @SequenceGenerator(
        name = "order_seq",
        sequenceName = "order_id_seq",
        allocationSize = 50)
    private Long id;

    private Instant createdAt;
}

数据库 DDL:

create sequence order_id_seq increment by 50 start with 1;

create table sales_orders (
    id          bigint not null,
    created_at  timestamp(6),
    primary key (id)
);

运行 SQL:

[SQL] select nextval('order_id_seq')
[SQL] insert into sales_orders (created_at, id) values (?, ?)
[SQL] insert into sales_orders (created_at, id) values (?, ?)
[SQL] insert into sales_orders (created_at, id) values (?, ?)
[batch] 50 条 INSERT 合并为 1 个 JDBC batch

这里发生的事情值得仔细看:Hibernate 调一次 nextval('order_id_seq') 拿到一个值(比如 1050),然后反推出本批可用范围 [1001..1050],接下来 50 次 persist() 都直接从内存范围里取 ID,完全不再访问数据库,直到第 51 次 persist 才会再发一次 nextval 取下一个号段 [1051..1100]。在 1 万行批量写入下,数据库的 nextval 调用从"1 万次"压缩到"200 次",同时 50 条 INSERT 进了一个 JDBC batch——这就是 SEQUENCE + Optimizer 把吞吐量推到 IDENTITY 的 5-10 倍的真实机制。

5.2.2 allocationSize 与 INCREMENT BY 必须一致

@SequenceGenerator(
    name = "product_seq",
    sequenceName = "product_id_seq",
    allocationSize = 50)             // 应用侧期望 50
create sequence product_id_seq increment by 1;   -- DB 实际 1

Hibernate 6 在启动时校验该差异:

org.hibernate.MappingException:
The increment size of the [product_id_seq] sequence is set to [50]
in the entity mapping while the associated database sequence
increment size is [1].

旧版本未校验时,多 JVM 同时预取相同 ID 范围,最终触发主键冲突。修复方式:数据库与应用配置同步变更,migration 脚本里 create sequence ... increment by 50 与 Java 注解里的 allocationSize = 50 必须成对。Liquibase / Flyway 的 changelog 里改一处必须连带改另一处,review 时检查"两个数字是否一致"是必查项。

Optimizer 选型

Optimizer行为适用场景风险
no-op (allocationSize=1)每次 INSERT 前一次 nextval对 ID 连续性要求高吞吐低
pooled取号段后内存分配,连续段高写入单 JVM重启 / 回滚 ID gap
pooled-lo取号段后从 low 端递增多 JVM 高吞吐同 pooled
自定义业务号段治理多租户分段需证明并发不重叠

ID gap 是 SEQUENCE + Optimizer 的预期行为。预取号段后未用完即被新事务跳过、应用重启遗留范围、事务回滚均产生 gap。生产监控应关注唯一性、单调性、写入延迟,而非 ID 连续性。要"ID 一定连续"只能 allocationSize=1 + 关闭 Optimizer,代价是吞吐回到 nextval-per-insert 的水平,通常得不偿失。

ID gap 不是 bug,是 SEQUENCE 的"原生定价"——pooled Optimizer 一次取 50 个号段后,只用了 30 个就回滚 / 重启 / 切换 leader,剩下 20 个号段就此作废。这种 gap 在亿行表里的累计可能是几百万 ID 缺口,看上去很可怕,但只要 ID 类型是 bigint (容量 9.2×10¹⁸),完全没必要担心耗尽。真正应当告警的是"ID 不唯一"而不是"ID 不连续"。把这一点写进生产 SOP 能避免运维同事每隔几个月就跑来问"为什么 ID 跳了 50 个"。

5.3 TABLE:兼容性最高,扩展性最低

TABLE 策略用一张辅助表 + 行锁模拟序列——它的卖点是"任何关系库都能跑",连不支持 sequence 的 MySQL 5.x 也能用。代价是所有需要分配 ID 的节点都得在同一行上 SELECT ... FOR UPDATE,并发写入退化为单线程串行。在小规模兼容性要求严苛(比如老项目用 MyBatis 转 JPA、跨多个数据库厂商部署)时是个保底选项,但在任何高并发场景里都不该出现。

TABLE 行锁瓶颈

图 5-3-1 · TABLE 策略下的全局行锁串行化

5.3.1 TABLE 配置示例

@Entity
@Table(name = "payments")
public class Payment {
    @Id
    @GeneratedValue(strategy = GenerationType.TABLE, generator = "payment_gen")
    @TableGenerator(
        name = "payment_gen",
        table = "id_generator",
        pkColumnName = "gen_name",
        valueColumnName = "gen_value",
        pkColumnValue = "payment",
        allocationSize = 50)
    private Long id;
}

辅助表 DDL:

create table id_generator (
    gen_name   varchar(64) primary key,
    gen_value  bigint not null
);

insert into id_generator values ('payment', 0);

运行 SQL:

select gen_value
from id_generator
where gen_name = 'payment'
for update;

update id_generator
set gen_value = ?
where gen_name = 'payment';

行锁在 SELECT FOR UPDATE 时获取,UPDATE 完成后释放。所有需要分配 ID 的节点串行化在该行上。100 个并发 INSERT 退化为单线程串行——即使 allocationSize=50 把锁竞争降低 50 倍,这一行仍然是整个集群写入吞吐的硬上限。

三种数据库内策略对比

维度IDENTITYSEQUENCETABLE
应用侧预知 ID否是是
JDBC batch否是是 (有锁开销)
数据库支持部分 (MySQL/PG/H2)部分 (PG/Oracle/DB2/H2)全部
高并发吞吐中高低

TABLE 是"最差选择中的最佳选择",而不是"通用兜底"。它存在的全部理由是"必须支持那个不支持 sequence 的老库" + "MySQL 5.5 之前不支持 AUTO_INCREMENT 跨表 ID"。一旦目标库是 PostgreSQL / Oracle / MySQL 8.0+,就没有理由选 TABLE——SEQUENCE 在所有这些库上都更好。生产里见过的 TABLE 滥用基本都是"教程抄过来的",根本没人想过它的代价是 100 倍写入延迟下降。

5.4 UUID:去中心化生成

当 ID 不再来自单一数据库,而是来自多个独立服务、移动端、边缘节点时,UUID 是事实标准。它的核心价值是"不需要协调就能保证全局唯一"——每个节点本地生成,没有 nextval 往返、没有 sequence 锁竞争、没有 worker id 治理负担。代价是 16 字节比 8 字节 bigint 大一倍,而 UUID v4 的全随机性会让 B-tree 索引疯狂页分裂。RFC 9562 (2024) 标准化的 UUID v7 用前 48 位嵌入毫秒时间戳解决了索引局部性问题,Hibernate 6 内建了 @UuidGenerator(style = TIME) 直接生成 v7,无需第三方库。

5.4.2 Hibernate 6 UUID 生成器

import org.hibernate.annotations.UuidGenerator;

@Entity
@Table(name = "event_records")
public class EventRecord {

    @Id
    @UuidGenerator(style = UuidGenerator.Style.TIME)   // UUID v7 时间有序
    private UUID id;

    @Column(nullable = false)
    private String payload;
}

生成 DDL (PostgreSQL):

create table event_records (
    id       uuid primary key,
    payload  varchar(255) not null
);

生成 DDL (MySQL):

create table event_records (
    id       binary(16) primary key,
    payload  varchar(255) not null
);

运行 SQL (UUID 在 INSERT 前生成,JDBC batch 可用):

insert into event_records (payload, id) values (?, ?);
-- 多条 INSERT 进入 JDBC batch

UUID 存储形态选型

数据库推荐列类型字节数索引性能备注
PostgreSQLuuid16最优内建类型,索引、JOIN、序列化都最快
MySQL 5.7+binary(16)16优紧凑,Hibernate 6 默认

Oracleraw(16)16优内建支持
H2uuid16优内建支持
从 char(36) 迁移到 binary(16) 的真实收益——某订单系统 1.2 亿行 events 表用 char(36) 存 UUID 主键 + 5 个 UUID 外键,主键 + 索引总占用 38 GB;迁到 binary(16) 后 17 GB,InnoDB Buffer Pool 命中率从 78% 升到 94%,P99 查询延迟从 45ms 降到 18ms。迁移过程是"加 binary(16) 列 + 后台 backfill + 切换索引 + 删旧列",分四步在线完成。这种"列类型选错会持续付几年的税"是 UUID 用 char(36) 最直观的代价证据。新表一律 binary(16) 或 PostgreSQL 的原生 uuid 类型,任何 PR review 看到 char(36) UUID 直接打回。
UUID v7 在 2024 年成为 RFC 9562 标准,Hibernate 6.2 起原生支持。在此之前生产里要用时间有序 UUID 通常引 cohort/uuid-creator 或 jchambers/ms-uuid 这种第三方库,或者用 ULID / TSID 替代。Hibernate 6.2 的 @UuidGenerator(style = TIME) 直接给出 v7 后,这些第三方依赖可以全部移除。Style.RANDOM = v4(随机)、Style.TIME = v7(时间有序)、Style.AUTO = 默认 v4(向后兼容旧版本行为)。生产新代码统一选 Style.TIME,把 v4 的"页分裂税"留给历史遗留表。

5.5 Snowflake:64-bit 分布式 Long

Snowflake 是 Twitter 开源的分布式 ID 算法,核心是把时间戳 + 机器号 + 序列号塞进一个 64-bit 整数,既保证全局唯一,又保留了 Long 类型的紧凑性 (8 字节 vs UUID 的 16 字节)。它在亿级 + 海量短链 / 订单 / 消息 ID 场景下是事实标准——美团 Leaf、百度 UidGenerator、Sony sonyflake 都是变体。代价是worker id 必须治理(否则两个节点拿到相同 worker id 直接产 ID 冲突),以及时钟回拨必须有应对方案(NTP 同步抖动 / 闰秒 / 跨时区迁移)。

Snowflake 字节布局

Snowflake 64-bit 字节布局:

   bit  63  62                         22  21       12  11        0
       ┌──┬─────────────────────────────┬─────────────┬───────────┐
       │S │       timestamp (41 bit)     │ worker (10) │ seq (12)  │
       └──┴─────────────────────────────┴─────────────┴───────────┘
        ↑           ↑                          ↑           ↑
       sign      毫秒时间戳                   机器号    毫秒内序列号
       =0         相对 epoch 偏移             1024 个     4096 个

  字段含义:
    sign       固定 0 (正数 Long)
    timestamp  当前毫秒 - epoch 起点 (约 69.7 年容量)
    worker     部署节点编号 (datacenter + machine)
    seq        同毫秒内的递增序列

5.5.1 Snowflake 自定义生成器

public class SnowflakeGenerator implements IdentifierGenerator {

    private final SnowflakeWorker worker = SnowflakeWorker.fromEnvironment();

    @Override
    public Object generate(SharedSessionContractImplementor session, Object entity) {
        return worker.nextId();
    }
}

@Entity
@Table(name = "distributed_orders")
public class DistributedOrder {

    @Id
    @GenericGenerator(name = "snowflake", type = SnowflakeGenerator.class)
    @GeneratedValue(generator = "snowflake")
    private Long id;
}

Snowflake 治理要点

维度必须保证失败模式
节点唯一worker id 启动期校验 (Zookeeper / Redis 注册)ID 冲突

Snowflake 的"难"不在算法,而在 worker id 治理与时钟回拨。几十行代码就能写出生成器,但生产里能稳跑的实现都做了三件事:(1) worker id 从中心化注册中心(ZK / 数据库行 / K8s StatefulSet 序号)获取,启动期 CAS 防冲突;(2) 检测到时钟回拨时,如果回拨 < 5ms 自旋等待,> 5ms 直接拒绝服务并报警(宁可挂掉也不发 ID);(3) 同毫秒 4096 序列号用尽时阻塞到下一毫秒,而不是抛异常。美团 Leaf-snowflake 的源码值得一读,把这些边界处理得很完整。"自己写一个 Snowflake"在面试里是加分项,在生产里是风险——优先用经过线上验证的开源实现。

5.6 复合主键:@EmbeddedId

"一行的身份由多列共同决定"——这种业务复合键在订单行 (订单号 + 行号)、用户角色绑定 (userId + roleId)、库存批次 (skuId + warehouseId + lotNo) 等场景里非常常见。JPA 提供两种映射方式:@EmbeddedId 把所有键字段聚合到一个 @Embeddable 类里,@IdClass 把键字段平铺在实体里另维护一个键类。两者功能等价,但 @EmbeddedId 在 OO 抽象、引用语义、@MapsId 兼容性上更优,新代码统一推荐 @EmbeddedId。

5.6.1 标准用法

@Embeddable
public class OrderLineId implements Serializable {

    private Long orderId;
    private Integer lineNo;

    protected OrderLineId() {}

    public OrderLineId(Long orderId, Integer lineNo) {
        this.orderId = orderId;
        this.lineNo = lineNo;
    }

    @Override
    public boolean equals(Object o) {
        return (o instanceof OrderLineId other)
            && Objects.equals(orderId, other.orderId)
            && Objects.equals(lineNo, other.lineNo);
    }

    @Override
    public int hashCode() {
        return Objects.hash(orderId, lineNo);
    }
}

@Entity
@Table(name = "order_lines")
public class OrderLine {

    @EmbeddedId
    private OrderLineId id;

    @MapsId("orderId")
    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    private SalesOrder order;

    @Column(nullable = false, precision = 19, scale = 2)
    private BigDecimal amount;
}

DDL:

create table order_lines (
    order_id  bigint not null,
    line_no   integer not null,
    amount    numeric(19, 2) not null,
    primary key (order_id, line_no)
);

查询 SQL:

select o1_0.order_id, o1_0.line_no, o1_0.amount
from order_lines o1_0
where o1_0.order_id = ?
  and o1_0.line_no = ?;

复合主键里 @MapsId 是"双声明去重"的关键——上面例子里 OrderLineId.orderId 与 OrderLine.order(指向 SalesOrder)都映射到 order_id 列。如果不用 @MapsId,要么就两个字段都设 @Column(name="order_id") 但 Hibernate 会抱怨"多个属性映射到同一列",要么就把 order 关联设 insertable=false, updatable=false 但语义不对(关联是只读的吗?不是)。@MapsId("orderId") 告诉 Hibernate "这个 ManyToOne 的外键值与 ID 里的 orderId 是同一个",生成的 SQL 自然合并成一列。这是 JPA 关联映射里最容易被忽视但最优雅的注解之一,第 6 章会再次详细讲。

5.7 源码深潜

Hibernate 把"主键生成"抽象成 IdentifierGenerator 接口,所有内置策略 (IdentityGenerator / SequenceStyleGenerator / TableGenerator / UuidGenerator) 与用户自定义生成器都实现这一接口。EntityPersister 在 persist() 调用栈里通过 ID 是否在 INSERT 前已知分流到两条路径:before-execution(已知 ID,进 ActionQueue 等 flush)与 on-execution(未知 ID,立即 INSERT 取 generatedKeys)。这条分流是 IDENTITY 不能 batch 的源码层根因。

5.7.1 SequenceStyleGenerator 内部流程

SequenceStyleGenerator.generate(session, entity)
  └─ Optimizer.generate(accessCallback)
     ├─ 检查内存池
     │  ├─ 池中有可用 ID → 直接返回
     │  └─ 池耗尽 → accessCallback.getNextValue()
     │              └─ select nextval('order_id_seq')
     └─ return 内存池中的下一个 ID

5.7.2 设计取舍

取舍 1:IDENTITY 与 PC 一致性

IDENTITY 必须立即 INSERT 才能获取 ID。该约束源于 Identity Map:Hibernate 不能将无 ID 的 Managed 实体长期保留在 PC 中,否则同 ID 的双重持久化、级联引用、equals 行为均无定义。代价是高写入路径无法 batch。

取舍 2:SEQUENCE 预取与 ID gap

Optimizer 通过预取号段减少数据库往返。代价是事务回滚、应用重启会留下 ID gap。该 gap 是设计取舍的可观察结果,而非数据丢失。强行实现 ID 连续性 (例如关闭 Optimizer 或 allocationSize=1) 牺牲吞吐量,在多数业务场景中得不偿失。

取舍 3:UUID 索引局部性

UUID v4 全随机分布带来全局唯一,代价是 B-tree 页分裂频繁。UUID v7 在前 48 位嵌入毫秒时间戳,使得新写入接近右端追加,索引性能接近自增。代价是 UUID v7 仍是 16 字节,比 8 字节 bigint 大一倍。

取舍 4:Snowflake 部署治理

Snowflake 算法本身简单,真正的工程难度在于 worker id 唯一性、时钟监控、epoch 选择、序列号溢出处理。未经过部署治理的 Snowflake 在多副本环境下可能产生重复 ID。

5.8 性能基准

选型不能只看"听起来快不快",必须看实测数据。下面是一组在 PostgreSQL 14、典型生产服务器、10 万行 INSERT 的批量写入下的实测对比——目标是把"哪种策略快多少"从感性认知变成可被引用的数字。

"没配 batch_size 的 SEQUENCE = 比 IDENTITY 还慢"——这是上表里最反直觉的一条。SEQUENCE 默认 allocationSize=50,但如果 persistence.xml 或 application.yml 里漏配 hibernate.jdbc.batch_size=50,JDBC 仍然不会合并 INSERT,结果就是每条 INSERT 前先 SELECT 一次 nextval(虽然有内存池摊薄),然后单条 INSERT,比 IDENTITY 多了 nextval 调用。SEQUENCE 的优势必须 batch_size + allocationSize 双配齐才能体现,缺一不可。生产配置硬规定:hibernate.jdbc.batch_size=50 + order_inserts=true + order_updates=true + 主键 SEQUENCE allocationSize=50,四件套必须同步。

5.9 陷阱与限制

主键策略陷阱大致分四类:性能陷阱(配错策略 / 配错 size 让 batch 失效)、数据一致性陷阱(双 JVM 预取冲突 / 应用 ID 重复)、跨库迁移陷阱(AUTO 默认值在升级中漂移 / columnDefinition 写死方言)、分布式治理陷阱(Snowflake worker 冲突 / 时钟回拨)。下面用决策树 + 陷阱明细表把这些一次性列清楚。

AUTO 是"看起来最方便,实际最危险"的策略。它让 Hibernate 自己挑——在 Hibernate 4 / 5 / 6 之间,这个"挑选"逻辑悄悄变过几次:Hibernate 5 在 MySQL 上 AUTO = IDENTITY;Hibernate 6 在 MySQL 上 AUTO 可能变成 TABLE(基于"SequenceStyleGenerator 在不支持 sequence 的库上回退到 TABLE")。一次 Hibernate 大版本升级后,生产数据库突然多出 hibernate_sequences 表,数据库 DBA 跑来问"这表谁建的"——这就是 AUTO 漂移的典型现场。生产代码一律显式声明 strategy = GenerationType.SEQUENCE / IDENTITY / UUID,绝不写 AUTO,也绝不依赖 @GeneratedValue 的默认值。

@Test
void sequenceUsesOneNextvalPerAllocationBlock() {

    tx(() -> {

            em.persist(new SalesOrder(Instant.now()));
