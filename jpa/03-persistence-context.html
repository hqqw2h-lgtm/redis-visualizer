第 3 章 · 持久化上下文与一级缓存

PersistenceContext · Identity Map · Snapshot · Dirty Checking · ActionQueue · FlushMode · EntityEntry · PersistentBag · 字节码增强 · 批处理 · 唯一键替换

如果说 EntityManager 是 JPA 暴露给应用代码的"方向盘",那 PersistenceContext (PC) 就是方向盘背后那一整套传动系统——所有 find / persist / merge / remove 的"看似操作数据库",其实都先停在 PC 这块工作区里,等到 flush 才被翻译成真正的 SQL。理解 PC 的内部结构是 JPA 进阶的分水岭:不懂 PC,你只会看到"为什么我改了字段没生成 UPDATE"、"为什么我 remove 后又 persist 一下报唯一约束"、"为什么 100 万条插入会 OOM"这些零散现象;懂 PC,你能立刻指出 ActionQueue 顺序、Snapshot 比对、 Identity Map 命中、Collection 跟踪四个根因之一,并给出 flush + clear、字节码增强、FlushMode.COMMIT 这些精准修复。

本章把 PC 拆成四个内部结构来讲清楚:Identity Map(主键到对象的一对一映射)、Snapshot(加载时字段值的副本,用于 dirty checking)、ActionQueue(待执行的 INSERT/UPDATE/DELETE 队列)、持久化集合跟踪表(PersistentBag/Set 的元数据)。每一块都有它独立的生命周期与失效语义,任何一个用错都会出现"代码看上去对、SQL 行为出乎意料"的现象。我们不仅画出每块的内部布局图,还给出运行时观察方法、Hibernate 源码入口、生产调优的真实代价。最终目的是让你在排查 dirty checking 性能、批量导入 OOM、唯一约束冲突时,有一套可复用的决策路径,而不是每次靠"猜哪里不对"。

  • PC 的四大内部结构:Identity Map / Snapshot / ActionQueue / Collection 跟踪表 — 各自职责与 flush 时的角色

3.1 Identity Map

Identity Map 是 PC 的"地基"——所有其他三个组件都建立在"同主键映射到同 Java 对象"这条不变量之上。它的实现极其简单:一个以 EntityKey 为键、以实体引用为值的 HashMap。但这个简单结构带来的语义保证,是 JPA 整个对象模型一致性的基础。

图 3-1-1 · Identity Map 的命中(cache hit)与穿透(miss)
同一个 Java 引用 即 (c1 == c2) 永远为 true,这是 dirty checking / 级联 / 双向关联的基础

EntityKey 由实体类型、主键值、租户标识(多租户场景)三者共同组成。三者全部一致才视为同一对象,Hibernate 必须返回同一 Java 引用。这个机制使得"同一事务内同主键访问"始终获得一致视图——你不需要担心 find 两次拿到两个对象然后两边各自修改导致后者覆盖前者。

3.1.1 重复 find 的 SQL 行为

em.getTransaction().begin();

Customer a = em.find(Customer.class, 1L);   // SELECT
Customer b = em.find(Customer.class, 1L);   // 命中 Identity Map

boolean same = (a == b);                     // true

em.getTransaction().commit();
select c1_0.id, c1_0.email, c1_0.name
from customers c1_0
where c1_0.id = ?;
-- 第二次 find 无 SQL

第二次 find() 直接命中 Identity Map,返回首次加载的同一对象。你打开 Hibernate SQL 日志(hibernate.show_sql=true)能看到第二次没产生 SQL——这是诊断 PC 是否生效的最快方法。如果你期望命中却看到了第二条 SELECT,只能是两种情况之一:(1) PC 已经被 clear 或 close 清空;(2) 两次 find 在不同的 EM/事务里。

3.1.2 EntityKey 的字节布局

EntityKey 内部字段 (Hibernate ORM):

  ┌────────────────────────────────────────────┐
  │  EntityKey                                  │
  ├────────────────────────────────────────────┤
  │  identifier      Object   主键值             │
  │  persister       EntityPersister  实体元数据 │
  │  tenantId        String   多租户标识(可选)   │
  │  hashCode        int      预计算缓存          │
  │  rootEntityName  String   继承根类型名称      │
  └────────────────────────────────────────────┘

  hashCode 计算:
    identifier.hashCode() ^ rootEntityName.hashCode()

  equals 判定:
    identifier 相等 + rootEntityName 相等 + tenantId 相等

注意 rootEntityName 用的是继承根类而不是具体子类——这意味着 EntityKey(Manager, 1) 和 EntityKey(Employee, 1) 在 Manager extends Employee 的继承结构下被视为同一个 key,从而在多态加载时不会产生重复对象。hashCode 是预计算并缓存的,因为 EntityKey 经常被放入 HashMap 频繁查询,每次都重新计算 String/Object 的 hashCode 不划算。

3.1.3 跨 EM 的对象身份

不同 EntityManager 之间,同主键的实体是不同 Java 实例。这是 JPA 规范明确允许的——每个 EM 维护独立的 PC,不共享 Identity Map。这带来一个直接后果:equals/hashCode 不应依赖 ==,而应基于稳定业务键(订单号、邮箱、UUID)或明确的主键策略。

反例:依赖默认 Object.equals

@Entity
public class Customer {
    @Id @GeneratedValue
    private Long id;
    // 不重写 equals/hashCode
}

// 两个 EM 加载同一行:
Customer c1 = em1.find(Customer.class, 1L);
Customer c2 = em2.find(Customer.class, 1L);

c1.equals(c2);             // false (Object.equals = ==)

Set<Customer> cache = new HashSet<>();
cache.add(c1);
cache.contains(c2);        // false — 业务上是同一个客户却找不到

正例:基于稳定业务键

@Entity
public class Customer {
    @Id @GeneratedValue
    private Long id;

    @Column(unique = true)
    private String email;

    @Override
    public boolean equals(Object o) {
        return (o instanceof Customer other)
            && Objects.equals(email, other.email);
    }

    @Override
    public int hashCode() {
        return Objects.hashCode(email);
    }
}

// 跨 EM 也能正确判等
c1.equals(c2);             // true
cache.contains(c2);        // true

业务键 equals 在跨上下文比较中保持稳定,但要求业务键不可变。如果业务键可能变更(例如邮箱可改),则需使用 Hibernate 推荐的"使用主键 + 显式判空"模式——只要在对象进入 Set/Map 之前已经拿到了主键(即"已 persist 并 flush"),用主键就稳定。但要注意 Transient 状态的对象主键还没分配,放进 HashSet 会有问题。

Hibernate 官方文档关于 equals/hashCode 的明确指引 — Hibernate 6 的官方文档(《Hibernate User Guide》第 2.4.2 节)给出三条规则:(1) 不要使用主键作 equals 实现,因为 transient 对象主键尚未分配;(2) 不要同时使用所有字段,因为修改任意字段会破坏 hashCode 稳定性;(3) 应当使用业务键(natural identifier)——例如订单号、ISBN、邮箱+用户名组合。如果实体真的没有自然业务键(罕见),那么 Hibernate 推荐使用 UUID 作为额外标识符,在对象创建时就分配好。这条规则在 Vlad Mihalcea 的《High-Performance Java Persistence》一书中也专门用了一章讨论(Chapter 7 "Identity"),并给出了一段经典的"主键 equals + 是否已 persist 检查"的兜底实现。

3.1.4 Identity Map 的容量与监控

生产环境里 PC 中 Managed 实体数量是必须监控的指标。Hibernate 提供了 SessionStatistics API 让你拿到当前 PC 的统计:

SessionImplementor s = em.unwrap(SessionImplementor.class);
SessionStatistics stats = s.getStatistics();

int entityCount     = stats.getEntityCount();        // PC 中 Managed 实体数
int collectionCount = stats.getCollectionCount();    // PC 中跟踪的集合数

Set<EntityKey> keys = stats.getEntityKeys();         // 所有 EntityKey
for (EntityKey key : keys) {
    System.out.println(key.getEntityName() + " id=" + key.getIdentifier());
}

把这个数据接到监控系统(Micrometer / Prometheus),配置告警阈值——例如 entityCount > 10000 即触发 P3 告警,提醒检查是否有"全表加载到 PC"的反模式。这种监控比"等到 commit P99 飙升再排查"早 30 分钟拦下事故。

3.2 Dirty Checking 与 Snapshot

Dirty Checking 是 PC 的"魔法"——你只是调了个 setter,Hibernate 在 commit 时就自动生成了 UPDATE。这种"声明式更新"的实现机制就是 Snapshot 比对:实体加载时把当前字段值拍一份快照,flush 时对比当前字段值与快照,差异列就生成 UPDATE。

图 3-2-1 · Snapshot 比对生成 UPDATE 的全过程

3.2.1 loadedState 的存储

实体加载时,Hibernate 调用 EntityPersister.getPropertyValues() 读取所有持久化字段,存入 Object[] loadedState 数组。该数组与实体的 EntityEntry 共同保存在 PC 中。这个数组本质上是字段值的"影子"——和实体本身完全独立的另一份字节,这就是为什么 PC 越大内存占用越高:每个实体在 PC 内都对应一份等长的字段副本。

loadedState 数组布局:

  字段映射顺序由 EntityPersister 决定 (注解扫描顺序):

  loadedState[0] = "alice@example.com"   // email
  loadedState[1] = "Customer C1001"       // name
  loadedState[2] = 0L                      // version
  // id 不在 loadedState 中,单独存储

  flush 时调用:
    Object[] currentState = persister.getPropertyValues(entity);
    int[] dirtyProps = persister.findDirty(currentState, loadedState, ...);

  示例字段差异判定:
    loadedState  = ["a@old", "C1001", 0L]
    currentState = ["a@new", "C1001", 0L]
    findDirty 返回 [0]   ← 只有 email (索引 0) 不同
    生成 UPDATE 时仅更新 email 列 (开启 @DynamicUpdate)

3.2.2 Dirty Checking 的成本

em.getTransaction().begin();

Customer c = em.find(Customer.class, 1L);
c.setEmail("alice.new@example.com");

em.getTransaction().commit();
select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = ?;

update customers
set email = ?, name = ?, version = ?
where id = ? and version = ?;

flush 阶段的 dirty checking 时间复杂度为 O(N × F),其中 N 是 PC 中 Managed 实体数,F 是单实体持久化字段数。注意这是每次 commit 都要全量跑一遍——不是增量。100 个实体 × 20 个字段 = 2000 次比对,几乎可忽略;但 10 万个实体 × 50 个字段 = 500 万次,加上每次比对涉及反射 + 类型转换,典型耗时几百毫秒到几秒。这就是"长事务 + 大对象图"在生产里的典型瓶颈来源。

3.2.3 字节码增强示例

// build.gradle
plugins {
    id 'org.hibernate.orm' version '6.4.0.Final'
}

hibernate {
    enhancement {
        enableDirtyTracking = true
        enableLazyInitialization = true
    }
}

启用后,Hibernate 编译时改写实体字节码——具体来说,实体类的 setter 方法被注入"标记字段为 dirty"的代码,实体类还会自动实现 SelfDirtinessTracker 接口暴露 $$_hibernate_getDirtyAttributes()。flush 阶段不再需要全字段比对,直接读这个集合即可。生产环境中典型收益是 flush 耗时下降 30%-60%,具体值取决于 PC 大小:PC 越大、字段越多,字节码增强的收益越显著。

Spring Data JPA + Hibernate 字节码增强的真实收益 — Vlad Mihalcea 在他的博客上做过一组对比测试:对一个有 50 个字段的实体,加载 1000 个进 PC 然后修改其中 100 个,默认 Snapshot 比对的 flush 耗时约 480ms,启用字节码增强后降到 95ms,差距约 5 倍。但他也强调字节码增强不是"开了就赚到":(1) 增强后的实体类不能与未增强的混用,所有 entity jar 必须统一构建;(2) 增强会让实体类的 .class 文件膨胀约 20-40%,需要监控 Permgen / Metaspace;(3) 调试时打断点看到的字段值可能与代码里看起来不一致(因为有"影子字段");(4) 对于很多业务系统 PC 通常只有几百个实体,默认 Snapshot 已经足够快,过度优化反而增加构建复杂度。结论:只在确实定位到 dirty checking 是瓶颈、且无法通过短事务解决时才启用。

3.2.4 @DynamicUpdate 的副作用

@DynamicUpdate 让 Hibernate 在每次 UPDATE 时只更新真正变化的列。看起来是无脑收益,但有两个隐藏成本必须考虑:

  1. SQL 不能缓存 — Hibernate 默认会预编译每个实体的 INSERT/UPDATE SQL 并复用。开 @DynamicUpdate 后每次都要根据 dirty 列动态拼 SQL,prepared statement cache 命中率下降。
  2. 不减少 dirty checking 的 CPU 成本 — 你仍然要全字段比对才能知道哪些变了,只是比对结果用于"少 update 几列"而不是"少 sql 文本"。
  3. 对乐观锁不友好 — 如果有 @Version 列,version 列每次都要更新,实际"少更新的列"比想象中少。

真正的减负来自字节码增强 + 短事务两个维度。@DynamicUpdate 只在"实体有 50+ 字段且每次只改少数几个、并且绑定的索引列特别多导致 update 全列代价大"的特殊场景才显著有用。

3.3 Write-Behind 与 ActionQueue

JPA 的另一个"魔法":你 persist 一个新实体后,代码立刻继续往下跑,中间没有任何 SQL 出现——直到 flush 或 commit 才统一发出去。这就是 Write-Behind 模式,实现它的就是 ActionQueue:把所有"待执行的 SQL 操作"按类型放进有序队列,flush 时按固定顺序批量执行。

图 3-3-1 · ActionQueue 的六类 Action 与固定执行顺序

ActionQueue 不按代码顺序执行操作,而是按预设顺序:Insert → Update → Collection Remove → Collection Update → Collection Recreate → Delete。这个顺序经过设计,保护外键约束:新对象的 INSERT 先于旧对象的 DELETE,避免"DELETE 删掉了被 INSERT 引用的父行"的 FK 失败。但代价是"代码顺序等于 SQL 顺序"的直觉被打破——这是 ActionQueue 引发生产事故的根源。

3.3.2 AUTO 模式下的查询前 flush

em.setFlushMode(FlushModeType.AUTO);
em.getTransaction().begin();

Order order = new Order("ORD-1001");
em.persist(order);

    "select count(o) from Order o", Long.class)

insert into orders (order_no, status, id) values (?, ?, ?);

select count(o1_0.id) from orders o1_0;

AUTO 模式确保 JPQL 查询能看到当前事务内的修改——这就是为什么你 persist 后立刻 count 能看到 +1。Hibernate 内部由 DefaultAutoFlushEventListener 在每个 query 执行前判断:(1) PC 中是否有 dirty 实体;(2) 该 query 涉及的表是否与 dirty 实体的表重叠。两条都成立就触发提前 flush。

这个语义带来的代价是只读查询可能触发提前写入,放大流量峰值时的资源占用。FlushMode.COMMIT 仅在 commit 时 flush,可避免该副作用,但需开发者确认查询不依赖未 flush 的修改。

Action 执行顺序对生产的影响

顺序约束效果生产风险
Insert 先于 Delete保护新对象引用旧对象的 FK唯一键替换需手动 flush
同类 Action 内部按入队顺序保留代码意图跨类型顺序被打乱
Update 在 Collection 之前集合修改基于最新实体状态复杂级联可能反直觉
Delete 最后保护被引用对象父子级联删除顺序倒置

3.3.3 唯一键替换场景 — 经典生产陷阱

反例:remove + persist 期望"原地替换"

em.getTransaction().begin();

Customer old = em.find(Customer.class, 1L);
em.remove(old);
em.persist(new Customer(
    "Customer C1001 V2",
    "alice@example.com"));

em.flush();

ActionQueue 先执行 INSERT 后执行 DELETE。
INSERT 时旧记录还存在,触发唯一索引冲突。
抛 ConstraintViolationException → 整个事务回滚。

正例:显式 flush 切割顺序边界

em.remove(old);
em.flush();             // 先执行 DELETE

em.persist(new Customer(
    "Customer C1001 V2",
    "alice@example.com"));
em.flush();             // 再执行 INSERT

手动 flush 切割两个 ActionQueue 周期。
第一次 flush 只有 DELETE,第二次只有 INSERT。
顺序问题消失,唯一约束不再冲突。 

-- 反例错误:
[ERROR] duplicate key value violates unique constraint "uk_customers_email"
[Detail] Key (email)=(alice@example.com) already exists.

-- 正例 SQL 序列:
delete from customers where id = ? and version = ?;
insert into customers (email, name, version, id) values (?, ?, ?, ?);
"代码顺序 ≠ SQL 顺序"是 JPA 最容易翻车的语义 — 我见过的最离奇的版本是这样:开发者发现某个唯一约束冲突,以为是并发问题,加了悲观锁;锁加了之后冲突没消失,又加 @Retryable;重试也没用,最后改成"先 sleep 100ms 再 persist"试图绕开。整个调试过程花了两天,根本没意识到是 ActionQueue 顺序问题——因为代码读起来确实是 remove 在前 persist 在后。这种 bug 一旦掌握"代码顺序 ≠ SQL 顺序"的心智模型 30 秒就能定位:任何"看起来按顺序写但 SQL 报顺序冲突"的现象,先排查 ActionQueue。这也是为什么 Hibernate 官方文档强烈建议在做"原地替换"时用 UPDATE 而不是 remove + persist——除非你确实需要主键变更。

3.3.4 ActionQueue 的批量化与 jdbc.batch_size

ActionQueue 不仅是"延迟",还是"批量化"的入口。当你设置 hibernate.jdbc.batch_size=50,Hibernate 在 flush 阶段会把同类 Action 的 SQL 合并成 JDBC batch:

em.getTransaction().begin();
for (int i = 0; i < 50; i++) {
    em.persist(new Product("P-" + i));
}
em.flush();   // → 一次 JDBC batch,50 条 INSERT 一并发送
em.getTransaction().commit();

注意 jdbc.batch_size 只影响"一次 flush 内"的批量,不替代 PC 清理:如果你 persist 10 万条不 clear,即使 batch_size=50 也照样 OOM。两者解决不同问题:batch_size 减少网络/protocol 往返,clear 减少 PC 内存占用。批处理章节会详细展开。

3.4 clear / detach / close

把对象从 PC "驱逐"出去有三种 API:em.detach、em.clear、em.close。它们的范围和副作用各不相同,选错就会出现"数据没保存"或"内存没释放"的 bug。

图 3-4-1 · 三种 PC 失效路径与丢失语义

3.4.1 clear 的副作用

em.getTransaction().begin();

em.persist(new Product("A"));
em.persist(new Product("B"));
em.clear();                       // 清空 PC,丢弃 ActionQueue 中的 INSERT

em.getTransaction().commit();
-- 无 INSERT
select count(*) from products;
-- 结果: 0

em.clear() 在 flush 前调用会丢弃 ActionQueue 中所有未执行的操作。这个行为对批处理场景至关重要——flush 与 clear 必须成对使用,顺序不可颠倒。先 clear 后 flush 等于"丢弃所有操作再尝试持久化空 PC",没有任何 SQL 产生,数据丢失且无任何错误日志。

三种 API 行为对比

API范围是否丢弃 ActionQueue适用场景
em.detach(entity)单个实体仅该实体的 Action选择性脱离管理
em.clear()整个 PC全部 Action批处理 reset
em.close()整个 EM全部 Action + 资源请求结束

3.4.2 批处理标准模式

反例:不 flush+clear 的批量插入

em.getTransaction().begin();
for (int i = 1; i <= 100_000; i++) {
    em.persist(new Product("P-" + i));
}
em.getTransaction().commit();

全部 10 万实体堆在 PC 中。
Snapshot 数组占用 + EntityEntry 元数据占用。
typically 几百 MB heap → OutOfMemoryError。

正例:每 500 条 flush+clear

for (int i = 1; i <= 100_000; i++) {
    em.persist(new Product("P-" + i));

    if (i % 500 == 0) {
        em.flush();          // 发送批量 INSERT
        em.clear();          // 清空 PC,释放快照
    }
}
em.flush();
em.clear();

每 500 条把 PC 重置一次。
PC 中 Managed 实体峰值始终 ≤ 500。
内存占用与总条数解耦,可放心跑亿级。 

执行特征:
  insert rows attempted    100000
  flush/clear checkpoints  200
  PC 中 Managed 实体峰值    ≤ 500
  Heap 占用峰值             < 50 MB

未配置 flush + clear 的批处理将累积所有 Managed 实体在 PC 中,heap 占用线性增长直至 OOM。Hibernate 的 hibernate.jdbc.batch_size 参数仅影响 JDBC 层批量,不替代 PC 清理——这两个参数解决不同维度的问题。

批处理还要配合 IDENTITY 主键的"陷阱" — 如果实体用 @GeneratedValue(strategy = IDENTITY),Hibernate 无法做 INSERT batch,因为它必须在每条 INSERT 后立刻读出 IDENTITY 值才能继续。这种情况下 batch_size 完全失效,SQL 变成"100 个独立的 INSERT 各取 generated key"。批量导入场景应该用 SEQUENCE 策略,提前批量预取序列号(allocationSize 设大一些,比如 50),才能让 batch_size 真正生效。这也是 Vlad Mihalcea 的高频建议:批量导入用 SEQUENCE,日常 OLTP 用 IDENTITY 没关系。

3.4.3 detach 的"半离线"语义

em.detach(entity) 把单个实体踢出 PC,但保留其他实体。这在以下场景有用:

  • 解耦传给前端:把已加载的实体 detach 掉,改字段不会触发 dirty checking,避免误生成 UPDATE。
  • 避免大对象常驻 PC:加载完处理完一个大实体(包含 BLOB / 长 VARCHAR)立刻 detach,释放 Snapshot 内存。
  • 临时跳过级联:detach 父对象后再做某些只该影响子对象的修改,父对象的修改不会传播。

但 detach 不会触发该实体已入队的 INSERT/UPDATE 立刻执行——这些操作直接被丢弃。如果你刚 persist 了一个对象立刻 detach,等于这个对象从未被持久化过。要先 flush 再 detach。

3.5 Flush 模式

FlushMode 决定"什么时候触发 flush"。JPA 规范定义两种:AUTO(默认)和 COMMIT;Hibernate 扩展了第三种 MANUAL。三者性能特性、数据可见性都不同。

图 3-5-1 · FlushMode 在查询时机的判定逻辑

FlushMode 选型

模式触发 flush 时机数据可见性推荐场景
AUTOcommit / 查询前 / 显式 flush当前事务内修改对查询可见通用业务
COMMIT仅 commit / 显式 flush当前事务修改对查询不可见报表只读路径
MANUAL (Hibernate)仅显式 flush严格控制时机测试或复杂事务

3.5.1 AUTO 引发的副作用

em.setFlushMode(FlushModeType.AUTO);
em.getTransaction().begin();

Customer c = em.find(Customer.class, 1L);
c.setEmail("dup@example.com");          // 假设已有此邮箱

long count = em.createQuery(
    "select count(c) from Customer c", Long.class)
    .getSingleResult();
[ERROR] PersistenceException:
ConstraintViolationException: duplicate key value violates
unique constraint "uk_customers_email"

唯一索引冲突在 count 查询前的 auto flush 中暴露。这个现象常被误解为"count 查询本身失败",实际是 setEmail → AUTO flush → INSERT/UPDATE → 触发约束。如果代码看起来是这样:

// 业务代码:先改字段,再查统计
customer.setEmail(newEmail);
long count = orderRepo.countByCustomerId(customer.getId());

开发者完全不会想到 countByCustomerId 这条只读查询会触发 UPDATE。这种"远距离副作用"是 AUTO 模式最大的认知陷阱。Spring Data JPA 默认 FlushModeType.AUTO,所以这个陷阱在 Spring 项目里非常普遍。

3.5.2 COMMIT 模式的适用场景

COMMIT 模式让 flush 仅在 commit 时发生,所有事务内的查询不会触发提前写入。适合:

  • 只读报表:整段事务都是 SELECT,没有修改,COMMIT 模式让查询路径完全无意外。
  • 批量导出:大量 SELECT + 偶尔修改一些"导出状态"字段,COMMIT 避免每次 SELECT 触发 flush 让大 PC 反复扫描。
  • 分阶段处理:先收集大量改动,最后 commit 时统一发出 SQL,中间 SELECT 用 PC 视图即可。
em.setFlushMode(FlushModeType.COMMIT);
em.getTransaction().begin();

Customer c = em.find(Customer.class, 1L);
c.setEmail("dup@example.com");

long count = em.createQuery(
    "select count(c) from Customer c", Long.class)
    .getSingleResult();          // 不触发 flush,仍读旧 count

em.getTransaction().commit();    // 这里才真正 INSERT/UPDATE

风险是开发者要明确知道"事务内的 query 看不到事务内的修改"。如果你 persist 了一个新订单后立刻在同一事务里 query 该客户的所有订单,COMMIT 模式下查不到刚 persist 的——可能误判为 bug。

Spring 的 @Transactional(readOnly=true) 与 FlushMode 的关系 — Spring 的 @Transactional(readOnly = true) 不仅是个事务标志,它还会通过 SessionImpl.setDefaultReadOnly(true) 把整个 session 标为只读,效果类似把每个加载进来的实体调用 setReadOnly(true) + 把 FlushMode 切到 MANUAL。这意味着只读事务内的查询不会触发 dirty checking、不会触发 auto flush、即使你不小心改了字段也不会生成 UPDATE(commit 时也不会)。这是 Spring 给只读路径的隐式优化,也是为什么"加 readOnly=true 后某段代码的 P99 突然下降几百 ms"——通常是因为之前 PC 里有大量实体在每次 query 前都要做 dirty 检查。但反过来,如果你期望"先改后 commit"却写了 readOnly=true,你的修改会被静默丢弃。这是 Spring + JPA 项目里非常常见的低级 bug,review 时务必盯住。

3.6 EntityEntry 与 EntityKey

前面把 PC 抽象成 "Identity Map + Snapshot + ActionQueue + Collection 跟踪表",但在 Hibernate 实现里这些不是四个独立结构,而是聚合在 StatefulPersistenceContext 这个类里。它内部维护几张关键的映射表,每个 Managed 实体都对应一份元数据:EntityEntry。

StatefulPersistenceContext 内部数据结构

StatefulPersistenceContext 主要字段:

  ┌─────────────────────────────────────────────────────────┐
  │  StatefulPersistenceContext                              │
  ├─────────────────────────────────────────────────────────┤
  │  entitiesByKey         Map<EntityKey, Object>            │
  │                        主键到实体的映射                    │
  ├─────────────────────────────────────────────────────────┤
  │  entityEntries         IdentityMap<Object, EntityEntry>  │
  │                        实体到 EntityEntry 的反向映射         │
  ├─────────────────────────────────────────────────────────┤
  │  collectionEntries     IdentityMap<集合, CollectionEntry>  │
  │                        持久化集合元数据                     │
  ├─────────────────────────────────────────────────────────┤
  │  collectionsByKey      Map<CollectionKey, 集合>           │
  │                        集合主键映射                       │
  ├─────────────────────────────────────────────────────────┤
  │  proxiesByKey          Map<EntityKey, Object>            │
  │                        延迟代理映射                       │
  └─────────────────────────────────────────────────────────┘

注意 entityEntries 用的是 IdentityMap 而非 HashMap——它的 key 比较走 System.identityHashCode,即引用相等而非 equals。这一点很关键:反向查找 EntityEntry 不依赖实体的 equals/hashCode,即使你写了一个 buggy 的 equals 也不会让 PC 内部混乱。

EntityEntry 关键字段

字段类型含义
statusStatus 枚举MANAGED / DELETED / GONE / READ_ONLY

图 3-6-1 · EntityEntry 的 5 状态生命周期

EntityEntry 的 status 字段精确反映实体在 PC 中的运行时状态。每个状态对 dirty checking 与 SQL 生成的影响不同:

  • MANAGED:正常状态,dirty checking 全字段比对,变化生成 UPDATE。
  • DELETED:已被 remove 但还没 flush,flush 时生成 DELETE。在此状态下再 persist 同一对象会回到 MANAGED,DELETE 被取消。
  • GONE:DELETE 已执行,实体不再存在于 PC,但引用还在 Java 堆上。
  • READ_ONLY:Hibernate 跳过 dirty checking,适合大量加载但不修改的报表场景。可显著减少 flush 时间。

3.6.2 实战:用 setReadOnly 优化报表查询

List<Order> orders = em.createQuery(
    "select o from Order o where o.createdAt > :since",
    Order.class)
    .setParameter("since", oneMonthAgo)
    .setHint(QueryHints.HINT_READONLY, true)   // 整批 READ_ONLY
    .getResultList();

for (Order o : orders) {
    report.add(toReportRow(o));   // 只读不修改
}
// commit 时不进行任何 dirty checking

这是 Hibernate 的"批量只读模式"。对于报表场景一次加载 10 万条订单,READ_ONLY 让 commit 时的 dirty checking 直接跳过,节省的时间可观——典型从 800ms 降到 30ms。

3.7 持久化集合

JPA 实体里的 List<Order> 在加载后不再是普通的 ArrayList——Hibernate 会偷偷把它替换成 PersistentBag / PersistentSet 之类的代理。这些代理跟踪集合的 add / remove / clear 操作,并在 flush 时翻译成对应的 SQL。理解这个替换机制,是理解"为什么集合操作有时不产生 SQL"的关键。

3.7.1 PersistentBag / PersistentSet

@Entity
public class Customer {
    @OneToMany(mappedBy = "customer")
    private List<Order> orders = new ArrayList<>();
}

// 加载后实际类型:
Customer c = em.find(Customer.class, 1L);
System.out.println(c.getOrders().getClass());
// → org.hibernate.collection.spi.PersistentBag

常见持久化集合类型

Java 类型Hibernate 持久化集合特性
ListPersistentBag允许重复,无序
List + @OrderColumnPersistentList索引列维护顺序
SetPersistentSet不允许重复
Set + LinkedHashSetPersistentSet维持插入顺序
SortedSetPersistentSortedSet自然排序
MapPersistentMapkey-value 结构

3.7.2 集合修改的 SQL 时机

em.getTransaction().begin();

Customer c = em.find(Customer.class, 1L);
c.getOrders().add(new Order("ORD-1001", c));

em.getTransaction().commit();
select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = ?;

select o1_0.customer_id, o1_0.id, o1_0.order_no
from orders o1_0
where o1_0.customer_id = ?;

insert into orders (customer_id, order_no, id) values (?, ?, ?);

集合操作在 flush 时由 CollectionUpdateAction 执行。mappedBy 指明关系所有方,Hibernate 通过所有方端的外键列管理关系。注意第二条 SELECT——访问 getOrders() 触发了集合的 lazy 初始化,Hibernate 把整个集合加载进 PC 才能跟踪后续的 add 操作。如果你只想 add 不需要读取已有项,可以考虑改用 cascade=PERSIST 直接 em.persist(newOrder),避免触发集合初始化的额外 SELECT。

3.7.3 集合替换破坏跟踪 — 经典陷阱

反例:替换为新集合

em.getTransaction().begin();

Customer c = em.find(Customer.class, 1L);
c.setOrders(new ArrayList<>());          // ❌ 替换字段引用
em.getTransaction().commit();

新 ArrayList 不是 PersistentBag,Hibernate 无法跟踪。
原集合的清空意图丢失。
无 DELETE,无 SQL,数据库里订单全部保留。

正例:操作原集合

em.getTransaction().begin();

Customer c = em.find(Customer.class, 1L);
c.getOrders().clear();                    // ✅ 调 PersistentBag.clear
em.getTransaction().commit();

PersistentBag.clear() 内部标记为 dirty。
flush 时 CollectionRemoveAction 生成 DELETE。
级联设置 + orphanRemoval 让子实体也被删除。

"重新 new 集合"是 JPA 集合最常见的 bug — 这个 bug 的可怕之处在于它没有任何错误提示:代码跑通,事务 commit,日志里也没异常,但数据库里"应该被清空"的子表数据完好无损。下次代码逻辑读出来发现还在,就把"我刚清空了"的认知和"为什么数据还在"的事实之间撕开一个沟,排查起来非常痛苦。生产防御:(1) 在团队规范里禁止 entity 类提供集合 setter,只暴露 getter;(2) 集合字段用 final + 在构造里 new ArrayList,从源头堵住 setter;(3) 用 Lombok 的话不要对集合字段加 @Setter。

3.8.2 设计取舍

取舍 1:Identity Map 与对象身份

PC 选择维护 Identity Map 而非每次返回新对象。该选择保证了同主键的对象身份唯一性,使得级联、双向关联、equals/hashCode 行为可预期。代价是 PC 内存随实体数量线性增长,长事务下需主动清理。如果不维护 Identity Map,find(Customer, 1) 两次返回两个独立对象,你修改其中一个不影响另一个,然后 commit 时根本不知道用谁的状态——整个 ORM 模型都崩了。

取舍 2:全字段 Snapshot 与字节码增强

默认 Snapshot 比较保留通用性,实体类无须实现任何接口或继承基类。代价是大对象在 PC 中产生字段级副本,大量 Managed 实体下 flush 阶段成本显著。字节码增强通过编译期改写 setter,将 dirty 标记从 flush 阶段下沉到 setter 调用,牺牲构建复杂度换取运行时性能。

取舍 3:Write-Behind 与 ActionQueue 排序

ActionQueue 的固定执行顺序保护外键约束,使得多数级联场景无须手动 flush。代价是开发者必须了解"代码顺序不等于 SQL 顺序",特别是在唯一键替换、跨实体顺序敏感场景下需显式插入 flush 边界。如果让 SQL 严格按代码顺序发出,Hibernate 必须在每个 persist/remove 立刻 flush,实际上等于退化成"无 write-behind",所有 batch 优化都丢失。

取舍 4:AUTO 默认 FlushMode

JPA 规范默认 FlushMode 为 AUTO,确保查询可读到当前事务内的修改。该选择保证语义一致性,但带来"只读查询触发写入"的副作用。生产中复杂事务可考虑切换 COMMIT 模式,以显式控制 flush 边界。

3.9.1 集合替换破坏跟踪示例

em.getTransaction().begin();

Customer c = em.find(Customer.class, 1L);
c.setOrders(new ArrayList<>());          // 错误:替换为新集合
em.getTransaction().commit();
-- 无 DELETE,无 SQL

c.setOrders(new ArrayList<>()) 替换了实体字段引用,新 ArrayList 不是 PersistentBag,Hibernate 无法跟踪原集合的清空操作。正确做法是 c.getOrders().clear(),该方法在 PersistentBag 内部标记为脏并在 flush 时生成 DELETE。

3.9.2 Bulk JPQL 绕过 PC 的副作用

em.getTransaction().begin();

Customer c = em.find(Customer.class, 1L);
String oldEmail = c.getEmail();   // alice@example.com

em.createQuery(
    "update Customer c set c.email = :ne where c.id = :id")
    .setParameter("ne", "new@example.com")
    .setParameter("id", 1L)
    .executeUpdate();             // 绕过 PC 直接 SQL

String stillOld = c.getEmail();   // 仍 alice@example.com
em.refresh(c);                    // 显式同步
String nowNew = c.getEmail();     // 终于 new@example.com

em.getTransaction().commit();

JPQL 的 UPDATE/DELETE 语句直接发 SQL,不经过 PC——它们不更新 PC 中已加载的实体状态。这导致"数据库已改、PC 中对象未改"的不一致。修复方式是 bulk 后立刻 em.clear()(整个 PC 重置)或 em.refresh(entity)(单个实体重新加载)。如果你不做这一步,后续访问该对象会读到旧值,可能引发逻辑错误。

3.10.2 Visibility Trace API:5 秒判定 PC 状态

// 暴露一个 admin endpoint,让 oncall 快速判定 PC 状态
@GetMapping("/admin/pc-status")
public Map<String, Object> pcStatus() {

    PersistenceContext pc = s.getPersistenceContext();
    SessionStatistics stats = s.getStatistics();

    return Map.of(
        "entityCount", stats.getEntityCount(),
        "collectionCount", stats.getCollectionCount(),
        "actionQueueSize", queueSize(s),
        "flushMode", em.getFlushMode().toString(),
        "isOpen", em.isOpen(),
        "topEntities", topEntitiesByCount(stats));
}

把这个 endpoint 接到内部监控面板,oncall 看到 P99 飙升时一秒查到 entityCount 是不是异常高,立刻判断要不要 kill 掉占用 PC 的长事务。这种"内部可观测性"是生产 JPA 服务的硬性配套。

3.10.3 完整闭环:最小回归用例

em.getTransaction().begin();
Customer c1 = em.find(Customer.class, 1L);
Customer c2 = em.find(Customer.class, 1L);
boolean same = (c1 == c2);                                   // true

c1.setEmail("test@example.com");
em.flush();                                                   // UPDATE
em.getTransaction().rollback();

em.getTransaction().begin();
String emailNow = em.find(Customer.class, 1L).getEmail();     // 旧值
em.getTransaction().commit();
[same]      true
[emailNow]  alice@example.com (旧值)
[结论]      flush 后 rollback 撤销修改

这段代码可以放在 CI 里作为"PC 行为不变量回归测试"——任何后续改动让 same 变 false 或让 emailNow 变成新值,立刻知道有人破坏了 PC 的核心语义。
