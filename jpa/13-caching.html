第 13 章 · 缓存体系

L1 PersistenceContext · L2 共享缓存 · Query Cache · 失效与一致性 · 命中率决策 · 集群与 Region

JPA / Hibernate 提供三层缓存:L1(EntityManager 私有)、L2(SessionFactory 共享)、Query Cache(JPQL 结果)。三层各自的作用域、生命周期、失效机制完全不同,把它们误当成同一种"开关"使用是大部分缓存 bug 的根因。

缓存的核心问题不是开关切换,而是判断业务场景:数据读多写少否?能容忍短暂不一致否?命中率能稳定高于维护成本否?未匹配场景的缓存常造成伪优化——表面 SQL 数减少,实际多了缓存维护开销与一致性 bug。本章按"L1 → L2 → Query Cache → 监控 → 失效 → 集合 → 源码 → 陷阱 → 回归 → 配置 → 拒绝清单"组织,聚焦"该不该开、开了能挣到什么、错了会丢什么"。

  • L1 = Identity Map:同一 EM 内 find() 两次返回同一引用,em.clear() 后行为变化
  • L2 工作流:事务 1 入缓存 脱水状态,事务 2 命中后由 EntityPersister 重新水合为 managed 实体
  • 四种 CacheConcurrencyStrategy:READ_ONLY / NONSTRICT_READ_WRITE / READ_WRITE / TRANSACTIONAL 的语义与适用场景
  • Query Cache 只存 ID 列表;仅启用 Query Cache 不启用 L2 = 退化为 1 + N 次按 ID 查询
  • Micrometer 集成 hibernate.l2.hit.rate / query.hit.rate,Prometheus 指标实际样例
  • 命中率决策表:L2 hit > 90% 保留 / hit < 80% 排查 / Query hit < 50% 关掉
  • 写操作触发的失效:JPQL DML 自动失效 / Native DML 不失效 需手动 Cache.evict
  • UpdateTimestampsCache 表级时间戳如何保证 Query Cache 的"读到最新数据"
  • 集合缓存仅存 ID 列表,关联实体必须同时启用 L2 才有意义
  • 核心源码:StatefulPersistenceContext / EntityCacheImpl / QueryResultsCacheImpl / SingleIdEntityLoader
  • 四类设计取舍:L1 默认开 / L2 默认关 / L2 存脱水状态 / Query Cache 仅存 ID 的根本原因
  • ehcache 完整配置 + 五类业务场景的 Region 设计(字典 / 主数据 / 热点 / 写频繁 / 强一致)

前置知识:第 2 章实体生命周期、第 3 章 PersistenceContext、第 11 章 fetch 策略、第 12 章 N+1 问题;JCache(JSR-107)与 ehcache 基础;事务隔离级别;集群一致性基础。

13.0 章首速览

一句话总览。L1 默认开,事务期内的引用一致与脏检查全靠它;L2 默认关,需明确启用、明确选 ConcurrencyStrategy、明确管理 Region 容量;Query Cache 只缓存"哪些 ID 满足这个 JPQL",必须配 L2 才有意义。三者协作的关键不是"开几层",而是"业务场景是否匹配缓存假设"。

13.1 一级缓存(L1)

L1 本质是Identity Map:同一 EntityManager 内,相同 ID 的实体永远返回同一 Java 引用。这是脏检查、双向关联管理、merge 语义等所有 ORM 行为的底层假设。

13.2 二级缓存(L2)

L2 跨 EntityManager / 跨事务共享,缓存的是脱水状态(简单字段值的数组)而非 managed 实体引用。命中后由 EntityPersister 重新水合为该事务的 managed 实体。

图 13-2-1 · L2 工作流 — 事务 1 入缓存,事务 2 重建实体

spring.jpa.properties.hibernate.cache.region.factory_class=org.hibernate.cache.jcache.JCacheRegionFactory
spring.jpa.properties.hibernate.javax.cache.uri=classpath:ehcache.xml
spring.jpa.properties.hibernate.generate_statistics=true
@Entity
@Cacheable
@org.hibernate.annotations.Cache(
    usage = CacheConcurrencyStrategy.READ_WRITE,
    region = "product-cache")
public class Product {
    @Id
    private Long id;
    private String name;
    private BigDecimal price;
}

READ_ONLY不会修改的字典强最优

13.3 查询缓存(Query Cache)

图 13-3-1 · Query Cache 工作流 — QueryKey → ID 列表,实体仍走 L2
spring.jpa.properties.hibernate.cache.use_query_cache=true
List<Product> hotProducts = em.createQuery("""
    select p from Product p
    where p.productCategory.code = :code
    order by p.sales desc
    """, Product.class)
    .setParameter("code", "PHONE")
    .setHint("org.hibernate.cacheable", true)
    .setHint("org.hibernate.cacheRegion", "query.hot-products")
    .setMaxResults(20)
    .getResultList();
[首次执行]
select p.id, p.name, p.price, p.product_category_id, p.sales
from products p
join product_categories pc on pc.id = p.product_category_id
where pc.code = ?
order by p.sales desc
fetch first ? rows only;
[bind]  [PHONE, 20]

[第二次执行]
[Query Cache hit]: query.hot-products
[L2 hits]:         20
[JDBC statements]: 0
配置实体在 L2?第二次执行
Query Cache + L2是1 次 Query Cache hit + 20 次 L2 hit
Query Cache + L2否(大集合超容量)1 次 Query Cache hit + 20 次按 ID 查询
Query Cache 无 L2否1 次 Query Cache hit + 20 次按 ID 查询(退化)
仅 L2是仍执行 SQL,但实体水合命中 L2

常见误区:仅启用 Query Cache 而不启用 L2。Query Cache 仅缓存 ID 列表,实体仍须从某处获取——退化结果是 1 + N 次按 ID 查询,比直接执行 SQL 还慢。

13.4 监控与决策

@Bean
MeterBinder hibernateCacheMetrics(EntityManagerFactory emf) {
    Statistics stats = emf.unwrap(SessionFactory.class).getStatistics();
    return registry -> {
        Gauge.builder("app.hibernate.l2.hit.rate", stats, s -> {
            long hit = s.getSecondLevelCacheHitCount();
            long miss = s.getSecondLevelCacheMissCount();
            return hit + miss == 0 ? 0 : (double) hit / (hit + miss);
        }).register(registry);

        Gauge.builder("app.hibernate.query.hit.rate", stats, s -> {
            long hit = s.getQueryCacheHitCount();
            long miss = s.getQueryCacheMissCount();
            return hit + miss == 0 ? 0 : (double) hit / (hit + miss);
        }).register(registry);
    };
}

app_hibernate_query_hit_rate                                     0.18

hibernate_second_level_cache_miss_total{region="product-cache"}  211

观测信号判断动作
L2 hit > 90%, 写少有效保留 + 监控 TTL
L2 hit < 80%, put/miss 高容量不足或写频繁增大容量或取消
Query hit < 50%维护成本 > 收益关闭 cacheable
集群节点旧值本地 L2 不一致改 Redisson / Infinispan

13.5 失效与一致性

图 13-5-1 · 写操作触发的失效 — Hibernate 写 vs Native 写
em.flush();

int updated = em.createNativeQuery(
    "update products set price = ?1 where id = ?2")
    .setParameter(1, new BigDecimal("99.00"))
    .setParameter(2, 100L)
    .executeUpdate();

em.clear();
em.getEntityManagerFactory().getCache().evict(Product.class, 100L);

Native DML 不自动失效 L2。Hibernate 不解析 native SQL,不知道动了哪张表 → 缓存的旧值仍被命中 → 跨事务读到错误数据。修复:手动 Cache.evict 或重启实例。

操作L2 同步须手动
em.merge / setter自动否
em.remove自动否
JPQL Bulk自动(按表)否
Native DML不同步Cache.evict
直接 JDBC不同步Cache.evict
数据库直接修改不同步重启或定时刷新

13.6 集合缓存

@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Customer {
    @Id private Long id;

    @OneToMany(mappedBy = "customer", fetch = FetchType.LAZY)
    @org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
    private List<Order> orders = new ArrayList<>();
}

@Entity
@Cacheable
@org.hibernate.annotations.Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class Order {
    @Id private Long id;
    @ManyToOne private Customer customer;
}
[Customer.orders 缓存内容]
  customer_id: 100
  order_ids:   [501, 502, 503]

[加载 customer.getOrders()]
  1. L2 lookup Customer.orders 100      → ID 列表
  2. 按 ID 查 Order L2                   → 实体

集合缓存仅保存关联实体的 ID 列表。访问集合元素仍须从 Order 的 L2 获取实体,因此 Order 也必须启用 L2,否则集合缓存退化为按 ID 的逐条查询。

13.7.1 设计取舍

取舍 1:L1 默认开启。L1 几乎免维护成本,且事务内引用一致是必要语义(脏检查、关联管理依赖)。代价是长事务可能积累大量 managed 实体,内存压力增加。

取舍 2:L2 默认关闭。L2 引入分布式一致性、Provider 配置、过期策略等复杂度。Hibernate 选择默认关闭,业务按需启用。代价是新手忘记启用而错过显著性能提升。

取舍 3:L2 存储脱水状态。脱水状态是属性的简单值,而非 managed 实体引用。这避免了集群中传输代理对象,代价是每次命中须重新水合(调用 EntityPersister)。

取舍 4:Query Cache 仅存 ID。实体可能被其他查询或 L2 操作改变,缓存完整实体引发一致性问题。仅存 ID 让 Query Cache 与 L2 解耦,代价是实体仍须从 L2 获取。

13.8 陷阱与限制

#陷阱现象修复
1Query Cache 无 L2第二次 N+1 按 ID 查询启用 L2 + 实体 Cacheable
2Native DML 不失效跨事务读旧数据手动 Cache.evict
3集群本地 L2节点间数据不一致Redisson / Infinispan
4Region 过小频繁 evict增大容量或取消
5写频繁实体加缓存维护成本 > 收益移除 Cacheable
6TTL 过长业务变更不可见TTL = 业务可容忍延迟
7大对象(Blob)内存爆缓存元数据,Blob 走 CDN
8双向关联缓存循环内存膨胀限制集合缓存层级

13.9 回归护栏

@Test
void productEntityShouldHitL2OnSecondTransaction() {
    SessionFactory sf = emf.unwrap(SessionFactory.class);
    Statistics stats = sf.getStatistics();
    stats.clear();

    txTemplate.execute(status -> em.find(Product.class, 100L));
    txTemplate.execute(status -> em.find(Product.class, 100L));

    long l2Hits = stats.getSecondLevelCacheHitCount();
    long l2Misses = stats.getSecondLevelCacheMissCount();

    assertThat(l2Hits).isEqualTo(1L);
    assertThat(l2Misses).isEqualTo(1L);
}

@Test
void nativeUpdateShouldRequireExplicitEvict() {
    em.find(Product.class, 100L);
    em.flush();
    em.createNativeQuery("update products set price = ?1 where id = ?2")
        .setParameter(1, new BigDecimal("99.00"))
        .setParameter(2, 100L)
        .executeUpdate();

    em.clear();
    assertThat(emf.getCache().contains(Product.class, 100L)).isTrue();

    emf.getCache().evict(Product.class, 100L);
    assertThat(emf.getCache().contains(Product.class, 100L)).isFalse();
}
