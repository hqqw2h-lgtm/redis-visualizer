第 4 章 · 基础映射与类型转换

@Entity · @Table · @Id · @GeneratedValue · @Column · @Enumerated · @Temporal · @Embeddable · AttributeConverter · BasicType · JdbcType · Dialect · DDL 生成 · 命名策略 · 跨方言对比

JPA 基础映射看起来只是"在字段上贴几个注解",但它实际是注解层与数据库 schema 之间的桥梁——你写下的每一个 @Column(length = ...)、@Enumerated(...)、@Temporal(...) 都会以三种身份同时生效:它决定启动期生成的 DDL(varchar(80) not null 还是 varchar(255))、决定运行期 INSERT/UPDATE 的列列表(updatable=false 的列不出现在 UPDATE 语句里)、决定JDBC 绑定的参数类型(BigDecimal 走 NUMERIC 还是 DECIMAL)。任何一处默认值偏离生产意图,都会以"数据精度静默截断"、"枚举重排后语义错位"、"跨库迁移 DDL 报错"这种代价昂贵的形式回来。

本章把最高频的八个基础映射注解(@Entity、@Table、@Id、@Column、@Enumerated、@Temporal、@Embeddable、AttributeConverter)放在同一个观测框架下展开:每节给出 Java 代码 → 生成 DDL → 运行 SQL → 回归断言四段,确保你不只看到"应该这么写",还能看到"这么写之后数据库里到底变成什么样"。我们不止讲"如何写对",更强调"哪里容易写错"以及"如何用测试守护"——因为基础映射的特点是错了不会立刻报错,而是把错误延迟到几个月后的生产事故里。

  • @Entity / @Table 如何定义"表身份":显式命名 vs 隐式命名策略,以及为什么生产强烈推荐显式
  • @Id + @GeneratedValue 四种主键策略(IDENTITY / SEQUENCE / TABLE / AUTO)的时序差异与批量插入性能影响
  • @Column 八大属性(name / nullable / length / precision / scale / unique / insertable / updatable / columnDefinition)分别影响 DDL 还是运行 SQL
  • Java 类型到 SQL 类型的两级映射:BasicType → JdbcType → Dialect 三层抽象如何让同一份代码在 PostgreSQL / MySQL / Oracle 生成不同 DDL
  • @Enumerated 的 STRING vs ORDINAL:为什么 ORDINAL 是"静默数据腐败"的高频根源
  • @Temporal 仅用于 java.util.Date/Calendar 旧 API,java.time 包内类型(LocalDate/LocalDateTime/Instant)直接由方言推导精度
  • AttributeConverter 在 Java 值对象与 JDBC 值之间做双向转换,但不感知列约束——精度/非空仍需显式 @Column
  • @Embeddable 把值对象摊平到宿主表,以及 @AttributeOverrides 处理"同实体重复嵌入同一组件"的列名冲突
  • 三层回归护栏:启动期 schema validation + PR 期 DDL snapshot + 单测期 SQL snapshot 的覆盖差异
  • 10 个高频陷阱(从"缺无参构造"到"columnDefinition 绑死方言")的"现象 / 根因 / 修复 / 守护"完整闭环

4.0 章首速览

基础映射的本质是把 Java 字段映射到三种东西:DDL 列定义、运行时 SQL 列列表、JDBC 参数绑定类型。任何注解都是为了影响这三者之一(或多个)。理解这一点,你看每个注解都能立刻判断"它影响的是 schema 还是 runtime"。

4.1 @Entity 与 @Table

@Entity 把一个 Java 类纳入 JPA 持久化管辖,@Table 进一步描述这个实体对应的数据库表(表名、唯一约束、索引)。两个注解共同回答一个问题:"这个类对应数据库里哪张表,这张表有什么表级约束"。

图 4-1-1 · 命名策略管线:Java 符号如何变成数据库标识

JPA 名字推导分两步:隐式命名(ImplicitNamingStrategy)从 Java 符号推出"逻辑名",物理命名(PhysicalNamingStrategy)把逻辑名转成数据库实际标识(大小写、下划线、加前缀等)。显式 @Table(name=...) / @Column(name=...) 直接跳过隐式推导,把指定的字符串作为逻辑名传给物理策略。

4.1.1 显式命名示例

import jakarta.persistence.*;

@Entity
@Table(
    name = "t_customer",
    uniqueConstraints = @UniqueConstraint(
        name = "uk_customer_email",
        columnNames = "email")
)
public class Customer {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 80)
    private String name;

    @Column(nullable = false, length = 120)
    private String email;

    protected Customer() {}

    public Customer(String name, String email) {
        this.name = name;
        this.email = email;
    }
}

4.1.2 生成 DDL

create table t_customer (
    id     bigint generated by default as identity,
    email  varchar(120) not null,
    name   varchar(80) not null,
    primary key (id),
    constraint uk_customer_email unique (email)
);

4.1.3 运行时 INSERT

insert into t_customer (email, name) values (?, ?);
-- bindings:
-- [1] alice@example.com
-- [2] Customer C1001

id 不出现在 INSERT 列表里,因为 IDENTITY 策略由数据库自增列生成。email 和 name 都带 not null 与明确长度,说明实体注解约束已经落到数据库层,而不是仅停留在 Bean Validation。

命名策略对照

配置默认行为推荐做法
隐式命名camelCase → snake_case(Spring Boot 默认 SpringPhysicalNamingStrategy)不依赖隐式策略,显式声明
@Table 显式 name表名直接生效强烈推荐,所有实体显式写
@Column 显式 name列名直接生效强烈推荐,关键列必须写
跨数据库迁移PhysicalNamingStrategy 仍可改写使用 SqlIdentifierAwareStrategy 锁定
关键字冲突列名 user / order 在某些库报错强制加表前缀或用反引号
"默认命名策略"是新项目 6 个月后的第一个事故源 — 真实场景见过太多:团队 A 在 Spring Boot 项目里全程不写 @Table / @Column,因为"反正会自动 snake_case"。半年后接手 DBA 想给某张表加分区,发现表名是 t_customer_order_detail,而隔壁服务 B 的同名实体在 MyBatis 里硬编码成了 customer_order_detail(无 t_ 前缀)。两套服务都"对"——只是 ImplicitNamingStrategy 不一样,一个走 SpringPhysicalNamingStrategy 加了 t_ 前缀(因为某个 starter 默认配置),另一个没加。后果是数据库里同一类业务实体有两套表名,跨服务 JOIN 报错,需要全量 schema migration 修复。显式 @Table(name = "...") 不是"啰嗦",是"把命名约定从代码 convention 提升到 DB schema 契约"——任何 reviewer 看到这一行就知道"这张表名永远是它"。

4.2 @Id 与主键策略

@Id 声明哪个字段是主键。@GeneratedValue 描述这个主键如何生成。JPA 提供四种生成策略,各自时序、性能、跨库可移植性差异巨大——选错策略不会让你的代码不能跑,但会在批量插入或跨库迁移时悄悄付出代价。

图 4-2-1 · 三种主键策略的时序差异:何时获得 ID 决定何时发 INSERT

四种主键策略对比

策略来源数据库支持批量插入友好典型场景
IDENTITY数据库自增列MySQL / PostgreSQL / H2否(须立即 INSERT)简单业务
SEQUENCE数据库序列PostgreSQL / Oracle / H2是(allocationSize 批量预分配)高并发批量
TABLE模拟序列表通用(所有库)否跨库兼容
AUTOProvider 决定——不推荐(隐式)

4.2.1 IDENTITY 策略示例

@Entity
@Table(name = "payments")
public class Payment {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 32)
    private String paymentNo;

    protected Payment() {}

    public Payment(String paymentNo) {
        this.paymentNo = paymentNo;
    }
}

DDL:

create table payments (
    id          bigint generated by default as identity,
    payment_no  varchar(32) not null,
    primary key (id)
);

运行时:

insert into payments (payment_no) values (?);
-- binding [1] PAY-2026-0001
-- generatedKeys = 1

4.2.2 SEQUENCE 策略示例

@Id
@GeneratedValue(strategy = GenerationType.SEQUENCE, generator = "payment_seq_gen")
@SequenceGenerator(name = "payment_seq_gen", sequenceName = "payment_seq", allocationSize = 50)
private Long id;

allocationSize = 50 让 Hibernate 一次 nextval 取 50 个 ID 区段并在内存中分发,减少 SEQUENCE 调用次数。批量插入 1000 条记录时,SEQUENCE 调用从 1000 次降至 20 次——在高并发批量场景,这一行配置等同于把 ID 分配的网络往返从 1000 次降到 20 次。

IDENTITY 与 JDBC 批量插入不兼容 — 这是面试常考但很多人答错的点:IDENTITY 策略下,Hibernate 在 persist() 那一刻就要立即发 INSERT 拿到自增 ID,因为没有 ID 就无法把对象放进持久化上下文(EntityKey 缺失)。这一行为直接破坏 JDBC batch——每个 INSERT 都立即往返,无法批量合并。如果你设置了 hibernate.jdbc.batch_size = 50 但实体用 IDENTITY,看起来配置生效,但跑起来批量插入还是 N 次往返,性能完全没改善。批量场景必须用 SEQUENCE,IDENTITY 只适合"单条/低并发"的简单业务。

4.3 @Column 列约束

@Column 是基础映射里使用频率最高、属性最多的注解。它同时影响 DDL(列定义)和运行时 SQL(列列表)——理解这"双重影响"是 @Column 用对的前提。

图 4-3-1 · @Column 属性影响域:哪些影响 DDL,哪些影响运行 SQL

4.3.1 完整属性示例

import java.math.BigDecimal;
import java.time.LocalDateTime;

@Entity
@Table(name = "t_product")
public class Product {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(name = "product_name", nullable = false, length = 100)
    private String name;

    @Column(nullable = false, precision = 12, scale = 2)
    private BigDecimal price;

    @Column(unique = true, length = 13)
    private String isbn;

    @Column(columnDefinition = "text")
    private String description;

    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;
}

4.3.2 生成 DDL

create table t_product (
    id            bigint generated by default as identity,
    created_at    timestamp(6) not null,
    description   text,
    isbn          varchar(13) unique,
    product_name  varchar(100) not null,
    price         numeric(12,2) not null,
    primary key (id)
);

4.3.3 运行时 SQL

INSERT(包含所有可插入列):

insert into t_product
    (created_at, description, isbn, product_name, price)
values
    (?, ?, ?, ?, ?);

UPDATE(自动排除 updatable=false 列):

update t_product
set description = ?, isbn = ?, product_name = ?, price = ?
where id = ?;
-- created_at 不在 UPDATE 列表中

"nullable = false 不是双重验证,是数据库的最后防线" — 很多开发以为"我已经在 Service 里校验过非空,DB 加 not null 是多余的"。这个想法在事故复盘里反复被打脸:应用层校验是可以被绕过的——直接连数据库跑修复脚本、跨服务的 ETL 抽取、异步消费者反序列化失败重写、运维人员手动 UPDATE 修数据......所有这些路径都不经过你的 Service 层。@Column(nullable = false) 落到 DDL 的 not null 才是唯一的"任何路径都绕不开"的约束。把这一行从注解里漏掉,等于把数据库的最后防线拆掉。在数据完整性敏感的领域(金融、医疗、合规),"应用层校验"压根不算约束,只有 schema 级别的 not null 才算。

4.4 Java 类型到 SQL 类型

同一个 BigDecimal 字段,在 PostgreSQL 是 numeric,在 MySQL 是 decimal,在 Oracle 是 number。这种"同代码不同 DDL"是 Hibernate 两级类型抽象(BasicType → JdbcType → Dialect)的直接结果——理解这个链路是跨库项目的必修课。

4.4.1 跨方言验证示例

@Entity
@Table(name = "t_basic_sample")
public class BasicSample {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, precision = 12, scale = 2)
    private BigDecimal amount;

    @Column(nullable = false)
    private Boolean active;

    @Column(nullable = false)
    private UUID externalId;
}

PostgreSQL DDL:

create table t_basic_sample (
    id           bigint generated by default as identity,
    active       boolean not null,
    amount       numeric(12,2) not null,
    external_id  uuid not null,
    primary key (id)
);

MySQL DDL:

create table t_basic_sample (
    id           bigint not null auto_increment,
    active       bit not null,
    amount       decimal(12,2) not null,
    external_id  binary(16) not null,
    primary key (id)
);

同一份 Java 代码,两份截然不同的 DDL:active 在 PostgreSQL 是 boolean 在 MySQL 是 bit,external_id 在 PostgreSQL 是原生 uuid 在 MySQL 是 binary(16)(后者索引行为完全不同)。跨数据库项目应该把目标数据库的 DDL 快照纳入 PR 评审——任何字段类型变化都会在快照 diff 里直接暴露,不会有"我以为差不多"的误判。

跨库迁移最先翻车的永远是 BigDecimal 和 Boolean — 真实迁移案例:某团队从 MySQL 迁到 PostgreSQL,代码不动,只换连接串和方言。预生产跑通了所有单测,上线两天后业务方报告"对账金额对不上"。排查发现:MySQL 里 BigDecimal 没写 precision/scale 时默认是 decimal(19, 2)(MySQL 的默认 scale 行为),PostgreSQL 里则是无 scale 限制的 numeric(可以存任意精度的小数)。从 MySQL 抽出来再灌进 PostgreSQL 的历史数据保留了 2 位小数,但新写入的金额因为业务计算产生了 4 位小数(比如折扣计算 100 * 0.333 = 33.3000...),数据库照单全收。等到月底对账,新数据的精度跟旧数据不一致,所有汇总值都出现微小漂移,无法对账。修复要写一个 PostgreSQL migration ALTER COLUMN amount TYPE numeric(12, 2),但这一步会拒绝所有超出精度的现有行,需要先清洗数据。教训:金额字段必须显式 precision/scale,跨库时方言默认值不一致是隐形地雷。

4.5 @Enumerated 枚举存储

枚举存储是基础映射里"错一次毁三年"的典型场景。@Enumerated 决定 Java 枚举字段如何序列化到数据库,两种选项 STRING 和 ORDINAL 在 DDL 上看起来都"能跑",但ORDINAL 的失败模式是"静默数据腐败"——错了不报错,直到几个月后才在业务报表里露馅。

图 4-5-1 · STRING vs ORDINAL:同一枚举重排后两种策略的命运分叉

4.5.1 推荐写法(STRING)

public enum OrderStatus {
    PENDING,
    CONFIRMED,
    SHIPPED,
    DELIVERED
}

@Entity
@Table(name = "t_order")
public class Order {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    private OrderStatus status;
}

DDL:

create table t_order (
    id      bigint generated by default as identity,
    status  varchar(20) not null,
    primary key (id)
);

运行 SQL:

insert into t_order (status) values (?);
-- binding [1] CONFIRMED

select id, status from t_order where id = ?;
-- 行: id=1, status='CONFIRMED'
-- Java: order.status == OrderStatus.CONFIRMED

4.5.2 ORDINAL 的隐性风险

@Enumerated(EnumType.ORDINAL)        // 不推荐
private OrderStatus status;
枚举重排前:
  PENDING(0), CONFIRMED(1), SHIPPED(2), DELIVERED(3)
  数据库中 id=10 的行: status = 1 (语义为 CONFIRMED)

枚举插入新值后:
  PENDING(0), PROCESSING(1), CONFIRMED(2), SHIPPED(3), DELIVERED(4)
  数据库中 id=10 的行: status = 1 (现读为 PROCESSING)

结果: 静默数据语义偏移

跨语言互通高低(依赖序号)
推荐生产使用是否
ORDINAL 的失败模式是"静默"的——不会报错,只会让你在几个月后才发现 — 你今天用 ORDINAL,代码跑得通、测试通过、上线无事。三个月后产品提需求:"在 CONFIRMED 之前加一个 PROCESSING 状态"。开发把 PROCESSING 加在枚举常量的"恰当位置"(逻辑上 PROCESSING 应该在 PENDING 和 CONFIRMED 之间),发版。从这一刻起,数据库里所有 status = 1 的历史 CONFIRMED 订单,Java 端读出来全部变成 PROCESSING——业务方收到工单"为什么我已经确认的订单显示在处理中",定位时根本不会怀疑这是几个月前那个 @Enumerated(ORDINAL) 注解的锅。修复要回滚枚举顺序 + 写 migration UPDATE t_order SET status = status + 1 WHERE status >= 1(且必须在原代码版本下执行),整个修复链路在事故响应时间窗口里几乎不可能完成。所以 @Enumerated(EnumType.STRING) 不是"推荐做法",而是"唯一可接受的做法",任何 ORDINAL 都应该在 code review 阶段被否决。

4.6 @Temporal 与 java.time

时间映射有一条简单的规则:新代码用 java.time,旧代码兼容 @Temporal。@Temporal 只用于 java.util.Date / Calendar 旧 API——这两个类没有精度信息(它们既能当日期用也能当时间戳用),必须通过 @Temporal 显式告诉 JPA 该当 DATE 还是 TIMESTAMP。java.time 包内的 LocalDate / LocalDateTime / Instant 类型自身就携带精度信息,直接由方言推导出对应 DDL,不需要也不应该加 @Temporal。

时间类型语义对照

Java 类型含义时区推荐场景
LocalDate日历日期无生日 / 纪念日
LocalTime时刻无营业时间
LocalDateTime日期时间无业务局部时间
InstantUTC 瞬间UTC事件时间戳
OffsetDateTime带 offset 时间显式 offset跨时区交易
ZonedDateTime带时区时间完整时区复杂时区调度

选型口径:事件类时间戳推荐 Instant,业务局部时间使用 LocalDateTime,展示层根据用户时区转换。把"何时发生(物理时刻,Instant)"和"在什么本地时间发生(语义时间,LocalDateTime)"分开是设计原则——混在同一个字段里早晚出时区漂移问题。

"LocalDateTime 用于跨时区"是工程上最常见的时间字段误用 — 一个全球应用,事件时间字段用 LocalDateTime,服务器跑在 UTC 时区,前端用户在东八区。用户晚上 10 点下单,前端发送 "2026-05-08T22:00:00"(没带时区),Java 端 LocalDateTime.parse 得到 22:00,数据库存 22:00。问题是数据库不知道这个 22:00 是 UTC 还是东八区——一年后想做"按 UTC 小时统计订单分布"的报表,只能假设它是某个固定时区,但实际上不同用户的写入逻辑可能用了不同的"时区假设"。等到团队规模大了,这种数据没法回溯纠正。用 Instant(强制 UTC 瞬间)能从源头杜绝这类问题——存的永远是物理时刻,展示层换算成用户本地时间,语义清晰且可逆。

4.7 AttributeConverter

AttributeConverter 是 JPA 提供的值对象 ↔ JDBC 值双向转换 SPI。它让你的领域模型可以保持 Money / EmailAddress / PhoneNumber 这种值对象类型,同时数据库列保持基础类型(numeric / varchar)。但 AttributeConverter 有一个关键边界——它只负责值转换,不感知列约束,精度/长度/非空仍然必须显式 @Column。

4.7.1 值对象与数据库列的解耦

public record Money(BigDecimal amount) {
    public Money {
        if (amount.scale() > 2) {
            throw new IllegalArgumentException("Money supports at most 2 decimals");
        }
    }
}

@Converter(autoApply = true)
public class MoneyConverter implements AttributeConverter<Money, BigDecimal> {

    @Override
    public BigDecimal convertToDatabaseColumn(Money money) {
        return money == null ? null : money.amount();
    }

    @Override
    public Money convertToEntityAttribute(BigDecimal value) {
        return value == null ? null : new Money(value);
    }
}

4.7.2 使用 Converter 的实体

@Entity
@Table(name = "t_subscription")
public class Subscription {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, precision = 12, scale = 2)
    private Money monthlyFee;
}

DDL:

create table t_subscription (
    id           bigint generated by default as identity,
    monthly_fee  numeric(12,2) not null,
    primary key (id)
);

运行时:

insert into t_subscription (monthly_fee) values (?);
-- Java: new Money(new BigDecimal("19.90"))
-- binding [1] 19.90

select monthly_fee from t_subscription where id = ?;
-- 数据库: 19.90
-- Java: Money[amount=19.90]

Converter 的边界

维度行为注意事项
Java 值转 DBconvertToDatabaseColumn必须处理 null
DB 值转 JavaconvertToEntityAttribute必须处理 null
列约束仍由 @Column 控制precision / nullable 必须显式
自动应用autoApply=true 全字段生效局部覆盖用 @Convert
主键 / version不可用于这两类字段JPA 规范限制
Native Query不自动应用需手动转换
Converter 的最大陷阱:它"看起来"应该带列约束,但实际不带 — 写完 MoneyConverter,直觉上会觉得"Money 这个值对象已经在构造器里保证了 scale ≤ 2,数据库列应该也跟着 2 位小数"。错——AttributeConverter 只是个值转换器,完全不参与 DDL 生成。如果你忘了在使用它的字段上写 @Column(precision = 12, scale = 2),数据库列会变成 numeric(无精度限制)。然后某个绕过 Converter 的 native query INSERT 进来一个 19.90001,数据库照单全收,后续 SELECT 出来传给 new Money(...) 时构造器抛 IllegalArgumentException——异常发生在读数据时,但根因是写数据时缺约束。修复:每个用 Converter 的字段都必须显式 @Column,明确写出 precision/length/nullable,因为 Converter 不会替你做这件事。

4.8 @Embeddable 与 @Embedded

@Embeddable 把一个"值对象"的字段摊平到宿主实体的表里——没有独立表,没有独立主键,没有独立生命周期。和 AttributeConverter 的"单字段值转换"不同,@Embeddable 处理的是多字段组合(地址 = 街道 + 城市 + 邮编)。

图 4-8-1 · @Embeddable 摊平:同组件多次嵌入靠 @AttributeOverrides 区分列名

4.8.1 完整示例

@Embeddable
public class ShipmentAddress {

    @Column(name = "street", nullable = false, length = 120)
    private String street;

    @Column(name = "city", nullable = false, length = 80)
    private String city;

    @Column(name = "zip_code", nullable = false, length = 12)
    private String zipCode;

    protected ShipmentAddress() {}
}

@Entity
@Table(name = "shipments")
public class Shipment {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Embedded
    private ShipmentAddress deliveryAddress;

    @Embedded
    @AttributeOverrides({
        @AttributeOverride(name = "street",
            column = @Column(name = "return_street", nullable = false, length = 120)),
        @AttributeOverride(name = "city",
            column = @Column(name = "return_city", nullable = false, length = 80)),
        @AttributeOverride(name = "zipCode",
            column = @Column(name = "return_zip_code", nullable = false, length = 12))
    })
    private ShipmentAddress returnAddress;
}

4.8.2 生成 DDL

create table shipments (
    id                bigint generated by default as identity,
    city              varchar(80) not null,
    street            varchar(120) not null,
    zip_code          varchar(12) not null,
    return_city       varchar(80) not null,
    return_street     varchar(120) not null,
    return_zip_code   varchar(12) not null,
    primary key (id)
);

注意 @AttributeOverrides 的作用——如果不显式覆盖,两个 @Embedded ShipmentAddress 会尝试映射到相同的列名(street/city/zip_code),启动期 Hibernate 会抛 ColumnNameMappingException。

Embeddable 与 Entity 对比

维度@Embeddable@Entity
主键无必须
表摊平到宿主表独立表
生命周期跟随宿主独立
Identity Map无有
Repository无独立通常有
双向引用不能可以
适用场景值对象(地址、坐标)业务实体
@Embeddable 是"避免地址表"和"避免地址字段污染主表"之间的最佳折中 — 当业务对象有"附属的多字段属性"(用户的收货地址、商品的尺寸规格、订单的金额明细),你有三种选择:(1) 平铺到主表 — 字段散乱,语义不聚合;(2) 单独建表 + 关联 — 多一张表多一次 JOIN,事务边界变复杂;(3) @Embeddable 摊平 — Java 层有独立的值对象类(ShipmentAddress),数据库层却是宿主表里的几列,既保持代码的语义聚合,又避免表设计膨胀。这是 DDD 中"值对象"在 JPA 里最直接的落地方式。判定原则:这组字段是否有"独立生命周期 / 独立标识 / 多对一引用关系"?有则用 @Entity,无则用 @Embeddable。地址在大多数业务里没有独立标识(同一个地址不需要被两个用户共享),所以用 @Embeddable;但如果是"快递站点"(有 stationId、多个订单共享),那就必须 @Entity。

4.10.2 设计取舍

取舍 1:启动期冻结映射元模型

JPA 在启动期解析所有注解并构建 EntityPersister。运行期不再回查注解,SQL 生成路径几乎无反射开销。代价是注解修改必须重启 + 重跑 schema validation,无法热更新。这种"启动期一次解析、运行期纯查表"的设计让 Hibernate 的 SQL 生成可以做到非常激进的优化(SQL 文本缓存、参数绑定计划缓存),否则每次 flush 都要扫一遍注解,性能至少差一个数量级。

取舍 2:Java 类型与 JDBC 类型分层

BasicType 与 JdbcType 的二级抽象使得不同 Dialect 可以定制 SQL 类型代码。代价是同一 Java 类型在不同方言下生成不同 DDL,跨库项目须保留目标库快照。如果只有一层抽象(Java 类型直接到 SQL),Hibernate 就无法支持"在 PostgreSQL 用 uuid 但在 MySQL 用 binary(16) 存同一个 UUID 字段"这种跨方言差异。

取舍 3:@Column 同时影响 DDL 与运行 SQL

@Column 既参与 DDL 生成,又参与运行 SQL 列列表(insertable / updatable)。该统一带来声明简洁,但 columnDefinition 直接绑定方言文本,迁库时须人工处理。"一个注解搞定 schema 和 runtime"是 JPA 设计哲学的核心,但也意味着任何 @Column 的修改都同时影响两个面。

取舍 4:Converter 不感知列约束

AttributeConverter 仅在 Java 与 JDBC 值之间转换,不涉及 nullable / length / precision。开发者必须显式书写 @Column,避免 Converter 隐藏列约束。这条边界是有意而非疏忽——如果 Converter 既管值转换又管列约束,API 会变得复杂(Converter 接口要 return 一组列定义),而且无法处理同一 Converter 用在不同字段需要不同精度的场景。
