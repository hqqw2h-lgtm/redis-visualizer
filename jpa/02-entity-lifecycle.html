第 2 章 · 实体生命周期与状态机

Transient · Managed · Detached · Removed · persist / merge / flush / refresh / detach / remove · Dirty Checking · 乐观锁 · LazyInitialization

JPA 实体的"状态"不是对象上的字段,也不是某个 enum 标记——它是对象与当前 PersistenceContext 之间的关系。同一个 Java 对象,在事务 A 里是 Managed,事务 A 结束就变成 Detached;同一个 Detached 对象通过 merge() 进入事务 B 时,Provider 不接管原对象,而是返回一个新的 Managed 副本——你继续用原对象修改的字段全部丢失。这种"状态由关系决定、关系由 PC 生命周期决定"的语义,是初学者最容易踩坑的地方,也是高级开发者排查"代码看起来改了对象但数据库未变"的必经之路。

本章按状态机的每条边展开:每条边给出 Java 调用、SQL 日志、容器状态(em.contains)、可观测信号(异常 / 日志关键字 / 行为差异)。读完本章你应该能在生产事故里听到"merge 后修改没保存"立刻判断是用了原对象继续改;听到"事务外懒加载抛异常"立刻判断是PC 关闭后访问代理;听到"查询前突然出现 UPDATE"立刻判断是FlushMode.AUTO 的自动刷新。

"状态是关系,不是属性"——这是本章最重要的一句话。同一个 Customer 对象,在事务边界内 em.contains(c)=true 即 Managed,事务结束 em.contains(c)=false 即 Detached。换一个 EntityManager,它从未见过这个对象,也是 Detached。状态从来不"写在对象上"——你 System.out.println(c.state) 是看不到的。判定状态的唯一权威途径是询问当前 PC(em.contains / PersistenceUnitUtil)。把这一点想清楚,后面所有"为什么改了没生效"的问题都迎刃而解——因为本质上你修改的对象当时不在某个 PC 的管辖范围内。

2.1 公共示例实体

后续示例统一基于以下 Customer 实体。SEQUENCE 主键策略让 INSERT 可以推迟到 flush 时机执行,便于观察"持久化登记"与"SQL 落地"的时序分离;@Version 字段使每一次 UPDATE/DELETE 都带乐观锁条件,便于在日志里看到 version 的变化轨迹。

@Entity
@Table(name = "customers")
public class Customer {
    @Id
    @GeneratedValue(strategy = GenerationType.SEQUENCE)
    private Long id;

    private String name;
    private String email;

    @Version
    private long version;

    @OneToMany(mappedBy = "customer", fetch = FetchType.LAZY)
    private List<Order> orders = new ArrayList<>();
}

观察状态切换与 SQL 时序需要打开如下日志级别——其中 orm.jdbc.bind=TRACE 会把绑定参数也打印出来,便于看清"INSERT 时 name 字段的最终值",这是判断 dirty checking 行为的关键。

spring.jpa.show-sql=true
spring.jpa.properties.hibernate.format_sql=true
logging.level.org.hibernate.SQL=DEBUG
logging.level.org.hibernate.orm.jdbc.bind=TRACE
不要用 GenerationType.IDENTITY 做生命周期实验 — IDENTITY 策略要求 persist() 立即执行 INSERT(因为 ID 由数据库自增列产生,不 INSERT 拿不到 ID)。这会破坏"persist 不一定立即 SQL"的语义,也会让"persist 后修改字段,INSERT 用最新值"的实验失败——因为 INSERT 已经在 persist 那一刻发完了,后续修改只能走 UPDATE。SEQUENCE 与 TABLE 策略下 Provider 才能延迟 INSERT 到 flush 时机,这是观察生命周期的正确实验环境。

2.2 Transient → Managed:persist 与 merge(new)

从一个 new Customer() 让对象进入 PC 管辖,有两条路径:persist() 直接接管原对象,merge() 返回一个新的 Managed 副本。两者最大差异不是性能,而是"原对象的命运"——这一点错了,后续所有修改可能默默丢失。

图 2-2-1 · persist 的"登记 + 延迟 INSERT"时序

2.2.1 persist 的语义

persist() 把 Transient 对象登记进当前 PersistenceContext,状态切换为 Managed,但不必然立即 INSERT。SEQUENCE 与 TABLE 策略下 INSERT 推迟到 flush;IDENTITY 策略下 Provider 必须执行 INSERT 才能拿到自增主键,因此 INSERT 与 persist 同步发生。

tx.begin();

Customer c = new Customer();              // Transient
c.setName("Customer C1001");
c.setEmail("alice@example.com");

em.persist(c);                            // Managed,排队插入
c.setName("Customer C1001 V2");           // 仍 Managed,INSERT 使用最新值

tx.commit();                              // flush → INSERT

预期 SQL 日志:

select next value for customers_seq;

insert into customers (email, name, version, id)
values ('alice@example.com', 'Customer C1001 V2', 0, 1);

persist() 后对象进入 Managed,后续在同一事务内对字段的修改不会触发额外 INSERT——Provider 在 flush 时使用当时的对象快照生成最终 INSERT 语句。这就是为什么日志里 name 是 "Customer C1001 V2" 而不是 "Customer C1001":INSERT 看到的是 commit 时刻的字段值。

2.2.2 merge(new) 的特殊语义

merge() 接受一个对象,但不接管这个对象本身,而是返回一个新的 Managed 副本。原 Transient 对象在调用结束后仍然是 Transient,继续修改原对象的字段不会被 dirty checking 跟踪。

tx.begin();

Customer draft = new Customer();
draft.setName("Customer C1002");
draft.setEmail("bob@example.com");

Customer managed = em.merge(draft);       // 返回 Managed 副本
draft.setName("Customer C1002 Lost");     // 忽略,draft 仍为 Transient
managed.setName("Customer C1002 Saved");  // 跟踪,Managed 字段变化

tx.commit();
select next value for customers_seq;

insert into customers (email, name, version, id)
values ('bob@example.com', 'Customer C1002 Saved', 0, 2);

容器状态可以用 em.contains 直接验证——这是辨别"哪个对象真正被 PC 跟踪"的最快方式:

em.contains(draft)   = false
em.contains(managed) = true

Spring Data JPA 的 save() 把这个坑藏起来了——但它仍然存在。SimpleJpaRepository.save(entity) 内部判断:实体是 new 走 persist,否则走 merge。它返回 save 后的 entity,所以 Spring 推荐写法是 entity = repo.save(entity) 而不是 repo.save(entity)。我们曾经在一段代码里看到:repo.save(dto.toEntity()) 后没接收返回值,然后从 dto 构造的实体上做更多操作并期望它进入 PC——结果在 IDENTITY 策略下走 persist 还能凑合(原对象会被填回 ID),在 SEQUENCE 策略下走 merge 直接全丢。修复极其简单(改成 x = repo.save(x)),但定位花了半天——调试时所有人都盯着 SQL 看,没想到是 Java 对象引用换了。

2.3 Managed → Database:Dirty Checking 与 flush

Managed 实体有个最迷人的特性:setter 不需要显式 save。改完字段,事务一提交,UPDATE 自然就出现在数据库里。这背后的机制叫 Dirty Checking——Provider 在 flush 时拿对象的当前值与 PC 里保存的 loadedState 快照逐字段比对,差异部分自动生成 UPDATE。

图 2-3-1 · flush 时刻的字段比对与 Action 入队

2.3.1 自动 UPDATE 路径

Managed 实体的字段修改无需显式 save。事务提交或显式 flush 触发字段比对,差异部分生成 UPDATE。

tx.begin();

Customer c = em.find(Customer.class, 1L);  // SELECT,加入 PC,记录 snapshot
c.setEmail("alice.v2@example.com");        // dirty 标记

em.flush();                                 // 立即发送 UPDATE
tx.commit();                                // 无第二次 UPDATE
select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = 1;

update customers
set email = 'alice.v2@example.com',
    name = 'Customer C1001 V2',
    version = 1
where id = 1
  and version = 0;

@Version 字段是乐观锁的核心。UPDATE 的 where 子句带原 version 值,Provider 期望影响行数为 1。若并发事务已修改该行,version 不匹配将导致 0 行受影响,Provider 抛 OptimisticLockException——这是分布式系统里最便宜也最常用的并发控制手段。

2.3.2 flush 与 commit 的差异

flush 仅发送 SQL,事务在 commit 前仍可回滚。该机制可用于"早失败检查"——例如提前暴露唯一索引冲突——但不应误读为业务已成功。

tx.begin();

Customer c = em.find(Customer.class, 1L);
c.setEmail("will-rollback@example.com");

em.flush();        // SQL 已发送至数据库
tx.rollback();     // 事务回滚,数据库撤销
update customers
set email = 'will-rollback@example.com',
    name = 'Customer C1001 V2',
    version = 2
where id = 1
  and version = 1;
-- ROLLBACK
-- 后续查询 email 仍为 'alice.v2@example.com'
flush 发送 SQL: true
事务提交: false
持久化生效: false

表 2-3-1:flush 触发场景

场景触发条件默认行为
事务提交tx.commit()必触发
显式调用em.flush()必触发
查询前 (AUTO)createQuery / find 可能受脏数据影响触发
Native querycreateNativeQuery + auto flush触发
FlushMode.COMMIT显式配置仅 commit 时触发
不要根据"flush 没抛异常"判断业务成功 — flush 把 INSERT/UPDATE/DELETE 发到数据库,数据库的约束(unique / foreign key / check)能在那一刻被检测到,但事务并没有提交。我们见过反模式:Service 层先 em.flush() 然后立刻调下游远程服务"通知库存系统",再在最后 tx.commit()。问题:flush 之后 commit 之前如果发生 rollback(后续业务校验失败、或下游服务返回 4xx 触发本地异常),数据库会撤销,但远程服务已经被通知了。外部副作用必须放在 commit 之后——可以用 TransactionSynchronization 注册 afterCommit 回调,或者使用 Outbox 模式让"通知"也变成事务内的一行 INSERT。

2.4 Database → Managed:find 与 getReference

从数据库把已存在的行变成 Managed 实体,有两条路径:find 立即 SELECT 并填字段;getReference 返回一个仅持有 ID 的代理,字段访问时才触发 SELECT。在"批量构建外键关联"场景下,getReference 能省下大量无谓的 SELECT。

图 2-4-1 · find 即时加载 vs getReference 延迟代理

2.4.1 即时加载与延迟加载

tx.begin();

Customer loaded = em.find(Customer.class, 1L);          // 立即 SELECT
Customer proxy = em.getReference(Customer.class, 2L);   // 无 SELECT

Long id = proxy.getId();        // 无 SELECT,代理可返回 ID
String name = proxy.getName();   // 触发 SELECT

tx.commit();
select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = 1;

select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = 2;

类型差异——这是getReference给出的代理对象的标志:

loaded.getClass()  = com.example.Customer
proxy.getClass()   = com.example.Customer$HibernateProxy

2.4.2 何时使用 getReference

getReference() 适用于"仅需将关联指向某 ID,无需读取字段"的场景。典型用例是设置外键:

Order order = new Order();
order.setCustomer(em.getReference(Customer.class, customerId));
em.persist(order);

该场景下无需加载 Customer 字段,getReference 比 find 节省一次 SELECT。若访问非 ID 字段时 PC 已关闭,代理初始化将抛 LazyInitializationException——这是 §2.11 要重点讨论的话题。

表 2-4-1:find 与 getReference 选型

场景推荐 API原因
立即读取实体字段find直接返回数据
仅设置外键getReference节省 SELECT
校验存在性find + null 检查getReference 在不存在时抛延迟错误
批量构建关联getReference减少加载次数
事务外使用find代理在事务外失效
"getReference 是性能优化,但要会读它的失败信号"。getReference 的代价不是 0——它"延迟"了一次 SELECT,但不能消除。如果你后面访问了非 ID 字段,SELECT 还是要发;如果你只用 ID(典型外键场景),才真正省下来。判断是否真的省了 SELECT,看运行时日志:打开 SQL DEBUG 后,getReference 一行不出 SELECT 说明优化生效;出现 SELECT 说明你访问了字段,getReference 退化成了 find。生产里这两种情形混在同一段代码,得靠日志辨别——很多团队"觉得自己用了 getReference 就快了",实际访问了字段反而多了一次代理初始化的开销。

2.5 Managed → Detached:detach / clear / close

把对象从 PC 中"摘出来",最显式的途径是 detach;隐式途径是 clear(摘所有)、close(关 PC)、事务结束(PC 销毁)、跨 PC(对象从前一事务持有)。这些隐式场景是初学者最容易忽视的——很多人不知道"事务结束 = 实体自动 detach"。

2.5.1 Detached 的核心特征

Detached 实体仍是普通 Java 对象,但已脱离 PersistenceContext 跟踪。字段修改仅改变内存,flush 不会生成 UPDATE。

tx.begin();

Customer c = em.find(Customer.class, 1L);   // Managed
em.detach(c);                                // Detached

c.setEmail("lost@example.com");              // 仅修改内存

tx.commit();                                  // 无 UPDATE
select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = 1;
-- 无 UPDATE

2.5.2 隐式 Detached 的常见场景

隐式 Detached 触发场景:

  1. em.clear()              所有 Managed 实体 → Detached
  2. em.close()              EM 关闭,所有实体 → Detached
  3. 事务结束 (TRANSACTION 范围)  PC 销毁,Managed → Detached
  4. 事务边界外的实体引用       从前一事务保留的对象自动 Detached
  5. 反序列化的对象            从 byte[] 还原,JPA 不感知,实质 Detached

表 2-5-1:Detached 的 SQL 影响

操作Managed 时Detached 时
setterflush 时 UPDATE无 SQL
em.removeDELETEIllegalArgumentException
关联 setter (双向)维护双向引用仅修改内存

"Service 返回 entity 给 Controller,Controller 改字段"——这是分布式系统里隐式 Detached 最经典的事故现场。Service 方法标了 @Transactional,事务在方法返回时结束;返回的 entity 已经是 Detached;Controller 在事务外做 customer.setStatus("CONFIRMED") 期望状态被持久化——什么都不会发生。修复方式是不要让 entity 跨越事务边界:Service 应返回 DTO;如果非要返回 entity,确保 Controller 不修改它,或者用 OSIV(Open Session in View)把 PC 延长到 view 层。OSIV 的代价见 §2.11。

2.6 Detached → Managed:merge

把一个 Detached 对象重新纳入 PC,标准 API 是 merge。但 merge 不接管原对象,而是复制字段值到一个新的 Managed 副本——这一点已经在 §2.2.2 里讨论过 merge(new) 的特殊语义,这里再深入它在"已有数据库行"场景下的行为。

图 2-6-1 · merge 的"加载 + 复制 + 标 dirty"四步

2.6.1 merge 的标准用法

Customer detached = new Customer();
detached.setId(1L);
detached.setName("Customer C1001 From Request");
detached.setEmail("request@example.com");

tx.begin();

Customer managed = em.merge(detached);
managed.setName("Customer C1001 Final");
detached.setEmail("ignored@example.com");   // 忽略

tx.commit();
select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = 1;

update customers
set email = 'request@example.com',
    name = 'Customer C1001 Final',
    version = 2
where id = 1
  and version = 1;

容器状态——这是辨别"哪个对象真正进了 PC"的关键:

detached == managed       false
em.contains(detached)     false
em.contains(managed)      true

2.6.2 merge 的覆盖风险

merge 将 detached 对象的所有字段值复制到 Managed 副本。若 detached 对象来自旧表单或半空 DTO,可能将数据库中的较新字段覆盖为旧值或 null——这是生产里被吐槽最多的"merge 玄学"。

// 数据库当前状态: name="Customer C1001", email="updated-by-other@example.com"
Customer stale = new Customer();
stale.setId(1L);
stale.setName("Customer C1001");
stale.setEmail(null);                         // 旧表单未提交 email

tx.begin();
em.merge(stale);                              // 将 email 覆盖为 null
tx.commit();
update customers
set email = null,
    name = 'Customer C1001',
    version = 3
where id = 1
  and version = 2;

2.6.3 安全的修复模式

把 "detached → merge → flush" 改写为 "按 ID 加载 Managed → 白名单字段复制":

反例:merge 全字段覆盖

// 表单只勾选了 name 字段
Customer dto = parseForm(request);
dto.setId(1L);
dto.setName("New Name");
// dto.email 为 null(表单没传)

tx.begin();
em.merge(dto);   // email 被覆盖为 null
tx.commit();

正例:加载后白名单复制

tx.begin();

Customer managed = em.find(
    Customer.class, command.id());
managed.setName(command.name());  // 仅复制业务允许字段
// email 保持数据库当前值

tx.commit();
update customers
set name = 'Customer C1001',
    version = 3
where id = 1
  and version = 2;

该模式将"哪些字段可被请求修改"显式表达在代码中,避免 merge 带来的隐式覆盖。这是大多数业务表单更新场景应该走的路径。

Spring Data JPA save 默认走 merge,所以这个坑也藏在 Spring 项目里。SimpleJpaRepository.save 对已有 ID 的实体走 em.merge(entity)。我们的订单系统曾经有过 P0 事故:用户在订单 detail 页只修改了"备注"字段,Controller 接收一个仅含 id + remark 的 OrderDTO,转成 Order entity 后 orderRepo.save(o)——结果数据库里 amount / status / shippingAddress 全部被覆盖为 null。修复后所有"局部更新"接口都改成 orderService.updateRemark(id, remark),内部走"find + setter + 不写 save"的路径,依赖 dirty checking 自动产生 UPDATE。这种"Service 层提供细粒度更新方法"是表单驱动业务的稳态架构。

2.7 Managed → Removed:remove

remove 标记一个 Managed 实体为待删除,真实 DELETE 在 flush 时才发出。和 INSERT/UPDATE 一样,DELETE 也带乐观锁版本号检查,这意味着"并发删除"也会触发 OptimisticLockException。

2.7.1 标准删除路径

tx.begin();

Customer c = em.find(Customer.class, 1L);    // Managed
em.remove(c);                                 // Removed

tx.commit();                                   // flush → DELETE
select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = 1;

delete from customers
where id = 1
  and version = 3;

DELETE 同样使用 version 进行乐观锁检查。若版本不匹配,Provider 抛 OptimisticLockException。这一行为常被忽略——很多人以为乐观锁只防 UPDATE,实际 DELETE 也吃这一套。

2.7.2 detached 直接 remove 的限制

Customer detached = sessionStore.get(customerId);

tx.begin();
em.remove(detached);                          // 抛异常
tx.commit();
java.lang.IllegalArgumentException:
Removing a detached instance com.example.Customer#1

修复方式是先 find 加载 Managed 实例,或 merge 后再 remove:

tx.begin();
Optional.ofNullable(em.find(Customer.class, customerId))
        .ifPresent(em::remove);
tx.commit();

表 2-7-1:删除路径对比

路径SQL 数对 detached 友好适用场景

批量 DELETE 用 createQuery("delete ...").executeUpdate()——但它绕开 PC 与级联。bulk DELETE 不经过 PC,不触发 @PreRemove 回调,不级联到关联实体,不维护内存中已加载实体的状态。这意味着批量 DELETE 后,PC 里如果还有同 ID 的 Managed 实体,它"以为自己还存在",后续 dirty UPDATE 会针对一个已经不存在的行。修复方式:bulk 操作前后调 em.clear() 或确保 bulk 在独立的 PC 里执行。这是数据迁移 / 清理脚本最常见的踩坑点。

2.8 Removed → Managed:flush 前撤销删除

Removed 状态在 flush 前是可逆的。在同一 PC 内对 Removed 实体调用 persist,可使其回到 Managed 状态——这是状态机里少有的"反向边"。

tx.begin();

Customer c = em.find(Customer.class, 1L);
em.remove(c);                                 // Removed
em.persist(c);                                // Managed
c.setName("Customer C1001 Restored");

tx.commit();
select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = 1;

update customers
set name = 'Customer C1001 Restored',
    version = 4
where id = 1
  and version = 3;
-- 无 DELETE

该机制在业务流程中不应作为常规手段。撤销删除应通过软删除字段或显式业务流程表达,而非依赖状态机的隐式行为。把"撤销"建模成 enum status 字段(如 ACTIVE / DELETED)比依赖 PC 内部状态更稳妥——后者是 PC 内部"还没 flush"窗口内的特性,跨进程 / 跨事务都不存在。

2.9 refresh:数据库覆盖内存

refresh() 使用数据库当前值重新填充 Managed 实体,丢弃所有未 flush 的内存修改。它是状态机里另一条"自反边"——Managed → Managed,但内存值被数据库值覆盖。

tx.begin();

Customer c = em.find(Customer.class, 1L);
c.setName("Will Be Lost");                    // 内存修改

em.refresh(c);                                 // 数据库值覆盖

tx.commit();
select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = 1;

select c1_0.id, c1_0.email, c1_0.name, c1_0.version
from customers c1_0
where c1_0.id = 1;
-- 无 UPDATE
refresh 后: c.name = "Customer C1001 Restored"

refresh 的典型用途包括:数据库存在触发器修改字段、业务需要读取最新版本(例如长事务场景)、调试中验证数据库当前状态、撤销当前事务内已做的内存修改但保留事务本身。

2.10 Flush 时机:SQL 为何提前出现

"明明只查了一下,为什么日志里先打了一条 UPDATE?"——这是 FlushMode.AUTO 在工作。Provider 默认在执行 JPQL/Native 查询前判断"PC 中是否有脏实体可能影响查询结果",若是,先 flush 再查询。

批处理"flush + clear"组合是 JPA 处理百万级数据的唯一标准答案。不 flush 等于把所有 INSERT 攒到事务结束 commit 那一刻,数据库瞬时压力极大;不 clear 等于让 PC 持续累积 Managed 实体,每个实体带 loadedState 快照,内存以 O(N) 增长直到 OOM。我们见过最惨的批处理事故:导入 200 万行 CSV,程序跑到 60 万行 OOM,heap dump 里 80% 是 Customer 实体的 loadedState 数组。修复就是在循环里加上 flush + clear 这两行——内存稳定在 100MB,跑完只用 8 分钟。batch size 取 100~500,太小达不到批量效果,太大每次 flush 内存压力又上来。

2.11 LazyInitialization:Detached 后访问懒加载

LazyInitializationException 是 JPA 入门最有名的异常,本质上就是"事务 / PC 关闭后访问尚未初始化的代理"。它在 web 应用里出现的频率极高,因为 Service 标了 @Transactional,Service 返回 entity 给 Controller,Controller 渲染时才访问 customer.getOrders()——此时 PC 早就关了。

图 2-11-1 · "事务结束后访问懒集合"的事故时序

2.11.1 异常重现

Customer customer;

tx.begin();
customer = em.find(Customer.class, 1L);
tx.commit();
em.close();                                   // PC 关闭,Customer Detached

int size = customer.getOrders().size();       // 抛异常
org.hibernate.LazyInitializationException:
failed to lazily initialize a collection of role:
com.example.Customer.orders: could not initialize proxy - no Session

2.11.2 修复路径

最稳定的修复是在事务内通过 JOIN FETCH 显式加载所需关联——把"懒"在加载时刻就解决,而不是等到访问时刻才初始化。

tx.begin();

Customer customer = em.createQuery("""
        select c
        from Customer c
        left join fetch c.orders
        where c.id = :id
        """, Customer.class)
    .setParameter("id", 1L)
    .getSingleResult();

int size = customer.getOrders().size();       // 已在事务内初始化

tx.commit();
select c1_0.id, c1_0.email, c1_0.name, c1_0.version,
       o1_0.customer_id, o1_0.id, o1_0.total_amount
from customers c1_0
left join orders o1_0 on c1_0.id = o1_0.customer_id
where c1_0.id = 1;

OSIV(spring.jpa.open-in-view=true,默认开)只是把问题"延后"到视图层,不是真正修复。OSIV 让 PC 一直存活到 HTTP 请求结束,代价是:(1) 视图层在渲染时仍在执行 SQL,N+1 在视图层爆发——你看到的"慢请求"很可能是 Thymeleaf 模板里 th:each 触发的几百次 SELECT;(2) 数据库连接被持有到响应回包,连接池利用率下降;(3) 异常被向后推迟,排查时栈帧已经离根因很远。生产推荐:关 OSIV(open-in-view: false),靠 Service 层显式 JOIN FETCH 或 DTO 投影把数据"装够"。关掉的瞬间会爆出一堆 LazyInitializationException——这正是它一直在掩盖的真问题,终于浮上水面。

2.12 生命周期回调

JPA 提供 7 个回调注解,让你在状态切换的特定时刻插入自己的代码——典型用途是填充审计字段(createdAt/updatedAt)、计算 transient 派生字段(displayName)、触发缓存失效。

2.12.1 七类回调时机

@Entity
public class Customer {
    @PrePersist
    void prePersist() {
        createdAt = Instant.now();
    }

    @PreUpdate
    void preUpdate() {
        updatedAt = Instant.now();
    }

    @PostLoad
    void postLoad() {
        displayName = name + " <" + email + ">";
    }
}

表 2-12-1:回调适用边界

回调触发点适合场景不适合场景
@PrePersistpersist 后 INSERT 前createdAt / UUID 填充远程调用
@PostPersistINSERT 后审计日志强一致外部系统

回调应保持轻量,不应在回调内部调用 EntityManager 进行新查询或修改其他实体——否则可能引发递归 flush 或破坏 ActionQueue 顺序。生产里发生过 @PreUpdate 里调 em.persist(auditLog) 导致死循环 flush 的事故,就是这个原因。

2.13 状态调试

生命周期问题有时表现得很隐晦——"对象看起来对的,SQL 却没出来"。这种时候不要靠猜,直接用 JPA 标准 API 把状态问出来。

2.13.1 标准 API 检查

PersistenceUnitUtil util = emf.getPersistenceUnitUtil();

Customer c = em.find(Customer.class, 1L);
boolean managed = em.contains(c);                       // true
boolean ordersLoaded = util.isLoaded(c, "orders");      // false

em.detach(c);
boolean managedAfter = em.contains(c);                  // false
boolean ordersInit = Hibernate.isInitialized(c.getOrders());  // false

2.13.2 集成测试护栏

@Test
void detachedChangeShouldNotUpdateDatabase() {
    Customer customer = tx(() -> em.find(Customer.class, 1L));

    customer.setEmail("lost@example.com");          // 事务外修改

    String email = tx(() -> em.find(Customer.class, 1L).getEmail());
    assertThat(email).isNotEqualTo("lost@example.com");
}

该测试验证 detached 修改不影响数据库,可在 CI 中防止重构时误改事务边界。把"行为期望"写成断言,比文档里写"小心 detached"有效 100 倍——重构者一旦改坏了 CI 立刻红。

2.14.2 设计取舍

取舍 1:Identity Map 支撑 dirty checking

PC 中每个实体保留 loadedState 快照。该快照在 flush 时与当前对象逐字段比对,差异部分生成 UPDATE。代价是大对象快照占用内存。Hibernate 提供字节码增强(bytecode enhancement)选项,可改为字段级跟踪(SelfDirtinessTracker),用增强代价换取减少快照对比开销。

取舍 2:write-behind 到 flush

setter 不立即生成 SQL,而是延迟到 flush。该机制带来批处理、级联排序、唯一性合并等优化空间,代价是错误暴露时机偏晚。AUTO 模式下,Provider 在查询前自动 flush,以维持"查询能看到当前事务内未提交的修改"的语义。

取舍 3:merge 不接管原对象

merge 选择创建副本而非接回原对象,避免 PC 中已存在同 ID Managed 实例时的冲突。代价是开发者必须使用返回值;若继续使用原对象,修改将丢失。该取舍在 JPA 规范中明确定义,Provider 不允许偏离——这意味着所有 JPA 实现(Hibernate / EclipseLink / OpenJPA)在 merge 行为上完全一致。
