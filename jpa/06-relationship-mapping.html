第 6 章 · 关联映射

Owning Side · @JoinColumn · @JoinTable · mappedBy · @MapsId · CascadeType · orphanRemoval · 双向同步 · LAZY 代理 · PersistentCollection · ActionQueue · JOIN FETCH · @EntityGraph · N+1

关联映射是 JPA 把 "Java 引用图" 与 "关系数据库表+外键" 这两个完全不同的世界粘在一起的桥梁。在对象层,你写的是 order.getItems()、orderItem.getOrder()、customer.getPreferences() 这样自然的字段引用;在数据库层,这些引用最终落到 子表的外键列、父表的共享主键 或 独立的中间表 上。两边的对应关系不是 Hibernate 智能猜出来的——它由你在注解里明确指定的 owning side(关系所有方) 决定:@ManyToOne 这一端默认拥有外键列,@OneToMany(mappedBy="order") 是反向镜像,@ManyToMany 中没有 mappedBy 的一端拥有 join table 行。一旦 owning side 选错,你会看到"集合修改完全没产生 SQL""明明加了行却外键是 NULL""删父对象顺手把共享商品也删掉了"这类离奇现象。

本章把四类关联(@OneToOne、@ManyToOne、@OneToMany、@ManyToMany)放在同一张坐标系下讲清楚:Java 注解形态、数据库 DDL 形态、运行时 SQL 形态、owning side 归属、推荐 fetch 策略、典型陷阱。同时把 CascadeType 与 orphanRemoval 这两个"生命周期传播"的开关单独拆出来,因为它们在生产事故里出现的频率远高于关联本身的语义错误——一句 @ManyToMany(cascade = CascadeType.ALL) 配上一次 em.remove(order),就足以让线上的共享 Product 表批量消失。每一节都从"推荐的基线写法"开始,再展开"为什么这种写法是基线",最后给出"做错会怎样"的对照例,让你不仅知其然,还知其所以然。

  • 关联四象限:@OneToOne / @ManyToOne / @OneToMany / @ManyToMany 与三种数据库形态(子表 FK / 共享主键 / join table)的精确对应
  • Owning Side 的语义本质:外键由谁写入 — @JoinColumn 在哪一端,FK 就由哪一端的 setter 触发 UPDATE

在进入每种关联类型的细节之前,先把"四类关联"与"三种数据库形态"并排放在一张图里——所有后续讨论都在这张图的某条箭头上。

四类关联与数据库形态的映射

图 6-0 · 四类关联映射到数据库的三种物理形态

注意 @ManyToOne 和 @OneToMany(mappedBy=...) 共同指向同一个外键列——它们是同一关系的两个 Java 视角,但只有 @ManyToOne 这端会触发 FK 写入,@OneToMany 反向集合只是读镜像。这是关联映射的第一条铁律:一个外键列只能有一个 owning side,FK 写入只由 owning side 触发。

"owning side"不是哲学问题,是物理写入权问题 — 很多人在选 owning side 时纠结"业务上谁更重要",其实这个问题在 Hibernate 看来没有意义。owning side 的真正含义是"这个外键列(或这一行 join table)由哪一端的 setter 调用触发 INSERT/UPDATE"。一条 orders 表里的 customer_id 列,只能由 OrderItem.setOrder(order) 这种 owning side 写入,反向的 customer.getOrders().add(order) 在双向 mappedBy 模式下根本不会写库——它只更新 Java 内存中的集合。理解这一点后,"为什么我加了 item 但 order_id 是 NULL"这种问题瞬间找到答案:你只改了 inverse 集合,没改 owning side 的字段。

6.1 @OneToOne:外键唯一或共享主键

@OneToOne 在四类关联里最容易"看起来简单实际复杂"——简单的部分在它就是"两边各持一引用",复杂的部分在 反向 LAZY 几乎不能真正实现,以及 共享主键 vs 唯一外键 两种物理方案的取舍直接影响 ID 生成、生命周期和懒加载行为。

6.1.1 外键唯一方案

最直观的实现:子表加一列外键 + 唯一索引。这种方案与 @ManyToOne 在 DDL 层几乎一致,只多了一个 unique 约束。

@Entity
@Table(name = "customers")
public class Customer {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String username;

    @OneToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "preferences_id", nullable = false, unique = true)
    private CustomerPreferences preferences;
}

@Entity
@Table(name = "customer_preferences")
public class CustomerPreferences {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String bio;
}

对应的 DDL:

create table customer_preferences (
    id   bigint generated by default as identity primary key,
    bio  varchar(255)
);

create table customers (
    id              bigint generated by default as identity primary key,
    username        varchar(255),
    preferences_id  bigint not null unique,
    constraint fk_customers_preferences
        foreign key (preferences_id) references customer_preferences(id)
);

运行时的 SQL 序列(先建 preferences、再建 customer 引用它,最后两次 find):

insert into customer_preferences (bio) values ('JPA reader');
insert into customers (preferences_id, username) values (1, 'alice');

-- find(Customer, 1) + 触发 preferences 懒加载
select c.id, c.preferences_id, c.username from customers c where c.id = ?;
select p.id, p.bio from customer_preferences p where p.id = ?;

这里值得注意:@OneToOne(fetch = LAZY) 在 owning side(持 @JoinColumn 的 Customer)上是真懒加载——Hibernate 会为 preferences 字段生成代理,只有访问该字段时才发出第二条 SELECT。但反向端(CustomerPreferences 如果反过来声明 @OneToOne(mappedBy="preferences"))就不再是真 LAZY,详见 6.1.3。

6.1.2 共享主键方案 (@MapsId)

如果 CustomerPreferences 在业务上是 "Customer 的扩展属性",生命周期完全依附于 Customer——不会独立存在、不会被多人共享、永远跟 Customer 一一绑定——那么共享主键方案是更优解:让 preferences.id 直接复用 customer.id。

@Entity
@Table(name = "customer_preferences")
public class CustomerPreferences {
    @Id
    private Long id;

    private String bio;

    @OneToOne(fetch = FetchType.LAZY, optional = false)
    @MapsId
    @JoinColumn(name = "id")
    private Customer customer;
}

DDL 里 customer_preferences.id 同时是主键和外键,省一列、省一个唯一索引:

create table customer_preferences (
    id   bigint primary key,
    bio  varchar(255),
    constraint fk_preferences_customer
        foreign key (id) references customers(id)
);

@MapsId 让 Hibernate 知道:这个字段的 ID 不是自己生成的,而是直接拷贝自关联实体的 ID。所以 persist(preferences) 之前必须先 persist(customer) 拿到 ID,Hibernate 才能拼出 preferences.id。

6.1.3 两种方案的精确对比

维度唯一外键@MapsId 共享主键
列数多一列 unique FK复用主键
索引数多一个唯一索引复用主键索引
反向 LAZY 行为反向端不可真正 LAZY双向 LAZY 友好
生命周期子可独立存在完全依附父对象
插入顺序顺序灵活必须父在前
典型场景关联可空可换真一对一扩展

6.1.4 反向 @OneToOne 的 LAZY 限制

这是 @OneToOne 最阴险的陷阱。在唯一外键方案中,如果你在 CustomerPreferences 一端加上 @OneToOne(mappedBy="preferences", fetch=LAZY) 反向引用 Customer,即使写了 fetch = LAZY,Hibernate 也无法实现真懒加载。原因如下:

反向 @OneToOne 的"假 LAZY"原理:

  Customer ── @OneToOne @JoinColumn(preferences_id)
            ──→ CustomerPreferences   (owning, 真 LAZY)

  CustomerPreferences ── @OneToOne(mappedBy="preferences")
                       ──→ Customer  (inverse, "假 LAZY")

  加载 CustomerPreferences 时,Hibernate 必须知道:
    - 反向 customer 字段是否为 null?
    - 是 null:不能放代理,要放 null
    - 不是 null:可以放代理,等访问时再 SELECT

  问题:Hibernate 没有"customer_preferences.customer_id"列可读,
       它必须发一条 SELECT 去 customers 表反查
       (where preferences_id = ?) 才能知道是否存在。

  结论:即使声明 fetch = LAZY,实际行为是 EAGER。

解决方案有三种:

  1. 改用 @MapsId 共享主键(最干净):反向端的 ID 直接是父对象 ID,不需要反查。
  2. 启用字节码增强:Hibernate 6 的 enableLazyInitialization 可以为反向 @OneToOne 生成"延迟初始化"的字段访问拦截,把"是否存在"的判定也推迟。
  3. 避免反向引用:如果业务不需要从 Preferences 反查 Customer,直接不声明这个反向字段。
"我加了 fetch=LAZY 为什么还是 EAGER"的元凶 No.1 — 在生产 N+1 排查中,反向 @OneToOne 是出现频率仅次于"懒集合循环触发"的元凶。典型场景:列表页查 100 个 CustomerPreferences,每行触发一次 customers 表的反查 SELECT,日志里干净利落地铺出 100 条同样模式的 SELECT。代码 review 时这种问题用肉眼读不出来,必须靠 SQL 计数测试。Vlad Mihalcea 在他的博客上专门写过一篇文章《The best way to map a @OneToOne relationship with JPA and Hibernate》,他的最终建议非常直接:"如果你有选择,永远用 @MapsId,不要用反向 @OneToOne"。

6.2 @ManyToOne 与 @OneToMany

这对组合是 JPA 关联映射里使用频率最高的——任何"一个父对象有多个子对象"的业务关系(订单 → 订单行、文章 → 评论、客户 → 订单、用户 → 收藏)都落在这里。多端 @ManyToOne 永远是 owning side,这条规则没有例外。

订单与订单行的 ER 模型

图 6-2 · 订单与订单行的 ER 关系与外键归属

外键 order_items.order_id 物理上属于子表,所以"谁拥有 owning side"在数据库层面已经被决定了——OrderItem 的 @ManyToOne 端。Order.items 那端只能写 mappedBy = "order",意思是"这个集合是 OrderItem.order 的反向镜像"。

6.2.1 推荐基线:多端拥有外键

@Entity
@Table(name = "orders")
public class Order {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String status = "NEW";

    @OneToMany(
        mappedBy = "order",
        cascade = {CascadeType.PERSIST, CascadeType.MERGE},
        orphanRemoval = true
    )
    private List<OrderItem> items = new ArrayList<>();

    public void addItem(OrderItem item) {
        items.add(item);
        item.setOrder(this);
    }

    public void removeItem(OrderItem item) {
        items.remove(item);
        item.setOrder(null);
    }
}

@Entity
@Table(name = "order_items")
public class OrderItem {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String productName;
    private int quantity;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @JoinColumn(name = "order_id", nullable = false)
    private Order order;

    protected OrderItem() {}

    public OrderItem(String productName, int quantity) {
        this.productName = productName;
        this.quantity = quantity;
    }

    public void setOrder(Order order) {
        this.order = order;
    }
}

这段代码包含了所有"双向 @OneToMany 的最佳实践":

  • 显式 fetch = LAZY — @ManyToOne 默认 EAGER 是 JPA 规范的历史包袱,生产代码必须显式覆盖。
  • optional = false + nullable = false — 业务上订单行必须属于某个订单,这两条联手让数据库和 ORM 一起拦住"游离的 OrderItem"。
  • cascade = {PERSIST, MERGE} — 保存订单时同时保存订单行;不级联 REMOVE,删除语义交给 orphanRemoval 控制。
  • orphanRemoval = true — 从集合移除一个 item 等于删除该 item 行,而不是把 order_id 置 NULL。
  • addItem / removeItem 双向同步方法 — 同时维护内存里的双向引用和数据库的 FK,封装好了不让外部直接操作集合。

6.2.2 持久化与 SQL

Order order = new Order();
order.addItem(new OrderItem("Keyboard", 1));
order.addItem(new OrderItem("Mouse", 2));

em.persist(order);
em.flush();
insert into orders (status) values ('NEW');
insert into order_items (order_id, product_name, quantity) values (1, 'Keyboard', 1);
insert into order_items (order_id, product_name, quantity) values (1, 'Mouse', 2);

addItem 同时维护双向引用:既添加到集合(让 Java 内存图保持一致),又设置 item.order = this(让 owning side 的 FK 字段被正确写入)。后者是写库的关键——前者只是让你下面 order.getItems().size() == 2 这种代码不踩空。

6.2.3 仅修改反向集合的错误

反例:只改集合不改 owning side

Order order = em.find(Order.class, 1L);
OrderItem item = new OrderItem("Monitor", 1);

order.getItems().add(item);    // ❌ 仅添加到集合
em.persist(item);              // item.order 仍为 null
em.flush();

items.add(item) 只更新内存集合。
item.order 字段从未被设置。
flush 时 owning side FK 是 NULL,被 nullable=false 拦下。

正例:走双向同步方法

Order order = em.find(Order.class, 1L);
OrderItem item = new OrderItem("Monitor", 1);

order.addItem(item);           // ✅ 双向同步
em.flush();

addItem 同时设置 items 和 item.order。
cascade=PERSIST 触发 item 的 INSERT。
FK 列正确写入 order_id。

反例触发的错误日志通常是:

[Error] org.hibernate.PropertyValueException:
not-null property references a null or transient value: OrderItem.order

  或数据库约束兜底:

[Error] insert into order_items (order_id, ...) values (NULL, ...)
        ERROR: NULL not allowed for column "ORDER_ID"

修复方式:所有添加 / 删除均通过 addItem / removeItem,在团队规范里禁止 entity 类对集合字段加 Lombok 的 @Setter,从源头堵住"直接 setOrders(newList)"或"order.getItems().add(item)"这种破坏 owning side 的写法。

6.2.4 单向 @OneToMany 的隐式中间表

如果你只想要"父对象到子对象的引用",不想在子对象上声明反向 @ManyToOne,可以写单向 @OneToMany:

@OneToMany       // 缺少 mappedBy 也缺少 @JoinColumn
private List<OrderItem> items = new ArrayList<>();

这种写法看起来"省事",但 Hibernate 在缺少明确指示时会自动生成 join table:

create table orders_order_items (
    order_id  bigint not null,
    items_id  bigint not null unique
);

这是 JPA 规范定的默认行为。后果:

  • 每次集合操作触发 INSERT / DELETE 中间表行,而不是 UPDATE 子表 FK。
  • 子表 order_items 上根本没有 order_id 列,反向 SQL 查询不能用。
  • 多了一张表的维护成本,DDL 评审时容易被忽略。

修复:补 @JoinColumn(name="order_id"),让 Hibernate 在子表加 FK 列;或者改为双向 mappedBy。

@OneToMany 配置选项对照

配置DDL 形态SQL 行为推荐度
双向 mappedBy子表 FK子端 INSERT/UPDATE FK★★★ 推荐
单向 + @JoinColumn子表 FK父端 UPDATE 子表 FK★ 一般

6.3 @ManyToMany 与中间表

@ManyToMany 看起来是"两端各持一个集合,中间一张 join table 连接"的对称结构,但 owning side 概念在这里依然存在——没有 mappedBy 的那一端是 owning side,join table 的行只由它的修改触发写入。

6.3.1 标准 @JoinTable

@Entity
@Table(name = "orders")
public class Order {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    @ManyToMany
    @JoinTable(
        name = "order_product",
        joinColumns = @JoinColumn(name = "order_id"),
        inverseJoinColumns = @JoinColumn(name = "product_id")
    )
    private Set<Product> products = new HashSet<>();

    public void addProduct(Product p) {
        products.add(p);
        p.getOrders().add(this);
    }
}

@Entity
@Table(name = "products")
public class Product {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String title;

    @ManyToMany(mappedBy = "products")
    private Set<Order> orders = new HashSet<>();

    public Set<Order> getOrders() { return orders; }
}

对应 DDL:

create table orders   (id bigint primary key, name  varchar(255));
create table products (id bigint primary key, title varchar(255));

create table order_product (
    order_id    bigint not null,
    product_id  bigint not null,
    primary key (order_id, product_id),
    constraint fk_op_order   foreign key (order_id)   references orders(id),
    constraint fk_op_product foreign key (product_id) references products(id)
);

运行时:

insert into orders   (name)  values ('Customer C1001');
insert into products (title) values ('JPA');
insert into order_product (order_id, product_id) values (1, 1);

仅 owning side(Order.products)端的修改触发 join 行写入。inverse side(Product.orders)仅作为读镜像,即使你写 product.getOrders().add(order),也不会产生任何 SQL。这是 @ManyToMany 双向最容易踩的坑——大量初学者以为两端都要改,结果两端都改也只有一端真正生效。

注意我们这里用 Set 而不是 List。原因是 @ManyToMany 的 join table 主键是 (order_id, product_id),天然不允许重复;Java 端用 Set 与之对应可以让 Hibernate 跳过昂贵的"查重 + DELETE-then-INSERT"逻辑(List 默认每次集合改动都会先 DELETE 全部 join 行再 INSERT 当前所有项)。这是 Vlad Mihalcea 的强烈建议,生产里也建议遵循。

6.3.2 业务型多对多:升级为中间实体

纯 @ManyToMany 适合"标签/角色/权限"这类不带业务字段的关系。一旦关系本身需要带字段(数量、单价、加入时间、状态、是否已删除……),就不应再用 @ManyToMany,而应升级为独立的中间实体。

图 6-3 · "订单 ↔ 商品" 升级为带业务字段的中间实体
@Embeddable
public class OrderItemId implements Serializable {
    @Column(name = "order_id")
    private Long orderId;

    @Column(name = "product_id")
    private Long productId;

    @Override
    public boolean equals(Object o) {
        return (o instanceof OrderItemId other)
            && Objects.equals(orderId, other.orderId)
            && Objects.equals(productId, other.productId);
    }

    @Override
    public int hashCode() { return Objects.hash(orderId, productId); }
}

@Entity
@Table(name = "order_items")
public class OrderItem {
    @EmbeddedId
    private OrderItemId id = new OrderItemId();

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @MapsId("orderId")
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    @MapsId("productId")
    @JoinColumn(name = "product_id")
    private Product product;

    private Integer quantity;
    private BigDecimal unitPrice;
}

@MapsId("orderId") 让 OrderItemId.orderId 这个 ID 字段直接复用 order.id,不需要单独的列。这种"复合主键 + 两个 @MapsId"是 JPA 表达"业务型多对多"的标准范式,Spring Data JPA 也完全兼容(JpaRepository<OrderItem, OrderItemId>)。

@ManyToMany 与中间实体对比

维度@ManyToMany中间实体
业务字段不支持支持 (quantity / unitPrice / status / 时间戳)
唯一约束仅复合主键任意业务约束
历史审计不可可加 @CreatedDate 等
可查询性关系不可被独立查询独立 Repository
fetch 控制整批 lazy / eager每端独立配置
推荐场景纯标签 / 角色权限业务关系
"先用 @ManyToMany 等以后有需求再改"是头号架构债 — 这种思路在代码上线后会让整个团队后悔。从 @ManyToMany 升级到中间实体不是改两个注解的事:你需要新建一个实体类、新建复合主键、改两端的集合泛型、改所有 Repository 接口、改所有调用点,还要写一个数据迁移脚本把旧 order_product 数据搬到新结构。任何只要怀疑未来可能加业务字段的关系,从第一天就用中间实体。代价是多写一个类,收益是永远不需要做这次痛苦的迁移。

6.4 CascadeType 与生命周期传播

CascadeType 控制"对父对象的操作是否传播到子对象"。它本质上是一个开关组合,但每种 type 的语义边界很容易混,生产里出过的事故大多源自于"以为只是个写起来方便的快捷方式,实际是把生命周期完全绑死"。

七种 Cascade 语义

CascadeType触发动作适合场景风险
PERSISTparent persist 时子对象一并 persist保存订单时一并保存订单行子对象必须独占
MERGEparent merge 时子对象一并 merge表单合并明细detached 图过大不易控
REMOVEparent remove 时一并删除子对象删父删独占子误删共享数据
REFRESHparent refresh 时一并 refresh 子同步父子状态频繁触发额外 SQL
DETACHparent detach 时一并脱管子父子同步脱离罕用
ALL包含上述五种严格独占聚合根放在 @ManyToMany 是事故源

6.4.1 推荐配置

// 推荐:订单行独占 → PERSIST + MERGE + orphanRemoval
@OneToMany(
    mappedBy = "order",
    cascade = {CascadeType.PERSIST, CascadeType.MERGE},
    orphanRemoval = true
)
private List<OrderItem> items = new ArrayList<>();

// 危险:Product 是共享实体 → 不应级联 REMOVE
@ManyToMany(cascade = CascadeType.ALL)         // ❌ 反例
private Set<Product> products = new HashSet<>();

这两段代码并排放是有意的——它们在语法上几乎一样,但语义后果天差地别。OrderItem 是订单的独占子,删订单时删它没问题;Product 是被多个订单共享的全局实体,删订单时绝对不能跟着把 Product 删了。

6.4.2 误用 @ManyToMany + ALL 的后果

-- em.remove(order) 触发 cascade ALL,SQL 序列:
delete from order_product where order_id = ?;
delete from products where id = ?;     -- ❌ 错误:删除共享 Product
delete from orders where id = ?;

如果数据库的 order_product 表对 product_id 列有 FK 约束,而其他订单还在引用这个 product,这条 DELETE 会被 FK 约束兜底失败,事务回滚——但失败信息常常被误诊为"数据库锁问题"。如果 Product 当前没被其他订单引用,删除会成功,导致共享实体丢失:其他依赖 Product 的代码下次跑时报"找不到 product"。

生产规则:

  • @ManyToMany 最多 使用 PERSIST / MERGE,绝不使用 REMOVE 或 ALL。
  • @OneToMany 仅在子对象是聚合根独占子时使用 REMOVE 或 ALL;否则只用 PERSIST + MERGE。
  • 任何 cascade = CascadeType.ALL 的代码都应该在 review 阶段被特别盯一下,确认子对象的生命周期。
真实事故:某中台用 @ManyToMany(cascade = ALL) 删掉了全局商品库 — 我亲历过的一次事故:某团队在订单实体上用 @ManyToMany(cascade = CascadeType.ALL) Set<Product> 关联商品。运行半年没事,因为业务上从来不删订单。某天产品同学要清理"测试订单",批量调了 orderRepository.deleteAll(testOrders)。该批 testOrders 引用了若干商品,这些商品同时也被真实订单引用,但 cascade = ALL 让 Hibernate 对每个 Product 都触发了 DELETE。数据库的 order_product 表上 FK 约束设的是 ON DELETE CASCADE,于是真实订单的 join 行也被一起删除。最终结果:整个商品库被删掉一半,真实订单丢失了商品引用,商家端展示的"订单详情"全部空白。恢复花了 6 小时,期间下单功能停摆。事故根因 review 时,大家盯着 cascade = ALL 那一行愣了很久——它"看起来只是个简化代码的开关"。

6.5 orphanRemoval

orphanRemoval 容易被当作 CascadeType.REMOVE 的别名,但二者的触发条件完全不同——前者跟着 "从父对象的集合中移除子对象",后者跟着 "em.remove(parent)"。

Cascade.REMOVE 与 orphanRemoval 的区别

图 6-5 · 两种"删除子对象"的触发路径

6.5.1 行为对照

Order order = em.find(Order.class, 1L);
OrderItem removed = order.getItems().get(0);

order.removeItem(removed);
em.flush();
-- orphanRemoval = true
delete from order_items where id = ?;

-- orphanRemoval = false, FK nullable
update order_items set order_id = null where id = ?;

-- orphanRemoval = false, FK not null
ERROR: NULL not allowed for column "ORDER_ID"

这三种结果对应同一段 Java 代码,只因 orphanRemoval 配置不同。生产里"我从订单移除了一个 item,为什么 DB 里这一行还在,只是 order_id 变成 NULL"这种问题,99% 是 orphanRemoval 没开。

两种触发场景对照

场景Cascade.REMOVEorphanRemoval=true
em.remove(order)删除全部子行删除全部子行(同样行为)
order.removeItem(item)解除外键或约束错误删除该子行
order.getItems().clear()解除全部外键删除全部子行
子可被多人共享不应使用不应使用
推荐场景真聚合根独占子实体
"orphanRemoval 不要用在 @ManyToOne 反向集合以外的任何关联" — orphanRemoval 在 JPA 规范里只对 @OneToMany 和 @OneToOne 有意义。但实际上你只应该把它用在 @OneToMany(mappedBy=...) 这种"父对象独占一组子对象"的场景。任何形如"我有一组共享的对象,父对象只是引用"的场景都不应用——一旦你 collection.remove(x),Hibernate 就直接 DELETE 了 x,而 x 可能还在被另一个父对象引用。这种情况下 DB 层的 FK 约束会兜底失败,但失败时已经是事务异常,业务流程被打断。安全的判断标准:能不能写出"这个子对象只属于这一个父对象,父对象一删它就不该存在"——能,才用 orphanRemoval。

6.6 读取与 SQL 数量

关联映射对 SQL 数量的影响,是生产性能问题里最常见的根源——N+1、懒加载触发风暴、反向 @OneToOne 假 LAZY,都属于"代码看起来对,SQL 数量爆炸"的范畴。本节把 fetch 策略组合的 SQL 数量基线钉死,让排查时能直接对照。

fetch 策略组合

图 6-6 · 四种 fetch 策略的 SQL 形态对比

6.6.1 显式 LAZY 与 fetch join

public interface OrderRepository extends JpaRepository<Order, Long> {
    @Query("""
        select distinct o
        from Order o
        left join fetch o.items
        where o.id = :id
        """)
    Optional<Order> findWithItems(@Param("id") Long id);
}

对照两种调用方式的 SQL 形态:

-- findById + 触发懒加载: 2 次 SELECT
select o.id, o.status from orders o where o.id = ?;
select i.order_id, i.id, i.product_name, i.quantity
from order_items i where i.order_id = ?;

-- findWithItems: 1 次 SELECT (但行数被笛卡尔放大)
select distinct o.id, o.status, i.id, i.order_id, i.product_name, i.quantity
from orders o
left join order_items i on i.order_id = o.id
where o.id = ?;

注意 distinct 关键字。JOIN FETCH 的语义是 SQL 层的笛卡尔积——一个订单 3 个 items 会返回 3 行,Hibernate 在 ResultSetTransformer 里把同一 Order 对象去重,但 SQL 文本里的 distinct 让 DB 在网络层提前去重,减少传输量。Hibernate 6 引入了 PassThroughDistinct 行为来避免不必要的 DB 端 distinct,但旧版还是建议加上。

fetch 策略性能基线

操作期望 SELECT 数风险点
findById + 访问懒集合2可接受
findWithItems (JOIN FETCH)1需 distinct 去重
列表 20 条 + 集合访问21 (N+1)必须修复

6.6.2 @EntityGraph 的优势

@EntityGraph(attributePaths = {"items"})
@Query("select o from Order o where o.status = :status")
List<Order> findActiveOrders(@Param("status") String status);

@EntityGraph 在 JPQL 之外声明"这次查询额外要加载哪些关联",Hibernate 会自动转成 LEFT JOIN FETCH。它的好处是同一个 JPQL 可以被不同调用点用不同 EntityGraph 复用——只读列表用最小图,详情页用全图,而不需要写多份 JPQL。Spring Data JPA 也直接支持。

6.6.3 DTO 投影的极致性能

public record OrderListItem(Long id, String status, int itemCount) {}

@Query("""
    select new com.example.OrderListItem(o.id, o.status, size(o.items))
    from Order o
    where o.status = :status
    """)
List<OrderListItem> findActiveOrderSummary(@Param("status") String status);

当列表场景只需要少数几个字段(如订单状态 + 行数),DTO 投影是最优解:不加载实体进 PC、没有 dirty checking 成本、网络传输列最少。代价是写多个 DTO 类、不能再走 PC 缓存(下次 find 同 ID 还要查一次)。

6.7.1 PersistentCollection 的差异计算

PersistentBag 内部状态:

  ┌────────────────────────────────────────────────┐
  │  PersistentBag                                  │
  ├────────────────────────────────────────────────┤
  │  bag           List<E>      实际持有元素          │
  │  storedSnapshot List<E>     加载时的快照副本       │
  │  dirty         boolean      是否有变更             │
  │  initialized   boolean      是否已从 DB 加载完成    │
  │  owner         Object       拥有该集合的实体        │
  │  role          String       例如 "Order.items"   │
  └────────────────────────────────────────────────┘

  flush 时差异计算 (getOrphans):
    storedSnapshot = [item1, item2, item3]   ← 加载时
    bag            = [item1, item3, item4]   ← 当前
    → orphans = [item2]   ← 被移除的
    → adds    = [item4]   ← 新增的

  对于 @OneToMany + orphanRemoval=true:
    orphans 触发 DELETE
    adds    触发 INSERT

这是为什么"重新 new 一个 ArrayList 替换字段"会破坏跟踪——新 ArrayList 不是 PersistentBag,根本没有 storedSnapshot 这个字段,Hibernate 拿不到差异基准。具体原理参见第 3 章持久化集合一节。

6.7.2 设计取舍

取舍 1:owning side 唯一

外键只由 owning side 写入,避免双向关系两端同时维护同一列。代价是开发者必须了解 mappedBy 的方向,并通过同步方法保持双向一致。Hibernate 不尝试"智能猜测"哪一端是数据来源,因为双向对象图可被任意修改至不一致——只有显式声明才能避免歧义。

取舍 2:懒加载用代理

@ManyToOne / @OneToOne 默认通过代理实现懒加载,父实体加载时不查询子实体,直到访问字段触发 SELECT。代价是序列化、equals/instanceof 检查、JSON 输出可能因代理类型而异常(序列化器看到的是 Customer$HibernateProxy$xxx 而不是 Customer)。生产中需通过 DTO 或 Hibernate.unproxy 处理。

取舍 3:集合用 PersistentCollection

JPA 加载实体后,Hibernate 将集合字段替换为 PersistentBag / PersistentSet 代理。该代理跟踪 add / remove 操作,在 flush 时计算差异并生成 SQL。代价是直接替换字段(entity.setItems(new ArrayList<>()))会破坏跟踪,导致集合修改不被记录。这是为什么团队规范应该禁止 entity 类对集合字段加 setter。

取舍 4:cascade 与 orphanRemoval 独立

cascade 控制操作传播,orphanRemoval 控制集合移除时的删除。两者语义独立,使得"删除父对象时保留子对象"与"从集合移除子对象时删除子对象"可分别配置。代价是开发者须区分两者的触发条件——这正是 6.5 节专门用一节讲清楚的内容。

6.8.1 双向 JSON 序列化循环

双向关联的另一个生产高频问题是 JSON 序列化时无限循环——order.items[0].order.items[0].order...,直到 StackOverflowError。常见三种修复:

// 方案 1: Jackson @JsonManagedReference / @JsonBackReference
@OneToMany(mappedBy = "order")
@JsonManagedReference
private List<OrderItem> items;

@ManyToOne
@JoinColumn(name = "order_id")
@JsonBackReference
private Order order;

// 方案 2: @JsonIgnore 在反向端
@ManyToOne
@JsonIgnore
private Order order;

// 方案 3: DTO 投影,完全脱离实体序列化
public record OrderDto(Long id, String status, List<OrderItemDto> items) {}

推荐方案 3。前两种方案能解决眼前问题但耦合了"持久化模型"和"前端契约";一旦前端要新字段,你要改实体类,而实体类改动会触发整个 ORM 的回归测试。

6.8.2 equals/hashCode 与集合关联

第 3 章讨论过 equals/hashCode 应基于业务键。关联映射场景里多了一条规则:equals/hashCode 永远不要参与关联字段。原因:

  • 关联字段是懒加载代理,访问会触发 SELECT,在 hashCode() 里触发是隐藏成本。
  • 关联字段可能是另一个实体的引用,它的 equals 又可能依赖关联,递归循环。
  • HashSet/HashMap 在第一次 add 时计算 hashCode,关联字段值在事务后期可能变化,造成 contains() 失败。

6.9 回归护栏

关联映射的护栏不是单元测试就够,需要"行为级"的集成测试——直接对照 SQL 数量、表行数、FK 列值断言。这种护栏跑得起来,关联类的回归就稳了。

6.9.1 owning side 守护

@Test
void addItemShouldWriteForeignKeyOnOwningSide() {
    Order order = new Order();
    order.addItem(new OrderItem("Mouse", 2));

    em.persist(order);
    em.flush();
    em.clear();

    Long count = em.createQuery("""
        select count(i) from OrderItem i where i.order.id = :orderId
        """, Long.class)
        .setParameter("orderId", order.getId())
        .getSingleResult();

    assertThat(count).isEqualTo(1L);
}

这个测试用 em.clear() 强制清空 PC,再用 JPQL 查询确认 DB 里 order_items.order_id 真正写入。如果有人把 addItem 改成只 items.add(item) 不 item.setOrder(this),这个测试会失败,在 CI 里拦下。

6.9.2 SQL 数量护栏

@Test
void findWithItemsShouldUseSingleSelect() {
    sqlCounter.clear();

    Order order = orderRepository.findWithItems(1L).orElseThrow();
    int size = order.getItems().size();

    assertThat(sqlCounter.selectCount()).isEqualTo(1);
}

sqlCounter 通常用 Hypersistence Utils 的 SQLStatementCountValidator 或自己用 Hibernate Statistics 实现。这个测试守护"findWithItems 必须只发一条 SELECT"——任何让它退化成 N+1 的改动(比如有人不小心删了 JOIN FETCH)都会被拦下。

6.9.3 join table 守护

@Test
void addingProductShouldOnlyInsertJoinRow() {
    Order order = orderRepository.save(new Order("Customer C1001"));
    Product product = productRepository.save(new Product("JPA"));

    order.addProduct(product);
    em.flush();

    assertThat(countRows("order_product")).isEqualTo(1);
    assertThat(em.find(Product.class, product.getId())).isNotNull();
}

这个测试守护两件事:(1) 加 product 只新增一行 join table,不删旧行;(2) Product 实体本身没被误删。如果有人把 @ManyToMany 改成 cascade = ALL,某些后续操作可能让 Product 消失,这个测试会失败。

@OneToMany mappedBy反向镜像不写库PersistentCollection不暴露集合 setter
@ManyToMany真实数据在 join table@JoinTable业务字段需中间实体
双向同步方法addItem 同时维护两端用户代码仅改集合不改 owning
Cascade生命周期边界Cascade 引擎共享实体不级联 REMOVE
orphanRemoval集合移除等于 DELETECollectionEntry.getOrphans用于共享子的灾难
fetch 策略显式 LAZY + JOIN FETCHJPQL / EntityGraphN+1 隐藏在懒加载
性能回归SQL 数量基线Hibernate Statistics无护栏退化无人发现
JSON 序列化用 DTO 而非 entity用户代码双向引用 StackOverflow
equals/hashCode业务键,不含集合用户代码关联字段参与身份

最小回归用例:

@Test
void onlyOwningSideChangeWritesForeignKey() {
    Order order = new Order();
    OrderItem item = new OrderItem("Mouse", 2);

    order.getItems().add(item);     // 仅修改集合
    em.persist(order);

    boolean rejected = false;
    try {
        em.flush();
    } catch (PropertyValueException ex) {
        rejected = true;
    }
    assertThat(rejected).isTrue();
}
[rejected]  true
[结论]      仅修改 inverse 集合不会写入 FK,Hibernate 在 flush 时拒绝
