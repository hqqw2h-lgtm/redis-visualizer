第 19 章 · 连接池与 HikariCP

ConcurrentBag · FastList · 字节码代理 · 利特尔法则 · 泄漏检测 · OSIV · 多数据源

每一条 SQL 的执行始于一次数据库连接。从零建立连接需要 TCP 三次握手、TLS 协商与数据库认证三层开销,典型耗时同机房 5 至 15 毫秒、跨区域 100 至 500 毫秒。一次普通 OLTP 查询本身只需 0.5 至 2 毫秒,建连成本远超查询本身。

HikariCP 是 Spring Boot 2.x 起的默认连接池,以 ConcurrentBag 无锁结构、字节码代理、固定池策略三项设计把连接池自身开销压到纳秒级。本章按工作原理、关键参数、监控指标、泄漏检测、性能量化五维展开。

  • 建连 3 阶段(TCP + TLS + Auth)的耗时分布,以及池复用如何把毫秒级开销降到纳秒
  • HikariCP 5 大组件:HikariPool / ConcurrentBag / PoolEntry / ProxyConnection / HouseKeeper
  • ConcurrentBag 三级查找:ThreadLocal → sharedList → SynchronousQueue 阻塞等待
  • PoolEntry 4 态状态机:NOT_IN_USE / IN_USE / RESERVED / REMOVED
  • FastList 去边界检查 + 字节码代理 close 拦截的微观优化
  • 12 个核心参数:maxPoolSize / minIdle / connectionTimeout / maxLifetime / idleTimeout / keepaliveTime / leakDetectionThreshold
  • 3 种池容量公式:PostgreSQL 经典公式 / 利特尔法则 / 监控驱动法
  • Micrometer + Prometheus 7 大指标 + 告警阈值
  • leakDetectionThreshold + 堆栈定位的连接泄漏排查
  • OSIV 对连接持有时间的影响:从 service 层延长到整个请求
  • 多数据源(主写从读)+ AbstractRoutingDataSource 主从路由
  • HikariCP 选择 ConcurrentBag 而非 BlockingQueue 的设计取舍

前置知识:第 15 章事务管理;TCP / TLS 握手基础;JDBC DataSource API;Spring Boot Actuator 与 Micrometer;PostgreSQL / MySQL 连接进程模型。

// FastList: ArrayList 去边界检查 + 尾部反向 remove
public T get(int index) {
    return (T) elementData[index];
}

public boolean remove(Object element) {
    for (int index = size - 1; index >= 0; index--) {
        if (element == elementData[index]) {
            int numMoved = size - index - 1;
            if (numMoved > 0) {
                System.arraycopy(elementData, index + 1, elementData, index, numMoved);
            }
            elementData[--size] = null;
            return true;
        }
    }
    return false;
}

HikariCP 的 ConnectionProxy / StatementProxy / ResultSetProxy 在构建期由 JavassistProxyFactory 生成,部署时已经是直接调用的 final 子类,运行期不再触发反射。代理的核心职责是把 close() 重定向为归还池。

19.5 池容量计算

[PostgreSQL 经典公式]
pool = (core_count × 2) + effective_spindle_count

4 核 + SSD:  pool = 8 + 1 = 9        → maxPoolSize = 10
16 核 + NVMe: pool = 32 + 2 = 34     → maxPoolSize = 32-36
32 核 + 读副本: pool = 64 + 4 = 68    → 主库 32 + 副本 32

[利特尔法则]
L = λ × W
L = 池大小, λ = 请求到达率(/s), W = 单请求平均连接持有时长(s)
服务画像RPS平均持有安全因子推荐池
轻读取(高缓存命中)2005 ms1.55(下限)
标准 OLTP50050 ms1.540
复杂 join 报表50300 ms2.030
写密集订单80080 ms1.596(需读写分离)
后台批处理52000 ms1.212
 
[监控驱动法 - 一周观测]
hikaricp_connections_active{pool="shop-write-pool"}
  P50 = 6, P95 = 14, P99 = 18, max = 22, stddev = 4

[修正]
maxPoolSize = ceil(P95 + 2 × stddev) = ceil(14 + 8) = 22

19.10 多数据源

@Configuration
public class DataSourceConfig {

    @Primary
    @Bean
    @ConfigurationProperties("spring.datasource.write.hikari")
    DataSource writeDataSource() {
        return new HikariDataSource();
    }

    @Bean
    @ConfigurationProperties("spring.datasource.read.hikari")
    DataSource readDataSource() {
        return new HikariDataSource();
    }

    @Bean
    DataSource routingDataSource(DataSource writeDataSource,
                                  DataSource readDataSource) {
        Map<Object, Object> targets = Map.of(
            "write", writeDataSource,
            "read", readDataSource);

        AbstractRoutingDataSource routing = new AbstractRoutingDataSource() {
            @Override
            protected Object determineCurrentLookupKey() {
                return TransactionSynchronizationManager.isCurrentTransactionReadOnly()
                    ? "read" : "write";
            }
        };
        routing.setTargetDataSources(targets);
        routing.setDefaultTargetDataSource(writeDataSource);
        return routing;
    }
}

19.13 设计取舍

设计选择HikariCP替代方案
借还容器ConcurrentBag(线程亲和)BlockingQueue(严格 FIFO)
代理生成Javassist 静态代理JDK 动态代理(反射)
池容量策略固定池(min=max)动态扩容(C3P0)
List 实现FastList 去检查ArrayList 带边界检查
状态同步AtomicInteger CASsynchronized 锁

为什么 ConcurrentBag 而非 BlockingQueue? 标准 BlockingQueue 保证 FIFO,但每次 take 都要竞争同一把锁。ConcurrentBag 牺牲 FIFO 公平性,改用 ThreadLocal 缓存 + CAS 抢占,允许同一线程优先复用上次借的连接。代价是某些连接可能长时间不被借出,需要 HouseKeeper 兜底回收。
