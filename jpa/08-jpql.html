第 8 章 · JPQL 深度解析

SQM · SQL AST · QueryInterpretationCache · JOIN FETCH · 集合 fetch 分页 · Bulk update/delete · DTO 构造器投影 · 命名参数 · type/treat · auto flush · HHH90003004

JPQL 长得像 SQL,但它根本不是 SQL——它是面向实体的查询语言。当你写 select c from Customer c join c.orders o,你查询的对象是 Java 类 Customer 而不是表 customers;你导航的是 Java 属性 c.orders(JPA 关联映射)而不是 SQL 的 JOIN ON customer_id = id。Hibernate 在运行期把 JPQL 解析为 SQM(Semantic Query Model)语义模型,再把 SQM 转成 SQL AST,最后由 JdbcSelectExecutor 渲染成具体方言的 SQL 文本并执行。这个"实体语言→SQL"的翻译管线是 JPQL 力量的源头——字段重命名 IDE 自动传播、分页语法跨数据库统一、关联导航天然类型安全;但也是 JPQL 陷阱的来源——JOIN 不加载关联(只过滤)、JOIN FETCH 限一个集合、Bulk JPQL 绕过 PersistenceContext 留下脏对象、DTO 构造器参数类型只在运行期检查。这些"长得像 SQL 但语义不同"的差异,如果不在脑子里建立"JPQL 是另一种语言"的认知,就会反复踩坑。

本章按"语法层 → 执行层 → 陷阱层"三层展开。第 1-2 节讲清楚 JPQL 与 SQL 的根本差异,以及为什么参数必须用绑定不能拼字符串;第 3 节 JOIN vs JOIN FETCH 是整个 JPA 学习曲线最重要的一对概念,误用就是 N+1 灾难的源头;第 4 节集合 fetch + 分页是另一个生产高频事故(HHH90003004 警告下的内存爆炸),给出"两步分页"的标准修复模式;第 5 节 DTO 构造器投影是 API 列表页的最佳实践,但它的运行期类型检查有反直觉陷阱;第 6 节 Bulk update/delete 是大批量场景必须用、但用完必须 em.clear() 的逃生通道;第 7-8 节进入 SQM 内部和 Hibernate 翻译链路;最后两节把生产里反复出现的 10 个陷阱与 4 个回归测试模板整理出来。

  • JPQL 与 SQL 的本质差异:实体类 vs 表名、属性名 vs 列名、路径表达式 vs JOIN ON、跨厂商方言统一
  • 命名参数 / 位置参数 / 字符串拼接三种参数绑定方式的安全性 + 查询计划缓存命中率差异
  • JOIN(过滤)vs JOIN FETCH(加载)的本质区别 — 普通 JOIN 仍可能 N+1
  • 多集合 JOIN FETCH 引发笛卡尔积,distinct c 仅去重根实体
  • HHH90003004 警告:集合 fetch + 分页 = 内存分页,生产事故的高频源
  • "两步分页"安全模式:第一步查 ID 页,第二步 IN 列表 + JOIN FETCH 不带 LIMIT
  • DTO 构造器投影 select new ... 的语法与运行期类型检查陷阱(COUNT 是 Long,不是 Integer)
  • Bulk JPQL update/delete 直接走 JDBC,不更新 PersistenceContext,需 em.clear() 或 refresh
  • Bulk JPQL 不触发 @PreUpdate / @PostUpdate 等生命周期回调,审计字段会缺失
  • JPQL 内置函数(LENGTH/COALESCE/CASE WHEN)+ 子查询限制(WHERE 允许、FROM 不允许)
  • type(p) = CreditCardPayment 与 treat(p as CreditCardPayment).cardNumber 在继承结构下的多态查询
  • SQM 翻译链路:JPQL → HqlParser → SqmSelectStatement → SQL AST → Dialect 渲染 → PreparedStatement
  • "现象 / 根因 / 修复"10 个陷阱清单 — 从 N+1 到 IN 列表过大到 FROM 子查询

前置知识:第 3 章持久化上下文(理解 auto flush 与 PC 的 dirty checking,本章 6.1 节 Bulk JPQL 绕过 PC 的现象正是基于第 3 章 PC 的语义)、第 6 章关联映射(JPQL 路径表达式 c.orders 直接对应 @OneToMany 映射)、第 7 章继承映射(type(p) 函数对应 SINGLE_TABLE 的 discriminator 列)。本章是把"对象在 PC 中的生命周期"和"实体之间的映射关系"组合起来,看它们在查询语言层如何被表达。后续第 9 章 Criteria API 是 JPQL 字符串语法的"类型安全替代品",共享后端 SQM,本章的所有 SQL 行为在 Criteria 里都成立。

JPQL 力量来自"实体语言"这个抽象,陷阱也来自这个抽象 — JPQL 看起来像 SQL,但它的语义层基础是实体模型 + 关联映射 + PersistenceContext三件套。每次你写 c.orders,Hibernate 在背后查 @OneToMany(mappedBy = "customer") 映射、生成 orders.customer_id = customers.id 的 JOIN ON、把结果集 hydrate 成 Order 对象、放进 PC、参与 dirty checking。这一长串"自动行为"在你写对的时候是力量(改字段名 IDE 自动跟、跨厂商分页统一、对象身份保证),但写错的时候是反直觉陷阱(JOIN 看起来加载了关联其实没有、UPDATE 看起来改了对象其实没有、集合 fetch 加分页看起来用 LIMIT 其实在内存切)。把 "JPQL 是面向实体的语言、不是 SQL"这个认知钉死,本章后面的所有陷阱都不需要死记——你能从"实体语义"推出"为什么这种行为"。

8.1 JPQL 查询的是实体

JPQL 第一条原则:FROM 后面跟实体类名,不是表名;属性表达式跟 Java 字段名,不是列名。这是 JPQL 与 SQL 最核心的差异。

实体名到表名的翻译

图 8-1-1 · 实体语言到 SQL 的映射:每个 token 都经过实体元数据翻译

8.1.1 路径表达式翻译

List<String> emails = em.createQuery("""
    select c.email
    from Customer c
    join c.orders o
    where o.totalAmount > :min
    """, String.class)
    .setParameter("min", new BigDecimal("1000.00"))
    .getResultList();
select c1_0.email_address
from customers c1_0
join orders o1_0 on o1_0.customer_id = c1_0.id
where o1_0.total_amount > ?;

JPQL 的 c.orders 在 SQL 中根据 @JoinColumn 翻译为 orders.customer_id = customers.id。重命名 Java 属性 email 为 emailAddress 时,只需更新映射,JPQL 自动跟进。表名重命名亦然——只要 @Table(name = "...") 改了,所有 JPQL 自动指向新表。

8.1.2 字段重命名安全性对比

[场景] 将 Customer.email 重命名为 emailAddress

JPQL 写法:                              SQL 写法:
select c.email from Customer c          select email_address from customers
↓ 重构 IDE 自动更新                       ↓ 仅文本替换,可能漏

select c.emailAddress from Customer c   select email from customers ← 漏改导致运行期错误
↓ 启动期映射检查                          ↓ 启动期通过,运行期 NPE / 错位

JPQL 的字段名是 Java 属性名,IDE rename 会自动更新所有引用——包括 JPQL 字符串内的引用。原生 SQL 字符串用的是数据库列名,IDE 不参与重构,只能靠全局文本搜索,漏改是常态。这就是为什么能用 JPQL 就别用原生 SQL,只有 JPQL 表达不了的场景(CTE、窗口函数、厂商优化器 hint)才退到原生 SQL。

8.2 参数绑定

JPQL 的参数绑定有三种语法,但安全和性能差异巨大:命名参数(:name)、位置参数(?1)、字符串拼接(+)。前两种走 JDBC PreparedStatement + 命中查询计划缓存,第三种是 SQL 注入和缓存淹没的双重陷阱。

命名参数与查询计划缓存

图 8-2-1 · 命名参数命中 QueryInterpretationCache 的执行路径

8.2.1 命名参数(推荐)

TypedQuery<Customer> q = em.createQuery("""
    select c from Customer c
    where c.status = :status and c.createdAt >= :since
    """, Customer.class);

q.setParameter("status", CustomerStatus.ACTIVE);
q.setParameter("since", Instant.parse("2026-01-01T00:00:00Z"));
List<Customer> customers = q.getResultList();
Hibernate:
    select c1_0.id, c1_0.created_at, c1_0.email_address, c1_0.status
    from customers c1_0
    where c1_0.status = ? and c1_0.created_at >= ?
binding parameter [1] as [VARCHAR] - [ACTIVE]
binding parameter [2] as [TIMESTAMP_UTC] - [2026-01-01T00:00:00Z]

命名参数有两大好处:可读性高(知道每个 ? 对应什么业务概念)、不依赖参数顺序(改 SQL 顺序不需要改 setParameter 顺序)。

8.2.2 字符串拼接的两层风险

String jpql = "select c from Customer c where c.name = '" + input + "'";
em.createQuery(jpql, Customer.class).getResultList();

输入 ' or '1'='1 时,有效 JPQL 变为:

select c from Customer c where c.name = '' or '1'='1'
[结果] 全表返回
[risk] SQL 注入 + QueryInterpretationCache 失效

字符串拼接的两个独立危害:

  1. SQL 注入:用户输入直接进 JPQL 文本。' or '1'='1 是经典注入字符串,绕过 WHERE 条件返回全表。
  2. 查询计划缓存失效:Hibernate 的 QueryInterpretationCache 以 normalized JPQL 字符串为 key 缓存解析结果。每次查询用户输入不同,JPQL 字符串就不同,cache 永远 miss,每次都重新跑一遍 HQL Parser。在高并发场景下这就是隐藏的 CPU 浪费,几十毫秒变几百毫秒。

参数绑定方式对比

方式语法安全计划缓存推荐
命名参数:name是命中推荐
位置参数?1是命中兼容旧代码
字符串拼接+注入风险miss禁止
"动态 SQL"的需求要用 Criteria 满足,不是字符串拼接 — 我经常遇到的辩护是"我这个查询条件是动态的,不能用命名参数,只能拼字符串"。这是错误的。"动态条件"的正确解法是 Criteria API(下一章详细讨论),它的整个设计目的就是类型安全地动态拼装查询:你根据用户输入往 Predicate[] 里塞条件,最后 cb.and(predicates) 一次组合。生成的 SQL 是参数化的、安全的、命中查询缓存的。任何"我必须用字符串拼接"的诉求,深挖下去都是"我不会写 Criteria"或"我懒得学 Criteria",而不是真的"无法用参数化"。

8.3 JOIN 与 JOIN FETCH

JPQL 最容易踩坑的语义点:JOIN 和 JOIN FETCH 长得一样,生成的 SQL 也长得一样,但语义完全不同。JOIN 只用于过滤/聚合,不加载关联;JOIN FETCH 加载关联到 PersistenceContext。

过滤 vs 加载

图 8-3-1 · JOIN(只过滤,不加载)vs JOIN FETCH(过滤 + 加载到 PC)

8.3.1 普通 JOIN 仍可能 N+1

List<Customer> customers = em.createQuery("""
    select distinct c from Customer c
    join c.orders o
    where o.totalAmount > :min
    """, Customer.class)
    .setParameter("min", new BigDecimal("1000"))
    .getResultList();

customers.forEach(c -> System.out.println(c.getOrders().size()));
[SELECT 1] select distinct c1_0.id, c1_0.email_address from customers c1_0
           join orders o1_0 on o1_0.customer_id = c1_0.id
           where o1_0.total_amount > ?
[SELECT 2..N+1] select o1_0.customer_id, o1_0.id, o1_0.total_amount
                from orders o1_0 where o1_0.customer_id = ?
[结论] N + 1 SELECT,集合迭代时触发懒加载

普通 JOIN 把关联表纳入 SQL 用于过滤条件,但 c.orders 集合仍是 LAZY 状态——你拿到 customers 后调 c.getOrders().size(),每个 customer 都会触发一条独立 SELECT 加载它的 orders。这就是经典的 N+1 问题。

8.3.2 JOIN FETCH 一次加载

List<Customer> customers = em.createQuery("""
    select distinct c from Customer c
    join fetch c.orders
    where c.status = :status
    """, Customer.class)
    .setParameter("status", CustomerStatus.ACTIVE)
    .getResultList();
select distinct c1_0.id, c1_0.email_address,
       o1_0.customer_id, o1_0.id, o1_0.total_amount
from customers c1_0
join orders o1_0 on o1_0.customer_id = c1_0.id
where c1_0.status = ?;

JOIN FETCH 一次性把客户和订单全部 SELECT 出来,Hibernate 把订单 hydrate 成 Order 对象并放到 customer 的 orders 集合里。后续 c.getOrders() 不会再触发 SQL。

JOIN 类型对比

类型SQL加载关联多集合 fetch
JOININNER JOIN不加载任意
LEFT JOINLEFT OUTER JOIN不加载任意
JOIN FETCHINNER JOIN加载限 1 个集合
LEFT JOIN FETCHLEFT OUTER JOIN加载限 1 个集合

多集合 fetch 引发笛卡尔积。例如同时 fetch c.orders 和 c.addresses,SQL 会返回 customers × orders × addresses 行积。即使 distinct c 也只去重根实体,SQL 结果集仍是笛卡尔行,内存膨胀几个数量级。Hibernate 6 实际不允许这样做(运行期抛异常),只允许同时 fetch 多个 @ManyToOne 单端关联。

JPA 学习曲线最大门槛是分清 JOIN 与 JOIN FETCH — 我教过的每一个 JPA 新人都在这一对概念上踩过至少一次坑。表面原因是名字相似,深层原因是 JPA 抽象层"延迟加载"和"过滤条件"是两件事但用同一个 JOIN 关键字。SQL 没这个问题——SQL 的 JOIN 永远是过滤 + 投影合一。JPA 拆出来一个独立的"是否加载到对象图"维度,再用 FETCH 关键字标记,大多数人需要踩过 1-2 次 N+1 才能内化这个区分。建议团队 review JPQL 时把 JOIN 全部默认怀疑:不带 FETCH 的 JOIN,如果后续有访问被 JOIN 的关联,几乎肯定是 N+1。

8.4 分页

JPQL 分页的优雅在于跨厂商方言统一——你只调 setMaxResults(10).setFirstResult(20),Hibernate 根据数据库方言(MySQL / PostgreSQL / Oracle / SQL Server)生成对应的 LIMIT / OFFSET / FETCH NEXT 语法。但分页与集合 fetch 组合时有重大陷阱。

Dialect 翻译分页

图 8-4-1 · 同一个 JPQL 在四种方言下生成不同 SQL

8.4.1 普通分页

List<Customer> page = em.createQuery("""
    select c from Customer c
    where c.status = :status
    order by c.createdAt desc
    """, Customer.class)
    .setParameter("status", CustomerStatus.ACTIVE)
    .setFirstResult(20)
    .setMaxResults(10)
    .getResultList();
select c1_0.id, c1_0.created_at, c1_0.email_address, c1_0.status
from customers c1_0
where c1_0.status = ?
order by c1_0.created_at desc
limit ? offset ?;

这种分页正常工作——单实体查询,无集合 fetch。LIMIT 在 SQL 层完成,数据库只扫描 10 条目标行。

8.4.2 集合 JOIN FETCH + 分页 = 内存分页(陷阱)

em.createQuery("""
    select c from Customer c
    join fetch c.orders
    order by c.createdAt desc
    """, Customer.class)
    .setFirstResult(0)
    .setMaxResults(20)
    .getResultList();
WARN HHH90003004:
  firstResult/maxResults specified with collection fetch;
  applying in memory.
[执行] 加载所有 customer × order 笛卡尔行,然后内存截取
[风险] 内存爆炸,假设 10 万订单 × 100 客户 = 1000 万行内存

这是 JPA 生产里最危险的"沉默退化"之一——SQL 的 LIMIT 子句被悄悄丢弃,Hibernate 加载全部笛卡尔行到内存,然后在内存里截取 20 条。日志里会有 HHH90003004 警告,但默认日志级别是 INFO,大部分团队看不到这条警告。直到 OOM 或者响应时间从 50ms 飙到 30s,才发现"分页根本没走 LIMIT"。

8.4.3 安全模式:两步分页

List<Long> ids = em.createQuery("""
    select c.id from Customer c
    order by c.createdAt desc, c.id desc
    """, Long.class)
    .setFirstResult(0)
    .setMaxResults(20)
    .getResultList();

if (ids.isEmpty()) return List.of();

List<Customer> customers = em.createQuery("""
    select distinct c from Customer c
    left join fetch c.orders
    where c.id in :ids
    order by c.createdAt desc, c.id desc
    """, Customer.class)
    .setParameter("ids", ids)
    .getResultList();
[SQL 计数] 2 条
  - SELECT 1: 主键页 (LIMIT 20 OFFSET 0)
  - SELECT 2: IN 列表 + JOIN FETCH (无 LIMIT)
[内存] 仅 20 条 customer 及关联订单

两步分页是 JPA 生态的"标准模式":第一步只查根实体的 ID,带 LIMIT 拿到精确的 20 个 ID;第二步用 IN 列表 + JOIN FETCH,不带 LIMIT(因为已经被 IN 列表限定到 20 个根实体了)。Spring Data JPA 的 @EntityGraph 注解 + Pageable 也在内部走这套模式。

8.5 DTO 投影

API 列表页通常只需要少数几个字段,完整加载实体 + 进 PC + 参与 dirty checking 是浪费。JPQL 的 select new ... 构造器投影直接把 SELECT 列拼成 DTO,跳过 PC。

实体查询 vs DTO 查询

图 8-5-1 · 实体加载 vs DTO 投影:是否进入 PersistenceContext

8.5.1 SELECT NEW 构造器投影

public record CustomerOrderSummary(Long id, String email, Long orderCount) {}

List<CustomerOrderSummary> summaries = em.createQuery("""
    select new com.example.CustomerOrderSummary(c.id, c.email, count(o))
    from Customer c
    left join c.orders o
    group by c.id, c.email
    order by count(o) desc
    """, CustomerOrderSummary.class)
    .getResultList();
select c1_0.id, c1_0.email_address, count(o1_0.id)
from customers c1_0
left join orders o1_0 on o1_0.customer_id = c1_0.id
group by c1_0.id, c1_0.email_address
order by count(o1_0.id) desc;

注意 select new com.example.CustomerOrderSummary(...) 必须用全限定类名。Java record 和普通 class 都可以。这条 SQL 的结果集是三列原始值,Hibernate 用反射调用 DTO 构造器把它们包装成对象。

8.5.2 构造器参数类型陷阱

COUNT 返回 Long,AVG 返回 Double,SUM 随字段类型变化(SUM(BigDecimal) 返回 BigDecimal,SUM(Integer) 返回 Long)。构造器参数类型不匹配在运行期才暴露。

public record OrderStat(Long id, Integer count) {}

em.createQuery("""
    select new com.example.OrderStat(c.id, count(o))
    from Customer c left join c.orders o
    group by c.id
    """, OrderStat.class)
    .getResultList();
[运行时异常]
  IllegalArgumentException: argument type mismatch
  at constructor OrderStat(Long, Integer)
  expected Integer, got Long
[修复] 改为 Long count 字段或使用 cast() 表达式

这是 JPQL 反射构造的固有限制——SQL 解析期不知道 count(o) 应当转成 Integer 还是 Long。修复方式有三种:(1) DTO 字段改 Long;(2) JPQL 用 cast(count(o) as integer) 显式转型;(3) DTO 提供两个构造器,Hibernate 选第一个签名匹配的。

8.6 Bulk Update / Delete

"我要把 1287 个 inactive 用户的 status 设为 LOCKED" — 这种大批量修改用 em.find 一个个改是疯狂的(1287 次 SELECT + 1287 次 UPDATE)。JPQL 提供 UPDATE 和 DELETE 语句一次性更新所有匹配行,但这种"逃生通道"绕过 PersistenceContext,有独立陷阱。

Bulk 绕过 PersistenceContext

图 8-6-1 · Bulk JPQL 直接 JDBC,不更新 PC,Java 对象与 DB 不一致

8.6.1 Bulk 后读到旧值

em.getTransaction().begin();

Customer customer = em.find(Customer.class, 1L);

int updated = em.createQuery("""
    update Customer c
    set c.status = :inactive
    where c.id = :id
    """)
    .setParameter("inactive", CustomerStatus.INACTIVE)
    .setParameter("id", 1L)
    .executeUpdate();

System.out.println(updated);
System.out.println(customer.getStatus());

em.getTransaction().commit();
Hibernate: update customers set status = ? where id = ?
[输出 1] 1            ← bulk 影响行数
[输出 2] ACTIVE       ← 内存对象仍是旧值
[根因] bulk 直接 JDBC 执行,不更新 PersistenceContext

Bulk JPQL 的实现是直接走 JDBC PreparedStatement,绕过 PC 的所有跟踪机制。数据库行已经改了,但 PC 里那个 customer 对象的 status 字段没人去同步——Hibernate 不知道哪些 PC 实体被你的 UPDATE 影响。

8.6.2 正确模式

em.flush();

int updated = em.createQuery("""
    update Customer c set c.status = :inactive
    where c.lastLoginAt < :cutoff
    """)
    .setParameter("inactive", CustomerStatus.INACTIVE)
    .setParameter("cutoff", cutoff)
    .executeUpdate();

em.clear();
[bulk updated] 1287 行
[em.clear] 清空 PersistenceContext
[后续] em.find / 查询从数据库重新读取

Bulk 操作前后两步:前 flush 防止 PC 中已有 dirty 实体的修改丢失,后 clear 强制后续查询重新加载。如果你只想清掉某一个实体而不是整个 PC,用 em.refresh(entity) 单实体重载。

Bulk vs 单实体修改

操作路径PC 同步触发回调适用

Bulk 不触发 @PreUpdate / @PostUpdate / @AuditTrail 等生命周期回调 — 这是另一个隐藏陷阱。如果你的实体有 @PreUpdate 自动设置 updatedAt 字段、或者用了 Hibernate Envers 做审计、或者依赖 @PostUpdate 发送领域事件——Bulk JPQL 全部跳过这些回调。结果:数据库行被改了,但审计表没记录、updatedAt 没更新、领域事件没发出。这种"沉默的不一致"在合规系统里是大事故。修复方法:Bulk SQL 里手动 set updated_at = current_timestamp 显式更新审计字段;关键路径不要用 Bulk,用单条 setter 让回调正常触发。

8.7.1 子查询限制

List<Customer> customers = em.createQuery("""
    select c from Customer c
    where exists (
        select 1 from Order o
        where o.customer = c and o.totalAmount > :min
    )
    """, Customer.class)
    .setParameter("min", new BigDecimal("1000"))
    .getResultList();

JPQL 允许 WHERE 与 HAVING 子查询,不允许 FROM 子查询。需要 FROM 子查询(派生表)时改用 Native SQL 或 jOOQ。Hibernate 6 部分支持 CTE,但跨数据库兼容性有限。

8.8.1 设计取舍

取舍 1:JPQL 绑定实体模型

绑定实体使得 Java 重构(字段 / 类重命名)自动传播到查询。代价是 JPQL 不能直接表达 CTE、窗口函数、厂商优化器 hint。复杂场景必须切换 Native SQL。

取舍 2:SQM 中间层

JPQL 与 Criteria API 共享 SQM,后续管线(SQL AST、Dialect、JDBC)统一。代价是源码层级深,排查问题需理解 SQM 结构。但好处是 Hibernate 6 加新功能(比如新增窗口函数支持)只需在 SQM/SQL AST 层加,JPQL 与 Criteria 同时受益。

取舍 3:Query plan cache

热查询缓存解析结果,显著降低 CPU 开销。代价是字符串拼接造成的查询变体可能淹没缓存。QueryInterpretationCache 默认上限 2048 条——如果你的应用动态拼接 JPQL,很快就被各种变体填满,后续查询全都 miss。这就是 8.2 节提到的"缓存淹没"。

取舍 4:auto flush before query

JPQL 查询前根据 FlushMode.AUTO 自动 flush 受影响的 query space,确保查询读到本事务内已修改状态。代价是只读查询可能触发写 SQL,反向导致难以排查的写库时机问题。生产中用 FlushMode.COMMIT 关闭这一行为是常见优化(见第 3 章)。

8.10 回归护栏

JPQL 的特点是大部分错误"运行起来才暴露"——启动期 Hibernate 不可能预知 N+1 或集合 fetch + 分页。这意味着必须用集成测试守护关键 JPQL 行为。

8.10.1 fetch + 分页守护

@Test
void fetchJoinPaginationShouldNotFallBackToMemory() {
    LogCaptor logs = LogCaptor.forLogger("org.hibernate.orm.query");

    repository.findCustomerPageWithOrders(PageRequest.of(0, 20));

    assertThat(logs.getWarnLogs())
        .noneMatch(line -> line.contains("HHH90003004"));
}

这条测试钉死"我们的分页查询不会触发内存分页"。任何人加了 join fetch c.orders 又忘了改两步分页,这条测试立刻失败。

8.10.2 SQL 数量守护

@Test
void listEndpointShouldNotRegressToNPlusOne() {
    SqlRecorder recorder = sqlRecorder.clear();

    customerService.listCustomerOrderSummaries();

    assertThat(recorder.selectCount()).isLessThanOrEqualTo(2L);
    assertThat(recorder.sqlText()).contains("left join");
}

8.10.3 Bulk 后 PC 同步守护

@Test
void bulkUpdateShouldRequireClearForFreshRead() {
    em.persist(new Customer("alice@example.com", CustomerStatus.ACTIVE));
    em.flush();

    em.createQuery("update Customer c set c.status = :s where c.id = :id")
        .setParameter("s", CustomerStatus.INACTIVE)
        .setParameter("id", 1L)
        .executeUpdate();

    Customer stale = em.find(Customer.class, 1L);
    em.refresh(stale);

    assertThat(stale.getStatus()).isEqualTo(CustomerStatus.INACTIVE);
}
