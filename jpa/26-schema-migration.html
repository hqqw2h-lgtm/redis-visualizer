第 26 章 · Schema 生成与数据库迁移

ddl-auto · Flyway · Liquibase · drift guard · expand-contract

实体模型描述应用期望看到的结构,但它不是数据库变更的发布计划。开发机可以由 Hibernate 临时生成表,生产环境必须把每次 DDL 变成可审查、可回放、可阻断漂移的迁移脚本。

核心问题:① 生产环境为什么禁止 ddl-auto=update;② 如何让"实体改动 → 迁移脚本 → 数据库变更 → 应用启动校验"形成闭环。

  • ddl-auto 五种策略(none/validate/update/create/create-drop)的行为对比 + 生产禁忌
  • Hibernate Schema SPI 调用链:SchemaManagementToolCoordinator → SchemaValidator
  • Flyway 版本化 SQL(V1__init.sql + flyway_schema_history)
  • Liquibase ChangeSet(YAML + databasechangelog)的对比与选型
  • 大表迁移的锁表评估:online DDL / pt-online-schema-change / 分批 backfill
  • CI drift guard:Testcontainers + JPA DDL snapshot diff,合并前阻断漂移
  • 破坏性变更的 expand-contract 三阶段:加新字段 → 双写迁移 → 删旧字段
  • PostgreSQL / MySQL / Oracle 方言差异(BIGSERIAL vs AUTO_INCREMENT 等)

前置知识:JPA 实体注解;SQL DDL;基本的 CI/CD 流程。

26.1 基线实体与 DDL

@Entity
@Table(
    name = "customers",
    uniqueConstraints = @UniqueConstraint(
        name = "uk_customers_email", columnNames = "email"),
    indexes = {
        @Index(name = "idx_customers_display_name", columnList = "display_name"),
        @Index(name = "idx_customers_region", columnList = "region")
    })
public class Customer {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false, length = 160)
    private String email;

    @Column(name = "display_name", nullable = false, length = 80)
    private String displayName;

    @Column(nullable = false, length = 20)
    private String tier;

    @Column(name = "created_at", nullable = false, updatable = false)
    private Instant createdAt;
}
CREATE TABLE customers (
    id           BIGSERIAL    NOT NULL,
    email        VARCHAR(160) NOT NULL,
    display_name VARCHAR(80)  NOT NULL,
    tier         VARCHAR(20)  NOT NULL DEFAULT 'STANDARD',
    region       VARCHAR(40)  NOT NULL,
    created_at   TIMESTAMPTZ  NOT NULL DEFAULT now(),
    CONSTRAINT pk_customers PRIMARY KEY (id),
    CONSTRAINT uk_customers_email UNIQUE (email),
    CONSTRAINT ck_customers_tier CHECK (tier IN ('STANDARD','VIP','ENTERPRISE'))
);
CREATE INDEX idx_customers_display_name ON customers (display_name);

26.2 ddl-auto 五种策略

策略启动行为是否触碰数据库适用环境生产结论
none不生成、不验证否迁移工具完全接管可用,但少一层启动校验
validate比对实体与 schema仅 SELECT metadataProd / CI推荐
update自动补缺失列 / 索引ALTER个人开发机生产禁止
create删除后重建DROP / CREATE单元测试 / Demo生产禁止
create-drop启动建表,关闭删表DROP / CREATE集成测试生产禁止
 
# application-prod.properties — 推荐生产配置
spring.jpa.hibernate.ddl-auto=validate
spring.jpa.properties.jakarta.persistence.schema-generation.database.action=none
spring.flyway.enabled=true
spring.flyway.locations=classpath:db/migration
spring.flyway.baseline-on-migrate=false
spring.flyway.clean-disabled=true
spring.flyway.out-of-order=false

update 的隐患:字段从 customerName 改为 displayName 后,update 产生 ALTER TABLE customers ADD COLUMN display_name VARCHAR(255)。新列添加成功,但旧列 customer_name 数据未迁移。应用读到的 display_name 全为 NULL,业务无声地损坏。

26.4 JPA Schema Generation — 导出 DDL 快照

jakarta.persistence.schema-generation.scripts.action=create
jakarta.persistence.schema-generation.scripts.create-target=build/schema-create.sql
jakarta.persistence.schema-generation.scripts.drop-target=build/schema-drop.sql
jakarta.persistence.schema-generation.create-source=metadata
jakarta.persistence.schema-generation.drop-source=metadata
jakarta.persistence.schema-generation.database.action=none

把 build/schema-create.sql 提到 CR 里 review,确认 Hibernate 渲染出的 DDL 符合预期。注意:这份 SQL 不能直接作为生产脚本,因为它没有 ALTER、回滚、数据迁移逻辑。

-- V1__create_customers_and_orders.sql
CREATE TABLE customers (
    id BIGSERIAL NOT NULL,
    email VARCHAR(160) NOT NULL,
    display_name VARCHAR(80) NOT NULL,
    tier VARCHAR(20) NOT NULL DEFAULT 'STANDARD',
    region VARCHAR(40) NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT pk_customers PRIMARY KEY (id),
    CONSTRAINT uk_customers_email UNIQUE (email)
);

CREATE INDEX idx_customers_region ON customers (region);

-- V2__add_customer_region_column.sql(已存在则 SKIP)
ALTER TABLE customers ADD COLUMN IF NOT EXISTS region VARCHAR(40) NOT NULL DEFAULT 'CN';
CREATE INDEX IF NOT EXISTS idx_customers_region ON customers (region);

-- V4__rename_customer_name_to_display_name.sql
ALTER TABLE customers RENAME COLUMN customer_name TO display_name;
-- 数据已通过应用层双写迁移完成
-- Flyway 自管表
SELECT installed_rank, version, description, success, execution_time
FROM flyway_schema_history ORDER BY installed_rank;

26.7 大表迁移性能

数据量裸 ALTER TABLE ADD COLUMN NOT NULLonline DDL (MySQL)pt-online-schema-change
10 万行1s 锁表~ 几秒~ 几秒
1 亿行~ 几分钟,可能锁表~ 分钟级,不阻塞 R/W~ 分钟级,可控锁
10 亿行不可用分钟到小时小时级 + 监控

大表加 NOT NULL 列:① 先加 nullable 列;② 批量 backfill;③ 加 NOT NULL 约束。三阶段提交,避免一次性锁表。

26.8 CI drift guard

@Testcontainers
class SchemaDriftGuardTest {

    @Container
    static PostgreSQLContainer<?> pg = new PostgreSQLContainer<>("postgres:16");

    @Test
    void flywayShouldProduceSchemaMatchingEntities() throws Exception {
        // 1. 启动 Postgres,跑 Flyway 迁移
        Flyway.configure()
            .dataSource(pg.getJdbcUrl(), pg.getUsername(), pg.getPassword())
            .locations("classpath:db/migration")
            .load()
            .migrate();

        // 2. 启动 Spring Boot 用 ddl-auto=validate
        Map<String, Object> props = Map.of(
            "spring.datasource.url", pg.getJdbcUrl(),
            "spring.jpa.hibernate.ddl-auto", "validate");

        // 3. 校验启动成功 → Entity 与 Flyway schema 一致
        new SpringApplicationBuilder(App.class).properties(props).run();
    }
}

26.11 陷阱与限制

#陷阱现象修复
1生产用 ddl-auto=update无声破坏改 validate + 迁移工具
2Flyway 脚本未提到 review脱离审计合并前必须 review
3Flyway baseline-on-migrate=true历史脚本跳过显式 baseline
4迁移脚本不可重放回放失败幂等设计
5大表 ALTER 锁表线上不可用online DDL / pt-osc
6Entity 改后没加迁移启动失败CI drift guard
7枚举字段加新值历史数据违反 CHECK先 DROP CHECK 再加
8外键索引缺失JOIN 慢显式 @Index
9Flyway clean-disabled=false误清生产始终 true
10多方言混用方言差异坑统一目标 DB
