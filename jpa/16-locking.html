第 16 章 · 乐观锁与悲观锁

@Version 自动版本管理 · SELECT FOR UPDATE / SHARE · SKIP LOCKED 队列 · 死锁预防 · LockOptions 源码

并发控制是数据一致性的最后一道防线。JPA 提供两类:乐观锁通过 @Version 在 commit 时检测;悲观锁通过 SELECT ... FOR UPDATE 在读取时阻塞。

选择依据是冲突概率与重试成本。读多写少用乐观锁;热点账户、资金转账用悲观锁;队列任务领取用 SKIP LOCKED。

  • @Version 的实现原理与三种 LockModeType(OPTIMISTIC / OPTIMISTIC_FORCE_INCREMENT / READ)
  • PESSIMISTIC_READ / PESSIMISTIC_WRITE / PESSIMISTIC_FORCE_INCREMENT 三种悲观锁 + 共享锁 vs 排他锁
  • lock.timeout = 0 / -2 / 正数 在 PostgreSQL/MySQL/Oracle 的方言差异
  • SKIP LOCKED 实现的高并发队列任务领取(无阻塞,无竞争)
  • 死锁产生的循环等待条件 + 加锁顺序一致的预防做法
  • OptimisticLockException / LockTimeoutException 的 Spring 重试模板
  • Hibernate 源码:LockOptions / ForUpdateClause / Dialect#getForUpdateString / UpdateCoordinatorStandard
  • 抢购场景的 UPDATE WHERE stock >= ? 单 SQL 方案,与乐观锁 + 重试方案的性能对比
  • PostgreSQL advisory lock 作为应用级分布式锁
  • 版本字段类型选型:int / long / Timestamp / Instant 的取舍

前置知识:第 15 章事务管理;数据库锁机制(行锁、共享锁、排他锁);死锁检测原理;JPQL 与原生 SQL。

16.3 悲观锁

图 16-3-1 · 悲观锁排队 — Tx2 等待 Tx1 commit 才能继续
LockModeType行为SQL
PESSIMISTIC_READ共享锁SELECT FOR SHARE / FOR UPDATE
PESSIMISTIC_WRITE排他锁SELECT FOR UPDATE
PESSIMISTIC_FORCE_INCREMENT排他锁 + 版本递增SELECT FOR UPDATE + UPDATE version

16.3.1 资金转账(按 ID 升序加锁)

@Transactional
public void transfer(Long fromId, Long toId, BigDecimal amount) {
    Long firstId = Math.min(fromId, toId);
    Long secondId = Math.max(fromId, toId);

    Account first = em.find(Account.class, firstId,
        LockModeType.PESSIMISTIC_WRITE);
    Account second = em.find(Account.class, secondId,
        LockModeType.PESSIMISTIC_WRITE);

    Account from = (firstId.equals(fromId)) ? first : second;
    Account to   = (firstId.equals(fromId)) ? second : first;

    from.debit(amount);
    to.credit(amount);
}
SELECT * FROM account WHERE id = 1 FOR UPDATE;
SELECT * FROM account WHERE id = 2 FOR UPDATE;

UPDATE account SET balance = balance - 100 WHERE id = 1;
UPDATE account SET balance = balance + 100 WHERE id = 2;

COMMIT;

按 ID 升序加锁是预防死锁的标准做法。如果两个事务以相反顺序加锁,就会形成循环等待。

16.7 实战场景

16.7.1 库存扣减(乐观锁 + 重试)

@Service
public class InventoryService {

    @Retryable(
        value = OptimisticLockException.class,
        maxAttempts = 5,
        backoff = @Backoff(delay = 50)
    )
    @Transactional
    public void deduct(Long productId, int quantity) {
        Product p = repo.findById(productId).orElseThrow();
        if (p.getStock() < quantity) {
            throw new InsufficientStockException();
        }
        p.setStock(p.getStock() - quantity);
    }
}

16.7.2 抢购(数据库自增 + UPDATE)

@Modifying
@Query("UPDATE Product p SET p.stock = p.stock - :qty " +
       "WHERE p.id = :id AND p.stock >= :qty")
int deductIfEnough(@Param("id") Long id, @Param("qty") int qty);

public boolean trySeckill(Long productId) {
    int updated = repo.deductIfEnough(productId, 1);
    return updated == 1;
}
方案特点性能
乐观锁 + 重试简单,高重试中
UPDATE WHERE stock >= qty一次 SQL 完成高
Redis 预扣 + 异步落库复杂,极高极高
队列串行严格,慢低

16.7.3 队列任务领取(SKIP LOCKED)

@Transactional
public List<Job> claimJobs(int batchSize) {
    return em.createNativeQuery("""
        SELECT * FROM job_queue
        WHERE status = 'PENDING'
        ORDER BY priority, created_at
        LIMIT ?
        FOR UPDATE SKIP LOCKED
        """, Job.class)
        .setParameter(1, batchSize)
        .getResultList();
}

16.8 陷阱与限制

#陷阱现象修复
1无 @Version 字段无版本检查添加 @Version
2version 类型不当溢出 / 精度问题用 long
3死锁频繁 rollback加锁顺序一致
4NOWAIT 立即失败业务感知差配 timeout

16.9 回归护栏

@Test
void optimisticLockShouldDetectConcurrentUpdate() {
    Long id = createProduct(100);

    Product p1 = repo.findById(id).orElseThrow();
    Product p2 = repo.findById(id).orElseThrow();

    p1.setStock(50);
    repo.saveAndFlush(p1);

    p2.setStock(80);

    assertThatThrownBy(() -> repo.saveAndFlush(p2))
        .isInstanceOf(ObjectOptimisticLockingFailureException.class);
}

@Test
void pessimisticLockShouldBlockOtherTx() throws Exception {
    Long id = createAccount(BigDecimal.valueOf(1000));

    CountDownLatch lockAcquired = new CountDownLatch(1);
    CountDownLatch lockReleased = new CountDownLatch(1);

    Future<?> tx1 = executor.submit(() -> txTemplate.execute(s -> {
        em.find(Account.class, id, LockModeType.PESSIMISTIC_WRITE);
        lockAcquired.countDown();
        try { lockReleased.await(); } catch (InterruptedException e) {}
        return null;
    }));

    lockAcquired.await();

    Map<String, Object> hints = Map.of("jakarta.persistence.lock.timeout", 0);

    assertThatThrownBy(() -> txTemplate.execute(s -> {
        em.find(Account.class, id, LockModeType.PESSIMISTIC_WRITE, hints);
        return null;
    })).isInstanceOf(LockTimeoutException.class);

    lockReleased.countDown();
    tx1.get();
}

16.12.2 PostgreSQL advisory lock

public boolean tryAcquireGlobalLock(long lockKey) {
    Number result = (Number) em.createNativeQuery(
        "SELECT pg_try_advisory_lock(?1)").setParameter(1, lockKey)
        .getSingleResult();
    return result.intValue() > 0;
}

public void releaseGlobalLock(long lockKey) {
    em.createNativeQuery("SELECT pg_advisory_unlock(?1)")
        .setParameter(1, lockKey)
        .getSingleResult();
}
类型适用优点缺点
int中等并发简单21 亿溢出
long高并发不溢出多 4 字节
Timestamp历史数据可读时钟漂移
Instant现代 Java可读需精度
