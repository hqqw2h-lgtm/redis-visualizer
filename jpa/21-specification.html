第 21 章 · 动态查询与 Specification

null-safe 工厂 · AND/OR/NOT 组合 · Metamodel · Predicate 树 · 回归保护

业务搜索的过滤条件不会按固定顺序出现:品牌可能为空,价格区间可能只给一端,关键词可能与分类一起过滤。Specification 的价值不在于让 Criteria API 更短,而在于把请求过滤条件稳定翻译成可组合、可测试、可回归保护的 Predicate 树。

本章按概念、Spring Data 集成、Predicate 工厂、组合模式、回归测试、源码六维展开。

  • Specification 翻译链:Request filter → 工厂 → Predicate 树 → CriteriaQuery → SQM → SQL
  • JpaSpecificationExecutor 7 个核心方法(findAll / findOne / count / exists / delete)
  • null-safe 工厂方法的 5 条约定(null 返 null / Trim / IgnoreCase / between 三分支 / join 复用)
  • Specification 组合操作:where / and / or / not 嵌套形成 Predicate 树
  • jpamodelgen 编译期 Metamodel(Customer_.email)消除字符串字段名
  • Specification 单元测试 + 集成测试 + SQL 快照三层守护
  • OR 条件 + count 查询的小坑(distinct 自动加 vs 不加)
  • 动态查询的性能取舍:Specification vs @Query vs Querydsl vs Native

前置知识:第 9 章 Criteria API;第 18 章 SQL 生成;第 20 章 Spring Data Repository;函数式接口(三参 lambda)。

@Service
public class ProductSearchService {

    @Autowired
    private ProductRepository repo;

    public Page<Product> search(ProductSearchForm form, Pageable pageable) {
        Specification<Product> spec = Specification
            .where(ProductSpecs.hasBrand(form.getBrand()))
            .and(ProductSpecs.priceBetween(form.getMinPrice(), form.getMaxPrice()))
            .and(ProductSpecs.keywordLike(form.getKeyword()));

        return repo.findAll(spec, pageable);
    }
}

21.2 Specification 工厂

图 21-2-1 · 工厂模式 — null 入参返 null,Spring Data 自动忽略
public final class ProductSpecs {

    private ProductSpecs() {}

    public static Specification<Product> hasBrand(String brand) {
        return (root, query, cb) -> {
            if (brand == null || brand.isBlank()) return null;
            return cb.equal(root.get(Product_.brand), brand);
        };
    }

    public static Specification<Product> priceBetween(BigDecimal min, BigDecimal max) {
        return (root, query, cb) -> {
            if (Objects.isNull(min) && Objects.isNull(max)) return null;
            if (Objects.nonNull(min) && Objects.nonNull(max))
                return cb.between(root.get(Product_.price), min, max);
            if (Objects.nonNull(min))
                return cb.ge(root.get(Product_.price), min);
            return cb.le(root.get(Product_.price), max);
        };
    }

    public static Specification<Product> keywordLike(String keyword) {
        return (root, query, cb) -> {
            if (keyword == null || keyword.isBlank()) return null;
            String like = "%" + keyword.toLowerCase() + "%";
            return cb.or(
                cb.like(cb.lower(root.get(Product_.name)), like),
                cb.like(cb.lower(root.get(Product_.description)), like));
        };
    }

    public static Specification<Product> inCategory(Long categoryId) {
        return (root, query, cb) -> {
            if (categoryId == null) return null;
            Join<Product, Category> cat = root.join(Product_.category);
            return cb.equal(cat.get(Category_.id), categoryId);
        };
    }
}
约定实现
入参为 null返回 null(Spring Data 自动忽略)
入参列表为空返回 null
字符串忽略大小写cb.lower(...) + .toLowerCase()
范围查询between / ge / le 三分支
关联条件root.join(Product_.category)

21.4 复杂关联

public static Specification<Order> customerEmailLike(String pattern) {
    return (root, query, cb) -> {
        if (pattern == null || pattern.isBlank()) return null;
        Join<Order, Customer> c = root.join(Order_.customer, JoinType.INNER);
        return cb.like(cb.lower(c.get(Customer_.email)),
                       "%" + pattern.toLowerCase() + "%");
    };
}

public static Specification<Order> hasItemQuantityAbove(int threshold) {
    return (root, query, cb) -> {
        Subquery<Long> sub = query.subquery(Long.class);
        Root<OrderItem> item = sub.from(OrderItem.class);
        sub.select(item.get(OrderItem_.order).get(Order_.id))
           .where(cb.gt(item.get(OrderItem_.quantity), threshold));
        return cb.in(root.get(Order_.id)).value(sub);
    };
}
场景API
关联条件root.join(Entity_.assoc)
子查询query.subquery(Long.class)
聚合query.groupBy + having
distinctquery.distinct(true)
fetch joinroot.fetch(Entity_.assoc) (仅 select 阶段)

子查询 vs Join 的取舍。子查询(IN)对优化器更友好,适合一对多 EXISTS 语义;Join 适合需要取关联字段。Specification 中应优先用子查询避免行膨胀。

21.6 回归测试

@Test
void hasBrandShouldGenerateEqualPredicate() {
    SqlCapture capture = sqlCapture.clear();

    Specification<Product> spec = ProductSpecs.hasBrand("Sony");
    repo.findAll(spec);

    assertThat(capture.normalizedSql())
        .contains("where p1_0.brand = ?");
    assertThat(capture.binds()).hasSize(1)
        .extracting(Bind::value).containsExactly("Sony");
}

@Test
void allNullsShouldGenerateNoWhereClause() {
    SqlCapture capture = sqlCapture.clear();

    Specification<Product> spec = Specification
        .where(ProductSpecs.hasBrand(null))
        .and(ProductSpecs.priceBetween(null, null))
        .and(ProductSpecs.keywordLike(""));

    repo.findAll(spec);

    assertThat(capture.normalizedSql()).doesNotContain("where");
}

@Test
void searchShouldExecuteTwoSqlForPage() {
    long before = stats.getPrepareStatementCount();

    Page<Product> page = service.search(
        new ProductSearchForm("Sony", new BigDecimal("100"), null, "wireless"),
        PageRequest.of(0, 20));

    long after = stats.getPrepareStatementCount();
    assertThat(after - before).isEqualTo(2L); // 1 主查 + 1 count
}
