第 9 章 · EntityGraph 与 Fetch Graph

@NamedEntityGraph · @NamedAttributeNode · @NamedSubgraph · EntityManager.createEntityGraph · jakarta.persistence.fetchgraph / loadgraph · LoadQueryInfluencers · GraphSemantic · @EntityGraph for Spring Data · 多 bag 与笛卡尔积 · 加载形状测试守护

前八章把 JPA 的"读"切成了三条主线:映射注解 决定关联默认是 LAZY 还是 EAGER,JPQL / Criteria 决定 WHERE 和投影,而加载形状(本次到底要把哪些关联一并捞回来)长期没有第三条独立的声明位置 —— 它要么藏在映射注解里写死成 EAGER,要么散在每条 JPQL 字符串里的 join fetch。这两种做法都各有顽疾:写死 EAGER 等于强制所有接口都 EAGER(列表页根本不需要的关联也被一并拉回),散在 JPQL 里则等于每个查询都要重写一遍加载路径,Repository 派生方法(findByStatus)更是连一行 JPQL 都没地方塞 fetch。JPA 2.1 引入 EntityGraph(JSR 338, 2013) 就是为了切出这第三条独立轴线 —— "加载形状作为一等公民",和查询语句、映射注解解耦,可命名、可复用、可在不同查询中互换。

本章把 EntityGraph 的规范语义、API 形态、运行时挂接、Spring Data 集成、与 JOIN FETCH 的边界一次讲透。读完本章,你应当能回答:为什么 fetchgraph 和 loadgraph 行为差这么远?为什么同一个 @NamedEntityGraph 在 em.find 和 JPQL 里走不同的 SQL?为什么两个 List 集合一起 attach 到 graph 上会抛 MultipleBagFetchException?Spring Data 的 @EntityGraph 注解和 JPA 标准 EntityGraph 之间是什么关系?并且能在团队里规定一条清晰的 "列表查询用 EntityGraph、详情查询用 JOIN FETCH、只读 API 用 DTO" 三分法,把 N+1 的"复发率"压到接近零 —— 这是本章的目标,而下一章(10 · Native SQL)在此基础上给出"绕开 JPQL 的最后一公里"。

  • JPA 加载形状的三种声明方式:映射默认 / JPQL JOIN FETCH / EntityGraph,以及为什么第三种是规范层的"正确解"
  • @NamedEntityGraph / @NamedAttributeNode / @NamedSubgraph 三个注解的语义、嵌套规则与命名约定
  • EntityManager.createEntityGraph() / addAttributeNodes() / addSubgraph() 动态编程式 API 的运行时模型
  • FETCH graph 与 LOAD graph 的语义差 —— 图外属性的不同处理是 EntityGraph 最大的坑
  • EntityGraph 怎样通过 jakarta.persistence.fetchgraph / loadgraph hint 注入到 em.find / JPQL / Criteria / Spring Data
  • Spring Data 的 @EntityGraph 注解的 4 种用法:静态 named graph、attributePaths 即写即用、动态名字注入、与 @Query 共存
  • 多层 Subgraph 树形加载:Order → items → product → category 怎样表达 4 层关联,以及行积爆炸的临界点
  • EntityGraph 与 JOIN FETCH 的三大差异:JOIN 类型(LEFT vs INNER 默认)、WHERE 过滤能力、可复用性
  • MultipleBagFetchException:为什么 2 个 List 集合不能同时 fetch,以及 Set / 拆查 / Hibernate @Fetch(SUBSELECT) 三种修复路径
  • Hibernate 源码层:LoadQueryInfluencers / RootGraphImpl / AttributeNodeImpl / GraphSemantic 怎样把 EntityGraph 解析成 SQL AST 上的 fetch 节点
  • 加载形状的测试守护范式:Hibernate.isInitialized() + Statistics + SQL 形状捕获三层断言

前置知识:第 2 章实体生命周期(理解 PersistenceContext 与 Proxy);第 6 章关联映射(@ManyToOne / @OneToMany 与 FetchType);第 8 章 JPQL(理解 join fetch 与 hint 机制)。本章是把第 6 章的"加载默认值"和第 8 章的"查询加载形状"统一到 JPA 2.1 规范的 EntityGraph 这一抽象上。后续第 11 章会把 EntityGraph 放进抓取策略全景,第 12 章把它列入 N+1 修复方案矩阵 —— 本章的"图作为一等公民"是后续两章的概念基石。

9.0 章首速览

图 9-0-2 · 加载形状的三条轴线 — 边界与适用
EntityGraph 是 JPA 2.1 引入的"加载形状的第三条声明轴线" —— 它不是 JOIN FETCH 的语法糖,而是把"加载契约"提升为可独立命名、可独立维护的对象。判断什么时候该用 EntityGraph 的口诀只有一句:"加载形状要在多个查询入口之间复用,或者你压根没 JPQL 字符串可写"。Spring Data 派生方法(findByStatus)、em.find(id)、Criteria API 都没有可以塞 join fetch 的位置 —— 这些场景下 EntityGraph 是唯一的标准答案。

9.1 业务模型

本章贯穿一个比 12 章更丰富的电商模型 —— 不再只是 Customer 1-N Order,而是 4 层关联:Customer → Order → OrderItem → Product → Category。这种深度才能展示 EntityGraph 的 Subgraph 嵌套能力 —— 一层关联用 JOIN FETCH 也能写,两层勉强,三层及以上几乎只有 EntityGraph 写得清楚。

@Entity
@Table(name = "customers")
public class Customer {
    @Id
    private Long id;

    private String name;

    @Enumerated(EnumType.STRING)
    private CustomerStatus status;

    @OneToMany(mappedBy = "customer", fetch = FetchType.LAZY)
    private List<Order> orders = new ArrayList<>();
}

@Entity
@Table(name = "orders")
public class Order {
    @Id
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "customer_id")
    private Customer customer;

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY, cascade = CascadeType.ALL)
    private List<OrderItem> items = new ArrayList<>();

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<AuditEvent> auditEvents = new ArrayList<>();

    private BigDecimal totalAmount;
    private OrderStatus status;
}

@Entity
@Table(name = "order_items")
public class OrderItem {
    @Id
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    private Order order;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "product_id")
    private Product product;

    private Integer quantity;
    private BigDecimal unitPrice;
}

@Entity
@Table(name = "products")
public class Product {
    @Id
    private Long id;

    private String name;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "category_id")
    private Category category;
}

@Entity
@Table(name = "categories")
public class Category {
    @Id
    private Long id;

    private String name;
}

@Entity
@Table(name = "audit_events")
public class AuditEvent {
    @Id
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    private Order order;

    private String eventType;
    private Instant occurredAt;
}

关键约定:所有关联都显式声明 fetch = FetchType.LAZY —— 这是 EntityGraph 真正发挥威力的起点。一旦映射里有任何关联保持默认 EAGER(@ManyToOne / @OneToOne),后面 9.5 节讨论的 loadgraph 漏网就会出现。这套实体设计代表"生产标准"。

本章贯穿的 4 个典型查询场景

#场景需要加载典型加载形状
① 列表客户列表页(列出 100 个客户,显示订单数)Customer + orders2 层 graph
② 详情订单详情(订单 + 客户 + 订单行 + 商品)Order + customer + items + items.product3 层 graph (含 subgraph)
③ 深详情订单完整快照(含商品分类)+ items.product.category4 层 graph
④ 不要审计日志页(只要订单 + 事件)Order + auditEvents2 层,但和详情场景互斥

4 个场景在 Repository 里要写 4 个查询。关键观察:场景 ② 和 ③ 只差一个 items.product.category,场景 ④ 和 ② 加载完全不同的子集 —— 如果用 EAGER 默认全加载,任何场景都浪费;如果用 JPQL 各写一遍,4 个独立 fetch 字符串维护成本高。EntityGraph 的甜蜜点正在于此:声明 4 个图,4 个查询挂 4 个图。

9.2 三种加载声明

"加载形状"在 JPA 历史上有三次进化。理解这三次进化的不可逆动机,你就能明白 EntityGraph 为什么不是"又一个语法糖"。

9.2.1 第一代:映射默认 FetchType —— 全局静态

@Entity
public class Order {
    @ManyToOne(fetch = FetchType.EAGER)
    private Customer customer;

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items;
}
[优点]   声明位置集中 —— Entity 内一眼能看到
[缺点]   全局生效 —— 列表 / 详情 / 审计页都强制按这个加载形状走
[隐患]   @ManyToOne 默认 EAGER —— 你不写 fetch,就拿到 N+1 隐患种子

9.2.2 第二代:JPQL join fetch —— 查询级显式

// 详情查询:一次性把 customer + items 拉回
Order order = em.createQuery("""
    select distinct o from Order o
    join fetch o.customer
    left join fetch o.items
    where o.id = :id
    """, Order.class)
    .setParameter("id", 10L)
    .getSingleResult();
[优点]   每条查询自由控制 —— 列表不 fetch,详情 fetch,清楚明确
[缺点]   写在字符串里 —— 重构改字段没编译保护,fetch 路径散落各处
[硬伤]   派生方法 findByStatus() 没有 JPQL 字符串可写 fetch
[硬伤]   Criteria API 写 fetch 要嵌套 fetch().fetch() 链,可读性差
[硬伤]   em.find(Order, id) 完全无 fetch 接入点

9.2.3 第三代:EntityGraph —— 加载形状作为一等公民

@NamedEntityGraph(
    name = "Order.detail",
    attributeNodes = {
        @NamedAttributeNode("customer"),
        @NamedAttributeNode(value = "items", subgraph = "items-subgraph")
    },
    subgraphs = @NamedSubgraph(
        name = "items-subgraph",
        attributeNodes = @NamedAttributeNode("product")
    ))
@Entity
public class Order { /* ... */ }

EntityGraph<?> graph = em.getEntityGraph("Order.detail");
Order order = em.find(Order.class, 10L,
    Map.of("jakarta.persistence.fetchgraph", graph));
[优点 1]  可命名复用 —— "Order.detail" 在 5 个 Repository 方法里挂同一份
[优点 2]  和查询语句解耦 —— em.find / JPQL / Criteria / Spring Data 都能用
[优点 3]  可静态(注解)也可动态(API)—— 静态适合稳定形状,动态适合参数化
[优点 4]  Spring Data 派生方法上可挂 @EntityGraph 注解 —— 第二代死区被覆盖
[局限]   只声明"加载哪些"(fetch plan)—— 不能加 WHERE / SET / 排序
"EntityGraph 不是更快的 JOIN FETCH" —— 它是为了把加载契约从查询字符串里抽离出来,变成 Repository 间可共享、可测试、可重命名的对象。我们团队曾经维护过一个 Repository,里头有 8 个方法各自写着 join fetch o.customer left join fetch o.items,某次需求要把详情页加上 items.product.category,8 个方法的 JPQL 字符串要逐个改一遍 —— 漏改了 3 个,生产报了 3 起 N+1。第二轮重构改用 @NamedEntityGraph(name = "Order.detail"),8 个方法都改成 @EntityGraph(value = "Order.detail"),后续维护只动那一个注解定义,所有挂载点自动同步。这是"把加载形状变成对象"带来的工程红利。

9.3 静态命名图

静态命名图通过 @NamedEntityGraph 写在实体类上,启动时被 Hibernate 解析进 metamodel,运行时通过 em.getEntityGraph(name) 取回。它适合形状稳定、跨 Repository 共享的加载契约 —— "详情图"、"列表图"、"导出图"这类业务级语义名是它的甜蜜区。

图 9-3-1 · @NamedEntityGraph 的三层注解结构

9.3.1 多层 NamedEntityGraph —— 四层关联的完整声明

@Entity
@NamedEntityGraph(
    name = "Order.fullDetail",
    attributeNodes = {
        @NamedAttributeNode("customer"),
        @NamedAttributeNode(value = "items", subgraph = "items-subgraph")
    },
    subgraphs = {
        @NamedSubgraph(
            name = "items-subgraph",
            attributeNodes = {
                @NamedAttributeNode(value = "product", subgraph = "product-subgraph")
            }
        ),
        @NamedSubgraph(
            name = "product-subgraph",
            attributeNodes = {
                @NamedAttributeNode("category")
            }
        )
    }
)
@Table(name = "orders")
public class Order {
    @Id private Long id;
    // ... 其他字段
}

9.3.2 同实体多图 —— 用 @NamedEntityGraphs 包裹

@Entity
@NamedEntityGraphs({
    @NamedEntityGraph(
        name = "Order.list",
        attributeNodes = @NamedAttributeNode("customer")
    ),
    @NamedEntityGraph(
        name = "Order.detail",
        attributeNodes = {
            @NamedAttributeNode("customer"),
            @NamedAttributeNode(value = "items", subgraph = "items-sg")
        },
        subgraphs = @NamedSubgraph(
            name = "items-sg",
            attributeNodes = @NamedAttributeNode("product")
        )
    ),
    @NamedEntityGraph(
        name = "Order.audit",
        attributeNodes = {
            @NamedAttributeNode("auditEvents")
        }
    )
})
@Table(name = "orders")
public class Order { /* ... */ }
[命名约定]   实体名.场景名  ——  "Order.list" / "Order.detail" / "Order.audit"
[加载路径]   每个图都是一棵从根实体出发的关联树
[复用性]     同一个图可在 em.find / JPQL / Spring Data 三个入口共用
[启动开销]   Hibernate 启动时一次性解析为 RootGraphImpl 对象,运行期 O(1) 查找

9.3.3 取出与使用

// 路径 1:em.find + hint
EntityGraph<?> graph = em.getEntityGraph("Order.detail");
Order order = em.find(Order.class, 10L,
    Map.of("jakarta.persistence.fetchgraph", graph));

// 路径 2:JPQL setHint
Order order2 = em.createQuery(
        "select o from Order o where o.id = :id", Order.class)
    .setParameter("id", 10L)
    .setHint("jakarta.persistence.fetchgraph",
             em.getEntityGraph("Order.detail"))
    .getSingleResult();

// 路径 3:Spring Data Repository 注解
public interface OrderRepository extends JpaRepository<Order, Long> {
    @EntityGraph(value = "Order.detail", type = EntityGraph.EntityGraphType.FETCH)
    Optional<Order> findById(Long id);
}
-- 三种路径生成的 SQL 形状一致(假设 fetchgraph 语义)
select o.id, o.customer_id, o.total_amount, o.status,
       c.id, c.name, c.status,
       i.id, i.order_id, i.product_id, i.quantity, i.unit_price,
       p.id, p.name, p.category_id
from   orders o
left   join customers c on c.id = o.customer_id
left   join order_items i on i.order_id = o.id
left   join products p on p.id = i.product_id
where  o.id = ?;

9.4 动态编程式图

静态命名图适合"形状稳定"的场景,但生产里有一类需求是"形状随参数动态变化" —— 比如同一个详情接口,带 ?expand=items,product 参数时拉到 Product 层,不带时只拉到 Order + customer。这种场景必须用编程式 EntityGraph:em.createEntityGraph(Class) 返回一个空图,然后 addAttributeNodes / addSubgraph 动态拼装。

9.4.1 基础 API

EntityGraph<Order> graph = em.createEntityGraph(Order.class);

graph.addAttributeNodes("customer");

Subgraph<OrderItem> itemsSg = graph.addSubgraph("items");
itemsSg.addAttributeNodes("product");

Subgraph<Product> productSg = itemsSg.addSubgraph("product");
productSg.addAttributeNodes("category");

Order order = em.find(Order.class, 10L,
    Map.of("jakarta.persistence.fetchgraph", graph));

9.4.2 形状随业务参数动态拼装

public Order findOrderWithExpansions(Long id, Set<String> expansions) {
    EntityGraph<Order> graph = em.createEntityGraph(Order.class);

    graph.addAttributeNodes("customer");

    if (expansions.contains("items")) {
        Subgraph<OrderItem> itemsSg = graph.addSubgraph("items");

        if (expansions.contains("product")) {
            Subgraph<Product> productSg = itemsSg.addSubgraph("product");

            if (expansions.contains("category")) {
                productSg.addAttributeNodes("category");
            }
        }
    }

    if (expansions.contains("audit")) {
        graph.addAttributeNodes("auditEvents");
    }

    return em.find(Order.class, id,
        Map.of("jakarta.persistence.fetchgraph", graph));
}
[expansions={}]                    SQL JOIN: customer
[expansions={items}]                SQL JOIN: customer + items
[expansions={items,product}]        SQL JOIN: customer + items + product
[expansions={items,product,cat}]    SQL JOIN: customer + items + product + categories
[expansions={audit}]                SQL JOIN: customer + audit_events
[expansions={items,audit}]          SQL JOIN: customer + items + audit_events  ⚠ MultipleBag

9.4.3 API 方法对比

方法返回作用典型用
em.createEntityGraph(Class)EntityGraph<T>从空开始动态拼装
em.createEntityGraph(String name)EntityGraph<?>复制 named graph命名图 + 动态扩展
em.getEntityGraph(String)EntityGraph<?>取 named graph 引用纯命名图使用
graph.addAttributeNodes(...)void顶层加属性非关联标量也能加(JPA 规范允许但通常无效果)
graph.addSubgraph(String)Subgraph<T>顶层加 subgraph多层关联
subgraph.addAttributeNodes(...)void子层加属性下一层关联
subgraph.addSubgraph(String)Subgraph<T>子层再嵌套三层及以上
编程式 API 没有"声明顺序唯一性"的检查 —— 重复添加同一节点不会报错,但会被去重。看起来无害,但如果你在循环里 for (String f : fields) graph.addAttributeNodes(f); 而 fields 里有重复,只会得到一份。更危险的是 graph.addSubgraph("items") 被调用两次 —— 第二次返回的是新对象,第一次的子图节点都被静默丢弃。生产里见过一次调用方法被 inline 了两遍,第一次的 subgraph 配置完全消失,加载形状退化成"只到 Order"。Hibernate 5+ 在重复 subgraph 节点上的行为有过多次微调,建议保持单赋值约束:Subgraph<X> sg = graph.addSubgraph("..."); 一次,后续操作都在变量上,绝不重新 addSubgraph 同名节点。

9.5 FETCH vs LOAD

JPA 2.1 规范定义了两种 EntityGraph 语义:jakarta.persistence.fetchgraph 与 jakarta.persistence.loadgraph。两者都通过 setHint 注入,差别只在"图外的属性怎么处理" —— 但这一个差别足以决定接口最终有没有 N+1。

图 9-5-1 · FETCH 与 LOAD 的语义对比

9.5.1 一段代码看清差异

// 假设 Order.customer 是 @ManyToOne 默认值(即 EAGER,且没人改)
// 而 graph 只列出了 "items"(没列 customer)

EntityGraph<Order> graph = em.createEntityGraph(Order.class);
graph.addAttributeNodes("items");

// 路径 1:FETCH 语义
Order o1 = em.find(Order.class, 10L,
    Map.of("jakarta.persistence.fetchgraph", graph));

// 路径 2:LOAD 语义
Order o2 = em.find(Order.class, 10L,
    Map.of("jakarta.persistence.loadgraph", graph));

路径 2 · LOAD 语义

-- SQL 1: customer 因为 @ManyToOne 默认 EAGER,被漏网加载
select o.id, o.customer_id, o.total_amount,
       i.id, i.order_id, i.product_id
from orders o
left join order_items i on i.order_id = o.id
where o.id = ?;

-- SQL 2: 第二条触发 customer 的 secondary select
select c.id, c.name from customers c where c.id = ?;

[SQL 数] 2 条 (而不是 1 条)
[根因] LOAD 不取消图外的 EAGER 默认 —— customer 自动 EAGER 漏出 N+1
[后果] 接口看上去用了 EntityGraph,实际仍有 N+1

路径 1 · FETCH 语义

-- SQL: 只 fetch items,customer 严格 LAZY
select o.id, o.customer_id, o.total_amount,
       i.id, i.order_id, i.product_id
from orders o
left join order_items i on i.order_id = o.id
where o.id = ?;

[SQL 数] 1 条
[根因] FETCH 强制图外全 LAZY —— customer 是 Proxy,业务代码访问才会触发 SQL
[推荐] 任何时候都用 FETCH,LOAD 几乎只在向后兼容场景才出现

9.5.2 语义对比表

图外属性 (映射 EAGER)强制 LAZY仍 EAGER (漏网)

推荐★ 99% 场景遗留兼容
规范引用JSR 338 § 3.7.4.1JSR 338 § 3.7.4.2
"如果你不能确定团队里所有人都把 @ManyToOne 显式写成 LAZY,那就永远用 fetchgraph 而不是 loadgraph"。LOAD 的规范设计假设你"知道映射默认值且能控制它",但工程现实里 @ManyToOne 默认 EAGER 是新手最容易遗忘的一项 —— LOAD 在这个生态环境里几乎就是"漏网器"。Spring Data 的 @EntityGraph 默认 type 是 FETCH,正是基于"保守即正确"。

9.8 多层 Subgraph

EntityGraph 真正区别于 JOIN FETCH 的能力是多层 Subgraph。一层 fetch 谁都能写,但当业务要"订单 + 订单行 + 商品 + 商品分类"4 层关联一并加载时,JOIN FETCH 的 SQL 字符串会膨胀到难以维护,而 EntityGraph 的 Subgraph 让 4 层结构在代码里依然像一棵树。

9.8.3 行积爆炸的边界

4 层 JOIN 的行积是 1 父 × N items × N products × 1 category —— 看起来只乘 N。但如果同层有两个 to-many 集合(比如同时 fetch items 和 auditEvents),行积变成 1 × N_items × M_audit,Hibernate 会抛 MultipleBagFetchException(详见 9.10);即便用 Set,行积也可能从几百行飙到几万行。

[危险形状]   双 to-many 同层 —— Order → items(10) + auditEvents(50)

9.10 多 bag 与笛卡尔陷阱

多层 Subgraph 让 EntityGraph 看起来"什么都能拉",但 Hibernate 在两个 List 集合同时 fetch 时会抛 MultipleBagFetchException —— 这是 EntityGraph 上手最容易踩的硬性边界。

9.10.1 复现 MultipleBagFetchException

@Entity
public class Order {
    @OneToMany(mappedBy = "order")
    private List<OrderItem> items;   // ← List = "bag"

    @OneToMany(mappedBy = "order")
    private List<AuditEvent> auditEvents;   // ← List = "bag"
}

@NamedEntityGraph(
    name = "Order.itemsAndAudit",
    attributeNodes = {
        @NamedAttributeNode("items"),
        @NamedAttributeNode("auditEvents")   // ← 第二个 bag
    }
)
-- 第一次调用就抛
Caused by: org.hibernate.loader.MultipleBagFetchException:
cannot simultaneously fetch multiple bags:
[com.acme.Order.items, com.acme.Order.auditEvents]

9.10.2 为什么 Hibernate 拒绝?

根因:List(在 Hibernate 术语里叫 bag,因为不保证顺序、允许重复)同时 fetch 会导致无法区分父子行积。假设 Order#10 有 3 个 items + 5 个 auditEvents:

SQL JOIN 结果行数: 1 × 3 × 5 = 15 物理行
每行携带:        order_id, item_id, audit_id

Hibernate 看到这 15 行,要去重 items —— 但 items 在 List 里允许重复,
Hibernate 无法判断 "这 5 个相同 item_id 是真重复还是被 audit 维度放大"。
items.size() 该返回 3 还是 15?语义模糊 —— 直接拒绝。

9.10.3 三种修复路径

方案实现SQL 数优缺点
① 改 Set把其中一个 List 改成 Set1行积仍是 1×3×5,内存代价大;Set 要求 equals/hashCode
② 拆两次查第 1 次 fetch items,第 2 次 fetch auditEvents2每条都干净,无笛卡尔;★ 推荐
③ @Fetch(SUBSELECT)把其中一个集合标 SUBSELECT1 + 1Hibernate 专属,父查询不复杂时优雅
④ 限 1 个大集合EntityGraph 里只放 items,auditEvents 第二次拿2同 ②,更显式

9.10.4 推荐做法:拆查 + 同 PersistenceContext 合并

@Transactional(readOnly = true)
public Order findFullDetail(Long orderId) {
    // 第 1 次:Order + customer + items + product
    Order order = orderRepository.findDetailById(orderId).orElseThrow();

    // 第 2 次:在同 Session 里再 fetch 另一个集合
    Order ordered = em.createQuery("""
        select o from Order o
        left join fetch o.auditEvents
        where o.id = :id
        """, Order.class)
        .setParameter("id", orderId)
        .getSingleResult();

    // ordered == order(PersistenceContext 一级缓存命中)
    return order;
}
[关键]      同事务、同 EntityManager —— 第二次查不创建新对象,
            而是命中一级缓存,把 auditEvents 集合"填"到已存在的 Order 上

[SQL 数]    2 条(各自走 LEFT JOIN,行积可控)
[内存]      不放大,因为没有跨集合笛卡尔

[守护]      assertSame(order, ordered) —— 这一行测试值得加
"Set 不是 MultipleBagFetchException 的真正修复"。Hibernate 接受 Set + List 组合(因为 Set 强制去重)或 Set + Set,但 SQL 仍然返回 N_items × M_audit 行,Hibernate 在 ResultTransformer 里做去重。对 Order#10(3 items + 5 audit)是 15 行;但生产数据 Order#X 有 100 items + 200 audit 时是 2 万行,Set 化只是"不抛异常"而不是"性能修好"。真正干净的修复是拆两次查(方案 ②),或者承认"详情接口不需要两个大集合一起拉",在接口层就拆成两个端点。

9.11.2 启动时 —— 命名图注册

// Hibernate 启动时扫描 @NamedEntityGraph,转化为 RootGraphImpl
// 关键路径(简化):

class NamedEntityGraphDefinition {
    String registrationName;          // 如 "Order.detail"
    NamedEntityGraph annotation;      // 注解原型
    RootGraphImplementor<?> graph;    // 解析后的实体图

    void resolve(SessionFactoryImplementor sf) {
        EntityPersister persister = sf.getMappingMetamodel()
            .getEntityDescriptor(entityName);
        this.graph = new RootGraphImpl<>(registrationName, persister, sf);

        for (NamedAttributeNode an : annotation.attributeNodes()) {
            AttributeNodeImpl node = graph.addAttribute(an.value());
            if (!an.subgraph().isEmpty()) {
                // 嵌套 subgraph 引用,解析为 SubGraphImpl
                resolveSubgraph(node, an.subgraph(), annotation.subgraphs());
            }
        }

        sf.getQueryEngine().getNamedObjectRepository()
            .registerNamedGraph(registrationName, graph);
    }
}

9.11.4 设计取舍

取舍 1:为什么 LOAD 保留 EAGER 默认 —— 向后兼容代价

JPA 2.1 规范定义两种语义,其实是历史包袱。LOAD 的设计是为了"在不破坏既有 EAGER 默认的前提下,补充加载额外路径";FETCH 才是"清洁室"语义。Hibernate 在实现上完全实现了两套语义,运行时根据 hint key 切换 —— 这一个枚举值的差,导致生产里 80% 的 "EntityGraph 不生效" 实际是"用了 loadgraph 但 @ManyToOne 没改 LAZY"。

取舍 2:NamedGraph 是不可变快照 —— em.getEntityGraph 返回引用还是副本?

JPA 规范没强制,Hibernate 实现里 em.getEntityGraph(name) 返回的是注册时的对象,如果你在运行时 graph.addAttributeNodes("...") 修改它,会污染所有后续从同一个 name 拿到的 graph。生产规约:从 getEntityGraph 拿到的图绝不修改;要修改请用 em.createEntityGraph(name) 拿副本(JPA 2.1 起支持)。

取舍 3:为什么 EntityGraph 不能加 WHERE —— 关注点分离

EntityGraph 只声明"加载哪些",不能加"过滤什么"。这看起来限制,实际上是有意为之:加载形状跨多个查询复用,而过滤条件每个查询不同 —— 把两者放在一个对象里会破坏复用性。同一个 Order.detail 图,既能挂在 findById 上,也能挂在 findByStatusAndAmount 上,正是因为 EntityGraph 不掺和 WHERE。

9.12.3 层 3 · SQL 形状断言

@Component
public class SqlRecorder {
    private final List<String> capturedSql = new CopyOnWriteArrayList<>();

    public void record(String sql) { capturedSql.add(sql); }
    public void clear() { capturedSql.clear(); }
    public List<String> all() { return capturedSql; }
    public String singleSelect() {
        return capturedSql.stream()
            .filter(s -> s.toLowerCase().startsWith("select"))
            .findFirst().orElseThrow();
    }
}

@Test
void detailGraphSqlMustContainJoins() {
    sqlRecorder.clear();

    repository.findDetailById(10L).orElseThrow();

    String sql = sqlRecorder.singleSelect().toLowerCase();

    assertThat(sql).contains("left outer join customers");
    assertThat(sql).contains("left outer join order_items");
    assertThat(sql).contains("left outer join products");
    assertThat(sql).doesNotContain("audit_events");
}

9.13.1 类型安全的 attributePaths —— 用 metamodel 替代字符串

反例:裸字符串

@EntityGraph(attributePaths = {
    "customer",
    "items",
    "items.product",
    "items.product.category"
})
Optional<Order> findById(Long id);

重命名 category 字段 → 编译通过 → 运行期默默不 fetch
没有 IDE 字段跳转
打错字符串("catagory")到生产才发现

正例:JPA Metamodel(@StaticMetamodel 生成)

// 编译期生成 Order_, OrderItem_, Product_ 静态类
@EntityGraph(attributePaths = {
    Order_.CUSTOMER,
    Order_.ITEMS,
    OrderItem_.PRODUCT,
    Product_.CATEGORY
})
Optional<Order> findById(Long id);

字段重命名 → 静态字段同步生成 → 编译失败立刻暴露
IDE 可跳转 / 重构安全
需要 jpa-metamodel-generator 注解处理器

"EntityGraph 字符串路径里的每个点都是字段名 —— 全靠拼写,IDE 不保护"。生产里出现过一次最离奇的事故:某次自动化重构把 Order.items 字段重命名为 orderLines,Hibernate metamodel 自动跟随,但 @EntityGraph(attributePaths = "items") 这个字符串没人改 —— Hibernate 启动期对未知 path 默认是静默忽略(取决于版本),运行期所有 items 仍是 LAZY,接口 SQL 数从 1 飙升到 100,直到 P99 报警才发现。所有 attributePaths 推荐用 JPA Metamodel 替代裸字符串,Spring Data 也支持 @EntityGraph(attributePaths = { Order_.CUSTOMER, ... }) 这种类型安全形式。

@Test
@Transactional
void detailGraphCollapsesFourLayersIntoSingleSql() {

    Order order = repository.findFullDetailById(10L).orElseThrow();

    String customerName = order.getCustomer().getName();

        .getCategory().getName();

        .as("Order.fullDetail must collapse 4 layers into 1 SQL")

    assertThat(Hibernate.isInitialized(order.getItems())).isTrue();

        .as("auditEvents not in graph, must stay LAZY (fetchgraph semantic)")
        .isFalse();
}
