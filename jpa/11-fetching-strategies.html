第 11 章 · 抓取策略与 EntityGraph

FetchType.LAZY/EAGER · 代理与 PersistentCollection · JOIN FETCH · @EntityGraph · @BatchSize · @Fetch(SUBSELECT) · DTO 投影 · SQL 数量预算 · LoadQueryInfluencers · BatchFetchQueue · OSIV · 回归护栏

同样一段代码 em.find(Order.class, 1L) 可能产生 1 条 SQL,也可能产生 100 条 — 差异不在 ORM 代码,在 抓取策略(fetching strategy)。一个 Order 实体后面挂着 customer / items / payment / auditEvents / shippingAddress / discounts 等关联,每一个都有"是否本次访问"、"几条 SQL"、"什么时候触发"的具体决策。所有这些决策的乘积就是一个接口的 SQL 预算与延迟分布。理解抓取策略 = 理解"对象图遍历"被翻译成"SQL 集合"的全部规则。

本章把 JPA + Hibernate 提供的全部抓取工具列成清单 — JPA 标准的 6 件武器:默认 FetchType / JOIN FETCH / EntityGraph(fetchgraph + loadgraph) / DTO 投影,加上 Hibernate 扩展的 2 件:@BatchSize / @Fetch(SUBSELECT)。每件工具都有"擅长的形状 + 不能解决的问题",没有任何一件能覆盖所有场景。本章的核心论点是:抓取策略不是某一个方法的属性,而是某一个用例的契约。同一个 Order 实体在"详情查询"用 JOIN FETCH、在"列表 + 过滤"用 EntityGraph、在"导出报表"用 DTO,这三条路径互不冲突,也都不能省略。把"我整个项目用 LAZY 还是 EAGER"问错的人,八成已经踩到 N+1 或 LazyInitializationException 至少一次。

  • JPA 标准默认值:@ManyToOne / @OneToOne 默认 EAGER, @OneToMany / @ManyToMany / @ElementCollection 默认 LAZY — 为什么 EAGER 默认是"标准的失误"
  • LAZY 代理的内部结构:HibernateProxy / $$ $$_hibernate_initialize / AbstractLazyInitializer — 为什么 customer.getId() 不触发 SQL 而 customer.getName() 触发
  • LazyInitializationException 的根因:Session 已关闭后访问代理 — 修复方向是 声明加载形状 而不是 延长 Session(为什么 OSIV 是反模式)
  • JOIN FETCH 的边界:单页详情完美,集合 + 分页冲突(HHH90003004),多 List 同时 fetch(MultipleBagFetchException)
  • @EntityGraph 的两种类型:fetchgraph(图外转 LAZY) vs loadgraph(图外尊重原注解)— 为什么 fetchgraph 是更安全的默认
  • @NamedEntityGraph + @NamedSubgraph 的多层结构,Spring Data 集成,编程式 EntityGraph 与 query hint
  • @BatchSize 的"分批加载"机制:1 + ceil(N/size) 公式,16-32 是 Web API 的合理默认
  • @Fetch(FetchMode.SUBSELECT) 的"二次子查询"模式:严格 2 条 SQL,适合"必访问全集合"
  • DTO 投影 + record:为什么报表 API 应该绕过实体图直接走 SQL → DTO
  • SQL 数量预算表:不同策略下加载 100 个 customer + orders 的 SQL 数公式与权衡
  • Hibernate 内部类:LoadQueryInfluencers / BatchFetchQueue / AbstractPersistentCollection — 它们如何把策略翻译成 SQL
  • 三个回归护栏模板:SQL count 断言、N+1 上限断言、EntityGraph shape 断言

前置知识:第 5-6 章关联映射(@ManyToOne / @OneToMany / @ManyToMany 的方向、所有方、外键列);第 3 章 PersistenceContext 与 Identity Map(本章解释 BatchFetchQueue 如何利用 PC 中收集 owner id);第 7-8 章 JPQL(本章 JOIN FETCH 是 JPQL 语法的一部分)。后续第 12 章 N+1 问题(把本章工具按"修复 N+1"重新组织)、第 13 章二级缓存(本章 @BatchSize 与 L2 的协作)、第 15 章性能调优(SQL 预算的实战调优)。

11.1 业务模型

本章使用一个典型的电商对象图:Order 关联 Customer(单端),Order 拥有 OrderItem 集合,OrderItem 关联 Product。同时 Order 有 AuditEvent 集合作为"低频访问的旁路集合"。

@Entity
@Table(name = "orders")
public class Order {
    @Id
    private Long id;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "customer_id")
    private Customer customer;

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<OrderItem> items = new ArrayList<>();

    @OneToMany(mappedBy = "order", fetch = FetchType.LAZY)
    private List<AuditEvent> auditEvents = new ArrayList<>();

    private BigDecimal total;
}

@Entity
@Table(name = "customer")
public class Customer {
    @Id private Long id;
    private String name;
}

@Entity
@Table(name = "order_item")
public class OrderItem {
    @Id private Long id;
    @ManyToOne(fetch = FetchType.LAZY) private Order order;
    @ManyToOne(fetch = FetchType.LAZY) private Product product;
    private String sku;
    private int quantity;
}
insert into customer(id, name) values (1, 'Ada');
insert into product(id, name) values (501, 'Redis Guide'), (502, 'JPA Visualizer');
insert into orders(id, customer_id, total) values (10, 1, 99.00);
insert into order_item(id, order_id, product_id, sku, quantity) values
    (100, 10, 501, 'SKU-REDIS-001', 1),
    (101, 10, 502, 'SKU-KAFKA-002', 3);

注意映射上所有关联都被显式标记为 FetchType.LAZY,这是经验法则。本章后续每个例子都基于这个模型展开。

11.2 默认 FetchType

JPA 规范规定的默认 FetchType 是个争议焦点:@ManyToOne / @OneToOne 默认 EAGER,而 @OneToMany / @ManyToMany 默认 LAZY。

11.2.1 EAGER 默认带来的列表 N+1

@Entity
public class Order {
    @ManyToOne   // 默认 EAGER!
    private Customer customer;
}

List<Order> orders = em.createQuery("""
    select o from Order o order by o.id
    """, Order.class).getResultList();
// 应用代码什么都没做,只是查列表
-- SELECT 1: 查 Order 列表
select o.id, o.customer_id, o.total
from orders o
order by o.id;

-- SELECT 2..N+1: 每个 Order 触发一次 customer 查询
select c.id, c.name from customer c where c.id = ?;
select c.id, c.name from customer c where c.id = ?;
... 重复 N 次 ...
[100 orders + 100 distinct customers]
SQL count = 101
[根因] EAGER 默认在业务代码触碰前就触发 N+1
[特征] 你看不到任何 customer.getName() 调用,SQL 已经发出去了
"@ManyToOne 默认 EAGER"是 JPA 规范的历史包袱 — JPA 1.0 时代认为"单端关联开销小,默认加载方便",这个判断在"列表查询返回 100 行"的场景立刻翻车。修改默认值需要破坏向后兼容,Jakarta EE 委员会到 2026 年仍未修订。今天的工程实践共识:所有 @ManyToOne / @OneToOne 都显式写 fetch = FetchType.LAZY,把默认值当不存在。Spring Initializr 的 Spring Data JPA starter 已经在文档里把这条规则标为 P0 best practice,Hibernate 6 的 ddl validator 也提供了 hibernate.default_eager_to_one_associations=false 但默认仍未启用——它知道改默认会破坏老项目。

11.2.2 显式 LAZY 修正

@Entity
public class Order {
    @ManyToOne(fetch = FetchType.LAZY, optional = false)
    private Customer customer;
}
select o.id, o.customer_id, o.total
from orders o
order by o.id;
-- 仅 1 条 SQL
SQL count = 1
customer  = uninitialized proxy (访问时才发 SELECT)

optional = false 告诉 Hibernate "这个关联在数据库层是 NOT NULL 的",结合 FetchType.LAZY 可以让 Hibernate 用更精确的代理(无需做 null 判断)。生产代码里 optional 应该按数据库列定义如实声明,而不是为了"图省事"全写 true。

11.3 LAZY 代理机制

LAZY 之所以能"延后加载",是因为 Hibernate 给关联字段塞了一个 代理对象(HibernateProxy)。代理在访问非 id 字段时触发底层加载,这是 LAZY 的核心机制。理解代理 = 理解何时会触发 SQL、何时会抛 LIE。

Proxy 初始化时机

11.3.1 代理观察实验

@Transactional
public void inspectLazyProxy() {
    Order order = em.find(Order.class, 10L);
    Customer customer = order.getCustomer();

    String proxyClass = customer.getClass().getName();
    boolean initBeforeId = Hibernate.isInitialized(customer);
    Long id = customer.getId();
    boolean initAfterId = Hibernate.isInitialized(customer);
    String name = customer.getName();
    boolean initAfterName = Hibernate.isInitialized(customer);

    System.out.println("proxyClass = " + proxyClass);
    System.out.println("initBeforeId = " + initBeforeId);
    System.out.println("id = " + id);
    System.out.println("initAfterId = " + initAfterId);
    System.out.println("name = " + name);
    System.out.println("initAfterName = " + initAfterName);
}
[proxyClass]      com.acme.Customer$HibernateProxy$xxxx
[initBeforeId]    false
[id]              1                   ← 不触发 SQL
[initAfterId]     false               ← getId() 不初始化
[name]            Ada                 ← 触发 SQL
[initAfterName]   true

关键观察:访问 getId() 不会触发初始化。原因是外键列已经在父实体加载时拿到了,代理不需要查询数据库就能返回 id。getName() 才需要去 customer 表 SELECT。这一点对"只需要 id 的场景"非常友好——例如序列化 customerId 时不会意外触发 N+1。

11.3.2 LazyInitializationException 的根因

@Service
class OrderService {
    @Transactional(readOnly = true)
    Order loadOrder(Long id) {
        return em.find(Order.class, id);
    }
}

@RestController
class OrderController {
    @GetMapping("/orders/{id}")
    OrderResponse get(@PathVariable Long id) {
        Order order = orderService.loadOrder(id);  // 事务在这里结束
        // 现在 Session 已关闭,代理失去上下文
        String customerName = order.getCustomer().getName();
        return new OrderResponse(order.getId(), customerName);
    }
}
[异常]
org.hibernate.LazyInitializationException:
could not initialize proxy [com.acme.Customer 1] - no Session

[根因]
Service 方法返回时事务已 commit + Session close
Controller 拿到的 order 持有的 customer proxy 失去 Session
访问 getName() 试图触发 SQL,但没有连接

反例:OSIV 掩盖问题

spring:
  jpa:
    open-in-view: true   # 默认

OSIV 把 Session 延长到 View 渲染结束。
LIE 不再出现,但 Controller 序列化时
仍触发 LAZY → 隐式 N+1 + DB 连接占用整个请求。

正例:声明加载形状 + DTO

spring:
  jpa:
    open-in-view: false
@Transactional(readOnly = true)
OrderDetailDto loadOrder(Long id) {
    return em.createQuery("""
        select new OrderDetailDto(
            o.id, c.name, o.total)
        from Order o join o.customer c
        where o.id = :id
        """, OrderDetailDto.class)
        .setParameter("id", id)
        .getSingleResult();
}

Service 直接返回 DTO,无 Proxy 越界。
Controller 不持有任何懒加载对象。

修复方向是显式声明加载形状(JOIN FETCH / EntityGraph / DTO),而非延长会话(OSIV)。OSIV 让 Session 跨越 Service / Controller / View 三层,看起来"修了" LIE,实际上把 SQL 触发分散到 View 层 — 这是序列化期 N+1 的主要来源。

11.4 JOIN FETCH

JOIN FETCH 是 JPQL 在查询级别声明加载形状的语法:它告诉 Hibernate "在这条查询里,顺带把这些关联也加载好,加进 PC 后标记为已初始化"。结果是一条 SQL 完成多实体加载。

查询级精确加载

图 11-4-1 · JOIN FETCH 把关联折成单条 SQL

11.4.1 单实体详情查询

public Order findOrderDetail(Long id) {
    return em.createQuery("""
        select distinct o from Order o
        join fetch o.customer
        left join fetch o.items
        where o.id = :id
        """, Order.class)
        .setParameter("id", id)
        .getSingleResult();
}
select distinct
    o.id, o.customer_id, o.total,
    c.id, c.name,
    i.id, i.order_id, i.product_id, i.sku, i.quantity
from orders o
join customer c on c.id = o.customer_id
left join order_item i on i.order_id = o.id
where o.id = ?;
[order.id]                10
[customer initialized]     true
[items initialized]        true
[items.size]               2
[SQL count]                1
[JDBC 行]                  2 (Order × items)
[实体]                     1 Order + 2 OrderItem

distinct 关键字在 JOIN FETCH 集合时不影响 SQL 层(SQL 仍然返回笛卡尔行),它只在 Hibernate 应用层去重 — 把 (Order, item1) 与 (Order, item2) 两行去重为同一个 Order 实体的 items 集合。Hibernate 6 后 spring.jpa.properties.hibernate.query.passDistinctThrough=false 默认开启,SQL 不带 distinct 也能正确去重,执行计划更优。

JOIN FETCH 限制

限制现象替代
集合 fetch + 分页HHH90003004 in-memory pagination warning两步分页(先查 id,再 JOIN FETCH)
多 List 同时 fetchMultipleBagFetchException一个改 Set 或拆查
多集合笛卡尔行数爆炸EntityGraph 限 1 大集合
count 查询count 含 fetch 列count 单独查

"集合 fetch + 分页"是 JOIN FETCH 最大的禁区 — 当 JPQL 含 join fetch o.items 又用 setMaxResults / setFirstResult 时,Hibernate 在日志里抛 HHH000104: firstResult/maxResults specified with collection fetch; applying in memory!。它的处理方式是:把所有数据一次性拉回内存,然后在 Java 里分页。如果你以为"分页 10 条"就只查 10 条,实际可能查了几十万条,内存直接爆。修复模式是"两步分页":第一步 select o.id from Order o ... limit 10 offset N 拿到 10 个 id,第二步 select o from Order o join fetch o.items where o.id in (:ids)。这是 Hibernate 6 推荐的标准做法。

11.5 EntityGraph

@EntityGraph 是 JPA 2.1 引入的"加载形状声明" — 把"本次需要加载哪些路径"从查询语句里抽出来,作为可复用的注解或编程式对象。它解决的核心问题是:同一个对象图在 5 个不同的 Repository 方法里都需要加载,JOIN FETCH 要把同样的语句写 5 次,EntityGraph 只写 1 次共享。

fetchgraph vs loadgraph

维度FETCH (fetchgraph)LOAD (loadgraph)
图内关联加载加载
图外关联转 LAZY尊重原实体注解
推荐是 (默认)仅遗留兼容
风险低EAGER 漏网
典型用法明确声明本次形状在 EAGER 基础上加补充
fetchgraph 是更安全的默认 — fetchgraph 把"图外关联一律 LAZY"作为统一保证,意味着无论你的实体上有多少 EAGER 漏网,fetchgraph 都强制"只加载我列出的"。loadgraph 反其道而行 — 它说"图内的我加载,图外的看你映射怎么写"。如果实体有遗留的 EAGER(常见于老项目),loadgraph 会让你的"看似精简"的图实际触发额外 SQL。所以 Spring Data 的 @EntityGraph 默认就是 FETCH 类型 — 这是经过深思熟虑的安全姿态。如果你看到代码里写 EntityGraphType.LOAD,通常是历史代码迁移过来的,可以考虑改成 FETCH 后跑回归测试观察是否有 SQL 变少(变少 = 之前有 EAGER 漏网 = 现在更精确)。

11.5.3 编程式 EntityGraph

EntityGraph<Order> graph = em.createEntityGraph(Order.class);
graph.addAttributeNodes("customer");

Order order = em.find(Order.class, 10L,
    Map.of("jakarta.persistence.fetchgraph", graph));

[custInit]   true
[itemsInit]  false

编程式 EntityGraph 适合"加载形状由运行时决定"的场景 — 例如 GraphQL 查询根据用户 selection set 动态决定加载哪些字段。em.find 接受 hint Map,jakarta.persistence.fetchgraph 是标准 hint 名称。

11.6 BatchSize 与 SUBSELECT

当查询入口已经写好(没法改成 JOIN FETCH),但代码会循环访问 LAZY 关联,JOIN FETCH 不适用,EntityGraph 也不在调用方掌控,这时 @BatchSize 和 @Fetch(SUBSELECT) 是兜底方案 — 它们在映射层声明"如果有人触发我,我应该怎样批量加载"。

11.6.3 全局兜底

spring.jpa.properties.hibernate.default_batch_fetch_size=16

这一行配置开启全局 BatchSize,所有未显式标注 @BatchSize 的 LAZY 关联都按 16 批量加载。生产经验:这是每个新 Spring + JPA 项目第一行该加的配置,它把"潜在的 N+1"转成"最差 N/16",10 倍以上数量级降低事故频率。

BatchSize 取值建议

范围适用风险
5-10小批量 / 关联较少仍多 SQL
16-32Web API 默认平衡
50-100后台批量IN 列表过长
> 200不推荐计划缓存抖动
BatchSize 太大会触发"绑定参数过多"错误 — Oracle JDBC 默认单 PreparedStatement 参数上限是 1000;PostgreSQL 是 32767;SQL Server 是 2100;DB2 是 32767。如果 @BatchSize(size = 1000) 加上某次查询正好凑到 1000 个 customer 一起初始化,Oracle 会抛 ORA-01795: maximum number of expressions in a list is 1000。解决:BatchSize 不要超过 100(给应用层留 buffer),如果某些超大批量场景需要,改用 @Fetch(SUBSELECT) 把"IN 列表"换成"子查询"。Hibernate 6 还内置了 BatchLoader 的 IN 切片机制(hibernate.dialect.batch_versioned_data),会自动按方言上限切分,但仍以"单查询 ≤ 数据库参数上限"为前提。

11.7 DTO 投影

DTO 投影是绕开实体图的策略 — 直接 select 列,封装到 DTO/record,不进入 PersistenceContext。适合"只读 API"、"报表"、"导出"场景:这些场景不需要"对象身份"和"dirty checking",反而需要"少加载、少内存、快响应"。

11.7.1 record DTO

public record OrderSummaryDto(
    Long orderId,
    String customerName,
    BigDecimal total,
    Long itemCount) {}

public OrderSummaryDto findSummary(Long orderId) {
    return em.createQuery("""
        select new com.acme.OrderSummaryDto(
            o.id, c.name, o.total, count(i)
        )
        from Order o
        join o.customer c
        left join o.items i
        where o.id = :orderId
        group by o.id, c.name, o.total
        """, OrderSummaryDto.class)
        .setParameter("orderId", orderId)
        .getSingleResult();
}
select o.id, c.name, o.total, count(i.id)
from orders o
join customer c on c.id = o.customer_id
left join order_item i on i.order_id = o.id
where o.id = ?
group by o.id, c.name, o.total;

{
  "orderId": 10,
  "customerName": "Ada",
  "total": 99.00,
  "itemCount": 2
}

[特点]
- 一条 SQL 完成所有聚合
- 不进入 PersistenceContext
- 无 dirty checking 开销
- 不能修改后保存(本来就不该)

DTO 不进入 PersistenceContext,无脏检查与级联。修改场景应重新加载实体。这是 DTO 模式的核心约束:读路径与写路径分离。CQRS 架构在这里有天然的契合度。

11.9.1 设计取舍

取舍 1:三层职责分工

映射层提供安全默认值 (LAZY),查询层声明本次形状 (JOIN FETCH / EntityGraph),运行时根据 PersistenceContext 与代理状态决定补取。任一层当作万能解决方案都会制造新问题。映射 EAGER → 列表 N+1;查询 JOIN FETCH 应付所有 → 笛卡尔行积;运行时 BatchSize 全靠 → 仍随 N 增长。三层各司其职是 Hibernate 加载架构的核心。

取舍 2:LoadQueryInfluencers 不修改谓词

EntityGraph 只声明加载路径,不能表达 WHERE 过滤。复杂过滤须 JPQL / Criteria。这种限制使得加载契约可复用,与查询过滤独立演进。如果 EntityGraph 同时表达过滤,就会出现"同一个图 + 不同过滤"的语义混乱。

取舍 3:BatchFetchQueue 限于补取

BatchSize 只能减少 SQL 次数,不能让接口变成严格 1 条 SQL。需要严格 1 条 SQL 时必须用 JOIN FETCH 或 EntityGraph。BatchFetchQueue 的设计目标是"已经在 LAZY 触发的现实下,把 N 条 SQL 合并成 N/B 条",而不是"消灭 LAZY"。

取舍 4:HibernateProxy 用字节码生成,不依赖反射

代理类在启动期由 Byte Buddy(以前是 Javassist / CGLIB)生成,不走 JDK 动态代理(后者只能代理接口)。Hibernate 选择字节码代理是为了支持"实体类是普通 class 而非接口",这与 JPA 规范契合 — 实体不要求实现任何 marker interface。代价是构建期需要字节码增强工具链,Spring Boot starter 默认配置好了所以业务无感。

11.11.3 EntityGraph Shape 守护

@Test
void orderDetailGraphShape() {
    Order order = orders.findDetailById(10L).orElseThrow();

    boolean custInit = Hibernate.isInitialized(order.getCustomer());
    boolean itemsInit = Hibernate.isInitialized(order.getItems());
    boolean auditInit = Hibernate.isInitialized(order.getAuditEvents());

    assertThat(custInit).isTrue();
    assertThat(itemsInit).isTrue();
    assertThat(auditInit).isFalse();   // 关键:图外不该被加载
}
"shape 断言"是被严重低估的回归护栏 — SQL count 只能告诉你"几条 SQL",但不能告诉你"是哪些关联被加载/没被加载"。Shape 断言用 Hibernate.isInitialized(field) 直接验证"customer 已初始化、items 已初始化、auditEvents 未初始化"。这种断言对 EntityGraph 重构特别有用 — 当你修改 NamedEntityGraph 的 attributeNodes,shape 断言立刻告诉你"这个图比之前少加载了 X" 或 "多加载了 Y"。Vlad Mihalcea 的 db-util 库提供了 SQLStatementCountValidator 自动化这类断言,值得在测试 dependency 里引入。
