第 10 章 · Native SQL 与结果映射

createNativeQuery · @SqlResultSetMapping · @ConstructorResult · @NamedNativeQuery · 存储过程 · 同步空间 · 二级缓存失效 · 方言差异 · Testcontainers · 回归护栏

JPQL 的世界是抽象的:你写 select c from Customer c where c.email = :e,Hibernate 翻译成方言相关的 SQL,自动绑定参数、自动维护一级缓存、查询前自动 flush——这是 JPA 给应用代码的"贴身管家"。但管家总有它的边界。当你需要 窗口函数、递归 CTE、JSON 操作符、数据库 hint、UPSERT、全文检索、厂商专属函数(generate_series、connect by、match against),JPQL 立刻无能为力——这些是表达力之外的真空地带,JPA 标准刻意没有把它们抽象进规范。Native SQL 就是这个真空地带的"逃生通道":你直接写厂商方言 SQL,Hibernate 只负责执行 + 把结果集装回对象。

但"逃生通道"这个比喻同时也是警告:这条路径放弃了 JPQL 的诸多保护机制。JPQL 编译期就能检查"字段名是否拼错"、"实体名是否存在"、"参数类型是否匹配";Native SQL 全部要等到运行时才能由数据库报错。JPQL 在事务内的写入会自动触发 auto flush 让查询读到一致视图;Native SQL 默认不知道你刚刚改了什么,可能读到 stale 数据。JPQL 返回的实体进入 PersistenceContext 自动参与 dirty checking;Native SQL 如果误用 resultClass = Entity.class 加载报表行,会让 PC 在不知不觉中膨胀到几十万条。本章的核心论点是:Native SQL 的难度不在 SQL 编写本身,而在结果映射、缓存同步、方言绑定、回归测试这四件事的工程化。会写 SQL 的人很多,能让 Native SQL 在生产里长期稳定不出 bug 的人少得多。

  • createNativeQuery 三种返回形态的差异:Object[] / resultClass=Entity / resultSetMapping=name — 各自的内存与 PC 影响
  • JDBC 驱动类型映射的"驱动方差":同一个 BIGINT 列在 PostgreSQL/MySQL/Oracle 驱动下可能返回 Long / BigInteger / BigDecimal,代码必须经 Number 接口转换
  • @SqlResultSetMapping 的三种映射:@EntityResult(实体行) / @ColumnResult(标量列) / @ConstructorResult(DTO),三者可在一个 mapping 内共存
  • @ConstructorResult + record 的现代 DTO 投影模式 — 用窗口函数 + DTO 组合做报表的标准范式
  • @NamedNativeQuery 在启动期注册的好处:语法/映射的"早失败"机制,但 SQL 语义/权限/计划仍要集成测试
  • Native DML(INSERT/UPDATE/DELETE)的三步缓存同步:em.flush() + em.clear() + Cache.evict() — 任何一步缺失都会留下脏数据
  • StoredProcedureQuery 的 IN/OUT/INOUT/REF_CURSOR 四种参数模式 + 厂商驱动差异
  • 方言差异矩阵:JSON 取值、UPSERT、分页、递归、全文检索 — 每种厂商语法都有"看起来一样实际不同"的陷阱
  • StatefulPersistenceContext 与 NativeQueryImpl 的源码协作:实体如何"水合"进 PC、scalar 如何走纯 JDBC 路径
  • 同步空间(synchronized query space)的概念:addSynchronizedEntityClass 是怎么让 Hibernate "知道" Native SQL 要读哪些表,从而触发 auto flush 并失效查询缓存
  • Testcontainers 真实方言测试 vs H2 兼容模式 — 为什么 H2 跑过的 SQL 在 PostgreSQL 上会失败
  • 10 个高频陷阱表 + 2 个回归护栏测试模板:别名稳定守护、缓存同步守护
  • "拒绝清单":什么时候 不应该用 Native SQL — Native SQL 不是免费的,过度使用会让整个项目失去 JPQL 抽象

前置知识:第 7-9 章 JPQL/Criteria 表达式、绑定参数、动态查询;第 3 章 PersistenceContext 的 Identity Map 与 ActionQueue(本章解释为什么 native UPDATE 会让 PC 中的对象状态过期);第 8 章 JPQL 的 bulk update/delete(它们也绕过 PC,与 native DML 同病相怜)。后续第 13 章二级缓存(本章 Cache.evict 的细节展开)、第 14 章批处理(本章 jdbc.batch_size 与 native batch 的对比)、第 15 章性能调优(本章 native window function 报表的执行计划)。

Native SQL 的"放弃"是刻意的 — JPA 规范把 Native SQL 暴露出来不是因为"JPQL 不够好",而是因为"任何抽象都有边界,边界外必须有逃生通道"。如果 JPA 试图把所有数据库厂商专属功能都抽象进 JPQL,会出现两种灾难:(1) JPQL 标准变得极其复杂,新手永远学不会;(2) 抽象层封住了 50% 的功能,Native SQL 反而成了 80% 真实项目的隐藏路径。把 Native 暴露成显式 API 让"我现在用了 PostgreSQL 专属能力"这件事在代码里有迹可循——code review 看到 createNativeQuery 立刻知道要审视方言绑定 + 缓存同步,如果是 JPQL 反而容易被忽略。这是一种"刻意暴露丑陋以提示风险"的设计哲学。

10.1 createNativeQuery 基础

最朴素的入口:不指定 mapping、不指定 resultClass,直接拿原始 ResultSet 的列 → Object[] 数组。这种用法适合临时调试、一次性脚本、运维查询;不适合长期运行的业务代码,因为列序变更立即破坏调用方,且需要应用代码自己做类型转换。

Object[] 映射路径

图 10-1-1 · Object[] 映射路径:JDBC ResultSet 到 Java 类型的转换链

10.1.1 Object[] 显式转换示例

Query q = em.createNativeQuery("""
    select e.id, e.name, e.price
    from product e
    where e.inventory_id = ?1
    order by e.price desc
    """);
q.setParameter(1, 10L);

List<Object[]> rows = q.getResultList();
for (Object[] row : rows) {
    Long id = ((Number) row[0]).longValue();
    String name = (String) row[1];
    BigDecimal price = (BigDecimal) row[2];
    System.out.printf("%d  %s  %s%n", id, name, price);
}
[输出示例]
101  Customer C1001  180000.00
109  Customer C1002  156000.00

[注意] BIGINT 在不同 JDBC 驱动可能返回 Long / BigInteger / BigDecimal
[规则] 数字列必须经 Number 接口转换,禁止直接强转

注意 ((Number) row[0]).longValue() 这种"经 Number 中转"的写法不是冗余防御,而是必要的可移植性保障——下一节的"驱动差异表"会展示同一个 SQL 在三种数据库下返回的运行时类型不同,直接 (Long) row[0] 在 Oracle 上会立刻 ClassCastException。

"H2 测过 = PostgreSQL 也过"是经典误判 — H2 默认使用自己的 BIGINT → Long 映射,与 PostgreSQL 一致;但 JSONB / UUID / 数组类型 H2 完全没有真实实现,只能"近似"。如果你在 H2 上写 where payload->>'status' = 'OPEN',H2 直接语法报错;反过来,你在 H2 上把 BIGINT 强转成 (Integer) row[0] 凑巧能跑(因为 H2 驱动友好转换),到 PostgreSQL 上立刻 ClassCastException。结论:Native SQL 必须用 Testcontainers 跑真数据库,H2 只能作为不涉及方言专属语法时的"快速检查"。

10.1.2 Object[] 的隐藏问题:列序敏感

// 第 1 版 SQL
String sql1 = "select id, name, price from product where id = ?1";
Object[] row = (Object[]) em.createNativeQuery(sql1)
    .setParameter(1, 1L).getSingleResult();
String name = (String) row[1];  // OK

// 半年后,SQL 维护时加了一列
String sql2 = "select id, sku, name, price from product where id = ?1";
Object[] row2 = (Object[]) em.createNativeQuery(sql2)
    .setParameter(1, 1L).getSingleResult();
String name2 = (String) row2[1];  // ❌ 现在返回的是 sku!

下标访问 + 列序变更 = 静默错位。最可怕的是 sku 与 name 都是 String,类型转换不报错,name 字段被赋值成 sku 数据,业务逻辑处理后写回数据库 — 这种 bug 排查难度极高。修复方式:下标访问只用于一次性脚本,业务代码用 @SqlResultSetMapping 或 record 投影,让"列名 → 字段"形成显式契约。

10.2 实体映射

第二种返回形态:resultClass = Entity.class,Native SQL 的查询结果直接装成实体。这种用法最像 JPQL,实体进入 PersistenceContext,享受一级缓存与 dirty checking。代价是 Native SQL 的列必须严格匹配实体的字段映射,否则装载失败。

Native 返回 Managed 实体

图 10-2-1 · Native 返回 Managed 实体的水合路径

10.2.1 实体映射代码

List<Product> products = em.createNativeQuery("""
    select id, name, price, inventory_id
    from product
    where inventory_id = ?1
    """, Product.class)
    .setParameter(1, 10L)
    .getResultList();

products.get(0).raisePrice(new BigDecimal("1000.00"));
em.flush();
-- 第 1 条:Native SELECT
select id, name, price, inventory_id
from product
where inventory_id = ?;

-- 第 2 条:setter 触发 dirty checking,flush 时自动 UPDATE
update product
set inventory_id = ?, name = ?, price = ?
where id = ?;

实体路径与普通 find 完全一致:Native 查询返回的实体进入 PersistenceContext,参与 Dirty Checking、级联、回调。这是 Native SQL 与 JPQL 在"返回类型"维度上唯一相同的地方。但要注意一个隐藏成本:报表查询如果误映射为实体,会让 PersistenceContext 装入 N 个 Managed 实体,每个实体都有 Snapshot 副本,commit 时全部参与 dirty checking,当 N=10 万时这是显著的内存与 CPU 浪费。

"报表用实体映射"是中后端最常见的隐性性能 bug — 我见过的真实案例:某金融系统的"客户账单查询"接口用 Native SQL + resultClass=Customer.class 加载客户列表,前端展示完关闭页面。表面上看一切正常,但实际上每个请求都让 1000 个客户进入 PC,在事务 commit 时跑 1000 次 dirty checking。后来切到 @ConstructorResult + record DTO,接口 P99 从 320ms 降到 45ms,heap 占用降低 60%。这条规则是:只读路径用 DTO,写路径才用实体。判断依据非常简单:这个查询的结果会不会被业务逻辑修改并 setter? 不会就用 DTO,会就用实体。

10.3 @SqlResultSetMapping

第三种返回形态(也是生产里最强大的):@SqlResultSetMapping。它把"SQL 列别名 → 实体字段 / DTO 构造参数"的对应关系外置成注解,SQL 与映射解耦,可以处理"多实体 join"、"实体 + scalar 混合"、"DTO 投影 + 嵌套关联"等复杂场景。

三种映射类型

图 10-3-1 · @SqlResultSetMapping 的三种映射类型

10.3.1 多实体映射

@SqlResultSetMapping(
    name = "ProductWithInventory",
    entities = {
        @EntityResult(
            entityClass = Product.class,
            fields = {
                @FieldResult(name = "id", column = "product_id"),
                @FieldResult(name = "name", column = "product_name"),
                @FieldResult(name = "price", column = "product_price"),
                @FieldResult(name = "inventory.id", column = "product_inventory_id")
            }),
        @EntityResult(
            entityClass = Inventory.class,
            fields = {
                @FieldResult(name = "id", column = "inventory_id"),
                @FieldResult(name = "name", column = "inventory_name")
            })
    })
@Entity
class Product { /* mapping holder */ }
List<Object[]> result = em.createNativeQuery("""
    select e.id           as product_id,
           e.name         as product_name,
           e.price        as product_price,
           e.inventory_id as product_inventory_id,
           d.id           as inventory_id,
           d.name         as inventory_name
    from product e
    join inventory d on d.id = e.inventory_id
    """, "ProductWithInventory")
    .getResultList();

for (Object[] row : result) {
    Product product = (Product) row[0];
    Inventory inventory = (Inventory) row[1];
    System.out.println(product.getName() + " @ " + inventory.getName());
}

10.4 ConstructorResult DTO 投影

@ConstructorResult 是 Native SQL 的"DTO 投影"模式 — Native SQL 的窗口函数 / 聚合 / JSON 处理能力,加上 record 类型的简洁,这是现代 Java 项目里报表查询的最佳实践。结果不进入 PersistenceContext,无脏检查、无级联、无内存累积。

10.4.1 窗口函数 + DTO

public record ProductPriceRankDto(
    Long productId,
    String productName,
    BigDecimal price,
    Long inventoryRank) {}

@SqlResultSetMapping(
    name = "PriceRankMapping",
    classes = @ConstructorResult(
        targetClass = ProductPriceRankDto.class,
        columns = {
            @ColumnResult(name = "product_id", type = Long.class),
            @ColumnResult(name = "product_name", type = String.class),
            @ColumnResult(name = "price", type = BigDecimal.class),
            @ColumnResult(name = "inventory_rank", type = Long.class)
        }))
@Entity
class Product { /* mapping holder */ }
List<ProductPriceRankDto> ranks = em.createNativeQuery("""
    select e.id   as product_id,
           e.name as product_name,
           e.price as price,
           dense_rank() over (
               partition by e.inventory_id
               order by e.price desc
           ) as inventory_rank
    from product e
    where e.active = true
    """, "PriceRankMapping")
    .getResultList();

ranks.forEach(System.out::println);
[输出示例]
ProductPriceRankDto[productId=P1001, productName=Redis Guide, price=128.00, inventoryRank=1]
ProductPriceRankDto[productId=P1002, productName=JPA Visualizer, price=99.00, inventoryRank=2]

[要求] PostgreSQL 14+ / MySQL 8+ / Oracle 12c+ (window function)
[特点] 不进入 PC,即使加载 100 万条也不会膨胀 PersistenceContext
record + @ConstructorResult 是 Java 17+ 的"零样板 DTO"模式 — 在 record 出现之前,DTO 类要写 constructor + getter + equals + hashCode + toString,即使 Lombok 加 @Value 也比 record 多一层。record 一行声明所有字段且不可变,与 Native SQL 的"列 → 构造参数"映射完美契合。生产建议:report 类的 DTO 一律用 record + @ConstructorResult,不要用普通类 + setter。这样还能避免 setter 反射注入带来的运行时类型不匹配 — record 的构造器在编译期就绑定字段类型,SQL 列类型与 record 字段类型不匹配会立刻 ClassCastException 而不是隐式转换出错值。

10.4.2 嵌套 DTO 与 List 折叠

如果 DTO 内部还需要嵌套子 DTO(例如"客户列表,每个客户带最近 3 个订单摘要"),Native SQL 一条 SQL 无法直接构建嵌套结构。两种典型做法:

  1. 两条 SQL + 应用层组装:第一条查客户,第二条用 where customer_id in (...) 一次性查所有订单,在 Java 里用 Map<customerId, List<OrderDto>> 聚合。简单稳健,SQL 各自独立易优化。
  2. JSON 聚合(PostgreSQL 14+ / MySQL 8+):用 json_agg / json_object_agg 在 SQL 层把子集合聚成 JSON 字符串,Java 端用 Jackson 反序列化。一条 SQL 完成,但失去类型校验。

10.5 @NamedNativeQuery

@NamedNativeQuery 把 SQL + 结果映射在启动期注册到 Hibernate 元模型,有两个工程价值:(1) 启动期就能验证 mapping 与 SQL 别名是否一致,提早暴露 misconfig;(2) SQL 集中放在实体类附近,避免散落在 Repository 各处。

@NamedNativeQuery(
    name = "Product.expensiveProducts",
    query = """
        select id, name, price, inventory_id
        from product
        where price > ?1
        order by price desc
        """,
    resultClass = Product.class)
@Entity
class Product { }
List<Product> products = em
    .createNamedQuery("Product.expensiveProducts", Product.class)
    .setParameter(1, new BigDecimal("100000"))
    .getResultList();
[启动日志]
INFO  Named query registered: Product.expensiveProducts
Hibernate:
    select id, name, price, inventory_id
    from product
    where price > ?
    order by price desc

命名查询在启动时注册,Hibernate 验证结果映射定义。SQL 语法、权限、执行计划仍须集成测试覆盖——Hibernate 启动期不会真的去数据库执行这条 SQL,所以"列名拼错"、"权限不足"、"索引缺失"这类运行时问题它检测不到。

10.6 Native 写操作

Native INSERT / UPDATE / DELETE 是 Native SQL 最危险的部分——它不经过 PersistenceContext,直接发 SQL 到数据库。带来三层不一致风险:(1) PC 中已加载的同主键实体仍是旧值;(2) 二级缓存(L2)中的对象仍是旧值;(3) 查询缓存(query cache)中包含该表的查询结果仍是旧值。修复模式是固定的"三步同步"。

Native DML 缓存同步

图 10-6-1 · Native DML 之后的三层缓存同步

10.6.1 安全写法

// 第 1 步:把 PC 中待 flush 的变更写出去,避免被丢失
Product product = em.find(Product.class, 1L);
em.flush();

// 第 2 步:执行 Native UPDATE
int updated = em.createNativeQuery("""
    update product
    set price = ?1
    where id = ?2
    """)
    .setParameter(1, new BigDecimal("99.00"))
    .setParameter(2, 1L)
    .executeUpdate();

// 第 3 步:清空 PC + L2 + 同步空间
em.clear();
em.getEntityManagerFactory().getCache().evict(Product.class, 1L);
[流程]
1. em.flush()                  - 写入待保存的 ORM 变更
2. native UPDATE 执行          - 数据库行更新
3. em.clear()                  - 清空 PersistenceContext
4. L2 evict Product 1          - 失效二级缓存

[关键不变量]
flush 前 PC 有未 flush 改动 → native UPDATE 不会读到这些改动
flush 后再 native UPDATE     → 数据库视图统一
clear 前 PC 仍持有旧 product → 后续 find 读 PC 旧值
clear 后再 find              → 重新 SELECT,读最新值

Native DML 与缓存层级

操作PC 同步L2 同步查询缓存同步触发回调
em.persist / em.merge自动自动自动 (按 query space)是 (@PrePersist 等)
JPQL Bulk update/delete否 (须 clear)自动 (按 query space)自动否

JDBC 直接执行否否否否
"我 update 完 find 还是旧值"99% 是缓存同步漏了 — 这种 bug 的可怕之处是它在测试环境通常不复现:测试事务结束后 PC 与 L2 都释放了,下次请求重新 SELECT 数据自然对。但生产环境长事务、长 EM 生命周期、Spring Bean 单例 EMF + L2 启用,所有缓存层都活着,PC/L2 中的旧值会被持续返回,直到某个请求触发 evict 才"治好"。生产排查路径:看代码里 Native DML 后是否有 em.clear() + Cache.evict(),缺一即定位。如果你团队大量使用 Native DML,建议封装一个 NativeDmlExecutor 工具类,把"3 步同步"做成强制模板,避免每个开发者各自实现。

JPA 通过 StoredProcedureQuery 提供存储过程调用 API。比起 Native SQL 它多了"参数模式"(IN/OUT/INOUT/REF_CURSOR)的概念,这些都是为了适配厂商存储过程的真实接口形态。

"业务流程进存储过程"在 2026 年是反模式 — 1990 年代到 2010 年代上半,把业务逻辑放进存储过程是为了"减少网络往返、贴近数据"。今天大多数情况下这条逻辑已经不成立:连接池 + JDBC 批处理足以减少往返,而存储过程的版本管理、调试体验、单元测试、跨数据库可移植性都远逊于 Java 代码。建议:存储过程仅用于"数据库厂商专属优化"(比如 Oracle PL/SQL 的批量游标 BULK COLLECT、PostgreSQL 的 PL/pgSQL CTE 优化、需要 trigger 紧耦合的场景),业务逻辑回到 Java 端。新项目原则:除非 DBA 团队明确要求,否则不写 sp。

10.9.1 设计取舍

取舍 1:保留 EntityManager 入口

Native 查询仍走 EntityManager,事务、参数绑定、实体管理全可复用。代价是 SQL 语义已经绑定目标数据库,跨数据库迁移须重写。如果设计成"独立的 NativeJdbcSession"对象,可以更明确地表达"这是逃生通道,不享受 JPA 服务",但会增加 API 复杂度。Hibernate 选择后者:统一入口 + 显式风险。

取舍 2:实体结果进入 PersistenceContext

Native 返回的实体与 JPQL 返回的实体行为一致,可参与脏检查、级联、回调。代价是报表查询若误映射为实体,会放大 PersistenceContext。设计选择"行为一致"优于"性能保护",因为不一致行为会让开发者混乱(同样的 Entity 怎么 Native 加载就不能修改了?)。

取舍 3:DTO 映射显式化

@SqlResultSetMapping 要求 SQL 别名与 mapping 配置精确对应。代价是修改 record 构造器或 SQL 别名时,两端必须同步,运行期才能发现错配。这是有意为之 — 让"SQL 列 → DTO 字段"成为代码契约,任何变更都强制评审,避免隐式变化。

取舍 4:同步空间显式声明

JPQL Hibernate 自动推断查询涉及的表 (query space)。Native SQL 因方言复杂(子查询、CTE、视图、JSON 路径),Hibernate 默认无法解析,须 addSynchronizedEntityClass 显式声明,以触发 auto flush 与查询缓存失效。设计选择"显式优于隐式",避免错误推断造成的性能/一致性 bug。

10.10 同步空间

同步空间(synchronized query space)是 Native SQL 的"知识接口" — 你告诉 Hibernate "这条 SQL 要读/写哪些表",Hibernate 利用这个信息在合适时机 auto flush 与失效查询缓存。不声明同步空间 = Hibernate 当成黑盒,任何 PC 中的 dirty 数据都不会自动 flush,导致 Native query 读到 stale 视图。

10.10.1 显式声明同步空间

List<ProductReportRow> rows = em.createNativeQuery("""
    select p.product_category_id, count(*) as product_count
    from products p
    group by p.product_category_id
    """, "ProductReportMapping")
    .unwrap(org.hibernate.query.NativeQuery.class)
    .addSynchronizedEntityClass(Product.class)
    .getResultList();

10.10.2 Bulk Native DML 安全模式

@Transactional
public int bulkDiscountDiscontinued() {
    em.flush();   // 1. 写出 PC 中已有 dirty 改动

    int updated = em.createNativeQuery("""
        update product
        set price = price * 0.90
        where discontinued = true
        """)
        .unwrap(org.hibernate.query.NativeQuery.class)
        .addSynchronizedEntityClass(Product.class)   // 2. 声明涉及表
        .executeUpdate();                              // 3. 数据库批量更新

    em.clear();  // 4. PC 清空,防 stale 读
    em.getEntityManagerFactory().getCache().evict(Product.class);  // 5. L2 失效
    return updated;
}
[before] em.flush()                       - 写入待 flush 变更
[exec]   native UPDATE                    - 数据库批量更新
[after]  em.clear()                       - 清空一级缓存
[after]  L2 evict Product region          - 失效二级缓存
[after]  Query Cache 同步失效              - 涉及 product 表的缓存查询

[这 5 步任何一步漏掉的现象]
- 漏 flush:之前的改动被 native 查询覆盖
- 漏 synchronized:auto flush 不触发,事务内 query 读旧值
- 漏 clear:之后的 find 读到 PC 中的旧 entity
- 漏 L2 evict:跨事务请求读到 L2 旧 entity
- 漏 query cache:依赖 product 表的缓存查询返回旧聚合值

10.11.1 字符串拼接 → SQL 注入

反例:字符串拼接

String name = userInput;  // 攻击者输入
String sql = "select * from product " +
             "where name = '" + name + "'";

em.createNativeQuery(sql).getResultList();

用户输入 ' OR '1'='1 → 全表导出。
每个不同的 name 都生成独立 SQL → prepared statement cache 命中 0%。

正例:setParameter

em.createNativeQuery("""
    select * from product where name = ?1
    """)
    .setParameter(1, userInput)
    .getResultList();

JDBC 层把 ? 与值分离,注入不可能。
SQL 文本固定 → prepared statement cache 100% 复用。

10.11.2 集合关联无法被 Native 自动加载

@Entity
public class Customer {
    @Id Long id;
    @OneToMany(mappedBy = "customer", fetch = FetchType.LAZY)
    private List<Order> orders;
}

// Native 加载实体
Customer c = (Customer) em.createNativeQuery(
    "select id, name from customers where id = ?1",
    Customer.class)
    .setParameter(1, 1L)
    .getSingleResult();

// 访问集合 → 触发 LAZY 加载,需 PC 仍打开
c.getOrders().size();   // 触发 SELECT orders

Native SQL 不支持像 JPQL 的 join fetch 那样在一条 SQL 里把集合关联加载进来。要么:(1) 接受 LAZY 触发额外 SELECT;(2) 用 @SqlResultSetMapping 同时映射两个实体并应用层组装。集合的"主从行积"装载本质上是 JPQL 层的能力。
