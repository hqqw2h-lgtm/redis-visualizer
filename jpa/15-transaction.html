第 15 章 · 事务管理与传播行为

RESOURCE_LOCAL vs JTA · @Transactional 代理 · 七种传播行为 · 四种隔离 · 异常回滚 · PersistenceContext 边界

事务是 JPA 的核心控制单元。@Transactional 不仅决定 commit / rollback,还决定 PersistenceContext 的生命周期、flush 时机、异常回滚规则、传播行为对外层事务的影响。

事务边界与 PersistenceContext 边界、JDBC 连接占用、Spring AOP 代理、同类内部调用四个层次共同决定行为。本章按事务模型、传播行为、隔离级别、异常回滚、PersistenceContext 关系五维讨论,并附 6 条事务调试 checklist。

  • RESOURCE_LOCAL 与 JTA 两种事务模型的对比,以及"何时该上 JTA"
  • Spring @Transactional 的完整调用链:Proxy → TransactionInterceptor → JpaTM → EM → DB
  • 七种传播行为决策树:REQUIRED / REQUIRES_NEW / NESTED / MANDATORY / NEVER / SUPPORTS / NOT_SUPPORTED
  • NESTED 的 SAVEPOINT 实现 + 为何 JpaTransactionManager 不支持 NESTED
  • 四种隔离级别 + 五个主流数据库的默认值;何时该用 SERIALIZABLE 何时该用乐观锁
  • Spring 回滚规则:RuntimeException + Error 自动 rollback;受检异常默认 commit;rollbackFor 显式控制
  • try-catch 吞异常的隐藏陷阱:rollback-only 标记 + 方法返回时仍 rollback
  • PC 与事务的两种范围:Transaction-scoped(默认)vs Extended(Stateful Bean)
  • 同类内部调用绕过代理的根因,以及三种修复(独立 Service / AopContext / 编程式)
  • TransactionTemplate 编程式事务在"循环内单条事务"的应用
  • 核心源码:TransactionInterceptor / JpaTM / TransactionSynchronizationManager / EntityManagerHolder
  • 事务调试 6 条 checklist + 三种取舍(AOP 而非字节码 / 默认 Runtime rollback / PC 默认 Tx-scoped)

前置知识:Spring AOP 基础;JDBC commit/rollback/SAVEPOINT 语义;数据库隔离级别基础;受检异常 vs 运行时异常的 Java 语义;第 3 章 PersistenceContext;第 13 章 L2 失效。

15.0 章首速览

一句话总览。@Transactional 不只是"加事务",它是一个 AOP 代理切面,定义了"事务边界 = PC 边界 = 异常回滚边界"三合一。同类内部调用会绕过代理,受检异常默认 commit,这两件事是 90% 事务 bug 的根因。

15.1 RESOURCE_LOCAL

EntityManager em = emf.createEntityManager();
EntityTransaction tx = em.getTransaction();

try {
    tx.begin();
    Order order = new Order("TX-1001", new BigDecimal("99.00"));
    em.persist(order);
    tx.commit();
} catch (RuntimeException ex) {
    if (tx.isActive()) {
        tx.rollback();
    }
    throw ex;
} finally {
    em.close();
}
[begin]
[SQL]    insert into orders (amount, order_no, id) values (?, ?, ?)
[bind]   [99.00, TX-1001, 1]
[commit]
操作时机说明
begin显式调用启动事务,获取连接
persist业务代码加入 PC,SQL 暂未发
flushcommit / 显式触发 SQL
commit显式调用JDBC COMMIT
rollback异常撤销
close显式 / try-with释放 EM

15.2 Spring @Transactional 调用链

图 15-2-1 · 声明式事务时序 — Proxy → Interceptor → TM → EM → DB
组件职责
TransactionInterceptor拦截 @Transactional 方法
JpaTransactionManagerJPA 事务管理
TransactionDefinition传播 / 隔离 / 超时配置
TransactionStatus当前事务状态
TransactionSynchronizationManagerThreadLocal 资源绑定
 
@Service
public class OrderService {
    @Autowired private OrderRepository repository;

    public void importOrders(List<Order> orders) {
        for (Order o : orders) {
            saveOne(o);  // ⚠ this. 调用,绕过代理!
        }
    }

    @Transactional
    public void saveOne(Order o) {
        repository.save(o);
    }
}

同类内部调用陷阱。saveOne 的 @Transactional 不生效——importOrders 直接以 this 调用 saveOne,绕过了 AOP 代理。修复:① 抽到独立 Service;② 注入自身代理(AopContext.currentProxy());③ 使用 TransactionTemplate 编程式。

15.3 传播行为

图 15-3-1 · 七种传播行为 — 决策树
传播外层存在行为适用
REQUIRED加入默认大多数业务
REQUIRES_NEW挂起 + 新建独立审计 / 日志
NESTEDSAVEPOINT外层 rollback 不影响部分回滚
MANDATORY必须有否则异常仅供子调用
NEVER必须无否则异常强制无事务
SUPPORTS加入 / 无事务灵活只读查询
NOT_SUPPORTED挂起 / 无事务显式跳过长任务
 
@Service
public class OrderService {
    @Transactional
    public void createOrder(Order order) {
        em.persist(order);
        auditService.logCreate(order.getId());
        throw new BusinessException("回滚但日志保留");
    }
}

@Service
public class AuditService {
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    public void logCreate(Long orderId) {
        em.persist(new AuditLog("CREATE", orderId));
    }
}
[执行]
1. createOrder 启动事务 1
2. logCreate 挂起事务 1, 启动事务 2
3. 事务 2 commit (审计日志保留)
4. createOrder 抛异常, 事务 1 rollback (订单未保存)

[结果]
- order 表: 无新增行
- audit_log 表: 有 CREATE 记录
[NESTED 示例]
SAVEPOINT SP_xxx;       -- nestedSave 开始
INSERT into orders ...; -- 失败 (唯一约束)
ROLLBACK TO SAVEPOINT SP_xxx;

SAVEPOINT SP_yyy;       -- 重试
INSERT into orders ...; -- 成功
RELEASE SAVEPOINT SP_yyy;
COMMIT;                 -- 外层 commit

15.4 隔离级别

级别脏读不可重复读幻读性能
READ_UNCOMMITTED可能可能可能最高
READ_COMMITTED否可能可能高
REPEATABLE_READ否否可能中
SERIALIZABLE否否否低
 
PostgreSQL:    READ_COMMITTED
MySQL InnoDB:  REPEATABLE_READ
Oracle:        READ_COMMITTED
SQL Server:    READ_COMMITTED
H2:            READ_COMMITTED
@Transactional(isolation = Isolation.SERIALIZABLE)
public void transferMoney(Long fromId, Long toId, BigDecimal amount) {
    Account from = em.find(Account.class, fromId);
    Account to = em.find(Account.class, toId);
    from.debit(amount);
    to.credit(amount);
}
场景推荐理由
多数业务读READ_COMMITTED平衡
报表统计REPEATABLE_READ同事务一致快照
资金 / 库存SERIALIZABLE 或乐观锁严格一致
高并发只读READ_COMMITTED最少锁
长事务批处理READ_COMMITTED + 乐观锁避免阻塞

图 15-5-1 · 回滚决策 — RuntimeException 自动,受检异常默认 commit

@Transactional(rollbackFor = {Exception.class})
public void process(Order order) throws BusinessException {
    em.persist(order);
    if (order.isInvalid()) {
        throw new BusinessException("invalid");
    }
}

@Transactional(rollbackFor = {SQLException.class},
               noRollbackFor = {OptionalException.class})
public void complex() throws SQLException, OptionalException {
    // ...
}
// ⚠ try-catch 吞异常的陷阱
@Transactional
public void importBatch(List<Order> orders) {
    for (Order o : orders) {
        try {
            em.persist(o);
        } catch (Exception ex) {
            log.error("import failed", ex);
        }
    }
}

15.7 编程式事务

@Autowired
private TransactionTemplate txTemplate;

public void importBatch(List<Order> orders) {
    for (Order order : orders) {
        try {
            txTemplate.executeWithoutResult(status -> {
                em.persist(order);
            });
        } catch (Exception ex) {
            log.error("import failed for {}", order.getOrderNo(), ex);
        }
    }
}
维度声明式编程式
语法@TransactionalTransactionTemplate
边界粒度方法任意代码块
同类调用受代理限制不受限
异常处理注解配置try-catch
适用大多数场景细粒度 / 循环内

15.8.1 设计取舍

取舍 1:AOP 代理而非字节码增强。Spring 选择 JDK 动态代理 / CGLIB 实现声明式事务,而非编译期字节码增强。代价是同类内部调用绕过代理。优势是配置灵活,无需修改字节码,可与 IoC 容器无缝集成。

取舍 2:RuntimeException 默认 rollback。JPA 抛出的多数异常(PersistenceException)是运行时异常,默认 rollback 符合业务期望。受检异常默认 commit 是为了兼容 EJB 规范,但实际多数场景须显式 rollbackFor=Exception.class。

取舍 3:PC 默认 transaction-scoped。PC 与事务同生命周期是默认(无状态会话 bean / Spring Service)。Extended PC 仅用于 Stateful Bean(JSF / 长会话),Spring 中较少使用。

15.9 陷阱与限制

#陷阱现象修复
1同类内部调用@Transactional 不生效抽到独立 Service
2受检异常 commit业务异常未回滚rollbackFor=Exception
3catch 吞异常rollback-only 状态REQUIRES_NEW
4OSIV + 事务外修改LIE / 无效修改关闭 OSIV
5NESTED 不工作JpaTM 不支持DataSourceTM
6REQUIRES_NEW 误用过多新连接仅审计场景
7长事务持有锁阻塞其他事务短事务
8默认隔离与数据库不符业务行为差异显式声明

@Test

    assertThat(exists).isFalse();

@Test
