第 14 章 · 批量处理与 JDBC Batching

10000 行从 30 秒到 2 秒 · IDENTITY 与 SEQUENCE · order_inserts · StatelessSession · Bulk vs Batch · 性能基准

批量处理是 JPA / Hibernate 性能优化的关键场景。10000 行数据导入,默认配置耗时 30 秒以上;经过 JDBC Batch 优化后可降至 2 秒。性能差异来源于:网络往返次数、PreparedStatement 复用、PersistenceContext 内存控制。

JDBC Batch 不是单一开关。hibernate.jdbc.batch_size、ID 生成策略、order_inserts / order_updates、PersistenceContext flush 节奏共同决定最终行为。本章覆盖 Batch 的源码机制、生效条件、监控方式与回归测试,并给出"先验证 batch 生效、再验证内存上限"的两段测试模板。

  • JDBC Batch 的核心约束:同 SQL 模板才能 addBatch,IDENTITY 立即返回 ID 阻止 batch
  • 批量优化检查清单 6 项:ID 策略 / batch_size / order_inserts / order_updates / flush+clear / 关 L2
  • SEQUENCE + allocationSize=50 的内存分配机制,与 batch_size 同步的对齐策略
  • 10000 行导入的内存曲线:有 clear 稳定 128MB,无 clear 飙到 1GB 触发 OOM
  • order_inserts=true / order_updates=true / batch_versioned_data=true 三个互相搭配的开关
  • StatelessSession 完全绕开 PC:无脏检查 / 无级联 / 无缓存,适合纯 ETL 通道
  • JPQL Bulk 与 JDBC Batch 的本质区别:1 条 UPDATE WHERE vs 多条 INSERT addBatch
  • Spring Data saveAll 的真相:仅是 forEach(save),无特殊优化
  • Hibernate Statistics 三个核心指标:prepareCount / batchCount / 平均 batch 大小
  • 核心源码链:ActionQueue → EntityPersister → JdbcCoordinator → BatchingBatch → PreparedStatement
  • 四类设计取舍:同 SQL 才 Batch / IDENTITY 不支持 / allocationSize 默认 50 / Stateless 绕开 PC
  • 1k / 10k / 100k / 1M 行的性能基准表 + Spring Batch / R2DBC 反应式两种扩展路径

前置知识:第 5 章 ID 生成、第 13 章缓存、JDBC PreparedStatement.addBatch/executeBatch API、Spring @Transactional 基础、堆内存监控基础。

14.0 章首速览

一句话总览。批量处理的核心瓶颈是每行一次 SQL 一次网络往返;Batch 把它压成"每 50 行一次往返"。前提是 ID 不要 IDENTITY(让 ID 内存可知),batch_size 与 allocationSize 对齐(避免 sequence 频繁取),并定期 flush() + clear()(避免 PC 爆内存)。

14.4 flush + clear 控制内存

@Transactional
public ImportResult importOrders(List<ImportOrder> rows) {
    int batchSize = 50;

    for (int i = 0; i < rows.size(); i++) {
        em.persist(rows.get(i));

        if ((i + 1) % batchSize == 0) {
            em.flush();
            em.clear();
        }
    }

    em.flush();
    em.clear();
    return new ImportResult(rows.size());
}
调用作用性能影响
em.flush()写入待保存变更触发 SQL,clear 前必需
em.clear()清空 PC释放内存,实体 detached
不调用 flush提交时一次 flush大批量内存爆
不调用 clearPC 持续累积脏检查变慢
 
[每 50 行 flush + clear]
flushed 50 rows    heap=128M
flushed 100 rows   heap=130M
flushed 150 rows   heap=128M
... (稳态)
flushed 10000 rows heap=132M

[无 clear]
flushed 5000 rows  heap=512M
flushed 10000 rows heap=1024M (OOM 风险)

14.6 StatelessSession

图 14-6-1 · Session vs StatelessSession — 功能 vs 性能
@Transactional
public void bulkImport(List<ImportOrder> rows) {
    Session session = em.unwrap(Session.class);
    StatelessSession stateless = session
        .getSessionFactory()
        .openStatelessSession();

    Transaction tx = stateless.beginTransaction();
    try {
        for (ImportOrder row : rows) {
            stateless.insert(row);
        }
        tx.commit();
    } finally {
        stateless.close();
    }
}

14.8 Spring Data JPA saveAll 真相

@Repository
public interface ImportOrderRepository extends JpaRepository<ImportOrder, Long> {
}

@Service
public class ImportService {

    @Autowired
    private ImportOrderRepository repository;

    @Transactional
    public void doImport(List<ImportOrder> rows) {
        repository.saveAll(rows);
    }
}
配置saveAll 行为
ID 策略 IDENTITY内部 forEach persist,无 batch
ID 策略 SEQUENCE + batch_size内部 forEach persist,可 batch
不调用 flush + clear大批量内存累积
@Transactional 包裹提交时一次 flush

saveAll 内部仅是 forEach(repository::save),无特殊优化。是否真正 batch 取决于 ID 策略与 hibernate.jdbc.batch_size 配置。需要内存控制时仍要显式 em.flush() + em.clear()。

14.10.1 设计取舍

取舍 1:同 SQL 模板才 Batch。JDBC Batch 限制——同一 PreparedStatement 才能 addBatch。Hibernate 按 BatchKey(SQL 模板)分组。order_inserts=true 重排实体顺序,使同表 INSERT 连续,提升 batch 利用率。

取舍 2:IDENTITY 不支持 Batch。JPA 规范要求 persist 返回前 ID 可用。IDENTITY 必须立即执行 INSERT 才能拿到 ID,与 Batch 延迟执行冲突。该取舍优先正确性而非性能。

取舍 3:SEQUENCE allocationSize 默认 50。在内存分配 50 个 ID,平衡了 sequence 取值次数与 ID 浪费风险。修改时须同步评估应用重启的 ID gap 与并发分配。

取舍 4:Stateless 完全绕开 PC。StatelessSession 不维护 PersistenceContext,所有特性(脏检查、级联、回调、缓存)失效。代价是不能用普通业务事务。设计意图是为纯 ETL 提供高性能通道。

14.12 回归护栏

@Test
void importShouldUseJdbcBatch() {
    Statistics stats = sf.getStatistics();
    stats.clear();

    importService.importOrders(generateRows(1000));

    long prepareCount = stats.getPrepareStatementCount();
    long entityInsertCount = stats.getEntityInsertCount();

    assertThat(entityInsertCount).isEqualTo(1000L);
    assertThat(prepareCount).isLessThan(50L);
}

@Test
void importShouldNotAccumulatePersistenceContext() {
    long heapBefore = Runtime.getRuntime().totalMemory()
                    - Runtime.getRuntime().freeMemory();

    importService.importOrders(generateRows(10000));

    long heapAfter = Runtime.getRuntime().totalMemory()
                   - Runtime.getRuntime().freeMemory();

    long delta = heapAfter - heapBefore;
    assertThat(delta).isLessThan(100L * 1024 * 1024);
}

@Configuration
public class OrderImportJobConfig {

    @Bean
    public Job importOrdersJob(JobRepository jobRepo, Step importStep) {
        return new JobBuilder("importOrders", jobRepo)
            .start(importStep)
            .build();
    }

    @Bean
    public Step importStep(JobRepository jobRepo,
                            PlatformTransactionManager txm,
                            ItemReader<ImportOrder> reader,
                            ItemWriter<Order> writer) {
        return new StepBuilder("importStep", jobRepo)
            .<ImportOrder, Order>chunk(100, txm)
            .reader(reader)
            .processor(o -> mapToEntity(o))
            .writer(writer)
            .build();
    }
}
@Bean
MeterBinder batchMetrics(EntityManagerFactory emf) {
    Statistics stats = emf.unwrap(SessionFactory.class).getStatistics();
    return registry -> {
        Gauge.builder("hibernate.batch.entity.inserts", stats,
                Statistics::getEntityInsertCount).register(registry);
        Gauge.builder("hibernate.batch.prepare.statements", stats,
                Statistics::getPrepareStatementCount).register(registry);
    };
}
