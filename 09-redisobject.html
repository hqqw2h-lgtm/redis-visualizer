第 9 章 · redisObject 对象封装

16 字节对象头 · 4bit type + 4bit encoding · 24bit lru/lfu · 引用计数与共享整数池 · 命令路径 type 检查 · 编码多态分派 · WRONGTYPE 防护

前 8 章讲的 SDS / adlist / dict / skiplist / intset / ziplist / listpack / quicklist / rax 都是底层"零件"。Redis 用户看到的 String / List / Hash / Set / ZSet / Stream 类型并不直接暴露这些零件 — 而是统一用一个 仅 16 字节 的对象头 redisObject(简称 robj)封装,五个字段串起类型系统、编码多态、引用计数与淘汰策略。

db.dict 哈希表的 value 类型是 robj *,所有命令实现拿到的都是 robj *。命令先看 type 字段做安全检查(避免 WRONGTYPE),再根据 encoding 字段把 ptr 解释为对应底层结构,最后调用对应模块的命令实现。这一套设计让"同一个用户类型"可以根据数据规模动态切换底层编码(如 String 在不同场景用 int / embstr / raw 三种编码),也让"同一种底层结构"被多个用户类型复用(如 listpack 同时被 Hash / ZSet / List 用)。

16 字节这个数字不是凑巧 — 它是 64 位平台对齐规则下"包含 1 个 8 字节指针 + 8 字节元数据"的最小自然边界。Redis 用 位字段(bit-field)把 type + encoding + lru 压进同一个 32 bit 字,再用 32 bit refcount + 64 bit ptr 凑齐 16 字节,正好两个 cache line 的 1/4,千万 key 场景下 robj 头总共也只占 160 MB,放得下日活几亿用户的 KV 元数据。

  • redisObject 5 字段 16 字节 字节布局:type 4b + encoding 4b + lru 24b + refcount 32b + ptr 64b
  • type(7 种用户类型)与 encoding(11 种底层编码)的多对多关系矩阵
  • String 三态 INT / EMBSTR / RAW 的三种内存布局(整数内嵌指针 / robj+SDS 同块 / 独立两块)
  • 引用计数语义 + 共享整数池(0–9999 预分配 + refcount=INT_MAX 哨兵)
  • 同一 24 bit 的 lru 字段在 LRU 模式(秒级时钟)与 LFU 模式(LDT 16b + counter 8b)下的双重语义
  • createObject / tryObjectEncoding / decrRefCount 的逐帧调用链
  • 命令前的 checkType WRONGTYPE 防护与 encoding 分派路径
  • 对象多态的 4 种候选方案对比:每类 struct / C++ 虚表 / union+tag / redisObject
  • 陷阱:APPEND 引发 EMBSTR→RAW、maxmemory 关闭共享池、LFU counter 上限 255

前置知识:第 1 章 SDS(理解 EMBSTR 与 RAW 的存储差异);第 3 章 dict(db.dict 是 robj 的容器);第 5–7 章 listpack / quicklist(理解多种 encoding 共用同一类型);第 8 章 rax / Stream(理解 OBJ_STREAM 编码)。读者需要熟悉位字段(C bit-field)与指针类型转换。

9.1 解决什么问题

Redis 是动态类型的内存数据库,需要在底层用 C 静态语言实现"对象多态"。每个 key 的 value 必须同时满足"统一类型系统 + 编码多态 + 内存极致紧凑 + 引用计数 + 淘汰元数据 + 命令分派统一"六个相互冲突的目标。这一节先列出业务需求拆解,再分析为什么"每类型独立 struct"、"C++ 虚表"、"union + tag"三种自然方案都不合适,最后落到 16 字节的硬约束。

9.1.3 为什么 16 字节而不是 24 / 32

64 位平台的对齐规则下,16 字节是"包含一个 8 字节指针 + 8 字节元数据"的最小自然边界。再小则需要内嵌 SDS 等容器,反而复杂。这 16 字节里已经塞下了:

  • 4 bit type — 标识用户类型(7 种,4 bit 还有 9 种富余)
  • 4 bit encoding — 标识底层编码(11 种,4 bit 还有 5 种富余)
  • 24 bit lru/lfu — 淘汰元数据(LRU 秒级时钟 / LFU LDT+counter)
  • 32 bit refcount — 引用计数(INT_MAX 表示永不释放)
  • 64 bit ptr — 指向底层数据(整数编码下直接存 long)

9.2 redisObject 字节布局

这一节把 16 字节字节序逐位拆开,讲清楚 5 个字段在内存里的物理排布,以及 type / encoding 的多对多关系。理解这个布局之后,后续每条命令"为什么先看 type 再看 encoding"就是显然的。

9.2.1 C 结构定义

/* src/server.h */
#define LRU_BITS 24

typedef struct redisObject {
    unsigned type:4;          /* OBJ_STRING / OBJ_LIST / OBJ_HASH / OBJ_SET /
                                 OBJ_ZSET / OBJ_STREAM / OBJ_MODULE */
    unsigned encoding:4;      /* OBJ_ENCODING_RAW / INT / HT / INTSET /
                                 SKIPLIST / EMBSTR / QUICKLIST / STREAM /
                                 LISTPACK 等 */
    unsigned lru:LRU_BITS;    /* LRU 时间戳 (秒级,LRU_BITS=24) 或
                                 LFU 数据 (高 16b LDT + 低 8b counter) */
    int refcount;             /* 引用计数,INT_MAX 表示共享对象 */
    void *ptr;                /* 指向底层数据,INT 编码下直接存 (void*)long */
} robj;

/* 头部 32 bit (一个 unsigned 字) 解读:
 *   bits 0-3   : type
 *   bits 4-7   : encoding
 *   bits 8-31  : lru (24 bit)
 * 紧跟 32 bit refcount + 64 bit ptr,合计 16 字节
 */
3 个位字段紧凑挤入 32 bit + 32 bit refcount + 64 bit ptr:头部 32 bit 一个字段,数据 32 bit 一个字段,指针 64 bit 一个字段,正好 16 字节自然对齐,不需要任何 padding。这是 Redis 极致内存优化的标志写法 — 一旦 type/encoding/lru 任一字段加 1 bit,整个结构就要扩到 24 字节,千万 key 场景内存开销直接 +50%。

9.4 引用计数 + 共享整数池

9.4.1 设计动机

引用计数解决"对象生命周期"问题:每个引用 +1,引用消失 -1,降到 0 时释放。Redis 在以下场景使用 refcount:

  1. 客户端命令:命令处理过程中可能多次引用同一对象,完成后 decrRefCount
  2. 共享整数池:0–9999 的整数对象在启动时预创建并共享,所有引用同一 robj
  3. AOF/RDB 序列化:持久化时短暂增加引用避免被释放

9.4.4 incrRefCount / decrRefCount 流程

图 9-4-2 · 引用计数加减的状态分支
/* src/object.c - 引用计数核心 */
void incrRefCount(robj *o) {
    if (o->refcount != OBJ_SHARED_REFCOUNT) o->refcount++;
}

void decrRefCount(robj *o) {
    if (o->refcount == 1) {
        switch(o->type) {
        case OBJ_STRING: freeStringObject(o); break;
        case OBJ_LIST:   freeListObject(o); break;
        case OBJ_SET:    freeSetObject(o); break;
        case OBJ_ZSET:   freeZsetObject(o); break;
        case OBJ_HASH:   freeHashObject(o); break;
        case OBJ_STREAM: freeStreamObject(o); break;
        case OBJ_MODULE: freeModuleObject(o); break;
        default: serverPanic("Unknown object type"); break;
        }
        zfree(o);
    } else {
        if (o->refcount <= 0) serverPanic("decrRefCount against refcount <= 0");
        if (o->refcount != OBJ_SHARED_REFCOUNT) o->refcount--;
    }
}

9.5 lru/lfu 字段:24 bit 双重用途

9.5.1 设计动机

lru:24 字段在 LRU 与 LFU 两种淘汰策略下有完全不同的语义:

  • LRU 模式:存对象的最后访问时间(秒级时钟,24 bit 周期 ≈ 194 天)
  • LFU 模式:高 16 bit 存最后访问时间(分钟级),低 8 bit 存"对数访问频率计数器"

LRU / LFU 互斥(由 maxmemory-policy 决定),所以同一字段可以双重用途。这是 24 bit 字段的精妙之处 — 不增加任何内存开销支持两种淘汰策略。

/* src/object.c - 各 type 的释放函数选择性展示 */
void freeStringObject(robj *o) {
    if (o->encoding == OBJ_ENCODING_RAW) {
        sdsfree(o->ptr);   /* RAW: SDS 独立块,需单独释放 */
    }
    /* INT 与 EMBSTR 都在 robj 同块,zfree(o) 即可 */
}

void freeListObject(robj *o) {
    switch (o->encoding) {
    case OBJ_ENCODING_QUICKLIST:
        quicklistRelease(o->ptr);
        break;
    case OBJ_ENCODING_LISTPACK:
        lpFree(o->ptr);
        break;
    default:
        serverPanic("Unknown list encoding type");
    }
}

void freeHashObject(robj *o) {
    switch (o->encoding) {
    case OBJ_ENCODING_HT:
        dictRelease((dict *)o->ptr);
        break;
    case OBJ_ENCODING_LISTPACK:
        lpFree(o->ptr);
        break;
    default:
        serverPanic("Unknown hash encoding type");
    }
}

C++ 虚表 + 继承+ 8 B vptr间接调用子类切换复杂C 不支持;vptr 千万 key 时浪费 80 MB
union + tagtag + 最大成员switch切换 union中联合体最大成员浪费;不能内嵌动态结构
redisObject16 Btype 4 bencoding 4 b中本场景最优

9.8.2 Redis 选择 redisObject 的 5 个核心原因

redisObject 设计要点:
① C 兼容:不依赖 C++ 特性,实现可移植;
② 位域紧凑:type / encoding / lru 共占 32 bit,连同 refcount 32 bit + ptr 64 bit 正好 16 字节;
③ 统一 ABI:所有命令实现拿到的都是 robj *,无需为不同类型写不同接口;
④ 运行时多态:encoding 字段支持同类型多编码,可根据数据规模动态切换;
⑤ 共享对象池:refcount = INT_MAX 表示"永不释放",节省内存。

9.8.3 三个关键设计取舍

取舍 1:为什么不每对象独立分配 type 字段而用位域

千万 key 场景下,每对象省 1–3 字节 × 1000 万 = 10–30 MB。位域虽然访问稍慢(读位域有移位 + 掩码),但与底层数据结构操作开销相比可忽略。CPU 端对 32 bit 字内的位字段读取几乎只需一个移位 + 掩码指令,在 Redis 单命令几十 μs 的尺度上完全淹没。

取舍 2:为什么 INT 编码直接把整数存 ptr 而不是单独分配

省 8 字节 SDS 头 + 整数字节,等于省 1 次 malloc。ptr = (void*)100 完全合法(指针视作 long 整数),命令实现处用 (long)o->ptr 取回。这是 Redis 内存优化的"小技巧":在 64 位平台上,void* 与 long 都是 8 字节,可以直接互相转换。

取舍 3:为什么共享池只 0–9999

10000 个对象 × 16 B = 160 KB,启动时全部分配。覆盖了大多数计数器 / 状态码场景。再扩大不仅占内存,还需要更长的查找时间(共享池本身是数组,O(1) 查),且超过这个范围的整数访问频率指数下降。9999 这个上限是 Antirez 在抽样真实业务流量后选定的,90% 以上的整数 key 都落在 0–9999 区间。

9.10.2 createEmbeddedStringObject 关键代码

/* src/object.c - 同块分配 robj + sdshdr8 + buf */
robj *createEmbeddedStringObject(const char *ptr, size_t len) {
    /* 1. 一次 malloc 拿到 robj + sdshdr8 + buf + '\0' */
    robj *o = zmalloc(sizeof(robj) + sizeof(struct sdshdr8) + len + 1);
    struct sdshdr8 *sh = (void *)(o + 1);   /* sds 紧跟 robj */

    /* 2. 初始化 robj 头 */
    o->type = OBJ_STRING;
    o->encoding = OBJ_ENCODING_EMBSTR;
    o->ptr = sh + 1;                         /* ptr 指向 sds buf */
    o->refcount = 1;

    /* 3. 初始化 LRU/LFU 字段 */
    if (server.maxmemory_policy & MAXMEMORY_FLAG_LFU) {
        o->lru = (LFUGetTimeInMinutes() << 8) | LFU_INIT_VAL;
    } else {
        o->lru = LRU_CLOCK();
    }

    /* 4. 初始化 sdshdr8 */
    sh->len = len;
    sh->alloc = len;
    sh->flags = SDS_TYPE_8;

    /* 5. 拷贝字符串数据 */
    if (ptr == SDS_NOINIT)
        sh->buf[len] = '\0';
    else if (ptr) {
        memcpy(sh->buf, ptr, len);
        sh->buf[len] = '\0';
    } else {
        memset(sh->buf, 0, len + 1);
    }
    return o;
}

/* ★ 关键不变量:
 *   1. robj、sdshdr8、buf 在同一个 malloc 块内
 *   2. o->ptr 指向 sh->buf,客户端访问 sds 时与 RAW 编码无差异
 *   3. 释放时只需 zfree(o),不需要 sdsfree
 *   4. 修改时必须先转 RAW (因为 EMBSTR 设计为只读)
 */

9.10.3 tryObjectEncoding 关键代码

/* src/object.c - 尝试把 String 升降为最紧凑编码 */
robj *tryObjectEncoding(robj *o) {
    long value;
    sds s = o->ptr;
    size_t len;

    /* 1. 只对 STRING 类型尝试编码 */
    serverAssertWithInfo(NULL, o, o->type == OBJ_STRING);
    if (!sdsEncodedObject(o)) return o;
    if (o->refcount > 1) return o;

    len = sdslen(s);
    /* 2. 短字符串 (len <= 20) 尝试解析为 long */
    if (len <= 20 && string2l(s, len, &value)) {
        /* 3. 在共享池范围内,且没启用 maxmemory,直接复用 */
        if ((server.maxmemory == 0 ||
             !(server.maxmemory_policy & MAXMEMORY_FLAG_NO_SHARED_INTEGERS)) &&
            value >= 0 &&
            value < OBJ_SHARED_INTEGERS)
        {
            decrRefCount(o);
            incrRefCount(shared.integers[value]);
            return shared.integers[value];
        } else {
            /* 4. 超出共享池但仍是整数 → 转 INT 编码 */
            if (o->encoding == OBJ_ENCODING_RAW) {
                sdsfree(o->ptr);
                o->encoding = OBJ_ENCODING_INT;
                o->ptr = (void *) value;
                return o;
            } else if (o->encoding == OBJ_ENCODING_EMBSTR) {
                decrRefCount(o);
                return createStringObjectFromLongLongForValue(value);
            }
        }
    }

    /* 5. 短字符串但不是数字 → 尝试转 EMBSTR (节省内存) */
    if (len <= OBJ_ENCODING_EMBSTR_SIZE_LIMIT) {
        robj *emb;
        if (o->encoding == OBJ_ENCODING_EMBSTR) return o;
        emb = createEmbeddedStringObject(s, sdslen(s));
        decrRefCount(o);
        return emb;
    }

    /* 6. 都不满足,保持 RAW (并尝试 sdstrim 释放冗余 SDS 头容量) */
    trimStringObjectIfNeeded(o, 0);
    return o;
}

/* ★ 关键事实:
 *   1. SET 命令在 setCommand 里不会主动调 tryObjectEncoding
 *      → 读路径 / argv 处理时统一调用,把客户端给的字符串变最紧凑编码
 *   2. 一旦改成 INT 编码,后续 GET 时用 sdsfromlonglong 临时转 SDS
 *   3. EMBSTR 与 RAW 的边界 OBJ_ENCODING_EMBSTR_SIZE_LIMIT = 44 (jemalloc 64B class)
 */

9.10.4 设计取舍

取舍 1:type 4 bit 限制为 16 种类型

Redis 当前用了 7 种(STRING / LIST / HASH / SET / ZSET / STREAM / MODULE),还有 9 种富余。即使将来加新类型(如新版 Stream 或 SortedSet),16 种也充裕。

取舍 2:encoding 4 bit 限制为 16 种编码

当前已用 11 种(去掉弃用的 zipmap / linkedlist / ziplist 还有 7–8 种活跃),还有富余。新底层结构可继续添加。

取舍 3:lru 24 bit 在多场景复用

LRU 时钟用 24 bit 秒级,周期 194 天,够用;LFU 用同 24 bit 编码 LDT 16 b + counter 8 b,设计紧凑。代价是 LRU / LFU 不能共存(必须二选一),但实际运维不会同时需要。

取舍 4:为什么 INT 编码与 EMBSTR / RAW 都属于 STRING 类型

表面看 INT 编码下 ptr 是整数,与 EMBSTR / RAW 的 SDS 指针完全不同。但从用户视角,SET k 100 与 SET k "hello" 都是 String 类型。Redis 让命令处理函数在拿到 robj 之后,根据 encoding 字段决定把 ptr 当整数还是当 SDS,客户端完全无感知。这种"用户类型与底层编码解耦"是 Redis 内存优化能持续演进的根本 — Redis 7.0 把 ziplist 全部换成 listpack,也只动了 encoding 这一层,客户端看到的 type 完全没变。

9.12 真实用例

用例 1:HTTP 状态码计数器(共享整数池)(亿级 PV 网关)
某 CDN 网关用 Redis 计数 HTTP 200 / 404 / 500 等状态码:SET status:200 12345678 / INCR status:200。关键点:大多数请求是 200,占 90%+;404 / 500 占比稳定;计数器值常常落在 0–9999 区间(按分钟粒度统计)。实际收益:每个状态码 1M key(按 minute 拆分),总占用预期 1M × 80 B = 80 MB,但因为整数 0–9999 被共享池命中,实际只占 1M × 40 B(dictEntry + key SDS) = 40 MB,省一半。注意:运维曾经把 maxmemory 设成 8GB,共享池失效,内存反而涨到 95 MB — 后来根据实际数据量调到 24GB 后保持 maxmemory=0 才恢复。
用例 2:大字符串 token 缓存(EMBSTR vs RAW 边界)(SaaS API 网关)
网关把 JWT token 缓存进 Redis 做去重 / 黑名单:SET token:abc123 ""。初始版本:JWT 平均 60 字节(带 header + payload + sig),全走 RAW 编码,1M token 占 ~ 96 MB。优化版本:改用 token hash(MD5 16 字节作 value)而不是完整 JWT,OBJECT ENCODING 返回 embstr,内存降到 ~ 40 MB(节省 58%)。权衡:用 hash 而不是完整 JWT 牺牲了"日志可追溯"(看到 hash 不知道是哪个 token),但在去重场景下 hash 已经够用。
用例 3:LFU 抗缓存击穿(秒杀活动)
电商秒杀活动期间,商品详情 key 访问从平时 100 QPS 飙到 10 万 QPS。问题:LRU 策略下,秒杀流量瞬间把"老商品详情"挤出缓存,秒杀结束后查"老商品"反而全部回源 DB,DB 压力暴涨。解决:切换 maxmemory-policy=allkeys-lfu,LFU 计数器对突发流量有对数衰减,秒杀流量虽然多但 counter 增长有限,"老商品"高频访问的累积 counter 仍然较高,不会被挤走。实际:秒杀期间 DB 压力下降 60%,缓存命中率从 84% 升到 96%。调优:lfu_log_factor 默认 10,该业务调到 8(增长更快,适合短时差异)。
用例 4:不要这么用 — 把 List 当大队列(失败案例复盘)
反例:某团队用 List 做异步任务队列(RPUSH tasks ""),JSON 平均 200 字节。初期:几千任务,List 是 quicklist + listpack,内存正常。暴涨:某天积压 100 万任务,quicklist 节点剧增到几千个,每节点 listpack ~ 4KB,总内存 4 GB。问题进一步恶化:消费端处理失败,任务一直没消费完,quicklist 内存只增不减(因为 LISTPACK→QUICKLIST 不可逆,即使消费完也保持 quicklist)。正解:改用 Stream + 消费组(第 8 章),Stream 自带 MAXLEN 控长 + PEL 流转,消费失败的消息自动重投。Stream + 100 万消息 ≈ 80 MB(50 倍内存节省)。
