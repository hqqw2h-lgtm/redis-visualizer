第 20 章 · I/O 多路复用

C10K · select / poll / epoll / kqueue · LT vs ET · ae 抽象层 · Reactor 模型

Redis 主线程以单进程单线程方式处理所有命令,其性能上限的关键约束有二:其一,命令执行路径必须保持串行,以维持数据结构的强一致语义;其二,事件循环对成千上万个客户端连接的等待成本必须接近常数。第二项约束的工程解决方案即为 I/O 多路复用——由内核维护"哪些 fd 已就绪",事件循环只处理就绪 fd,空闲连接不消耗主线程时间。

主流操作系统提供的多路复用 API 包括 select / poll / epoll / kqueue / evport 等。Redis 通过 ae.c 抽象层在编译期选择平台最优后端,上层 networking.c 只依赖统一接口。该抽象使得上层代码无需感知平台差异,但运维必须通过 INFO server 的 multiplexing_api 字段确认实际后端。

  • C10K 问题的本质:三种连接处理模型(线程/轮询/多路复用)的成本对比
  • select 的三大约束:FD_SETSIZE、全量拷贝、O(N) 线性扫描
  • poll 解决了什么、未解决什么——为什么 Redis 主线 ae 不使用 poll
  • epoll 内核数据结构:长期兴趣集合 + 就绪队列 → 等待成本接近 O(ready)
  • 水平触发(LT)与边缘触发(ET)的语义差异和 Redis 选择 LT 的工程依据
  • kqueue 的 filter 抽象与 Solaris evport 的差异
  • Reactor vs Proactor:为何 Redis ae 主线仍采用就绪通知模型
  • ae.c 编译期后端选择优先级:evport → epoll → kqueue → select 兜底
  • fd 容量诊断:ulimit、maxclients + 32、fs.file-max、somaxconn
  • 性能边界:多路复用何时是瓶颈,何时不是
  • 从 accept 到命令执行的完整调用链(ae + networking + server.c)

前置知识:第 19 章 ae 事件循环(知道有 aeProcessEvents / aeApiPoll);第 21 章 RESP(知道事件回调内做了什么);Linux/POSIX 文件描述符与 socket 编程基础。

20.0 章首速览

图 20-0-1 · I/O 多路复用全景

本章的逻辑主线:select 与 poll 是全量扫描模型,成本与总连接数线性相关;epoll 与 kqueue 通过内核维护兴趣集合,成本与就绪连接数线性相关。Redis 通过 ae.c 屏蔽平台差异,上层无需感知后端。

20.1 C10K 问题的本质

图 20-1-1 · 三种处理 N 个连接的模型对比

C10K 问题的核心命题是:在单机 10,000 连接规模下,等待 I/O 的开销必须低于命令执行开销。模型 A 的线程开销随连接数线性增长,模型 B 在大量空闲连接下产生持续轮询开销,只有模型 C 将"等待"卸载给内核。

选择多路复用的真实原因: 不只是性能,更是命令串行化的简化。所有命令在主线程顺序执行,数据结构无需考虑并发修改的语义复杂性——这一点甚至比性能本身更关键。

20.2 select:第一代多路复用 API

图 20-2-1 · select 的工作模型 — 4 个 O(N) 步骤

select 的开销结构:用户态准备 bitmap O(N) + 内核拷贝 bitmap O(N) + 内核扫描 bitmap O(N) + 用户态扫描结果 O(N)。即使只有少量 fd 就绪,所有四个 O(N) 步骤仍需执行。

图 20-2-2 · select 的三项硬约束
约束 1: FD_SETSIZE 上限
  • fd_set 是固定大小 bitmap, FD_SETSIZE 默认 1024
  • 需要扩大时必须重编译 libc 或应用
  • 连接数超过 1024 时 select 直接不可用
约束 2: 全量数据传递
  • 每次调用都需要传递完整的 fd_set
  • fd 集合稳定时, 内核必须重复扫描相同集合
  • 事件循环每秒数千次调用, 拷贝开销显著
约束 3: 线性扫描
  • 内核遍历整个 [0, max_fd] 检查就绪状态
  • 就绪 fd 占比越低, 无效检查越多
  • N=10000, ready=3 时, 9997 次检查无意义
fd_set rfds;
int maxfd = 0;

FD_ZERO(&rfds);
FD_SET(listenfd, &rfds);
if (listenfd > maxfd) maxfd = listenfd;

for (int i = 0; i < n_clients; i++) {
    FD_SET(clients[i].fd, &rfds);
    if (clients[i].fd > maxfd) maxfd = clients[i].fd;
}

struct timeval timeout = {1, 0};
int n = select(maxfd + 1, &rfds, NULL, NULL, &timeout);

for (int fd = 0; fd <= maxfd; fd++) {
    if (FD_ISSET(fd, &rfds)) {
        handle_ready(fd);
    }
}
表 20-2-1 · N=10000, ready=3 时 select 成本剖析
操作工作量
FD_SET 调用 N 次O(N)
内核拷贝 bitmapO(N) 字节
内核扫描 bitmapO(N) 比特
内核拷贝结果回用户O(N) 字节
FD_ISSET 检查 N 次O(N)
实际处理 ready 数O(3)
无效操作占比≈ (N - ready) / N ≈ 99.97%

20.3 poll:取消 fd 上限

struct pollfd fds[N];
fds[0].fd = listenfd;
fds[0].events = POLLIN;

for (int i = 0; i < n_clients; i++) {
    fds[i+1].fd = clients[i].fd;
    fds[i+1].events = POLLIN | POLLOUT;
}

int n = poll(fds, n_clients + 1, 1000);

for (int i = 0; i <= n_clients; i++) {
    if (fds[i].revents & POLLIN)  handle_read(fds[i].fd);
    if (fds[i].revents & POLLOUT) handle_write(fds[i].fd);
}
✅ 改进
  • 取消 FD_SETSIZE 上限,fd 数量仅受系统资源限制
  • events 与 revents 字段分离,请求与结果不共用同一 bitmap
  • 事件类型更丰富(POLLPRI / POLLRDHUP / POLLERR)
❌ 未解决
  • 仍需传递完整 pollfd 数组,O(N) 用户/内核拷贝
  • 内核仍需线性扫描每个 pollfd,O(N) 检查
  • 用户仍需遍历整个数组找就绪项,O(N) 用户扫描
  • 与活跃连接数无关的成本仍然存在
表 20-3-1 · select / poll / epoll 算法复杂度对比
API注册 fd 复杂度等待复杂度用户处理复杂度总复杂度
selectO(1) (FD_SET)O(N) 内核扫描 + O(N) 拷贝O(N) FD_ISSETO(N)
pollO(1) (设置 pollfd)O(N) 内核扫描 + O(N) 拷贝O(N) 遍历O(N)
epollO(log N)接近 O(ready)O(ready)主要随 ready 增长

Redis 主线 ae 后端不使用 poll。poll 在概念上是 select 与 epoll 之间的过渡,实际部署中要么使用 select(兜底,极少数平台),要么使用 epoll / kqueue / evport(主流生产平台)。

20.4 epoll:按就绪 fd 返回的通知机制

图 20-4-1 · epoll 的兴趣集合与就绪队列

epoll 将 select 的"每次全量传 fd"改为"长期维护兴趣集合"。fd 注册一次后驻留内核兴趣集合,等待时仅返回已就绪的 fd 子集。

int epfd = epoll_create1(0);

struct epoll_event ev;
ev.events = EPOLLIN;
ev.data.fd = clientfd;
epoll_ctl(epfd, EPOLL_CTL_ADD, clientfd, &ev);

struct epoll_event events[1024];
int n = epoll_wait(epfd, events, 1024, timeout_ms);

for (int i = 0; i < n; i++) {
    int fd = events[i].data.fd;
    if (events[i].events & EPOLLIN)  handle_read(fd);
    if (events[i].events & EPOLLOUT) handle_write(fd);
}
表 20-4-1 · N=10000, ready=3 时 epoll 成本
操作工作量
epoll_ctl ADD(建立连接时)O(log N)
epoll_wait 返回(连接稳定后)主要随 ready 数量增长
遍历就绪事件O(3)
事件循环成本连接稳定后主要与活跃 fd 数相关
图 20-4-2 · epoll_wait 的内核路径 — 主动通知模型
epoll vs select 的根本区别: epoll_wait 的睡眠/唤醒由内核协议栈直接驱动,不需要轮询。这是 epoll 与 select/poll 在系统层面最根本的差异:select/poll 是"应用主动检查",epoll 是"内核主动通知"。

20.5 LT 与 ET 触发模式

图 20-5-1 · 水平触发与边缘触发的语义

20.5.1 反例 → 修复对照(ET 模式)

❌ 错误:仅读一次就退出
if (events[i].events & EPOLLIN) {
    int n = read(fd, buf, sizeof(buf));
    process(buf, n);
}
/* ET 下:
 * 数据 4096 → 应用读 1024
 * 剩余 3072 可能长期得不到新通知
 * 连接进入假死状态 */
✅ 正确:循环读至 EAGAIN
while (1) {
    int n = read(fd, buf, sizeof(buf));
    if (n > 0)       process(buf, n);
    else if (n == 0) { close(fd); break; }
    else if (errno == EAGAIN ||
             errno == EWOULDBLOCK) break;
    else             { close(fd); break; }
}
/* ET 必须 drain 至 EAGAIN */
图 20-5-2 · ET 错误处理的失败时间线

场景: socket 接收缓冲 4096 字节, 应用 buf 1024 字节

时刻事件内核 rcv_buf就绪队列应用状态
t=0数据到达4096{fd}epoll_wait 返回
t=1触发 ET 通知4096(已弹出)处理回调
t=2read 1024 字节3072空仅处理 1024
t=3下次 epoll_wait3072空(无新边缘)永久阻塞
剩余 3072 字节 → 可能长期得不到新通知
连接进入假死状态, 客户端可能一直等待响应,直到后续边缘事件或 keepalive 超时才被检测。 修复: ET 模式下必须 while(read() > 0) drain 至 EAGAIN。
表 20-5-1 · Redis 选择 LT 的依据
维度LTET
通知次数较多较少
实现复杂度简单复杂(必须 drain 至 EAGAIN)
漏处理后果下次循环再通知,数据不丢fd 假死,需断开重连
Redis 命令路径单次 read 一批,下次循环继续必须循环 read 直至 EAGAIN
Redis 选择默认不采用
选 LT 的根本原因: 命令处理的可中断性。命令执行串行,单次 readQueryFromClient 处理一批数据后退出事件回调,下次循环继续处理剩余数据。LT 自然支持该模型;ET 则需要在事件回调内 drain 全部数据,违背了"事件循环让出主线程"的基本原则。

表 20-6-1 · 四种 Reactor / Proactor 模型对比

Reactor 与 Proactor 的本质差异:Reactor 通知"可以做了",应用自己执行 I/O;Proactor 通知"已经做完了",I/O 由内核完成,应用读取结果。Redis ae 是 Reactor 模型,与 epoll/kqueue/evport 天然契合。

int kq = kqueue();

struct kevent change;
EV_SET(&change, fd, EVFILT_READ, EV_ADD | EV_ENABLE, 0, 0, NULL);

struct kevent events[64];
struct timespec ts = {1, 0};
int n = kevent(kq, &change, 1, events, 64, &ts);

for (int i = 0; i < n; i++) {
    if (events[i].filter == EVFILT_READ) {
        handle_read(events[i].ident);
    }
}

epoll 仅处理 fd 就绪;kqueue 通过 filter 抽象统一处理多类事件。

图 20-6-1 · 为什么 Redis ae 主线仍选择 Reactor
Redis 不以 Proactor 模型为 ae 主线,并不是因为 Proactor 天然无法保持命令串行;只要把完成事件重新派发回主线程,语义仍可成立。真正的取舍在于复杂度、平台覆盖和收益:Redis 当前瓶颈更多在命令执行、网络栈和输出缓冲,Reactor + 可选 I/O 线程已经能以较小改动缓解网络读写成本。

/* src/ae_epoll.c (简化) */
typedef struct aeApiState {
    int epfd;
    struct epoll_event *events;
} aeApiState;

static int aeApiAddEvent(aeEventLoop *eventLoop, int fd, int mask) {
    aeApiState *state = eventLoop->apidata;
    struct epoll_event ee = {0};

    int op = eventLoop->events[fd].mask == AE_NONE ?
             EPOLL_CTL_ADD : EPOLL_CTL_MOD;

    ee.events = 0;
    int merged = mask | eventLoop->events[fd].mask;
    if (merged & AE_READABLE) ee.events |= EPOLLIN;
    if (merged & AE_WRITABLE) ee.events |= EPOLLOUT;
    ee.data.fd = fd;

    if (epoll_ctl(state->epfd, op, fd, &ee) == -1) return -1;
    return 0;
}

static int aeApiPoll(aeEventLoop *eventLoop, struct timeval *tvp) {
    aeApiState *state = eventLoop->apidata;
    int retval, numevents = 0;

    retval = epoll_wait(state->epfd, state->events, eventLoop->setsize,
        tvp ? (tvp->tv_sec*1000 + (tvp->tv_usec + 999)/1000) : -1);
    if (retval > 0) {
        numevents = retval;
        for (int j = 0; j < numevents; j++) {
            int mask = 0;
            struct epoll_event *e = state->events + j;
            if (e->events & EPOLLIN)  mask |= AE_READABLE;
            if (e->events & EPOLLOUT) mask |= AE_WRITABLE;
            if (e->events & EPOLLERR) mask |= AE_WRITABLE | AE_READABLE;
            if (e->events & EPOLLHUP) mask |= AE_WRITABLE | AE_READABLE;
            eventLoop->fired[j].fd = e->data.fd;
            eventLoop->fired[j].mask = mask;
        }
    }
    return numevents;
}

aeApiPoll 将平台原生事件结构(epoll_event)转换为 ae 统一的 aeFiredEvent 结构,屏蔽了平台差异。AE_READABLE / AE_WRITABLE 是上层使用的统一掩码,各后端按各自语义映射。

20.8 fd 容量与诊断

图 20-8-1 · 连接数的四层约束

Redis 启动时执行 adjustOpenFilesLimit(),自动尝试将进程 fd 软上限调整到 maxclients + 32(预留 AOF / RDB / cluster bus / 监听 socket / pidfile / 日志)。若调整失败,Redis 会降低 maxclients 并打印警告。

20.9 性能边界与基线

表 20-9-1 · 多路复用的性能贡献基线

epoll 的优势在"高总连接数 + 低活跃比例"场景最显著。所有连接都活跃的极端场景下,epoll 与 select 的差异较小,因为 select 需要扫描的 fd 与 epoll 需要返回的就绪 fd 数量相同。

/* src/server.c: aeMain 主循环 */
void aeMain(aeEventLoop *eventLoop) {
    eventLoop->stop = 0;
    while (!eventLoop->stop) {
        aeProcessEvents(eventLoop, AE_ALL_EVENTS|
                                    AE_CALL_BEFORE_SLEEP|
                                    AE_CALL_AFTER_SLEEP);
    }
}

/* src/ae.c: aeProcessEvents 单次迭代 */
int aeProcessEvents(aeEventLoop *eventLoop, int flags) {
    int processed = 0, numevents;

    if (eventLoop->beforesleep != NULL && flags & AE_CALL_BEFORE_SLEEP)
        eventLoop->beforesleep(eventLoop);

    numevents = aeApiPoll(eventLoop, tvp);

    if (eventLoop->aftersleep != NULL && flags & AE_CALL_AFTER_SLEEP)
        eventLoop->aftersleep(eventLoop);

    for (int j = 0; j < numevents; j++) {
        aeFileEvent *fe = &eventLoop->events[eventLoop->fired[j].fd];
        int mask = eventLoop->fired[j].mask;
        int fd = eventLoop->fired[j].fd;

        if (fe->mask & mask & AE_READABLE) {
            fe->rfileProc(eventLoop, fd, fe->clientData, mask);
        }
        if (fe->mask & mask & AE_WRITABLE) {
            fe->wfileProc(eventLoop, fd, fe->clientData, mask);
        }
        processed++;
    }

    return processed;
}

20.10.1 设计取舍

抽象层的代价与收益: Redis 选择 ae 抽象层的根本原因是平台中立性。Linux 的 epoll、BSD/macOS 的 kqueue、Solaris 的 evport 在 API 形态、事件语义、错误处理上各有差异,但本质都是 Reactor 模型。ae.c 提供统一的 aeCreateFileEvent / aeApiPoll 抽象,上层 networking.c 与 server.c 完全不感知后端,代价是抽象层引入了一次函数调用开销(aeApiPoll 内部转换事件结构)。该开销相对系统调用与命令执行可忽略。

LT 优于 ET 的选择依据是命令处理的可中断性。命令执行需要让出主线程以处理时间事件、AOF flush、cluster bus 等其他任务,事件回调内不应 drain 全部数据。LT 模式下 readQueryFromClient 处理一批数据后退出,下次循环继续,自然契合事件循环的轮转模型。ET 强制 drain 至 EAGAIN,违背该模型。

I/O 多线程(Redis 6+)将网络 I/O 卸载到辅助线程,但命令执行仍保持单线程串行,这一选择维持了数据结构无并发安全保证的约束。Proactor 模型(IOCP / io_uring)若要用于 Redis,也必须把完成事件重新排回主线程;这会带来较大的 ae 改造、平台差异和调度复杂度,因此当前主线仍采用 Reactor。
