第 11 章 · HyperLogLog 基数估算

12 KiB 固定内存 · 0.81% 标准误差 · 零-run 启发式 · 调和平均 · 稀疏 / 稠密双存储 · LinearCounting 修正 · 可分布式合并不累加误差

精确统计 N 个元素的去重基数(distinct count)需要 O(N) 量级的内存:SET 要把每个元素本体存起来才能判断"出现过没"。HyperLogLog(以下简称 HLL)用固定 12 KiB对任意大小的数据集给出基数估算,标准误差稳定在 0.81%。算法核心来自 Flajolet 等人 2007 年的论文 HyperLogLog: the analysis of a near-optimal cardinality estimation algorithm,Redis 在 PFADD/PFCOUNT/PFMERGE 三条命令中将其工程化。

HLL 的"魔法"看起来违反直觉:用 12 KiB 估出 1 亿 UV,误差只有 ±81 万。秘密是 "hash 函数把元素映射到均匀 bit 串 → 观察前导零最大值 → 用最大值反推 N"。零-run 启发式给出粗估,分桶 + 调和平均把方差压下来,LinearCounting 修正小基数区,稀疏编码把低基数下的 12 KiB 内存压到几百字节。本章按这条逻辑链一节一节展开,每个公式都给出可在 Redis 源码 src/hyperloglog.c 中对应的代码片段。

HLL 不是万能锤子。它不能反查元素是否存在(无 PFEXISTS),不能枚举元素(无 PFRANGE),不能删除单个元素(无 PFDEL),不能精确计数。需要任何一项请用 SET / BITMAP / Bloom Filter。HLL 的甜点场景是 "只问 N 是多少,不问 N 由谁组成":网站日活 UV、广告独立曝光数、独立访问 IP 数、唯一搜索词数 ── 这些场景下,12 KiB vs Set 的 4-6 GiB 是百万倍的内存差距。

  • 零-run 启发式的概率公式 P(ρ ≥ k) = 2-k 与 N ≈ 2ρ_max 反推
  • 为什么单寄存器估算方差大,要用 m=16384 个寄存器分桶 + 调和平均
  • 调和平均 H = m / Σ 2-R[i] 抑制零-run 长尾的数学直觉
  • Redis HLL 稠密结构 12304 字节字节级布局(magic+encoding+cached+16384×6bit)
  • 稀疏存储的 ZERO / XZERO / VAL 三种 opcode 与游程编码
  • 稀疏 → 稠密单向升级流程,触发条件:ρ>32 或长度超 hll-sparse-max-bytes
  • PFADD / PFCOUNT / PFMERGE 完整调用链与 cached_cardinality 缓存机制
  • LinearCounting 修正小基数偏差,触发阈值 linear < 2.5 × m
  • PFMERGE 寄存器逐位取最大,合并不引入额外误差,可分布式预聚合
  • HLL 的不可逆 / 不可枚举本质:寄存器只保留 hash 特征位
  • 典型陷阱:小集合用 HLL 反而费内存、稀疏→稠密内存突变、错把 HLL 当 Set

前置知识:第 6 章 String / SDS(HLL 在底层就是 String,OBJECT ENCODING 返回 raw);第 10 章 Bitmap(对比"按位独立"vs"按桶聚合"两类紧凑结构);基础概率:几何分布、调和平均、最大似然估计。读者需要熟悉 hash 函数(MurmurHash 即可)与位操作(leading zeros count)。

11.1 零-run 启发式

HLL 的全部 "魔法" 都建立在一个关于独立均匀 bit 串的概率事实上。把每个元素 x 通过一个高质量 hash 函数(Redis 用 MurmurHash64A,seed = 0xadc83b19ULL)映射成 64 位均匀分布的二进制串。定义 ρ(h(x)) = h(x) 二进制表示中前导零个数 + 1(即首个 1 出现的位置,1-indexed)。ρ 服从几何分布:

核心概率事实:P(ρ ≥ k) = 2-k ⇔ P(ρ = k) = 2-k
也就是说:观察到一个 hash 值前导零至少 10 个,概率约 1/1024。在 N 个独立 hash 中,出现至少一个 ρ ≥ 10 的概率约 1 - (1 - 2-10)N。当 N ≈ 2k 时,该概率接近 1 — 因此 "观察到的最大 ρ 反推 N"。

11.1.1 零-run 与基数的因果链

图 11-1-1 · 零-run 启发式的逻辑流

直观解释:扔 N 次硬币,记录"连续正面的最长游程"。游程长度 k 与 N 之间近似 k ≈ log₂ N。同理,在 N 个均匀 hash 中,最大前导零数与 log₂ N 成正比。这就是 HLL 名字里 "Log Log" 的由来:用 log(log N) bit 即可粗估 N。一个 6 bit 的寄存器可表示 ρ ∈ [0, 63],能估算的 N 上限是 263 ≈ 9 × 1018。

11.2 分桶与调和平均

把 64-bit hash 拆成两段:前 14 bit 作桶下标(共 m = 214 = 16384 个桶),后 50 bit 用于计算 ρ。每个桶维护一个寄存器 R[i],记录"路过该桶的所有元素中,后 50 bit ρ 的最大值"。

11.2.1 m 个寄存器的分桶机制

图 11-2-1 · 64-bit hash 的桶下标 + 后 50 bit ρ 拆分

Redis 使用 m = 16384 = 214。每寄存器存 6 bit(因为 ρ ≤ 50 + 1 = 51,6 bit 可表示 0..63)。总计 16384 × 6 / 8 = 12288 B = 12 KiB,加上 16 字节头部(HYLL magic + encoding flags + cached cardinality),稠密结构总长 12304 字节。

11.2.2 为什么用调和平均

图 11-2-2 · 调和平均 vs 算术平均(6 寄存器示例)

寄存器最大值 R = [3, 4, 4, 5, 4, 30] (6 寄存器, 异常值 30)

❌ 算术平均 (被异常值拉高)
avg = (3 + 4 + 4 + 5 + 4 + 30) / 6
    = 8.33

N ≈ m × 2^avg
  ≈ 6 × 322
  = 1932    ← 严重高估
→ 真实 N 应在几十量级
✅ 调和平均 (异常值被压制)
Z = 2^-3 + 2^-4 + 2^-4 + 2^-5 + 2^-4 + 2^-30
  = 0.125 + 0.0625 + 0.0625 + 0.03125 + 0.0625 + 9.3e-10
  ≈ 0.34375

H = m / Z = 6 / 0.34375 ≈ 17.45
N ≈ α_m × m² / Z ≈ 75
→ 异常值 30 因 2-30 ≈ 0 几乎不影响
HyperLogLog 选择调和平均 + 偏差校正常数 αm, 通过 2-R[i] 把大 R 的贡献压成 ≈ 0, 抑制零-run 长尾对估算的拉高效应。
调和平均的物理意义:2-R[i] 是"该桶第一次出现 ρ ≥ R[i] 的概率倒数"。把所有桶的概率倒数相加(Σ),得到一个对 "独立 hash 之间发生碰撞的总质量" 的估计。然后 m / Σ 反推每桶平均承载的 hash 数,乘以 m 得总基数。算术平均会被 "少数极端 ρ"(小概率事件)拉高,而调和平均通过 2-R[i] 把大 R 的影响压成 ≈ 0。

表 11-2-1 · HLL 估算公式 E = αm × m² / Z 的成分

这个公式是 HLL 的灵魂。m² 来自"每桶承载 N/m 个元素时,每桶最大 ρ ≈ log₂(N/m)" 的反推 + 调和平均的归一。αm 是 Flajolet 论文通过解析积分得出的偏差修正常数(详见原论文 Theorem 1)。

11.3 标准误差 1.04/√m

HLL 的精度由 m 决定。每翻倍一次 m,标准误差按 1/√2 ≈ 0.707 衰减,内存按 ×2 增加。Redis 选择 m = 16384 的依据是 12 KiB 内存与 0.81% 误差之间的性价比甜点。

11.3.1 不同 m 值下的精度-内存权衡

图 11-3-1 · m 翻倍 → 误差减 √2 倍 → 内存增 2 倍

m 翻倍到 65536 仅将误差从 0.81% 降到 0.41%,但内存翻为 48 KiB。对绝大多数 UV 场景,0.81% 已远优于业务的统计精度需求(广告 / 风控 / BI 通常容忍 1-5% 误差),没必要为 0.4% 多花 4 倍内存。

11.4.3 稠密存储宏与代码

/* src/hyperloglog.c — 关键宏(简化版) */

#define HLL_P 14                          /* 桶下标位数 */
#define HLL_REGISTERS (1 << HLL_P)        /* 16384 */
#define HLL_BITS 6                        /* 每寄存器 6 bit */
#define HLL_REGISTER_MAX ((1 << HLL_BITS) - 1)  /* 63 */
#define HLL_HDR_SIZE 16                   /* magic + encoding + cache */
#define HLL_DENSE_SIZE (HLL_HDR_SIZE + ((HLL_REGISTERS * HLL_BITS + 7) / 8))
                                          /* = 16 + 12288 = 12304 */

#define HLL_DENSE_GET_REGISTER(target, p, regnum) do {  \
    uint8_t *_p = (uint8_t *)(p);                       \
    unsigned long _byte = (regnum) * HLL_BITS / 8;      \
    unsigned long _fb   = (regnum) * HLL_BITS & 7;      \
    unsigned long _fb8  = 8 - _fb;                      \
    unsigned long b0 = _p[_byte];                       \
    unsigned long b1 = _p[_byte + 1];                   \
    target = ((b0 >> _fb) | (b1 << _fb8)) & HLL_REGISTER_MAX; \
} while(0)

#define HLL_DENSE_SET_REGISTER(p, regnum, val) do {     \
    uint8_t *_p = (uint8_t *)(p);                       \
    unsigned long _byte = (regnum) * HLL_BITS / 8;      \
    unsigned long _fb   = (regnum) * HLL_BITS & 7;      \
    unsigned long _fb8  = 8 - _fb;                      \
    unsigned long _v    = (val);                        \
    _p[_byte]   &= ~(HLL_REGISTER_MAX << _fb);          \
    _p[_byte]   |= _v << _fb;                           \
    _p[_byte+1] &= ~(HLL_REGISTER_MAX >> _fb8);         \
    _p[_byte+1] |= _v >> _fb8;                          \
} while(0)
为什么不直接用 8 bit 寄存器?8 bit 内存为 16 KiB,比 6 bit 多 33%。HLL 的设计哲学是 "每比特都有意义":6 bit 已能表示 ρ ∈ [0, 63],覆盖 N ≤ 263 的所有现实场景。多出的 2 bit 是浪费。代价是 GET/SET 需要跨字节位操作,但实测在现代 CPU 上 200 ns 可完成 ── 完全可接受。

11.7 小基数偏差与 LinearCounting 修正

HLL 在低基数区(N < ~2.5m,即 N < 40000 左右,m=16384 时)估算明显偏低,因为:大部分桶仍是 0,Z 中有大量 20 = 1 项,把 Z 拉大,从而把 E 拉小。Redis 用 LinearCounting 在低基数区接管。

11.7.2 修正阈值与代码

/* src/hyperloglog.c — hllCount() 简化版 */
double hllCount(struct hllhdr *hdr, int *invalid) {
    double m  = HLL_REGISTERS;       /* 16384 */
    double E;
    int   j;
    int   ez = 0;                    /* 零寄存器数 V */
    double zl = 0;                   /* Z = Σ 2^(-R[i]) */

    /* 1. 遍历 16384 寄存器,累加 Z 与统计 V */
    for (j = 0; j < HLL_REGISTERS; j++) {
        unsigned long reg;
        HLL_DENSE_GET_REGISTER(reg, hdr->registers, j);
        if (reg == 0) {
            ez++;
            zl += 1.0;               /* 2^0 */
        } else {
            zl += 1.0 / (1ULL << reg);  /* 2^(-reg) */
        }
    }

    /* 2. 主估算公式 E = α_m × m² / Z */
    E = (1.0/zl) * m * m * HLL_ALPHA_INF;  /* α_m ≈ 0.7213 */

    /* 3. LinearCounting 修正(低基数区) */
    if (E < m * 2.5 && ez != 0) {
        E = m * log(m / (double)ez);  /* LinearCounting */
    } else if (m == 16384 && E < 72000) {
        /* Redis 额外的高精度 HLL++ 偏差校正(略) */
    }

    return E;
}

11.8 PFMERGE 误差不累加

HLL 最优雅的工程性质之一:合并误差不累加。无论合并多少个 HLL,结果的标准误差仍是 0.81%。这一性质源于"合并 = 寄存器逐位取最大",而取最大不引入新的随机性,与"在新数据集上重新计算 HLL"的结果数学等价。

11.9 HLL 不可逆与不可枚举

HLL 的所有不支持操作都源于一个本质事实:寄存器只保留 hash 的"最大前导零"特征,不保留原始元素。需要反查 / 枚举的场景必须使用 Set 或 Bloom Filter。

11.11.3 hllAdd 关键代码

/* src/hyperloglog.c — hllAdd 简化版 */
int hllAdd(robj *o, unsigned char *ele, size_t elesize) {
    struct hllhdr *hdr = o->ptr;
    long index;
    uint8_t count;

    /* 1. 计算 hash 与 ρ */
    count = hllPatLen(ele, elesize, &index);  /* 内部用 MurmurHash64A
                                                 + leading_zeros + 1 */

    /* 2. 根据 encoding 走不同分支 */
    switch (hdr->encoding) {
    case HLL_DENSE:
        return hllDenseAdd(hdr->registers, index, count);
    case HLL_SPARSE:
        return hllSparseAdd(o, index, count);
    default:
        return -1;  /* invalid */
    }
}

/* hllDenseAdd: 仅当 new_ρ > old_ρ 时才更新 */
int hllDenseAdd(uint8_t *registers, long index, uint8_t count) {
    uint8_t oldcount;
    HLL_DENSE_GET_REGISTER(oldcount, registers, index);
    if (count > oldcount) {
        HLL_DENSE_SET_REGISTER(registers, index, count);
        return 1;  /* 寄存器变更,需要失效缓存 */
    }
    return 0;
}

11.11.4 设计取舍

图 11-11-2 · HLL 数据结构设计决策树

取舍 1:固定 m = 16384,不开放配置

不开放 m 配置参数,使所有 HLL 共享同一精度。优势是 PFMERGE 可直接逐寄存器合并;若不同 HLL 用不同 m 则合并需先重采样,代价远高于固定值带来的灵活性。劣势是无法用更小内存(如 256 寄存器、192 B)做粗略估计 ── 但 12 KiB 已经够小,不构成实际问题。

取舍 2:稀疏结构默认 3000 B 阈值

阈值定在 3000 B 而非 12 KiB:稀疏结构 PFCOUNT 复杂度 O(K) 与 opcode 数量线性相关,接近 12 KiB 时 K 已大,稀疏的内存优势消失而 CPU 开销已显著高于稠密。3000 B 是经验值,可通过 hll-sparse-max-bytes 调整。

取舍 3:cached_cardinality 缓存 PFCOUNT 结果

PFCOUNT 复杂度 O(m) ≈ 30~50 µs,在高频调用下不可忽视。通过缓存机制,无写入期间 PFCOUNT 退化为 O(1)。代价是写入需额外标记缓存失效(单字节最高位置 1)── 这个额外开销可以忽略不计,设计上是纯赚。

取舍 4:HLL 在底层是 String

OBJECT ENCODING hll:demo 返回 "raw",因为 HLL 在底层存为 SDS String,encoding 字段反映的是 String 编码而非 HLL 内部稀疏 / 稠密标记。稀疏 / 稠密标记在 HLL 头部第 4 字节(HYLL[4])。这种 "复用 String 类型作为承载容器" 的做法让 HLL 自动获得 String 的所有基础设施(持久化、复制、过期、迁移),减少内核工作量。

11.12 常见陷阱与反例

11.12.1 反例 1 · 用 HLL 算精确数据

错误:小集合追求精确

> PFADD users:online u1 u2 u3 u4 u5
> PFCOUNT users:online
5

> PFADD users:online u6 u7 ... u100
> PFCOUNT users:online
99                          # 应该 100, 但 HLL 0.81% 误差
                            # 在 N=100 时 ±1 已经 1% 误差
[场景判断]
- 算 1000 万 UV: 8 万误差可接受 -> HLL ✅
- 算 10 个用户的精确去重: 1 个误差就是 10% -> HLL ❌
- 财务相关基数: 需要精确 -> 不要用 HLL

修复:小集合用 SET

> SADD users:online u1 u2 u3 u4 u5
> SCARD users:online
5                            # 精确

> SADD users:online u6 u7 ... u100
> SCARD users:online
100                          # 精确

[规则]
- N < 1000: 用 SET (精确 + 内存可控)
- 1k < N < 1M: HLL 与 SET 都可
  内存敏感选 HLL
- N > 1M: HLL 几乎是唯一选择
- 财务/合规: 不论 N 多大,
  都要用 SET 或精确算法

11.12.2 反例 2 · 超大 PFMERGE 集合

错误:一次合并 365 个 HLL

> PFMERGE total:uv:weekly daily:1 ... daily:7
                                      # 7 个还行 ~350 µs
> PFMERGE total:uv:yearly daily:1 ... daily:365
                                      # 365 个,每次 ~15 ms
                                      # 阻塞主线程 15 ms

# 现象:
# - 主线程被占 15 ms,其他请求积压
# - 每次年报查询都要 15 ms 阻塞
# - 高 QPS 场景下连锁雪崩

修复:分级预聚合

# 提前增量合并,建立金字塔
PFMERGE weekly_w1 daily_d1 ... daily_d7
PFMERGE weekly_w2 daily_d8 ... daily_d14
...
PFMERGE monthly_m1 weekly_w1 ... weekly_w4
PFMERGE quarterly_q1 monthly_m1 monthly_m2 monthly_m3
PFMERGE yearly quarterly_q1 ... quarterly_q4

# 年报查询只合 4 个 quarterly:
> PFMERGE total:uv:yearly quarterly_q1 ... q4
                                      # 4 个,~200 µs
> PFCOUNT total:uv:yearly             # 缓存命中,<1 µs

11.12.3 反例 3 · 不知 sparse / dense 切换

现象:应用监控看到 hll:online 内存从 200 B 突变到 12 KiB,团队怀疑有 bug。
根因:distinct N 超过约 3000(hll-sparse-max-bytes 默认 3000),自动从 sparse 转 dense。
判断:这是正常行为,不是 bug。
注意:转换不可逆 ── dense 不会回到 sparse。即使后续删掉所有元素(实际上 HLL 也不支持删),encoding 也不会回退。

11.12.4 反例 4 · 把 HLL 当 Set 用

错误:对 HLL 调 Set 命令

        against a key holding the
        wrong kind of value
