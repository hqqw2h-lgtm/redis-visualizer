第 23 章 · 淘汰策略

八种 maxmemory-policy · 采样 + 16 槽淘汰池 · 24-bit LRU 时钟 · Morris 概率计数器 · LFU 衰减 · volatile-ttl 预测性淘汰 · Cluster 一致性

Redis 是内存数据库,内存容量是硬约束。当 Redis 通过分配器申请的内存接近 maxmemory 时,写命令必须在"淘汰旧 key"与"返回 OOM"之间作出选择。注意:maxmemory 约束的是 Redis 统计内存,不是 OS RSS、网络缓冲、复制缓冲或持久化 fork 峰值。Redis 的淘汰机制由三部分组成:八种策略覆盖"范围 × 算法"二维取舍空间,随机采样 + 淘汰池实现 O(1) 时间复杂度的近似 LRU/LFU,redisObject 内嵌 24-bit lru 字段使元数据开销近零。

精确 LRU 需要全局双向链表,每节点至少 16 字节指针开销,千万级 key 下额外内存约 152 MB,且每次访问都需链表移动维护。精确 LFU 需要全局优先队列,空间 O(N) + 维护 O(log N)。Redis 的近似方案以"命中率损失约 3-5%"换取"元数据开销近零 + 维护 O(1)",该取舍在缓存场景下显著优于精确算法。本章按"必要性 → 八种策略 → 采样机制 → LRU/LFU 算法 → 部署适配 → 陷阱"组织,聚焦淘汰决策的工程权衡与生产可观测性。

23.0 章首速览

图 23-0-1 · 淘汰决策全景

淘汰由 processCommand 在写命令前置检查中触发。检查路径上同步执行 performEvictions(),直至内存压回上限或候选集耗尽。该路径的延迟会反馈到客户端,因此淘汰算法的复杂度直接影响主线程性能。

核心洞察: Redis 淘汰是"同步阻塞"操作——写命令在执行业务逻辑之前必须先把内存压回 maxmemory。这意味着淘汰算法的 CPU 开销直接进入客户端 P99 延迟。这就是 Redis 选择"采样近似 O(s)"而非"全局有序 O(log N)"的根本原因。

noeviction 是默认策略,等价于"内存满则拒绝写入"。其余七种策略沿"范围(allkeys / volatile)" × "算法(lru / lfu / random / ttl)"两维度组合。volatile-* 仅在候选集非空时工作,所有 key 无 TTL 时退化为 noeviction。

23.1 淘汰的必要性

23.1.1 Redis 的两种使用角色

图 23-1-1 · 缓存角色 vs 数据库角色

两种角色的策略选择截然相反。缓存角色优先选择 allkeys-lru 或 allkeys-lfu,允许淘汰任意 key 以保持服务可用;数据库角色选择 noeviction,以 OOM 拒绝写入换取数据完整性。

生产环境最大的配置错误: "在数据库角色实例上配置 allkeys-*",会静默丢失主副本数据。一旦内存达到上限,业务关键数据会被悄无声息地淘汰,而应用层往往要等到读取时才发现数据"消失"。

23.1.2 processCommand 的淘汰触发点

/* src/server.c: 简化的写命令前置检查 */
int processCommand(client *c) {
    if (server.maxmemory && !mustObeyClient(c)) {
        int out_of_memory = (performEvictions() == EVICT_FAIL);
        if (out_of_memory && is_denyoom_command(c->cmd)) {
            rejectCommand(c, shared.oomerr);
            return C_OK;
        }
    }
    /* 继续执行命令 */
}

淘汰检查在命令分发前同步执行。performEvictions() 内部循环采样并删除候选 key,直至内存满足条件或返回 EVICT_FAIL。该路径阻塞当前命令的执行,故淘汰量大时会推高客户端 P99 延迟。mustObeyClient 用于排除从复制流执行命令的 replica 客户端,以保持主从淘汰逻辑由主节点统一控制。

23.1.3 maxmemory 与物理内存的关系

配置层次:
  系统物理内存            32 GB
    ├── 操作系统           ~2 GB
    ├── 其他进程           ~4 GB
    └── Redis 总开销       ~26 GB
          ├── used_memory          maxmemory 控制
          ├── 复制 backlog         repl-backlog-size
          ├── 客户端输出缓冲       client-output-buffer-limit
          ├── lua 引擎             少量
          ├── COW (RDB/AOF rewrite) 取决于写流量
          └── 内存碎片             mem_fragmentation_ratio

示例经验(不是固定比例):
  maxmemory 可先按物理内存的 60-70% 估算
  其余空间留给:
    fork() COW 内存峰值
    输出缓冲、复制缓冲
    内存碎片(jemalloc)

例: 32 GB 系统物理内存
  maxmemory 20 GB
  保留 12 GB 容纳 RDB/AOF 重写期间的 COW

maxmemory 控制的是 Redis 统计内存口径,不是 OS RSS 或 cgroup 上限;fork COW、输出缓冲、复制 backlog、allocator dirty page 与内核开销仍需额外预留。生产配置应基于本实例写入峰值、持久化窗口与碎片指标校准,否则 BGREWRITEAOF / BGSAVE 期间可能触发 OOM Killer。

真实事故案例: 某电商大促前夜,运维将 maxmemory 设为系统物理内存的 90%(28.8 GB / 32 GB),意图最大化缓存容量。大促开始后写流量激增,触发 BGREWRITEAOF,fork 子进程时 COW 在 5 分钟内增长 6 GB,系统总内存使用达到 35 GB,kernel OOM Killer 选择 Redis 主进程(RSS 最大)杀掉,服务中断 12 分钟。事后将 maxmemory 调回 20 GB(62.5%),并增加 repl-backlog-size 监控。

23.3 采样机制与淘汰池

23.3.1 为什么不维护全局有序结构

表 23-3-1 · 三种 LRU 实现的内存与时间开销
方案元数据开销(每 key)访问时维护成本淘汰成本1000 万 key 总开销
精确 LRU 双向链表16 字节 (两指针)O(1) 链表移动O(1)~152 MB

robj.lru 是 redisObject 中的 24-bit 字段,无需额外内存。采样近似的代价是命中率较精确算法略低(典型 3-5%),但元数据开销节省可达数百 MB,在内存数据库语境下是显著正向收益。

23.3.2 performEvictions 主循环

/* src/evict.c: 简化的淘汰主循环 */
int performEvictions(void) {
    long long mem_used = zmalloc_used_memory();
    long long mem_tofree = mem_used - server.maxmemory;
    int result = EVICT_OK;

    while (mem_tofree > 0) {
        /* 1. 选择候选集字典 */
        int policy = server.maxmemory_policy;
        struct evictionPoolEntry *pool = EvictionPoolLRU;

        if (policy & MAXMEMORY_FLAG_ALLKEYS) {
            sampledict = db->dict;
        } else {
            sampledict = db->expires;
        }
        if (dictSize(sampledict) == 0) {
            result = EVICT_FAIL;  /* 候选集为空 */
            break;
        }

        /* 2. 采样填充淘汰池 */
        evictionPoolPopulate(dbid, sampledict, db->dict, pool);

        /* 3. 从淘汰池尾部取出最差候选 */
        bestkey = pool[EVPOOL_SIZE - 1].key;
        if (bestkey == NULL) {
            result = EVICT_FAIL;
            break;
        }

        /* 4. 删除该 key */
        long long delta = (long long)zmalloc_used_memory();
        if (server.lazyfree_lazy_eviction)
            dbAsyncDelete(db, keyobj);
        else
            dbSyncDelete(db, keyobj);
        delta -= (long long)zmalloc_used_memory();
        mem_tofree -= delta;

        server.stat_evictedkeys++;
    }
    return result;
}

主循环每轮采样 maxmemory-samples 个 key,填充至 16 槽淘汰池,从池尾取出 idle 值最大的候选删除。删除一个 key 释放的内存可能不足以将 used_memory 压回 maxmemory,故循环可能多次迭代,直至条件满足或候选集耗尽。

23.3.3 淘汰池数据结构

/* src/evict.c */
#define EVPOOL_SIZE 16
#define EVPOOL_CACHED_SDS_SIZE 255

struct evictionPoolEntry {
    unsigned long long idle;  /* idle time (LRU) 或 idle value (LFU/TTL) */
    sds key;                  /* key 名 SDS 拷贝 */
    sds cached;               /* 可复用的 SDS 缓冲 */
    int dbid;                 /* DB 编号 */
};

static struct evictionPoolEntry *EvictionPoolLRU;
淘汰池槽位排列:

  slot 0    slot 1    slot 2  ...  slot 14   slot 15
  ┌─────┐  ┌─────┐  ┌─────┐      ┌─────┐  ┌─────┐
  │idle │  │idle │  │idle │      │idle │  │idle │
  │ 100 │  │ 250 │  │ 480 │  ... │1500 │  │2000 │
  └─────┘  └─────┘  └─────┘      └─────┘  └─────┘
   最佳                                   最差
   候选                                   候选
   (保留)                                 (淘汰)

idle 值含义:
  LRU 模式: idle = now_clock - obj.lru,值越大越久未访问
  LFU 模式: idle = 255 - LFUDecrAndReturn(obj),值越大频率越低
  TTL 模式: idle = ULLONG_MAX - expire_time,过期越早 idle 越大

槽位按 idle 值升序排列,槽 15 始终持有当前已观察到的"最差候选"。新采样的 key 通过二分查找插入到合适位置,如果新 key 的 idle 值小于槽 0,则丢弃。淘汰时从槽 15 取出。

23.3.4 采样如何填充淘汰池

图 23-3-4 · evictionPoolPopulate 时序

dictGetSomeKeys 在 hash table 上随机起点采样,返回 N 个 key(可能少于 N,取决于 hash 表填充率)。每个 key 计算 idle 值并尝试插入淘汰池。该过程的关键工程价值在于"跨轮次累积":第一轮可能采到平庸候选,但好的候选会留在池中,后续轮次只替换为更差的候选。

23.3.5 采样质量的概率分析

设 N 为候选集总 key 数, k 为目标 (期望淘汰的 top-k 最差候选数)
单次采样 s 个 key,至少命中一个目标的概率:

  P(命中) = 1 - C(N-k, s) / C(N, s)
         ≈ 1 - (1 - k/N)^s   (N >> k, s 时近似)

数值示例 (N = 1,000,000, k = 10,000 即 top-1%):

  s = 10  (默认):
    P = 1 - (1 - 0.01)^10 = 1 - 0.99^10 ≈ 9.6%
  
  连续 10 轮采样后,池中至少有一个 top-1% 的概率:
    P_10轮 = 1 - (1 - 0.096)^10 ≈ 63.7%
  
  连续 50 轮:
    P_50轮 = 1 - (1 - 0.096)^50 ≈ 99.4%

结论: 采样数 s=10 单轮命中率 9.6%,但跨轮次淘汰池累积后,
      50 轮内几乎必然将 top-1% 候选淘汰。
核心洞察: 淘汰池的工程价值即"用 16 个槽位的 O(1) 内存,将单轮 9.6% 的命中率放大到多轮的 99%+"。该机制是 Redis 采样近似能逼近精确算法命中率的根本原因。

23.4 近似 LRU:24-bit 时钟

23.4.1 redisObject 中的 lru 字段

/* src/server.h */
typedef struct redisObject {
    unsigned type:4;       /* 数据类型 */
    unsigned encoding:4;   /* 内部编码 */
    unsigned lru:24;       /* LRU/LFU 复用字段 */
    int refcount;          /* 引用计数 */
    void *ptr;             /* 数据指针 */
} robj;
24-bit lru 字段的两种解读:

  LRU 模式 (allkeys-lru / volatile-lru):
    24 bits 全部记录访问时间戳的低 24 位 (秒级)
    
        bit 23                    bit 0
        ┌─────────────────────────────┐
        │   Redis LRU clock 低 24 位  │
        └─────────────────────────────┘

  LFU 模式 (allkeys-lfu / volatile-lfu):
    高 16 bits 衰减时间, 低 8 bits 频率计数
    
        bit 23      bit 8  bit 7   bit 0
        ┌───────────────┐ ┌──────────┐
        │ ldt 衰减时间  │ │ logc 计数│
        └───────────────┘ └──────────┘
        分钟级时间戳        Morris 计数器

LRU 与 LFU 模式复用同一个 24-bit 字段,运行时根据 maxmemory-policy 动态决定该字段的语义。切换策略不需要重启 Redis,但已有 key 的 lru 字段会被新策略重新解读,可能导致短暂的命中率波动。

23.4.2 LRU clock 的更新

/* src/evict.c */
unsigned int LRU_CLOCK(void) {
    unsigned int lruclock;
    if (1000 / server.hz <= LRU_CLOCK_RESOLUTION) {
        atomicGet(server.lruclock, lruclock);
    } else {
        lruclock = getLRUClock();
    }
    return lruclock;
}

unsigned int getLRUClock(void) {
    return (mstime() / LRU_CLOCK_RESOLUTION) & LRU_CLOCK_MAX;
}

#define LRU_BITS 24
#define LRU_CLOCK_MAX ((1 << LRU_BITS) - 1)  /* 16,777,215 */
#define LRU_CLOCK_RESOLUTION 1000             /* 1 秒 */

LRU 时钟分辨率 1 秒。serverCron 以 server.hz 频率(默认 10 Hz)更新全局 server.lruclock,key 访问时直接读取该原子变量,避免每次访问都调用 mstime()。低于 1000 ms 的访问间隔无法区分,这是 24-bit 精度的工程取舍。

23.4.3 idle 时间计算与回绕

/* src/evict.c: 简化 */
unsigned long long estimateObjectIdleTime(robj *o) {
    unsigned long long lruclock = LRU_CLOCK();
    if (lruclock >= o->lru) {
        return (lruclock - o->lru) * LRU_CLOCK_RESOLUTION;
    } else {
        /* 时钟回绕处理 */
        return (lruclock + (LRU_CLOCK_MAX - o->lru))
               * LRU_CLOCK_RESOLUTION;
    }
}
24-bit clock 的回绕周期:

  LRU_CLOCK_MAX = 2^24 - 1 = 16,777,215
  LRU_CLOCK_RESOLUTION = 1 秒
  回绕周期 = 16,777,215 秒 ≈ 194.18 天

回绕场景:
  obj.lru = 16,000,000 (约 185 天前)
  当前 lruclock = 100,000 (回绕后)
  
  estimateObjectIdleTime:
    lruclock < obj.lru → 回绕分支
    idle = 100,000 + (16,777,215 - 16,000,000) = 877,215 秒
    ≈ 10.15 天

精度损失分析:
  实际 idle = (24*3600 - lruclock 启动时刻) + 回绕已过秒数
  对于已经 194 天未访问的 key,精度损失数小时,
  在淘汰决策中无影响 (该 key 在任何合理候选集中都是最差候选)

23.4.4 LRU 与精确 LRU 的差异

图 23-4-4 · 精确 vs 近似 LRU

精确 LRU 维护全局有序链表,需要每次访问时调整链表结构,且每个 key 增加两个指针(64-bit 系统 16 字节)。Redis 近似 LRU 仅在访问时更新一个 24-bit 字段,淘汰时采样 10 个 key 选择 idle 最大者。后者牺牲部分精度(单次淘汰可能选不到全局最差候选),但通过淘汰池跨轮次累积,最终命中率与精确 LRU 接近。

23.5 近似 LFU:Morris 计数器

23.5.1 LFU 字段的双重含义

24-bit lru 字段在 LFU 模式下:

  bit 23                    bit 8  bit 7        bit 0
  ┌─────────────────────────────┐ ┌──────────────┐
  │   ldt: 衰减时间 (16 bits)   │ │ logc: 8 bits │
  │   分钟级时间戳              │ │ 0-255 计数器 │
  └─────────────────────────────┘ └──────────────┘

ldt (logarithmic decay time):
  分辨率: 1 分钟
  范围: 16 bits = 65,535 分钟 ≈ 45.5 天
  超过范围回绕,与 LRU 同样需要回绕处理

logc (logarithmic counter):
  范围: 0-255 (8 bits)
  初始值: LFU_INIT_VAL = 5 (新 key 默认)
  增长: Morris 概率递增
  衰减: 根据 lfu-decay-time 配置

LFU 字段的设计目标:用 24-bit 同时表达"频率"与"衰减时间",支持百万级访问的频率区分,且热度可随时间衰减。8-bit logc 表面上仅 0-255,但通过对数概率递增,实际可表达 10^6 量级的访问频率。

23.5.2 Morris 概率计数器

/* src/evict.c */
#define LFU_INIT_VAL 5

uint8_t LFULogIncr(uint8_t counter) {
    if (counter == 255) return 255;
    double r = (double)rand() / RAND_MAX;
    double baseval = counter - LFU_INIT_VAL;
    if (baseval < 0) baseval = 0;
    double p = 1.0 / (baseval * server.lfu_log_factor + 1);
    if (r < p) counter++;
    return counter;
}
Morris 计数器递增概率表 (lfu_log_factor = 10, 默认):

  当前 logc   递增概率           等价访问量级直觉
  ────────  ─────────────────  ─────────────────
  5 (初始)    100%               新 key
  6           1/(1*10+1) = 9.1%   ~10 次访问
  10          1/(5*10+1) = 1.96%  ~100 次访问
  18          1/(13*10+1) = 0.77% ~1000 次访问
  50          1/(45*10+1) = 0.22% ~10000 次访问
  142         1/(137*10+1)≈0.073% ~100000 次访问
  255         0 (饱和)            ~1000000+ 次访问

含义:
  低频 key 的 logc 增长快,容易区分访问 1 次 vs 10 次
  高频 key 的 logc 增长慢,饱和需要百万次访问
  对数关系使 8 bits 可表达 6 个数量级的频率范围
历史脉络: Morris 计数器的核心思想是"用概率递增模拟对数计数"。当 counter = c 时,以概率 p ∝ 1/c 递增,使 counter 随访问次数对数增长。该机制最早由 Robert Morris 在 1978 年提出,用于在有限位宽下计数大数量级事件。Redis 复用该方案在 8 bits 内表达 10^6 量级访问频率。

23.5.3 LFU 衰减机制

/* src/evict.c */
unsigned long LFUDecrAndReturn(robj *o) {
    unsigned long ldt = o->lru >> 8;
    unsigned long counter = o->lru & 255;
    unsigned long num_periods = server.lfu_decay_time
        ? LFUTimeElapsed(ldt) / server.lfu_decay_time : 0;
    if (num_periods)
        counter = (num_periods > counter) ? 0 : counter - num_periods;
    return counter;
}

unsigned long LFUTimeElapsed(unsigned long ldt) {
    unsigned long now = LFUGetTimeInMinutes();
    if (now >= ldt) return now - ldt;
    return 65535 - ldt + now;  /* ldt 回绕处理 */
}
衰减计算示例 (lfu-decay-time = 1):

  场景 1: 热 key 持续访问
    访问时刻 t=0:  ldt=0, logc=5
    访问时刻 t=10: 增 logc 到 ~10
    访问时刻 t=60: 1 小时后,logc 已饱和到 ~50
    
  场景 2: 热 key 停止访问
    最后访问 t=0: ldt=0, logc=50
    查询时刻 t=120 分钟:
      num_periods = 120 / 1 = 120
      counter = 50 - 120 = max(0, 50-120) = 0
    
  场景 3: 中等访问
    最后访问 t=0: ldt=0, logc=20
    查询时刻 t=10 分钟:
      num_periods = 10 / 1 = 10
      counter = 20 - 10 = 10

衰减目的:
  避免历史热点永久占据高频位
  允许新热点在合理时间内取代旧热点
  lfu-decay-time 越大,衰减越慢,历史热点保留越久

衰减机制是 LFU 抵抗"僵尸热度"的关键。无衰减的 LFU 会让历史热点永久驻留(其 logc 已饱和),即使长期不再访问也不会被淘汰。lfu-decay-time 配置衰减速度,默认 1 分钟衰减 1 单位。

23.5.4 OBJECT FREQ 命令的语义

$ redis-cli OBJECT FREQ user:hot:1
(integer) 142

$ redis-cli OBJECT FREQ user:cold:1
(integer) 5
OBJECT FREQ 返回的是 logc 值,不是真实访问次数:

  logc = 5  → 新 key 或已衰减完
  logc = 10 → 约百次访问
  logc = 50 → 约万次访问
  logc = 142 → 约十万次访问
  logc = 255 → 已饱和,百万级访问

误区:
  ✗ logc = 100 不等于访问 100 次
  ✗ logc 不可线性比较 (100 vs 200 不是 2 倍频率)

正确用法:
  ✓ 比较两 key logc 的相对大小,识别冷/热分布
  ✓ 监控 logc 分布,验证 LFU 是否正确识别热点
  ✓ logc 持续 < 10 的 key 是淘汰候选
常见误用: 监控大盘把 OBJECT FREQ 当作"访问次数"画折线图,实际是 logc 值。两 key logc=50 与 logc=100 不是"2 倍频率",而是大致"10000 vs 100000 量级"的访问差。只能用作冷/热分级,不能做精确流量统计。

23.6 volatile-ttl:预测性淘汰

23.6.1 volatile-ttl 的评分计算

/* src/evict.c: evictionPoolPopulate 中 */
if (server.maxmemory_policy == MAXMEMORY_VOLATILE_TTL) {
    idle = ULLONG_MAX - (long)dictGetVal(de);
    /* dictGetVal(de) 是 expire 字典的 value, 即过期时间戳(毫秒) */
}
淘汰池 score 计算逻辑(变量名仍叫 idle):

  expire_time   key 的过期时间戳 (毫秒)
  score = ULLONG_MAX - expire_time

  过期越早 → expire_time 越小 → score 越大 → 越优先淘汰

示例:
  key A: 1 分钟后过期, expire_time = now + 60_000
  key B: 1 小时后过期, expire_time = now + 3_600_000
  
  A 的 score > B 的 score
  A 更可能被采样选为淘汰候选

设计依据:
  即将过期的 key 对业务的剩余价值最低
  主动淘汰它们等价于"提前过期",释放空间给新写入
  优于等待惰性删除/定期扫描

23.6.2 volatile-ttl 与过期删除的关系

图 23-6-2 · 三种 TTL 删除机制

volatile-ttl 是独立于过期删除的机制。过期删除处理"已到期的 key",volatile-ttl 在内存压力下处理"即将到期的 key"。两者的触发条件、时机、目标 key 集合完全不同,但都作用于 db->expires 字典。

23.7.3 Cluster 各 master 独立淘汰的风险

风险场景: maxmemory 配置不一致

  shard A master: maxmemory 4 GB, 当前 used 3.5 GB
  shard B master: maxmemory 2 GB, 当前 used 1.9 GB
  
  写入分布: 30% 进入 shard A, 70% 进入 shard B (slot 倾斜)
  
  结果:
    shard B 先达到 maxmemory,触发淘汰
    shard A 仍有 0.5 GB 空闲
    业务热数据在 shard B 被淘汰,
    但 shard A 上较冷的 key 未被淘汰
    总命中率下降

修复:
  cluster-maxmemory: 统一所有 master 的 maxmemory
  redis-cli --cluster call $ENTRY CONFIG SET maxmemory 4gb
  
  slot 分布检查:
    redis-cli --cluster check
    redis-cli CLUSTER COUNTKEYSINSLOT
    
  rebalance 不均时:
    redis-cli --cluster rebalance
真实事故: 某社交平台扩容时新加 2 个 master 节点,运维忘记同步 maxmemory 配置(新节点默认 0,即无限制)。一个月后老节点频繁淘汰、命中率从 95% 跌到 78%,但新节点内存增长到接近物理上限。排查发现 redis-cli --cluster call 显示新老节点 maxmemory 不一致。修复后命中率回升,后续将 maxmemory 检查加入 cluster 健康检查脚本。

❌ 反模式 · volatile-lru + 无 TTL

SET user:1 "..."         # 无 EXPIRE
SET session:1 "..."      # 无 EXPIRE
SET cache:1 "..."        # 无 EXPIRE

# 内存达上限后写入失败
SET new:1 "..."
# (error) OOM command not allowed when used memory > 'maxmemory'

db->expires 为空,volatile-* 等价于 noeviction。所有"看起来配置了 LRU 淘汰"的实例,实际仍是 noeviction 行为。

✅ 正确做法 · allkeys-lfu

# 缓存角色用 allkeys-lfu,不依赖 TTL
CONFIG SET maxmemory 4gb
CONFIG SET maxmemory-policy allkeys-lfu
CONFIG SET maxmemory-samples 10
CONFIG SET lazyfree-lazy-eviction yes

# 业务 key 可以不设 TTL
SET user:1 "..."
SET session:1 "..."  # 可选:加 EXPIRE 加速淘汰冷数据

# 内存达上限自动淘汰 logc 最低的 key
SET new:1 "..."      # 触发淘汰,新 key 写入成功

缓存角色优先 allkeys-lfu,不依赖业务给所有 key 加 TTL;同时开启 lazyfree 避免大 key 淘汰阻塞主线程。

23.9.4 lazyfree 对淘汰的影响

lazyfree-lazy-eviction yes:
  
  performEvictions 调用 dbAsyncDelete
  ┌───────────────────────────────┐
  │ dbAsyncDelete                 │
  │   1. 主线程移除 key 索引       │  
  │   2. 估算 value 大小           │
  │   3. value 大 → 提交 bio 线程  │  ← 异步释放
  │      value 小 → 主线程释放    │
  └───────────────────────────────┘
  主线程仅承担索引移除 O(1)
  bio 线程异步释放数据结构内存

lazyfree-lazy-eviction no:
  
  performEvictions 调用 dbSyncDelete
  ┌───────────────────────────────┐
  │ dbSyncDelete                  │
  │   1. 主线程移除 key 索引       │  
  │   2. 主线程递归释放 value      │  ← 同步释放
  │      list/hash/set/zset 全遍历 │
  └───────────────────────────────┘
  大 hash/set/zset 淘汰可能阻塞主线程数十毫秒
生产强烈推荐: lazyfree-lazy-eviction yes 是生产环境的强烈推荐配置。淘汰 1 GB hash 时同步释放可能阻塞主线程 100 ms+,异步释放仅占用主线程 O(1) 索引移除时间。该配置在 §24 lazyfree 章详细分析。

23.10.3 lookupKey 访问时更新元数据

/* src/object.c: 访问 key 时更新 LRU/LFU 元数据 */
robj *lookupKey(redisDb *db, robj *key, int flags) {
    dictEntry *de = dictFind(db->dict, key->ptr);
    if (de) {
        robj *val = dictGetVal(de);

        if (server.maxmemory_policy & MAXMEMORY_FLAG_LFU) {
            updateLFU(val);
        } else {
            val->lru = LRU_CLOCK();
        }
        return val;
    }
    return NULL;
}

void updateLFU(robj *val) {
    unsigned long counter = LFUDecrAndReturn(val);
    counter = LFULogIncr(counter);
    val->lru = (LFUGetTimeInMinutes() << 8) | counter;
}

23.10.4 淘汰池插入

/* src/evict.c: 淘汰池插入 */
void evictionPoolPopulate(int dbid, dict *sampledict, dict *keydict,
                          struct evictionPoolEntry *pool) {
    int j, k, count;
    dictEntry *samples[server.maxmemory_samples];

    count = dictGetSomeKeys(sampledict, samples, server.maxmemory_samples);

    for (j = 0; j < count; j++) {
        unsigned long long idle;
        sds key = dictGetKey(samples[j]);
        robj *o = dictGetVal(samples[j]);

        if (server.maxmemory_policy & MAXMEMORY_FLAG_LRU) {
            idle = estimateObjectIdleTime(o);
        } else if (server.maxmemory_policy & MAXMEMORY_FLAG_LFU) {
            idle = 255 - LFUDecrAndReturn(o);
        } else if (server.maxmemory_policy == MAXMEMORY_VOLATILE_TTL) {
            idle = ULLONG_MAX - (long)dictGetVal(samples[j]);
        }

        /* 二分查找插入位置 */
        k = 0;
        while (k < EVPOOL_SIZE && pool[k].key && pool[k].idle < idle) k++;
        if (k == 0 && pool[EVPOOL_SIZE - 1].key != NULL) continue;
        else if (k < EVPOOL_SIZE && pool[k].key == NULL) {
            /* 插入空槽 */
        } else {
            /* 移位插入 */
        }
        pool[k].key = sdsdup(key);
        pool[k].idle = idle;
        pool[k].dbid = dbid;
    }
}

23.10.5 设计取舍

Redis 选择采样近似而非精确 LRU/LFU 的根本原因是元数据成本。精确 LRU 需要每 key 16 字节链表指针,千万级 key 累积 152 MB,且每次访问需链表移动维护。采样近似将元数据压缩到 24 bits 嵌入 robj,访问维护成本 O(1) 时间戳更新,淘汰成本 O(s) 采样,代价是命中率较精确算法低约 3-5%。在内存数据库语境下,该取舍是显著正向收益:节省的内存可多存数百万 key。

24-bit lru 字段复用为 LRU 时钟或 LFU(ldt + logc)的设计目标是"零额外内存"。若每个 key 增加 8 字节计数器,千万级 key 累积 80 MB,与采样近似的内存收益相消。复用方案的代价是 LRU 与 LFU 不能同时启用,切换策略需要重新解读字段语义。

淘汰池的 16 槽位是经验取舍。槽数过少则跨轮次累积效果弱,槽数过多则插入开销增加且淘汰收益边际递减。16 槽位在概率分析中已能将单轮 9.6% 命中率放大到 50 轮 99%+,继续扩容收益不显著。

23.11 本章要点

#要点
1淘汰由 processCommand 在写命令前置检查触发,同步执行直至内存压回 maxmemory 或候选集耗尽
2八种策略按"范围 (allkeys/volatile) × 算法 (lru/lfu/random/ttl/none)"二维组合,noeviction 是默认
3volatile-* 候选集为 db->expires,所有 key 无 TTL 时退化为 noeviction,生产最常见配置错误
4数据库角色通常使用 noeviction;缓存角色按访问分布选择 allkeys-lfu(Zipfian)或 allkeys-lru(均匀)
5采样 + 16 槽淘汰池实现 O(1) 元数据 + O(s) 淘汰开销,跨轮次累积使命中率逼近精确算法
6redisObject 的 24-bit lru 字段在 LRU 模式存时间戳,LFU 模式分裂为 16-bit ldt + 8-bit logc
7LRU 时钟 24 bits 秒级精度,回绕周期 194 天,实际场景无影响;LFU 分钟级 ldt 回绕周期 45 天
8LFU 用 Morris 概率递增使 8-bit logc 表达 6 个数量级访问频率,衰减由 lfu-decay-time 控制
9volatile-ttl 是预测性淘汰,优先删除即将过期的 key,与过期删除机制独立
10常见配置起点:lazyfree-lazy-eviction yes 降低大 key 释放阻塞,maxmemory-samples 按命中率/CPU 权衡,Cluster 各 master 容量与 slot 规划一致

章末总结

第 23 章淘汰策略与第 22 章 jemalloc、第 24 章 lazyfree 共同构成 Redis 内存管理的三层基础:
