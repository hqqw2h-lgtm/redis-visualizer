第 10 章 · Leader Epoch 与日志截断

KIP-101 · KIP-279 · OffsetForLeaderEpoch · 任期权威边界 · FencedLeaderEpoch

第 9 章讲了 HW 是消费者可见边界,但有一个隐藏的暗角:follower 重启时,凭什么相信本地 HW 是对的?Kafka 0.11 以前的回答是"我读自己 highwatermark-checkpoint 文件就行" — 这个回答在连续故障场景下会让已提交的消息消失。KIP-101 (引入 Leader Epoch) 和 KIP-279 (修正 follower 截断逻辑) 把这个洞补上:截断点不再来自"本地观察",而是向当前 leader 查询"我那个任期的 endOffset 是多少"。

本章按 "旧问题 → epoch 数据模型 → OffsetForLeaderEpoch 协议 → 截断场景 → Fetch 校验 → 与 Raft Term 对比 → 排障 → 源码 → 取舍 → 陷阱" 组织,对应 Apache Kafka 3.7 的实现。读完后你不仅理解协议,还能预判每一种异常:为什么 follower 启动慢可能因为 OffsetForLeaderEpoch 卡住?为什么 FENCED_LEADER_EPOCH 出现在日志里不一定是 bug?

  • Pre-KIP-101 的 truncate-to-local-HW 在哪两类场景下会丢已提交数据
  • Leader Epoch + startOffset 的形式定义、持久化文件、写入时机
  • OffsetForLeaderEpoch 请求/响应的字节布局,以及三种错误码的语义
  • Follower 重启的完整恢复路径:读本地 epoch → 查 leader → 截断 → Fetch
  • 四种 Fetch 错误:FENCED_LEADER_EPOCH / UNKNOWN_LEADER_EPOCH / NOT_LEADER / NOT_LEADER_FOR_PARTITION
  • Kafka Leader Epoch 与 Raft Term / Paxos Ballot 的对应关系

图 10-0-1 · 三句话浓缩
本章三句话:

  1 Leader Epoch 是分区 Leader 任期编号
    每次 Leader 变更, epoch + 1
    每个 epoch 关联一个 startOffset

  2 Follower 恢复截断点不来自本地 HW
    而是向当前 Leader 查询该 epoch 的 endOffset
    确保删除孤儿数据但保留已提交数据

  3 Fetch 请求携带 currentLeaderEpoch
    Broker 用它拒绝过期请求 (FENCED_LEADER_EPOCH)
    避免使用过时元数据继续复制
把 Leader Epoch 当作"分区版本号"来看会更直观:每次 leader 切换 = 主版本号 +1。任何关于该分区的 RPC(Fetch / OffsetForLeaderEpoch)都要带上"我以为现在是哪个版本",broker 比对后才决定是接受、拒绝、还是要求你重新去拉元数据。这套机制让 Kafka 在网络分区、controller 切换、broker 重启等复杂情况下,始终能给出一个"权威答案",而不是各方互相猜。

10.1 旧截断逻辑的不安全性

Pre-KIP-101 的截断逻辑非常简单 — 简单到不安全。Follower 启动时读自己 highwatermark-checkpoint 文件,然后把 LEO 截到 HW 上。这个逻辑的隐含假设是:"本地 HW = 集群已提交位置"。可惜这个假设在两类场景下都不成立。

图 10-1-1 · 旧截断逻辑的灾难时序

旧截断逻辑的伪代码

// Pre-KIP-101 错误实现
void recoverWithLocalHw(Log log) {
    long localHw = log.readHighWatermarkCheckpoint();
    if (log.logEndOffset() > localHw) {
        log.truncateTo(localHw);    // 用本地 HW 截断 — 不安全
    }
}
失败条件:
条件触发概率影响
Follower 已复制但本地 HW 未推进每次 FetchResponse 间隙该 Follower 单独崩溃后重启时丢已提交数据
Leader 在 Follower 重启窗口崩溃低但非零失去权威 Leader,本地截断生效
后续选举该 Follower 为 LeaderISR 内选举可达新 Leader 历史缺数据,全集群感知数据丢失

KIP-101 的核心修正:截断点必须由"当前 leader 的任期权威边界"决定,而非"本地观察到的 HW"。本章其余各节展开这个修正的实现。

10.2 Epoch 数据模型

Leader Epoch 是一个 (epoch, startOffset) 二元组的有序表。每个分区维护一份,持久化到 leader-epoch-checkpoint。这个表的形式语义是:"epoch K 覆盖 offset 范围 [startOffset(K), startOffset(K+1))"。它是一个紧凑、可追溯的"任期历史"。

10.3 OffsetForLeaderEpoch 协议

OffsetForLeaderEpoch 是 Kafka 内部 RPC,只在 broker 之间(以及 consumer 检测 leader 切换时)使用。它的语义很简单:"给定 epoch E,告诉我 E 的 endOffset"。但响应里隐藏着一个关键巧思 — 它返回的 epoch 不一定等于请求的 epoch,可能返回"≤ 请求 epoch 的最大已知 epoch",这覆盖了 follower 落后多个任期的情况。

协议字节布局

OffsetForLeaderEpochRequest v4 关键字段:

  ┌────────────────┬──────────────┬─────────────────────┐
  │ replicaId      │ topics[]     │ partitions[]        │
  │ int32          │ array        │  ├ partition int32  │
  │                │              │  ├ currentLEpoch    │
  │                │              │  └ leaderEpoch      │
  └────────────────┴──────────────┴─────────────────────┘

  replicaId           调用方 broker id (consumer 用 -1)
  currentLeaderEpoch  调用方已知的当前 Leader epoch (用于元数据校验)
  leaderEpoch         调用方本地最新 epoch (待查询)

OffsetForLeaderEpochResponse 关键字段:

  ┌────────────────┬──────────────┬─────────────────────┐
  │ errorCode      │ leaderEpoch  │ endOffset           │
  │ int16          │ int32        │ int64               │
  └────────────────┴──────────────┴─────────────────────┘

  errorCode = NONE (0)                  正常返回
            = FENCED_LEADER_EPOCH (74)  请求 epoch 落后
            = UNKNOWN_LEADER_EPOCH (75) 请求 epoch 超前
            = NOT_LEADER (6)            非 Leader 收到查询

错误码含义Follower 动作

10.4 截断场景剖析

截断不是一个频繁动作 — 在 Kafka 长期稳定运行时,你可能几个月看不到一次截断。但一旦发生,它意味着一段"本地有 / 集群无"的孤儿数据要被永久删除。理解触发场景能帮你判断"刚刚发生了什么"。

场景 A:Follower 重启 — 本地超出权威边界

初始状态:
  Leader B (epoch 1) LEO=10
  Follower X (本地 epoch 表):
    (0, 0)
    (1, 5)
    本地 LEO = 8

X 重启:
  1 读本地 epoch=1, LEO=8
  2 OffsetForLeaderEpoch(epoch=1, currentLE=1) → B
  3 B 返回: epoch=1, endOffset=10
  4 比较: 8 < 10 → 保留本地日志
  5 Fetch(offset=8) 继续追赶

X 重启(变种,本地超出):
  1 读本地 epoch=1, LEO=12  (X 之前是分区中 LEO 最大的副本)
  2 OffsetForLeaderEpoch(epoch=1) → B
  3 B 返回: epoch=1, endOffset=10
  4 比较: 12 > 10 → truncateTo(10)
  5 Fetch(offset=10) 继续

场景 B:Leader 切换后旧 Leader 恢复

初始状态:
  Leader A (epoch 1), LEO=10, HW=10
  Follower B, LEO=10

T0: A 网络分区, 仍接受 Producer 写入
    A: LEO=15, 但 follower 未跟上
T1: Controller 选 B 为新 leader (epoch=2, startOffset=10)
    B 开始接受写入, LEO=12
T2: A 网络恢复, 收到 LeaderAndIsr (现在是 follower)
    A 本地: epoch=1, LEO=15
    新 leader: B, epoch=2

A 恢复流程:
  1 OffsetForLeaderEpoch(epoch=1, currentLE=2) → B
  2 B epoch 表: (1, 0), (2, 10) → epoch 1 endOffset = 10
  3 A 比较 LEO(15) > endOffset(10) → truncateTo(10)
  4 A 追加 epoch=2 项: (2, 10)
  5 Fetch(offset=10, epoch=2)
  6 A 现在跟得上 B,加入 ISR

→ A 多写的 5 条 record (offset 10..14) 被永久删除
→ 这是正确的,因为它们从未被复制,不算"已提交"

场景 C:Tiered Storage 的特殊截断

开启 KIP-405 Tiered Storage 后,本地只保留尾部,大部分历史已经 offload 到 S3。这种情况下的截断不再"删除字节",而是"调整 endOffset 标记 + 通知 RLM 重新计算"。详见第 18 章。

10.5 Fetch 携带 epoch 校验

OffsetForLeaderEpoch 解决"截断点",但还不够 — follower 进入正常 Fetch 后,如果元数据再次过期怎么办?Kafka 的方案是每次 Fetch 请求都带上 currentLeaderEpoch,broker 收到后比对,如果不匹配立即返回 FENCED_LEADER_EPOCH,强制 follower 刷新元数据。这让"使用过时 leader 信息继续复制"这种 bug 在协议层就不可能发生。

FetchRequest 字段

FetchRequest v13 (KIP-595 修订)
  replicaId         int32
  maxWaitMs         int32
  minBytes          int32
  maxBytes          int32
  topics[]
    topic           string
    partitions[]
      partition         int32
      currentLeaderEpoch int32   ← 关键字段
      fetchOffset       int64
      lastFetchedEpoch  int32    ← 用于 epoch 链一致性校验
      logStartOffset    int64
      partitionMaxBytes int32

broker 处理伪代码:

  if request.currentLeaderEpoch < partition.leaderEpoch:
      return FENCED_LEADER_EPOCH
  if request.currentLeaderEpoch > partition.leaderEpoch:
      return UNKNOWN_LEADER_EPOCH
  if request.lastFetchedEpoch != null:
      verify epoch chain (KIP-320)
  serveFetch(...)

KIP-320:Fetch 时校验完整 epoch 链

KIP-320 是 KIP-101 的补强 — 它解决"epoch 表跨越多次但 endOffset 仍然能对上"的边角案例。具体做法:Fetch 请求带上 lastFetchedEpoch,leader 不仅校验 currentLeaderEpoch,还校验fetchOffset 对应的 epoch 是不是真的等于 lastFetchedEpoch。 
KIP-320 校验流程:

  Follower epoch 表: (0, 0), (1, 5)
  Follower 本地 LEO = 8
  Follower 发 Fetch: fetchOffset=8, lastFetchedEpoch=1, currentLE=2

  Leader epoch 表: (0, 0), (2, 5)        ← epoch 1 被跳过
  Leader 检查 offset 8 在哪个 epoch:
    8 >= 5 → epoch 2
  比对: lastFetchedEpoch=1 ≠ 2 → 数据不一致 → 返回 OffsetOutOfRange

  Follower 收到错误后, 重新走 OffsetForLeaderEpoch 找截断点

10.6 与 Raft Term 的对比

读到这里你可能想:"这不就是 Raft 的 term 吗?"几乎是,但有几个微妙差异。下表对照三个常见的 leader-based 一致性协议:

概念Kafka Leader EpochRaft TermPaxos Ballot
语义分区 leader 任期整 raft 组任期提案编号
选举者Controller (KRaft 内部用 Raft)Raft 多数派投票Acceptor 多数派
单调性严格递增严格递增严格递增
日志关联每个 epoch 关联 startOffset每条 log entry 带 term每条提案带 ballot
截断时机follower 启动 / 切主append 时校验选主时
RPCOffsetForLeaderEpochAppendEntriesPrepare + Accept

Kafka 选择"分区级别的 epoch"而非"集群级别的 term",是因为 Kafka 的复制单元是分区,不是整个集群。每个分区有独立的 leader 选举,自然有独立的 epoch。这让横向扩展(partition 数)不会带来 term 协调的开销。

典型排障场景

场景:Follower 启动后长时间不进入 ISR 
现象:
  broker-3 重启后, kafka-topics.sh 显示 Isr 仍只有 1,2 多分钟
  broker-3 日志: 频繁收到 UNKNOWN_LEADER_EPOCH

  1 看 controller 是否已经向 broker-3 发送 LeaderAndIsr

  - 优先恢复 controller 健康

  - 如果持续 > 5 min, 重启 broker-3 强制重新拉元数据

endOffsetFor 简化代码

def endOffsetFor(requestedEpoch: Int, logEndOffset: Long): EpochEntry = {
  // 1 已超出最高 epoch → 返回当前 LEO
  if (requestedEpoch > latestEpoch) {
    return EpochEntry(latestEpoch, logEndOffset)
  }

  // 2 找到 ≤ requestedEpoch 的最大 epoch (entry 1)
  val entry1 = epochEntries.findLast(_.epoch <= requestedEpoch)
    .getOrElse(return EpochEntry(-1, -1))

  // 3 找到下一个 epoch (entry 2)
  val entry2 = epochEntries.find(_.epoch > entry1.epoch)

  // 4 endOffset = entry2.startOffset(如有), 否则 LEO
  val endOffset = entry2.map(_.startOffset).getOrElse(logEndOffset)
  EpochEntry(entry1.epoch, endOffset)
}

truncate 实现

def truncateTo(targetOffset: Long): Unit = lock.synchronized {
  if (targetOffset >= logEndOffset) return    // 已经在范围内

  // 1 截断 segments(可能删除整个 segment)
  segments.truncate(targetOffset)

  // 2 截断 epoch 表(移除 startOffset > targetOffset 的所有 entry)
  leaderEpochCache.foreach(_.truncateFromEnd(targetOffset))

  // 3 重新计算 LEO
  recoveryPoint = math.min(recoveryPoint, targetOffset)
  logEndOffsetMetadata = LogOffsetMetadata(targetOffset)

  // 4 刷新 producer state snapshot(KIP-360)
  producerStateManager.truncateFullyAndStartAt(targetOffset)
}

10.9 设计取舍

① 任期权威 vs 本地观察

截断点来自当前 leader 的"任期权威边界",而非 follower 自己看到的 HW。代价是多一次 RPC(OffsetForLeaderEpoch),但杜绝了"无声丢消息"的可能。

② 分区级 vs 集群级 epoch

每个分区独立 epoch,而非整个集群一个 term。让横向扩展不带来 term 协调开销,但每个分区单独维护 epoch 表(几 KB · 几乎无成本)。

③ 一并校验 Fetch 中的 epoch

每次 Fetch 都带 currentLeaderEpoch,broker 不匹配立即拒绝。代价是 RPC 多 4 字节,收益是协议层杜绝"用过时元数据继续复制"的 bug。

④ KIP-320 的链校验

Fetch 也带 lastFetchedEpoch,即使 endOffset 对得上也要求 epoch 链一致。这是对极端边角案例的兜底。

FENCED_LEADER_EPOCH 是机制,不是错误broker 日志当报错处理
Kafka epoch 是分区级别与 Raft term 不同误把 cluster 当统一 term

本章小结

Leader Epoch 是 Kafka 在 0.11 版本之后的"正确性基石" — 它把"截断到哪里"这个决定从 follower 的本地观察,移到了 leader 的任期权威。本质上是承认一个事实:只有当前 leader 知道"哪些数据真正属于这一任期"。OffsetForLeaderEpoch RPC、FENCED_LEADER_EPOCH 错误码、Fetch 携带 epoch 校验,三者环环相扣,让"用过时元数据继续复制"在协议层就不可能发生。下一章 KRaft 会把这个机制再向上抽象一层:不只分区,整个 controller 元数据本身也用类似的 epoch / term 来管理,Kafka 终于不再需要 ZooKeeper。
