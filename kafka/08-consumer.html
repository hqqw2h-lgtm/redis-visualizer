第 8 章 · 消费者与组协议

Consumer Group · GroupCoordinator · JoinGroup/SyncGroup · 四种 assignor · CooperativeSticky · OffsetCommit · 三大超时 · Static Membership · KIP-848 心跳推进式 · 监控护栏 · 源码地图

Consumer Group 是 Kafka 在 "一份数据广播给多个业务系统" 与 "单业务系统横向扩容" 之间的统一抽象。Producer 的复杂度集中在批次与路由,Broker 的复杂度集中在 log 与副本,而消费侧的复杂度全部压在 GroupCoordinator:成员加入、分区分配、心跳维护、offset 提交、成员变化时的 rebalance 调度。所有这些状态都坐落在一个 broker 上,所有这些请求都要在毫秒级内做出仲裁,所有这些超时都要在网络抖动与业务慢处理之间画一条恰当的线。

本章以 Apache Kafka 3.7 为基线,沿 "语义模型 → Coordinator 状态机 → JoinGroup/SyncGroup → 分配策略 → offset 提交 → 配置 → 监控 → static membership → KIP-848 → 陷阱" 这条主线一路深挖。我们不会停留在 "group.id 决定边界" 这种表面结论,而是逐项追问:为什么 Coordinator 选 __consumer_offsets 分区 leader?为什么分配算法跑在客户端 leader 而非 broker?为什么 CooperativeSticky 把 rebalance 拆成两轮?为什么 session.timeout.ms 与 max.poll.interval.ms 治理的是完全不同的失败?以及 KIP-848 凭什么用一个 ConsumerGroupHeartbeat RPC 替代了 Classic 的三段式握手。

  • 同 group.id 互斥 / 跨 group.id 广播的二元语义,以及 "组并行度上限 = 分区数" 的硬约束
  • GroupCoordinator 由 __consumer_offsets 分区 leader 决定 — 路由公式 hash(group.id) % offsetsTopicPartitionCount
  • 组状态机:Empty → PreparingRebalance → CompletingRebalance → Stable → Dead 的五态迁移
  • Classic 协议两阶段:JoinGroup 收集订阅、SyncGroup 下发分配,以及 "为什么分配在客户端 leader"
  • 四种 assignor 对比:Range / RoundRobin / Sticky / CooperativeSticky(生产推荐)
  • Eager vs Cooperative 的二轮 rebalance 时序 — 全局屏障 vs 局部撤销 + 增量分配
  • OffsetCommit 写 __consumer_offsets compacted topic:key = (group, topic, partition)、value = (offset, leaderEpoch, metadata, ts)
  • 四种提交方式取舍:auto-commit / commitSync / commitAsync / 分区级提交,以及 ILLEGAL_GENERATION 与 REBALANCE_IN_PROGRESS 的恢复策略
  • 三大超时分工:heartbeat.interval.ms(心跳频率)/ session.timeout.ms(失联检测)/ max.poll.interval.ms(慢处理保护)
  • 四条配置约束:心跳 ≤ session/3、max.poll.interval 覆盖 P99、records × 单条耗时 < poll interval、rebalance.timeout 覆盖 revoke
  • 消费者组健康四维 — Lag / Rebalance / Poll Loop / Commit,以及 records-lag-max 等核心 JMX 指标
  • Static Membership(group.instance.id)如何在滚动发布期间抑制 rebalance,以及 FENCED_INSTANCE_ID 的根因
  • KIP-848 服务端 assignor + 心跳推进式 rebalance 的迁移路径与监控指标变化
  • 十大常见陷阱 — 自动提交跳消息 / 回调慢 IO / session 过小 / RoundRobin 倾斜 / 重复 instance ID 等 — 的现象、根因、护栏指标与处理动作

前置知识:第 1 章 Kafka 数据流(topic / partition / broker 模型);第 2 章 Log 引擎(分区即一条日志);第 5 章 zero-copy 与 PageCache(决定 Fetch 路径性能);第 7 章 Partitioning(producer 端分区路由,与 consumer 端分区消费一一对照)。本章是 "consumer 侧" 的精解,后续第 9 章 ISR 副本基于 leader epoch 与 HW 给 OffsetCommit 写路径提供容错语义,第 13 章 Exactly-Once 在本章 "至少一次" 的基础上引入 transactional consumer。

主循环里有三种 "状态变化驱动力":成员上线下线、订阅集合变更、Coordinator 端 __consumer_offsets 分区迁移。任何一种触发,主循环都会从 Fetch 跳到 Rebalance,走完一轮 JoinGroup → SyncGroup 后再回到 Fetch。理解了这条主循环,后续每节都可以放在 "驱动力 + 转移路径" 的框架下去看。

地图三:三句话结论

本章三句话:

  1 同 group.id 内分区互斥消费
    不同 group.id 之间广播
    组内并行度上限 = 分区数(consumer 多于分区时,多出的实例 idle)

  2 GroupCoordinator 维护组状态机

  3 三个超时各管一件事
    heartbeat.interval.ms 心跳频率(默认 3 s)

    max.poll.interval.ms  慢处理保护(默认 300 s,主动离组)
关键洞察: Kafka 的消费者协议把 "谁正在消费哪个分区" 这个分布式状态完全收敛到 GroupCoordinator 上,而 Coordinator 又复用了已有的 __consumer_offsets compacted topic 做持久化。换句话说,Coordinator 既是仲裁器,又是日志写手 — 这个 "一鱼两吃" 的设计让 Kafka 不必引入额外的元数据存储,但也意味着 __consumer_offsets 分区 leader 抖动会同时影响该组的所有 consumer。

8.1 Consumer Group 的语义模型

语义模型是后续所有讨论的地基:它决定一条消息会被几个进程看到、决定扩容的硬上限、决定 lag 怎么读。Kafka 选择的语义在工程上极简 — 只有两件事:同组互斥、跨组广播。

8.1.1 互斥消费与广播消费

图 8-1-1 · 一个 Topic 被两个 group 同时消费的拓扑

同一条消息可以被 order-service 与 audit-service 各看一次,这是广播;而 order-service 内部的 consumer-a 与 consumer-b 永远不会同时拿到 P0 这条记录,这是互斥。互斥的实现机制就是分区分配 — 每个分区在某一代 (generation) 至多属于一个成员。

8.1.2 同组与跨组对比

表 8-1-1 · 同 group / 跨 group 的五个维度对比
维度同 group.id不同 group.id
消费方式组内互斥(同一分区只由一个成员消费)组间广播(每组独立看完整流)
扩容方式增加 consumer 实例分担分区新建消费者组

kafka-consumer-groups.sh \
  --bootstrap-server kafka-0:9092 \

order-service  order-created  0          1205            1210            5    consumer-a-...  10.0.0.11  order-a
order-service  order-created  1          1180            1180            0    consumer-a-...  10.0.0.11  order-a

读该输出关注三个量:CURRENT-OFFSET 是已提交位置(下一条要消费的 offset);LOG-END-OFFSET 是分区当前 HW 之前的末尾位置;LAG = LOG-END-OFFSET − CURRENT-OFFSET 是未处理消息数。如果 CONSUMER-ID 为空,通常意味着组目前无活跃成员,但仍有已提交 offset 留存 — 这本身不是错误,而是一个 "已经下班的服务" 的常态。

常见误解: "把分区数设得多一点,以后扩容方便" — 没错,但代价是每个 Consumer 都要为每个分区维持 fetch session 与 offset 存储,broker 端 Coordinator 也要为更多 (group, topic, partition) 写 compacted record。分区数过多会拖累 Coordinator 与控制面元数据,所以分区数的选择应基于第 7 章的估算公式而非 "尽量大"。

8.1.4 并行度公式

组内并行度:

    parallelism = min(consumer_count, partition_count)

  consumer_count = 6, partition_count = 4
    → 4 个 consumer 各拿一个分区,2 个 consumer idle

  consumer_count = 3, partition_count = 12
    → 平均 4 分区 / consumer,但实际由 assignor 决定具体映射

  consumer_count = 1, partition_count = N
    → 单实例消费 N 个分区,顺序按分区独立保证

8.2 GroupCoordinator 与组状态机

语义说完该看实现。GroupCoordinator 不是一个独立进程,而是某 broker 上的逻辑角色 — 这个角色由 __consumer_offsets 这个内部 compacted topic 的分区 leader 来兼任。理解 Coordinator 的路由方式,才能理解为什么一个 broker 故障会让若干消费者组同时进入 rebalance。

8.2.1 Coordinator 路由

图 8-2-1 · 从 group.id 到 Coordinator broker 的两步映射
int partition = Utils.toPositive(groupId.hashCode())
                % offsetsTopicPartitionCount;
Broker coordinator = leaderOf("__consumer_offsets", partition);

这段 12 行代码决定了一件大事:同一个 group.id 在集群内只有一个 Coordinator。当 __consumer_offsets 分区的 leader 发生切换(broker 故障 / 主动 reassign),该组的 Coordinator 也随之迁移到新 leader broker。客户端通过 FindCoordinator 请求获取当前负责该组的 broker,失败时收到 NOT_COORDINATOR 错误并重试。

8.2.3 状态语义与请求行为

表 8-2-1 · 五个状态下 Coordinator 与客户端的请求行为
状态Coordinator 行为客户端典型请求行为

8.2.4 Coordinator 端调用路径

图 8-2-3 · 一个 JoinGroup 请求在 broker 端的调用链

排障时常见的拒绝原因可以归为五类:

  • NOT_COORDINATOR — 请求发到了旧的 broker,客户端需重新 FindCoordinator。
  • ILLEGAL_GENERATION — 客户端 generation 已过期(自己还在 N,组已进入 N+1)。
  • UNKNOWN_MEMBER_ID — Coordinator 不认识这个 member.id,通常因为 session 已超时被踢。
  • FENCED_INSTANCE_ID — 同一个 group.instance.id 有新实例进来,旧的被 fence。
  • REBALANCE_IN_PROGRESS — 心跳期间组进入了 PreparingRebalance,需要重新 JoinGroup。
关键洞察: 这五个错误码不是孤立的 — 它们对应组状态机里的不同迁移路径。看到 ILLEGAL_GENERATION 不要简单重试,要先重走 JoinGroup;看到 FENCED_INSTANCE_ID 必须停止当前实例并报警,因为它意味着同名实例已上线。

8.3 JoinGroup 与 SyncGroup

Classic 协议把 "决定谁拿哪个分区" 这件事拆成了两段:第一段 JoinGroup 让 Coordinator 收齐成员名册并选出 Leader;第二段 SyncGroup 让 Leader 跑分配算法并把结果分发给所有 follower。两段中间夹着一个 "全局屏障" — 所有人都 join 完才能 sync。理解这个屏障,就能理解为什么大组容易被慢成员拖死。

8.3.5 为什么分配在客户端 Leader

设计决策:
  Coordinator 仅负责收集成员、广播分配
  分配算法在客户端 Leader 执行

理由:
  1 允许应用通过 ConsumerPartitionAssignor 自定义分配逻辑
  2 应用可以携带 metadata (如机架、IP) 影响分配
  3 broker 不必内置所有可能的 assignor 实现

代价:
  1 全局屏障: 所有成员必须 JoinGroup 后才能 SyncGroup
  2 大组被慢成员拖累
  3 Leader 故障需重新选举 Leader

KIP-848 改进:
  把可标准化部分收回 Coordinator (服务端 assignor)
  保留服务端可配置 assignor 接口
  消除全局屏障 (心跳推进式 rebalance)
真实案例 · 1200 个 consumer 的全局屏障

某物联网平台单组规模约 1200 个 consumer,使用 Classic 协议 + StickyAssignor。每次滚动升级,先撤销的成员 onPartitionsRevoked 平均耗时 8 秒(回调里同步 flush 本地 RocksDB)。因为是全局屏障,整个组等最慢成员,实测 rebalance 端到端时间约 12-15 秒,期间所有 1200 个实例都停止消费。改用 CooperativeSticky 后,撤销集合缩小到约 1/N,但全局屏障仍存在;迁移到 KIP-848 后,屏障被心跳推进式 rebalance 取代,端到端 P99 降到 2 秒。

8.4 分配策略与 rebalance 停顿

"谁拿哪个分区" 这件事,有四种实现各自代表了一种工程权衡:Range 简单但容易倾斜,RoundRobin 均匀但要求订阅一致,Sticky 在均匀的同时保留上次分配,Cooperative 把停顿从 "全员" 缩小到 "迁移者"。生产环境的默认建议是 CooperativeStickyAssignor。

8.4.5 Cooperative Rebalance 时序

图 8-4-2 · 新成员 C2 加入时的两轮 Cooperative Rebalance

Cooperative Rebalance 的核心改进:第一轮仅撤销需要迁移的分区,未变动分区继续消费;第二轮再分配新 owner。该设计将 "全员停止 → 重新分配" 的全局屏障改为 "局部撤销 + 增量分配",显著降低 rebalance 期间的吞吐损失。

8.4.6 Eager vs Cooperative

Eager(Range / RoundRobin / Sticky)
  • 撤销范围:全部分区
  • Rebalance 轮数:1 轮
  • 吞吐影响:全员停顿
  • 实现复杂度:简单,状态干净
  • 适用:小组、低频 rebalance
Cooperative(CooperativeSticky)
  • 撤销范围:仅迁移分区
  • Rebalance 轮数:2 轮
  • 吞吐影响:仅迁移期间局部停顿
  • 实现复杂度:需 assignor 与回调正确处理 "部分撤销"
  • 适用:大组、生产推荐
切换条件: Cooperative 协议要求所有成员同时使用 CooperativeStickyAssignor。一旦组里混入一个 Range/RoundRobin/Sticky 的成员,整组降级为 Eager。滚动升级时务必按 "全员就绪后再启用 Cooperative" 的顺序。

8.5 OffsetCommit:组进度的存储

"消费到哪了" 需要被持久化,Kafka 选择把它写进自己 — 一个名为 __consumer_offsets 的 compacted topic,key 由 (group, topic, partition) 三元组构成。Compaction 保证每个 (group, partition) 只保留最新 offset,过期记录被压实清理。这是一种 "用自己实现自己" 的优雅设计。

8.5.1 OffsetCommit 写入路径

图 8-5-1 · 从 commitSync 到 compacted record 的完整路径

关键事实: Compaction 保证每个 (group, partition) 只保留最新 offset。__consumer_offsets 是 compacted topic,key 包含 group + topic + partition,value 包含已提交 offset、leader epoch、应用 metadata 与提交时间戳。

8.5.2 手动提交代码模板

while (running) {
    ConsumerRecords<String, OrderEvent> records =
        consumer.poll(Duration.ofMillis(500));
    for (ConsumerRecord<String, OrderEvent> record : records) {
        handleOrder(record.value());
    }
    consumer.commitSync();
}

提交语义: commitSync 提交 "下一条要消费的 offset",即已成功处理的最后一条 offset + 1。处理失败时不提交,下次 poll 从未提交位置重新拉取(可能重复消费,但不丢失)。这是 Kafka "至少一次" 语义的核心。

8.5.3 CLI 验证 offset

8.5.6 OffsetCommit 校验时序

图 8-5-3 · commitSync 在 broker 端的校验时序

提交失败的常见原因:提交发生在 rebalance 窗口,应用仍在处理上一代 assignment,但 Coordinator 已进入新 generation。正确动作是先完成新一轮 poll() 与 assignment 回调,再提交当前仍拥有的分区进度。盲目重试旧 offset 可能覆盖新 owner 已经提交的更新位置,造成 "offset 倒退"。

关键洞察: "Kafka 提交的不是已处理 offset,而是下一条要消费的 offset" — 这个 +1 偏移让 "重启从已提交位置继续消费" 这条语义在代码上极简(直接 seek 到提交值即可),但也意味着 commitAsync 异步回调如果乱序覆盖,会让消费者悄悄退回到老位置而无报错。生产中通常对 commitAsync 加 "回调里只在 offset 单调递增时才接受" 的护栏。

8.6.4 超时关系约束

约束 1: heartbeat.interval.ms < session.timeout.ms / 3
  典型: 3000 < 45000 / 3 = 15000  ✓
  违反时: 心跳间隔过大, broker 在收到下一次心跳前已判定 session 超时

约束 2: max.poll.interval.ms 必须覆盖业务最慢批次
  典型: 业务 P99 100 秒 → max.poll.interval.ms ≥ 200 秒
  违反时: 慢批次触发主动离组, 引发 rebalance

约束 3: max.poll.records × 单条处理时间 < max.poll.interval.ms
  典型: 500 records × 200 ms = 100 秒 < 300 秒  ✓
  违反时: 单批处理超时, 主动离组

约束 4: rebalance.timeout.ms 应覆盖最慢成员的 onPartitionsRevoked
  典型: 业务 revoke 耗时 P99 30 秒 → rebalance.timeout.ms ≥ 60 秒
  违反时: rebalance 失败, 进入下一轮
真实案例 · session 太短被网络抖动反复踢

某金融交易系统跨 AZ 部署 Kafka 与 consumer,默认 session.timeout.ms=10000。机房间 RTT P99 约 150 ms,偶尔抖动到 200-300 ms;某次抖动持续 12 秒,所有 consumer 心跳延迟超过 10 秒被 Coordinator 全部踢出,组进入 PreparingRebalance。实际 consumer 进程健康,但被网络误判。调到 session.timeout.ms=45000(默认值)后,同等抖动不再触发 rebalance,生产 rebalance 频率从 12 次 / 小时降到 0.3 次 / 小时。

8.8 Static Membership 与滚动发布

动态成员的设计假设是 "consumer 实例随时来去",所以每次 JoinGroup 都会被分配新 member.id,组也因此进入 PreparingRebalance。但在 K8s StatefulSet 这种 "身份稳定的滚动发布" 场景下,这种假设反而成了麻烦 — 每次 Pod 滚动重启,都触发完整 rebalance。Static Membership 用 group.instance.id 把 "身份" 钉死,让短暂离线的实例回归时不需要 rebalance。

8.8.4 Static Membership 的正确滚动顺序

推荐滚动重启顺序:

  1 maxSurge=1, maxUnavailable=0
    每次仅一个 Pod 重启

  2 重启前等待新 Pod ready
    避免双副本短暂同 instance ID 在线

  3 旧 Pod terminationGracePeriodSeconds:
    至少覆盖一次 commit + LeaveGroup
    典型 30-60 秒

  4 新 Pod 启动后:
    若 session.timeout.ms 内回归 → 保留分配
    若超时 → 触发 rebalance, 新分配生效

避坑要点:
  1 重复 instance ID 会触发 FENCED_INSTANCE_ID
    新实例覆盖旧实例, 旧实例无法继续消费
  2 不能用动态生成的 UUID 作 instance ID
    每次重启 ID 变化, 失去 static 优势
  3 实例 ID 命名空间隔离
    不同环境 (dev/staging/prod) 不复用 ID
关键避坑: 在 K8s 中如果同时跑两个副本数 ≥ 1 的 Deployment,且都用同样的 POD_NAME 模板生成 group.instance.id — 两个 Deployment 的 order-worker-0 不一样,但跨 namespace 互不感知。如果两个 Deployment 共用同一个 group.id,会出现两边互相 fence 的死锁。生产中 instance ID 命名建议加上 namespace 或集群标识前缀。

8.9 KIP-848 新协议

Classic 协议解决了 Kafka 早期 "客户端自由分配" 的需求,但十年过去发现:绝大多数生产部署都只用内置的几个 assignor,真正自定义的极少;而 Classic 引入的 "全局屏障 + 客户端 leader" 代价巨大。KIP-848 把分配权回收到 Coordinator,用一个 ConsumerGroupHeartbeat RPC 替代三段式握手,把 rebalance 从 "事件式" 改为 "心跳推进式"。

8.9.3 协议设计取舍

被拒绝的方案及其原因:

  方案 A: 每个 consumer 自己抢分区锁
    优点: 不需要 Coordinator
    拒绝: 分布式锁 + 故障检测复杂, 脑裂导致重复消费

  方案 B: Broker 第一版起完全负责分配
    优点: 客户端简化
    拒绝: 旧生态需 ConsumerPartitionAssignor 自定义, broker 难兼容所有策略

  方案 C: Eager rebalance 一次撤销全部
    优点: 实现简单, 状态干净
    拒绝: 大组或慢回调全组停顿, cache 失效

  方案 D: Cooperative rebalance
    优点: 减少停顿与迁移
    限制: 需 assignor 与回调正确处理"部分撤销", 混用旧策略可能出错

  方案 E: Static Membership
    优点: 滚动发布稳
    限制: 不适合高弹性短生命周期 worker, 重复 ID 互相 fence

  方案 F: KIP-848 新协议
    优点: 服务端分配, 成员独立推进
    限制: 需 broker / client / 监控同步升级, 老集群不能仅改客户端

8.9.4 迁移路径

阶段 1: 升级 Broker 到 3.7+
  __consumer_offsets compaction policy 兼容
  Coordinator 同时支持 classic + consumer 协议

阶段 2: 升级客户端到 3.7+
  保持 group.protocol=classic (默认)
  验证 classic 协议下行为不变

阶段 3: 灰度迁移
  按消费者组逐组切换 group.protocol=consumer
  对比 rebalance 频率与 lag 变化

阶段 4: 全量切换
  所有消费者组使用 consumer 协议
  classic 路径作为 fallback 保留

注意:
  迁移期间监控指标名称变化
  rebalance-rate-per-hour 在新协议下含义不同
  服务端 assignor 配置入口为 group.consumer.assignors
关键洞察: KIP-848 的本质不是 "改写协议",而是 "把客户端 Leader 这个临时仲裁角色拿掉"。Classic 协议里,客户端 leader 是个让人头疼的存在:它是普通 consumer,但承担集群级仲裁;它故障了组就重选;它慢了组就等。新协议把这件事还给 Coordinator,逻辑回到了 "唯一仲裁器" 的简洁状态,代价是 broker 需要内置 assignor 实现。

8.11.3 ConsumerCoordinator.poll 简化

// clients/.../ConsumerCoordinator.java — 简化伪代码
public boolean poll(Timer timer, boolean waitForJoinGroup) {
    maybeUpdateSubscriptionMetadata();

    if (subscriptions.hasAutoAssignedPartitions()) {
        if (rejoinNeededOrPending()) {
            if (!ensureCoordinatorReady(timer)) return false;
            if (!ensureActiveGroup(timer)) return false;
        }
    }
    pollHeartbeat(timer.currentTimeMs());
    maybeAutoCommitOffsetsAsync(timer.currentTimeMs());
    return true;
}

这段代码体现了 "普通 Fetch 不走入组路径" 的设计:只有当 rejoinNeededOrPending() 为 true 时,才会进入 ensureCoordinatorReady 与 ensureActiveGroup。稳态消费的开销主要在 pollHeartbeat(定时发心跳)与 maybeAutoCommitOffsetsAsync 两条轻量路径上。

8.11.4 设计取舍

取舍 1:协议开销集中在组变化时

ensureActiveGroup 仅在需分配时触发。普通 Fetch 不必每次入组。该选择让协议开销集中在组变化时,稳态时几乎无开销。代价是组变化时需走完整 JoinGroup + SyncGroup + 回调流程 — 这恰好是 KIP-848 想优化的痛点。

取舍 2:回调在用户线程执行

onPartitionsRevoked / onPartitionsAssigned 在用户的 poll() 线程内同步执行。该设计避免了多线程同步复杂度,但要求回调耗时受控。生产中常见反例:回调内执行同步 DB flush 或外部 RPC,放大 rebalance 停顿到秒级甚至分钟级。

取舍 3:OffsetCommit 写 compacted topic

Offset 提交写入 __consumer_offsets(compacted topic),而非外部 KV 存储。该选择利用了 Kafka 已有的复制 + compaction 机制,无需引入新存储。代价是 Coordinator broker 同时承担数据 partition leader 与 offset 写入,需独立监控其负载。

取舍 4:分配在客户端 Leader 而非 Coordinator

把 assignor 留给应用自定义,是 Kafka 早期 "可扩展协议" 哲学的体现。十年后回看,99% 的生产部署只用四个内置 assignor — KIP-848 终于回到 "服务端默认,客户端可选" 的形态。

章末总结

第 8 章把 "消费侧的全部分布式状态" 在 12 节里讲完。回顾这条主线:

  • §8.1 语义 明确了组内互斥、跨组广播两条规则,以及 "分区数 = 组并行上限" 这条硬约束。
  • §8.2-8.3 Coordinator + Classic 协议 把 "谁仲裁、谁分配、谁通知" 三件事拆给 broker 与客户端 leader。
  • §8.4 分配策略 给出了四种 assignor 的取舍,以及 CooperativeSticky 把 rebalance 拆成两轮的工程价值。
  • §8.5 OffsetCommit 解释了 "自己实现自己" 的 compacted topic 存储 + 四种提交方式的取舍。
  • §8.6-8.7 配置 + 监控 把三大超时与四维健康指标落到了可抄走的基线上。
  • §8.8 Static Membership 让滚动发布在 K8s 时代不再频繁触发 rebalance。
  • §8.9 KIP-848 把仲裁权收回 Coordinator,用心跳推进式 rebalance 消除全局屏障。
  • §8.10-8.11 陷阱 + 源码 给出了十大常见事故的护栏指标与源码定位。
