第 23 章 · 安全体系

TLS · SASL/SCRAM · mTLS · OAUTHBEARER · ACL · StandardAuthorizer · Audit · Listener 隔离 · 证书轮换

"Kafka 集群对内网开放" — 这句话在生产事故复盘里出现得太多了。一个不开认证的 Kafka 集群,任何能 ping 通 9092 的进程都能 produce / consume / 删 topic。Kafka 的安全不是"能连上",而是每一次连接、读写、管理操作都能回答四个问题:是谁、从哪里来、能不能做、做了什么。这四件事分别对应:认证(Authentication)、网络层访问控制、授权(Authorization)、审计(Audit)。

本章按"威胁模型 → 四层安全 → Listener 隔离 → TLS → SASL → ACL → Audit → 证书轮换 → 性能 → 反例"组织,对应 Apache Kafka 3.7 的 StandardAuthorizer、SaslServerAuthenticator、SslTransportLayer 实现。生产部署的目标是"每一层都可独立验证"。

  • 四层安全:加密 / 认证 / 授权 / 审计
  • Listener 隔离 — INTERNAL/EXTERNAL/CONTROLLER 各走不同协议
  • TLS 协议(传输加密 + 可选 mTLS)
  • SASL 四种机制:PLAIN / SCRAM-SHA-256 / GSSAPI(Kerberos) / OAUTHBEARER
  • StandardAuthorizer + ACL 资源模型(Topic / Group / Cluster / TransactionalId)
  • "默认拒绝" vs "默认放行" 的安全含义
  • 证书/密码轮换的具体步骤
  • SSL CPU 飙升 + 如何卸载到 LB

TLS 1.3 基础;Kerberos / OAuth 概念;RBAC vs ABAC;基本网络安全(防火墙、零信任);第 15 章 SocketServer / Listener。

23.1 四层安全模型

威胁对策层落地配置
中间人嗅探加密SASL_SSL listener + 公司 CA
客户端伪装认证SASL/SCRAM、mTLS、OAUTHBEARER + JWT
横向越权授权StandardAuthorizer + 最小 ALLOW

23.2 Listener 隔离

把不同流量走不同 listener,各 listener 配不同安全协议。这是 Kafka 最简单也最有效的隔离手段。

listeners=INTERNAL://0.0.0.0:9092,EXTERNAL://0.0.0.0:9093,CONTROLLER://0.0.0.0:9094
listener.security.protocol.map=INTERNAL:SASL_PLAINTEXT,EXTERNAL:SASL_SSL,CONTROLLER:SSL
inter.broker.listener.name=INTERNAL
controller.listener.names=CONTROLLER
advertised.listeners=INTERNAL://broker-1.internal:9092,EXTERNAL://kafka.company.com:9093
Listener协议用途
INTERNALSASL_PLAINTEXTbroker 间 + follower fetch(同 VPC,信任网络)
EXTERNALSASL_SSL外部 client(需要 TLS + 认证)
CONTROLLERSSL(+ mTLS)KRaft 控制面(最敏感,锁死)

23.3 TLS 加密

# broker 端
listeners=EXTERNAL://0.0.0.0:9093
listener.security.protocol.map=EXTERNAL:SSL
ssl.keystore.location=/etc/kafka/ssl/server.keystore.jks
ssl.keystore.password=...
ssl.key.password=...
ssl.truststore.location=/etc/kafka/ssl/server.truststore.jks
ssl.truststore.password=...
ssl.client.auth=required          # mTLS, 强制客户端证书
ssl.enabled.protocols=TLSv1.3,TLSv1.2
ssl.cipher.suites=TLS_AES_128_GCM_SHA256,TLS_AES_256_GCM_SHA384
TLS 握手在 Network Thread 上做,会消耗 CPU(参见 15 章 SSL 握手生产惨案)。生产场景:① 长连接复用减少握手次数;② 在 LB 层卸载 TLS;③ 适当增加 num.network.threads。

23.4 SASL 认证

机制含义适合
SASL/PLAIN用户名密码明文(需要 SSL 配合)简单场景,与 LDAP 集成
SASL/SCRAM-SHA-256挑战-响应,密码哈希无外部依赖,生产推荐
SASL/GSSAPI(Kerberos)Kerberos ticket已有 Kerberos 基础设施
SASL/OAUTHBEAREROAuth2 JWT云原生,与 IdP 集成
mTLS客户端证书服务间,无密码

SCRAM 配置(生产最常用)

# broker
sasl.enabled.mechanisms=SCRAM-SHA-256
sasl.mechanism.inter.broker.protocol=SCRAM-SHA-256
listener.security.protocol.map=EXTERNAL:SASL_SSL

# 添加用户
kafka-configs --bootstrap-server ... --alter \
  --add-config 'SCRAM-SHA-256=[password=secret123]' \
  --entity-type users --entity-name order-svc

# 客户端
sasl.mechanism=SCRAM-SHA-256
sasl.jaas.config=org.apache.kafka.common.security.scram.ScramLoginModule required \
  username="order-svc" password="secret123";
security.protocol=SASL_SSL

23.5 ACL 授权

Kafka 4.x 后默认 Authorizer 是 StandardAuthorizer(KRaft 原生,ACL 存在 metadata 里)。Kafka 3.x ZK 版用 AclAuthorizer。

资源操作说明
TopicRead / Write / Create / Delete / Alter / Describe / DescribeConfigs / AlterConfigs主要业务资源
GroupRead(consume)/ Describe / DeleteConsumer Group
ClusterCreate / Alter / Describe / IdempotentWrite / ClusterAction(broker 间)集群级
TransactionalIdWrite / DescribeEOS 事务
DelegationTokenDescribe委托认证
 
# 给 order-svc 配 Produce + Consume 权限
kafka-acls --bootstrap-server ... --add \
  --allow-principal User:order-svc \
  --operation Write --topic orders

kafka-acls --bootstrap-server ... --add \
  --allow-principal User:order-svc \
  --operation Read --topic orders \
  --operation Read --group order-consumer-group

# 通配符
kafka-acls --bootstrap-server ... --add \
  --allow-principal User:metrics-collector \
  --operation Describe --topic '*'
默认拒绝:配 allow.everyone.if.no.acl.found=false(Kafka 4.x 默认)。这样未授权操作会立即报错(可见的失败),而不是默默放行(静默漏洞)。生产环境绝不打开 default allow。

23.6 Audit 审计

<!-- log4j2.xml -->
<Logger name="kafka.authorizer.logger" level="INFO" additivity="false">
  <AppenderRef ref="authorizerAppender"/>
</Logger>
<Appender name="authorizerAppender" type="File"
          fileName="${kafka.logs.dir}/kafka-authorizer.log">
  <PatternLayout pattern="[%d] %p %m (%c)%n"/>
</Appender>

每次 ALLOWED / DENIED 都会打到这个 logger。生产场景把它接到 SIEM(Splunk / ELK / Datadog),保留 90+ 天用于合规审计。

23.8 性能影响

项影响
TLS 1.3 握手RTT 1 次(比 TLS 1.2 少一次),CPU 主要在密钥协商
SSL 数据帧AES-GCM 硬件加速,CPU 影响 5-15%

23.9 反例与排障

反例 1: "allow.everyone.if.no.acl.found=true"
  现象: 漏配 topic 的 ACL → 所有人能读写
  修复: 改成 false, 现网逐 topic 补 ACL

反例 2: "用 DENY 当封禁"
  现象: 几百条 ACL 包含 DENY, 权限不可推理
  修复: 删除 DENY, 改用 ALLOW 显式列举

反例 3: "证书过期才发现"
  现象: ssl handshake error 突然爆
  修复: 监控 cert expiration, 提前 90 天告警

反例 4: "INTERNAL listener 也走 SSL"
  现象: broker 间 SSL CPU 飙
  修复: 内网 listener 用 PLAINTEXT (信任网络)

23.10 本章要点

本章小结

Kafka 安全是一个"四层栈" — 缺任何一层都不算"安全集群"。加密用 TLS,认证用 SASL/SCRAM 或 mTLS,授权用 StandardAuthorizer + ACL,审计用 authorizer logger + SIEM。生产实践三条铁律:① 默认拒绝(allow.everyone.if.no.acl.found=false);② DENY 只做应急封禁,长期用 ALLOW;③ Listener 隔离 — INTERNAL 走 PLAINTEXT(内网信任),EXTERNAL 走 SASL_SSL,CONTROLLER 锁死 mTLS。下一章我们看 Eligible Leader Replicas (ELR),这是 KIP-966 引入的、解决"unclean leader election 数据丢失"的新机制。
