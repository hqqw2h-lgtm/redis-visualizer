第 20 章 · 配额管理 Quotas

ClientQuotaManager · ControllerQuotaManager · 四类配额 · 8 层匹配 · 滑动窗口 · throttleTimeMs · 多租户隔离

"一个客户端用 ETL 半夜把全表导进 Kafka,导致同集群的实时业务全部抖动" — 这是多租户 Kafka 集群最常见的故障模式。Kafka Quotas 是协议层的"刹车" — 它不依赖 OS cgroup,而是在 broker 请求处理路径上识别 (user, client-id),对 4 类资源(Produce 字节、Fetch 字节、Request 时间、Controller 元数据变更)分别限流。被治理的客户端不是被 拒绝,而是被 延迟响应 — broker 在响应里塞一个 throttle_time_ms 字段,客户端 SDK 自然按这个值降速。

Quotas 是治理多租户集群的第一道防线。本章按"四类配额 → 8 层匹配 → 字节配额实现 → 请求时间配额 → Controller 配额 → 限流生命周期 → 滑动窗口 → 监控 → 取舍 → 陷阱"组织,对应 Apache Kafka 3.7 的 ClientQuotaManager。

  • 四类 Quota:produce_byte_rate / consumer_byte_rate / request_percentage / controller_mutation_rate
  • 身份模型:(user, client-id) 8 层匹配优先级
  • "延迟响应而非拒绝"的协议设计 + throttle_time_ms 字段
  • broker 局部计量 — 集群总 = N × broker 配额
  • 滑动窗口算法:Sensor + KafkaMetric + 时间衰减
  • Controller 配额对 CreateTopics / DeleteTopics 等元数据风暴的防护
  • chann mute + Purgatory DelayedThrottle 实现细节
  • 常见反例:对 ETL 限不住,对实时业务误伤

第 8 章 Consumer Group;第 11 章 KRaft;第 14 章 Purgatory;第 15 章 Network Reactor;基本的 Token Bucket / Sliding Window 算法。

20.1 四类配额

① Produce 字节配额

限 入站 字节速率(MB/s)。粒度按 (user, client-id)。超 quota 后 ProduceResponse 里 throttle_time_ms 增大。

kafka-configs --bootstrap-server ... --alter \
  --add-config 'producer_byte_rate=10485760' \
  --entity-type users --entity-name team-a

② Fetch 字节配额

限 出站 字节速率。粒度同上。注意:Follower fetch 也走这条,replication 用户配大 quota 或单独 listener。

--add-config 'consumer_byte_rate=20971520' \
  --entity-type users --entity-name team-a

③ 请求时间配额

限 broker 处理时间百分比。比如 5% 意味着该客户端的请求总时间占 broker 总能力的 5%。适合限"慢请求大客户"。

--add-config 'request_percentage=10' \
  --entity-type users --entity-name team-a

④ Controller 元数据配额

限 每秒 metadata mutation 数。防止恶意循环创建 / 删除 topic。Kafka 3.0+ 默认开启。

--add-config 'controller_mutation_rate=1.0' \
  --entity-type users --entity-name team-a

20.3 字节配额实现

字节配额的核心是滑动窗口测量。broker 在每次 Produce / Fetch 成功处理后,把该请求的字节数累加到 Sensor(一个时间衰减的计数器)。Sensor 用 N 个 sub-window(默认 11 个,每个 1 秒)做滑动窗口:

Sensor 状态:
  windowSize = 1000 ms
  numSubwindows = 11
  current = bucket[currentIndex]

每次 record(bytes):
  if (clock > current.endTime) {
    rotate to next bucket
    reset its count
  }
  current.count += bytes

观测速率 (rate over last 11s):
  totalBytes = sum(all buckets) / 11s

broker 局部计量,无全局令牌桶

每个 broker 独立计量自己接收 / 发送给该客户端的字节。集群总配额 = N × broker quota。这是简化设计 — 全局令牌桶需要跨 broker 协调,broker 间同步 quota 会变成新瓶颈。代价是分布不均的客户端(只跟一个 broker 通信)会过早被限,而 broker 全集群读 / 写的客户端能拿满 N × quota。

20.4 请求时间配额

request_percentage 限的是"该客户端的请求处理时间占 broker 总线程时间的百分比"。计算公式:

broker 一秒内总 wall time:
  totalWallTime = (numIoThreads + numNetworkThreads) × 1000 ms

某客户端一秒内的处理时间:
  clientWallTime = Σ(localTimeMs + ioWaitTimeMs) per request

观测百分比:
  pct = clientWallTime / totalWallTime × 100

throttleTimeMs:
  excess = clientWallTime - (quota_pct / 100) × totalWallTime
  throttleTimeMs = excess
request_percentage 比字节配额更适合"慢请求大客户" — 比如某客户端发的请求很小但处理慢(复杂的 admin API、大 SearchTopic),字节配额限不住。request_percentage 直接限 broker 时间预算。

20.5 Controller 元数据配额

KIP-599 引入的 controller_mutation_rate 限的是 CreateTopics / DeleteTopics / CreatePartitions 这种元数据写入的频率。设计动机:之前一个故障脚本可以在几秒内 createTopic 上万次,导致 Controller 元数据 log 写入风暴,影响整个集群的 metadata 推送。

受限 API计量
CreateTopics1 个 topic = 1 次 mutation
DeleteTopics1 个 topic = 1 次 mutation
CreatePartitions新增的 partition 数 = mutation 数

20.7 滑动窗口算法

Quota 用 org.apache.kafka.common.metrics.Sensor(Kafka 通用指标抽象)做滑动窗口。每个 Sensor 含一个或多个 KafkaMetric,后者用 SampledStat(默认实现是 Rate)做时间窗口测量。

参数默认含义
quota.window.size.seconds1子窗口大小
quota.window.num11子窗口数
(等效观察窗口)10 秒滑动窗口总长
窗口太大问题:默认 11 秒窗口意味着前 10 秒突发的字节会让后续 10 秒一直被限。短突发场景(大文件分块上传)如果业务允许,调小到 5-6 个 subwindow + 1s 子窗口 = 5-6 秒观察窗口。但太小会让限流"抖动"。

20.8 监控与排障

JMX含义异常
QuotaManager.ThrottleTimeMs (quotaType=Produce, user=...)给该 user 的 produce 限流持续 > 0 = 配额触发
QuotaManager.Throttle.byte-rate当前观测速率接近 quota = 即将限
BrokerTopicMetrics.BytesInPerSec per client.id各客户端贡献识别噪声邻居
ControllerQuotaManager.ThrottleTimeMs元数据限流> 0 = 元数据风暴

排障流程

症状: "客户端 producer 抖动, P99 突然涨到 500ms"

1 看客户端: ProduceResponse.throttle_time_ms > 0?
  是 → 被 quota 限了, 走 2
  否 → 网络 / GC 等其他原因

2 broker JMX: QuotaManager.ThrottleTimeMs (user=<user>) > 0?
  是 → 知道是哪个 user, 走 3

3 是哪类 quota? Produce / Fetch / Request / Controller?
  - 查 broker config 看 quota 设了多少
  - 查 BytesInPerSec 看实际速率

4 处理:
  - 若是噪声邻居: 收紧其配额 + 通知业务
  - 若是合理用量: 放大配额 或 拆 listener
  - 若是 Controller 风暴: 立即查谁在 createTopics 循环

20.9 设计取舍

① 延迟响应 vs 拒绝

延迟比拒绝友好:客户端不必处理新 error code,SDK 自动按 throttle_time 退避。代价是 broker 多一个 DelayedThrottle 入 Purgatory,Purgatory 大小可能涨。

② broker 局部 vs 全局令牌桶

broker 局部省去跨 broker 同步,简单可靠。代价是不均分布的客户端可能过早被限或拿满 N × quota。生产中通常通过合理配额值规避。

③ Sensor + 滑动窗口

滑动窗口比令牌桶更适合"测量平均速率"。代价是突发短时间内不限(burst 设计),需要业务侧配合限速。

④ 4 类 Quota 独立

分开限制不同资源,精细控制。代价是配置多。生产 89% 场景只用 Produce/Fetch 字节即可。

20.11 本章要点

本章小结

Quotas 是 Kafka 多租户治理的核心机制。它在协议层实现"延迟响应"的友好限流,不破坏现有客户端兼容性。四类 Quota(Produce 字节、Fetch 字节、Request 时间、Controller 变更)覆盖了 broker 几乎所有的"可被滥用资源"。8 层匹配让你能精细治理:某 team 的 ETL 客户端单独限,其他默认。

用好 Quota 的关键:① 先开认证,身份可信才有意义;② 监控 ThrottleTimeMs,识别噪声邻居;③ broker 局部计量,集群总 = N × broker;④ Follower fetch 用单独 user;⑤ 不要把 quota 当紧急刹车,它是预防性限流,故障时还需要应急扩容 / 隔离。下一章我们看 Kafka Connect,一个用于"把 Kafka 接入外部系统" 的集成框架。
