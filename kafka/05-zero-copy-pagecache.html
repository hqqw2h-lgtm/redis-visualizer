第 5 章 · Zero-Copy 与 Page Cache

sendfile + SG-DMA · transferTo 调用链 · page cache 双重角色 · 追尾 vs 追赶 · mmap 与 sendfile 分工 · TLS 退化 · 内核参数 · 反模式护栏

很多人对 Kafka 性能的第一印象是"它把数据全缓存在内存里"。事实正相反:Kafka 把数据缓存这件事完全外包给了操作系统。broker JVM 几乎不持有任何业务字节,它只负责"告诉内核从哪个文件的哪段偏移开始,把多少字节直接发给哪个 socket",剩下的全部交给 page cache 与 sendfile(2)。这种"把自己藏起来,让 Linux 干活"的设计,是 Kafka 单 broker 能跑出 25-30 GiB/s 出站带宽的根。

本章把 zero-copy 与 page cache 这一对耦合的工程决策展开:先把传统 read+write / sendfile / sendfile + SG-DMA 三种数据路径在字节、CPU、上下文切换三维度对齐;然后追踪 Kafka fetch 路径从 KafkaApis.handleFetchRequest 一路到 Linux sendfile64 系统调用的完整调用链;接着把 page cache 在写入与读取上的"双重身份"讲清楚 — 它既是 producer 的写入缓冲,又是 consumer 的读取来源;最后讨论追尾 vs 追赶、mmap vs sendfile、TLS 退化、dirty page 内核参数等四组工程取舍,并把九大反模式与排障决策树固化为可巡检的护栏。讨论锚定 Apache Kafka 3.7 与 Linux 5.x。

  • "Zero-Copy"的精确含义:用户态 0 次、CPU 拷贝 0 次,DMA 不可避免(物理 I/O 下界)
  • 三种数据路径的字节、CPU、上下文切换、内存带宽对比,以及为什么现代 10G+ 网卡才能真正做到"零 CPU 拷贝"
  • sendfile(2) 的内核流程:从 page cache 命中到 SG-DMA 描述符提交,broker 只负责一次系统调用
  • JDK FileChannel.transferTo() 在 Linux 下如何映射到 sendfile64,以及它在哪些条件下会"悄悄退化"为用户态拷贝
  • Kafka fetch 路径:KafkaApis → ReplicaManager → UnifiedLog → FileRecords → PlaintextTransportLayer → sendfile64 的完整调用栈
  • Page Cache 在写入与读取路径上"同一物理页"的双重角色,以及为什么 Kafka 故意把 JVM heap 配得很小
  • 追尾读 vs 追赶读:延迟分布差几个数量级,以及"追赶污染追尾"的典型故事
  • mmap 与 sendfile 的分工:索引用 mmap(随机点读)、日志用 sendfile(顺序大块)
  • TLS 为什么阻断 zero-copy:加密必须在用户态发生,sendfile 路径直接失效
  • 多 listener 拓扑:内网 PLAINTEXT 保 zero-copy,公网 SASL_SSL 保安全
  • 大内存机的 dirty page 参数陷阱:百分比制在 256 GiB 机器上变成 50 GiB 一次刷盘的隐形雷
  • 九大反模式 — 从"误以为 zero-copy = 不落盘"到"追赶 consumer 没配 quota"的完整排障矩阵

"Zero-Copy"是个营销词,精确说法是"零用户态拷贝"。物理磁盘到内存、内存到网卡这两次 DMA 是物理 I/O 的下界,谁都消除不了。Kafka 真正消除的是 用户进程地址空间 与 CPU 寄存器 上的数据拷贝。理解这个区分有什么用?— 当你看到 broker CPU 还是 30%、不是 0%,不要怀疑"sendfile 没生效",这部分 CPU 是花在系统调用入口、协议解析、网络中断处理上的,不是数据搬运。要确认 sendfile 真在工作,看 strace -e trace=sendfile -c 的统计,或者看 perf top 里有没有 do_softirq / tcp_sendpage 这种内核函数 — 它们在,sendfile 就在。

5.1 三种数据传输路径

"为什么 sendfile 比 read+write 快"这个问题如果只回答"少了几次拷贝",其实只说了一半。完整的答案要把 DMA 次数 / CPU 拷贝次数 / 上下文切换次数 / 内存带宽消耗 四个维度同时摊开。本节按 read+write、sendfile、sendfile + SG-DMA 三条路径逐一对照,目的是让你理解 Kafka 在 Linux 上跑的"真实路径"到底走的是哪一条。

路径一 · 传统 read + write

图 5-1-1 · 传统 read + write 路径
传统 read + write 路径(任何 Unix 内核):

  Disk
    ↓ DMA
  Kernel Buffer (page cache)
    ↓ CPU copy (read 系统调用,从内核到用户态)
  User Buffer (应用堆)
    ↓ CPU copy (write 系统调用,从用户态到内核)
  Socket Buffer (内核)
    ↓ DMA
  NIC

  数据拷贝次数:
    DMA: 2 次(磁盘→内核, 内核→网卡)
    CPU: 2 次(内核→用户, 用户→内核)

  上下文切换:
    read:  user → kernel → user(2 次)
    write: user → kernel → user(2 次)
    合计 4 次

  CPU 占用:
    每字节涉及 2 次内存拷贝
    1 GiB/s 数据 ≈ 2 GiB/s 内存带宽消耗

传统路径让磁盘字节经历 内核 → 用户态 → 内核 → 网卡 四段旅程,CPU 在每段都要参与内存拷贝。对一个想跑 1 GiB/s 出站带宽的 broker 来说,这意味着要消耗 2 GiB/s 的内存带宽 — 现代 DDR4 单通道理论带宽约 20-25 GiB/s,看似富余,但实际系统里还有缓存一致性、其他进程、内核工作队列在抢,2 GiB/s 的"纯数据搬运"在生产负载下足够把 L3 cache 击穿,显著拖慢同机器上的其他业务。这条路径今天只在必须经过用户态处理(如 TLS 加密、应用层过滤)的场景才会用。

路径二 · sendfile

图 5-1-2 · sendfile 路径
sendfile 路径(Linux 2.4+):

  Disk
    ↓ DMA
  Kernel Buffer (page cache)
    ↓ CPU copy (内核内部,从 page cache 到 socket buffer)
  Socket Buffer (内核)
    ↓ DMA
  NIC

  数据拷贝次数:
    DMA: 2 次
    CPU: 1 次(仅内核内部,不出内核态)

  上下文切换:
    sendfile: user → kernel → user(2 次)

  CPU 占用:
    用户态完全不参与
    仅内核态有一次拷贝

  系统调用签名:
    ssize_t sendfile(int out_fd, int in_fd, off_t *offset, size_t count);
    in_fd  - 文件描述符(数据来源,必须 mmap-able)
    out_fd - socket 描述符(数据去向)
    offset - 文件起始偏移,内核会自动推进
    count  - 本次最大字节数

sendfile 把"读 + 写"合并成一个系统调用,关键改进是数据全程不离开内核态。用户进程仅触发系统调用并等待完成,数据搬运由内核内部完成。从 4 次上下文切换降到 2 次,从 2 次 CPU 拷贝降到 1 次,内存带宽消耗减半。Linux 2.4 引入,后来在 4.5 版本起内核会自动选择 splice 路径优化某些场景,但接口不变 — Kafka 调用方完全不用关心底层实现的演进。

路径三 · sendfile + SG-DMA(零 CPU 拷贝)

图 5-1-3 · sendfile + SG-DMA 路径
sendfile + SG-DMA 路径(Linux 2.4+,需 SG-DMA 网卡):

  Disk
    ↓ DMA
  Kernel Buffer (page cache)
    ↓ descriptor (仅传指针,不传数据)
  NIC
    ↓ Scatter-Gather DMA(网卡按描述符直接读取多个页)

  数据拷贝次数:
    DMA: 2 次(磁盘→内核, 内核页→网卡)
    CPU: 0 次

  关键:
    socket buffer 不再存数据,仅存描述符列表
    NIC 通过 SG-DMA 按描述符从 page cache 直接读取
    现代 10G+ 网卡均支持(典型如 Intel X710 / X722 / Mellanox CX5)

  Kafka 在 Linux 上默认走该路径,前提:
    1. 内核版本 ≥ 2.4
    2. 网卡支持 SG-DMA(10G+ 几乎全支持)
    3. 文件 page 已在 page cache(否则触发磁盘读)
    4. 传输协议为 PLAINTEXT(TLS 退化到 read+write)

SG-DMA 是 "zero copy" 这个词在工程上能成立的真实路径。把 socket buffer 改造成"描述符列表",NIC 通过 Scatter-Gather DMA 直接从 page cache 读多个不连续的物理页,内核连"内核内部那次拷贝"都省了。从此点理解 zero-copy 的精确含义:用户态 0 次、CPU 0 次、DMA 不可避免。Kafka 在合规配置下默认享受这条路径,生产里只要四个前提都成立,broker 单核就足以推 25 Gbps 出站带宽。

实测 CPU 占用对比

图 5-1-4 · 28 核 broker 推 10 GiB/s 持续 1 分钟
传输 10 GiB/s 持续 1 分钟,broker CPU 占用近似:

  read + write:    ~70% (≈ 2 个核心专用于内存拷贝,12-15 核被吃)
  sendfile:        ~25% (1 个核心用于内核拷贝,7-8 核被吃)
  sendfile+SG-DMA: ~5%  (仅系统调用与中断处理,1-2 核被吃)

实测条件:
  - 28 核 broker,Intel Xeon Platinum 8280 / 8358
  - 25 Gbps 网卡(Intel X710,支持 SG-DMA)
  - NVMe SSD(本地数据,不走网络存储)
  - PLAINTEXT 协议
  - 数据热(全部在 page cache)

观察方式:
  top         看 broker 进程 %CPU 和 sy% / us% 比例
  perf top    看 do_softirq 与 sendfile 相关函数(应占首位)
  pidstat     看 minflt/s / majflt/s 是否持续 0
  iostat -x   看磁盘 r/s 是否近 0(数据全 cache 命中)
从 read+write 切到 sendfile,broker 单机吞吐翻 3 倍。2014 年某电商公司迁移日志收集系统,前端用 logstash 走 TCP 把行日志推到自研接收服务,接收服务再 read 文件并 write 到下游 Kafka。后来直接换成 Kafka producer + Kafka broker,broker 出站走 sendfile,同样的 25G 网卡同样的机器,出站从 4 GiB/s 涨到 12 GiB/s — 不是 Kafka 协议本身比 TCP 高效,而是数据彻底不再进出用户态,broker CPU 从 80% 降到 30%,机器还有富余继续承接。这件事让团队意识到:网络密集型服务的 CPU 瓶颈,80% 时间不是协议处理,而是字节搬运。

实测中,zero-copy 节省的 CPU 通常折算为更多分区或更高吞吐。25 Gbps 网卡 + SG-DMA 的 broker 单机可承载 30 GiB/s 入站 + 出站组合流量(假设入站 / 出站对称),分区数可推到 4000-5000 个 — 这个数字在没有 zero-copy 的架构下要 3-4 倍硬件才能达到。

5.2 sendfile 系统调用细节

把 sendfile 当黑盒用没问题,但排障时你必须能"打开盒子看里面"。本节从内核入口看到网卡 DMA,从 strace 实际输出看到 JDK 的 FileChannel.transferTo() 是怎么映射到 syscall,把这个调用从语义上、字节流上、时序上同时锚定。

内核侧时序

图 5-2-1 · sendfile 内核流程(单次系统调用全程)

这张时序图里两个分支必须看清楚:命中分支(追尾读)与未命中分支(追赶读)的差距是 1 微秒 vs 100 微秒以上。命中时内核只做"页表查找 + 描述符提交",几乎没有真正的 I/O;未命中时要触发 readahead,从磁盘搬数据进 page cache,期间 broker 进程会被内核挂起等待。这个差距就是后面 §5.5 "追尾 vs 追赶"延迟分布完全不在一个数量级的根因。

JDK 的 transferTo 映射

public abstract long transferTo(long position, long count,
                                 WritableByteChannel target) throws IOException;
JDK 内部决策(Linux 上,OpenJDK 17 实现):

  1. 检查 target 类型
  2. 若 target 是 SocketChannel 且未设 SSL
  3. 调用 sun.nio.ch.FileChannelImpl.transferToDirectly()
  4. 该方法通过 JNI 调用 sendfile64

实际调用栈:
  FileChannelImpl.transferToDirectlyInternal
    → IOUtil.transferTo
      → FileDispatcher.transferTo0(JNI 边界)
        → sendfile64(syscall)

退化条件(任何一项触发就退化):
  - target 不是 SocketChannel(如自定义 Channel 包装)
  - SSL 上下文已包装(SslEngine 必须看到明文)
  - 系统不支持 sendfile(极少数古老内核)
  - 文件在 ZFS / 某些奇异 FS 上(实现不支持)
  → 退化到 transferToTrustedChannel(经用户态 ByteBuffer 中转)

这里"退化"是个隐形坑:JDK 不会抛异常告诉你"sendfile 失败了",它会静默走慢路径。生产里要确认 sendfile 真的在工作,只能用 strace 或 eBPF 抓系统调用 — 这是排障 "broker CPU 莫名飙高"的标准检查项。

5.3 Kafka Fetch 调用链

把 sendfile 这个"末端动作"嵌回 Kafka 的整体调用链,你会看到这条链条短得令人惊讶:从 handleFetchRequest 到 sendfile64,一共 8-10 层调用,而真正涉及"字节拷贝"的层只有最后一层 — 其他全是偏移量 + 字节范围的元数据传递。这种"broker 不接触数据本体"的架构,是 Kafka 性能可预测的根本原因。

从 KafkaApis 到 sendfile64 的全调用栈

图 5-3-1 · Fetch 路径源码追踪:从 KafkaApis 到 sendfile64

注意 FileRecords.slice 这一步:它返回的不是字节,而是一个 "文件 + 起始位置 + 长度" 的三元组对象。从这里到 sendfile64,broker JVM 持有的全部信息只有这三元组,真实字节始终留在 page cache。这是 zero-copy 在架构层面能成立的关键 — Kafka 内部设计就把"字节搬运"这件事推到了可以最后一刻才发生的位置。

FileRecords.writeTo 的简洁实现

public class FileRecords extends AbstractRecords {

    @Override
    public long writeTo(GatheringByteChannel destChannel,
                        long offset,
                        int length) throws IOException {
        long position = start + offset;
        long count = Math.min(length, end - start - offset);
        return channel.transferTo(position, count, destChannel);
    }
}
调用方:Send 对象的 writeTo
  Send 是 Kafka network 层的发送抽象
  - 对于 FileRecords,直接走 transferTo(走 sendfile)
  - 对于 ByteBufferSend,走传统 write(已经在 heap)

关键判断在 PlaintextTransportLayer:

  public long transferFrom(FileChannel fileChannel,
                            long position, long count) {
      return fileChannel.transferTo(position, count, socketChannel);
  }

  其中 socketChannel 是 java.nio.channels.SocketChannel
  在未启用 SSL 时,JDK 内部走 sendfile

SslTransportLayer 的对应方法:

  public long transferFrom(FileChannel fileChannel,
                            long position, long count) {
      // 必须先把数据读进 ByteBuffer,再交给 SSLEngine 加密
      ByteBuffer buf = ByteBuffer.allocate((int)count);
      fileChannel.read(buf, position);  // 数据进入 JVM heap!
      ...
  }

对照看这两个 transportLayer 的实现就明白:SSL 不是"在 sendfile 上加一层加密"那么简单,而是直接绕开 sendfile,改成传统 read + 用户态加密 + write。FileRecords 那个优雅的"零拷贝"通道在 SSL 路径下根本不存在 — 这就是 §5.7 要展开的 TLS 退化的根。

图 5-3-2 · 单次 FetchResponse 同时推多分区
FetchResponse 中的 Send 结构:

  MultiRecordsSend
    ├── Send_for_partition_0  (FileRecords.writeTo,走 sendfile)

Selector → sendfile 的协作时序

图 5-3-3 · FetchResponse 与 sendfile 的步进协作
PlaintextTransportLayer 持有 SocketChannel
SocketChannel 在 JDK 实现中支持 transferFrom

调用顺序:
  1. KafkaApis 构造 FetchResponse(只持元数据)
  2. FetchResponse 包装 MultiRecordsSend
  3. Processor.processNewResponses 把 Send 加入 selector
  4. selector 在 OP_WRITE 触发时调用 channel.write(send)
  5. PlaintextTransportLayer.write 调用 send.writeTo(channel)
  6. send.writeTo 调用 FileRecords.writeTo
  7. FileRecords.writeTo 调用 fileChannel.transferTo(socketChannel)
  8. JDK 在 Linux 上调用 sendfile64
  9. 内核完成传输,系统调用返回字节数
  10. 若 send 未完成(返回值 < count),继续等下次 OP_WRITE

整体特点:
  数据从未进入 broker 用户空间
  broker 仅维护 offset 与字节范围
  内核完成实际传输工作
  socket 缓冲满 / 网络背压时,只是"下次再来",不阻塞 broker 线程
NIO Selector + sendfile 是 Kafka 高吞吐的"组合拳"。Selector 让一个 Processor 线程能管理几千个 socket(epoll 多路复用),sendfile 让每个 socket 的数据写入不消耗 CPU。两者结合的结果是:一个 broker 用 8-16 个 Processor 线程就能处理 5000+ 客户端连接、推 25 GiB/s 出站带宽。如果换成 read+write 模型,这个数字会被 CPU 卡在 5-8 GiB/s。这种"少线程 + 零拷贝"的组合在 Netty / Vert.x 等框架里都是核心思想,但 Kafka 把它做到了存储与网络耦合得最紧的程度 — 文件本身就是"待发送的字节流",中间不需要任何反序列化。

5.4 Page Cache 的核心地位

讨论 Kafka 性能而不讲 page cache 是空中楼阁。Linux page cache 在 Kafka 架构里扮演了三个角色:写入缓冲(producer 写入的 dirty page)、读取来源(consumer fetch 的 sendfile 输入)、跨进程数据共享(broker 重启后仍可命中)。这一节把这三个角色与"为什么 Kafka 拒绝 JVM heap 缓存"的设计决策一起讲清楚。

Page Cache 的双重身份

图 5-4-1 · 写入路径与读取路径在 page cache 上汇合

写入路径把字节存入 page cache 形成 dirty page,读取路径(若数据热)直接从同一份 page cache 读取。同一物理页既是 producer 的写入终点,又是 consumer 的读取起点。该设计让生产 + 消费在 page cache 上自然汇合,broker 不需要维护额外缓存层。这就是为什么 Kafka 在追尾读场景下磁盘 IOPS 几乎为 0 — 字节从未真正落盘就被 consumer 取走了(虽然 OS 仍会异步 writeback 到磁盘以保证持久性)。

方案 A vs 方案 B · JVM heap 还是 Page Cache?

图 5-4-2 · 数据缓存的两种架构对比
方案 A:JVM heap 缓存数据(被 Kafka 显式拒绝)

  数据流:
    磁盘 → page cache → broker JVM heap → socket
    每条消息在内存中存在 2 份(page cache + heap)

  问题:
    1. GC:消息字节数据被 JVM 视为对象,GC 标记扫描成本高
    2. 重复:同一字节在 page cache 和 heap 各占一份,内存利用率减半
    3. 重启:进程崩溃后 heap 缓存全失,page cache 仍可能存活
    4. zero-copy 失效:数据在 heap,无法 sendfile

方案 B:Linux Page Cache(Kafka 选择)

  数据流:
    磁盘 ↔ page cache ↔ socket(via sendfile)
    broker JVM heap 仅保存 offset + 索引引用 + 控制元数据

  优势:
    1. GC:JVM heap 小(典型 4-8 GiB),GC 影响小
    2. 单一副本:同一字节仅 page cache 存,内存利用率高
    3. 重启:broker 进程退出但内核 page cache 保留
    4. zero-copy:页直接喂给 sendfile

经验配置(64 GiB 机器):
  JVM heap        = 6 GiB (broker 元数据 + 网络缓冲)
  page cache 可用 ≈ 50 GiB(可缓存 50 GiB 热数据)
  其他            = 8 GiB(OS / 其他进程 / 临时缓冲)

重启后 page cache 是否保留

图 5-4-3 · 进程崩溃 vs 机器重启的 page cache 行为
场景 A:broker JVM 进程崩溃,机器仍正常运行

  T0:broker 运行,page cache 包含 30 GiB 热 segment
  T1:JVM 进程被 kill -9 或 OOM killer 干掉
  T2:进程退出,broker 持有的文件描述符释放
  T3:文件描述符释放不影响 page cache(内核全局管理)
  T4:broker 进程重启,重新打开文件
  T5:文件 mmap / read 直接命中已有 page cache

  结果:
    consumer fetch 立即恢复正常延迟
    无需"预热"
    P99 几乎不抖动

场景 B:机器重启(reboot / kernel panic / 电源)

  T0:正常运行
  T1:reboot
  T2:内核重启,page cache 全清(物理内存断电)
  T3:broker 启动,首次 fetch 触发磁盘读
  T4:逐步重建 page cache(取决于 fetch 顺序)
  → 重启后 30-300 秒 fetch 较慢(取决于读量与磁盘速度)

工程意义:
  错峰滚动重启,避免同时多 broker 缺乏 page cache
  SRE 团队的"重启 playbook"必须把这个差异写进去

5.5 追尾读 vs 追赶读

Kafka 的吞吐能力在两类读场景下表现完全不同:追尾读(consumer 与 producer 几乎同步,数据全在 page cache)是设计目标,延迟在毫秒级;追赶读(consumer 落后几小时甚至几天,数据需从磁盘读)是吞吐挑战,延迟跳到几十甚至几百毫秒。理解这两种模式的差异,以及"追赶污染追尾"的传染效应,是 SRE 容量规划的核心。

两类读模式的数据流差异

图 5-5-1 · 追尾读 vs 追赶读:两种模式对系统行为的影响

追尾读是 Kafka 的"主场":producer 与 consumer 之间通过 page cache 直接接力,磁盘 I/O 几乎不参与。一个典型的实时业务(consumer lag < 1 秒)在追尾读模式下,broker 磁盘 IOPS 可以低到 100-500(全是异步 writeback),CPU 也只有 10-20%,瓶颈是网卡带宽。追赶读则是另一个故事 — consumer 读历史 segment 时大概率 page cache 不命中,触发磁盘读取,且这种读取还会通过 LRU 把热页挤出 cache,把追尾 consumer 一并拖慢。

追赶读对追尾读的污染

  page cache 容量不足,内核 LRU 回收
  热 segment 部分被回收

  page cache 中热数据从 30 GiB 退化到 5 GiB

  consumer A 延迟抖动 30 分钟

  其他实时 consumer 也被波及
"追赶 consumer 没限速 = 实时业务延迟抖动 30 分钟"是大部分 Kafka 集群的隐形雷。最常见的诱因是数据科学团队启动一个新模型,把 auto.offset.reset=earliest 接到生产 topic 上,瞬间从 7 天前开始重放 — 这种"完全合法的消费请求"会把所有实时业务的 P99 拉高一个量级,而且没有任何报警直接命中"是这个 consumer 的锅"。修复路径:给所有 non-realtime consumer group 强制配 consumer_byte_rate,把回放限速到 10-50 MiB/s,让回放慢慢做、不挤热数据;或者用独立的 broker 集群专门跑历史重放。这是 SRE 在"集群规模上千节点"那一刻必须做的隔离。

5.6 mmap 与 sendfile 的分工

Kafka 同时使用 mmap 和 sendfile 两种"少拷贝"技术,这不是冗余,而是各自适配不同的访问模式:索引文件用 mmap(随机点读 / 二分查找),日志文件用 sendfile(顺序大块传输到 socket)。理解这种分工对正确解读"为什么 .index 用 mmap 而 .log 不能也用 mmap"很关键。

两类技术的应用场景

图 5-6-1 · mmap 与 sendfile 的职责边界

这种分工的本质:mmap 是"用户态访问 + page cache 共享",适合索引这种"读出后还要做计算(二分搜索)"的场景;sendfile 是"内核态直传 + 不进用户态",适合日志这种"读出后只要发出去"的场景。如果索引也用 sendfile,broker 拿不到字节做二分查找;如果日志也用 mmap,数据进入用户态就阻断了 zero-copy 路径。两者各得其所。

5.7 TLS 退化路径

TLS 是个看似无关紧要的开关,实际上对 Kafka 性能影响极大。它直接关闭 zero-copy 路径,因为加密必须在用户态发生 — broker 必须先把字节读进 JVM,经 SSLEngine 加密后再写回 socket。这一节把 TLS 的退化代价、AES-NI 硬件加速的缓解、多 listener 拓扑的隔离技巧讲清楚。

SSL 如何拦截 sendfile

图 5-7-1 · PLAINTEXT 与 SSL 路径分叉

SSL/TLS 协议要求加密发生在用户态,因为 SSLEngine 需要看到明文做对称加密。broker 必须把明文字节读入 JVM(fileChannel.read 进 ByteBuffer),经 SSLEngine 加密后写回 socket。该过程让 sendfile 完全失效,fetch 路径回到传统 read+write 模型,所有 §5.1 描述的 read+write 代价都重新回来:2 次用户态拷贝、4 次上下文切换、2 倍内存带宽消耗。

启用 TLS 1.3 + ChaCha20-Poly1305(无硬件加速但更轻):
  Fetch BytesOut: 7.5 GiB/s
  CPU usage:      55%

关键观察:
  sendfile 不再出现在 strace 输出
  CPU user% 显著上升(从 us=10 → us=40)
  SSLEngine 相关栈在 perf top 中占 30%+
  含 sun.security.ssl / com.sun.crypto.provider 大量调用

5.8 内核参数调优

page cache 是 Linux 全局资源,不归 Kafka 管,但 Kafka 性能强烈依赖它的行为。Linux 提供一组 sysctl 参数控制 dirty page 回写、swap 倾向、mmap 上限,这些默认值在 Kafka 大内存机器上多数不合适。本节给出生产推荐值与每个值背后的工程理由。

dirty page 控制流程

图 5-8-1 · dirty page 触发 writeback 的两个阈值

dirty page 参数清单

参数默认含义推荐
vm.dirty_background_ratio10后台 writeback 触发 %5(或用 bytes 替代)
vm.dirty_background_bytes0(取 ratio)绝对字节阈值大内存机推荐用此
vm.dirty_ratio20前台同步阈值 %10-60(权衡)
vm.dirty_bytes0(取 ratio)绝对字节阈值大内存机推荐

大内存机推荐配置

# 256 GiB 内存机器
sudo sysctl -w vm.dirty_background_bytes=1073741824   # 1 GiB
sudo sysctl -w vm.dirty_bytes=4294967296              # 4 GiB
sudo sysctl -w vm.swappiness=1
sudo sysctl -w vm.max_map_count=262144

# 持久化到 sysctl 配置
cat <<EOF | sudo tee /etc/sysctl.d/99-kafka.conf
vm.dirty_background_bytes=1073741824
vm.dirty_bytes=4294967296
vm.swappiness=1
vm.max_map_count=262144
EOF

sudo sysctl --system
理由:
  256 GiB × 10% = 25.6 GiB(默认 dirty_ratio)
  → 一次刷盘可能高达 25 GiB,持续数十秒
  → 期间所有写入线程被阻塞
  → 业务 P99 直接飙到秒级

  改为绝对值 4 GiB:
  → 单次最多 4 GiB
  → SSD 可在 5-10 秒内完成
  → 阻塞窗口可控

教训:
  百分比制在大内存机上变成异常大的绝对值
  推荐改用 dirty_bytes / dirty_background_bytes 直接指定
"默认 dirty_ratio 在 256 GiB 机器上 = 50 GiB 一次刷盘"是性能毛刺的常见根因。某金融客户 broker 是 384 GiB 机器,默认 vm.dirty_ratio=20 意味着脏页可累积到 76 GiB 才触发同步刷盘,一旦触发,即使 NVMe 也要 30-60 秒才能写完,期间所有写入阻塞。表现是"每 10-15 分钟一次集体延迟尖峰",每次持续 30 秒。改成 vm.dirty_bytes=4294967296(4 GiB)后,毛刺消失,P99 从尖峰 2000 ms 降到稳定 30 ms。线上看到 broker 周期性卡顿,先看 cat /proc/meminfo | grep Dirty 是不是周期性接近阈值。

swappiness 与其他参数

图 5-8-3 · swappiness 取值的工程含义
swappiness 取值范围 0-100:

  0   - 几乎不 swap,完全保留 page cache

实际效果:

  避免 broker JVM heap 被 swap

5.9 设计取舍

到这一节为止,所有"为什么这么做"的子问题都已铺开,可以集中讨论 Kafka 在 zero-copy 与 page cache 维度的四组核心取舍。每一个取舍都不是"我选了好的",而是"我接受了某些代价换取另一些收益"。

被显式拒绝的"缓存优化"备选

备选看似优势拒绝理由生产边界
JVM heap 缓存数据broker 内可控数据双份 + GC 干扰 + 阻断 sendfileheap 仅放元数据(4-8 GiB)
每写同步 fsync持久性直观吞吐受 disk fsync IOPS 限制用副本协议替代
direct I/O 绕过 page cache避免 LRU 污染追尾读失去命中,sendfile 失效仅冷归档场景
TLS 也零拷贝安全 + 吞吐双赢加密必须用户态,物理上做不到TLS 单独 CPU 规划
写入复制到第二缓存双缓存命中page cache 已是单一权威无收益

取舍 1 · Page Cache vs 自管缓存

选择 · Page Cache:GC 不扫数据 · 单一副本 · sendfile 兼容 · 进程崩溃仍存活
代价 · 内核 LRU 不可干预,Kafka 无法保证特定 segment 一定在内存
补救 · vm.swappiness=1 + 大内存 + page cache 监控 + 追赶 quota 联合保障

Kafka 选择不维护 broker 内自管缓存,依据有三:其一,GC 不需要扫描日志数据,broker 可保持 4-8 GiB 小堆;其二,数据仅 page cache 一份,内存利用率高;其三,sendfile 必须从 page cache 出发,自管缓存反而阻断快路径。代价是 page cache 由内核管理,Kafka 无法直接干预 LRU 行为 — 这就是为什么"追赶污染追尾"必须靠 quota 兜底。

取舍 2 · 同步 fsync vs 副本协议

选择 · 副本协议:RF=3 + acks=all + min.ISR=2,P99 5-20 ms,吞吐高
代价 · 单 broker 故障可能丢失最新未刷盘数据,但被副本覆盖
边界 · 全 broker 同时断电(rare 但可能)需 unclean leader election 兜底

每写 fsync 在 NVMe 上需 50-200 微秒,在 HDD 上需 5-20 毫秒。3 副本 + acks=all 在异步刷盘下提供同等持久性,P99 延迟降至 5-20 毫秒。Kafka 选择副本协议,代价被副本协议覆盖。这个取舍直接决定了 Kafka 不是"传统数据库"的角色 — 它的可靠性来自分布式协议,不是单机 fsync。

取舍 3 · PLAINTEXT 默认 vs TLS 默认

选择 · PLAINTEXT 默认:开箱即用享受 sendfile · 内网信任场景吞吐最大化
代价 · 公网部署需运维显式配 TLS,默认配置不"开箱即用安全"
补救 · 多 listener 拓扑:内网 PLAINTEXT,公网 SASL_SSL,broker 间 PLAINTEXT

取舍 4 · mmap 索引 vs read 索引

选择 · mmap 索引:二分搜索内全程零 syscall · 总开销 < 5 µs
代价 · 冷启动可能 page fault · 受 vm.max_map_count 限制(必须调大)
边界 · 大量 segment 时 mmap 数量爆炸,需要 segment.bytes 与 retention 配合

取舍矩阵汇总

维度选择拒绝方案决定因素
数据缓存Page CacheJVM heapsendfile + GC
持久性副本协议同步 fsync吞吐目标
默认协议PLAINTEXTTLS 默认sendfile 可用性
索引访问mmapread syscall二分性能
数据传输sendfileread+writeCPU 节省
追赶治理Quota + Tiered Storage无限制实时业务保护

5.11 本章小结

要点依据易错点
Zero-Copy 指 fetch 不进用户态sendfile + SG-DMA不等于不落盘 / 不等于无 DMA
Kafka 通过 transferTo 触发 sendfileFileRecords.writeTo仅 PLAINTEXT 路径,SSL 静默退化
Page Cache 是核心数据缓存不在 JVM heapheap 不应挤占 page cache
追尾读快,追赶读慢命中 vs miss 差 10-100x追赶污染热数据,需要 quota
TLS 阻断 sendfile加密必须用户态内网可保留 PLAINTEXT
mmap 用于索引,sendfile 用于日志各自适合的访问模式不要混用,不要一刀切
dirty page 参数大内存用绝对值dirty_bytes 4 GiB默认百分比在大机上过大
重启后 page cache 可能仍在进程崩溃 vs 机器重启不可作为强保证,错峰重启
swappiness 必须设为 1避免 JVM heap 被 swap0 在某些内核会 OOM kill
多 listener 隔离 PLAINTEXT 与 TLS内网快 + 公网安全broker 间通信不需要 TLS
