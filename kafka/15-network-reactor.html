第 15 章 · 网络层 Reactor 模型

SocketServer · Acceptor · Processor · RequestChannel · KafkaRequestHandler · KafkaApis · NetworkClient

如果一个 Kafka broker 同时挂着 5k 个长连接(Producer/Consumer/Follower 都算上),它内部并不是"一个连接一个线程"那种古典模型。Kafka 的网络层是经典的 主从 Reactor 流水线:Acceptor 单线程负责 accept() 新连接,N 个 Processor 各自跑一个 java.nio.channels.Selector 处理 TCP 读写,所有 Processor 共用一个 RequestChannel 把帧化好的请求扔给后端,M 个 KafkaRequestHandler 才进入 KafkaApis 这条业务路径。

这套流水线的好处是清晰:任何延迟现象都能映射到某一段线程或队列。坏处是参数多 — num.network.threads、num.io.threads、queued.max.requests、socket.*.buffer.bytes、SSL 配置…… 不理解流水线就调参,只会"调一个东西好了一阵又坏了"。本章按"流水线 → 各段细节 → 协议帧 → 时间分解 → 监控 → 取舍 → 陷阱"组织,目标是看完后你能不用启动 broker 就预测一次 ProduceRequest 在网络层的完整路径。

  • 主从 Reactor 模型:Acceptor 一个,Processor 多个,Handler 池更多
  • "为什么响应必须回到原 Processor" 的 Selector 状态约束
  • RequestChannel 共享请求队列 + 按 Processor 独立响应队列
  • mute/unmute:Kafka 在 Processor 层做的"per-connection 单飞行"
  • 请求时间六段拆分:Queue/Local/Remote/Throttle/Response/Total
  • SSL/SASL 握手在哪个线程发生,为什么会"把 Processor 吃光"
  • 5 个核心指标的健康基线 + 常见瓶颈的段到动作映射

前置知识:Java NIO Selector / Channel;TCP backlog;TLS 握手;第 6 章 Producer NetworkClient;第 9 章 acks=all 与 DelayedProduce;第 14 章 Purgatory。

15.1 生产视角 — 看流水线再调参

大多数"Kafka 慢"的 ticket 最后都不是 broker 自己的问题。但找到根因的前提是按段定位:先看是哪一段瓶颈,再考虑该段的调参。盲调 num.io.threads 是最常见的错误 — 它只调"Handler 池",对 SSL CPU 或 Follower fetch 慢都没用。

num.io.threads88-16Handler 数,按 CPU 核

生产健康基线:NetworkProcessorAvgIdlePercent > 0.5(Processor 仍有余量),RequestHandlerAvgIdlePercent > 0.3(Handler 还能扛),RequestQueueSize < queued.max.requests/2(队列不堆),TotalTime P99 < 100ms(端到端可控)。任何一个持续突破阈值都意味着瓶颈段已经出现。

15.2 SocketServer 装配

SocketServer 是 broker 启动时第一个起来的网络组件。它读取 listeners(可能是多个 listener,如 PLAINTEXT://broker1:9092, SSL://broker1:9093),为每个 listener 创建一个 Acceptor 和 N 个 Processor,并构建一个全局 RequestChannel。

// SocketServer 启动伪代码
class SocketServer(config: KafkaConfig) {
  val requestChannel = new RequestChannel(queueSize = config.queuedMaxRequests)
  val acceptors = mutable.Map.empty[EndPoint, Acceptor]

  def startup(): Unit = {
    config.listeners.foreach { ep =>
      val acceptor = new Acceptor(ep, requestChannel, ...)
      // 每个 listener 起 num.network.threads 个 Processor
      (0 until config.numNetworkThreads).foreach { i =>
        val processor = new Processor(id = nextId(), requestChannel, ...)
        acceptor.addProcessor(processor)
        processor.start()
      }
      acceptor.start()
      acceptors(ep) = acceptor
    }
  }
}

注意:每个 listener 都有独立的 Processor 池。这是 Kafka 给 SSL/PLAINTEXT 资源隔离的方式 — 你可以让控制面走 PLAINTEXT,数据面走 SSL,SSL 握手吃 CPU 不影响控制面的延迟。

多 listener 的典型用法

仅一个 listener

listeners=PLAINTEXT://:9092
num.network.threads=8

简单。如果 SSL 握手洪峰,会卡住数据面。

多 listener 分离

listeners=INTERNAL://:9092,EXTERNAL://:9093
listener.security.protocol.map=INTERNAL:PLAINTEXT,EXTERNAL:SSL
num.network.threads=8  # 每个 listener 都 8 个

SSL CPU 飙也不影响 INTERNAL listener。Follower 走 INTERNAL,Producer 走 EXTERNAL。

15.3 Acceptor 连接接入

Acceptor 单线程,只做一件事:accept() 拿到新 socket → 选一个 Processor 把 socket 塞过去。"选哪个 Processor"用 round-robin。"塞过去"通过 Processor 内部的 newConnections: ConcurrentLinkedQueue<SocketChannel>,Processor 在下次 poll() 后处理。

// Acceptor.run 伪代码
def run(): Unit = {
  val nioSelector = NSelector.open()
  serverChannel.register(nioSelector, SelectionKey.OP_ACCEPT)
  var index = 0
  while (isRunning) {
    val ready = nioSelector.select(500)
    if (ready > 0) {
      val keys = nioSelector.selectedKeys().iterator()
      while (keys.hasNext) {
        val key = keys.next(); keys.remove()
        if (key.isAcceptable) {
          val socket = serverSocketChannel.accept()
          configure(socket) // SO_LINGER, TCP_NODELAY, ...
          // round-robin 选一个 Processor
          val processor = processors(index % processors.size)
          processor.accept(socket) // 放入 processor.newConnections
          index += 1
        }
      }
    }
  }
}
常见误解:Acceptor 不参与读写、不持有 SelectionKey。它只是"分包员"。即使你跑 100k 连接,Acceptor 也基本不忙。所以 num.network.threads 调的是 Processor,不是 Acceptor。Acceptor 数等于 listener 数。

15.4 Processor 读写流

Processor 是网络层实际干活的线程。它一个循环里做四件事:

  1. 处理新连接:从 newConnections 取 SocketChannel,注册到自己的 Selector 上(OP_READ),建立 KafkaChannel 包装。
  2. 处理出站响应:从 responseQueue 取响应,挂到对应连接的 send 槽位上,enable OP_WRITE。
  3. poll Selector:阻塞最多 300ms,把 ready 的连接收/发各跑一次。
  4. 处理 completed:每个完整收到的请求扔进 RequestChannel.requestQueue。每个发完的响应回调 callback()。每个掉线/超时的连接,把它的"in-flight 请求"标记 disposed。
图 15-4 · Processor 一次 poll 循环

mute / unmute — Kafka 在 Processor 层做的"per-connection 单飞行"

Kafka 一个连接同时只能有一个正在被 broker 处理的请求(producer 端的 idempotent producer 有 max.in.flight.requests.per.connection 但那是 producer 自己的事;broker 这边的行为更严格)。Processor 在请求完成接收后会 mute 这个连接(取消 OP_READ),直到该请求的响应发完才 unmute(重新启用 OP_READ)。这样保证一个连接同时不会读到第二个请求。

// Processor.processCompletedReceives 简化
def processCompletedReceives(): Unit = {
  selector.completedReceives.forEach { receive =>
    val channelId = receive.source
    selector.mute(channelId)              // 暂停读
    val req = parseRequest(receive)
    requestChannel.sendRequest(req)
  }
}

// Processor.processCompletedSends 简化
def processCompletedSends(): Unit = {
  selector.completedSends.forEach { send =>
    selector.unmute(send.destination)     // 响应发完恢复读
  }
}

15.5 RequestChannel — 跨线程边界

RequestChannel 是 Processor(N 个)和 Handler(M 个)之间唯一的传递点。它结构是一个共享 requestQueue + 多个按 Processor 独立的 responseQueue。

class RequestChannel(queueSize: Int) {
  // 所有 Processor 共享
  val requestQueue: ArrayBlockingQueue[Request] =
    new ArrayBlockingQueue[Request](queueSize)

  // 每个 Processor 一个响应队列
  val responseQueues: ConcurrentHashMap[Int, BlockingQueue[Response]] =
    new ConcurrentHashMap()

  def sendRequest(req: Request): Unit = requestQueue.put(req)
  def receiveRequest(timeoutMs: Long): Request = requestQueue.poll(timeoutMs)

  def sendResponse(resp: Response): Unit = {
    val q = responseQueues.get(resp.processorId)
    q.put(resp)
  }
}
为什么响应队列要按 Processor 分:因为每个 Processor 持有自己的 Selector + 自己注册的 channel。响应只能在注册该 channel 的 Selector 上写。把响应发到错的 Processor 就找不到 channel,响应就丢了。所以 KafkaApis 在 sendResponse 时必须知道 processorId,这是 Request 对象一直携带的属性。

15.6 Handler 与 KafkaApis

Handler 池(kafka-request-handler-*)做的事很简单 — 从 requestQueue 取一个请求,调 KafkaApis.handle(req),处理完调 sendResponse。KafkaApis 是 broker 业务路由表 — 它根据 apiKey(Produce / Fetch / JoinGroup / Metadata 等)分发到对应处理函数。

class KafkaRequestHandler(...) extends Runnable {
  def run(): Unit = while (running) {
    val req = requestChannel.receiveRequest(timeoutMs = 300)
    if (req != null) {
      apis.handle(req)
    }
  }
}

class KafkaApis(...) {
  def handle(req: RequestChannel.Request): Unit = req.header.apiKey match {
    case ApiKeys.PRODUCE        => handleProduceRequest(req)
    case ApiKeys.FETCH          => handleFetchRequest(req)
    case ApiKeys.JOIN_GROUP     => handleJoinGroupRequest(req)
    case ApiKeys.OFFSET_FETCH   => handleOffsetFetchRequest(req)
    case ApiKeys.METADATA       => handleTopicMetadataRequest(req)
    // ... 70+ apiKey
  }
}

15.7 协议帧与请求时间分解

Kafka 协议是带长度前缀的二进制帧:4-byte int (length) + length bytes (header+body)。Processor 用 NetworkReceive 累积 byte buffer 直到一帧完整,再交给上游解码。

15.8 安全握手 SSL/SASL — 为什么会把 Processor 吃光

SSL/SASL 握手在 Kafka 的设计里跑在 Processor 线程。这是为了简化:Processor 持有 KafkaChannel,KafkaChannel 内部状态机包含 NOT_AUTHENTICATED / AUTHENTICATION_IN_PROGRESS / READY。握手字节流从 Selector 读出后,直接由 SslTransportLayer / SaslServerAuthenticator 推进状态机。

图 15-8 · SSL 握手在 Processor 上的状态机
生产惨案案例:某客户端集群滚动重启,5k 实例同时建连。num.network.threads=3,瞬时 5k 个 SSL 握手分到 3 个 Processor 上排队。这段时间内已建立的 Producer 连接全部 stall(因为 Processor 都在算 RSA)。Producer 端看到 request.timeout.ms 全部触发,大量重试,雪崩。

修复:① 把 num.network.threads 调到 8;② 拆 listener,业务数据走 PLAINTEXT,控制 / 跨 DC 走 SSL;③ 在 LB 层卸载 TLS;④ 客户端避免同时滚动重启。

15.10 设计取舍

① 主从 Reactor vs 单 Reactor

单 Reactor 一个线程做 accept + 读写,简单但 SSL 重时会卡 accept。Kafka 把 Acceptor 单独拆出,即使所有 Processor 都在做 SSL 握手,新连接的 accept 仍能完成,后端排队最多丢握手不丢 accept。

② Processor 持有 Selector

每个 Processor 自带 Selector,channel 一旦注册到某 Selector,所有读写都必须在该线程做。代价是响应必须按 ID 路由回原 Processor,收益是无锁。

③ 共享 requestQueue vs per-Processor queue

requestQueue 共享有动态负载均衡效果 — Handler 谁有空谁取。代价是 ArrayBlockingQueue 是有锁的。Kafka 没用 Disruptor,因为 broker 是 IO bound 不是 CPU bound。

④ mute / unmute per-connection

同一连接同时只允许一个 in-flight 请求,简化了响应路由(响应一到就 unmute)和顺序语义(同连接的请求按序处理)。代价是单连接的吞吐受 RTT 限制,需要客户端建多连接才能并行。

本章小结

Kafka 的网络层是一个教科书级的主从 Reactor实现:Acceptor 接连接,Processor 跑 Selector 做 TLS 和帧读写,RequestChannel 跨线程交付,Handler 调 KafkaApis。整条流水线的关键设计是"响应必须回到原 Processor"(因为 Selector 是 thread-confined)和"per-connection mute/unmute"(保证单连接请求顺序)。

定位生产延迟的最佳起点是请求时间六段拆分(Queue/Local/Remote/Throttle/ResponseQueue/ResponseSend)— 知道哪一段长了,才能针对性调参或修 bug。最常见的两个生产惨案是"SSL 握手把 Processor 吃光"和"盲调 num.io.threads"。下一章我们看 Kafka 3.6 之后的重磅特性:Tiered Storage 分层存储,即把冷数据自动卸载到 S3 / HDFS 等远端对象存储。
