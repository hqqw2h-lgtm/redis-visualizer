第 13 章 · Log Compaction(日志压缩)

基于 key 的最新值保留 · Cleaner 线程 · Tombstone · 删除墓碑 · __consumer_offsets

Kafka 默认是"基于时间/大小的滚动删除"(cleanup.policy=delete):过了 retention 就整段删除,不管你是否还需要。这对日志类数据合适,但对"每个 key 都需要保留最新值"的场景就不行 — 比如 changelog、KV 状态快照、__consumer_offsets。这时需要另一种策略 cleanup.policy=compact:Kafka 后台异步把同 key 的旧版本删除,只留最新一份。这就是日志压缩 (Log Compaction)。

本章按 "问题边界 → Cleaner 流程 → Tombstone → 配置 → __consumer_offsets → 排障 → 源码 → 取舍 → 陷阱" 组织,对应 Apache Kafka 3.7。Log Compaction 的实现细节很多人不熟悉,这是个相对低调的功能 — 但 Kafka 内部的所有"无限保留 KV 主题"都依赖它,理解它能帮你正确使用 Kafka Streams / Connect / Schema Registry 等基于 Kafka 的中间件。

  • Compaction 与 Delete 两种 cleanup.policy 的差异和适用场景
  • Cleaner 线程的两阶段工作流:扫描 + 重写
  • OffsetMap 哈希表的工作机制与内存占用估算
  • "clean / dirty" 区间的判定方式与 cleanable ratio
  • Tombstone(value=null)的作用与 delete.retention.ms
  • __consumer_offsets / __transaction_state 为什么必须用 compact
  • min.cleanable.dirty.ratio / segment.bytes / segment.ms 之间的关系
  • Compaction 在生产中的典型陷阱

前置知识:第 2 章日志存储引擎(segment 概念);第 4 章索引系统;第 12 章 EOS(__transaction_state 用了 compact)。

13.1 Compaction 要解决什么

设想一个场景:你想用 Kafka 主题做"用户档案的最新状态"。每次用户改了名字,你 publish 一条 user_id → updated_profile 的消息。如果用 cleanup.policy=delete,7 天后这些消息就没了 — 想恢复某个用户的当前档案,你得从头扫整个主题。Compact 解决的就是这个问题:让 Kafka 自己保留每个 key 的最新值,允许"无限保留"。

三个典型场景

① KV 状态主题

Kafka Streams 的 RocksDB state store 同时往一个 changelog topic 写,后者用 compact。如果 streams 任务重启,从 changelog 完整重建状态。

② Consumer Offsets

__consumer_offsets 把消费位移作为 (group, topic, partition) → offset 的 KV 来存。每次 commitSync 写一条新版本,旧的最终会被 cleaner 清掉。

③ Schema Registry / Config

Confluent Schema Registry 用 Kafka 主题做后端存储,所有 schema 都是 KV。任何配置类、元数据类的"最新值"系统都适合用 compact。

13.2 Cleaner 工作流程

Cleaner 是 broker 内的后台线程池(默认数量 log.cleaner.threads=1)。它的工作循环是:扫描所有 compact 主题 → 找出 dirty ratio 最高的分区 → 重写 segment。activeSegment(当前正在写入的)永远不参与 compaction,所以最新的写入永远不会被清理。

图 13-2 · Cleaner 两阶段工作流

OffsetMap 内存估算

OffsetMap 是一个 Open Addressing 的 hash 表:
  - 每条 entry: SHA-1(key) 16 bytes + offset 8 bytes = 24 bytes
  - 默认负载因子 0.75

可处理的 dirty 区大小:
  log.cleaner.dedupe.buffer.size / 24 / 0.75
  默认 128 MiB:
    128 * 1024 * 1024 / 24 / 0.75 ≈ 5.7M keys

如果 dirty 区超过这个量,Cleaner 会分批做:
  cleanedOffsetMap.full → 重新开始下一轮
  → Compaction 不一帧完成,持续推进

13.3 Tombstone 与 delete.retention.ms

Tombstone 是 value 为 null 的 record。它的语义是"删除这个 key"。Cleaner 看到 tombstone 会:① 把之前所有同 key 的 record 标记为可删除;② 自己也最终被删除,但不是马上 — 要保留 delete.retention.ms(默认 86400000 = 24 小时),给消费者时间看到这条 tombstone。

  --create --topic user-profiles \

compact,delete 混合策略

Kafka 也支持 cleanup.policy=compact,delete — 既按 key 保留最新值,又遵守 retention.ms 整体删除老 segment。适合 "状态有上限存活期" 的场景(比如最多保留半年的用户档案)。

13.5 与 __consumer_offsets 的关系

Kafka 内部主题 __consumer_offsets 是 compact 最经典的应用。它存储所有 consumer group 的 offset 提交记录,key 是 (group, topic, partition),value 是 offset + metadata。每次 commitSync 都写一条新记录。如果不 compact,这个主题会无限增长(尤其有高频 commit 的 group)。

__consumer_offsets 的默认配置:

  cleanup.policy=compact
  segment.bytes=104857600 (100 MiB)
  segment.ms=600000 (10 分钟)
  min.cleanable.dirty.ratio=0.5
  delete.retention.ms=86400000 (24 小时)

  partition 数: 50 (offsets.topic.num.partitions)
  RF: 3 (offsets.topic.replication.factor)

读取所有 group 的 offset:

  kafka-consumer-groups.sh --bootstrap-server kafka:9092 --list
  → my-app-1
  → my-app-2

  kafka-consumer-groups.sh --bootstrap-server kafka:9092 \
    --describe --group my-app-1
  → TOPIC PARTITION CURRENT-OFFSET LOG-END-OFFSET LAG

13.6 排障

关键 JMX

指标含义异常
kafka.log:type=LogCleanerManager,name=uncleanable-partitions-count无法 compact 的分区数> 0
kafka.log:type=LogCleanerManager,name=max-dirty-percent最高 dirty 比例持续 > 0.9
kafka.log:type=LogCleaner,name=cleaner-recopy-percentCleaner 重写比例过低说明 dedupe buffer 不够
kafka.log:type=LogCleaner,name=DeadThreadCountCleaner 死线程> 0

典型问题

问题:compact 主题持续增长,从不被清理 
现象:
  __consumer_offsets-0 持续涨,磁盘 90% 满
  max-dirty-percent = 0.95
  uncleanable-partitions-count = 1

排查:
  1 看 Cleaner 是否健康
     jstack 看 kafka-log-cleaner-thread-0 是否存活
  2 看 dedupe buffer 是否够大
     log.cleaner.dedupe.buffer.size 默认 128 MiB
     对于上亿 key, 可能不够
  3 看是否有未提交事务
     ongoing transaction 阻塞 LSO, 也会阻塞 compaction (KIP-360)

动作:
  - 重启 Cleaner: kill -SIGINT 让 Cleaner 重新启动
  - 调大 buffer: log.cleaner.dedupe.buffer.size=512MiB
  - 增加线程: log.cleaner.threads=4

核心循环简化

while (!shutdown) {
  // 1 选最 dirty 的 partition
  val partition = LogCleanerManager.grabFilthiestCompactedLog()
    .getOrElse { sleep(backoffMs); continue }

  // 2 build OffsetMap from dirty range
  val offsetMap = new SkimpyOffsetMap(dedupeBufferSize)
  val endOffset = buildOffsetMap(partition.dirtyRange, offsetMap)

  // 3 rewrite each segment
  for (segment <- partition.cleanableSegments) {
    val cleanedSegment = createNewSegment(segment.baseOffset, ".cleaned")
    segment.iterator.foreach { record =>
      if (offsetMap.get(record.key) == record.offset)
        cleanedSegment.append(record)
    }
    cleanedSegment.flush()
    swapAtomically(segment, cleanedSegment)
  }

  partition.cleanerCheckpoint = endOffset
}

13.8 设计取舍

① 异步 vs 同步 compact

Kafka 选异步:Cleaner 后台跑,不影响 producer 写入路径。代价是 dirty 区可能短期较大,需要监控。

② 重写 vs 原地标删

选重写(写新文件 → 原子 rename),不做原地删除。代价是 IO 翻倍,但避免锁竞争和半成品状态。

③ SHA-1 hashing vs 全 key

OffsetMap 只存 SHA-1(key) 16 bytes,不存原 key。代价是极低概率的哈希碰撞 → 误删某个 key 的最新值;收益是内存可控。

④ tombstone 保留期

tombstone 不立即删除,留 24 小时给慢消费者看到。代价是磁盘多占一点,收益是删除事件不会被静默吃掉。

13.9 陷阱与限制

本章小结

Log Compaction 让 Kafka 主题不只是"append-only 事件流",还可以成为"基于 key 的最新值存储"。Cleaner 异步重写 segment,把同 key 的旧版本删除;tombstone 表示"删除某个 key";__consumer_offsets 和 __transaction_state 都依赖这一机制实现"无限保留"。理解 compaction 是理解 Kafka 内部主题(以及 Kafka Streams / Connect)的关键。下一章我们换回数据面性能维度,看 Timing Wheel — Kafka 用来高效管理大量定时任务(produce/fetch purgatory)的核心数据结构。
