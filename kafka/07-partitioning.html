第 7 章 · 分区策略与哈希算法

分区动机 · 路由决策树 · Murmur2 + ZigZag · toPositive · Sticky Partitioner · 自定义分区器 · 一致性哈希对比 · 分区数估算 · key 设计 · 扩容漂移

Kafka 的"分区"看起来只是把一条 topic 切成 N 条独立日志,但它其实是整个 Kafka 并行模型与顺序保证的核心边界。它决定 producer 同 key 是否进同一台 broker、决定 consumer group 能横向扩到多少实例、决定一次扩容是否会让业务侧出现"乱序",还决定一个 hot key 是否会把单台 broker 的单核 CPU 打满。这一章把"一条 ProducerRecord 怎么被路由到具体 partition"摊开到字节级 — 从 Murmur2 哈希、toPositive 位运算,到 Sticky Partitioner 的批次状态机,再到 hot key 加盐与扩容漂移的数学推导。

讨论以 Apache Kafka 3.7 客户端的 BuiltInPartitioner 为基线 — 它从 2.4 替换了历史的 DefaultPartitioner,默认行为是"有 key 走 Murmur2 取模、无 key 走 Sticky"。本章不会停留在"分区是用来扩容的"这种表层结论,而是逐项追问:为什么 Kafka 选 Murmur2 而不是 xxHash?为什么 toPositive 用位与而不是 Math.abs?为什么 Sticky 短期粘附长期均匀?以及最容易被忽略但最致命的:为什么 alter --partitions 是一次破坏性操作。

分区是 Kafka 唯一的并行维度,也是唯一的顺序边界。一个 consumer group 内,同一 partition 永远只能被一个 consumer 实例消费;反过来,同一 consumer 实例可以同时消费多个 partition。这意味着 partition 数 直接等于 consumer group 的并行上限。如果你的 topic 只有 3 个 partition,部署 10 个 consumer 实例毫无意义 — 7 个会一直闲置,直到某个活跃 consumer 离群。这一点在容量规划阶段就要拍板,事后再扩 partition 数虽然技术上可行,但代价是同 key 跳分区(详见 §7.11)。

7.1 分区的设计动机

"为什么 Kafka 要分区"看似显而易见 — 单台机器扛不住,自然要切片。但具体切到几片、按什么切、怎么路由,这些都不是显而易见的。这一节先把分区带来的并行/吞吐/容量收益与"顺序粒度变细"这个代价摊开,让后面所有讨论都有一个共同的"权衡基线"。

每个分区有 1 个 leader 与 N-1 个 follower(N = replication.factor)。leader 分布通常按 broker 数轮转,follower 分布由 controller 在创建 topic 时按 rack-aware 算法决定 — 这一点在第 9 章会详细展开。本节关心的是分区编号(P0/P1/P2/P3)是稳定的:即使某个分区 leader 切换到其他 broker,它的分区号不会变 — 这一点直接决定了 consumer offset 的稳定性,也是后面 §7.8 解释"为什么不用一致性哈希"的关键依据。

真实事故:某电商平台为了"提升性能"把单 topic 从 6 分区扩到 60 分区。运维团队的逻辑是"多 10 倍 partition = 多 10 倍并行度",但他们没注意到该 topic 的 consumer group 只有 8 个实例。扩容后吞吐只涨了 5%,但 broker 启动时间从 15s 涨到 90s,fetch metadata 单次响应从 2KB 涨到 200KB。问题根源是 partition 数远超 consumer 并行度,导致每个 consumer 实例平均承担 7 个 partition 的拉取与提交开销,网络往返数翻倍。修复路径是"先把 consumer 扩到 60,再用 60 partition;否则保持 6 partition + 6 consumer"。教训:partition 数不是越多越好,应该匹配 consumer 并行度。

public int partition(ProducerRecord<K, V> record,
                     byte[] serializedKey,
                     byte[] serializedValue,
                     Cluster cluster) {

    Integer specifiedPartition = record.partition();
    if (specifiedPartition != null) {
        return specifiedPartition;
    }

    if (partitionerIgnoreKeys || serializedKey == null) {
        return BuiltInPartitioner.partitionForKey(null,
            cluster.partitionsForTopic(record.topic()).size());
    }

    return BuiltInPartitioner.partitionForKey(serializedKey,
        cluster.partitionsForTopic(record.topic()).size());
}

public static int partitionForKey(byte[] keyBytes, int numPartitions) {
    if (keyBytes == null) {
        return -1;
    }
    return Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions;
}

partitioner.ignore.keys=true 是生产里最容易误开的开关。它的语义是"即使消息带 key,也按 Sticky 路由,完全忽略 key 哈希"。配置文档把它描述为"可以提升吞吐",听起来很美好;但它的真实含义是同 key 不再保证同分区,直接破坏所有依赖 entity-level 顺序的下游业务。比如订单系统假设"同 orderId 的事件按顺序消费",一旦误开这个开关,你会看到 OrderPaid 出现在 OrderCreated 之前,审计与对账全乱。除非你 100% 确定这个 topic 不依赖任何 key 顺序,否则应该保持默认 false。一个稳妥的护栏是在平台层禁止业务方设置该参数,只允许走 SDK 默认。

7.3 Murmur2 哈希算法

Murmur2 是 Kafka 把 key 字节均匀打散到 partition 的核心工具。它不是 Kafka 自创,而是 Austin Appleby 在 2008 年发布的非加密通用哈希(Murmur = "MUltiply and Rotate")。Kafka 当年选它的理由是"速度足够快、雪崩效应足够好、跨语言实现成熟"。这一节把它的算法逐字节拆开,让你不仅知道"它返回一个 int",还能在排障时手动复现一次哈希。

算法整体流程

图 7-3-1 · Murmur2 算法流程

Kafka 的 Murmur2 实现源码

// org.apache.kafka.common.utils.Utils
public static int murmur2(final byte[] data) {
    int length = data.length;
    int seed = 0x9747b28c;
    final int m = 0x5bd1e995;
    final int r = 24;

    int h = seed ^ length;
    int length4 = length / 4;

    for (int i = 0; i < length4; i++) {
        final int i4 = i * 4;
        int k = (data[i4 + 0] & 0xff)
              + ((data[i4 + 1] & 0xff) << 8)
              + ((data[i4 + 2] & 0xff) << 16)
              + ((data[i4 + 3] & 0xff) << 24);
        k *= m;
        k ^= k >>> r;
        k *= m;
        h *= m;
        h ^= k;
    }

    switch (length % 4) {
        case 3:
            h ^= (data[(length & ~3) + 2] & 0xff) << 16;
        case 2:
            h ^= (data[(length & ~3) + 1] & 0xff) << 8;
        case 1:
            h ^= data[length & ~3] & 0xff;
            h *= m;
    }

    h ^= h >>> 13;
    h *= m;
    h ^= h >>> 15;
    return h;
}

Kafka 选 Murmur2 而不是其他哈希的理由

算法速度雪崩效应Kafka 适用性拒绝理由
MD5慢(100 MiB/s)优否加密级开销过重
SHA-256慢(100 MiB/s)优否加密级开销过重
CRC32快(2 GiB/s)校验级否雪崩弱,相似 key 易碰撞

"雪崩效应"是分区均匀性的数学基础。如果哈希函数对相似输入产生相似输出,那么相邻 key(如 "user-100"、"user-101"、"user-102"...)就会聚集到相邻 partition,出现明显倾斜。Murmur2 的雪崩效应让相邻 1 bit 的输入产生约 50% bit 不同的输出,等价于"近似随机分布",这才是Murmur2 % N能均匀分布到 N 个 partition 的根本原因。CRC32 之所以不适合,正是因为它的雪崩效应弱 — 相邻 key 的 CRC32 输出仅有少量 bit 变化,取模后会形成可见的倾斜模式。

7.4 toPositive 与取模

Murmur2 返回 32-bit signed int,值域是 [-2³¹, 2³¹-1]。直接拿这个值 % N 会返回负数 — Java 的 % 运算保留符号。partition 编号必须是非负的 [0, N-1],所以取模前必须先把 hash 转为非负。这一节解释为什么 Kafka 选 & 0x7fffffff 而不是 Math.abs,以及这背后藏着的一个 32 位整数的经典边界陷阱。

负数取模的问题

图 7-4-1 · Java 负数取模的符号保留
Java int 的范围: [-2^31, 2^31 - 1]

murmur2 返回 32-bit signed int:
  可能为正(0..2^31-1)
  可能为负(-2^31..-1)

负数取模在 Java 中保留符号:
  -940440895 % 12 = -3   (无效分区号)
  -940440895 % 6  = -3
   213560324 % 12 = 8

需要先转为非负:
  方案 A: Math.abs(murmur2(...))
    问题:Math.abs(Integer.MIN_VALUE) = Integer.MIN_VALUE(仍负)
    边界值崩溃

  方案 B: hash & 0x7fffffff(Kafka 选择)
    清除符号位
    所有输入返回非负值
    无边界问题

Math.abs 的边界问题

图 7-4-3 · Integer.MIN_VALUE 的溢出陷阱
Integer.MIN_VALUE = -2147483648 = 0x80000000 = 1000 0000 ... 0000

Math.abs(Integer.MIN_VALUE):
  公式: -x = ~x + 1
  ~0x80000000 = 0x7FFFFFFF
  0x7FFFFFFF + 1 = 0x80000000  (溢出回到原值)
  返回 -2147483648

后果:
  Math.abs(Integer.MIN_VALUE) % 12 = -2147483648 % 12 = -8
  仍为负数,bug 触发概率 1/2^32

Kafka 选 & 0x7fffffff:
  toPositive(0x80000000) = 0x80000000 & 0x7fffffff = 0
  所有输入均返回非负
  概率 100% 安全

hash=-940440895 positive=1207042753 partition=1

同一 key 重复运行得到相同分区。这是分区内顺序保证的基础。Java、Go、Python 客户端实现 murmur2 时只要使用相同 seed(0x9747b28c)即可保持跨语言一致 — 这一点在 §7.13 的"跨语言哈希不一致"陷阱里会展开。

不要"自己优化" Murmur2 实现。曾经有团队在 Go 客户端里用了不同 seed 或者忘了对 length 做 XOR,结果同 key 在 Go producer 与 Java producer 写出去后落到不同 partition,下游 consumer 看到"同一订单的事件被两个实例同时处理"。这种 bug 极难复现,因为它只在某些特定 key 长度下触发。修复方式只有一个:所有语言的客户端都用 Kafka 官方的 murmur2 实现 / librdkafka,不要自己重写。

7.5 历史 Round-Robin(< 2.4)

2.4 之前,Kafka 对 null key 的默认策略是 Round-Robin — 每条消息按计数器递增,依次分配到 P0、P1、P2、...、PN-1、P0、...。这个策略很直观,但它有一个严重问题:它把消息均匀打散,反而让批次饱和不起来。这一节回顾 Round-Robin 的实现与它带来的"批次碎片化"问题,解释为什么 Kafka 在 KIP-480 改成了 Sticky。

Round-Robin 算法

private final AtomicInteger counter = new AtomicInteger(0);

public int partition(...) {
    if (keyBytes == null) {
        int nextValue = counter.getAndIncrement();
        int partition = Utils.toPositive(nextValue) % numPartitions;
        return partition;
    }
    // key 路由保持 murmur2 % N
}

批次碎片化问题

    R2 → P1
    R3 → P2
    R4 → P0

    R6 → P2

    P0: [R1, R4]   (2 条,batch 未满)

    P2: [R3, R6]   (2 条,batch 未满)

  linger.ms 到期后:

7.6 Sticky Partitioner

Sticky Partitioner 由 KIP-480 在 2.4 引入,是当前 Kafka 的默认 null-key 策略。它的核心思想是"短期粘附 + 长期均匀":一段时间内所有 null-key 消息都进同一个 partition(让 batch 攒满),batch 满后切换到下一个 partition。这样既保留了 Round-Robin 的均匀性,又解决了批次碎片化问题。

Strict Uniform Sticky(KIP-794)

Kafka 3.3+ 的内置分区逻辑改进:在选新 sticky 分区时,不再纯随机,而是结合各分区的 in-flight 字节数等负载信息,避免长期粘在慢 broker 上。该机制保证长期分布均匀,同时避免 hot leader 上的累积压力。这一改动很微妙但很重要 — 它让 Sticky 在 broker 异构(部分 broker 性能低)的环境下依然能保持流量均衡。

Sticky 与 batch.size + linger.ms 是"绑定生效"的。如果你保留了 Sticky 默认但配了 linger.ms=0,Sticky 的粘附窗口被压缩到几乎为零(每条消息一发就被 flush,相当于每条都触发 onNewBatch),退化成接近 Round-Robin 的行为。要让 Sticky 真正发挥威力,linger.ms 必须 ≥ 5(让 producer 有机会攒批),batch.size 应配 32-128 KiB 让 batch 容纳足够 record。生产实践:对低延迟敏感的 topic 也至少配 linger.ms=2,几乎不影响 P99 延迟,但能显著提升吞吐与压缩比。

7.7 自定义分区器

当业务规则无法用"key 哈希"或"Sticky"覆盖时,就需要自定义 Partitioner 接口实现。典型场景:按地区路由(同一地区的请求集中到固定 partition)、按优先级分流(高优先级走专用 partition)、按租户分流(不同租户隔离)。这一节给出两个生产级示例,以及自定义分区器必须遵守的设计原则。

Partitioner 接口

public interface Partitioner extends Configurable, Closeable {
    int partition(String topic,
                  Object key, byte[] keyBytes,
                  Object value, byte[] valueBytes,
                  Cluster cluster);

    void close();

    default void onNewBatch(String topic,
                            Cluster cluster,
                            int prevPartition) {}
}

按地区分区的实现

public final class RegionPartitioner implements Partitioner {

    private static final Map<String, Integer> REGION_MAP = Map.of(
        "ASIA",   0,
        "EUROPE", 1,
        "US",     2
    );

    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        int numPartitions = cluster.partitionCountForTopic(topic);

        if (key == null) {
            return ThreadLocalRandom.current().nextInt(numPartitions);
        }

        String region = extractRegion(key.toString());
        Integer fixedPartition = REGION_MAP.get(region);

        if (fixedPartition != null) {
            return Math.floorMod(fixedPartition, numPartitions);
        }

        return Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions;
    }

    private String extractRegion(String key) {
        int idx = key.indexOf(':');
        return idx > 0 ? key.substring(0, idx).toUpperCase() : "UNKNOWN";
    }

    @Override
    public void close() {}

    @Override
    public void configure(Map<String, ?> configs) {}
}

按优先级分区的实现

public final class PriorityPartitioner implements Partitioner {

    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        int numPartitions = cluster.partitionCountForTopic(topic);

        if (key == null) {
            return ThreadLocalRandom.current().nextInt(numPartitions);
        }

        String priority = key.toString();
        return switch (priority) {
            case "HIGH"   -> 0;
            case "MEDIUM" -> 1 % numPartitions;
            case "LOW"    -> 2 % numPartitions;
            default       -> Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions;
        };
    }

    @Override
    public void close() {}

    @Override
    public void configure(Map<String, ?> configs) {}
}

自定义分区器的设计原则

原则说明反例
始终回退到默认 hash未识别 key 不应集中到 0return 0;
处理 null key单独分支抛 NPE
使用 floorMod保证非负% numPartitions(可能负)
避免阻塞业务线程调用,不能慢调用远程 RPC
测试边界numPartitions 变化下的行为仅测当前分区数

配置使用

partitioner.class=com.example.RegionPartitioner
Properties props = new Properties();
props.put(ProducerConfig.PARTITIONER_CLASS_CONFIG,
          RegionPartitioner.class.getName());
KafkaProducer<String, String> producer = new KafkaProducer<>(props);
自定义分区器最常见的事故:在 partition 方法里调用远程 RPC。partition 方法运行在业务调用 send() 的线程上,任何阻塞都会直接卡住业务线程。曾经有团队在 partition 方法里查询 Redis 决定路由策略,Redis 抖动时直接把整个业务服务的 send 线程池打满,业务接口超时。修复方式是把决策数据预加载到内存(如 ConcurrentHashMap),或者用本地 ttl 缓存,绝不能在 partition 方法里做任何 I/O 操作。

7.8 一致性哈希对比

很多分布式系统(Cassandra、DynamoDB、Memcached、Redis Cluster)用一致性哈希环来路由 key。Kafka 没用 — 它用最朴素的 hash % N。这一节把两种方案的设计目标摊开,解释 Kafka 为什么"反潮流"。

一致性哈希环

图 7-8-1 · Dynamo 风格的一致性哈希环
一致性哈希环(以 Dynamo 为例):

           Node A
            ↑
            │
      Key Z ┤ ← 顺时针
            │
            │
   Node D ──┼── Node B
            │
            │
            │
            │
           Node C

  路由:
    每个 key 映射到环上一点
    顺时针找最近 node

  增删 node:
    新增 Node E 在 A 与 B 之间
    仅 A-B 范围的 key 重定位
    其他 key 不受影响

Kafka vs Dynamo 设计对比

维度KafkaDynamo

Kafka 不用一致性哈希的四个原因

图 7-8-2 · Kafka 拒绝一致性哈希的工程理由
原因 1:分区数极少变化
  topic 创建时确定分区数
  生产环境通常不调整
  一致性哈希优势(最小迁移)用不上

原因 2:分区号是稳定的
  partition 0 始终是 partition 0
  即使 leader 切换到其他 broker
  partition 号不变 → consumer offset 稳定

原因 3:简单取模足够
  在均匀哈希假设下
  murmur2 % N 分布偏差 < 5%
  额外引入环结构无收益

原因 4:分区数变化时
  顺序边界已被新分区打散
  一致性哈希也无法保留旧分区内顺序
  Kafka 让该问题在业务层显式处理

两种系统的扩容行为差异

图 7-8-3 · 增加节点 / 增加分区的对比
场景:增加 1 个节点

Dynamo:
  T0:5 节点,数据在环上分布
  T1:加入节点 X
  T2:仅 X 的 hash 范围数据迁移到 X
  T3:其他节点数据不动
  迁移量约 1/6 总数据

Kafka(增加分区):
  T0:topic 6 分区
  T1:alter --partitions 12
  T2:新消息按 12 分区路由
  T3:旧消息留在原分区(不迁移)
  T4:同 key 新旧消息可能在不同分区
  → 顺序断裂,业务可见

Kafka(增加 broker):
  T0:3 broker,12 分区
  T1:加入 broker 4
  T2:执行 reassignment 工具迁移分区
  T3:partition 路由不变(% N 不变)
  T4:仅副本物理位置变化
  → 业务无感
"加 broker"和"加 partition"是两个完全不同的操作。加 broker 只是把现有 partition 的副本搬到新 broker 上,partition 编号与路由公式都不变,业务无感知;加 partition 才会改变 % N 的取模分母,让旧 key 跳分区。运维实践里这两个动作经常被混为一谈("我们集群扩容了"),导致业务方完全没意识到风险。沟通时一定要明确说清楚是"扩 broker"还是"扩 partition"。

7.9 分区数选择

"应该开多少个 partition"是 topic 创建时最难的决定 — 因为它几乎无法回退。分区数太少,后续扩容会引发 key 漂移;太多,broker 启动慢、metadata 大、客户端内存压力大。这一节给出一个可操作的估算公式与典型业务范围参考。

目标分区数 = ceil(max(T / Tp, T / Tc))

  T  = 业务目标总吞吐(MiB/s)
  Tp = 单分区 producer 吞吐上限 ≈ 10-30 MiB/s

  T = 100 MiB/s
  Tp = 20 MiB/s, Tc = 10 MiB/s
  分区数 = ceil(max(100/20, 100/10)) = max(5, 10) = 10

  T = 1000 MiB/s
  Tp = 30 MiB/s, Tc = 15 MiB/s
  分区数 = ceil(max(1000/30, 1000/15)) = max(34, 67) = 67 → 取 72(整除)

经验:
  Tp 与 Tc 取决于消息大小、压缩、网络
  小消息(< 1 KiB):Tp ≈ 10 MiB/s,Tc ≈ 5 MiB/s
  大消息(> 10 KiB):Tp ≈ 50 MiB/s,Tc ≈ 30 MiB/s

7.10 Key 设计模式

Key 设计是 Kafka 业务层最难的工程之一 — 它直接决定流量是否均衡、顺序是否保证、扩容是否平稳。这一节把三种典型 key 模式(实体 key / 加盐 / 复合 key)摊开,并给出 hot key 的发现与排查命令。

hot key 形成机制

图 7-10-1 · 全国电商 SKU 维度的 hot key 形成
场景:全国电商,SKU 维度发送

  Producer 发送:
    key = "SKU-IPHONE15-PRO" → partition 3
    key = "SKU-IPHONE15-PRO" → partition 3
    key = "SKU-IPHONE15-PRO" → partition 3
    ...(每秒 10000 条同 key)
    key = "SKU-OBSCURE-001" → partition 7(每秒 1 条)

  分区流量:
    P0: 100 records/s
    P1: 100 records/s
    P2: 100 records/s

    P3 leader broker 单核 100% 满载
    P3 fetch 延迟 300+ ms
    其他分区正常 < 10 ms
    P3 followers fetch 落后 → ISR 收缩
    可能触发 leader 切换抖动

加盐解决 hot key

图 7-10-2 · 加盐与时间窗口加盐
反模式:
  key = "SKU-IPHONE15-PRO"
  → 全部进同分区

正模式 1(随机加盐):
  String saltedKey = "SKU-IPHONE15-PRO-" +
                     ThreadLocalRandom.current().nextInt(10);

  → 分散到最多 10 个分区

代价:
  消费侧需聚合 SKU-IPHONE15-PRO-* 全部分区
  无法保证单 SKU 全局有序
  仅保证同 saltedKey 内有序

正模式 2(时间窗口加盐):
  String compositeKey = sku + ":" + LocalDate.now();
  → "SKU-IPHONE15-PRO:2026-05-08"
  → 同一天同一 SKU 在同分区
  → 不同天可能在不同分区,天然分散

80% 流量来自单一 SKU,确认 hot key 来源,继续在业务层加盐。

真实事故:某直播平台用 streamerId 当 key,头部主播开播时单分区被打挂。该平台的 chat-message topic 用 streamerId 作 key,假设是"同主播间的弹幕保持顺序"。设计期间这个假设没问题,因为长尾分布;但当某个头部主播开播,瞬时弹幕 50 万 QPS 全部进入同一个 partition,leader broker 的 NIC 被打满,该 partition 的 follower 复制跟不上,ISR 缩到 1,minISR 触发拒写,整个直播间弹幕停止。修复方式:把 key 改成 streamerId + ":" + (timestamp / 1000)(每秒切换 partition),牺牲"绝对全序"换取"秒级窗口内有序",压力立即均匀。

7.11 分区扩容兼容性

"我们 topic 流量涨了,加点 partition 吧" — 这是一个看似无害的运维操作,但它会直接导致同 key 跳分区,破坏所有依赖实体级顺序的下游业务。这一节用数学推导漂移率,并给出生产里可行的扩容方案。

扩容后 key 重路由示例

图 7-11-1 · 6 → 12 分区扩容时的 key 漂移
场景:topic 从 6 分区扩到 12 分区

  扩容前 hash → 6 分区 路由:
    key A: positive=1592744578, % 6 = 4
    key B: positive=1411423576, % 6 = 4
    key C: positive=318708676,  % 6 = 4
    key D: positive=807723753,  % 6 = 3
    key E: positive=1479937309, % 6 = 1

  扩容后 hash → 12 分区 路由:
    key A: 1592744578 % 12 = 10  ← 变化
    key B: 1411423576 % 12 = 4   ← 不变
    key C: 318708676  % 12 = 4   ← 不变
    key D: 807723753  % 12 = 9   ← 变化
    key E: 1479937309 % 12 = 1   ← 不变

  漂移率约 50%(理论值,均匀哈希)

漂移率的数学推导

图 7-11-2 · 漂移率公式
定理:
  分区数从 N_old 扩到 N_new,key 路由不变的概率为
  P(stable) = N_old / N_new (当 N_new 是 N_old 的整数倍)

证明:
  对均匀哈希 h,h % N_old 与 h % N_new 都均匀分布
  h % N_old == h % N_new 等价于 h 落在 [k * N_new, k * N_new + N_old) 区间
  该区间在 [0, lcm(N_old, N_new)) 范围内的占比为 N_old / N_new

应用:
  6 → 12: stable = 50%, 漂移 = 50%
  6 → 18: stable = 33%, 漂移 = 67%
  6 → 7:  stable ≈ 14%, 漂移 ≈ 86%
  N → 2N: stable = 50%
  N → kN(整数倍): stable = 1/k

四种扩容方案对比

方案做法适用场景代价
预留分区创建时分区数 × 2-4长期规划初期资源
双写过渡新旧 topic 并行写,逐步切严格有序业务数据冗余
容忍漂移直接 alter partitions短期不一致可容部分 key 乱序

扩容风险评估脚本

import hashlib

def murmur2_simulate(key_bytes):
    h = 0x9747b28c ^ len(key_bytes)
    for i in range(0, len(key_bytes) - 3, 4):
        k = int.from_bytes(key_bytes[i:i+4], 'little')
        k = (k * 0x5bd1e995) & 0xFFFFFFFF
        k ^= k >> 24
        k = (k * 0x5bd1e995) & 0xFFFFFFFF
        h = (h * 0x5bd1e995) & 0xFFFFFFFF
        h ^= k
    return h if h < 0x80000000 else h - 0x100000000

def to_positive(h):
    return h & 0x7fffffff

def calc_drift(keys, old_n, new_n):
    drift = 0
    for key in keys:
        h = murmur2_simulate(key.encode('utf-8'))
        old_p = to_positive(h) % old_n
        new_p = to_positive(h) % new_n
        if old_p != new_p:
            drift += 1
    return drift, len(keys)

keys = ["O1001", "O1002", "O1003", "C2001", "C2002"]
drift, total = calc_drift(keys, 6, 12)
print(f"drift: {drift}/{total} = {drift*100/total:.1f}%")
drift: 2/5 = 40.0%

5 个 key 的小样本仅说明计算方法,生产决策应使用至少 1000 条真实 key 样本。

partition 方法简化代码

private int partition(ProducerRecord<K, V> record,
                      byte[] serializedKey,
                      byte[] serializedValue,
                      Cluster cluster) {
    if (record.partition() != null) {
        return record.partition();
    }

    if (partitioner != null) {
        return partitioner.partition(
            record.topic(), record.key(), serializedKey,
            record.value(), serializedValue, cluster);
    }

    return BuiltInPartitioner.partitionForKey(
        serializedKey,
        cluster.partitionsForTopic(record.topic()).size());
}

BuiltInPartitioner 简化代码

public static int partitionForKey(byte[] keyBytes, int numPartitions) {
    if (keyBytes == null) {
        return -1;
    }
    return Utils.toPositive(Utils.murmur2(keyBytes)) % numPartitions;
}

四个关键设计取舍

设计取舍 1:Murmur2 vs xxHash — xxHash 速度更快但 Kafka 选定 Murmur2 时已足够。改换需破坏跨客户端兼容(Java/Go/Python 均实现 Murmur2),收益与代价不匹配。

设计取舍 2:简单取模 vs 一致性哈希 — 一致性哈希在节点频繁变动时优势明显,但 Kafka 分区数稳定的特性让该优势用不上。简单取模 + 显式分区数管理让运维边界清晰。

设计取舍 3:Sticky vs Round-Robin — Sticky 短期粘附换长期均匀,batch 充分利用率从 50% 提升到 90%+。Round-Robin 在小消息高频场景下产生大量小 batch,网络与磁盘 I/O 浪费严重。

设计取舍 4:扩容不迁移历史 — 历史数据自动按新分区迁移会破坏 offset 语义与日志顺序边界。Kafka 选择保留旧分区不动,新消息用新路由,代价是同 key 短期顺序断裂,由业务层显式处理。

反例与正例

反模式:null key + 后聚合

producer.send(new ProducerRecord<>(
    "user-events", null, payload));
// 但下游聚合需要按 userId 顺序

现象:同一 userId 消息分散到所有分区
消费侧无法保证单用户事件顺序
业务报错乱

正模式:实体 key 路由

String userId = event.getUserId();
producer.send(new ProducerRecord<>(
    "user-events", userId, payload));

现象:同 userId 消息进同分区
消费侧按分区顺序处理
业务有序
代价:若 userId 分布不均会有 hot partition

"我们用了 null key 但需要顺序"是分区策略最危险的反模式。null key + Sticky 是为无顺序要求的事件流设计的(指标、日志、临时通知)。任何依赖实体级顺序的业务都必须显式提供 key — 即使你觉得"反正下游有去重",也不要信赖默认行为。一个稳妥做法:平台层强制要求所有业务 topic 在 schema 中标注 "ordering-key" 字段,SDK 层自动用该字段当 Kafka key,业务方甚至感知不到 partition 概念,只需要在 schema 里说清楚自己依赖什么粒度的顺序。

要点依据易错点
显式 partition 优先级最高record.partition() != null跳过分区器
key 路由 = toPositive(murmur2) % NBuiltInPartitioner同 key 同分区
Sticky 短期粘附,长期均匀KIP-480仅无 key 场景
