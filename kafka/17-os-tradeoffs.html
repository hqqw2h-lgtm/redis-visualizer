图 17-1 · Produce 内核路径(序列)
关键观察:Producer 收到 ack 时,数据可能还没真正落盘!仅在 Page Cache 里。是 Linux kthread 异步把 dirty pages 写回。这就是 17.3 节"持久性靠副本不靠 fsync" 的根因 — 单机 Page Cache 不保证持久,但只要 N 个副本各自的 Page Cache 都有,就能容忍 N-1 个节点掉电。

17.2 Page Cache — Kafka 真正的缓存

Page Cache 是 Linux 在所有空闲内存里建的文件系统缓存。Kafka 的 append 行为是顺序写 + 无 fsync,这恰好与 Page Cache 的最优工作模式吻合:

  • 顺序写 → Page Cache 的 dirty page 容易合并成大 IO,writeback 高效。
  • 无 fsync → 调用立刻返回,响应延迟与磁盘解耦。
  • 读路径:tail consumer 读最新数据,几乎 100% 命中 dirty/clean cache,无磁盘 IO。

"不要给 JVM 太多堆"的根因

错误配置问题推荐
broker 机器 64GB 内存,JVM 堆设 32GBOS 只剩 30GB 缓存日志,fetch 频繁 missJVM 堆 6GB,留 56GB 给 OS
swappiness=60(默认)OS 在内存紧时会把 dirty page 换出到 swapswappiness=1(或 0,但 0 太极端)

关键 vm 参数

# /etc/sysctl.d/kafka.conf
vm.swappiness=1                    # 几乎不用 swap
vm.dirty_background_ratio=5        # 后台 writeback 触发阈值: 5% dirty
vm.dirty_ratio=80                  # 同步 writeback 阈值: 80% dirty
vm.dirty_expire_centisecs=12000    # dirty page 最长保留 2 分钟
vm.dirty_writeback_centisecs=500   # writeback 周期 5 秒
vm.max_map_count=262144            # mmap 上限,默认 65530 不够大集群

# 应用
sysctl -p /etc/sysctl.d/kafka.conf

17.3 fsync 与副本持久化

fsync 是把 Page Cache 的 dirty page 强制刷盘的系统调用。Kafka 在正常路径不调 fsync — 这一点是 Kafka 区别于很多数据库的关键。

传统数据库:WAL fsync

每条事务提交都 fsync WAL → 单机断电不丢。代价是 fsync 是 SSD/NVMe 上最慢的操作(数百 μs - 几 ms)。

Kafka:复制 + Page Cache

append 只写 Page Cache 立即 ack(给 leader 内 follower)。持久性靠 acks=all + min.insync.replicas = N 跨机房副本。代价是单机 Page Cache 仍可能丢小窗口。

配置含义影响
log.flush.interval.messages每 N 条 fsync 一次设了就走 fsync,慢
log.flush.interval.ms每 N ms fsync 一次同上
flush.messages / flush.ms(topic 级)同上同上
不要盲目配 log.flush.interval.messages=1 期望"更安全"!这会让吞吐下降 5-10 倍,而正确做法是增加副本数(min.insync.replicas=2-3)。Kafka 设计哲学是用副本而非 fsync 保证持久性。

17.4 mmap 与索引

Kafka 的 .index、.timeindex、.txnindex 都用 mmap(memory-mapped file)。这样的好处:

  • 查找时不用 syscall,直接内存访问。
  • OS 自动管理换入换出,无需手工 buffer。
  • 多线程读同一个 index 时,共享物理 page。

陷阱:vm.max_map_count

大集群 broker 上每个 partition 至少 2-4 个 .index/.timeindex mmap。
假设 broker 持有 10000 partition, 每个 50 segment, 每 segment 3 个 index:
  10000 × 50 × 3 = 1,500,000 mmap

Linux 默认 vm.max_map_count = 65530
   → broker 启动时 "Out of memory: Failed to map ..." 报错

修复:vm.max_map_count = 1048576 (1M)

# 5 epoll 上限
fs.epoll.max_user_watches=1048576
sendfile zero-copy 失效场景:① TLS(必须解密后才能发送 → 走 user-space buffer);② 压缩格式不匹配(broker 是 lz4,consumer 要 gzip → 需要重新编码);③ 远程 fetch(KIP-405,数据从 S3 来不在 PageCache)。任何一种都会让 fetch CPU 翻倍。

17.8 NUMA 与 IRQ

多 socket(物理 CPU)的服务器有 NUMA(Non-Uniform Memory Access)结构 — 每个 CPU socket 有"本地"内存,跨 socket 访问慢 30-50%。Kafka 在双 socket 机器上不做 NUMA 优化能损失 15-30% 吞吐。

# 1 检查 NUMA 结构
numactl --hardware

# 2 broker 绑到一个 NUMA node
numactl --cpunodebind=0 --membind=0 \
  java -Xmx6g -cp ... kafka.Kafka server.properties

# 3 IRQ 亲和性:NIC 中断绑到对应 NUMA node 的 CPU
# 看 NIC 在哪个 NUMA
cat /sys/class/net/ens3/device/numa_node
# 设 IRQ smp_affinity
echo <cpu mask> > /proc/irq/<num>/smp_affinity

17.11 设计取舍

① Page Cache vs 自管缓存

Kafka 选择把缓存交给 OS,而非自己实现。代价是缓存策略可调性低,收益是 sendfile 直接走 cache → 用户态。

② 无 fsync vs 强 fsync

无 fsync 让 produce 延迟从 ms 级降到 μs 级。代价是单机断电可能丢小窗口数据(几百 ms 以内),用副本兜底。

③ mmap 索引 vs read+seek

mmap 把索引访问降到 memory lookup。代价是 max_map_count 限制。

④ epoll edge-trigger 在 Selector 上

Java NIO 用 epoll level-trigger,简化编程但每次需要重新查 ready。Netty 用 edge-trigger 更高效但更复杂。Kafka broker 走 NIO,client 走 NIO 都是 level-trigger,够用。

17.13 本章要点

本章小结

Kafka 的极致性能从来都不是单纯的"broker 调参",而是建立在 Page Cache、sendfile、epoll、mmap 等一系列 OS 原语之上。理解每条数据路径对应的内核接口,你就能在故障排查时"顺藤摸瓜" — 看到 fetch CPU 飙就想到 sendfile 失效,看到 produce P99 飙就想到 swappiness / dirty_*,看到 broker 启动 OOM 就想到 max_map_count。这些知识不能被 Kafka 文档替代,它们才是真正决定 broker 性能上下限的地基。下一章我们看 Kafka 设计中引用的几个经典分布式论文与算法(Raft、Paxos、CAP、LSM 等),理解它们与 Kafka 实现的对应关系。
