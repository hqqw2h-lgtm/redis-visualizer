第 14 章 · 层级时间轮 (Hierarchical Timing Wheel)

SystemTimer · TimingWheel · TimerTaskList · DelayQueue · Purgatory · O(1) add & cancel

Kafka broker 内部一直存在着"大量短期延迟任务":Produce 等 ISR 副本 Ack、Fetch 等数据到达、JoinGroup 等成员到齐、事务等 marker 写入。这些任务有三个共性 — 数量多(单 broker 1k - 100k)、取消多(90%+ 在 timeout 前被业务事件提前完成)、精度容忍(毫秒级误差不影响业务)。如果用普通的 ScheduledExecutorService.schedule,百万级短期任务的插入与取消会让堆排序成为 CPU 瓶颈。

Kafka 选了一个更适合这个场景的数据结构:层级时间轮 (Hierarchical Timing Wheel),把 add / cancel 都降到 O(1),再用 DelayQueue 仅唤醒非空 bucket。这套机制隐藏在 DelayedOperationPurgatory 背后,几乎所有 broker 内部的"等待 X 完成"都跑这套。本章把它从数据结构、Cascade 推进、与 Purgatory 协作的全栈摊开,让你读 broker 源码时一眼看出"这个延迟操作什么时候会被叫醒"。

  • 普通 ScheduledExecutor 在大量短期任务场景下的瓶颈
  • 单层时间轮的 tickMs / wheelSize / interval 三参数关系
  • 层级 cascade:高层 bucket 到期不是执行,而是降级到低层
  • DelayQueue 为什么只放非空 bucket 而不放 task
  • Purgatory 的 watch key 表与 Timer 的协作模式
  • "业务条件先行 vs Timer 超时" 两条完成路径的 CAS 协调
  • Produce/Fetch/JoinGroup 五大延迟操作类型
  • 典型监控指标 + 内存估算 + GC 影响

前置知识:第 6 章 Producer 内部(linger.ms 与 batch);第 9 章 acks=all 的 DelayedProduce;第 11 章 KRaft 内部组件;Java 并发基础(Atomic、CAS、DelayQueue)。

14.1 为什么需要时间轮

"超时控制"在工程上看起来是一件简单的事 — Java 给了 ScheduledExecutorService.schedule,Netty 给了 HashedWheelTimer,Linux 给了 timerfd。但 Kafka 选择了自己实现一套多层级的时间轮。这一选择不是"为了酷",而是基于 broker 场景的具体测量。

普通 ScheduledExecutor 的瓶颈

操作JDK ScheduledThreadPoolExecutorKafka Timing Wheel
add(task)O(log n) — 加入 DelayQueue / PriorityQueueO(1) — 算出 bucket,挂入双向链表
cancel(task)O(log n) 或 lazy remove(留尸体)O(1) — 从链表 unlink
peek nextO(1) — heap topO(log B) — DelayQueue of buckets, B ≪ n
典型场景延迟百万级任务:P99 50-200msP99 < 1ms(只取决于 ExpiredReaper)

Kafka 场景的关键特征

Kafka 内部延迟任务的三大特点决定了时间轮是最优解:
  • 数量大:单 broker 同时挂起 1k - 100k 操作。100k 任务用堆排序,每次 add/cancel 都是 17 次比较。
  • 取消多:90%+ 操作在 timeout 前被业务事件提前完成。堆里大量 "tombstone" 拖慢 peek。
  • 精度容忍:Produce 等 ISR ACK,毫秒级误差对业务无影响。无需 ns 精度。

14.2 单层时间轮模型

先理解单层时间轮(就像一个真实的时钟刻度)。它由三个参数定义:

  • tickMs — 一个 bucket 代表的时间跨度(类似时钟"一秒")
  • wheelSize — bucket 数量(类似时钟 60 个秒刻度)
  • interval = tickMs × wheelSize — 整轮覆盖的时间跨度
图 14-2 · 单层时间轮:tickMs=1ms, wheelSize=20, interval=20ms

单层不够用 — 超出 interval 怎么办

单层时间轮的硬限制:它只能放 delay < interval 的任务。delay = 30ms 在 wheel(interval=20ms)里没地方放。这时就需要层级。

14.3 层级 cascade

层级时间轮的思路类似数字时钟:秒表满了进位到分针,分针满了进位到时针。每层 wheel 都比下一层 wheelSize 倍大。Kafka 默认 tickMs=1ms, wheelSize=20:

层级tickMs覆盖范围含义
L01ms0 - 20ms毫秒粒度
L120ms20ms - 400ms20ms 粒度
L2400ms400ms - 8s400ms 粒度
L38s8s - 160s8s 粒度
L4160s160s - ~53分160s 粒度

层级是按需创建 — 一开始只有 L0,有 task delay 超过 20ms 时才创建 L1。cascade 是层级时间轮的核心机制:高层 bucket 到期时,任务不立即执行,而是降级到低层 wheel,继续等待精度更高的位置。

图 14-3 · Cascade:从 L1 降级到 L0
cascade 到 L0 重新分配 每个 task 算出精确的 L0 bucket L1 (tickMs=20ms, 20-400ms) now b1 b2 b3 b4 L1 b1 装着 delay = 25ms 的 task 未到 L1 currentBucket,继续等 cascade 重新分配
为什么 cascade 而非直接执行
反思:L1 b1 到期了,装的 task 都 delay 25-44ms。如果直接执行,精度只到 20ms。

cascade 重新分配后:
  task delay=25ms → L0 b5 (现在 currentBucket=b0, b5 = 5ms 后)
  task delay=30ms → L0 b10
  task delay=40ms → L0 b19

精度从 20ms 提升到 1ms,但是付出的代价:
  cascade 时把 L1 b1 整桶 task 重新分配
  → 仅在 bucket 到期那一刻发生,平摊到每 ms 仍是 O(1) 期望

→ 这是层级时间轮的核心 trick

14.4 DelayQueue 优化

问题:如果时间轮已经把 add/cancel 降到 O(1),"何时检查 bucket 到期"还需要一个推进者。最直接的方案是定时每 1ms 推进一次 — 但 broker 上大量延迟操作可能瞬间集中在某 bucket,定时推进既浪费 CPU 又精度低。Kafka 的方案:用一个 DelayQueue<TimerTaskList>(注意是 bucket 不是 task!),仅非空 bucket 入队。

// SystemTimer 简化代码
class SystemTimer {
    private final DelayQueue<TimerTaskList> delayQueue = new DelayQueue<>();
    private final TimingWheel root;

    void add(TimerTask task) {
        long expireAt = task.expirationMs;
        // 1 算出 bucket(O(1)),挂入 TaskList
        TimerTaskList bucket = root.add(task);
        // 2 如果 bucket 是新非空(原来空),才入 DelayQueue
        if (bucket != null && bucket.setExpiration(bucket.getExpiration())) {
            delayQueue.offer(bucket);
        }
    }

    void advanceClock(long timeoutMs) {
        TimerTaskList bucket = delayQueue.poll(timeoutMs, TimeUnit.MILLISECONDS);
        while (bucket != null) {
            // 把 bucket 中所有 task 重新分配(cascade 或 forceComplete)
            bucket.flush(task -> reinsert(task));
            bucket = delayQueue.poll(0, TimeUnit.MILLISECONDS);
        }
    }
}
关键 trick:DelayQueue 装的是 bucket(几百-几千个)而非 task(几十万),所以 DelayQueue 的 O(log n) 操作里的 n 实际是几千,可以接受。这就是"用 bucket 化把堆排序压缩到对数级别 + 用桶链表把 task 操作降到 O(1)"的组合拳。

14.5 Purgatory 与 Timer 协作

SystemTimer 解决了"超时兜底",但 broker 的延迟操作还有另一个完成路径:业务条件满足。比如 DelayedProduce 等的是"ISR 全员复制完" — 这事件可能发生在 timeout 之前。DelayedOperationPurgatory 是负责协调这两条路径的组件。

图 14-5 · DelayedOperation 的两条完成路径

Watch Key 机制

"watch key" 是 Purgatory 的索引手段。当 DelayedProduce 加入 Purgatory 时,它声明"我在关注 topicPartition=order-created-0 的状态变化"。Purgatory 维护一个 Map<Watchable, List<DelayedOperation>>。当任何 broker 内部代码调用 checkAndComplete(key),Purgatory 就把那个 key 下的所有 DelayedOperation 试一遍 tryComplete()。

// DelayedOperationPurgatory 简化代码
def tryCompleteElseWatch(op: DelayedOperation, watchKeys: Seq[Any]): Boolean = {
  // 1 先试一下能否立即完成
  if (op.tryComplete()) return true

  // 2 不能 → 加入每个 watch key 的列表
  watchKeys.foreach { key =>
    watchersFor(key).addOperation(op)
  }

  // 3 再加入 timer
  timeoutTimer.add(op)
  true
}

def checkAndComplete(key: Any): Int = {
  val watchers = watchersFor(key)
  // 把所有等这个 key 的 op 都试一遍
  watchers.tryCompleteWatched()
}

add 核心代码

// TimingWheel.add
def add(timerTaskEntry: TimerTaskEntry): Boolean = {
  val expiration = timerTaskEntry.expirationMs
  if (timerTaskEntry.cancelled) return false

  if (expiration < currentTime + tickMs) {
    // 已经过期 — 不放入 wheel
    false
  } else if (expiration < currentTime + interval) {
    // 在本 wheel 范围内
    val virtualId = expiration / tickMs
    val bucket = buckets((virtualId % wheelSize).toInt)
    bucket.add(timerTaskEntry)
    // 如果是新非空,设 expiration 并返回需要入 DelayQueue
    if (bucket.setExpiration(virtualId * tickMs)) {
      queue.offer(bucket)
    }
    true
  } else {
    // 超出本 wheel,递归到上一层
    if (overflowWheel == null) addOverflowWheel()
    overflowWheel.add(timerTaskEntry)
  }
}

14.8 设计取舍

① 层级 vs 单层

单层时间轮覆盖的 interval 有限,要覆盖 1 小时就要 3.6M 个 bucket。层级用 log_w(N) 层级压缩,实际 5-6 层够用。

② Bucket-level DelayQueue

DelayQueue 装 bucket 而非 task,把堆排序的规模从 n=百万 缩到 b=几千。cancel 不需要从 DelayQueue 删除(只需 unlink 链表)。

③ Cascade vs 直接超时

高层 bucket 到期不立即执行,而是 cascade 到低层。代价是 cascade 时的瞬时 O(bucket size),收益是整体精度上升到 tickMs。

④ CAS 双路完成

Purgatory 业务路径和 Timer 超时路径并行运行,用 CAS 保证只有一条路径完成响应。代价是少量原子操作,收益是不需要锁。

本章小结

层级时间轮是 Kafka 内部一个"不显眼但极致工程化"的数据结构。它把延迟操作的 add / cancel 都降到 O(1),用 bucket-level DelayQueue 把堆排序的规模从百万压到千级,用 cascade 解决"高层精度低"的问题,再用 Purgatory 协调"业务条件满足 vs Timer 超时"两条完成路径。所有这一切的目的是让 broker 能同时挂起 100k+ 延迟操作而不影响主路径性能。下一章我们看 Kafka 网络层另一个核心组件:Reactor + IO Multiplexing,即 broker 如何用一组 N 个 Network Thread 处理上万并发连接。
