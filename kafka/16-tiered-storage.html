第 16 章 · 分层存储 Tiered Storage (KIP-405)

RemoteLogManager · RemoteStorageManager · RemoteLogMetadataManager · S3/GCS/Azure · local.retention · catch-up consumer

"我有 10 万 partition,每个 partition 想保留 90 天数据用于回放" — 这个需求在 KIP-405 之前几乎只能靠"加 SSD"硬扛。10 万 partition × 100MB/min × 90 天 = 几 PB SSD,成本爆炸,扩缩容滚动几天结束。Kafka 3.6 正式 GA 的分层存储 (Tiered Storage) 通过一个新抽象解决这个矛盾:active segment + 近期 closed segment 留在本地 SSD,历史 closed segment 自动上传到对象存储(S3 / GCS / Azure Blob / HDFS),broker 仅在catch-up consumer 跨过本地 retention 边界时才从远端读回。

分层存储的核心承诺:节省 SSD,不影响热路径性能。这是真的吗?本章我们从组件拓扑开始,逐层拆解:① 三件套 RLM/RSM/RLMM 各自职责;② 上传状态机和 idempotency;③ __remote_log_metadata internal topic 的设计;④ 远程 Fetch 怎样从 Network Thread "不阻塞"地完成;⑤ 双重 retention 策略(local + total);⑥ 失败恢复;⑦ 监控关键指标。看完后,你能不靠 SDK 就预测一次远程 Fetch 的完整代码路径。

  • 什么场景适合分层存储,什么场景不适合
  • RLM/RSM/RLMM 三件套 + __remote_log_metadata internal topic
  • Segment 上传的状态机:STARTED → FINISHED → DELETE_STARTED → DELETE_FINISHED
  • "元数据先于数据"的可见性规则
  • 本地保留 (local.retention.*) 与全局保留 (retention.*) 的二维设计
  • 远程 Fetch 在 Network Thread 之外的 "异步 fetch" 实现
  • 缓存层:Segment + Index + LeaderEpoch 都被 LRU cache
  • Broker 失联 → 上传半途时的恢复策略

前置知识:第 2 章日志存储引擎;第 4 章索引体系;第 9 章 ISR;Java SPI;S3 / 对象存储基础(分段上传、Range Get)。

16.1 适用与不适用场景

✅ 适合分层存储

  • 业务保留 > 30 天
  • 消费者最大 lag < 本地 retention(冷读罕见)
  • 有合规要求需要长期可回溯
  • 对象存储成本 ≤ SSD 的 1/5
  • topic 数据量 PB 级

❌ 不适合分层存储

  • 业务保留 < 7 天 — 直接本地 retention
  • 大量 catch-up 消费者(冷读频繁,远端 IO 费用爆炸)
  • Compacted topic — 不支持(KIP-405 仅 delete policy)
  • 极低延迟要求(冷读 P99 数百 ms)
  • 对象存储不稳定 / RTT 高

16.2 三件套组件

RemoteLogManager (RLM)

RLM 是 broker 内置组件,启动时初始化。它有一个 专用线程池(remote.log.manager.thread.pool.size,默认 10),负责调度三件事:

  • Copy:扫描每个 leader partition,把已 close 且未上传的 segment 上传到 RSM。
  • Cleanup:按 retention 策略删除过期 remote segment。
  • Read:在 KafkaApis.handleFetch 时,如果 fetch offset < local log start offset,委托给 RLM 异步从 RSM 读。

RemoteStorageManager (RSM) — SPI

RSM 是 Kafka 暴露的 SPI 接口,没有内置实现。供应商(AWS / Confluent / Aiven)各自提供:

public interface RemoteStorageManager extends Configurable, Closeable {
    void copyLogSegmentData(RemoteLogSegmentMetadata m, LogSegmentData data) throws RemoteStorageException;

    InputStream fetchLogSegment(RemoteLogSegmentMetadata m, int startPos) throws RemoteStorageException;
    InputStream fetchLogSegment(RemoteLogSegmentMetadata m, int startPos, int endPos) throws RemoteStorageException;

    InputStream fetchIndex(RemoteLogSegmentMetadata m, IndexType indexType) throws RemoteStorageException;

    void deleteLogSegmentData(RemoteLogSegmentMetadata m) throws RemoteStorageException;
}

RemoteLogMetadataManager (RLMM) — SPI

RLMM 维护"哪些 segment 上传过、状态如何、对应的 leader epoch、index 位置"等元数据。默认实现 TopicBasedRLMM 把元数据持久化到 internal topic __remote_log_metadata 里,每个 broker 都消费这个 topic 在内存里建索引。

16.3 上传:本地 → 远程

当一个 partition 的 segment roll(close,变成只读)后,RLM 调度器会在下一次扫描时把它选中上传。上传是一个有状态机的过程,这个状态机存在 RLMM 里:

图 16-3 · Segment 上传状态机

上传过程的"元数据先行"原则

// RLM.copySegment 简化
void copyLogSegment(LogSegmentData segment) throws Exception {
    RemoteLogSegmentId id = new RemoteLogSegmentId(...);
    RemoteLogSegmentMetadata meta = new RemoteLogSegmentMetadata(id, state: COPY_SEGMENT_STARTED, ...);

    // 1 元数据先写
    rlmm.addRemoteLogSegmentMetadata(meta).get();

    try {
        // 2 实际上传字节
        rsm.copyLogSegmentData(meta, segment);

        // 3 更新元数据为 FINISHED
        rlmm.updateRemoteLogSegmentMetadata(meta.withState(COPY_SEGMENT_FINISHED)).get();
    } catch (Exception e) {
        // 失败 → 留着 STARTED 状态,后续清理流程会兜底删除
        log.error("upload failed", e);
        throw e;
    }
}
为什么元数据先写:如果先上传字节再写元数据,broker 在中间挂了,远程有"孤儿字节"找不到所属 segment,无法清理。先写元数据(状态=STARTED),即使 broker 挂了,后续 cleanup 也能根据"STARTED 状态超期"找到孤儿 segment 并触发清理。

16.4 元数据 RLMM 内部

__remote_log_metadata 是个特殊 topic — 50 个 partition(默认),replication factor 3。它的 partition 数决定了元数据并发上限。每个 user topic-partition 的元数据会落到固定的 metadata partition,用 (topicIdPartition.hashCode % numMetadataPartitions) 计算。

Record 类型含义
RemoteLogSegmentMetadataRecord新 segment 元数据(创建)
RemoteLogSegmentMetadataUpdateRecord状态变化(STARTED → FINISHED / DELETE_*)
RemotePartitionDeleteMetadataRecord整个 partition 删除标记
RemoteLogSegmentMetadataSnapshotRecordsnapshot,加快 broker 重启

每个 broker 都消费 __remote_log_metadata

不只是 leader broker 关心元数据。每个 broker 都启动一个 ConsumerTask 消费 __remote_log_metadata,在本地内存里维护:

broker-1 内存视图:
  topic A-partition-0 (leader 在 broker-2):
    segment[0..999]    → COPY_SEGMENT_FINISHED, URI=s3://bucket/...
    segment[1000..1999] → COPY_SEGMENT_FINISHED, URI=s3://bucket/...

  topic A-partition-1 (leader 在 broker-1):
    segment[0..999]    → COPY_SEGMENT_FINISHED, URI=s3://bucket/...
    segment[1000..1500] → COPY_SEGMENT_STARTED  (正在上传)

→ 任何 broker leadership 变更时, 新 leader 立刻知道哪些 segment 已上传

16.5 远程 Fetch 路径

客户端 fetch 一个历史 offset,broker 怎样从远端读回来?关键是不能在 Network Thread 阻塞(网络线程同步等 S3 IO 会卡死所有连接)。Kafka 用一个异步 fetch 池(remote.log.reader.threads,默认 10)接手。

图 16-5 · 远程 Fetch 完整路径
关键点:Network Thread / KafkaApis 自始至终不阻塞。DelayedRemoteFetch 是 Purgatory 注册的延迟操作(同第 14 章机制),等远程 IO 完成后由 RemoteLogReaderThread 回调 forceComplete。这样即使 S3 偶发 200ms 抖动,也不会影响热路径的 fetch P99。

16.7 失败恢复

场景状态恢复机制
上传中 broker 挂状态仍是 STARTEDRLM 重启扫描:超过 remote.log.manager.task.interval.ms 仍 STARTED → 调 rsm.delete 清孤儿

远端 segment 损坏fetch 时校验失败记录 alarm,标记 segment 不可用

16.8 性能与缓存

分层存储在冷读路径上的性能取决于三件事:RSM 单次 Range Get RTT、并发数、本地缓存命中率。Kafka 在 RLM 里内置了几层 cache:

Cache对象大小配置命中作用
Segment index cacheOffsetIndex / TimeIndexremote.log.index.file.cache.total.size.bytes = 1GiBfetch lookup 时不必再下 index
Leader epoch cacheLeaderEpochCheckpoint同上truncation 检查不必拉远端
(用户可选) PageCache本地 segment cache 文件OS PageCache多消费者同 offset 重复读
性能基线(参考 Confluent benchmark):本地 fetch P99 < 5ms,远程 fetch P99 50-200ms(S3 same-region)、500ms-2s(S3 cross-region)。一旦 catch-up consumer 数量过多,S3 请求并发被限流,P99 可能飙到秒级。这就是 16.1 节"不适合频繁冷读"的根因。

16.10 设计取舍

① 本地 + 远程 二级 vs 全远程

全远程会让 tail fetch 走 S3 — 延迟数十 ms,无法承接 Kafka 主要场景(实时流)。二级架构让热路径完全保持原 Kafka 行为,只在冷路径承担额外延迟。

② RSM 为 SPI 不内置

对象存储 API 差异大(S3 / GCS / Azure / HDFS 协议都不同),内置任何一个都偏袒。SPI 让供应商各自实现。代价:开源用户需挑选可用实现。

③ 元数据用 internal topic 而非外部 KV

用 Kafka 自己的 topic 做元数据存储,broker 重启自动 replay,无需引入额外 DB。代价:50 partition 元数据 topic 是热点,在 broker 重启大量时 replay 慢。

④ DelayedRemoteFetch 用 Purgatory

复用现有的 Purgatory + Timer 机制,无新增组件。代价:Purgatory 大小可能因冷读暴涨。

本章小结

分层存储 (KIP-405) 是 Kafka 3.6+ 最重要的存储侧演进 — 它让"保留 90 天甚至 7 年的全量数据"从昂贵 SSD 集群变成"SSD + S3"组合,成本下降 5-10 倍而对热路径性能几乎无影响。核心架构是三件套(RLM / RSM / RLMM)+ internal metadata topic(__remote_log_metadata)+ 状态机(STARTED → FINISHED → DELETE_*),所有这些一起保证"元数据先于数据"的可见性规则。

用好分层存储要记住两条:① 评估业务最大 consumer lag,确保 < local.retention(否则冷读频繁,远端 IO 费用爆炸);② 监控 RemoteCopyLagBytes / RemoteDeleteLagSegments,任何一个持续涨都意味着上传 / 清理被卡。下一章我们看 broker 的另一面:OS 取舍,即 PageCache / mmap / sendfile / TCP 参数,这些"看似 OS 知识"如何决定 Kafka 极限吞吐。
