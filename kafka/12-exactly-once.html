第 12 章 · Exactly-Once 语义

幂等 Producer · 事务 Producer · Transaction Coordinator · read_committed · LSO · 两阶段提交

"Exactly-Once" 一词容易引发误解。它不是"消息只发送一次"的物理保证(物理上 producer 仍可能重发),而是一组工程约束的组合,让消费者看到的结果等价于"消息只被处理一次"。Kafka 的 EOS 由三件套实现:幂等 Producer 用 (PID, epoch, sequence) 三元组在 broker 端做单分区去重;事务 Producer 用 Transaction Coordinator 驱动跨分区原子提交;Consumer 用 isolation.level=read_committed 配合 LSO (Last Stable Offset) 屏蔽未提交数据。

本章按 "问题边界 → 幂等 Producer → Sequence/PID 机制 → 事务 Producer → TC 状态机 → read_committed → 监控 → 源码 → 取舍 → 陷阱" 组织,对应 Apache Kafka 3.7。EOS 是从 0.11 引入(KIP-98),3.0 进入第二代(KIP-447 让消费者-生产者模式更简洁),3.7 仍在持续完善。读完后你能区分什么场景需要 EOS、什么场景不需要,并预判每种异常的根因。

  • At-least-once 的两类重复:Producer 重试重复 + Consumer 重启重复
  • EOS 的边界 — 它能覆盖什么,不能覆盖什么(外部数据库副作用)
  • 幂等 Producer 的 (PID, partition, sequence) 三元组去重原理
  • Sequence 鸿沟检测:OutOfOrderSequence vs DuplicateSequence
  • 事务 Producer 的两阶段提交:Prepare → WriteTxnMarker
  • Transaction Coordinator 的状态机和 __transaction_state 日志
  • LSO (Last Stable Offset) 与 HW 的关系,read_committed 如何屏蔽未提交
  • aborted transaction index 文件的作用与 consumer 端跳过
  • Producer Epoch Fencing 如何防止 zombie producer 写入
  • EOS 性能开销与是否值得

前置知识:第 6 章 Producer 内部 (acks / retries / max.in.flight);第 9 章 ISR (acks=all 是 EOS 前提);第 11 章 KRaft (TC 元数据在 KRaft 中如何承载)。

EOS 的本质是 "把不可避免的重发" 变成 "对消费者透明的去重"。物理网络上消息可能被重发 N 次,但 broker 端用 sequence 去重 + consumer 端用 LSO 屏蔽 + transaction marker 控制可见性,让"业务下游看到的结果"等价于消息只被处理一次。它不是无成本的:吞吐降低 5-30%、延迟升高 5-50ms、运维多一个 Transaction Coordinator 要看。

12.1 EOS 要解决什么

At-least-once 的两类重复来源是 EOS 要解决的核心问题。理解这两类问题再看 EOS,逻辑就非常清晰。

图 12-1 · At-least-once 的两类重复

EOS 不能解决什么

EOS 是 Kafka 内部的端到端保证。它不能解决:
  • Consumer 处理过程中调用的外部 HTTP / DB / 第三方服务 — 这些副作用 Kafka 看不到,业务侧仍需自己做幂等
  • Consumer 输出到非 Kafka 系统 — 必须自己实现 sink 端幂等
  • Producer 在事务外发送的消息 — 它们不受 EOS 保护
  • 由于 client bug 重复发送的逻辑级重复(同一业务事件被发了两次)
EOS 覆盖的范围是 "Kafka → Kafka" 的消费-处理-生产链路。要把 EOS 延展到外部系统,需要 outbox pattern / 2PC connector(如 Debezium)。

12.2 幂等 Producer

幂等 Producer 是 EOS 的底座。开启 enable.idempotence=true 后,Producer 启动时会向任意一个 broker 申请一个全局唯一的 Producer ID (PID) + epoch,之后每个 batch 都带上 (PID, partition, sequence) 三元组。Broker 在分区 leader 上维护一个滑动窗口的 sequence 状态,可以识别"我已经见过这个 batch 了"或"出现了 gap"。

开启幂等的最小配置

enable.idempotence=true

# Kafka 自动检查的前置条件(violation 时启动失败):
acks=all                              # 否则没意义 — 单副本写入算什么幂等
retries > 0 (默认 Integer.MAX_VALUE)  # 否则失败直接抛,不重试
max.in.flight.requests.per.connection <= 5

Sequence 状态机

图 12-2 · Broker 端 sequence 三种情形
静默丢弃,不写入 响应: 200 OK + 原 offset → Producer 看到的 ACK,如同新写入 情形 3 · 鸿沟(seq > lastSeq+1) 收到 batch seq=103 期望: lastSeq + 1 = 101 实际: seq=103 (跳了 101,102) 动作: 拒绝 错误: OutOfOrderSequence → Producer 端不可恢复,需重启 关键:broker 维护的 sequence 状态在分区 leader 切换时通过 ProducerState snapshot 同步给新 leader, 保证幂等保证不会因为 leader failover 失效。

ProducerState snapshot

分区目录:
/var/kafka-logs/order-created-0/
  ├── 00000000000000000000.log
  ├── 00000000000000000000.snapshot      ← Producer 状态快照
  ├── 00000000000000005000.snapshot
  └── leader-epoch-checkpoint

snapshot 内容(每个活跃 PID 一项):

  PID=42  epoch=3  lastSeq=199  lastOffset=2103  lastTimestamp=...
  PID=43  epoch=1  lastSeq=88   lastOffset=2104  ...

作用:
  - leader failover 时, 新 leader 从 snapshot 恢复 ProducerState
  - log compaction 不删除 PID 状态(即使该 PID 没有新写入)
  - Producer 端 retries 时, broker 仍能识别去重

12.3 Sequence 与 PID

PID 申请流程

Producer.send() 首次调用前:

  1 Producer 发 InitProducerIdRequest (无 transactional.id)
    任意 broker 收到, 转给 ProducerIdManager 分配
    ProducerIdManager 在 controller 端按 block 分配 (1000 个 PID 一批)

  2 broker 返回 (PID, epoch=0)
    Producer 缓存这两个值

  3 Producer 之后所有 send 都带上 PID + epoch + 递增 seq

幂等 Producer 与事务 Producer 的区别:
  - 幂等:每次启动新申请 PID, 进程死了 PID 就废了
  - 事务:transactional.id 是用户提供的字符串, PID 在重启后被恢复

Sequence 越界场景

错误触发影响恢复
OutOfOrderSequenceException序列跳号(broker 收到 seq=103 而期望 101)本 Producer 实例对该分区不可用抛给上层,通常重启
DuplicateSequenceException序列重复(静默处理,通常不抛)无,broker 返回原 offset不需要
UnknownProducerIdExceptionbroker 的 ProducerState 已过期(默认 24h)该 PID 失效Producer 自动重新申请 PID

12.4 事务 Producer

幂等 Producer 只能保证单分区的去重。当业务需要"一次 send 写多个分区,要么全成功要么全失败"时,需要事务 Producer。它通过 transactional.id + Transaction Coordinator 的两阶段提交协议实现。

事务 Producer 使用示例

// Producer 配置
props.put("enable.idempotence", "true");
props.put("transactional.id", "order-tx-producer-1");

KafkaProducer<String, String> producer = new KafkaProducer<>(props);

producer.initTransactions();   // 1 注册 transactional.id, 申请 PID/epoch, 还可能 abort 上次未完成事务

try {
    producer.beginTransaction();              // 2 标记事务开始

    producer.send(record1);                   // 3 多分区写入
    producer.send(record2);

    // 4 同时把 consumer offset 也"写入"事务
    producer.sendOffsetsToTransaction(
        consumerOffsets, consumerGroupMetadata);

    producer.commitTransaction();             // 5 两阶段提交
} catch (ProducerFencedException e) {
    producer.close();                         // 被新 producer 实例 fence
} catch (KafkaException e) {
    producer.abortTransaction();              // 6 任何异常 → 回滚
}

12.5 Transaction Coordinator

TC 是一个特殊的"逻辑组件",由某个 broker 担任(由 transactional.id 的 hash 决定哪个 broker)。它不是一个独立进程,而是 broker 内部的子系统。所有事务的状态都写入内部主题 __transaction_state(默认 50 个 partition,RF=3,log-compacted)。

TC 状态机

状态含义下一步
EMPTY新事务 ID,无活跃事务InitProducerId → ONGOING
ONGOING事务进行中EndTxn → PREPARE_COMMIT/ABORT
PREPARE_COMMIT第一阶段已记录,准备写 markermarker 全部写入 → COMPLETE_COMMIT
COMPLETE_COMMIT事务完成可被覆盖 / log-compacted
PREPARE_ABORT准备回滚marker 写入 → COMPLETE_ABORT
COMPLETE_ABORT回滚完成同 COMPLETE_COMMIT
DEADTC failover 检测到 hanging 事务,触发清理fence + abort

Producer Epoch Fencing

场景:Producer A 进程被杀,新进程 B 启动重连同一 transactional.id

  1 Producer A 用 transactional.id="tx1" 拿到 (PID=42, epoch=5)
    正在写 record, 但还未 commit
    A 进程被 kill -9

  2 Producer B 启动, 重连同一 transactional.id="tx1"
    TC 收到 InitProducerIdRequest:
      - 发现已有 PID=42, epoch=5
      - 分配新 epoch=6
      - 标记 epoch=5 为 fenced
      - 如有 ongoing 事务, 触发 abort

  3 Producer A 进程恢复(网络分区结束?), 用 epoch=5 继续发 record
    broker 检查: lastSeenEpoch=6 > request.epoch=5
    → 返回 ProducerFenced 错误
    → A 端 catch 后必须 close, 不能再继续写

→ 这是 EOS 的关键 "zombie fencing"
→ 保证同一 transactional.id 只有最新进程实例能写

12.6 read_committed 与 LSO

Producer 写完事务后,数据已经在分区日志里,但 consumer 看到没看到取决于 isolation.level:

  • read_uncommitted(默认):看到所有数据,包括未提交的
  • read_committed:仅看到 offset < LSO 的数据,自动跳过 aborted 事务

LSO (Last Stable Offset) 是 HW 的进一步约束:它是 HW 和"最早未完成事务的 firstOffset" 中较小的那个。开启的事务会阻塞 LSO 推进,但不影响 HW。

图 12-6 · LSO 与 HW 的关系

aborted transaction index

分区目录里有一个特殊文件 *.txnindex,记录所有 aborted 事务的 (firstOffset, lastOffset, PID)。FetchResponse 给 read_committed consumer 时会一并返回这个范围列表,consumer 收到后自动跳过对应 offset 范围。

/var/kafka-logs/order-created-0/
  ├── 00000000000000000000.log
  ├── 00000000000000000000.index
  ├── 00000000000000000000.timeindex
  ├── 00000000000000000000.txnindex     ← aborted 事务范围索引
  └── ...

文件内容(每条 32 bytes):
  ┌──────────┬──────────┬──────────┬──────────┐
  │ version  │ pid      │ first    │ last     │
  │ int16    │ int64    │ int64    │ int64    │
  └──────────┴──────────┴──────────┴──────────┘

12.7 监控与排障

JMX含义异常阈值
kafka.coordinator.transaction:type=TransactionMarkerChannelManagermarker 队列长度持续 > 0
TransactionExpirationCount事务过期数突增
OngoingTransactionMaxDurationMs最长 ongoing 事务> transaction.timeout.ms
kafka.server:type=BrokerTopicMetrics,name=PartitionsCount分区数对照 baseline
ProducerExpirationCountPID 过期数突增

典型问题

问题:read_committed 消费者突然 lag 飙升 
现象:
  consumer lag 持续增加, 但 producer 仍在以正常速率写入
  jmx OngoingTransactionMaxDurationMs = 600000 (10 分钟)

根因:
  某个 transactional.id 的 Producer 实例死了,事务没 commit 也没 abort
  TC 在等 transaction.timeout.ms (默认 60s) 后自动 abort
  在 abort 之前, LSO 卡在该事务的 firstOffset

排查:
  1 查 __transaction_state 找出 ongoing 事务的 transactional.id
     bin/kafka-transactions.sh --bootstrap-server ... list
  2 找到长期 ongoing 的 transaction
  3 通常自动 abort 后恢复,如果 producer 还在运行则要确认是否 hang

动作:
  - 等 transaction.timeout.ms 自动 abort
  - 或手动 kill 该 transactional.id:
    bin/kafka-transactions.sh --bootstrap-server ... abort \
      --transactional-id order-tx-producer-1

12.9 设计取舍

① Sequence 单分区去重

幂等 Producer 仅做单分区去重,不做跨分区原子性。代价是简单可靠,broker 状态轻量。要跨分区原子性需要事务 Producer。

② TC 中心化

事务状态集中在 TC(单 broker),用 __transaction_state 持久化。代价是 TC 是事务热点;收益是协议简单,无需多 broker 共识。

③ WriteTxnMarker 控制消息

commit/abort marker 写入参与分区作为"事务边界"。代价是每个分区多了几条控制消息;收益是 consumer 无需查 TC 即可判断事务状态。

④ Producer Epoch Fencing

同 transactional.id 重连 → epoch+1,旧实例被 fence。代价是 epoch 持久化开销;收益是 zombie producer 不能写脏数据。

本章小结

EOS 是 Kafka 在"分布式正确性"这一维度的最高保证 — 但它的范围严格限定在 Kafka 集群内部。幂等 Producer 用 sequence 解决重试重复,事务 Producer 用两阶段提交解决跨分区原子性,read_committed Consumer 用 LSO 屏蔽未提交。三者环环相扣,但只覆盖"Kafka → Kafka"链路。任何外部副作用(HTTP、DB、缓存)仍需业务自己做幂等。下一章 Log Compaction 我们换个角度看 Kafka 日志 — 不再是"只追加",而是基于 key 的最新值保留机制,这正是 __consumer_offsets 和 __transaction_state 之所以能"无限保留"的关键。
