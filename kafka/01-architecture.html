第 1 章 · 全局架构

三平面分离 · KRaft Quorum · RecordAccumulator · KafkaApis · UnifiedLog · ISR / HW · LeaderEpoch · 故障 Playbook

Kafka 不是普通的消息队列,也不只是分布式日志系统——它是一台把分区追加日志包装成高吞吐流式中间件的精密机械。打开外壳,你会看到三个独立运转的平面:数据面(Producer / Broker / Consumer)在毫秒级把字节往磁盘上推,控制面(KRaft Controller Quorum)用 Raft 协议维护拓扑事实,存储面(UnifiedLog + LogSegment + 稀疏 index)把每个分区物化为磁盘上有序的 segment 文件。三个平面共享同一个 broker 进程,但线程模型独立、协议格式独立、故障语义独立——任何看不清三平面边界的认知,都会在事故复盘时把你引向错误的根因。

这一章的目标是把这台机械的全局拓扑、关键链路、故障语义一次性铺平。从三节点 KRaft 集群的最小可观测拓扑,到 Producer 通过 RecordAccumulator + Sender 线程把 batch 推上网络;到 Broker 用 Acceptor + Processor + Handler 三级线程模型把 ProduceRequest 分派进 ReplicaManager;到 Controller Quorum 在 __cluster_metadata 上跑 Raft 共识、broker 端 MetadataPublisher 把变更应用到本地 cache;到 Consumer Group 的 JoinGroup → SyncGroup 协议;到 UnifiedLog 的 segment + 稀疏 index 物理布局;到 ISR + HW 的复制推进规则;再到 broker 失联后 controller fence + leader 重选的故障传播序列;最后是 KafkaServer.startup 的源码主干和九条架构陷阱。所有讨论锚定 Apache Kafka 3.7 的 KRaft 模式,ZooKeeper 模式仅作历史对照。

  • Kafka 三平面(数据面 / 控制面 / 存储面)的边界、职责、故障敏感度,以及它们在 broker 进程内如何共享线程
  • 三节点 KRaft 拓扑的最小启动序列:kafka-storage.sh format → controller leader 选举 → broker 装配 → MetadataLoader 应用快照
  • Producer 的内部数据流:Serializer → Partitioner(murmur2 / Sticky) → RecordAccumulator deque → Sender 线程 → NetworkClient,业务线程与 I/O 解耦的设计
  • Broker 的 Reactor 线程模型:Acceptor(1 个) + Processor(num.network.threads) + KafkaRequestHandler(num.io.threads),以及请求耗时的六段分解(QueueTime / LocalTime / RemoteTime / ThrottleTime / ResponseQueueTime / ResponseSendTime)
  • KRaft 的 metadata log 模型:__cluster_metadata 由 Controller Quorum 维护、broker 通过 MetadataFetch 拉取、BrokerMetadataPublisher 应用到本地子系统
  • Consumer Group 的协议演进:Eager → Cooperative Sticky → Static Membership → KIP-848(server-side rebalance);JoinGroup / SyncGroup / Heartbeat / OffsetCommit 四类请求
  • 分区日志的物理布局:baseOffset.log + .index + .timeindex + .snapshot,UnifiedLog 与 LocalLog / ProducerStateManager / LeaderEpochFileCache 的组合关系
  • 复制协议:Follower 通过 Fetch 模拟 consumer 拉取 leader 日志,LEO 与 HW 的推进规则,ISR 由 replica.lag.time.max.ms 决定
  • 故障传播序列:Broker 心跳超时 → Controller fence → metadata log 写 PartitionChangeRecord → broker 应用切换 → 客户端 NOT_LEADER_OR_FOLLOWER → metadata 刷新
  • KafkaServer.startup 的源码主干:SocketServer / LogManager / ReplicaManager / KafkaController / GroupCoordinator / TransactionCoordinator 的装配顺序
  • 九条架构陷阱:bootstrap 当全集群 / acks=all 不丢 / ISR == replicas / fsync 才持久 / controller 是热点 / KRaft 单点 / partition 数随便加 / metadata 永不刷新 / log.flush 同步刷盘

前置知识:基础的 TCP / 多线程 / 文件 I/O 概念;了解过任意一种 MQ(RabbitMQ / RocketMQ / SQS)有助于对照;了解 Raft 共识或 ZooKeeper 的角色概念能更快理解控制面。后续 第 2 章会深入 UnifiedLog 与 LogSegment 的 segment 滚动 + 稀疏索引 + 零拷贝读路径,第 3 章会展开 KRaft Quorum 的 Raft 协议细节,第 4 章会进入 Producer / Consumer 的网络层与协议帧。本章是后续所有章节的"地图"——读懂它再读细节,不容易迷失方向。

1.1 最小可观测集群

理解 Kafka 的最快路径是先搭起一个三节点 KRaft 拓扑——这是生产能用的最小配置,而不是教学示例:3 个 controller voter 构成 Raft Quorum,允许任意 1 个节点失联仍可写。每个节点同时承担 broker + controller 双重角色,通过 process.roles 配置开启。

图 1-1-1 · 三节点 KRaft 拓扑(controller + broker 合并部署)

KRaft 模式下每个节点可以同时承担 broker 与 controller 两种角色,由 process.roles 控制。生产部署常用 3 个 controller voter 形成 quorum,其余节点仅作为 broker 加入。Controller 之间通过端口 9093 运行 Raft 协议,broker 与客户端通过端口 9092 交互。

kafka-storage.sh format 仅在首次启动时执行,生成 meta.properties 锁定 cluster.id。后续重启不重复 format,避免覆盖元数据。Controller leader 转换日志由 RaftManager 输出,与 partition leader 是不同概念。

1.2 Producer 到 Partition 的路径

Producer 不会直接将记录发给 topic,而是先选定 TopicPartition,然后定位该分区当前的 leader broker。选定分区的逻辑由 Partitioner 实现,Kafka 默认提供两种策略:有 key 的记录通过 murmur2(key) % numPartitions 映射,无 key 的记录由 sticky partitioner 在批次粒度上保持分区粘附。

图 1-2-1 · Producer 内部数据流(业务线程 → Sender → 网络)
图 1-2-2 · RecordAccumulator 批次组装时序

业务线程调用 send() 后,记录被序列化并追加到 RecordAccumulator 中对应分区的 deque 末尾的 ProducerBatch。Sender 线程在每轮循环中检查所有 deque,挑出已满或停留时间超过 linger.ms 的 batch,按目标 broker 分组后封装为 ProduceRequest。该模型将业务线程的 send() 与网络 I/O 解耦,业务线程仅完成内存拷贝,后续重试与回调由 Sender 线程驱动。

1.3 Broker 请求处理模型

Broker 采用经典的 Reactor + 线程池模型,把"接连接 / 读写 socket / 处理请求"三件事分到三个不同的线程上做。这样的好处是慢的请求(等 fsync、等 ISR ACK)不会卡住快的请求(metadata 查询),各类请求的 P99 延迟互不污染。

图 1-3-1 · Broker Reactor 线程模型(Acceptor + Processor + Handler)

Broker 采用经典的 Reactor + 线程池模型。Acceptor 单线程接受新连接,Processor(Network 线程,默认 3 个)负责 socket 读写与协议解码;解码后的请求进入 RequestChannel 的 request queue。KafkaRequestHandler(I/O 线程,默认 8 个)从 queue 中取出请求,调用 KafkaApis.handle() 分派到具体处理器。处理结果回写到 response queue,由 Network 线程发送回客户端。

图 1-3-2 · 请求耗时分解(六段)

任何一个 broker 端请求的总耗时由六段时间组成,Broker 通过 JMX 暴露每段的 P50/P99。其中 LocalTimeMs 反映纯计算与磁盘 I/O,RemoteTimeMs 反映 acks=all 等待 ISR 复制或 follower fetch 等待数据可见的 purgatory 时间。排障时优先按段定位,而不是整体看 P99。

KafkaApis.handle() 根据 RequestHeader.apiKey 分派到具体处理函数。Produce/Fetch 走 ReplicaManager,Group/Heartbeat 走 GroupCoordinator,Transaction 相关走 TransactionCoordinator,Metadata 直接读 broker 本地的 MetadataCache。该分派结构使得不同请求类型的故障面相互隔离:Group 路径阻塞不影响 Produce 路径。

1.4 Controller Quorum 与元数据传播

KRaft(Kafka Raft)是 Kafka 3.x 内置的控制平面共识协议,它把所有拓扑事实(topic 列表、partition leader、replica 集合、broker 注册)都序列化为 __cluster_metadata 上的一条 metadata record,并通过 Raft 协议在 Controller Quorum 内部复制。这取代了过去 ZooKeeper 模式下分散在多个 znode 中的状态。

图 1-4-1 · KRaft 元数据日志结构

KRaft 将 ZooKeeper 中分散的拓扑信息合并为单一 Raft 日志 __cluster_metadata。该日志由 Controller Quorum 维护,通过 Raft 协议实现强一致复制。Active Controller(Raft leader)接受所有写入,如 CreateTopics、AlterPartition,并将变更追加为 metadata records。Follower Controller 复制日志,broker 通过 MetadataFetch 拉取已提交的 records 并应用到本地 MetadataCache。

图 1-4-2 · Metadata 变更传播链(CreateTopics → 客户端可见)

控制面变更的可见性边界由两次延迟决定:Quorum 多数 ACK 决定何时返回给 AdminClient,broker 拉取与应用决定何时被客户端看到。在 3.7 版本中,broker 端的 MetadataLoader 与 BrokerMetadataPublisher 解耦,前者负责按 offset 顺序读取 records,后者负责应用到具体子系统(如 ReplicaManager、GroupCoordinator)。

1.5 Consumer Group 协调

Consumer Group 是 Kafka 提供"分摊消费"语义的核心抽象。多个 consumer 进程通过同一个 group.id 加入同一个 group,Kafka 保证同一时刻每个分区只被组内的一个 consumer 消费。这一约束由 GroupCoordinator 执行。

图 1-5-1 · Consumer Group 角色关系

Consumer Group 由 GroupCoordinator 协调,每个 group 通过 Math.abs(group.id.hashCode) % numPartitions 映射到 __consumer_offsets 的某个分区,该分区的 leader broker 即为该 group 的 coordinator。Member 通过 JoinGroup → SyncGroup → Heartbeat → OffsetCommit 四类请求与 coordinator 交互,具体分区到 member 的映射由 partition.assignment.strategy 配置的策略类(如 CooperativeStickyAssignor)在 leader member 上执行。

图 1-5-2 · JoinGroup → SyncGroup 时序

JoinGroup 请求由所有 member 在加入 group 时发出,coordinator 收集到稳定的成员集合后返回 leader member id 与 member 列表。Leader member 在本地执行 PartitionAssignor.assign() 计算分区到 member 的映射,然后通过 SyncGroup 将映射上报给 coordinator,coordinator 再将每个 member 的 assignment 返回给对应 member。该设计将分配逻辑放在客户端,broker 端无需理解具体策略。

1.6 分区日志的物理布局

所有控制面、协调器最终都在为存储面服务——分区日志在磁盘上的物理布局,才是 Kafka 真正的"持久层数据结构"。每个 TopicPartition 对应一个目录,目录内由若干 segment 文件组合而成,每个 segment 又被一组稀疏 index 配套覆盖。

图 1-6-1 · TopicPartition 目录结构

每个 TopicPartition 对应一个目录,目录内的文件按 segment 组织。文件名前缀是该 segment 第一条记录的 baseOffset,固定 20 位前导零便于字典序排序。三类文件构成最小持久化单元:.log 存放消息,.index 是稀疏的 offset → 文件位置索引,.timeindex 是稀疏的 timestamp → offset 索引。Active segment 是当前接受追加的 segment,其余 segment 已封存,只读。

图 1-6-2 · UnifiedLog 内部结构

UnifiedLog 是分区在 broker 端的逻辑表示,包含本地分段集合、producer 状态机、leader epoch 缓存。每个 LogSegment 持有一个数据文件和三个索引文件,索引通过 mmap 映射到内存以避免读放大。appendAsLeader() 在 active segment 上执行追加,当 segment 大小达到 log.segment.bytes 或时间达到 log.roll.hours 时触发 roll,生成新的 active segment。

图 1-6-3 · Append 路径(KafkaApis → ReplicaManager → UnifiedLog → 磁盘)

写入路径不调用 fsync,数据通过 FileChannel.write() 进入 page cache,由内核根据 dirty_ratio 与 dirty_writeback_centisecs 异步刷盘。Kafka 通过副本冗余而非单机 fsync 提供持久性,该设计将单 broker 的同步开销转化为副本间的复制开销,在多副本场景下显著降低 P99 延迟。

1.7 复制与 ISR 推进

有了存储面的物理布局,接下来要看的就是多副本之间的一致性协议。Kafka 的复制协议跟 Raft 不一样——它没有"日志投票"环节,而是让 follower 主动 Fetch leader 的日志、leader 维护 ISR 集合并据此推进 HW(High Watermark)。

图 1-7-1 · Leader-Follower 复制时序

Follower 通过持续 Fetch 模拟 consumer 行为复制 leader 日志。Follower 的 fetchOffset 同时携带"已成功 append 到本地的下一条 offset",leader 据此判断每个 follower 的 LEO(Log End Offset)。当 ISR 集合内所有 follower 的 LEO 都 ≥ 某个值 K 时,leader 将 HW(High Watermark)推进到 K,客户端 consumer 只能读到 HW 以下的数据。

图 1-7-2 · HW 与 LEO 关系

ISR 是 leader 维护的"复制足够新"的 follower 集合,判定标准由 replica.lag.time.max.ms(默认 30 秒)控制:follower 必须在该时间内至少完成一次成功的 fetch 请求,否则被移出 ISR。注意该判定不依据 LEO 落差字节数(早期版本曾使用 replica.lag.max.messages,在 0.9 移除)。

1.8 故障处理

所有的架构铺陈最终都是为了应对故障。Kafka 的故障传播链非常清晰:controller 通过 heartbeat 感知 broker 失联 → 写入 metadata log → broker 应用切换 → 客户端通过 metadata 刷新发现新 leader。理解这条链路,排障时就能从症状走回根因,而不是反向猜测。

图 1-8-1 · Broker 失联恢复时序

Broker 失联由 controller 通过 heartbeat 超时检测。检测到失联后,controller 将该 broker 标记为 fenced,并对其 leader 分区从 ISR 中选择新 leader(优先选 ISR 内 broker,若 unclean.leader.election.enable=true 且 ISR 为空则可选 out-of-sync 副本)。新 leader 信息通过 metadata log 传播,broker 端的 BrokerMetadataPublisher 应用该变更后切换分区角色。客户端收到 NOT_LEADER_OR_FOLLOWER 错误后刷新 metadata 并重试新 leader。

图 1-8-2 · 故障决策树(从症状到根因)

线上 Kafka 问题通常以 Producer timeout、Consumer lag、Leader 频繁迁移、疑似丢消息四种症状呈现。每种症状对应一组定位入口,排障应从症状走回架构层,而非反向从配置表开始猜测。
