第 24 章 · Eligible Leader Replicas (KIP-966)

ISR · ELR · Last-Replica-Standing · Unclean Leader Election · HWM 不回退

"我设了 acks=all + min.insync.replicas=2,为什么还是丢了数据?" — 这是 Kafka 论坛最经典的"事故复盘"问题。答案藏在传统 leader 选举的悬崖效应里:当 ISR 缩到只剩 leader 一个、leader 又经历不干净关机、其本地尾部日志在 fsync 之前断电丢失后,这台机器恢复后再次被选为 leader,其他副本被迫 truncate 到这个新 leader 的 LEO — 已经 commit 给 producer 的数据就这样回退。

KIP-966 引入的 Eligible Leader Replicas (ELR) 不是"更大胆的 unclean election",而是把"可当 leader"从"是否还在 ISR 内"这一刚性条件中拆出来。当 ISR 跌破 min.insync.replicas 且 HWM 停止前进时,Controller 把那些"至少拥有当前 HWM 的副本" 标记为 ELR。leader 故障时先选 ISR,再选 ELR,最后才考虑 unclean — 既避免了 last-replica-standing 的 HWM 回退,又给可用性留了喘息。

  • 传统 ISR-only 选举在 last-replica-standing 场景的悬崖效应
  • ELR 与 ISR 的语义差异:ELR 只保证"有 HWM",不保证"跟 leader 同步"
  • 三层选举顺序:ISR → ELR → Unclean → Offline
  • ELR 的副本状态机:加入 / 离开 / 清空 触发条件
  • describe topic / partition 新增字段
  • 与 min.insync.replicas / acks 的协同关系
  • 典型生产演练:模拟 last-replica 故障
  • 监控指标 + JMX

24.6 演练:故障场景

演练:模拟 last-replica-standing

1 创建 topic, RF=3, minISR=2
2 用 chaos 工具断 broker-2 网络 → ISR={0,1}, ELR={2}
3 让 broker-1 卡 GC 30s → ISR={0}, ELR={1,2}, HW 停推
4 kill -9 broker-0 (模拟掉电)
5 broker-0 恢复, 但其本地 LEO 比 commit 时少 (fsync 未完成)

期望(ELR=ON):
  controller 选举 → 看 ISR={}, 看 ELR={1,2} 非空
  → 等其中一个恢复 (e.g. broker-2)
  → 选 broker-2 为新 leader, HWM 保留
  → broker-0 加入时反向 truncate

期望(ELR=OFF, 旧行为):
  controller 选举 → ISR={} 等 broker-0 恢复
  → broker-0 成 leader, broker-1/2 truncate 到其 LEO
  → 数据丢失!

24.7 与 minISR / acks 的关系

ELR 不替代 acks=all + minISR。它们是互补的:
  • min.insync.replicas 决定写入需要多少 ISR 同步 → 控制"commit 之前的不变量"
  • ELR 决定故障选举时谁可信 → 控制"故障后的不变量"
两者必须一起评审。仅有 minISR 而无 ELR:写入安全但故障可能回退;仅有 ELR 而 minISR=1:故障安全但写入可能 stale。

24.8 设计取舍

① ELR 牺牲什么

故障恢复时间变长 — 不再"当 ISR 空就立刻选 unclean",而是先尝试等 ELR 中某个副本恢复。代价是分钟级 offline 窗口(等 ELR 副本 broker 启动)。

② 仅 HWM 而非 LEO

ELR 只要求"有 HWM" 不要求"追平 LEO"。这放宽了候选集,提高了 ELR 选举成功的概率。代价是 ELR 选出来后,leader-0 上未 commit 的 (LEO, HWM] 区间消失。但这本来就是"未 commit",不算丢数据。

③ Controller 内存开销

每个 partition 在 metadata 里多记 ELR 列表 + LastKnownELR。集群 1M partition × ~3 副本 = 几 MB,可接受。

④ describe 输出兼容

新增 ELR 字段不破坏旧 client。但 unclean recovery 行为变化,可能让自动化运维脚本误判。

24.10 本章要点

本章小结

ELR(KIP-966)是 Kafka 4.x 在副本协议上一次重要演进。它通过在 ISR 之外引入"仍有 HWM 的可信候选集",优雅地解决了 last-replica-standing 场景的 HWM 回退问题。核心规则:选举顺序 ISR → ELR → Unclean → Offline,四层边界不可混淆。生产部署:① RF=3 + minISR=2 + ELR 启用 + unclean=false 是黄金组合;② 监控 EligibleLeaderReplicas,故障演练验证;③ 把 ELR 与 minISR 一起评审,前者是故障侧不变量,后者是写入侧不变量。
