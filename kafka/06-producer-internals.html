第 6 章 · Producer 内部流水线

RecordAccumulator · BufferPool · Sender 单线程 · in-flight requests · 幂等 PID/Sequence · 事务 EOS · linger.ms 与 batch.size

Kafka Producer 的核心不是单次远程调用,而是一条本地异步流水线。业务线程把 record 序列化后追加到分区缓冲区,Sender 线程独立地把缓冲区中的批次合并为请求、处理响应、重试和回调。该解耦让业务线程在 1-2 毫秒内拿到 future,真正的网络 I/O 与 broker 确认在后台进行——这正是 Kafka 在百万级吞吐场景下仍能保持业务线程"非阻塞感"的根本原因。

本章按 "整体流水线 → send 时序 → 拦截器 → 序列化与分区 → RecordAccumulator → BufferPool → Sender 与请求 → 重试与幂等 → 响应与 callback → 配置 → 排障 → 源码 → 取舍 → 陷阱" 十四节组织。所有讨论锚定 Apache Kafka 3.7 客户端实现:KafkaProducer#doSend 是业务入口,Sender#runOnce 是 I/O 入口,RecordAccumulator 与 BufferPool 是两条线程在共享内存上的会面点;TransactionManager 维护 PID/epoch/sequence 让重试不会引入重复;InFlightRequests 限制单连接未确认请求数,确保顺序与 fail-fast。

  • Producer 完整流水线:App → Interceptors → Serializer → Partitioner → RecordAccumulator → Sender → NetworkClient → Broker 的线程边界
  • 业务线程 1-2 ms 与 Sender 线程独立推进的时间分解,以及为何同步 future.get() 会让吞吐降低 10-100 倍
  • 拦截器执行顺序:onSend 在业务线程、onAcknowledgement 在 Sender 线程,以及拦截器异常的吞噬规则
  • 三种分区策略:显式 partition、有 key 的 murmur2 + toPositive % numPartitions、无 key 的 sticky partitioner 的具体行为
  • RecordAccumulator 的 ConcurrentMap<TopicPartition, Deque<ProducerBatch>> 数据模型,以及双重 tryAppend 锁内 / 锁外两段式分配
  • BufferPool 三种路径:复用池化 (<1 µs) / 新分配池化 (5-10 µs) / 非池化分配 (10-50 µs),以及 max.block.ms 等待超时的语义

6.0 章首速览

Producer 内部由两类独立线程共享一段共享内存构成。业务线程负责把 ProducerRecord 转换为字节并追加到分区缓冲;Sender 线程独立扫描缓冲、按 broker 分组组装 ProduceRequest、维护重试与幂等状态。理解这两类线程在 RecordAccumulator 上的"接力"是理解 Kafka Producer 一切性能与正确性问题的起点。

图 6-0 · Producer 全流水线(主线程 + 共享内存 + Sender 线程)
核心结论:Producer 是"业务线程入队 + Sender 线程出队送网络"的两段式异步流水线。send() 返回 Future 仅表示本地处理完成(序列化 + 分区计算 + 入 deque),broker ack 在 Future.get() 或 callback 触发时才到达。理解这一点可以避免 90% 的 Producer 性能问题——你以为在调远程 RPC,其实只是在做内存拷贝。

业务线程仅消耗 1-2 ms(序列化 + 分区 + 入 deque),Sender 线程在 ~5-10 ms 后完成 broker 往返。如果业务调用 future.get() 等待 broker ack,业务线程会被阻塞约 13 ms,无法继续准备下一条 record——批量化彻底失效,吞吐降到几百每秒。

推荐的异步 callback

ProducerRecord<String, String> record =
    new ProducerRecord<>("order-created", orderId, payload);

producer.send(record, (metadata, exception) -> {
    if (exception == null) {
        log.info("sent topic={} partition={} offset={}",
            metadata.topic(), metadata.partition(), metadata.offset());
        return;
    }
    log.warn("send failed key={}", orderId, exception);
});

不推荐的同步阻塞

for (Order order : orderEvents) {
    ProducerRecord<String, String> record =
        new ProducerRecord<>("order-created", order.id(), toJson(order));

6.2 拦截器:观测而非业务

拦截器(ProducerInterceptor)是 Kafka 给 Producer 提供的非侵入式扩展点。它适合做观测、打点、trace 注入、轻量改写 header;不适合做业务逻辑,因为它不是业务事务的一部分,异常会被吞噬而不传给业务代码。

图 6-2-1 · 拦截器执行顺序(链式)

interceptor.classes 配置的顺序即调用顺序,前一个拦截器的输出作为下一个的输入。两个回调点:onSend 在业务线程执行(阻塞 send 调用),onAcknowledgement 在 Sender 线程执行(阻塞 callback 链)。

trace header 拦截器实现

public final class TraceHeaderInterceptor
        implements ProducerInterceptor<String, String> {

    @Override
    public ProducerRecord<String, String> onSend(
            ProducerRecord<String, String> record) {
        record.headers().add("trace-id",
            currentTraceId().getBytes(StandardCharsets.UTF_8));
        return record;
    }

    @Override
    public void onAcknowledgement(RecordMetadata metadata,
                                   Exception exception) {
        String name = exception == null
            ? "producer.acked"
            : "producer.failed";
        metrics.counter(name).increment();
    }

    @Override public void close() {}
    @Override public void configure(Map<String, ?> configs) {}
}

自定义 Partitioner

public final class TenantPartitioner implements Partitioner {
    @Override
    public int partition(String topic, Object key, byte[] keyBytes,
                         Object value, byte[] valueBytes, Cluster cluster) {
        if (key == null) {
            throw new IllegalArgumentException("tenant key is required");
        }
        List<PartitionInfo> parts = cluster.partitionsForTopic(topic);
        String tenant = String.valueOf(key).split(":")[0];
        byte[] tenantBytes = tenant.getBytes(StandardCharsets.UTF_8);
        return Utils.toPositive(Utils.murmur2(tenantBytes)) % parts.size();
    }

    @Override public void close() {}
    @Override public void configure(Map<String, ?> configs) {}
}
分区是顺序的边界。Kafka 仅在同一分区内保证顺序;一旦跨分区,broker 端写入顺序与 consumer 端消费顺序都没有任何承诺。如果业务需要"同 user 严格顺序",就必须把 user id 当 key,让 hash 把同 user 路由到同分区——但代价是热点 user 会让某个分区压力远高于其他。

6.4 RecordAccumulator

RecordAccumulator 是业务线程与 Sender 线程的会面点。它的核心数据结构是 ConcurrentMap<TopicPartition, Deque<ProducerBatch>>:每个分区一个独立 deque,业务线程仅追加到尾部、Sender 线程从头部取已就绪的 batch。这种"按分区分桶 + 锁粒度按 deque"的设计让两类线程在共享数据结构上仅在末尾节点存在锁竞争。

图 6-4-1 · RecordAccumulator 数据模型

每个 TopicPartition 维护独立的 Deque<ProducerBatch>。业务线程仅追加到尾部当前活跃 batch,Sender 线程从头部取已就绪的 batch。该设计让两类线程在共享数据结构上仅在末尾节点存在锁竞争——跨分区 append 完全无竞争,同分区 append 也只在 deque 锁内做毫秒级操作。

append 实现(双重 tryAppend)

Deque<ProducerBatch> dq = batches.computeIfAbsent(tp, k -> newDeque());

synchronized (dq) {
    ProducerBatch last = dq.peekLast();
    if (last != null && last.tryAppend(record, callback)) {
        return last.future();
    }
}

ByteBuffer buffer = bufferPool.allocate(size, maxBlockTimeMs);

synchronized (dq) {
    ProducerBatch last = dq.peekLast();
    if (last != null && last.tryAppend(record, callback)) {
        bufferPool.deallocate(buffer);
        return last.future();
    }

    ProducerBatch batch = new ProducerBatch(tp, buffer);
    batch.tryAppend(record, callback);
    dq.addLast(batch);
    sender.wakeup();
    return batch.future();
}

双重 tryAppend 是性能关键:第一次在锁内尝试追加,失败则在锁外分配 buffer(开销大),再次进锁后重试。该模式让在 buffer 等待期间其他线程可能填满当前 batch,本线程可复用而非创建新 batch。

6.5 BufferPool 与内存管理

BufferPool 是 Producer 的内存配额管理器。所有 ProducerBatch 都从 BufferPool 申请内存,使用完毕归还。它通过 固定大小的 freelist + 可用内存池 两段结构,把"频繁分配"的开销降到 < 1 µs,同时把"超大单条 record"通过非池化分配兜底,把"内存不足"通过等待队列兜底。

真实事故:某团队 buffer.memory=32 MiB + linger.ms=50 + 200 个分区。每分区一个开放 batch + freelist 留 1-2 个,32 MiB 几乎全部被活跃 batch 占用。突发流量打来时 BufferPool 频繁 wait,send P99 飙到 60 秒触发 TimeoutException。修复:把 buffer.memory 提到 128 MiB,并把 linger.ms 降到 20——前者扛住峰值,后者让活跃 batch 周转更快。

6.6 Sender 线程与请求组装

Sender 是 Producer 内的独立 I/O 线程。它运行一个无限循环 runOnce,每轮做五件事:扫描 ready 分区、drain 出 batch、检查过期 batch、按 broker 分组组装 ProduceRequest、通过 NetworkClient 异步发送。它是单线程的——这是 Kafka 保证分区内顺序的根本机制。

图 6-6-1 · Sender.runOnce 主循环

runOnce 是 Sender 主循环。每次循环找出所有 ready broker,取出 batch,组装为按 broker 分组的 ProduceRequest,然后通过 NetworkClient 异步发送。整个循环单线程串行,确保同一分区内 batch 的发送顺序与 append 顺序一致——这是 Kafka 顺序保证的根基。

6.7 重试与幂等

Producer 的重试不是简单的"再发一次"。它必须在以下三件事之间取得平衡:可重试错误(NotLeader、超时)需要自动恢复;不可重试错误(消息超大、PID fenced)必须 fail-fast 不重试;幂等必须保证重试不引入重复。这一节把决策树画清楚。

事务 Producer 与 EOS

幂等只保证单分区内不重;跨分区原子需要事务 Producer。事务 Producer 通过 transactional.id 跨进程重启保持身份,通过两阶段提交把多分区写入与 consumer offset 提交合在一个原子单位里。这是"读 → 处理 → 写"流式应用实现 Exactly-Once Semantics(EOS)的基础。

Properties props = new Properties();
props.put("bootstrap.servers", "kafka-0:9092");
props.put("transactional.id", "billing-consumer-tx");
props.put("enable.idempotence", true);
props.put("acks", "all");

KafkaProducer<String, String> producer = new KafkaProducer<>(props);
producer.initTransactions();

try {
    producer.beginTransaction();
    producer.send(new ProducerRecord<>("settled", orderId, payload));
    producer.send(new ProducerRecord<>("audit",   orderId, audit));
    producer.sendOffsetsToTransaction(consumerOffsets, consumerGroupMetadata);
    producer.commitTransaction();
} catch (ProducerFencedException | OutOfOrderSequenceException
       | AuthorizationException e) {
    producer.close();
} catch (KafkaException e) {
    producer.abortTransaction();
}

6.8 响应与 callback

响应处理是 Sender 线程"出坑"的最后一步。它从 broker 收到 ProduceResponse、用 correlationId 找回 InFlightRequest、对每个 batch 内部的 thunk 依次触发 onAcknowledgement 与用户 callback、最后归还 buffer。所有 callback 都在 Sender 这一根线程上执行——这是性能上最敏感的设计点。

doSend 简化代码

private Future<RecordMetadata> doSend(ProducerRecord<K, V> record,
                                       Callback callback) {
    record = interceptors.onSend(record);

    Cluster cluster = waitOnMetadata(record.topic(), maxBlockTimeMs);

    byte[] keyBytes = keySerializer.serialize(record.topic(),
                                               record.headers(), record.key());
    byte[] valueBytes = valueSerializer.serialize(record.topic(),
                                                   record.headers(), record.value());

    int partition = partition(record, keyBytes, valueBytes, cluster);
    TopicPartition tp = new TopicPartition(record.topic(), partition);

    setReadOnly(record.headers());
    Header[] headers = record.headers().toArray();

    RecordAccumulator.RecordAppendResult result =
        accumulator.append(tp, record.timestamp(),
                            keyBytes, valueBytes, headers,
                            interceptCallback, maxBlockTimeMs - elapsed);

    if (result.batchIsFull || result.newBatchCreated) {
        sender.wakeup();
    }
    return result.future;
}

6.13 陷阱与限制

Producer 的陷阱基本可分四类:阻塞类(同步 get、callback 慢、buffer 不足)、可靠性类(acks=1 + 单副本、重试关 + 重试需要、幂等未启)、性能类(linger=0 + 期望 batch、压缩比为 1、callback 重)、顺序类(非幂等 + 高 inflight、partitioner 错、分区数变更)。

反例与正例对比

反模式(单线程思维 + 同步等): 
Properties props = new Properties();

props.put("retries", 0);
props.put("enable.idempotence", false);

  producer.send(record).get();    // 每条同步等
}

现象:吞吐 100 records/s(应能 10 万+);leader 故障丢消息;网络重试可能重复;buffer 偶发 timeout。

正模式(异步 + 幂等 + 批量): 
Properties props = new Properties();
props.put("acks", "all");
props.put("retries", Integer.MAX_VALUE);
props.put("enable.idempotence", true);
props.put("delivery.timeout.ms", 120000);
props.put("max.in.flight.requests.per.connection", 5);
props.put("linger.ms", 20);
props.put("batch.size", 65536);
props.put("buffer.memory", 67108864);
props.put("compression.type", "zstd");

for (Order order : orders) {
  producer.send(record, lightCallback);
}
producer.flush();
producer.close(Duration.ofSeconds(30));

期望:吞吐 100k+ records/s;单 broker 故障不丢;网络重试不重复;优雅关闭不丢。

12 条陷阱的共同根因:把 Kafka Producer 当作"本地 RPC 客户端"看待——以为 send 完了就稳了、以为 get() 才安全、以为 callback 是"消息已落 broker 的回调"。真实模型是:Producer 是一条本地异步流水线。所有"为什么 send 不阻塞"、"为什么 callback 在 Sender 上跑"、"为什么幂等需要 sequence"的问题,都要从"线程边界 + 共享内存 + broker 状态机"三个维度找,而不是从"事务 / 锁 / 同步"找。
