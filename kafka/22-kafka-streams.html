第 22 章 · Kafka Streams

DSL · Topology · StreamTask · RocksDBStore · changelog · repartition · EOS · Standby

"实时 join 订单和用户表"、"每 5 分钟窗口统计 GMV"、"检测连续 3 笔失败登录" — 这类有状态流处理是 Kafka 之上最常见的应用形态。你可以用 Flink / Spark Streaming(独立集群),但如果你的 service mesh 已经是"普通 JVM 应用" 形态,引入新集群是个大决策。Kafka Streams 不是独立集群,而是嵌入 Java 应用的原生库 — 你的 Spring Boot 应用启动时 new KafkaStreams(topology, config).start(),流处理就跑起来了,扩容靠多启动相同 application.id 实例,Consumer Group 协议自动分 task。

这种"无集群" 形态的代价是部分能力受限(没有跨 JVM 任务调度、没有专门的流处理 SQL 引擎),但对中小规模 / 微服务架构非常友好。本章按"运行时 → DSL → KStream/KTable → 状态存储 → 窗口 → Repartition → EOS → Standby"组织,对应 Apache Kafka 3.7 的 org.apache.kafka.streams。

  • Streams 是 库 不是集群:扩容 = 多启动实例
  • DSL + Topology + StreamTask 三层抽象
  • KStream(事件流) vs KTable(实时物化表)
  • RocksDB 本地状态 + changelog topic 容灾
  • 窗口:Tumbling / Hopping / Session / Sliding
  • Repartition topic 内部生成的时机
  • EOS in Streams = idempotent producer + read-process-write 事务
  • Standby 任务持续追平 changelog,加速 failover

22.1 库 vs 集群

Streams(库)

  • 嵌入普通 Java 应用
  • 扩容靠多启动实例 + Consumer Group 协议
  • 无独立集群,运维简单
  • 仅 Java/Scala/Kotlin/Clojure
  • 适合 ≤ 100 GB 状态

Flink / Spark Streaming(集群)

  • 独立 JobManager / TaskManager 集群
  • 跨语言(Python / Scala / Java)
  • 需要单独运维集群
  • 原生支持复杂窗口 / EventTime watermark
  • 适合 TB+ 状态、复杂 SQL

22.2 DSL 与拓扑

// DSL 示例:统计每个用户的下单次数(每分钟)
StreamsBuilder builder = new StreamsBuilder();

KStream<String, Order> orders = builder.stream("orders");

orders
    .filter((key, order) -> order.amount > 100)
    .groupBy((key, order) -> order.userId)            // 重分组 → repartition topic
    .windowedBy(TimeWindows.ofSizeWithNoGrace(Duration.ofMinutes(1)))
    .count(Materialized.as("user-order-count-store"))  // state store
    .toStream()
    .map((wKey, count) -> KeyValue.pair(wKey.key(), count))
    .to("user-order-count-per-minute");                // output topic

Topology topology = builder.build();
System.out.println(topology.describe());  // 打印 DAG

KafkaStreams streams = new KafkaStreams(topology, props);
streams.start();

编译后会生成的内部 topic:

  • my-app-user-order-count-store-repartition — repartition
  • my-app-user-order-count-store-changelog — state 备份(compacted)

22.3 KStream vs KTable

这是 Streams 最重要的抽象区分。KStream(事件流) ≠ KTable(实时物化表)。

维度KStreamKTable
本质无界事件流(append-only)实时物化表(key 唯一,留最新)
语义"发生了什么""当前状态"
同 key 多 record各自独立保留覆盖,只留最新
底层 topic普通 topiccompacted topic(或本地 KTable)
join 行为KStream-KStream:窗口 joinKStream-KTable:实时 lookup
典型来源事件、点击日志用户表、产品表(CDC)

22.4 状态存储 RocksDB

每个 StreamTask 内的 stateful 操作(aggregate / count / reduce / join)都需要持久化状态。Streams 默认用嵌入式 RocksDB(LSM-tree KV store)做本地状态,同时把所有变更写到changelog topic(compacted)做容灾。

读路径:
  task.aggregate() → store.get(key) → RocksDB (本地, sub-ms)

写路径:
  store.put(key, value)
  → RocksDB write (本地)
  → changelog topic 写(producer.send)
  → 下次 task.commit() 才 producer.flush()

崩溃恢复:
  Streams 启动 → 看 changelog 与 RocksDB checkpoint 的差距
  → replay changelog 上的 missing record → RocksDB
  → restore 完成才进 RUNNING

22.5 窗口

类型含义典型场景
Tumbling不重叠,固定大小每分钟 GMV
Hopping固定大小,有 advance(可重叠)每分钟看过去 5 分钟
Session由"不活跃间隔"分隔用户会话统计
Sliding (KIP-450)固定大小,每条记录新窗口实时风控

每种窗口都有 grace 参数 — 处理乱序事件的"宽限期"。3.x 之后要求显式声明 ofSizeAndGrace 或 ofSizeWithNoGrace。

22.6 Repartition

任何"key 改变" 的操作(groupBy、selectKey、through)都会触发 repartition — 因为 stateful 操作要求"相同 key 路由到相同 task",而 input topic 的分区可能与新 key 不一致。框架自动建一个 repartition topic 把数据重新路由:

stream
  .selectKey((k, v) -> v.userId)        // ← key 改变, 标记 repartition needed
  .groupByKey()                          // ← 这里触发 repartition topic 创建
  .aggregate(...)
陷阱:repartition topic 是未压缩 + 短 retention(默认 retention.ms=Long.MAX_VALUE 直到 commit 后删除)— broker 重启或 stream 长时间停可能积累。监控 repartition topic 大小,必要时调 repartition.purge.interval.ms。

22.7 EOS in Streams

props.put(StreamsConfig.PROCESSING_GUARANTEE_CONFIG, StreamsConfig.EXACTLY_ONCE_V2);
// 自动:
//  - producer.idempotence = true
//  - producer.transactional.id = <applicationId>-<taskId>
//  - consumer.isolation.level = read_committed
//  - 每次 commit 是一个 Kafka transaction:
//    * 写 output topic
//    * 写 changelog topic
//    * commit consumer offset
//    都在同一事务内, 要么都成功要么都回滚

22.8 Standby & Restore

Streams 支持配置 num.standby.replicas — 让其他实例预先追平 changelog 的本地 RocksDB(但不处理 input)。当 active task 所在实例挂了,standby 立刻接管,几乎无 restore 时间。

实例 A: ACTIVE task 0_0    实例 B: STANDBY task 0_0
  RocksDB (local)           RocksDB (local, replicated from changelog)
  poll input                no input processing
  process                   只追 changelog topic
  write output              -
  commit (txn)              -

A 挂了:
  Rebalance → B 升级为 ACTIVE
  B 已有 RocksDB, 仅需 catch-up 几秒 changelog
  → P99 接管时间从几分钟 → 几秒

#陷阱现象修复

22.11 本章要点

本章小结

Kafka Streams 是 Kafka 自家的嵌入式流处理库 — 没有独立集群,扩容即多启动实例。它把"事件流 ↔ 实时表 ↔ 状态存储" 三者用 Consumer Group + Kafka topic + RocksDB 优雅地编排。生产用好要懂:① 选对 KStream / KTable 抽象;② 配 standby 加速 failover;③ 监控 RocksDB 与 repartition topic;④ 显式开 EOS_V2。下一章我们看 Kafka 的安全机制 — SASL/SSL 认证、ACL 授权、加密。
