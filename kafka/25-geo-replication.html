第 25 章 · 跨集群复制 (Geo-replication)

MirrorMaker 2 · MirrorSource/Checkpoint/Heartbeat · Cluster Linking · Active-Passive · Active-Active · RPO/RTO

"把 Kafka 部署到两个数据中心,一边挂了切到另一边" — 这事说起来简单,做起来九成团队会踩坑。最常见的错误是"把一个集群横跨 DC" — 让 broker 同时落在 us-east + eu-west,认为这样就是"跨地域高可用"。错。Kafka 副本协议假设 broker 间稳定低延迟(单数字 ms RTT),50-200ms 跨 DC RTT 会让 ISR / Controller / acks=all 全部承担跨地域开销,P99 飙升到秒级,故障恢复非常痛苦。

正确做法是多集群异步复制:每个 DC 一个独立 Kafka 集群,跨 DC 用 MirrorMaker 2(KIP-382)或 Cluster Linking(Confluent)做异步复制。这种架构下,本地写入仍是低延迟,跨 DC 复制延迟以"RPO"形式被显式管理 — 业务知道"切流时最多丢 X 秒数据"。

本章按"为什么不跨 DC → MM2 三件套 → MirrorSource → Offset Sync → Heartbeat → Failover → Active-Active → Cluster Linking → 性能 → 取舍"组织,对应 Kafka 4.x 的 MirrorMaker 2.0。

  • "不要单集群跨 DC" 的根因(ISR / KRaft / acks=all 的网络假设)
  • MM2 三件套分工:Source / Checkpoint / Heartbeat
  • 远程 topic 命名约定(us-east.orders)
  • Offset Sync 怎样让 consumer failover 时找到正确 offset
  • Heartbeat 监控复制延迟
  • Active-Passive failover 的标准 runbook
  • Active-Active 的循环防护与冲突解决
  • Cluster Linking vs MM2 取舍

25.1 为什么不单集群跨 DC

把一个 Kafka 集群横跨 50-200ms RTT 的 DC,会触发以下连锁反应:

机制跨 DC 的代价
ISR 同步follower 在远端,fetch RTT 飙,replica.lag.time.max.ms 设大会让 ISR 失去意义
acks=all等远端 follower ACK,Producer P99 飙到 100ms+
KRaft Controllerquorum vote 跨 DC RTT,选举时间变长
Consumer fetch消费者只能从 leader fetch(除 follower fetch),远端消费高延迟
网络分区DC 间网络抖动直接表现为 ISR 收缩 / leader 变更风暴

25.2 MM2 三件套

Connector职责核心 topic
MirrorSourceConnector复制 records + topic 配置 + ACL远端目标 topic + mm2-offset-syncs.<cluster>.internal
MirrorCheckpointConnector复制 consumer group offset(翻译后)<source>.checkpoints.internal
MirrorHeartbeatConnector定期写 heartbeat,测延迟heartbeats(在源)+ <source>.heartbeats(在目标)

远程 topic 命名

源集群 us-east 的 topic:
  orders

复制到 eu-west 后, 默认 IdentityReplicationPolicy:
  us-east.orders

(可换成 IdentityReplicationPolicy 保持相同名, 但 Active-Active 会循环)

25.3 MirrorSourceConnector

name=us-east-to-eu-west-source
connector.class=org.apache.kafka.connect.mirror.MirrorSourceConnector
topics=orders,inventory,payments
source.cluster.alias=us-east
target.cluster.alias=eu-west
source.cluster.bootstrap.servers=kafka.use1.com:9092
target.cluster.bootstrap.servers=kafka.euw1.com:9092
replication.factor=3
refresh.topics.enabled=true               # 自动镜像新 topic
refresh.topics.interval.seconds=60
sync.topic.configs.enabled=true
sync.topic.acls.enabled=true
emit.checkpoints.enabled=true
tasks.max=8

25.4 Offset Sync

跨集群复制最难的不是搬数据,而是搬 consumer group offset。同一条消息在源集群 offset=12345,复制到目标集群可能 offset=98765(因为目标 topic 是新建的,offset 序列不一样)。MirrorCheckpointConnector 维护一张映射表帮助 consumer 在 failover 时找到等价位置。

图 25-4 · Offset Sync 工作流

failover 时用 RemoteClusterUtils

// failover 时, 把 consumer 切到 eu-west
Map<TopicPartition, OffsetAndMetadata> offsets =
    RemoteClusterUtils.translateOffsets(
        props,                  // eu-west 连接配置
        "us-east",              // source alias
        "order-svc",            // consumer group
        Duration.ofSeconds(10)
    );

KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props);
consumer.subscribe(List.of("us-east.orders"));
consumer.poll(0);  // 触发 assignment
offsets.forEach((tp, oam) -> consumer.seek(tp, oam.offset()));
// 继续消费, 从正确位置开始

25.5 Heartbeat

Heartbeat 是一个特殊的小 topic — MirrorHeartbeatConnector 每秒往源集群 heartbeats 写一条带时间戳的消息,MM2 把它复制到目标 us-east.heartbeats。运维通过对比"消息时间戳 vs 当前时间"算复制延迟(replication lag),即 RPO 指标。

监控复制延迟:

  现在时间:      2026-05-12 10:00:30 UTC
  最新 heartbeat: 2026-05-12 10:00:25 UTC (从 us-east.heartbeats 读)
  → 复制延迟 = 5 秒
  → failover RPO ≤ 5 秒 (近似)

25.7 Active-Active

两个集群都接 producer。MM2 配双向:

# us-east → eu-west
us-east->eu-west.enabled=true
us-east->eu-west.topics=orders,inventory

# eu-west → us-east
eu-west->us-east.enabled=true
eu-west->us-east.topics=orders,inventory
循环防护:MM2 用 DefaultReplicationPolicy 给远程 topic 加 source alias 前缀(us-east.orders),防止循环复制。但如果你用 IdentityReplicationPolicy(保持同名),必须用 topics.exclude 显式列出"不要再复制回来"的 topic。漏配 = 循环复制 = 数据指数膨胀。

冲突解决

两端同时写同一 key 的数据,业务侧必须设计冲突解决策略:

  • 用户 ID 分区:user 1-1000 写 us-east,1001-2000 写 eu-west。无冲突。
  • Last Write Wins:用消息时间戳,后写覆盖。需要时钟同步。
  • CRDT:用合并友好的数据结构(计数、集合)。

#陷阱现象修复

25.11 本章要点

本章小结

跨集群复制是 Kafka 生产部署的必备能力,但也是踩坑最多的领域。三个核心原则:① 不要单集群跨 DC — Kafka 副本协议假设低 RTT,跨 DC 用异步复制;② MM2 三件套都要装 — Source / Checkpoint / Heartbeat,缺任何一个 failover 都不完整;③ Active-Active 必须设计冲突解决 — 业务侧策略,不靠 Kafka 魔法。监控 heartbeat 时差就是监控 RPO,定期演练 failover 就是验证 RTO。
