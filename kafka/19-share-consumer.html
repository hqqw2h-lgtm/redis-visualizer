第 19 章 · Share Consumer 与 Share Groups (KIP-932)

ShareConsumer · Share Group Coordinator · Record Lock · per-record ACK · __share_group_state · ACCEPT / RELEASE / REJECT

十多年来 Kafka 一直坚持"一个 partition 同一时间只能被一个 consumer 独占"这条铁律。这条铁律让分区内严格有序成为可能,也让 offset commit 变成简单的"整型前缀推进"。但代价同样硬:并行度上限 = partition 数。如果你的业务是慢任务池 — 一条记录处理 10 秒,但 partition 数只有 10,你最多并行 10 个 worker,扩容只能加 partition,而加 partition 又会影响下游 key 分布。

Kafka 4.1 preview 引入的 Share Consumer + Share Group(KIP-932)就是为了打破这条铁律。它把"分区独占" 改成"记录级竞争" — 同一个 partition 的不同 record 可以分发给不同 ShareConsumer 并行处理,每条记录单独 ACK,失败的记录可以 RELEASE 重投。这种"Pub/Sub-like 工作池"语义之前只能用 RabbitMQ / SQS / ActiveMQ 实现,现在 Kafka 原生支持了。

但它不是替代 Consumer Group,而是补齐。流处理、ETL、严格顺序场景继续用 Consumer Group。任务池、慢任务、异步处理用 Share Group。本章详细讲两者的差异、Record Lock 实现、ACK 三态语义、Coordinator 状态机、压测与监控要点。

  • Consumer Group 在"慢任务池" 场景的扩容瓶颈
  • Record-level lock:每条 record 在 Coordinator 上的 owner / deadline / count
  • ShareFetch RPC:Coordinator 怎样按可用 record 分发
  • 三态 ACK:ACCEPT(终态成功)/ RELEASE(重投)/ REJECT(终态失败)
  • Delivery limit + Lock timeout 两道防线
  • __share_group_state internal topic + Coordinator 状态机
  • 顺序退化和去重(at-least-once 仍然成立)
  • 压测要点:Coordinator 是状态热点

19.1 适用与拒绝场景

✅ 适合 Share Group

  • 慢任务池(每条 record 处理 1s - 60s)
  • 需要逐条 retry / 死信(DLQ)
  • 不依赖分区内顺序(record 之间独立)
  • Worker 数远大于 partition 数
  • 任务时长方差大(快慢混合)

❌ 不适合(用 Consumer Group)

  • 需要分区内严格有序
  • 流处理 / Kafka Streams
  • 事务(EOS)读取(Share Group 不兼容 transactional consumer)
  • 下游有 stateful join,依赖 partition 路由
  • 毫秒级延迟敏感(Coordinator 多一跳)

19.2 Record Lock 机制

Share Group 的核心数据结构是 Record Lock Table:每条被分发的 record 都有一个锁条目,记录"当前被谁拿走、什么时候到期、已投递几次"。

// Coordinator 内的 Lock 条目(概念)
class RecordLock {
    long offset;           // partition 内 offset
    String ownerMemberId;  // 当前 Worker member id
    long lockExpiresAt;    // 锁到期时间(ms)
    int deliveryCount;     // 已投递次数(从 1 开始)
    LockState state;       // ACQUIRED / RELEASED / ACCEPTED / REJECTED
}

enum LockState {
    ACQUIRED,   // Worker 正在处理
    RELEASED,   // Worker 主动放弃 / 超时被收回, 等待重投
    ACCEPTED,   // 终态: 成功
    REJECTED    // 终态: 失败(超过 delivery limit 或显式 REJECT)
}

19.3 ShareFetch RPC

客户端用一个新的 RPC ShareFetch(替代普通 Fetch)。它返回的不是"从 offset X 起的 N 条记录",而是"当前可用的、未被锁的、还在 delivery limit 内的记录子集"。响应里每条 record 都带一个 deliveryCount 和 acquisitionLockTimeout。

// 客户端使用 ShareConsumer
ShareConsumer<String, Order> consumer = new KafkaShareConsumer<>(props);
consumer.subscribe(List.of("orders"));

while (running) {
    ConsumerRecords<String, Order> records = consumer.poll(Duration.ofSeconds(1));
    for (ConsumerRecord<String, Order> rec : records) {
        try {
            process(rec.value());
            consumer.acknowledge(rec, AcknowledgeType.ACCEPT);  // 成功
        } catch (TransientException e) {
            consumer.acknowledge(rec, AcknowledgeType.RELEASE); // 重投
        } catch (PoisonPillException e) {
            consumer.acknowledge(rec, AcknowledgeType.REJECT);  // 死信
        }
    }
    consumer.commitSync();  // 批量提交 ACK
}

19.4 三态 ACK

ACK 类型含义下次该 record 行为
ACCEPT成功状态 → ACCEPTED,不再分发
RELEASE主动放弃(瞬时失败)状态 → RELEASED,等下次 ShareFetch 重投(deliveryCount++)
REJECT显式失败(永久问题,如 schema 解析错)状态 → REJECTED,送 DLQ 或丢弃
(无 ACK 直到 timeout)Worker 挂了或慢锁到期,等同 RELEASE
Delivery limit 是 Share Group 的"第二道防线"。默认 5:同一条 record 被 RELEASE / timeout 5 次后,即使没 REJECT 也会自动 → REJECTED 防止毒丸消息无限重试。生产建议:配 DLQ topic + delivery limit = 3-5。

19.5 Coordinator 状态机

Share Group 仍由 Group Coordinator(同 Consumer Group)托管,但用一个新的 internal topic __share_group_state 持久化 Lock Table。这个 topic 是 compacted(同第 13 章),key = (groupId, topicPartition, offset),value = 最新 Lock 状态。

__share_group_state record (key, value):

  KEY:   GroupId="fulfillment", TopicPartition="orders-0", Offset=12345
  VALUE: { ownerMemberId="worker-a-7f3e",
           deliveryCount=2,
           lockExpiresAt=1715432100000,
           state=ACQUIRED }

每个 broker 都消费这个 topic, 在内存里维护 Lock Table view.
broker 重启后, replay 该 topic 重建 Lock 状态.

19.6 顺序退化与去重

Share Group 不保证分区内顺序。同一 partition 的 record 100 和 101 可能被不同 Worker 同时处理,甚至 101 可能先完成。如果业务依赖顺序(如"先创建订单再修改"),不要用 Share Group,继续用 Consumer Group。

去重仍是 at-least-once:RELEASE / timeout 后重投,业务侧仍可能看到同一 record 两次(Worker 处理成功但 ACK 未到达 Coordinator 时崩溃)。需要业务侧幂等保证。

19.8 设计取舍

① per-record lock vs offset commit

per-record 提供细粒度但代价是 Coordinator 状态爆炸:1k record/s × 60s lock = 60k 活动锁条目。需要 compacted topic + 内存索引兜底。

② 失去顺序换取吞吐

不依赖 partition 数扩并行,Worker 1000 个都能跑。代价是分区内顺序不保证。补救:业务保证幂等,或对顺序关键 topic 不用 Share Group。

③ DLQ 由业务实现

REJECT 之后 Kafka 不自动入 DLQ。业务在 catch 块里把 record 写 dlq topic + REJECT,逻辑显式。优点是 DLQ 格式自由,缺点是业务多一步代码。

④ 与 EOS 不兼容

Share Group 是 at-least-once,与 read_committed transactional consumer 不兼容(KIP-932 v1)。EOS 流处理继续用 Consumer Group。

19.9 陷阱

#陷阱现象修复
1认为分区内有序"先创建订单后修改"乱序顺序场景用 Consumer Group
2忘了幂等RELEASE 重投后业务双写业务侧加 idempotency key
3delivery limit 设无限毒丸消息无限循环设 3-5 + DLQ
4lock timeout 太短慢任务全 RELEASE调整到业务 P99 + 30s
5Coordinator 单点压力__share_group_state 写入瓶颈分散到多 group
6事务读取场景误用EOS 失效用 Consumer Group + read_committed

19.10 本章要点

本章小结

Share Group(KIP-932)是 Kafka 4.x 一次重要的架构扩展 — 它在不破坏现有 Consumer Group 的基础上,补齐了"记录级竞争消费 / 工作池 / 慢任务"场景。核心机制是 Record Lock(每条 record 在 Coordinator 上有 owner + deadline + count)+ 三态 ACK(ACCEPT / RELEASE / REJECT)+ delivery limit(毒丸防御)。它不是替代 Consumer Group,而是覆盖 RabbitMQ / SQS / ActiveMQ 之前占据的场景。

用好 Share Group 需要记住:① 没有顺序保证;② 仍然是 at-least-once,需要业务幂等;③ Coordinator 是状态热点,压测必看 __share_group_state 写入延迟;④ 与 EOS 不兼容。下一章我们讲 Quotas — broker 怎样限流 Producer / Consumer / Replication 流量,避免噪声邻居问题。
