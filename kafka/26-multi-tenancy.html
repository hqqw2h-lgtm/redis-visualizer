第 26 章 · 多租户与监控

Tenant → Quota → ACL → Namespace → Metrics → Alert · 闭环治理

"生产 Kafka 上有 47 个团队、120 个微服务、3000+ topic — 谁该用多少资源?故障时谁该被叫醒?" — 这是大公司生产 Kafka 的真实形态。把多租户治理做好,是把"共享集群" 从"定时炸弹" 变成"稳定平台" 的关键工程。它要回答四个问题:谁能用(认证)、能用多少(配额)、能访问哪些(ACL)、用了多少 + 何时介入(监控 + Runbook)。

本章把所有"多租户 + 监控"主题汇成一张闭环图,然后逐项展开:Topic Namespace 命名规范、Quota 与 ACL 协同、关键 JMX 指标、PromQL 告警表达式、Runbook 模板、Consumer Lag、反例与陷阱。这是图解 Kafka 的最后一章,把前面所有章节(replication / consumer / KRaft / quota / security)整合到运维实践。

  • 多租户闭环:Tenant → Quota → ACL → Namespace → Metrics → Alert
  • Topic 命名规范(domain.team.project.type.name.vN)
  • Quota + ACL 协同 — 缺一不可
  • 必看 JMX:UnderReplicatedPartitions / RequestQueueSize / RequestHandlerAvgIdlePercent
  • Consumer Lag 的"四种异常形态"
  • PromQL 告警模板
  • Runbook 该写什么(避免"叫醒人但没指引")
  • 反例:无 default quota、ACL 通配符过宽、告警无 owner

26.1 租户模型

层级颗粒度示例
BU(业务部门)10-100 team"电商"、"金融"、"物流"
Team(团队)5-20 app"order"、"payment"、"shipment"
App(应用)1-10 service"order-svc"、"order-worker"
Principal(认证主体)1 service = 1 principalUser:order-svc
Principal 是治理的最小单位。Quota / ACL 都按 principal 配置。一个 app 一个 principal,这样可以独立调整 quota、独立分析监控。多个 app 共用一个 principal 会让责任不清。

26.2 Topic Namespace

推荐命名:

  domain.team.project.type.name.vN

例:
  ecom.order.checkout.event.order-created.v1
  finance.payment.refund.command.refund-request.v2
  logistics.shipment.tracking.snapshot.shipment-state.v1

字段:
  domain  - 业务域 (ecom/finance/logistics/...)
  team    - 团队
  project - 子系统
  type    - event / command / query / snapshot / dlq
  name    - 具体业务名
  vN      - schema 版本

命名规范的三个收益

收益怎样用到
Prefixed ACLkafka-acls --resource-pattern-type PREFIXED --topic ecom.order.
租户标签自动化解析 topic 名前缀 → Prometheus label domain=ecom, team=order
变更评审看到 v2 知道是 schema 升级,新老共存

26.5 PromQL 告警

groups:
- name: kafka-cluster
  rules:
  - alert: KafkaUnderReplicated
    expr: kafka_cluster_partition_under_replicated > 0
    for: 5m
    labels:
      severity: high
    annotations:
      summary: "Kafka under-replicated partitions"
      runbook: "https://wiki.example.com/runbook/kafka-urp"

  - alert: KafkaOffline
    expr: kafka_cluster_partition_offline > 0
    for: 0m
    labels:
      severity: critical
    annotations:
      summary: "Kafka has offline partitions"
      runbook: "https://wiki.example.com/runbook/kafka-offline"

  - alert: KafkaRequestHandlerBusy
    expr: kafka_server_kafkarequesthandlerpool_requesthandleravgidlepercent < 0.3
    for: 5m
    labels:
      severity: warn
    annotations:
      summary: "Kafka request handlers near saturation"

- name: kafka-tenant
  rules:
  - alert: KafkaConsumerLag
    expr: kafka_consumergroup_lag{group=~"order-.*"} > 100000
    for: 10m
    labels:
      severity: high
      tenant: order
    annotations:
      summary: "Consumer group {{ $labels.group }} lag > 100k"
      runbook: "https://wiki.example.com/runbook/kafka-lag"
      owner: "team-order@example.com"

  - alert: KafkaProducerThrottled
    expr: rate(kafka_server_brokertopicmetrics_bytesinpersec_throttle_time_ms[5m]) > 1000
    for: 10m
    labels:
      severity: warn
    annotations:
      summary: "Producer throttled, tenant={{ $labels.user }}"

26.6 Runbook 模板

无 Runbook 的告警等于没告警。Oncall 接到告警如果不知道怎么做,就会"静音"或"等下班再说"。每个告警必须配 Runbook。 
Runbook: KafkaConsumerLag(订单消费滞后)

1 上下文
   - SLA: order-svc 5 分钟内消费完
   - 业务影响: 订单延迟出库
   - Owner: team-order@ (PD 工作时间) / @oncall (24x7)

2 立即诊断
   - kafka-consumer-groups --describe --group order-svc
   - 看是哪个 partition lag 最大
   - 查 partition leader broker 的 RequestQueueSize / CPU

3 常见根因
   a) Consumer 实例少 / OOM 重启
      → 看 K8s Pod 状态, 重启或扩容
   b) 业务处理慢 (调下游变慢)
      → 看 application APM, 找下游瓶颈
   c) Partition 偏斜 (单 partition 远高于其他)
      → 检查 producer key 分布
   d) Broker 网络 / 磁盘问题
      → 看 BytesOutPerSec 是否到 NIC 限

4 缓解
   - 临时:扩 consumer 实例数 (≤ partition 数)
   - 紧急:跳过老数据 (seek to latest, 接受丢消息)
   - 长期:加 partition / 优化下游

5 通知
   - 通知 owner: team-order
   - 通知 SRE: kafka-platform
   - 故障复盘: 5 工作日内

26.9 性能 + 容量规划

容量规划三层维度:

1 集群容量 (broker / 磁盘 / 网络)
   - 公式: peak_throughput × replication_factor × 1.5 安全余量
   - 看 BytesInPerSec / BytesOutPerSec, 留 30% headroom

2 租户配额 (per principal)
   - 总配额 = 集群可用 × tenant 占比
   - 按 BU/Team 划分 (谁付钱谁分多)
   - 每 6 个月评审一次

3 监控容量 (Prometheus / Grafana / Alertmanager)
   - 每 partition 几十个 metric × 几千 partition × 几十 broker
   - 估算: kafka cluster 50 broker × 5k partition → ~100k metrics
   - Prom retention: 短 (1d) + 长 (1y) 分层

26.10 本章要点

本章小结

多租户治理是 Kafka 生产化的最后一公里。核心是闭环:Tenant → Quota → ACL → Namespace → Metrics → Alert → 反馈调整。任何一环缺失都会让"共享集群" 退化成"定时炸弹"。生产实践的三条铁律:① 1 app = 1 principal,治理颗粒度精确到服务;② Topic 命名规范是基础设施,Prefixed ACL + 监控 label 都靠它;③ 告警必须连 Runbook 与 Owner,否则只是"叫醒人"。Consumer Lag 是最重要的业务指标,要看四种形态(健康 / 尖峰 / 上升 / 卡死),不同响应。

这是图解 Kafka 的最后一章。把 26 章串起来 — 你已经从 broker 内部存储(zero-copy / sendfile)、消息分发(reactor / purgatory)、副本协议(ISR / ELR / Leader Epoch)、控制平面(KRaft Raft)、客户端模型(Producer EOS / Consumer Group / Share Group),走到生产运维(quota / security / geo-replication / multi-tenancy / monitoring)。Kafka 不是一个"消息队列" 而是一个"分布式日志" 平台 — 它的设计哲学是"把每个层次的不变量做对,就能撑起整个 ecosystem"。希望这本书帮你看到这些不变量。
