第 11 章 · KRaft 与元数据管理

KIP-500 · KIP-595 · KIP-866 · Controller Quorum · __cluster_metadata · MetadataImage · BrokerHeartbeat

KRaft (Kafka Raft) 让 Kafka 终于摆脱了 ZooKeeper。事实来源(source of truth)从外部 ZK znode 迁回 Kafka 自身的内部主题 __cluster_metadata,通过自实现的 Raft 协议在 Controller Quorum 之间复制。所有元数据变更 — topic 创建、分区 leader 切换、broker 注册、ACL、动态配置 — 都成为这条 log 上的一条 record,broker 用 Fetch 拉取增量来构建本地 MetadataImage。

本章按 "动机 → 拓扑 → Raft 选举 → 元数据日志 → broker 生命周期 → snapshot → 迁移 → 排障 → 取舍 → 陷阱" 组织,对应 Apache Kafka 3.7 的实现。从 3.3 起 KRaft 是 production-ready,从 3.5 起新集群默认 KRaft,4.0 完全移除 ZooKeeper。读完本章你能不依赖任何 ZK 知识地理解 Kafka 元数据的复制、传播、恢复全过程。

  • ZK 模式的双源问题(ZK znode + Controller 内存缓存)以及失败模式
  • KRaft 的 Isolated 部署拓扑:为什么 controller 节点和 broker 节点必须分开
  • controller.quorum.voters 静态配置 vs. KIP-853 动态成员管理(3.7 起)
  • kraft-controller 内部 Raft 的 election timeout、fetch heartbeat、Pre-vote 优化
  • __cluster_metadata 单分区单副本 quorum 日志的格式与 record 类型
  • Broker 启动后默认 Fenced → BrokerHeartbeat → Unfenced 的状态机
  • MetadataImage 与 MetadataDelta:批量应用元数据变更
  • Snapshot 触发条件、文件名格式、读取时机
  • kafka-metadata-shell 和 kafka-storage 工具的排障用法
  • 从 ZK 模式迁移到 KRaft 的 Dual-Write 桥接方案

前置知识:第 9 章 ISR/Leader Epoch 概念(KRaft 元数据日志也用 epoch);Raft 协议基础(term/election/log replication 三件套)。读 §11.3 时如果对 Raft 不熟,可以先复习 etcd 或 raft.pdf 论文。

11.1 ZooKeeper 模式的瓶颈

要理解 KRaft 为什么这么设计,得先理解它要解决什么。ZK 模式的核心问题不是 "ZK 慢",而是 "双源不一致":同一份元数据存在两处 — ZK 的 znode 和 Controller 进程的内存缓存。这两者靠 watcher + RPC 同步,任一侧滞后或漏通知,都会让 broker 路由错乱。

图 11-1 · ZK 模式的双源问题
瓶颈触发场景量化影响
Controller failover 慢元数据规模 200 000 partitions重建耗时 60-120 秒
ZK 写入延迟ISR 频繁变化单次写入 P99 5-50 ms
Watch 通知竞态高频元数据变更漏通知率 < 0.1% 但影响一致性
双系统运维升级 / 备份 / 监控运维复杂度 × 2

KRaft 的修正方向:元数据单一事实来源 · Controller failover < 1 秒 · 元数据可作为日志回放 · 移除 ZK 运维负担。这一切的核心都是把元数据视为一条可复制、可快照、可回放的日志。

11.2 KRaft 拓扑与配置

生产环境 KRaft 必须采用 Isolated 模式:Controller 节点只跑控制平面,Broker 节点只跑数据平面。Combined 模式(单进程同时承担 controller + broker)适合本地实验或 < 5 节点的小集群,但生产中 broker 的 IO/GC 压力会传染到控制面,引发 Controller 选举抖动。

图 11-2 · Isolated 部署:控制面与数据面分离

Controller 配置

# config/kraft/controller-0.properties
process.roles=controller
node.id=3000

controller.listener.names=CONTROLLER
listeners=CONTROLLER://controller-0.example.com:9093
listener.security.protocol.map=CONTROLLER:SSL

# 仲裁组成员(KIP-595 静态配置)
controller.quorum.voters=\
  3000@controller-0.example.com:9093,\
  3001@controller-1.example.com:9093,\
  3002@controller-2.example.com:9093

metadata.log.dir=/var/lib/kafka/kraft-controller
metadata.log.max.snapshot.interval.ms=3600000     # 1 小时一次 snapshot
metadata.log.segment.bytes=1073741824             # 1 GiB 一段
metadata.log.retention.ms=604800000               # 7 天后清理 snapshot 之前的 log

Broker 配置

# config/kraft/broker-100.properties
process.roles=broker
node.id=100

listeners=PLAINTEXT://broker-100.example.com:9092
controller.listener.names=CONTROLLER
controller.quorum.voters=\
  3000@controller-0.example.com:9093,\
  3001@controller-1.example.com:9093,\
  3002@controller-2.example.com:9093

log.dirs=/var/lib/kafka/data
broker.heartbeat.interval.ms=2000
broker.session.timeout.ms=9000

存储格式化

11.3 Raft 选举与心跳

KRaft 内部用 Raft 协议在 Controller Quorum 之间选举。它和 etcd Raft 大同小异,但有几个 Kafka 特定优化:Pre-vote 减少不必要的 term bump、Fetch-based replication 复用 Kafka 现有 Fetch RPC、不存日志中带 noop commit 条目。

图 11-3 · Raft 状态机:Follower → Candidate → Leader

Pre-vote 优化

Pre-vote 是 KRaft 借鉴 etcd 的优化:候选人不直接 bump term,而是先发 PreVoteRequest 试探"如果我成为候选人,你会投我吗?"。多数派同意了才真正 bump term。这避免了一种常见抖动 — broker 网络瞬断后恢复,如果直接 bump term 会无谓地推高全集群 term,而集群明明本来工作得好好的。

Fetch-based replication

KRaft 不用 Raft 论文里的 AppendEntries,而是复用 Kafka 的 Fetch RPC。Follower 主动拉,leader 不主动推。和数据面副本协议(第 9 章)用的是同一套 Fetch 实现,这让代码复用率极高。

11.4 元数据日志结构

元数据日志住在 __cluster_metadata 这个特殊主题里。它有几个独特属性:单分区、单副本组(controller quorum 全员)、永不删除(只 truncate snapshot 之前)、record 是 schema 化的。

常见 Record 类型

PartitionChangeRecordISR / leader 变更topicId, partitionId, isr, leader

MetadataImage 与 MetadataDelta

图 11-4 · MetadataImage 的批量应用模型
Broker 应用元数据流程:

  1 当前快照状态 = MetadataImage @ offset 100
    {
      brokers: {100, 101, 102},
      topics:  {order-created: ...},
      configs: {...}
    }

  2 Fetch 拉到一批新 record (offset 101..105)
    建立 MetadataDelta:
      [101] PartitionChange (order-created-3, leader: 101 -> 102)
      [102] RegisterBroker  (broker 103)
      [103] PartitionChange (order-created-3, isr: [101,102] -> [102])
      [104] ConfigRecord    (topic order-created, retention.ms=...)
      [105] PartitionChange (...)

  3 一次性 apply delta → 新 MetadataImage @ offset 105
    (单次锁 + 批量更新,而非每条 record 一次锁)

  4 通知 client / 内部组件: metadata changed at offset 105

11.5 Broker 生命周期

KRaft 中 broker 的生命周期比 ZK 模式严格得多。新启动的 broker 默认是 Fenced 状态,不接受 leader 角色,直到追上 metadata offset + 心跳通过。这避免了"broker 元数据还旧着就被选为新 leader 导致数据丢失"这种 ZK 时代的暗病。

图 11-5 · Broker 生命周期状态机

BrokerHeartbeat 协议

BrokerHeartbeatRequest 字段:
  brokerId               int32
  brokerEpoch            int64
  currentMetadataOffset  int64     ← 本地已 apply 到哪
  wantFence              bool      ← 优雅退出时 true
  wantShutDown           bool

Controller 响应:
  errorCode              int16
  isCaughtUp             bool      ← 是否追上
  isFenced               bool
  shouldShutDown         bool

→ broker 每 broker.heartbeat.interval.ms (默认 2s) 发一次
→ controller 超过 broker.session.timeout.ms (默认 9s) 未收到 → 标记 Fenced

11.6 元数据 Snapshot

元数据日志会不断增长,如果没有清理机制,Controller 启动时要回放数月的历史。Snapshot 是解决方案:周期性把 MetadataImage @ offset N 序列化到磁盘,然后可以丢弃 offset N 之前的所有 log record。

Snapshot 触发条件

  • 大小触发:metadata.log.bytes / committed metadata 累计超过 metadata.log.segment.bytes
  • 时间触发:距上次 snapshot 超过 metadata.log.max.snapshot.interval.ms(默认 1 小时)
  • 手动触发:kafka-storage.sh snapshot(罕用)

Snapshot 文件结构

/var/lib/kafka/kraft-controller/__cluster_metadata-0/

  00000000000000000000.log               ← 日志段 0
  00000000000000000000-0000000000.checkpoint  ← snapshot 文件
  00000000000000050000.log               ← 日志段 50000(snapshot 之后)
  00000000000000050000.index
  partition.metadata
  quorum-state

snapshot 文件名: -.checkpoint
  committedOffset 包含的最大 offset
  epoch           生成 snapshot 时的 leader epoch

Snapshot 加载流程

Controller 启动恢复:

  1 在 __cluster_metadata-0 目录找最新的 snapshot 文件
  2 读取 snapshot → 初始化 MetadataImage
  3 从 snapshot.committedOffset+1 开始重放后续 log
  4 应用完所有 log → MetadataImage 最终态
  5 准备好接受 client 请求

→ 假设 snapshot @ offset 50000, log 最新 @ offset 50200
→ Controller 启动仅需读取 1 个 snapshot + 200 条 log record
→ 通常 < 1 秒就能完成,而非 ZK 时代的分钟级

11.7 ZK 到 KRaft 迁移

从 Kafka 3.5 起官方提供了 ZK → KRaft 在线迁移路径,核心思路是 Dual-Write:Controller 同时写 ZK 和 KRaft 日志,直到所有 broker 都升级到 KRaft 模式,然后切换。整个过程不需要重启业务,但需要仔细规划。

迁移核心步骤
1 准备阶段 (Kafka ≥ 3.5)
  - 升级所有 broker 到 3.5+
  - 启用 zookeeper.metadata.migration.enable=true

2 部署 KRaft Controller Quorum
  - 单独部署 3 个 controller 节点
  - controller 知道 zookeeper.connect, 进入 Dual-Write 模式

3 Migration In Progress
  - controller 将 ZK 中所有元数据写入 KRaft 日志
  - 同时维护 ZK 与 KRaft 双向同步
  - 持续到 inter.broker.protocol.version 升到 3.5

4 Broker 滚动升级
  - 每个 broker: process.roles=broker + controller.quorum.voters
  - 不再连 ZK, 改连 controller quorum

5 切换最终状态
  - 所有 broker 完成升级
  - controller: zookeeper.metadata.migration.enable=false
  - 移除 ZK 集群
迁移最大的风险是回退路径 — 一旦切换完成,无法回退到 ZK 模式(KRaft 日志格式与 ZK znode 不对等)。生产迁移强烈建议先在克隆环境演练一遍,确认所有客户端兼容性、监控覆盖、运维剧本都到位。

11.9 设计取舍

① 自托管 Raft vs 外部 ZK

不再依赖 ZK 简化运维,但要求 Kafka 自己实现 Raft。Kafka 选了一个工程性最强的实现 — Fetch-based replication,与现有 Kafka log 协议共享代码,可靠性可量化。

② Pull vs Push

Broker 主动 Fetch 元数据,而非 Controller 主动 Push。流控由 broker 控制,慢 broker 不阻塞 controller。代价是有 1 RTT 的元数据传播延迟(通常 < 100ms)。

③ Fenced 默认安全

新 broker 默认 Fenced,等追上 metadata + 心跳通过才 Unfenced。代价是启动慢一点(几百毫秒),收益是不会因为元数据陈旧选错 leader。

④ Snapshot 周期

过短 snapshot 频繁占 IO,过长 controller 启动慢。默认 1 小时是经验值;高吞吐元数据场景可缩短到 15 分钟。

本章小结

KRaft 让 Kafka 从一个需要 ZooKeeper 的复杂系统,变成了一个完整、自洽、可控的分布式日志平台。核心创新不是"自己实现了 Raft",而是把元数据视为可复制的日志。Active Controller 唯一写入,Follower Controller 持续 replicate,Broker 通过 Fetch 增量拉取并构建 MetadataImage。这套模型让 controller failover < 1 秒,百万级 partitions 不再是奢望。Fenced 状态机进一步杜绝了"元数据陈旧的 broker 当选 leader 导致数据丢失"的可能。下一章我们转向另一个维度的正确性 — Exactly-Once Semantics(EOS),看看 Kafka 如何在生产者层面、消费者层面、Streams 层面分别给出 exactly-once 保证。
