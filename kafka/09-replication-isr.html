第 9 章 · 副本协议与 ISR

LEO · HW · ISR 状态机 · acks 与 minISR · 干净选举 · Leader Epoch · Rack Awareness · 排障

Kafka 的副本协议不是"写三份"那么简单。它把写入集中在 leader,把复制委托给 follower 主动拉取,再用 ISR、HW、LEO 与 min.insync.replicas 四个核心概念共同回答"哪些数据已经安全到可以暴露给消费者"。生产环境最大的边界是:在可用性和不丢已提交消息之间,你要做出明确选择 — 这一选择会直接影响 SLA 与告警规则。

本章按 "目标 → leader/follower 模型 → LEO 与 HW → ISR 收缩与扩张 → acks 与 minISR → leader 选举 → leader epoch → rack-awareness → 排障 → 源码 → 取舍 → 陷阱" 组织,对应 Apache Kafka 3.7 的 KRaft 模式实现。读完后你应能看着 kafka-topics.sh --describe 的输出说出某个分区的安全级别,并准确预判某种故障会带来什么后果。

  • LEO / HW / ISR 三个核心指标的定义、推进时机与持久化机制
  • Follower 拉取(Fetch)而非 leader 推送(Push)的工程理由
  • HW 必须经过 2 个 RTT 才能推进的原因 — remoteLEO 更新时序
  • ISR 状态机的四态:InISR / OutOfSync / CatchingUp / Offline,以及 replica.lag.time.max.ms 的真实语义
  • acks=all 在 minISR 配错时如何"静悄悄地"退化成 acks=1
  • 干净选举(Clean Election)与不干净选举(Unclean Election)的安全/可用性取舍
  • Leader Epoch + OffsetForLeaderEpoch 协议如何防止"脑裂截断"
  • Rack Awareness 副本布局对单 AZ 故障的容灾收益与跨 AZ 流量代价
  • UnderReplicatedPartitions / UnderMinIsrPartitionCount / IsrShrinksPerSec 三个告警指标的根因排查路径
  • 八个生产环境最常见的副本陷阱与对应修复

ISR 不是配置,而是运行时状态。它每秒都在变化:某个 broker GC 一停就被踢出去,网络一恢复就被加回来。所以 acks=all 的真实语义不是"等所有副本",而是"等当前这一刻的 ISR 全员"。这一点是 90% 生产事故的根源 — 配了 RF=3 + acks=all,以为安全得不得了,结果 minISR 没设 / 设成了 1,在 ISR 缩到 1 副本的瞬间,acks=all 静悄悄地退化成了 acks=1。

9.1 副本协议的目标

设计副本协议要回答三个问题:写入算成功了吗?消费者能读到哪里?谁能当新 leader?三个问题分别对应"持久性、可见性、可用性"三个维度。Kafka 用 acks + minISR + ISR + HW + leader epoch 这套机制把它们串起来,但代价是这套机制有不少边界条件,配错任何一个都会让其他几个失去意义。

9.2 Leader / Follower 模型

Kafka 写入只走 leader,follower 不接受写。但 follower 也不是被动接收 — 它是主动 Fetch,像普通 consumer 一样去 leader 拉数据。这个看似"反直觉"的设计,其实是 Kafka 在工程上的一次关键取舍。

图 9-2-1 · 写入与复制时序:LEO/HW 如何随时间推进
角色写请求读请求副本职责状态维护
Leader接收并持久化服务消费者维护 ISR / HWLEO + remoteLEO[*]
Follower不直接接收通常不服务(KIP-392 例外)主动 Fetch leader本地 LEO + epoch cache

Follower Fetch 的工程理由

图 9-2-2 · Follower 像消费者一样主动拉
ReplicaFetcherThread loop:
  while running:
    currentLEO = log.endOffset
    sendFetchRequest(
      leader,
      offset    = currentLEO,
      maxBytes  = replica.fetch.max.bytes,
      maxWait   = replica.fetch.wait.max.ms
    )
    response = receive()

    for batch in response.records:
      log.appendAsFollower(batch)

    sleep(0 or fetch.wait.max.ms)

  收益:
    1 复用 Fetch 协议(与 consumer 完全一致)
    2 leader 无需推送状态机,代码简单
    3 流控由 follower 自身控制,慢 follower 不阻塞 leader
    4 backpressure 天然存在 — follower 拉得慢 = 自己被踢出 ISR

remoteLEO 推进时机 — 为什么需要 2 个 RTT

图 9-2-3 · 隐式 ACK 协议 — Fetch 是 push 的反向
关键点:remoteLEO 仅在 follower **下一次** Fetch 时推进

  T0: Leader LEO=100 HW=100  remoteLEO[F1]=100 remoteLEO[F2]=100
  T1: Producer append 100..102
      Leader LEO=103 HW=100 (F1/F2 还未拉)
  T2: F1 Fetch(offset=100)
      Leader 返回 100..102, remoteLEO[F1] 仍 100
      (F1 即将拉,不是已确认)
  T3: F1 完成本地 append, 下次 Fetch(offset=103)
      → remoteLEO[F1] = 103
  T4: F2 完成本地 append, 下次 Fetch(offset=103)
      → remoteLEO[F2] = 103
      HW = min(103, 103, 103) = 103

  → HW 推进至少需要 2 个 RTT
  → follower 通过"下次 Fetch 的 offset"隐式 ACK 上次写入
  → 没有显式 commit / ack 协议,所有信息复用 Fetch 协议
这种"隐式 ACK"的设计有一个微妙后果:即使 follower 已经写入本地,leader 不会立刻知道 — 必须等下一轮 Fetch。这意味着 HW 永远比真实复制状态滞后 1 个 RTT。在 P99 抖动的网络环境里,这一滞后可能成倍放大,表现为"明明 producer 已经收到 ACK 几百毫秒了,consumer 还看不到"。这不是 bug,是协议的本质。

9.3 LEO 与 HW

LEO 和 HW 是副本协议中最容易混淆的两个概念。简单说:LEO 是每个副本各自的"下一条可写位置";HW 是leader 全局唯一的"消费者可见边界"。HW 必然 ≤ 任何 ISR 副本的 LEO,且严格等于 min(LEO over ISR)。

HW 计算公式

long computeHighWatermark(long leaderLeo,
                          int leaderBrokerId,
                          Map<Integer, Long> remoteLeo,
                          Set<Integer> isr) {
    long hw = leaderLeo;
    for (int brokerId : isr) {
        if (brokerId != leaderBrokerId) {
            Long followerLeo = remoteLeo.get(brokerId);
            if (followerLeo == null) {
                throw new IllegalStateException("missing ISR replica LEO: " + brokerId);
            }
            hw = Math.min(hw, followerLeo);
        }
    }
    return hw;
}

9.4 ISR 收缩与扩张

ISR 是动态变化的集合。Kafka 不要求所有副本都同步,只要求"跟得上"的副本参与提交。"跟得上"的判定是基于时间(replica.lag.time.max.ms 默认 30s),不是基于"字节数滞后"(replica.lag.max.messages 在 0.9 已删除)。这个改动是 KIP-1 — 它让突发流量不再因为"瞬间落后 10MB"而触发 ISR 抖动。

9.5 acks 与 min.insync.replicas

这是整章最容易踩坑的地方。acks=all 听起来很安全,但它只等"当前 ISR 全员",而 ISR 是动态的。如果 minISR=1,在 ISR 缩到 1 个副本的瞬间,acks=all 就静悄悄地退化成 acks=1 — 仅 leader 本地 append 即返回成功。这种"伪 acks=all"在故障时表现为已确认的数据消失,而监控却显示 producer 的 ACK 率 100%。

图 9-5-1 · acks 与 minISR 的组合矩阵

acks 三种模式

acks等待对象写入延迟数据安全典型场景
0不等最低最低指标 / 心跳
1leader 本地 append中中(leader 故障可能丢)日志
all / -1当前 ISR 全员复制高最高订单 / 支付

acks=all 退化反例

反例(minISR=1): 
T0: ISR={1,2,3}, RF=3, minISR=1, Producer acks=all
    Leader 1 等 follower 2+3 → 持久性正常

T1: 网络抖动, follower 3 滞后 30s
    ISR 收缩到 {1, 2}
    Producer 仍可写, 等 2 副本

T2: follower 2 也滞后
    ISR 收缩到 {1}
    Producer 仍可写, acks=all 退化为 acks=1
    ↑ 实际上仅 leader 本地 append 即返回成功
    ↑ 若 leader 立即崩溃 → 数据丢失

根因: minISR=1 时, ISR 缩到 1 仍允许写
正确: minISR=2 (RF=3 时), ISR={1} 直接拒写

9.6 Leader 选举

Leader 故障后,Controller 要选出新 leader。选举有两条路径:干净选举(从 ISR 选)和 不干净选举(从 OSR 选)。前者保证不丢已提交数据,后者用可用性换数据安全。生产环境绝大多数 topic 都应禁用 unclean,宁可分区不可用也不接受数据消失。

Unclean 选举风险

场景: RF=3, ISR={1}, Replicas={1,2,3}
      (Follower 2/3 离线超 30s, 已移出 ISR)
      Leader=1, LEO=200, HW=200

T0: Leader 1 崩溃 → ISR 已只剩 1, 现在为空
    关闭 unclean → 分区不可用
    打开 unclean → 选 Follower 2 (LEO=100, 落后 100 条)
                → 100..199 已提交数据丢失
                → consumer 看到 offset 回退

unclean 含义:
  ✗ 用可用性换数据安全
  ✗ 可能导致已提交数据消失
  → 生产环境通常关闭

四种选举类型对照

类型触发来源数据安全配置
干净选举leader 故障 + ISR 非空ISR 第一存活副本✓ 不丢unclean=false
Unclean 选举leader 故障 + ISR 空任何存活副本⚠️ 可能丢unclean=true
Preferred 选举手动 / 自动再平衡Replicas[0]✓ 安全auto.leader.rebalance
Reassignment分区迁移新分配位置✓ 安全kafka-reassign-partitions

9.7 Leader Epoch

仅靠 HW 不足以处理"网络分区 + 双主"的场景 — 旧 leader 在分区期间多写了几条数据,新 leader 选出后,旧 leader 恢复时该如何对账?Kafka 引入 Leader Epoch 作为单调递增的时代戳,每次 leader 切换 epoch + 1,follower 用 OffsetForLeaderEpoch 协议向新 leader 查"我最后那个 epoch 的 endOffset 是多少",超出的本地日志直接截断。KIP-101 和 KIP-279 分别引入和修正了这个机制。

Leader Epoch 设计动机

图 9-7-1 · 仅靠 HW 处理不了的场景
T0: Leader 1, ISR={1,2,3}, HW=100
T1: Leader 1 网络分区, 写入 101..105
    Leader 1 LEO=105, 但 follower 2/3 联系不上
T2: Controller 检测 Leader 1 失联, 选 Follower 2 为新 leader
T3: 新 Leader 2 LEO=100
    Producer 重连后写 101..103 到新 leader
T4: Leader 1 恢复网络, 看到自己 LEO=105
    Follower 2 LEO=103
    若仅按 HW 同步, 旧 leader 的 101..105 和新 leader 的 101..103 冲突

解决:
  引入 leader epoch
  每次 leader 切换 epoch + 1
  HW 与 epoch 配对
  不同 epoch 的 LEO 不可直接比较

KIP-101 引入 leader epoch
KIP-279 修正 follower 截断逻辑
图 9-7-2 · Leader Epoch 时序:旧 leader 恢复后截断

Leader Epoch Checkpoint 文件

/var/kafka-logs/order-created-0/leader-epoch-checkpoint:

  0           ← 版本(0)
  3           ← 条目数(3)
  42 0        ← epoch 42 从 offset 0 开始
  43 100      ← epoch 43 从 offset 100 开始
  44 245      ← epoch 44 从 offset 245 开始

解读:
  epoch 42 区间 [0, 99]
  epoch 43 区间 [100, 244]
  epoch 44 区间 [245, ∞)

作用:
  follower 启动 → 读此文件
  → 向 leader 查询自己最后 epoch 的 endOffset
  → 若本地 LEO > 该 endOffset → 截断到 endOffset
  → 避免数据冲突

OffsetForLeaderEpoch 协议

Follower 启动恢复流程:

  1 读本地 leader-epoch-checkpoint
    最后 entry: epoch=42, startOffset=0
    本地 LEO = 105

  2 向 leader 发 OffsetForLeaderEpoch:
    request:
      topic              : order-created
      partition          : 0
      currentLeaderEpoch : 43
      lastFetchedEpoch   : 42

  3 leader 响应:
    response:
      epoch     : 42
      endOffset : 100   (epoch 42 的最后 offset)

  4 follower 比较:
    本地 LEO (105) > endOffset (100)
    → 截断本地日志到 offset 100
    → 删除 offset 100..104 的 records

  5 follower 用新 epoch (43) 继续 Fetch

9.8 Rack Awareness

Rack Awareness 是把"机架/AZ 拓扑"告诉 Kafka,让它在分配副本时尽量分散。配置 broker.rack 后,Kafka 会确保同一分区的 3 个副本来自不同 rack,防止"整个机架掉电时全部副本一起丢"。代价是跨 AZ 复制流量 = (RF-1) × 写入流量,需要靠压缩控制绝对带宽。

跨 AZ 流量成本

图 9-8-2 · 跨 AZ 流量计算 + KIP-392 优化
RF=3 跨 3 AZ, leader 在 az-a:

  producer (az-a) → leader (az-a):     0 跨 AZ 流量
  leader (az-a)  → follower-1 (az-b):  1× 写入流量
  leader (az-a)  → follower-2 (az-c):  1× 写入流量
  consumer (az-a) ← leader (az-a):     0 跨 AZ 流量

  跨 AZ 总流量 = 2× 写入流量 = (RF-1) × 写入流量

优化方案 (KIP-392 Follower Fetching):
  consumer 可从 **同 AZ 的 follower** 读取
  → 降低跨 AZ 读流量
  → 需 client 配合 client.rack 配置
  → 适合大流量读场景(写少读多)

最佳实践:
  - producer 写入用 acks=all + zstd 压缩(降绝对带宽)
  - 监控 BytesInPerSec(跨 AZ 即可推算成本)
  - 大流量读场景启用 KIP-392

appendRecords 简化代码

def appendRecords(timeout: Long,
                  requiredAcks: Short,
                  internalTopicsAllowed: Boolean,
                  origin: AppendOrigin,
                  entriesPerPartition: Map[TopicPartition, MemoryRecords],
                  responseCallback: Map[TopicPartition, PartitionResponse] => Unit) {

  if (isValidRequiredAcks(requiredAcks)) {

    val localProduceResults = appendToLocalLog(
      internalTopicsAllowed, origin, entriesPerPartition, requiredAcks)

    val produceStatus = localProduceResults.map { case (tp, result) =>
      tp -> ProducePartitionStatus(
        result.info.lastOffset + 1,
        new PartitionResponse(result.error, result.info.firstOffset, ...))
    }

    // acks=all 时进入 Purgatory 异步等待
    if (delayedProduceRequestRequired(requiredAcks, entriesPerPartition, localProduceResults)) {
      val produceMetadata = ProduceMetadata(requiredAcks, produceStatus)
      val delayedProduce = new DelayedProduce(timeout, produceMetadata, this, responseCallback)
      delayedProducePurgatory.tryCompleteElseWatch(delayedProduce, ...)
    } else {
      responseCallback(produceStatus.mapValues(_.responseStatus))
    }
  }
}

HW 更新核心逻辑

def maybeIncrementHighWatermark(): Boolean = {
  val leaderLogEndOffset = leaderLog.logEndOffset
  val newHighWatermark = inSyncReplicaIds.map { replicaId =>
    if (replicaId == localBrokerId)
      leaderLogEndOffset
    else
      remoteReplicas(replicaId).logEndOffset
  }.min

  val oldHighWatermark = leaderLog.highWatermark
  if (newHighWatermark > oldHighWatermark) {
    leaderLog.updateHighWatermark(newHighWatermark)
    true
  } else {
    false
  }
}

ISR 收缩判断

def maybeShrinkIsr(): Unit = {
  val currentTimeMs = time.milliseconds()
  val outOfSyncReplicas = inSyncReplicaIds.filter { replicaId =>
    if (replicaId == localBrokerId) false
    else {
      val replica = remoteReplicas(replicaId)
      val timeSinceLastFetch = currentTimeMs - replica.lastCaughtUpTimeMs
      timeSinceLastFetch > replicaLagTimeMaxMs
    }
  }

  if (outOfSyncReplicas.nonEmpty) {
    val newInSyncReplicaIds = inSyncReplicaIds -- outOfSyncReplicas
    expandIsrOrShrinkIsr(newInSyncReplicaIds)
  }
}

四大设计取舍

① Leader-Only 写入

所有写入集中到 leader,简化顺序与冲突。代价是 leader 是单点瓶颈,需靠分区扩展水平扩容。Kafka 选这个是因为分区天然是并行单位,leader 热点可通过分区数缓解。

② Follower Pull 拉取

Follower 主动 Fetch,而非 leader 推送。复用 Fetch 协议,流控由 follower 控制,leader 不被慢 follower 阻塞。代价是复制延迟取决于 follower fetch 频率。

③ HW = ISR LEO 最小值

消费者仅可读 ISR 全员复制的数据,确保任意 ISR 副本当选新 leader 后已提交数据不丢。代价是慢 ISR 拖慢 HW 推进,replica.lag.time 控制拖累窗口。

④ minISR 拒写而非降级

ISR < minISR 时直接拒绝写入,而非降级到更弱的 acks。用不可用换数据安全。生产端必须接受短暂不可写,而非接收单副本 ACK 的假成功。

反例 1:RF=2 + minISR=1 
问题:
  1 broker 故障 → ISR={1} → 退化为 acks=1

  1 broker 故障仍能正常写入
  ISR 不会跌破 minISR
反例 2:开 unclean leader election

场景:
  Producer 写 100..200, HW=200, ISR={1,2,3}

  Leader 1 崩溃, ISR 为空
  Follower 2 (LEO=100) 当选新 leader

  unclean.leader.election.enable=false
  minISR=2 防止 ISR 缩到 1

本章小结

副本协议把"写入成功的定义"从单机变成了集合语义。LEO 在本地推进,HW 经过 2 个 RTT 才能让 consumer 看到,ISR 是这一切的动态参与名单。acks=all + min.insync.replicas + unclean=false 三件套缺一不可 — 单独配 acks=all 在生产环境是危险的虚假安全感。理解 Leader Epoch 后,就不会再担心"网络分区 + 旧 leader 恢复"导致的数据冲突,因为 OffsetForLeaderEpoch 会让旧 leader 主动截断超出的本地日志。下一章我们会专门把 Leader Epoch 摊开到字节级:它如何在 segment 重启、日志清理、分区迁移等场景下保持单调,以及它和 HW 如何共同构成 Kafka "不丢已提交数据"的最终保证。
