第 21 章 · Kafka Connect

DistributedHerder · Worker · Source/SinkTask · Converter · SMT · connect-configs/offsets/status · DLQ · REST API

"把 MySQL CDC 写到 Kafka,再从 Kafka 写到 Elasticsearch + S3 + Snowflake" — 这种"把 Kafka 当数据总线" 的场景每个公司都有。手写 Producer / Consumer 跑通一条链路简单,但在生产环境会重复造轮子:任务拆分、failure retry、offset commit、schema 演进、监控告警、滚动升级。Kafka Connect 是 Kafka 官方提供的数据集成框架,把上面这些公共关注点抽成框架,你只需写 Connector / Task 类(或装 Confluent / Debezium 提供的现成 connector),其他都框架代办。

本章按"运行时拓扑 → 核心模型 → Source/Sink → Distributed → Rebalance → Converter → SMT → Offset 与 EOS → DLQ → REST → 监控"组织,对应 Apache Kafka 3.7 的 org.apache.kafka.connect.runtime。

  • Connect 三大角色:Connector(声明性配置)/ Task(实际搬数据)/ Worker(执行环境)
  • Distributed 模式:worker 集群、internal topics、rebalance
  • Converter(JSON / Avro / Protobuf / String)+ SMT 变换链
  • Source offset 持久化在 connect-offsets + EOS Source(KIP-618)
  • DLQ:errors.tolerance + errors.deadletterqueue.topic.name
  • REST API:管理 connector 全生命周期
  • Schema Registry 与 Connect 的协作
  • 典型反例:offset 不推进、connector 不停 rebalance

21.1 适用场景

✅ 用 Connect

  • 定型的 source / sink 集成(MySQL CDC、S3 sink、ES sink)
  • 需要 task 拆分 / failover / rebalance
  • 需要 Schema Registry 协作
  • 不想自己管 offset 持久化
  • 需要 DLQ + retry policy

❌ 自己写 Producer/Consumer

  • 需要复杂业务逻辑(不仅是 search-replace 变换)
  • 需要跨数据源 join 或聚合(用 Streams)
  • 极致延迟要求(Connect 多一跳 framework 开销)
  • 单次 ETL 任务(Connect 部署成本高)

21.3 Distributed 模式

Connect 有两种部署模式:Standalone(单 JVM,本地文件存 offset)和 Distributed(多 worker 集群)。生产必用 Distributed。它的核心是用 Kafka 自己的 Consumer Group 协议做 worker 间协调 — 所有 worker 共享 group.id,通过 GroupCoordinator rebalance 决定哪个 worker 跑哪个 task。

# worker 配置
bootstrap.servers=kafka:9092
group.id=connect-cluster-A           # 同 group.id 的 worker 加入同一集群
config.storage.topic=connect-configs
offset.storage.topic=connect-offsets
status.storage.topic=connect-status
config.storage.replication.factor=3
offset.storage.replication.factor=3
status.storage.replication.factor=3
plugin.path=/opt/connect-plugins

21.4 Rebalance

Connect 用 Incremental Cooperative Rebalance(KIP-415)— 与 Consumer Group 的 cooperative 类似但更精细。worker 加入 / 退出 / 配置变化都会触发 rebalance,但只有真正受影响的 task 会迁移,无关 task 继续跑。

触发场景受影响 task停顿
worker 加入分配新 task 给它无,新 task 启动
worker 退出它的 task 重分配仅那些 task 短停顿
配置变更(scale tasks.max)所有该 connector 的 task该 connector 短停顿
连接 broker 失败所有 task全部停顿,直到恢复

21.5 Converter & SMT

Converter

序列化器,负责"SourceTask 输出 Struct → byte[]" 或反向。常见:

  • StringConverter — UTF-8
  • JsonConverter — JSON,可带 schema
  • AvroConverter — Avro + Schema Registry
  • ProtobufConverter — Protobuf + Registry

SMT(Single Message Transform)

一条消息的内联变换链。例子:

transforms=AddTopicPrefix,ExtractField
transforms.AddTopicPrefix.type=org.apache.kafka.connect.transforms.RegexRouter
transforms.AddTopicPrefix.regex=(.*)
transforms.AddTopicPrefix.replacement=prod.$1

transforms.ExtractField.type=org.apache.kafka.connect.transforms.ExtractField$Value
transforms.ExtractField.field=payload

21.6 Source offset 与 EOS

Source connector 的 offset 是"外部系统的位置"(MySQL binlog position、File line number、API cursor),与 Kafka offset 没关系。Connect 把它存到 connect-offsets internal topic。

connect-offsets record(简化):

  KEY: { connector: "mysql-cdc",
         partition: { server: "mysql-1", database: "shop", table: "orders" } }
  VAL: { binlog_file: "mysql-bin.000123",
         binlog_pos: 456789,
         row: 42 }

Worker SourceTask 每完成一批 poll():
  1 把 records 写 Kafka(异步)
  2 records 都 ack 后, 把 SourceOffset 写 connect-offsets
  3 崩溃后, 新 worker 从 connect-offsets 读出位置, 继续 poll

EOS Source(KIP-618)

从 Kafka 3.0 起,Connect 支持Source EOS:把"写 Kafka" 和"提交 source offset 到 connect-offsets" 包在一个事务。两者要么都成功要么都失败。开启:

# worker level
exactly.once.source.support=enabled

# producer 自动开启 idempotent + transactional

21.7 DLQ 死信

Connect 内置 DLQ 机制 — 当 Sink task 处理一条 record 失败(反序列化错、下游写失败),框架可以选:跳过、停 task、或写 DLQ topic。

errors.tolerance=all              # 容忍所有错(否则会停 task)
errors.deadletterqueue.topic.name=es-sink-dlq
errors.deadletterqueue.topic.replication.factor=3
errors.deadletterqueue.context.headers.enable=true  # 加诊断 header

POST /connectors/{name}/restart重启 connector

PUT /connectors/{name}/config更新配置(触发 rebalance)
DELETE /connectors/{name}删 connector

21.10 取舍与陷阱

#陷阱现象修复
1internal topic RF=1worker 集群崩塌后无法恢复RF ≥ 3
2tasks.max 太小并行度不够= partition 数 (sink) 或 = 资源单元 (source)
3Converter 不匹配反序列化失败客户端 / Connect 用同 Converter
4不开 EOS Source故障重启可能丢/重3.0+ 开 exactly.once.source.support
5忽视 DLQ一条坏数据停 taskerrors.tolerance=all + DLQ
6plugin.path 错connector 加载不到检查目录权限
7不监控 rebalance频繁 rebalance 导致延迟看 connect-worker-rebalance-metrics

21.11 本章要点

本章小结

Kafka Connect 是把"Kafka 与外部系统的胶水代码" 工程化的框架。三大角色清晰:Connector 声明(不搬数据)、Task 实际搬、Worker 提供执行环境。状态全在 Kafka internal topics(connect-configs / offsets / status),所以 worker 是无状态可重启。生产部署的关键:RF ≥ 3 的 internal topics、Incremental Cooperative Rebalance、Source EOS + Sink 幂等、DLQ + errors.tolerance。下一章我们讲 Connect 的"近亲" Kafka Streams — 它在 Kafka 之上跑流处理(state store / window / KTable / 拓扑)。
