第 3 章 · 消息格式 V0 至 V2

三代演进 · RecordBatch 头部 · varint + ZigZag · CRC-32C · attributes 位域 · headers · down-conversion · dump-log 取证

Kafka 的"消息格式"听起来像协议文档里的字节排列细节,但它其实是一条贯穿性能、兼容、运维的核心边界。它决定 broker 能不能跳过用户态把字节零拷贝发出去,决定批量消息能不能共享一次 CRC 校验,决定老 consumer 拉新数据时 broker 要不要付出几倍 CPU 的"翻译税",还决定你 dump-log 时看到的字段名能不能准确对应一段日志故障。这一章把 V0 / V1 / V2 三代格式逐字节摊开,把"为什么 V2 是今天唯一的现实选项"说透。

讨论以 Apache Kafka 3.7 的 V2 格式(magic=2)为主——它从 0.11 引入,十年间经受了幂等 producer、事务、headers、KRaft、tiered storage 一系列重大特性的考验,几乎所有现代功能都依赖它的批次结构与字段位。V0 与 V1 仅作历史参照,实际生产里如果你还看到 magic=1,那一定是有客户端没升或者有 broker 配了 log.message.format.version=0.10——两种情况都是事故待发的信号。本章后半段的 down-conversion 与陷阱章节会反复回到这一点。

  • V0 / V1 / V2 三代字节布局对比,以及 V2 把"单条信封"改成"批次共享头部"的根本原因
  • RecordBatch 61 字节固定头部的每一个字段的语义、字节偏移、是否在 CRC 范围内
  • Record 内部用 varint + ZigZag 编码 offsetDelta / timestampDelta 的算法细节,以及为何批内连续场景每条只占 1-2 字节
  • CRC-32C 为什么不覆盖 baseOffset:让 broker 在分配 offset 时不必重算每条 CRC
  • kafka-dump-log 输出每个字段的解读、伪 hexdump 字节切片对照、CRC 校验范围可视化
  • ProducerRecord → MemoryRecordsBuilder → ProduceRequest → UnifiedLog 的字段映射全链路
  • linger.ms 与 batch.size 如何决定 V2 头部均摊 — 为什么 linger.ms=0 让 V2 退化为接近 V1 的逐条结构
  • 16 位 attributes 位域:压缩 codec(bit 0-2)、timestampType(bit 3)、transactional(bit 4)、control batch(bit 5)、delete horizon(bit 6)
  • 批次压缩在 broker 端"不解压、整批落盘"的设计,以及 zstd / lz4 / snappy / gzip 的速度与压缩比对照
  • Headers 字段的字节结构与编码;为什么 header 没有 schema、必须由平台层强制类型一致
  • Down-conversion 的代价:V2 → V1/V0 转换会丢失幂等、事务、headers 与时间检索语义,broker CPU 与 P99 显著上升
  • 消除 down-conversion 的检查清单:broker 配置、topic 覆盖、客户端版本、JMX 指标、fetch CPU 五维度
  • 九大常见陷阱(仅看 consumer 输出 / linger=0 / 老 consumer 触发转换 / header 类型不一 / control batch 误读 等)与排障护栏指标

前置知识:第 1 章 Kafka 数据流与 partition 模型;第 2 章 broker 架构、UnifiedLog、segment 文件命名。本章是"日志格式"维度的精解,后续第 4 章索引系统会基于本章定义的 baseOffset / firstTimestamp 字段讨论 .index / .timeindex 如何稀疏映射,第 7 章幂等与事务会基于本章的 producerId / baseSequence / control batch 解释端到端语义。

3.1 三代格式演进

消息格式的三代演进不是"多加几个字段"的版本号递增,而是三次重新思考"消息在网络与磁盘上应该长什么样"。每一次演进都对应一类生产痛点的缓解。

V2 RecordBatch Header 之后:
  records[] - 每条 record 使用 varint + delta 编码

V2 的 batch header 看似比 V0/V1 的单条信封更大(61 字节 vs 26-34 字节),但该开销由批次内所有 record 共担。当批次包含 100 条 record 时,header 均摊到每条仅 0.6 字节,显著低于 V0/V1 的 26-34 字节。这种"集中头部 + 共享元数据"的思想后面在第 4 章的稀疏索引、第 5 章的事务 control batch 都会反复出现 — 它是 Kafka 在小消息高频场景保持高吞吐的核心抽象。

每一代解决了什么、留下了什么

小消息元数据占比的实算

图 3-1-3 · 100 字节业务 value 的格式开销对比
单条 100 字节业务 value:

V0 单条开销:
  offset(8) + msgSize(4) + crc(4) + magic(1) + attributes(1)
  + keyLen(4) + valueLen(4) = 26 字节
  开销占比 = 26 / (26 + 100) = 20.6%

V1 单条开销:
  V0 字段 + timestamp(8) = 34 字节
  开销占比 = 34 / (34 + 100) = 25.4%

V2 单批 100 条:
  batch header(61) + 100 × (record overhead ≈ 4 字节 varint)
  = 61 + 400 = 461 字节
  数据总计 = 100 × 100 = 10000 字节
  开销占比 = 461 / (461 + 10000) = 4.4%

V2 在批次粒度足够大时,元数据占比可降至 V0 的 1/5 左右。这种"每加一条 record 边际开销骤降"的特性,是后续 producer 调优为什么必谈 linger.ms + batch.size 的根因 — 跑出 V2 真正性能的前提是让批次有足够的 record。

生产里 V0 / V1 真的还存在吗 — 一个还在跑 0.10 broker 的机房故事。某金融客户的"接入网关日志"在 2023 年还跑着 0.10.2.1 broker,topic 配置 message.format.version=0.10,producer 用的是 0.11+ 客户端 — 也就是说 producer 发出的是 V2 格式,broker 接收时反向 down-convert 成 V1 才落盘。事故现象是 broker CPU 持续 60-80%、fetch P99 飙到秒级。最后排查发现:这台 broker 既没 zstd(V0/V1 不支持),也没幂等保护,单核 CRC 重算把 SSE4.2 都打不满 — 历史包袱不只是"少几个字段",而是整套现代特性都用不了。修复路径是先升 broker 到 2.x、移除 message.format.version、平滑滚动重启,P99 立刻回到 50ms。结论:如果你今天的 magic 不是 2,优先级最高的工程就是把它升上来。

3.2 V2 RecordBatch 结构

把 V2 RecordBatch 拆开看,它是"61 字节固定头部 + 变长 records 序列"。头部里每个字段都有一个具体的工程理由,不是"觉得需要就加"。本节按字段、按 CRC 范围、按"为什么 baseOffset 在 CRC 之外"逐项展开。

字段全景图

batchLength4 字节跳到下一个 batch 的字节长度

为什么 CRC 不覆盖 baseOffset

图 3-2-2 · CRC 不覆盖 baseOffset 的工程原因(producer 与 broker 时序)

V0/V1 中,offset 写在 message 信封内并在 CRC 范围内,broker 重写 offset 时必须重算每条 CRC。V2 把 baseOffset 移出 CRC 范围,broker 仅需修改 8 字节即可完成 offset 分配 — 这一项设计在生产里直接体现为 broker write 路径的 CPU 占比从两位数降到单位数。配合 SSE4.2 硬件 CRC32C 指令(CRC-32C 校验本身可达 10 GiB/s),整个 append 路径在大多数场景里被 disk write 限制,而非 CPU 限制。

绝对值规则:
  absoluteOffset = baseOffset + offsetDelta
  absoluteTimestamp = firstTimestamp + timestampDelta

Record 内部不保存绝对值,完全依赖 batch header 提供的基准。该设计让批内连续的 offset 与时间戳都能压缩到 1-2 字节 varint — 接下来 §3.3 详细展开 varint 的算法,以及为什么 ZigZag 编码对负 delta 仍然友好。

3.3 Record 的 varint 编码

Record 是批内的"普通乘客",所有字段都用变长 varint 编码:length / attributes / timestampDelta / offsetDelta / keyLength / key / valueLength / value / headersCount / headers。其中 timestampDelta 与 offsetDelta 还经过 ZigZag 编码 — 因为它们是有符号 delta(虽然连续场景下永远 ≥ 0,但格式必须支持乱序)。

Record 的字段顺序

图 3-3-1 · Record 字段顺序(批内每条 record 的字节布局)

varint + ZigZag 的算法本质

图 3-3-2 · varint 与 ZigZag 编码规则
varint 规则:
  每字节使用低 7 位存储数据,最高位 1 表示 "还有后续字节",0 表示结束
  仅适用于非负整数

ZigZag 规则(用于有符号 delta):
  zigzag(n) = (n << 1) ^ (n >> 31)   for int32
  正数: zigzag(0)=0, zigzag(1)=2, zigzag(2)=4
  负数: zigzag(-1)=1, zigzag(-2)=3
  → 负数也能压到小字节数,而不是用 4-byte 补码

示例:
  零             → varint 1 字节: 00
  delta = 1      → ZigZag → 2  → varint: 02
  delta = 12     → ZigZag → 24 → varint: 18
  delta = 25     → ZigZag → 50 → varint: 32
  delta = 127    → ZigZag → 254 → varint 2 字节: fe 01
  delta = 128    → ZigZag → 256 → varint 2 字节: 80 02

ZigZag + varint 在批内连续 offset 与时间戳场景下,通常每条仅占 1 字节。批次越大,这种压缩效率越显著。这一点和 Protocol Buffers 的字段编码思想完全一致,Kafka 当年设计 V2 时直接借鉴了 protobuf 的 varint 实现 — 这也是"大多数二进制协议趋同"的一个例子。

实测:批内 100 条 record 的字节占用

图 3-3-3 · 批内 100 条 record 的 delta 字节分布

  压缩比约 6 倍

varint 的"代价"是无法直接 random-access — 但 Kafka 不需要。protobuf 当年也面临这个权衡:用 varint 节省字节,但读取一条 record 必须从头扫到尾才能找到第 N 个字段的偏移,无法像 fixed-size 那样 O(1) 跳转。Kafka 完全接受这个代价 — 因为 consumer 永远是顺序消费一个 batch 内的所有 record(不会跳读),broker 也永远是顺序 transferTo 整个 batch(也不会逐 record 操作)。只有 batch header 是定长 61 字节、可以 O(1) 解析,这刚好满足 broker 在 leader 写入路径上"先校验头部、再扫一眼 records"的需求。这种"内层用 varint 省字节、外层用定长支持快速跳转"的双层设计,在很多高效二进制协议里反复出现 — Lucene 倒排链、HBase HFile、protobuf wire format 都是同一思路。

批次粒度的压缩边界

图 3-6-3 · 批次压缩在 producer / broker / consumer 三端的责任分布

V2 压缩发生在批次粒度,batch header 与 records 字节序列分别处理。batch header 不压缩(broker 需快速解析 baseOffset / producerId 等元数据),records 字节序列整体压缩后存放。该设计的两大收益:(1) 相似消息共享压缩字典,压缩比通常比逐条压缩高 1.5-3 倍;(2) broker 完全不需要解压就能完成 append 与 fetch — 配合 sendfile 零拷贝,broker 几乎不付出压缩 CPU 成本,只让 producer 与 consumer 两端付出。

压缩算法对照表

算法压缩比压缩速度解压速度CPU 占用适用场景
none1.0x000%已压缩数据(图片 / 视频)

zstd 是 2024+ 的默认推荐,但要确认 client / broker 都 ≥ 2.1。zstd 在 Kafka 2.1+ 提供原生支持,压缩比接近 gzip,速度接近 lz4。但有一个常见陷阱:topic 设了 compression.type=zstd,然后老 producer(< 2.1)发数据进来时会直接报 UnsupportedCompressionTypeException。生产升级路径是先把 broker 全部升到 2.1+,再升 producer,最后才把 topic 默认 codec 切到 zstd — 反过来会引发 producer 层 fail-fast。如果你接管的是一套 Kafka 1.x 的旧集群,先把 codec 留在 lz4(2.x 之前广泛支持),等 broker 升级完成后再切 zstd。

3.7 Headers 与元数据规范

Headers 是 V2 引入的"消息侧带 metadata 通道"。它解决一个核心痛点:trace、schema、retry count 这些路由 / 审计 / 中间件信号不应该耦合在业务 value 里。Headers 让中间件直接读 metadata 而不解析业务 payload,但它的弱约束(无 schema)也带来另一类陷阱。

headers vs 把 metadata 塞 value 的对比

反例:把 trace / retry / schema 塞 value

String json = mapper.writeValueAsString(Map.of(
    "traceId",   ctx.traceId(),
    "schema",    "order-created-v3",
    "retry",     ctx.retry(),
    "payload",   businessEvent
));
producer.send(new ProducerRecord<>(
    "order-created", orderId, json));

Consumer 必须先反序列化整个 JSON 才能看 traceId。
路由 / 审计 / DLQ 中间件被迫理解业务 schema。
schema 升级时所有 metadata 抽取代码同步改。

正例:用 headers 通道

ProducerRecord<String, byte[]> rec =
    new ProducerRecord<>("order-created", orderId, payloadBytes);
rec.headers()
   .add("traceId", ctx.traceId().getBytes(UTF_8))
   .add("schema",  "order-created-v3".getBytes(UTF_8))
   .add("retry",   ByteBuffer.allocate(4)
            .putInt(ctx.retry()).array());
producer.send(rec);

Trace 中间件 record.headers().lastHeader("traceId") 即可。
业务 value 只装业务事件,schema 升级互不影响。
DLQ / 审计组件不需要懂业务格式。

Header 类型不一致的真实风险

图 3-7-4 · 同名 header 不同编码的解析灾难
Producer A:
  header.put("retry", "3")  → bytes: 33 (字符 '3')

Producer B:
  header.put("retry", new byte[]{0x03}) → bytes: 03 (单字节整数)

Consumer 1 期望字符串:
  String s = new String(header.value(), UTF_8);
  s = "3"  ✓ 来自 A
  s = "\u0003"  ✗ 来自 B,误读为 "Ctrl-C"

Consumer 2 期望字节:
  int retry = header.value()[0];
  retry = 51 ('3' 的 ASCII)  ✗ 来自 A,数值错误
  retry = 3  ✓ 来自 B
Header 没有 schema 系统,所有约定由应用平台层强制。Kafka 协议层只承诺 headers 是 List<Header> 其中 Header = (String key, byte[] value),完全不约束 value 的编码。这意味着两个 producer 写同名 header 不同编码,broker 不会拒绝,consumer 拿到的就是字节流自己解码 — 一旦解码方式假设错了,整个下游就漂移。生产级建议:为每个 topic 维护 header 白名单与编码规范,新增 header 走变更流程,新增前两个版本同时支持新旧 header 名称,稳定后再淘汰旧名。这种规范化的工作通常由平台团队做"infra 治理"承担,业务团队按 SDK 调用即可。

3.8 Down-conversion 兼容路径

Down-conversion 是 Kafka 给"老 client + 新 broker"的兼容补丁。它的存在让你不必"全量同步升级",但代价远比想象中重。这一节把代价摊开,并给出"什么时候必须消除 down-conversion"的硬指标。

Down-conversion 在指标上的真实代价

图 3-8-3 · 不同比例老 consumer 下的 broker 资源消耗
基线(无 down-conversion):
  Fetch P99 latency = 8 ms
  Broker CPU usage  = 25%
  BytesOut/sec     = 800 MiB/s

10% 客户端为 V1(触发 down-conversion):
  Fetch P99 latency = 40 ms (5× 增长)
  Broker CPU usage  = 65% (2.6× 增长)

50% 客户端为 V1:
  Fetch P99 latency = 200+ ms
  Broker CPU usage  = 90%+

down-conversion 是兼容性边界而非长期模式。生产环境应在客户端升级完成后立即移除 log.message.format.version 的旧版本固定。10% 老 consumer 看起来无伤大雅,但已经能让 broker CPU 翻倍 — 这是大多数团队在"客户端升完 90%"后还放任 down-conversion 的常见误判。

3.9 设计取舍

V2 不是凭空产生的"完美方案",而是 Kafka 团队在多个维度做了具体取舍后的工程结论。把这些取舍逐个看清楚,你就会理解为什么有些"看起来更优"的方案 Kafka 没选 — 那些方案在某个隐含维度上都更糟。

V2 设计取舍矩阵

设计点Kafka 选择拒绝的替代方案影响
offset 表达batch baseOffset + record delta每条绝对 offset减少 per-record 字节,append 时不重算 CRC
时间表达batch firstTimestamp + delta每条绝对 timestamp批内时间差小,varint 高效
校验粒度batch CRC-32C每条 CRCbroker 批量校验,CPU 显著降低
压缩粒度batch每条相似消息共享字典,压缩比提升 1.5-3 倍
CRC 范围不含 baseOffset含 baseOffsetbroker append 时无需重算 CRC
元数据扩展record headers塞入 value中间件可读元数据而不解析业务
兼容旧客户端fetch 时 down-conversion强制全量同步升级降低升级门槛但有 CPU 代价

V2 在不同场景下的边际收益

图 3-9-1 · 三类典型场景下 V2 的收益

  V1:每条 34 字节开销,带宽占用 134 字节/条 → 13.4 MiB/s
  V2:批 100 条,header 0.6 字节/条,带宽 100.6 字节/条 → 10.06 MiB/s

  V1:每条 34 字节,占比 0.3%

  节省带宽 < 0.3%(几乎无差异)

场景 C:跨区复制 + 网络瓶颈:
  V2 + zstd 压缩比 V1 + gzip 通常节省 30-50% 带宽
  显著降低跨区流量成本

V2 引入的新限制

图 3-9-2 · V2 优点对应的隐性代价
V2 不是免费午餐:幂等 producer 的 ProducerStateManager 需要 broker 维护 producerId → sequence cache。每个活跃 producerId 在 broker 端占用约几百字节 + 周期性 snapshot。生产里偶尔会遇到"客户端用了大量短生命周期 producer(每个请求 new 一个 KafkaProducer)",broker 端 producerId 数量飙升到百万级,内存被吃光。这种场景的修复不是限制 V2,而是教育业务方用常驻 producer 池而不是按请求 new。Kafka 3.0+ 也加了 producer.id.expiration.ms 默认 24h 来回收无活动的 producerId,缓解这个问题。

control batch 误读是自研 consumer 最常见的事故。事务 producer 在 commit / abort 时,broker 会写入一种特殊 batch:attributes bit 4=1 (transactional) + bit 5=1 (control),records 内容是 commit / abort marker 而不是业务数据。Kafka 官方 client 会自动跳过这些 batch,但很多团队基于 Kafka 协议自研了"轻量 consumer"(C / Go / Rust),如果忘了实现 isControl 的判断,就会把 commit marker 反序列化为业务消息,通常表现为"偶尔反序列化抛异常"或者"莫名其妙多了一些不知道从哪来的事件"。修复:在解析每个 batch 时第一步就检查 attributes bit 5,等于 1 直接跳到下一个 batch。

V2 把单条信封变成批次共享RecordBatch 61 字节 headerheader 越大,均摊越关键
