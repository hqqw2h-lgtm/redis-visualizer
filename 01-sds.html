第 1 章 · SDS 动态字符串

Redis 为什么不用裸 C 字符串 · header 如何藏在 char* 前面 · len / alloc 如何解决长度和追加

Redis 里到处都是字符串:key 是字符串,很多 value 也是字符串。问题是,C 原生字符串只给你一个起点指针,它不知道自己有多长,也不知道后面还剩多少空间。

所以每次求长度,C 字符串都要一路扫到 \0;如果内容里本来就有 0x00,很多字符串函数会提前停;如果要追加内容,还得调用者自己保证空间够。Redis 不能把这些成本和风险放在高频读写路径里。

SDS(Simple Dynamic String)的做法很直接:数据还是从 char * 指向的位置开始,但在数据前面藏一个 header,记录 len、alloc、flags。这样外面看起来仍然像 C 字符串,Redis 内部却能 O(1) 拿长度、保存任意字节,并且知道什么时候该扩容。

先抓住一句话:SDS = 兼容 char * 的外壳 + 可管理长度和容量的 header。后面所有 header 分级、预分配、embstr/raw、MEMORY USAGE 诊断,都是从这句话展开。

  • 长度从哪来:C 字符串靠扫描 \0,SDS 直接读 header.len。
  • 容量谁来管:C 字符串把空间管理留给调用者,SDS 用 alloc 记录剩余空间并按规则扩容。
  • 为什么还像 C 字符串:SDS 指针仍指向数据区 buf,所以 printf("%s", s) 这类代码还能直接用。

如果你熟悉 char *、结构体内存布局和 malloc/realloc,可以直接看图;如果不熟,先记住“指针指向数据,header 藏在前面”即可。

1.1 C 字符串的三处约束

C 字符串的规则很朴素:从指针开始往后读,读到第一个 0x00 就认为字符串结束。这个规则足够简单,但它也意味着两件事:长度要扫出来,容量没人替你记。Redis 的 key 和 value 会被频繁读写、追加、复制,还可能保存图片头、压缩包、序列化结果这类二进制数据,所以这两个"没人记"会变成真实成本。

1.2 SDS 结构定义与字节布局

现在把 SDS 摊到内存里看。读图时先不要急着记 flags 的每一位,只抓住一个位置关系:外部拿到的 sds 指针指向 buf[0],而 len/alloc/flags 藏在它前面。Redis 内部向前退几字节就能读 header,外部 C 代码仍把它当普通 char * 用。

图 1-2-2 · sdshdr8 字节级布局(存储 "hello")

__attribute__((__packed__)) 禁止编译器为对齐插入 padding。对千万级 key 场景,每个 key 节省 5 字节,1000 万 key 总共节省约 47 MiB:

typedef char *sds;

struct __attribute__ ((__packed__)) sdshdr8 {
    uint8_t len;          /* used */
    uint8_t alloc;        /* excluding the header and null terminator */
    unsigned char flags;  /* 3 lsb of type, 5 unused bits */
    char buf[];
};

1.3 长度查询的 O(1) 实现

C 字符串的 strlen 是 O(n);Redis 把这个常见操作降到 O(1) — 不是靠缓存,而是靠在结构里直接存了一个 len 字段。这一节把 STRLEN 命令从客户端到 SDS 内部的完整时序拆开,再讲 sdslen() 如何根据 flags 分派到不同 header。

图 1-3-1 · STRLEN 命令调用时序(Client → 命令层 → object → SDS header)
图 1-3-2 · sdslen 的类型分派分支(读 flags → switch 到对应 header → 返回 len)

源码实现:

static inline size_t sdslen(const sds s) {
    unsigned char flags = s[-1];
    switch(flags & SDS_TYPE_MASK) {
        case SDS_TYPE_5:
            return SDS_TYPE_5_LEN(flags);
        case SDS_TYPE_8:
            return SDS_HDR(8, s)->len;
        case SDS_TYPE_16:
            return SDS_HDR(16, s)->len;
        case SDS_TYPE_32:
            return SDS_HDR(32, s)->len;
        case SDS_TYPE_64:
            return SDS_HDR(64, s)->len;
    }
    return 0;
}

APPEND 命令的返回值由 sdssetlen() 写入新长度后直接读取,无需扫描 value:

127.0.0.1:6379> SET sds:len "hello world"
OK
127.0.0.1:6379> STRLEN sds:len
(integer) 11
127.0.0.1:6379> APPEND sds:len "!"
(integer) 12
127.0.0.1:6379> STRLEN sds:len
(integer) 12
无论字符串大小为 5 字节或 5 MiB,STRLEN 时间复杂度恒为 O(1)。这是 Redis 把 C 字符串改造为 SDS 的第一项核心收益。

1.4 五种 header 类型设计

所有 SDS 都以"header + buf"的方式排布,但 header 的大小是动态的:1 字节 / 3 字节 / 5 字节 / 9 字节 / 17 字节五个档位,按字符串长度选择最小够用的那个。这是 Redis 在"千万级小 key"场景下榨取内存的关键设计。

类型选择的源码逻辑:

char sdsReqType(size_t string_size) {
    if (string_size < (1 << 5)) return SDS_TYPE_5;
    if (string_size < (1 << 8)) return SDS_TYPE_8;
    if (string_size < (1 << 16)) return SDS_TYPE_16;
    if (string_size < ((long long)1 << 32)) return SDS_TYPE_32;
    return SDS_TYPE_64;
}
sdsReqType(5)       -> SDS_TYPE_5
sdsReqType(200)     -> SDS_TYPE_8
sdsReqType(70000)   -> SDS_TYPE_32
embstr("hello")     -> 对象层强制 sdshdr8(非 sdshdr5)
对象层 embstr 编码强制使用 sdshdr8,不回退到 sdshdr5。原因:sdshdr5 无 alloc 字段,无法支持后续扩容;embstr 虽设计为不可变,仍需保留 alloc 字段供 sdsavail() 等内部接口查询。

1.5 容量管理与预分配策略

SDS 解决"长度查询"是直接存字段;解决"追加成本"则要复杂得多——既不能每次 realloc(O(n²) 累积),也不能预留太多(浪费内存)。Redis 的答案是两阶段策略:小串(< 1 MiB)双倍预留,大串(≥ 1 MiB)线性 +1 MiB。

源码核心(抽掉错误处理后的主干):

sds sdsMakeRoomFor(sds s, size_t addlen) {
    size_t avail = sdsavail(s);
    size_t len = sdslen(s);
    size_t newlen = len + addlen;

    if (avail >= addlen) return s;

    if (newlen < SDS_MAX_PREALLOC)
        newlen *= 2;
    else
        newlen += SDS_MAX_PREALLOC;

    return sdsResize(s, newlen);
}

1.6 惰性释放策略

1.7 二进制安全语义

SDS 的"二进制安全"不是某个新增特性,而是边界不再来自数据流带来的副作用:既然 header.len 已经显式记录了"用了多少字节",数据区里出现 \0 也不再是终止信号 — 它就是普普通通的一个字节,和其他字节一样。

图 1-7-1 · 二进制安全的字节边界来源
图 1-7-2 · 显式长度写入流程(以 Redis String 短值 / sdshdr8 为例)

源码(概念化摘录;Redis 7.x 实现名为 _sdsnewlen,还包含溢出检查、trymalloc 分支和 allocator usable size 处理):

sds sdsnewlen(const void *init, size_t initlen) {
    void *sh;
    sds s;
    char type = sdsReqType(initlen);
    if (type == SDS_TYPE_5 && initlen == 0) type = SDS_TYPE_8;
    int hdrlen = sdsHdrSize(type);

    sh = s_malloc(hdrlen + initlen + 1);
    s = (char*)sh + hdrlen;
    if (initlen && init) memcpy(s, init, initlen);
    s[initlen] = '\0';
    if (type == SDS_TYPE_5) {
        s[-1] = type | (initlen << SDS_TYPE_BITS);
    } else {
        s[-1] = type;
        sdssetlen(s, initlen);
        sdssetalloc(s, initlen);
    }
    return s;
}
sdshdr5 与 Redis String 对象不要混在一起:独立 SDS 库对很短的非空字符串可能选择 SDS_TYPE_5;但 Redis 对象层的 embstr 短字符串使用 sdshdr8,因为对象层仍需要 alloc 字段和统一的后续处理路径。

Python 客户端的存取一致性验证:

import redis

r = redis.Redis()
payload = bytes([0xff, 0xd8, 0xff, 0x00, 0x10, 0x4a, 0x46])
r.set("sds:jpeg-head", payload)

got = r.get("sds:jpeg-head")
print(len(got))
print(got == payload)
print(got.hex(" "))
7
True
ff d8 ff 00 10 4a 46
"二进制安全"的本质 = 数据完整性的保证条件由"数据流中无 0x00"放宽到"调用者传入正确的 initlen"。只要客户端 SDK 在写入和读取两端都用 (ptr, len) 二元组,JPEG 头、protobuf 序列化结果、加密 blob 等任意字节序列都能完整保留。

        continue

    memory = r.memory_usage(key) or 0
    ratio = memory / max(strlen, 1)
    if strlen > 1024 * 1024 or ratio > 4:
        print(key, "strlen=", strlen, "memory=", memory, "ratio=", round(ratio, 2))

1.11.1 设计取舍

取舍 1:SDS 指针指向 buf 而非 header

将 sds 类型设为 char *,指针指向数据区,使 SDS 在与 C 标准库交互时无需修改,代价是访问 header 需要 s[-1] 的指针运算。这一选择在工程总成本上更优:Redis 内部已有数百处 printf("%s", ...) 和 strncpy 调用,若指针指向 header,所有这些调用都需要包装。把"反向访问 header"这条偏离主路径的代价收在少数几个 inline 函数里(sdslen / sdsavail),换取主路径无修改成本,是典型的"局部代价换全局简化"。

取舍 2:五种 header 而非单一大 header

按容量分级 header 的设计动机是节省千万级 key 场景下的元数据开销。极端情况下(全部为 5 字节 key + 5 字节 value),每个 key 节省 6 字节,1000 万 key 节省 57 MiB。代价是 sdslen() 等接口需要 switch 类型分派,引入 5-7 个 CPU 周期 — 对一个 inline 调用而言可接受。统一 64 位 header 的设计内存放大约 2-3 倍,在小 value 场景下不可接受。

取舍 3:预分配双倍 vs +1 MiB 两阶段

阈值设为 1 MiB 的依据是分布式系统中"小到中等字符串频繁扩容、超大字符串扩容稀疏"的经验观察。对 100 KiB 字符串,双倍预分配浪费 100 KiB,占总占用 50%——该量级可接受;对 1 GiB 字符串,双倍预分配浪费 1 GiB,占内存 50%——不可接受。1 MiB 阈值在两种模式间提供平滑过渡。

取舍 4:embstr 不可变 + 单向转 raw

embstr 设计为不可变的核心动机是让短字符串读路径更简单:redisObject 与 SDS 在同一内存块中,一次分配、局部性好。只要发生容量变化,整块都要重新分配;既然已经要重建,直接转成更适合后续修改的 raw,比继续维持 embstr 更省判断。raw 是修改路径上的终态,无回退路径,这条单向边看起来“少了一条边”,实际是在简化所有关于“是否可降级”的判断。
