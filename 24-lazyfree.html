第 24 章 · Lazy Free 与碎片整理

UNLINK · bio_thread[2] · LAZYFREE_THRESHOLD = 64 · Active Defrag · jemalloc defrag_hint · 主线程时间片切片

Redis 是单线程命令处理模型,所有命令在主线程顺序执行。该模型的核心约束是命令路径上的任何同步操作都直接转化为客户端延迟。删除一个 500 万元素的 hashtable 需要遍历释放每个 entry,主线程被占用 1-2 秒,期间所有客户端命令被阻塞。该现象在 Redis 4.0 之前是单线程模型的固有限制。

Redis 4.0 引入两项异步化机制以解决该限制。其一,Lazy Free 将大对象的内存释放从主线程卸载到 bio 后台线程,主线程仅承担索引摘除的 O(1) 开销;其二,Active Defrag 在事件循环空隙渐进式整理 jemalloc 碎片,以"主线程时间片切片"形式控制对命令延迟的影响。两者共同将"释放即阻塞"与"长时间运行碎片累积"这两类工程问题从单线程模型中剥离。

  • DEL 大 key 阻塞主线程的实测延迟(1M 元素 hashtable ~ 480 ms)与 Sentinel 误判 failover 的连锁反应
  • bio 三线程架构(BIO_CLOSE_FILE / BIO_AOF_FSYNC / BIO_LAZY_FREE)的任务提交与消费
  • UNLINK 的两阶段:主线程 dictUnlink(O(1))+ bio 线程 decrRefCount
  • lazyfreeGetFreeEffort 估算逻辑与 LAZYFREE_THRESHOLD = 64 阈值的工程依据
  • 5 个 lazyfree-lazy-* 配置项默认 no,升级 4.0+ 后必须显式开启
  • DEL vs UNLINK 语义差异:DEL 默认同步,UNLINK 始终走 dbAsyncDelete
  • jemalloc size class 与 slab 空洞:碎片产生的根本原因
  • mem_fragmentation_ratio 解读:> 1.5 触发,< 1.0 警告 swap
  • Active Defrag 主线程时间片切片(cycle-min/max)与 jemalloc defrag_hint
  • 主从复制陷阱:复制流传播 DEL 不是 UNLINK,Replica 必须独立配置 lazyfree
  • MIGRATE 仍同步释放,Cluster 大 key Slot 迁移的工程考量

图 24-2-1 · bio 三线程架构

bio (Background I/O) 是 Redis 长期存在的后台线程子系统,Redis 4.0 之前已用于 close fd 与 AOF fsync。Redis 4.0 新增 BIO_LAZY_FREE 类型,复用同一线程框架处理异步释放。三线程互不干扰,任意类型积压不影响其他类型处理。

void bioCreateBackgroundJob(int type, void *arg1, void *arg2, void *arg3) {
    struct bio_job *job = zmalloc(sizeof(*job));
    job->arg1 = arg1; job->arg2 = arg2; job->arg3 = arg3;

    pthread_mutex_lock(&bio_mutex[type]);
    listAddNodeTail(bio_jobs[type], job);
    bio_pending[type]++;
    pthread_cond_signal(&bio_newjob_cond[type]);
    pthread_mutex_unlock(&bio_mutex[type]);
}

void *bioProcessBackgroundJobs(void *arg) {
    int type = (intptr_t)arg;
    pthread_mutex_lock(&bio_mutex[type]);
    while (1) {
        if (listLength(bio_jobs[type]) == 0) {
            pthread_cond_wait(&bio_newjob_cond[type], &bio_mutex[type]);
            continue;
        }
        listNode *ln = listFirst(bio_jobs[type]);
        struct bio_job *job = ln->value;
        listDelNode(bio_jobs[type], ln);
        pthread_mutex_unlock(&bio_mutex[type]);

        if (type == BIO_CLOSE_FILE)      close((int)(long)job->arg1);
        else if (type == BIO_AOF_FSYNC)  redis_fsync((int)(long)job->arg1);
        else if (type == BIO_LAZY_FREE)  lazyfreeFreeObject(job->arg1);

        zfree(job);
        pthread_mutex_lock(&bio_mutex[type]);
        bio_pending[type]--;
    }
}

24.4 UNLINK 完整流程

图 24-4-1 · UNLINK 的四步流程
/* src/lazyfree.c */
size_t lazyfreeGetFreeEffort(robj *key, robj *obj, int dbid) {
    if (obj->type == OBJ_LIST &&
        obj->encoding == OBJ_ENCODING_QUICKLIST) {
        quicklist *ql = obj->ptr;
        return ql->len;  /* 节点数 */
    } else if (obj->type == OBJ_SET &&
               obj->encoding == OBJ_ENCODING_HT) {
        return dictSize((dict*)obj->ptr);
    } else if (obj->type == OBJ_ZSET &&
               obj->encoding == OBJ_ENCODING_SKIPLIST) {
        return ((zset*)obj->ptr)->zsl->length;
    } else if (obj->type == OBJ_HASH &&
               obj->encoding == OBJ_ENCODING_HT) {
        return dictSize((dict*)obj->ptr);
    } else if (obj->type == OBJ_STREAM) {
        size_t effort = 0;
        stream *s = obj->ptr;
        effort += s->groups ? raxSize(s->groups) : 0;
        effort += raxSize(s->rax);
        return effort;
    } else {
        return 1;  /* 标量 string、紧凑编码、小容器 */
    }
}

#define LAZYFREE_THRESHOLD 64
free effort 不是字节数,而是"释放复杂度估算"。listpack 即使占用 64 KB 内存,释放仍是 zfree() 单次系统调用,effort 计为 1;hashtable 包含 100 个 8 字节 entry 共 800 字节,但需要 100 次独立 free,effort 计为 100。
图 24-4-2 · 阈值 64 的工程依据

阈值 64 的推导

分析项同步释放异步 (bio) 释放
主线程耗时N × t_freet_bio (提交)
实测参数t_free ≈ 50-100 ns (jemalloc free 单元素)t_bio ≈ 3-5 μs (mutex lock + list 追加 + cond_signal + unlock)
收益条件主线程节省 N × t_free − t_bio > 0 ⇒ N > t_bio / t_free
阈值计算N > 5000 ns / 75 ns ≈ 67 → Redis 选 64 (2⁶, 略保守)
线程安全的根源:主线程与 bio 线程的并发安全通过"先脱离引用,再异步释放"的两阶段设计保证。任何时刻,某个 robj 要么在 keyspace 中可被命令访问,要么已脱离 keyspace 待 bio 释放,不存在共享状态。该设计避免了通用读写锁、RCU 宽限期等复杂并发原语。

24.6 Active Defragmentation

图 24-6-1 · Defrag 的核心思路
void activeDefragCycle(void) {
    if (!server.active_defrag_running) return;

    long long start = ustime();
    long long timelimit = computeDefragTimelimit();

    while (ustime() - start < timelimit) {
        defragKeysScan();  /* 扫描 keyspace */
        if (defrag_done) {
            server.active_defrag_running = 0;
            break;
        }
    }
}

void *activeDefragAlloc(void *ptr) {
    void *newptr;
    size_t size = zmalloc_size(ptr);
    /* MALLOCX_TCACHE_NONE bypass tcache 以填充已有 slab */
    newptr = mallocx(size, MALLOCX_TCACHE_NONE);
    memcpy(newptr, ptr, size);
    free(ptr);
    return newptr;
}
Defrag 的核心约束是"在主线程时间片内执行"。jemalloc 的 defrag_hint() 提示"该对象搬迁后能填充非满 slab 的空洞",hint 为真时调用 mallocx(MALLOCX_TCACHE_NONE) 重新分配,bypass tcache 以确保新分配填充已有 slab 空洞而非新 slab。

24.10 源码深潜

图 24-10-1 · 释放路径调用图
/* src/lazyfree.c: dbAsyncDelete 主入口 */
int dbAsyncDelete(redisDb *db, robj *key) {
    if (db->expires) dictDelete(db->expires, key->ptr);

    dictEntry *de = dictUnlink(db->dict, key->ptr);
    if (de) {
        robj *val = dictGetVal(de);
        size_t free_effort = lazyfreeGetFreeEffort(key, val, db->id);

        if (free_effort > LAZYFREE_THRESHOLD &&
            val->refcount == 1) {
            atomicIncr(lazyfree_objects, 1);
            bioCreateBackgroundJob(BIO_LAZY_FREE, val, NULL, NULL);
            dictSetVal(db->dict, de, NULL);
        }
        dictFreeUnlinkedEntry(db->dict, de);
        return 1;
    }
    return 0;
}

/* src/lazyfree.c: bio 线程释放 */
void lazyfreeFreeObject(void *args[]) {
    robj *o = (robj *)args[0];
    decrRefCount(o);
    atomicDecr(lazyfree_objects, 1);
}

24.10.1 设计取舍

取舍 1 · "脱离引用 + 异步释放" vs 通用并发控制:Redis 选择前者而非读写锁/RCU 宽限期的根本原因是单线程命令模型的简洁性。dictUnlink 后 keyspace 不再持有该 value 的引用,主线程任何后续命令都无法找到该对象,bio 线程的释放与命令路径不存在共享状态。
取舍 2 · LAZYFREE_THRESHOLD = 64 不通过配置暴露:修改需要权衡 t_bio 与 t_free 的实测值,普通运维难以正确判断。jemalloc 改进或 mutex 优化可能改变比值,但现代硬件上 64 仍是合理选择。
取舍 3 · Active Defrag 主线程时间片 vs 独立后台线程:Redis 对象内部持有大量引用(dict entry 的 key/value、skiplist node 的 ele/forward、quicklist 的 prev/next),搬迁后必须更新所有持有者的指针。独立线程搬迁需要在搬迁某对象时阻止主线程访问,这会引入读写锁,违背单线程命令路径模型。
取舍 4 · replica-lazy-flush yes 必备:psync 失败时 Replica 需要 FLUSHALL 清空本地数据后接收新 RDB,大数据集的 FLUSHALL 同步释放可能阻塞 Replica 主线程数十秒,导致复制超时,陷入死循环。

8mem_fragmentation_ratio = RSS / used_memory > 1.5 表示碎片严重,需启用 Active Defrag
9Active Defrag 在主线程事件循环时间片内渐进搬迁,cycle-min/cycle-max 控制 CPU 占用比
10Defrag 不能用独立线程因指针更新需串行化;Lazy Free 不消灭释放成本,仅卸载至 bio,内存释放总耗时不变
