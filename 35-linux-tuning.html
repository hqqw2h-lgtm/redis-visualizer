第 35 章 · Linux 内核调优

overcommit · THP · CoW 容量模型 · somaxconn / tcp-backlog · NUMA · cgroup v2 · swappiness · I/O scheduler · 闭环验证

Redis 主线程在用户态执行命令,但其性能与可用性大量依赖内核行为:fork() 系统调用是否被 overcommit 策略拒绝、Transparent Huge Pages 是否放大写时复制、somaxconn 是否截断 accept 队列、fdatasync 是否被 I/O 调度延迟、setrlimit 是否被 systemd 覆盖。这些边界决定了相同 Redis 二进制在不同宿主机上呈现差异显著的吞吐与尾延迟。

本章按内核子系统组织:内存(overcommit、THP、swappiness)、网络(somaxconn、tcp-backlog)、CPU(affinity、NUMA)、文件描述符(ulimit、systemd)、I/O 调度(块设备 scheduler)。每个参数与 Redis 源码路径绑定,并给出验证命令与发布阈值。

  • 为什么 Redis 的 fork()、CoW 和 THP 会把"内存够不够"变成内核问题
  • 怎样用容量模型而不是固定倍数评估 maxmemory、cgroup 与客户端缓冲
  • 网络 backlog、systemd nofile、NUMA 与 I/O scheduler 哪些值得调,哪些必须先压测
  • 每个调优项如何回到 Redis 指标与系统指标做闭环验证

前置知识:第 17 章 RDB(理解 fork + CoW);第 18 章 AOF(理解 fsync 三策略);第 22 章 jemalloc(理解 mem_fragmentation_ratio);第 23 章淘汰(理解 maxmemory 容量预算);Linux 内核基础(虚拟内存、page cache、cgroup v2、systemd、NUMA、块设备调度器)。读者最好亲手跑过 sysctl -a 并查过几个内核参数的含义。

核心思想: 调优不是"改一堆参数"。每个参数变更必须有明确的 Redis 指标 + 系统指标 + 发布阈值三联绑定,否则就是迷信调优。本章把"参数 → Redis 影响 → 验证命令 → 发布阈值"四要素全部对齐,让调优变成可审计的工程动作。

35.1 vm.overcommit_memory 与 fork 可靠性

Redis 的 RDB 与 AOF rewrite 通过 fork() 创建子进程,子进程继承父进程内存映射并通过 Copy-on-Write 共享物理页。Linux 内核的 overcommit 策略决定 fork() 是否被拒绝。

图 35-1-1 · fork 调用链

35.1.2 三种 overcommit 策略对比

表 35-1-2 · vm.overcommit_memory 选项
vm.overcommit_memory内核行为Redis 影响生产建议
0(默认启发式)估算可用内存,大进程 fork 可能被拒大数据集 BGSAVE 间歇性失败❌ 不推荐
1(始终允许)放宽 fork 准入,延迟到实际写入更不易被预先拒绝,但物理内存/cgroup 余量仍需保证✅ Redis 常见建议
2(严格记账)限制总虚拟内存 ≤ 物理内存 + swap × ratioRedis fork 经常被拒❌ 不推荐

启发式策略(0)在父进程 RSS 较大时可能将 fork 视为"需要再准备一份同等内存"而拒绝。Redis 利用 CoW 仅在写入时复制页面,启发式估算与实际行为不符。

$ redis-cli INFO persistence | grep rdb_last_bgsave_status

vm.overcommit_memory=1 解决"内核允许 fork",不解决"宿主机有足够物理内存"。后者需要单独的容量模型(见 §35.3)。

35.2 Transparent Huge Pages 与 CoW 放大

Linux Transparent Huge Pages(THP)将 4 KiB 页面合并为 2 MiB 大页。对随机访问、读多写少的工作负载有缓存命中收益。对 Redis 的 fork + 小写入场景,THP 反而放大 CoW 复制粒度。

35.2.1 CoW 复制粒度对比

图 35-2-1 · 4 KiB vs 2 MiB CoW

写入 1 字节即触发整页复制。THP 下每次写入复制 2 MiB,放大比相对 4 KiB 页为 512 倍。BGSAVE 期间父进程持续接收写命令,RSS 快速增长。

35.2.2 THP 三种状态行为对比

表 35-2-2 · enabled 配置选项
transparent_hugepage/enabled内核行为Redis 影响
always所有匿名映射均尝试合并大页CoW 放大严重
madvise仅显式建议的区域使用大页Redis 通常不主动请求,但仍应核验发行版与依赖库行为
never完全禁用CoW 粒度恒为 4 KiB

madvise 不是"一定会合并",但也不应只看配置名做假设。生产基线通常把 Redis 所在宿主机统一设为 never,并用 /sys/kernel/mm/transparent_hugepage/enabled 与 fork 延迟指标复核。

35.3 fork CoW 余量估算

35.3.3 容量公式

used_memory_gb = 32
write_dirty_rate = 0.08
client_buffer_gb = 2
repl_backlog_gb = 1
fragmentation_gb = used_memory_gb * 0.10
os_reserved_gb = 3

cow_gb = used_memory_gb * write_dirty_rate
required_free_gb = cow_gb + client_buffer_gb + repl_backlog_gb + fragmentation_gb + os_reserved_gb
print(f"CoW reserve     : {cow_gb:.1f} GiB")
print(f"Required free   : {required_free_gb:.1f} GiB")

# 输出:
# CoW reserve     : 2.6 GiB
# Required free   : 11.8 GiB

发布评审时,该数字与 free -g 的可用内存对比,若可用内存低于 Required free,则禁止启用 BGSAVE 或先调整数据规模。

35.4 somaxconn 与 tcp-backlog

Redis tcp-backlog 配置项作为 listen() 系统调用的 backlog 参数。内核通过 net.core.somaxconn 截断该值,通过 net.ipv4.tcp_max_syn_backlog 限制半连接队列。

35.4.1 连接队列与内核截断关系

图 35-4-1 · 双队列结构

35.4.2 三个 backlog 参数的关系

表 35-4-2 · backlog 参数全维度
参数层级默认值控制对象
net.ipv4.tcp_max_syn_backlog内核全局1024 - 8192半连接队列(SYN 已收到)
net.core.somaxconn内核全局128(旧)/4096(新)accept 队列上限
tcp-backlog(redis.conf)Redis 进程511listen() 请求值
核心截断规则: 实际 accept 队列长度为 min(tcp-backlog, somaxconn)。Redis 配置 tcp-backlog 511、内核默认 somaxconn=128 时,实际队列仅 128。仅改 redis.conf 无效。

35.5 CPU affinity 与 NUMA

Redis 命令执行主路径运行在主线程。进程在 CPU 核心间迁移会导致 L1/L2 cache 失效;在多 socket NUMA 服务器上,远端 NUMA 节点访问内存的延迟显著高于本地节点。

35.6 文件描述符与 cgroup

Redis maxclients 配置项的实际上限受限于进程文件描述符。Redis 还需额外 fd 给监听 socket、AOF 文件、RDB pipe、复制连接、集群总线。

35.7 Swap 与 I/O 调度

vm.swappiness 控制内核将匿名页交换到 swap 的倾向。Redis 单次内存访问从纳秒级被 swap 拉到毫秒级时,p99 延迟即抖动。

35.7.1 swappiness 取值语义

表 35-7-1 · swappiness 四档
取值行为Redis 建议
0尽量避免匿名页换出,具体语义随内核变化可选,需和 OOM 策略一起验证
1极少使用 swap常见保守值
60(默认)平衡 page cache 与匿名页❌ 不推荐
100倾向 swap❌ 不推荐

vm.swappiness 没有跨内核的唯一安全值。很多 Redis 基线会从 1 开始,但是否使用 0、是否禁用 swap,应结合 OOM 策略、cgroup、监控与故障演练决定。

35.9 核查脚本

将关键参数收集为发布前检查输出,生产环境先跑只读核查,再由变更单执行修改。

#!/usr/bin/env bash
set -euo pipefail

pid="$(pgrep -n redis-server || true)"

echo "== sysctl =="
sysctl vm.overcommit_memory vm.swappiness net.core.somaxconn net.ipv4.tcp_max_syn_backlog

echo "== thp =="
cat /sys/kernel/mm/transparent_hugepage/enabled

echo "== redis =="
redis-cli INFO persistence | grep -E 'rdb_last_bgsave_status|aof_last_bgrewrite_status'
redis-cli INFO stats | grep -E 'rejected_connections|instantaneous_ops_per_sec'

if [[ -n "$pid" ]]; then
    echo "== process limits =="
    grep 'Max open files' "/proc/$pid/limits"
    grep -i '^Swap:' "/proc/$pid/smaps_rollup"
fi

35.10.3 反例与正例对照

❌ 反例 · cgroup memory.max 等于 maxmemory

[Service]
MemoryMax=32G
# Redis maxmemory 32gb
# BGSAVE/AOF rewrite 触发 OOM kill

cgroup 边界与 Redis 数据集大小重合,CoW + buffer + 子进程内存无处可放,fork 期间必然被 cgroup OOM Killer 杀掉。

✅ 正例 · 按容量模型留余量

[Service]
MemoryHigh=42G
MemoryMax=48G
# Redis maxmemory 32gb
# 余量覆盖 CoW + buffer + 子进程

这里的 48G 是由写脏率、客户端缓冲、backlog、碎片和子进程开销估算得出;在别的实例上应重新建模并用 BGSAVE/AOF rewrite 压测验证。

35.11.3 被拒绝的"看似合理"方案

vm.overcommit_memory=2严格限制内存超卖Redis fork 频繁被拒,持久化可用性下降
THP 设为 madvise比 never 温和不能只凭配置名判断,仍需验证实际 THP 与 fork 指标
