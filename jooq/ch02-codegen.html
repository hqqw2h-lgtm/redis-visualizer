第 2 章 · 代码生成引擎

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

jOOQ 的类型安全在编译期就能拦截列名漂移与类型不一致,核心机制是构建期的代码生成管线。该管线把数据库 schema 投影为 Java 元模型,使列引用、外键关系、序列与函数都成为编译器可见的静态符号。第 1 章建立了 jOOQ 在三类抽象方案中的位置,本章展开这条管线的每个阶段:从元数据来源到 Tables / Records / POJOs / DAOs 输出,从 GeneratorStrategy / Matcher 命名定制到 CI 集成与 DDLDatabase 无连接生成。

构建期管线分为 7 个阶段:入口工具 → 元数据来源 → 元数据提取 → Definition 模型 → includes / excludes / Matcher 过滤 → GeneratorStrategy 命名 → JavaWriter 写源码。每个阶段的产物与失败信号都需要在 CI 流水线中可观测,否则 schema 漂移会延迟到运行期才暴露。本章按"管线总览 → 元数据 → 产物 → 命名 → CI → 陷阱 → 源码"组织。

2.1.3 入口配置示例

最常见的入口是 Maven 插件 jooq-codegen-maven。下例针对 PostgreSQL,生成路径锁定在 target/generated-sources/jooq,该路径默认进入 source root。

<plugin>
  <groupId>org.jooq</groupId>
  <artifactId>jooq-codegen-maven</artifactId>
  <version>3.19.10</version>
  <dependencies>
    <dependency>
      <groupId>org.postgresql</groupId>
      <artifactId>postgresql</artifactId>
      <version>${postgresql.version}</version>
    </dependency>
  </dependencies>
  <executions>
    <execution>
      <phase>generate-sources</phase>
      <goals><goal>generate</goal></goals>
      <configuration>
        <jdbc>
          <driver>org.postgresql.Driver</driver>
          <url>jdbc:postgresql://localhost:5432/mydb</url>
          <user>codegen_user</user>
          <password>${jooq.codegen.password}</password>
        </jdbc>
        <generator>
          <database>
            <name>org.jooq.meta.postgres.PostgresDatabase</name>
            <inputSchema>public</inputSchema>
          </database>
          <target>
            <packageName>com.example.jooq</packageName>
            <directory>target/generated-sources/jooq</directory>
          </target>
        </generator>
      </configuration>
    </execution>
  </executions>
</plugin>

2.2.3 INFORMATION_SCHEMA 直接抽样

直接对 INFORMATION_SCHEMA 抽样,可以审查 jOOQ 即将读取的实际数据。该步骤在排查"jOOQ 找不到列"时是必须的诊断动作。

SELECT table_name, column_name, data_type, is_nullable
FROM information_schema.columns
WHERE table_schema = 'public'
  AND table_name IN ('customers', 'orders')
ORDER BY table_name, ordinal_position;

table_name | column_name      | data_type         | is_nullable
-----------+------------------+-------------------+------------
customers  | id               | bigint            | NO
customers  | name             | character varying | NO
customers  | status           | character varying | NO
orders     | id               | bigint            | NO
orders     | customer_id      | bigint            | NO
orders     | total_amount     | numeric           | NO
orders     | created_at       | timestamp         | NO

2.2.4 方言差异

不同数据库暴露元数据的程度差异显著。MySQL 8.0.16 之前忽略 CHECK 约束;Oracle 使用 ALL_TAB_COLUMNS 而非标准 INFORMATION_SCHEMA;DB2 使用 SYSCAT。jOOQ 为每种方言实现独立 Database 类,以抹平这些差异。

方言 Database 实现 元数据特征
PostgreSQL PostgresDatabase 完整 INFORMATION_SCHEMA + pg_catalog 扩展
MySQL MySQLDatabase INFORMATION_SCHEMA 不全
Oracle OracleDatabase ALL_* 视图
SQL Server SQLServerDatabase sys.* 视图
H2 H2Database 标准 INFORMATION_SCHEMA
DB2 DB2Database SYSCAT.*

2.3.4 Table 类示例

TableImpl 子类是表的 Java 投影,每个列由 createField 生成,携带名称、类型、约束。getPrimaryKey 与 getReferences 提供主外键导航入口,这些方法在 onKey() 与 selectFrom().where().fetch() 等 DSL 中被运行期消费。

public class Orders extends TableImpl<OrdersRecord> {

    public static final Orders ORDERS = new Orders();

    public final TableField<OrdersRecord, Long> ID =
        createField(DSL.name("id"),
            SQLDataType.BIGINT.nullable(false).identity(true), this);

    public final TableField<OrdersRecord, Long> CUSTOMER_ID =
        createField(DSL.name("customer_id"),
            SQLDataType.BIGINT.nullable(false), this);

    public final TableField<OrdersRecord, BigDecimal> TOTAL_AMOUNT =
        createField(DSL.name("total_amount"),
            SQLDataType.NUMERIC.precision(12, 2).nullable(false), this);

    public final TableField<OrdersRecord, LocalDateTime> CREATED_AT =
        createField(DSL.name("created_at"),
            SQLDataType.LOCALDATETIME.nullable(false), this);

    @Override
    public UniqueKey<OrdersRecord> getPrimaryKey() {
        return Keys.ORDERS_PKEY;
    }

    @Override
    public List<ForeignKey<OrdersRecord, ?>> getReferences() {
        return List.of(Keys.ORDERS__FK_ORDERS_CUSTOMER_ID);
    }
}

2.3.6 Record 与 POJO 的语义边界

Record 绑定 Configuration,可调用 store / refresh / delete 直接持久化。POJO 是无依赖 Java Bean,适合跨层传输。两者不应互相替代:Record 用于事务边界内的写入,POJO 用于 REST 返回与跨服务通信。

public class OrdersRecord extends UpdatableRecordImpl<OrdersRecord> {
    public Long getId()              { return (Long) get(0); }
    public void setId(Long value)    { set(0, value); }
    public Long getCustomerId()      { return (Long) get(1); }
    public BigDecimal getTotalAmount() { return (BigDecimal) get(2); }
}

public class Order implements Serializable {
    private Long id;
    private Long customerId;
    private BigDecimal totalAmount;
    private LocalDateTime createdAt;
    // 标准 getter / setter
}

2.4.3 自定义 Strategy 示例

继承 DefaultGeneratorStrategy 是常见做法。下例为表类加 J 前缀,POJO 加 Dto 后缀,Record 加 Rec 后缀。该策略在多模块工程中区分 jOOQ 生成类与业务领域模型类。

public class PrefixedStrategy extends DefaultGeneratorStrategy {

    @Override
    public String getJavaClassName(Definition def, Mode mode) {
        String name = super.getJavaClassName(def, mode);
        return switch (mode) {
            case DEFAULT -> "J" + name;
            case POJO    -> name + "Dto";
            case RECORD  -> name + "Rec";
            default      -> name;
        };
    }
}

2.5.2 includes / excludes 粗筛

includes 与 excludes 是 schema 侧的第一层过滤,用于排除迁移历史表、临时表与备份表。两者均接受 Java 正则表达式,匹配的对象使用合格名(qualified name)进行测试。

<database>
  <inputSchema>public</inputSchema>
  <includes>t_.*|v_.*</includes>
  <excludes>flyway_schema_history|tmp_.*|_bak$</excludes>
</database>

Included objects:
  public.t_customer
  public.t_order
  public.v_order_sales

Excluded objects:
  public.flyway_schema_history
  public.tmp_import_buffer
  public.customer_bak

2.5.3 Matcher 重命名

matchers 是声明式重命名工具,通过正则捕获组与 transform 规则将数据库标识符映射为 Java 命名。下例去掉表的 t_ 前缀与列的 f_ 前缀,并对 POJO 与 Record 增加 Dto / Record 后缀。

<generator>
  <strategy>
    <matchers>
      <tables>
        <table>
          <expression>t_(.*)</expression>
          <tableClass>
            <transform>PASCAL</transform>
            <expression>$1</expression>
          </tableClass>
          <pojoClass>
            <transform>PASCAL</transform>
            <expression>$1_dto</expression>
          </pojoClass>
          <recordClass>
            <transform>PASCAL</transform>
            <expression>$1_record</expression>
          </recordClass>
        </table>
      </tables>
      <fields>
        <field>
          <expression>f_(.*)</expression>
          <fieldMember>
            <transform>UPPER</transform>
            <expression>$1</expression>
          </fieldMember>
        </field>
      </fields>
    </matchers>
  </strategy>
</generator>

t_customer.f_first_name -> tables.Customers.FIRST_NAME
t_customer.f_last_name  -> tables.Customers.LAST_NAME
t_order.f_customer_id   -> tables.Orders.CUSTOMER_ID
t_customer              -> pojos.CustomerDto / records.CustomersRecord
t_order                 -> pojos.OrderDto / records.OrdersRecord

2.5.4 Matcher 与 GeneratorStrategy 协作

维度 Matcher GeneratorStrategy
形式 声明式 XML Java 代码
输入 名称模式 完整 Definition 对象
范围 按对象类型与名称 全局或按 Mode
复杂度 简单 高
推荐场景 批量去前缀 / 加后缀 跨 schema 区分 / 注释驱动命名

两者可叠加使用:Matcher 先做 schema 侧重命名,Strategy 再做 Java 侧最终修饰。当规则需要读取对象类型、schema、注释或外部配置时,优先使用 Strategy。

2.6.2 Flyway 迁移与 codegen 串联

将 Flyway 与 jOOQ codegen 同时配置在 Maven 构建中,确保每次构建从最新迁移开始。Flyway 在 initialize 阶段执行,codegen 在 generate-sources 阶段执行,两者顺序由 Maven 生命周期保证。

<plugin>
  <groupId>org.flywaydb</groupId>
  <artifactId>flyway-maven-plugin</artifactId>
  <executions>
    <execution>
      <phase>initialize</phase>
      <goals><goal>migrate</goal></goals>
    </execution>
  </executions>
  <configuration>
    <url>jdbc:postgresql://${db.host}:5432/${db.name}</url>
    <user>${db.user}</user>
    <password>${db.password}</password>
  </configuration>
</plugin>

2.6.5 DDL-based codegen 配置

DDLDatabase(jOOQ 3.16+)允许直接从 DDL 脚本生成代码,完全消除构建期数据库依赖。该方案在安全性与 CI 简化两方面均优于真实数据库连接。

<plugin>
  <groupId>org.jooq</groupId>
  <artifactId>jooq-codegen-maven</artifactId>
  <dependencies>
    <dependency>
      <groupId>org.jooq</groupId>
      <artifactId>jooq-meta-extensions</artifactId>
      <version>${jooq.version}</version>
    </dependency>
  </dependencies>
  <configuration>
    <generator>
      <database>
        <name>org.jooq.meta.extensions.ddl.DDLDatabase</name>
        <properties>
          <property>
            <key>scripts</key>
            <value>src/main/resources/db/migration/*.sql</value>
          </property>
          <property>
            <key>sort</key>
            <value>flyway</value>
          </property>
          <property>
            <key>defaultNameCase</key>
            <value>lower</value>
          </property>
        </properties>
      </database>
    </generator>
  </configuration>
</plugin>

2.6.7 多模块工程的 source root

Maven 多模块工程中,codegen 通常放在独立 jooq-codegen 子模块。生成目录必须通过 build-helper-maven-plugin 注册为 source root,IDE 与 javac 才会参与编译。

<plugin>
  <groupId>org.codehaus.mojo</groupId>
  <artifactId>build-helper-maven-plugin</artifactId>
  <executions>
    <execution>
      <phase>generate-sources</phase>
      <goals><goal>add-source</goal></goals>
      <configuration>
        <sources>
          <source>target/generated-sources/jooq</source>
        </sources>
      </configuration>
    </execution>
  </executions>
</plugin>

2.7.3 凭证安全实践

构建期 JDBC 凭证不能直接写在 pom.xml。推荐三层方案:专用只读 codegen 用户、CI 环境变量注入、Vault 或 Secret Manager 二次保护。最小权限原则要求 codegen 用户仅有 SELECT on information_schema 与目标 schema。

<jdbc>
  <driver>org.postgresql.Driver</driver>
  <url>${env.JOOQ_CODEGEN_URL}</url>
  <user>${env.JOOQ_CODEGEN_USER}</user>
  <password>${env.JOOQ_CODEGEN_PASSWORD}</password>
</jdbc>

2.7.5 折中方案

CI 中执行"重新生成 + diff 检查"的双步策略,既保留生成代码的可审查性,又避免开发者忘记重新生成。该方案适用于 schema 变更中等频率的项目。

mvn -q generate-sources
git diff --exit-code src/generated/java || {
  echo "Generated jOOQ sources are stale";
  echo "Run 'mvn generate-sources' locally and commit the diff";
  exit 1;
}

2.8.3 最小入口示例

GenerationTool 的最小入口在测试与脚本场景中可直接使用。该入口接受 JAXB Configuration,与 Maven 插件最终调用的 API 一致。

GenerationTool.generate(new org.jooq.meta.jaxb.Configuration()
    .withJdbc(new Jdbc()
        .withDriver("org.postgresql.Driver")
        .withUrl("jdbc:postgresql://localhost:5432/order-fulfillment")
        .withUser("codegen_user"))
    .withGenerator(new Generator()
        .withName("org.jooq.codegen.JavaGenerator")
        .withDatabase(new Database()
            .withName("org.jooq.meta.postgres.PostgresDatabase")
            .withInputSchema("public"))
        .withTarget(new Target()
            .withPackageName("com.example.jooq.generated")
            .withDirectory("target/generated-sources/jooq"))));

2.8.4 设计取舍

设计点 取舍 后果
生成源码 vs 运行时反射 编译期检查 + 启动期零反射 schema 变更必须重新生成
Database 抽象元数据来源 JDBC / DDL / Liquibase 可互换 CI 可用 DDLDatabase 避开生产连接
GeneratorStrategy 命名集中 Strategy + Matcher 双层 业务代码不应硬编码命名假设
生成 TableField<R, T> SQL 类型投影为 Java 泛型 ORDERS.ID.eq("x") 编译失败
输出可读 .java 非 .class 字节码 可在 IDE 中查看 + Code Review

2.8.5 边界

jooq-codegen 只负责把数据库事实编译成 Java API,不做业务建模。聚合根边界、DTO 划分、迁移顺序仍是 schema 设计与 Repository 层的责任。codegen 输出的命名与结构是中间产物,业务代码不应依赖未公开的实现细节。

codegen 边界:

  ┌────────────────────┐
  │ schema (DDL)       │  ← 真相源头
  └─────────┬──────────┘
            │
            ▼
  ┌────────────────────┐
  │ jooq-codegen       │  ← 本章范围
  └─────────┬──────────┘
            │
            ▼
  ┌────────────────────┐
  │ Java 元模型        │  ← 业务代码消费
  │ Tables / Records   │
  └─────────┬──────────┘
            │
            ▼
  ┌────────────────────┐
  │ Repository / DSL   │  ← 业务责任
  └────────────────────┘
