第 18 章 · QueryPart 表达式树与渲染管线

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

前 17 章讲"如何使用",本章讲"如何工作"。当一行 dsl.select(ORDERS.ID).from(ORDERS).where(ORDERS.ID.eq(1)) 被写下时,实际产物是一棵表达式树;调用 .fetch() 时,jOOQ 对树做深度优先遍历,通过 RenderContext 生成 SQL 字符串、通过 BindContext 绑定参数。这就是 jOOQ 引擎的全部秘密。

核心设计:QueryPart 是万物之祖。所有 SQL 元素(语句、表、列、条件、常量)都实现该接口,通过统一的 accept(Context) visitor 方法驱动渲染。

18.1.1 接口定义

QueryPart 是公共组合基类型;真正驱动渲染的 accept(Context) 位于内部 visitor / SPI 层。业务代码通常不直接实现裸 QueryPart,而应使用 DSL、CustomField、CustomCondition 或 Binding / Converter。

public interface QueryPart extends Serializable {
    String toString();

    QueryPart $replace(Replacer replacer);
    <R> R $traverse(Supplier<R> supplier, BiFunction<R, QueryPart, R> before);
}

abstract class AbstractQueryPart implements QueryPart {
    public abstract void accept(Context<?> ctx);
}

18.3.3 BindContext 简化逻辑

class DefaultBindContext extends AbstractContext<BindContext> {
    private final PreparedStatement stmt;
    private int nextIndex = 1;

    @Override
    public void visit(Val<?> val) {
        val.bind(stmt, nextIndex++);
    }

    @Override
    public void visit(QueryPart part) {
        part.accept(this);
    }
}

图 18-5 · 同一棵树多种 SQL — LIMIT/OFFSET 的四方言分发

18.5.1 Limit 节点伪代码

class Limit extends AbstractQueryPart {
    @Override
    public void accept(Context<?> ctx) {
        switch (ctx.family()) {
            case POSTGRES:
            case MYSQL:
                ctx.keyword(LIMIT).sql(' ').visit(numberOfRows)
                   .sql(' ').keyword(OFFSET).sql(' ').visit(offset);
                break;

            case ORACLE:
                ctx.keyword(OFFSET).sql(' ').visit(offset)
                   .sql(' ').keyword(ROWS)
                   .sql(' ').keyword(FETCH_NEXT).sql(' ').visit(numberOfRows)
                   .sql(' ').keyword(ROWS_ONLY);
                break;

            case SQLSERVER:
                ctx.keyword(OFFSET).sql(' ').visit(offset)
                   .sql(' ').keyword(ROWS)
                   .sql(' ').keyword(FETCH_NEXT).sql(' ').visit(numberOfRows)
                   .sql(' ').keyword(ROWS_ONLY);
                break;
        }
    }
}

18.8 陷阱与调试

表 18-8:四大陷阱

# 陷阱 现象 根因 修复
1 自定义 QueryPart 漏方言 PG 通过 / MySQL 失败 accept 只处理一个 family 覆盖目标方言,默认抛 SQLDialectNotSupportedException
2 RenderContext 状态污染 后续字段无表名前缀 / 缩进错乱 改 qualify / castMode 后未恢复 try-finally 恢复或作用域 data API
3 线程不安全树共享 请求条件串到另一请求 共享可变 SelectQuery 每请求新建 query,只共享 Table / Field 常量
4 深表达式树栈溢出 大量 .or() 触发 SOE 极深二叉树 DSL.or(Collection) / field.in(Collection) 扁平化

18.8.1 自定义 QueryPart 示例

final class JsonText extends AbstractField<String> {
    private final Field<?> json;
    private final String key;

    JsonText(Field<?> json, String key) {
        super(name("json_text"), SQLDataType.VARCHAR);
        this.json = json;
        this.key = key;
    }

    @Override
    public void accept(Context<?> ctx) {
        switch (ctx.family()) {
            case POSTGRES:
                ctx.sql('(').visit(json).sql(" ->> ").visit(val(key)).sql(')');
                break;
            case MYSQL:
                ctx.sql("json_unquote(json_extract(")
                   .visit(json)
                   .sql(", concat('$.', ").visit(val(key)).sql(")))");
                break;
            default:
                throw new SQLDialectNotSupportedException("JSON text path");
        }
    }
}

Field<String> title = new JsonText(ORDERS.META, "title");
var query = dsl.select(title).from(ORDERS).where(ORDERS.ID.eq(42));

System.out.println(query.getSQL());
System.out.println(query.getSQL(ParamType.INLINED));
System.out.println(query.getBindValues());

select ("orders"."meta" ->> ?) from "orders" where "orders"."id" = ?
select ("orders"."meta" ->> 'title') from "orders" where "orders"."id" = 42
[title, 42]

不变量(visitor invariant):

RenderContext: visit(JsonText) -> sql + ?
BindContext  : same Val visited in same order -> setString
violation = SQL placeholder count != bind values count

18.9.2 visitor 不变量

@Test
void render_and_bind_have_consistent_count() {
    Query q = dsl.select(ORDERS.ID).from(ORDERS)
        .where(ORDERS.ID.like("%jOOQ%"))
        .and(ORDERS.AMOUNT.between(
            new BigDecimal("30.00"), new BigDecimal("99.00")));

    String sql = q.getSQL(ParamType.INDEXED);
    long placeholders = sql.chars().filter(c -> c == '?').count();
    int  binds = q.getBindValues().size();

    assertThat(binds).isEqualTo((int) placeholders);
}

18.10 测试护栏

18.10.1 方言矩阵

@ParameterizedTest
@EnumSource(value = SQLDialect.class,
    names = {"POSTGRES", "ORACLE", "SQLSERVER", "MYSQL"})
void limit_offset_renders_per_dialect(SQLDialect dialect) {
    DSLContext ctx = DSL.using(dialect);
    String sql = ctx.renderInlined(
        select(ORDERS.ID).from(ORDERS).orderBy(ORDERS.ID).limit(10).offset(20));

    if (dialect == SQLDialect.POSTGRES || dialect == SQLDialect.MYSQL) {
        assertThat(sql).contains("limit 10").contains("offset 20");
    } else {
        assertThat(sql).contains("offset 20 rows").contains("fetch next 10 rows only");
    }
}

设计取舍

取舍 1:统一 QueryPart 而非按节点类型分别接口

所有 SQL 元素实现同一接口。代价是 QueryPart API 看起来较"宽",收益是 visitor 可统一遍历整棵树。

取舍 2:Render / Bind 双 Context 而非合一

SQL 文本与参数值分离匹配 JDBC PreparedStatement 协议,天然防注入。代价是自定义节点必须保证两趟遍历顺序一致。

取舍 3:方言逻辑写在 accept 内分支

节点根据 ctx.family() 输出不同语法。方言边界集中,但源码阅读复杂。建议为每方言写快照测试。

取舍 4:ParamType.INLINED 可切换

INLINED 便于 DDL / 调试,但生产 SELECT/DML 应保留绑定变量。

取舍 5:Visitor 不内置 SQL 优化

jOOQ 不试图在渲染期做 SQL 重写或优化,把这部分留给数据库自身的优化器。
