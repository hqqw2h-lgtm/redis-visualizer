第 13 章 · ExecuteListener 与性能调优

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

每条 SQL 从 Java 对象到数据库结果集要经历 render / prepare / bind / execute / fetch 五个阶段。jOOQ 在每个阶段前后都埋设了回调钩子 — ExecuteListener。掌握这套生命周期可在不修改业务代码的前提下植入 SQL 日志、慢查询告警、指标采集,并把 SQL 改写需求交给更合适的 VisitListener、Settings 或统一查询构造器。

本章按"生命周期 → SQL 日志 → 慢查询 → 批量写入 → Loader API → fetchLazy / Cursor → 陷阱 → 回归守护 → 源码深潜"展开。ExecuteListener 是 jOOQ 执行引擎的"观察窗",嵌在自身执行模板方法里(而非 JDBC proxy 外层),因此能看到 Query、ExecuteContext、SQL、bind values、异常和 ResultSet。

13.1.4 SqlProbeListener 示例

public final class SqlProbeListener extends DefaultExecuteListener {
    private static final Logger log = LoggerFactory.getLogger(SqlProbeListener.class);

    @Override public void renderEnd(ExecuteContext ctx) {
        if (ctx.query() != null) {
            log.info("SQL : {}", ctx.sql());
            log.info("BIND: {}", ctx.query().getBindValues());
        }
    }

    @Override public void executeStart(ExecuteContext ctx) {
        ctx.data("started", System.nanoTime());
    }

    @Override public void executeEnd(ExecuteContext ctx) {
        Long t = (Long) ctx.data("started");
        if (t != null) {
            long ms = (System.nanoTime() - t) / 1_000_000;
            log.info("EXEC: {}ms rows={}", ms, ctx.rows());
        }
    }
}

public class SqlLogListener extends DefaultExecuteListener {
    private static final Logger log = LoggerFactory.getLogger(SqlLogListener.class);

    @Override public void renderEnd(ExecuteContext ctx) {
        if (ctx.query() != null) {
            String inlined = DSL.using(ctx.configuration()).renderInlined(ctx.query());
            log.info("SQL: {}", inlined);
        }
    }
}

阈值 级别 处理

13.3.1 SlowQueryListener

public class SlowQueryListener extends DefaultExecuteListener {
    private static final Logger log = LoggerFactory.getLogger(SlowQueryListener.class);
    private final long thresholdMs;
    private final MeterRegistry mr;

    public SlowQueryListener(long thresholdMs, MeterRegistry mr) {
        this.thresholdMs = thresholdMs;
        this.mr = mr;
    }

    @Override public void executeStart(ExecuteContext ctx) {
        ctx.data("slow.start", System.nanoTime());
    }

    @Override public void executeEnd(ExecuteContext ctx) {
        Long start = (Long) ctx.data("slow.start");
        if (start == null) return;
        long ms = (System.nanoTime() - start) / 1_000_000;

        mr.timer("db.query.duration").record(ms, TimeUnit.MILLISECONDS);

        if (ms > 5_000) {
            log.error("CRITICAL SLOW QUERY [{}ms]: {}", ms, ctx.sql());
            mr.counter("db.slow_query.critical").increment();
        } else if (ms > 1_000) {
            log.warn("SLOW QUERY [{}ms]: {}", ms, ctx.sql());
            mr.counter("db.slow_query.warn").increment();
        } else if (ms > thresholdMs) {
            log.debug("Query [{}ms]: {}", ms, ctx.sql());
        }
    }
}

端到端耗时(SQL + 大结果集抓取)应把结束点放到 fetchEnd() 或 end()。

13.8.2 事件顺序测试

@Test
void executeListener_event_order_is_stable() {
    List<String> events = new CopyOnWriteArrayList<>();
    ExecuteListener listener = new DefaultExecuteListener() {
        @Override public void renderStart(ExecuteContext c) { events.add("renderStart"); }
        @Override public void renderEnd(ExecuteContext c) { events.add("renderEnd"); }
        @Override public void prepareStart(ExecuteContext c) { events.add("prepareStart"); }
        @Override public void prepareEnd(ExecuteContext c) { events.add("prepareEnd"); }
        @Override public void bindStart(ExecuteContext c) { events.add("bindStart"); }
        @Override public void bindEnd(ExecuteContext c) { events.add("bindEnd"); }
        @Override public void executeStart(ExecuteContext c) { events.add("executeStart"); }
        @Override public void executeEnd(ExecuteContext c) { events.add("executeEnd"); }
        @Override public void fetchStart(ExecuteContext c) { events.add("fetchStart"); }
        @Override public void fetchEnd(ExecuteContext c) { events.add("fetchEnd"); }
    };

    DSLContext dsl = DSL.using(new DefaultConfiguration()
        .set(dataSource).set(SQLDialect.H2)
        .set(new DefaultExecuteListenerProvider(listener))
        .set(new Settings().withExecuteLogging(false)));

    dsl.selectOne().where(DSL.val(1).eq(1)).fetch();

    assertThat(events).containsSubsequence(
        "renderStart", "renderEnd",
        "prepareStart", "prepareEnd",
        "bindStart", "bindEnd",
        "executeStart", "executeEnd",
        "fetchStart", "fetchEnd");
}

13.8.3 SQL 与 bind 同时捕获

@Test
void sql_and_bind_values_are_captured_together() {
    AtomicReference<String> sql = new AtomicReference<>();
    AtomicReference<List<Object>> binds = new AtomicReference<>();

    ExecuteListener listener = new DefaultExecuteListener() {
        @Override public void renderEnd(ExecuteContext c) {
            if (c.query() != null) {
                sql.set(c.sql());
                binds.set(c.query().getBindValues());
            }
        }
    };

    DSLContext dsl = DSL.using(new DefaultConfiguration()
        .set(dataSource).set(SQLDialect.H2)
        .set(new DefaultExecuteListenerProvider(listener)));

    dsl.select(ORDERS.ID).from(ORDERS)
        .where(ORDERS.STATUS.eq("PAID"))
        .fetch();

    assertThat(sql.get()).contains("where").contains("?");
    assertThat(binds.get()).containsExactly("PAID");
}

13.8.4 慢查询阈值

@Test
void slow_query_threshold_emits_warning() {
    ListAppender<ILoggingEvent> appender = attachListAppender(SlowQueryListener.class);

    DSLContext dsl = DSL.using(new DefaultConfiguration()
        .set(postgresDataSource).set(SQLDialect.POSTGRES)
        .set(new DefaultExecuteListenerProvider(
            new SlowQueryListener(10, new SimpleMeterRegistry()))));

    dsl.fetch("select pg_sleep(0.020)");

        .extracting(ILoggingEvent::getFormattedMessage)

}

表 13-9:源码入口

入口 包 / 类 职责
接口 org.jooq.ExecuteListener 10+ 回调
默认实现 org.jooq.impl.DefaultExecuteListener 空骨架
上下文 org.jooq.ExecuteContext sql / query / statement / resultSet / exception
提供器 org.jooq.ExecuteListenerProvider 实例化 listener
默认提供器 org.jooq.impl.DefaultExecuteListenerProvider 注入

Diagnostic org.jooq.DiagnosticsListener / DiagnosticsContext 诊断
内置 org.jooq.tools.LoggerListener executeLogging

设计取舍

取舍 1:阶段回调 + 上下文,而非"只暴露最终 SQL"

执行问题不在单点:渲染慢、prepare 慢、bind 类型错、执行慢、fetch 慢根因不同。阶段回调让用户精确定位;ExecuteContext 提供 SQL、query、statement、resultSet、exception 等全链路对象。

取舍 2:嵌入 jOOQ 模板方法,而非 JDBC proxy

JDBC proxy 看不到 jOOQ AST、Settings、Configuration、Record 映射阶段。jOOQ 选择把 listener 嵌入自身执行模板方法,因此能看到业务对象层信息。

取舍 3:四类 listener 分工而非合并

ExecuteListener / VisitListener / RecordListener / DiagnosticsListener 分别覆盖执行 / 渲染 / 记录 / 诊断四个层面。合并会让接口臃肿且语义模糊;分工让每个 listener 职责单一,组合使用时挂到同一 Configuration。

取舍 4:Listener 不替代 SQL 改写器

VisitListener 是局部观察 / 改写挂点,不是完整 SQL 重写器。复杂多租户应使用 QOM / Policy / 统一 Repository,Listener 不应承载业务语义,否则后期维护困难。

取舍 5:ctx.data() 共享上下文

阶段间共享数据(如 executeStart 记录起始时间,executeEnd 计算耗时)通过 ctx.data(key, value) 完成。该 API 隔离不同 listener 的状态,避免共享变量竞态。
