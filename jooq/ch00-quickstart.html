第 0 章 · Quick Start:从 Schema 到类型安全 SQL

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

jOOQ 的入门路径与 ORM 框架不同。它不通过反射映射类与表,而是把数据库 schema 当作真相源头,在构建期生成 Java 元模型,在编译期约束查询表达式,在运行期把 DSL 树渲染为方言 SQL。本章把这条闭环拆成六个可验证阶段:codegen、DSL、render、execute、mapping、transaction。

通常在排错过程中,问题会被笼统归因为"jOOQ 异常"。更有效的判断是先确定故障落在哪一阶段:代码生成属于构建期,DSL 表达式属于编译期,SQL 渲染与 JDBC 执行属于运行期,提交回滚由连接或 Spring 边界控制。本章按这一结构组织 11 节内容,所有示例围绕 customers/orders 两表展开,在 H2 + Maven 环境下可重复运行。

0.1 项目骨架与四阶段边界

0.1.1 三类目录与两类构件

最小可运行项目由三类目录构成:源码目录、构建输出目录、运行时目录。每一类对应不同的生命周期边界,问题排查时优先确认目录是否符合预期状态。

0.1.2 四阶段边界对照

阶段 触发命令 产物 失败信号

0.2 Schema 与代码生成

0.2.1 codegen 的执行顺序

图 0-2-1 · codegen 时序 — Maven 串起 Flyway / jooq-meta / JavaGenerator

0.2.4 TableField 的类型来源

public class Customers extends TableImpl<CustomersRecord> {
    public static final Customers CUSTOMERS = new Customers();

    public final TableField<CustomersRecord, String> ID =
        createField(DSL.name("ID"),
                    SQLDataType.VARCHAR.length(20), this);

    public final TableField<CustomersRecord, String> NAME =
        createField(DSL.name("NAME"),
                    SQLDataType.VARCHAR.length(100).nullable(false), this);

    public final TableField<CustomersRecord, String> EMAIL =
        createField(DSL.name("EMAIL"),
                    SQLDataType.VARCHAR.length(200).nullable(false), this);
}

TableField<R, T> 携带两条静态信息:绑定的 Record 类型 R 与 SQL 列类型 T。这两个泛型参数在编译期参与所有 DSL 表达式的类型检查,是 jOOQ 类型安全的物理基础。

0.3 DSLContext 运行期入口

0.3.1 DSLContext 的角色定位

DSLContext 是运行期单一入口。它持有 Configuration,后者封装数据源、方言、设置、监听器、连接提供者。每条 DSL 表达式经由 DSLContext 转化为 QueryPart 树,再由内部 Renderer 输出 SQL。

图 0-3-1 · DSLContext 执行时序 — App / Driver / Conn / DSL / DB

0.3.2 第一条查询

package com.example;

import org.jooq.DSLContext;
import org.jooq.SQLDialect;
import org.jooq.impl.DSL;

import java.sql.Connection;
import java.sql.DriverManager;

import static com.example.jooq.generated.tables.Customers.CUSTOMERS;
import static com.example.jooq.generated.tables.Orders.ORDERS;

public class Main {
    public static void main(String[] args) throws Exception {
        try (Connection conn = DriverManager.getConnection(
                "jdbc:h2:./target/db", "sa", "")) {
            DSLContext dsl = DSL.using(conn, SQLDialect.H2);

            dsl.select(CUSTOMERS.NAME, CUSTOMERS.EMAIL, ORDERS.ID)
               .from(CUSTOMERS)
               .join(ORDERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
               .where(CUSTOMERS.EMAIL.eq("chen@example.com"))
               .fetch()
               .forEach(row -> System.out.printf(
                   "%s placed %s%n",
                   row.get(CUSTOMERS.NAME),
                   row.get(ORDERS.ID)));
        }
    }
}

$ mvn -q exec:java -Dexec.mainClass=com.example.Main
Customer Wei placed O1001

0.3.4 DSLContext 与 Configuration 关系

图 0-3-2 · DSLContext 与 Configuration — Configuration 是真正的"运行期上下文"

Configuration 在所有运行期组件之间传递,DSLContext 仅是其上的便利门面。所有自定义扩展(监听器、转换器、命名策略、SQL 方言)都通过 Configuration 接入。

0.4 DSL 到 SQL 的渲染流水线

0.4.1 渲染流程的五个阶段

图 0-4-1 · DSL → SQL 五阶段渲染流水线 — 双遍历模型(RenderContext + BindContext)

每条 DSL 表达式构造一棵抽象语法树,称为 QueryPart 树。RenderContext 与 BindContext 在两次遍历中分别产出 SQL 文本与参数列表,确保参数化与防注入。

0.4.2 getSQL 与 renderInlined 对比

方法 输出特征 主要用途 风险
query.getSQL() 含 ? 占位符 日志、缓存键、单元测试 不可直接复制执行
dsl.renderInlined(query) 内联字面量 排查、复制到客户端 字面量参与 SQL 注入
query.getParams() 仅参数列表 与 getSQL() 配对 顺序敏感
query.getBindValues() 参数值列表 配合 getSQL() 重建调用 类型已转 Java

0.4.3 渲染设置矩阵

Settings 字段 默认 影响
renderFormatted false 输出多行格式化 SQL
renderSchema true 是否在表名前加 schema
renderQuotedNames EXPLICIT_DEFAULT_QUOTED 标识符引号策略
renderNameCase AS_IS 标识符大小写
renderKeywordCase LOWER 关键字大小写
renderOptionalAsKeywordForFieldAliases DEFAULT as 关键字策略

Settings 是渲染期的全局开关,通过 DSL.using(..., settings) 在 DSLContext 实例化时注入。生产环境与本地排查可使用不同的 Settings,避免日志与执行 SQL 不一致。

0.4.4 ASCII:QueryPart 树结构示意

SelectQueryImpl
  ├─ select   : Field[] = { CUSTOMERS.NAME, ORDERS.ID }
  ├─ from     : Table[] = { CUSTOMERS }
  ├─ joins    : Join[]  = { JoinTable(ORDERS, INNER, ON) }
  │                          └─ on: Compare(ORDERS.CUSTOMER_ID, EQ, CUSTOMERS.ID)
  ├─ where    : Condition = Compare(CUSTOMERS.EMAIL, EQ, Param("chen@example.com"))
  ├─ groupBy  : 空
  ├─ having   : 空
  ├─ orderBy  : 空
  └─ limit    : 空

每个节点都是 QueryPart 的子类型。RenderContext.visit(part) 递归下行,BindContext.visit(part) 在第二遍历时收集参数值。该双遍历模型在第 18 章源码深潜中详述。

0.5 Result 与 Record 映射

0.5.1 三类映射目标

图 0-5-1 · 三类映射目标 — fetch / fetchInto(Record) / fetchInto(DTO)

Record 是 jOOQ 的核心行抽象,生成的 CustomersRecord 继承 UpdatableRecord 并绑定主键,使得 store/delete/refresh 自动生成 INSERT/UPDATE/DELETE。Pojo(record 类或 JavaBean)只承载字段,无数据库语义。

0.5.2 三种映射写法对照

record CustomerDto(String id, String name, String email) {}

var generic = dsl.select(CUSTOMERS.ID, CUSTOMERS.NAME, CUSTOMERS.EMAIL)
                 .from(CUSTOMERS)
                 .fetch();
generic.forEach(row -> System.out.println(row.get(CUSTOMERS.NAME)));

var typedRecord = dsl.fetchOne(CUSTOMERS, CUSTOMERS.ID.eq("C1001"));
typedRecord.setEmail("chen.updated@example.com");
typedRecord.store();

var dtoList = dsl.select(CUSTOMERS.ID, CUSTOMERS.NAME, CUSTOMERS.EMAIL)
                 .from(CUSTOMERS)
                 .fetchInto(CustomerDto.class);
dtoList.forEach(System.out::println);

0.5.3 列名别名规则

record OrderView(String customerName, String orderId) {}

var rows = dsl.select(
        CUSTOMERS.NAME.as("customerName"),
        ORDERS.ID.as("orderId"))
    .from(CUSTOMERS)
    .join(ORDERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
    .fetchInto(OrderView.class);

rows.forEach(System.out::println);

OrderView[customerName=Customer Wei, orderId=O1001]

DTO 映射依赖列名匹配。当 select 列与 DTO 字段名不一致时,必须使用 as("fieldName") 显式别名。否则 jOOQ 默认按列原名查找属性或构造器参数,匹配不到的字段在 DTO 中保留为 null。

0.5.4 映射性能开销分布

阶段 主要开销 优化点
ResultSet → Record 列遍历 + 类型转换 复用 Record 类型缓存
Record → DTO 反射查找属性 使用 record 类或显式 RecordMapper
Converter 调用 用户实现 避免每行新建对象
Result 持有 JVM 堆内存 大结果用 Cursor 流式读取

大结果集场景下应使用 dsl.fetchLazy(...) 返回 Cursor,逐行处理而不全量加载到内存。第 25 章给出 10M 行级别的具体性能数据。

0.6 事务边界与连接管理

0.6.1 三种典型事务模型

图 0-6-1 · 三种典型事务模型 — JDBC / jOOQ transaction / Spring @Transactional

0.6.2 纯 JDBC 边界示例

try (Connection conn = DriverManager.getConnection(
        "jdbc:h2:./target/db", "sa", "")) {
    conn.setAutoCommit(false);
    DSLContext tx = DSL.using(conn, SQLDialect.H2);

    try {
        CustomersRecord customer = tx.newRecord(CUSTOMERS);
        customer.setId("C1002");
        customer.setName("Li Lei");
        customer.setEmail("li@example.com");
        customer.store();

        tx.insertInto(ORDERS)
          .set(ORDERS.ID, "O1002")
          .set(ORDERS.CUSTOMER_ID, customer.getId())
          .set(ORDERS.AMOUNT, new BigDecimal("128.00"))
          .set(ORDERS.STATUS, "PAID")
          .execute();

        conn.commit();
    } catch (Exception e) {
        conn.rollback();
        throw e;
    }
}

0.6.3 jOOQ transaction API

dsl.transaction(cfg -> {
    DSLContext tx = DSL.using(cfg);
    tx.insertInto(CUSTOMERS)
      .set(CUSTOMERS.ID, "C1003")
      .set(CUSTOMERS.NAME, "Wang Fang")
      .set(CUSTOMERS.EMAIL, "wang@example.com")
      .execute();

    tx.insertInto(ORDERS)
      .set(ORDERS.ID, "O1003")
      .set(ORDERS.CUSTOMER_ID, "C1003")
      .set(ORDERS.AMOUNT, new BigDecimal("256.00"))
      .set(ORDERS.STATUS, "PAID")
      .execute();
});

transaction 接受 TransactionalRunnable,在闭包返回时调用 commit,抛出异常时自动 rollback。该 API 不依赖 Spring,仅利用 Configuration.connectionProvider() 提供的连接。

0.6.4 Spring 集成边界示例

@Service
public class OrderService {
    private final DSLContext dsl;

    public OrderService(DSLContext dsl) {
        this.dsl = dsl;
    }

    @Transactional
    public String createOrder(String customerId, BigDecimal amount) {
        return dsl.insertInto(ORDERS)
                  .set(ORDERS.ID, "O" + System.currentTimeMillis())
                  .set(ORDERS.CUSTOMER_ID, customerId)
                  .set(ORDERS.AMOUNT, amount)
                  .set(ORDERS.STATUS, "PAID")
                  .returningResult(ORDERS.ID)
                  .fetchOne(ORDERS.ID);
    }
}

Spring Boot starter 通过 DataSourceTransactionManager 与 DataSource 关联事务上下文,DSLContext 通过 SpringConnectionProvider 取到当前线程绑定连接。jOOQ 不重复管理事务,这是 Spring 集成的核心设计。

0.8 陷阱与限制

图 0-8 · Quickstart 陷阱分类

图 0-8-1 · Quickstart 陷阱分类 — 4 阶段 8 个陷阱

表 0-8:陷阱明细

# 阶段 症状 根因 修复 监控指标
1 构建期 codegen 跳过目标表 inputSchema 大小写或 includes 不匹配 数据库客户端确认表名后调整正则 JavaGenerator 日志
2 构建期 Driver class not found Flyway / codegen 插件未声明 JDBC 驱动 在两个插件的 <dependencies> 中各加一次 Maven 报错
3 构建期 CI codegen 数十秒 扫描整库或系统表 配置 includes / excludes 白名单 构建耗时
4 编译期 字段已删却仍编译通过 旧 generated-sources 残留 mvn clean generate-sources cannot find symbol
5 编译期 IDE 红线但 mvn 通过 源码根未注册 IDE 重新导入 Maven 模块 IDE 设置
6 运行期 SQL 语法在目标库不识别 SQLDialect.DEFAULT 显式配置目标方言 SQL 异常
7 运行期 日志只能看到 ? 默认日志级别 启用 LoggerListener DEBUG 日志
8 事务期 部分行已落库 autoCommit 仍为 true setAutoCommit(false) 或 @Transactional 业务校验

反模式与正模式对照

反模式:
  pom.xml:
    flyway-maven-plugin 缺少 H2 driver
    jooq-codegen-maven 直接写 <jdbc><url>jdbc:h2:./target/db</url>
  application.yml:
    spring.datasource.url=jdbc:postgresql://prod
    jooq SQLDialect 使用 DEFAULT
  Service:
    没有 @Transactional,捕获 SQLException 后吞掉

正模式:
  pom.xml:
    两个插件各自声明 H2 driver
    inputSchema=PUBLIC, includes=CUSTOMERS|ORDERS
  application.yml:

@Test
void query_returnsExpectedRows() throws Exception {
    try (Connection conn = openConnection()) {
        DSLContext dsl = DSL.using(conn, SQLDialect.H2);
        Result<?> rows = dsl.select(CUSTOMERS.NAME, ORDERS.ID)
                            .from(CUSTOMERS)
                            .join(ORDERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
                            .where(CUSTOMERS.EMAIL.eq("chen@example.com"))
                            .fetch();

        assertThat(rows).hasSize(1);
        assertThat(rows.get(0).get(ORDERS.ID)).isEqualTo("O1001");
    }
}

@Test
void failedUnitOfWork_rollsBack() throws Exception {
    try (Connection conn = openConnection()) {
        conn.setAutoCommit(false);
        DSLContext tx = DSL.using(conn, SQLDialect.H2);

        try {
            tx.insertInto(CUSTOMERS)
              .columns(CUSTOMERS.ID, CUSTOMERS.NAME,
                       CUSTOMERS.EMAIL, CUSTOMERS.CREATED_AT)
              .values("C9999", "Wang Fang", "wang@example.com",
                      LocalDate.of(2024, 1, 2))
              .execute();
            throw new IllegalStateException("force rollback");
        } catch (RuntimeException expected) {
            conn.rollback();
        }

        Integer remaining = tx.selectCount()
                              .from(CUSTOMERS)
                              .where(CUSTOMERS.ID.eq("C9999"))
                              .fetchOne(0, int.class);
        assertThat(remaining).isZero();
    }
}

设计取舍

取舍 1:构建期 codegen 与运行时反射

ORM 框架普遍依赖运行时反射建立类与表的映射,代价是启动期扫描类路径并维持元数据缓存。jOOQ 选择构建期生成 Java 源码,把元模型固化到 .class 文件,运行时只读取静态字段。该方案使类型检查左移到编译期,代价是 schema 变更后必须重新构建。

取舍 2:DSL 同步 API 与流式异步

主线 DSL 返回 Result<R> 的同步 API,语义清晰且与 JDBC 模型对齐。流式与异步路径通过 fetchLazy/fetchAsync/fetchStream 与 R2DBC 模块单独提供,避免主路径承担反压与背压复杂性。第 21 章详述 R2DBC 集成。

取舍 3:连接由调用者提供

DSLContext 不持有连接池,而是通过 ConnectionProvider 在每次执行时按需获取与释放。该设计使 jOOQ 与 HikariCP、Spring DataSource、应用自有连接池兼容,事务边界由调用者(纯 JDBC、transaction API、Spring 代理)负责。

0.11 本章要点

要点 验证依据 易错点
jOOQ 类型安全来自构建期 codegen mvn generate-sources 后 Tables.java 存在 schema 变更未重建
DSLContext 是运行期单一入口 DSL.using(conn, dialect) 未指定 SQLDialect
QueryPart 树支持双遍历渲染 getSQL 与 renderInlined 把 renderInlined 用于生产
Record 行模型可直接落库 record.store/delete/refresh 与 POJO 混淆
DTO 映射依赖列名 as("fieldName") 字段为 null
事务边界由调用者控制 setAutoCommit / transaction / @Transactional 部分提交
故障落在四阶段之一 决策树 把所有错误归因为 jOOQ
回归守护至少三条断言 SQL snapshot + 行数 + 回滚 仅做行数断言
