第 9 章 · Record 与结果集

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

.fetch() 返回的 Result<Record> 不是 JDBC ResultSet,而是 jOOQ 在 SQL 元数据与 Java 对象之间的中间层。该层保留列别名、Field<T> 类型与 changed 标志位,使后续的 POJO 映射、批量更新、流式处理在同一抽象上运作。本章按"Record 继承体系 → 六种获取策略 → 三种映射路径 → 函数式管道 → fetchInto 快捷 → 陷阱 → 源码"展开。

理解 Record 体系的关键在于其工程目标:既保留 SQL 视图的形态(列、类型、别名),又支持 Active Record 风格的可变状态(store / delete / refresh)。前者服务于复杂查询与多表 JOIN,后者服务于简单 CRUD。两者共存于同一继承树,使业务代码可在不同复杂度间平滑切换。

9.1 Record 与 Result 体系

9.1.1 类层次

图 9-1-1 · Record 类层次 — Record → RecordN / TableRecord / UpdatableRecord + Result

9.1.2 Record 类型来源

来源 返回 说明
select(F1).from(T) Record1<T1> 单列泛型
select(F1, F2).from(T) Record2<T1, T2> 双列泛型
select(F1..F22) Record1..Record22 至 22 列
select(F1..F23+) Record(无泛型) 超 22 列退化
selectFrom(T) TableRecord<R> 表绑定
fetchOne(T, c) UpdatableRecord<R> 含主键的表绑定
selectCount() Record1<Integer> 聚合

Record1..Record22 上限源自 Java 泛型限制(无可变长泛型参数)。超过 22 列时退化为通用 Record。实际项目中,SELECT 超过 22 列通常应改用 fetchInto(POJO.class) 或拆分查询。

9.2.4 fetchStream 与 fetchLazy

图 9-2-4 · fetch vs fetchStream — 全量 vs 流式的内存差异
try (Stream<OrdersRecord> s = ctx.selectFrom(ORDERS).fetchStream()) {
    s.filter(r -> r.getAmount().compareTo(BigDecimal.TEN) > 0)
     .map(OrdersRecord::getTitle)
     .forEach(System.out::println);
}

try (Cursor<OrdersRecord> c = ctx.selectFrom(ORDERS).fetchLazy()) {
    Result<OrdersRecord> batch;
    while ((batch = c.fetchNext(500)).isNotEmpty()) {
        processBatch(batch);
    }
}

fetchStream / fetchLazy 持有 JDBC ResultSet 与连接,必须 try-with-resources 关闭。是否真正服务端流式还取决于 JDBC 驱动、事务边界与 fetch size 配置。

9.2.5 大结果集策略对照

行数 推荐 原因
< 1,000 fetch() 全量简洁
1k-100k fetch() + 分页 LIMIT/OFFSET / SEEK
> 100k fetchLazy().fetchNext(N) 批量恒定内存
全表导出 fetchLazy() + 文件 IO 平衡网络与内存

9.3.3 反射映射 into

public class OrdersDTO {
    private Long id;
    private String title;
    private String customerName;

    public void setId(Long id) { this.id = id; }
    public void setTitle(String t) { this.title = t; }
    public void setCustomerName(String n) { this.customerName = n; }
}

OrdersDTO dto = ctx.select(
        ORDERS.ID,
        ORDERS.TITLE,
        CUSTOMERS.NAME.as("customer_name"))
    .from(ORDERS)
    .join(CUSTOMERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
    .where(ORDERS.ID.eq(1L))
    .fetchOne()
    .into(OrdersDTO.class);

DefaultRecordMapper 的常见匹配分支:setter 注入(默认构造器 + 同名 setter)、字段注入(同名字段)、构造器注入(Java record 或 -parameters 编译)。匹配失败时字段静默为 null,而非异常。

9.3.4 列名标准化规则

SQL 列 标准化 匹配 Java
ORDER_ID orderId setOrderId / orderId
customer_name customerName setCustomerName / customerName
ID id setId / id
PUBLISHED_IN publishedin setPublishedIn / publishedIn

去下划线 + 大小写归一是 jOOQ 默认列名匹配策略。多表 JOIN 时若有重名列,必须 .as("alias") 消除歧义。

DSLContext ctx = DSL.using(

关键反例与正例

    .where(ORDERS.STATUS.eq("ACTIVE"))
    .fetchOne();   // → TooManyRowsException

// 正例 1:fetchAny

    .where(ORDERS.STATUS.eq("ACTIVE"))
    .fetchAny();

// 正例 2:limit(1) + 排序
OrdersRecord latest = ctx.selectFrom(ORDERS)
    .where(ORDERS.STATUS.eq("ACTIVE"))
    .orderBy(ORDERS.CREATED_AT.desc())
    .limit(1)
    .fetchOne();

// 反例:Cursor 未关闭

c.fetchNext();
// 资源泄漏 → 连接池耗尽

// 正例:try-with-resources
try (Cursor<OrdersRecord> cur = ctx.selectFrom(ORDERS).fetchLazy()) {
    while (cur.hasNext()) {
        OrdersRecord rec = cur.fetchNext();

    }

设计取舍

取舍 1:Record 作为中间层

不直接暴露 ResultSet(生命周期危险、类型弱),也不强制 POJO(失去列元数据、动态投影)。Record 作为中间层保留 SQL 视图形态,可向 POJO / DTO / Map / JSON 转换。

取舍 2:Record1..Record22 上限

Java 泛型无可变长泛型参数,Record1..Record22 是工程折中。超过 22 列退化为通用 Record。该上限是已知妥协,促使大查询使用 DTO。

取舍 3:可变 vs 不可变 Record

UpdatableRecord.store() 需要可变状态(changed tracking)。完全不可变虽然线程安全语义清晰,但会牺牲 Active Record 模式。jOOQ 选择可变 + 文档化使用约束(单线程、不跨事务共享)。

取舍 4:RecordMapperProvider 全局扩展

通过 Configuration 级别注册 RecordMapperProvider,into / fetchInto 自动使用。该设计避免业务代码每处都传 mapper,与 DI 框架(Spring)集成自然。

9.11 性能场景与监控

9.11.2 反例:在循环中 fetchOne

// ❌ 错误 — N+1 查询
List<Order> orders = dsl.selectFrom(ORDERS).fetchInto(Order.class);
for (Order o : orders) {
    Customer c = dsl.selectFrom(CUSTOMERS)
                    .where(CUSTOMERS.ID.eq(o.getCustomerId()))
                    .fetchOneInto(Customer.class);  // ❌ 每个订单 1 次查询
    o.setCustomer(c);
}

// 1000 个订单 = 1001 次 DB 往返

// ✅ 修复 — 一次 JOIN
Map<Long, Customer> customers = dsl.select(...)
    .from(ORDERS)
    .join(CUSTOMERS).on(CUSTOMERS.ID.eq(ORDERS.CUSTOMER_ID))
    .fetchMap(CUSTOMERS.ID, Customer.class);

// 或者用 MULTISET (见第 11 章)

9.11.3 反例:fetchLazy 忘记关闭

// ❌ 错误
Cursor<OrdersRecord> cursor = dsl.selectFrom(ORDERS).fetchLazy();
while (cursor.hasNext()) {
    OrdersRecord r = cursor.fetchNext();
    process(r);
}
// ❌ 没 close,JDBC ResultSet 泄漏,数据库 cursor 不释放

// ✅ 修复 — try-with-resources
try (Cursor<OrdersRecord> cursor = dsl.selectFrom(ORDERS).fetchLazy()) {
    while (cursor.hasNext()) {
        OrdersRecord r = cursor.fetchNext();
        process(r);
    }
}
// 或更简洁的 fetchStream
try (Stream<OrdersRecord> stream = dsl.selectFrom(ORDERS).fetchStream()) {
    stream.forEach(this::process);
}

9.11.4 反例:Record 跨线程共享

// ❌ 错误
OrdersRecord r = dsl.selectFrom(ORDERS).where(ORDERS.ID.eq(42)).fetchOne();
executor.submit(() -> {
    r.setStatus("CONFIRMED");
    r.store();  // ❌ Record 不是线程安全的
});
executor.submit(() -> {
    r.setAmount(r.getAmount() + 100);
    r.store();  // ❌ 与上面竞争
});

// ✅ 修复 — 不共享 Record,每个线程独立查询
executor.submit(() -> {
    OrdersRecord r = dsl.selectFrom(ORDERS).where(ORDERS.ID.eq(42)).fetchOne();
    r.setStatus("CONFIRMED");
    r.store();
});

9.12 设计取舍

设计点 jOOQ 选择 替代方案 影响
Record 体系 Record / RecordN / Updatable 分层 单一 Record API 兼顾通用与 Active Record 模式
changed 标志位 字段级 dirty tracking 全字段 update 减少 SQL 字段, 防止误覆盖 NULL
fetch 默认 eager List 全部加载 默认 lazy 简单场景方便, 大结果集需要显式 lazy
映射策略 fetchInto / map / 手动 build 三种 仅 ORM 映射 灵活, 支持复杂 DTO 拼接
乐观锁 Settings.executeWithOptimisticLocking 业务层判断 version 配置化, 减少业务代码

9.13 复盘 / Recap

维度 重点
核心 Record 是 SQL 元数据与 Java 对象之间的中间层,既保留列别名/类型,又支持 Active Record。
取数 6 种策略覆盖单行 (fetchOne/Optional/Single)、多行 eager (fetch)、流式 (fetchLazy/fetchStream)、聚合 (fetchMap)。
映射 fetchInto 反射 vs map(lambda) 显式两条路径;反射方便,lambda 类型安全且更快。
性能 10k 以下 eager,10k-100k 视场景,> 100k 必 lazy + setFetchSize 防止 cursor OOM。
易错 N+1 (循环 fetchOne)、cursor 不关闭 (fetchLazy 泄漏)、Record 跨线程共享 (非线程安全)。
监控 jooq_fetch_duration / fetch_row_count / lazy_cursor_open / record_store_failed 必跟踪。
