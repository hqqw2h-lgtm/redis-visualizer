第 12 章 · SQLDialect 与方言引擎

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

SQLDialect 是 jOOQ "一次编写,多库适配" 的核心机制。同一段 DSL 在 PostgreSQL 上生成 LIMIT 10 OFFSET 20,在 Oracle 12c 上变成 OFFSET 20 ROWS FETCH NEXT 10 ROWS ONLY,在 SQL Server 上又是另一种语法 — 这背后由一棵 QueryPart AST 与一个 Context.visit() 深度优先遍历驱动,每个节点根据当前 SQLDialect 选择渲染分支。当目标方言缺少某项特性时,jOOQ 通过 emulation(模拟)填补,使业务代码无需感知数据库差异。

本章按"原理 → 方言差异矩阵 → emulation → VisitListener → Parser → 陷阱 → 生产守卫 → 测试 → 源码深潜"展开。SQLDialect 的关键设计是分层判断:方言族(family)→ 数据库版本能力 → 部署环境能力,每层都需要不同的验证手段(快照测试、Testcontainers、生产 smoke test)。

12.2.1 UPSERT 同 DSL 不同 SQL

var upsert = DSL.insertInto(PRODUCTS)
    .set(PRODUCTS.SKU, "SKU-REDIS-001")
    .set(PRODUCTS.NAME, "Redis 实战课程")
    .set(PRODUCTS.PRICE, new BigDecimal("128.00"))
    .onDuplicateKeyUpdate()
    .set(PRODUCTS.NAME, excluded(PRODUCTS.NAME))
    .set(PRODUCTS.PRICE, excluded(PRODUCTS.PRICE));

PostgreSQL:

insert into products (sku, name, price)
values ('SKU-REDIS-001', 'Redis 实战课程', 128.00)
on conflict (sku) do update
set name = excluded.name,
    price = excluded.price

MySQL:

insert into products (sku, name, price)
values ('SKU-REDIS-001', 'Redis 实战课程', 128.00)
on duplicate key update
  name = values(name),
  price = values(price)

Oracle:

merge into products
using (
  select 'SKU-REDIS-001' sku,
         'Redis 实战课程' name,
         128.00 price
  from dual
) src
on (products.sku = src.sku)
when matched then update
  set products.name = src.name,
      products.price = src.price
when not matched then insert (sku, name, price)
  values (src.sku, src.name, src.price)

H2:

merge into products (sku, name, price)
key (sku)
values ('SKU-REDIS-001', 'Redis 实战课程', 128.00)

12.2.2 类型 / 函数差异组合

var typed = DSL.select(
        ORDERS.ID,
        ORDERS.PUBLISHED.eq(true).as("is_published"),
        DSL.concat(ORDERS.ID, inline(" / "), ORDERS.SKU).as("label"),
        DSL.trunc(ORDERS.ORDER_DATE, DatePart.MONTH).as("published_month"),
        DSL.jsonValue(ORDERS.METADATA, "$.format").as("format"))
    .from(ORDERS)
    .where(ORDERS.PUBLISHED.eq(true));

PostgreSQL:

select
  orders.id,
  orders.published = true as is_published,
  orders.id || ' / ' || orders.sku as label,
  date_trunc('month', orders.order_date) as published_month,
  orders.metadata ->> 'format' as format
from orders
where orders.published = true

MySQL:

select
  orders.id,
  orders.published = 1 as is_published,
  concat(orders.id, ' / ', orders.sku) as label,
  str_to_date(date_format(orders.order_date, '%Y-%m-01'), '%Y-%m-%d') as published_month,
  json_unquote(json_extract(orders.metadata, '$.format')) as format
from orders
where orders.published = 1

Oracle:

select
  orders.id,
  case when orders.published = 1 then 1 else 0 end as is_published,
  orders.id || ' / ' || orders.sku as label,
  trunc(orders.order_date, 'MM') as published_month,
  json_value(orders.metadata, '$.format') as format
from orders
where orders.published = 1

业务代码使用 DSL.concat / DSL.trunc / DSL.jsonValue 等统一 API,渲染器自动选择方言函数;若改用 DSL.field("col->>'k'") 注入原始 SQL 片段,则绕过方言适配,跨库即失败。

12.3 Emulation 模拟

图 12-3:Emulation 决策

图 12-3 · Emulation 决策树 — native / emulation / unsupported 三路分流

表 12-3:常见 emulation

特性 原生方言 模拟方言 / 方案 精度损失
BOOLEAN PG MySQL → 1/0,Oracle → 1/0 无
LIMIT/OFFSET PG / MySQL Oracle 11g → ROWNUM 深分页性能差
MERGE Oracle / PG 15+ MySQL → ON DUPLICATE KEY 复合唯一键语义不同
RETURNING PG MySQL → SELECT LAST_INSERT_ID 仅自增 ID

12.3.1 LATERAL 模拟示例

var lateral = DSL.select(CUSTOMERS.ID,
        field(name("t", "cnt"), Integer.class))
    .from(CUSTOMERS)
    .crossApply(
        select(count().as("cnt"))
        .from(ORDERS)
        .where(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
        .asTable("t"));

PostgreSQL:

select customers.id, t.cnt
from customers
cross join lateral (
  select count(*) as cnt
  from orders
  where orders.customer_id = customers.id
) as t

MySQL 5.7:

select customers.id,
       (select count(*)
        from orders
        where orders.customer_id = customers.id) as cnt
from customers

12.3.2 RETURNING 模拟

var inserted = DSL.insertInto(CUSTOMERS)
    .set(CUSTOMERS.NAME, "Customer C1001")
    .set(CUSTOMERS.EMAIL, "c1001@example.com")
    .returning(CUSTOMERS.ID, CUSTOMERS.NAME, CUSTOMERS.EMAIL);

PostgreSQL:

insert into customers (name, email)
values ('Customer C1001', 'c1001@example.com')
returning customers.id, customers.name, customers.email

MySQL 模拟:

insert into customers (name, email)
values ('Customer C1001', 'c1001@example.com');
select last_insert_id();

Emulation ≠ 完美等效:必须为关键 emulation 路径建立真实数据库集成测试,覆盖唯一键冲突、复合主键、默认值返回、空集语义等边界。

12.5.2 Oracle → PostgreSQL 迁移

DSLContext oracle = DSL.using(SQLDialect.ORACLE);
DSLContext postgres = DSL.using(SQLDialect.POSTGRES);

String oracleSQL = """
    SELECT NVL(customer_name, 'N/A') AS first_name,
           SYSDATE AS exported_at
    FROM customers
    WHERE ROWNUM <= 5
    """;

Query parsed = oracle.parser().parseQuery(oracleSQL);
String pgSQL = postgres.render(parsed);

PostgreSQL 输出:

select coalesce(customers.customer_name, 'N/A') as first_name,
       current_timestamp as exported_at
from customers
limit 5

12.5.3 转换映射

Oracle 语法 jOOQ 自动转换 PostgreSQL 输出
NVL(a, b) → COALESCE coalesce(a, b)
SYSDATE → CURRENT_TIMESTAMP current_timestamp
ROWNUM <= N → LIMIT N(简单 Top-N) limit N
DECODE(a,b,c,d) → CASE WHEN case when a = b then c else d end
FROM dual → 移除 (无)
FETCH FIRST N ROWS ONLY → LIMIT N limit N

12.5.4 批量迁移工具

List<String> oracleStmts = readSQLFile("oracle-app.sql");
List<String> failed = new ArrayList<>();

for (String sql : oracleStmts) {
    try {
        Queries parsed = oracle.parser().parse(sql);
        for (Query q : parsed) {
            String pgSQL = postgres.render(q);
            writeToFile("pg-migration.sql", pgSQL + ";\n");
        }
    } catch (ParserException e) {
        failed.add(sql);
        log.warn("Manual review needed: {}", sql);
    }
}
log.info("Migrated {} stmts, {} need review",
    oracleStmts.size() - failed.size(), failed.size());

Parser 局限:PL/SQL 包(PACKAGE)、存储过程体、触发器等过程式代码超出解析范围,需人工迁移。Parser 解析成功不等于业务语义等价,目标库 smoke test 必不可少。

表 12-6:陷阱明细

# 陷阱 现象 根因 修复
1 ON DUPLICATE 语义 错命中其他唯一键 MySQL 不区分 复合键集成测试

7 PG MERGE 版本差 模拟为 ON CONFLICT 12 无 MERGE 升级到 15+
8 MySQL 5.7 CTE 模拟为派生表 5.7 无 CTE 升级到 8+
9 PL/SQL 不可解析 ParserException 过程式代码 人工迁移

void limit_offset_renders_correctly_per_dialect() {

        .set(PRODUCTS.SKU, "SKU-001")
        .onDuplicateKeyUpdate()

           .set(PRODUCTS.VENDOR, "V1").set(PRODUCTS.SKU, "S2")

设计取舍

取舍 1:统一 AST + 方言分支,而非每方言独立生成器

每方言独立生成器隔离清晰但重复严重,新增 DSL 特性需改 N 套实现。jOOQ 选择统一 QueryPart AST,在 accept(Context) 内部根据方言切换分支,新特性只需在节点级别加分支。

取舍 2:Emulation 而非"最低公分母 SQL"

最低公分母放弃 PG RETURNING、MySQL upsert、Oracle 分析函数等强能力,业务无法用到方言原生优势。jOOQ 选择"原生优先,emulation 兜底",再用集成测试验证模拟语义。

取舍 3:不做"运行时检测数据库版本自动改写"

JDBC metadata 不完全可靠,版本能力还受兼容模式、权限、发行版影响。jOOQ 让用户显式声明 SQLDialect,版本差异由 SQL 快照与集成测试守住,而非运行时猜测。

取舍 4:Parser 与 Renderer 解耦

Parser 把 SQL 还原为方言无关 AST,Renderer 再渲染到任意目标方言。该设计使 Oracle → PostgreSQL 迁移、SQL 标准化、跨库重写等场景共享同一套 AST 接口。

取舍 5:VisitListener 限于观察,不替代 QOM / Settings

VisitListener 是"观察 / 局部改写"挂点,而非完整 SQL 重写器。复杂多租户应使用 Query Object Model 或 Policy(商业版),动态 schema 应使用 renderMapping。这避免 listener 成为业务逻辑黑洞。

12.10 本章要点

要点 验证依据 易错点

Emulation 不完美 集成测试 仅看 SQL 快照
