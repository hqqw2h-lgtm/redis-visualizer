第 11 章 · MULTISET 与嵌套集合

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

MULTISET 是 jOOQ 3.15+ 的核心结构化查询能力:用一条 SQL 取回任意深度的嵌套树形结构,零数据冗余,完全类型安全。它在 AST、方言渲染、结果反序列化、DTO mapper 四层协作完成"一对多"与"多层嵌套"的查询任务,使 N+1 与 JOIN 冗余两大问题同时消失。

本章按"原理 → 一对多 → 多层嵌套 → row 与 Records → 性能 → 陷阱 → 生产守卫 → 测试 → 源码深潜"展开。MULTISET 的关键洞察是:数据库返回的仍然是普通行,只是其中一列承载嵌套集合(JSON / XML / ARRAY 载体),jOOQ 在客户端反序列化为 Result<RecordN>,再由 convertFrom + Records.mapping 转为业务 DTO。

11.1 N+1 与传统解法

11.1.1 经典 N+1 时序

图 11-1-1 · 经典 N+1 时序 — 1 次父查询 + 100 次子查询 = 101 次往返

11.1.2 三种策略对照

策略 SQL 次数 网络往返 数据冗余 Java 后处理
N+1 1 + N 1 + N 无 简单但延迟爆炸
JOIN + Java groupBy 1 1 父行随子行重复 需手动分组
MULTISET 1 1 父子均不重复 jOOQ 自动还原

11.1.3 JOIN 方案的代价

var rows = dsl.select(CUSTOMERS.ID, CUSTOMERS.NAME, ORDERS.ID, ORDERS.AMOUNT)
    .from(CUSTOMERS)
    .leftJoin(ORDERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
    .fetchGroups(
        r -> r.into(CUSTOMERS),
        r -> r.into(ORDERS));
  • 父行随子行展开:CUSTOMERS 的 20 列在每个 ORDER 行重复
  • LEFT JOIN 需手动过滤"无订单客户"的空 ORDERS 行
  • 三层嵌套时数据膨胀(N × M × K)

11.2 MULTISET 原理

11.2.1 SQL 翻译路径

图 11-2-1 · SQL 翻译路径 — DSL → AST → 方言 → DB → 解析 → DTO

11.2.2 PostgreSQL 生成 SQL

SELECT
  customers.id,
  customers.name,
  (
    SELECT COALESCE(
      JSONB_AGG(JSONB_BUILD_ARRAY(orders.id, orders.amount)),
      JSONB_BUILD_ARRAY()
    )
    FROM orders
    WHERE orders.customer_id = customers.id
  ) AS orders
FROM customers

COALESCE(..., JSONB_BUILD_ARRAY()) 确保空集映射为空数组而非 NULL,jOOQ 自动注入,无需手工处理。

11.2.3 方言映射

方言 聚合函数 元素函数 备注
PostgreSQL JSONB_AGG JSONB_BUILD_ARRAY 默认 JSONB,最佳支持
MySQL 5.7.22+ JSON_ARRAYAGG JSON_ARRAY 注意 max_allowed_packet
MariaDB 10.5+ JSON_ARRAYAGG JSON_ARRAY group_concat_max_len
Oracle 12c+ JSON_ARRAYAGG JSON_ARRAY VARCHAR2 4000 上限或 CLOB
H2 2.x JSON_ARRAYAGG JSON_ARRAY 适合轻量回归
SQL Server 2016+ FOR JSON 变体 - 嵌套支持有限

11.3 一对多映射

11.3.1 基础查询

Result<Record3<Integer, String, Result<Record2<String, BigDecimal>>>> result =
    dsl.select(
        CUSTOMERS.ID,
        CUSTOMERS.NAME,
        multiset(
            select(ORDERS.ID, ORDERS.AMOUNT)
            .from(ORDERS)
            .where(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
            .orderBy(ORDERS.AMOUNT.desc())
        ).as("orders")
    )
    .from(CUSTOMERS)
    .fetch();

子查询内 orderBy 必须显式声明,否则不同方言可能返回不稳定顺序(参考 §11.7)。

11.3.2 DTO 映射

record OrdersDTO(String orderId, BigDecimal amount) {}
record CustomersDTO(Integer id, String name, List<OrdersDTO> orders) {}

List<CustomersDTO> customers = dsl.select(
        CUSTOMERS.ID,
        CUSTOMERS.NAME,
        multiset(
            select(ORDERS.ID, ORDERS.AMOUNT)
            .from(ORDERS)
            .where(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
            .orderBy(ORDERS.AMOUNT.desc())
        ).as("orders")
         .convertFrom(r -> r.map(Records.mapping(OrdersDTO::new)))
    )
    .from(CUSTOMERS)
    .fetch(Records.mapping(CustomersDTO::new));

11.3.3 输出形态

[
  CustomersDTO[id=1, name=Customer C1001,
    orders=[OrdersDTO[O1001, 128.00], OrdersDTO[O1002, 256.00]]],
  CustomersDTO[id=2, name=Customer C1002,
    orders=[OrdersDTO[O2001, 79.00]]],
  CustomersDTO[id=3, name=Customer C1003, orders=[]]
]

convertFrom 把嵌套 Result<RecordN> 转为业务 List;Records.mapping(Ctor::new) 提供编译期类型安全:构造器签名与列顺序不匹配则编译失败。

11.4 多层嵌套

11.4.1 三层 SQL 形态

SELECT
  customers.id,
  customers.name,
  (
    SELECT COALESCE(JSONB_AGG(JSONB_BUILD_ARRAY(
      o.id, o.amount,
      (
        SELECT COALESCE(JSONB_AGG(JSONB_BUILD_ARRAY(
          i.product_id, i.quantity, i.unit_price)),
          JSONB_BUILD_ARRAY())
        FROM order_items i
        WHERE i.order_id = o.id
      )
    )), JSONB_BUILD_ARRAY())
    FROM orders o
    WHERE o.customer_id = customers.id
  ) AS orders
FROM customers

11.4.2 三层 DSL

record OrderItemDTO(String productId, int quantity, BigDecimal unitPrice) {}
record OrdersWithItemsDTO(String orderId, BigDecimal amount, List<OrderItemDTO> items) {}
record CustomersFullDTO(Integer id, String name, List<OrdersWithItemsDTO> orders) {}

List<CustomersFullDTO> result = dsl.select(
    CUSTOMERS.ID,
    CUSTOMERS.NAME,
    multiset(
        select(
            ORDERS.ID,
            ORDERS.AMOUNT,
            multiset(
                select(ORDER_ITEMS.PRODUCT_ID, ORDER_ITEMS.QUANTITY, ORDER_ITEMS.UNIT_PRICE)
                .from(ORDER_ITEMS)
                .where(ORDER_ITEMS.ORDER_ID.eq(ORDERS.ID))
            ).as("items")
             .convertFrom(r -> r.map(Records.mapping(OrderItemDTO::new)))
        )
        .from(ORDERS)
        .where(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
    ).as("orders")
     .convertFrom(r -> r.map(Records.mapping(OrdersWithItemsDTO::new)))
)
.from(CUSTOMERS)
.fetch(Records.mapping(CustomersFullDTO::new));

11.4.4 嵌套深度建议

深度 评价 行为
1 层 推荐 SQL 简洁,可读
2 层 可管理 SQL 较长但清晰
3 层 注意 SQL 极长,JSON 解析成本上升
4+ 层 反模式 不可调试,拆为多次查询

嵌套深度不影响查询次数,但会显著增加 SQL 长度与 JSON 解析开销。生产环境建议 ≤ 3 层。

11.5.2 multiset vs multisetAgg

API 使用位置 数据来源 典型形态
multiset(select) SELECT 子句 独立相关子查询 父行 + 子集合
multisetAgg(f1, f2) GROUP BY 聚合 当前 FROM/JOIN 后分组 array_agg 风格 
var grouped = dsl.select(
    CUSTOMERS.ID,
    CUSTOMERS.NAME,
    multisetAgg(ORDERS.ID, ORDERS.AMOUNT)
        .as("orders")
        .convertFrom(r -> r.map(Records.mapping(OrdersDTO::new)))
)
.from(CUSTOMERS)
.join(ORDERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
.groupBy(CUSTOMERS.ID, CUSTOMERS.NAME)
.fetch();

SELECT
  customers.id,
  customers.name,
  JSONB_AGG(JSONB_BUILD_ARRAY(orders.id, orders.amount)) AS orders
FROM customers
JOIN orders ON orders.customer_id = customers.id
GROUP BY customers.id, customers.name

multisetAgg 不生成相关子查询,而是在 JOIN 分组内聚合。注意 INNER JOIN 会丢失"无订单客户";若需保留,改用 LEFT JOIN 并配合 coalesce 空集合处理。

11.5.3 综合 DTO 构建

record CustomerOrderBundle(

List<CustomerOrderBundle> bundles = dsl.select(

11.7.1 MariaDB 与 MySQL 会话参数

-- MariaDB JSON_ARRAYAGG 受 group_concat_max_len 影响
SET SESSION group_concat_max_len = 1048576;  -- 1MB

-- MySQL 大结果集需保证 packet 足够
SHOW VARIABLES LIKE 'max_allowed_packet';

应通过连接池 init SQL 或数据库参数统一设置,避免 ExecuteListener 中递归 SET SESSION。

11.7.2 Oracle CLOB 返回

SELECT JSON_ARRAYAGG(... RETURNING CLOB) FROM ...

ALTER SYSTEM SET MAX_STRING_SIZE = EXTENDED;  -- VARCHAR2 → 32767

表 11-10:源码入口

入口 包 / 类 职责
DSL DSL.multiset / multisetAgg API
AST org.jooq.impl.Multiset / MultisetAgg 节点

设计取舍

取舍 1:用 JSON 载体而非数据库原生 MULTISET

主流数据库对 SQL:2003 MULTISET 类型支持参差,JDBC 返回格式不统一。jOOQ 选择 JSON 作为跨方言传输层,在客户端恢复结构,获得"一次实现 N 个方言可用"的工程价值。

取舍 2:数组的数组而非对象数组

JSON 列名不参与反序列化,数组形态传输更小且避免每行重复 key。JSONB_BUILD_ARRAY(v0, v1, v2) 比 JSONB_BUILD_OBJECT('id', v0, 'amount', v1) 字节数显著更小。

取舍 3:嵌套深度无上限,但有性能边界

API 不限制嵌套深度,但每层 JSON 聚合都增加 SQL 长度与解析成本。生产实践应在 3 层封顶,超出改为多次查询 + Map 拼装。

取舍 4:multiset vs multisetAgg 双 API

二者语义近似但来源不同:multiset 是相关子查询(独立 FROM),multisetAgg 是分组聚合(共享 FROM)。提供两套 API 让用户按场景选择,避免单 API 强行覆盖两种用法导致语义模糊。

取舍 5:Records.mapping 编译期校验

通过 Records.mapping(Ctor::new) 让 DTO 构造器签名参与编译期类型检查,列顺序与构造器参数不匹配则编译失败,避免运行时反射 mapping 的隐蔽错误。

11.11 性能场景与监控

MULTISET 的最大魅力是消除 N+1,但任何"一条 SQL 取所有"都有上限。生产前需要量化嵌套深度、子集合大小、JSON 载体内存。

join_distinct,1k,50,join+distinct,1700,11000,520,JOIN 笛卡儿积

教学结论:
  N+1 (传统 for 循环): 10k 父对象 50 子 = 500k+1 次往返 = 数十秒
  MULTISET: 同样数据一条 SQL = 80-100 ms
  嵌套深度 ≤ 3 层为佳,4 层就开始失控
  JSON 载体 PG 表现最好 (jsonb_agg + jsonb_build_object)
  MySQL/MariaDB 注意 group_concat_max_len / GROUP_CONCAT 截断
  Oracle 大集合用 CLOB 或 XMLTYPE

11.11.2 反例:MULTISET 内部还在循环

// ❌ 错误
List<OrderSummary> orders = dsl.select(ORDERS.ID,
        multiset(select(ITEMS.ID, ITEMS.NAME).from(ITEMS).where(ITEMS.ORDER_ID.eq(ORDERS.ID)))
            .as("items"))
    .from(ORDERS)
    .fetchInto(OrderSummary.class);

for (OrderSummary o : orders) {
    // ❌ 这里又对每个订单查 customer
    Customer c = dsl.selectFrom(CUSTOMERS).where(CUSTOMERS.ID.eq(o.customerId())).fetchOneInto(Customer.class);
    o.setCustomer(c);
}
// MULTISET 解决了 ITEMS 的 N+1,但 customer 又来了 N+1

// ✅ 修复 — 把 customer 也放进 SELECT
List<OrderSummary> orders = dsl.select(
        ORDERS.ID, ORDERS.TOTAL,
        row(CUSTOMERS.ID, CUSTOMERS.NAME).convertFrom(Records.mapping(Customer::new)),
        multiset(...).as("items"))
    .from(ORDERS)
    .join(CUSTOMERS).on(CUSTOMERS.ID.eq(ORDERS.CUSTOMER_ID))
    .fetchInto(OrderSummary.class);

11.11.3 反例:MULTISET 没 orderBy

// ❌ 错误
multiset(
    select(ITEMS.ID, ITEMS.NAME)
        .from(ITEMS)
        .where(ITEMS.ORDER_ID.eq(ORDERS.ID))
).as("items")
// 测试断言依赖顺序,但 SQL 标准不保证 MULTISET 内部顺序

// ✅ 修复
multiset(
    select(ITEMS.ID, ITEMS.NAME)
        .from(ITEMS)
        .where(ITEMS.ORDER_ID.eq(ORDERS.ID))
        .orderBy(ITEMS.ID.asc())  // ✅ 显式排序
).as("items")

11.11.4 反例:嵌套过深 SQL 失控

// ❌ 错误 — 5 层嵌套
multiset(select(LEVEL1.fields(),
    multiset(select(LEVEL2.fields(),
        multiset(select(LEVEL3.fields(),
            multiset(select(LEVEL4.fields(),
                multiset(...).as("level5")
            )).as("level4")
        )).as("level3")
    )).as("level2")
)).as("level1")
// SQL 长度 > 10 KB, 解析慢, 内存爆

[修复策略]
- 限制嵌套 ≤ 3 层
- 超过 3 层用两次 query (父 + 子)
- 或重新设计 DTO 结构

11.12 设计取舍

设计点 jOOQ 选择 替代方案 后果
载体格式 JSON (PG) / JSON_ARRAYAGG (其他) XML / 自定义二进制 JSON 跨方言兼容性最好
渲染策略 各 dialect 模拟 强制 JSON 跨数据库可移植
反序列化 jOOQ 客户端 数据库直接 row 客户端 CPU + 内存压力
convertFrom Records.mapping 静态构造 反射 类型安全 + 性能好
子集合空集合 返回空 Result, 非 null null 简化业务空检查

11.13 复盘 / Recap

维度 重点
核心 MULTISET 用一条 SQL 取嵌套树形结构,消除 N+1 与 JOIN 笛卡儿积冗余;原理是数据库返回 JSON/XML 载体,客户端反序列化为嵌套 Result。
用法 multiset(select...) 子查询 + .convertFrom(Records.mapping(Type::new)) 反序列化 DTO;支持任意深度嵌套。
性能 1k-10k 父对象 50 子 ≤ 100 ms;嵌套深度 ≤ 3 层为佳;PG jsonb 表现最好,MySQL 注意 group_concat_max_len。
易错 MULTISET 内部还嵌 for 循环;子查询没 orderBy 顺序不稳;嵌套过深 SQL 失控;空集合非 null 不要错检查。
监控 multiset_query_duration / result_size / nesting_depth / truncate_total 必跟踪。
