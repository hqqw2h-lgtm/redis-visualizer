23.4 方言渲染分歧

同一棵表达式树,不同 SQLDialect 在两处产生分歧:标识符引号风格和 LIMIT 子句语法。

图 23-4:分歧决策点

图 23-4 · 方言分歧两大决策点 — 标识符 + LIMIT

表 23-4:四方言对照

方言 标识符 LIMIT 语法 LIMIT 位置
PostgreSQL "table"."column" LIMIT ? 末尾
MySQL `table`.`column` LIMIT ? 末尾
Oracle "table"."column" FETCH FIRST ? ROWS ONLY 末尾

23.4.1 四方言完整 SQL

PostgreSQL:

select "orders"."id", count(*)
from "orders"
  join "customers" on "orders"."customer_id" = "customers"."id"
where "customers"."country" = ?
group by "orders"."id"
having count(*) > ?
order by count(*) desc
limit ?

MySQL:

select `orders`.`id`, count(*)
from `orders`
  join `customers` on `orders`.`customer_id` = `customers`.`id`
where `customers`.`country` = ?
group by `orders`.`id`
having count(*) > ?
order by count(*) desc
limit ?

Oracle:

select "orders"."id", count(*)
from "orders"
  join "customers" on "orders"."customer_id" = "customers"."id"
where "customers"."country" = ?
group by "orders"."id"
having count(*) > ?
order by count(*) desc
fetch first ? rows only

SQL Server:

select top (?) [orders].[id], count(*)
from [orders]
  join [customers] on [orders].[customer_id] = [customers].[id]
where [customers].[country] = ?
group by [orders].[id]
having count(*) > ?
order by count(*) desc

23.4.2 循环输出方言 SQL

import static org.jooq.SQLDialect.*;
import org.jooq.impl.DSL;

var query = DSL.select(ORDERS.ID, count())
    .from(ORDERS)
    .join(CUSTOMERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
    .where(CUSTOMERS.COUNTRY.eq("China"))
    .groupBy(ORDERS.ID)
    .having(count().gt(3))
    .orderBy(count().desc())
    .limit(10);

for (SQLDialect d : new SQLDialect[]{POSTGRES, MYSQL, ORACLE, SQLSERVER}) {
    System.out.println("--- " + d.getName() + " ---");
    System.out.println(DSL.using(d).renderInlined(query));
    System.out.println();
}

SQL Server 特殊处理:在使用 TOP 的渲染路径里,TOP 必须紧跟在 SELECT 之后,而不是查询末尾。jOOQ 在渲染时提前检测 Limit 节点,在输出 SELECT 关键字时插入 TOP(?)。因此不要把一个方言渲染出的 SQL 和另一个方言配置下的 getBindValues() 手工拼在一起;SQL 与绑定顺序必须来自同一个 Configuration。

输出:

SQL: select ... where "customers"."country" = ? ... having count(*) > ? ... limit ?
Bind values: [China, 3, 10]
  1 -> China (varchar)
  2 -> 3 (integer)
  3 -> 10 (integer)

两趟遍历的必要性:RenderContext 与 BindContext 遍历相同树,但职责分离。Render 关心 SQL 语法(Param 输出 ?),Bind 关心值绑定(非 Param 节点直接跳过)。两趟使用相同的 DFS 序,确保绑定索引与 SQL 中 ? 位置严格一致。

23.9.1 反模式

// 错误 1 跨 Configuration 拼接
var query = DSL.using(POSTGRES).select(...).from(...).limit(10);
String sqlServerSql   = DSL.using(SQLSERVER).render(query);
List<Object> pgBinds  = query.getBindValues();

// 正确
DSLContext sqlServer = DSL.using(connection, SQLSERVER);
sqlServer.select(...).from(...).limit(10).fetch();

// 错误 2 直接执行 inlined SQL

// 正确 走 PreparedStatement

性能注意:每次 getSQL() 都触发完整 RenderContext 遍历。性能敏感循环避免反复调用;若要评估成本,记录循环次数、平均耗时(ms / µs)、查询节点规模、目标方言。如需缓存 SQL,在循环外调用一次保存结果。

class OrderProbe extends DefaultExecuteListener {
    final List<String> events = new ArrayList<>();
    @Override public void renderStart(ExecuteContext ctx)  { events.add("renderStart"); }
    @Override public void renderEnd(ExecuteContext ctx)    { events.add("renderEnd:" + ctx.sql()); }

23.12 架构设计取舍

23.12.1 为什么两趟遍历

jOOQ 完全可以在 DSL 链式调用时同步拼接 SQL,但它没有这么做。原因是 SQL 文本与参数绑定虽来自同一棵树,却服务于不同目标:前者按方言输出语法,后者按 JDBC 类型安全绑定值。拆成两趟遍历,牺牲一点即时性,换来更强的正确性边界。

设计选择 被拒绝方案 取舍
构建阶段保留 AST 链式调用直接拼字符串 AST 可重渲染为不同方言 / 参数策略
Render 与 Bind 分离 渲染时把值拼进去 保留 PreparedStatement、计划缓存、SQL 注入防护
Configuration 驱动方言 查询对象硬编码方言 同一逻辑查询可观察不同 SQL;不能跨配置混用
Listener 生命周期开放 让用户继承内部类 扩展点稳定,内部实现可演进
RecordN 类型安全映射 全部返回 Map<String,Object> 编译期知道列数 / 类型,动态投影靠 mapper
