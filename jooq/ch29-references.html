29.3 排障索引：症状、资料和第一条命令

参考章最常被打开的时刻，是“我知道 jOOQ 出问题了，但不知道查哪里”。下面这张表按事故症状组织。

症状 优先怀疑 先跑什么 输出怎么看 去哪里读
查询结果对，但线上慢 N+1、OFFSET、函数包裹索引列 SQL logging + SQL count guard 同一请求 SQL 数量是否暴涨；WHERE 是否包函数 Ch.13、Ch.25、Use The Index Luke
本地 SQL 正常，换 DB 失败 方言差异、模拟语法不支持 query.getSQL(ParamType.INLINED) + Online SQL Translator SQL 是否用了目标 DB 不支持的语法 Ch.12、Ch.22、Manual: SQL Dialect
codegen 少表或字段类型不对 includes/excludes、schema、forcedTypes generated sources check 生成目录里是否有目标表；字段 Java 类型是否符合预期 Ch.2、Manual: Code Generation
升级 jOOQ 后 SQL 变了 Settings 默认值、dialect render 策略变化 Settings snapshot + SQL snapshot test diff 是否只发生在格式，还是语义变化 Ch.15、Release notes、Javadoc Settings
事务内还是读到脏状态 transaction provider、连接来源不一致 打印 connection id / transaction boundary 同一事务是否用了不同 connection Ch.14、Manual: Transactions
批量写入慢 没用 batch、逐条 fetch、缺少 bind SQL log + JDBC driver metrics 是否每行一条 round-trip Ch.6、Ch.25、数据库 driver 文档

诊断 1：SQL logging 必须同时看 SQL 和 bind values

最小可用做法是在测试里打开 jOOQ 的 execute logging，并把查询渲染成可复制的 SQL。生产日志不要长期打开 bind 明文，排障窗口内打开即可。

import static com.acme.jooq.Tables.CUSTOMERS;
import static org.jooq.conf.ParamType.INLINED;

import org.jooq.DSLContext;
import org.jooq.Query;
import org.jooq.SQLDialect;
import org.jooq.impl.DSL;
import org.junit.jupiter.api.Test;

class SqlLoggingSmokeTest {
  @Test
  void printSqlThatWillBeSentToDatabase() {
    DSLContext dsl = DSL.using(SQLDialect.POSTGRES);

    Query query = dsl.select(CUSTOMERS.ID, CUSTOMERS.NAME)
        .from(CUSTOMERS)
        .where(CUSTOMERS.EMAIL.eq("Orwell"))
        .limit(5);

    System.out.println(query.getSQL());
    System.out.println(query.getBindValues());
    System.out.println(query.getSQL(INLINED));
  }
}

期望输出要成对保存到 PR 或测试快照里，不能只说“SQL 看起来没问题”。

select CUSTOMERS.ID, CUSTOMERS.NAME
from CUSTOMERS
where CUSTOMERS.EMAIL = ?
limit ?

[Orwell, 5]

select CUSTOMERS.ID, CUSTOMERS.NAME
from CUSTOMERS
where CUSTOMERS.EMAIL = 'Orwell'
limit 5

判断动作：

看到什么 说明 下一步
SQL 数量比预期多很多 可能是 N+1 或循环内 fetch 加 SQL count assertion，回到 Ch.25
? bind 正常但 inline SQL 报错 可能是字符串转义或方言差异 用目标数据库直接执行 inline SQL
limit ? 在某些 DB 不支持 方言或驱动限制 查 jOOQ SQL Translator 和目标 DB 文档

诊断 2：Settings snapshot 防止“默认值漂移”

jOOQ 的 Settings 会影响 SQL 渲染、大小写、schema 映射、参数类型和执行行为。升级 jOOQ 前后，先把关键配置打印出来。

import org.jooq.SQLDialect;
import org.jooq.conf.ParamType;
import org.jooq.conf.RenderQuotedNames;
import org.jooq.conf.Settings;
import org.jooq.impl.DSL;
import org.junit.jupiter.api.Test;

class JooqSettingsSnapshotTest {
  @Test
  void printEffectiveSettingsForOrderItem() {
    Settings settings = new Settings()
        .withRenderSchema(false)
        .withRenderQuotedNames(RenderQuotedNames.EXPLICIT_DEFAULT_UNQUOTED)
        .withParamType(ParamType.INDEXED);

    var dsl = DSL.using(SQLDialect.POSTGRES, settings);

    System.out.println("dialect=" + dsl.dialect());
    System.out.println("renderSchema=" + dsl.settings().isRenderSchema());
    System.out.println("renderQuotedNames=" + dsl.settings().getRenderQuotedNames());
    System.out.println("paramType=" + dsl.settings().getParamType());
  }
}

把输出作为升级前基线：

dialect=POSTGRES
renderSchema=false
renderQuotedNames=EXPLICIT_DEFAULT_UNQUOTED
paramType=INDEXED

如果升级后输出变成下面这样，就不是“格式变化”，而是生产 SQL 可能会变：

dialect=POSTGRES
renderSchema=true
renderQuotedNames=ALWAYS
paramType=INLINED

判断动作：

变化项 生产风险 守护方式
renderSchema SQL 指向不同 schema 或权限失败 固定 Settings，并在 SQL snapshot 中断言不带 schema
renderQuotedNames 大小写敏感对象名突然失效 对关键表名做 snapshot
paramType bind 变 inline，影响计划缓存和日志敏感信息 CI 中断言 getSQL() 仍包含 ?

诊断 3：generated sources check 比“编译过了”更具体

codegen 问题不要只靠 IDE 能不能补全。用脚本检查生成目录里是否真的有目标表和字段。

$generated = "target/generated-sources/jooq"
$mustHave = @(
  "tables/Customers.java",
  "tables/Orders.java",
  "tables/records/CustomersRecord.java",
  "tables/records/OrdersRecord.java"
)

$missing = $mustHave | Where-Object {
  -not (Test-Path (Join-Path $generated $_))
}

if ($missing) {
  Write-Error ("Missing generated jOOQ sources: " + ($missing -join ", "))
  exit 1
}

Write-Host "Generated jOOQ sources OK:"
$mustHave | ForEach-Object { Write-Host " - $_" }

正常输出：

Generated jOOQ sources OK:
 - tables/Customers.java
 - tables/Orders.java
 - tables/records/CustomersRecord.java
 - tables/records/OrdersRecord.java

异常输出：

Missing generated jOOQ sources: tables/Orders.java, tables/records/OrdersRecord.java

判断动作：

缺什么 常见根因 下一步
整张表缺失 <includes> 没匹配、schema 错、迁移未执行 打印 codegen JDBC URL 和 schema
Record 缺失 generator 策略或配置禁用了 records 查 <generate><records>
字段 Java 类型不对 forcedTypes 正则没命中 打印数据库元数据列类型

29.5 常见陷阱表：参考资料看对了，问题才能修对

陷阱 现象 根因 一行修复
只看 inline SQL，不看 bind values 本地复制 SQL 能跑，应用里慢或错 inline SQL 改变了参数类型、转义和计划缓存行为 同时保存 getSQL()、getBindValues()、getSQL(INLINED)
只读 jOOQ Manual，不读目标数据库文档 jOOQ 能生成 SQL，但数据库执行失败 jOOQ 负责渲染，不保证数据库版本支持所有能力 用 Online SQL Translator + 目标 DB 手册双查
codegen 漏表时先改 Java 代码 DSL 里找不到表或字段 schema、migration、includes/excludes 才是源头 先跑 generated sources check
升级 jOOQ 只跑单元测试 SQL 文本、Settings 默认值悄悄变化 单元测试 mock 了数据库，不覆盖渲染差异 加 SQL snapshot 和 Settings snapshot
把 ExecuteListener 当业务日志 日志量暴涨，敏感参数泄露 SQL 诊断与审计日志混用 排障窗口开启，生产默认脱敏和限采样
相信 EXPLAIN 的估算等于真实耗时 执行计划看着好，线上仍慢 统计信息、缓存、参数分布会影响真实执行 同时看 actual plan、p95、SQL count
在循环里临时写 fetchOne() 小数据没问题，大数据 N+1 DSL 太方便，容易把 round-trip 藏进循环 对服务层加 SQL 数量断言
所有资料都收藏，但没有本地最小复现 issue 来回讨论，没人能重现 缺 SQL、DDL、Settings、dialect 版本 issue 模板固定四件套：DDL、DSL、SQL、输出

29.6 回归守护：把 References 变成 CI 门禁

参考资料告诉你“应该怎么做”，回归守护保证下一次改动不会忘。

守护 1：关键 SQL snapshot

import static com.acme.jooq.Tables.ORDERS;
import static org.assertj.core.api.Assertions.assertThat;
import static org.jooq.conf.ParamType.INDEXED;

import org.jooq.SQLDialect;
import org.jooq.impl.DSL;
import org.junit.jupiter.api.Test;

class OrdersQuerySnapshotTest {
  @Test
  void activeOrdersQueryShapeDoesNotDrift() {
    var dsl = DSL.using(SQLDialect.POSTGRES);

    var query = dsl.select(ORDERS.ID, ORDERS.ID)
        .from(ORDERS)
        .where(ORDERS.DELETED_AT.isNull())
        .orderBy(ORDERS.ID.desc())
        .limit(20);

    assertThat(query.getSQL(INDEXED)).isEqualTo("""
        select "orders"."id", "orders"."status"
        from "orders"
        where "orders"."deleted_at" is null
        order by "orders"."id" desc
        limit ?
        """.trim());

    assertThat(query.getBindValues()).containsExactly(20);
  }
}

期望测试输出：

OrdersQuerySnapshotTest > activeOrdersQueryShapeDoesNotDrift() PASSED

如果升级 jOOQ 后 SQL 变成 offset pagination、schema 前缀变化、字段被 quote 策略影响，测试会先于生产发现。

守护 2：SQL count assertion 防 N+1

class SqlCounterListener extends org.jooq.impl.DefaultExecuteListener {
  private int count;

  @Override
  public void executeStart(org.jooq.ExecuteContext ctx) {
    count++;
  }

  int count() {
    return count;
  }
}

测试里把 listener 注入 Configuration，然后断言一个服务方法的 SQL 数量：

@Test
void loadingCustomerWithOrdersDoesNotBecomeNPlusOne() {
  SqlCounterListener counter = new SqlCounterListener();
  DSLContext dsl = dslWith(counter);

  authorService.loadCustomerWithOrders(dsl, 50);

  assertThat(counter.count()).isLessThanOrEqualTo(2);
}

期望输出：

loadingCustomerWithOrdersDoesNotBecomeNPlusOne() PASSED
SQL count: 2

失败输出的行动含义：

Expected SQL count <= 2 but was 51

这时不要去翻十篇博客，先回到 Ch.25 的 N+1/MULTISET/批量查询方案。

守护 3：codegen 目录进入构建门禁

把 §29.3 的 generated sources check 放到 Maven/Gradle verify 阶段，目标不是“检查文件存在”本身，而是确认数据库迁移、codegen 配置、Java 编译三者同步。

守护 防什么 触发时机
SQL snapshot jOOQ 升级或 Settings 改动导致 SQL 形状变化 每次 PR
SQL count assertion 服务层循环查询、隐式 N+1 Repository/Service 测试
generated sources check migration/codegen 不同步 generate-sources 后
Settings snapshot 默认值漂移、环境覆盖 应用启动 smoke test
dialect snapshot 本地 DB 与生产 DB 方言不一致 多数据库支持模块

29.9.1 Codd 1970 与 jOOQ 类型系统

Codd 1970 把关系代数定义为"基于域的运算" (domain-based)。在 jOOQ 里这直接体现为 Field<T> 而非 String name:

// 不是
String orderId = "id";

// 而是
Field<Long> orderId = ORDERS.ID;  // T = Long, 编译期类型可信

读懂 Codd 之后,你不会再问"为什么不能直接用字符串列名" — 那相当于丢掉关系代数的域,把类型安全交还给运行期。

29.9.2 SEQUEL 1974 与 DSL 词序

SEQUEL (后来改名 SQL) 选择 SELECT-FROM-WHERE 子句序,因为它接近自然语言。jOOQ DSL 严格保持这个词序而不是任意 fluent:

// jOOQ DSL 强制词序
dsl.select(...)
   .from(...)
   .where(...)
   .orderBy(...)
   .limit(...);

// 编译器拒绝:
// dsl.from(...).select(...);  // 不允许颠倒 SELECT/FROM

这是 §11 章 / §18 章 QueryPart 体系的根本约束:SelectFromStep 必须先于 SelectWhereStep,在编译期保证 SQL 子句顺序合法。

29.9.3 LINQ 2006 与 Field 泛型

LINQ 把"查询作为表达式树"引入主流类型系统。Java 没有表达式树,但泛型可以承担类似职责:

Condition c = ORDERS.AMOUNT.gt(BigDecimal.valueOf(100))   // Field<BigDecimal>.gt(BigDecimal)
                  .and(ORDERS.STATUS.in("PAID", "SHIPPED"));  // Field<String>.in(String...)
// 编译器拒绝:
// ORDERS.AMOUNT.in("hello")  // Field<BigDecimal>.in(String) 不通过

Type erasure 让运行期看不到 T,但编译期已经把所有非法组合拒绝。这是 §4 章类型系统的核心,也是 jOOQ 与 JdbcTemplate / MyBatis 的本质区别。

29.9.4 Generative Programming 与 codegen 决策

Czarnecki 的论点:运行期反射不可避免有性能、可读性、IDE 友好性损失;能在编译期生成的就不要运行期生成。jOOQ 把整个表元数据生成为 Java 代码,代价是构建依赖数据库,收益是 IDE 补全、refactor 跟随、编译期 schema 校验。

这也解释了 §29.4 生产检查清单为什么把 generated sources check 列为强制项:codegen 是 jOOQ 的核心承诺,丢了它整个类型安全链路就断了。

29.10 排障案例集:从真实事故看资料怎么用

§29.3 给了症状索引表,但症状到资料还隔着一层判断。这一节挑 6 个真实事故,每个完整走一遍"症状 → 证据 → 资料 → 修复 → 守护"。

案例 1:Repository 加了 N+1 但小数据没暴露

[症状]
- 测试环境 100 个订单,Repository 调用 200 ms
- 生产环境 5000 个订单,同一调用 12 秒
- 应用 logs 没有 ERROR

[证据收集]
- 加 SqlCounterListener (§29.6 守护 2)
- 跑测试: SQL count = 5001
- 看 SQL log: 1 条 select orders + 5000 条 select customers

[查资料]
- 第一站: §29.3 排障索引 → "查询结果对但慢" → N+1 嫌疑
- 第二站: Ch.11 MULTISET (一对多嵌套查询)
- 第三站: Ch.25 §25.x N+1 模式

[修复]
原代码:
  for (Order o : orders) { o.setItems(itemRepo.findByOrderId(o.getId())); }

修复:
  dsl.select(ORDERS.fields(),
       multiset(select(ITEMS.fields()).from(ITEMS).where(ITEMS.ORDER_ID.eq(ORDERS.ID))).as("items")
     ).from(ORDERS).fetch(...)

[守护]
- 测试加 assertThat(counter.count()).isLessThanOrEqualTo(2);

案例 2:升级 jOOQ 3.18 → 3.19 后 SQL 突然带 schema 前缀

[症状]
- 升级当晚, 部分查询报 "permission denied for schema public"
- 同一应用在 staging 环境一切正常

[证据收集]
- 用 §29.3 诊断 1 打印查询 SQL
- 升级前: select "orders"."id" from "orders"
- 升级后: select "public"."orders"."id" from "public"."orders"

[根因]
- jOOQ 3.19 把 RenderSchema 的某个默认值翻转
- staging 用的应用启动账号有 public schema 权限,生产没有

[修复]
- Settings.withRenderSchema(false) 显式关闭
- 不要相信 jOOQ 默认值跨大版本一致

[守护]
- 加 §29.6 守护 1: SQL snapshot 测试
- 加 Settings snapshot 测试 (§29.3 诊断 2)

案例 3:codegen 漏一张表,Service 编译却过

[症状]
- migration 加了一张 audit_logs 表
- DSL 写 dsl.selectFrom(AUDIT_LOGS) 报 "cannot find symbol"
- 重跑 mvn generate-sources 仍然没有

[证据收集]
- 跑 §29.3 诊断 3: generated sources check
- 输出: Missing generated jOOQ sources: tables/AuditLogs.java
- 看 codegen 配置: <includes>orders|customers|items</includes>

[根因]
- includes 是白名单, audit_logs 不在
- migration 加表时没改 codegen 配置

[修复]
- includes 加 |audit_logs
- 或改 .* 通配符 (注意排除系统表)

[守护]
- generated sources check 进 CI verify 阶段 (§29.4)

案例 4:同一事务内 fetch 看不到自己 insert 的行

[症状]
- service 方法内: 先 dsl.insertInto.execute(), 然后 dsl.selectFrom.fetch()
- fetch 没看到刚 insert 的行
- 同一方法标了 @Transactional

[证据收集]
- 打印 connection 标识符 (System.identityHashCode(connection))
- 发现 insert 走 connection-A, fetch 走 connection-B

[根因]
- DSLContext 用了独立 DataSourceConnectionProvider
- 没用 SpringTransactionProvider, 不参与 Spring 事务
- 两次操作分别从连接池借连接,看到不同的事务隔离视图

[修复]
- @Bean 配置 DSLContext 时注入 SpringTransactionProvider
- 或者用 dsl.transaction(...) 替代 @Transactional

[资料]
- §29.3 排障索引 "事务内还是读到脏状态"
- Ch.14 Transactions
- Ch.15 Spring 集成

案例 5:本地 H2 一切正常,部署 PG 报 "function regexp_like does not exist"

[症状]
- DSL: WHERE field("regexp_like(name, ?)", "...^.*$")
- 本地 H2 测试通过
- 部署 PostgreSQL 报错

[证据收集]
- §29.3 诊断 1: query.getSQL(INLINED)
- 看到原始 SQL 直接是 regexp_like(name, '...')
- regexp_like 是 Oracle/H2 函数, PG 用 ~ 或 SIMILAR TO

[修复 1: 用 jOOQ DSL 抽象]
NAME.likeRegex("...^.*$")  // jOOQ 自动按 dialect 渲染

[修复 2: 改 SQL]
field("name ~ ?", "...^.*$")  // PG 语法

[守护]
- 多 dialect SQL snapshot 测试 (Ch.12, Ch.22)
- 不再用 plain SQL 字符串绕开 jOOQ DSL 抽象

案例 6:CRUD 模板 update 把没改的字段也写回了 NULL

[症状]
- patch API: 只更新一个字段
- 数据库其他字段被覆盖为 NULL

[根因]
- new 出 OrdersRecord, 只 set 了一个字段就 store()
- jOOQ 默认更新所有列,因为 changed 标志位都没设

[证据]
- 看生成的 UPDATE SQL: SET name=?, address=?, phone=? WHERE id=?
- 应该只: SET name=? WHERE id=?

[修复]
方式 1: fetch + modify
  OrdersRecord r = dsl.selectFrom(ORDERS).where(ORDERS.ID.eq(id)).fetchOne();
  r.setName(newName);
  r.store();  // 只更新 changed 字段

方式 2: 显式 update DSL
  dsl.update(ORDERS).set(ORDERS.NAME, newName).where(ORDERS.ID.eq(id)).execute();

[资料]
- Ch.9 Record 与 changed 标志位
- jOOQ Manual: UpdatableRecord

模板 1:SqlCounterListener 完整版

import org.jooq.ExecuteContext;
import org.jooq.impl.DefaultExecuteListener;
import java.util.concurrent.atomic.AtomicInteger;

public final class SqlCounterListener extends DefaultExecuteListener {
    private final AtomicInteger count = new AtomicInteger();
    private final AtomicInteger errorCount = new AtomicInteger();

    @Override
    public void executeStart(ExecuteContext ctx) {
        count.incrementAndGet();
    }

    @Override
    public void exception(ExecuteContext ctx) {
        errorCount.incrementAndGet();
    }

    public int count() { return count.get(); }
    public int errorCount() { return errorCount.get(); }
    public void reset() { count.set(0); errorCount.set(0); }
}

// 在测试 Configuration 注入
DSLContext dslWith(SqlCounterListener counter) {
    Configuration config = baseConfig.derive(
        new DefaultExecuteListenerProvider(counter));
    return DSL.using(config);
}

import org.assertj.core.api.SoftAssertions;

        soft.assertAll();
    }

模板 3:慢查询监控 Listener

public final class SlowQueryListener extends DefaultExecuteListener {

    private static final long SLOW_THRESHOLD_MS = 200;
    private static final ThreadLocal<Long> START = new ThreadLocal<>();
    private final MeterRegistry registry;

    @Override
    public void executeStart(ExecuteContext ctx) {
        START.set(System.nanoTime());
    }

    @Override
    public void executeEnd(ExecuteContext ctx) {
        long startNanos = START.get() != null ? START.get() : 0;
        long elapsedMs = (System.nanoTime() - startNanos) / 1_000_000;
        START.remove();

        registry.timer("jooq.query.duration",
            "type", ctx.type().name().toLowerCase()).record(Duration.ofMillis(elapsedMs));

        if (elapsedMs > SLOW_THRESHOLD_MS) {
            registry.counter("jooq.query.slow",
                "type", ctx.type().name().toLowerCase()).increment();
            log.warn("slow jooq query elapsed_ms={} sql={}",
                elapsedMs, ctx.sql());
        }
    }
}
