第 6 章 · DML 操作

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

DML 决定数据库状态的演化路径。jOOQ 的 DML API 不是把 INSERT / UPDATE / DELETE / MERGE 包装为四个方法,而是把"写哪张表、改哪些列、带不带条件、要不要取回结果"分解为可检查、可渲染、可复用的 DSL 链。每条链在编译期约束语法,在运行期渲染为目标方言 SQL,在执行期返回受影响行数或 RETURNING 子句的记录。

本章按"心智模型 → 四条分支 → 写后读 → Record CRUD → 陷阱 → 源码"展开。SQL 片段为提升可读性多采用 renderInlined() 风格,真实执行时 jOOQ 默认使用 bind variables 与 PreparedStatement。

6.4.2 基础 UPDATE 与行数断言

int rows = dsl.update(ORDERS)
    .set(ORDERS.AMOUNT, new BigDecimal("49.90"))
    .set(ORDERS.UPDATED_AT, currentTimestamp())

行数断言是 UPDATE 的最低安全门。即使开启了乐观锁,也应在业务层显式比较 rows 与预期,避免静默成功。

    .set(ORDERS.CUSTOMER_NAME,
        select(CUSTOMERS.NAME)
            .from(CUSTOMERS)
            .where(CUSTOMERS.ID.eq(ORDERS.CUSTOMER_ID)))
    .where(ORDERS.CUSTOMER_NAME.isNull())
    .execute();

set customer_name = (
  select customers.name from customers
  where customers.id = orders.customer_id
)
where orders.customer_name is null

    .set(ORDERS.DISCOUNT_RATE, new BigDecimal("0.15"))
    .from(CUSTOMERS)
    .where(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))

6.6 MERGE 与 UPSERT

6.6.1 UPSERT 状态机

图 6-6-1 · UPSERT 状态机 — 比较唯一键 → INSERT 或 UPDATE

6.6.2 PostgreSQL ON CONFLICT

int rows = dsl.insertInto(ORDERS, ORDERS.ISBN, ORDERS.TITLE, ORDERS.AMOUNT)
    .values("978-1-234", "jOOQ Field Guide", new BigDecimal("59.90"))
    .onConflict(ORDERS.ISBN)
    .doUpdate()
    .set(ORDERS.TITLE,  DSL.excluded(ORDERS.TITLE))
    .set(ORDERS.AMOUNT, DSL.excluded(ORDERS.AMOUNT))
    .execute();

insert into orders (isbn, title, amount)
values (?, ?, ?)
on conflict (isbn) do update
set title = excluded.title,
    amount = excluded.amount

6.6.3 MySQL ON DUPLICATE KEY

int rows = dsl.insertInto(ORDERS, ORDERS.ISBN, ORDERS.TITLE, ORDERS.AMOUNT)
    .values("978-1-234", "jOOQ Field Guide", new BigDecimal("59.90"))
    .onDuplicateKeyUpdate()
    .set(ORDERS.TITLE,  DSL.excluded(ORDERS.TITLE))
    .set(ORDERS.AMOUNT, DSL.excluded(ORDERS.AMOUNT))
    .execute();

-- MySQL 经典语法,jOOQ 可按版本渲染为新语法
insert into orders (isbn, title, amount)
values (?, ?, ?)
on duplicate key update
  title = values(title),
  amount = values(amount)

6.6.4 标准 MERGE(Oracle / SQL Server)

var src = select(val("978-1-234").as("isbn"),
                 val("jOOQ Field Guide").as("title"))
    .asTable("src");

int rows = dsl.mergeInto(ORDERS)
    .using(src)
    .on(ORDERS.ISBN.eq(field(name("src", "isbn"), String.class)))
    .whenMatchedThenUpdate()
        .set(ORDERS.TITLE, field(name("src", "title"), String.class))
    .whenNotMatchedThenInsert(ORDERS.ISBN, ORDERS.TITLE)
        .values(field(name("src", "isbn"),  String.class),
                field(name("src", "title"), String.class))
    .execute();

-- Oracle
merge into orders
using (select '978-1-234' isbn, 'jOOQ Field Guide' title from dual) src
on (orders.isbn = src.isbn)
when matched then update set title = src.title
when not matched then insert (isbn, title) values (src.isbn, src.title)

关键反例与正例

// 反例:UPSERT 覆盖创建时间
.onConflict(ORDERS.ISBN)
.doUpdate()
.set(ORDERS.TITLE, excluded(ORDERS.TITLE))
.set(ORDERS.CREATED_AT, excluded(ORDERS.CREATED_AT));

// 正例:仅更新可变字段
.onConflict(ORDERS.ISBN)
.doUpdate()
.set(ORDERS.TITLE, excluded(ORDERS.TITLE))
.set(ORDERS.AMOUNT, excluded(ORDERS.AMOUNT))
.set(ORDERS.UPDATED_AT, currentTimestamp());

// 反例:行数判断 UPSERT 分支
int rows = upsertOrder(...);
if (rows == 1) log("inserted");   // MySQL 错
else if (rows == 2) log("updated");

// 正例:returning + 业务键查询
OrdersRecord saved = upsertReturningOrder(...);
boolean wasInsert = saved.getCreatedAt().equals(saved.getUpdatedAt());

// 反例:乐观锁异常被吞
try {
    rec.store();
} catch (Exception e) {
    log.warn("ignore");   // 业务覆盖
}

// 正例:捕获 + 重试 / 提示
try {
    rec.store();
} catch (DataChangedException ex) {
    OrdersRecord fresh = dsl.fetchOne(ORDERS, ORDERS.ID.eq(rec.getId()));
    return mergeAndRetry(fresh, rec);
}

设计取舍

取舍 1:四类 DML 分开建模

INSERT / UPDATE / DELETE / MERGE 分别拥有独立的 *QueryImpl,而非统一为通用 MutationQuery。该设计使各 DML 的方言差异隔离,但带来分支多、需独立测试的代价。

取舍 2:RETURNING 抽象统一

returning(...) 在 API 层是一个方法,内部根据方言能力选择 SQL RETURNING / OUTPUT 或退化为 getGeneratedKeys + SELECT。该抽象屏蔽了方言差异,但生产代码不应假设所有方言行为完全一致。

取舍 3:DML 与 SELECT 共用渲染引擎

DML 仍是 QueryPart。安全策略(WHERE 守护)、监听器(ExecuteListener)、SQL snapshot 测试可在读写两端共用。该统一性是 jOOQ 测试基础设施的根基。

取舍 4:UpdatableRecord 与 DSL 共存

store() 提供 ORM 风格的 active record,而 DSL 提供完全控制的语句构建。两者并不互斥,而是覆盖不同复杂度:简单 CRUD 用 Record,复杂批量与跨表写入用 DSL。

TRUNCATE 是 DDL 触发器 / 回滚差异 当 DELETE 用
store() 适合简单 CRUD changed 标志位 复杂写入应用 DSL
