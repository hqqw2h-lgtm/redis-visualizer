第 5 章 · SELECT 查询构建

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

SELECT 是 SQL 中最复杂的语句,也是 jOOQ 类型系统价值集中体现的场景。jOOQ 的 SELECT API 通过 Step 接口链实现类型驱动的状态机:每一步返回更具体的中间类型,编译器在每步约束下一步合法操作。该设计既不是 Builder 模式的随意拼接,也不是字符串拼接的延后报错,而是把 SQL 子句顺序固化为类型签名。

本章按"流水线 → 条件 → JOIN → 排序与分页 → 子查询 → 动态构建 → 陷阱 → 源码"的结构展开。所有示例使用三表 schema:customers(用户)、orders(订单)、products(商品),覆盖单表查询、多表关联、子查询、动态过滤、Keyset 分页等典型场景。

5.1 SELECT 构建流水线

5.1.1 Step 接口链

图 5-1-1 · SELECT Step 接口链 — 类型驱动的状态机

每个 Step 接口仅暴露当前状态合法的下一步方法。SelectSelectStep 不能直接调用 .where(),因为 FROM 子句尚未指定。该约束使 SQL 子句顺序在编译期被检查,无需运行时校验。

DSL.and / or 对空集合返回布尔常量,使空过滤器不破坏 SQL 语义。该对称性是动态查询构建的基石。

from orders

5.5.5 EXISTS 与 IN 选型

维度 EXISTS IN
短路 找到第一条即停 需扫描结果集
NULL 语义 三值逻辑安全 NULL 处理需注意
优化器重写 通常已优化 现代 DB 自动转半连接
大子查询 优势明显 视优化器
小子查询 等价 表达直观

PostgreSQL 12+ 与 MySQL 8.0+ 的优化器能将 IN 子查询自动重写为半连接,两者性能差异在多数场景下接近。EXISTS 在大结果集与相关子查询场景仍具优势。

5.6.2 动态 WHERE 模式

public Result<CustomersRecord> searchUsers(
        String name, String email, String role,
        Integer minAge, Integer maxAge) {

    List<Condition> conditions = new ArrayList<>();
    if (name != null)   conditions.add(CUSTOMERS.NAME.containsIgnoreCase(name));
    if (email != null)  conditions.add(CUSTOMERS.EMAIL.eq(email));
    if (role != null)   conditions.add(CUSTOMERS.TIER.eq(role));
    if (minAge != null) conditions.add(CUSTOMERS.AGE.ge(minAge));
    if (maxAge != null) conditions.add(CUSTOMERS.AGE.le(maxAge));

    return ctx.selectFrom(CUSTOMERS)
              .where(conditions)
              .orderBy(CUSTOMERS.NAME)
              .fetch();
}

5.6.3 动态 JOIN 模式

SelectJoinStep<Record> query = ctx.select(ORDERS.fields())

5.6.4 动态排序模式

SortField<?> sortField = switch (sortBy) {
    case "name"  -> asc ? CUSTOMERS.NAME.asc()  : CUSTOMERS.NAME.desc();
    case "age"   -> asc ? CUSTOMERS.AGE.asc()   : CUSTOMERS.AGE.desc();
    case "email" -> asc ? CUSTOMERS.EMAIL.asc() : CUSTOMERS.EMAIL.desc();
    default      -> CUSTOMERS.ID.asc();
};

ctx.selectFrom(CUSTOMERS)
   .where(conditions)
   .orderBy(sortField)
   .limit(pageSize)
   .fetch();

switch 白名单是动态排序的安全模式。直接把前端字段名传给 field(userInput) 会引入 SQL 注入风险,应避免。

5.6.5 noCondition 透明条件

ctx.selectFrom(CUSTOMERS)
   .where(
       name != null ? CUSTOMERS.NAME.like(name) : noCondition(),
       role != null ? CUSTOMERS.TIER.eq(role)   : noCondition())
   .fetch();

DSL.noCondition() 在最终 SQL 中不渲染,不影响 AND / OR 语义。该机制使三元表达式风格的动态条件无需 if 分支。

        .where(ORDERS.CUSTOMER_ID.eq(c.getId()))

           .where(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))

5.8 SELECT 测试护栏

5.8.1 SQL Snapshot + bind

@Test
void recent_orders_query_keeps_sql_and_bind() {
    var query = dsl
        .select(ORDERS.ID, ORDERS.STATUS, ORDERS.AMOUNT)
        .from(ORDERS)
        .join(CUSTOMERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
        .where(CUSTOMERS.COUNTRY.eq("CN"))
        .and(ORDERS.AMOUNT.gt(new BigDecimal("50.00")))
        .orderBy(ORDERS.ID.asc())
        .seek(100L)
        .limit(20);

    assertThat(query.getSQL(ParamType.INDEXED))
        .isEqualToNormalizingWhitespace("""
            select orders.id, orders.status, orders.amount
            from orders
            join customers on orders.customer_id = customers.id
            where customers.country = ?
              and orders.amount > ?
              and orders.id > ?
            order by orders.id asc
            limit ?
            """);

    assertThat(query.getBindValues())
        .containsExactly("CN", new BigDecimal("50.00"), 100L, 20);
}

设计取舍

取舍 1:Step 接口与可变 SelectQueryImpl

Step 接口提供编译期类型导航,SelectQueryImpl 提供运行期可变状态。该双层设计使流畅 DSL 与动态构建在同一对象图上协作:fluent API 隐藏可变性,动态 API 显式利用可变性。

取舍 2:Condition 一等公民

Condition 不依附于查询对象,可独立创建、组合、传递、测试。该设计使动态 WHERE 不依赖字符串拼接,业务层可以把条件作为参数在多个查询间复用。

取舍 3:隐式连接默认 LEFT JOIN

隐式连接渲染为 LEFT JOIN 而非 INNER JOIN。理由:可空外键场景下 INNER JOIN 会过滤主表行,违反"路径导航不应改变主表行数"的直觉。需 INNER JOIN 时仍应显式 .join().on()。
