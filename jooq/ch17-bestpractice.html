前 16 章覆盖了 jOOQ 的全部主要特性。"会用"和"用好"之间还有一道鸿沟:连接泄漏、方言差异、架构争论。本章把前 16 章的经验浓缩成一份可操作的避坑指南与最佳实践手册。

最佳实践不是孤立规则,而是对 jOOQ 架构边界的尊重:schema 变化交给 codegen,SQL 表达交给 QueryPart 树,执行观测交给 listener,类型边界交给 Converter / Binding,事务边界交给 TransactionProvider 或外部事务管理器。团队踩坑,通常是把一个边界的责任塞进了另一个边界。

17.1.1 JPA → jOOQ 等价写法

// JPQL: 字符串拼接,无类型安全;DISTINCT 避免一张订单因多明细重复
@Query("SELECT DISTINCT o FROM Order o JOIN o.items i " +
       "WHERE o.status = :status AND i.price > :minAmount")
List<Order> findByStatusAndMinPrice(@Param("status") String status,
                                     @Param("minAmount") BigDecimal minAmount);

// Criteria API: 类型安全但极度冗长
CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<Order> cq = cb.createQuery(Order.class);
Root<Order> root = cq.from(Order.class);
Join<Order, Item> items = root.join("items");
cq.select(root).distinct(true);
cq.where(cb.and(
    cb.equal(root.get("status"), status),
    cb.gt(items.<BigDecimal>get("price"), minAmount)));
return em.createQuery(cq).getResultList();

// jOOQ DSL: 类型安全 + SQL 原生表达力;EXISTS 避免 JOIN 放大父表行数
dsl.selectFrom(ORDERS)
   .where(ORDERS.STATUS.eq(status))
   .andExists(selectOne()
       .from(ITEMS)
       .where(ITEMS.ORDER_ID.eq(ORDERS.ID))
       .and(ITEMS.PRICE.gt(minAmount)))
   .fetchInto(OrderDto.class);

// 动态条件
Condition cond = DSL.noCondition();
if (status != null)    cond = cond.and(ORDERS.STATUS.eq(status));
if (minAmount != null) cond = cond.andExists(selectOne()
        .from(ITEMS)
        .where(ITEMS.ORDER_ID.eq(ORDERS.ID))
        .and(ITEMS.PRICE.gt(minAmount)));
dsl.selectFrom(ORDERS).where(cond).fetchInto(OrderDto.class);

17.2.9 陷阱 9:枚举迁移

图 17-2-9 · 枚举迁移陷阱 — DBA 新增枚举值,旧应用解析失败
public final class OrderStatusConverter implements Converter<String, OrderStatus> {
    private static final Logger log = LoggerFactory.getLogger(OrderStatusConverter.class);

    @Override public OrderStatus from(String dbVal) {
        if (dbVal == null) return null;
        try { return OrderStatus.valueOf(dbVal); }
        catch (IllegalArgumentException e) {
            log.warn("Unknown status: {}", dbVal);
            return OrderStatus.UNKNOWN;
        }
    }

    @Override public String to(OrderStatus userVal) {
        return userVal == null || userVal == OrderStatus.UNKNOWN ? null : userVal.name();
    }

    @Override public Class<String>      fromType() { return String.class; }
    @Override public Class<OrderStatus> toType()   { return OrderStatus.class; }
}

UNKNOWN 仅作为读取旧数据的兜底,不应写回数据库;写入前业务层应拒绝未知枚举。

17.4.1 Repository 模式

@Repository
public class OrderRepository {
    private final DSLContext dsl;

    public OrderRepository(DSLContext dsl) { this.dsl = dsl; }

    public List<OrderWithItems> findWithItems(OrderStatus status) {
        return dsl.select(
                ORDERS.ID, ORDERS.CREATED_AT,
                multiset(
                    select(ITEMS.NAME, ITEMS.PRICE, ITEMS.QTY)
                    .from(ITEMS)
                    .where(ITEMS.ORDER_ID.eq(ORDERS.ID))
                ).as("items").convertFrom(r -> r.into(ItemDto.class)))
            .from(ORDERS)
            .where(ORDERS.STATUS.eq(status))
            .fetchInto(OrderWithItems.class);
    }

    public List<OrderDto> search(OrderSearchCriteria c) {
        Condition cond = DSL.noCondition();
        if (c.status()   != null) cond = cond.and(ORDERS.STATUS.eq(c.status()));
        if (c.minTotal() != null) cond = cond.and(ORDERS.TOTAL.ge(c.minTotal()));
        if (c.dateFrom() != null) cond = cond.and(ORDERS.CREATED_AT.ge(c.dateFrom()));
        return dsl.selectFrom(ORDERS)
                  .where(cond)
                  .orderBy(ORDERS.CREATED_AT.desc())
                  .limit(c.pageSize())
                  .offset(c.offset())
                  .fetchInto(OrderDto.class);
    }
}

表 17-8:关键源码入口

设计取舍

取舍 1:类型安全 DSL 而非 ORM 实体图

jOOQ 拒绝透明脏检查、隐式懒加载、自动实体图,换来 SQL 形状可预测、编译期 schema 检查、方言渲染控制。

取舍 2:Plain SQL 仅作为逃生口

DSL.field(String) / DSL.condition(String) 提供完整 SQL 控制权,代价是放弃类型安全。jOOQ 不阻止使用,但通过文档与 API 设计鼓励团队自律。

取舍 3:DAO 仅覆盖单表 CRUD

复杂查询不在 DAO 内部实现,而是放在 Query Class 或 Repository 中。代价是要写更多类,收益是 DAO 不会膨胀。

取舍 4:fetchOne / fetchSingle / fetchAny / fetchOptional 各司其职

四个方法语义完全不同,迫使开发者声明意图。代价是 API 表面变大,收益是结果集语义在调用处显式可见。

取舍 5:不内置 N+1 检测

jOOQ 提供 ExecuteListener 和 DiagnosticsListener,但默认不开启 N+1 检测。理由是检测策略因业务而异,框架统一拦截会误报。团队应在测试层加 QueryCountListener 自行守护。
