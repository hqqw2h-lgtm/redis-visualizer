25.1 执行计划分析

任何优化第一步是理解当前发生什么。执行计划是 SQL 的 X 光片——告诉查询走哪条路、扫描多少行、用了什么索引。jOOQ 3.15+ 提供 explain() API,无需离开 Java 代码即可获取计划。

25.1.1 获取执行计划

Explain explain = dsl.explain(
    dsl.select(ORDERS.ID, CUSTOMERS.NAME)
       .from(ORDERS)
       .join(CUSTOMERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
       .where(CUSTOMERS.COUNTRY.eq("China")));

System.out.println(explain);
System.out.println("estimated rows: " + explain.rows());
System.out.println("estimated cost: " + explain.cost());

25.2 索引感知查询

索引是性能基石,但查询写法决定索引能否被使用。

25.2.1 陷阱一:函数包裹

// 反例:函数包裹列 -> 全表扫描
dsl.selectFrom(ORDERS).where(DSL.year(ORDERS.CREATED_AT).eq(2024)).fetch();
// SELECT * FROM "orders" WHERE extract(year from "created_at") = 2024
// 计划:Seq Scan

// 正例:范围查询 -> B-Tree 索引
dsl.selectFrom(ORDERS)
   .where(ORDERS.CREATED_AT.ge(LocalDate.of(2024, 1, 1).atStartOfDay()))
   .and(ORDERS.CREATED_AT.lt(LocalDate.of(2025, 1, 1).atStartOfDay()))
   .fetch();
// 计划:Index Scan using idx_orders_created_at

25.2.2 陷阱二:LIKE 模式

模式 SQL 索引可用
LIKE 'john%' 前缀匹配 可走 B-Tree
LIKE '%@gmail.com' 后缀匹配 不走索引
LIKE '%key%' 中间通配 不走索引
LIKE 'john%'(pattern_ops) 显式索引 可走 
dsl.selectFrom(CUSTOMERS).where(CUSTOMERS.EMAIL.like("john%")).fetch();
// SELECT * FROM "customers" WHERE "email" LIKE 'john%'
// pattern_ops 索引:Index Range Scan

后缀搜索可建函数索引 CREATE INDEX idx_email_rev ON customers(reverse(email)),然后:

dsl.selectFrom(CUSTOMERS)
   .where(DSL.field("reverse({0})", String.class, CUSTOMERS.EMAIL)
       .like(DSL.val(new StringBuilder("@gmail.com").reverse().toString() + "%")))
   .fetch();

25.3 N+1 问题检测与解决

N+1 问题是 ORM 世界的沉默杀手:1 次主查询 + N 次关联查询。jOOQ 提供 DiagnosticsListener 自动检测,通过 MULTISET 从根本解决。

图 25-3:N+1 检测流程

图 25-3 · N+1 检测流程 — DiagnosticsListener 监听 SQL 模板重复

25.3.1 DiagnosticsListener

DiagnosticsListener listener = new DefaultDiagnosticsListener() {
    @Override public void repeatedStatements(DiagnosticsContext ctx) {
        log.warn("N+1 / repeated SQL detected: {}", ctx.message());
    }
    @Override public void tooManyRowsFetched(DiagnosticsContext ctx) {
        log.warn("Too many rows fetched: {}", ctx.message());
    }
};

try (Connection c = dataSource.getConnection();
     Connection diag = new DiagnosticsConnection(c, listener)) {
    DSLContext diagDsl = DSL.using(diag, SQLDialect.POSTGRES);
    for (CustomersRecord cust : diagDsl.selectFrom(CUSTOMERS).fetch()) {
        diagDsl.selectFrom(ORDERS).where(ORDERS.CUSTOMER_ID.eq(cust.getId())).fetch();
    }
}

25.3.2 MULTISET 一步解决

var result = dsl.select(
    CUSTOMERS.ID,
    CUSTOMERS.NAME,
    multiset(
        select(ORDERS.ID, ORDERS.AMOUNT)
        .from(ORDERS)
        .where(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
    ).as("orders").convertFrom(r -> r.map(Records.mapping(OrderDto::new))))
.from(CUSTOMERS)
.fetch();

SELECT
  "customers"."id",
  "customers"."name",
  (SELECT coalesce(jsonb_agg(jsonb_build_array("v0", "v1")), jsonb_build_array())
     FROM (SELECT "orders"."id" AS "v0", "orders"."amount" AS "v1"
             FROM "orders" WHERE "orders"."customer_id" = "customers"."id") "t"
  ) AS "orders"
FROM "customers"

仅 1 条 SQL,不论客户数量。多层嵌套同样只需 1 条 SQL,三层嵌套全部在数据库端完成。

25.5 分页优化

OFFSET 分页直觉,但深页性能急剧下降。Keyset 分页通过游标实现恒定性能。

表 25-5:OFFSET vs Keyset 性能

页码 OFFSET (ms) Keyset (ms) 倍数
第 1 页 2 2 同
第 100 页 15 2 7.5x
第 1000 页 120 2 60x
第 5000 页 480 3 160x
第 10000 页 780 3 260x

25.5.1 OFFSET 深页问题

dsl.selectFrom(ORDERS).orderBy(ORDERS.ID).limit(20).offset(100_000);
// SELECT * FROM "orders" ORDER BY "id" LIMIT 20 OFFSET 100000
// 扫描 100020 行 -> 丢弃 100000 行 -> 返回 20 行

25.5.2 Keyset 恒定性能

var firstPage = dsl.selectFrom(ORDERS).orderBy(ORDERS.ID).limit(20).fetch();
Long lastSeenId = firstPage.get(firstPage.size() - 1).getId();

var nextPage = dsl.selectFrom(ORDERS)
    .where(ORDERS.ID.gt(lastSeenId))
    .orderBy(ORDERS.ID)
    .limit(20)
    .fetch();

25.5.3 jOOQ seek

var page = dsl.selectFrom(ORDERS)
    .orderBy(ORDERS.CREATED_AT.desc(), ORDERS.ID.desc())
    .seek(lastCreatedAt, lastId)
    .limit(20)
    .fetch();

SELECT * FROM "orders"
 WHERE ("created_at", "id") < (timestamp '2024-06-15 10:30:00', 98765)
 ORDER BY "created_at" DESC, "id" DESC
 LIMIT 20

seek() 自动处理多列比较,等价于 (created_at < ?) OR (created_at = ? AND id < ?),但 ROW 值表达式更高效。

    select(EMPLOYEE.ID, EMPLOYEE.NAME, EMPLOYEE.PARENT_ID, val(0).as("level"))

25.9.1 声明式 Policy

public class TenantContext {
    private static final ThreadLocal<String> CURRENT = new ThreadLocal<>();
    public static String current()           { return CURRENT.get(); }
    public static void set(String tenantId)  { CURRENT.set(tenantId); }
    public static void clear()               { CURRENT.remove(); }
}

DSLContext tenantDsl = DSL.using(
    configuration.derive((PolicyProvider) ctx -> List.of(
        DSL.policy("tenant_orders").on(ORDERS)
           .using(ORDERS.TENANT_ID.eq(DSL.val(TenantContext.current()))),
        DSL.policy("tenant_items").on(ORDER_ITEMS)
           .using(ORDER_ITEMS.TENANT_ID.eq(DSL.val(TenantContext.current()))),
        DSL.policy("tenant_users").on(CUSTOMERS)
           .using(CUSTOMERS.TENANT_ID.eq(DSL.val(TenantContext.current()))))));

25.10.1 自动审计 Listener

public class AuditRecordListener extends DefaultRecordListener {
    @Override public void insertStart(RecordContext ctx) { setAuditFields(ctx.record(), true); }
    @Override public void updateStart(RecordContext ctx) { setAuditFields(ctx.record(), false); }

    private void setAuditFields(Record rec, boolean isNew) {
        LocalDateTime now = LocalDateTime.now();
        String user = SecurityContext.currentUser();
        Field<LocalDateTime> uat = rec.field("updated_at", LocalDateTime.class);
        Field<String>        uby = rec.field("updated_by", String.class);
        Field<LocalDateTime> cat = rec.field("created_at", LocalDateTime.class);
        Field<String>        cby = rec.field("created_by", String.class);

        if (uat != null) rec.set(uat, now);
        if (uby != null) rec.set(uby, user);
        if (isNew) {
            if (cat != null) rec.set(cat, now);
            if (cby != null) rec.set(cby, user);
        }
    }
}

DSLContext dsl = DSL.using(
    configuration.derive(RecordListenerProvider.of(AuditRecordListener::new)));

    @Autowired private OrderRepository jpaRepo;
    @Autowired private DSLContext dsl;

    public Order save(Order o) { return jpaRepo.save(o); }

    public List<OrderReportDto> monthlyReport(YearMonth m) {
        return dsl.select(ORDERS.STATUS, count().as("count"), sum(ORDERS.TOTAL_AMOUNT).as("revenue"))
            .from(ORDERS)
            .where(ORDERS.CREATED_AT.ge(m.atDay(1).atStartOfDay()))
            .and(ORDERS.CREATED_AT.lt(m.plusMonths(1).atDay(1).atStartOfDay()))
            .groupBy(ORDERS.STATUS)
            .fetchInto(OrderReportDto.class);
    }
}

25.13.2 N+1 自动检测

@Test
void detectsN1() {
    var probe = new ArrayList<String>();
    DiagnosticsListener l = new DefaultDiagnosticsListener() {
        @Override public void repeatedStatements(DiagnosticsContext ctx) {
            probe.add("repeat:" + ctx.message());
        }
    };

    try (Connection c = dataSource.getConnection();
         Connection diag = new DiagnosticsConnection(c, l)) {
        DSLContext d = DSL.using(diag, SQLDialect.POSTGRES);
        for (int i = 0; i < 3; i++)
            d.selectFrom(ORDERS).where(ORDERS.CUSTOMER_ID.eq((long) i)).fetch();
    }

    assertThat(probe).isNotEmpty();
}

25.13.3 Keyset 不重复不漏

@Test
void seek_keysetCoverage() {
    Set<Long> seen = new HashSet<>();
    LocalDateTime cursorAt = LocalDateTime.MAX;
    long cursorId = Long.MAX_VALUE;

    while (true) {
        var page = dsl.selectFrom(ORDERS)
            .orderBy(ORDERS.CREATED_AT.desc(), ORDERS.ID.desc())
            .seek(cursorAt, cursorId)
            .limit(20).fetch();
        if (page.isEmpty()) break;
        for (OrdersRecord r : page) assertThat(seen.add(r.getId())).isTrue();
        OrdersRecord last = page.get(page.size() - 1);
        cursorAt = last.getCreatedAt();
        cursorId = last.getId();
    }
}

设计取舍

取舍 1:优化围绕查询形状

jOOQ 保留 SQL 可观测性,不隐藏数据库计划。代价是每次优化要保存 SQL、bind、EXPLAIN 三件证据;收益是优化决策不靠感觉而靠证据。

取舍 2:DiagnosticsListener 只报警

N+1、重复 SQL 可零侵入发现。代价是诊断只报告不自动修业务模型,需要开发者改查询形状;收益是诊断成本极低,可在开发 / CI 全程开启。

取舍 3:Keyset 替代深 OFFSET

大页码性能稳定。代价是牺牲随机跳页能力,API 改为 cursor 模型;收益是页码越深性能越好。

取舍 4:方言能力显式分支

全文、JSON、递归 CTE 用原生能力。代价是方言耦合,需 repository / extension 封装;收益是性能上限不被抽象层压制。
