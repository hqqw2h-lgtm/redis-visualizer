第 4 章 · 类型系统:Field<T> 与 DataType

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

jOOQ 的类型安全不是注解约定,而是 Java 泛型与代码生成的物理产物。每张表的每个列在 codegen 阶段被生成为 TableField<R, T> 静态字段,泛型参数 R 锁定 Record 类型,T 锁定列的 Java 类型。所有 DSL 表达式 eq / gt / lt / in / between 等方法,在 Field<T> 接口上以 T 为约束重载,使类型不匹配的比较在 javac 阶段就被拒绝。

本章拆解类型系统的四层结构:SQL 类型与 Java 类型的映射规则、Field<T> 泛型传播机制、DataType<T> SPI 与自定义类型扩展、cast / coerce / convert 三种类型转换的语义边界。最后回到陷阱(泛型擦除、MULTISET 推断、方言差异)与源码深潜(Field / DataType / Converter / Binding 协作)。

4.2 Field 泛型体系

4.2.1 类层次

图 4-2-1 · Field 类层次 — 多接口继承使 Field 能用于 select/groupBy/orderBy

Field<T> 同时实现多个接口,使其能出现在 select、group by、order by 等不同上下文。TableField<R, T> 增加 Table<R> 与列名信息,是 codegen 产物的实际类型。

4.2.2 类型约束实例

Condition c1 = ORDERS.STATUS.eq("PAID");
Condition c2 = ORDERS.ID.gt(100L);
Condition c3 = ORDERS.AMOUNT.le(new BigDecimal("128.00"));

Condition wrong1 = ORDERS.ID.eq("PAID");
Condition wrong2 = ORDERS.STATUS.eq(42);

[ERROR] no suitable method found for eq(String)
[ERROR]   method Field.eq(Long) is not applicable
[ERROR]     argument mismatch; String cannot be converted to Long

[ERROR] no suitable method found for eq(int)
[ERROR]   method Field.eq(String) is not applicable
[ERROR]     argument mismatch; int cannot be converted to String

4.2.3 重载方法签名

方法 签名 用途
eq(T) Condition eq(T value) 字段与字面量
eq(Field<T>) Condition eq(Field<T> other) 字段与字段
eq(Select<? extends Record1<T>>) 子查询 单值子查询
eq(QuantifiedSelect<? extends Record1<T>>) eq ANY 等 量化子查询

每条 eq 重载共享同一 T,使字段、字面量、子查询在编译期保持类型一致。该重载策略适用于所有比较与运算方法。

4.2.4 类型传播路径

图 4-2-4 · 类型传播路径 — T 在派生方法中保持,直到 cast/coerce/convert 显式改变

Field<T> 的 T 在所有派生方法中保持,直到 cast / coerce / convert 显式改变类型参数。该传播链使 SQL 表达式整体保持类型一致。

4.4.2 SQLDataType 标准常量

SQLDataType 是方言无关的标准常量,渲染时由 RenderContext 转换为目标方言语法。

4.4.3 DataType 不可变构建链

图 4-4-3 · DataType 不可变构建链 — 每次调用返回新实例

DataType<T> 是不可变对象,所有修饰方法返回新实例。该设计与 Field<T> 配合,使类型元信息可在多个 Field 间共享而不被修改。

4.4.4 自定义 Converter

Converter<String, OrderStatus> converter = Converter.ofNullable(
    String.class, OrderStatus.class,
    OrderStatus::valueOf,
    OrderStatus::name);

DataType<OrderStatus> statusType =
    SQLDataType.VARCHAR.asConvertedDataType(converter);

Field<OrderStatus> status = DSL.field("status", statusType);

Condition c = status.eq(OrderStatus.PAID);

select status
from orders
where status = ?

bind[1] = "PAID"           (String, from OrderStatus.PAID)
record.get(status)         = PAID  (OrderStatus)
status.getDataType().getType() = OrderStatus.class

4.4.5 Converter 失败模式

org.jooq.exception.DataTypeException: Error while reading field: status
Caused by: java.lang.IllegalArgumentException: No enum constant OrderStatus.ARCHIVED_OLD
  at java.lang.Enum.valueOf(Enum.java:273)
  at OrderStatus.valueOf(OrderStatus.java:1)

数据库出现 Java 枚举不识别的字面量时,Converter 在读取阶段抛出 DataTypeException。生产防护:在 Converter 内捕获并降级为默认值,或在数据库层加 CHECK 约束。

4.5.4 Kotlin Generator 的 null 安全

元信息 Kotlin 生成类型 编译器约束

nullable=true String? 必须 ?. 或 !!

jOOQ 3.17+ 的 KotlinGenerator 把 nullable 元信息映射为 Kotlin 可空类型,使空值检查由 Kotlin 编译器强制。Java Generator 仅在 Javadoc 中注明,不引入 @Nullable 注解。

4.5.5 INSERT NOT NULL 拒绝

dsl.insertInto(ORDERS)
   .set(ORDERS.ID, (Long) null)
   .set(ORDERS.STATUS, "PAID")
   .execute();

org.jooq.exception.DataAccessException:
SQL [insert into orders (id, status) values (?, ?)];
ERROR: null value in column "id" of relation "orders" violates not-null constraint

类型系统不阻止 null 写入(Java 类型仍是 Long),最终由数据库 NOT NULL 约束拒绝。该层防护对应"防御深度"原则:类型系统是第一层,数据库约束是最后兜底。

4.6 cast、coerce 与 convert

4.6.1 三种转换的决策树

图 4-6-1 · cast / coerce / convert 决策树 — 三种转换语义边界

4.6.2 cast 数据库级转换

Field<String> amountText = ORDERS.AMOUNT.cast(SQLDataType.VARCHAR);
dsl.select(ORDERS.ID, amountText).from(ORDERS).fetch();

select orders.id, cast(orders.amount as varchar)
from orders

record.get(ORDERS.ID)   = 100        (Long)
record.get(amountText)  = "129.90"   (String)
amountText.getDataType().getType() = String.class

4.6.3 coerce 仅改 Java 泛型

Field<Object> raw = DSL.field("metadata->>'count'");
Field<Integer> count = raw.coerce(SQLDataType.INTEGER);

dsl.select(count).from(ORDERS).fetch();

select metadata->>'count'
from orders

coerce 不在 SQL 中加 CAST,仅把 Java 泛型从 Object 改为 Integer。该机制适用于纯 SQL 模板返回值类型 jOOQ 无法推断的场景。

4.6.4 coerce 失败模式

database value = "many"

org.jooq.exception.DataTypeException: Cannot convert from many (class java.lang.String) to class java.lang.Integer
Caused by: java.lang.NumberFormatException: For input string: "many"

coerce 是开发者承诺类型兼容的机制。若实际值不能转换,jOOQ 在读取阶段抛出 DataTypeException。生产代码应配合 SQL snapshot 测试与运行时断言。

4.6.5 三方法对比表

方法 Java 类型 SQL 输出 转换执行方 风险
cast(T) 改为 Field<T> CAST(... AS T) 数据库 SQL 报错
coerce(T) 改为 Field<T> 原样 jOOQ Java 端 读取异常
convert(Converter) 改为 Field<U> 原样 Converter Converter 自定义
场景 推荐方法
INT → TEXT 用于拼接 cast

4.8 SQL Snapshot 守护

4.8.1 类型 + SQL 联合断言

@Test
void status_filter_keeps_typed_bind() {
    var query = dsl.select(ORDERS.ID, ORDERS.STATUS, ORDERS.AMOUNT)
        .from(ORDERS)
        .where(ORDERS.STATUS.eq("PAID")
            .and(ORDERS.AMOUNT.gt(new BigDecimal("100.00"))));

    assertThat(query.getSQL())
        .isEqualToNormalizingWhitespace("""
            select orders.id, orders.status, orders.amount
            from orders
            where (orders.status = ? and orders.amount > ?)
            """);

    assertThat(query.getBindValues())
        .containsExactly("PAID", new BigDecimal("100.00"));
}

4.8.2 Converter 路径断言

@Test
void status_converter_round_trips() {
    Field<OrderStatus> typed = ORDERS.STATUS.convert(
        Converter.ofNullable(
            String.class, OrderStatus.class,
            OrderStatus::valueOf,
            OrderStatus::name));

    var query = dsl.select(ORDERS.ID, typed)
        .from(ORDERS)
        .where(typed.eq(OrderStatus.PAID));

    assertThat(query.getBindValues())
        .containsExactly(OrderStatus.PAID);

    assertThat(query.getSQL())
        .contains("orders.status = ?");
}

设计取舍

取舍 1:Field 泛型与运行期擦除

Java 泛型在 javac 后被擦除,运行期无法保留 T。jOOQ 的类型安全完全发生在编译期,运行期通过 DataType<T> 间接保留类型元信息。代价是运行期类型判断必须经 getDataType().getType()。

取舍 2:Converter 与 Binding 分层

Converter 适用于值对象转换(枚举、UUID 字符串),不感知 JDBC。Binding 适用于 JDBC 协议级控制(JSON、自定义二进制),需实现 set / get / register 等方法。日常场景 Converter 足够,Binding 仅用于复杂类型映射。

取舍 3:cast 与 coerce 二选一

cast 把转换责任交给数据库,代价是 SQL 文本变化与方言差异。coerce 把转换责任交给 jOOQ Java 端,代价是开发者承诺类型兼容。两者互补:数据库能直接转的用 cast,jOOQ 已知类型用 coerce,两者都不便用 convert + Converter。
