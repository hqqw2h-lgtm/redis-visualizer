28.0 生产级 CRUD 的组合架构

生产级 jOOQ 项目不能把所有能力都塞进 Repository。多租户、软删除、审计、慢查询、乐观锁、 指标、健康检查属于基础设施能力；Repository 只应该描述"这个业务用例要访问哪些表、生成什么 SQL"。 一旦把租户过滤、审计字段、慢查询计时手写进每个 Repository 方法，就会出现遗漏和行为不一致。

能力 推荐落点 为什么放这里
多租户过滤 PolicyProvider / DefaultConfigurationCustomizer 所有 SELECT/UPDATE/DELETE 自动带条件，避免手写遗漏
软删除过滤 PolicyProvider + 专用恢复 DSL 普通查询默认过滤，恢复/审计用 raw DSL 明确绕过
审计字段 RecordListener record.store()、insertInto().returning() 前统一填充
SQL 审计与慢查询 ExecuteListener 可以拿到最终 SQL、绑定参数、耗时、异常
乐观锁 Settings + version 字段 jOOQ 自动在 store() 中附加版本条件
只读/写事务 Service @Transactional 一个业务操作可能调用多个 Repository，事务属于用例边界
HTTP 合约 Controller + DTO 不暴露 Record，不让数据库字段直接决定 API

生产配置的关键原则是：优先通过 Spring Boot 暴露的 jOOQ 自定义入口增强现有 Configuration， 而不是随手 new DefaultDSLContext(new DefaultConfiguration())。后者容易丢掉 Boot 自动注入的 SpringTransactionProvider、DataSourceConnectionProvider、异常翻译和已有 Listener。

生产排错可以按这条链路切分：HTTP 是否带租户 → TenantContext 是否设置 → Policy 是否注入 → SQL 是否参数化 → 事务连接是否复用 → Listener 是否记录耗时和异常 → HTTP 是否映射为稳定错误码。

生产扩展的设计原则：增强同一个 Configuration

第 26、27 章的 Repository 只关心"我要什么数据"。到了生产环境，额外能力变多： 租户隔离、软删除、审计、慢查询、指标、乐观锁、诊断监听、健康检查。它们的共同点是都依赖 jOOQ 的 Configuration。如果项目里出现多个互不相干的 DSLContext 或 DefaultConfiguration， 这些能力就会随机失效。

错误做法 表面现象 生产风险
Repository 里 DSL.using(dataSource, POSTGRES) 查询能跑 丢失 Boot 注入的事务、Policy、Listener
为乐观锁单独声明 DefaultConfiguration Bean 乐观锁示例能跑 覆盖 Boot 自动配置，连接和异常翻译不一致
审计 Listener 内继续用同一个 DSL 写审计表 审计代码复用简单 递归触发 Listener，甚至无限审计自己
管理员查询直接绕过 Policy 临时排查方便 变成没有审计的后门数据访问路径

推荐做法是所有生产能力都通过 DefaultConfigurationCustomizer、ExecuteListenerProvider、 RecordListenerProvider、PolicyProvider、Settings 加到同一个 Boot 管理的配置对象上。 需要绕过 Policy 的少数场景（如恢复软删除数据、审计写入）必须用命名清晰的 rawDsl， 并由 Service 方法和审计日志包住。

生产边界深潜：哪些能力必须横切，哪些必须留在业务层

生产级 CRUD 最容易失控的地方，是把“全局不变量”和“业务用例规则”混在一起。多租户、软删除、 审计、慢查询、异常翻译这些规则的目标是“每一条正式 SQL 都必须一致”；库存扣减、订单状态机、 退款窗口这些规则的目标是“当前业务用例要做什么”。前者适合放在 jOOQ Configuration、Spring 事务和 Listener 中，后者应该留在 Service 层。

28.1 多租户隔离 — Policy 行级安全

SaaS 产品中，多个租户共享同一张数据库表。如果每条 SQL 都手动拼接 WHERE tenant_id = ?，不仅容易遗漏，更可能造成数据泄露。 jOOQ 3.19+ 的 Policy API 将行级安全策略声明式注入到所有查询中——开发者写的 SQL 完全不需要感知租户字段。

Policy 声明

PolicyProvider 在 DefaultConfiguration 中注册，对所有使用该 Configuration 的查询生效——SELECT、UPDATE、DELETE 都自动附加租户过滤条件。

@Configuration
public class MultiTenantConfig {

    @Bean
    DefaultConfigurationCustomizer tenantPolicyCustomizer() {
        return configuration -> configuration.set((PolicyProvider) ctx -> List.of(
                DSL.policy("tenant_isolation")
                   .on(ORDERS)
                   .using(ORDERS.TENANT_ID.eq(
                       DSL.val(TenantContext.current())
                   ))
            )
        );
    }
}

public class TenantContext {
    private static final ThreadLocal<String> HOLDER = new ThreadLocal<>();

    public static void set(String tenantId) { HOLDER.set(tenantId); }
    public static String current()          { return HOLDER.get(); }
    public static void clear()              { HOLDER.remove(); }
}

TenantFilter — 从 HTTP 请求提取租户

@Component
public class TenantFilter implements Filter {

    @Override
    public void doFilter(ServletRequest req, ServletResponse resp,
                         FilterChain chain) throws IOException, ServletException {
        String tenantId = ((HttpServletRequest) req).getHeader("X-Tenant-ID");
        if (tenantId == null || tenantId.isBlank()) {
            ((HttpServletResponse) resp).sendError(400, "Missing X-Tenant-ID");
            return;
        }
        TenantContext.set(tenantId);
        try {
            chain.doFilter(req, resp);
        } finally {
            TenantContext.clear();
        }
    }
}

@Bean

28.2 审计日志 — RecordListener + ExecuteListener

合规要求每条数据都记录"谁、在什么时间、做了什么操作"。jOOQ 的 RecordListener 在 Record 级别拦截增/改操作，自动填充审计字段—— 不需要在每个 Repository 里重复 setCreatedBy()。

RecordListener 实现

@Component
public class AuditRecordListener extends DefaultRecordListener {

    @Override
    public void insertStart(RecordContext ctx) {
        setAuditFields(ctx.record(), true);
    }

    @Override
    public void updateStart(RecordContext ctx) {
        setAuditFields(ctx.record(), false);
    }

    private void setAuditFields(Record r, boolean isNew) {
        LocalDateTime now = LocalDateTime.now();
        String user = SecurityContext.currentUser();
        Field<LocalDateTime> updatedAt = r.field("updated_at", LocalDateTime.class);
        Field<String> updatedBy = r.field("updated_by", String.class);
        Field<LocalDateTime> createdAt = r.field("created_at", LocalDateTime.class);
        Field<String> createdBy = r.field("created_by", String.class);

        if (updatedAt != null) r.set(updatedAt, now);
        if (updatedBy != null) r.set(updatedBy, user);
        if (isNew) {
            if (createdAt != null) r.set(createdAt, now);
            if (createdBy != null) r.set(createdBy, user);
        }
    }
}

// 注册到 Configuration
@Bean
DefaultConfigurationCustomizer auditRecordCustomizer(AuditRecordListener auditListener) {
    return configuration -> configuration.set(
        new DefaultRecordListenerProvider(auditListener)
    );
}

ExecuteListener — SQL 级审计追踪

@Component
public class AuditExecuteListener extends DefaultExecuteListener {
    private final AuditLogRepository auditRepo;

    @Override
    public void executeEnd(ExecuteContext ctx) {
        if (ctx.type() == ExecuteType.WRITE) {
            // 使用独立 DataSource / raw DSL 写审计表，避免审计 INSERT 再次触发本 listener。
            auditRepo.save(AuditLog.builder()
                .sql(ctx.sql())
                .user(SecurityContext.currentUser())
                .timestamp(Instant.now())
                .rowsAffected(ctx.rows())
                .build());
        }
    }
}

28.3 慢查询检测 — ExecuteListener + Micrometer

一条慢 SQL 就能拖垮整个系统。通过 ExecuteListener 记录每条查询的执行时间， 结合 Micrometer 发送到 Prometheus/Grafana，实现自动慢查询告警。

PerformanceListener 实现

@Component
public class PerformanceListener extends DefaultExecuteListener {
    private static final Logger log = LoggerFactory.getLogger(PerformanceListener.class);
    private static final long SLOW_THRESHOLD_NS = 3_000_000_000L; // 3 秒

    private final MeterRegistry meterRegistry;
    private final ThreadLocal<Long> startTime = new ThreadLocal<>();

    public PerformanceListener(MeterRegistry meterRegistry) {
        this.meterRegistry = meterRegistry;
    }

    @Override
    public void executeStart(ExecuteContext ctx) {
        startTime.set(System.nanoTime());
    }

    @Override
    public void executeEnd(ExecuteContext ctx) {
        long elapsed = System.nanoTime() - startTime.get();
        startTime.remove();

        // 记录到 Micrometer
        meterRegistry.timer("jooq.query.duration",
            "type", ctx.type().name()
        ).record(elapsed, TimeUnit.NANOSECONDS);

        // 慢查询告警
        if (elapsed > SLOW_THRESHOLD_NS) {
            log.warn("SLOW QUERY [{}ms]: {}",
                elapsed / 1_000_000, ctx.sql());
            meterRegistry.counter("jooq.query.slow").increment();
        }
    }

    @Override
    public void exception(ExecuteContext ctx) {
        meterRegistry.counter("jooq.query.error",
            "type", ctx.type().name()
        ).increment();
    }
}

28.4 批量操作优化

电商场景中，"商品导入"和"库存同步"需要一次写入成千上万条记录。 逐条 INSERT 的方式会产生上万次网络往返，batch() 将多条语句打包成单次请求。

batch() 批量导入

@Transactional
public int batchImportProducts(List<ProductImportDto> products) {
    var inserts = products.stream()
        .map(p -> dsl.insertInto(PRODUCTS)
            .set(PRODUCTS.NAME, p.name())
            .set(PRODUCTS.PRICE, p.amount())
            .set(PRODUCTS.STOCK, p.stock())
            .set(PRODUCTS.CATEGORY_ID, p.categoryId()))
        .toList();

    int[] result = dsl.batch(inserts).execute();
    return Arrays.stream(result).sum();
}

MERGE / UPSERT — 幂等导入

外部系统推送的数据可能重复。ON CONFLICT ... DO UPDATE 实现"存在则更新、不存在则插入"—— 一条 SQL 替代"查询 + 判断 + 插入/更新"的三步逻辑。

public void upsertProduct(String sku, String name, BigDecimal price) {
    dsl.insertInto(PRODUCTS)
       .set(PRODUCTS.SKU, sku)
       .set(PRODUCTS.NAME, name)
       .set(PRODUCTS.PRICE, price)
       .onConflict(PRODUCTS.SKU)
       .doUpdate()
       .set(PRODUCTS.NAME, name)
       .set(PRODUCTS.PRICE, price)
       .execute();
}

-- 生成的 SQL
INSERT INTO products (sku, name, price)
VALUES ('SKU-001', 'Widget', 29.99)
ON CONFLICT (sku) DO UPDATE
SET name  = EXCLUDED.name,
    price = EXCLUDED.price;

// 批量 UPSERT：结合 batch + onConflict
@Transactional
public int batchUpsertProducts(List<ProductImportDto> products) {
    var upserts = products.stream()
        .map(p -> dsl.insertInto(PRODUCTS)
            .set(PRODUCTS.SKU, p.sku())
            .set(PRODUCTS.NAME, p.name())
            .set(PRODUCTS.PRICE, p.amount())
            .set(PRODUCTS.STOCK, p.stock())
            .onConflict(PRODUCTS.SKU)
            .doUpdate()
            .set(PRODUCTS.NAME, p.name())
            .set(PRODUCTS.PRICE, p.amount())
            .set(PRODUCTS.STOCK, p.stock()))
        .toList();

    int[] result = dsl.batch(upserts).execute();
    return Arrays.stream(result).sum();
}

public PageResult<OrderDto> listOrders(Long afterId, int size) {
    var query = dsl.select(
            ORDERS.ID, ORDERS.TOTAL_AMOUNT, ORDERS.STATUS,
            ORDERS.CREATED_AT, CUSTOMERS.NAME)
        .from(ORDERS)
        .join(CUSTOMERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
        .where(ORDERS.STATUS.ne("DELETED"));

    if (afterId != null) {
        query = query.and(ORDERS.ID.gt(afterId));
    }

    var results = query
        .orderBy(ORDERS.ID.asc())
        .limit(size + 1)    // 多取一条判断 hasMore
        .fetchInto(OrderDto.class);

    boolean hasMore = results.size() > size;
    if (hasMore) results = results.subList(0, size);

    Long lastId = results.isEmpty()
        ? null
        : results.get(results.size() - 1).id();

    return new PageResult<>(results, lastId, hasMore);
}

28.7 软删除 — 数据保护与合规

直接 DELETE 后数据永远消失。生产系统通常用"软删除"——标记 deleted_at 而非物理删除，所有查询自动过滤已删除数据。

-- 添加软删除列

ALTER TABLE customers ADD COLUMN deleted_at TIMESTAMPTZ DEFAULT NULL;

-- 变成：
-- UPDATE orders SET deleted_at = now() WHERE id = 1;

@Repository
public class SoftDeleteRepository {

    public int softDelete(Table<?> table, Condition condition) {
        return dsl.update(table)
            .set(field("deleted_at", LocalDateTime.class), LocalDateTime.now())
            .set(field("deleted_by", String.class), SecurityContext.currentUser())
            .where(condition)
            .and(field("deleted_at").isNull())
            .execute();
    }

    public int restore(Table<?> table, Condition condition) {
        // 使用原始 DSLContext 绕过 Policy 进行恢复
        return rawDsl.update(table)
            .setNull(field("deleted_at", LocalDateTime.class))
            .setNull(field("deleted_by", String.class))
            .where(condition)
            .execute();
    }
}

@Bean
public ExecuteListenerProvider metricsListener(MeterRegistry registry) {
    return () -> new DefaultExecuteListener() {
        private final ThreadLocal<Long> start = new ThreadLocal<>();

        @Override
        public void executeStart(ExecuteContext ctx) {
            start.set(System.nanoTime());
        }

        @Override
        public void executeEnd(ExecuteContext ctx) {
            long elapsed = System.nanoTime() - start.get();
            start.remove();

            String queryType = switch (ctx.type()) {
                case READ  -> "select";
                case WRITE -> "write";
                case DDL   -> "ddl";
                default    -> "other";
            };

            registry.timer("jooq.query.duration",
                "type", queryType,
                "table", extractTableName(ctx.sql())
            ).record(elapsed, TimeUnit.NANOSECONDS);

            registry.counter("jooq.query.count",
                "type", queryType
            ).increment();
        }

        @Override
        public void exception(ExecuteContext ctx) {
            registry.counter("jooq.query.errors",
                "type", ctx.type().name(),
                "exception", ctx.exception().getClass().getSimpleName()
            ).increment();
        }

        private String extractTableName(String sql) {
            if (sql == null) return "unknown";
            var m = Pattern.compile("(?i)(?:FROM|INTO|UPDATE)\\s+(\\w+)")
                           .matcher(sql);
            return m.find() ? m.group(1) : "unknown";
        }
    };
}

// 统一的 JooqConfig — 整合所有生产特性
@Configuration
public class JooqConfig {

    @Bean
    DefaultConfigurationCustomizer productionJooqCustomizer(
            AuditRecordListener auditListener,
            PerformanceListener perfListener,
            PerformanceProperties perfProps) {

        var settings = new Settings()
            .withExecuteWithOptimisticLocking(true)
            .withExecuteWithOptimisticLockingExcludeUnversioned(true)
            .withRenderSchema(false)
            .withRenderFormatted(perfProps.logAllQueries());

        return configuration -> configuration
            .set(settings)
            // 多租户 + 软删除策略
            .set(combinedPolicyProvider())
            // 审计 RecordListener
            .set(new DefaultRecordListenerProvider(auditListener))
            // 性能 ExecuteListener
            .set(new DefaultExecuteListenerProvider(perfListener));
    }

    private PolicyProvider combinedPolicyProvider() {
        return ctx -> {
            var policies = new java.util.ArrayList<Policy<?>>();

            // 多租户
            for (var t : List.of(ORDERS, CUSTOMERS, PRODUCTS)) {
                if (t.field("tenant_id", String.class) == null) continue;
                policies.add(DSL.policy("tenant_" + t.getName())
                    .on(t)
                    .using(t.field("tenant_id", String.class)
                        .eq(DSL.val(TenantContext.current()))));
            }

            // 软删除
            for (var t : List.of(ORDERS, CUSTOMERS)) {
                policies.add(DSL.policy("softdel_" + t.getName())
                    .on(t)
                    .using(t.field("deleted_at").isNull()));
            }

            return policies;
        };
    }

护栏 ⑦ 审计写入不能递归触发自身

SQL 审计最常见的生产事故，是 ExecuteListener 在 executeEnd() 中继续用同一个业务 DSLContext 写 audit_logs。这样审计 INSERT 也会触发同一个 Listener，轻则重复记录，重则无限递归。

@Test
void auditExecuteListener_shouldUseRawDslAndAvoidRecursion() {
    var auditCounter = new AtomicInteger();

    ExecuteListener listener = new DefaultExecuteListener() {
        @Override
        public void executeEnd(ExecuteContext ctx) {
            if (ctx.sql() != null && ctx.sql().contains("audit_logs")) {
                auditCounter.incrementAndGet();
            }
        }
    };

    DSLContext businessDsl = dsl.configuration()
        .derive(new DefaultExecuteListenerProvider(listener))
        .dsl();

    orderService.createOrderThrough(businessDsl, command);

    assertThat(auditCounter)
        .as("audit INSERT must not be audited by the business listener again")
        .hasValue(0);
}

实现上更推荐给审计写入一个明确命名的边界：

@Bean
@Qualifier("auditRawDsl")
DSLContext auditRawDsl(DataSource dataSource) {
    return DSL.using(dataSource, SQLDialect.POSTGRES);
}

这个 rawDsl 不是给业务绕过 Policy 的后门，只能由审计仓库、恢复软删除数据、运维修复脚本等少数 Service 用例使用，并且每一次调用都要留下操作者、原因和影响行数。

本章小结

  • 多租户隔离：PolicyProvider 声明式注入行级安全， TenantFilter + ThreadLocal 从 HTTP 请求传递租户身份， 对开发者代码完全透明。

  • 审计日志：RecordListener 自动填充 created_at/by、updated_at/by 四个审计字段； ExecuteListener 记录完整 SQL 审计追踪。

  • 慢查询检测：通过 ExecuteListener 记录查询耗时到 Micrometer，配合 Prometheus/Grafana 实现实时告警。

  • 批量操作：batch() 打包多条 INSERT 为单次请求， ON CONFLICT ... DO UPDATE 实现幂等 UPSERT， 多值 INSERT 进一步提升吞吐。

  • Keyset 分页：用 WHERE id > ? 替代 OFFSET，深分页性能保持恒定——百万行数据仍可 2ms 返回。

  • 乐观锁：Settings.withExecuteWithOptimisticLocking(true) 自动在 UPDATE 中附加 WHERE version = ?，配合 @Retryable 优雅处理冲突。

  • 软删除：Policy 自动为所有 SELECT 添加 deleted_at IS NULL，物理删除变为标记删除。

  • 测试：Testcontainers 启动与生产一致的 PostgreSQL， @JooqTest 切片测试快速反馈，MockDataProvider 做纯单元测试。

  • 监控：Micrometer 采集查询计数/耗时/错误， HealthIndicator 检查数据库连通性和连接池水位，Grafana 面板一目了然。

  • 生产检查清单：从连接池到 SQL 安全、从性能到运维，18 项逐一验证。
