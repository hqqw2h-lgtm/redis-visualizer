第 20 章 · SQL 解析器

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

jOOQ 不只是查询构建器,它内置覆盖常见 DML / DDL 的 SQL 解析器,能把受支持的 SQL 字符串解析为类型安全的 QueryPart 表达式树,然后用任意目标方言重新渲染。这意味着可以把 MySQL 写的 SQL 自动翻译成 PostgreSQL、Oracle、SQLite,甚至在不修改应用代码的情况下实现透明的方言转换。

本章覆盖三件事:Parser API 的入口与解析管线、方言翻译引擎的工作流程、ParsingConnection 的透明拦截。前置:第 12 章(方言渲染)与第 19 章(QOM)。

20.2.2 代码示例

Settings settings = new Settings()
    .withParseDialect(ParseDialect.MYSQL);

DSLContext mysql = DSL.using(SQLDialect.MYSQL, settings);
Parser parser = mysql.parser();

ResultQuery<?> query = parser.parseSelect(
    "SELECT id, name, price FROM product " +
    "WHERE price > 100 ORDER BY name LIMIT 10"
);

DSLContext postgres = DSL.using(SQLDialect.POSTGRES);
String pgSql = postgres.renderInlined(query);
System.out.println(pgSql);

输出:

SELECT id, name, price
FROM product
WHERE price > 100
ORDER BY name
LIMIT 10

Condition cond = parser.parseCondition(
    "status = 'ACTIVE' AND created_at > DATE '2024-01-01'");

var result = postgres
    .select(PRODUCT.ID, PRODUCT.NAME)
    .from(PRODUCT)
    .where(cond)
    .fetch();

Queries ddl = parser.parse(

    "CREATE INDEX idx_name ON t (name);");

20.3.3 复合迁移示例

String mysqlSource = """
    SELECT IFNULL(`nickname`, CONCAT(`first_name`, ' ', `last_name`)) AS `display_name`
    FROM `customers`
    WHERE `deleted` = 0
    LIMIT 5
    """;

Query query = mysql.parser().parseSelect(mysqlSource);

String postgresSql  = DSL.using(SQLDialect.POSTGRES).renderInlined(query);
String sqlServerSql = DSL.using(SQLDialect.SQLSERVER).renderInlined(query);

PostgreSQL 输出:

SELECT COALESCE("nickname", "first_name" || ' ' || "last_name") AS "display_name"
FROM "customers"
WHERE "deleted" = 0
LIMIT 5

SQL Server 输出:

SELECT TOP 5 ISNULL([nickname], [first_name] + ' ' + [last_name]) [display_name]
FROM [customers]
WHERE [deleted] = 0

20.4 ParsingConnection — 透明翻译

ParsingConnection 是一个 JDBC Connection 包装器,拦截所有通过它执行的 SQL,自动解析并用目标方言重新渲染后再发送给底层数据库。应用代码无需修改。

图 20-4:透明翻译时序

图 20-4 · 透明翻译时序 — ParsingConnection 在 JDBC 层拦截 / 翻译 / 透传

20.4.1 配置

DataSource pgDataSource = ...;

Settings settings = new Settings()
    .withParseDialect(ParseDialect.MYSQL);

DSLContext dsl = DSL.using(pgDataSource, SQLDialect.POSTGRES, settings);
Connection parsingConn = dsl.parsingConnection();

try (Statement stmt = parsingConn.createStatement()) {
    ResultSet rs = stmt.executeQuery(
        "SELECT id, name FROM product LIMIT 10, 20");
}

实际发送到 PostgreSQL 的 SQL:

SELECT id, name FROM product LIMIT 20 OFFSET 10

20.4.2 Spring Boot 包装

@Configuration
public class ParsingDataSourceConfig {
    @Bean
    public DataSource parsingDataSource(DataSource original) {
        Settings settings = new Settings()
            .withParseDialect(ParseDialect.MYSQL);
        DSLContext dsl = DSL.using(original, SQLDialect.POSTGRES, settings);
        return new DataSource() {
            @Override
            public Connection getConnection() { return dsl.parsingConnection(); }
        };
    }
}

注意:ParsingConnection 拦截所有 SQL,包括 ORM 框架(如 Hibernate)生成的 SQL。如果 Hibernate 已针对目标方言生成正确 SQL,再次翻译可能引入二次转换错误。建议仅在纯原生 SQL 或 JDBC 直连场景使用。

20.6.1 错误处理模式

图 20-6-1 · 错误处理模式 — parseUnknownFunctions + 其它解析设置
Settings failSettings = new Settings().withParseDialect(ParseDialect.MYSQL);
DSLContext failDsl = DSL.using(SQLDialect.POSTGRES, failSettings);

try {
    failDsl.parser().parseSelect("SELECT id, name FROM product WHERE");
} catch (ParserException e) {
    System.err.println("解析失败: " + e.getMessage());
}

异常输出:

解析失败: Token ')' expected: [1:24] SELECT id, name FROM product WHERE

Settings ignoreSettings = new Settings()
    .withParseDialect(ParseDialect.MYSQL)
    .withParseUnknownFunctions(ParseUnknownFunctions.IGNORE);

DSLContext ignoreDsl = DSL.using(SQLDialect.POSTGRES, ignoreSettings);
var query = ignoreDsl.parser().parseSelect("SELECT CUSTOM_FUNC(id) FROM t");
System.out.println(ignoreDsl.renderInlined(query));

输出:

SELECT CUSTOM_FUNC(id) FROM t

性能提示:jOOQ 解析器针对生产 SQL 做了优化,普通 SELECT 解析开销远低于一次数据库往返。但不要把 SQL 解析放进逐行热路径,迁移工具、启动期 SQL 校验、CI 批量扫描更适合它。

20.7 陷阱与限制

表 20-7:解析器支持覆盖度

20.7.1 DDL 保真度

DDL 翻译保真度显著低于 DML。MySQL 的 AUTO_INCREMENT、PostgreSQL 的 SERIAL、Oracle 的 SEQUENCE + TRIGGER 在概念上等价,但所有约束、注释、存储选项无法保证完整保留。

String mysqlDdl = """
    CREATE TABLE product (
      id BIGINT AUTO_INCREMENT PRIMARY KEY,
      name VARCHAR(100) NOT NULL,
      price DECIMAL(10, 2),
      created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
    ) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4
    """;

Queries parsed = mysql.parser().parse(mysqlDdl);
String pgDdl   = postgres.renderInlined(parsed);
System.out.println(pgDdl);

PostgreSQL 初稿:

CREATE TABLE product (
  id bigint GENERATED BY DEFAULT AS IDENTITY,
  name varchar(100) NOT NULL,
  price decimal(10, 2),
  created_at timestamp DEFAULT CURRENT_TIMESTAMP,
  CONSTRAINT product_pkey PRIMARY KEY (id)
);

DDL 审查输出:

[OK]     AUTO_INCREMENT -> GENERATED BY DEFAULT AS IDENTITY
[OK]     PRIMARY KEY / NOT NULL / DECIMAL precision preserved
[REVIEW] ENGINE=InnoDB removed, PostgreSQL has no engine clause
[REVIEW] DEFAULT CHARSET=utf8mb4 removed, set at database encoding instead

20.7.2 CI 批量分流

record CheckResult(String file, String status, String detail) {}

List<String> scripts = List.of(
    "001-products.sql::SELECT * FROM product LIMIT 10, 20",
    "002-ddl.sql::CREATE TABLE product (id BIGINT AUTO_INCREMENT PRIMARY KEY)",
    "003-load.sql::LOAD DATA INFILE 'product.csv' INTO TABLE product"
);

for (String item : scripts) {
    String[] parts = item.split("::", 2);
    String file = parts[0];
    String sql = parts[1];

    try {
        Queries parsed = mysql.parser().parse(sql);
        String translated = postgres.renderInlined(parsed);
        if (sql.contains("AUTO_INCREMENT")) {
            System.out.printf("[REVIEW] %s -> DDL translated, storage options need review%n", file);
        } else {
            System.out.printf("[OK]     %s -> %s%n", file, translated);
        }
    } catch (ParserException e) {
        System.out.printf("[FAIL]   %s -> %s%n", file, e.getMessage());
    }
}

输出:

[OK]     001-products.sql -> SELECT * FROM product LIMIT 20 OFFSET 10
[REVIEW] 002-ddl.sql -> DDL translated, storage options need review
[FAIL]   003-load.sql -> Unsupported query type: LOAD DATA

20.8.1 启动期 SQL 校验

@PostConstruct
void verifySqlScripts() {
    Path dir = Paths.get("src/main/resources/db/sql");
    try (var stream = Files.walk(dir)) {
        stream.filter(p -> p.toString().endsWith(".sql"))
              .forEach(this::verifyOne);
    }
}

private void verifyOne(Path path) {
    String sql = Files.readString(path);
    try {
        mysqlParser.parse(sql);
    } catch (ParserException e) {
        throw new IllegalStateException(
            "SQL parse failed at " + path + ": " + e.getMessage(), e);
    }
}

设计取舍

取舍 1:手写递归下降

不使用 ANTLR / JavaCC。代价是手写代码量大,收益是与 jOOQ AST 紧密贴合,易做方言分支与精细错误恢复。

取舍 2:解析与渲染解耦

一次解析可以输出多种目标方言。代价是方言边界要靠 golden SQL 覆盖,收益是迁移工具可以复用同一棵树渲染多个目标。

取舍 3:ParsingConnection 透明拦截

JDBC 层包装,业务代码无感。代价是运行时翻译增加延迟,收益是迁移过渡期可以无侵入地把旧 SQL 适配到新数据库。

取舍 4:不做语义验证

解析器只关心语法,不验证表 / 列存在。代价是可写出语法对、语义错的 SQL,收益是不依赖 schema 也能解析,方便离线工具链。

取舍 5:存储过程不在范围内

只覆盖 DML 与基本 DDL。代价是涉及存储过程的迁移要靠手工或第三方工具,收益是核心范围保持小而稳定。
