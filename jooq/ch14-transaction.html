第 14 章 · 事务与连接管理

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

一条 SQL 能执行成功只是起点;多条 SQL 要么全部成功、要么全部回滚 — 这才是生产级代码的底线。jOOQ 没有重新发明事务轮子,而是通过 TransactionProvider SPI 将事务控制委托给底层实现 — 可以是 JDBC 原生事务,也可以是 Spring PlatformTransactionManager。

本章按"SPI → 生命周期 → 嵌套与 Savepoint → Spring 集成 → 连接池 → 陷阱 → 守护 → 源码深潜"展开。事务的核心理念是"派生 Configuration":事务内 DSL.using(configuration) 创建的 DSLContext 绑定同一连接、同一 TransactionProvider 状态,通过 TransactionListener 暴露生命周期。

14.1.2 Lambda 事务 API

dsl.transaction(configuration -> {
    DSLContext tx = DSL.using(configuration);

    Long orderId = tx.insertInto(ORDERS)
        .set(ORDERS.CUSTOMER_ID, customerId)
        .set(ORDERS.AMOUNT, amount)
        .returning(ORDERS.ID)
        .fetchOne(ORDERS.ID);

    tx.insertInto(ORDER_ITEMS, ORDER_ITEMS.ORDER_ID, ORDER_ITEMS.PRODUCT_ID)
      .values(orderId, productId)
      .execute();
});

Long newOrderId = dsl.transactionResult(configuration -> {
    DSLContext tx = DSL.using(configuration);
    return tx.insertInto(ORDERS)
             .set(ORDERS.CUSTOMER_ID, customerId)
             .set(ORDERS.AMOUNT, amount)
             .returning(ORDERS.ID)
             .fetchOne(ORDERS.ID);
});

PostgreSQL 输出:

begin;
insert into orders (customer_id, amount) values (?, ?) returning orders.id;
insert into order_items (order_id, product_id) values (?, ?);
commit;

Lambda 内必须用 DSL.using(configuration) 创建事务专属 DSLContext,该实例绑定事务连接。直接使用外部 dsl 会在不同连接上执行,脱离事务范围。

14.2.2 auto-commit vs 显式事务

// 反例:auto-commit 无原子性
dsl.insertInto(ORDERS).set(ORDERS.AMOUNT, amount).execute();   // 自动 COMMIT
dsl.insertInto(ORDER_ITEMS).set(...).execute();               // 失败则订单已提交

// 正例:显式事务保证原子性
dsl.transaction(configuration -> {
    DSLContext tx = DSL.using(configuration);
    tx.insertInto(ORDERS).set(ORDERS.AMOUNT, amount).execute();
    tx.insertInto(ORDER_ITEMS).set(...).execute();
});

图 14-3:嵌套状态机

图 14-3 · 嵌套事务状态机 — Savepoint 实现部分回滚

14.3.1 部分回滚示例

dsl.transaction(outerConfig -> {
    DSLContext otx = DSL.using(outerConfig);

    Long orderId = otx.insertInto(ORDERS)
       .set(ORDERS.CUSTOMER_ID, 1001L)
       .set(ORDERS.AMOUNT, new BigDecimal("199.99"))
       .returning(ORDERS.ID)
       .fetchOne(ORDERS.ID);

    try {
        otx.transaction(innerConfig -> {
            DSLContext itx = DSL.using(innerConfig);
            itx.insertInto(ORDER_ITEMS)
               .set(ORDER_ITEMS.ORDER_ID, orderId)
               .set(ORDER_ITEMS.PRODUCT_ID, 99999L)  // 不存在
               .execute();
        });
    } catch (DataAccessException e) {
        log.warn("明细插入失败,改写默认条目", e);

        otx.insertInto(ORDER_ITEMS)
           .set(ORDER_ITEMS.ORDER_ID, orderId)
           .set(ORDER_ITEMS.PRODUCT_ID, DEFAULT_PRODUCT_ID)
           .execute();
    }
});

PostgreSQL 输出:

begin;
insert into orders (customer_id, amount) values (1001, 199.99);
savepoint jooq_savepoint_1;
insert into order_items (order_id, product_id) values (10042, 99999);
rollback to savepoint jooq_savepoint_1;
insert into order_items (order_id, product_id) values (10042, 1);
commit;

14.3.2 三种结果

图 14-3-2 · 嵌套事务三种结果 — 成功 / 部分回滚 / 全部回滚

表 14-3:嵌套层与回滚范围

层级 JDBC 操作 回滚范围
第 1 层(外层) setAutoCommit(false) 全部 SQL
第 2 层 setSavepoint("sp_1") sp_1 之后
第 3 层 setSavepoint("sp_2") sp_2 之后
第 N 层 setSavepoint("sp_(N-1)") sp_(N-1) 之后

不 catch 内层异常 = 整体回滚。Savepoint 只是事务内的检查点,RELEASE SAVEPOINT 不等于 COMMIT。

14.3.3 nested 标志

new DefaultTransactionProvider(connectionProvider, true);   // 嵌套用 Savepoint
new DefaultTransactionProvider(connectionProvider, false);  // 内层加入外层事务

nested=false 时内层 transaction() 不创建 Savepoint,行为类似 Spring REQUIRED 传播。

14.4 Spring 集成

图 14-4:全链路

图 14-4 · Spring 集成全链路 — @Transactional 与 jOOQ 共享 Connection

14.4.1 配置

@Configuration
public class JooqConfig {

    @Bean
    public DefaultDSLContext dslContext(DataSource ds, PlatformTransactionManager tx) {
        DefaultConfiguration cfg = new DefaultConfiguration();
        cfg.set(SQLDialect.POSTGRES);
        cfg.set(new DataSourceConnectionProvider(
            new TransactionAwareDataSourceProxy(ds)));
        cfg.set(new SpringTransactionProvider(tx));
        return new DefaultDSLContext(cfg);
    }
}

@Service
public class OrderService {
    @Autowired private DSLContext dsl;

    @Transactional
    public void createOrder(Long customerId, List<OrderItemDTO> items) {
        OrdersRecord order = dsl.newRecord(ORDERS);
        order.setCustomerId(customerId);
        order.store();

        for (OrderItemDTO item : items) {
            dsl.insertInto(ORDER_ITEMS)
               .set(ORDER_ITEMS.ORDER_ID, order.getId())
               .set(ORDER_ITEMS.PRODUCT_ID, item.productId())
               .execute();
        }
    }
}

表 14-4:Spring vs jOOQ 原生

场景 Spring? @Transactional? 推荐
无框架 否 - dsl.transaction() + DefaultTransactionProvider
Spring 声明式 是 是 @Transactional + SpringTransactionProvider
Spring 编程式 是 否 dsl.transaction() + SpringTransactionProvider

14.4.2 TransactionAwareDataSourceProxy

该 Spring 代理类确保 DataSource.getConnection() 返回 Spring 事务管理器绑定到当前线程的连接,而非从池新借。缺失会让 jOOQ 在事务外的连接上执行 SQL,直接破坏事务原子性。

14.6.2 长事务反例

// 反例:事务内 HTTP
dsl.transaction(cfg -> {
    DSLContext tx = DSL.using(cfg);
    tx.insertInto(ORDERS).set(...).execute();
    HttpResponse resp = httpClient.send(req, ...);  // 5 s+
    tx.update(ORDERS).set(ORDERS.PAYMENT_STATUS, resp.body()).execute();

// 正例:先 I/O 再事务

    DSLContext tx = DSL.using(cfg);
    tx.insertInto(ORDERS).set(...).execute();
    tx.update(ORDERS).set(ORDERS.PAYMENT_STATUS, resp.body()).execute();

14.8 测试护栏

14.8.1 提交与回滚断言

@Test
void transaction_commits_on_success() {
    dsl.transaction(cfg -> {
        DSL.using(cfg).insertInto(ORDERS)
           .set(ORDERS.ID, 1001L).set(ORDERS.AMOUNT, BigDecimal.TEN)
           .execute();
    });

    assertThat(dsl.fetchExists(ORDERS, ORDERS.ID.eq(1001L))).isTrue();
}

@Test
void transaction_rolls_back_on_exception() {
    assertThatThrownBy(() -> dsl.transaction(cfg -> {
        DSL.using(cfg).insertInto(ORDERS)
           .set(ORDERS.ID, 1002L).set(ORDERS.AMOUNT, BigDecimal.TEN)
           .execute();
        throw new RuntimeException("rollback me");
    })).isInstanceOf(RuntimeException.class)
       .hasMessage("rollback me");

    assertThat(dsl.fetchExists(ORDERS, ORDERS.ID.eq(1002L))).isFalse();
}

14.8.2 嵌套部分回滚

@Test
void nested_savepoint_partially_rolls_back() {
    dsl.transaction(outer -> {
        DSLContext otx = DSL.using(outer);
        otx.insertInto(LOG).set(LOG.MSG, "step1").execute();

        try {
            otx.transaction(inner -> {
                DSL.using(inner).insertInto(LOG).set(LOG.MSG, "step2-bad").execute();
                throw new RuntimeException("inner fail");
            });
        } catch (RuntimeException ignored) {
        }

        otx.insertInto(LOG).set(LOG.MSG, "step3").execute();
    });

    List<String> msgs = dsl.select(LOG.MSG).from(LOG).fetch(LOG.MSG);
    assertThat(msgs).containsExactly("step1", "step3");
}

14.8.3 长事务时长断言

@Test
@Timeout(value = 200, unit = TimeUnit.MILLISECONDS)
void transaction_completes_under_threshold() {
    long start = System.currentTimeMillis();
    dsl.transaction(cfg -> {
        DSLContext tx = DSL.using(cfg);
        tx.insertInto(ORDERS).set(ORDERS.ID, 1003L)
          .set(ORDERS.AMOUNT, BigDecimal.TEN).execute();
    });
    assertThat(System.currentTimeMillis() - start).isLessThan(200);
}

设计取舍

取舍 1:TransactionProvider SPI 而非内置事务

不绑定 Spring / JTA / JDBC 任一模型,业务可按部署形态选择实现。代价是自研 provider 必须严格保证 begin / commit / rollback 对称性,否则连接泄漏。

取舍 2:派生 Configuration 而非线程绑定

Lambda 内传入派生 Configuration,与外部上下文隔离。该设计支持并发事务(同线程内多个事务、异步事务),代价是业务必须自觉使用 DSL.using(configuration) 而非外层 dsl。

取舍 3:嵌套映射 Savepoint 而非独立事务

JDBC 不支持真正的嵌套事务,SQL Server / PG / MySQL 都用 Savepoint。jOOQ 的内层 transaction() 自然映射 Savepoint;若需独立事务边界,使用 Spring REQUIRES_NEW 或 outbox 模式。

取舍 4:事务不托管外部 I/O

事务 lambda 可执行任意代码,但 jOOQ 不强制限制 I/O。设计假定开发者了解长事务危害;生产应通过静态扫描或代码审查禁止事务内 HTTP / 文件 / 消息操作。

取舍 5:TransactionListener 用于观察而非改写

Listener 只能观察 begin / commit / rollback 事件,无法改变结果。该约束保证事务语义的可预测性;改写需求应在更上层(TransactionProvider 实现 / Spring TX Manager)解决。
