第 3 章 · DSLContext 与 Configuration

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

DSLContext 是 jOOQ 运行期的唯一入口。所有查询构造、SQL 渲染、参数绑定、JDBC 执行、Record 映射都从它开始。但 DSLContext 本身不持有连接、不维护事务,它只是 Configuration 之上的薄门面,把所有运行期决策委托给 Configuration 中的可插拔组件。

Configuration 是 jOOQ 的运行期依赖注入容器,持有七类组件:ConnectionProvider、TransactionProvider、ExecuteListenerProvider、RecordMapperProvider、ConverterProvider、Settings、SQLDialect。该七元组完整描述一次 SQL 执行的所有可定制行为。本章逐一拆解 DSLContext 的实例化路径、Configuration 的 derive 模型、SQL 渲染管线、Settings 调参面板、以及多数据源 / 多租户场景下的扩展模式。

3.1 DSLContext 的角色定位

3.2.2 组件对照

3.2.3 derive 模型

System.out.println(base.connectionProvider() == noSchema.connectionProvider());

3.2.4 derive 的不可变语义

Configuration.derive(...) 返回新实例,共享原配置中其他组件引用,仅替换显式提供的部分。该机制使多租户、多数据源、监控注入场景能在不污染全局 DSLContext 的前提下定制单次操作。

派生方式 替换组件 共享组件
derive(Settings) Settings 其他六类
derive(SQLDialect) Dialect 其他六类
derive(ConnectionProvider) 连接提供者 其他六类
derive(ExecuteListenerProvider) 监听器 其他六类

3.3 ConnectionProvider 协议

3.3.1 acquire 与 release 时序

图 3-3-1 · acquire / release 时序 — 一次完整的 SQL 执行生命周期

每次 SQL 执行触发一次完整的 acquire-execute-release。release 调用 Connection.close(),该方法在连接池下被解释为"归还到池",而非真正关闭物理连接。

3.3.2 两种内置 ConnectionProvider

Provider 连接来源 生命周期 适合场景 并发风险
DataSourceConnectionProvider DataSource / 池 每次 acquire / release 生产、Spring 低
DefaultConnectionProvider 单 Connection 调用方关闭 测试、脚本 高

3.3.3 Spring 事务感知集成

DSLContext ctx = DSL.using(
    new DefaultConfiguration()
        .set(dataSource)
        .set(SQLDialect.POSTGRES)
        .set(new SpringTransactionProvider())
        .set(new DataSourceConnectionProvider(
            new TransactionAwareDataSourceProxy(dataSource))));

Spring Boot 自动配置中,DataSourceConnectionProvider 包装的是 TransactionAwareDataSourceProxy,使 DataSourceUtils.getConnection() 能取到当前线程绑定的连接。事务未结束时 release 不真正关闭连接,而是交由 Spring 事务管理器统一管理。

3.3.4 jOOQ transaction API 与派生配置

ctx.transaction(configuration -> {
    DSLContext tx = DSL.using(configuration);

    int inserted = tx.insertInto(CUSTOMERS)
        .columns(CUSTOMERS.ID, CUSTOMERS.NAME, CUSTOMERS.STATUS)
        .values(10L, "Ada", "ACTIVE")
        .execute();

    int updated = tx.update(ORDERS)
        .set(ORDERS.CUSTOMER_ID, 10L)
        .where(ORDERS.ID.in(1L, 2L))
        .execute();

    System.out.println("inserted=" + inserted + " updated=" + updated);
});

[DEBUG] o.j.t.LoggerListener -- Begin transaction
[DEBUG] o.j.t.LoggerListener -- Executing query : insert into "customers" ...
[DEBUG] o.j.t.LoggerListener -- Executing query : update "orders" ...
[DEBUG] o.j.t.LoggerListener -- Commit transaction
inserted=1 updated=2

transaction(...) 不是把外层 DSLContext 变为事务对象。jOOQ 给闭包传入派生 Configuration,绑定当前事务连接;闭包内必须用该派生配置创建新 DSLContext,否则 SQL 执行不在事务上下文。

3.4 SQL 渲染管线

3.4.1 渲染流程

图 3-4-1 · SQL 渲染管线 — AST 两次遍历产出 SQL 文本 + bind values

3.4.2 访问者模式核心

每个 QueryPart(Field、Table、Condition、Param)实现 accept(Context),将自身片段追加到 RenderContext 的内部 StringBuilder。该模式使 SQL 文本输出与参数收集在两次遍历中独立完成。

3.4.3 Indexed 与 Inline 对比

SelectConditionStep<Record1<String>> q = ctx
    .select(ORDERS.STATUS)
    .from(ORDERS)
    .where(ORDERS.ID.eq(1L));

System.out.println("indexed: " + q.getSQL());
System.out.println("binds  : " + q.getBindValues());
System.out.println("inlined: " + q.getSQL(ParamType.INLINED));

indexed: select "orders"."status" from "orders" where "orders"."id" = ?
binds  : [1]
inlined: select "orders"."status" from "orders" where "orders"."id" = 1
ParamType 输出形式 用途 注入风险
INDEXED(默认) ? 生产执行 安全
INLINED 字面量 调试 / 复制 不安全
NAMED :p1 命名参数驱动 安全
NAMED_OR_INLINED 命名或内联 兼容驱动 视场景

3.4.4 render 与 execute 边界

Query rename = ctx.update(ORDERS)
    .set(ORDERS.STATUS, "CANCELLED")
    .where(ORDERS.ID.eq(1L));

System.out.println("rendered: " + rename.getSQL());
System.out.println("binds   : " + rename.getBindValues());

int rows = rename.execute();
System.out.println("affected: " + rows);

[DEBUG] o.j.t.LoggerListener -- Executing query : update "orders" set "status" = ? where "orders"."id" = ?
[DEBUG] o.j.t.LoggerListener -- -> with bind values : [CANCELLED, 1]
rendered: update "orders" set "status" = ? where "orders"."id" = ?
binds   : [CANCELLED, 1]
affected: 1

getSQL() 仅渲染,不接触 JDBC。execute() 才触发 acquire 连接、绑定参数、调用 PreparedStatement.executeUpdate、release 连接的完整周期。该分离使单元测试可以在不连接数据库的前提下断言 SQL 形状。

3.6 多数据源与多租户

3.6.1 多数据源拓扑

图 3-6-1 · 多数据源拓扑 — 每个数据源独立 DSLContext + 独立连接池

DSLContext 是轻量对象,创建开销可忽略。多数据源场景应为每个库实例化独立 DSLContext,而非共享单一"万能 Context"。

3.6.2 ThreadLocal 多租户

图 3-6-2 · ThreadLocal 多租户时序 — TenantFilter 在请求入口注入 + finally 清理
public class TenantContext {
    private static final ThreadLocal<DSLContext> CTX = new ThreadLocal<>();
    public static void set(DSLContext ctx) { CTX.set(ctx); }
    public static DSLContext dsl()         { return CTX.get(); }
    public static void clear()             { CTX.remove(); }
}

@Override
public void doFilter(ServletRequest req, ServletResponse res, FilterChain chain)
        throws IOException, ServletException {
    String tenantId = ((HttpServletRequest) req).getHeader("X-Tenant-Id");
    DataSource ds = tenantDataSources.get(tenantId);
    TenantContext.set(DSL.using(ds, SQLDialect.POSTGRES));
    try {
        chain.doFilter(req, res);
    } finally {
        TenantContext.clear();
    }
}

3.6.3 业务代码透明感知

public List<OrderDto> findOrders() {
    return TenantContext.dsl()
        .selectFrom(ORDERS)
        .fetchInto(OrderDto.class);
}

业务代码不感知租户。Filter 在请求入口注入,finally 清理避免 ThreadLocal 内存泄漏。该模式与 Spring Security 的 SecurityContextHolder 同构。

3.6.4 Virtual Threads 与 ScopedValue

JDK 21+ 虚拟线程数量可达数十万,ThreadLocal 模式仍可用但要警惕 remove() 必须执行。JDK 21 预览的 ScopedValue 自动跟随结构化并发的作用域,无需手动清理:

public static final ScopedValue<DSLContext> TENANT = ScopedValue.newInstance();

ScopedValue.where(TENANT, DSL.using(acmeDS, SQLDialect.POSTGRES))
           .run(() -> service.findOrders());

3.7 陷阱与限制

图 3-7:DSLContext 陷阱分类

图 3-7 · DSLContext 陷阱分类 — 4 大类 8 个高频问题

表 3-7:陷阱明细

# 陷阱 症状 根因 修复 监控指标
1 单 Connection 跨线程 偶发 SQL 异常 多线程共享同一连接 改用 DataSourceConnectionProvider 异常日志
2 闭包用旧 ctx 写不在事务 未用派生 Configuration DSL.using(configuration) 数据回滚测试
3 共享可变 Settings 配置漂移 修改已绑定 Settings 使用 derive 单元测试
4 ThreadLocal 未 clear 内存泄漏 finally 缺 remove try / finally 强制清理 heap dump
5 Settings 不生效 期望未达成 错配阶段(渲染 vs 执行) 查阶段对照表 snapshot 测试
6 derive 后污染 全局行为变化 修改 derive 来源 用 new Settings(orig) 拷贝 code review
7 Cursor 未关闭 连接耗尽 fetchLazy 后未 close try-with-resources 连接池告警
8 break 跳出 同上 for 循环 break 同上 同上

Cursor 泄漏反例与正例

// 反模式:Cursor 未关闭
Cursor<Record> cursor = ctx.selectFrom(ORDERS).fetchLazy();
for (Record r : cursor) {
    if (someCondition) break;
}

// 正模式:try-with-resources
try (Cursor<Record> cursor = ctx.selectFrom(ORDERS).fetchLazy()) {
    for (Record r : cursor) {
        if (someCondition) break;
    }
}

Settings 可变性反例

反模式:
  Settings shared = new Settings().withRenderSchema(true);
  DSLContext a = DSL.using(ds, POSTGRES, shared);
  DSLContext b = DSL.using(ds, POSTGRES, shared);
  shared.setRenderSchema(false);   // a 与 b 都被影响

正模式:
  Settings base = new Settings().withRenderSchema(true);
  DSLContext a = DSL.using(ds, POSTGRES, base);
  Configuration noSchemaCfg = a.configuration()
      .derive(new Settings(base).withRenderSchema(false));
  DSLContext b = DSL.using(noSchemaCfg);   // 互不污染

3.8 SQL Snapshot 守护

3.8.1 基础 snapshot 测试

@Test
void coreOrdersQuery_snapshot() {
    var query = ctx.select(ORDERS.ID, ORDERS.STATUS)
        .from(ORDERS)
        .where(ORDERS.ID.eq(1L));

    assertThat(query.getSQL())
        .isEqualToNormalizingWhitespace("""

            from "orders"

@ParameterizedTest
@EnumSource(value = SQLDialect.class,
    names = {"POSTGRES", "MYSQL", "H2", "ORACLE"})

3.8.3 Listener 事件顺序断言

final class TraceListener extends DefaultExecuteListener {
    final List<String> events = new ArrayList<>();

    @Override public void renderStart(ExecuteContext ctx) { events.add("renderStart"); }
    @Override public void bindEnd(ExecuteContext ctx)     { events.add("bindEnd"); }
    @Override public void executeEnd(ExecuteContext ctx)  { events.add("executeEnd"); }
}

@Test
void execute_lifecycle_observes_events() {
    var trace = new TraceListener();
    var local = base.configuration()
        .derive(new DefaultExecuteListenerProvider(trace))
        .dsl();

    local.select(ORDERS.ID).from(ORDERS).where(ORDERS.ID.eq(1L)).fetch();

    assertThat(trace.events)
        .containsSubsequence("renderStart", "bindEnd", "executeEnd");
}

设计取舍

取舍 1:门面与配置分离

DSLContext 仅作为 API 入口,所有可变状态压到 Configuration。该分离使 DSLContext 可以是单例线程安全对象,而 Configuration 通过 derive 实现局部覆盖。代价是排查问题时需同时检查门面引用与配置组件。

取舍 2:Provider SPI 全可替换

连接获取、事务管理、监听器、记录映射、类型转换均为 SPI。该设计使 jOOQ 与 Spring、HikariCP、自定义连接池透明集成,代价是初学者面对 7 类接口学习曲线陡峭。

取舍 3:渲染在运行期发生

同一 AST 可由不同 DSLContext 渲染为不同方言 SQL。该机制使 SQL 重写、方言迁移、跨库测试成为可能,代价是 SQL snapshot 测试必须固定 SQLDialect 与 Settings。
