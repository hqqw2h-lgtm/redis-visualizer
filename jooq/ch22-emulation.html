第 22 章 · SQL 子句模拟引擎

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

"Write SQL once, run on any database" 不是口号,而是 jOOQ 模拟引擎每天在生产环境默默完成的工作。当一段 DSL 写下 isDistinctFrom() 而目标库是 Oracle、当 limit(10).offset(20) 被发到没有原生 LIMIT 的方言,jOOQ 通常不会要求改写 DSL,而是在幕后将 SQL 意图翻译成目标数据库能理解的等价语义——这就是 SQL 子句模拟引擎。

本章覆盖:模拟引擎的设计哲学、四大高频模拟(LIMIT / OFFSET、MERGE、IS DISTINCT FROM、Boolean)、高级结构模拟(MULTISET、ARRAY、ROW、LATERAL)、性能代价与跨数据库测试策略。前置:第 12 章(SQL Dialect Engine)。

22.1 模拟引擎的设计哲学

SQL 标准定义了统一语法,但每款数据库的支持范围千差万别。jOOQ 的策略不是"最小公倍数"(只用所有库都支持的语法),而是按特性检测、按能力翻译:检查目标方言是否原生支持该 SQL 特性;若支持则直接渲染,若不支持则拆解为语义等价的替代构造。

22.2 LIMIT / OFFSET 模拟

分页是最常见的 SQL 需求之一,但 LIMIT x OFFSET y 并非 SQL 标准,源自 MySQL / PostgreSQL,其他数据库各有方言。

表 22-2:方言对照矩阵

方言 典型渲染 备注
SQL:2008 标准 OFFSET n ROWS FETCH NEXT m ROWS ONLY 现代方言优先
PostgreSQL / MySQL LIMIT m OFFSET n 原生支持
Oracle 12c+ OFFSET n ROWS FETCH NEXT m ROWS ONLY 原生分页子句
Oracle 11g 双层子查询 + ROWNUM 必须先排序

22.2.1 共享 DSL

Result<Record> result = dsl
    .select(CUSTOMER.NAME, CUSTOMER.SALARY)
    .from(CUSTOMER)
    .orderBy(CUSTOMER.SALARY.desc())
    .limit(10).offset(20)
    .fetch();

PostgreSQL / MySQL:

SELECT customer.name, customer.salary
FROM customer
ORDER BY customer.salary DESC
LIMIT 10 OFFSET 20

Oracle 12c+:

SELECT customer.name, customer.salary
FROM customer
ORDER BY customer.salary DESC
OFFSET 20 ROWS
FETCH NEXT 10 ROWS ONLY

Oracle 11g:

SELECT *
FROM (
  SELECT t.*, ROWNUM rn
  FROM (
    SELECT customer.name, customer.salary
    FROM customer
    ORDER BY customer.salary DESC
  ) t
  WHERE ROWNUM <= 30
)
WHERE rn > 20

SQL Server 2012+:

SELECT customer.name, customer.salary
FROM customer
ORDER BY customer.salary DESC
OFFSET 20 ROWS
FETCH NEXT 10 ROWS ONLY

SQL Server 2008:

SELECT sub.name, sub.salary
FROM (
  SELECT customer.name, customer.salary,
         ROW_NUMBER() OVER (ORDER BY customer.salary DESC) AS rn
  FROM customer
) sub
WHERE sub.rn BETWEEN 21 AND 30
ORDER BY sub.rn

DB2:

SELECT customer.name, customer.salary
FROM customer
ORDER BY customer.salary DESC
OFFSET 20 ROWS
FETCH FIRST 10 ROWS ONLY

22.3 MERGE / UPSERT 模拟

"如果记录存在就更新,不存在就插入"——业务需求常见,但 SQL 标准的 MERGE 在各数据库实现差异堪称"方言鸿沟"最深的区域之一。

表 22-3:UPSERT 形态对比

方言 UPSERT 形态 关键差异
SQL 标准 / Oracle MERGE INTO target USING source ON ... 源表 / 源查询驱动
PostgreSQL INSERT ... ON CONFLICT (key) DO UPDATE 冲突目标必须明确
MySQL INSERT ... ON DUPLICATE KEY UPDATE 依赖唯一键 / 主键冲突
SQL Server MERGE INTO ... USING (VALUES ...) ... 注意并发与锁语义
SQLite 3.24+ INSERT ... ON CONFLICT (key) DO UPDATE 接近 PostgreSQL

图 22-3:mergeInto() 渲染路径

图 22-3 · mergeInto() 渲染路径 — UPSERT 5 种方言形态

22.3.1 共享 DSL

dsl.mergeInto(PRODUCT, PRODUCT.ID, PRODUCT.NAME, PRODUCT.PRICE)
   .key(PRODUCT.ID)
   .values(100, "Widget", 9.99)
   .execute();

PostgreSQL:

INSERT INTO product (id, name, price)
VALUES (100, 'Widget', 9.99)
ON CONFLICT (id)
DO UPDATE SET
  name  = EXCLUDED.name,
  price = EXCLUDED.price

MySQL:

INSERT INTO product (id, name, price)
VALUES (100, 'Widget', 9.99)
ON DUPLICATE KEY UPDATE
  name  = VALUES(name),
  price = VALUES(price)

Oracle:

MERGE INTO product t
USING (SELECT 100 id, 'Widget' name, 9.99 price FROM dual) s
ON (t.id = s.id)
WHEN MATCHED THEN UPDATE SET t.name = s.name, t.price = s.price
WHEN NOT MATCHED THEN INSERT (id, name, price) VALUES (s.id, s.name, s.price)

SQL Server:

MERGE INTO product AS t
USING (VALUES (100, 'Widget', 9.99)) AS s (id, name, price)
ON t.id = s.id
WHEN MATCHED THEN
  UPDATE SET name = s.name, price = s.price
WHEN NOT MATCHED THEN
  INSERT (id, name, price) VALUES (s.id, s.name, s.price);

SQLite:

INSERT INTO product (id, name, price)
VALUES (100, 'Widget', 9.99)
ON CONFLICT (id)
DO UPDATE SET
  name  = excluded.name,
  price = excluded.price

22.4 IS DISTINCT FROM 模拟

IS DISTINCT FROM 是 SQL 标准的 NULL 安全比较运算符。A IS DISTINCT FROM B 当且仅当 A ≠ B(含 NULL 比较)时返回 true。Oracle、SQL Server、MySQL 不一定原生支持;MySQL 通常使用 NULL 安全等值 <=>。

图 22-4:CASE 展开

图 22-4 · IS DISTINCT FROM CASE 展开 — 三值逻辑等价改写

22.4.1 共享 DSL

dsl.select(CUSTOMER.ID, CUSTOMER.NAME)
   .from(CUSTOMER)
   .where(CUSTOMER.MANAGER_ID.isDistinctFrom(CUSTOMER.DEPT_ID))
   .fetch();

PostgreSQL(原生):

SELECT customer.id, customer.name
FROM customer
WHERE customer.manager_id IS DISTINCT FROM customer.dept_id

Oracle / SQL Server(CASE):

SELECT customer.id, customer.name
FROM customer
WHERE CASE
  WHEN customer.manager_id IS NULL AND customer.dept_id IS NULL THEN 0
  WHEN customer.manager_id IS NULL OR  customer.dept_id IS NULL THEN 1
  WHEN customer.manager_id = customer.dept_id THEN 0
  ELSE 1
END = 1

MySQL(NULL 安全等值取反):

SELECT customer.id, customer.name
FROM customer
WHERE NOT (customer.manager_id <=> customer.dept_id)

表 22-4:三值逻辑回归

manager_id dept_id isDistinctFrom
NULL NULL false
NULL 10 true
10 NULL true
10 10 false
10 20 true

生成 SQL 尾部 = 1 因 Oracle / SQL Server 没有原生 boolean 类型,CASE 必须返回整数。这与下一节 Boolean 模拟直接关联。

22.6 其他模拟策略

22.6.1 MULTISET 嵌套集合

MULTISET 是 jOOQ 3.15+ 的标志性特性,几乎没有数据库原生支持 SQL 标准 MULTISET,jOOQ 必须翻译为 JSON_ARRAYAGG 或 XMLAGG。

图 22-6 · MULTISET 翻译 — JSON / XML / 字符串聚合三选一
var result = dsl
    .select(
        ORDER.ID,
        ORDER.CUSTOMER_ID,
        multiset(
            select(ORDER_LINE.PRODUCT_ID, ORDER_LINE.QTY)
                .from(ORDER_LINE)
                .where(ORDER_LINE.ORDER_ID.eq(ORDER.ID))
        ).as("lines"))
    .from(ORDER)
    .fetch();

PostgreSQL:

SELECT
  o.id,
  o.customer_id,
  (SELECT COALESCE(jsonb_agg(jsonb_build_array(v.product_id, v.qty)), jsonb_build_array())
   FROM (SELECT ol.product_id, ol.qty
         FROM order_line ol
         WHERE ol.order_id = o.id) v) AS lines
FROM orders o

MySQL 8:

SELECT
  o.id,
  o.customer_id,
  (SELECT COALESCE(JSON_ARRAYAGG(JSON_ARRAY(v.product_id, v.qty)), JSON_ARRAY())
   FROM (SELECT ol.product_id, ol.qty
         FROM order_line ol
         WHERE ol.order_id = o.id) v) AS lines
FROM orders o

Oracle:

SELECT
  o.id,
  o.customer_id,
  (SELECT COALESCE(JSON_ARRAYAGG(JSON_ARRAY(v.product_id, v.qty)), JSON_ARRAY())
   FROM (SELECT ol.product_id, ol.qty
         FROM order_line ol
         WHERE ol.order_id = o.id) v) lines
FROM orders o

22.6.3 JSON 构造

Field<JSON> summary = jsonObject(
    key("id").value(ORDER.ID),
    key("customer").value(ORDER.CUSTOMER_ID),
    key("status").value(ORDER.STATUS));

dsl.select(summary.as("summary"))
   .from(ORDER)
   .where(ORDER.ID.eq(100L))

SELECT jsonb_build_object(

  'customer', orders.customer_id,
  'status', orders.status) AS summary

SELECT JSON_OBJECT(
  'id', orders.id,

  KEY 'id'       VALUE orders.id,
  KEY 'customer' VALUE orders.customer_id,
  KEY 'status'   VALUE orders.status) AS summary
FROM orders
WHERE orders.id = 100

22.6.5 LATERAL / APPLY

图 22-6b · LATERAL / APPLY — 4 种方言渲染形态
var rows = dsl
    .select(ORDER.ID, field(name("x", "first_product")))
    .from(ORDER)
    .crossApply(
        select(ORDER_LINE.PRODUCT_ID.as("first_product"))
            .from(ORDER_LINE)
            .where(ORDER_LINE.ORDER_ID.eq(ORDER.ID))
            .orderBy(ORDER_LINE.POSITION)
            .limit(1)
            .asTable("x"))
    .fetch();

PostgreSQL:

SELECT o.id, x.first_product
FROM orders o
CROSS JOIN LATERAL (
  SELECT ol.product_id AS first_product
  FROM order_line ol
  WHERE ol.order_id = o.id
  ORDER BY ol.position
  FETCH FIRST 1 ROW ONLY
) x

SQL Server:

SELECT o.id, x.first_product
FROM orders o
CROSS APPLY (
  SELECT TOP 1 ol.product_id AS first_product
  FROM order_line ol
  WHERE ol.order_id = o.id
  ORDER BY ol.position
) x

MySQL(相关子查询):

SELECT
  o.id,
  (SELECT ol.product_id
   FROM order_line ol
   WHERE ol.order_id = o.id
   ORDER BY ol.position
   LIMIT 1) AS first_product
FROM orders o

22.9 源码深潜

模拟引擎不是独立的 SQL 字符串翻译器,它分散在 QueryPart.accept(Context<?>) 方法的方言分支中。每个语法节点最了解自己的语义,因此最适合决定原生渲染或改写。

表 22-9:模拟主题源码入口

模拟主题 源码入口 重点函数
LIMIT / OFFSET org.jooq.impl.Limit、SelectQueryImpl.accept Limit.accept(ctx) 决定 LIMIT / FETCH;SQL Server TOP 位置搬移
MERGE / UPSERT MergeImpl、InsertQueryImpl key(...) / conflict target 影响 PG ON CONFLICT、MySQL ON DUPLICATE KEY
NULL 安全比较 IsDistinctFrom.accept 原生方言输出 IS DISTINCT FROM,其他展开 CASE 或 MySQL <=> 反向
Boolean 条件 Condition.accept、True / False 节点 TRUE / FALSE 映射 1/0 / BIT
MULTISET Multiset.accept、DefaultBinding SQL 渲染 JSON / XML 聚合,Java 通过 binding 解码
RETURNING InsertQueryImpl / UpdateQueryImpl / DeleteQueryImpl 原生 RETURNING、SQL Server OUTPUT、JDBC generated keys 选路
JSON / ARRAY / ROW JSONArrayAgg / JSONObject / Array / Row 节点 同结构按方言渲染为 JSON / XML / 拆解字段

图 22-9 · 模拟引擎源码地图 — 三条值得跟读的路径
跟读目标 推荐入口 关键观察点 为什么重要

22.10 架构设计取舍

模拟引擎有一个隐含边界:它优先模拟"语义可证明等价"的 SQL 结构,不承诺把任意现代 SQL 自动翻译到任意老数据库。

设计选择 被拒绝的方案 取舍
按节点局部模拟 字符串级全局替换 节点知道语义和绑定,字符串替换不知子查询 / 别名 / 参数顺序
方言能力内置渲染层 业务自写 if (dialect) 应用更干净,但需相信方言矩阵,关键路径必须有 SQL 快照
不强行模拟所有窗口函数 自连接 / 变量 / 临时表 部分可行,但 NULL / 排序 / 性能难保证
MULTISET 用 JSON / XML 多次查询客户端拼装 单 SQL 保事务一致性,可能增 JSON 聚合成本
保留 Plain SQL 逃生舱 禁止绕过 性能关键路径手写 SQL,但失部分可移植性

拒绝在 ExecuteListener.renderEnd() 里正则替换 SQL 修复语义。原因有三:渲染后的 SQL 已丢失 QueryPart 类型信息;替换可能命中子查询、字面量;绑定参数顺序已确定,插入新占位符破坏 PreparedStatement。需要全局约束,应在 DSL 工厂层或数据库视图 / 策略 / 触发器层做。

22.11 证据与回归护栏

模拟功能最容易出现"语法变了但测试没发现"的回归。测试不要只断言行数,还要断言 SQL 形状、绑定顺序、关键语义。

22.11.1 LIMIT / OFFSET 形状

var page = DSL.select(CUSTOMER.NAME, CUSTOMER.SALARY)
    .from(CUSTOMER)
    .orderBy(CUSTOMER.SALARY.desc())
    .limit(10).offset(20);

assertThat(DSL.using(POSTGRES).renderInlined(page)).contains("limit 10 offset 20");
assertThat(DSL.using(ORACLE).renderInlined(page))
    .contains("offset 20 rows").contains("fetch next 10 rows only");
assertThat(DSL.using(SQLSERVER).renderInlined(page))
    .contains("offset 20 rows").contains("fetch next 10 rows only");

22.11.2 NULL 安全比较

Condition c = CUSTOMER.MANAGER_ID.isDistinctFrom(CUSTOMER.DEPT_ID);

assertThat(DSL.using(POSTGRES).render(c)).contains("is distinct from");
assertThat(DSL.using(MYSQL).render(c)).contains("<=>");
assertThat(DSL.using(ORACLE).render(c).toLowerCase()).contains("case");

22.11.3 MULTISET 同时锁 SQL 与 Java 结构

var orders = DSL.select(
        ORDER.ID,
        multiset(
            select(ORDER_LINE.PRODUCT_ID, ORDER_LINE.QTY)
                .from(ORDER_LINE)
                .where(ORDER_LINE.ORDER_ID.eq(ORDER.ID))
        ).as("lines"))
    .from(ORDER)
    .where(ORDER.ID.eq(100L));

assertThat(DSL.using(POSTGRES).renderInlined(orders).toLowerCase()).contains("jsonb_agg");
assertThat(DSL.using(MYSQL).renderInlined(orders).toLowerCase()).contains("json_arrayagg");

var row = dsl.fetchOne(orders);
Result<Record2<Long, Integer>> lines = row.get("lines", Result.class);
assertThat(lines).extracting(r -> r.value2()).containsExactly(2, 1, 5);
