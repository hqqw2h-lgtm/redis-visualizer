第 7 章 · 高级 SQL

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

第 5 章与第 6 章覆盖了 SELECT 与 DML 的基础形态。真实业务中的排名、累计、层级遍历、Top-N per Group 等需求,需要进入 SQL 的"高级区":窗口函数、CTE、递归查询、派生表、LATERAL JOIN、集合操作。本章按"窗口 → CTE → 递归 → 派生表与 LATERAL → 集合操作 → 陷阱 → 源码"展开,所有特性以 jOOQ 类型安全 DSL 驱动,并对方言差异给出渲染对照。

高级 SQL 的工程价值有三层:第一,把复杂数据处理留在数据库,减少内存中的程序聚合;第二,把可读性从嵌套子查询提升到线性 CTE;第三,把方言差异收拢到 jOOQ 渲染层,业务代码只表达 SQL 意图。本章后半部分专门讨论这三层背后的源码与设计取舍。

7.1 窗口函数

7.1.1 窗口函数与 GROUP BY

维度 GROUP BY 窗口函数
输入 N 行 N 行
输出 M 行(M ≤ N) N 行
折叠 是 否
同时看明细 + 聚合 否 是
典型用途 汇总报表 排名 / 累计 / 偏移

窗口函数与 GROUP BY 的本质区别在于是否折叠行。窗口函数保留每行明细,在每行上附加基于"窗口(一组相关行)"的计算结果。该特性使一个查询可同时展示"本行价格"与"该作者所有书的均价"。

dsl.select(
       ORDERS.ID,
       ORDERS.CUSTOMER_ID,
       ORDERS.SALES,
       rank().over()
             .partitionBy(ORDERS.CUSTOMER_ID)
             .orderBy(ORDERS.SALES.desc())
             .as("sales_rank"),
       denseRank().over()
             .partitionBy(ORDERS.CUSTOMER_ID)
             .orderBy(ORDERS.SALES.desc())
             .as("dense_rank"))
   .from(ORDERS)
   .fetch();

select
  orders.id, orders.customer_id, orders.sales,
  rank() over (
    partition by orders.customer_id
    order by orders.sales desc) as sales_rank,
  dense_rank() over (
    partition by orders.customer_id
    order by orders.sales desc) as dense_rank
from orders

       lag(SALES.AMOUNT, 1).over()

select
  sales.month, sales.amount,
  lag(sales.amount, 1) over (order by sales.month) as prev_month,
  sales.amount - lag(sales.amount, 1) over (order by sales.month) as growth
from sales

7.1.6 窗口帧(Window Frame)

帧规范 jOOQ API 含义

7.2 CTE 公共表表达式

7.2.1 CTE 数据流

图 7-2-1 · CTE 数据流 — WITH 定义 + 主查询引用

CTE 通过 WITH ... AS (...) 命名子查询,主查询用名字引用。最大价值在于把深层嵌套子查询拆为线性步骤,使逻辑可按定义顺序自上而下阅读。

7.2.2 单 CTE 实例

var topCustomers      = name("top_customers").fields("customer_id", "total_sales");
var topCustomersTable = table(name("top_customers"));
Field<Long> topCustomersId = field(name("top_customers", "customer_id"), Long.class);
Field<BigDecimal> topTotalSales = field(name("top_customers", "total_sales"), BigDecimal.class);
Field<BigDecimal> totalSales = sum(ORDERS.SALES).as("total_sales");

dsl.with(topCustomers).as(
       select(ORDERS.CUSTOMER_ID.as("customer_id"), totalSales)
       .from(ORDERS)
       .groupBy(ORDERS.CUSTOMER_ID)
       .having(sum(ORDERS.SALES).gt(new BigDecimal("1000"))))
   .select(CUSTOMERS.NAME, topTotalSales)
   .from(CUSTOMERS)
   .join(topCustomersTable).on(CUSTOMERS.ID.eq(topCustomersId))
   .fetch();

with top_customers as (
  select orders.customer_id, sum(orders.sales) as total_sales
  from orders
  group by orders.customer_id
  having sum(orders.sales) > 1000)
select customers.name, top_customers.total_sales
from customers
join top_customers on customers.id = top_customers.customer_id

7.2.4 CTE 与子查询对比

维度 嵌套子查询 CTE
阅读顺序 内向外跳读 WITH 顺序自上而下
中间结果复用 重复书写 CTE 名引用
调试 手动拆出内层 单独运行 CTE
命名 通常无 命名清晰
优化器 内联 内联或物化

现代数据库(PG 12+ / MySQL 8.0+)优化器多数会内联 CTE,性能与等价子查询相当。CTE 的核心收益是可读性,尤其当逻辑超过 3 层嵌套时。

7.3.3 组织架构树

var orgTree = name("org_tree").fields("id", "name", "manager_id", "level");
var org = table(name("org_tree"));
Field<Long>    orgId        = field(name("org_tree", "id"), Long.class);
Field<String>  orgName      = field(name("org_tree", "name"), String.class);
Field<Long>    orgManagerId = field(name("org_tree", "manager_id"), Long.class);
Field<Integer> orgLevel     = field(name("org_tree", "level"), Integer.class);

dsl.withRecursive(orgTree).as(
       select(EMPLOYEE.ID, EMPLOYEE.NAME, EMPLOYEE.MANAGER_ID, val(0).as("level"))
       .from(EMPLOYEE)
       .where(EMPLOYEE.MANAGER_ID.isNull())
       .unionAll(
           select(EMPLOYEE.ID, EMPLOYEE.NAME, EMPLOYEE.MANAGER_ID,
                  orgLevel.plus(1))
           .from(EMPLOYEE)
           .join(org).on(EMPLOYEE.MANAGER_ID.eq(orgId))))
   .select(orgId, orgName, orgManagerId, orgLevel)
   .from(org)
   .orderBy(orgLevel, orgName)
   .fetch();

with recursive org_tree (id, name, manager_id, level) as (
  select id, name, manager_id, 0
  from employee
  where manager_id is null
  union all
  select e.id, e.name, e.manager_id, ot.level + 1
  from employee e
  join org_tree ot on e.manager_id = ot.id)
select * from org_tree
order by level, name

7.3.4 分类目录与面包屑

dsl.withRecursive(catTree).as(
       select(CATEGORY.ID, CATEGORY.NAME, CATEGORY.PARENT_ID,
              CATEGORY.NAME.as("path"), val(0).as("depth"))
       .from(CATEGORY)
       .where(CATEGORY.PARENT_ID.isNull())
       .unionAll(
           select(CATEGORY.ID, CATEGORY.NAME, CATEGORY.PARENT_ID,
                  concat(catPath, val(" / "), CATEGORY.NAME).as("path"),
                  catDepth.plus(1))
           .from(CATEGORY)
           .join(product).on(CATEGORY.PARENT_ID.eq(catId))))
   .select(catId, catName, catParentId, catPath, catDepth)
   .from(product)
   .fetch();

withRecursive 的"三件套":锚点 SELECT(WHERE 过滤根)、UNION ALL + 递归 SELECT(JOIN CTE 自身)、终止条件(无新行自动停)。jOOQ 把这三者映射为类型安全的链式 API。

7.4.3 LATERAL Top-N 示例

dsl.select(
       CUSTOMERS.NAME,
       field(name("top3", "id"), Long.class),
       field(name("top3", "sales"), Integer.class))
   .from(CUSTOMERS)
   .crossApply(
       select(ORDERS.ID, ORDERS.SALES)
       .from(ORDERS)
       .where(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
       .orderBy(ORDERS.SALES.desc())
       .limit(3)
       .asTable("top3"))
   .fetch();

-- PostgreSQL
select customers.name, top3.id, top3.sales
from customers,
lateral (
  select orders.id, orders.sales
  from orders
  where orders.customer_id = customers.id
  order by orders.sales desc
  limit 3) as top3

-- SQL Server
select customers.name, top3.id, top3.sales
from customers
cross apply (
  select top 3 orders.id, orders.sales
  from orders
  where orders.customer_id = customers.id
  order by orders.sales desc) as top3

7.5 集合操作

7.5.1 集合操作语义

操作 数学语义 去重 含义
UNION A ∪ B 是 合并去重
UNION ALL A ∪ B(含重复) 否 合并保留重复
INTERSECT A ∩ B 是 两边都有
INTERSECT ALL A ∩ B(含重复) 否 两边都有保留重复
EXCEPT A − B 是 A 有 B 没有
EXCEPT ALL A − B(含重复) 否 A 有 B 没有保留重复

7.5.2 集合操作示例

dsl.select(ORDERS.CUSTOMER_ID).from(ORDERS).where(ORDERS.YEAR.eq(2024))
   .union(
       select(ORDERS.CUSTOMER_ID).from(ORDERS).where(ORDERS.YEAR.eq(2025)))
   .fetch();

dsl.select(ORDERS.CUSTOMER_ID).from(ORDERS).where(ORDERS.YEAR.eq(2024))
   .intersect(
       select(ORDERS.CUSTOMER_ID).from(ORDERS).where(ORDERS.YEAR.eq(2025)))
   .fetch();

dsl.select(ORDERS.CUSTOMER_ID).from(ORDERS).where(ORDERS.YEAR.eq(2024))
   .except(
       select(ORDERS.CUSTOMER_ID).from(ORDERS).where(ORDERS.YEAR.eq(2025)))
   .fetch();

7.5.3 jOOQ API 与方言支持

操作 jOOQ API MySQL 支持
UNION .union() 全版本
UNION ALL .unionAll() 全版本

UNION 默认去重,底层需排序或哈希去重。若确认结果集无重复或不在意重复,使用 UNION ALL 可避免去重开销。

陷阱 1:窗口函数不能在 WHERE 中

SQL 标准规定窗口函数在 WHERE 与 HAVING 之后计算,因此不能直接 WHERE RANK() OVER(...) <= 3,需派生表包裹。

var ranked = table(
    select(ORDERS.fields(),
           rank().over()
               .partitionBy(ORDERS.CUSTOMER_ID)
               .orderBy(ORDERS.SALES.desc())
               .as("rk"))
    .from(ORDERS))
    .as("ranked");

dsl.select(ranked.fields())
   .from(ranked)
   .where(ranked.field("rk", Integer.class).le(3))
   .fetch();

select *
from (
  select orders.*,
         rank() over (partition by orders.customer_id
                      order by orders.sales desc) as rk
  from orders) as ranked
where ranked.rk <= 3

陷阱 3:CTE 物化与内联

方言 默认 控制
PG 11- 强制物化 无
PG 12+ 自动 AS MATERIALIZED / NOT MATERIALIZED
MySQL 8.0+ 自动倾向内联 优化器 hint
Oracle 自动 /*+ MATERIALIZE / INLINE */

PostgreSQL 11- 的 CTE 性能陷阱在于强制物化,过滤后取少量结果时会先物化巨大临时表再过滤。升级到 12+ 或手写子查询可解决。

设计取舍

取舍 1:节点化而非字符串模板

每个高级 SQL 特性都是 QueryPart 节点,而非追加字符串。该设计带来"重",但保证了类型检查、方言搬移、绑定顺序的统一性。字符串模板在 LIMIT 子句位置(SQL Server 的 TOP / Oracle 的 FETCH)就会失败。

取舍 2:方言判断在渲染层

crossApply 在 PostgreSQL 渲染为 CROSS JOIN LATERAL,在 SQL Server 渲染为 CROSS APPLY。该判断由 Configuration.dialect() 驱动,业务代码只表达意图。此设计避免业务层散落 if (dialect) 分支。

取舍 3:不自动优化高级查询

CTE 物化、LATERAL 嵌套循环、递归边界都依赖数据规模与执行计划。jOOQ 仅保证语义正确,性能由 EXPLAIN、索引、专项测试兜底。该边界使 jOOQ 与查询优化器保持职责分离。

7.10 本章要点

要点 验证依据 易错点
窗口函数不折叠行 OVER (...) 当 GROUP BY 用
RANK / ROW_NUMBER / DENSE_RANK 重复值处理 选错语义
LAG / LEAD 偏移 NULL 边界 忽略 NULL
窗口帧默认值 RANGE / ROWS 默认 frame 因数据库而异
CTE 提升可读性 WITH 顺序 期望性能
递归三件套 锚点 + 递归 + 终止 缺终止
LATERAL 引用左侧 CROSS APPLY 普通派生表
集合操作去重 UNION vs UNION ALL 误用 ALL
