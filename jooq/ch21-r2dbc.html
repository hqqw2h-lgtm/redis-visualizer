第 21 章 · R2DBC 响应式查询

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

JDBC 伴随 Java 走过二十多年,简单可靠,但有一个根本限制:每次查询阻塞当前线程。在高并发微服务时代,线程是昂贵资源——一千个并发请求需要一千个线程等待数据库返回。R2DBC(Reactive Relational Database Connectivity)用非阻塞 I/O 打破这个瓶颈;jOOQ 从 3.15 开始原生支持 R2DBC,让同一套类型安全 DSL 既可以走 JDBC 同步路径,也可以走 R2DBC 响应式路径。

本章覆盖:JDBC 阻塞模型与 R2DBC 非阻塞模型的本质差异、ConnectionFactory 配置、Flux<Record> / Mono<Record> 接入、背压机制、响应式事务、JDBC / R2DBC 共存策略、常见陷阱。前置:第 3 章(DSLContext)、第 14 章(事务)。

表 21-1:JDBC vs R2DBC 资源模型

维度 JDBC R2DBC
I/O 模型 调用线程等待数据库响应 非阻塞驱动以 Publisher 形式推送
线程占用 查询期间占用工作线程 通过事件循环 / 驱动线程推进

关键结论:R2DBC 并非全场景优于 JDBC。低并发场景下 JDBC 更直接;R2DBC 真正价值在高并发 + I/O 密集场景,例如 API 网关、实时推送、流式数据处理。

21.2.3 订阅生命周期

OrderHandler.list()
  -> Select<?> query = dsl.select(...).from(ORDERS)
     // only builds QueryPart tree; no connection yet
  -> Flux.from(query)
     // wraps jOOQ Publisher; still lazy
  -> WebFlux subscribes
     -> request(n)
     -> jOOQ obtains R2DBC Connection from ConnectionFactory
     -> render SQL + bind parameters to R2DBC Statement
     -> statement.execute()
     -> map Row -> Record -> DTO
     -> onComplete / onError / cancel closes connection

验证点:创建 query 时连接池 acquired = 0;订阅后增加;正常完成、异常、取消后必须回到 0。

// 1 响应式 SELECT
Flux<Record2<String, BigDecimal>> orders = Flux.from(dsl
    .select(ORDERS.ID, ORDERS.AMOUNT)
    .from(ORDERS)
    .where(ORDERS.AMOUNT.gt(BigDecimal.TEN))
    .orderBy(ORDERS.ID));

orders.subscribe(
    record -> log.info("{} -- ¥{}", record.get(ORDERS.ID), record.get(ORDERS.AMOUNT)),
    error  -> log.error("查询失败", error),
    ()     -> log.info("所有记录处理完毕")
);

// 2 响应式 INSERT
Mono<Integer> inserted = Mono.from(dsl
    .insertInto(ORDERS)
    .set(ORDERS.ID, "O1001")
    .set(ORDERS.AMOUNT, BigDecimal.valueOf(128.00))
    .set(ORDERS.CUSTOMER_ID, 1L));

inserted.subscribe(rows -> log.info("插入了 {} 行", rows));

// 3 响应式 UPDATE
Mono<Integer> updated = Mono.from(dsl
    .update(ORDERS)
    .set(ORDERS.AMOUNT, ORDERS.AMOUNT.mul(0.8))
    .where(ORDERS.ORDER_DATE.lt(LocalDate.of(2020, 1, 1))));

updated.subscribe(rows -> log.info("折扣更新影响 {} 行", rows));

// 4 WebFlux Controller
@GetMapping("/orders")
public Flux<OrderDto> listOrders() {
    return Flux.from(dsl
        .select(ORDERS.ID, ORDERS.AMOUNT, CUSTOMERS.NAME)
        .from(ORDERS)
        .join(CUSTOMERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID)))
        .map(r -> new OrderDto(
            r.get(ORDERS.ID),
            r.get(ORDERS.AMOUNT),
            r.get(CUSTOMERS.NAME)));
}

执行 SQL:

select orders.id, orders.amount
from orders
where orders.amount > ?
order by orders.id
-- bind: [10.00]

惰性求值:jOOQ 查询对象在 R2DBC 模式下是"蓝图",只有被 Flux.from / Mono.from 订阅后才真正执行。未订阅,SQL 永不发送。

21.4 背压机制

图 21-4:需求驱动的数据流

图 21-4 · 需求驱动的数据流 — request(n) 控制拉取节奏

表 21-4:背压策略

策略 行为 适用场景
limitRate(n) 每次向上游 request(n) 通用场景控制拉取
buffer(n) 攒满 n 个批量交付 批量写入 / 批量发送
onBackpressureBuffer(n) 缓冲 n 个,溢出回调 消费者偶尔慢
onBackpressureDrop() 来不及时丢弃 监控采样 / 实时流
onBackpressureLatest() 只保留最新 UI 状态 / 传感器

21.4.1 缓冲策略示例

Flux.from(dsl.select(ORDERS.ID, ORDERS.AMOUNT).from(ORDERS))
    .buffer(100)
    .subscribe(batch -> batchWriter.write(batch));

Flux.from(dsl.selectFrom(ORDERS))
    .limitRate(256)
    .map(record -> transform(record))
    .subscribe(result -> send(result));

Flux.from(dsl.selectFrom(ORDERS))
    .onBackpressureBuffer(1024,
        dropped -> log.warn("丢弃: {}", dropped))
    .subscribe(this::process);

背压操作符不会自动给 SQL 加 limit。它们改变 Reactive Streams 的 request(n) 信号,要限制结果集大小、分页或 keyset,必须在 DSL 里显式表达。

21.5.1 转账示例

Mono<Void> transfer = Mono.from(
    dsl.transactionPublisher(configuration -> {
        DSLContext tx = DSL.using(configuration);

        Mono<Integer> debit = Mono.from(tx
            .update(ACCOUNT)
            .set(ACCOUNT.BALANCE, ACCOUNT.BALANCE.minus(amount))
            .where(ACCOUNT.ID.eq(fromId)));

        Mono<Integer> credit = Mono.from(tx
            .update(ACCOUNT)
            .set(ACCOUNT.BALANCE, ACCOUNT.BALANCE.plus(amount))
            .where(ACCOUNT.ID.eq(toId)));

        Mono<Integer> log = Mono.from(tx
            .insertInto(TRANSFER_LOG)
            .set(TRANSFER_LOG.FROM_ID, fromId)
            .set(TRANSFER_LOG.TO_ID,   toId)
            .set(TRANSFER_LOG.AMOUNT,  amount));

        return Flux.concat(debit, credit, log).then();
    }));

transfer.subscribe(
    null,
    err  -> log.error("转账失败 已回滚", err),
    ()   -> log.info("转账成功 已提交")
);

commit;

21.6.1 Spring Boot 双 DSL 配置

@Configuration
public class JooqConfig {

    @Bean @Qualifier("jdbc")
    DSLContext jdbcDsl(DataSource dataSource) {
        return DSL.using(dataSource, SQLDialect.POSTGRES);
    }

    @Bean @Qualifier("r2dbc")
    DSLContext r2dbcDsl(ConnectionFactory connectionFactory) {
        return DSL.using(connectionFactory, SQLDialect.POSTGRES);
    }
}

@Service
public class OrderService {
    private final DSLContext jdbcDsl;
    private final DSLContext r2dbcDsl;

    public Flux<OrderDto> listOrdersReactive() {
        return Flux.from(r2dbcDsl
            .select(ORDERS.ID, ORDERS.AMOUNT)
            .from(ORDERS))
            .map(r -> new OrderDto(r.get(ORDERS.ID), r.get(ORDERS.AMOUNT)));
    }

    @Transactional
    public void importOrders(List<OrderDto> orders) {
        jdbcDsl.batchInsert(
            orders.stream()
                  .map(b -> jdbcDsl.newRecord(ORDERS, b))
                  .toList()
        ).execute();
    }
}

21.9.1 未订阅不得执行 SQL

@Test
void r2dbc_query_should_be_lazy_until_subscription() {
    var counter = new AtomicInteger();
    DSLContext traced = dsl.configuration()
        .derive(new DefaultExecuteListenerProvider(new DefaultExecuteListener() {
            @Override public void executeStart(ExecuteContext ctx) {
                counter.incrementAndGet();
            }
        })).dsl();

    Flux<Record1<Long>> flux = Flux.from(
        traced.select(ORDERS.ID).from(ORDERS).orderBy(ORDERS.ID));

    assertThat(counter).hasValue(0);

    StepVerifier.create(flux.take(1))
        .expectNextCount(1)
        .verifyComplete();

    assertThat(counter).hasValue(1);
}

订阅语义 未订阅时 listener 不触发 构建即拿连接

设计取舍

取舍 1:DSL 与 AST 复用

JDBC 与 R2DBC 共享 QueryPart 树。代价是测试必须覆盖 subscribe / cancel / error 三条分支,收益是迁移代价大幅降低。

取舍 2:Publisher 惰性执行

构建 query 不等于执行 query。代价是日志与事务必须在订阅链内观察,收益是与 Reactor / WebFlux 模型一致。

取舍 3:背压交给 Reactive Streams

减少线程阻塞与内存堆积。代价是数据库驱动是否真正尊重 request(n) 要压测验证,收益是高并发吞吐显著提升。

取舍 4:响应式事务绑定 Publisher 链

成功 commit、错误 rollback、cancel 释放连接。代价是不能在链内混入阻塞 JDBC 或 block(),否则 Event Loop 卡死。

取舍 5:不替代 JDBC

R2DBC 与 JDBC 共存而非取代。代价是双依赖、双配置,收益是渐进迁移与场景化选型。
