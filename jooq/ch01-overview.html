第 1 章 · jOOQ 全景

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

Java 应用访问关系数据库的主流路径分为三层抽象:JDBC 直接发送 SQL 字符串,ORM 框架以对象图为中心管理状态,SQL DSL 在保留 SQL 语义的基础上提供编译期类型检查。jOOQ 属于第三层,目标是把 SQL 表达式变成 Java 类型系统的一等公民,同时不接管对象生命周期。

本章建立判断 jOOQ 在架构中位置的整体框架。讨论顺序为:与 JDBC、ORM 的语义边界,Schema 到 Record 的代码生成管线,运行期 DSL 渲染流水线,选型决策矩阵,以及 Overview 阶段必须建立的回归守护底线。所有示例围绕 customers 与 orders 两表展开,在 PostgreSQL + jOOQ 3.19 环境下可重复运行。

1.1 三层抽象的语义边界

1.1.1 JDBC、ORM、SQL DSL 在抽象层的位置

JDBC 是数据库访问的最底层抽象,通过 PreparedStatement 发送 SQL 字符串并按列名或下标读取 ResultSet。ORM 框架在 JDBC 之上构建对象关系映射,将 SQL 隐藏在持久化上下文之后。SQL DSL 不隐藏 SQL,而是把 SQL 表达式建模为 Java 类型,编译期校验列名、参数类型与结果形状。

图 1-1-1 · JDBC / ORM / jOOQ 在抽象层的位置 — 三条路径都收敛到 JDBC Driver

1.1.2 三层抽象的特征对照

维度 JDBC JPA / Hibernate jOOQ
主要表达式 SQL 字符串 JPQL / Criteria / Entity Java DSL 表达式
编译期校验 无 Entity 字段层 列名 + 列类型
SQL 可见性 完全暴露 默认隐藏 完全暴露
对象状态管理 无 持久化上下文 无
方言适配 由调用方处理 由 Provider 处理 由 jOOQ 内置 28 种方言渲染器处理
主要风险 拼写错误延迟到运行期 N+1 / 隐式 flush 仍需阅读执行计划
适配场景 极少量查询 CRUD 聚合根 复杂读、报表、跨方言

1.1.3 同一查询的三种表达对照

JDBC 的表达直接而冗长,SQL 字符串与参数在编译期对编译器是不透明的。

String sql = """
    select c.name, o.id, o.total_amount
    from customers c
    join orders o on o.customer_id = c.id
    where c.status = ?
    order by o.created_at desc
    limit ?
    """;

try (PreparedStatement ps = connection.prepareStatement(sql)) {
    ps.setString(1, "ACTIVE");
    ps.setInt(2, 2);
    try (ResultSet rs = ps.executeQuery()) {
        while (rs.next()) {
            System.out.printf("%s #%d %.2f%n",
                rs.getString("name"),
                rs.getLong("id"),
                rs.getBigDecimal("total_amount"));
        }
    }
}

JPA 的 JPQL 表达从 Entity 关系出发,SQL 由 Provider 在运行期生成。

List<RecentOrder> rows = entityManager.createQuery("""
        select new com.example.RecentOrder(c.name, o.id, o.totalAmount)
        from Customer c join c.orders o
        where c.status = :status
        order by o.createdAt desc
        """, RecentOrder.class)
    .setParameter("status", "ACTIVE")
    .setMaxResults(2)
    .getResultList();

jOOQ 的 DSL 表达保留 SQL 形状,所有列引用在编译期被类型系统检查。

record RecentOrder(String customerName, long orderId, BigDecimal total) {}

List<RecentOrder> rows = dsl
    .select(CUSTOMERS.NAME, ORDERS.ID, ORDERS.TOTAL_AMOUNT)
    .from(CUSTOMERS)
    .join(ORDERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
    .where(CUSTOMERS.STATUS.eq("ACTIVE"))
    .orderBy(ORDERS.CREATED_AT.desc())
    .limit(2)
    .fetch(r -> new RecentOrder(
        r.get(CUSTOMERS.NAME),
        r.get(ORDERS.ID),
        r.get(ORDERS.TOTAL_AMOUNT)));

1.1.4 渲染产物对照

-- jOOQ 输出的 SQL(关闭引号与 schema 前缀)
select customers.name, orders.id, orders.total_amount
from customers
join orders on orders.customer_id = customers.id
where customers.status = ?
order by orders.created_at desc
limit ?
-- bind values: ["ACTIVE", 2]

jOOQ 输出的 SQL 与 JDBC 手写版本结构一致,差异仅在标识符引号与 schema 前缀策略,由 Settings 控制。该一致性使 jOOQ 查询能直接被 DBA 审阅与 EXPLAIN。

1.2 jOOQ 的设计前提

1.2.1 三条不可妥协的设计前提

图 1-2-1 · jOOQ 三条不可妥协的设计前提 — SQL / Schema / JDBC 优先

1.2.2 SQL 优先的具体表现

jOOQ 的每个 DSL 方法都映射到 SQL 中的一个语法元素。select 对应 SELECT 子句,from 对应 FROM,join...on 对应 INNER JOIN ... ON,leftJoin...onKey 在保留 LEFT JOIN 语义的同时使用 codegen 元数据推导外键条件。

DSL 方法 SQL 对应 备注
select(...) SELECT 列表 投影由 Field 序列定义
from(...) FROM 子句 接受 Table 与子查询
join(...).on(...) INNER JOIN ... ON 显式条件
leftJoin(...).onKey() LEFT JOIN ... ON fk codegen 提供外键
where(...) WHERE Condition 树
groupBy(...) GROUP BY 接受 Field 与表达式
having(...) HAVING 仅作用于聚合
orderBy(...) ORDER BY 含 asc/desc/nullsFirst
limit(n).offset(m) 方言相关分页 PostgreSQL 与 H2 用 LIMIT/OFFSET
forUpdate() FOR UPDATE MySQL 5.7 不支持 NOWAIT

1.2.3 Schema 优先的代价与收益

代码生成发生在构建期。Schema 是真相源头,Java 元模型是其投影。代价是 Schema 变更必须经过构建管线,收益是列名、列类型、外键、索引、唯一约束都被编译器与 IDE 直接感知。

Schema → codegen → Java 元模型对照:

  数据库定义                       Java 元模型
  ──────────────────────────────────────────────────────────
  customers.name VARCHAR(100)      CUSTOMERS.NAME : TableField<CustomersRecord, String>
  customers.id   BIGINT PK         CUSTOMERS.ID   : TableField<CustomersRecord, Long>
  orders.amount  NUMERIC(12,2)     ORDERS.TOTAL_AMOUNT : TableField<OrdersRecord, BigDecimal>
  fk customer_id → customers.id    Keys.ORDERS__FK_ORDERS_CUSTOMER_ID
  ──────────────────────────────────────────────────────────

1.2.4 JDBC 优先的执行边界

jOOQ 不重写 JDBC 协议。DSLContext 通过 ConnectionProvider 获取 Connection,在执行期把渲染后的 SQL 与绑定参数交给 PreparedStatement,然后从 ResultSet 读出结果并按 Record 或 DTO 形态返回。事务由调用方负责,可以是手动 setAutoCommit(false)、jOOQ transaction(...) API,或 Spring @Transactional。

1.3.3 字段引用不再是字符串

ORDERS.TOTAL_AMOUNT 的类型为 TableField<OrdersRecord, BigDecimal>,该类型携带表绑定与值类型两条静态信息。把金额字段当作字符串比较的代码无法通过编译。

var positivePaidOrders = dsl
    .select(ORDERS.ID, ORDERS.TOTAL_AMOUNT)
    .from(ORDERS)
    .where(ORDERS.TOTAL_AMOUNT.ge(new BigDecimal("100.00")))
    .orderBy(ORDERS.TOTAL_AMOUNT.desc());

select orders.id, orders.total_amount
from orders
where orders.total_amount >= ?

1.3.4 关系来自 codegen 元数据

onKey() 利用生成代码中的外键元数据推导 join 条件,渲染结果与显式 on(...) 完全一致。该方法在外键关系唯一时使用,有歧义时需要显式选择 Keys.ORDERS__FK_ORDERS_CUSTOMER_ID。

Field<Integer> orderCount = count(ORDERS.ID).as("order_count");

var summary = dsl
    .select(CUSTOMERS.NAME, orderCount)
    .from(CUSTOMERS)
    .leftJoin(ORDERS).onKey()
    .groupBy(CUSTOMERS.ID, CUSTOMERS.NAME)
    .orderBy(orderCount.desc());

select customers.name, count(orders.id) as order_count
from customers
left join orders on orders.customer_id = customers.id
group by customers.id, customers.name
order by order_count desc

将 customers.status 重命名为 customers.lifecycle_status 后,旧的 jOOQ 代码在 mvn compile 阶段直接失败。错误信号离 Schema 变更最近,修复成本最低。

1.4.3 运行期仍是普通 JDBC

适配新字段后,运行期路径不变。jOOQ 不绕过数据库执行计划,也不修改 JDBC 驱动行为。

System.out.println(q.getSQL(ParamType.INDEXED));
System.out.println(q.getBindValues());

select customers.name
from customers

-- bind values: ["ACTIVE"]

1.5.2 写读分离的混合架构

写路径常见使用 JPA 管理聚合根,读路径使用 jOOQ 表达精确查询。两者通过同一份 Spring 事务上下文协作,共享 DataSource 与 DataSourceTransactionManager。

@Service
public class OrderApplicationService {
    private final OrderRepository orderRepository;
    private final DSLContext dsl;

    @Transactional
    public OrderView createOrderAndReadView(CreateOrderCommand command) {
        Order order = orderRepository.save(Order.from(command));
        return dsl
            .select(ORDERS.ID, CUSTOMERS.NAME, ORDERS.TOTAL_AMOUNT)
            .from(ORDERS)
            .join(CUSTOMERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
            .where(ORDERS.ID.eq(order.getId()))
            .fetchOne(r -> new OrderView(
                r.get(ORDERS.ID),
                r.get(CUSTOMERS.NAME),
                r.get(ORDERS.TOTAL_AMOUNT)));
    }
}

# 陷阱 症状 根因 修复 监控指标
1 把 jOOQ 当 ORM 期望对象改字段后自动 update jOOQ 不维护持久化上下文 显式 update/insert/delete 业务校验

1.6.3 反例与正例对比

反模式:
  Service 方法:
    var rows = dsl.selectFrom(ORDERS)
                  .where(ORDERS.CUSTOMER_ID.eq(1L))
                  .fetch();
  测试:
    assertThat(rows).hasSize(2);
  问题:
    SQL 形状漂移不可见
    bind 顺序错不可见
    数据 fixture 变化即误报

正模式:
  关键查询单独抽出 buildXxxQuery() 方法返回 Query
  测试三段断言:

    2. assertThat(query.getBindValues()).containsExactly(...)
    3. assertThat(query.fetch()).hasSize(2)

1.7.2 三层守护的最小测试模板

private static String compactSql(String sql) {
    return sql.replaceAll("\\s+", " ").trim();
}

@Test
void recent_orders_query_keeps_sql_shape_and_bind_order() {
    var q = dsl
        .select(CUSTOMERS.NAME, ORDERS.ID, ORDERS.TOTAL_AMOUNT)
        .from(CUSTOMERS)
        .join(ORDERS).on(ORDERS.CUSTOMER_ID.eq(CUSTOMERS.ID))
        .where(CUSTOMERS.STATUS.eq("ACTIVE"))
        .orderBy(ORDERS.CREATED_AT.desc())
        .limit(2);

    assertThat(compactSql(q.getSQL(ParamType.INDEXED))).isEqualTo(
        "select customers.name, orders.id, orders.total_amount " +
        "from customers " +
        "join orders on orders.customer_id = customers.id " +
        "where customers.status = ? " +
        "order by orders.created_at desc " +
        "limit ?");

    assertThat(q.getBindValues()).containsExactly("ACTIVE", 2);
}

Record / DTO 显式映射 持久化上下文 不接管对象生命周期 写路径需显式写出
复用 JDBC 协议 自建数据库协议 复用驱动 / 连接池 / 事务 性能瓶颈仍在数据库
Configuration 不可变 全局可变单例 支持多租户与 derive derive 是横切配置的标准入口

1.8.4 扩展边界

扩展点 推荐入口 不推荐做法
列类型映射 Converter / Binding / codegen forcedTypes 业务代码到处手动 String → Enum
SQL 渲染规则 Settings / VisitListener 拼字符串后再正则替换
查询审计 / 指标 ExecuteListener 每个 Repository 重复写日志
多租户 / 多数据源 Configuration.derive() 修改全局单例的 Settings
自定义函数 封装 Field<T> / Condition 命名 helper 直接 field(String) 拼用户输入

1.8.5 边界守护断言

@Test
void jooq_query_keeps_sql_and_bind_separate() {
    var q = dsl
        .select(CUSTOMERS.NAME)
        .from(CUSTOMERS)
        .where(CUSTOMERS.STATUS.eq("ACTIVE"));

    assertThat(q.getSQL(ParamType.INDEXED)).contains("customers.status = ?");
    assertThat(q.getBindValues()).containsExactly("ACTIVE");
    assertThat(q.getSQL(ParamType.INDEXED)).doesNotContain("ACTIVE");
}

该断言的意义在于守护 jOOQ 的底层心智模型:SQL 结构进入渲染管线,用户值进入绑定管线,两者在 PreparedStatement 层才合流。任何破坏该不变量的实现都会让安全性回到 JDBC 字符串拼接的水平。

1.9 性能特征与开销分布

1.9.1 阶段开销分布

单查询开销分布(典型 OLTP,P50):

  阶段                  耗时     占比      说明
  ──────────────────────────────────────────────────────
  DSL → QueryPart 树     5-50 μs   1%-2%   纯 Java 对象构造
  Render → SQL 文本      20-100 μs 2%-5%   字符串拼接 + 引号策略
  Bind → 参数列表        5-30 μs   1%-2%   值收集
  PreparedStatement      0.1-1 ms  10%-15% JDBC 驱动 + 连接池
  网络 + 数据库执行       1-10 ms   70%-85% 主要耗时
  ResultSet → Record     0.1-2 ms  5%-10%  类型转换
  ──────────────────────────────────────────────────────
  合计                   1.5-13 ms 100%

1.9.2 开销定位策略

现象 优先排查 工具
单查询慢 数据库执行计划 EXPLAIN ANALYZE
大批量慢 bind 数量 / batch 配置 dsl.batch(...)
启动慢 codegen 输出量 includes/excludes 收紧
内存高 大结果集全量加载 fetchLazy + Cursor
反射多 DTO 映射策略 record 类或显式 RecordMapper

1.9.3 jOOQ 的非性能优势

性能上 jOOQ 与 JDBC 同量级,真正的优势在于可维护性与可测试性。下列指标是 jOOQ 与字符串 SQL 在中长期维护中的差异。

指标 字符串 SQL jOOQ DSL
Schema 变更感知 运行期 SQL 异常 编译期 cannot find symbol
条件组合 字符串拼接 + 占位符计数 Condition 可组合
方言适配 多分支 if / 多份 SQL 文件 单一 DSL + 切换 SQLDialect
测试覆盖 仅可端到端 SQL snapshot + bind + 映射 三段
IDE 重构支持 字符串内重命名困难 字段 / 表的引用一致更新

1.10 本章要点

要点 验证依据 易错点
jOOQ 是 SQL DSL,不是 ORM DSLContext API + 生成的 Tables 期望对象状态自动管理
类型安全主要来自构建期 codegen target/generated-sources/jooq/** Schema 变更未重建
