第 16 章 · Schema 迁移管线

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

数据库 Schema 是 jOOQ 类型安全的根基:ORDERS.ID 之所以是 Field<String>,是因为 codegen 读取了真实的 VARCHAR 列定义。当 Schema 在演进时(加列、改类型、建索引),如果生成代码与数据库不同步,编译会断裂。本章构建一条黄金管线:让 Schema 迁移与代码生成像齿轮一样自动咬合。

核心契约:Schema 迁移是代码生成的唯一真相来源。所有 Schema 变更必须通过迁移脚本表达,确保开发、CI、预发布、生产环境使用完全相同的 DDL 演进路径。

16.2.1 端到端示例

迁移脚本(故意把 published_at 放在 V3,模拟真实增量演进):

-- V1__create_customers.sql
create table customers (
  id          bigint generated always as identity primary key,
  first_name  varchar(80) not null,
  last_name   varchar(80) not null,
  created_at  timestamp not null default current_timestamp
);

-- V2__create_orders.sql
create table orders (
  id          bigint generated always as identity primary key,
  customer_id bigint not null references customers(id),
  amount      numeric(10, 2) not null,
  status      varchar(20) not null default 'NEW'
);
create index idx_orders_customer_id on orders(customer_id);

-- V3__add_orders_published_at.sql
alter table orders add column published_at date;
create index idx_orders_published_at on orders(published_at);

        createField(DSL.name("id"), SQLDataType.BIGINT.nullable(false).identity(true), this);

    public final TableField<OrdersRecord, Long>       CUSTOMER_ID =

业务代码:

List<OrderSummary> fetchRecentOrders(DSLContext ctx, LocalDate since) {
    return ctx.select(ORDERS.ID, ORDERS.AMOUNT, ORDERS.STATUS, CUSTOMERS.LAST_NAME)
        .from(ORDERS)
        .join(CUSTOMERS).on(CUSTOMERS.ID.eq(ORDERS.CUSTOMER_ID))
        .where(ORDERS.PUBLISHED_AT.ge(since))
        .orderBy(ORDERS.PUBLISHED_AT.desc(), ORDERS.ID.desc())
        .fetch(r -> new OrderSummary(
            r.get(ORDERS.ID),
            r.get(ORDERS.AMOUNT),
            r.get(ORDERS.STATUS),
            r.get(CUSTOMERS.LAST_NAME)));
}

      <groupId>org.flywaydb</groupId>
      <artifactId>flyway-maven-plugin</artifactId>

      <executions>
        <execution>
          <phase>generate-sources</phase>

        </execution>
      </executions>
      <configuration>
        <url>${db.url}</url>
        <user>${db.user}</user>
        <password>${db.password}</password>
        <locations>
          <location>filesystem:src/main/resources/db/migration</location>
        </locations>
      </configuration>
    </plugin>

    <plugin>
      <groupId>org.jooq</groupId>
      <artifactId>jooq-codegen-maven</artifactId>
      <version>3.19.8</version>
      <executions>
        <execution>
          <phase>generate-sources</phase>
          <goals><goal>generate</goal></goals>
        </execution>
      </executions>
      <configuration>
        <jdbc>
          <url>${db.url}</url><user>${db.user}</user><password>${db.password}</password>
        </jdbc>
        <generator>
          <database><inputSchema>public</inputSchema></database>
          <target>
            <packageName>com.myapp.jooq</packageName>
            <directory>target/generated-sources/jooq</directory>
          </target>
        </generator>
      </configuration>
    </plugin>
  </plugins>
</build>

Maven 同 phase 多插件按声明顺序执行。Flyway 必须写在 jOOQ codegen 之前。

16.3 Liquibase 集成

表 16-3:Liquibase vs Flyway

维度 Flyway Liquibase
迁移单元 纯 SQL changeSet(XML/YAML/JSON/SQL)
版本表 flyway_schema_history DATABASECHANGELOG
回滚 手写反向脚本(Teams 才有 Undo) changeSet 可声明 rollback
抽象层 贴近数据库 跨方言抽象 DDL
适用场景 单方言、SQL 团队 多方言、需要回滚声明

class SqlSnapshotTest {
    private final DSLContext ctx = DSL.using(SQLDialect.POSTGRES);

    @Test
    void recent_orders_query_sql_stays_stable() {
        var query = ctx.select(ORDERS.ID, ORDERS.AMOUNT, ORDERS.STATUS, CUSTOMERS.LAST_NAME)
            .from(ORDERS)
            .join(CUSTOMERS).on(CUSTOMERS.ID.eq(ORDERS.CUSTOMER_ID))

            select "orders"."id", "orders"."amount", "orders"."status", "customers"."last_name"
            from "orders"
            join "customers" on "customers"."id" = "orders"."customer_id"
            where "orders"."published_at" >= date '2026-01-01'
            order by "orders"."published_at" desc, "orders"."id" desc
            """);
    }
}

16.7.1 启动验证

@PostConstruct
void verifyMigration(DSLContext dsl, Flyway flyway) {
    var current = flyway.info().current();
    if (current == null || !current.getVersion().getVersion().equals(EXPECTED_VERSION)) {
        throw new IllegalStateException(
            "Schema version mismatch: expected=" + EXPECTED_VERSION
                + ", actual=" + (current == null ? "null" : current.getVersion()));
    }

    int orderCount = dsl.fetchCount(ORDERS);
    log.info("Migration verified: version={}, orders.count={}", current.getVersion(), orderCount);
}

设计取舍

取舍 1:迁移工具不内嵌进 jOOQ

jOOQ 选择只负责"读已迁移的 Schema",不实现自己的 migration runner。代价是用户必须组合两个工具,收益是与 Flyway / Liquibase 等成熟 migration 工具自由集成。

取舍 2:DDL-based codegen 牺牲完整性换零依赖

DDLDatabase 用 jOOQ SQL Parser 解析 DDL 在内存中构建 Schema 模型,不启动数据库。代价是 routines / UDT / 物化视图等无法还原,收益是开发循环从分钟级降至毫秒级。

取舍 3:Testcontainers 选择临时容器而非长驻 DB

每次构建启动新容器,确保 codegen 输入与迁移脚本完全一致。代价是 5-15 s 启动开销,收益是开发、CI、生产环境 100% 可复现。

取舍 4:生成代码的版本控制取决于团队

jOOQ 不强制策略 A 或 B。两种策略各有充分理由,差异在于团队对 PR 可读性、仓库体积、CI 速度的权衡。

取舍 5:同 phase 多插件按声明顺序执行

Maven / Gradle 都没有"先 migrate 后 codegen"的约定。jOOQ 不试图自动在 codegen 前注入 migrate,而是要求用户显式声明顺序或 dependsOn。代价是多一处配置,收益是不破坏构建工具的常规语义。
