第 8 章 · SQL 函数与表达式

v7 图解版 · 自动从 v6 markdown 生成 · Mermaid 图原样保留

SQL 的表达力集中在函数与表达式中。CONCAT 拼接字符串、SUM 汇总金额、CAST 转换类型、调用数据库私有的存储过程,这些操作在原生 SQL 中是裸字符串,在 jOOQ 中是类型安全的 Java 方法调用,且最终生成的 SQL 由方言层负责适配。

本章按"源码入口 → 五大内置函数 → 聚合与分组 → 类型强制 → Plain SQL → 存储过程与 UDT → 陷阱 → 测试护栏"展开。每个函数节点最终都进入同一渲染管线,理解这条管线是后续章节(第 10 章 Converter / Binding、第 12 章 Dialect、第 18 章 QueryPart)的基础。

8.2 聚合与 GROUP BY

8.2.1 GROUP BY 数据流

图 8-2-1 · GROUP BY 数据流 — 原始行 → 分组 → 聚合 → HAVING → 最终行
阶段 操作 与其他子句关系
WHERE 过滤原始行 在 GROUP BY 之前

Aggregate 聚合计算 函数级
HAVING 过滤聚合结果 在 GROUP BY 之后

8.2.2 基本 GROUP BY 实例

ctx.select(
       ORDERS.CUSTOMER_ID,
       DSL.count().as("order_count"),
       DSL.sum(ORDERS.AMOUNT).as("total_amount"))
   .from(ORDERS)
   .groupBy(ORDERS.CUSTOMER_ID)
   .having(DSL.count().ge(5))
   .orderBy(DSL.sum(ORDERS.AMOUNT).desc())
   .fetch();

select orders.customer_id,
       count(*) as order_count,
       sum(orders.amount) as total_amount
from orders
group by orders.customer_id
having count(*) >= 5
order by sum(orders.amount) desc

8.2.3 ROLLUP / CUBE / GROUPING SETS

写法 产生分组 适用
ROLLUP(A, B) (A,B) → (A) → () 层级报表
CUBE(A, B) (A,B), (A), (B), () 多维透视
GROUPING SETS((A,B),(A),()) 显式列出 精准控制 
ctx.select(ORDERS.REGION, ORDERS.PRODUCT_ID,
           DSL.sum(ORDERS.AMOUNT).as("subtotal"))
   .from(ORDERS)
   .groupBy(DSL.rollup(ORDERS.REGION, ORDERS.PRODUCT_ID))
   .fetch();

ctx.select(ORDERS.REGION, ORDERS.CATEGORY,
           DSL.sum(ORDERS.AMOUNT).as("subtotal"))
   .from(ORDERS)
   .groupBy(DSL.cube(ORDERS.REGION, ORDERS.CATEGORY))
   .fetch();

-- ROLLUP
select orders.region, orders.product_id, sum(orders.amount)
from orders
group by rollup (orders.region, orders.product_id)

-- 输出: 明细行 + (region 小计 product=NULL) + (总计 region=NULL)

8.2.4 GROUPING 函数判断小计

ctx.select(

8.3 类型强制函数

8.3.1 cast 与 coerce 决策

图 8-3-1 · cast vs coerce 决策 — 数据库转换 vs Java 类型断言

8.3.2 cast 与 coerce 对比

维度 cast coerce
生成 SQL CAST(x AS type) 无
转换执行 数据库 Java(JDBC 隐式)
失败模式 SQL 异常 ClassCastException
性能 可能影响索引 零成本
用途 真实类型转换 兼容类型断言

8.3.3 cast 实例

Field<Integer>    numericId = CUSTOMERS.EXTERNAL_ID.cast(Integer.class);
Field<BigDecimal> rounded   = ORDERS.AMOUNT.cast(SQLDataType.NUMERIC(10, 2));
Field<String>     idStr     = DSL.cast(CUSTOMERS.ID, String.class);

select cast(customers.external_id as integer)   as numeric_id,
       cast(orders.amount as numeric(10, 2))    as rounded,
       cast(customers.id as varchar)            as id_str
from customers, orders

8.3.4 coerce 实例与风险

// 数据库返回 Number,业务断言为 Long
Field<Long> id = LEGACY_TABLE.ID.coerce(Long.class);

// 与自定义 Converter 协作
Field<MyEnum> status = ORDERS.STATUS_CODE
    .coerce(SQLDataType.VARCHAR)
    .convert(MyEnumConverter.INSTANCE);

coerce 不生成 SQL,仅在 Java 侧改变 Field<T> 的泛型。运行时若数据库返回值无法转换为目标类型,将抛 ClassCastException。仅在确信类型兼容时使用。

8.3.5 cast 破坏索引的反例

// 反例:CAST 阻止索引使用
ctx.selectFrom(CUSTOMERS)
   .where(CUSTOMERS.CREATED_AT.cast(SQLDataType.DATE)
          .eq(LocalDate.of(2024, 6, 15)))
   .fetch();

// 正例:半开区间
LocalDate target = LocalDate.of(2024, 6, 15);
ctx.selectFrom(CUSTOMERS)
   .where(CUSTOMERS.CREATED_AT.ge(target.atStartOfDay()))
   .and(CUSTOMERS.CREATED_AT.lt(target.plusDays(1).atStartOfDay()))
   .fetch();

8.4.5 封装可复用 helper

public final class CustomDSL {
    public static Field<String> jsonExtract(Field<?> json, String path) {
        return DSL.field("JSON_EXTRACT({0}, {1})",
                String.class, json, DSL.val(path));
    }

    public static Condition jsonContains(Field<?> json, String value) {
        return DSL.condition("JSON_CONTAINS({0}, {1})",
                json, DSL.val(value));
    }
}

图 8-6:函数陷阱分类

表 8-6:陷阱明细

# 陷阱 现象 根因 修复
1 Plain SQL 注入 DROP TABLE 字符串拼接 {0} + val + 白名单
2 函数名方言差异 跨库报错 手写函数名 用 DSL.* 内置

关键反例与正例

// 反例:NULL 影响 AVG
DSL.avg(ORDER_ITEMS.QUANTITY);
// 数据 [5, NULL, 3, NULL, 4] → AVG = 4.0(忽略 NULL)

// 正例:把 NULL 当作 0 参与平均
DSL.avg(DSL.coalesce(ORDER_ITEMS.QUANTITY, 0));
// → AVG = (5+0+3+0+4)/5 = 2.4

// 反例:CAST 列
.where(CUSTOMERS.CREATED_AT.cast(SQLDataType.DATE).eq(target));

// 正例:半开区间
.where(CUSTOMERS.CREATED_AT.ge(target.atStartOfDay()))
.and(CUSTOMERS.CREATED_AT.lt(target.plusDays(1).atStartOfDay()));

8.7 测试护栏

8.7.1 cast 与 coerce SQL 形状断言

@Test
void cast_and_coerce_have_different_sql_shapes() {
    DSLContext pg = DSL.using(SQLDialect.POSTGRES);

    Field<Integer> casted  = CUSTOMERS.EXTERNAL_ID.cast(Integer.class);
    Field<Integer> coerced = CUSTOMERS.EXTERNAL_ID.coerce(Integer.class);

    assertThat(pg.select(casted).from(CUSTOMERS).getSQL(ParamType.INLINED))
        .containsIgnoringCase("cast(")
        .containsIgnoringCase("as integer");

    assertThat(pg.select(coerced).from(CUSTOMERS).getSQL(ParamType.INLINED))
        .doesNotContainIgnoringCase("cast(");
}

8.7.2 Plain SQL 注入断言

@Test
void plain_sql_keeps_input_as_bind() {
    String input = "'; DROP TABLE customers; --";
    Field<String> ext = DSL.field("MY_FUNC({0})", String.class, DSL.val(input));

    var query = ctx.select(ext).from(CUSTOMERS);
    assertThat(query.getSQL(ParamType.INDEXED))
        .contains("?")
        .doesNotContain("DROP TABLE");
    assertThat(query.getBindValues()).contains(input);
}

8.8 源码深潜

图 8-8:函数渲染内部架构

图 8-8 · 函数渲染内部架构 — DSL 工厂 → Field 节点 → AST → 双访问器输出
文件 职责

org.jooq.impl.RoutineImpl 存储过程调用

设计取舍

取舍 1:函数实现 Field<T> 而非返回字符串

每个函数节点继续实现 Field<T>,可参与 select / having / orderBy / as / cast / convertFrom 链式调用。被拒绝的方案是"每个函数返回 SQL 字符串":实现简单,但失去类型、方言重写、绑定参数与 AST 访问能力。

取舍 2:方言差异藏在渲染层

concat / substring / length / dateAdd 等的方言差异由 DefaultRenderContext 翻译。业务代码只需 DSL.concat(...),无需感知 PostgreSQL || 与 MySQL CONCAT 的差异。

取舍 3:cast 与 coerce 分离

cast 在数据库层执行真转换;coerce 仅在 Java 侧改变 Field<T> 泛型。分离的代价是用户需要理解两者差异,但收益是性能与安全语义清晰。

取舍 4:Plain SQL 模板而非字符串拼接

{0}, {1} 占位符 + QueryPart 参数,使 jOOQ 能在不修改 SQL 文本的前提下完成绑定与方言识别。该设计避免了 SQL 注入,同时保留了对数据库私有函数的调用能力。

8.9 本章要点

要点 验证依据 易错点
五大函数返回 Field<T> 编译期类型 当返回 SQL 字符串
GROUP BY 在 WHERE 之后 子句顺序 把 HAVING 当 WHERE
ROLLUP 一次出小计 NULL 行 不区分汇总与明细
cast 在数据库执行 SQL 含 CAST 误以为零成本
coerce 仅改 Java 泛型 SQL 不变 期待数据库转换
Plain SQL 必须 {0} bind values 字符串拼接注入
函数包裹列破坏索引 EXPLAIN 期望仍命中索引
AVG 忽略 NULL 三值逻辑 期待按 0 平均
