第 37 章 · 性能优化实战

命令路径分段 · 基线建立 · 单线程瓶颈 · Pipeline 调参 · 编码与 Big Key · 持久化副作用 · 排查决策树 · perf / flame graph · 发布护栏

Redis 性能优化的目标不是抬高单次 redis-benchmark 的吞吐数字,而是把端到端延迟分解到具体源头:主线程是否被慢命令占住、网络是否在 RTT 空等、连接池是否积压、持久化是否引入 fork 毛刺、内存编码是否将对象撑胖。优化的每一步都必须能用"指标 → 源码入口 → 修复手段"的因果链解释清楚。

诊断框架可概括为"先找等待,再找放大器":先确认延迟来自 Redis 内部还是外部,再确认内部延迟由哪类操作放大。本章按命令路径(socket → 解析 → 执行 → 持久化 → 回复)展开,逐段给出指标、源码入口、修复工具。

  • 如何把一次命令拆成网络、解析、执行、持久化和回复五段来看 P99
  • 为什么优化前必须先建立包含分位数、副作用和数据形状的基线
  • 哪些问题用 SCAN/UNLINK/Pipeline 能缓解,哪些只是把成本换了位置
  • 何时从内建命令升级到 perf、flame graph 与 bpftrace

前置知识:第 5 章字符串与编码(理解 listpack / hashtable 切换);第 17-18 章 RDB/AOF(理解 fork / fsync 物理成本);第 21 章 RESP 协议(理解 processInputBuffer);第 28 章 Lua / Functions(理解脚本阻塞主线程);第 34 章 ACL(理解命令权限校验在 call() 前);Linux 性能分析基础(perf / bpftrace / 火焰图阅读)。

37.0 端到端命令路径与源码入口

核心结论: 一次 Redis 请求穿过的源码路径决定了性能问题的诊断起点。理解每段代码对应的指标,可避免"端到端慢"被误判为"命令慢"。推荐定位顺序是:网络 → 命令 → 内部事件 → 客户端 → 系统调用 → 火焰图。
案例化场景: 某团队接到告警"Redis P99 抖动到 80 ms",立刻把 SLOWLOG 全员调阈值到 1 ms,结果只看到几条 HGETALL 在 3 ms 左右。继续追 LATENCY DOCTOR 才发现是 fork 事件 850 ms — 业务在凌晨低峰窗口触发了 BGSAVE,服务端的 SLOWLOG 不会记录这种非命令延迟。教训:慢日志只看命令执行时间,不包括 fork、fsync、过期、淘汰、复制等内部事件,排查时必须把 SLOWLOG、LATENCY、--latency 三件套并列拉一遍。

37.1 基线与指标拆分

未建立基线的"性能优化"无法判断是否有效。基线必须固定四个维度:命令类型、value 大小、并发连接、key 分布。任何变更前后采集同一套指标,才能形成可复现的因果链。

基线必填字段

维度字段示例易错点
吞吐ops/s100 K仅记 avg,不带分位
延迟P50 / P95 / P99 / max0.5 / 1.0 / 1.4 / 8.9 ms"P99 漂亮但 max 巨大"
副作用fork、fsync、过期、淘汰、复制 backloglatest_fork_usec、aof_delayed_fsync只看吞吐忽略后台事件
负载连接数、pipeline 深度、CPU 单核占用50、1、72%CPU 总占低 ≠ 单核空
数据形状命令类型、value 大小、key 分布SET/GET、256B、1M 随机 key"压 GET 通过"不代表 HGETALL 通过

37.2 单线程瓶颈

Redis 主线程不是不能忙,而是不能被不可切分的工作长时间占住。单次大集合遍历、单次同步删除、单次全库扫描都会让后续轻量请求排队。优化的关键是把不可切分的工作拆成可切分的(SCAN 替代 KEYS,HMGET 替代 HGETALL),或者推到后台线程(UNLINK 替代 DEL)。

图 37-2-1 · 单线程事件循环阻塞路径

典型阻塞命令与替代方案

命令复杂度阻塞原因替代方案
KEYS patternO(N)全库扫描SCAN 游标分批
HGETALL key(大 hash)O(N)整个 hash 遍历HMGET 字段读取或 HSCAN
SMEMBERS key(大 set)O(N)整个 set 遍历SSCAN
LRANGE key 0 -1(大 list)O(N)全量返回LRANGE 分页
DEL bigkeyO(N)同步释放UNLINK、lazyfree-lazy-user-del yes
FLUSHALL(大 db)O(N)同步清空FLUSHALL ASYNC
DEBUG SLEEP nO(N)主线程 sleep生产禁用
SORT 大集合O(N+M·log M)排序占主线程预排好的 ZSet
长 Lua / FCALL视脚本脚本运行期独占拆短 + 限时

37.4 Pipeline 与网络往返

Pipeline 通过批量发送和批量读取响应减少 RTT 次数。该机制不改变单条命令复杂度,只压缩客户端与服务端间的等待空隙。批次过大时,尾延迟与输出缓冲区会反向损耗收益。

37.5 内存编码与 Big Key

Redis 命令的实际成本由对象编码决定。HGET 看似 O(1),底层数据结构若为 hashtable,涉及指针追逐与内存分配;若为 listpack,则连续内存读取。编码切换点由配置阈值决定,跨越阈值的对象路径完全变化。

37.6 持久化副作用

RDB、AOF rewrite、AOF fsync 均不是"后台无感"。fork() 复制页表,CoW 放大内存压力,fdatasync() 与块设备调度共同决定 P99。理解这三条物理路径是评估持久化策略的前提。

37.7 网络与系统调用

每次命令穿过客户端序列化 → TCP 栈 → Redis RESP 解析 → 命令执行 → 输出缓冲 → 内核发送队列。Pipeline、批量命令、连接复用减少这条路径上的无效往返。慢消费者会让输出缓冲反向积压,导致实例 OOM 风险。

设计取舍

取舍 1 · 命令执行单线程。Redis 命令在主线程串行执行,获得简单一致性语义。代价是热 key 必致单核饱和,无法通过加 CPU 核数线性扩展。该选择源于"内存数据库读写极快,锁竞争反而成为瓶颈"的工程观察。

取舍 2 · I/O threads 仅做读写。Redis 6.0 引入 I/O threads,仅并行化 RESP 解析与回复编码,命令执行仍单线程。该选择保留命令语义简单性,同时让网络密集场景(高 QPS、大 payload)可受益于多核。命令计算密集场景(Lua、Sort)收益有限。

取舍 3 · Pipeline 不改变命令复杂度。Pipeline 仅压缩 RTT,服务端仍逐条执行命令。该设计避免了"命令依赖前一条结果"的复杂性,代价是单一 O(N) 命令仍占住主线程。批量操作命令(MGET、HMGET)是 Pipeline 的语义化版本。

取舍 4 · LATENCY 事件采样而非全量。LATENCY 仅记录超过阈值的事件,不维护每请求耗时直方图。该设计开销极低(几乎可忽略),代价是无法回答"P99.99 是多少"。需要分位数时,需在客户端或外部 APM 维护直方图。
