第 1 章 · 全景概览

JPA 增强 · 类型安全 · Entity View · CriteriaBuilder · 框架对比

JPA 是 Java 持久化的标准，但当查询复杂度上升——子查询嵌套、CTE、窗口函数、DTO 投影——标准 Criteria API 就变得力不从心。 Blaze-Persistence 正是为填补这一空白而生的开源框架：它在 JPA 之上构建了一套类型安全、表达力极强的查询 DSL， 同时通过 Entity View 将 DTO 映射提升到声明式水平。本章带你从全局视角了解 Blaze-Persistence 的定位、架构与核心概念。

  • 理解 JPA Criteria API 的局限性以及 Blaze-Persistence 的解决思路
  • 掌握 Core / Entity View / 集成层 的三层架构
  • 认识 CriteriaBuilderFactory、EntityViewManager 等核心接口
  • 在 JPA Criteria、QueryDSL、jOOQ 之间做出合理技术选型
  • 了解 JPQL 生成管线的完整流程
  • 提前知晓常见陷阱与限制

1.1 JPA 的困境与 Blaze-Persistence 的诞生

JPA（Java Persistence API）通过 EntityManager 和 JPQL 为 Java 应用提供了统一的 ORM 层。 对于简单的 CRUD 操作，JPA 工作得很好。但一旦需求演变为：

  • 多层嵌套子查询与关联过滤
  • CTE（公用表表达式）递归查询
  • 窗口函数（ROW_NUMBER、RANK）
  • 批量 UPDATE / DELETE 的 DML 语句
  • 高效 DTO 投影（避免 N+1 和全字段加载）

标准 JPA Criteria API 就暴露出冗长、类型不安全、SQL 功能覆盖不足等问题。

代码对比：同一查询的两种写法

假设我们需要查询"年龄大于 18 且姓名以 'J' 开头的用户，并按注册时间降序排列"。 来看看 JPA Criteria API 和 Blaze-Persistence CriteriaBuilder 的差异。 

CriteriaBuilder cb = em.getCriteriaBuilder();
CriteriaQuery<User> cq = cb.createQuery(User.class);
Root<User> root = cq.from(User.class);

Predicate agePred = cb.greaterThan(
    root.get("age"), 18);
Predicate namePred = cb.like(
    root.get("name"), "J%");

cq.where(cb.and(agePred, namePred));
cq.orderBy(cb.desc(
    root.get("registeredAt")));

List<User> users = em
    .createQuery(cq)
    .getResultList();
CriteriaBuilder<User> cb = cbf
    .create(em, User.class);

List<User> users = cb
    .where("age").gt(18)
    .where("name").like().value("J%")
    .orderByDesc("registeredAt")
    .getResultList();
💡 关键洞察
Blaze-Persistence 的 CriteriaBuilder 通过流式 API（Fluent API）将查询构建代码量减少 60%–70%， 同时保留完整的类型安全。每一个方法调用对应一个 SQL 子句，阅读和维护成本大幅降低。

1.3 核心概念速览

在深入各模块之前，先建立对核心接口和注解的整体认知。Blaze-Persistence 的 API 围绕两个工厂展开：

  • CriteriaBuilderFactory（CBF）— 创建查询构建器，是 Core 层的入口
  • EntityViewManager（EVM）— 管理 Entity View 的注册与实例化，是 Entity View 层的入口

核心注解一览

Entity View 通过 Java 接口 + 注解实现声明式 DTO 定义。以下是最常用的注解：

注解位置用途
@EntityView 接口/抽象类 标记该类型为某个实体的视图（DTO 定义的入口）
@IdMapping getter 方法 映射实体的主键字段
@Mapping("expr") getter 方法 映射任意 JPQL 表达式到视图属性
@UpdatableEntityView 接口/抽象类 允许通过视图执行 UPDATE 操作
@CreatableEntityView 接口/抽象类 允许通过视图创建新实体
@MappingSubquery getter 方法 将子查询结果映射为视图属性
@MappingCorrelated getter 方法 关联查询映射，支持自定义关联策略

Entity View 示例

@EntityView(User.class)
public interface UserSummaryView {

    @IdMapping
    Long getId();

    String getName();

    @Mapping("SIZE(orders)")
    long getOrderCount();

    @Mapping("TREAT(address AS DomesticAddress).city")
    String getCity();
}

// 使用
CriteriaBuilder<User> cb = cbf.create(em, User.class);
List<UserSummaryView> views = evm
    .applySetting(EntityViewSetting.create(UserSummaryView.class), cb)
    .getResultList();
💡 关键洞察
Entity View 的核心优势在于只 SELECT 视图中声明的字段。上例生成的 JPQL 不会加载 User 的所有列， 而是精确投影 id、name、SIZE(orders)、address.city， 从根本上避免了全字段加载的性能浪费。
▸ Entity View 工作流
sequenceDiagram participant App as 应用代码 participant EVM as EntityViewManager participant CB as CriteriaBuilder participant JPA as JPA Provider participant DB as 数据库 App->>EVM: applySetting(ViewClass, cb) EVM->>CB: 注入 SELECT / JOIN 子句 CB->>CB: 构建完整 JPQL App->>CB: getResultList() CB->>JPA: 提交 JPQL JPA->>DB: 执行 Native SQL DB-->>JPA: ResultSet JPA-->>CB: 原始结果 CB-->>EVM: 实例化 View 对象 EVM-->>App: List<ViewClass>

1.5 JPQL 生成管线

Blaze-Persistence 不直接生成原生 SQL，而是先生成增强的 JPQL（兼容 HQL），再委托给 JPA Provider 翻译为具体数据库的 SQL。 理解这条管线有助于排查性能问题和理解框架行为。

管线各阶段详解

1

API 调用 → 表达式树

每一次 .where()、.orderBy()、.select() 调用， 都会在内部构建一棵表达式树（AST）。树的节点包括：谓词节点、路径表达式、函数调用、子查询引用等。

2

表达式树优化

框架在渲染前会对表达式树做优化：合并冗余 JOIN、提取公共子表达式、决定是否使用子查询或 EXISTS 改写。

3

JPQL 渲染

JpqlRenderer 遍历表达式树，生成标准 JPQL 字符串。对于 JPA 不支持的特性（CTE、窗口函数）， 框架会生成扩展 JPQL或直接注入原生 SQL 片段。

4

JPA Provider 翻译

Hibernate 或 EclipseLink 接收 JPQL，经过其内部解析器和方言（Dialect）转换为目标数据库的原生 SQL。

5

数据库执行

原生 SQL 被发送到数据库，返回 ResultSet，再经由 JPA Provider 映射回 Java 对象。

调试技巧：查看生成的 JPQL

在开发阶段，可以通过以下方式查看 Blaze-Persistence 生成的 JPQL：

CriteriaBuilder<User> cb = cbf.create(em, User.class)
    .where("age").gt(18);

// 获取生成的 JPQL 字符串
String jpql = cb.getQueryString();
System.out.println(jpql);
// 输出: SELECT user FROM User user WHERE user.age > :param_0

配合 Hibernate 的 hibernate.show_sql=true 可以同时看到最终的原生 SQL。

1.6.3 Entity View 常见陷阱

  • 循环引用：A View 引用 B View，B View 又引用 A View → 需要用 @MappingCorrelated 或拆分子视图打破循环
  • 延迟加载误区：Entity View 默认即时加载所有声明的属性，不存在 JPA 实体的 LAZY 语义
  • 集合映射性能：在 View 中映射 Set<SubView> 时，如果数据量大，要配合分页或限制条数
  • 更新冲突：@UpdatableEntityView 使用乐观锁，需要实体定义 @Version 字段
💡 最佳实践
在项目初期使用 cb.getQueryString() 配合 SQL 日志验证每一个关键查询的生成结果。 不要等到上线才发现生成的 SQL 不符合预期。第 2 章将详细介绍如何配置日志级别以便调试。

本章小结

  • JPA Criteria API 在复杂查询场景下存在冗长、类型安全不足、SQL 功能覆盖有限等问题
  • Blaze-Persistence 通过 Core（CriteriaBuilder）和 Entity View 两大模块分别解决了查询构建和 DTO 投影的痛点
  • 框架采用四层架构：JPA Provider → Core → Entity View → 集成层，各层职责清晰、按需引入
  • CriteriaBuilderFactory 和 EntityViewManager 是框架的两大入口
  • 与 QueryDSL、jOOQ 相比，Blaze-Persistence 在"JPA 集成 + 高级 SQL 特性"这一象限具有独特优势
  • JPQL 生成管线经历：API 调用 → 表达式树 → JPQL 渲染 → JPA Provider 翻译 → 原生 SQL
  • Hibernate 是最佳 JPA Provider；使用 EclipseLink 或 DataNucleus 时需评估特性支持
