17.1 GraphQL 与 Entity View 的天然契合

GraphQL 和 Entity View 在设计哲学上高度一致：前者让客户端声明"我要哪些字段"，后者让服务端声明"只查询哪些列"。将两者串联起来，可以实现端到端的精准投影。

为什么 Entity View 天然解决 GraphQL N+1

传统 GraphQL resolver 采用"逐字段解析"模式 — 每个嵌套类型触发一次数据库查询。而 Entity View 在查询构建阶段就确定了所有需要的关联和列，生成单条优化 SQL。

传统 DataFetcher (N+1) vs Entity View (单次查询)
sequenceDiagram participant Client as GraphQL Client participant Resolver as GraphQL Resolver participant DB as 数据库 Note over Client,DB: ❌ 传统 DataFetcher 方式 Client->>Resolver: query { orders { id items { sku } } } Resolver->>DB: SQL #1: SELECT * FROM orders DB-->>Resolver: 返回 N 条订单 loop 每条订单 (N 次) Resolver->>DB: SQL #(1+i): SELECT * FROM items WHERE order_id = ? DB-->>Resolver: 返回子项 end Resolver-->>Client: 合并结果 Note over Client,DB: ✅ Entity View 方式 Client->>Resolver: query { orders { id items { sku } } } Resolver->>DB: SQL #1: SELECT o.id, i.sku FROM orders o
LEFT JOIN items i ON ... DB-->>Resolver: 返回全部数据 Resolver-->>Client: 直接返回 Note over DB: 传统方式: 1+N 条 SQL ❌
Entity View: 1 条 SQL ✅
💡 核心洞见：GraphQL 的字段选择机制与 Entity View 的投影机制在语义上完全等价。将两者结合，意味着客户端的 GraphQL 查询直接驱动数据库只 SELECT 和 JOIN 必要的列与表 — 无需中间 DTO 转换层。

代码实现：Entity View + Relay 分页

@Controller
public class OrderGraphQLController {

    @Autowired
    private CriteriaBuilderFactory cbf;
    @Autowired
    private EntityViewManager evm;
    @Autowired
    private EntityManager em;

    @QueryMapping
    public OrderConnection orders(
            @Argument Integer first,
            @Argument String after) {

        EntityViewSetting<OrderView, PaginatedCriteriaBuilder<OrderView>>
            setting = EntityViewSetting.create(OrderView.class, 0, first);

        // 解码 Relay cursor → keyset
        if (after != null) {
            KeysetPage keyset = decodeCursor(after);
            setting = EntityViewSetting.create(OrderView.class, keyset, first);
        }

        CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
            .orderByAsc("createdAt")
            .orderByAsc("id");

        PagedList<OrderView> page = evm.applySetting(setting, cb)
            .getResultList();

        return toRelayConnection(page);
    }

    private OrderConnection toRelayConnection(PagedList<OrderView> page) {
        List<OrderEdge> edges = page.stream()
            .map(node -> new OrderEdge(
                encodeCursor(page.getKeysetPage(), node),
                node))
            .collect(Collectors.toList());

        PageInfo pageInfo = new PageInfo(
            page.getTotalSize() > page.getMaxResults() + page.getFirstResult(),
            page.getFirstResult() > 0,
            edges.isEmpty() ? null : edges.get(0).getCursor(),
            edges.isEmpty() ? null : edges.get(edges.size()-1).getCursor());

        return new OrderConnection(edges, pageInfo, (int) page.getTotalSize());
    }
}
💡 Relay = Keyset：Relay cursor 本质上就是序列化的 keyset 元组。Blaze-Persistence 的 KeysetAwarePage 直接提供 keyset 信息，只需 Base64 编码/解码即可生成符合 Relay 规范的 cursor。

@QueryMapping
public OrderView order(@Argument Long id,
                       DataFetchingEnvironment env) {

    EntityViewSetting<OrderView, CriteriaBuilder<OrderView>>
        setting = EntityViewSetting.create(OrderView.class);

    // 从 GraphQL 选择集提取需要的字段
    Set<String> selectedFields = env.getSelectionSet()
        .getImmediateFields().stream()
        .map(SelectedField::getName)
        .collect(Collectors.toSet());

    // 仅 fetch 被选中的字段
    setting.fetch(selectedFields.toArray(new String[0]));

    CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
        .where("id").eq(id);

    return evm.applySetting(setting, cb).getSingleResult();
}

批量加载嵌套关联

当 Entity View 包含多个集合关联时，使用 MULTISET 策略可以在单条 SQL 中完成所有嵌套数据的加载：

@EntityView(Order.class)
public interface OrderGraphQLView {
    @IdMapping
    Long getId();
    String getCustomerName();

    @Mapping(fetch = FetchStrategy.MULTISET)
    List<OrderItemView> getItems();

    @Mapping(fetch = FetchStrategy.MULTISET)
    List<ShipmentView> getShipments();

    @Mapping(fetch = FetchStrategy.JOIN)
    CustomerView getCustomer();
}

@Controller
public class OrderGraphQLController {

    @Autowired private OrderService orderService;

    @QueryMapping
    public List<OrderView> orders(
            DataFetchingEnvironment env) {
        Set<String> fields = extractFields(env);
        return orderService.findAll(fields);
    }

    @QueryMapping
    public OrderDetailView order(
            @Argument Long id,
            DataFetchingEnvironment env) {
        return orderService.findById(id, extractFields(env));
    }

    @QueryMapping
    public OrderConnection orderConnection(
            @Argument Integer first,
            @Argument String after,
            DataFetchingEnvironment env) {
        return orderService.findPaginated(first, after, extractFields(env));
    }

    @MutationMapping
    public OrderDetailView createOrder(
            @Argument OrderCreateInput input) {
        return orderService.create(input);
    }

    private Set<String> extractFields(DataFetchingEnvironment env) {
        return env.getSelectionSet().getImmediateFields()
            .stream()
            .map(SelectedField::getName)
            .collect(Collectors.toSet());

3. Service 层 — EntityViewSetting 动态构建

@Service
@Transactional(readOnly = true)
public class OrderService {

    @Autowired private CriteriaBuilderFactory cbf;
    @Autowired private EntityViewManager evm;
    @PersistenceContext private EntityManager em;

    public List<OrderView> findAll(Set<String> fields) {
        EntityViewSetting<OrderView, CriteriaBuilder<OrderView>>
            setting = EntityViewSetting.create(OrderView.class);

        if (fields != null && !fields.isEmpty()) {
            setting.fetch(fields.toArray(new String[0]));
        }

        return evm.applySetting(setting,
                   cbf.create(em, Order.class))
                   .getResultList();
    }

    public OrderConnection findPaginated(
            Integer first, String after, Set<String> fields) {

        int pageSize = first != null ? first : 20;

        CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
            .orderByAsc("createdAt")
            .orderByAsc("id");

        PaginatedCriteriaBuilder<OrderView> pcb;
        if (after != null) {
            KeysetPage keyset = CursorCodec.decode(after);
            pcb = evm.applySetting(
                EntityViewSetting.create(OrderView.class, keyset, pageSize),
                cb);
        } else {
            pcb = evm.applySetting(
                EntityViewSetting.create(OrderView.class, 0, pageSize),
                cb);
        }

        PagedList<OrderView> page = pcb.getResultList();
        return RelayMapper.toConnection(page);
    }
}

@Configuration
public class GraphQLBatchConfig {

    @Bean
    public BatchLoaderRegistry batchLoaderRegistry(
            CriteriaBuilderFactory cbf,
            EntityViewManager evm,
            EntityManager em) {

        BatchLoaderRegistry registry = new BatchLoaderRegistry();

        registry.forTypePair(Long.class, CustomerView.class)
            .registerMappedBatchLoader((ids, env) -> {
                CriteriaBuilder<Customer> cb = cbf.create(em, Customer.class)
                    .where("id").in(ids);

                List<CustomerView> results = evm.applySetting(
                    EntityViewSetting.create(CustomerView.class), cb)
                    .getResultList();

                return Mono.just(results.stream()
                    .collect(Collectors.toMap(
                        CustomerView::getId, v -> v)));
            });

        return registry;
    }
}
💡 双重优化：Entity View 提供列级精准投影（仅 SELECT 需要的列），BatchLoader 提供行级批量加载（合并 N 次查询为 1 次 IN 查询）。两者组合使 GraphQL 查询性能接近手写 SQL。

@GraphQLApi
@ApplicationScoped
public class OrderGraphQLResource {

    @Inject CriteriaBuilderFactory cbf;
    @Inject EntityViewManager evm;
    @Inject EntityManager em;

    @Query("orders")
    @Description("查询所有订单，支持分页")
    public List<OrderView> getOrders(
            @DefaultValue("20") int pageSize,
            @DefaultValue("0") int offset) {

        EntityViewSetting<OrderView, PaginatedCriteriaBuilder<OrderView>>
            setting = EntityViewSetting.create(
                OrderView.class, offset, pageSize);

        CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
            .orderByAsc("id");

        return evm.applySetting(setting, cb).getResultList();
    }

    @Query("order")
    @Description("按 ID 查询单个订单详情")
    public OrderDetailView getOrder(@Name("id") Long id) {
        EntityViewSetting<OrderDetailView, CriteriaBuilder<OrderDetailView>>
            setting = EntityViewSetting.create(OrderDetailView.class);

        CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
            .where("id").eq(id);

        return evm.applySetting(setting, cb).getSingleResult();
    }

    @Mutation("createOrder")
    @Description("创建新订单")
    @Transactional
    public OrderDetailView createOrder(OrderCreateInput input) {
        OrderCreateView view = evm.create(OrderCreateView.class);
        view.setCustomerName(input.getCustomerName());
        evm.save(em, view);
        return getOrder(view.getId());
    }

陷阱 1：循环引用

当 OrderView 包含 CustomerView，而 CustomerView 又引用 List<OrderView> 时，GraphQL schema 生成会陷入无限递归。

循环引用 — 问题与解法
// ✅ 打断循环：为不同方向创建精简视图
@EntityView(Customer.class)
public interface CustomerSummaryView {
    @IdMapping Long getId();
    String getName();
    // 不包含 getOrders() — 打断循环
}

@EntityView(Order.class)
public interface OrderView {
    @IdMapping Long getId();
    String getCustomerName();
    CustomerSummaryView getCustomer(); // 引用精简视图
}

陷阱 2：Mutation 复杂性

GraphQL Mutation 的输入类型（Input Types）与 Entity View 的 Updatable/Creatable 视图并非直接等价。需要独立定义 Input 类型，在 Resolver 层完成转换。

// GraphQL Input 类型 — 独立于 Entity View
public class OrderCreateInput {
    private String customerName;
    private List<OrderItemInput> items;
    // getters & setters
}

// Resolver 层负责 Input → UpdatableView 转换
@MutationMapping
public OrderDetailView createOrder(@Argument OrderCreateInput input) {
    OrderCreateView view = evm.create(OrderCreateView.class);
    view.setCustomerName(input.getCustomerName());

    for (OrderItemInput item : input.getItems()) {
        OrderItemCreateView itemView = evm.create(OrderItemCreateView.class);
        itemView.setSku(item.getSku());
        itemView.setQuantity(item.getQuantity());
        view.getItems().add(itemView);
    }

    evm.save(em, view);
    return findById(view.getId());
}

陷阱 3：深层嵌套性能

⚠️ 查询深度失控：GraphQL 允许客户端无限嵌套查询（如 order.customer.orders.customer.orders...），每增加一层嵌套都会产生额外的 JOIN。务必设置查询深度和复杂度限制。 
# application.properties — Spring for GraphQL
spring.graphql.graphiql.enabled=true

# SmallRye GraphQL (Quarkus) — 查询复杂度限制
quarkus.smallrye-graphql.events.enabled=true
mp.graphql.maxComplexity=100
mp.graphql.maxDepth=5

陷阱 4：Cursor 编解码

Relay cursor 应当是不透明的字符串。直接暴露 keyset 值（如数据库 ID）可能导致安全问题。

public class CursorCodec {
    private static final String HMAC_KEY = "your-secret-key";

    public static String encode(Serializable[] keyset) {
        String json = new Gson().toJson(keyset);
        String base64 = Base64.getUrlEncoder()
            .encodeToString(json.getBytes(StandardCharsets.UTF_8));
        String hmac = computeHmac(base64, HMAC_KEY);
        return base64 + "." + hmac;
    }

    public static Serializable[] decode(String cursor) {
        String[] parts = cursor.split("\\.");
        if (parts.length != 2) throw new IllegalArgumentException("Invalid cursor");
        if (!computeHmac(parts[0], HMAC_KEY).equals(parts[1])) {
            throw new SecurityException("Cursor tampered");
        }
        String json = new String(
            Base64.getUrlDecoder().decode(parts[0]),
            StandardCharsets.UTF_8);
        return new Gson().fromJson(json, Serializable[].class);
    }
}

📝 本章小结

  • GraphQL 与 Entity View 在"按需投影"理念上天然同构 — GraphQL field ≈ Entity View getter ≈ SQL SELECT 列。
  • Entity View 接口可自动映射为 GraphQL 类型，Subview 嵌套直接对应 GraphQL 嵌套类型，实现 Schema-from-Code。
  • Relay 规范分页与 Keyset 分页一一对应：keyset 元组 Base64 编码后即为 cursor，PagedList 直接转换为 Connection/Edge/PageInfo。
  • GraphQL 字段选择通过 DataFetchingEnvironment 传递给 EntityViewSetting.fetch()，实现端到端精准投影，从根本上消除 N+1 问题。
  • Spring for GraphQL 集成通过 @QueryMapping + Entity View 实现；BatchLoader 模式进一步优化关联数据批量加载。
  • Quarkus + SmallRye GraphQL 通过 @GraphQLApi + CDI 注入 Blaze-Persistence 组件，完全兼容GraalVM 原生编译。
  • 实际集成需注意循环引用、Mutation 建模、查询深度限制、Cursor 安全、Schema 演进等陷阱。
