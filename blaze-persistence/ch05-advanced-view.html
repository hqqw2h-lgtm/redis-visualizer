5.1 子视图嵌套（Subview）

当 @Mapping 方法的返回类型本身也是一个 @EntityView 时，Blaze-Persistence 自动将其识别为子视图（Subview）。这是构建复杂 DTO 对象树的基础。

三个 @EntityView 接口形成嵌套链。框架在编译期（EntityViewManager 初始化）就完成了嵌套解析，运行时直接生成优化后的单条查询。

@EntityView(Product.class)
public interface ProductView {
    @IdMapping
    Long getId();
    String getName();
    String getSku();
}

@EntityView(OrderItem.class)
public interface OrderItemView {
    @IdMapping
    Long getId();
    Integer getQuantity();
    BigDecimal getUnitPrice();

    // 返回类型是另一个 EntityView → 自动子视图
    ProductView getProduct();
}

@EntityView(Order.class)
public interface OrderView {
    @IdMapping
    Long getId();
    String getStatus();
    BigDecimal getTotalPrice();

    // List<子视图> → 自动嵌套集合投影
    List<OrderItemView> getItems();
}

💡 关键优势：无论嵌套多少层，Blaze-Persistence 仅生成一条 SQL（默认 JOIN 策略）。 SELECT 子句只包含视图中实际声明的字段，避免 SELECT * 带来的多余列传输。

5.2 继承映射（Inheritance）

JPA 实体可以有继承层次（SINGLE_TABLE、JOINED、TABLE_PER_CLASS）。Blaze-Persistence Entity View 同样支持多态投影：定义基础视图接口，派生子视图映射到不同实体子类。

Entity View 继承层次：AnimalView → CatView / DogView
classDiagram class AnimalView { <<@EntityView(Animal)>> +Long getId() +String getName() +Double getWeight() } class CatView { <<@EntityView(Cat)>> +Boolean isIndoor() +String getBreed() } class DogView { <<@EntityView(Dog)>> +String getBarkVolume() +Boolean isTrained() } AnimalView <|-- CatView : @EntityViewInheritance AnimalView <|-- DogView : @EntityViewInheritance class Animal { <<JPA Entity>> -Long id -String name -Double weight -String dtype } class Cat { <<JPA Entity>> -Boolean indoor -String breed } class Dog { <<JPA Entity>> -String barkVolume -Boolean trained } Animal <|-- Cat Animal <|-- Dog 
@EntityView(Animal.class)
@EntityViewInheritance
public interface AnimalView {
    @IdMapping
    Long getId();
    String getName();
    Double getWeight();
}

@EntityView(Cat.class)
@EntityViewInheritanceMapping("TYPE(this) = Cat")
public interface CatView extends AnimalView {
    Boolean isIndoor();
    String getBreed();
}

@EntityView(Dog.class)
@EntityViewInheritanceMapping("TYPE(this) = Dog")
public interface DogView extends AnimalView {
    String getBarkVolume();
    Boolean isTrained();
}

5.3 集合映射

Entity View 可以映射 List、Set、Map 类型的关联集合。关键在于选择正确的抓取策略——不同策略产生截然不同的 SQL 形态和性能特征。

三种集合抓取策略对比
JOIN 策略
单条 SQL
LEFT JOIN 集合表
⚠️ 主实体行重复
适合小集合
SUBSELECT 策略
两条 SQL
主查询 + 子查询
✅ 无行重复
适合中等集合
⭐ MULTISET 策略
单条 SQL
嵌套集合作为子列
✅ 无行重复
推荐策略
策略SQL 条数行重复嵌套支持适用场景
FetchStrategy.JOIN 1 是（笛卡尔积风险） 多层 JOIN 可能爆炸 小集合、单层嵌套
FetchStrategy.SUBSELECT N+1（但可批量） 否 良好 中等集合、需要分页
FetchStrategy.MULTISET 1 否 优秀 复杂嵌套、大集合 ⭐ 
@EntityView(Order.class)
public interface OrderWithItemsView {
    @IdMapping
    Long getId();

    // 默认 JOIN 策略
    List<OrderItemView> getItems();

    // 显式指定 MULTISET 策略（推荐）
    @Mapping(fetch = FetchStrategy.MULTISET)
    List<OrderItemView> getItemsMultiset();

    // SUBSELECT 策略
    @Mapping(fetch = FetchStrategy.SUBSELECT)
    Set<TagView> getTags();
}

DB-->>EVM: N × M 行（笛卡尔积） EVM-->>App: List<OrderView>（去重后） Note over App,DB: MULTISET 策略 App->>EVM: applySetting(OrderView.class) EVM->>DB: SELECT o.*, (SELECT ... 💡 最佳实践：优先使用 MULTISET 策略。它在单条 SQL 中内嵌子集合，避免笛卡尔积， 是 Blaze-Persistence 相比原生 JPA 的核心差异化特性之一。

5.4 @MappingCorrelated 关联子查询

有些派生字段无法通过简单的路径映射表达——例如统计关联实体数量、计算聚合值、或引用不相关的实体。@MappingCorrelated 允许在视图中嵌入关联子查询。

correlator = OrderCountProvider.class) EVM->>CP: 调用 provideCorrelation() CP->>CP: cb.from(Order.class)
.where("customerId").eqExpression("EMBEDDING_VIEW(id)") CP-->>EVM: 返回关联查询片段 EVM->>SQL: 合并为主查询的

// 1. 定义 CorrelationProvider
public class OrderCountProvider implements CorrelationProvider {
    @Override
    public void applyCorrelation(
            CorrelationBuilder correlationBuilder,
            String correlationExpression) {
        correlationBuilder
            .correlate(Order.class)
            .on("customer.id")
            .eqExpression(correlationExpression)
            .end();
    }
}

// 2. 在 EntityView 中使用关联子查询
@EntityView(Customer.class)
public interface CustomerSummaryView {
    @IdMapping
    Long getId();
    String getName();

    // 关联子查询：统计该客户的订单数
    @MappingCorrelated(
        correlationBasis = "id",
        correlator = OrderCountProvider.class,
        correlationResult = "id",
        fetch = FetchStrategy.JOIN
    )
    Long getOrderCount();

    // 关联子查询：计算总消费金额
    @MappingCorrelated(
        correlationBasis = "id",
        correlator = TotalSpentProvider.class,
        correlationResult = "totalPrice",
        fetch = FetchStrategy.JOIN
    )
    BigDecimal getTotalSpent();
}
// 原生方式：先查客户，再循环查订单数
List<Customer> customers = em
    .createQuery("FROM Customer")
    .getResultList();
for (Customer c : customers) {
    Long count = em.createQuery(
        "SELECT COUNT(o) FROM Order o " +
        "WHERE o.customer = :c", Long.class)
        .setParameter("c", c)
        .getSingleResult();
    // N+1 查询！
}
// @MappingCorrelated 方式
// 生成单条 SQL：
// SELECT c.id, c.name,
//   (SELECT COUNT(*)
//    FROM orders o
//    WHERE o.customer_id = c.id)
//   AS orderCount
// FROM customers c

List<CustomerSummaryView> result =
    evm.applySetting(
        EntityViewSetting.create(
            CustomerSummaryView.class),
        cb)
    .getResultList();
⚠️ 何时使用关联 vs 常规映射： 如果目标属性可以通过 JPA 关联路径（如 order.customer.name）到达，使用 @Mapping 即可。 只有当需要聚合计算、跨无关联实体或复杂过滤时才需要 @MappingCorrelated。

5.5 Flat View

常规子视图会生成嵌套对象结构（如 orderView.getAddress().getCity()），而 Flat View 将关联实体的字段直接扁平化到父视图层级，省去中间层。

嵌套子视图 vs Flat View 的结构对比
常规子视图（嵌套结构）
OrderView
↳ getAddress() → AddressView
↳ getStreet()
↳ getCity()
↳ getZipCode()
Flat View（扁平结构）
OrderView
↳ getAddressStreet()
↳ getAddressCity()
↳ getAddressZipCode()

使用 @MappingFlat 或直接在 @Mapping 中指定路径表达式即可将关联属性拍平：

@EntityView(Order.class)
public interface OrderFlatView {
    @IdMapping
    Long getId();
    String getStatus();

    // 扁平化：直接映射关联实体 Address 的字段
    @Mapping("shippingAddress.street")
    String getShippingStreet();

    @Mapping("shippingAddress.city")
    String getShippingCity();

    @Mapping("shippingAddress.zipCode")
    String getShippingZipCode();

    @Mapping("shippingAddress.country.name")
    String getShippingCountryName();
}

5.6 @MappingParameter 与上下文参数

@EntityView(Product.class)
public interface LocalizedProductView {
    @IdMapping
    Long getId();

    // 根据运行时 locale 参数选择名称列
    @Mapping("CASE WHEN :locale = 'zh' THEN nameZh ELSE nameEn END")
    String getDisplayName();

    // 注入参数本身
    @MappingParameter("locale")
    String getLocale();
}

// 调用方设置参数
EntityViewSetting<LocalizedProductView, CriteriaBuilder<LocalizedProductView>> setting =
    EntityViewSetting.create(LocalizedProductView.class);
setting.withOptionalParameter("locale", "zh");

List<LocalizedProductView> products =
    evm.applySetting(setting, cb).getResultList();
💡 参数类型：withOptionalParameter 允许视图在未传参时使用默认值； withParameter 则为必需参数——缺失时抛出异常。参数可以是任意 Java 对象，不限于基本类型。

5.7 陷阱与限制

高级 Entity View 功能强大但有边界。以下是实践中最常见的陷阱：

高级 Entity View 常见陷阱全景
🔴 深层嵌套
3+ 层子视图 → 多重 JOIN、笛卡尔积风险
✅ 解法：MULTISET
🔴 集合 + 分页
集合 JOIN → 行重复导致分页数量不准
✅ 解法：SUBSELECT 或 MULTISET
🔴 继承层数
多层继承 → 判别逻辑复杂、SELECT 列爆炸
✅ 解法：限制 2 层，用组合替代
🔴 Correlated N+1
@MappingCorrelated + JOIN → 默认单行执行
✅ 解法：设置 batchSize 或 SUBSELECT
🔴 Flat + Optional
Flat View 映射可选关联 → NULL 时全部字段为空
✅ 解法：子视图整体可空更清晰

5.7.1 深层嵌套与笛卡尔积

当子视图嵌套超过 3 层、且多层都包含集合映射时，默认 JOIN 策略会产生笛卡尔积——查询结果行数 = 各层集合大小之积。

🚨 危险模式：Order(10行) → Items(5/单) → Tags(3/单) = 150 行结果。 切换为 MULTISET 后回到 10 行。

5.7.2 集合映射与分页冲突

使用 JOIN 抓取集合时，LIMIT / OFFSET 作用于 JOIN 后的行而非实体。10 个 Order 各含 5 个 Item，LIMIT 10 只返回 2 个 Order 的数据。

// ❌ 错误：集合 JOIN + 分页导致数据截断
CriteriaBuilder<OrderView> cb = cbf.create(em, OrderView.class)
    .orderByAsc("id")
    .page(0, 10); // LIMIT 10 作用于 JOIN 后的行！

// ✅ 正确：集合使用 MULTISET，分页作用于主实体
// 或将集合改为 SUBSELECT 策略

5.7.3 @MappingCorrelated 的 N+1 风险

默认情况下关联子查询随主查询一起内联。但如果配合 SUBSELECT fetch 且未设置 batchSize，每行会触发一次独立的关联查询。

Correlated Subquery 批处理 vs 逐行执行

5.7.4 Flat View 与可选关联

当被 Flat 映射的关联是 @ManyToOne(optional = true) 时，关联为 NULL 意味着所有扁平字段都是 NULL——无法区分"关联存在但字段为空"和"关联本身不存在"。此时使用可空的子视图更加语义明确。

实战经验

在电商系统中，订单的 billingAddress 可能为空（使用发货地址结算）。 如果对 billingAddress 使用 Flat View，当地址不存在时，billingStreet、billingCity 全部为 null， 客户端无法区分是"没有独立结算地址"还是"地址信息未填"。

推荐方案：使用 @Mapping("billingAddress") AddressView getBillingAddress()，关联不存在时整个子视图返回 null。

📋 本章小结

  • 子视图嵌套：返回类型为 @EntityView 时自动识别为子视图，合并为单条优化 SQL
  • 继承映射：@EntityViewInheritance + @EntityViewInheritanceMapping 实现多态 DTO 投影
  • 集合策略：JOIN（小集合）、SUBSELECT（需分页）、MULTISET（推荐，无笛卡尔积）
  • 关联子查询：@MappingCorrelated + CorrelationProvider 嵌入聚合计算，优于手动 N+1 循环
  • Flat View：路径表达式扁平化嵌套结构，适合表格/CSV 场景，注意可选关联的语义陷阱
  • @MappingParameter：运行时参数注入，实现 Locale 切换等动态逻辑
  • 陷阱警示：深层嵌套用 MULTISET、集合分页避免 JOIN、关联子查询注意批处理
