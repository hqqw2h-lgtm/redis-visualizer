第 10 章 · 分页

Offset 分页 · Keyset 分页 · COUNT 优化 · PagedList · Entity View 分页

分页是几乎每个后端系统都要面对的基本问题，然而传统的 OFFSET / LIMIT 方案在数据量增长后性能急剧恶化， 并且在并发写入场景下会产生"丢行/重复行"的一致性缺陷。Blaze-Persistence 提供了基于 Keyset 分页 的 O(1) 方案，同时保留了传统 Offset 分页的 API 兼容性，并可无缝结合 Entity View 投影。

10.1 Offset 分页的问题

传统分页依赖 SQL 的 OFFSET n LIMIT m 语法。数据库引擎必须先扫描并跳过前 n 行， 才能返回目标页的 m 行。当 n 增大时，跳过的行数线性增长，导致查询时间随页码线性上升。

Offset 分页的行扫描模型 — 越往后越慢
第 1 页 · OFFSET 0
✅ 返回 20 行
第 50 页 · OFFSET 980
⏭ 跳过 980 行 → ✅ 返回 20 行
第 500 页 · OFFSET 9980
⏭ 跳过 9,980 行 → ✅ 返回 20 行
第 5000 页 · OFFSET 99,980
⏭ 跳过 99,980 行 → ✅ 返回 20 行

除了性能问题，Offset 分页在并发写入场景下还存在一致性缺陷：

并发修改导致的丢行与重复行
sequenceDiagram participant U as 用户浏览器 participant DB as 数据库 U->>DB: SELECT ... OFFSET 0 LIMIT 3（第 1 页） DB-->>U: [A, B, C] Note over DB: 另一用户在 B 前面插入了新行 X U->>DB: SELECT ... OFFSET 3 LIMIT 3（第 2 页） DB-->>U: [C, D, E] Note over U: C 出现两次！因为插入 X 后行整体后移
第 2 页本应是 [D, E, F]
⚠ 生产事故场景：在高写入频率的订单系统中，运营人员翻页导出数据时， Offset 分页可能导致部分订单漏掉或重复导出，引发资金对账异常。

性能退化的根本原因是 OFFSET 迫使数据库做 O(n) 的行扫描：

页码OFFSET 值扫描行数相对耗时（100万行表）
1020~1ms
1001,9802,000~5ms
1,00019,98020,000~40ms
10,000199,980200,000~350ms
50,000999,9801,000,000~1.8s

10.2 Keyset 分页原理

Keyset 分页（也称 Seek Method 或 Cursor-based Pagination）利用上一页最后一条记录的排序键值作为下一页查询的 WHERE 条件，从而绕过 OFFSET 的行扫描开销。

Keyset 分页核心算法
flowchart TD A["客户端请求第 1 页"] --> B["SELECT ... ORDER BY created_at DESC, id DESC LIMIT 20"] B --> C["返回 20 行 + 提取最后一行的 Keyset"] C --> D{"用户点击下一页?"} D -->|是| E["SELECT ... WHERE (created_at, id) < (lastDate, lastId) ORDER BY created_at DESC, id DESC LIMIT 20"] E --> F["返回 20 行 + 提取新的 Keyset"] F --> D D -->|否| G["结束"] style A fill:#1e3a5f,stroke:#58a6ff style B fill:#1e3a5f,stroke:#58a6ff style E fill:#1a3a2a,stroke:#3fb950 style F fill:#1a3a2a,stroke:#3fb950 style G fill:#2d1f4e,stroke:#bc8cff

这个方法的数学本质是：给定排序列 (c1, c2, ... cn) 和上一页最后行的值 (v1, v2, ... vn)， 下一页的条件等价于字典序比较：

WHERE (c1, c2, ..., cn) < (v1, v2, ..., vn)   -- DESC 排序
WHERE (c1, c2, ..., cn) > (v1, v2, ..., vn)   -- ASC 排序

由于查询条件直接命中索引的 B-Tree 叶节点，无论请求第几页，数据库只需一次索引查找 + 顺序读取 LIMIT 行， 复杂度为 O(log n + pageSize)，约等于 O(1)。

Offset O(n) vs Keyset O(1) 性能对比
graph LR subgraph Offset["Offset 分页 — O(n)"] direction TB O1["第 1 页: 1ms"] --> O2["第 100 页: 5ms"] O2 --> O3["第 1000 页: 40ms"] O3 --> O4["第 10000 页: 350ms"] O4 --> O5["第 50000 页: 1800ms ⚠"] end subgraph Keyset["Keyset 分页 — O(1)"] direction TB K1["第 1 页: 1ms"] --> K2["第 100 页: 1ms"] K2 --> K3["第 1000 页: 1ms"] K3 --> K4["第 10000 页: 1ms"] K4 --> K5["第 50000 页: 1ms ✅"] end style O4 fill:#3a1a1a,stroke:#f85149 style O5 fill:#3a1a1a,stroke:#f85149 style K1 fill:#1a3a2a,stroke:#3fb950 style K2 fill:#1a3a2a,stroke:#3fb950 style K3 fill:#1a3a2a,stroke:#3fb950 style K4 fill:#1a3a2a,stroke:#3fb950 style K5 fill:#1a3a2a,stroke:#3fb950
💡 索引利用：Keyset 分页的高性能前提是排序列上存在合适的复合索引。 例如 ORDER BY created_at DESC, id DESC 需要 INDEX(created_at DESC, id DESC)， 这样数据库可以直接从 B-Tree 的目标位置开始顺序扫描。

逐行扫描到目标位置"] -.->|"扫描 L1→L2→L3→..." | L1 KEYSET["Keyset: 直接定位
目标叶子节点"] -.->|"索引查找 O(log n)"| L4 style OFFSET fill:#3a1a1a,stroke:#f85149 style KEYSET fill:#1a3a2a,stroke:#3fb950 style L4 fill:#1a3a2a,stroke:#3fb950

10.3 Blaze-Persistence PaginatedCriteriaBuilder

Blaze-Persistence 通过 PaginatedCriteriaBuilder 统一了 Offset 和 Keyset 两种分页模式。 调用 page(firstResult, maxResults) 使用 Offset 模式，调用 page(keysetPage, firstResult, maxResults) 则启用 Keyset 模式。两种模式返回相同的 PagedList<T> 结果。

PaginatedCriteriaBuilder API 全景
classDiagram class CriteriaBuilder~T~ { +where(expression) +orderByAsc(path) +orderByDesc(path) +page(int firstResult, int maxResults) PaginatedCriteriaBuilder~T~ +page(KeysetPage keyset, int firstResult, int maxResults) PaginatedCriteriaBuilder~T~ } class PaginatedCriteriaBuilder~T~ { +withCountQuery(boolean enable) +withExtractAllKeysets(boolean enable) +withKeysetExtraction(boolean enable) +getResultList() PagedList~T~ +getPageSize() int +getFirstResult() int } class PagedList~T~ { +getTotalSize() long +getTotalPages() int +getFirstResult() int +getMaxResults() int +getKeysetPage() KeysetPage } class KeysetPage { +getFirstResult() int +getMaxResults() int +getLowest() Keyset +getHighest() Keyset +getKeysets() List~Keyset~ } CriteriaBuilder --> PaginatedCriteriaBuilder : page() PaginatedCriteriaBuilder --> PagedList : getResultList() PagedList --> KeysetPage : getKeysetPage()

Offset 分页示例

CriteriaBuilder<Cat> cb = cbf.create(em, Cat.class)
    .orderByAsc("name")
    .orderByAsc("id");

// 第 3 页，每页 20 条（firstResult = 40）
PagedList<Cat> page3 = cb.page(40, 20).getResultList();

long total  = page3.getTotalSize();   // 总行数
int  pages  = page3.getTotalPages();  // 总页数
// page3 包含 cat 实体列表

Keyset 分页示例

// 第 1 页：普通 Offset 调用获取初始 KeysetPage
PagedList<Cat> page1 = cb.page(0, 20).getResultList();
KeysetPage keyset = page1.getKeysetPage();

// 第 2 页：传入上一页的 KeysetPage
PagedList<Cat> page2 = cb.page(keyset, 20, 20).getResultList();
// 数据库生成 WHERE (name, id) > (lastName, lastId) 而非 OFFSET 20
Offset vs Keyset 分页 — 完整请求流程
sequenceDiagram participant C as 客户端 participant BP as Blaze-Persistence participant DB as 数据库 rect rgba(30,58,95,.3) Note over C,DB: Offset 模式 C->>BP: page(40, 20) BP->>DB: SELECT ... ORDER BY name, id OFFSET 40 LIMIT 20 BP->>DB: SELECT COUNT(*) ... DB-->>BP: 20 行 + 总数 BP-->>C: PagedList (含 KeysetPage) end rect rgba(26,58,42,.3) Note over C,DB: Keyset 模式 C->>BP: page(keysetPage, 60, 20) BP->>DB: SELECT ... WHERE (name, id) > (?, ?) ORDER BY name, id LIMIT 20 BP->>DB: SELECT COUNT(*) ...（可关闭） DB-->>BP: 20 行 + 总数 BP-->>C: PagedList (含新 KeysetPage) end
💡 无缝切换：第一次请求必须使用 Offset 模式（因为还没有 KeysetPage）， 后续请求传入 KeysetPage 即可自动切换到 Keyset 模式。API 返回的 PagedList 始终包含 KeysetPage，方便客户端持续翻页。

KeysetPage 的序列化

在 REST API 场景中，KeysetPage 需要在客户端和服务端之间传递。Blaze-Persistence 提供了 KeysetPageable 与 Spring Data 集成的序列化方案。也可以手动将 KeysetPage 序列化为 JSON：

// 序列化：将 KeysetPage 转为可传输的 DTO
KeysetPage kp = pagedList.getKeysetPage();
Map<String, Object> dto = Map.of(
    "firstResult", kp.getFirstResult(),
    "maxResults",  kp.getMaxResults(),
    "lowest",      kp.getLowest().getTuple(),   // Object[]
    "highest",     kp.getHighest().getTuple()    // Object[]
);
// 转为 JSON 发送给客户端

// 反序列化：从客户端接收后重建
KeysetPage restored = new DefaultKeysetPage(
    firstResult, maxResults,
    new DefaultKeyset(lowestTuple),
    new DefaultKeyset(highestTuple),
    Collections.emptyList()
);

10.4 COUNT 查询优化

默认情况下，PaginatedCriteriaBuilder 会自动生成一条 COUNT(*) 查询以计算总行数。 在大表上，这条 COUNT 查询可能比分页查询本身更慢。Blaze-Persistence 提供了多种优化手段。

COUNT 查询生成与优化决策树
flowchart TD A["PaginatedCriteriaBuilder 执行"] --> B{"withCountQuery?"} B -->|"true（默认）"| C{"有自定义 COUNT?"} B -->|"false"| D["跳过 COUNT
totalSize = -1"] C -->|是| E["使用自定义 COUNT 查询"] C -->|否| F["自动生成 COUNT 查询"] F --> G{"查询包含 GROUP BY?"} G -->|是| H["包装为 SELECT COUNT(*)
FROM (原查询) sub"] G -->|否| I["直接 SELECT COUNT(*)
FROM entity WHERE ..."] E --> J["执行 COUNT + 分页"] H --> J I --> J D --> K["仅执行分页查询"] style D fill:#1a3a2a,stroke:#3fb950 style E fill:#1a3a2a,stroke:#3fb950 style K fill:#1a3a2a,stroke:#3fb950

关闭 COUNT 查询

PagedList<Cat> page = cb.page(0, 20)
    .withCountQuery(false)     // 不执行 COUNT
    .getResultList();

// page.getTotalSize() == -1
// page.getTotalPages() == -1

自定义 COUNT 查询

当自动生成的 COUNT 查询性能不理想时，可以提供一个优化过的自定义 COUNT：

PagedList<Cat> page = cb.page(0, 20)
    .withCountQuery(
        cbf.create(em, Long.class)
           .from(Cat.class, "c")
           .select("COUNT(c.id)")
           .where("c.age").gt(3)
    )
    .getResultList();
withExtractAllKeysets 批量预计算
flowchart LR subgraph 默认["默认模式"] direction TB D1["提取第一行 Keyset"] --> D2["提取最后一行 Keyset"] D2 --> D3["其他行无 Keyset 信息"] end subgraph 全量["withExtractAllKeysets(true)"] direction TB E1["提取第 1 行 Keyset"] --> E2["提取第 2 行 Keyset"] E2 --> E3["提取第 3 行 Keyset"] E3 --> E4["..."] E4 --> E5["提取第 20 行 Keyset"] end 默认 -->|"需要跳页时"| SKIP["只能顺序翻页"] 全量 -->|"支持跳到任意行"| JUMP["可跳转到任意行位置"] style SKIP fill:#3a1a1a,stroke:#f85149 style JUMP fill:#1a3a2a,stroke:#3fb950 
PagedList<Cat> page = cb.page(0, 20)
    .withExtractAllKeysets(true)   // 为每一行计算 Keyset
    .getResultList();

KeysetPage ks = page.getKeysetPage();
List<Keyset> allKeysets = ks.getKeysets();  // 20 个 Keyset
// 可实现"跳转到第 N 行"的精确定位
📊 性能权衡：withExtractAllKeysets(true) 会在 SELECT 列表中增加额外列以提取所有行的排序键值， 这会增加约 10-15% 的查询开销。仅在需要精确跳页时启用。

10.5 Entity View 分页

Blaze-Persistence 最大的优势之一就是将 Entity View 投影与分页 API 无缝结合。 通过 EntityViewSetting 包装分页参数，可以直接返回 PagedList<ViewClass>。

Entity View + Pagination 管线全景
flowchart TD A["EntityViewSetting.create(CatView.class)"] --> B["setting.withPagination(firstResult, maxResults)"] B --> C["setting.withKeysetPage(keysetPage)"] C --> D["evm.applySetting(setting, cb)"] D --> E["PaginatedCriteriaBuilder<CatView>"] E --> F["自动生成优化 SELECT
仅投影 View 需要的列"] F --> G["自动生成 COUNT 查询"] F --> H["自动提取 Keyset"] G --> I["PagedList<CatView>"] H --> I style A fill:#2d1f4e,stroke:#bc8cff style D fill:#2d1f4e,stroke:#bc8cff style E fill:#1e3a5f,stroke:#58a6ff style I fill:#1a3a2a,stroke:#3fb950

完整示例

// 1. 定义 Entity View
@EntityView(Cat.class)
public interface CatView {
    @IdMapping Long getId();
    String getName();
    @Mapping("age") Integer getAge();
    @Mapping("owner.name") String getOwnerName();
}

// 2. 构建基础查询
CriteriaBuilder<Cat> cb = cbf.create(em, Cat.class)
    .where("age").gt(1)
    .orderByAsc("name")
    .orderByAsc("id");

// 3. 创建 EntityViewSetting 并配置分页
EntityViewSetting<CatView, PaginatedCriteriaBuilder<CatView>> setting =
    EntityViewSetting.create(CatView.class, 0, 20);

// 4. 应用 Setting，获取分页结果
PaginatedCriteriaBuilder<CatView> pcb = evm.applySetting(setting, cb);
PagedList<CatView> page = pcb.getResultList();

// 5. 使用结果
page.forEach(v -> System.out.println(v.getName()));
long total = page.getTotalSize();
KeysetPage keyset = page.getKeysetPage();  // 传给下一页请求

💡 SELECT 优化：Entity View 分页不会 SELECT *。 Blaze-Persistence 根据 CatView 的 @Mapping 注解，自动生成仅包含需要列的 SELECT， 减少 I/O 和反序列化开销。

10.6 多列 Keyset

在实际业务中，单列排序（如仅按 id）往往不够用。常见场景是按"创建时间 + ID"复合排序， 或按"状态 + 优先级 + ID"多列排序。Keyset 分页需要将所有排序列作为 Keyset 的组成部分。

多列 Keyset 的字典序比较逻辑
flowchart TD A["WHERE 条件生成
ORDER BY created_at DESC, id ASC"] --> B{"created_at < lastDate ?"} B -->|是| C["✅ 匹配 — 取这一行"] B -->|否| D{"created_at = lastDate ?"} D -->|是| E{"id > lastId ?"} D -->|否| F["❌ 不匹配 — 跳过"] E -->|是| G["✅ 匹配 — 取这一行"] E -->|否| H["❌ 不匹配 — 跳过"] style C fill:#1a3a2a,stroke:#3fb950 style G fill:#1a3a2a,stroke:#3fb950 style F fill:#3a1a1a,stroke:#f85149 style H fill:#3a1a1a,stroke:#f85149

Blaze-Persistence 自动将多列排序转换为等价的 SQL 条件：

-- ORDER BY created_at DESC, id ASC 的 Keyset 条件展开
WHERE (
    created_at < ?lastDate
    OR (created_at = ?lastDate AND id > ?lastId)
)

NULL 值处理

当排序列可能为 NULL 时，Keyset 比较需要额外处理 NULL 语义。Blaze-Persistence 根据 NULLS FIRST / NULLS LAST 配置自动生成正确的 IS NULL 条件：

CriteriaBuilder<Task> cb = cbf.create(em, Task.class)
    .orderByDesc("dueDate")  // dueDate 可能为 NULL
    .orderByAsc("id");

// 如果 lastDueDate 为 NULL，Blaze-Persistence 自动处理:
// WHERE (
//     dueDate IS NOT NULL
//     OR (dueDate IS NULL AND id > ?lastId)
// )
⚠ NULL 排序陷阱：不同数据库对 NULL 的默认排序位置不同： PostgreSQL 默认 NULLS LAST（ASC 时），Oracle 默认 NULLS LAST（DESC 时）。 务必显式指定 NULLS FIRST 或 NULLS LAST 以保证跨数据库一致性。

10.7 陷阱与限制

Keyset 分页虽然性能优异，但并非万能。理解其限制才能在合适的场景中选择合适的分页策略。

限制一览表

限制原因应对策略
不能跳转到任意页 Keyset 依赖上一页的排序键值，无法凭空计算第 N 页的 Keyset 结合 withExtractAllKeysets 或使用 Offset 兜底
必须有确定性排序 排序必须唯一确定每一行的位置，否则 Keyset 会跳过或重复行 始终在排序末尾追加主键列（如 id）
排序列变更后 Keyset 失效 如果排序列的值被修改（如 status 变化），已发出的 Keyset 指向的位置可能不再正确 使用不可变排序列（如 createdAt）或在变更后重新获取第 1 页
集合 JOIN 导致行膨胀 JOIN FETCH 一对多关联会让每行变成多行，LIMIT 计数错误 Blaze-Persistence 自动使用 ID 子查询分页策略
COUNT 在超大表上仍然慢 即使 Keyset 分页本身很快，COUNT(*) 仍需全表扫描 使用 withCountQuery(false) 或维护近似计数器

WHERE cat.id IN (id1, id2, ..., id20)
ORDER BY cat.name, cat.id DB-->>BP: 完整结果（含关联集合） Note over BP: 按 ID 分组，组装为 20 个实体
💡 ID 子查询策略：当分页查询包含集合 JOIN 时，Blaze-Persistence 自动拆分为两步： 先用子查询按 LIMIT 取出主键列表，再用 IN 条件获取完整数据。这避免了笛卡尔积导致的 LIMIT 计数错误，是 Blaze-Persistence 相比原生 JPA 的重要优势。

REST API 中的 Keyset 序列化最佳实践

REST API Keyset 分页流程
sequenceDiagram participant Client as 前端 participant API as REST API participant BP as Blaze-Persistence participant DB as 数据库 Client->>API: GET /cats?page=0&size=20 API->>BP: page(0, 20) BP->>DB: SELECT + COUNT DB-->>BP: 结果 BP-->>API: PagedList + KeysetPage API-->>Client: { data: [...], keyset: "base64..." } Client->>API: GET /cats?size=20&after=base64... API->>BP: page(decode(keyset), 20, 20) BP->>DB: SELECT WHERE (col) > (?) LIMIT 20 DB-->>BP: 结果 BP-->>API: PagedList + 新 KeysetPage API-->>Client: { data: [...], keyset: "newBase64..." } 
// REST Controller 示例
@GetMapping("/cats")
public ResponseEntity<PagedResponse<CatView>> listCats(
        @RequestParam(defaultValue = "0") int offset,
        @RequestParam(defaultValue = "20") int size,
        @RequestParam(required = false) String afterKeyset) {

    CriteriaBuilder<Cat> cb = cbf.create(em, Cat.class)
        .orderByAsc("name")
        .orderByAsc("id");

    EntityViewSetting<CatView, PaginatedCriteriaBuilder<CatView>> setting;

    if (afterKeyset != null) {
        KeysetPage keyset = deserializeKeyset(afterKeyset);
        setting = EntityViewSetting.create(CatView.class, offset, size);
        setting.withKeysetPage(keyset);
    } else {
        setting = EntityViewSetting.create(CatView.class, offset, size);
    }

    PagedList<CatView> result = evm.applySetting(setting, cb).getResultList();

    return ResponseEntity.ok(new PagedResponse<>(
        result,
        result.getTotalSize(),
        serializeKeyset(result.getKeysetPage())
    ));
}

本章小结

  • Offset 分页的 O(n) 扫描复杂度在大表上不可接受，且并发写入时存在丢行/重复行风险
  • Keyset 分页通过 WHERE (cols) > (lastVals) 条件实现 O(1) 翻页，前提是排序列有合适索引
  • PaginatedCriteriaBuilder 统一了两种分页模式：page(offset, size) 与 page(keyset, offset, size)
  • COUNT 查询可通过 withCountQuery(false) 关闭或提供自定义 COUNT 查询优化性能
  • withExtractAllKeysets(true) 为每行提取 Keyset，支持更灵活的跳页策略
  • Entity View 与分页 API 无缝结合，返回 PagedList<ViewClass>，自动优化 SELECT 列
  • 多列 Keyset 自动展开为字典序比较条件，NULL 值根据 NULLS FIRST/LAST 配置处理
  • 集合 JOIN 场景下 Blaze-Persistence 自动采用 ID 子查询策略避免笛卡尔积
