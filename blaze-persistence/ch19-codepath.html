第 19 章 · 代码到 SQL 全路径追踪

CriteriaBuilder → 表达式树 → JPQL → HQL → SQL · 每一步都附真实输出

一行 Java 代码到数据库执行的 SQL，中间到底经过了多少次变换？本章以一条真实的聚合查询为例，追踪它穿越 CriteriaBuilder API、ExpressionFactory、隐式 JOIN 解析、QueryGenerator、Hibernate HQL 解析器直至最终原生 SQL 的每一帧变化。每个步骤都同时给出框架状态图和实际的 JPQL / SQL 输出——图看结构，代码看结果。

  • 观察每个 API 调用后框架内部 Manager 状态的逐帧变化，并用 getQueryString() 验证中间 JPQL
  • 理解 ExpressionFactory 如何将字符串解析为类型化的表达式树，并给出解析后的数据结构
  • 追踪隐式 JOIN 从路径表达式推导到 JoinNode 树扩展的完整过程，对比解析前后的 JPQL 差异
  • 看到 QueryGenerator 如何遍历所有 Manager 拼接 JPQL，展示逐子句的拼接过程
  • 跟随 JPQL 进入 Hibernate 内部被翻译为原生 SQL，给出完整的 PostgreSQL 输出
  • 对比 Entity View 应用后路径差异，附完整 View 定义和生成的 JPQL

前置知识：第 3 章 · CriteriaBuilder 核心 的 API 使用，以及 第 14 章 · 渲染管线 对 QueryGenerator 的初步了解。

本章追踪的示例查询 — 查询已完成订单中电子产品类别、按客户分组、总额超过 1000 的前 20 名客户： 
CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .select("o.customer.name")
    .select("SUM(o.totalAmount)")
    .join("o.items", "i", JoinType.LEFT)
    .where("o.status").eq(OrderStatus.COMPLETED)
    .where("i.product.category").eq("Electronics")
    .groupBy("o.customer.name")
    .having("SUM(o.totalAmount)").gt(1000)
    .orderByAsc("o.customer.name")
    .setMaxResults(20);

下面的实体模型是本章 SQL 追踪的基础：

@Entity @Table(name = "orders")
public class Order {
    @Id private Long id;
    @ManyToOne private Customer customer;
    @OneToMany(mappedBy = "order") private Set<OrderItem> items;
    @Enumerated(EnumType.STRING) private OrderStatus status;
    @Column(name = "total_amount") private BigDecimal totalAmount;
}

@Entity @Table(name = "customers")
public class Customer {
    @Id private Long id;
    private String name;
}

@Entity @Table(name = "order_items")
public class OrderItem {
    @Id private Long id;
    @ManyToOne private Order order;
    @ManyToOne private Product product;
    private int quantity;
}

@Entity @Table(name = "products")
public class Product {
    @Id private Long id;
    private String category;
    private String name;
}

JPQL 对比：隐式 JOIN 解析前 vs 解析后

这是隐式 JOIN 解析最核心的变化——对比用户视角的"简写 JPQL"和框架生成的"展开 JPQL"：

-- ═══════════════════════════════════════════════════
-- BEFORE: 隐式 JOIN 解析前 (用户视角的路径表达式)
-- ═══════════════════════════════════════════════════
SELECT o.customer.name, SUM(o.totalAmount)
FROM Order o
LEFT JOIN o.items i
WHERE o.status = :param_0
  AND i.product.category = :param_1
GROUP BY o.customer.name
HAVING SUM(o.totalAmount) > :param_2
ORDER BY o.customer.name ASC

-- ═══════════════════════════════════════════════════
-- AFTER: 隐式 JOIN 解析后 (实际生成的 JPQL)
-- ═══════════════════════════════════════════════════
SELECT customer_1.name, SUM(o.totalAmount)
FROM Order o
LEFT JOIN o.items i
JOIN o.customer customer_1        -- 隐式 INNER JOIN
JOIN i.product product_1          -- 隐式 INNER JOIN
WHERE o.status = :param_0
  AND product_1.category = :param_1   -- 路径重写: i.product.category → product_1.category
GROUP BY customer_1.name               -- 路径重写: o.customer.name → customer_1.name
HAVING SUM(o.totalAmount) > :param_2
ORDER BY customer_1.name ASC           -- 路径重写: o.customer.name → customer_1.name
三个关键变化 — ① o.customer.name → customer_1.name（路径重写），② 自动插入 JOIN o.customer customer_1（隐式 JOIN 生成），③ i.product.category → product_1.category（路径重写 + 隐式 JOIN JOIN i.product product_1）。

19.4 第四步：QueryGenerator JPQL 生成

调用 cb.getQueryString()（或内部触发查询执行）时，QueryGenerator 遍历所有 Manager，按固定顺序拼接 JPQL 字符串。

JPQL 字符串逐子句生长

每访问一个 Manager，StringBuilder 追加对应片段。以下展示每一步 StringBuilder 的内容：

// ── QueryGenerator 内部 StringBuilder 逐步追踪 ──

// ① visitSelect(SelectManager):
sb = "SELECT customer_1.name, SUM(o.totalAmount)"

// ② visitFrom(JoinManager):
sb += " FROM Order o"
// → "SELECT customer_1.name, SUM(o.totalAmount) FROM Order o"

// ③ visitJoins(JoinManager):
sb += " LEFT JOIN o.items i"
sb += " JOIN o.customer customer_1"
sb += " JOIN i.product product_1"
// → "... FROM Order o LEFT JOIN o.items i JOIN o.customer customer_1 JOIN i.product product_1"

// ④ visitWhere(WhereManager):
sb += " WHERE o.status = :param_0 AND product_1.category = :param_1"

// ⑤ visitGroupBy(GroupByManager):
sb += " GROUP BY customer_1.name"

// ⑥ visitHaving(HavingManager):
sb += " HAVING SUM(o.totalAmount) > :param_2"

// ⑦ visitOrderBy(OrderByManager):
sb += " ORDER BY customer_1.name ASC"

    INNER JOIN products p1_

WHERE

    AND p1_.category = ?
GROUP BY
    c1_.name
HAVING

ORDER BY
    c1_.name ASC
LIMIT ?

19.8 与 Entity View 的路径差异

当查询通过 EntityViewSetting 应用 Entity View 时，管线中的多个步骤发生变化。

Entity View 定义

@EntityView(Order.class)
public interface OrderSummaryView {

    @Mapping("customer.name")
    String getCustomerName();

    @Mapping("SUM(totalAmount)")
    BigDecimal getTotalAmount();
}

使用 Entity View 的查询代码

// 基础查询 (不含 SELECT — 由 Entity View 自动生成):
CriteriaBuilder<Order> baseCb = cbf.create(em, Order.class, "o")
    .join("o.items", "i", JoinType.LEFT)
    .where("o.status").eq(OrderStatus.COMPLETED)
    .where("i.product.category").eq("Electronics")
    .groupBy("o.customer.name")
    .having("SUM(o.totalAmount)").gt(1000)
    .orderByAsc("o.customer.name")
    .setMaxResults(20);

// 应用 Entity View:
EntityViewSetting<OrderSummaryView, CriteriaBuilder<OrderSummaryView>> setting =
    EntityViewSetting.create(OrderSummaryView.class);
CriteriaBuilder<OrderSummaryView> cb = evm.applySetting(setting, baseCb);

// 查看生成的 JPQL:
System.out.println(cb.getQueryString());

// 输出 (与手写版本完全一致):
// SELECT customer_1.name, SUM(o.totalAmount)
// FROM Order o
// LEFT JOIN o.items i
// JOIN o.customer customer_1
// JOIN i.product product_1
// WHERE o.status = :param_0
//   AND product_1.category = :param_1
// GROUP BY customer_1.name
// HAVING SUM(o.totalAmount) > :param_2
// ORDER BY customer_1.name ASC

本章小结

  • CriteriaBuilder 链式调用逐步填充 6 个内部 Manager，每步调用 getQueryString() 可实时查看中间 JPQL。
  • ExpressionFactory 将字符串经词法/语法分析后解析为类型化的表达式节点：PathExpression 表示路径遍历，FunctionExpression 表示聚合函数。
  • 隐式 JOIN 解析是核心变换步骤——o.customer.name 被拆解为显式 JOIN o.customer customer_1 + customer_1.name，使用属性路径去重避免重复 JOIN。
  • QueryGenerator 以 Visitor 模式按固定顺序（SELECT → FROM → JOIN → WHERE → GROUP BY → HAVING → ORDER BY）遍历所有 Manager，将表达式树序列化为 JPQL 字符串。
  • Hibernate 接收 JPQL 后经 HQL Parser → Semantic Analyzer → SQL Generator 翻译为原生 SQL：Order → orders、totalAmount → total_amount、关联属性 JOIN → 外键 ON 条件。
  • 结果映射在 Tuple 模式下通过 TupleTransformer 完成（t.get(0)）；在 Entity View 模式下通过 ObjectBuilder 构造代理对象（view.getCustomerName()），提供类型安全的访问。
  • Entity View 差异仅体现在 SELECT 自动生成（从 @Mapping 推导）和结果映射（ObjectBuilder → View Proxy）两步，生成的 JPQL 和 SQL 完全相同。
