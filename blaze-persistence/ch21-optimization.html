21.1 Entity View 获取策略优化

最简单也最有效的优化——从 SELECT * 到精确投影。当业务只需要 3 个字段时，查询 15+ 列不仅浪费带宽，还会阻止数据库使用覆盖索引。Entity View 自动裁剪查询列。

问题：SELECT * 全字段查询

// ❌ 问题：没有 Entity View → SELECT * 全字段查询
List<Order> orders = em.createQuery(
    "SELECT o FROM Order o", Order.class)
    .getResultList();

// SQL: SELECT o.id, o.status, o.customer_id, o.total_amount,
//      o.shipping_address, o.billing_address, o.notes,
//      o.created_at, o.updated_at, o.deleted, o.version,
//      o.discount_code, o.tax_amount, o.currency, o.source
// → 查询了 15+ 列，但只用到 3 列！

解决：Entity View 精确投影

// ✅ 解决：Entity View 精确投影
@EntityView(Order.class)
public interface OrderListView {
    @IdMapping Long getId();
    String getStatus();
    @Mapping("customer.name") String getCustomerName();
}

CriteriaBuilder<OrderListView> cb = evm.applySetting(
    EntityViewSetting.create(OrderListView.class),
    cbf.create(em, Order.class));
List<OrderListView> views = cb.getResultList();

-- 生成的 SQL（只查询 3 列 + 自动 JOIN）:
SELECT o.id, o.status, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.id
-- → 只查询 3 列 ✅
-- → 数据库可以使用 (id, status) 覆盖索引
-- → 网络传输数据量减少 80%+

进阶：多层级 Entity View 仍精确投影

@EntityView(Order.class)
public interface OrderDetailView {
    @IdMapping Long getId();
    String getStatus();
    BigDecimal getTotalAmount();
    @Mapping("customer") CustomerBriefView getCustomer();
    @Mapping("items") Set<OrderItemBriefView> getItems();
}

@EntityView(Customer.class)
public interface CustomerBriefView {
    @IdMapping Long getId();
    String getName();
    String getEmail();
}

@EntityView(OrderItem.class)
public interface OrderItemBriefView {
    @IdMapping Long getId();
    String getProductName();
    Integer getQuantity();
    BigDecimal getUnitPrice();
}

-- 即使嵌套 3 层 View，仍只查询需要的列:
SELECT o.id, o.status, o.total_amount,
       c.id, c.name, c.email,
       i.id, i.product_name, i.quantity, i.unit_price
FROM orders o
JOIN customers c ON o.customer_id = c.id
LEFT JOIN order_items i ON i.order_id = o.id
-- 每层 View 只投影声明的字段
-- 不会加载 customer.address、customer.phone 等未映射字段

21.2 N+1 问题检测与解决

N+1 问题是 ORM 最经典的性能杀手——查询 1 次获取 N 个父实体，再逐条查询 N 次获取子集合。在 Entity View 中，当 Subview 集合使用默认 SELECT 获取策略时也会触发。

Fig 21-3 N+1 问题发生机制
sequenceDiagram participant App as 应用代码 participant BP as Blaze-Persistence participant DB as 数据库 App->>BP: 查询 100 个订单 + 子项 BP->>DB: SELECT o.id, o.status FROM orders (1 条 SQL) DB-->>BP: 100 个 Order loop 每个 Order 触发一次子查询 BP->>DB: SELECT i.* FROM order_items WHERE order_id = 1 DB-->>BP: items for order 1 BP->>DB: SELECT i.* FROM order_items WHERE order_id = 2 DB-->>BP: items for order 2 Note over BP,DB: ... 重复 98 次 ... BP->>DB: SELECT i.* FROM order_items WHERE order_id = 100 DB-->>BP: items for order 100 end Note over App,DB: 总计: 1 + 100 = 101 条 SQL！

问题代码：默认 SELECT 策略

// ❌ N+1: Subview 默认 SELECT 策略
@EntityView(Order.class)
public interface OrderWithItems {
    @IdMapping Long getId();
    String getStatus();
    Set<OrderItemView> getItems(); // 每个 Order 触发一次子查询!
}

@EntityView(OrderItem.class)
public interface OrderItemView {
    @IdMapping Long getId();
    String getProductName();
    Integer getQuantity();
}

// 查询 100 个订单:
List<OrderWithItems> orders = evm.applySetting(
    EntityViewSetting.create(OrderWithItems.class),
    cbf.create(em, Order.class))
    .getResultList();

-- 生成的 SQL（N+1 问题）:
-- 第 1 条: 查询所有 Order
SELECT o.id, o.status FROM orders o

-- 第 2 ~ 101 条: 每个 Order 逐条查询 items
SELECT i.id, i.product_name, i.quantity FROM order_items i WHERE i.order_id = 1
SELECT i.id, i.product_name, i.quantity FROM order_items i WHERE i.order_id = 2
SELECT i.id, i.product_name, i.quantity FROM order_items i WHERE i.order_id = 3
...
SELECT i.id, i.product_name, i.quantity FROM order_items i WHERE i.order_id = 100
-- 总计: 101 条 SQL

解决方案 1：@BatchFetch 批量加载

// ✅ 解决1: @BatchFetch 批量加载
@EntityView(Order.class)
public interface OrderWithItemsBatched {
    @IdMapping Long getId();
    String getStatus();

    @BatchFetch(size = 50)
    Set<OrderItemView> getItems();
    // 将 100 个 order_id 分为 2 批，用 IN (...) 批量加载
}

-- 生成的 SQL（@BatchFetch 优化后）:
-- 第 1 条: 查询所有 Order
SELECT o.id, o.status FROM orders o

-- 第 2 条: 批量加载前 50 个 Order 的 items
SELECT i.id, i.product_name, i.quantity, i.order_id
FROM order_items i
WHERE i.order_id IN (1, 2, 3, ..., 50)

-- 第 3 条: 批量加载后 50 个 Order 的 items
SELECT i.id, i.product_name, i.quantity, i.order_id
FROM order_items i
WHERE i.order_id IN (51, 52, 53, ..., 100)

-- 总计: 1 + 2 = 3 条 SQL（从 101 条降到 3 条）

解决方案 2：Correlated Subquery

// ✅ 解决2: @MappingCorrelated 子查询关联
@EntityView(Order.class)
public interface OrderWithItemsCorrelated {
    @IdMapping Long getId();
    String getStatus();

    @MappingCorrelated(
        correlator = OrderItemCorrelator.class,
        correlationBasis = "id",
        correlationResult = "this",
        fetch = FetchStrategy.SUBSELECT
    )
    Set<OrderItemView> getItems();
}

public class OrderItemCorrelator implements CorrelationProvider {
    @Override
    public void applyCorrelation(
            CorrelationBuilder correlationBuilder, String correlationExpression) {
        correlationBuilder
            .correlate(OrderItem.class, "i")
            .on("i.order.id").eqExpression(correlationExpression)
            .end();
    }
}

-- 生成的 SQL（Correlated Subquery — SUBSELECT 策略）:
-- 第 1 条: 查询所有 Order
SELECT o.id, o.status FROM orders o

-- 第 2 条: 一次性子查询加载所有 items
SELECT i.id, i.product_name, i.quantity, i.order_id
FROM order_items i
WHERE i.order_id IN (
    SELECT o.id FROM orders o
)
-- 总计: 2 条 SQL（无论有多少 Order）

解决方案 3：JOIN 获取（单查询）

// ✅ 解决3: JOIN 获取 — 单条 SQL 搞定
@EntityView(Order.class)
public interface OrderWithItemsJoined {
    @IdMapping Long getId();
    String getStatus();

    @MappingCorrelated(
        correlator = OrderItemCorrelator.class,
        correlationBasis = "id",
        correlationResult = "this",
        fetch = FetchStrategy.JOIN
    )
    Set<OrderItemView> getItems();
}

-- 生成的 SQL（JOIN — 单条查询）:
SELECT o.id, o.status,
       i.id AS items_id,
       i.product_name AS items_productName,
       i.quantity AS items_quantity
FROM orders o
LEFT JOIN order_items i ON i.order_id = o.id
-- 总计: 1 条 SQL
-- 注意: JOIN 会产生笛卡尔积，行数 = Order数 × 平均Items数

21.3 分页优化 — Offset vs Keyset

OFFSET 分页是最常见的分页方式，但在深翻页场景下性能急剧恶化——第 5000 页意味着数据库要扫描 100000 行然后丢弃前 99980 行。Keyset 分页通过 WHERE 条件代替 OFFSET，实现恒定 O(1) 的分页性能。

Fig 21-5 Offset vs Keyset 分页性能特征

问题：OFFSET 深翻页

// ❌ OFFSET 分页 — 第 5000 页性能极差
PaginatedCriteriaBuilder<OrderView> pcb = evm.applySetting(
    EntityViewSetting.create(OrderView.class, 99980, 20),
    cbf.create(em, Order.class)
        .orderByAsc("id")
);
PagedList<OrderView> page = pcb.getResultList();

-- OFFSET 分页生成的 SQL:
SELECT o.id, o.status, o.total_amount
FROM orders o
ORDER BY o.id ASC
LIMIT 20 OFFSET 99980
-- → 扫描 100,000 行，丢弃 99,980 行
-- → 页码越大越慢，第 5000 页比第 1 页慢 500 倍

解决：Keyset 分页 — 恒定性能

// ✅ Keyset 分页 — 恒定性能
// 第一页（无 keyset，等价于普通 LIMIT）:
PaginatedCriteriaBuilder<OrderView> firstPage = evm.applySetting(
    EntityViewSetting.create(OrderView.class, 0, 20),
    cbf.create(em, Order.class)
        .orderByAsc("id")
);
PagedList<OrderView> page1 = firstPage.getResultList();

// 后续页（使用前一页的 KeysetPage）:
KeysetPage keyset = page1.getKeysetPage();
PaginatedCriteriaBuilder<OrderView> nextPage = evm.applySetting(
    EntityViewSetting.create(OrderView.class, keyset, 20),
    cbf.create(em, Order.class)
        .orderByAsc("id")
);
PagedList<OrderView> page2 = nextPage.getResultList();

-- Keyset 分页生成的 SQL:
-- 第一页:
SELECT o.id, o.status, o.total_amount
FROM orders o
ORDER BY o.id ASC
LIMIT 20

-- 第二页（使用 Keyset）:
SELECT o.id, o.status, o.total_amount
FROM orders o
WHERE o.id > 12345       -- 上一页最后一条的 id
ORDER BY o.id ASC
LIMIT 20
-- → Index Seek, 只扫描 20 行 ✅
-- → 无论第几页，性能恒定 O(1)

进阶：复合排序键的 Keyset 分页

// 复合排序键: createdAt DESC, id DESC
PaginatedCriteriaBuilder<OrderView> pcb = evm.applySetting(
    EntityViewSetting.create(OrderView.class, keyset, 20),
    cbf.create(em, Order.class)
        .orderByDesc("createdAt")
        .orderByDesc("id")
);
PagedList<OrderView> page = pcb.getResultList();

-- 复合 Keyset 生成的 SQL:
SELECT o.id, o.status, o.total_amount, o.created_at
FROM orders o
WHERE (o.created_at, o.id) < (?, ?)
ORDER BY o.created_at DESC, o.id DESC
LIMIT 20
-- 参数: [lastCreatedAt, lastId]
-- → 行值比较 (row value comparison) 利用复合索引
-- → 推荐创建索引: CREATE INDEX idx_order_created_id
--   ON orders(created_at DESC, id DESC)

21.4 CTE 优化复杂查询

当报表查询涉及聚合 + 排名 + 过滤时，嵌套子查询会让 SQL 难以阅读和优化。CTE 将复杂逻辑分解为命名步骤，既提高可读性，又让数据库优化器有更大的优化空间。

Fig 21-6 CTE 将嵌套子查询分解为线性步骤

实战：月度客户销售排名 Top 10

// CTE 实体定义
@CTE
@Entity
public class MonthlySalesCTE {
    @Id Long customerId;
    String customerName;
    BigDecimal monthlyTotal;
    Integer salesRank;
}

// 构建查询
LocalDate startOfMonth = LocalDate.of(2026, 4, 1);
LocalDate endOfMonth = LocalDate.of(2026, 5, 1);

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .with(MonthlySalesCTE.class)
        .from(Order.class, "o")
        .bind("customerId").select("o.customer.id")
        .bind("customerName").select("o.customer.name")
        .bind("monthlyTotal").select("SUM(o.totalAmount)")
        .bind("salesRank").select(
            "WINDOW_ROW_NUMBER() OVER (ORDER BY SUM(o.totalAmount) DESC)")
        .where("o.createdAt").ge(startOfMonth)
        .where("o.createdAt").lt(endOfMonth)
        .groupBy("o.customer.id")
        .groupBy("o.customer.name")
    .end()
    .from(MonthlySalesCTE.class, "m")
    .select("m.salesRank")
    .select("m.customerName")
    .select("m.monthlyTotal")
    .where("m.salesRank").le(10)
    .orderByAsc("m.salesRank");

-- 生成的 SQL:
WITH MonthlySalesCTE(customerId, customerName, monthlyTotal, salesRank) AS (
    SELECT c.id,
           c.name,
           SUM(o.total_amount),
           ROW_NUMBER() OVER (ORDER BY SUM(o.total_amount) DESC)
    FROM orders o
    JOIN customers c ON o.customer_id = c.id
    WHERE o.created_at >= '2026-04-01'
      AND o.created_at < '2026-05-01'
    GROUP BY c.id, c.name
)
SELECT m.salesRank, m.customerName, m.monthlyTotal
FROM MonthlySalesCTE m
WHERE m.salesRank <= 10
ORDER BY m.salesRank ASC

-- 执行计划优势:
-- 1. CTE 物化一次，主查询只扫描 ≤10 行
-- 2. GROUP BY + SUM 在 CTE 内完成，窗口函数紧随其后
-- 3. 外层过滤 salesRank <= 10 极快（只过滤聚合结果）
为什么不用嵌套子查询？嵌套方案需要 SELECT * FROM (SELECT * FROM (SELECT ... GROUP BY) t1) t2 WHERE rank <= 10——三层嵌套。CTE 版本将每步拆为独立命名块，SQL 可读性提升且方便调试（可以单独执行 CTE 部分验证中间结果）。

21.5 批量 DML 优化

逐条 DML 是生产环境最常见的性能反模式——N 条记录产生 N 条 SQL 语句，每条都有网络往返、事务日志写入、索引更新的开销。Blaze-Persistence 的 CriteriaBuilder 支持批量 UPDATE、DELETE 和 INSERT...SELECT，将 N 条语句合并为 1 条。

Fig 21-7 逐条 DML vs 批量 DML

问题：逐条更新

// ❌ 逐条更新 — N 条 UPDATE
List<Order> expiredOrders = em.createQuery(
    "SELECT o FROM Order o WHERE o.status = :s AND o.createdAt < :cutoff",
    Order.class)
    .setParameter("s", OrderStatus.PENDING)
    .setParameter("cutoff", cutoffDate)
    .getResultList();

for (Order o : expiredOrders) {
    o.setStatus(OrderStatus.CANCELLED);
    o.setUpdatedAt(Instant.now());
    em.merge(o);
}
// → 先 SELECT 加载所有实体，再逐条 UPDATE
// → 1000 条过期订单 = 1 SELECT + 1000 UPDATE = 1001 条 SQL

解决 1：批量 UPDATE

// ✅ 批量 UPDATE — 1 条 SQL 更新所有匹配行
int updated = cbf.update(em, Order.class, "o")
    .set("o.status", OrderStatus.CANCELLED)
    .set("o.updatedAt", Instant.now())
    .where("o.status").eq(OrderStatus.PENDING)
    .where("o.createdAt").lt(cutoffDate)
    .executeUpdate();

System.out.println("已取消 " + updated + " 个过期订单");

-- 生成的 SQL（1 条搞定）:
UPDATE orders
SET status = 'CANCELLED',
    updated_at = '2026-04-21T10:30:00Z'
WHERE status = 'PENDING'
  AND created_at < '2026-04-01T00:00:00Z'
-- → 1 条 SQL 更新 1000 行
-- → 无需先 SELECT 加载实体到内存

解决 2：批量 DELETE

// ✅ 批量 DELETE — 清理一年前的审计日志
Instant oneYearAgo = Instant.now().minus(365, ChronoUnit.DAYS);

int deleted = cbf.delete(em, AuditLog.class, "a")
    .where("a.createdAt").lt(oneYearAgo)
    .executeUpdate();

System.out.println("已清理 " + deleted + " 条过期审计日志");

-- 生成的 SQL:
DELETE FROM audit_logs
WHERE created_at < '2025-04-21T10:30:00Z'
-- → 单条 SQL 批量删除

解决 3：INSERT...SELECT（数据归档）

// ✅ INSERT...SELECT — 将旧订单归档到 order_archive 表
int archived = cbf.insert(em, OrderArchive.class)
    .from(Order.class, "o")
    .bind("id").select("o.id")
    .bind("status").select("o.status")
    .bind("customerName").select("o.customer.name")
    .bind("totalAmount").select("o.totalAmount")
    .bind("createdAt").select("o.createdAt")
    .bind("archivedAt").select("CURRENT_TIMESTAMP")
    .where("o.createdAt").lt(archiveDate)
    .where("o.status").in(OrderStatus.COMPLETED, OrderStatus.CANCELLED)
    .executeUpdate();

// 归档后删除原表数据:
int cleaned = cbf.delete(em, Order.class, "o")
    .where("o.createdAt").lt(archiveDate)
    .where("o.status").in(OrderStatus.COMPLETED, OrderStatus.CANCELLED)
    .executeUpdate();

System.out.println("已归档 " + archived + " 条，清理 " + cleaned + " 条");

-- 生成的 SQL — INSERT...SELECT:
INSERT INTO order_archive(id, status, customer_name,
                          total_amount, created_at, archived_at)
SELECT o.id, o.status, c.name,
       o.total_amount, o.created_at, CURRENT_TIMESTAMP
FROM orders o

// 每日销售报表：日销售额 + 7日移动平均 + 累计总额
CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    // 日期维度
    .select("FUNCTION('DATE_TRUNC', 'day', o.createdAt)")
    // 日销售额
    .select("SUM(o.totalAmount)")
    // 7日移动平均
    .select("WINDOW_AVG(SUM(o.totalAmount)) OVER ("
        + "ORDER BY FUNCTION('DATE_TRUNC', 'day', o.createdAt) "
        + "ROWS BETWEEN 6 PRECEDING AND CURRENT ROW)")
    // 累计总额
    .select("WINDOW_SUM(SUM(o.totalAmount)) OVER ("
        + "ORDER BY FUNCTION('DATE_TRUNC', 'day', o.createdAt) "
        + "ROWS UNBOUNDED PRECEDING)")
    // 日环比（前一天销售额）
    .select("WINDOW_LAG(SUM(o.totalAmount), 1) OVER ("
        + "ORDER BY FUNCTION('DATE_TRUNC', 'day', o.createdAt))")
    .where("o.createdAt").ge(startDate)
    .where("o.createdAt").lt(endDate)
    .groupBy("FUNCTION('DATE_TRUNC', 'day', o.createdAt)")
    .orderByAsc("FUNCTION('DATE_TRUNC', 'day', o.createdAt)");

List<Tuple> report = cb.getResultList();

-- 生成的 PostgreSQL SQL:
SELECT DATE_TRUNC('day', o.created_at) AS sale_date,
       SUM(o.total_amount) AS daily_total,
       AVG(SUM(o.total_amount)) OVER (
           ORDER BY DATE_TRUNC('day', o.created_at)
           ROWS BETWEEN 6 PRECEDING AND CURRENT ROW

       SUM(SUM(o.total_amount)) OVER (

           ROWS UNBOUNDED PRECEDING
       ) AS cumulative_total,
       LAG(SUM(o.total_amount), 1) OVER (
           ORDER BY DATE_TRUNC('day', o.created_at)
       ) AS prev_day_total
FROM orders o
WHERE o.created_at >= '2026-04-01'
  AND o.created_at < '2026-05-01'
GROUP BY DATE_TRUNC('day', o.created_at)
ORDER BY DATE_TRUNC('day', o.created_at) ASC

Step 1：注册自定义全文搜索函数

public class PgSearchFunction implements JpqlFunction {

    @Override
    public boolean hasArguments() {
        return true;
    }

    @Override
    public boolean hasParenthesesIfNoArguments() {
        return true;
    }

    @Override
    public Class<?> getReturnType(Class<?> firstArgumentType) {
        return Boolean.class;
    }

    @Override
    public void render(FunctionRenderContext ctx) {
        ctx.addChunk("to_tsvector('chinese', ");
        ctx.addArgument(0);   // 被搜索的字段
        ctx.addChunk(") @@ to_tsquery('chinese', ");
        ctx.addArgument(1);   // 搜索关键词
        ctx.addChunk(")");
    }
}

// 搜索排名函数
public class PgTsRankFunction implements JpqlFunction {

    @Override
    public boolean hasArguments() { return true; }

    @Override
    public boolean hasParenthesesIfNoArguments() { return true; }

    @Override
    public Class<?> getReturnType(Class<?> firstArgumentType) {
        return Float.class;
    }

    @Override
    public void render(FunctionRenderContext ctx) {
        ctx.addChunk("ts_rank(to_tsvector('chinese', ");
        ctx.addArgument(0);
        ctx.addChunk("), to_tsquery('chinese', ");
        ctx.addArgument(1);
        ctx.addChunk("))");
    }
}

// 注册
CriteriaBuilderConfiguration config = Criteria.getDefault();
config.registerFunction(
    new JpqlFunctionGroup("pg_search", new PgSearchFunction()));
config.registerFunction(
    new JpqlFunctionGroup("pg_ts_rank", new PgTsRankFunction()));

Step 2：在查询中使用全文搜索

// 全文搜索文章 + 按相关度排序
String searchKeyword = "Blaze-Persistence & 优化";

CriteriaBuilder<ArticleSearchView> cb = evm.applySetting(
    EntityViewSetting.create(ArticleSearchView.class),
    cbf.create(em, Article.class, "a")
        .where("FUNCTION('pg_search', a.content, :keyword)").eq(true)
        .orderByDesc("FUNCTION('pg_ts_rank', a.content, :keyword)")
);
cb.setParameter("keyword", searchKeyword);

List<ArticleSearchView> results = cb.getResultList();

-- 生成的 PostgreSQL SQL:
SELECT a.id, a.title, a.author, a.created_at
FROM articles a
WHERE to_tsvector('chinese', a.content)
      @@ to_tsquery('chinese', 'Blaze-Persistence & 优化')
ORDER BY ts_rank(
    to_tsvector('chinese', a.content),
    to_tsquery('chinese', 'Blaze-Persistence & 优化')
) DESC

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .withRecursive(OrgTreeCTE.class)
        // ====== Anchor: 根节点（parent IS NULL）======
        .from(Department.class, "d")
        .bind("id").select("d.id")
        .bind("name").select("d.name")
        .bind("parentId").select("d.parent.id")
        .bind("level").select("0")
        .bind("path").select("CONCAT('/', d.name)")
        .where("d.parent").isNull()
    .unionAll()
        // ====== Recursive: 子节点关联父节点 ======
        .from(Department.class, "d")
        .from(OrgTreeCTE.class, "t")
        .bind("id").select("d.id")
        .bind("name").select("d.name")
        .bind("parentId").select("d.parent.id")
        .bind("level").select("t.level + 1")
        .bind("path").select("CONCAT(t.path, '/', d.name)")
        .where("d.parent.id").eqExpression("t.id")
    .end()
    // ====== 主查询: 从 CTE 读取完整树 ======
    .from(OrgTreeCTE.class, "t")
    .select("t.level")
    .select("t.name")
    .select("t.path")
    .select("t.parentId")
    .orderByAsc("t.path");

           CONCAT('/', d.name)

    WHERE d.parent_id IS NULL

    JOIN OrgTreeCTE t ON d.parent_id = t.id
)
SELECT t.level, t.name, t.path, t.parentId

ORDER BY t.path ASC

陷阱 1: em.find()

陷阱 4: CTE + MySQL 5.x
❌ MySQL 5.x 不支持 CTE，递归 CTE 无法降级
✅ 升级到 MySQL 8.0+ 或使用 PostgreSQL
陷阱 5: ViewFilter 遗漏
❌ 软删除数据泄漏，已删除订单出现在列表
✅ 用 ViewFilterProvider 自动激活过滤器

陷阱 1：Entity View 不等于 em.find()

// ❌ 常见错误: 用 em.find() 查询单个实体 → 绕过 Entity View
Order order = em.find(Order.class, orderId);
// → SELECT * FROM orders WHERE id = ? (全字段)

// ✅ 正确: 使用 evm.find() 获取 View 投影
OrderView view = evm.find(em, OrderView.class, orderId);
// → SELECT o.id, o.status, c.name FROM orders o
//   JOIN customers c ON o.customer_id = c.id
//   WHERE o.id = ?

陷阱 2：批量 DML 后一级缓存不同步

// ❌ 陷阱: 批量 UPDATE 后已加载的实体仍是旧值
Order order = em.find(Order.class, 1L);
System.out.println(order.getStatus()); // → PENDING

cbf.update(em, Order.class, "o")
    .set("o.status", OrderStatus.CANCELLED)
    .where("o.id").eq(1L)
    .executeUpdate();

// order 对象仍在 L1 缓存中，值未更新!
System.out.println(order.getStatus()); // → 仍然是 PENDING ❌

// ✅ 解决: 批量 DML 后清除一级缓存
cbf.update(em, Order.class, "o")
    .set("o.status", OrderStatus.CANCELLED)
    .where("o.id").eq(1L)
    .executeUpdate();
em.clear();  // 清除 L1 缓存

Order refreshed = em.find(Order.class, 1L);
System.out.println(refreshed.getStatus()); // → CANCELLED ✅

陷阱 3：什么时候不该用 Blaze-Persistence

场景推荐方案原因
超简单 CRUD (单表、无关联)Spring Data JPA 原生引入框架的复杂度不值得
存储过程/原生 SQL 强依赖MyBatis / jOOQBlaze-Persistence 基于 JPQL，无法调用存储过程
NoSQL 数据库Spring Data MongoDB 等Blaze-Persistence 只支持关系型数据库
极致性能 (微秒级)JDBC + 手写 SQL任何 ORM 抽象都有开销
遗留系统表结构极不规范MyBatis XML 映射JPA 实体映射困难时，先解决映射问题

JPA Provider 特定注意事项

// Hibernate 特定: 启用 hibernate.query.plan_cache_max_size
// Blaze-Persistence 生成大量动态 JPQL → 查询计划缓存可能溢出
// 建议值: 4096 (默认 2048)
properties.put("hibernate.query.plan_cache_max_size", "4096");

// EclipseLink 特定: 共享缓存与批量 DML 冲突
// 批量 UPDATE/DELETE 不会通知 EclipseLink L2 缓存
// 解决: 批量 DML 后手动失效相关缓存
em.getEntityManagerFactory().getCache()
    .evict(Order.class);  // 失效 Order 的 L2 缓存

// Hibernate 6.x / Jakarta EE 10:
// Blaze-Persistence 1.6+ 同时支持 javax.persistence 和 jakarta.persistence
// 检查依赖版本匹配:
// blaze-persistence-integration-hibernate-6.2 → Hibernate 6.2+
// blaze-persistence-integration-hibernate-5.6 → Hibernate 5.6

本章小结

  • Entity View 精确投影 将 SELECT * 裁剪为只查询需要的列——减少 80%+ 的网络传输数据量，启用覆盖索引扫描，消除 EntityManager 一级缓存的内存压力
  • N+1 问题有三种解决方案：@BatchFetch 最简单（一行注解），Correlated SUBSELECT 最均衡（2 条 SQL 无膨胀），JOIN 获取最激进（1 条 SQL 但有笛卡尔积）
  • Keyset 分页用 WHERE id > ? 替代 OFFSET——第 5000 页的性能与第 1 页相同 (~5ms)，而 OFFSET 分页在深翻页时性能下降 500 倍
  • CTE 将嵌套 3 层的子查询分解为线性命名步骤——提高 SQL 可读性，让优化器有更大的优化空间
  • 批量 DML 用 1 条 SQL 替代 N 条——批量 UPDATE/DELETE 比逐条操作快 300 倍，INSERT...SELECT 在数据库内部完成数据迁移，不经过应用层
  • 窗口函数让报表查询在单条 SQL 中返回日销售额、7 日移动平均、累计总额、日环比 4 个维度——无需多次查询拼装
  • @ViewFilter 将软删除和多租户条件抽为命名过滤器——一次定义、查询时按名称激活，配合 ViewFilterProvider 实现完全透明的条件注入
  • 全文搜索通过 JpqlFunction 扩展点将 PostgreSQL 的 tsvector/tsquery 集成到 Blaze-Persistence——在 JPQL 层面使用，SQL 渲染时输出原生语法
  • 递归 CTE 在单条 SQL 中展开任意深度的树形结构——组织架构、分类目录、评论回复等场景不再需要应用层递归查询
  • 生产调优需要关注连接池大小（CPU核心数 × 2 + 磁盘数）、查询计划缓存、批量 DML 后的缓存失效、Keyset 排序列的唯一性
