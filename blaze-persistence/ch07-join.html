7.1 隐式 JOIN vs 显式 JOIN

在 Blaze-Persistence 中，当你通过点号表达式（如 order.customer.name）访问关联属性时， 框架会自动推断所需的 JOIN 并在生成的 JPQL / SQL 中隐式添加。 这种「路径导航触发关联」的机制称为隐式 JOIN。与之对应， join() / innerJoin() 等 API 让你显式声明关联关系及其别名。

隐式 JOIN vs 显式 JOIN — 决策流程

隐式 JOIN 路径解析

隐式 JOIN 最大的优势是简洁——开发者只关注要访问的属性路径， JOIN 的生成完全交给框架。Blaze-Persistence 会对路径进行解析， 按照实体元模型将每一段路径映射为相应的 SQL JOIN。 

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .select("o.customer.name")          // 隐式 LEFT JOIN Customer
    .select("o.customer.address.city")  // 复用已有 JOIN + 追加 Address JOIN
    .where("o.customer.vip").eq(true)   // 复用同一个 Customer JOIN
    .orderByAsc("o.customer.name");
// 生成 SQL:
// SELECT c.name, a.city FROM Order o
//   LEFT JOIN Customer c ON o.customer_id = c.id
//   LEFT JOIN Address  a ON c.address_id  = a.id
// WHERE c.vip = true ORDER BY c.name
路径去重：多次引用同一路径段（如 o.customer）只会产生一次 JOIN， Blaze-Persistence 内部通过路径注册表进行去重。

显式 JOIN 声明

当你需要控制 JOIN 类型、附加额外 ON 条件或跨实体关联时，使用显式 JOIN API： 

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .innerJoin("o.customer", "c")           // 显式 INNER JOIN
    .leftJoin("o.items", "i")               // 显式 LEFT JOIN
    .select("c.name")
    .select("i.productName")
    .where("c.country").eq("CN");
// 开发者完全控制 JOIN 类型与别名

隐式 JOIN 的问题场景

当实体中存在多条路径指向同一目标实体时，隐式 JOIN 可能导致意想不到的重复关联。 

// 实体模型：Order 同时有 createdBy 和 approvedBy 两个 User 关联
// 隐式 JOIN 会为每条路径生成独立的 JOIN
cb.select("o.createdBy.name")    // LEFT JOIN User u1 ON ...
  .select("o.approvedBy.name")   // LEFT JOIN User u2 ON ...（不同路径 → 不同 JOIN）

// 显式 JOIN 让你精确控制别名和复用
cb.leftJoin("o.createdBy", "creator")
  .leftJoin("o.approvedBy", "approver")
  .select("creator.name")
  .select("approver.name");
注意：隐式 JOIN 默认使用 LEFT JOIN。如果业务语义要求 INNER JOIN（即关联必须存在）， 必须改用显式 JOIN 以避免返回不必要的 NULL 行。

7.2 JOIN 类型

Blaze-Persistence 支持 SQL 标准定义的四种 JOIN 类型，每种类型对未匹配行的处理方式不同。

四种 JOIN 类型 — 数据包含关系
INNER JOIN
只返回两表都匹配的行
Order ∩ Customer
LEFT JOIN
返回左表所有行
右表无匹配 → NULL
Order ∪ (Order ∩ Customer)
RIGHT JOIN
返回右表所有行
左表无匹配 → NULL
(Order ∩ Customer) ∪ Customer
FULL JOIN
返回两表所有行
任一方无匹配 → NULL
Order ∪ Customer

四种 JOIN 的 API

JOIN 类型API 方法SQL 生成典型场景
INNER innerJoin(path, alias) INNER JOIN ... ON ... 关联必须存在（如订单必有客户）
LEFT leftJoin(path, alias) LEFT JOIN ... ON ... 关联可选（如订单可能没有优惠券）
RIGHT rightJoin(path, alias) RIGHT JOIN ... ON ... 以右表为主（少见，可用子查询替代）
FULL join(path, alias, JoinType.FULL) FULL OUTER JOIN ... ON ... 全量匹配报表（如对账） 
// INNER JOIN — 只查有客户的订单
CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
    .from(Order.class, "o")
    .innerJoin("o.customer", "c")
    .where("c.country").eq("CN");

// LEFT JOIN — 包含无明细的订单
CriteriaBuilder<Tuple> cb2 = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .leftJoin("o.items", "i")
    .select("o.id").select("i.productName");

// RIGHT JOIN
CriteriaBuilder<Tuple> cb3 = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .rightJoin("o.customer", "c")
    .select("o.id").select("c.name");

// FULL OUTER JOIN
CriteriaBuilder<Tuple> cb4 = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .join("o.customer", "c", JoinType.FULL)
    .select("o.id").select("c.name");

joinOn() — 自定义 JOIN 条件

joinOn() 允许你在 JOIN 的 ON 子句中附加额外条件， 而不是放在 WHERE 中——这在 LEFT JOIN 场景下语义完全不同。

ON 条件 vs WHERE 条件 — 语义差异
flowchart LR subgraph ON_CLAUSE ["ON 条件（joinOn）"] O1["LEFT JOIN items i\n ON o.id = i.order_id\n AND i.status = 'ACTIVE'"] O2["效果: 未匹配的 Order 仍保留\ni 列为 NULL"] end subgraph WHERE_CLAUSE ["WHERE 条件"] W1["LEFT JOIN items i\n ON o.id = i.order_id\nWHERE i.status = 'ACTIVE'"] W2["效果: 过滤掉了 i 为 NULL 的行\nLEFT JOIN 退化为 INNER JOIN"] end O1 --> O2 W1 --> W2 style ON_CLAUSE fill:#1a3a2a,stroke:#3fb950 style WHERE_CLAUSE fill:#3a1a1a,stroke:#f85149 
CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .leftJoinOn("o.items", "i")             // 开始 ON 条件
        .on("i.status").eq(ItemStatus.ACTIVE)
    .end()                                   // 结束 ON 条件
    .select("o.id")
    .select("i.productName");
// SQL: LEFT JOIN OrderItem i ON o.id = i.order_id AND i.status = 'ACTIVE'

7.3 FETCH JOIN

FETCH JOIN 是解决 JPA 延迟加载 N+1 问题的核心手段。 Blaze-Persistence 通过 fetch() API 让你在查询时声明需要预加载的关联， 将原本需要 N+1 次查询的场景合并为单次 JOIN 查询。

N+1 问题 vs FETCH JOIN — 查询次数对比
sequenceDiagram participant App as 应用代码 participant BP as Blaze-Persistence participant DB as 数据库 Note over App,DB: ❌ 无 FETCH JOIN（N+1） App->>BP: 查询 10 个 Order BP->>DB: SELECT * FROM orders (1 次) DB-->>BP: 10 行 Order loop 每个 Order 访问 customer App->>BP: order.getCustomer() BP->>DB: SELECT * FROM customers WHERE id=? (N 次) DB-->>BP: 1 行 Customer end Note over App,DB: 总计: 1 + 10 = 11 次查询 Note over App,DB: ✅ 使用 FETCH JOIN App->>BP: 查询 Order + fetch customer BP->>DB: SELECT o.*, c.* FROM orders o JOIN customers c ON ... (1 次) DB-->>BP: 10 行 (Order + Customer) Note over App,DB: 总计: 1 次查询

fetch() API 使用

// 使用 fetch() 预加载关联
CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
    .from(Order.class, "o")
    .fetch("o.customer")        // FETCH JOIN Customer
    .fetch("o.items")           // FETCH JOIN OrderItem
    .where("o.status").eq(OrderStatus.PENDING);
// SQL: SELECT o FROM Order o
//        JOIN FETCH o.customer
//        JOIN FETCH o.items
//      WHERE o.status = 'PENDING'
// JPA 原生 JOIN FETCH
em.createQuery(
  "SELECT o FROM Order o " +
  "JOIN FETCH o.customer " +
  "JOIN FETCH o.items " +
  "WHERE o.status = :s",
  Order.class)
  .setParameter("s", "PENDING")
  .getResultList();
// Blaze-Persistence fetch()
cbf.create(em, Order.class)
  .from(Order.class, "o")
  .fetch("o.customer")
  .fetch("o.items")
  .where("o.status")
    .eq(OrderStatus.PENDING)
  .getResultList();
与 JPA JOIN FETCH 的区别：Blaze-Persistence 的 fetch() 在 Entity View 场景下会自动优化为仅 SELECT 所需字段，而 JPA 的 JOIN FETCH 总是加载完整实体。

笛卡尔积陷阱

同时 FETCH JOIN 多个集合关联会导致笛卡尔积—— 如果 Order 有 3 个 Item 和 2 个 Payment，结果集膨胀为 3 × 2 = 6 行。

多集合 FETCH JOIN 的笛卡尔积问题
flowchart TD O["Order #1"] --> I1["Item A"] O --> I2["Item B"] O --> I3["Item C"] O --> P1["Payment X"] O --> P2["Payment Y"] subgraph CARTESIAN ["笛卡尔积结果集 = 3 × 2 = 6 行"] R1["(Order#1, Item A, Payment X)"] R2["(Order#1, Item A, Payment Y)"] R3["(Order#1, Item B, Payment X)"] R4["(Order#1, Item B, Payment Y)"] R5["(Order#1, Item C, Payment X)"] R6["(Order#1, Item C, Payment Y)"] end I1 --> R1 I1 --> R2 I2 --> R3 I2 --> R4 I3 --> R5 I3 --> R6 style O fill:#1e3a5f,stroke:#58a6ff style CARTESIAN fill:#3a1a1a,stroke:#f85149
规避策略：对于多集合关联，改用 SUBSELECT 或 MULTISET FetchStrategy（详见 7.6 节），或将查询拆分为多次单集合 FETCH JOIN。 Hibernate 在检测到多 bag FETCH 时会抛出 MultipleBagFetchException。

7.4 关联路径表达式

Blaze-Persistence 允许使用点号分隔的路径表达式进行多级实体导航。 框架会按路径的每一段自动创建所需的 JOIN，形成从根实体到目标属性的完整关联链。

点号导航与 SQL 解析

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    // 四级路径导航
    .select("o.customer.address.city.name")
    // 等效 SQL:
    // SELECT city.name FROM Order o
    //   LEFT JOIN Customer c    ON o.customer_id = c.id
    //   LEFT JOIN Address  a    ON c.address_id  = a.id
    //   LEFT JOIN City     city ON a.city_id     = city.id
    .getResultList();
Embeddable 导航：如果路径中某段是 @Embeddable 而非独立实体， Blaze-Persistence 不会为该段生成 JOIN，而是直接引用嵌入属性所在表的列。 例如 customer.contactInfo.phone 中，contactInfo 是 Embeddable， 则只 JOIN Customer 表，直接读取其 contact_phone 列。
Embeddable vs Entity 路径解析差异
flowchart TD subgraph ENTITY_PATH ["实体路径: o.customer.address"] EP1["Order o"] -->|"JOIN"| EP2["Customer c"] EP2 -->|"JOIN"| EP3["Address a"] end subgraph EMBED_PATH ["Embeddable 路径: c.contactInfo.phone"] EB1["Customer c"] -->|"无 JOIN\n直接列引用"| EB2["c.contact_phone"] end style ENTITY_PATH fill:#1e3a5f,stroke:#58a6ff style EMBED_PATH fill:#1a3a2a,stroke:#3fb950

路径表达式还支持集合导航。对集合属性使用路径会隐式产生 JOIN， 随后可用聚合函数或 SIZE 表达式对集合进行计算： 

// 集合路径 + 聚合
CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .select("o.id")
    .select("SIZE(o.items)")             // 隐式 JOIN + COUNT
    .select("o.items.price").max()       // 隐式 JOIN + MAX
    .groupBy("o.id");

7.5 跨实体关联与 Lateral JOIN

标准 JPA 只能 JOIN 实体间已声明的 @ManyToOne / @OneToMany 等映射关联。 Blaze-Persistence 突破了这一限制，允许跨实体关联—— 即 JOIN 两个没有映射关系的实体，并通过 ON 条件手动指定关联条件。

跨实体关联与 Lateral JOIN 架构
标准 JPA JOIN
只能 JOIN 已映射关联
Order →(ManyToOne)→ Customer
受限于实体映射
跨实体关联 (BP)
JOIN 无映射关系的实体
通过 ON 条件手动关联
Order ←ON→ AuditLog
灵活的跨表查询
Lateral JOIN
子查询可引用外层表的列
每行外层数据驱动一次子查询
类似 SQL 的 LATERAL / APPLY
强大的行级子查询
// 跨实体关联：Order 与 AuditLog 没有 JPA 映射
CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .leftJoinOn(AuditLog.class, "log")
        .on("log.entityId").eqExpression("o.id")
        .on("log.entityType").eq("ORDER")
    .end()
    .select("o.id")
    .select("log.action")
    .select("log.timestamp");

Lateral JOIN 支持矩阵

Lateral JOIN（也称 CROSS APPLY / OUTER APPLY）允许子查询引用外层查询的列。 Blaze-Persistence 在支持该特性的数据库上可生成对应 SQL。

数据库Lateral JOIN语法备注

Oracle ✅ 支持 LATERAL 12c+ 原生支持
SQL Server ✅ 支持 CROSS APPLY / OUTER APPLY 2005+ 支持
MySQL ✅ 支持 LATERAL 8.0.14+ 支持
H2 ⚠️ 有限 — 部分版本不支持，测试时注意

7.6 Entity View 中的 JOIN 控制

在 Entity View 层面，Blaze-Persistence 提供四种 FetchStrategy 来控制关联数据的加载方式。每种策略生成截然不同的 SQL 模式， 适用于不同的数据规模和性能需求。

四种 FetchStrategy — SQL 生成模式对比
JOIN 策略
单一查询 + JOIN
SELECT o.*, i.* FROM Order o LEFT JOIN Item i
✅ 最少查询次数
⚠️ 多集合有笛卡尔积风险
SELECT 策略
主查询 + N 次子查询
SELECT * FROM Item WHERE order_id = ? (× N)
✅ 无笛卡尔积
⚠️ N+1 查询
SUBSELECT 策略
主查询 + 1 次批量子查询
SELECT * FROM Item WHERE order_id IN (SELECT ...)
✅ 无笛卡尔积
✅ 只需 2 次查询
MULTISET 策略
单一查询 + 嵌套集合
SELECT o.*, (SELECT ... FROM Item) FROM Order
✅ 单次查询 · ✅ 无笛卡尔积
⚠️ 需数据库支持

代码示例与策略选择

@EntityView(Order.class)
public interface OrderView {

    @IdMapping
    Long getId();

    // JOIN 策略（默认）— 单一查询
    @Mapping(fetch = FetchStrategy.JOIN)
    CustomerView getCustomer();

    // SUBSELECT 策略 — 批量加载集合
    @Mapping(fetch = FetchStrategy.SUBSELECT)
    List<OrderItemView> getItems();

    // MULTISET 策略 — 嵌套集合，单次查询
    @Mapping(fetch = FetchStrategy.MULTISET)
    List<PaymentView> getPayments();
}
策略查询次数笛卡尔积适用场景
JOIN 1 ⚠️ 多集合时有风险 单一集合或 @ManyToOne 关联
SELECT N+1 ✅ 无 少量主记录时可接受
SUBSELECT 2 ✅ 无 大量主记录 + 集合关联
MULTISET 1 ✅ 无 多集合场景的最优解（需 DB 支持）
策略选择决策树
flowchart TD START["需要加载关联数据"] --> Q1{"关联类型?"} Q1 -->|"@ManyToOne / 单值"| A1["✅ JOIN 策略\n单次查询即可"] Q1 -->|"@OneToMany / 集合"| Q2{"有几个集合需要加载?"} Q2 -->|"1 个集合"| A2["✅ JOIN 策略\n无笛卡尔积风险"] Q2 -->|"多个集合"| Q3{"数据库支持 MULTISET?"} Q3 -->|"是"| A3["✅ MULTISET 策略\n单查询 + 无笛卡尔积"] Q3 -->|"否"| Q4{"主记录数量?"} Q4 -->|"少量 (<100)"| A4["SELECT 策略\n简单但有 N+1"] Q4 -->|"大量"| A5["✅ SUBSELECT 策略\n2 次查询 + 无笛卡尔积"] style START fill:#1e3a5f,stroke:#58a6ff style A1 fill:#1a3a2a,stroke:#3fb950 style A2 fill:#1a3a2a,stroke:#3fb950 style A3 fill:#2d1f4e,stroke:#bc8cff style A4 fill:#3a2a1a,stroke:#d29922 style A5 fill:#1a3a2a,stroke:#3fb950
📖 参考：@MappingFetch 注解允许在 Entity View 的集合映射上 指定 FetchStrategy，覆盖默认行为。结合 @BatchSize 可进一步优化 SELECT 策略的批量加载粒度。

7.7 陷阱与限制

JOIN 是性能调优中最需要谨慎的部分。以下列出生产环境中最常见的陷阱。

JOIN 常见陷阱全景图
陷阱 1: 笛卡尔积
多集合 FETCH JOIN
结果集行数爆炸 3×2=6 行
解决: SUBSELECT / MULTISET
陷阱 2: 隐式 JOIN 去重
相同路径自动去重
不同路径到同一实体 → 多个 JOIN
解决: 使用显式 JOIN + 别名
陷阱 3: FETCH + 分页
Hibernate 警告 HHH000104
内存中分页! 全量加载后裁剪
解决: 先分页 ID, 再 FETCH JOIN
陷阱 4: 深层 JOIN 链
o.a.b.c.d.e.f — 6 层 JOIN
SQL 复杂度指数增长
解决: 子查询 / Entity View 拆分

陷阱 1: 多集合 FETCH JOIN 的笛卡尔积

// ❌ 危险: 同时 FETCH 两个集合
CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
    .from(Order.class, "o")
    .fetch("o.items")       // 集合 1
    .fetch("o.payments");   // 集合 2 → 笛卡尔积!

// ✅ 安全: 使用 SUBSELECT 策略
@EntityView(Order.class)
public interface SafeOrderView {
    @Mapping(fetch = FetchStrategy.JOIN)
    List<OrderItemView> getItems();        // 只有一个用 JOIN

    @Mapping(fetch = FetchStrategy.SUBSELECT)
    List<PaymentView> getPayments();       // 另一个用 SUBSELECT
}

陷阱 2: FETCH JOIN 与分页冲突

Hibernate 内存分页：当 FETCH JOIN 集合与 setFirstResult / setMaxResults 同时使用时，Hibernate 会发出警告 HHH000104， 并在内存中执行分页（先加载全部数据再裁剪）。这在大数据量下会导致 OOM。 
// ❌ 危险组合
CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
    .from(Order.class, "o")
    .fetch("o.items")
    .setFirstResult(0)
    .setMaxResults(20);  // Hibernate 内存分页!

// ✅ 安全: 两阶段查询
// 阶段 1: 分页取 ID
List<Long> ids = cbf.create(em, Long.class)
    .from(Order.class, "o")
    .select("o.id")
    .setFirstResult(0)
    .setMaxResults(20)
    .getResultList();

// 阶段 2: 用 ID 批量 FETCH
CriteriaBuilder<Order> cb = cbf.create(em, Order.class)
    .from(Order.class, "o")
    .fetch("o.items")
    .where("o.id").in(ids);

📝 本章小结

  • 隐式 vs 显式 JOIN：路径表达式触发隐式 LEFT JOIN，显式 API 提供完整的类型和别名控制
  • 四种 JOIN 类型：INNER / LEFT / RIGHT / FULL 各有语义，joinOn() 支持自定义 ON 条件
  • FETCH JOIN：通过 fetch() 将 N+1 查询合并为单次查询，但需警惕多集合的笛卡尔积
  • 路径表达式：点号导航自动解析为 JOIN 链，Embeddable 段不产生额外 JOIN
  • 跨实体关联：Blaze-Persistence 突破 JPA 映射限制，支持无关联实体的 JOIN 和 Lateral JOIN
  • FetchStrategy：JOIN / SELECT / SUBSELECT / MULTISET 四策略适配不同场景，MULTISET 是多集合最优解
  • 安全清单：避免多集合 FETCH + 分页组合，限制 JOIN 深度，注意 Provider 行为差异
