18.1 Quarkus 扩展架构

18.2 构建时优化 vs 运行时初始化

传统的 Spring Boot 集成（第 12 章）在应用启动时才执行 Entity View 的扫描、校验和代理生成。Quarkus 将这些步骤全部前置到编译阶段，带来数量级的启动加速。

Fig 18-3 Spring Boot vs Quarkus 启动流程对比

❷ 加载 Spring Context
❸ 扫描 @EntityView 注解
❹ 校验映射合法性
❺ 生成动态代理 (字节码生成)
❻ 创建 EntityViewManager
❼ 应用就绪 ✅
Quarkus — 构建时优化

反射配置

Quarkus 扩展在构建时自动生成 reflect-config.json，将所有 Entity View 接口及其方法注册为可反射访问。手动配置示例：

// reflect-config.json (由 Quarkus 扩展自动生成)
[
  {
    "name": "com.example.view.CatView",
    "allDeclaredMethods": true,
    "allDeclaredConstructors": true
  },
  {
    "name": "com.example.view.CatCreateView",
    "allDeclaredMethods": true,
    "allDeclaredConstructors": true
  }
]
⚠️ 自定义 View：如果 Entity View 使用了自定义的 @PostCreate、@PostLoad 回调方法，这些方法也必须注册到反射配置中，否则原生镜像运行时会抛出 IllegalAccessError。

18.4 静态 Entity View 实现

生成的实现类示例

// 由 APT 自动生成 — CatView_Impl.java
@Generated("com.blazebit.persistence.view.processor")
public class CatView_Impl implements CatView {

    private final Long id;
    private final String name;
    private final Integer age;

    public CatView_Impl(Long id, String name, Integer age) {
        this.id = id;
        this.name = name;
        this.age = age;
    }

    @Override
    public Long getId() { return id; }

    @Override
    public String getName() { return name; }

    @Override
    public Integer getAge() { return age; }
}
💡 零反射运行时：静态实现类是普通的 Java 类，所有 getter 方法都是直接方法调用——不经过反射、不经过代理。这不仅满足 GraalVM 的要求，同时也带来微小的运行时性能提升。

服务层注入示例

@ApplicationScoped
public class CatService {

    @Inject
    CriteriaBuilderFactory cbf;

    @Inject
    EntityViewManager evm;

    @Inject
    EntityManager em;

    public List<CatListView> findAll() {
        CriteriaBuilder<Cat> cb = cbf.create(em, Cat.class);
        return evm.applySetting(
            EntityViewSetting.create(CatListView.class), cb
        ).getResultList();
    }

    @Transactional
    public void create(CatCreateView cat) {
        evm.save(em, cat);
    }
}

18.7 陷阱与限制

原生编译为 Blaze-Persistence 带来了极致的启动性能，但也引入了若干限制和注意事项。以下是生产环境中最常遇到的问题。

Fig 18-13 原生编译限制分类
功能限制
• 部分 @Mapping 表达式不支持
• 动态 View 创建不可用
构建限制
• 编译时间显著增加
• 编译内存需 8GB+
运行时限制
• 无法运行时新增 Entity View
• 调试困难，堆栈不完整
兼容性限制
• 仅支持 Hibernate
• Hibernate 版本需 6.2+

功能限制详解

功能JVM 模式原生模式说明
基本 Entity View✅✅通过静态实现完全支持
@UpdatableEntityView✅✅需要 APT 生成可变实现
@CreatableEntityView✅✅需要 APT 生成构建器
MULTISET Fetch✅✅完全支持
CTE 查询✅⚠️需确保 CTE 实体注册到反射配置
动态 EntityViewSetting✅⚠️运行时动态添加排序/过滤受限
自定义 TypeConverter✅⚠️需手动注册到反射配置
运行时注册新 View✅❌原生镜像封闭世界假设

📝 本章小结

  • Quarkus 扩展通过 @BuildStep 处理器将 Entity View 的扫描、校验和代理生成全部移至构建阶段，实现运行时零初始化开销。
  • 构建时优化让 Quarkus JVM 模式启动时间降至 ~0.7s（对比 Spring Boot ~3.5s），原生模式更可压至 ~30ms。
  • GraalVM 原生镜像面临反射、动态代理、ServiceLoader 三大挑战，Quarkus 扩展通过自动生成反射配置和预解析 SPI 逐一解决。
  • 静态 Entity View 实现是原生编译的核心方案——APT 注解处理器在编译期生成完整的 Java 源码，彻底消除运行时字节码生成需求。
  • 原生测试分两层：@QuarkusTest 进行 JVM 模式白盒测试，@QuarkusIntegrationTest 对原生可执行文件执行黑盒验证。
  • CDI 集成完全自动化——CriteriaBuilderFactory 和 EntityViewManager 由扩展自动注册为 CDI Bean，支持多持久化单元场景。
  • 已知限制包括：仅支持 Hibernate、运行时无法新增 Entity View、部分复杂 @Mapping 表达式需额外配置。
