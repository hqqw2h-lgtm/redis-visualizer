23.1 动态条件查询 — EntityViewSpecificationExecutor

电商后台最常见的需求——运营人员在订单管理页面输入多种筛选条件（状态、客户、金额范围、日期区间），后端需要动态拼接查询条件。Blaze-Persistence 提供两种方式：JPA Specification 整合和原生 CriteriaBuilder 流式拼接。

搜索请求 DTO

public record OrderSearchRequest(
    OrderStatus status,
    Long customerId,
    BigDecimal minAmount,
    BigDecimal maxAmount,
    LocalDateTime dateFrom,
    LocalDateTime dateTo,
    String keyword
) {}

方式一：Specification + EntityViewSpecificationExecutor

Spring Data 集成模块提供了 EntityViewSpecificationExecutor 接口，让 Repository 直接接受 Specification 参数并返回 Entity View 投影结果。

public interface OrderRepository
        extends EntityViewRepository<OrderListView, Long>,
                EntityViewSpecificationExecutor<OrderListView, Order> {
}

@Service
@RequiredArgsConstructor
public class OrderSearchService {

    private final OrderRepository orderRepo;

    public Page<OrderListView> searchOrders(OrderSearchRequest req,
                                             Pageable pageable) {
        return orderRepo.findAll(
            (root, query, cb) -> {
                var predicates = new ArrayList<Predicate>();

                if (req.status() != null)
                    predicates.add(
                        cb.equal(root.get("status"), req.status()));

                if (req.customerId() != null)
                    predicates.add(
                        cb.equal(root.get("customer").get("id"),
                                 req.customerId()));

                if (req.minAmount() != null)
                    predicates.add(
                        cb.greaterThanOrEqualTo(
                            root.get("totalAmount"), req.minAmount()));

                if (req.maxAmount() != null)
                    predicates.add(
                        cb.lessThanOrEqualTo(
                            root.get("totalAmount"), req.maxAmount()));

                if (req.dateFrom() != null)
                    predicates.add(
                        cb.greaterThanOrEqualTo(
                            root.get("createdAt"), req.dateFrom()));

                if (req.dateTo() != null)
                    predicates.add(
                        cb.lessThanOrEqualTo(
                            root.get("createdAt"), req.dateTo()));

                return cb.and(predicates.toArray(new Predicate[0]));
            },
            pageable
        );
    }
}

-- 当 status=PAID, minAmount=100, dateFrom=2026-01-01 时生成的 SQL:
SELECT o.id, o.status, o.total_amount, c.name AS customer_name,
       o.created_at
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'PAID'
  AND o.total_amount >= 100
  AND o.created_at >= '2026-01-01'
ORDER BY o.created_at DESC
LIMIT 20 OFFSET 0

方式二：CriteriaBuilder 流式拼接

对于需要更细粒度控制的场景（如动态排序、跨表联合条件），直接用 Blaze-Persistence 的 CriteriaBuilder 提供更大灵活性。

@Service
@RequiredArgsConstructor
public class OrderCriteriaSearchService {

    private final CriteriaBuilderFactory cbf;
    private final EntityViewManager evm;
    private final EntityManager em;

    public List<OrderListView> searchWithCriteria(OrderSearchRequest req) {
        CriteriaBuilder<Order> cb = cbf.create(em, Order.class, "o");

        if (req.status() != null)
            cb.where("o.status").eq(req.status());

        if (req.customerId() != null)
            cb.where("o.customer.id").eq(req.customerId());

        if (req.minAmount() != null)
            cb.where("o.totalAmount").ge(req.minAmount());

        if (req.maxAmount() != null)
            cb.where("o.totalAmount").le(req.maxAmount());

        if (req.dateFrom() != null)
            cb.where("o.createdAt").ge(req.dateFrom());

        if (req.dateTo() != null)
            cb.where("o.createdAt").le(req.dateTo());

        if (req.keyword() != null && !req.keyword().isBlank())
            cb.where("o.customer.name").like()
              .value("%" + req.keyword() + "%").noEscape();

        return evm.applySetting(
            EntityViewSetting.create(OrderListView.class),
            cb
        ).getResultList();
    }
}

23.2 聚合统计 — 仪表盘

产品经理需要一个运营仪表盘：按月统计订单数量、总收入、平均客单价、去重客户数。传统做法需要写原生 SQL 或复杂 JPQL；用 Entity View 的 @Mapping 聚合表达式，可以在编译期就定义好所有聚合逻辑。

聚合 Entity View 定义

@EntityView(Order.class)
public interface MonthlySalesView {

    @Mapping("FUNCTION('EXTRACT_MONTH', createdAt)")
    Integer getMonth();

    @Mapping("FUNCTION('EXTRACT_YEAR', createdAt)")
    Integer getYear();

    @Mapping("COUNT(id)")
    Long getOrderCount();

    @Mapping("SUM(totalAmount)")
    BigDecimal getTotalRevenue();

    @Mapping("AVG(totalAmount)")
    BigDecimal getAvgOrderValue();

    @Mapping("COUNT(DISTINCT customer.id)")
    Long getUniqueCustomers();
}

@Repository
public interface SalesReportRepository
        extends EntityViewRepository<MonthlySalesView, Long> {
}

@Service
@RequiredArgsConstructor
public class DashboardService {

    private final CriteriaBuilderFactory cbf;
    private final EntityViewManager evm;
    private final EntityManager em;

    public List<MonthlySalesView> getMonthlySales(int year) {
        CriteriaBuilder<Order> cb = cbf.create(em, Order.class, "o")
            .where("o.status").notEq(OrderStatus.CANCELLED)
            .where("FUNCTION('EXTRACT_YEAR', o.createdAt)").eq(year);

        return evm.applySetting(
            EntityViewSetting.create(MonthlySalesView.class),
            cb
        ).getResultList();
    }
}

Tuple 方式——更灵活的聚合查询

public List<Tuple> getMonthlySalesTuples(int year) {
    CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
        .from(Order.class, "o")
        .select("FUNCTION('EXTRACT_MONTH', o.createdAt)", "month")
        .select("COUNT(o.id)", "orderCount")
        .select("SUM(o.totalAmount)", "totalRevenue")
        .select("AVG(o.totalAmount)", "avgOrderValue")
        .select("COUNT(DISTINCT o.customer.id)", "uniqueCustomers")
        .where("o.status").notEq(OrderStatus.CANCELLED)
        .where("FUNCTION('EXTRACT_YEAR', o.createdAt)").eq(year)
        .groupBy("FUNCTION('EXTRACT_MONTH', o.createdAt)")
        .orderByAsc("FUNCTION('EXTRACT_MONTH', o.createdAt)");

    return cb.getResultList();
}
Entity View vs Tuple 聚合的选择：如果聚合字段是固定的（仪表盘），优先用 Entity View——类型安全、接口稳定、可测试。如果聚合字段需要动态变化（用户自选指标），用 Tuple 更灵活。

多维度聚合——按状态 + 月份交叉统计

public interface StatusMonthlySalesView {

23.3 窗口函数 — 排名与趋势

窗口函数是 SQL 分析的杀手级特性——在不改变行数的前提下，为每行计算排名、累计和、移动平均等聚合值。Blaze-Persistence 在 JPQL 层面提供了完整的窗口函数支持。

客户消费排名

public List<Tuple> getCustomerSpendingRank() {
    CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
        .from(Order.class, "o")
        .select("o.customer.id", "customerId")
        .select("o.customer.name", "customerName")
        .select("SUM(o.totalAmount)", "totalSpent")
        .select("COUNT(o.id)", "orderCount")
        .select("WINDOW_RANK() OVER ("
              + "ORDER BY SUM(o.totalAmount) DESC)", "spendingRank")
        .select("WINDOW_DENSE_RANK() OVER ("
              + "ORDER BY SUM(o.totalAmount) DESC)", "denseRank")
        .where("o.status").eq(OrderStatus.COMPLETED)
        .groupBy("o.customer.id")
        .groupBy("o.customer.name")
        .orderByAsc("WINDOW_RANK() OVER ("
                  + "ORDER BY SUM(o.totalAmount) DESC)");

    return cb.getResultList();
}

-- 生成的 SQL:
SELECT o.customer_id,
       c.name,
       SUM(o.total_amount) AS total_spent,
       COUNT(o.id) AS order_count,
       RANK() OVER (ORDER BY SUM(o.total_amount) DESC) AS spending_rank,
       DENSE_RANK() OVER (ORDER BY SUM(o.total_amount) DESC) AS dense_rank
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'COMPLETED'
GROUP BY o.customer_id, c.name
ORDER BY spending_rank

月度销售趋势——累计与环比

public List<Tuple> getMonthlySalesTrend(int year) {
    CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
        .from(Order.class, "o")
        .select("FUNCTION('EXTRACT_MONTH', o.createdAt)", "month")
        .select("SUM(o.totalAmount)", "monthlyRevenue")
        .select("WINDOW_SUM(SUM(o.totalAmount)) OVER ("
              + "ORDER BY FUNCTION('EXTRACT_MONTH', o.createdAt) "
              + "ROWS UNBOUNDED PRECEDING)", "cumulativeRevenue")
        .select("WINDOW_LAG(SUM(o.totalAmount), 1) OVER ("
              + "ORDER BY FUNCTION('EXTRACT_MONTH', o.createdAt))",
              "prevMonthRevenue")
        .where("o.status").notEq(OrderStatus.CANCELLED)
        .where("FUNCTION('EXTRACT_YEAR', o.createdAt)").eq(year)
        .groupBy("FUNCTION('EXTRACT_MONTH', o.createdAt)")
        .orderByAsc("FUNCTION('EXTRACT_MONTH', o.createdAt)");

    return cb.getResultList();
}

-- 生成的 SQL（一条语句同时返回月度、累计、环比）:
SELECT EXTRACT(MONTH FROM o.created_at) AS month,
       SUM(o.total_amount) AS monthly_revenue,
       SUM(SUM(o.total_amount)) OVER (
           ORDER BY EXTRACT(MONTH FROM o.created_at)
           ROWS UNBOUNDED PRECEDING
       ) AS cumulative_revenue,
       LAG(SUM(o.total_amount), 1) OVER (
           ORDER BY EXTRACT(MONTH FROM o.created_at)
       ) AS prev_month_revenue
FROM orders o
WHERE o.status <> 'CANCELLED'
  AND EXTRACT(YEAR FROM o.created_at) = 2026
GROUP BY EXTRACT(MONTH FROM o.created_at)
ORDER BY month

23.4 CTE — 月度销售排名报表

CTE（公用表表达式）将复杂查询分解为命名步骤。在报表场景中，先用 CTE 计算每位客户的月度消费总额和排名，再从 CTE 结果中筛选 Top 10——比嵌套子查询可读性高一个量级。

CTE 实体定义

@CTE
@Entity
public class MonthlySalesCTE {
    @Id Long customerId;
    String customerName;
    Integer salesMonth;
    BigDecimal monthlyTotal;
    Long orderCount;
    Integer salesRank;
}
CTE 实体必须满足三个条件：① 标注 @CTE 和 @Entity；② 必须有 @Id 字段；③ 必须在 persistence.xml 或 @EntityScan 中注册。缺少任何一项都会导致运行时异常。 
@Service
@RequiredArgsConstructor
public class SalesRankingService {

    private final CriteriaBuilderFactory cbf;
    private final EntityManager em;

    public List<Tuple> getTopCustomersByMonth(int year, int month,
                                               int topN) {
        CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
            .with(MonthlySalesCTE.class)
                .from(Order.class, "o")
                .bind("customerId").select("o.customer.id")
                .bind("customerName").select("o.customer.name")
                .bind("salesMonth")
                    .select("FUNCTION('EXTRACT_MONTH', o.createdAt)")
                .bind("monthlyTotal").select("SUM(o.totalAmount)")
                .bind("orderCount").select("COUNT(o.id)")
                .bind("salesRank")
                    .select("WINDOW_RANK() OVER ("
                          + "PARTITION BY "
                          + "FUNCTION('EXTRACT_MONTH', o.createdAt) "
                          + "ORDER BY SUM(o.totalAmount) DESC)")
                .where("o.status").eq(OrderStatus.COMPLETED)
                .where("FUNCTION('EXTRACT_YEAR', o.createdAt)").eq(year)
                .groupBy("o.customer.id")
                .groupBy("o.customer.name")
                .groupBy("FUNCTION('EXTRACT_MONTH', o.createdAt)")
            .end()
            .from(MonthlySalesCTE.class, "c")
            .select("c.customerName")
            .select("c.salesMonth")
            .select("c.monthlyTotal")
            .select("c.orderCount")
            .select("c.salesRank")
            .where("c.salesMonth").eq(month)
            .where("c.salesRank").le(topN)
            .orderByAsc("c.salesRank");

        return cb.getResultList();
    }
}

-- 生成的 SQL:
WITH MonthlySalesCTE(customer_id, customer_name, sales_month,
                     monthly_total, order_count, sales_rank) AS (
    SELECT o.customer_id,
           c.name,
           EXTRACT(MONTH FROM o.created_at),
           SUM(o.total_amount),
           COUNT(o.id),
           RANK() OVER (
               PARTITION BY EXTRACT(MONTH FROM o.created_at)
               ORDER BY SUM(o.total_amount) DESC
           )
    FROM orders o
    JOIN customers c ON o.customer_id = c.id
    WHERE o.status = 'COMPLETED'
      AND EXTRACT(YEAR FROM o.created_at) = 2026
    GROUP BY o.customer_id, c.name,
             EXTRACT(MONTH FROM o.created_at)
)
SELECT c.customer_name, c.sales_month,
       c.monthly_total, c.order_count, c.sales_rank
FROM MonthlySalesCTE c
WHERE c.sales_month = 3
  AND c.sales_rank <= 10
ORDER BY c.sales_rank

CTE vs 子查询：CTE 的优势不仅是可读性——PostgreSQL 12+ 的优化器会自动内联（inline）非递归 CTE，生成与子查询等价的执行计划。但递归 CTE 不会被内联，这是设计上的必然选择。

23.5 递归 CTE — 分类树

电商系统的商品分类是典型的树形结构——"电子产品 → 手机 → 智能手机 → 折叠屏手机"可能有 4-6 层深度。用递归 CTE 可以在一条 SQL 中展开任意深度的分类树，无需应用层递归查询。

分类实体

@Entity
@Table(name = "categories")
public class Category {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private String name;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "parent_id")
    private Category parent;

    @OneToMany(mappedBy = "parent")
    private List<Category> children;

    private Integer sortOrder;
    private Boolean active;
}

递归 CTE 实体

@CTE
@Entity

递归 CTE 查询实现

@Service
@RequiredArgsConstructor
public class CategoryTreeService {

    private final CriteriaBuilderFactory cbf;
    private final EntityManager em;

    public List<Tuple> getCategoryTree(Long rootId) {
        CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
            .withRecursive(CategoryTreeCTE.class)
                // Anchor: 根节点
                .from(Category.class, "c")
                .bind("id").select("c.id")
                .bind("name").select("c.name")
                .bind("parentId").select("c.parent.id")
                .bind("level").select("0")
                .bind("path").select("c.name")
                .where("c.id").eq(rootId)
            .unionAll()
                // Recursive: 子节点递归
                .from(Category.class, "child")
                .innerJoinOn(CategoryTreeCTE.class, "parent")
                    .on("parent.id").eqExpression("child.parent.id")
                .end()
                .bind("id").select("child.id")
                .bind("name").select("child.name")
                .bind("parentId").select("child.parent.id")
                .bind("level").select("parent.level + 1")
                .bind("path")
                    .select("CONCAT(parent.path, ' → ', child.name)")
            .end()
            .from(CategoryTreeCTE.class, "t")
            .select("t.id")
            .select("t.name")
            .select("t.parentId")
            .select("t.level")
            .select("t.path")
            .orderByAsc("t.path");

        return cb.getResultList();
    }
}

-- 生成的 SQL:
WITH RECURSIVE CategoryTreeCTE(id, name, parent_id, level, path) AS (
    -- Anchor: 根节点
    SELECT c.id, c.name, c.parent_id, 0, CAST(c.name AS VARCHAR(1000))
    FROM categories c
    WHERE c.id = 1

    UNION ALL

    -- Recursive: 每层子节点
    SELECT child.id, child.name, child.parent_id,
           parent.level + 1,
           CONCAT(parent.path, ' → ', child.name)
    FROM categories child
    INNER JOIN CategoryTreeCTE parent
        ON parent.id = child.parent_id
)
SELECT t.id, t.name, t.parent_id, t.level, t.path
FROM CategoryTreeCTE t
ORDER BY t.path

public List<Tuple> getCategoryTreeWithProductCount(Long rootId) {

        .withRecursive(CategoryTreeCTE.class)
            .from(Category.class, "c")
            .bind("id").select("c.id")
            .bind("name").select("c.name")

        .unionAll()
            .from(Category.class, "child")
            .innerJoinOn(CategoryTreeCTE.class, "parent")
                .on("parent.id").eqExpression("child.parent.id")
            .end()
            .bind("id").select("child.id")
            .bind("name").select("child.name")
            .bind("parentId").select("child.parent.id")
            .bind("level").select("parent.level + 1")
            .bind("path")
                .select("CONCAT(parent.path, ' → ', child.name)")

        .from(CategoryTreeCTE.class, "t")
        .leftJoinOn(Product.class, "p")
            .on("p.category.id").eqExpression("t.id")
        .end()
        .select("t.id")

        .select("t.level")

23.9 Correlated Subquery in Entity View

在 Entity View 中通过 @MappingCorrelatedSimple 声明关联子查询，可以为每个实体计算汇总统计字段——如"该客户的总订单数"、"该商品的平均评分"——而无需在 Service 层手动拼接。

客户 View 带关联统计

@EntityView(Customer.class)
public interface CustomerDetailView {
    @IdMapping Long getId();
    String getName();
    String getEmail();

    @MappingCorrelatedSimple(
        correlated = Order.class,
        correlationBasis = "this",
        correlationExpression =
            "correlationKey = customer.id",
        correlationResult = "COUNT(id)"
    )
    Long getTotalOrders();

    @MappingCorrelatedSimple(
        correlated = Order.class,
        correlationBasis = "this",
        correlationExpression =
            "correlationKey = customer.id",
        correlationResult = "SUM(totalAmount)"
    )
    BigDecimal getTotalSpent();

    @MappingCorrelatedSimple(
        correlated = Order.class,
        correlationBasis = "this",
        correlationExpression =
            "correlationKey = customer.id",
        correlationResult = "AVG(totalAmount)"
    )
    BigDecimal getAvgOrderValue();

    @MappingCorrelatedSimple(
        correlated = Order.class,
        correlationBasis = "this",
        correlationExpression =
            "correlationKey = customer.id",
        correlationResult = "MAX(createdAt)"
    )
    LocalDateTime getLastOrderDate();
}

性能注意：每个 @MappingCorrelatedSimple 字段都会生成一个 Correlated Subquery。如果一个 View 有 5 个关联统计字段且查询返回 100 行，数据库需要执行 500 次子查询。对于列表场景，考虑将统计字段拆分到单独的 View（详情页使用），或改用 @BatchFetch 策略减少数据库往返。

陷阱 1：CTE 实体缺少 @Id

// ❌ 错误：CTE 实体没有 @Id → 运行时异常
@CTE @Entity

    String name;

// IllegalArgumentException: Entity type 'BrokenCTE'

// ✅ 正确：必须有 @Id
@CTE @Entity

    @Id Long id;  // 即使 CTE 不需要主键概念，JPA 要求必须有
    String name;
    BigDecimal total;

陷阱 3：窗口函数 JPQL 语法与标准 SQL 差异

// ❌ 错误：直接用 SQL 语法写窗口函数
cb.select("RANK() OVER (ORDER BY SUM(o.totalAmount) DESC)");
// Blaze-Persistence 无法解析！

// ✅ 正确：用 WINDOW_ 前缀
cb.select("WINDOW_RANK() OVER ("
        + "ORDER BY SUM(o.totalAmount) DESC)");

// 常用窗口函数前缀映射:
// SQL RANK()       → WINDOW_RANK()
// SQL DENSE_RANK() → WINDOW_DENSE_RANK()
// SQL ROW_NUMBER() → WINDOW_ROW_NUMBER()
// SQL SUM()        → WINDOW_SUM()
// SQL LAG()        → WINDOW_LAG()
// SQL LEAD()       → WINDOW_LEAD()
// SQL FIRST_VALUE()→ WINDOW_FIRST_VALUE()
// SQL LAST_VALUE() → WINDOW_LAST_VALUE()
// SQL NTILE()      → WINDOW_NTILE()

陷阱 5：隐式 JOIN 性能问题

// ❌ 性能陷阱：多次引用隐式路径 → 多次 JOIN
cb.select("o.customer.name")
  .select("o.customer.email")
  .select("o.customer.phone")
  .where("o.customer.status").eq("ACTIVE");
// 某些 JPA 实现可能生成 4 次 JOIN customer 表！

// ✅ 正确：显式 JOIN 一次
cb.innerJoin("o.customer", "c")
  .select("c.name")
  .select("c.email")
  .select("c.phone")
  .where("c.status").eq("ACTIVE");
// → 只有 1 次 JOIN

-- ❌ 可能生成的低效 SQL:
SELECT c1.name, c2.email, c3.phone
FROM orders o
JOIN customers c1 ON o.customer_id = c1.id
JOIN customers c2 ON o.customer_id = c2.id
JOIN customers c3 ON o.customer_id = c3.id
WHERE c4.status = 'ACTIVE'

-- ✅ 正确的单次 JOIN:
SELECT c.name, c.email, c.phone
FROM orders o
INNER JOIN customers c ON o.customer_id = c.id
WHERE c.status = 'ACTIVE'

陷阱 7：聚合 View 缺少 GROUP BY 隐式行为

// ⚠️ 注意：Entity View 的聚合字段会自动推导 GROUP BY
// 但非聚合字段必须全部出现在 GROUP BY 中

@EntityView(Order.class)
public interface BrokenAggView {
    @Mapping("status")
    OrderStatus getStatus();        // 非聚合 → GROUP BY

    @Mapping("customer.name")

本章小结

  • 动态条件查询有两条路径：Specification 集成 Spring Data 分页排序、CriteriaBuilder 支持 CTE/窗口函数等高级表达式——按团队技术栈和查询复杂度选择
  • 聚合 Entity View 通过 @Mapping("COUNT(id)") / SUM / AVG 在接口层声明聚合逻辑——类型安全、可测试，一条 SQL 返回完整仪表盘数据
  • 窗口函数在不改变行数的前提下计算排名（WINDOW_RANK）、累计（WINDOW_SUM）、环比（WINDOW_LAG）——单条 SQL 同时返回多个维度
  • CTE 将复杂查询分解为命名步骤——先计算排名，再过滤 Top N，可读性和性能兼优
  • 递归 CTE 在一条 SQL 中展开任意深度的分类树——Anchor 定义根节点，UNION ALL 递归展开子节点，path 字段跟踪完整路径
  • JpqlFunction 在 SQL 渲染阶段将 FUNCTION('pg_search', ...) 翻译为 to_tsvector() @@ to_tsquery()——集成数据库特有功能的标准方式
  • JpqlMacro 在 JPQL 解析阶段展开——MACRO(CURRENT_USER_NAME) 变为当前用户字面量，适合审计和多租户上下文注入
  • Entity View 继承通过 @EntityViewInheritanceMapping 按条件自动分派子 View——PENDING、SHIPPED、COMPLETED 各显示不同字段
  • Correlated Subquery 通过 @MappingCorrelatedSimple 为每行计算关联统计——CorrelationProvider 支持更复杂的关联条件
  • 七大陷阱：CTE 需 @Id、CTE 需注册、窗口函数用 WINDOW_ 前缀、Function/Macro 按时机选、显式 JOIN 避免重复、继承条件需互斥、聚合字段影响 GROUP BY 粒度
