第 0 章 · Quick Start：5 分钟跑通你的第一个 Blaze-Persistence 应用

从 zero 到 Entity View DTO · CriteriaBuilder 第一条查询 · SQL 日志解读 · N+1 自检 · 三种集成方式 · 出问题诊断手册

本章是全书唯一的"动手"章节——读完就能跑起来一个用 Blaze-Persistence 查 JPA Entity + 投影成 DTO 的程序。从此后每章原理讲解都建立在「你曾亲手运行过 cbf.create(em, Order.class).where(...).getResultList()」这个共同基础上。

如果你已经熟悉 JPA Criteria 但没用过 Blaze-Persistence，强烈建议至少读完 0.2 第一个 Entity View——那里会用一段对比代码让你看到「Entity View 到底解决了什么 JPA Criteria 解决不了的问题」。

💡 为什么这么多模块？ Blaze 把"API"和"实现"分开：业务代码只 import API（接口稳定），运行时才用 impl 模块。这样未来切换 EclipseLink/DataNucleus 只换 blaze-persistence-integration-* 模块即可。
💡 为什么 jakarta 后缀？Jakarta EE 9+ 的包名从 javax.persistence 迁到 jakarta.persistence，Blaze 提供两套 artifact。Spring Boot 3.x / Hibernate 6+ 一律用 jakarta。

配 CBF + EVM

新建一个 Entity（最简单的 1 对多）和一个启动类：

// src/main/java/com/example/Order.java
@Entity
public class Order {
    @Id @GeneratedValue Long id;
    String customerName;
    Instant placedAt;
    @OneToMany(mappedBy = "order", cascade = CascadeType.ALL)
    List<OrderLine> lines = new ArrayList<>();
}

@Entity
public class OrderLine {
    @Id @GeneratedValue Long id;
    @ManyToOne Order order;
    String product;
    BigDecimal price;
}
// src/main/java/com/example/Main.java
import com.blazebit.persistence.*;
import com.blazebit.persistence.view.*;
import com.blazebit.persistence.spi.*;

public class Main {
    public static void main(String[] args) {
        EntityManagerFactory emf = Persistence.createEntityManagerFactory("demo");

        // ★ 第一步：构建 CriteriaBuilderFactory（CBF）
        CriteriaBuilderConfiguration cbc = Criteria.getDefault();
        CriteriaBuilderFactory cbf = cbc.createCriteriaBuilderFactory(emf);

        // ★ 第二步：构建 EntityViewManager（EVM）
        EntityViewConfiguration evc = EntityViews.createDefaultConfiguration();
        evc.addEntityView(OrderSummaryView.class);   // ← 0.2 节定义
        EntityViewManager evm = evc.createEntityViewManager(cbf);

        try (EntityManager em = emf.createEntityManager()) {
            em.getTransaction().begin();
            // 准备数据
            Order o = new Order();
            o.customerName = "Alice";
            o.placedAt = Instant.now();
            o.lines.add(line(o, "Coffee", new BigDecimal("3.50")));
            o.lines.add(line(o, "Bagel",  new BigDecimal("2.00")));
            em.persist(o);
            em.getTransaction().commit();

            // ★ 第三步：用 CriteriaBuilder 查
            List<Order> results = cbf.create(em, Order.class)
                                     .where("customerName").eq("Alice")
                                     .getResultList();
            results.forEach(r -> System.out.println(r.id + " " + r.customerName));
        }
    }
}

运行结果

$ mvn compile exec:java -Dexec.mainClass=com.example.Main
1 Alice
💡 这条 Blaze CriteriaBuilder 跑了什么 SQL？ 
SELECT order0_.id, order0_.customer_name, order0_.placed_at FROM orders order0_ WHERE order0_.customer_name = ?
对比原生 JPA Criteria 等价代码（10+ 行 CriteriaQuery.from()/where(builder.equal(...))），Blaze 用 1 个 fluent 调用表达了同样的语义。原理在 第 3 章 · CriteriaBuilder 查询引擎。

0.2 第一个 Entity View（Blaze 的精华）

纯 CriteriaBuilder 已经比原生 JPA 好用，但 Blaze 真正的杀手锏是 Entity View——它解决了 JPA 一直没解决的"DTO 投影"问题。

声明视图

package com.example;
import com.blazebit.persistence.view.*;

@EntityView(Order.class)
public interface OrderSummaryView {

    @IdMapping
    Long getId();

    String getCustomerName();

    // ★ 自动 SUM 聚合，跑 GROUP BY
    @Mapping("SIZE(lines)")
    int getLineCount();

    @Mapping("SUM(lines.price)")
    BigDecimal getTotalAmount();
}

查询并解读 SQL

List<OrderSummaryView> views =
    evm.applySetting(EntityViewSetting.create(OrderSummaryView.class),
                     cbf.create(em, Order.class))
       .getResultList();

views.forEach(v -> System.out.printf("Order #%d: %s, %d lines, total=$%s%n",
    v.getId(), v.getCustomerName(), v.getLineCount(), v.getTotalAmount()));
// 输出：Order #1: Alice, 2 lines, total=$5.50
▎图 0-2 · Entity View vs JPA 原生 vs CriteriaBuilder 三种写法对比
维度 原生 JPA Criteria Blaze CriteriaBuilder Blaze Entity View
代码行数15+ 行 builder 链3-5 行 fluent1 行 + 声明式接口
类型安全部分（Path<T>）字符串路径，编译期 弱接口 getter，编译期 强
SELECT 列数全字段指定才有自动按需投影
N+1需手动 fetch join需手动 fetch join自动消除
聚合（SUM/COUNT）手写 GROUP BY手写 GROUP BY@Mapping("SUM(...)") 一行
💡 为什么叫 "Entity View"？ 类比数据库视图——Entity View 是 Entity 的"投影 + 聚合 + 派生属性"声明式定义。Blaze 把这个 view 翻译成最优 SQL（最小列集 + 自动 join + 集合 fetch）。这就是它能干掉 N+1 的根本原因。

0.4 SQL 日志解读 + N+1 自检

开发期 90% 的 Entity View 调试是「我声明了这个 view，跑出来的 SQL 长啥样？是不是 N+1？」。最简单的方式——开 Hibernate 的 show_sql。

跑 0.2 节那段查询，控制台输出：

Hibernate:
    select                                                         ← 自动 SELECT 列表（只 SELECT view 用到的列）
        order0_.id as id1_0_,
        order0_.customer_name as customer_name2_0_,
        count(lines1_.id) as col_2_0_,                                ← @Mapping("SIZE(lines)") 翻译成 COUNT
        sum(lines1_.price) as col_3_0_                                ← @Mapping("SUM(...)") 翻译成 SUM
    from
        orders order0_
    left outer join order_lines lines1_                              ← Blaze 自动 LEFT JOIN，不写 JOIN 提示
        on order0_.id = lines1_.order_id
    group by order0_.id, order0_.customer_name                       ← 自动 GROUP BY 主键和投影列
💡 这一条 SQL 就跑出了 N+1 问题的 0 次 N——因为 Blaze 把"集合 fetch + 聚合"压在一条带 GROUP BY 的 SELECT 里。如果你直接用 JPA findById() 然后访问 order.lines.size()，会触发额外的 SELECT * FROM order_lines WHERE order_id = ?。N+1 自检详见 第 13 章 · 最佳实践。
开关能看到性能影响
hibernate.show_sql=true所有 SQL（带占位符 ?）~3% 慢
hibernate.format_sql=trueSQL 格式化（含换行）~5% 慢
org.hibernate.SQL=DEBUGSQL（同 show_sql 但走 logger）~3% 慢
org.hibernate.orm.jdbc.bind=TRACE参数绑定值（？被替换）~10% 慢
org.hibernate.SQL_SLOW=INFO仅慢查询（> 配置阈值）0%

0.5 Spring Boot 自动配置速查

用 Spring Boot 比 0.1 节的纯 JPA 配置短 80%。

<dependency>
  <groupId>com.blazebit</groupId>
  <artifactId>blaze-persistence-integration-spring-data-3.1</artifactId>
  <version>1.6.11</version>
</dependency>
# application.yml
spring:
  datasource:
    url: jdbc:h2:./target/db
  jpa:
    properties:
      hibernate.dialect: org.hibernate.dialect.H2Dialect
      hibernate.show_sql: true
// 注入用，无需手动 createCriteriaBuilderFactory
@Service
public class OrderService {
    private final CriteriaBuilderFactory cbf;     // ★ 自动注入
    private final EntityViewManager evm;          // ★ 自动注入
    private final EntityManager em;

    public List<OrderSummaryView> findByCustomer(String name) {
        return evm.applySetting(EntityViewSetting.create(OrderSummaryView.class),
                                cbf.create(em, Order.class).where("customerName").eq(name))
                  .getResultList();
    }
}

// EntityViewRepository 让你写更少代码
public interface OrderViewRepository
        extends EntityViewSpecificationRepository<OrderSummaryView, Long> {
    List<OrderSummaryView> findByCustomerName(String name);
}

0.7 三种集成方式对比

方式依赖启动配置适用
① 纯 JPA + Blaze（本章用的） core-impl + entity-view-impl + integration-hibernate-X 手动 Criteria.getDefault().createCriteriaBuilderFactory(emf) 命令行工具、批处理、嵌入到非 Spring 容器
② Spring Boot + Blaze integration-spring-data-3.x application.yml 自动配置 + 注入 CBF/EVM 大多数 Web/微服务（推荐）
③ Quarkus + Blaze（GraalVM Native） blaze-persistence-integration-quarkus-3 Quarkus 扩展，构建期生成所有元数据 低延迟 Lambda、需要快速冷启动

详见 第 12 章 Spring Data 和 第 18 章 Quarkus & GraalVM。

0.8 出问题怎么办（FAQ 速查）

症状大概率原因解法
NoClassDefFoundError: com.blazebit.persistence.spi.CriteriaBuilderConfigurationContributor缺 integration-hibernate-X 模块加上对应 Hibernate 版本的 integration jar（hibernate-6.2 / hibernate-5.4）
EntityView OrderView is not registered没 evc.addEntityView(OrderView.class)在 createEntityViewManager 之前注册；或用 Spring Boot 自动扫描
跑出来的 SQL 多了几次额外 SELECTEntity View 里用了 JPA 关联但没声明 @Mapping把所有 getter 都用 @Mapping 显式声明，避免 Blaze 回退到 lazy load
QuerySyntaxException: SIZE not supported in SELECTHibernate 版本太老（< 6）升级到 Hibernate 6.2+，或用 @MappingCorrelated
Spring Boot 启动报 circular reference EVM ↔ TX managerEntityViewListener 注入了 TransactionManager用 @Lazy，或改为 ApplicationEventPublisher
Quarkus native image 启动 NoSuchMethodException静态元模型没生成开启 blaze-persistence-entity-view-processor-jakarta 注解处理器
分页 OFFSET 大时变慢Offset 分页 O(n) 病灶切到 Keyset 分页，详见 Ch.10
批量 UPDATE 后查询返回旧值JPA L1 缓存没刷新UPDATE 后 em.flush();em.clear()；详见 Ch.11 DML

💡 学习心法：Blaze 看似 API 很多（CriteriaBuilder + EntityView + DbmsDialect + JpqlFunction...），但真正写业务时只用 2 个 Factory + 2 个注解（@EntityView, @Mapping）。其他都是"高级用法保留地"。先把这 4 个用熟，再按需深入。
▎ 本章重点提炼
  • ① 核心概念：Blaze-Persistence = JPA Criteria 的类型安全替代 + EntityView 投影框架
  • ② 关键设计：CriteriaBuilder 链式 API 生成优化 JPQL/SQL；EntityView 实现 DTO 自动映射
  • ③ 主流程：添加 Maven 依赖 → 注入 CriteriaBuilderFactory + EntityViewManager → 链式构建查询
  • ④ 性能关键：EntityView 只 SELECT 需要的列（投影裁剪），避免 JPA 的全字段 SELECT *
  • ⑤ 易错点：@EntityView 和 @Mapping 是 90% 场景的全部注解，不要过早引入高级特性
