第 8 章 · CTE 与集合操作

CTE · 递归 CTE · UNION · INTERSECT · EXCEPT · JPA 增强

SQL 的 WITH 子句（Common Table Expression）是编写复杂查询的利器： 它把中间结果命名为临时表，让主查询逻辑清晰分层。然而 JPA / JPQL 规范完全不支持 CTE， 也不支持 UNION 等集合操作。本章演示 Blaze-Persistence 如何在 JPA 之上 补齐这两块关键拼图。

  • 使用 UNION / INTERSECT / EXCEPT 组合查询结果

8.1 CTE 基本概念

SQL 中的 CTE

Common Table Expression（CTE）通过 WITH 关键字将子查询定义为具名临时结果集， 在主查询中可像表一样引用。它的最大价值在于消除子查询嵌套， 把"由内到外"的阅读顺序改为"从上到下"。

图 8-1 · CTE 将嵌套子查询展开为线性结构

JPA 为何不支持 CTE

JPQL 的设计哲学是面向对象查询语言——它操作的是实体与关联路径， 而 CTE 本质是一种临时的"匿名表"，不属于 JPA 的实体元模型。因此 JPA 2.x 和 Jakarta Persistence 3.x 规范均未引入 CTE 支持。

核心思路：Blaze-Persistence 把 CTE 建模为一种特殊的 JPA 实体（@CTE）， 在 JPQL 生成阶段将其翻译为原生 SQL 的 WITH 子句——JPA Provider 只看到最终 SQL， 完全不感知 CTE 的存在。

8.2 CTE 实体定义

@CTE 注解

要使用 CTE，首先需要定义一个 CTE 实体类。 它用 @CTE 注解标记，看起来像普通 JPA 实体，但并不映射到真实数据库表。 CTE 实体的字段定义了 CTE 的"列"——主查询通过这些字段引用 CTE 结果。 

@CTE                          // Blaze-Persistence 标记
@Entity                       // JPA 要求
public class TopCustomerCTE {

    @Id
    private Long customerId;

    private BigDecimal totalAmount;

    private Long orderCount;

    // getters & setters ...
}

CTE 实体 vs 普通实体

图 8-3 · CTE 实体与普通实体结构对比
classDiagram class Customer { <<Entity>> +Long id +String name +String email +List~Order~ orders 真实数据库表 customer } class Order { <<Entity>> +Long id +BigDecimal amount +LocalDate orderDate +Customer customer 真实数据库表 orders } class TopCustomerCTE { <<CTE Entity>> +Long customerId +BigDecimal totalAmount +Long orderCount 无真实表 - 运行时 WITH 子句 } Customer "1" --> "*" Order : has TopCustomerCTE ..> Customer : 引用 customerId TopCustomerCTE ..> Order : 聚合 amount/count style TopCustomerCTE fill:#2d1f4e,stroke:#bc8cff style Customer fill:#1e3a5f,stroke:#58a6ff style Order fill:#1e3a5f,stroke:#58a6ff

JPA 无法表达"临时表"概念。要在纯 JPQL 中实现类似效果，只能写嵌套子查询或多次查询后在 Java 内存中拼合。

Blaze-Persistence 的 @CTE 实体让"WITH 子句"成为一等公民， 在编译期提供类型安全的列绑定。

注册要求：CTE 实体必须像普通 JPA 实体一样注册到 persistence.xml （或 Spring Boot 的实体扫描路径中），否则 Blaze-Persistence 无法识别它。

8.3 CTE 查询构建

with() API 工作流

CTE 查询的核心 API 是 with(CTEEntity.class)。调用后进入 CTE 绑定模式， 使用 .bind("column") 将 CTE 实体的各列绑定到子查询的 SELECT 项。

图 8-4 · CTE 查询构建时序
sequenceDiagram participant Dev as 开发者 participant CB as CriteriaBuilder participant CTE as CTE 注册器 participant SQL as SQL 渲染器 Dev->>CB: cbf.create(em, Customer.class) Dev->>CB: .with(TopCustomerCTE.class) CB->>CTE: 注册 CTE 定义 Dev->>CTE: .from(Order.class, "o") Dev->>CTE: .bind("customerId").select("o.customer.id") Dev->>CTE: .bind("totalAmount").select("SUM(o.amount)") Dev->>CTE: .bind("orderCount").select("COUNT(o.id)") Dev->>CTE: .groupBy("o.customer.id") CTE-->>CB: CTE 绑定完成（end） Dev->>CB: 主查询: .from(TopCustomerCTE.class) Dev->>CB: .where("totalAmount").gt(10000) Dev->>CB: .getResultList() CB->>SQL: 合并 CTE + 主查询 SQL-->>Dev: WITH top_cust AS (...) SELECT ...

完整代码示例

以下示例查询"订单总金额超过 10000 的顶级客户"：

CriteriaBuilder<Customer> cb = cbf.create(em, Customer.class);

// Step 1: 定义 CTE —— 按客户汇总订单
cb.with(TopCustomerCTE.class)
    .from(Order.class, "o")
    .bind("customerId").select("o.customer.id")
    .bind("totalAmount").select("SUM(o.amount)")
    .bind("orderCount").select("COUNT(o.id)")
    .groupBy("o.customer.id")
  .end();

// Step 2: 主查询引用 CTE 进行过滤
cb.from(TopCustomerCTE.class, "tc")
  .innerJoinOn(Customer.class, "c")
    .on("c.id").eqExpression("tc.customerId")
  .end()
  .where("tc.totalAmount").gt(10000)
  .orderByDesc("tc.totalAmount");

List<Customer> topCustomers = cb.getResultList();

生成的 SQL：

WITH top_cust AS (
    SELECT o.customer_id, SUM(o.amount), COUNT(o.id)
    FROM orders o
    GROUP BY o.customer_id
)
SELECT c.*
FROM top_cust tc
JOIN customer c ON c.id = tc.customer_id
WHERE tc.total_amount > 10000
ORDER BY tc.total_amount DESC

8.4 递归 CTE

递归 CTE 的结构

递归 CTE 由两部分组成：基础查询（锚点）和递归查询， 通过 UNION ALL 连接。每次递归迭代将上一轮的结果作为输入，直到不再产生新行。 这是遍历层级数据（组织架构、分类树、BOM 物料清单）的标准技术。

首先定义 CTE 实体：

@CTE
@Entity
public class OrgHierarchyCTE {
    @Id
    private Long employeeId;
    private String employeeName;
    private Long managerId;
    private Integer depth;
}

然后构建递归查询：

CriteriaBuilder<OrgHierarchyCTE> cb = cbf.create(em, OrgHierarchyCTE.class);

cb.withRecursive(OrgHierarchyCTE.class)
    // 基础查询: CEO（无上级）
    .from(Employee.class, "e")
    .bind("employeeId").select("e.id")
    .bind("employeeName").select("e.name")
    .bind("managerId").select("e.manager.id")
    .bind("depth").select("0")
    .where("e.manager").isNull()
  // UNION ALL 连接递归部分
  .unionAll()
    // 递归查询: 查找上一层结果的直接下属
    .from(Employee.class, "e")
    .from(OrgHierarchyCTE.class, "parent")
    .bind("employeeId").select("e.id")
    .bind("employeeName").select("e.name")
    .bind("managerId").select("e.manager.id")
    .bind("depth").select("parent.depth + 1")
    .where("e.manager.id").eqExpression("parent.employeeId")
  .end();

List<OrgHierarchyCTE> orgTree = cb.getResultList();

循环检测与深度限制

无限递归风险：如果数据存在循环引用（例如 A 的上级是 B，B 的上级又是 A）， 递归 CTE 将无限执行。不同数据库处理方式不同：
数据库默认行为限制手段
PostgreSQL 无限递归直到 OOM SET statement_timeout 或应用层 WHERE depth < N
MySQL 8+ 达到 cte_max_recursion_depth 后报错 默认 1000 次，可通过 SET 调整
Oracle 报 ORA-32044 错误 CYCLE 子句检测循环
SQL Server 默认 100 次递归后报错 OPTION (MAXRECURSION N)
最佳实践：在递归查询中始终加入 depth 字段并在 WHERE 中限制最大深度（如 WHERE parent.depth < 20）， 这比依赖数据库默认限制更可靠。

8.5 集合操作：UNION / INTERSECT / EXCEPT

四种集合操作

集合操作将多个 SELECT 的结果按集合理论组合。JPA 规范同样不支持这些操作， Blaze-Persistence 提供了完整的类型安全 API。

图 8-8 · 四种集合操作的语义
UNION ALL
集合 A: 1,2,3   集合 B: 2,3,4
结果: 1,2,3,2,3,4
保留全部（含重复）
UNION
集合 A: 1,2,3   集合 B: 2,3,4
结果: 1,2,3,4
去重合并
INTERSECT
集合 A: 1,2,3   集合 B: 2,3,4
结果: 2,3
仅交集
EXCEPT
集合 A: 1,2,3   集合 B: 2,3,4
结果: 1
A 中排除 B
图 8-9 · 集合操作 Venn 图示意
🔵 UNION = A ∪ B（全部元素，去重）
🟢 UNION ALL = A + B（全部元素，保留重复）
🟣 INTERSECT = A ∩ B（共有元素）
🟠 EXCEPT = A − B（A 独有元素）

API 实战

示例：合并 VIP 客户和高频客户名单

CriteriaBuilder<CustomerDTO> cb = cbf.create(em, CustomerDTO.class);

// 查询 1: VIP 客户（总金额 > 50000）
cb.startUnion()
    .from(Customer.class, "c")
    .innerJoin("c.orders", "o")
    .select("c.id")
    .select("c.name")
    .groupBy("c.id", "c.name")
    .having("SUM(o.amount)").gt(50000)

// 查询 2: 高频客户（订单数 > 100）
  .union()
    .from(Customer.class, "c")
    .innerJoin("c.orders", "o")
    .select("c.id")
    .select("c.name")
    .groupBy("c.id", "c.name")
    .having("COUNT(o.id)").gt(100)
  .endUnion();

List<CustomerDTO> result = cb.getResultList();

示例：查找"同时属于两个分类"的产品（INTERSECT）

CriteriaBuilder<Long> cb = cbf.create(em, Long.class);

cb.startUnion()
    .from(Product.class, "p")
    .select("p.id")
    .where("p.category.name").eq("Electronics")
  .intersect()
    .from(Product.class, "p")
    .select("p.id")
    .where("p.category.name").eq("On Sale")
  .endUnion();

示例：查找"在 A 类别但不在 B 类别"的产品（EXCEPT）

CriteriaBuilder<Long> cb = cbf.create(em, Long.class);

cb.startUnion()
    .from(Product.class, "p")
    .select("p.id")
    .where("p.category.name").eq("Premium")
  .except()
    .from(Product.class, "p")
    .select("p.id")
    .where("p.discontinued").eq(true)
  .endUnion();

8.6 数据库支持矩阵

不同数据库对 CTE 和集合操作的支持存在差异。 Blaze-Persistence 会在运行时检测目标数据库的能力， 对不支持的特性采用降级策略（如内联子查询替代 CTE）。

降级策略：当目标数据库不支持 CTE 时（如 MySQL 5.x）， Blaze-Persistence 会尝试将 CTE 内联为子查询。但递归 CTE 无法降级—— 如果数据库不支持递归，将抛出运行时异常。

陷阱 1：CTE 实体未注册

ClassNotFoundException / Unknown entity： CTE 实体虽然不对应真实表，但仍必须被 JPA 持久化单元管理。 在 persistence.xml 中添加 <class>， 或确保 Spring Boot 的 @EntityScan 覆盖 CTE 所在包。

陷阱 2：MySQL 5.x 不支持 CTE

MySQL 在 8.0 版本才引入 CTE 支持。如果项目仍在使用 5.x， 所有 CTE 查询将失败。Blaze-Persistence 能将非递归 CTE 降级为内联子查询， 但递归 CTE 没有替代方案——必须升级到 MySQL 8.0+。

陷阱 3：Materialized vs Inline CTE 性能

某些数据库（如 PostgreSQL 12+）允许显式控制 CTE 是否被物化（materialized）。 物化的 CTE 先执行并缓存结果，后续引用直接读取缓存； 内联的 CTE 在每次引用时重新计算。

Materialized CTE（默认 PG < 12）

优化器无法将主查询的过滤条件下推到 CTE 内部，可能导致全表扫描后再过滤。

WITH big_cte AS MATERIALIZED (
  SELECT ... FROM large_table
)
SELECT * FROM big_cte
WHERE id = 42  -- 无法下推

Inline CTE（PG 12+ 默认）

优化器可以将谓词下推，让 CTE 内的查询只扫描必要行。

WITH big_cte AS NOT MATERIALIZED (
  SELECT ... FROM large_table
)
SELECT * FROM big_cte
WHERE id = 42  -- 下推到内部

陷阱 4：Hibernate CTE 渲染问题

Hibernate 兼容性：Blaze-Persistence 的 CTE 渲染完全绕过 Hibernate 的 HQL 解析器——它在最终 SQL 层面注入 WITH 子句。因此需要匹配的 Hibernate 集成模块版本（blaze-persistence-integration-hibernate-5.x 或 hibernate-6.x），版本不匹配可能导致 SQL 注入位置错误。

陷阱 5：集合操作的列对齐

使用 UNION / INTERSECT / EXCEPT 时，所有子查询必须返回相同数量和类型的列。 Blaze-Persistence 会在构建阶段做类型检查，但复杂表达式（如 CASE） 可能导致类型推断不一致——建议显式 cast。

实战：多表统一搜索

电商系统中需要实现"全局搜索"——同时搜索产品名、订单号和客户名， 返回统一的搜索结果列表。使用 UNION 可以将三个独立查询合并： 

cb.startUnion()
    .from(Product.class).select("id").select("'PRODUCT'")
    .select("name").where("name").like().value("%keyword%").noEscape()
  .union()
    .from(Order.class).select("id").select("'ORDER'")
    .select("orderNo").where("orderNo").like().value("%keyword%").noEscape()
  .union()
    .from(Customer.class).select("id").select("'CUSTOMER'")
    .select("name").where("name").like().value("%keyword%").noEscape()
  .endUnion();

本章小结

  • CTE（WITH 子句）将复杂子查询扁平化为具名临时结果集，Blaze-Persistence 通过 @CTE 实体 + with() API 在 JPA 之上实现完整支持
  • CTE 实体用 @CTE @Entity 标注，定义 CTE 的列结构，必须注册到持久化单元
  • CTE 查询通过 .bind("col").select(...) 将子查询结果绑定到 CTE 列，主查询像引用普通实体一样引用 CTE
  • 递归 CTE使用 withRecursive() + unionAll() 实现层级遍历（组织树、分类树），务必加 depth 限制防止无限递归
  • 集合操作（UNION / INTERSECT / EXCEPT）通过 startUnion() / endUnion() API 组合多个查询结果
  • 数据库兼容性是核心关注点：MySQL 5.x 不支持 CTE，INTERSECT/EXCEPT 需要 MySQL 8.0.31+，Oracle 用 MINUS 替代 EXCEPT
  • 常见陷阱包括 CTE 实体未注册、递归死循环、Materialized 性能问题、集合操作列不对齐
