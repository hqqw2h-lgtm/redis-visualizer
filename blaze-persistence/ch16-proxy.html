第 16 章 · Entity View 代理生成

运行时代理 · 字节码生成 · 脏检查实现 · 不可变 vs 可变 · 静态元模型

Entity View 以接口或抽象类定义 — 但运行时谁来实例化它们？本章深入 Blaze-Persistence 最核心的黑魔法：代理类的字节码生成机制。我们将逐层剥开代理的内存布局、脏检查位图、集合录制器，以及编译期静态元模型的替代方案。

  • 理解 Entity View 从接口到可执行对象的三种实现策略
  • 掌握运行时字节码生成的完整流程（ProxyFactory → ByteBuddy/ASM → Class）
  • 区分不可变视图与可变视图在代理层面的差异
  • 深入脏检查位图（dirty bits）的工作原理与部分更新机制
  • 理解集合代理（RecordingList / RecordingSet / RecordingMap）的变更追踪策略
  • 使用注解处理器生成静态实现，加速启动并兼容 GraalVM
  • 识别代理生成的常见陷阱与模块化 Java 限制

前置知识：建议先完成 第 4 章 · Entity View 基础 和 第 6 章 · 可更新 Entity View 的学习。

16.1 Entity View 实现策略

当你定义一个 @EntityView 接口时，其中没有任何实现代码 — 只有 getter 方法签名。Blaze-Persistence 必须在运行时生成真正的类来实现这些接口，才能将查询结果映射为类型安全的对象。

16.2 运行时代理生成

运行时代理生成是 Entity View 的默认实现策略。当 EntityViewManager 初始化时，它会为每个已注册的 View 接口生成一个代理类，然后在查询执行时通过代理类的构造器将结果元组映射为 View 实例。

16.3 不可变视图的代理

只读的 Entity View（未标注 @UpdatableEntityView）获得的是精简版代理：只有 getter 方法，没有 setter，不追踪脏状态。这类代理的内存开销最小、生成逻辑最简单。

TupleConstructor：查询结果到代理的映射

Blaze-Persistence 的 ObjectBuilder 系统负责将 SELECT 元组转换为代理实例。核心流程：先确定元组中每个位置对应 View 的哪个属性，再调用代理构造器完成映射。

SELECT 元组 → 代理实例的映射流程

确保 tuple[i] 对应正确的 getter
💡 性能优势：只读代理的实例化路径非常短 — 仅一次数组赋值，无任何拷贝或快照。在 getResultList() 返回 10,000 条记录时，只读代理比可更新代理节约约 40% 的内存分配。

@EntityView(Cat.class)
public interface CatSummaryView {
    @IdMapping
    Long getId();
    String getName();

    @Mapping("UPPER(breed)")
    String getBreedUpper();

    @Mapping("SIZE(kittens)")
    long getKittenCount();
}

16.4 可变视图的脏检查

标注了 @UpdatableEntityView 的视图代理拥有完整的变更追踪能力。每次调用 setter 时，代理会自动在位图中标记对应字段为"脏"。最终 flush 时，只有脏字段才会被写入 UPDATE 语句。

// 可更新代理生成的 setter 伪代码
@Override
public void setName(String name) {
    if (!Objects.equals(this.$$_tuple[1], name)) {
        this.$$_tuple[1] = name;
        this.$$_dirtyBits |= (1L << 1);  // 标记 bit 1 为脏
        if (this.$$_parent != null) {
            this.$$_parent.$$_markDirty(this.$$_parentIndex);
        }
    }
}

@Override
public boolean $$_isDirty() {
    return this.$$_dirtyBits != 0L;
}

@Override
public String[] $$_getDirtyProperties() {
    List<String> dirty = new ArrayList<>();
    if (($$_dirtyBits & (1L << 1)) != 0) dirty.add("name");
    if (($$_dirtyBits & (1L << 2)) != 0) dirty.add("age");
    // ...
    return dirty.toArray(new String[0]);
}

初始状态快照与重置

可更新代理在加载时会拍摄一份初始状态快照 $$_initialState。当调用 $$_unmarkDirty() 时，快照同步更新为当前值。这确保了二次修改能被正确追踪。

初始快照的生命周期
stateDiagram-v2 [*] --> 加载态: 查询返回元组 加载态 --> 修改态: setter 调用 修改态 --> 修改态: 更多 setter 调用 修改态 --> 持久化: evm.save() 成功 持久化 --> 加载态: 快照更新为当前值\ndirtyBits 重置为 0 修改态 --> 回滚态: evm.save() 失败 回滚态 --> 修改态: 修正后重试 note right of 加载态 $$_initialState = copy($$_tuple) $$_dirtyBits = 0 end note note right of 修改态 $$_tuple 已变更 $$_dirtyBits ≠ 0 $$_initialState 保持不变 end note
⚠️ 位图容量限制：long 类型的脏位图最多追踪 64 个字段。当 View 的属性超过 64 个时，框架会自动切换到 long[] 数组。不过，拥有 64+ 属性的 View 本身就是反模式（参见第 13 章 God View 反模式）。

静态元模型 API

APT 同时生成静态元模型类 CatView_，提供编译期类型安全的属性引用，类似 JPA 的 CriteriaMetamodel。

// APT 生成的静态元模型（简化）
@StaticMetamodel(CatView.class)
public class CatView_ {
    public static volatile MethodSingularAttribute<CatView, Long> id;
    public static volatile MethodSingularAttribute<CatView, String> name;
    public static volatile MethodSingularAttribute<CatView, Integer> age;
    public static volatile MethodPluralAttribute<CatView, List<KittenView>, KittenView> kittens;
}

// 使用元模型进行类型安全的排序和过滤
EntityViewSetting<CatView, CriteriaBuilder<CatView>> setting =
    EntityViewSetting.create(CatView.class);
setting.addAttributeSorter(CatView_.name, Sorters.ascending());
setting.addAttributeFilter(CatView_.age, "3");

16.7 陷阱与限制

代理生成是 Blaze-Persistence 最强大也最脆弱的部分。以下问题在生产环境中反复出现，需提前识别和预防。

代理生成六大陷阱及应对策略
① 启动时间膨胀
✅ 使用 APT 静态实现或延迟初始化
② 模块化 Java 限制
✅ 添加 --add-opens java.base/java.lang
③ GraalVM 不兼容
✅ 必须使用 APT + 注册 reflect-config.json
④ 深层视图层次
✅ 扁平化视图层次，避免 >3 层嵌套
⑤ 序列化问题
✅ 配置 Jackson Mixin 或 @JsonIgnoreProperties
⑥ ClassLoader 冲突
✅ 确保 View 和代理在同一 ClassLoader

陷阱 1：启动时间膨胀

每个 Entity View 的代理生成需要 5–15ms。当项目中有 100+ 个 View 时，累计启动时间可达数秒。

// 测量代理生成耗时
long start = System.nanoTime();
EntityViewConfiguration config = EntityViews.createDefaultConfiguration();
config.addEntityView(CatView.class);
config.addEntityView(DogView.class);
// ... 100+ Views
EntityViewManager evm = config.createEntityViewManager(cbf);
long elapsed = (System.nanoTime() - start) / 1_000_000;
System.out.println("EVM init: " + elapsed + "ms");
// 典型输出: EVM init: 850ms  (100 Views)

陷阱 2：模块化 Java (JPMS) 限制

Java 9+ 的模块系统默认禁止跨模块反射访问。代理生成需要访问 java.lang 包的内部 API。

# JVM 启动参数 — 为代理生成开放必要的模块
--add-opens java.base/java.lang=ALL-UNNAMED
--add-opens java.base/java.lang.reflect=ALL-UNNAMED
--add-opens java.base/java.util=ALL-UNNAMED
⚠️ 忘记 add-opens：如果缺少上述 JVM 参数，启动时会抛出 InaccessibleObjectException，错误信息中包含 "module java.base does not 'opens' ... to unnamed module"。这是 Java 17+ 环境中最常见的代理生成失败原因。

陷阱 5：代理实例的 Jackson 序列化

代理类包含框架内部字段（$$_tuple, $$_dirtyBits 等），直接序列化会泄漏内部状态。需要配置 Jackson 过滤。

// 方案 1: 全局配置 Jackson ObjectMapper
ObjectMapper mapper = new ObjectMapper();
mapper.addMixIn(EntityViewProxy.class, EntityViewProxyMixin.class);

@JsonIgnoreProperties({"$$_tuple", "$$_dirtyBits", "$$_parent",
    "$$_parentIndex", "$$_initialState", "$$_kind"})
abstract class EntityViewProxyMixin {}

// 方案 2: 使用 blaze-persistence-integration-jackson
// 自动处理代理序列化，推荐方案
<dependency>
    <groupId>com.blazebit</groupId>
    <artifactId>blaze-persistence-integration-jackson</artifactId>
    <version>${blaze-persistence.version}</version>
</dependency>

陷阱 6：应用服务器中的 ClassLoader 问题

WildFly / JBoss 部署注意事项

在 EAR 部署中，如果 Entity View 接口定义在一个 EJB 模块、而 EntityViewManager 在另一个 Web 模块中初始化，代理类可能由错误的 ClassLoader 加载。解决方案：将 View 接口和 EVM 配置放在共享的 lib/ 目录下，确保同一 ClassLoader 可见。在 Spring Boot Fat JAR 部署中不存在此问题。

📝 本章小结

  • Entity View 接口通过运行时代理（ByteBuddy/ASM）、编译期静态实现（APT）或 Builder 模式三种策略实例化。
  • 代理类实现用户定义接口 + 内部接口（EntityViewProxy、DirtyTracker），字段存储于 Object[] 元组数组中。
  • 只读代理只有 getter、无脏追踪，内存开销最小；可更新代理包含 setter、脏位图和初始状态快照。
  • 脏检查通过 long 位图的位操作实现：setter 写入时 OR 标记对应 bit，flush 时只更新脏字段，生成部分 UPDATE SQL。
  • 集合字段被替换为录制集合（RecordingList/Set/Map），精确记录增删操作，支持 Diff-based 和 Replace-based 两种 flush 策略。
  • APT 注解处理器在编译期生成 *_Impl.java 和 *_.java 静态元模型，消除启动开销并兼容 GraalVM Native Image。
  • 代理生成的主要陷阱包括 JPMS add-opens、Jackson 序列化泄漏、深层嵌套膨胀和 ClassLoader 冲突。
