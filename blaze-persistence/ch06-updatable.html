前置知识：第 4 章 · Entity View 基础 和 第 5 章 · 高级 Entity View

6.1 从只读到可写：Updatable Entity View

在第 4 章中，Entity View 被设计为只读接口——只有 getter，没有 setter。@UpdatableEntityView 注解打破了这一限制：它告诉 Blaze-Persistence 为该视图生成可变代理，允许通过 setter 修改属性值，并在调用 save() 时将变更同步到数据库。

只读视图 → 可更新视图 → 数据库 状态流转
stateDiagram-v2 [*] --> ReadOnlyView : @EntityView（只读） ReadOnlyView --> UpdatableView : 添加 @UpdatableEntityView UpdatableView --> DirtyState : 调用 setter 修改属性 DirtyState --> FlushPhase : EntityViewManager.save() FlushPhase --> GenerateSQL : 脏检查 → 仅更新变更列 GenerateSQL --> Database : UPDATE ... SET col=? WHERE id=? Database --> CleanState : 提交成功 CleanState --> UpdatableView : 重置脏标志 note right of DirtyState : 内部维护 dirtySet\n记录哪些字段被修改 note right of GenerateSQL : 只生成必要的 SET 子句

接口 vs 抽象类定义

可更新视图支持接口和抽象类两种定义方式。接口方式需要同时声明 getter 和 setter；抽象类方式可以利用字段直接赋值。

@UpdatableEntityView
@EntityView(User.class)
public interface UserUpdateView {

    @IdMapping
    Long getId();

    String getName();
    void setName(String name);

    String getEmail();
    void setEmail(String email);
}
@UpdatableEntityView
@EntityView(User.class)
public abstract class UserUpdateView {

    @IdMapping
    public abstract Long getId();

    public abstract String getName();
    public abstract void setName(String name);

    public abstract String getEmail();
    public abstract void setEmail(String email);
}
Updatable View 代理生成架构
flowchart LR DEF["@UpdatableEntityView\n接口/抽象类定义"] --> EVM["EntityViewManager\n启动扫描"] EVM --> PROXY["代理类生成"] PROXY --> GET["getter 方法\n委托到内部字段"] PROXY --> SET["setter 方法\n写入 + 标记 dirty"] PROXY --> DT["DirtyTracker\n接口实现"] style DEF fill:#2d1f4e,stroke:#7c3aed style PROXY fill:#1e3a5f,stroke:#2563eb style DT fill:#1a3a2a,stroke:#22c55e

使用示例

// 1. 从数据库加载可更新视图
UserUpdateView user = entityViewManager.find(em, UserUpdateView.class, userId);

// 2. 修改属性 — 内部自动标记脏字段
user.setName("新名称");
user.setEmail("new@example.com");

// 3. 持久化变更 — 只 UPDATE 被修改的列
entityViewManager.save(em, user);
核心优势：与 JPA 的 merge() 不同，save() 不会生成包含所有列的 UPDATE 语句，而是只更新实际发生变更的字段。这在宽表场景下能显著减少 I/O。

6.2 创建新实体：@CreatableEntityView

如果说 @UpdatableEntityView 对应 UPDATE，那么 @CreatableEntityView 就是 INSERT 的对应物。它允许你通过 EntityViewManager.create() 实例化一个空视图，填充字段后调用 save() 将其作为新行插入数据库。

@CreatableEntityView 完整生命周期
flowchart TD A["EntityViewManager.create(UserCreateView.class)"] --> B["空视图实例\n所有字段为 null/默认值"] B --> C["setter 填充字段\nsetName / setEmail / ..."] C --> D{"所有必填字段\n已设置?"} D -- 是 --> E["EntityViewManager.save(em, view)"] D -- 否 --> F["抛出异常\n缺少必填属性"] E --> G["生成 INSERT SQL"] G --> H["INSERT INTO user\n(name, email, ...) VALUES (?, ?, ...)"] H --> I["数据库分配 ID"] I --> J["回填 ID 到视图"] style A fill:#2d1f4e,stroke:#7c3aed style E fill:#1e3a5f,stroke:#2563eb style G fill:#1a3a2a,stroke:#22c55e style H fill:#1a3a2a,stroke:#22c55e style F fill:#3a1a1a,stroke:#f85149

定义可创建视图

通常会将 @CreatableEntityView 和 @UpdatableEntityView 组合在同一个视图上，使其同时支持 INSERT 和 UPDATE：

@CreatableEntityView
@UpdatableEntityView
@EntityView(User.class)
public interface UserEditView {

    @IdMapping
    Long getId();

    String getName();
    void setName(String name);

    String getEmail();
    void setEmail(String email);

    @Mapping("department")
    DepartmentIdView getDepartment();
    void setDepartment(DepartmentIdView dept);
}

创建并保存新实体

// 1. 创建空视图
UserEditView newUser = entityViewManager.create(UserEditView.class);

// 2. 填充属性
newUser.setName("张三");
newUser.setEmail("zhangsan@example.com");

// 3. 设置关联
DepartmentIdView dept = entityViewManager.getReference(DepartmentIdView.class, 1L);
newUser.setDepartment(dept);

// 4. 保存 → INSERT
entityViewManager.save(em, newUser);
// newUser.getId() 现在已有数据库分配的 ID
@CreatableEntityView vs 传统 JPA persist
注意：@CreatableEntityView 要求视图中所有非空（non-nullable）字段在 save() 之前必须被设置，否则会抛出 IllegalArgumentException。建议在视图接口中使用 @EmptyFlatViewCreation 或默认值来规避此限制。

6.3 脏检查机制

脏检查（Dirty Tracking）是可更新视图的核心引擎。每当你调用 setter 时，代理对象会将对应的属性名加入内部 dirtySet。调用 save() 时，Blaze-Persistence 只会对 dirtySet 中的字段生成 UPDATE SQL。

脏检查完整时序 — 从 setter 到 SQL
sequenceDiagram participant App as 应用代码 participant Proxy as 可更新视图代理 participant DT as DirtyTracker participant EVM as EntityViewManager participant DB as 数据库 App->>Proxy: setName("李四") Proxy->>DT: markDirty("name") App->>Proxy: setEmail("lisi@co.com") Proxy->>DT: markDirty("email") Note over DT: dirtySet = {name, email} App->>EVM: save(em, view) EVM->>DT: getDirtyFields() DT-->>EVM: ["name", "email"] EVM->>EVM: 生成部分 UPDATE EVM->>DB: UPDATE user SET name=?, email=? WHERE id=? DB-->>EVM: 1 row affected EVM->>DT: clearDirty() Note over DT: dirtySet = {}

DirtyTracker 接口

所有可更新视图代理都实现了 DirtyTracker 接口，你可以在应用代码中直接查询脏状态：

UserUpdateView user = entityViewManager.find(em, UserUpdateView.class, 1L);
DirtyTracker tracker = (DirtyTracker) user;

// 修改前
tracker.isDirty();          // false
tracker.getDirtyProperties(); // []

user.setName("王五");

// 修改后
tracker.isDirty();          // true
tracker.getDirtyProperties(); // ["name"]

// 手动重置（不保存到 DB）
tracker.markClean();
JPA 全量 UPDATE vs Entity View 部分 UPDATE 对比
❌ JPA merge() — 全量 UPDATE
em.merge(entity) → UPDATE user SET name=?, email=?, phone=?, address=?, created_at=?, ... 所有列 WHERE id=?
✅ Entity View save() — 部分 UPDATE
evm.save(em, view) → UPDATE user SET name=? WHERE id=?
仅脏字段 — 网络 I/O 大幅减少
1
脏字段 → 1 个 SET 子句
50+
JPA merge 可能生成的 SET 子句
~60%
宽表场景网络 I/O 节省
设计哲学：Entity View 的脏检查发生在应用层，无需依赖 Hibernate 的 ByteBuddy 字节码增强或代理拦截。这意味着即使在脱管（detached）状态下，脏状态仍然完整保留。

6.4 级联策略

当可更新视图包含子视图（如 OrderView 内嵌 OrderItemView）时，你需要通过级联策略告诉 Blaze-Persistence 如何处理子对象的持久化、更新和删除。

WHERE id=?"] SAVE --> SCAN["扫描子视图集合\nitems: List<OrderItemView>"] SCAN --> NEW{"新增的子视图?"} NEW -- 是 --> INS["INSERT INTO order_items ..."] SCAN --> MOD{"已修改的子视图?"} MOD -- 是 --> UPD_C["UPDATE order_items SET ... 代码示例：父子级联

@UpdatableEntityView
@EntityView(Order.class)
public interface OrderUpdateView {

    @IdMapping
    Long getId();

    String getStatus();
    void setStatus(String status);

    @UpdatableMapping(cascade = { CascadeType.PERSIST, CascadeType.UPDATE })
    @Mapping("items")
    List<OrderItemUpdateView> getItems();
    void setItems(List<OrderItemUpdateView> items);
}

@CreatableEntityView
@UpdatableEntityView
@EntityView(OrderItem.class)
public interface OrderItemUpdateView {

    @IdMapping
    Long getId();

    String getProductName();
    void setProductName(String name);

    Integer getQuantity();
    void setQuantity(Integer qty);
}
// 加载订单视图（包含子项）
OrderUpdateView order = evm.find(em, OrderUpdateView.class, orderId);

// 修改子项
order.getItems().get(0).setQuantity(5);

// 新增子项
OrderItemUpdateView newItem = evm.create(OrderItemUpdateView.class);
newItem.setProductName("鼠标");
newItem.setQuantity(2);
order.getItems().add(newItem);

// 一次 save() 同时 UPDATE 父 + UPDATE 子 + INSERT 新子项
evm.save(em, order);

Orphan Removal 陷阱：如果 @UpdatableMapping 没有设置 orphanRemoval = true，从集合中移除的子视图不会被 DELETE，仅会断开外键关联（SET FK = NULL）。如果外键列不可空，数据库将抛出约束异常。

6.5 乐观锁

在并发环境下，多个用户可能同时修改同一条记录。Blaze-Persistence 通过 @Version 映射支持乐观锁——在 UPDATE 时检查版本号是否与读取时一致，不一致则抛出 OptimisticLockException。

乐观锁完整时序
sequenceDiagram participant U1 as 用户 A participant U2 as 用户 B participant EVM as EntityViewManager participant DB as 数据库 U1->>EVM: find(View, id=1) EVM->>DB: SELECT ... WHERE id=1 DB-->>EVM: {name="旧值", version=3} EVM-->>U1: view (version=3) U2->>EVM: find(View, id=1) EVM->>DB: SELECT ... WHERE id=1 DB-->>EVM: {name="旧值", version=3} EVM-->>U2: view (version=3) U1->>EVM: setName("A修改") → save() EVM->>DB: UPDATE ... SET name='A修改', version=4 WHERE id=1 AND version=3 DB-->>EVM: 1 row updated ✓ Note over DB: version 变为 4 U2->>EVM: setName("B修改") → save() EVM->>DB: UPDATE ... SET name='B修改', version=4 WHERE id=1 AND version=3 DB-->>EVM: 0 rows updated ✗ EVM-->>U2: 抛出 OptimisticLockException

视图中映射版本号

@UpdatableEntityView
@EntityView(Product.class)
public interface ProductUpdateView {

    @IdMapping
    Long getId();

    @Version
    Long getVersion();

    String getName();
    void setName(String name);

    BigDecimal getPrice();
    void setPrice(BigDecimal price);
}

冲突处理代码

ProductUpdateView product = evm.find(em, ProductUpdateView.class, productId);
product.setPrice(new BigDecimal("99.99"));

try {
    evm.save(em, product);
} catch (OptimisticLockException e) {
    // 方案 1：提示用户刷新后重试
    // 方案 2：重新加载最新版本并合并
    ProductUpdateView latest = evm.find(em, ProductUpdateView.class, productId);
    latest.setPrice(product.getPrice());
    evm.save(em, latest);
}

flowchart TD CONFLICT["OptimisticLockException"] --> S1["策略 1：提示用户\n刷新页面重试"] CONFLICT --> S2["策略 2：自动重载\n合并变更后重试"] CONFLICT --> S3["策略 3：Last-Write-Wins\n忽略版本直接覆盖"] S1 --> UI["适合：Web 表单编辑"] S2 --> BG["适合：后台批处理"] S3 --> LOW["适合：低冲突场景\n⚠ 可能丢失数据"] style CONFLICT fill:#3a1a1a,stroke:#f85149 style S1 fill:#1a3a2a,stroke:#22c55e style S2 fill:#1e3a5f,stroke:#2563eb style S3 fill:#3a2a1a,stroke:#d97706
参考：JPA 2.2 规范 §3.4.2 定义了 @Version 的语义。Blaze-Persistence 在 Entity View 层面完全兼容此规范，并在生成的 UPDATE 语句中自动加入 AND version = ? 条件。

6.6 EntityViewManager.save() 深度解析

save() 是可更新视图最核心的方法。它内部执行一条复杂的管线，从类型检测到 SQL 生成，每个阶段都有明确的职责。

save() 完整执行管线
flowchart TD ENTRY["evm.save(em, view)"] --> TYPE{"视图类型检测"} TYPE -- "@CreatableEntityView\n且 id 为 null" --> NEW_PATH["新实体路径"] TYPE -- "@UpdatableEntityView\n且 id 非 null" --> UPD_PATH["更新路径"] NEW_PATH --> VALIDATE_N["校验必填字段"] VALIDATE_N --> BUILD_INS["构建 INSERT 语句"] BUILD_INS --> EXEC_INS["执行 INSERT"] EXEC_INS --> GEN_ID["获取生成的 ID"] GEN_ID --> CASCADE_N["级联处理子视图"] CASCADE_N --> DONE["完成"] UPD_PATH --> DIRTY{"isDirty()?"} DIRTY -- 否 --> CASCADE_U["仅级联处理子视图"] DIRTY -- 是 --> FLUSH{"FlushMode?"} FLUSH -- FULL --> FULL_F["全字段 UPDATE\n（覆盖所有映射列）"] FLUSH -- LAZY --> LAZY_F["仅脏字段 UPDATE\n（部分 SET 子句）"] FULL_F --> VERSION["版本号检查 + 递增"] LAZY_F --> VERSION VERSION --> EXEC_UPD["执行 UPDATE"] EXEC_UPD --> CASCADE_U CASCADE_U --> CLEAR["清除脏标记"] CLEAR --> DONE style ENTRY fill:#2d1f4e,stroke:#7c3aed style NEW_PATH fill:#1a3a2a,stroke:#22c55e style UPD_PATH fill:#1e3a5f,stroke:#2563eb style DONE fill:#1a3a2a,stroke:#22c55e

配置 FlushMode

@UpdatableEntityView(mode = FlushMode.LAZY)  // 默认值
@EntityView(User.class)
public interface UserUpdateView {
    // ...
}

// 也可在 save 时指定
evm.save(em, view, FlushMode.FULL);

批量操作

对于批量更新场景，Blaze-Persistence 支持在单次事务中保存多个视图。框架会尽可能合并同类 SQL 以利用 JDBC 批处理：

List<UserUpdateView> users = evm.applySetting(
    EntityViewSetting.create(UserUpdateView.class),
    criteriaBuilder
).getResultList();

for (UserUpdateView u : users) {
    u.setStatus("ACTIVE");
}

// 批量 save
for (UserUpdateView u : users) {
    evm.save(em, u);
}
em.flush();
性能提示：批量保存时，FULL 模式因为 SQL 语句固定，更容易被 JDBC 驱动合并为 batch，而 LAZY 模式因每条语句 SET 子句不同，batch 效果有限。在批量场景下评估两种模式的实际吞吐量。

6.7 陷阱与限制

可更新视图功能强大，但也有一些需要注意的限制和常见陷阱。

常见陷阱全景图

陷阱 1：注解混用

错误用法：单独使用 @CreatableEntityView 而不加 @UpdatableEntityView 时，视图只能用于 INSERT。对已存在的实体调用 save() 将会尝试 INSERT 而非 UPDATE，导致主键冲突。 
// ✗ 错误：只有 @CreatableEntityView
@CreatableEntityView
@EntityView(User.class)
public interface UserView { ... }

// 试图"更新"会变成 INSERT → 主键冲突！
UserView existing = evm.find(em, UserView.class, 1L);
existing.setName("new");
evm.save(em, existing);  // → PK violation!

// ✓ 正确：同时标注两个注解
@CreatableEntityView
@UpdatableEntityView
@EntityView(User.class)
public interface UserView { ... }

陷阱 2：反向集合更新

JPA 中的双向关联有"拥有方"（owning side）概念。如果你的可更新视图映射的是被拥有方的集合（如 @OneToMany(mappedBy=...)），直接修改这个集合不会生效：

反向集合陷阱
flowchart LR subgraph WRONG["✗ 修改 inverse 集合"] W1["department.getUsers().add(newUser)"] --> W2["save(department)"] W2 --> W3["无效！外键在 User 表"] end subgraph RIGHT["✓ 修改拥有方"] R1["user.setDepartment(dept)"] --> R2["save(user)"] R2 --> R3["UPDATE user SET dept_id=?"] end style WRONG fill:#3a1a1a,stroke:#f85149 style RIGHT fill:#1a3a2a,stroke:#22c55e

陷阱 3：脱管视图的脏状态

Entity View 的脏状态存储在代理对象的内存中。如果视图被序列化（如通过 HTTP 传输到前端再返回），反序列化后的对象不再是原始代理，脏状态将丢失。解决方案是使用 EntityViewManager.saveTo() 进行合并：

// 前端返回的反序列化对象（非代理）
UserEditView fromClient = objectMapper.readValue(json, UserEditView.class);

// 使用 saveTo 将变更合并到托管视图
evm.saveTo(em, fromClient);

陷阱 4：深层级联的性能影响

级联深度 vs SQL 数量关系

性能建议：级联深度超过 2 层时，请认真评估 SQL 数量。考虑将深层子对象的操作拆分为独立的 save() 调用，或使用批量 DML（第 11 章）替代逐条级联更新。

陷阱 5：并发修改无 @Version

如果可更新视图没有映射 @Version 字段，save() 生成的 UPDATE 不会包含版本检查，Last-Write-Wins 将默默生效——后一次写入覆盖前一次，无任何警告。

本章小结

  • @UpdatableEntityView 将只读视图升级为可写视图，通过 setter 修改属性并用 save() 持久化。
  • @CreatableEntityView 支持 INSERT 操作，通过 evm.create() 实例化空视图。
  • 脏检查在应用层实现，仅对变更字段生成 UPDATE SQL，显著减少 I/O。
  • 级联策略（PERSIST / UPDATE / DELETE）控制子视图的传播行为，orphanRemoval 决定是否物理删除。
  • @Version 映射启用乐观锁，在并发修改时抛出 OptimisticLockException。
  • save() 内部执行完整管线：类型检测 → 脏检查 → FlushMode 决策 → 版本校验 → SQL 执行 → 级联。
  • 注意五大陷阱：注解混用、反向集合、脱管合并、深层级联性能、无版本并发覆盖。
