14.1 渲染管线全景

Blaze-Persistence 的查询渲染并非一步到位的字符串拼接。从用户 API 调用到数据库执行，数据流经六个清晰的阶段，每个阶段都承担着不同的转换职责。

两种渲染模式

Blaze-Persistence 支持两种截然不同的查询渲染路径。选择哪条路径取决于查询中使用的特性是否超出了标准 JPQL 的表达能力。

💡 核心设计决策：Blaze-Persistence 优先生成标准 JPQL，仅在 JPQL 表达能力不足时才回退到 provider-native 路径。这保证了最大程度的 JPA 可移植性，同时不牺牲高级功能的可用性。

14.2 ExpressionFactory 与表达式树

用户通过 CriteriaBuilder 传入的字符串（如 "name"、"SIZE(items)"）并非直接拼接到查询中。它们首先被 ExpressionFactory 解析为强类型的表达式节点，形成一棵内部表达式树。

解析示例

来看一个具体例子：用户构建一个带聚合与嵌套条件的查询时，表达式树的构造过程。

14.3 查询节点结构

AbstractCommonQueryBuilder 是所有查询构建器的基类。它不直接持有查询的各组成部分，而是通过一组 Manager 对象来分治管理查询的不同子句。

当多个 WHERE 条件或 SELECT 表达式引用同一关联路径时，JoinManager 会自动去重，确保同一路径只产生一个 JOIN 子句。

💡 去重的关键：JoinManager 以归一化后的关联路径作为去重键。即使 select("customer.name") 和 where("customer.country") 在不同的 Manager 中声明，它们引用的是同一个 customer JOIN 节点，最终只会输出一条 LEFT JOIN 子句。

14.4 JPQL 字符串构建

QueryGenerator 是渲染管线的核心输出阶段。它实现了 Visitor 模式，按固定顺序遍历各 Manager 持有的表达式树，将每个节点转化为对应的 JPQL 片段，最终拼接为完整的查询字符串。

QueryGenerator 渲染时序 — 各子句按序输出
sequenceDiagram participant Caller as getQueryString() participant QG as QueryGenerator participant SM as SelectManager participant JM as JoinManager participant WM as WhereManager participant GM as GroupByManager participant HM as HavingManager participant OM as OrderByManager participant PM as ParameterManager Caller->>QG: buildQueryString() QG->>SM: renderSelect(sb) SM-->>QG: "SELECT customer_1.name, SUM(o.amount)" QG->>JM: renderFrom(sb) JM-->>QG: "FROM Order o LEFT JOIN o.customer customer_1" QG->>WM: renderWhere(sb) WM-->>QG: "WHERE o.status = :param_0" QG->>GM: renderGroupBy(sb) GM-->>QG: "GROUP BY customer_1.name" QG->>HM: renderHaving(sb) HM-->>QG: "HAVING SUM(o.amount) > :param_1" QG->>OM: renderOrderBy(sb) OM-->>QG: "ORDER BY customer_1.name ASC" QG->>PM: collectParameters() PM-->>QG: {param_0: ACTIVE, param_1: 1000} QG-->>Caller: 完整 JPQL + 参数 Map

别名管理

AliasManager 在渲染过程中负责分配和解析别名。隐式 JOIN 的别名由框架自动生成，避免与用户定义的别名冲突。

别名类型命名规则示例

隐式 JOIN 别名路径名 + 递增序号customer → customer_1
子查询别名sub_ + 序号sub_1, sub_2

参数收集

渲染过程中，ParameterManager 负责收集所有绑定参数。参数名由框架自动生成（param_0, param_1, ...），确保唯一性。最终通过 query.setParameter() 绑定到 JPA Query 对象。

// 内部渲染结果示例

// JPQL 字符串:
String jpql =
  "SELECT customer_1.name, SUM(o.amount) " +
  "FROM Order o " +
  "LEFT JOIN o.customer customer_1 " +
  "WHERE o.status = :param_0 " +
  "  AND customer_1.country IN (:param_1, :param_2) " +
  "GROUP BY customer_1.name " +
  "HAVING SUM(o.amount) > :param_3 " +
  "ORDER BY customer_1.name ASC";

// 参数 Map:
Map<String, Object> params = Map.of(
    "param_0", Status.ACTIVE,
    "param_1", "CN",
    "param_2", "US",
    "param_3", 1000
);

// 执行:
Query query = em.createQuery(jpql);
params.forEach(query::setParameter);
List results = query.getResultList();

14.5 高级特性的 SQL 降级

标准 JPQL 的表达能力有限 — 它不支持 CTE、窗口函数、LATERAL JOIN、VALUES 子句等高级 SQL 特性。当用户使用这些特性时，Blaze-Persistence 必须绕过标准 JPQL，直接生成 provider-native 查询。

SQL Function 注入机制

Blaze-Persistence 利用 Hibernate 的 SQL Function 扩展点，将原生 SQL 片段"伪装"为 JPQL 函数调用。Hibernate 解析 HQL 时，遇到这些自定义函数会直接输出对应的 SQL 片段。

CTE 渲染详解

CTE（公用表表达式）是 SQL 降级的典型场景。Blaze-Persistence 将 CTE 渲染为 WITH 子句，通过 SQL Function 注入到最终查询中。

// 用户代码 — 使用 CTE
CriteriaBuilder<Tuple> cb = cbf.create(em, MyCte.class)
    .with(MyCte.class)
        .from(Order.class, "o")
        .bind("id").select("o.id")
        .bind("totalAmount").select("SUM(o.amount)")
        .groupBy("o.customer.id")
    .end()
    .select("myCte.id")
    .select("myCte.totalAmount");

// 内部渲染结果（Hibernate 最终 SQL）:
// WITH MyCte(id, totalAmount) AS (
//     SELECT o.id, SUM(o.amount)
//     FROM orders o
//     GROUP BY o.customer_id
// )
// SELECT m.id, m.totalAmount
// FROM MyCte m

14.6 查询计划缓存交互

Blaze-Persistence 生成的 JPQL 字符串最终由 Hibernate 解析执行。Hibernate 内部维护了一个查询计划缓存（Query Plan Cache），用于缓存已解析的查询 AST 和 SQL 翻译结果。渲染管线的输出质量直接影响缓存的命中率。

缓存友好 vs 缓存不友好

渲染管线生成的查询字符串是否稳定，直接决定了缓存命中率。参数化的查询模板是缓存友好的，而动态拼接的字符串会导致缓存污染。

缓存友好 vs 缓存不友好的渲染

提高缓存命中率的策略

1

固定查询模板

尽量让查询的 WHERE 条件数量和结构保持稳定。对可选条件使用 1=1 AND ... 或空条件占位技术，避免生成不同数量条件的查询变体。

2

使用参数而非字面量

永远通过 :param 绑定参数，不要将值直接嵌入查询字符串。WHERE id = 42 和 WHERE id = 43 是两个不同的缓存条目，但 WHERE id = :param_0 始终只有一个。

3

控制 IN 子句参数数量

Hibernate 对 IN(:params) 列表中参数数量不同的查询生成不同的缓存条目。Blaze-Persistence 可通过参数填充（padding）将参数列表长度归一化到 2 的幂次（如 3 个参数填充为 4 个）。

4

调整缓存大小

根据应用查询种类数量调整 plan_cache_max_size，默认值 2048 适用于大多数应用。查询种类超过 500 种时应考虑增大。

💡 Blaze-Persistence 的优势：由于 Blaze-Persistence 统一使用参数绑定而非字符串拼接，其生成的查询天然具备较高的缓存友好性。只要避免过度动态的条件组合，缓存命中率通常可以保持在 90% 以上。

陷阱 1：复杂嵌套子查询渲染 Bug

当查询包含三层或更深的嵌套子查询时，别名解析可能出现交叉引用错误。这是表达式树遍历中的已知边界情况。

⚠️ 嵌套限制：超过 3 层的子查询嵌套可能触发别名解析错误。建议将深层嵌套的子查询重构为 CTE，既能规避渲染 Bug，也能提升查询可读性。

陷阱 2：Provider 特定的渲染差异

不同 JPA Provider 对同一 JPQL 的解析行为存在差异，特别是在边界情况下。

陷阱 3：查询字符串长度问题

复杂的查询（大量 OR 条件、多级嵌套、长 IN 列表）可能生成超长的查询字符串，导致以下问题：

  • Hibernate 解析耗时显著增长（与字符串长度近似线性相关）
  • 部分数据库对 SQL 长度有硬性限制（如 Oracle 的 64KB）
  • 查询计划缓存中单条目占用过大内存
  • 日志输出不完整，难以调试

陷阱 4：表达式树内存开销

每个表达式节点都是 Java 对象。对于极端复杂的查询（如动态生成数百个 OR 条件的搜索引擎），表达式树本身的内存占用不可忽视。

生产案例：动态搜索的表达式膨胀

某电商平台的高级搜索功能支持 20+ 个可选过滤条件，每个条件有多种运算符。测试发现，当用户同时使用 15 个条件时，表达式树包含超过 200 个节点，getQueryString() 耗时从正常的 0.1ms 上升到 8ms。解决方案：将复杂筛选拆分为多个预定义查询模板，通过策略模式组合。

📝 本章小结

  • 渲染管线六阶段：API 调用 → ExpressionFactory 解析 → 表达式树 → QueryGenerator 遍历 → 渲染模式选择 → JPA Provider 执行。
  • ExpressionFactory 通过 JPQL.Next 解析器将字符串转化为类型化的表达式节点（PathExpression、FunctionExpression、SubqueryExpression 等）。
  • Manager 分治：AbstractCommonQueryBuilder 通过 SelectManager、JoinManager、WhereManager 等组件分别管理查询的各个子句。
  • JoinNode 树负责 JOIN 去重，确保同一关联路径只生成一条 JOIN 子句。
  • QueryGenerator 以 Visitor 模式遍历表达式树，按 SELECT → FROM → WHERE → GROUP BY → HAVING → ORDER BY 的顺序输出 JPQL 字符串。
  • SQL 降级通过 Hibernate SQL Function 扩展点实现，将 CTE / 窗口函数等高级特性注入为原生 SQL 片段。
  • 缓存友好的查询应使用参数绑定而非字面量，保持查询模板稳定以提高 Hibernate 查询计划缓存命中率。
  • 调试渲染问题应遵循管线前端到后端的排查顺序：JPQL → SQL → 参数绑定 → 执行结果。
