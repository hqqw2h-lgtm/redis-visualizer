第 3 章 · CriteriaBuilder 查询引擎

CriteriaBuilder · WHERE · ORDER BY · 表达式体系 · 动态查询 · JPQL 生成

CriteriaBuilder 是 Blaze-Persistence 的核心入口：它以流式 API 构建类型安全的 JPQL 查询，最终由底层 JPA 提供者翻译为原生 SQL。 本章从接口继承体系出发，逐一拆解 SELECT、WHERE、ORDER BY、表达式、动态拼接与 JPQL 生成管线，用 12 张 Mermaid 图可视化每一个关键步骤。

  • CriteriaBuilder 接口继承体系与 FullQueryBuilder 角色
  • SELECT / WHERE / ORDER BY 子句的流式构建方法
  • RestrictionBuilder 谓词树与 JPA Criteria 对比
  • 表达式类型体系：路径、函数、子查询、CASE WHEN
  • 运行时动态查询拼接策略与参数化查询
  • JPQL 生成管线与调试技巧

3.1 CriteriaBuilder API 全景

Blaze-Persistence 的查询 API 围绕一组精心设计的接口层层继承。理解这棵继承树，才能正确使用链式调用的每一步返回值。

3.2 SELECT 子句构建

SELECT 子句决定查询返回的投影字段。Blaze-Persistence 支持路径表达式、聚合函数、CASE 表达式以及子查询作为 SELECT 项。

代码示例

路径 + 聚合投影

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .select("o.customer.name", "customerName")
    .select("SUM(o.amount)",   "totalAmount")
    .select("COUNT(o.id)",     "orderCount")
    .groupBy("o.customer.name");

// JPQL: SELECT o.customer.name AS customerName,
//              SUM(o.amount) AS totalAmount,
//              COUNT(o.id) AS orderCount
//       FROM Order o
//       GROUP BY o.customer.name

CASE WHEN 表达式

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .select("o.id")
    .selectCase("statusLabel")
        .when("o.status").eqExpression("'SHIPPED'")
            .thenExpression("'已发货'")
        .when("o.status").eqExpression("'PENDING'")
            .thenExpression("'待处理'")
        .otherwise("'未知'")
    .select("o.amount");

// JPQL: SELECT o.id,
//              CASE WHEN o.status = 'SHIPPED' THEN '已发货'
//                   WHEN o.status = 'PENDING' THEN '待处理'
//                   ELSE '未知' END AS statusLabel,
//              o.amount
//       FROM Order o

3.3 WHERE 条件与谓词

WHERE 子句由 RestrictionBuilder 链式构建。每次调用 where("path") 返回一个 RestrictionBuilder，支持 eq()、gt()、like() 等操作符。多个条件默认以 AND 组合。

动态 WHERE 构建

CriteriaBuilder<Cat> cb = cbf.create(em, Cat.class, "c");

if (nameFilter != null) {
    cb.where("c.name").like().value("%" + nameFilter + "%").noEscape();
}
if (minAge != null) {
    cb.where("c.age").ge(minAge);
}
if (statuses != null && !statuses.isEmpty()) {
    cb.where("c.status").in(statuses);
}

return cb.getResultList();
💡 动态拼接的优势：不需要像 JPA Criteria 那样预先构建 Predicate[]， 直接按条件追加 where() 即可，框架自动以 AND 组合。这使得条件查询代码更加直观。

3.5 表达式体系

Blaze-Persistence 内部维护了一套表达式 AST（抽象语法树），每个 select()、where()、orderBy() 中的字符串都会被解析为对应的表达式节点。

常用函数表达式

// COUNT — 聚合计数

3.6 动态查询拼接

在实际业务中，查询条件往往根据用户输入或业务逻辑动态变化。Blaze-Persistence 的流式 API 天然支持条件拼接——只要不调用某个 where()，该条件就不会出现在最终 JPQL 中。

图 3-10 · 动态查询决策树
flowchart TD START["创建 CriteriaBuilder"] --> CHECK_NAME{"nameFilter != null?"} CHECK_NAME -->|是| ADD_NAME["where('c.name').like(nameFilter)"] CHECK_NAME -->|否| CHECK_AGE ADD_NAME --> CHECK_AGE{"minAge != null?"} CHECK_AGE -->|是| ADD_AGE["where('c.age').ge(minAge)"] CHECK_AGE -->|否| CHECK_STATUS ADD_AGE --> CHECK_STATUS{"statuses 非空?"} CHECK_STATUS -->|是| ADD_STATUS["where('c.status').in(statuses)"] CHECK_STATUS -->|否| CHECK_SORT ADD_STATUS --> CHECK_SORT{"sortField != null?"} CHECK_SORT -->|是| ADD_SORT["orderByAsc(sortField)"] CHECK_SORT -->|否| EXEC ADD_SORT --> EXEC["getResultList()"] style START fill:#1e3a5f,stroke:#58a6ff style CHECK_NAME fill:#2d1f4e,stroke:#bc8cff style CHECK_AGE fill:#2d1f4e,stroke:#bc8cff style CHECK_STATUS fill:#2d1f4e,stroke:#bc8cff style CHECK_SORT fill:#2d1f4e,stroke:#bc8cff style EXEC fill:#1a3a2a,stroke:#3fb950 
public List<Cat> searchCats(CatSearchCriteria criteria) {
    CriteriaBuilder<Cat> cb = cbf.create(em, Cat.class, "c");

    // 动态 WHERE
    if (criteria.getName() != null) {
        cb.where("c.name").like()
          .value("%" + criteria.getName() + "%").noEscape();
    }
    if (criteria.getMinAge() != null) {
        cb.where("c.age").ge(criteria.getMinAge());
    }
    if (criteria.getMaxAge() != null) {
        cb.where("c.age").le(criteria.getMaxAge());
    }
    if (criteria.getStatuses() != null) {
        cb.where("c.status").in(criteria.getStatuses());
    }
    if (criteria.getOwnerId() != null) {
        cb.where("c.owner.id").eq(criteria.getOwnerId());
    }

    // 动态排序
    if (criteria.getSortField() != null) {
        if (criteria.isSortAsc()) {
            cb.orderByAsc("c." + criteria.getSortField());
        } else {
            cb.orderByDesc("c." + criteria.getSortField());
        }
    } else {
        cb.orderByAsc("c.id");
    }

    // 分页
    if (criteria.getPageSize() > 0) {
        cb.setFirstResult(criteria.getPage() * criteria.getPageSize());
        cb.setMaxResults(criteria.getPageSize());
    }

    return cb.getResultList();
}

💡 最佳实践：对于高频查询，尽量使用 :param 参数化表达式而非内联值。 这样 JPA 提供者（Hibernate）能复用同一份查询计划缓存，显著减少 JPQL → SQL 的解析开销。

3.8 陷阱与限制

使用 CriteriaBuilder 时有一些常见的陷阱需要注意。本节列出最常见的问题和解决方案。

⚠️ 隐式 JOIN 陷阱：路径 c.owner.address.city 会自动生成 2 个 JOIN。 如果多个 where() 使用同一前缀路径（如 c.owner.name 和 c.owner.age）， Blaze-Persistence 会智能复用同一个 JOIN，但跨不同关联路径时可能产生意料之外的笛卡尔积。

常见陷阱一览

陷阱症状解决方案
隐式 JOIN 重复 查询结果行数比预期多 使用 DISTINCT 或显式 JOIN
参数类型不匹配 运行时 IllegalArgumentException 确保 eq() 的参数类型与实体字段类型一致
LIKE 未转义 通配符 %、_ 被当作匹配符 使用 .like().value(val).noEscape() 或手动转义
NULL 比较 where("c.name").eq(null) 不生效 使用 .isNull() 代替 .eq(null)
集合 IN 空列表 .in(emptyList) 抛异常 调用前判空，或使用 1=0 短路
查询计划缓存膨胀 Hibernate QueryPlanCache 内存占用过高 参数化查询而非内联值

EclipseLink 注意事项

  • NULLS FIRST/LAST 通过 CASE WHEN 模拟
  • 部分 Blaze-Persistence 扩展函数需额外注册
  • CTE 支持有限
  • 批量 INSERT-SELECT 行为不同

Hibernate 优势

  • 原生 NULLS FIRST/LAST（PostgreSQL）
  • 完整的 CTE 和窗口函数支持
  • Blaze-Persistence 默认优化路径
  • 查询计划缓存更成熟

📝 经验法则：如果应用有超过 500 种不同的查询结构（去参数化后）， 应该检查是否存在内联值导致的缓存膨胀。使用 getQueryString() 输出并归一化对比， 识别可参数化的查询。

📝 本章小结

  • CriteriaBuilder<T> 继承自 FullQueryBuilder，汇聚 SELECT / WHERE / ORDER BY / 分页等全部查询能力
  • SELECT 子句支持路径、聚合、CASE WHEN 和子查询四种表达式类型
  • RestrictionBuilder 链式 API 大幅简化 WHERE 条件构建，相比 JPA Criteria 减少 50%+ 代码量
  • ORDER BY 支持精细的 NULLS FIRST / LAST 控制；LIMIT / OFFSET 通过 setFirstResult / setMaxResults 实现
  • 表达式 AST 体系支持路径、函数、聚合、参数、算术和 CASE 等节点类型
  • 动态查询通过条件式 where() 调用天然实现，无需预构建 Predicate 数组
  • getQueryString() 是调试的第一手段；配合 Hibernate SQL 日志可追踪完整管线
  • 注意隐式 JOIN 重复、NULL 比较、空 IN 列表和查询计划缓存膨胀等常见陷阱
