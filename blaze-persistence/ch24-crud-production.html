24.1 多租户隔离 — ViewFilter

SaaS 应用最核心的安全需求是租户数据隔离。传统做法是在每个 Repository 方法中手动加 WHERE tenant_id = ?——这种方式脆弱、易遗漏。Blaze-Persistence 的 @ViewFilter 提供声明式的全局过滤器，在 Entity View 层面自动注入租户条件。

租户上下文（ThreadLocal）

public class TenantContext {
    private static final ThreadLocal<String> CURRENT = new ThreadLocal<>();

    public static String current() {
        String tenant = CURRENT.get();
        if (tenant == null) throw new IllegalStateException("No tenant set");
        return tenant;
    }

    public static void set(String tenantId) { CURRENT.set(tenantId); }
    public static void clear() { CURRENT.remove(); }
}

租户拦截器

@Component
public class TenantInterceptor implements HandlerInterceptor {
    @Override
    public boolean preHandle(HttpServletRequest request,
                             HttpServletResponse response,
                             Object handler) {
        String tenantId = request.getHeader("X-Tenant-Id");
        if (tenantId == null) {
            response.setStatus(400);
            return false;
        }
        TenantContext.set(tenantId);
        return true;
    }

    @Override
    public void afterCompletion(HttpServletRequest request,
                                HttpServletResponse response,
                                Object handler, Exception ex) {
        TenantContext.clear();
    }
}

Service 层应用过滤器

@Service
@RequiredArgsConstructor
public class OrderQueryService {

    private final CriteriaBuilderFactory cbf;
    private final EntityViewManager evm;
    private final EntityManager em;

    public List<OrderListView> listOrders() {
        EntityViewSetting<OrderListView, CriteriaBuilder<OrderListView>> setting =
            EntityViewSetting.create(OrderListView.class);
        setting.addAttributeFilter("tenantFilter", TenantContext.current());

        CriteriaBuilder<OrderListView> cb = evm.applySetting(setting,
            cbf.create(em, Order.class));

        return cb.getResultList();
    }
}
自动注入原理：@ViewFilter 的 value 是 JPQL 表达式，:tenantId 是命名参数。当 addAttributeFilter("tenantFilter", value) 被调用时，Blaze-Persistence 将该条件合并到 WHERE 子句中——开发者无需手动拼接条件，也不会遗漏。

24.2 软删除 — ViewFilter

生产系统几乎不做物理删除。软删除通过 deletedAt 字段标记已删除记录，查询时自动排除。@ViewFilter 让这一逻辑对业务代码完全透明。

软删除实现

@Service
@RequiredArgsConstructor
public class OrderLifecycleService {

    private final CriteriaBuilderFactory cbf;
    private final EntityManager em;

    @Transactional
    public void softDelete(Long orderId) {
        int updated = cbf.update(em, Order.class, "o")
            .set("o.deletedAt", Instant.now())
            .set("o.deletedBy", SecurityContext.currentUser())
            .where("o.id").eq(orderId)
            .where("o.tenantId").eq(TenantContext.current())
            .where("o.deletedAt").isNull()
            .executeUpdate();

        if (updated == 0) {
            throw new OrderNotFoundException(orderId);
        }
    }

    @Transactional
    public void restore(Long orderId) {
        cbf.update(em, Order.class, "o")
            .setExpression("o.deletedAt", "NULL")
            .setExpression("o.deletedBy", "NULL")
            .where("o.id").eq(orderId)
            .where("o.tenantId").eq(TenantContext.current())
            .where("o.deletedAt").isNotNull()
            .executeUpdate();
    }
}

24.3 审计 — EntityViewListener

每条记录的 createdAt、createdBy、updatedAt、updatedBy 必须自动填充。Blaze-Persistence 的 EntityViewListener 在 Entity View 持久化/更新前自动注入这些字段——无需手动赋值，无需 JPA @PrePersist 回调。

审计监听器

@ViewListener
public class AuditViewListener implements PrePersistEntityListener<Object>,
                                           PreUpdateEntityListener<Object> {
    @Override
    public void onPrePersist(Object view) {
        if (view instanceof Auditable a) {
            a.setCreatedAt(Instant.now());
            a.setCreatedBy(SecurityContext.currentUser());
            a.setUpdatedAt(Instant.now());
            a.setUpdatedBy(SecurityContext.currentUser());
        }
    }

    @Override
    public void onPreUpdate(Object view) {
        if (view instanceof Auditable a) {
            a.setUpdatedAt(Instant.now());
            a.setUpdatedBy(SecurityContext.currentUser());
        }
    }
}

24.4 Keyset 分页 — 高性能列表

OFFSET 分页在数据量超过 10 万行后性能急剧下降——数据库必须扫描并丢弃前 N 行。Keyset 分页通过 WHERE (created_at, id) < (?, ?) 直接跳到目标位置，翻页耗时恒定 O(1)。

Controller 实现

@RestController
@RequestMapping("/api/orders")
@RequiredArgsConstructor
public class OrderController {

    private final CriteriaBuilderFactory cbf;
    private final EntityViewManager evm;
    private final EntityManager em;

    @GetMapping
    public KeysetPageResult<OrderListView> listOrders(
            @RequestParam(required = false) String afterKey,
            @RequestParam(defaultValue = "20") int size) {

        KeysetPage keyset = afterKey != null
            ? deserializeKeyset(afterKey) : null;

        EntityViewSetting<OrderListView,
            PaginatedCriteriaBuilder<OrderListView>> setting =
                keyset != null
                    ? EntityViewSetting.create(OrderListView.class, keyset, size)
                    : EntityViewSetting.create(OrderListView.class, 0, size);

        setting.addAttributeFilter("activeOnly");
        setting.addAttributeFilter("tenantFilter", TenantContext.current());

        PaginatedCriteriaBuilder<OrderListView> cb = evm.applySetting(setting,
            cbf.create(em, Order.class)
               .orderByDesc("createdAt")
               .orderByDesc("id"));

        PagedList<OrderListView> page = cb.getResultList();

        return new KeysetPageResult<>(
            page,
            page.getTotalSize(),
            serializeKeyset(page.getKeysetPage()),
            page.getTotalPages()
        );
    }

        if (keysetPage == null || keysetPage.getHighest() == null) return null;
        try {
            return Base64.getUrlEncoder().encodeToString(

        } catch (Exception e) {
            return null;
        }
    }

        // 反序列化 keyset 游标
        byte[] bytes = Base64.getUrlDecoder().decode(encoded);

24.5 批量操作 — 订单归档

历史订单如果一直留在热表中，查询性能会随数据量线性下降。生产系统需要定期将已完成/已取消的旧订单归档到冷表，然后从热表中删除。Blaze-Persistence 的 INSERT...SELECT + 批量 DELETE 可以在单条 SQL 中完成这项工作。

归档服务

@Service
@RequiredArgsConstructor
@Slf4j
public class OrderArchiveService {

    private final CriteriaBuilderFactory cbf;
    private final EntityManager em;

    @Transactional
    public ArchiveResult archiveOldOrders(LocalDateTime before) {
        // Step 1: INSERT INTO order_archive SELECT FROM orders
        int archived = cbf.insert(em, OrderArchive.class)
            .from(Order.class, "o")
            .bind("id").select("o.id")
            .bind("customerId").select("o.customer.id")
            .bind("status").select("o.status")
            .bind("totalAmount").select("o.totalAmount")
            .bind("originalCreatedAt").select("o.createdAt")
            .bind("archivedAt").select("CURRENT_TIMESTAMP")
            .bind("archivedBy").select("'SYSTEM'")
            .where("o.createdAt").lt(before)
            .where("o.status").in(OrderStatus.COMPLETED, OrderStatus.CANCELLED)
            .where("o.deletedAt").isNull()
            .executeUpdate();

        // Step 2: DELETE archived orders from hot table
        int deleted = cbf.delete(em, Order.class, "o")
            .where("o.createdAt").lt(before)
            .where("o.status").in(OrderStatus.COMPLETED, OrderStatus.CANCELLED)
            .where("o.deletedAt").isNull()
            .executeUpdate();

        log.info("Archived {} orders, deleted {} from hot table", archived, deleted);
        return new ArchiveResult(archived, deleted);
    }
}

public record ArchiveResult(int archived, int deleted) {}

WHERE  o.created_at < '2024-10-15T00:00:00'
AND    o.status IN ('COMPLETED', 'CANCELLED')
AND    o.deleted_at IS NULL;
事务一致性：INSERT 和 DELETE 必须在同一个事务中执行。如果 DELETE 在 INSERT 之前执行（或分属不同事务），数据会丢失。@Transactional 确保两步操作的原子性。

24.6 乐观锁 — @Version

当两个用户同时编辑同一订单时，后提交的更新会覆盖先提交的修改——这就是"丢失更新"问题。JPA 的 @Version 字段配合 Blaze-Persistence 的 Updatable Entity View 可以自动检测并拒绝冲突更新。

Controller 层冲突处理

@PutMapping("/api/orders/{id}")
public ResponseEntity<?> updateOrder(@PathVariable Long id,
                                      @RequestBody OrderUpdateRequest request) {
    try {
        OrderDetailView result = orderService.updateOrder(id, request);
        return ResponseEntity.ok(result);
    } catch (OptimisticLockException e) {
        return ResponseEntity.status(HttpStatus.CONFLICT)
            .body(Map.of(
                "error", "CONFLICT",
                "message", "订单已被其他用户修改，请刷新后重试",
                "currentVersion", orderService.getCurrentVersion(id)
            ));
    }
}

@Component
@RequiredArgsConstructor
public class EntityViewMetrics {

    private final MeterRegistry meterRegistry;

    public <T> T timeViewRendering(String viewName, Supplier<T> operation) {
        Timer.Sample sample = Timer.start(meterRegistry);
        try {
            T result = operation.get();
            sample.stop(Timer.builder("blaze.entityview.render")
                .tag("view", viewName)
                .tag("status", "success")
                .description("Entity View rendering time")
                .register(meterRegistry));
            return result;
        } catch (Exception e) {
            sample.stop(Timer.builder("blaze.entityview.render")
                .tag("view", viewName)
                .tag("status", "error")
                .register(meterRegistry));
            throw e;
        }
    }
}

Q2: 何时用 CriteriaBuilder，何时用 Spring Data 方法名查询？

// ✅ 方法名查询 — 简单条件
List<OrderListView> findByStatus(OrderStatus status);
Page<OrderListView> findByCustomerId(Long customerId, Pageable pageable);

// ✅ CriteriaBuilder — 动态条件组合
public List<OrderListView> search(OrderSearchCriteria criteria) {
    CriteriaBuilder<Order> cb = cbf.create(em, Order.class, "o");
    if (criteria.getStatus() != null)
        cb.where("o.status").eq(criteria.getStatus());
    if (criteria.getMinAmount() != null)
        cb.where("o.totalAmount").ge(criteria.getMinAmount());
    if (criteria.getCustomerName() != null)
        cb.where("o.customer.name").like()
          .value("%" + criteria.getCustomerName() + "%").noEscape();
    // ...
    return evm.applySetting(
        EntityViewSetting.create(OrderListView.class), cb)
        .getResultList();
}

// ✅ CriteriaBuilder — CTE / 窗口函数 / 批量 DML
// Spring Data 方法名不支持这些高级特性
场景推荐方式理由
固定条件查询Spring Data 方法名零样板代码
2-3 个可选过滤条件CriteriaBuilder动态组合更灵活
CTE / 窗口函数CriteriaBuilder方法名不支持
批量 DMLCriteriaBuilderINSERT/UPDATE/DELETE 批量
Keyset 分页CriteriaBuilder 或 EntityViewRepository两者均支持

Q3: 连接池如何调优？

# application-prod.yml
spring:
  datasource:
    hikari:
      # 最大连接数 = CPU 核心数 * 2 + 磁盘数
      # 4 核 CPU + 1 SSD → 4 * 2 + 1 = 9 ≈ 10
      maximum-pool-size: 10
      minimum-idle: 5

      # 等待连接超时（超过则抛异常）
      connection-timeout: 5000

      # 连接最大存活时间（避免数据库端超时）
      max-lifetime: 1800000   # 30 分钟

      # 空闲连接超时
      idle-timeout: 600000    # 10 分钟

      # 泄漏检测阈值
      leak-detection-threshold: 30000  # 30 秒
常见错误：把 maximum-pool-size 设置为 50 或 100。过多连接会导致数据库上下文切换开销增大，反而降低吞吐量。PostgreSQL 官方建议单实例不超过 CPU * 2 + disk。

Q4: 二级缓存与 Entity View 如何配合？

@Entity
@Cacheable  // 开启二级缓存
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
@Table(name = "products")
public class Product {
    @Id private Long id;
    private String name;
    private BigDecimal price;
    private String category;
}

// Entity View 查询时，如果底层 Entity 命中二级缓存，
// Hibernate 不会发送 SQL 查询
@EntityView(Product.class)
public interface ProductSummaryView {
    @IdMapping Long getId();
    String getName();
    BigDecimal getPrice();
}

spring:

          use_second_level_cache: true

缓存与 ViewFilter 的关系：查询缓存会缓存查询 SQL + 参数的结果 ID 列表。由于 @ViewFilter 会改变 WHERE 条件，不同租户的查询会产生不同的缓存键——缓存隔离是自动的。

24.13 陷阱与限制

陷阱 1: ViewFilter 遗漏

// ❌ 危险：忘记为新的 Entity View 添加 tenantFilter
@EntityView(Order.class)
public interface OrderExportView {
    @IdMapping Long getId();
    // ... 没有 @ViewFilter → 可以查到所有租户的数据！
}

// ✅ 正确：所有 Entity View 都声明 tenantFilter
@EntityView(Order.class)
@ViewFilter(name = "tenantFilter", value = "tenantId = :tenantId")
public interface OrderExportView {
    @IdMapping Long getId();
    // ...
}

陷阱 2: Keyset 分页缺少唯一列

// ❌ 错误：仅按 createdAt 排序（可能有重复值）
cbf.create(em, Order.class)
   .orderByDesc("createdAt");
// 如果两行 createdAt 相同 → 数据可能重复或丢失

// ✅ 正确：添加唯一列作为 tiebreaker
cbf.create(em, Order.class)
   .orderByDesc("createdAt")
   .orderByDesc("id");         // id 是唯一的

陷阱 3: 定时任务中的 TenantContext

// ❌ 错误：定时任务未设置 TenantContext
@Scheduled(cron = "0 0 3 * * SUN")
public void weeklyCleanup() {
    orderService.cleanupExpired();  // TenantContext.current() → NPE!
}

// ✅ 正确：遍历所有租户逐个处理
@Scheduled(cron = "0 0 3 * * SUN")
public void weeklyCleanup() {
    List<String> tenants = tenantService.getAllTenantIds();
    for (String tenant : tenants) {
        try {
            TenantContext.set(tenant);
            orderService.cleanupExpired();
        } finally {
            TenantContext.clear();
        }
    }
}

陷阱 4: 乐观锁与批量更新冲突

// ❌ 危险：批量 UPDATE 不走 @Version 机制
cbf.update(em, Order.class, "o")
   .set("o.status", OrderStatus.CANCELLED)
   .where("o.createdAt").lt(cutoff)
   .executeUpdate();
// 不会检查 version → 可能覆盖并发修改

// ✅ 如果需要版本检查，必须逐条处理
// 批量操作适用于没有并发冲突的场景（如归档、清理）

陷阱 5: EntityViewListener 中的事务边界

// ❌ 错误：在 Listener 中发起新的数据库查询
@ViewListener
public class BadListener implements PrePersistEntityListener<Object> {
    @Inject EntityManager em;

    @Override
    public void onPrePersist(Object view) {
        // 在 flush 过程中查询 → 可能导致死锁或脏读
        em.createQuery("...").getResultList();
    }
}

// ✅ 正确：Listener 仅做字段赋值，不做数据库操作
@ViewListener
public class GoodListener implements PrePersistEntityListener<Object> {
    @Override
    public void onPrePersist(Object view) {
        if (view instanceof Auditable a) {
            a.setCreatedAt(Instant.now());
            a.setCreatedBy(SecurityContext.currentUser());
        }
    }
}

本章小结

  • 多租户隔离：@ViewFilter("tenantFilter") 在 Entity View 层声明租户条件，配合 TenantInterceptor 和 TenantContext 实现请求级自动隔离，杜绝人工遗漏。
  • 软删除：@ViewFilter("activeOnly") 透明过滤已删除记录，cbf.update() 设置 deletedAt 实现软删除，管理员可跳过过滤器查看全部数据。
  • 审计：AuditViewListener 在 onPrePersist / onPreUpdate 回调中自动填充 createdAt、createdBy、updatedAt、updatedBy。
  • Keyset 分页：通过 WHERE (created_at, id) < (?, ?) 游标实现恒定 O(1) 翻页，百万行列表仍在 7ms 以内。
  • 批量归档：INSERT...SELECT + 批量 DELETE 在两条 SQL 中完成历史订单的归档和清理。
  • 乐观锁：JPA @Version 配合 Updatable Entity View，在 WHERE id = ? AND version = ? 条件下自动检测并发冲突。
  • 集成测试：Testcontainers 启动真实 PostgreSQL 容器，确保测试 SQL 方言与生产完全一致。
  • 可观测性：Hibernate Statistics + Micrometer + Actuator 构建三层指标体系，覆盖查询耗时、渲染指标、连接池健康。
  • Spring Data 集成：EntityViewRepository 提供方法名查询 + Keyset 分页 + 自动投影的完整能力。
  • 项目结构：表示层 → 应用层 → 领域层 → 基础设施层的四层分包，职责清晰、可测试。
