9.1 窗口函数概念

普通聚合函数（GROUP BY）将多行折叠为一行；而窗口函数在保留每一行的同时， 在一个逻辑"窗口"上执行计算。这意味着你可以在同一行中同时看到明细数据和聚合结果。

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Employee.class, "e")
    .select("e.name")
    .select("e.department.name")
    .select("e.salary")
    .selectWindow("ROW_NUMBER")
        .partitionBy("e.department.name")
        .orderBy("e.salary", false)  // DESC
    .end()
    .orderByAsc("e.department.name")
    .orderByAsc("e.salary");

SELECT e.name,
       d.name,
       e.salary,
       ROW_NUMBER() OVER (
           PARTITION BY d.name
           ORDER BY e.salary DESC
       ) AS rank_col
FROM Employee e
JOIN Department d ON e.department_id = d.id
ORDER BY d.name, e.salary

9.3 聚合窗口函数

任何标准聚合函数（SUM、COUNT、AVG、MIN、MAX） 都可以配合 OVER 子句变成窗口函数。最常见的场景是累计合计和滑动平均。

累计合计代码示例

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .select("o.orderDate")
    .select("o.amount")
    // 累计合计: SUM(amount) OVER (ORDER BY orderDate)
    .selectWindow("SUM", "o.amount")
        .orderBy("o.orderDate", true)
        .rows().betweenUnboundedPreceding().andCurrentRow()
    .end()
    .orderByAsc("o.orderDate");

生成 SQL：

SELECT o.order_date,
       o.amount,
       SUM(o.amount) OVER (
           ORDER BY o.order_date
           ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
       ) AS running_total
FROM orders o
ORDER BY o.order_date

3 行滑动平均代码

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(StockPrice.class, "s")
    .select("s.tradeDate")
    .select("s.closePrice")
    .selectWindow("AVG", "s.closePrice")
        .orderBy("s.tradeDate", true)
        .rows().between(2, WindowFrameExclusionType.NO_OTHERS)
              .andCurrentRow()
    .end()
    .orderByAsc("s.tradeDate");
⚡ 帧类型提示：ROWS 按物理行号计算帧边界， RANGE 按值的逻辑范围计算。绝大多数场景使用 ROWS 即可； RANGE 适合处理日期间隔不均匀的时序数据。

JpqlFunction 接口

public interface JpqlFunction {
    // 函数是否需要参数
    boolean hasArguments();

    // 无参数时是否保留括号（如 CURRENT_TIMESTAMP vs NOW()）
    boolean hasParenthesesIfNoArguments();

    // 渲染 SQL 片段
    void render(FunctionRenderContext context);
}

示例：注册 PostgreSQL 的 array_agg 函数

public class ArrayAggFunction implements JpqlFunction {

    @Override
    public boolean hasArguments() {
        return true;
    }

    @Override
    public boolean hasParenthesesIfNoArguments() {
        return true;
    }

    @Override
    public void render(FunctionRenderContext context) {
        context.addChunk("array_agg(");
        context.addArgument(0);  // 渲染第一个参数
        if (context.getArgumentsSize() > 1) {
            context.addChunk(" ORDER BY ");
            context.addArgument(1);
        }
        context.addChunk(")");
    }
}

注册与使用

// 注册
CriteriaBuilderConfiguration config = Criteria.getDefault();
config.registerFunction(
    new JpqlFunctionGroup("array_agg", new ArrayAggFunction())
);
CriteriaBuilderFactory cbf = config.createCriteriaBuilderFactory(emf);

// 使用
CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Employee.class, "e")
    .select("e.department.name")
    .select("FUNCTION('array_agg', e.name)")
    .groupBy("e.department.name");

Simple CASE vs Searched CASE

类型SQL 语法特点
Simple CASE CASE status WHEN 'A' THEN ... WHEN 'B' THEN ... END 比较单个表达式的多个值（类似 switch）
Searched CASE CASE WHEN score >= 90 THEN ... WHEN score >= 80 THEN ... END 每个分支独立布尔条件（类似 if-else if）

CaseWhenBuilder API

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Employee.class, "e")
    .select("e.name")
    .select("e.score")
    .selectCase()
        .when("e.score").geExpression("90").thenExpression("'S级'")
        .when("e.score").geExpression("80").thenExpression("'A级'")
        .when("e.score").geExpression("70").thenExpression("'B级'")
        .when("e.score").geExpression("60").thenExpression("'C级'")
        .otherwiseExpression("'D级'")
    .end();

⚠️ 陷阱 1：WHERE 中不能使用窗口函数

// ❌ 错误：WHERE 中直接用窗口函数
cb.where("ROW_NUMBER() OVER (...)").le(10);  // 编译报错或运行时异常

// ✅ 正确：用子查询包裹
CriteriaBuilder<Tuple> sub = cbf.create(em, Tuple.class)
    .from(Employee.class, "e")
    .select("e.id")
    .select("e.name")
    .selectWindow("ROW_NUMBER")
        .orderBy("e.salary", false)
    .end();

// 外层查询过滤排名
CriteriaBuilder<Tuple> outer = cbf.create(em, Tuple.class)
    .fromSubquery(sub, "ranked")
    .select("ranked.name")
    .where("ranked.row_number").le(10);
⚠️ 陷阱 2：自定义函数须匹配 JPA Provider
Hibernate 和 EclipseLink 处理 FUNCTION() 表达式的方式不同。 确保你的 JpqlFunction 实现与实际使用的 JPA 提供者兼容。
⚠️ 陷阱 3：大分区性能问题
窗口函数在每个分区内进行计算。如果 PARTITION BY 产生的分区过大 （例如不分区时全表作为一个分区），数据库需要在内存中缓存大量行，可能导致严重性能问题。
窗口函数性能优化策略
① 索引优化
为 PARTITION BY 和 ORDER BY 列创建复合索引
CREATE INDEX idx_dept_salary ON employee(dept_id, salary)
② 控制分区大小
先 WHERE 过滤缩小数据集，再应用窗口函数
③ 限制帧范围
N PRECEDING 比 UNBOUNDED PRECEDING 高效
④ CTE 物化
复杂窗口计算用 CTE 分步执行

📋 本章小结

  • 窗口函数三要素：PARTITION BY（分区）、ORDER BY（排序）、FRAME（帧范围）共同定义每行的计算窗口
  • 排名函数：ROW_NUMBER 强制唯一、RANK 跳号、DENSE_RANK 不跳号、NTILE 均匀分组
  • 聚合窗口：SUM/AVG OVER 实现累计合计和滑动平均，FRAME 子句决定计算范围
  • 偏移函数：LAG/LEAD 引用前后行数据，FIRST_VALUE/LAST_VALUE 取分区边界值
  • 自定义函数：实现 JpqlFunction 接口，通过 JpqlFunctionGroup 按数据库方言注册不同实现
  • CASE 表达式：CaseWhenBuilder 提供类型安全的条件分支，NULLIF/COALESCE 是常用简化形式
  • 关键限制：窗口函数不能在 WHERE 中使用；大分区需注意性能；MySQL 8.0+ 才支持
