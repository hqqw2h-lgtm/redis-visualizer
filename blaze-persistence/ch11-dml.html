第 11 章 · 批量 DML

UPDATE · DELETE · INSERT-SELECT · ReturningResult · 缓存一致性

逐行遍历实体、修改属性、逐个 merge()——当你需要把 10 万行的状态从 ACTIVE 改为 ARCHIVED 时，这种做法意味着 10 万次 SQL UPDATE。批量 DML 一条语句搞定，但 JPA 的原生批量操作只有字符串拼接的 JPQL，毫无类型安全。Blaze-Persistence 用 CriteriaBuilder 的流式 API 把 UPDATE、DELETE、INSERT-SELECT 统统变成编译时可检查的类型安全构建器。

  • JPA 原生 executeUpdate() 的局限：字符串拼接、无编译检查、重构不安全
  • UPDATE CriteriaBuilder：set() / setExpression() 流式设置列值 + WHERE 过滤
  • DELETE CriteriaBuilder：带子查询条件的批量删除
  • INSERT-SELECT：跨表数据迁移与归档的类型安全方式
  • ReturningResult：获取受影响的行数据，以及各数据库对 RETURNING 子句的支持差异
  • 批量 DML 对 JPA 一级缓存的影响及解决方案
  • 7 个常见陷阱：实体监听器失效、版本列不自增、JOIN 在 UPDATE/DELETE 中的兼容性等

11.1 JPA 批量操作的局限

JPA 规范提供了 Query.executeUpdate() 来执行批量 UPDATE 和 DELETE，但它接受的是原始 JPQL 字符串。当实体字段重命名时，编译器无法发现断裂的查询；当 WHERE 条件复杂化时，字符串拼接变成噩梦。

Fig 11-1 · JPA 字符串 DML vs Blaze-Persistence 类型安全 DML
// JPA 原生批量 UPDATE — 字符串拼接
String jpql = "UPDATE Order o " +
    "SET o.status = :newStatus " +
    "WHERE o.status = :oldStatus " +
    "AND o.createdAt < :cutoff";
int count = em.createQuery(jpql)
    .setParameter("newStatus", Status.ARCHIVED)
    .setParameter("oldStatus", Status.COMPLETED)
    .setParameter("cutoff", cutoffDate)
    .executeUpdate();
// 字段名拼错？编译通过，运行时才爆炸
// Blaze-Persistence 类型安全 UPDATE
int count = cbf.update(em, Order.class)
    .set("status", Status.ARCHIVED)
    .where("status").eq(Status.COMPLETED)
    .where("createdAt").lt(cutoffDate)
    .executeUpdate();
// 字段名由编译器检查，IDE 可安全重构

11.2 UPDATE CriteriaBuilder

Blaze-Persistence 的 UPDATE 操作从 CriteriaBuilderFactory.update() 开始，返回一个 UpdateCriteriaBuilder。它支持 set() 设置字面值、setExpression() 设置表达式（如数学运算、函数调用），以及完整的 WHERE 子句过滤。

Fig 11-2 · UPDATE 查询构建管线
flowchart TD START["cbf.update(em, Entity.class)"] --> ALIAS["设置别名 .from(Entity.class, 'e')"] ALIAS --> SET["设置列值"] SET --> SET_LIT[".set('col', literalValue)"] SET --> SET_EXP[".setExpression('col', 'UPPER(e.name)')"] SET --> SET_SUB[".set('col', subqueryBuilder)"] SET_LIT --> WHERE["WHERE 条件"] SET_EXP --> WHERE SET_SUB --> WHERE WHERE --> W1[".where('field').eq(val)"] WHERE --> W2[".where('field').in(subquery)"] WHERE --> W3[".whereExists(subquery)"] W1 --> EXEC[".executeUpdate()"] W2 --> EXEC W3 --> EXEC EXEC --> SQL["生成 SQL:\nUPDATE table SET ... WHERE ..."] SQL --> DB["数据库执行\n返回 int affectedRows"] style START fill:#1e3a5f,stroke:#2563eb style EXEC fill:#2d1f4e,stroke:#7c3aed style DB fill:#1a3a2a,stroke:#22c55e

基础 UPDATE：设置字面值

// 将所有过期订单标记为 CANCELLED
int cancelled = cbf.update(em, Order.class, "o")
    .set("status", OrderStatus.CANCELLED)
    .set("cancelReason", "auto-expired")
    .where("expiresAt").le(LocalDateTime.now())
    .where("status").eq(OrderStatus.PENDING)
    .executeUpdate();

表达式 UPDATE：setExpression()

当需要基于现有值进行计算时，setExpression() 接受 JPQL 表达式字符串：

// 所有商品价格上调 10%
int updated = cbf.update(em, Product.class, "p")
    .setExpression("price", "p.price * 1.10")
    .setExpression("updatedAt", "CURRENT_TIMESTAMP")
    .where("category").eq(Category.ELECTRONICS)
    .executeUpdate();
💡 set() vs setExpression()：set("price", 100) 将 price 设为字面值 100；setExpression("price", "p.price * 1.10") 将 price 设为当前值的 1.1 倍。前者绑定参数，后者嵌入表达式。

带子查询的 UPDATE

// 将每个客户的 totalOrders 更新为实际订单数
int synced = cbf.update(em, Customer.class, "c")
    .set("totalOrders")
        .from(Order.class, "o")
        .select("COUNT(o)")
        .where("o.customer.id").eqExpression("c.id")
    .end()
    .executeUpdate();

11.3 DELETE CriteriaBuilder

DELETE 操作的 API 更简洁——不需要 set()，只有 WHERE 条件。cbf.delete(em, Entity.class) 返回 DeleteCriteriaBuilder，支持所有 CriteriaBuilder 的 WHERE 功能，包括子查询、EXISTS 和复合条件。

Fig 11-3 · DELETE 查询构建流程
flowchart LR DEL["cbf.delete(em, Entity.class)"] --> WHERE["WHERE 子句"] WHERE --> SIMPLE[".where('field').eq(val)"] WHERE --> EXISTS[".whereExists(subquery)"] WHERE --> IN[".where('field').in(subquery)"] SIMPLE --> EXEC[".executeUpdate()"] EXISTS --> EXEC IN --> EXEC EXEC --> SQL["DELETE FROM table WHERE ..."] SQL --> RESULT["int affectedRows"] style DEL fill:#3a1515,stroke:#f85149 style EXEC fill:#2d1f4e,stroke:#7c3aed style RESULT fill:#1a3a2a,stroke:#22c55e

简单批量删除

// 删除 90 天前的审计日志
int deleted = cbf.delete(em, AuditLog.class, "a")
    .where("createdAt").lt(LocalDate.now().minusDays(90))
    .executeUpdate();

带子查询的删除

// 删除没有任何订单的客户
int purged = cbf.delete(em, Customer.class, "c")
    .whereNotExists()
        .from(Order.class, "o")
        .select("1")
        .where("o.customer.id").eqExpression("c.id")
    .end()
    .executeUpdate();

11.4 INSERT-SELECT

JPA 规范不支持 INSERT ... SELECT 语法，这是 Blaze-Persistence 的独有扩展。它将一个查询的结果直接插入目标表，特别适合数据归档、表间迁移和批量初始化场景。

Fig 11-5 · INSERT-SELECT 执行流程
sequenceDiagram participant App as 应用代码 participant ICB as InsertCriteriaBuilder participant Engine as 查询引擎 participant DB as 数据库 App->>ICB: cbf.insertInto(em, ArchiveOrder.class) App->>ICB: .from(Order.class, "o") App->>ICB: .bind("id").select("o.id") App->>ICB: .bind("orderNo").select("o.orderNo") App->>ICB: .bind("total").select("o.total") App->>ICB: .bind("archivedAt").select("CURRENT_TIMESTAMP") App->>ICB: .where("o.status").eq(COMPLETED) App->>ICB: .where("o.completedAt").lt(cutoff) ICB->>Engine: 解析 bind/select 映射 Engine->>Engine: 验证源/目标列类型匹配 Engine-->>ICB: INSERT INTO archive_order (id, order_no, total, archived_at)
SELECT o.id, o.order_no, o.total, CURRENT_TIMESTAMP
FROM orders o
WHERE o.status = 'COMPLETED' AND o.completed_at < ? ICB->>DB: executeUpdate() DB-->>App: insertedRows = 1,250

数据归档示例

// 将已完成超过 1 年的订单归档到 ArchiveOrder 表
int archived = cbf.insertInto(em, ArchiveOrder.class)
    .from(Order.class, "o")
    .bind("id").select("o.id")
    .bind("orderNumber").select("o.orderNumber")
    .bind("customerId").select("o.customer.id")
    .bind("totalAmount").select("o.totalAmount")
    .bind("originalStatus").select("o.status")
    .bind("archivedAt").select("CURRENT_TIMESTAMP")
    .where("o.status").eq(OrderStatus.COMPLETED)
    .where("o.completedAt").lt(LocalDate.now().minusYears(1))
    .executeUpdate();

// 归档后删除原表数据
int cleaned = cbf.delete(em, Order.class, "o")
    .where("o.status").eq(OrderStatus.COMPLETED)
    .where("o.completedAt").lt(LocalDate.now().minusYears(1))
    .executeUpdate();
Fig 11-6 · 归档流水线：INSERT-SELECT → DELETE
flowchart LR SOURCE["orders 表\n(100万行)"] -->|"INSERT-SELECT\n筛选已完成 > 1年"| ARCHIVE["archive_orders 表\n(+1250 行)"] SOURCE -->|"DELETE\n相同条件"| CLEANED["orders 表\n(清理后 998,750 行)"] SOURCE -.->|"⚠️ 必须在同一事务内"| TX["@Transactional"] TX -.-> ARCHIVE TX -.-> CLEANED style SOURCE fill:#3a2a1a,stroke:#d97706 style ARCHIVE fill:#1a3a2a,stroke:#22c55e style CLEANED fill:#1a3a2a,stroke:#22c55e style TX fill:#2d1f4e,stroke:#7c3aed
⚠️ INSERT-SELECT + DELETE 必须在同一事务：如果归档和删除不在同一事务中，中间发生故障会导致数据丢失（原表已删但归档未入库）或数据重复（归档成功但删除失败后重试）。始终用 @Transactional 包裹整个流程。

11.5 ReturningResult

标准的 executeUpdate() 只返回受影响行数（int）。但有时你需要知道具体哪些行被修改或删除了——例如删除后需要向被删用户发送通知。Blaze-Persistence 的 ReturningResult 机制利用 SQL 的 RETURNING 子句来解决这个需求。

Fig 11-7 · 各数据库对 RETURNING 子句的支持
✅ 原生支持
PostgreSQL — RETURNING *
Oracle — RETURNING INTO
SQL Server — OUTPUT clause
⚠️ 模拟支持
H2 — 先 SELECT 再 DML
MySQL — 先 SELECT 再 DML
❌ 不支持
旧版数据库
需手动两步操作

DELETE + RETURNING：删除并获取被删数据

// 删除过期会话并返回被删的用户 ID
ReturningResult<Long> result = cbf.delete(em, Session.class, "s")
    .where("s.expiresAt").lt(Instant.now())
    .executeWithReturning("s.userId");

int deletedCount = result.getUpdateCount();
List<Long> affectedUserIds = result.getResultList();

// 向被删用户发送"会话过期"通知
notificationService.notifySessionExpired(affectedUserIds);

UPDATE + RETURNING

// 锁定待处理任务并返回任务 ID
ReturningResult<Long> result = cbf.update(em, Task.class, "t")
    .set("status", TaskStatus.PROCESSING)
    .set("lockedBy", currentWorker)
    .set("lockedAt", Instant.now())
    .where("status").eq(TaskStatus.PENDING)
    .where("priority").ge(Priority.HIGH)
    .executeWithReturning("t.id");

List<Long> lockedTaskIds = result.getResultList();
Fig 11-8 · RETURNING 模拟机制（MySQL / H2）
sequenceDiagram participant App as 应用代码 participant BP as Blaze-Persistence participant DB as MySQL / H2 App->>BP: executeWithReturning("s.userId") Note over BP: 检测到数据库不支持原生 RETURNING BP->>DB: SELECT s.user_id FROM session s
WHERE s.expires_at < NOW() DB-->>BP: [userId: 101, 205, 378] BP->>DB: DELETE FROM session s
WHERE s.expires_at < NOW() DB-->>BP: affectedRows = 3 Note over BP: 两步操作封装在同一事务中 BP-->>App: ReturningResult{count=3, list=[101,205,378]}
📝 模拟模式的注意事项：对于不支持原生 RETURNING 的数据库，Blaze-Persistence 会先执行 SELECT 再执行 DML。这意味着两条 SQL 之间有微小的时间窗口，在高并发场景下可能产生竞态条件。生产环境建议优先选择支持原生 RETURNING 的数据库（PostgreSQL、Oracle、SQL Server）。

11.6 JPA 缓存一致性

批量 DML 最大的"暗坑"是绕过了 JPA 持久化上下文（一级缓存）。当你用 executeUpdate() 直接在数据库中修改了 1000 行，EntityManager 里缓存的那些实体对象仍然是旧值——读到的是脏数据。

Fig 11-9 · 批量 DML 导致的缓存不一致问题
flowchart TD subgraph PC["JPA 持久化上下文 (一级缓存)"] E1["Order#1\nstatus = PENDING"] E2["Order#2\nstatus = PENDING"] E3["Order#3\nstatus = PENDING"] end subgraph DB["数据库"] D1["Order#1\nstatus = ARCHIVED"] D2["Order#2\nstatus = ARCHIVED"] D3["Order#3\nstatus = ARCHIVED"] end BULK["批量 UPDATE\nSET status = ARCHIVED"] -->|"直接操作数据库\n绕过持久化上下文"| DB PC -.->|"❌ 缓存仍是旧值!\n应用读到 PENDING\n数据库已是 ARCHIVED"| STALE["💀 脏读"] style PC fill:#3a1515,stroke:#f85149 style DB fill:#1a3a2a,stroke:#22c55e style BULK fill:#2d1f4e,stroke:#7c3aed style STALE fill:#3a1515,stroke:#f85149,color:#f85149

解决方案：DML 前后的 EntityManager 操作

Fig 11-10 · 批量 DML 的正确 EntityManager 生命周期
flowchart TD A["开始事务"] --> B["em.flush()\n将待写变更推入数据库"] B --> C["执行批量 DML\ncbf.update/delete/insert"] C --> D{"后续还需要\n读取被修改的实体？"} D -->|"是"| E["em.clear()\n清空持久化上下文"] E --> F["重新加载实体\n em.find() / 查询"] D -->|"否"| G["直接提交事务"] F --> G G --> H["事务提交"] style B fill:#3a2a1a,stroke:#d97706 style C fill:#2d1f4e,stroke:#7c3aed style E fill:#3a2a1a,stroke:#d97706 style H fill:#1a3a2a,stroke:#22c55e 
@Transactional
public void archiveOldOrders(LocalDate cutoff) {
    // ① flush：确保之前的实体变更先写入数据库
    em.flush();

    // ② 执行批量 DML
    int count = cbf.update(em, Order.class, "o")
        .set("status", OrderStatus.ARCHIVED)
        .where("status").eq(OrderStatus.COMPLETED)
        .where("completedAt").lt(cutoff)
        .executeUpdate();

    // ③ clear：清空一级缓存，避免脏读
    em.clear();

    // ④ 后续查询会从数据库重新加载最新数据
    List<Order> orders = cbf.create(em, Order.class)
        .where("status").eq(OrderStatus.ARCHIVED)
        .getResultList();
}
💡 为什么 flush() 在 DML 之前？如果在批量 DML 前持久化上下文有未刷新的变更（比如刚 persist() 的实体），这些变更可能被批量 DML 覆盖或冲突。flush() 确保"先存后改"的顺序一致性。
时机操作目的
DML 之前em.flush()将持久化上下文中的 pending 变更写入数据库，避免被批量 DML 覆盖
DML 之后em.clear()清空一级缓存，后续查询从数据库获取最新数据
替代方案新事务 / 新 EntityManager在独立的持久化上下文中执行 DML，彻底隔离缓存

11.7 陷阱与限制

批量 DML 是一把双刃剑：性能极高，但绕过了 JPA 的大量安全机制。以下是生产环境中最常踩的坑。

Fig 11-11 · 批量 DML 的七大陷阱全景
① 实体监听器失效
→ 手动触发审计逻辑
② 版本列不自增
→ setExpression 手动 +1
③ JOIN 兼容性
→ 改用子查询代替 JOIN
④ INSERT-SELECT 类型匹配
→ 确认列顺序和类型
⑤ 一级缓存不一致
→ flush() → DML → clear()
⑥ 二级缓存未失效
→ evict 二级缓存 Region
⑦ 事务回滚语义
→ 合理划分事务边界

① 实体监听器失效

批量 DML 直接生成 SQL 发送给数据库，完全绕过 JPA 实体生命周期回调。@PreUpdate、@PostDelete、@EntityListeners 中的审计逻辑（如记录修改人、修改时间）不会被触发。

// ❌ 这些监听器在批量 DML 时不会执行
@PreUpdate
public void onUpdate() {
    this.updatedBy = SecurityContext.getCurrentUser();
    this.updatedAt = Instant.now();
}

// ✅ 解决方案：在批量 UPDATE 中手动 set 审计字段
cbf.update(em, Order.class, "o")
    .set("status", OrderStatus.ARCHIVED)
    .set("updatedBy", currentUser)           // 手动设置
    .setExpression("updatedAt", "CURRENT_TIMESTAMP")  // 手动设置
    .where("status").eq(OrderStatus.COMPLETED)
    .executeUpdate();

② 版本列不自增

JPA 乐观锁依赖 @Version 字段在每次更新时自动递增。批量 UPDATE 绕过了这个机制，版本号不会自动 +1，可能导致后续的乐观锁检查失效。

// ✅ 手动递增版本列
cbf.update(em, Order.class, "o")
    .set("status", OrderStatus.ARCHIVED)
    .setExpression("version", "o.version + 1")  // 手动 +1
    .where("status").eq(OrderStatus.COMPLETED)
    .executeUpdate();

③ JOIN 在 UPDATE/DELETE 中的兼容性

Fig 11-12 · UPDATE/DELETE 中 JOIN 的数据库兼容性
// ❌ UPDATE 中直接 JOIN 可能不兼容所有数据库
// ✅ 推荐：用子查询代替 JOIN
cbf.update(em, Order.class, "o")
    .set("status", OrderStatus.VIP_ORDER)
    .where("o.customer.id").in()
        .from(Customer.class, "c")
        .select("c.id")
        .where("c.tier").eq(CustomerTier.VIP)
    .end()
    .executeUpdate();

④ INSERT-SELECT 类型必须精确匹配

bind() 的目标列与 select() 的源表达式的 Java 类型必须严格一致。Integer 对 Long、String 对 Enum 都会导致运行时类型转换错误。列的绑定顺序也必须与目标表的实际列顺序对应。

⑤ 二级缓存未失效

即使你正确地 em.clear() 清空了一级缓存，JPA 二级缓存（如 Hibernate 的 Ehcache / Infinispan）仍然持有旧数据。批量 DML 后需要手动 evict 对应实体的缓存 Region：

// 清除 Hibernate 二级缓存中 Order 实体的缓存
em.getEntityManagerFactory().getCache()
    .evict(Order.class);

⑥ 事务边界与回滚

批量 DML 一旦执行就直接写入数据库（在事务内）。如果后续业务逻辑抛出异常导致事务回滚，DML 也会一并回滚——这是正确行为。但要注意：如果你在批量 DML 后发送了外部通知（如邮件、消息队列），回滚无法撤回已发送的通知。

⚠️ 副作用不可回滚：不要在批量 DML 和事务提交之间执行不可逆的外部副作用（发邮件、调用第三方 API、推送消息队列）。将副作用移到事务提交成功后的回调中，使用 TransactionSynchronizationManager.registerSynchronization() 注册 afterCommit() 回调。

⑦ 完整的安全操作清单

阶段操作说明
DML 前em.flush()写出 pending 变更
DML 构建手动 set 审计字段弥补 @PreUpdate 失效
DML 构建手动递增 version 列弥补乐观锁失效
DML 构建用子查询代替 JOIN确保跨数据库兼容
DML 后em.clear()清空一级缓存
DML 后cache.evict(Entity.class)清除二级缓存
事务后afterCommit() 执行副作用避免回滚后的"幽灵通知"

本章小结

  • 告别字符串拼接 — Blaze-Persistence 为 UPDATE、DELETE、INSERT-SELECT 提供了类型安全的流式 Builder API，编译时即可发现字段名错误
  • UPDATE CriteriaBuilder 通过 set() 设字面值、setExpression() 设表达式、子查询设动态值，支持完整的 WHERE 过滤
  • DELETE CriteriaBuilder 支持带 EXISTS / IN 子查询的条件删除
  • INSERT-SELECT 是 JPA 不支持的 Blaze-Persistence 独有能力，适用于数据归档、表间迁移、批量初始化
  • ReturningResult 在 PostgreSQL / Oracle / SQL Server 上原生支持 RETURNING 子句，其他数据库通过先 SELECT 再 DML 模拟
  • 缓存一致性是批量 DML 的最大风险——必须在 DML 前 flush()、DML 后 clear() + 二级缓存 evict
  • 七大陷阱：实体监听器失效、版本列不自增、JOIN 兼容性、INSERT-SELECT 类型匹配、一级/二级缓存不一致、事务回滚与副作用
