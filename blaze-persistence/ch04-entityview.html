第 4 章 · Entity View 基础

@EntityView · @IdMapping · @Mapping · 接口投影 · SELECT 优化

Entity View 是 Blaze-Persistence 最具特色的功能——用一个 Java 接口定义 DTO，框架自动生成只查询所需列的 SELECT。 本章从概念、注解、投影方式、SQL 优化原理到查询集成，完整拆解 Entity View 的基础用法。

4.1 什么是 Entity View

在传统 JPA 开发中，查询结果要么是完整的 Entity 对象（包含所有列），要么需要手写 JPQL 投影并手动构建 DTO。两种方式各有痛点：Entity 全列加载浪费带宽和内存，手动 DTO 映射则代码冗余且容易出错。

Entity View 的核心思想：用 Java 接口（或抽象类）声明你需要的字段，Blaze-Persistence 自动生成 只包含这些字段 的 SELECT 语句，并将结果映射为类型安全的 DTO 实例。

完整 Entity View 示例

下面展示一个包含所有核心注解的完整示例，从 JPA Entity 到 Entity View 的映射：

// ── JPA Entity ──────────────────────────────
@Entity
@Table(name = "t_user")
public class User {
    @Id @GeneratedValue
    private Long id;
    private String name;
    private String email;
    private LocalDateTime createdAt;

    @ManyToOne(fetch = FetchType.LAZY)
    private Department department;

    @OneToMany(mappedBy = "user")
    private Set<Order> orders;
}

// ── Entity View ─────────────────────────────
@EntityView(User.class)
public interface UserView {

    @IdMapping
    Long getId();

    // 隐式映射：方法名匹配实体属性名
    String getName();

    // 显式映射：可重命名
    @Mapping("email")
    String getEmailAddress();

    // 关联路径：自动 JOIN department
    @Mapping("department.name")
    String getDeptName();

    // 表达式映射：聚合计算
    @Mapping("SIZE(orders)")
    Long getOrderCount();

    // 格式化表达式
    @Mapping("CONCAT(name, ' <', email, '>')")
    String getDisplayName();
}
💡 隐式映射规则：当 getter 方法名去掉 get 前缀并首字母小写后，与实体属性名一致时，可以省略 @Mapping 注解。例如 getName() 自动映射到 name 属性。

4.4 SELECT 优化原理

Entity View 最强大的特性之一是自动 SELECT 裁剪：框架分析视图定义中声明的 getter 方法，只生成包含这些字段的 SELECT 子句，而不是像加载 Entity 那样 SELECT *。

N+1 查询自动消除

当 Entity View 包含关联路径（如 department.name）时，框架自动将其转换为 JOIN 查询，而非触发延迟加载。

N+1 问题 vs Entity View 解决方案
sequenceDiagram participant App as 应用代码 participant JPA as JPA (原生) participant DB as 数据库 Note over App,DB: ❌ 原生 JPA → N+1 问题 App->>JPA: findAll Users (10条) JPA->>DB: SELECT * FROM t_user DB-->>JPA: 10 行结果 loop 每个 User 访问 department App->>JPA: user.getDepartment().getName() JPA->>DB: SELECT * FROM t_department WHERE id=? DB-->>JPA: 1 行 end Note over DB: 共 1 + 10 = 11 条 SQL ❌ Note over App,DB: ✅ Entity View → 1 条 SQL App->>JPA: applySetting(UserView.class) JPA->>DB: SELECT u.id, u.name, d.name
FROM t_user u JOIN t_department d
ON u.department_id = d.id DB-->>JPA: 10 行（包含 dept 信息） Note over DB: 仅 1 条 SQL ✅
💡 优化原理：Entity View 在 JPQL 生成阶段就确定了所有需要的关联路径，并将其转换为 JOIN。这意味着无论结果集有多少行，关联数据获取永远是 O(1) 次查询——从根本上杜绝了 N+1 问题。

完整查询集成代码

// ── 步骤 1：定义 Entity View（带过滤器和排序器）──
@EntityView(User.class)
@ViewFilter(name = "activeFilter",
    value = "status = 'ACTIVE'")
public interface UserListView {

    @IdMapping
    Long getId();

    @Mapping("name")
    @AttributeFilter(ContainsFilter.class)
    String getName();

    @Mapping("email")
    String getEmail();

    @Mapping("department.name")
    @AttributeSorter
    String getDeptName();

    @Mapping("createdAt")
    LocalDateTime getCreatedAt();
}

// ── 步骤 2：构建查询 ──────────────────────────
CriteriaBuilder<User> cb = cbf.create(em, User.class);

// ── 步骤 3：创建 EntityViewSetting ────────────
EntityViewSetting<UserListView, CriteriaBuilder<UserListView>> setting =
    EntityViewSetting.create(UserListView.class);

// 添加过滤：名字包含 "张"
setting.addAttributeFilter("name", "张");

// 添加排序：按部门名称升序
setting.addAttributeSorter("deptName", Sorters.ascending());

// 启用命名过滤器
setting.withViewFilter("activeFilter");

// ── 步骤 4：应用并执行 ────────────────────────
CriteriaBuilder<UserListView> resultCb =
    evm.applySetting(setting, cb);

List<UserListView> users = resultCb.getResultList();

陷阱 1：代理生成开销

Entity View 在首次使用某个视图类型时，需要通过字节码技术生成代理类。这个过程存在一次性的性能开销。

⚠️ 注意：代理生成发生在 EntityViewManager 初始化阶段。在生产环境中，务必让 EntityViewManager 在应用启动时初始化完成（Spring Boot 自动完成），而不是在首个请求到达时才触发。 
// ✅ 推荐：在 @Configuration 中预先初始化
@Bean
public EntityViewManager evm(CriteriaBuilderFactory cbf,
                              EntityViewConfiguration config) {
    // 所有 Entity View 代理在此时生成
    return config.createEntityViewManager(cbf);
}

// ❌ 避免：延迟初始化
@Lazy
@Bean
public EntityViewManager evm(...) { ... } // 首次请求会变慢

陷阱 2：映射表达式运行时错误

@Mapping 注解中的表达式是字符串，编译器无法检查。拼写错误或路径不存在只会在运行时抛出异常。

陷阱 3：循环视图引用

⚠️ 危险：如果 UserView 包含 DepartmentView，而 DepartmentView 又包含 List<UserView>，会导致无限递归。Blaze-Persistence 会在初始化时检测到循环引用并抛出异常。 
// ❌ 循环引用 — 初始化失败
@EntityView(User.class)
public interface UserView {
    @IdMapping Long getId();
    DepartmentView getDepartment(); // → 引用 DepartmentView
}

@EntityView(Department.class)
public interface DepartmentView {
    @IdMapping Long getId();
    List<UserView> getUsers(); // → 又引用 UserView → 循环！
}

// ✅ 解决：为不同场景创建不同视图
@EntityView(Department.class)
public interface DepartmentSimpleView {
    @IdMapping Long getId();
    String getName();
    // 不包含 users 引用，打破循环
}

陷阱 5：JSON 序列化

Entity View 的代理对象不是普通 POJO，某些 JSON 库可能无法正确序列化。需要注册专用的 Jackson 模块。

// 注册 Blaze-Persistence Jackson 模块
@Bean
public ObjectMapper objectMapper(EntityViewManager evm) {
    ObjectMapper mapper = new ObjectMapper();
    mapper.registerModule(
        new EntityViewAwareObjectMapper(evm)  // 核心：注册视图感知模块
    );
    mapper.registerModule(new JavaTimeModule());
    return mapper;
}

📋 本章小结

  • Entity View 是类型安全的 DTO 投影——用 Java 接口声明所需字段，框架自动生成优化 SELECT
  • 三大核心注解：@EntityView（绑定实体）、@IdMapping（映射 ID）、@Mapping（映射表达式）
  • 接口投影是默认推荐方式，简洁不可变；抽象类投影适用于需要自定义 Java 逻辑的场景
  • SELECT 自动裁剪只查询视图声明的字段，减少网络传输和内存消耗
  • N+1 问题通过将关联路径转换为 JOIN 从根本上消除，始终保持 O(1) 查询
  • EntityViewSetting是视图与查询的桥梁，支持过滤、排序、分页的统一配置
  • 常见陷阱：注意代理生成开销、映射表达式的运行时校验、循环引用和深层嵌套
