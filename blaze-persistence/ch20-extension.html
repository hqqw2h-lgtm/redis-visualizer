20.2 JpqlFunction — 自定义函数注册

这个 SPI 是什么

JpqlFunction 是 Blaze-Persistence 最核心的扩展点。它让开发者在 JPQL 层面注册任意函数，框架在 SQL 渲染时调用函数的 render() 方法产出目标数据库的原生 SQL 片段。接口定义只有四个方法：

public interface JpqlFunction {

    boolean hasArguments();

    boolean hasParenthesesIfNoArguments();

    Class<?> getReturnType(Class<?> firstArgumentType);

    void render(FunctionRenderContext context);
}
Fig 20-2 JpqlFunction 渲染调用时序
sequenceDiagram participant U as 用户代码 participant CB as CriteriaBuilder participant EP as ExpressionParser participant FR as FunctionRenderer participant JF as JpqlFunction participant FRC as FunctionRenderContext participant SQL as SQL Buffer U->>CB: .select("FUNCTION('array_agg', o.product.name)") CB->>EP: 解析表达式 EP->>FR: 查找注册函数 "array_agg" FR->>JF: hasArguments()? → true FR->>JF: getReturnType(String.class) → String.class FR->>JF: render(FunctionRenderContext) JF->>FRC: addChunk("array_agg(") JF->>FRC: addArgument(0) FRC-->>FRC: 渲染参数: product_1.name JF->>FRC: addChunk(")") FRC->>SQL: 拼接: array_agg(product_1.name) SQL-->>U: 最终 SQL 片段就绪

Step 1: 定义自定义函数

public class PostgresArrayAggFunction implements JpqlFunction {

    @Override
    public boolean hasArguments() {
        return true;
    }

    @Override
    public boolean hasParenthesesIfNoArguments() {
        return true;
    }

    @Override
    public Class<?> getReturnType(Class<?> firstArgType) {
        return String.class;
    }

    @Override
    public void render(FunctionRenderContext ctx) {
        ctx.addChunk("array_agg(");
        ctx.addArgument(0);     // 渲染第一个参数
        ctx.addChunk(")");
    }
}

Step 2: 注册到 CriteriaBuilderConfiguration

CriteriaBuilderConfiguration config = Criteria.getDefault();
config.registerFunction(
    new JpqlFunctionGroup("array_agg", new PostgresArrayAggFunction())
);
CriteriaBuilderFactory cbf = config.createCriteriaBuilderFactory(emf);

Step 3: 在查询中使用 + 输出 SQL

CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .select("o.customer.name")
    .select("FUNCTION('array_agg', o.product.name)")
    .groupBy("o.customer.name");

System.out.println(cb.getQueryString());

-- 生成的 JPQL:
SELECT o.customer.name, array_agg(product_1.name)
FROM Order o JOIN o.product product_1
GROUP BY o.customer.name

-- PostgreSQL 原生 SQL:
SELECT customer_1.name, array_agg(product_1.name)
FROM orders o
JOIN customers customer_1 ON o.customer_id = customer_1.id
JOIN products product_1 ON o.product_id = product_1.id
GROUP BY customer_1.name

框架自己怎么用：30+ 内置函数

Blaze-Persistence 自身通过 JpqlFunction 注册了大量内置函数。这些函数覆盖集合运算、窗口函数、聚合增强、分页辅助等多个领域，且针对不同数据库有不同的渲染实现。

以 GROUP_CONCAT 为例——同一个 JPQL 函数调用，框架为不同数据库渲染完全不同的原生 SQL：

// 用户代码 — 与数据库无关:
CriteriaBuilder<Tuple> cb = cbf.create(em, Tuple.class)
    .from(Order.class, "o")
    .select("o.customer.name")
    .select("GROUP_CONCAT(o.product.name SEPARATOR ',')")
    .groupBy("o.customer.name");

-- PostgreSQL 渲染:
SELECT c.name, STRING_AGG(p.name, ',')
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN products p ON o.product_id = p.id
GROUP BY c.name

-- MySQL 渲染:
SELECT c.name, GROUP_CONCAT(p.name SEPARATOR ',')
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN products p ON o.product_id = p.id
GROUP BY c.name

-- Oracle 渲染:
SELECT c.name, LISTAGG(p.name, ',') WITHIN GROUP (ORDER BY p.name)
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN products p ON o.product_id = p.id
GROUP BY c.name

-- SQL Server 渲染:
SELECT c.name, STRING_AGG(p.name, ',')
FROM orders o
JOIN customers c ON o.customer_id = c.id
JOIN products p ON o.product_id = p.id
GROUP BY c.name
Dogfooding 设计：JpqlFunction 不是"给用户的附加功能"。框架自身的窗口函数（WINDOW_ROW_NUMBER）、集合运算（SET_UNION）、聚合增强（COUNT_TUPLE）全部通过同一 SPI 实现。你的自定义函数与框架内置函数享有完全相同的渲染管线。

20.3 JpqlMacro — 宏表达式

这个 SPI 是什么

JpqlMacro 工作在表达式解析的最前端——在 JpqlFunction 查找之前。它不产出 SQL，而是将宏调用展开为新的 JPQL 表达式片段。接口只有一个方法：

public interface JpqlMacro {

    void render(FunctionRenderContext context);
}

我们怎么用：NOW 时间戳宏

public class NowMacro implements JpqlMacro {

    @Override
    public void render(FunctionRenderContext ctx) {
        ctx.addChunk("CURRENT_TIMESTAMP");
    }
}

// 注册宏
config.registerMacro("NOW", new NowMacro());

// 使用宏进行查询:
CriteriaBuilder<Order> cb = cbf.create(em, Order.class, "o")
    .where("o.createdAt").ltExpression("NOW()");

System.out.println(cb.getQueryString());

-- 生成的 JPQL（宏展开后）:
SELECT o
FROM Order o
WHERE o.createdAt < CURRENT_TIMESTAMP

-- PostgreSQL 原生 SQL:
SELECT o.*
FROM orders o
WHERE o.created_at < CURRENT_TIMESTAMP

我们怎么用：多租户宏

public class CurrentTenantMacro implements JpqlMacro {

    @Override
    public void render(FunctionRenderContext ctx) {
        ctx.addChunk(":currentTenantId");
    }
}

config.registerMacro("CURRENT_TENANT", new CurrentTenantMacro());

// 在任何查询中自动注入租户条件:
CriteriaBuilder<Order> cb = cbf.create(em, Order.class, "o")
    .where("o.tenantId").eqExpression("CURRENT_TENANT()")
    .where("o.status").eq(OrderStatus.ACTIVE);

cb.setParameter("currentTenantId", SecurityContext.getTenantId());
System.out.println(cb.getQueryString());

-- 宏展开后生成的 JPQL:
SELECT o
FROM Order o
WHERE o.tenantId = :currentTenantId
  AND o.status = :param_0

-- PostgreSQL 原生 SQL:
SELECT o.*
FROM orders o
WHERE o.tenant_id = ?
  AND o.status = ?

// 框架内部使用 VIEW_ROOT 宏的场景:
@EntityView(Cat.class)
public interface CatDetailView {

    @IdMapping
    Long getId();

    String getName();

    // @MappingCorrelated 使用 VIEW_ROOT 宏引用根实体
    @MappingCorrelated(
        correlationBasis = "VIEW_ROOT(id)",
        correlationExpression = "kitten.parent.id = EMBEDDING_VIEW_ROOT(id)",
        correlationResult = "kitten"
    )
    List<KittenView> getKittens();

// EMBEDDING_VIEW_ROOT(id) 被替换为:   parentCat.id

SELECT c.id, c.name,
       k.id AS kittens_id, k.name AS kittens_name
FROM cats c
LEFT JOIN cats k ON k.parent_id = c.id

20.4 DbmsDialect — 数据库方言扩展

这个 SPI 是什么

DbmsDialect 是 Blaze-Persistence 能够跨数据库工作的关键。每个方言实现了同一套接口，控制 CTE、窗口函数、分页、LATERAL JOIN、DML 等高级特性在不同数据库上的渲染策略。

class ViewTypeObjectBuilder implements ObjectBuilder<Object> {

    private final ProxyFactory proxyFactory;
    private final int[] tupleMapping;
    private final ObjectBuilder<?>[] nestedBuilders;

    @Override
    public Object build(Object[] tuple) {
        Object proxy = proxyFactory.create(tupleMapping.length);
        for (int i = 0; i < tupleMapping.length; i++) {
            int tupleIndex = tupleMapping[i];
            if (nestedBuilders[i] != null) {
                proxy.$$_setAttribute(i,
                    nestedBuilders[i].build(tuple));
            } else {
                proxy.$$_setAttribute(i, tuple[tupleIndex]);
            }
        }
        proxy.$$_initDirtyTracker();
        return proxy;
    }
}

这个 SPI 是什么

// 定义可审计视图接口
public interface AuditableView {
    void setCreatedAt(Instant createdAt);
    void setCreatedBy(String user);
    void setUpdatedAt(Instant updatedAt);
    void setUpdatedBy(String user);
}

// 审计 Listener
@ViewListener
public class AuditViewListener implements EntityViewListener {

    @PrePersist
    public void onPrePersist(Object view) {
        if (view instanceof AuditableView av) {
            av.setCreatedAt(Instant.now());
            av.setCreatedBy(SecurityContext.getCurrentUser());
            av.setUpdatedAt(Instant.now());
            av.setUpdatedBy(SecurityContext.getCurrentUser());
        }
    }

    @PreUpdate
    public void onPreUpdate(Object view) {
        if (view instanceof AuditableView av) {
            av.setUpdatedAt(Instant.now());
            av.setUpdatedBy(SecurityContext.getCurrentUser());
        }
    }

    @PreRemove
    public boolean onPreRemove(Object view) {
        AuditLog.record("DELETE",
            view.getClass().getSimpleName(),
            ((Identifiable) view).getId(),
            SecurityContext.getCurrentUser());
        return true;   // false → 取消删除操作
    }

    @PostUpdate
    public void onPostUpdate(Object view) {
        EventBus.publish(new EntityUpdatedEvent(
            view.getClass().getSimpleName(),
            ((Identifiable) view).getId()));
    }
}

@EntityView(Order.class)
@ViewFilter(name = "activeOnly", value = "deleted = false")
@ViewFilter(name = "currentTenant", value = "tenantId = :tenantId")
public interface OrderView {

    @IdMapping
    Long getId();

    String getCustomerName();

    BigDecimal getTotalAmount();

    OrderStatus getStatus();

    boolean isDeleted();

// Step 2: 查询时激活过滤器

    EntityViewSetting.create(OrderView.class);

setting.addViewFilter("activeOnly");
setting.addViewFilter("currentTenant");
setting.addOptionalParameter("tenantId", currentTenantId);

CriteriaBuilder<OrderView> cb = cbf.create(em, Order.class);

    .getResultList();

用法 2：Specification 组合查询

// 定义 Specification:
class OrderSpecifications {
    static Specification<Order> hasStatus(OrderStatus s) {
        return (root, query, cb) ->
            cb.equal(root.get("status"), s);
    }

    static Specification<Order> totalGreaterThan(BigDecimal min) {
        return (root, query, cb) ->
            cb.greaterThan(root.get("totalAmount"), min);
    }

    static Specification<Order> customerNameLike(String keyword) {
        return (root, query, cb) ->
            cb.like(root.get("customer").get("name"),
                    "%" + keyword + "%");
    }
}

// 组合 Specification:
Page<OrderView> page = repo.findAll(
    hasStatus(CONFIRMED)
        .and(totalGreaterThan(new BigDecimal("100")))
        .and(customerNameLike("Alice")),
    PageRequest.of(0, 20, Sort.by("totalAmount").descending())
);

-- 三个 Specification 组合后生成:
SELECT o.id, o.customer_name, o.total_amount, o.status
FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = ?
  AND o.total_amount > ?
  AND c.name LIKE ?
ORDER BY o.total_amount DESC
LIMIT 20 OFFSET 0
-- 参数: ['CONFIRMED', 100.00, '%Alice%']

用法 3：游标分页（Keyset Pagination）

// 第一页:
KeysetAwarePage<OrderView> firstPage = repo.findAll(
    new KeysetPageRequest(null, Sort.by("id"), 20));

// 下一页 — 基于游标，不使用 OFFSET:
KeysetAwarePage<OrderView> nextPage = repo.findAll(
    firstPage.nextPageable());

-- 第一页:
SELECT o.id, o.customer_name, o.total_amount, o.status
FROM orders o
ORDER BY o.id ASC
LIMIT 20

-- 第二页（游标分页，避免 OFFSET）:
SELECT o.id, o.customer_name, o.total_amount, o.status
FROM orders o
WHERE o.id > ?              -- 上一页最后一条的 id
ORDER BY o.id ASC
LIMIT 20
-- 参数: [lastId]          -- 无 OFFSET！O(1) 性能
游标分页 vs OFFSET 分页：OFFSET 10000 需要数据库扫描前 10000 行后丢弃；游标分页用 WHERE id > ? 直接定位，性能恒定 O(1)，不随页码增长而下降。

本章小结

  • JpqlFunction 让我们在 JPQL 层面注册自定义函数——render() 方法产出原生 SQL。框架自身用它实现了 30+ 内置函数（SET_UNION、WINDOW_ROW_NUMBER、GROUP_CONCAT 等），且 GROUP_CONCAT 在 PostgreSQL、MySQL、Oracle、SQL Server 上分别渲染为 STRING_AGG、GROUP_CONCAT、LISTAGG、STRING_AGG
  • JpqlMacro 在表达式解析前完成宏展开——比 JpqlFunction 更早介入。框架用 VIEW_ROOT / EMBEDDING_VIEW_ROOT 实现 Entity View 子查询关联；开发者可注册 CURRENT_TENANT、NOW 等业务宏
  • DbmsDialect 是跨数据库的枢纽——同一 CTE API，PostgreSQL 输出 WITH ... AS，MySQL 5.7 降级为派生表，Oracle 用列别名语法。分页、窗口函数、LATERAL JOIN、DML + JOIN 同理
  • ObjectBuilder 将 Object[] 元组映射为目标对象——框架的 ViewTypeObjectBuilder 负责创建 Entity View 代理、初始化脏跟踪、递归构建嵌套 View
  • EntityViewListener 提供 @PreCreate / @PreUpdate / @PreRemove 等生命周期钩子——框架内部用于脏跟踪和级联传播，开发者用于审计、事件发布
  • ViewFilter 是纯用户扩展点——通过 @ViewFilter 注解定义命名过滤器，查询时按名称激活，适合软删除和多租户
  • ConfigurationContributor 通过 ServiceLoader 实现零配置自动注册——将函数和宏打包为 JAR，消费方只需添加 Maven 依赖
  • Spring Data Repository 扩展 让 Repository 具备 Entity View 感知、Specification 动态查询、游标分页能力，生成高效的 WHERE id > ? 分页 SQL
  • Dogfooding 设计哲学：每个扩展点都先被框架自身使用验证，再开放给开发者——没有二等公民
