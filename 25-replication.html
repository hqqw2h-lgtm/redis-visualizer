第 25 章 · 主从复制

PSYNC 2.0 · repl_backlog 环形缓冲 · replid + replid2 · Diskless · min-replicas · WAIT 命令

Redis 主从复制把主节点的写入流以 RESP 编码的命令字节流形式异步传播到从节点。设计目标包含三项:断线重连后避免重传完整数据集、故障转移后保留旧主节点身份以减少全量同步、单线程主进程不被复制路径阻塞。

三项目标共同决定了 Redis 复制协议的结构:全局复制偏移量(master_repl_offset)、固定大小环形缓冲(repl_backlog)、双 Replication ID(replid + replid2)、PSYNC 2.0 握手协议。任何一个组件缺失都会让上面三个目标之一回落到"全量同步"的高代价兜底。

本章按"协议基础 → 数据结构 → 运行时行为 → 生产实践 → 源码取舍"五层组织。每一节先给出图示,再给出代码或命令验证,最后给出容量推导公式与可按业务基线调整的监控阈值。

  • 异步复制为什么不能升级为强同步:min-replicas-* 是写入前健康门槛,WAIT 是写入后 ACK 等待
  • PSYNC 2.0 握手两条路径:首次连接强制全量、断线重连按 backlog 窗口判定
  • repl_backlog 5 字段:size / idx / histlen / off / master_repl_offset 的语义与不变量
  • backlog 容量公式 2 × W × T:为什么默认 1 MiB 仅适用于测试环境
  • replid + replid2 双身份如何让 failover 后兄弟从节点免全量同步
  • 命令传播路径 replicationFeedSlaves():同时写 backlog 与每个 replica 的 client output buffer
  • Diskless 复制 fork → pipe → socket:慢磁盘 + 快网络场景的最优解
  • INFO replication 8 个核心字段:从 master_link_status 到 repl_backlog_histlen
  • 9 类生产陷阱与诊断:从 backlog 太小到 replica-serve-stale-data 默认开启
  • 4 个核心源码取舍:内存环形 vs 磁盘日志、异步 vs 同步、双 ID、命令字节流 vs 操作日志

25.1 异步复制的三处约束

Redis 复制是异步的:主节点写入命令到内存即返回 +OK,不等待任何从节点 ACK。这个决策直接决定了写入路径的延迟特征,也决定了主节点崩溃时的数据丢失窗口。这一节用对照图把"单节点"与"主从"的写入路径并排画清楚。

异步复制核心权衡:写入路径默认不等待从节点 ACK,代价是主节点崩溃时存在数据丢失窗口。min-replicas-to-write 与 min-replicas-max-lag 是基于最近 ACK 的写入前健康门槛——主节点检测到健康从节点数不足时拒绝新写入,但已经返回 +OK 的写入仍可能丢失。

大数据集全量同步循环陷阱:replica 输出缓冲的硬限制(默认 256 MiB)是大数据集全量同步失败的常见原因。当主节点持续写入速率高、RDB 较大时,buffer 在传输完成前可能超限,触发连接关闭并循环全量同步,主节点长时间无法把这个 replica 带上线。修复方式是按"传输期间新增写入量 + 抖动余量"提高 replica/slave hard/soft 限制,避免简单设为无限;同时启用 diskless、降低写入峰值或分片。

25.4 repl_backlog 环形缓冲区

repl_backlog 是 Redis 复制的核心数据结构:一个固定大小的环形缓冲区,用于在断线重连时为从节点提供"最近 N 字节"的写流回放窗口。这一节把 5 个核心字段、字节布局、写入回绕、判定逻辑全部画清楚。

25.4.1 repl_backlog 数据结构

/* src/server.h - 简化 */
struct redisServer {
    char *repl_backlog;            /* 环形缓冲区指针 */
    long long repl_backlog_size;   /* 缓冲区总大小(repl-backlog-size) */
    long long repl_backlog_histlen;/* 当前有效数据长度 */
    long long repl_backlog_idx;    /* 下一次写入位置(写指针) */
    long long repl_backlog_off;    /* 缓冲区第一个字节的全局偏移量 */
    long long master_repl_offset;  /* 全局复制偏移量(单调递增) */
};

25.4.5 部分同步可行性判定

/* src/replication.c — masterTryPartialResynchronization 简化 */
long long psync_offset = client_request_offset;
long long backlog_start = server.repl_backlog_off;
long long backlog_end = backlog_start + server.repl_backlog_histlen;

if (psync_offset < backlog_start || psync_offset > backlog_end) {
    /* offset 已被覆盖或超前 → 必须全量同步 */
    goto need_full_resync;
}

/* 计算 backlog 中的实际位置 */
long long skip = psync_offset - backlog_start;
long long j = (server.repl_backlog_idx
               + (server.repl_backlog_size - server.repl_backlog_histlen))
              % server.repl_backlog_size;
j = (j + skip) % server.repl_backlog_size;
/* 从 j 起发送 (histlen - skip) 字节 */
部分同步充要条件:backlog_start ≤ psync_offset ≤ backlog_end。当主节点在断线期间写入数据量超过 repl_backlog_size 时,旧字节被覆盖,该条件失效,握手退化为全量同步。这就是为什么 repl-backlog-size 必须按写入速率推导,而非凭经验估算。

25.6 Replication ID 与故障转移身份

25.7.4 WAIT 与 min-replicas 的差异

图 25-7-3 · 选用 WAIT 还是 min-replicas 的决策
WAIT 与 min-replicas 均不提供强同步/共识语义。WAIT 用于单次写入后等待 ACK 计数;如果超时返回的 ACK 数不足,写入也不会被回滚。min-replicas-* 是写入前的健康门槛,基于最近 ACK 时间判断是否拒写。两者都不能保证已经返回 +OK 的写入在主节点崩溃后完全不丢。

25.7.5 写命令传播源码片段

/* src/replication.c — replicationFeedSlaves 简化 */
void replicationFeedSlaves(list *slaves, int dictid, robj **argv, int argc) {
    /* 1. 构造 RESP 编码字节流 */
    sds proto = catCommandToProtocol(argv, argc);
    /* 2. 写入 backlog 与全局 offset */
    feedReplicationBacklog(proto, sdslen(proto));
    server.master_repl_offset += sdslen(proto);
    /* 3. 写入每个从节点的 client output buffer */
    listIter li;
    listNode *ln;
    listRewind(slaves, &li);
    while ((ln = listNext(&li)) != NULL) {
        client *slave = ln->value;
        addReplyProto(slave, proto, sdslen(proto));
    }
    sdsfree(proto);
}

25.8 Diskless 复制

传统全量同步要求主节点 fork 子进程把 RDB 写到磁盘,主进程再读盘发送。Diskless 模式去掉中间的磁盘环节,fork 后子进程直接通过 pipe 把 RDB 字节流写到 socket。

子进程 CPU 占用写磁盘 I/O 阻塞较多pipe 写入,持续 CPU
适用 Redis 版本全部2.8.18+

25.13.3 设计取舍

取舍 1:内存环形缓冲 vs 磁盘日志

Redis 选择内存环形缓冲(类比 SRAM 寄存器),而非 Kafka / MySQL 的磁盘日志。代价是 backlog 大小有上界,断线时间乘写入速率超过 repl-backlog-size 后只能 FULLRESYNC。收益是主线程写入路径仅需 memcpy 到环形缓冲,延迟不依赖磁盘 I/O。

取舍 2:异步复制 vs 同步复制

Redis 选择异步复制,以最大化写入吞吐。代价是主节点崩溃时存在数据丢失窗口。min-replicas-to-write 提供写入前健康门槛,WAIT 提供写入后 ACK 等待,但二者都不改变异步复制本质。强一致需求场景应选择具备同步复制或共识提交语义的方案(如 etcd、ZooKeeper)。

取舍 3:replid + replid2 双身份

故障转移后,新主节点保留旧主 replid 为 replid2,允许兄弟从节点在不全量同步的前提下迁移。该设计将故障转移的复制成本从 O(N × RDB) 降至 O(增量字节),前提是 backlog 容量足够覆盖切换期间的写入。

取舍 4:命令字节流 vs 操作日志

Redis 复制传播 RESP 编码的命令字节流,而非 MySQL 风格的操作日志(逻辑 / 行式)。代价是从节点必须完整执行命令(无法跳过),且部分非确定命令(如 SPOP、SCRIPT EVAL)需要特殊处理。收益是协议格式简单,从节点代码与主节点共用同一套命令执行路径。
