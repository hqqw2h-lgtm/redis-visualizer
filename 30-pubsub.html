第 30 章 · Pub/Sub 与 Sharded Pub/Sub

双向索引 · glob 匹配 · fire-and-forget · Cluster Bus 网络放大 · Sharded 路由 · 慢消费者治理

Redis 的发布订阅由 两个 dict 加一个 list 实现完整的 channel/pattern 订阅模型,语义是 fire-and-forget:消息不持久化、不堆积、无 ACK,投递语义是 at-most-once。这种设计在单节点上提供低于毫秒的投递延迟,代价是任何环节(订阅者离线、缓冲溢出、消费者 crash)都可能丢消息。

在 Cluster 模式下,普通 PUBLISH 通过 Cluster Bus 广播到所有节点,产生 O(N) 网络放大;Redis 7.0 引入 Sharded Pub/Sub(SPUBLISH/SSUBSCRIBE),把 channel 当成 key 计算 slot,消息仅在归属节点投递,网络放大降至 O(1)。Pub/Sub 与 Stream 共同覆盖 Redis 的消息光谱:Pub/Sub 选择极致延迟与简单性,Stream 提供 backlog/ACK/offset/consumer group,牺牲部分延迟换取可靠投递。

30.1 Pub/Sub 解决的问题

Pub/Sub 之所以存在,是为了把"客户端反复拉" 改成"事件来了主动推"。一个具体场景能立刻说明差距。

图 30-1-1 · 轮询 vs 推送 — 50 实例感知配置变更的成本对比

Pub/Sub 适合"通知是状态而非历史"的场景,不适合需要可靠投递、负载均衡、回放历史的场景。下表把"该用"与"别用"分开。

✓ 适用

  • 配置变更广播(业务可容忍偶尔丢)
  • 实时仪表盘推送
  • 多实例缓存失效通知(有补偿机制)
  • Sentinel 心跳与服务发现
  • Streaming API(客户端在线接收)

✗ 不适用

  • 订单/支付/库存等需要可靠投递
  • 消费者重启后从断点恢复
  • 消费者组负载均衡
  • 大集群高频消息(Cluster Bus 放大)
  • 需要回放历史(无 backlog)

选型口令。需要可靠投递时改用 Stream(XADD / XREADGROUP / XACK)或外部 MQ(Kafka / RabbitMQ / RocketMQ);Pub/Sub 的最佳位置就是"丢一条不会有人来追责"的场景。

30.2 核心数据结构

Pub/Sub 的核心结构只有三处:服务端的 pubsub_channels dict、pubsub_patterns list、pubsubshard_channels dict;每个客户端再镜像一份"自己订阅了什么"。

/* server.h - redisServer 结构体相关字段 */
struct redisServer {
    dict *pubsub_channels;        /* channel → list<client*> */
    list *pubsub_patterns;        /* list<pubsubPattern> */
    dict *pubsubshard_channels;   /* Redis 7.0+ shard channel → list<client*> */
};

typedef struct pubsubPattern {
    client *client;        /* 订阅此 pattern 的客户端 */
    robj   *pattern;       /* glob 模式字符串 */
} pubsubPattern;

struct client {
    dict *pubsub_channels;        /* 此客户端订阅的 channel 集合 */
    list *pubsub_patterns;        /* 此客户端订阅的 pattern 列表 */
    dict *pubsubshard_channels;   /* Redis 7.0+ */
};

图 30-2-1 · 双向索引 — 服务端与客户端互相持有

"双向索引"是空间换时间的经典权衡。下表把单向与双向在四类操作上的成本对齐:

操作单向 (仅 server 持有)双向 (client 也持有)
SUBSCRIBE channelO(1)O(1)
PUBLISH channelO(订阅者数)O(订阅者数)
UNSUBSCRIBE channelO(1)O(1)
断连清理O(全部 channel)O(此客户端订阅数)
内存开销低略高(双倍指针)

设计同构。双向索引 = 数据库的二级索引:用一定空间换查询时间。Redis 在所有"反向需精确定位"的地方都用了这个模式(Cluster slot ↔ key 的反查、过期 key 的反查、复制 ID 的反查)。

30.3 SUBSCRIBE 与 PUBLISH 实现

核心实现都在 pubsub.c 中,大约 200 行。先看 SUBSCRIBE 的 4 步走。

/* pubsub.c - 简化版 */
int pubsubSubscribeChannel(client *c, robj *channel) {
    /* 1. 加入客户端的 pubsub_channels */
    if (dictAdd(c->pubsub_channels, channel, NULL) != DICT_OK)
        return 0;  /* 已订阅,跳过 */
    incrRefCount(channel);

    /* 2. 在服务端 dict 中查找/创建该 channel 的客户端链表 */
    list *clients;
    dictEntry *de = dictFind(server.pubsub_channels, channel);
    if (de == NULL) {
        clients = listCreate();
        dictAdd(server.pubsub_channels, channel, clients);
        incrRefCount(channel);
    } else {
        clients = dictGetVal(de);
    }

    /* 3. 客户端追加到链表尾部 */
    listAddNodeTail(clients, c);

    /* 4. 回复客户端当前订阅总数 */
    addReplyPubsubSubscribed(c, channel);
    return 1;
}

图 30-3-1 · SUBSCRIBE 4 步走 — 全程 O(1) 摊销

PUBLISH 分两个阶段:dict 找直接订阅者一次性投递,然后线性遍历 pattern list 做 glob 匹配。

/* pubsub.c - 简化版 */
int pubsubPublishMessageInternal(robj *channel, robj *message) {
    int receivers = 0;

    /* 阶段 1: 直接订阅者投递 */
    dictEntry *de = dictFind(server.pubsub_channels, channel);
    if (de) {
        list *clients = dictGetVal(de);
        listIter li;  listNode *ln;
        listRewind(clients, &li);
        while ((ln = listNext(&li)) != NULL) {
            client *c = listNodeValue(ln);
            addReplyPubsubMessage(c, channel, message);
            receivers++;
        }
    }

    /* 阶段 2: pattern 订阅者投递 */
    if (server.pubsub_patterns) {
        listIter li;  listNode *ln;
        listRewind(server.pubsub_patterns, &li);
        while ((ln = listNext(&li)) != NULL) {
            pubsubPattern *pat = listNodeValue(ln);
            if (stringmatchlen(pat->pattern->ptr, sdslen(pat->pattern->ptr),
                              channel->ptr, sdslen(channel->ptr), 0)) {
                addReplyPubsubPatMessage(pat->client, pat->pattern,
                                        channel, message);
                receivers++;
            }
        }
    }
    return receivers;
}

图 30-3-2 · PUBLISH 消息投递时序 — 两阶段

PUBLISH 返回值的真实含义。返回的整数 = 本次本节点投递的客户端数,包括直接订阅 + pattern 匹配,不包括:Cluster 其他节点的订阅者、业务层是否真处理了、TCP 是否成功送达。
0 不代表"无人收到"(本节点没订阅 + 其他节点有订阅 → 仍是 0);N>0 也不代表"N 人已处理"(只代表"写入了 N 个客户端的输出缓冲区")。

30.4 PSUBSCRIBE 与 glob 匹配

PSUBSCRIBE 用 glob 通配符订阅一组 channel,代价是每次 PUBLISH 都要对全部 pattern 跑一遍 stringmatchlen。

通配符含义示例 pattern匹配不匹配
*0+ 任意字符news.*news.tech, news.tech.ainewsfeed
?1 个任意字符log.?log.a, log.1log.ab
[abc]字符类ch.[ab]ch.a, ch.bch.c
[a-z]字符范围ch.[a-z]ch.xch.5
\*字面量转义ch\*ch*chx
 
/* util.c - 简化核心 */
int stringmatchlen(const char *pattern, int patternLen,
                   const char *string, int stringLen, int nocase) {
    while (patternLen && stringLen) {
        switch (pattern[0]) {
        case '*':
            while (patternLen && pattern[0] == '*') {
                pattern++; patternLen--;
            }
            if (patternLen == 0) return 1;  /* 尾部 * 匹配一切 */
            /* 对 string 的每个后缀尝试匹配剩余 pattern */
            while (stringLen) {
                if (stringmatchlen(pattern, patternLen,
                                  string, stringLen, nocase))
                    return 1;
                string++; stringLen--;
            }
            return 0;
        case '?':
            string++; stringLen--;
            break;
        case '[':
            /* 字符类匹配,省略 */
            break;
        default:
            if (pattern[0] != string[0]) return 0;
            string++; stringLen--;
        }
        pattern++; patternLen--;
    }
    while (patternLen && pattern[0] == '*') { pattern++; patternLen--; }
    return (patternLen == 0 && stringLen == 0);
}

图 30-4-1 · PSUBSCRIBE 成本模型 — P × M 的线性扫描

为何 pattern 不用 dict 索引?dict 加速的前提是"有可哈希的 key 直接定位到 value",但 pattern 是表达式,channel 是具体字符串,两者不可逆——news.* 不能反向构造出 news。每次 PUBLISH 必须对每个 pattern 调用 stringmatchlen,无法跳过。trie/NFA 可优化前缀共享场景,但生产环境通常 < 100 个 pattern,代码复杂度收益不显著,Redis 选择保留最简洁的 list。

30.6 Cluster 模式的网络放大

Cluster 中的 PUBLISH 有一个隐藏成本:每条消息都通过 Cluster Bus 转发到所有其他节点。

/* cluster.c - 简化逻辑 */
void clusterPropagatePublish(robj *channel, robj *message) {
    dictIterator *di = dictGetIterator(server.cluster->nodes);
    dictEntry *de;
    while ((de = dictNext(di)) != NULL) {
        clusterNode *node = dictGetVal(de);
        if (node == myself) continue;
        if (node->flags & (CLUSTER_NODE_HANDSHAKE |
                           CLUSTER_NODE_NOADDR)) continue;
        clusterSendPublish(node->link, channel, message);  /* 单条放大 N-1 倍 */
    }
    dictReleaseIterator(di);
}

图 30-6-1 · PUBLISH vs SPUBLISH — Cluster Bus 放大对比

定理 30-6-1 · Cluster PUBLISH 的网络开销公式。设 Cluster 节点数 N,PUBLISH 吞吐 R msg/s,单条消息 B 字节:

  • 每条 PUBLISH 触发 Bus 消息数:M = N - 1
  • 总 Bus 消息流量:R_bus = R × (N - 1) msg/s
  • Bus 带宽:BW_bus = R × (N - 1) × B bytes/s

数值示例(B = 200 字节,R = 10,000 msg/s):

  • 小集群 N = 12: R_bus = 110,000 msg/s, BW_bus ≈ 22 MB/s
  • 大集群 N = 100: R_bus = 990,000 msg/s, BW_bus ≈ 198 MB/s

Bus 带宽是 Redis 集群协议的"血管",承担 gossip、PFAIL/FAIL、SLOTUPDATE、manual failover。PUBLISH 抢占 Bus → 心跳排队 → PFAIL 误判 → 误判 failover → 业务可用性下降。

30.7 Sharded Pub/Sub(Redis 7.0)

Sharded Pub/Sub 是对 Cluster PUBLISH 不可扩展性的根本回应:把 channel 当成 key,按 hash slot 路由,只在 owner 节点投递,网络放大降至 O(1)。

图 30-7-1 · SSUBSCRIBE/SPUBLISH 命令模型 — slot 路由 + MOVED

/* pubsub.c - 简化 */
int pubsubPublishMessageShard(robj *channel, robj *message) {
    int receivers = 0;
    dictEntry *de = dictFind(server.pubsubshard_channels, channel);
    if (de) {
        list *clients = dictGetVal(de);
        listIter li;  listNode *ln;
        listRewind(clients, &li);
        while ((ln = listNext(&li)) != NULL) {
            client *c = listNodeValue(ln);
            addReplyPubsubMessage(c, channel, message);
            receivers++;
        }
    }
    /* 关键差异: 1) 无 clusterPropagatePublish 调用 2) 无 pubsub_patterns 遍历 */
    return receivers;
}

void spublishCommand(client *c) {
    int receivers = pubsubPublishMessageShard(c->argv[1], c->argv[2]);
    if (server.cluster_enabled) {
        /* 仅向该 slot 的 replica 传播 (走复制流, 不走 Bus 广播) */
        clusterPropagatePublishShard(c->argv[1], c->argv[2]);
    }
    addReplyLongLong(c, receivers);
}

slot 迁移期的订阅中断。当 slot 9572 从 Node A 迁移到 Node B:

  • T0: Node A 接收 CLUSTER SETSLOT 9572 MIGRATING B,Node B 接收 IMPORTING A
  • T1: 已连 A 的客户端仍接收消息;新连 B 的 SSUBSCRIBE 被 MOVED 回 A
  • T2: CLUSTER SETSLOT 9572 NODE B 广播,Node A 关闭相关订阅 → 客户端被踢
  • T3: 客户端重连 B 后重新 SSUBSCRIBE,期间发布的消息已丢

工程要求:客户端必须实现"断线 → 重新订阅",期间消息按 fire-and-forget 接受丢失;关键消息应使用 Stream。

30.9 多部署模式

不同部署形态下,PUBLISH/SUBSCRIBE 的行为有明显差别,尤其是主从复制中的"PUBLISH 不经复制流"。

图 30-9-1 · 主从复制下 PUBLISH 的行为 — 主从订阅完全独立

Sentinel 自身使用 Pub/Sub 做服务发现。每个 Sentinel 实例在所有监控的 master/replica 上 SUBSCRIBE __sentinel__:hello,每 2 秒 PUBLISH 自身信息(sentinel_ip:port@master_name/epoch)。通过共同监控的 master,任意一对 Sentinel 都能互相发现,不需要手动配置完整列表。这是 Pub/Sub 的经典适用场景:fire-and-forget 与"最新存活状态"的语义天然契合。

30.10 性能与基线

单节点 fan-out 是 Pub/Sub 的核心成本曲线:订阅者越多,P99 延迟下降越快。

图 30-10-1 · 单节点 fan-out 退化曲线 — 1 → 10,000 订阅者

fan-out 取舍:

  • 小 fan-out (< 100): Pub/Sub 几乎免费
  • 中 fan-out (100-1000): 注意 P99 延迟,准备分流
  • 大 fan-out (> 1000): 必须分层架构 → Redis → Kafka → 终端

PUBLISH 总成本 = 序列化 + S × 输出缓冲写入;S × T_buffer_write 在 S=10000 时占主导,主线程被复制操作打满,其他命令延迟也会上升。

方式吞吐 (msg/s)Bus 消息/sBus 带宽
PUBLISH(N=12)~180,000~1,980,000~242 MB/s
SPUBLISH(N=12)~460,000~0~0

吞吐差异 2.6×,带宽节省 ~99%。节点越多差距越大:N=30 时 PUBLISH 接近不可用,N=100 时必然瘫痪。

图 30-11-1 · 客户端自动重订阅模板 — 指数退避 + keepalive + health_check

import time, logging, redis
from redis.exceptions import ConnectionError

CHANNELS = ["config.changed", "feature.flag"]
RETRY_BACKOFF = [1, 2, 5, 10, 30]

def run_subscriber(host, port):
    backoff_idx = 0
    while True:
        try:
            conn = redis.Redis(host=host, port=port,
                              health_check_interval=30,
                              socket_keepalive=True)
            pubsub = conn.pubsub()
            pubsub.subscribe(*CHANNELS)
            logging.info("subscribed channels=%s", CHANNELS)
            backoff_idx = 0

            for msg in pubsub.listen():
                if msg["type"] == "message":
                    handle_message(msg["channel"], msg["data"])

        except ConnectionError as ex:
            wait = RETRY_BACKOFF[min(backoff_idx, len(RETRY_BACKOFF)-1)]
            backoff_idx += 1
            logging.warning("pubsub disconnected err=%s wait=%ds", ex, wait)
            time.sleep(wait)
        except Exception as ex:
            logging.error("pubsub fatal err=%s", ex)
            raise

关键要点:

  • ✓ 异常捕获后必须重新进入循环重订阅
  • ✓ 指数退避避免瞬时雪崩
  • ✓ socket_keepalive 检测 TCP 半开连接
  • ✓ health_check_interval 主动探活
  • ✗ 不要在 listen 内 catch 后 continue(订阅状态已丢)
  • ✗ 不要不退避立即重连(可能撑爆服务器)

void subscribeCommand(client *c) {
    int j;
    if ((c->flags & CLIENT_PUBSUB) && c->resp == 2) {
        /* RESP2 下,订阅状态有命令限制 */
        addReplyError(c, "...");
        return;
    }
    for (j = 1; j < c->argc; j++)
        pubsubSubscribeChannel(c, c->argv[j]);
    c->flags |= CLIENT_PUBSUB;
}

/* src/pubsub.c - PUBLISH 命令入口 */
void publishCommand(client *c) {
    int receivers = pubsubPublishMessage(c->argv[1], c->argv[2]);
    if (server.cluster_enabled) {
        clusterPropagatePublish(c->argv[1], c->argv[2]);
    } else {
        forceCommandPropagation(c, PROPAGATE_REPL);
    }
    addReplyLongLong(c, receivers);
}

/* src/cluster.c - Cluster 广播 */
void clusterPropagatePublish(robj *channel, robj *message) {
    dictIterator *di = dictGetIterator(server.cluster->nodes);
    dictEntry *de;
    while ((de = dictNext(di)) != NULL) {
        clusterNode *node = dictGetVal(de);
        if (node == myself) continue;
        if (node->flags & (CLUSTER_NODE_HANDSHAKE | CLUSTER_NODE_NOADDR))
            continue;
        clusterSendPublish(node->link, channel, message);
    }
    dictReleaseIterator(di);
}

设计取舍

Redis Pub/Sub 选择"双向索引"是空间换时间的经典权衡。若仅 server 端持有 channel → clients 映射,客户端断连时清理需遍历所有 channel,O(全 channel × 平均订阅者)。增加 client 端镜像后,断连清理为 O(此客户端的订阅数),代价是每个订阅消耗双倍指针。该模式与关系数据库的二级索引设计同构:用一定空间换查询时间。

pubsub_patterns 选择 list 而非 dict 的根本原因是 pattern 与 channel 的不可逆关系。dict 加速的前提是有可哈希的 key 直接定位到 value;但 pattern 是表达式,channel 是具体字符串,两者不能互相反推。每次 PUBLISH 必须对每个 pattern 调用 stringmatchlen 判定,无法跳过。trie 或 NFA 可优化前缀共享场景,但代码复杂度显著增加,对生产环境通常 < 100 个 pattern 的规模收益不大,Redis 选择保留最简洁的 list 实现。

PUBLISH 不经过主从复制流是 Pub/Sub 即时性的必然结果。复制流是异步的,从节点接收时间晚于主节点,若从节点重放 PUBLISH 会导致重复投递;若仅从节点重放,主节点订阅者无消息。Redis 选择"PUBLISH 仅在执行节点处理",主从订阅者独立,代价是从节点订阅者无法完整看到全局消息流,需应用层规定连接到主节点。

Redis 7.0 引入 Sharded Pub/Sub 是对 Cluster 模式 PUBLISH 不可扩展性的根本回应。原生 PUBLISH 通过 Cluster Bus 广播保证全局可达,但 O(N) 网络放大在大集群下不可接受。Sharded Pub/Sub 将 channel 提升为 key,通过 hash slot 路由,牺牲全局可达性换取 O(1) 网络成本。该设计与 Redis Cluster 的核心 key 分片模式一致,使消息系统与数据系统共用同一拓扑模型。代价是 pattern 订阅在 Sharded 模式下不存在(SPSUBSCRIBE 不实现),因为 pattern 的全局语义与 slot 隔离原则冲突。

30.13 本章要点

  1. Pub/Sub 由 dict(channels)+ list(patterns)实现,服务端与客户端双向索引以支持 O(1) 断连清理
  2. PUBLISH 投递分两阶段:dict 查找直接订阅者,list 遍历对每个 pattern 调用 stringmatchlen
  3. PUBLISH 返回值是本节点投递客户端数,不包括 Cluster 其他节点;不代表业务层处理完成
  4. Pub/Sub 投递语义为 at-most-once:无 backlog、无 ACK,断连/慢消费者/crash 均会丢消息
  5. Cluster 普通 PUBLISH 通过 Cluster Bus 广播到所有节点,产生 O(N) 网络放大
  6. Redis 7.0 Sharded Pub/Sub(SPUBLISH/SSUBSCRIBE)将 channel 视作 key,按 slot 路由消除 Bus 放大
  7. Sharded Pub/Sub 不支持 pattern,客户端需处理 MOVED 重定向,slot 迁移触发订阅中断
  8. PUBLISH 不经主从复制流传播,主从订阅者状态独立;实践中应规定客户端连主节点
  9. 慢消费者通过 client-output-buffer-limit pubsub 32mb 8mb 60 控制
  10. 需要可靠投递时应使用 Stream;Sentinel 自身用 __sentinel__:hello 通过 Pub/Sub 做服务发现
