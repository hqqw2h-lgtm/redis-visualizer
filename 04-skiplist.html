第 4 章 · skiplist 跳表

概率多层链表 · span 排名累加 · backward 反向遍历 · dict + skiplist 双结构 · O(log N) 范围 + rank

ZSet(Sorted Set)需要在同一个数据结构上同时提供四种访问能力:ZSCORE 按 member 查 score(O(1) 期望)、ZRANGEBYSCORE 按 score 范围扫描(O(log N + M))、ZRANK 计算排名(O(log N))、ZREVRANGE 反向遍历。dict 解决不了范围扫描;平衡树(红黑树/AVL)能解决但插删需要旋转,源码复杂且不易在单线程中精确计算 span(排名累加);B 树主要为磁盘 I/O 设计,内存场景常数大。

skiplist(跳表)用 多层概率性链表 替代树旋转:每个节点以概率 P=0.25 升一层,期望高度 O(log N)。Redis 在标准跳表基础上增加 span(排名累加值)与 backward(反向指针)两个工程化扩展,使 ZRANK/ZREVRANGE 也能在 O(log N) 完成。ZSet 编码超过阈值后,采用 dict + skiplist 双结构:dict 保证 ZSCORE O(1),skiplist 保证范围与排名 O(log N)。

  • 跳表为何在 Redis 中胜出红黑树/AVL/B 树/有序数组
  • zskiplistNode 与 zskiplist 字节级布局,以及柔性数组 level[] 的可变长度
  • zslRandomLevel 几何分布与 P=0.25 的工程取舍
  • 核心:zslInsert 7 帧逐帧动画 + span 修正的代数关系
  • zslDelete 的 5 帧链路重连与 level 收缩
  • zslFirstInRange 多层下降的 O(log N) 起点定位
  • zslGetRank 沿 search path 累加 span,把 rank 从 O(N) 降到 O(log N)
  • ZRANGEBYSCORE 完整命令链路:命令层 → 范围结构 → skiplist 起点 → 顺序遍历
  • 9 种典型陷阱:雪花 ID 做 score、ZRANGEBYSCORE 全量阻塞、ZRANK 误以为 O(1) 等
  • dict 与 skiplist 共享 sds 指针把内存代价从 2x 降到 ~1.4x 的优化

4.3 zslRandomLevel 随机层数生成

4.3.1 设计动机

跳表的关键不变量:期望高度 O(log N)。这要求每层节点数大约是下一层的 1/P。Redis 选择 P=0.25(每个节点 25% 概率升一层),理由:

  • 期望节点高度 = 1/(1-P) = 4/3 ≈ 1.33,平均每节点 1.33 个 forward 指针
  • 期望搜索路径 = log1/P(N) = log4(N),N=1M 时仅约 10 层
  • P=0.5 会让节点平均 2 个指针,内存多 50%

4.3.3 源码

#define ZSKIPLIST_MAXLEVEL 32
#define ZSKIPLIST_P 0.25

int zslRandomLevel(void) {
    static const int threshold = ZSKIPLIST_P * RAND_MAX;
    int level = 1;
    while (random() < threshold)
        level += 1;
    return (level < ZSKIPLIST_MAXLEVEL) ? level : ZSKIPLIST_MAXLEVEL;
}

4.4 zslInsert 插入操作(本章核心)

图 4-4-1 · zslInsert 调用链

图 4-4-2 · zslInsert 关键状态转移(插入 X = score 15)

span 修正是 zslInsert 的核心难点。普通跳表只需重连 forward,但 Redis 要支持 O(log N) rank,因此每个 forward 都附带"跨越节点数" span。插入后,新节点把"原 update→original forward"这一段跨越距离切成两段:update 到 X 一段,X 到 original forward 一段。两段之和 = 原 span + 1(因为多穿过新节点)。

4.4.2 span 修正的代数关系

表 4-4-2 · span 修正代数公式
层范围变更对象更新公式
i ∈ [0, k-1]
(新节点存在)		新节点 x.level[i].spanupdate[i].span − (rank[0] − rank[i])
i ∈ [0, k-1]update[i].level[i].span(rank[0] − rank[i]) + 1
i ∈ [k, zsl.level-1]
(新节点穿过, 无 forward)		update[i].level[i].span++ (穿过新节点)
代数验证 (level i):
  • 原 update[i] 到 update[i].forward 之间的 level 0 距离 = update[i].span
  • 新节点 x 在 level 0 的 rank = rank[0] + 1
  • update[i] 到 x 在 level 0 的距离 = rank[0] − rank[i] + 1
  • x 到原 forward 在 level 0 的距离 = update[i].span − (rank[0] − rank[i])
两段距离之和 = update[i].span + 1 ✓ (穿过新节点 +1)

4.9.2 设计动机

Redis 选择 skiplist + span 扩展 + dict 双结构 的组合,核心动机:

  1. 源码复杂度可控:跳表无旋转,代码简洁,易于维护与调试
  2. 同时支持范围 + rank:span 字段使 rank 计算从标准跳表的 O(N) 降到 O(log N),无需平衡树的子树大小维护
  3. 概率平衡的工程鲁棒性:期望性能 O(log N) 在 N 足够大时统计稳定,无最坏情况偏置
  4. dict 共享 sds 内存:双结构看似 2x 内存,实际通过 sds 指针共享降至 ~1.4x
  5. 反向遍历支持:level 0 的 backward 指针使 ZREVRANGE 与正向遍历同样 O(M)

2ZRANK 误以为 O(1)调用频繁时 CPU 与 N 正相关ZRANK 是 O(log N),不是 O(1)频繁场景缓存 rank;ZSCORE 替代
3大 ZSet 全量 ZRANGE 0 -1同 1同 1分页 + 限制
4雪花 ID 做 score排序错乱、相邻 ID 被合并double 53-bit 整数精度极限ID 放 member, score 用安全数值范围
5浮点 score 累加(ZINCRBY)长期累加后精度漂移double 浮点累加误差使用整数倍数(如 ×1000)规避
6小 ZSet 不是 skiplist 编码OBJECT ENCODING 显示 listpacklistpack 阈值未触发不要假设源码路径,用 OBJECT ENCODING 验证
7listpack 阈值调过大中等 ZSet 操作变慢listpack 内 O(N) 扫描保持默认 zset-max-listpack-entries=128
8同 score 排序假设错误业务依赖错误的稳定排序同 score 时按 member 字典序排显式包含次级排序键
9ZADD GT NX 同时使用命令报错NX 与 GT/LT 互斥二选一
陷阱 4 — 雪花 ID 做 score 是最常见的"看起来工作但实际错误"案例。 Redis score 是 IEEE 754 double,有效整数范围 ±253(≈ 9 × 1015);典型雪花 ID 64 bit,大于 253。当两个相邻 ID 的低位差异落在 53-bit 之外时,double 会"舍入到同值",导致 ZRANK 返回错误,或两条不同消息被视作同 score。
正确做法:把 ID 放进 member,用毫秒时间戳(13-14 位整数,远小于 253)做 score。

真实案例 — 排行榜雪花 ID 灾难
某社交 App 把"发帖时间"写成雪花 ID 后直接做 ZSet score(ZADD feed:user 1714000000123456789 post_xxx),上线两周后发现"同毫秒发布的不同帖子在 ZRANGE 中互相覆盖、ZRANK 偶发返回错误索引"。排查发现:1714000000123456789 ≈ 260.6,远超 double 的 253 整数精度,低位 8 bit 全部丢失。修复:score = floor(snowflake_id / 1024)(主键放 member,score 仅用于排序),配合 OBJECT ENCODING 监控编码切换。

4.11.1 关键源码片段

zskiplistNode *zslInsert(zskiplist *zsl, double score, sds ele) {
    zskiplistNode *update[ZSKIPLIST_MAXLEVEL], *x;
    unsigned long rank[ZSKIPLIST_MAXLEVEL];
    int i, level;

    x = zsl->header;
    for (i = zsl->level-1; i >= 0; i--) {
        rank[i] = i == (zsl->level-1) ? 0 : rank[i+1];
        while (x->level[i].forward &&
               (x->level[i].forward->score < score ||
                (x->level[i].forward->score == score &&
                 sdscmp(x->level[i].forward->ele, ele) < 0))) {
            rank[i] += x->level[i].span;
            x = x->level[i].forward;
        }
        update[i] = x;
    }

    level = zslRandomLevel();
    if (level > zsl->level) {
        for (i = zsl->level; i < level; i++) {
            rank[i] = 0;
            update[i] = zsl->header;
            update[i]->level[i].span = zsl->length;
        }
        zsl->level = level;
    }
    x = zslCreateNode(level,score,ele);

    for (i = 0; i < level; i++) {
        x->level[i].forward = update[i]->level[i].forward;
        update[i]->level[i].forward = x;
        x->level[i].span = update[i]->level[i].span - (rank[0] - rank[i]);
        update[i]->level[i].span = (rank[0] - rank[i]) + 1;
    }
    for (i = level; i < zsl->level; i++) {
        update[i]->level[i].span++;
    }

    x->backward = (update[0] == zsl->header) ? NULL : update[0];
    if (x->level[0].forward)
        x->level[0].forward->backward = x;
    else
        zsl->tail = x;
    zsl->length++;
    return x;
}

取舍 1 — 概率 vs 严格平衡:跳表理论最坏 O(N)(所有节点都是 level 1),但概率极小(P(全 level 1) = 0.75N,N=1000 时约 10-125)。Redis 接受这种概率风险,换得源码简洁。
取舍 4 — MAXLEVEL=32:理论上跳表无层数上限,但 P=0.25 + N ≤ 264 的场景下,期望最高层 ≈ log4(264) = 32。32 既够用又便于编译期分配 update[] 与 rank[] 数组。
