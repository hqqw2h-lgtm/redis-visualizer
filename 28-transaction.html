第 28 章 · 事务 MULTI/EXEC

MULTI / EXEC / DISCARD / WATCH 四件套 · 单线程串行执行 · CLIENT_MULTI 与 multiState 队列 · CAS 乐观锁与 signalModifiedKey 钩子 · 不回滚的工程取舍 · ACID 满足度分析 · Cluster 单 slot 约束

Redis 事务的语义边界与传统 RDBMS 事务存在显著差异。Redis 通过 MULTI / EXEC / DISCARD / WATCH 四个命令提供"批量提交 + 串行执行 + 乐观锁"语义,但不提供 ACID 中的回滚特性。理解 Redis 事务的关键是识别这一边界:Redis 事务的"原子性"指 EXEC 期间命令不被其他客户端命令插入,而非"全部成功或全部回滚"。

这套设计借助 Redis 主线程的顺序执行:EXEC 阶段的命令按队列连续执行,期间不会被其他客户端命令插入。注意,这只说明执行隔离,不等于传统数据库的"失败回滚"或提交日志。Redis 不做回滚,是为了避免在热路径维护 undo log / snapshot。WATCH 则提供乐观锁(OCC)语义,让客户端在 EXEC 前发现被监视 key 是否被其他写命令改过。

本章按"语义模型 → 四个命令 → WATCH 机制 → 不回滚的工程权衡 → ACID 分析 → 部署适配 → 陷阱"组织,强调事务的边界条件与生产可观测性。

  • 入队语法错误(CLIENT_DIRTY_EXEC)与执行期类型错误的两种处理路径区别
  • DISCARD 清理 multiState、解除 WATCH、清除 flag 的完整资源回收
  • watched_keys 字典与 signalModifiedKey 钩子链的零成本设计

28.0 章首速览

图 28-0-1 · 事务全景:三阶段 + 双分支

事务从客户端视角是三阶段:WATCH(可选注册监视) → MULTI(入队命令) → EXEC(批量提交)。EXEC 在服务端有双分支:无 dirty 标志时串行执行队列并返回结果数组;有 CLIENT_DIRTY_CAS 或 CLIENT_DIRTY_EXEC 时整体放弃,返回 nil 或 EXECABORT 错误。

核心洞察: Redis 事务的"原子性"是 scheduling atomicity(调度原子性) — EXEC 内部不被其他命令插入,而不是 RDBMS 的 all-or-nothing atomicity(全有全无原子性)。如果把 Redis 事务当成会自动回滚的数据库事务,就会误判错误处理方式:例如 INCR 字符串失败时,前面的 SET 仍可能已经生效。
图 28-0-2 · 事务的三个核心约束

三类约束的精确组合定义了 Redis 事务的能力边界,任何一项理解偏差都会导致生产事故。常见误区是将其等同于 RDBMS 事务,期待回滚保证 — 实际遇到 INCR 字符串等运行时错误时,部分命令已生效,数据出现非预期状态。把事务当作"减少网络 RTT + 防止被插入 + 检测并发冲突"三件套是正确认知,期待回滚保证是错误认知。

28.1 为什么需要事务

28.1.1 无事务下的"读-计算-写"竞争条件

场景: 两客户端并发扣余额

  时刻 t   Client A 视角             Client B 视角            balance
  ──────  ───────────────────────  ──────────────────────  ───────
  t = 0   GET balance → 100        -                        100
  t = 1   计算 new = 100 - 10 = 90  GET balance → 100       100
  t = 2   -                        计算 new = 100 - 20 = 80  100
  t = 3   SET balance 90           -                        90
  t = 4   -                        SET balance 80           80

  最终 balance = 80
  期望 balance = 70 (扣 10 + 扣 20)
  少扣 10,数据丢失

根因: 两客户端基于过期数据 (t=0 的 100) 各自计算并写回
      Redis 单线程不能防止"读-计算-写"序列被其他客户端插入

Redis 单线程模型只保证每条命令的原子性,不能阻止两条命令之间被其他客户端的命令插入。GET → SET 之间的客户端计算时间窗口是竞争条件的根源,该问题在 DECRBY balance 10 单命令时不存在(原子完成),但在"读取后做业务校验再写"模式下普遍存在。

28.1.2 WATCH + MULTI/EXEC 的解决方案

图 28-1-2 · 乐观锁冲突检测时序
客户端负责,Redis 不自动重试 — 退避策略避免活锁 ✗ DIRTY_CAS 生命周期:client 级别,持续到 EXEC/DISCARD/UNWATCH;过期/淘汰也会触发

WATCH 提供 CAS (Compare-And-Swap) 语义。客户端读取后到 EXEC 前,任何其他客户端对监视 key 的修改都会触发 DIRTY_CAS 标志,导致 EXEC 返回 nil。客户端必须有重试逻辑 — 该重试由业务代码控制,Redis 不自动重试。

重要约束: WATCH 检测的是"是否有任何修改",不区分 value 是否真正改变。对同一 key 重复 SET 相同 value 也会触发 DIRTY_CAS。这是简化实现的取舍 — 精确比较 value 需要保存旧值,代价不值得。

28.2 四大核心命令

28.2.1 MULTI - 仅设置标志位

/* src/server.h */
#define CLIENT_MULTI (1<<3)

/* src/multi.c */
void multiCommand(client *c) {
    if (c->flags & CLIENT_MULTI) {
        addReplyError(c, "MULTI calls can not be nested");
        return;
    }
    c->flags |= CLIENT_MULTI;
    addReply(c, shared.ok);
}
MULTI 的实际工作:
  仅设置一个 client flag,不创建任何数据结构

  设置后的副作用:
    后续命令进入 queueMultiCommand 而非 processCommand 主路径
    redis-cli 提示符变为 "(TX)"
    回复变为 "QUEUED" (除特殊命令)

不可嵌套:
  若已设置 CLIENT_MULTI 再发 MULTI → 报错
  Redis 事务是单层的,无嵌套概念
  代价: 客户端组合事务的工作必须在客户端完成

特殊命令在事务中:
  MULTI / EXEC / DISCARD / WATCH 不入队,直接处理
  其他所有命令一律入队
极简实现: MULTI 命令本身只有 6 行 C 代码 — 检查 flag 是否已设置(报错), 设置 flag, 回复 +OK。没有创建队列、没有分配内存、没有锁。所有事务复杂度都集中在 queueMultiCommand 与 execCommand 两个函数中,符合 Redis 命令"小函数 + 显式状态"的代码风格。

28.2.2 命令入队 - 仅记录不执行

/* src/multi.c */
typedef struct multiCmd {
    robj **argv;
    int argv_len;
    int argc;
    struct redisCommand *cmd;
} multiCmd;

typedef struct multiState {
    multiCmd *commands;
    int count;
    int cmd_flags;
    int cmd_inv_flags;
    size_t argv_len_sums;
    int alloc_count;
} multiState;

void queueMultiCommand(client *c, uint64_t cmd_flags) {
    multiCmd *mc;

    if (c->flags & CLIENT_DIRTY_EXEC) return;

    if (c->mstate.count == 0) c->mstate.cmd_flags = 0;
    if (c->mstate.alloc_count == c->mstate.count) {
        c->mstate.alloc_count = (c->mstate.alloc_count + 1) * 2;
        c->mstate.commands = zrealloc(c->mstate.commands,
            sizeof(multiCmd) * c->mstate.alloc_count);
    }
    mc = c->mstate.commands + c->mstate.count;
    mc->cmd = c->cmd;
    mc->argc = c->argc;
    mc->argv = zmalloc(sizeof(robj*) * c->argc);
    memcpy(mc->argv, c->argv, sizeof(robj*) * c->argc);
    for (int j = 0; j < c->argc; j++)
        incrRefCount(mc->argv[j]);
    c->mstate.count++;
    c->mstate.cmd_flags |= cmd_flags;
}
入队的内存开销:

  每条入队命令:
    multiCmd 结构: 32 bytes
    argv 指针数组: 8 bytes × argc
    每个 robj 引用计数 +1 (复用对象,不拷贝)

  事务总开销:
    O(commands_count × avg_argc)
    通常远小于 1 KB
    长事务 (1000+ 命令) 才需关注内存

入队期间的语法验证:
  命令是否存在 (cmd != NULL)
  参数个数是否匹配 cmd->arity
  Cluster 模式: 所有 key 是否在同一 slot

任一验证失败:
  client->flags |= CLIENT_DIRTY_EXEC
  当前命令返回错误
  EXEC 时整体拒绝事务

入队仅复制 argv 指针并增加引用计数,不拷贝 robj 对象本身,故内存开销远小于直接执行命令。cmd_flags 累积所有入队命令的标志位,EXEC 阶段可基于此判断事务是否包含写命令(CMD_WRITE)、是否需要复制传播等。

28.2.3 EXEC - 批量执行

/* src/multi.c (简化) */
void execCommand(client *c) {
    int j;
    robj **orig_argv;
    int orig_argc;
    struct redisCommand *orig_cmd;

    if (!(c->flags & CLIENT_MULTI)) {
        addReplyError(c, "EXEC without MULTI");
        return;
    }

    /* 1. WATCH 冲突检测 */
    if (c->flags & (CLIENT_DIRTY_CAS | CLIENT_DIRTY_EXEC)) {
        if (c->flags & CLIENT_DIRTY_EXEC)
            addReplyErrorFormat(c, "EXECABORT Transaction discarded "
                                  "because of previous errors.");
        else
            addReply(c, shared.nullarray[c->resp]);
        discardTransaction(c);
        return;
    }

    /* 2. 串行执行所有入队命令 */
    addReplyArrayLen(c, c->mstate.count);
    for (j = 0; j < c->mstate.count; j++) {
        c->argc = c->mstate.commands[j].argc;
        c->argv = c->mstate.commands[j].argv;
        c->cmd = c->mstate.commands[j].cmd;

        call(c, CMD_CALL_FULL);
    }

    discardTransaction(c);
}
EXEC 阶段的关键不可中断性:

  for 循环在主线程的 processCommand 调用栈内完成
  事件循环不会返回 epoll_wait
  其他客户端的命令在该期间被排队等待

  保证: EXEC 内的所有命令构成一个原子调度单元
        任意时刻最多一个事务的 EXEC 在执行
        多事务的 EXEC 形成全序关系 (无交错执行)

非原子性: 单命令失败
  call() 返回错误不会中断 for 循环
  错误对象作为该命令的结果加入 reply 数组
  下一条命令照常执行
  最终 reply 数组中既有成功也可能有错误
核心机制: EXEC 的执行循环期间不会回到事件循环处理其他客户端命令。因此,事务队列越长,其他客户端等待越久。这里要区分两件事:Redis 保证 EXEC 内部不被插队;它不保证队列里每条命令都成功,也不会在后续命令失败时回滚前面的写入。

28.2.4 DISCARD - 放弃事务

/* src/multi.c */
void discardCommand(client *c) {
    if (!(c->flags & CLIENT_MULTI)) {
        addReplyError(c, "DISCARD without MULTI");
        return;
    }
    discardTransaction(c);
    addReply(c, shared.ok);
}

void discardTransaction(client *c) {
    freeClientMultiState(c);
    initClientMultiState(c);
    c->flags &= ~(CLIENT_MULTI | CLIENT_DIRTY_CAS | CLIENT_DIRTY_EXEC);
    unwatchAllKeys(c);
}
DISCARD 的清理工作:
  释放所有已入队 multiCmd 的 argv
  减少 robj 引用计数
  清除 CLIENT_MULTI / DIRTY_CAS / DIRTY_EXEC flags
  解除所有 WATCH

使用场景:
  业务代码检测到无需提交事务时主动放弃
  入队过程中发现错误,主动清理
  连接断开前的资源回收

注意:
  DISCARD 后客户端回到普通命令模式
  之前 WATCH 的 key 不再有效
  需要重新 WATCH + MULTI 开始新事务

28.3 事务执行流程

28.3.1 processCommand 中的事务分支

图 28-3-1 · 命令分发的事务分支

processCommand 在事务模式下分两条路径:特殊命令(MULTI / EXEC / DISCARD / WATCH / RESET)直接执行,其他命令进入 queueMultiCommand。这个分支的设计使事务命令本身不入队(否则会陷入死循环),普通命令统一入队。

28.3.2 client 状态转移

图 28-3-2 · 客户端状态机
client flags 与状态对应:

  状态                CLIENT_MULTI  CLIENT_DIRTY_CAS  CLIENT_DIRTY_EXEC
  ─────────────────  ────────────  ────────────────  ─────────────────
  Normal              0             0                  0
  Watching            0             0                  0
  Watching (有冲突)    0             1                  0
  InTx (干净)          1             0                  0
  InTx (有冲突)        1             1                  0
  InTx (有错误)        1             0                  1

EXEC 决策依据:
  DIRTY_EXEC 优先于 DIRTY_CAS
  DIRTY_EXEC = 1 → EXECABORT (整体拒绝)
  DIRTY_CAS = 1  → null array (WATCH 冲突)
  全 0           → 正常执行

28.4 WATCH:乐观锁与 CAS 语义

28.4.1 WATCH 的内部数据结构

/* src/multi.c */
typedef struct watchedKey {
    listNode node;
    robj *key;
    redisDb *db;
    client *client;
    unsigned expired:1;
} watchedKey;

/* src/server.h - redisDb */
struct redisDb {
    /* ... */
    dict *watched_keys;  /* key -> list<watchedKey> */
    /* ... */
};
watched_keys 字典结构:

  redisDb.watched_keys:
    "balance" → [watchedKey{client_A}, watchedKey{client_C}]
    "stock"   → [watchedKey{client_B}]
    "name"    → [watchedKey{client_A}, watchedKey{client_B}]

  client A 同时 WATCH 了 "balance" 与 "name"

查找路径:
  写命令执行后:
    db.watched_keys[key] → 链表
    遍历链表,给每个 client 设 DIRTY_CAS

  EXEC 时:
    检查当前 client 的 DIRTY_CAS flag
    O(1)

watched_keys 是反向索引:从 key 反查"哪些 client 在监视它"。这避免了每次 WATCH 时查找全局所有 client 的低效扫描,使 signalModifiedKey 的开销与"该 key 的 watcher 数量"成正比,而非"全局 client 数量"。

28.4.2 signalModifiedKey 的触发链

/* src/db.c */
void signalModifiedKey(client *c, redisDb *db, robj *key) {
    touchWatchedKey(db, key);
    trackingInvalidateKey(c, key, 1);
}

void touchWatchedKey(redisDb *db, robj *key) {
    list *clients;
    listIter li;
    listNode *ln;

    if (dictSize(db->watched_keys) == 0) return;

    clients = dictFetchValue(db->watched_keys, key);
    if (!clients) return;

    listRewind(clients, &li);
    while ((ln = listNext(&li))) {
        watchedKey *wk = listNodeValue(ln);
        wk->client->flags |= CLIENT_DIRTY_CAS;
    }
}
signalModifiedKey 嵌入所有写命令路径:

  src/t_string.c:
    setGenericCommand → signalModifiedKey (SET 后)
    incrCommand       → signalModifiedKey (INCR 后)
    appendCommand     → signalModifiedKey (APPEND 后)

  src/t_list.c:
    listTypePush      → signalModifiedKey
    listTypePop       → signalModifiedKey

  src/t_hash.c:
    hashTypeSet       → signalModifiedKey

  src/db.c:
    dbDelete          → signalModifiedKey (DEL/UNLINK)
    expireGenericCommand → signalModifiedKey

  ... (所有修改 keyspace 的命令)

性能开销:
  if (dictSize(watched_keys) == 0) return;  ← 大多数情况零开销
  有 watcher 时:
    一次哈希查找
    一次链表遍历
    每个 watcher O(1) 设 flag
低成本设计: if (dictSize(watched_keys) == 0) return; 这一行让没有 WATCH 的写路径很快跳过 OCC 逻辑。只有真正有人监视某个 key 时,Redis 才需要查找 watcher 并标记 DIRTY_CAS。这体现的是"按需付费",不是"完全没有成本"。

监视的 key 不存在,新创建✅ 是新建 key 也触发 signalModifiedKey
监视的 key 内部修改但 value 相同(SET k v 已有 v)✅ 是不做 value 比较,简化实现
监视的 key 已过期但未删除(惰性删除中)❌ 否过期后续 expireIfNeeded 才触发
Cluster 模式下从 Replica 监视—不支持,Replica 拒绝 WATCH

28.4.5 典型的 WATCH 重试模式

def transfer(redis_cli, src, dst, amount, max_retries=5):
    for attempt in range(max_retries):
        try:
            with redis_cli.pipeline() as pipe:
                pipe.watch(src, dst)
                src_balance = int(pipe.get(src) or 0)
                if src_balance < amount:
                    pipe.unwatch()
                    raise ValueError("insufficient balance")
                pipe.multi()
                pipe.decrby(src, amount)
                pipe.incrby(dst, amount)
                pipe.execute()
                return True
        except WatchError:
            sleep(0.01 * (2 ** attempt))
    raise RuntimeError("max retries exceeded")
重试逻辑要点:

  ✓ 限制最大重试次数,避免无限循环
  ✓ 指数退避,避免冲突时多客户端同时重试
  ✓ 重试时重新读取数据,不能复用旧值
  ✓ 业务条件检查 (如余额) 必须在 WATCH 后

  ✗ 不要在 EXEC 失败后立即重试,可能立即再冲突
  ✗ 不要假设 EXEC 失败一定是 WATCH 冲突,
    EXECABORT 也可能,需区分错误类型

28.5.3 不回滚的设计依据

图 28-5-3 · 设计取舍的两条根因链

官方文档给出的设计依据可以概括为两点:常见的事务失败多来自命令写错或类型不匹配,应在上线前通过校验和测试发现;而回滚需要 undo log 或 snapshot,会显著增加复杂度与延迟。因此 Redis 选择快速暴露错误,而不是在服务端补做运行时回滚。

28.5.4 替代方案 - Lua 前置校验

-- 转账场景:先校验余额,再执行写入
EVAL "
  local src_balance = tonumber(redis.call('GET', KEYS[1]) or 0)
  local amount = tonumber(ARGV[1])

  if src_balance < amount then
    return {err = 'insufficient balance'}
  end

  redis.call('DECRBY', KEYS[1], amount)
  redis.call('INCRBY', KEYS[2], amount)
  return 'OK'
" 2 account_a account_b 100

MULTI / EXEC 适用场景

  • 纯写入,不需要基于读结果决策
  • 客户端组合的批量操作
  • 需要 WATCH 提供 OCC 检测的场景
  • 团队不熟悉 Lua 时的首选
  • 单批次 ≤ 10 命令的轻量事务

Lua 脚本适用场景

  • 需要基于读取结果先校验、再写入
  • 需要基于读结果做条件分支
  • 大量命令的原子打包
  • 需要多次访问同一 key 的复杂计算
  • 性能敏感(单 RTT 比 MULTI/EXEC 少 1 次)
典型风险案例: 用 MULTI/EXEC 实现转账时,如果余额检查留在客户端,而入队阶段只包含 DECRBY/INCRBY,WATCH 只能检测 key 是否被别人改过,不能替你判断余额是否足够。结果可能是"校验失败但写命令仍被提交"。这类场景更适合用 Lua / Functions 在服务端一次性完成"读 + 校验 + 写"。

28.6.2 EXEC 的无交错执行

结论: EXEC 阶段的命令不会与其他客户端命令交错

设 T_1, T_2, ..., T_n 为 n 个并发提交的事务

Redis 单线程模型保证:
  ∀ i ≠ j: time_range(EXEC(T_i)) ∩ time_range(EXEC(T_j)) = ∅

  即任意两事务的 EXEC 时间区间不重叠

证明:
  EXEC 的执行在 processCommand 调用栈内完成
  for (j = 0; j < c->mstate.count; j++) call(c, ...)
  for 循环不调用 aeProcessEvents,事件循环不返回

  因此 EXEC 内的所有命令构成不可分割的执行单元

  多事务的 EXEC 命令到达时间在事件循环中形成全序
  执行顺序 = 到达顺序
  执行结果等价于 (T_1, T_2, ..., T_n) 某种全排列的串行调度

  从调度角度看:
    多个 EXEC 可以排列成一个串行顺序
    其他客户端命令不会插入某个 EXEC 内部

  ⊢ 可以把 EXEC 看作一个连续执行片段,但不包含回滚语义  ∎

28.6.3 与传统数据库的对比

SQL 标准的四个隔离级别

  • Read Uncommitted — 允许脏读
  • Read Committed — 防脏读
  • Repeatable Read — 防脏读 + 不可重复读
  • Serializable — 防脏读 + 不可重复读 + 防幻读

实现方式:MVCC 多版本 + 行锁 / 谓词锁 / SSI 快照隔离。

Redis 事务的 Serializable

  • 不允许任何并发访问 EXEC 内部
  • 天然达到 Serializable
  • 无 MVCC,无锁,无快照
  • 实现成本几乎为零

代价:单线程瓶颈,无法多核扩展;长事务占用主线程,影响其他客户端;生产建议事务尽可能短,避免在事务中做计算。

对比 PostgreSQL Serializable Snapshot Isolation (SSI):
  PostgreSQL 通过快照 + 提交时检测读写依赖
  允许并发执行,提交时决定中止哪些事务

  Redis 通过单线程串行,不需要快照与依赖检测

  PostgreSQL: 高并发,可能中止大量事务 (吞吐受冲突影响)
  Redis: 单线程,执行确定但吞吐受限于单核

适用场景对比:
  PG SSI:  OLTP 高并发,事务可能很长 (秒级)
  Redis:   缓存 / 计数器 / 分布式锁,事务必须很短 (毫秒级)

# Cluster 模式 - 跨 slot 报错
127.0.0.1:7000> MULTI
OK

QUEUED
127.0.0.1:7000(TX)> SET payment:P9001 done

(error) CROSSSLOT Keys in request don't hash to the same slot
Cluster 事务约束:

  EXEC 期间所有 key 必须在同一节点

  解决方案: hash tag
    SET order:{customer:C001}      paid
    SET payment:{customer:C001}    done
    SET shipment:{customer:C001}   pending

    所有 key 用 {customer:C001} 作为 hash tag
    都映射到同一 slot,同一节点
    EXEC 可以正常执行

副作用:
  hash tag 设计导致负载倾斜
  高频 customer 的所有相关数据集中在一个 slot
  该 slot 所在节点 CPU 高,其他节点空闲
  生产建议: hash tag 粒度选择业务自然边界,避免过粗
hash tag 粒度陷阱: 若所有用户都用 {global} 作为 hash tag,所有 key 都路由到同一 slot,Cluster 退化为单节点!正确做法是按"业务原子单元"选择 tag — 例如订单系统按 {order_id},社交系统按 {user_id},确保每个 tag 对应可独立提交的事务范围,粒度过粗导致热点,过细导致跨 tag 操作无法原子化。

28.8.2 事务内常见命令耗时

单事务执行时间 = sum(每命令耗时)

  典型耗时 (8 核 32 GB):
    SET / GET            ~1-3 µs
    INCR / DECR          ~1-3 µs
    HSET / HGET          ~2-5 µs
    LPUSH / RPUSH        ~2-5 µs
    SADD                 ~3-7 µs
    ZADD                 ~5-10 µs
    HMGET (10 fields)    ~10-20 µs

  10 条命令的事务: 通常 < 100 µs
  100 条命令的事务: ~1 ms

  超过 1 ms 的事务影响其他客户端
  超过 10 ms 的事务严重阻塞主线程
  超过 100 ms 的事务可能触发 cluster failover

生产建议:
  事务命令数 ≤ 10
  避免在事务中执行 O(N) 命令 (HGETALL/SMEMBERS)
  超大事务拆分为多批
Cluster 联动: cluster-node-timeout 默认 15 s,但运维通常调到 5 s 甚至 2 s。如果事务执行时间接近这个阈值,主节点会被同 shard 的 replica + 其他 master 投票判定为"不可达",触发误 failover。这是"事务必须毫秒级"的硬性工程约束 — 事务时长直接决定 Cluster 稳定性。

❌ 反模式 · 仅检查 EXEC 是否报错

with redis_cli.pipeline(transaction=True) as pipe:
    pipe.set("name", "Bob")
    pipe.incr("name")           # 类型错误
    pipe.set("age", 30)
    results = pipe.execute()    # 不抛异常!

    print("transaction OK")     # 误报"成功"
# 实际 name='Bob' 已写入,业务以为完整一致

客户端只看 execute() 是否抛异常,忽略 results 数组中可能混入 Exception 对象。这是生产环境最常见的事务相关 bug。

✅ 正确做法 · 逐项检查 reply 数组

with redis_cli.pipeline(transaction=True) as pipe:
    pipe.set("name", "Bob")
    pipe.incr("name")
    pipe.set("age", 30)
    results = pipe.execute()

failed = [(i, r) for i, r in enumerate(results)
          if isinstance(r, Exception)]
if failed:
    log.error(f"transaction partial failure: {failed}")
    # 触发补偿/告警逻辑
    raise TransactionPartialFailure(results)

逐项检查每个 reply 是否为 Exception,区分 WatchError(可重试)与 PartialFailure(应告警 + 补偿)。

28.9.4 正确的客户端处理模板

def execute_transaction(redis_cli, watched_keys, transaction_fn,
                        max_retries=5):
    """
    执行 WATCH + MULTI/EXEC 事务,带重试与错误处理
    """
    for attempt in range(max_retries):
        try:
            with redis_cli.pipeline() as pipe:
                pipe.watch(*watched_keys)

                read_results = transaction_fn.read(pipe)

                if not transaction_fn.should_proceed(read_results):
                    pipe.unwatch()
                    return None

                pipe.multi()
                transaction_fn.write(pipe, read_results)

                results = pipe.execute()

                for i, r in enumerate(results):
                    if isinstance(r, Exception):
                        log.error(f"command {i} failed: {r}")
                        raise TransactionPartialFailure(results)

                return results
        except WatchError:
            sleep(0.01 * (2 ** attempt))
            continue
    raise TransactionRetriesExhausted()
模板要点:

  ✓ 限制最大重试次数
  ✓ 指数退避避免冲突风暴
  ✓ 业务前置校验在 MULTI 前
  ✓ 不应执行时 unwatch
  ✓ 检查 reply 数组每项是否为 Exception
  ✓ 区分 WatchError (重试) 与 PartialFailure (告警)

28.10.3 watchForKey 注册

/* src/multi.c: WATCH 注册 */
void watchForKey(client *c, robj *key) {
    list *clients = NULL;
    listIter li;
    listNode *ln;
    watchedKey *wk;

    /* 检查已监视,避免重复 */
    listRewind(c->watched_keys, &li);
    while ((ln = listNext(&li))) {
        wk = listNodeValue(ln);
        if (wk->db == c->db && equalStringObjects(key, wk->key))
            return;
    }

    /* 添加到 db 的 watched_keys 字典 */
    clients = dictFetchValue(c->db->watched_keys, key);
    if (!clients) {
        clients = listCreate();
        dictAdd(c->db->watched_keys, key, clients);
        incrRefCount(key);
    }
    /* clients 是 watcher 链表 */

    wk = zmalloc(sizeof(*wk));
    wk->key = key;
    wk->client = c;
    wk->db = c->db;
    incrRefCount(key);
    listAddNodeTail(c->watched_keys, wk);
    listAddNodeTail(clients, wk);
}

28.10.4 EXEC 主循环完整版

/* src/multi.c: EXEC 主循环 */
void execCommand(client *c) {
    if (!(c->flags & CLIENT_MULTI)) { /* ... 报错 */ }

    if (c->flags & (CLIENT_DIRTY_CAS | CLIENT_DIRTY_EXEC)) {
        if (c->flags & CLIENT_DIRTY_EXEC)
            addReplyErrorObject(c, shared.execaborterr);
        else
            addReply(c, shared.nullarray[c->resp]);
        discardTransaction(c);
        return;
    }

    int orig_argc = c->argc;
    robj **orig_argv = c->argv;
    struct redisCommand *orig_cmd = c->cmd;

    addReplyArrayLen(c, c->mstate.count);
    for (int j = 0; j < c->mstate.count; j++) {
        c->argc = c->mstate.commands[j].argc;
        c->argv = c->mstate.commands[j].argv;
        c->cmd = c->mstate.commands[j].cmd;

        call(c, CMD_CALL_FULL);
    }

    c->argv = orig_argv;
    c->argc = orig_argc;
    c->cmd = orig_cmd;
    discardTransaction(c);
}

28.10.5 设计取舍

Redis 事务选择"批量提交 + 连续执行 + 乐观锁"而非传统 ACID 事务,核心原因是保持单线程命令路径简单。EXEC 内部不被其他客户端命令插入,无需 MVCC、锁或依赖图;代价是没有回滚能力。业务代码应在 MULTI 前完成能做的校验,并用测试覆盖类型错误、参数错误等可预防问题。

WATCH 的设计利用了写命令路径上已存在的 signalModifiedKey() 钩子,无需引入额外锁结构。该机制的工程亮点是"开销与监视客户端数量成正比,不监视时近零开销"。if (dictSize(watched_keys) == 0) return; 使绝大多数无 WATCH 场景的写命令不承担任何 OCC 开销。

不回滚的工程权衡有其内在一致性:许多事务错误来自参数、命令或 key 类型不符合预期,这些问题更适合在测试与上线前校验中发现。回滚机制能减少部分事故影响,但也会让每次写入都背上额外记录成本。Redis 的选择是让错误尽快暴露,把补偿与重试交给业务层。

Cluster 中事务限于单 slot,根本原因是跨节点原子提交需要协调协议、超时处理和部分失败恢复。Redis Cluster 避免在核心路径引入这套分布式事务机制,因此选择"同一 slot 内执行,跨 slot 由应用拆分或改用其他方案"。

28.11 本章要点

#要点
1Redis 事务的"原子性"指 EXEC 期间不被插入,而非"全部成功或全部回滚",该差异是误用根源
2MULTI 仅设置 CLIENT_MULTI flag;命令入队为 multiCmd 结构;EXEC 串行执行队列
3EXEC 阶段连续执行,期间不处理其他客户端命令;这提供执行隔离,不提供失败回滚
4入队语法错误设 CLIENT_DIRTY_EXEC,EXEC 时整体拒绝(EXECABORT);执行类型错误不影响其他命令
5WATCH 注册 client 到 db.watched_keys[key] 链表,signalModifiedKey 钩子在写命令路径触发 DIRTY_CAS
6EXEC 检查 DIRTY_EXEC 优先于 DIRTY_CAS,DIRTY_EXEC → EXECABORT,DIRTY_CAS → null array
7不回滚的工程权衡:类型错误是代码 bug 不是运行时竞争,实现回滚需 undo log 显著增加复杂度
8需要基于读结果做条件分支时,优先用 Lua / Functions 在服务端先校验再写
9Cluster 模式事务要求所有 key 在同一 slot,通过 hash tag {} 强制,粒度选择需平衡热点与解耦
10生产模板:WATCH + GET + MULTI + 命令 + EXEC + 检查每项 reply,WatchError 重试,Exception 告警

章末总结

第 28 章事务与第 27 章 Cluster、第 29 章 Lua 共同构成 Redis 多命令编排的三层基础:

  • 第 27 章(Cluster)解决"命令在哪里执行":slot 路由、hash tag、节点拓扑;
  • 第 28 章(事务)解决"多命令如何打包":MULTI/EXEC 队列、无交错执行、WATCH 乐观锁;
  • 第 29 章(Lua / Functions)解决"如何在服务端做决策":条件分支、读后写原子、持久化业务逻辑。

三章合并起来才是完整的 Redis 命令编排画面。理解这些机制后,选择"用 Pipeline 还是 MULTI/EXEC 还是 Lua"就从"凭直觉"变成"看场景需求 → 看语义匹配 → 看性能目标"的工程决策。事务不是万能锁,而是调度原子性 + 乐观锁的精确组合工具。
