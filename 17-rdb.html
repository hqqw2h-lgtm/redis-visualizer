第 17 章 · RDB 持久化

周期快照 · fork + Copy-on-Write 一致性 · dump.rdb 二进制格式 · CRC64 校验 · CoW 内存上界 · THP 写放大 · 主从全量同步

Redis 在内存中维护 keyspace,进程退出后内存内容立即丢失。持久化的核心问题是:在不长时间阻塞主线程命令路径的前提下,获取一份内存数据的一致性副本并落到磁盘。

RDB 的解法是周期性快照:在某个时间点 T,将整个 keyspace 序列化为一个紧凑的二进制文件 dump.rdb,文件包含 T 时刻所有 key/value/过期时间。重启时通过 rdbLoad() 反序列化恢复 keyspace。

实现该语义有两个关键约束:① 序列化期间主线程不能停止接受写命令;② 序列化得到的快照必须对应一个具体的逻辑时间点,不能是"边写边读"的不一致状态。Redis 通过 fork() + Copy-on-Write 同时满足两者,代价是 fork 延迟、CoW 额外内存与可能的 OOM 风险。

  • SAVE(同步阻塞)与 BGSAVE(fork 后台)的语义边界与适用场景

  • fork() 的页表复制语义、Copy-on-Write 触发时序与状态机
  • CoW 额外内存上界 M_extra = r × M,r 接近 1 时的 OOM 风险
  • dump.rdb 文件结构:REDIS magic + version + AUX + DB section + EOF + CRC64
  • 常见 AUX 字段(redis-ver / used-mem / repl-id / repl-offset)
  • value type 操作码表(0/2/3/5/11/14/16-21)与 listpack 编码演进
  • BGSAVE 端到端时序、子进程序列化主循环、rio 抽象与 CRC 累加
  • 主从全量同步、Sentinel/Cluster/Diskless 模式下 RDB 的角色
  • INFO persistence 关键字段:latest_fork_usec / rdb_last_cow_size / rdb_last_bgsave_status

17.1 SAVE 与 BGSAVE 的语义边界

RDB 有两个命令入口:SAVE 在主线程同步序列化,BGSAVE 通过 fork() 把序列化扔给子进程。两者的差异不只是"是否阻塞",还涉及一致性来源、错误反馈方式与适用场景。这一节先把执行路径与对比表摆清楚。

SAVE 的真实角色:在生产环境中主要用于 SHUTDOWN SAVE 这类受控维护流程。常规持久化应通过 BGSAVE 完成。BGSAVE 命令的即时返回值为 Background saving started,实际成功与否需通过 INFO persistence 中 rdb_last_bgsave_status 字段判断。

17.1.3 BGSAVE 的状态查询路径

$ redis-cli BGSAVE
Background saving started

$ redis-cli INFO persistence | grep -E "rdb_bgsave_in_progress|rdb_last_bgsave_status|latest_fork_usec|rdb_last_cow_size"
rdb_bgsave_in_progress:1
rdb_last_bgsave_status:ok
latest_fork_usec:18432
rdb_last_cow_size:0

rdb_bgsave_in_progress=1 表示子进程正在写文件,该值由 serverCron() 在每轮事件循环中检查 wait3() 状态后更新。latest_fork_usec=18432 表示最近一次 fork() 系统调用阻塞了 18.4 ms。rdb_last_cow_size 在 BGSAVE 完成时更新,记录上一次成功的 CoW 增量。

17.2.3 serverCron 触发 BGSAVE 关键代码

/* src/server.c - serverCron 简化版 */
int serverCron(struct aeEventLoop *eventLoop, long long id, void *clientData) {
    if (server.rdb_bgsave_scheduled &&
        !hasActiveChildProcess() &&
        ldbPendingChildren() == 0)
    {
        rdbSaveInfo rsi, *rsiptr;
        rsiptr = rdbPopulateSaveInfo(&rsi);
        if (rdbSaveBackground(SLAVE_REQ_NONE, server.rdb_filename, rsiptr,
                              RDBFLAGS_NONE) == C_OK) {
            server.rdb_bgsave_scheduled = 0;
        }
    }
    /* ... 遍历 save 规则 ... */
}
主节点禁用自动 save 的常见做法:在主节点配置 save "" 关闭自动触发,改用运维编排在低峰期触发 BGSAVE 并等待完成后校验,或将 RDB 任务转移到从节点。原因:BGSAVE 在高峰期触发会叠加 CoW 内存与 fork 延迟,影响主线程响应。

17.3 fork() 与 Copy-on-Write 一致性

BGSAVE 的核心魔法在 fork()。子进程看到的 keyspace 视图必须等价于 fork 那一刻的快照,父进程同时还能继续处理写命令 — 这两件事看似矛盾,实际是 Linux 内核通过 页表复制 + 缺页中断 + 写时复制 三步联合实现的。这一节用三张图把 CoW 的语义彻底拆开,然后给出 CoW 额外内存的上界公式。

17.3.1 fork() 的页表复制语义

图 17-3-1 · fork() 仅复制页表,物理页延迟复制

fork() 仅复制父进程的页表(虚拟地址 → 物理地址映射),不复制实际物理页。复制完成后,内核将父子两份页表中的所有页项标记为只读。任一进程对只读页的写操作触发 CPU 缺页中断,内核在中断处理中分配新物理页、复制旧页内容、修改写入方页表项为可读写。子进程的页表项始终指向 fork 时刻的旧页,这就是 RDB 一致性的物理基础。

17.3.2 CoW 触发时序

图 17-3-2 · 父进程写触发 CoW 的完整时序

子进程通过 rdbSaveRio() 遍历 db.dict 时,所有未被父进程修改的页继续与父进程共享物理内存。一旦父进程修改某页,内核分配副本并切换父进程页表项,子进程继续看到旧页内容。子进程整个生命周期内观察到的视图等价于 fork 那一刻的内存快照。

17.3.4 定理 17.1:CoW 额外内存上界

设 BGSAVE 持续时间内父进程修改的内存页数为 K,系统页大小为 P,则 CoW 期间额外占用的物理内存 M_extra 为:

Mextra = K × P

设 BGSAVE 期间被修改的内存比例为 r ∈ [0, 1],父进程已用内存为 M,则:

Mextra = r × M
这里的 M 是 OS 物理页口径:CoW 额外占用发生在操作系统 RSS/物理内存层,不是业务逻辑数据大小,也不是 RDB 文件大小。used_memory 是 Redis/allocator 口径,rdb_last_cow_size 与 RSS 峰值更接近,规划容量时必须把三者分开看。
表 17-3-1 · 不同 r 取值对应的 10 GiB 实例额外内存
场景r 取值10 GiB 实例额外内存
只读高峰0.00≈ 0(仅页表)
读多写少0.05500 MiB
典型业务0.202 GiB
写密集0.505 GiB
全量重写1.0010 GiB(2× 翻倍)
OOM 红线:对 30 GiB 物理内存的实例,如果 Redis 已用内存接近 30 GiB,r=0.5 的写密集 BGSAVE 会额外占用约 15 GiB,总 RSS 可能超过机器可用内存。生产规划中,maxmemory 通常设为物理内存的 55%-60%,预留约 40% 作为 CoW + 内存碎片缓冲。

17.7.2 CoW 在 /proc/smaps 的可见性

$ pid=$(pidof redis-server)
$ awk '/Private_Dirty/ {sum += $2} END {print sum " kB"}' /proc/$pid/smaps
786432 kB

$ until redis-cli INFO persistence | grep -q 'rdb_bgsave_in_progress:0'; do sleep 1; done
$ redis-cli INFO persistence | grep rdb_last_cow_size
rdb_last_cow_size:805306368

Private_Dirty 是父进程在 BGSAVE 期间写脏的私有页总数,反映 CoW 实时增量。rdb_last_cow_size 是子进程退出后 Redis 记录的累计 CoW。两者量级接近时,说明 CoW 估算与实测一致。该值持续超过 maxmemory 的 30% 时,应将 BGSAVE 调度到低峰期或进行实例分片。

17.7.3 RDB Header 直接验证

from pathlib import Path

raw = Path("dump.rdb").read_bytes()[:9]
magic = raw[:5].decode("ascii", errors="replace")
version = raw[5:9].decode("ascii", errors="replace")
print({"magic": magic, "version": version})
{'magic': 'REDIS', 'version': '0011'}

magic != REDIS 表明文件并非 RDB 或前 5 字节已损坏。版本号超过当前 Redis 版本支持上限时,加载会被拒绝。该检查可作为备份链路的最低校验,实际入库前仍需 redis-check-rdb 通过。

17.10.3 反例与正例配置对比

反模式 — 大促期间 OOM Kill

redis.conf:
  maxmemory 32gb
  save 60 10000              # 高峰每分钟触发
  transparent_hugepage 默认开启
  vm.overcommit_memory 0

# 现象:
# - 大促流量进来, r ≈ 0.6
# - BGSAVE 触发, CoW 增量 ~ 19 GB
# - 系统总内存 32 GB, 无足够余量
# - vm.overcommit_memory=0 拒绝 fork
#   → fork: Cannot allocate memory
# - BGSAVE 失败, 持续累积 dirty
# - 同时 THP 在 madvise 模式偶尔
#   分配 2 MiB 页, 放大 CoW
# - 最终 OOM Killer 选择 redis-server
# - 整实例进程被杀, RTO ≥ 5 min

正模式 — 持久化稳定运行

redis.conf:
  maxmemory 18gb              # 32 GiB 物理 × 55%
  save ""                     # 禁用自动
  rdbchecksum yes
  rdbcompression yes
  dir /mnt/ssd-rdb/

# 系统:
  sysctl vm.overcommit_memory=1
  sysctl vm.swappiness=1
  echo never > /sys/.../transparent_hugepage/enabled

# 低峰期编排:
  redis-cli BGSAVE
  until redis-cli INFO persistence | grep -q 'rdb_bgsave_in_progress:0'; do sleep 1; done
  redis-cli INFO persistence | grep -q 'rdb_last_bgsave_status:ok'
  redis-check-rdb dump.rdb
  cp dump.rdb /backup/$(date +\%F).rdb

# 收益:
# - maxmemory 留出 14 GB CoW 余量
# - fork 成功概率显著提高
# - THP 关闭后 CoW 粒度严格 4 KiB
# - 备份必经 redis-check-rdb 校验

17.11.3 rdbSaveRio 核心代码摘要

/* src/rdb.c - rdbSaveRio 简化版 */
ssize_t rdbSaveRio(int req, rio *rdb, int *error, int rdbflags, rdbSaveInfo *rsi) {
    char magic[10];
    snprintf(magic, sizeof(magic), "REDIS%04d", RDB_VERSION);
    if (rdbWriteRaw(rdb, magic, 9) == -1) goto werr;
    if (rdbSaveInfoAuxFields(rdb, rdbflags, rsi) == -1) goto werr;

    for (int j = 0; j < server.dbnum; j++) {
        redisDb *db = server.db + j;
        if (dictSize(db->dict) == 0) continue;

        if (rdbSaveType(rdb, RDB_OPCODE_SELECTDB) == -1) goto werr;
        if (rdbSaveLen(rdb, j) == -1) goto werr;
        if (rdbSaveType(rdb, RDB_OPCODE_RESIZEDB) == -1) goto werr;
        if (rdbSaveLen(rdb, dictSize(db->dict)) == -1) goto werr;
        if (rdbSaveLen(rdb, dictSize(db->expires)) == -1) goto werr;

        dictIterator *di = dictGetSafeIterator(db->dict);
        dictEntry *de;
        while ((de = dictNext(di)) != NULL) {
            sds keystr = dictGetKey(de);
            robj key, *o = dictGetVal(de);
            initStaticStringObject(key, keystr);
            long long expire = getExpire(db, &key);
            if (rdbSaveKeyValuePair(rdb, &key, o, expire) == -1) {
                dictReleaseIterator(di);
                goto werr;
            }
        }
        dictReleaseIterator(di);
    }

    if (rdbSaveType(rdb, RDB_OPCODE_EOF) == -1) goto werr;
    uint64_t cksum = rdb->cksum;
    memrev64ifbe(&cksum);
    if (rioWrite(rdb, &cksum, 8) == 0) goto werr;
    return C_OK;
werr:
    return C_ERR;
}

17.11.4 设计取舍

取舍 1:fork + CoW vs 多线程共享地址空间

多线程方案下,序列化线程与主线程共享 keyspace,主线程的修改会让序列化线程读到不一致状态。为获得一致性,需要全局锁或 MVCC,代价是命令路径引入并发控制。fork 方案借助内核页表机制天然提供快照隔离,代价是 fork 延迟与 CoW 内存。在 Redis 单线程命令模型下,前者破坏了核心设计前提,fork 是更合适的选择。

取舍 2:RDB 与 AOF 互补

RDB 提供紧凑的二进制快照,加载快、文件小,但两次快照之间的写入会在崩溃时丢失。AOF 提供逐命令日志,数据丢失窗口小至 1 秒,但文件大、加载慢。两者并非二选一:Redis 4.0 引入的混合持久化在 AOF 重写时输出 RDB preamble + 增量命令,获得 RDB 的恢复速度与 AOF 的数据安全。

取舍 3:LZF 压缩 vs zstd

RDB 默认使用 LZF 压缩字符串。LZF 压缩比中等(典型 1.5-2×),但压缩/解压速度极快,对 CPU 占用低。Redis 选择 LZF 而非更高压缩比的算法(如 zstd),原因是 BGSAVE 子进程的 CPU 时间会与生产业务争抢内核调度,压缩算法的速度优先级高于压缩比。
