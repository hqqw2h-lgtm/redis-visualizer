第 26 章 · Sentinel 哨兵

独立进程 · Pub/Sub 自动发现 · SDOWN/ODOWN 两阶段判定 · epoch + Raft 风格 Leader 选举 · 故障转移五阶段 · 副本三层排序 · 脑裂保护

Sentinel 是一组独立部署的进程,负责对一组 Redis 主从节点执行三项任务:监控存活、触发故障转移、向客户端通告新主节点地址。其设计核心包括:基于 Pub/Sub 的拓扑自动发现、SDOWN/ODOWN 两阶段故障判定、Raft 风格的 Leader 选举、epoch 单调递增的版本号机制。Sentinel 自身保持无共享状态,仅通过 Pub/Sub 与 INFO 命令收集信息。

26.1 Sentinel 的三项设计目标

第 25 章的主从复制解决了"读扩展"与"数据冗余",但主节点崩溃后客户端如何切到新主这个问题没有答案 — 仍需运维介入手动 REPLICAOF NO ONE。Sentinel 正是为此而生:它独立于 Redis 数据进程运行,持续监控主从拓扑,在主节点不可用时自动选出新主,并向客户端通告新地址。

26.1.2 Sentinel 在拓扑中的角色

图 26-1-1 · Sentinel 集群、Redis 数据节点、客户端三层关系

Sentinel 进程与数据进程是分离的:Sentinel 不存储用户数据,Redis 数据节点不参与故障判定决策。Sentinel 通过订阅数据节点的 __sentinel__:hello 频道相互发现,通过 INFO 命令收集复制拓扑。

为什么独立部署:Sentinel 选择"独立进程而非内嵌共识"是有意为之。代价是部署复杂度增加(需额外维护 N+M 个进程),收益是故障域隔离 — 数据节点可保持单线程简洁性,Sentinel 可独立升级,主节点崩溃不会拖垮 Sentinel 本身。第 27 章 Cluster 走另一条路:把判定与选举内嵌到数据节点,代价是数据节点协议复杂度上升。

/* sentinel.c — hello 消息处理(简化) */
void sentinelProcessHelloMessage(char *hello, int hello_len) {
    int numtokens = sdssplit(hello, ',', &tokens);
    if (numtokens != 8) goto cleanup;

    /* 字段 0-3: 发送方信息 */
    char *ip = tokens[0];
    int port = atoi(tokens[1]);
    char *runid = tokens[2];
    uint64_t current_epoch = strtoull(tokens[3], NULL, 10);

    /* 检查或新增 known_sentinel */
    sentinelRedisInstance *si = getSentinelByRunid(runid);
    if (!si) {
        si = createSentinelRedisInstance(runid, SRI_SENTINEL, ip, port,
                                         master->quorum, master);
    }
    /* 字段 4-7: 当前主节点信息(用于检测拓扑分歧) */
    /* ... */
}
为什么不让 Sentinel 互相直连:让 master 充当消息总线避免了"发现协议本身需要发现"的循环。Sentinel 启动时只需要 1 个外部输入(master 地址),其余成员通过订阅同一频道自然汇聚。NAT/防火墙环境下只需开放 Sentinel → master 的 TCP 与 PUBSUB 通道,无需 Sentinel 之间两两可达。

26.4 SDOWN 主观下线判定

SDOWN(Subjectively Down,主观下线)是单个 Sentinel 基于自身 PING 视角做出的本地判断。它仅触发"询问其他 Sentinel"的动作,不直接触发故障转移。

26.4.4 SDOWN 判定源码

/* sentinel.c — SDOWN 判定(简化) */
void sentinelCheckSubjectivelyDown(sentinelRedisInstance *ri) {
    mstime_t elapsed = 0;
    if (ri->link->act_ping_time)
        elapsed = mstime() - ri->link->act_ping_time;
    else if (ri->link->disconnected)
        elapsed = mstime() - ri->link->last_avail_time;

    if (elapsed > ri->down_after_period) {
        if (!(ri->flags & SRI_S_DOWN)) {
            sentinelEvent(LL_WARNING, "+sdown", ri, "%@");
            ri->flags |= SRI_S_DOWN;
            ri->s_down_since_time = mstime();
        }
    } else {
        if (ri->flags & SRI_S_DOWN) {
            sentinelEvent(LL_WARNING, "-sdown", ri, "%@");
            ri->flags &= ~SRI_S_DOWN;
        }
    }
}

SDOWN 仅是单个 Sentinel 的本地判断。同一时刻不同 Sentinel 对同一实例的判断可能不同,网络分区或 Sentinel 自身的 GC 暂停均可能导致误判。SDOWN 不直接触发故障转移,必须经 ODOWN 集体仲裁。

26.5 ODOWN 客观下线仲裁

ODOWN(Objectively Down,客观下线)是 Sentinel 集群通过投票达成的集体共识:至少 quorum 个 Sentinel 都认为 master 不可达。它是触发 Leader 选举与故障转移的前置条件。

quorum 必须严格过半:quorum 必须严格大于 N/2 才能保证故障转移在分区场景下不会同时在两个分区触发(脑裂)。配置 quorum < ⌈(N+1)/2⌉ 等同于放弃脑裂保护。例如 N=3, quorum=1 时,网络一分为二的两侧都能各自达成 ODOWN,各自选 Leader,各自提升新主 — 分区恢复后两个"新主"互相覆盖,数据丢失不可逆。

26.7 Leader 选举

ODOWN 成立后,所有 Sentinel 都可能想发起故障转移。为避免多个 Sentinel 同时执行造成冲突,Sentinel 采用 Raft 风格的 Leader 选举:每次故障转移都需要先选出唯一 Leader,只有 Leader 才能执行后续提升与广播。

26.7.4 epoch 投票约束矩阵

图 26-7-2 · 收到投票请求时的 epoch 决策流程

epoch 单调递增 + 同 epoch 内每节点至多 1 票的两条约束,共同保证同一 epoch 内最多产生 1 个 Leader。配合 quorum 过半的要求,即使发生网络分区,任意时刻最多只有一个分区能选出 Leader 执行故障转移。

26.7.5 投票路径源码

/* sentinel.c — 投票路径(简化) */
char *sentinelVoteLeader(sentinelRedisInstance *master,
                          uint64_t req_epoch,
                          char *req_runid,
                          uint64_t *leader_epoch) {
    if (req_epoch > sentinel.current_epoch) {
        sentinel.current_epoch = req_epoch;
        sentinelFlushConfig();
        sentinelEvent(LL_WARNING, "+new-epoch", master, "%llu",
                      (unsigned long long)sentinel.current_epoch);
    }

    if (master->leader_epoch < req_epoch &&
        sentinel.current_epoch <= req_epoch)
    {
        sdsfree(master->leader);
        master->leader = sdsnew(req_runid);
        master->leader_epoch = sentinel.current_epoch;
        sentinelFlushConfig();
        sentinelEvent(LL_WARNING, "+vote-for-leader", master, "%s %llu",
                      master->leader, (unsigned long long)master->leader_epoch);
    }
    *leader_epoch = master->leader_epoch;
    return master->leader ? sdsdup(master->leader) : NULL;
}
为什么 epoch 必须持久化:sentinelFlushConfig() 会把 current_epoch 与 leader_epoch 写入 sentinel.conf。这是关键 — 如果 Sentinel 重启,它必须记住"我在 epoch=12 已经投给了 X",否则可能在重启后又把同 epoch 的票投给 Y,破坏"同 epoch 1 票"的安全性。Raft 的 votedFor 持久化要求是同一道理。

26.9.4 副本选择源码

/* sentinel.c — 副本选择(简化) */
int compareSlavesForPromotion(const void *a, const void *b) {
    sentinelRedisInstance **sa = (sentinelRedisInstance **)a;
    sentinelRedisInstance **sb = (sentinelRedisInstance **)b;

    if ((*sa)->slave_priority != (*sb)->slave_priority)
        return (*sa)->slave_priority - (*sb)->slave_priority;

    /* offset 大者优先(数据更新) */
    if ((*sa)->slave_repl_offset > (*sb)->slave_repl_offset) return -1;
    else if ((*sa)->slave_repl_offset < (*sb)->slave_repl_offset) return 1;

    /* runid 字典序回退 */
    return strcasecmp((*sa)->runid, (*sb)->runid);
}
为什么 priority 优先于 offset:运维侧应有最终决定权。设想一台跨机房高延迟副本即使 offset 最高,也不应当成为新主(网络抖动会让它看起来"领先"但实际写延迟高)。priority 给运维一个"硬性偏好"通道,offset 仅在 priority 相同时作为客观指标使用,runid 则保证排序结果稳定可重现。

26.13.3 设计取舍

取舍 1:独立进程 vs 内嵌共识

Sentinel 选择独立进程而非将共识算法嵌入 Redis 数据节点。代价是部署复杂度增加(需额外维护 N+M 个进程)。收益是数据节点保持单线程简洁性,Sentinel 可独立升级,故障域隔离。第 27 章 Cluster 走另一条路 — 把判定与选举内嵌进数据节点,因此 Cluster 节点的协议复杂度显著上升。

取舍 2:Pub/Sub 自动发现 vs 静态配置

Sentinel 通过 __sentinel__:hello 频道自动发现集群成员,不需要在配置中显式列出每个 Sentinel 地址。代价是 master 故障时 hello 频道短暂失效(已发现的成员仍可工作)。收益是动态扩缩容无需重启,部署成本低。

取舍 3:quorum + epoch 双约束 vs 仅过半

ODOWN 使用 quorum 配置项(可独立调节),故障转移使用 epoch 单调递增 + 同 epoch 1 票限制。代价是配置项增加。收益是允许细粒度调节:可设置 quorum < 过半实现"激进检测",或 quorum = 过半实现"保守检测",epoch 机制始终保证选举正确性。

取舍 4:不提供强一致 vs 提供 min-replicas 边界

Sentinel 故障转移过程仍是异步的,不保证零数据丢失。代价是分区场景下旧主可能丢失分区前的写入。Redis 通过 min-replicas-to-write + min-replicas-max-lag 提供事后边界,而非引入完整的 Paxos/Raft 数据共识。这与 Redis 整体"AP 优先"的设计哲学一致。

章末总结

第 26 章把第 25 章的"主从复制"提升到"高可用":通过独立部署的 Sentinel 进程把"主节点崩溃"从需要人工介入的事故转化为可在十秒级自动恢复的事件。本章的四层结构对应了一条清晰的因果链:

  • §26.1-26.3 拓扑与发现 — Sentinel 通过 __sentinel__:hello 自动发现集群成员,master 充当消息总线让 Sentinel 间无需直接连接;
  • §26.4-26.6 故障判定 — SDOWN 是单 Sentinel 视角的本地判断,ODOWN 是 ≥ quorum 的集体共识,两阶段判定通过 quorum 配置项调节灵敏度;
  • §26.7-26.9 选举与转移 — Raft 风格的 epoch + 同 epoch 1 票约束保证选举安全,五阶段故障转移在 ~10 秒内把流量切到新主,副本选择三层排序保证选择确定且尽可能保留数据;
  • §26.10-26.13 生产实践 — 多机房部署、quorum 必须过半、客户端订阅 +switch-master 是三条不可妥协的生产红线。
