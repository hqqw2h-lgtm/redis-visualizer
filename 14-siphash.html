第 14 章 · SipHash 与 HashDoS 防御

128 位密钥 PRF · ARX 三元混合 · 抗 HashDoS · 进程私有 seed · dict 桶索引专用

Redis 字典在均匀哈希假设下查找均摊为 O(1)。该假设依赖于桶位置不可被攻击者离线预测,否则可构造大量同桶键将查找退化为 O(N)、批量插入退化为 O(N²)。

SipHash-2-4 是为短输入哈希表设计的带密钥伪随机函数。Redis 在进程启动时生成 16 字节 dict hash seed,作为 SipHash 的 128 位密钥参与每次桶索引计算,使攻击者无法在脱机环境复现桶位置。本章按"威胁模型 → SipHash 算法结构 → SipRound 内部 → 尾块与最终化 → Redis 集成路径 → 与 CRC16 的边界 → 算法对比 → 部署与陷阱"组织。

  • HashDoS 攻击与普通哈希碰撞的本质差异(攻击者可控 vs 概率不可控)
  • SipHash-2-4 的 128 位密钥 + 256 位状态 + ARX 三元混合算法结构
  • 从 16 字节 seed 派生 v0..v3 的 4 个公开常量("somepseudorandomlygeneratedbytes")
  • SipRound 单轮 14 个 ARX 操作、抗时序侧信道的工程含义
  • 8 字节分块小端编码与尾块 (len & 0xff) << 56 长度位的设计意图
  • Redis 集成路径:dictHashKey → siphash,seed 来自 dictSetHashFunctionSeed
  • SipHash(dict 桶私有)与 CRC16(Cluster slot 公开)的严格职责分离
  • seed 独立性对主从、Cluster、failover、RDB/AOF 重载的工程影响
  • 8 类常见误区:固定 seed 调试、误用为 MAC、跨实例对比 hash 值等

前置知识:第 03 章 dict 哈希表(rehash、桶链);第 12 章 GeoHash(理解为何不同场景需要不同哈希);CRC 多项式校验和的基本概念。读者需理解 ARX 密码学结构与 64 位整数运算。

14.1 威胁模型:HashDoS 不是普通碰撞

普通哈希函数的随机碰撞概率符合生日悖论,链表长度均值有界。HashDoS 风险的核心是攻击者掌握哈希函数及其 seed 后,可在自有机器离线枚举出落入同一桶的键,然后在线一次性写入 Redis,使桶链长度由攻击者主动控制。

核心区分: "碰撞概率高"和 "HashDoS" 是两类不同问题。前者用更优散列函数即可缓解,后者必须引入攻击者无法获知的密钥使桶位置失去离线可复现性。SipHash 解决的是后者,xxHash/MurmurHash 解决的是前者。

14.2 SipHash-2-4 算法结构

SipHash-2-4 使用 128 位密钥派生 256 位内部状态,消息按 8 字节小端块逐块压缩,每块 2 轮 SipRound,所有块完成后追加 4 轮最终化,输出 64 位。"2-4" 中的 2 指压缩轮数 c,4 指最终化轮数 d,与密钥长度或输出长度无关。

128 位 seed拆分为 k0、k1 两个 64 位密钥dictSetHashFunctionSeed(启动时随机)
v0..v34 个 64 位内部状态字由 k0、k1 与 4 公开常量异或派生
公开常量 C0..C3ASCII "somepseudorandomlygeneratedbytes" 拆分算法标准定义

/* SipHash-2-4 完整骨架 */
uint64_t siphash24(const uint8_t *in, size_t len, const uint8_t key[16]) {
    uint64_t k0 = load64_le(key + 0);
    uint64_t k1 = load64_le(key + 8);
    uint64_t v0 = k0 ^ 0x736f6d6570736575ULL;  /* "somepseu" */
    uint64_t v1 = k1 ^ 0x646f72616e646f6dULL;  /* "dorandom" */
    uint64_t v2 = k0 ^ 0x6c7967656e657261ULL;  /* "lygenera" */
    uint64_t v3 = k1 ^ 0x7465646279746573ULL;  /* "tedbytes" */

    size_t off = 0;
    for (; off + 8 <= len; off += 8) {
        uint64_t m = load64_le(in + off);
        v3 ^= m;
        SIPROUND;
        SIPROUND;
        v0 ^= m;
    }

    uint64_t b = pack_tail_with_len(in + off, len);
    v3 ^= b;
    SIPROUND;
    SIPROUND;
    v0 ^= b;

    v2 ^= 0xff;
    SIPROUND;
    SIPROUND;
    SIPROUND;
    SIPROUND;

    return v0 ^ v1 ^ v2 ^ v3;
}

14.3 密钥派生:从 seed 到 v0..v3

128 位 seed 在进程启动时由 getRandomBytes 生成 16 字节随机字节,通过 dictSetHashFunctionSeed 注入字典哈希函数。该 seed 仅驻留进程内存,不持久化到 RDB/AOF,不通过复制流传播,不向任何客户端命令暴露。

14.4 SipRound 内部:ARX 三元混合

SipRound 仅使用三种 CPU 原语:加法 Add、循环左移 Rotate、异或 XOR。无查表、无输入相关分支、无内存随机访问,使其抗时序侧信道攻击且对编译器友好。每轮包含两阶段:并行混合(v0/v1 与 v2/v3 各自独立),交叉混合(v0/v3 与 v2/v1 跨组)。

#define ROTL(x, b) (((x) << (b)) | ((x) >> (64 - (b))))

#define SIPROUND do {                                            \
    v0 += v1;  v1 = ROTL(v1, 13);  v1 ^= v0;  v0 = ROTL(v0, 32); \
    v2 += v3;  v3 = ROTL(v3, 16);  v3 ^= v2;                     \
    v0 += v3;  v3 = ROTL(v3, 21);  v3 ^= v0;                     \
    v2 += v1;  v1 = ROTL(v1, 17);  v1 ^= v2;  v2 = ROTL(v2, 32); \
} while (0)

宏展开避免函数调用开销;循环左移在现代 CPU 上是单周期指令(rol 在 x86,ror 在 ARM)。

14.5 8 字节分块与最终化

SipHash 按 8 字节小端块切分输入。最后不足 8 字节的尾块需要填充:剩余字节放入 byte0..byte6,byte7 固定保存 len mod 256。该设计使 "ab"、"ab\0"、"ab\0\0" 在尾块编码上不同,避免补零混淆。

图 14-5-1 · 尾块字节布局("hello" 5 字节)

表 14-5-1 · 不同长度输入的尾块编码差异
输入len完整块 m尾块 b (16 进制)说明
""0—0x0000000000000000空尾块 + len = 0
"a"1—0x0100000000000061byte0=0x61, byte7=0x01
"abc"3—0x0300000000636261byte0..2 倒序填充, byte7=len
"abcdefgh"80x68676665646362610x0800000000000000完整块走主路径, 空尾块 + len=8
"abcdefghi"90x68676665646362610x09000000000000691 完整块 + 1 尾字节
byte7 永远是 len mod 256。所以 "ab"、"ab\0"、"ab\0\0" 的尾块完全不同 — 0x0200000000006261 / 0x0300000000006261 / 0x0400000000006261,补零混淆被彻底消除。 
uint64_t pack_tail(const uint8_t *tail, size_t len) {
    uint64_t b = ((uint64_t)len & 0xff) << 56;

    switch (len & 7) {
    case 7: b |= (uint64_t)tail[6] << 48; /* fallthrough */
    case 6: b |= (uint64_t)tail[5] << 40; /* fallthrough */
    case 5: b |= (uint64_t)tail[4] << 32; /* fallthrough */
    case 4: b |= (uint64_t)tail[3] << 24; /* fallthrough */
    case 3: b |= (uint64_t)tail[2] << 16; /* fallthrough */
    case 2: b |= (uint64_t)tail[1] << 8;  /* fallthrough */
    case 1: b |= (uint64_t)tail[0];
    case 0: break;
    }
    return b;
}

switch fallthrough 是有意的:从最高位字节倒序填充,使任意 0..7 字节尾巴都能正确组装。(len & 0xff) << 56 占据 byte7,确保不同长度输入的尾块编码唯一。

图 14-5-3 · 最终化阶段时序 — 4 轮 SipRound 扩散
4: 压缩阶段 2 轮(快), 最终化 4 轮(强扩散) return v0 ^ v1 ^ v2 ^ v3 → uint64_t 哈希值 (送给 dict 取桶)

v2 ^= 0xff 是消息结束的统一标记,使最终化阶段与压缩阶段的状态可区分。4 轮最终化提供额外扩散,确保任意单比特输入差异在输出 64 位中达到约 50% 翻转概率(雪崩效应)。

14.6 Redis 集成路径

SipHash 在 Redis 内的调用入口集中在 dictHashKey。命令处理路径中,所有需要查找数据库键、对象属性、命令表的位置都通过 dict 接口,最终都调用 SipHash 计算桶索引。

图 14-6-1 · 命令路径中的 dict 查找时序
dictEntry *dictFind(dict *d, const void *key) {
    if (dictSize(d) == 0) return NULL;
    if (dictIsRehashing(d)) _dictRehashStep(d);

    uint64_t h = dictHashKey(d, key);

    for (int table = 0; table <= 1; table++) {
        size_t idx = h & d->ht[table].sizemask;
        dictEntry *he = d->ht[table].table[idx];

        while (he) {
            if (key == he->key || dictCompareKeys(d, key, he->key))
                return he;
            he = he->next;
        }
        if (!dictIsRehashing(d)) break;
    }
    return NULL;
}

哈希值仅决定桶索引,真正的相等判断由 dictCompareKeys 完成。SipHash 的高质量分布使桶链长度接近泊松分布期望值,避免极端长链。

14.7 SipHash 与 CRC16 的边界

Redis 内部使用两类哈希,职责严格分离:dict 桶索引用 SipHash(进程私有、抗攻击),Cluster slot 路由用 CRC16(跨节点公开稳定)。该边界由功能需求决定:dict 桶布局是进程内部细节,Cluster slot 必须客户端与所有节点一致。

14.8 与 DJBX/MurmurHash 的对比

14.9 部署模式:seed 独立的工程含义

每个 Redis 进程独立生成 dict hash seed,不通过复制流传播。该设计的工程含义:dict 桶布局是进程内部实现细节,主从、Cluster、Sentinel failover 后桶布局可能完全不同,但不影响数据语义与命令正确性。

性能担忧"MurmurHash 更快,SipHash 浪费 CPU"只看平均吞吐,忽视恶意输入下退化把额外哈希成本视作 HashDoS 防御代价
误用为 MAC"SipHash 有密钥可用于签名"64 位输出对认证场景过短认证用 HMAC-SHA256 / BLAKE2
跨实例对比 hash"同一 key 的 SipHash 应一致"主从重启后桶位置不同,误判故障跨节点一致性看 CRC16,不看 SipHash
错解 "2-4" 含义"2 是密钥长度,4 是输出长度"调优时错误降低安全轮数2 是压缩轮 c,4 是最终化轮 d
暴露内部 hash 调试"把每个 key 的 hash 打出来排查"削弱防御,且无跨实例可比性只看链长分布、版本、延迟等间接信号
误判 dict rehash"换 SipHash 就不会有查找抖动"大 key、热点 key、rehash 仍可能慢综合诊断,不只看哈希函数

14.10.1 反例 → 修复对照(误用 SipHash 为 MAC)

❌ 反例
/* 用 SipHash 做请求签名 */
uint64_t sig = siphash24(body, body_len, api_secret);
send_header("X-Signature", hex64(sig));

/* 问题:
 * 1. 64 位 tag 对认证场景过短
 *    (2^32 次伪造尝试可成功)
 * 2. SipHash 不是设计用来抵抗
 *    MAC 伪造攻击
 * 3. Redis 中的 SipHash 用法
 *    不是协议认证边界
 */
✅ 修复
/* 用 HMAC-SHA256 做请求签名 */
uint8_t tag[32];
hmac_sha256(api_secret, secret_len,
            body, body_len, tag);
send_header("X-Signature", hex(tag, 32));

/* 256 位 tag,抗伪造
 * SipHash 仅用于哈希表桶索引
 * 不要混用为协议认证 */
