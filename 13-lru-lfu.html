第 13 章 · LRU/LFU 近似淘汰

24 位双语义字段 · 16 槽淘汰池 · Morris 概率计数器 · 8 种 maxmemory-policy · 采样 vs 精确取舍 · 主从淘汰传播 · 淘汰风暴诊断

Redis 在 used_memory 触及 maxmemory 时进入淘汰路径。经典 LRU 需要 HashMap 结合双向链表,每个键约 16 字节附加指针,千万键规模下约 160 MiB 仅用于维护淘汰顺序,且每次访问需要更新链表节点,导致写放大与缓存行抖动。

Redis 的工程取舍是放弃全局精确顺序,在 redisObject 复用 24 位字段表达访问时间或频率热度,在淘汰时采用随机采样配合 16 槽淘汰池近似最优。LFU 模式进一步用 8 位 Morris 概率计数器表达对数级访问量,以 16 位分钟时戳记录上次衰减时刻。本章按"为何近似 → 元数据布局 → 采样算法 → 频率算法 → 策略矩阵 → 部署模式 → 基准 → 陷阱"组织。

  • 放弃精确 LRU 的工程动机:160 MiB 指针 + 链表 cache line 抖动 vs 24-bit 嵌入字段
  • 24 位字段在 LRU/LFU 模式下的双语义切分:idle clock 24-bit · ldt:16 + counter:8
  • dictGetSomeKeys 随机采样 + evictionPoolPopulate 16 槽淘汰池跨轮保留候选
  • Morris 概率递增:p = 1 / ((counter - 5) × factor + 1) + 线性衰减 periods = elapsed / decay-time
  • 8 种 maxmemory-policy 在 LRU/LFU/random/TTL × allkeys/volatile 矩阵下的选型
  • 主从复制场景下淘汰风暴对复制带宽的放大,以及 Cluster 节点级独立淘汰

前置知识:第 9 章 redisObject(理解 16 B 总开销与 4 bit type/encoding 字段);第 3 章 dict(dictGetSomeKeys 抽样接口);第 10 章 编码体系(理解 lru:24 字段与 LFU/LRU 共享池禁用)。本章聚焦 src/evict.c 与 src/object.c 中的淘汰实现,数据布局以 src/server.h 中 redisObject 位域为准。

13.1 为何放弃精确 LRU

精确 LRU 维护 HashMap<key, Node> 与全局双向链表,任意访问需要将节点摘除后重新插入链表头部。该方案在内存数据库语境下代价过高:每个键额外存储 prev、next 指针共 16 字节,千万键规模累计约 160 MiB,且热路径每次 GET、SET 都改写链表节点,造成 cache line 频繁失效。

Redis 选择在 redisObject 复用一个 24 位字段保存访问时钟或频率元数据,淘汰时通过随机采样近似最旧或最冷键。该方案放弃了全局有序性,换取零额外指针、O(1) 元数据更新与可控的 CPU 开销。

13.3 采样 LRU 与淘汰池

Redis 在内存超限时调用 performEvictions(旧版本 freeMemoryIfNeeded),该函数通过 dictGetSomeKeys 从目标字典随机抽取 maxmemory-samples 个键,结合 16 槽淘汰池跨轮保留候选,最终从池尾选出最旧键并删除。该流程由写命令路径触发,会阻塞主线程,因此采样大小与淘汰速率直接影响写入 P99。

13.3.4 evictionPoolPopulate 源码

/* src/evict.c — 简化形式 */
void evictionPoolPopulate(int dbid, dict *sampledict,
                          dict *keydict, struct evictionPoolEntry *pool) {
    dictEntry *samples[server.maxmemory_samples];
    int count = dictGetSomeKeys(sampledict, samples,
                                server.maxmemory_samples);

    for (int j = 0; j < count; j++) {
        unsigned long long idle = estimateObjectIdleTime(samples[j]->v.val);

        int k = 0;
        while (k < EVPOOL_SIZE && pool[k].key != NULL
               && pool[k].idle < idle) k++;

        if (k == 0 && pool[EVPOOL_SIZE - 1].key != NULL) continue;

        if (pool[k].key == NULL) {
            /* 空槽,直接放置 */
        } else {
            memmove(pool + k + 1, pool + k,
                    sizeof(*pool) * (EVPOOL_SIZE - k - 1));
        }
        pool[k].idle = idle;
        pool[k].key  = sdsdup(samples[j]->key);
    }
}

13.4 LFU 概率计数与衰减

LRU 仅记录最近访问时刻,无法区分"过去高频但近 2 分钟未访问"与"近 2 分钟首次访问"两类键。Redis 4.0 引入 LFU 模式,在同一 24 位字段中拆分出 16 位上次衰减时刻 ldt 与 8 位 Morris 概率计数器 counter,使淘汰决策基于访问频率而非访问时间。

13.4.7 LFUDecrAndReturn 源码

/* src/evict.c — 简化形式 */
unsigned long LFUDecrAndReturn(robj *o) {
    unsigned long ldt     = o->lru >> 8;
    unsigned long counter = o->lru & 255;

    unsigned long num_periods = server.lfu_decay_time
        ? LFUTimeElapsed(ldt) / server.lfu_decay_time
        : 0;

    if (num_periods)
        counter = (num_periods > counter) ? 0 : counter - num_periods;
    return counter;
}
升温与遗忘的联调:lfu-log-factor 控制升温速度,lfu-decay-time 控制遗忘速度。高 factor + 高 decay 使新热点升温缓慢且老热点滞留;低 factor + 低 decay 使突发热点快速识别也快速退场。两者必须配套调,单独调一个会导致 counter 分布偏斜——例如 factor=100 + decay=1 会让所有键长期停留在 counter=5~10,失去区分度。

13.8.4 源码深潜:performEvictions 主循环

/* src/evict.c — 简化形式,展示主循环 */
int performEvictions(void) {
    int keys_freed = 0;
    size_t mem_reported, mem_tofree;
    long long mem_freed = 0;

    if (getMaxmemoryState(&mem_reported, NULL,
                          &mem_tofree, NULL) == C_OK)
        return EVICT_OK;

    while (mem_freed < (long long)mem_tofree) {
        sds bestkey = NULL;
        int bestdbid = -1;

        if (server.maxmemory_policy & (MAXMEMORY_FLAG_LRU |
                                       MAXMEMORY_FLAG_LFU |
                                       MAXMEMORY_FLAG_TTL)) {
            struct evictionPoolEntry *pool = EvictionPoolLRU;

            while (bestkey == NULL) {
                /* 在每个 db 上采样,填充 pool */
                for (int i = 0; i < server.dbnum; i++) {
                    db = server.db + i;
                    dict *d = (server.maxmemory_policy &
                               MAXMEMORY_FLAG_ALLKEYS)
                        ? db->dict : db->expires;
                    if (dictSize(d) == 0) continue;
                    evictionPoolPopulate(i, d, db->dict, pool);
                }
                /* 从 pool 末尾取 idle/frequency 最差的键 */
                for (int k = EVPOOL_SIZE - 1; k >= 0; k--) {
                    if (pool[k].key == NULL) continue;
                    bestkey  = pool[k].key;
                    bestdbid = pool[k].dbid;
                    break;
                }
            }
        }

        if (bestkey) {
            db = server.db + bestdbid;
            robj *keyobj = createStringObject(bestkey, sdslen(bestkey));
            if (server.lazyfree_lazy_eviction)
                dbAsyncDelete(db, keyobj);
            else
                dbSyncDelete(db, keyobj);
            server.stat_evictedkeys++;
            decrRefCount(keyobj);
            keys_freed++;
        }
    }
    return EVICT_OK;
}

主循环的工程细节:

  • 淘汰发生在写命令路径上,而非后台线程,因此每次淘汰都直接计入写入延迟。
  • 当 lazyfree-lazy-eviction = yes 时,dbAsyncDelete 将释放工作交给后台线程,主线程仅解除字典指针,降低主线程阻塞。
  • 池在每次循环中被多 DB 填充并按 idle 升序排列,末尾即本轮淘汰对象。
  • 单次 performEvictions 可淘汰多键直到 mem_freed >= mem_tofree,因此突发写入会触发"淘汰风暴"。
