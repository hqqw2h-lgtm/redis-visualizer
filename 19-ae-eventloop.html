第 19 章 · ae 事件循环:Redis 单线程的真相

单一主线程 · 文件事件 + 时间事件 · beforeSleep 写合并 · serverCron 心跳 · 慢命令阻塞 · I/O 线程边界 · epoll/kqueue/select 三后端

Redis 在数据库命令执行层面使用单一主线程,这个主线程围绕 ae 事件循环运转:监听 socket 文件描述符、读取请求、解析协议、执行命令、写回响应,以及周期性维护任务 serverCron。事件循环在任意时刻只串行执行一条命令,因此一条慢命令会推迟所有就绪 fd 的处理。

本章从 ae 的核心数据结构、一轮循环的完整流程、beforeSleep 收尾、serverCron 时间事件、慢命令阻塞机制、Redis 6 引入的 I/O 线程边界,到 epoll / kqueue / select 的多路复用后端,系统拆解 Redis 主线程的可观测性、可调度性、可预测性。每一节给出量化数据、反例与修复对比、源码切入点。

读完本章你会清楚:为什么"单线程的 Redis"实际是 1 个主线程 + 多个后台线程;为什么 hz 不能盲目调高;为什么开了 I/O 线程慢命令照样慢;为什么 KEYS 一次就能让 P99 飙到 200 ms;以及排查一个延迟问题的最短路径。

  • "Redis 单线程"的精确含义:命令执行串行,但进程内还有 bio / lazyfree / I/O 线程
  • aeEventLoop 结构:events[fd] O(1) 索引,文件事件 + 时间事件双类
  • 一条 SET 命令穿过 ae 的 5 步函数栈与每步 µs 级耗时
  • beforeSleep 写合并、AOF flush、快速过期、cluster 心跳的复合职责
  • serverCron 心跳与 hz / dynamic-hz 的相互作用
  • 慢命令(KEYS / SORT / 大 key DEL / 长 Lua)阻塞整轮循环的反例

19.1 单线程的精确含义

"Redis 是单线程"这一表述只覆盖了主路径。准确的表述是:数据库命令的执行在主线程中串行完成。Redis 进程内还有若干个后台线程,负责辅助任务:AOF 后台 fsync、大对象异步释放(UNLINK / lazyfree),以及 Redis 6 之后新增的 I/O 线程,负责并行读写 socket。

三个并存的事实:
① 命令执行只在 1 个主线程 — call() 是唯一入口,任意时刻只跑一条命令;
② I/O 与 fsync 可以并行 — bio_aof / bio_lazy_free / I/O 线程并行做"非语义"工作;
③ 后台线程不参与 keyspace — 不读不写 dict,因此无需锁,语义模型保持单线程的简单性。
表 19-1-1 · 进程内线程角色
线程 / 组件职责不参与的工作关键观察点

bio_aof_fsync后台 fsync AOF buffer不处理客户端命令aof_delayed_fsync 增长会反馈成尾延迟

I/O threads读 querybuf、写 reply buffer不执行 call()适合大 value 场景,小 value 收益有限

19.2 aeEventLoop 数据结构

ae 通过两类事件管理整个调度:文件事件(file event)处理 socket I/O,时间事件(time event)处理周期性维护。事件以 fd 为下标直接索引到 events[fd],查找复杂度 O(1)。

19.2.1 C 结构定义

typedef struct aeEventLoop {
    int maxfd;                      /* 当前注册的最大 fd */
    int setsize;                    /* events/fired 数组容量 */
    aeFileEvent *events;            /* events[fd] = 该 fd 的读写回调 */
    aeFiredEvent *fired;            /* 本轮 poll 返回的就绪 fd */
    aeTimeEvent *timeEventHead;     /* 时间事件链表 */
    void *apidata;                  /* epoll/kqueue/select 私有状态 */
    aeBeforeSleepProc *beforesleep; /* 进入 poll 前的收尾 */
    aeBeforeSleepProc *aftersleep;  /* poll 返回后的唤醒 */
    int stop;
} aeEventLoop;

typedef struct aeFileEvent {
    int mask;                       /* AE_READABLE / AE_WRITABLE */
    aeFileProc *rfileProc;
    aeFileProc *wfileProc;
    void *clientData;
} aeFileEvent;

typedef struct aeTimeEvent {
    long long id;
    monotime when;                  /* 触发时间(单调时钟) */
    aeTimeProc *timeProc;
    aeEventFinalizerProc *finalizerProc;
    void *clientData;
    struct aeTimeEvent *prev, *next;
} aeTimeEvent;

19.4 beforeSleep 是吞吐的"收银台"

每条命令执行完成后,Redis 不会立即向 socket 发送回复。reply buffer 累积在 client->reply 中,client 加入 clients_pending_write 队列。下一次进入 aeApiPoll(即 epoll_wait)之前,beforeSleep 钩子集中将这些回复写出。

图 19-4-1 · 写合并示意
事件循环 1 轮内:
  收到客户端 1 的 GET foo  → call() → addReply +"hello"
  收到客户端 2 的 PING       → call() → addReply +"PONG"
  收到客户端 3 的 SET k v    → call() → addReply +"OK"
  ...
  收到客户端 N 的 ...

  此时 clients_pending_write = [c1, c2, c3, ..., cN]

beforeSleep:
  for each c in clients_pending_write:
      while c->reply 有未发数据:
          write(c->fd, ...)   --- 一次 syscall 可能写完多条回复
      若 socket 写满, 注册 AE_WRITABLE 事件

★ 收益:
  - 同一客户端的多条回复合并为 1 次 write
  - 减少用户态/内核态切换
  - reply buffer 复用,堆/栈分配次数降低

19.4.1 设计动机:syscall 摊薄

void beforeSleep(struct aeEventLoop *el) {
    handleClientsWithPendingWritesUsingThreads();
    flushAppendOnlyFile(0);
    moduleHandleBlockedClients();
    activeExpireCycle(ACTIVE_EXPIRE_CYCLE_FAST);
    if (server.cluster_enabled) clusterBeforeSleep();
}

如果改为"每条命令执行完立即 write",每条命令至少多一次 syscall;集中写则可能在一次 write 中刷出多个客户端的回复,减少内核 / 用户态切换。

19.5 serverCron 是主线程心跳

serverCron 是 ae 时间事件的主体,以 hz 为频率执行,默认 hz=10 即每 100 ms 一次。它在主线程中运行,与命令执行抢占同一份 CPU 预算,因此提高 hz 实际是增加打断主线程的频率,而非提高吞吐。

int serverCron(struct aeEventLoop *eventLoop, long long id, void *clientData) {
    updateCachedTime(1);
    clientsCron();
    databasesCron();       /* expire + rehash */
    backgroundOperationCron();
    if (server.cluster_enabled) clusterCron();
    return 1000 / server.hz;
}

activeExpireCycle (慢)每轮≤ 25 ms抽样过期 key,删除

dynamic-hz 启用后,Redis 根据 connected_clients 自动放大 hz(每超过 MAX_CLIENTS_PER_CLOCK_TICK 倍频),既覆盖大量客户端的 idle 检测需求,又避免人为调高 hz 的副作用。

19.6 慢命令阻塞整轮循环

事件循环本身不含并发原语,任何在 call() 中执行超过若干毫秒的命令都会推迟其余 fd 的处理。

19.8.1 epoll 接入代码

/* src/ae_epoll.c 核心片段 */
static int aeApiCreate(aeEventLoop *eventLoop) {
    aeApiState *state = zmalloc(sizeof(aeApiState));
    state->events = zmalloc(sizeof(struct epoll_event) * eventLoop->setsize);
    state->epfd = epoll_create(1024);
    eventLoop->apidata = state;
    return 0;
}

static int aeApiAddEvent(aeEventLoop *eventLoop, int fd, int mask) {
    aeApiState *state = eventLoop->apidata;
    struct epoll_event ee = {0};
    int op = eventLoop->events[fd].mask == AE_NONE ? EPOLL_CTL_ADD : EPOLL_CTL_MOD;
    ee.events = 0;
    if (mask & AE_READABLE) ee.events |= EPOLLIN;
    if (mask & AE_WRITABLE) ee.events |= EPOLLOUT;
    ee.data.fd = fd;
    return epoll_ctl(state->epfd, op, fd, &ee);
}

static int aeApiPoll(aeEventLoop *eventLoop, struct timeval *tvp) {
    aeApiState *state = eventLoop->apidata;
    int retval = epoll_wait(state->epfd, state->events,
        eventLoop->setsize, tvp ? tvp->tv_sec * 1000 + tvp->tv_usec / 1000 : -1);
    if (retval > 0) {
        for (int j = 0; j < retval; j++) {
            int mask = 0;
            struct epoll_event *e = state->events + j;
            if (e->events & EPOLLIN)  mask |= AE_READABLE;
            if (e->events & EPOLLOUT) mask |= AE_WRITABLE;
            if (e->events & EPOLLERR) mask |= AE_WRITABLE;
            if (e->events & EPOLLHUP) mask |= AE_WRITABLE;
            eventLoop->fired[j].fd = e->data.fd;
            eventLoop->fired[j].mask = mask;
        }
    }
    return retval;
}

19.8.2 epoll 触发模式选择

Redis 使用水平触发(LT):只要 fd 仍处于可读 / 可写状态,每次 epoll_wait 都会返回。这与边缘触发(ET)相比的优势是处理代码简单,劣势是若回调未读尽数据,下一次 poll 仍会被唤醒。

表 19-8-2 · LT vs ET
触发模式编程复杂度唤醒次数漏读风险Redis 选择
水平触发 LT低较多低(兜底)是
边缘触发 ET高较少(一次到位)高(需循环读尽)否

水平触发与 Redis 的一次 read(c->fd, buf, PROTO_QUERYBUF_LEN_LIMIT) 限长策略相互配合:即使一次没读尽,下一轮 epoll_wait 还会再唤醒。

19.9.1 反例:大 pipeline 反向阻塞

反例:1M 条命令一次塞

import redis
client = redis.Redis(host="redis-prod", port=6379)

p = client.pipeline(transaction=False)
for i in range(1_000_000):
    p.set(f"k{i}", "v" * 1024)
p.execute()

[现象]
- 客户端发出 1M 条命令, 每条 1 KB value
- Redis 主线程在 processInputBuffer 循环中一次解析多条
- 但 reply buffer 累积达 ~ 1 GB
- beforeSleep 一次写不完, 后续轮反复登记 AE_WRITABLE
- 主线程被 reply 写阻塞, 新命令延迟飙升
- 客户端因 omem 超限被强制断开, 任务失败

修复:拆批 1000 条

batch_size = 1000
for batch in chunks(range(1_000_000), batch_size):
    p = client.pipeline(transaction=False)
    for i in batch:
        p.set(f"k{i}", "v" * 1024)
    p.execute()

[修复后量化]
- 每批 1000 条, reply buffer 占用 ≈ 1 MB
- beforeSleep 一次写完
- 总耗时 18 s (略慢于反例峰值, 但稳定)
- 主线程对其他客户端的 P99 = 1.8 ms (无抖动)

19.9.2 反例:大 key DEL 阻塞

反例:DEL 1 GB list

redis-cli MEMORY USAGE huge_list
# 1073741824   即 1 GB
redis-cli DEL huge_list
[反例量化]
- DEL 调用进入 call(), 需释放 list 内所有 listNode
- 1 GB list 含 ≈ 100M 个 16-byte 节点
- 释放过程线性遍历, 主线程占用 ≈ 250 ms
- 期间所有客户端 P99 飙至 250 ms+

修复:UNLINK + lazyfree

redis-cli UNLINK huge_list

redis-cli CONFIG SET lazyfree-lazy-eviction yes
redis-cli CONFIG SET lazyfree-lazy-expire yes
redis-cli CONFIG SET lazyfree-lazy-server-del yes
redis-cli CONFIG SET lazyfree-lazy-user-del yes
[修复]
- UNLINK 在主线程仅做 dictDelete, O(log N)
- 真正的对象释放放到 bio_lazy_free 后台线程
- 主线程对外 P99 影响 < 1 ms

启用 4 项 lazyfree 配置后,所有内置删除路径(eviction / expire / 命令覆盖 / DEL)都改走异步释放,主线程对外延迟基本不再被大 key 释放影响。

19.12 取舍

ae 的核心取舍是"主线程事件循环简单可预测" vs "多核并行执行的复杂度"。Redis 把命令执行、对象修改、数据结构维护全部压进一个线程,代价是:

  • 任何慢命令、大 key 释放、超大回复都会推迟所有 fd
  • 单实例 OPS 上限受单核性能制约(典型 100~200 K)

收益是:

  • 无需锁、事务冲突、内存可见性问题
  • 调度可预测,延迟可量化,SLOWLOG / LATENCY 等机制简单有效
  • 数据结构内部不需考虑并发安全,代码简洁、cache locality 好
文件事件与时间事件共享一轮调度,serverCron、AOF flush、过期扫描、pending writes 均占主线程预算。这是为何 hz 不能盲调、大 key 不能用 DEL、慢命令不能用 KEYS 的根本原因 — 它们没有"独立线程"可以躲。Redis 6 的 I/O 线程将网络读写搬到旁边,减少 syscall 与拷贝成本,但不改变命令串行的边界。

Redis 7 的 Function 与 Module 机制为长任务提供了部分卸载方案:FUNCTION CREATE 加载的脚本在主线程执行,但 Redis 通过 timeout 与 Module API 的协作机制使其在到点时主动让出主线程,缓解整轮阻塞。

如果只能记住一件事:Redis 不是没有线程, 而是主线程的事件循环高度可预测。一条慢命令、一个大 key 释放、一次 200 ms 的 Lua 脚本 — 任何"占用主线程超过几毫秒"的操作, 都会把所有客户端拖入排队。排查卡顿时, 先看 SLOWLOG, 再看 LATENCY, 最后才考虑加 I/O 线程或调内核参数。
