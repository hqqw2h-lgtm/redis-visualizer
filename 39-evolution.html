第 39 章 · Redis 版本演进与未来

1.0 → 8.0 主版本演进 · Valkey 分叉 · 升级流程模板 · 回滚证据链 · 版本选择决策树 · 源码边界映射

Redis 自 2009 年发布 1.0 起,每个大版本都重新划定了运行时与生产边界。1.0 引入核心数据结构与 RDB 持久化;2.x 增加 Pub/Sub、事务、Lua、毫秒级 TTL;2.8/3.0 引入 PSYNC、Sentinel 和 Cluster;4.0/5.0 加入模块与 Streams;6.0 补齐 ACL/TLS/I-O threads;7.0 引入 Functions 与 Multi-Part AOF;Redis 8 与 Valkey 分叉后,Redis 路线整合 Search/JSON/TimeSeries/Vector 等能力,Valkey 路线选择 BSD-3 与开放治理。

升级评审的核心是回答"哪些能力收益值得拿、哪些兼容风险必须先解决"。版本号大小不是决策依据,生产收益与回滚成本才是。本章按时间线梳理每代关键能力,并给出"先验证可回滚、再验证新能力"的升级流程模板。

  • Redis 1.0 → 8.0 + Valkey 分叉的关键事件时间线与对应的"该版本解决了什么生产问题"
  • 升级前盘点资产的五条命令:INFO server + MODULE LIST + ACL USERS + INFO persistence + COMMAND INFO
  • 能力 / 风险二维评估矩阵,以及每类能力的"升级必须验证项"
  • PSYNC + Sentinel + Cluster 三层复制能力的演进与各自的升级关注点
  • UNLINK / Lazy Free 6 个配置项的语义与默认值
  • Streams 与 Kafka 的本质差异(PEL、内存约束、Stream 不是磁盘日志)
  • ACL 用户拆分:default → admin/app/readonly/bot 的最小权限模板
  • Multi-Part AOF 目录结构,以及为何升级 7.x 必须保留 RDB 回滚点
  • Functions 治理流程:加载 / 复制 / 回滚 / 只读标记
  • Redis 8 与 Valkey 的许可证、治理、模块兼容三维分叉
  • 升级流程模板:盘点 → RDB 备份 → 沙箱加载 → 客户端矩阵 → 滚动 → 观测
  • 客户端兼容矩阵设计:Jedis / Lettuce / ioredis / redis-py 主流 SDK 在 7.x → 8.x 升级时的回归项

39.0 章首速览

核心结论: Redis 演进的本质是"生产能力逐步补齐":1.x 把数据结构放到服务端;2.x 让其可编程;3.x 让其水平扩展;4.x/5.x 让其成为平台;6.x 补安全与吞吐;7.x 补脚本治理与持久化格式;Valkey/Redis 8 沿两条治理路线继续。升级评审唯一不变的纪律是:先证明可回滚(RDB 备份 + 沙箱加载 + 客户端矩阵),再证明新能力收益值得风险。
真实事故: 某团队从 6.2 升 7.2,看中 Multi-Part AOF 的 rewrite 空间收益,但仅做了 AOF 备份未做 RDB。升级后业务发现 Functions 不兼容现有 Lua,需要回滚。结果:7.x AOF 目录结构(BASE + INCR + MANIFEST)无法被 6.2 直接读取,只能恢复到一周前的 RDB,丢失 7 天数据。教训:跨大版本升级必须保留 RDB 回滚点,AOF 不是回滚资产。

39.3 从五种数据结构到可编程服务器

Redis 1.x/2.x 的关键不是"另一个 KV 存储",而是把数据结构放到服务端,让客户端通过原子命令修改 List/Set/ZSet/Hash。Redis 2.6 的 Lua 脚本进一步将多步逻辑推到服务端,带来"原子复合操作"的能力。

Lua 解决原子性,但脚本仍运行在主线程。脚本超时即阻塞所有命令,因此每个长 Lua 脚本必须有耗时监控与回滚方案。Redis 7.0 引入 Functions 用于把脚本变成可管理资源:命名、加载、复制、回滚都可控。

39.5 Modules、Streams 与异步释放

Redis 4.0 的 Modules API 让 Redis 演进为平台,RediSearch/RedisJSON/RedisTimeSeries 等生态由此扩展。同一版本的 UNLINK 与 lazy free 解决大 key 删除阻塞。Redis 5.0 的 Streams 引入轻量事件流。

Streams 不是 Kafka 替代品。它适合 Redis 数据集内的低延迟队列,但需监控 PEL 大小、消费者滞后与内存增长。Streams 数据集占用受 maxmemory 约束,溢出时整个实例进入 OOM 处置流程。

39.6 6.0 与 7.0:安全、多线程、Functions、AOF 格式

Redis 6.0 补齐了企业安全能力(ACL、TLS、RESP3),并引入 I-O threads。Redis 7.0 引入 Functions 与 Multi-Part AOF,这是升级风险最高的一组变化。

图 39-6-1 · ACL 用户拆分模型

$ redis-cli ACL SETUSER app on '>apppass' '~app:*' '+@read' '+@write' '-@admin' '-@dangerous'
OK
$ redis-cli ACL LIST | grep '^user app'
user app on #... ~app:* resetchannels -@all +@read +@write -@admin -@dangerous

6.0 / 7.0 升级风险矩阵

能力风险等级主要兼容点
ACL中老客户端是否还用 AUTH 而非 AUTH user pass
TLS中证书旋转、握手开销、tls-replication
I-O threads低仅在网络瓶颈下收益,需 ≥ 4 核
RESP3中HELLO 3 协议、Push 类型客户端实现
Functions高脚本治理、ACL 类别、回滚
Multi-Part AOF高目录结构改变、旧版无法读
Sharded Pub/Sub中集群下广播范围由 slot 决定

39.7 Redis 8 与 Valkey:功能路线与治理路线分叉

2024 年 Redis 核心许可证变化后,Valkey 基于 Redis 7.2.x 成为 Linux Foundation 下的 BSD-3 分叉。Redis 8 起 Redis Open Source 提供 AGPLv3 / RSALv2 / SSPLv1 三许可选择,并把 Search、JSON、TimeSeries、Bloom 等能力作为 Redis Open Source 的组成部分。

图 39-7-1 · Redis 与 Valkey 路线对比

Redis 8 与 Valkey 决策矩阵

维度Redis 8Valkey
许可证AGPLv3 / RSALv2 / SSPLv1BSD-3
治理Redis Inc. 主导Linux Foundation 治理
多模能力内置 Search / JSON / TS / Vector / Bloom模块需自行集成
兼容基准Redis 7.x 命令兼容Redis 7.2 BSD 时代命令兼容
云厂商支持Redis Cloud + 商业AWS / GCP / Azure 大量加入
适用场景一站式多模数据平台开源合规 + 性能 + 兼容优先
判断准则:依赖 RediSearch/RedisJSON/RedisTimeSeries 时,迁移 Valkey 需要把模块兼容作为独立项目,不可直接二进制替换;若仅使用核心命令,Valkey 7.2 兼容路径较平滑。

39.8 升级流程:先验证可回滚,再验证新能力

39.12 源码与架构边界

版本演进最终落到三类边界:命令表多了什么命令,运行时路径多了什么状态,磁盘 / 网络格式是否仍能被旧版本理解。升级评审应从 release notes 走到源码入口,再走到可观测验证。

39.13 本章要点

要点命令 / 源码依据易错点
Redis 演进主线是生产能力逐步补齐release notes、INFO server升级目标必须明确要拿哪项能力
6.0 是安全与网络吞吐的分水岭ACL / TLS / I-O threads配最小权限,压测 I-O threads
7.0 是脚本治理与 AOF 格式分水岭FUNCTION LIST、appendonlydir升级前必须 RDB 备份
Redis 与 Valkey 是治理与功能路线分叉MODULE LIST、许可证合规与模块依赖一并评估
升级成功标准 = 可回滚 + 可观测 + 可验证RDB 校验、沙箱加载、Prometheus 指标滚动升级期间冻结新特性
客户端矩阵是上线门禁Jedis / Lettuce / ioredis 等服务端升级不能解决客户端问题
新版本不等于新语义自动安全I-O threads、Functions、Multi-Part AOF单命令互斥仍存在,长脚本仍阻塞
