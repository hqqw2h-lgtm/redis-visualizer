第 32 章 · 客户端缓存

Server-Assisted Client-Side Caching · CLIENT TRACKING · Tracking vs Broadcasting · RESP3 Push · NOLOOP / OPTIN / OPTOUT · 一致性数学 · 目录式与嗅探式

Redis 单条简单命令在服务端通常很快,而同 AZ 网络 RTT 往往高出一到两个数量级。对稳定热 key 来说,瓶颈常常不是命令本身,而是每次都要跨网络取值。Redis 6.0 引入的服务端辅助客户端缓存(Server-Assisted Client-Side Caching)允许客户端把热数据副本保留在进程内:命中时走本地读,写入时由 Redis 推送失效通知。这是一种目录式缓存一致性协议。

Redis 客户端缓存选择最终一致:失效通知是异步到达的,客户端在收到通知前可能短暂读到旧值。若要强一致读,就不能把本地副本当权威来源,而应直接读 Redis 或把校验与写入放到服务端完成。机制本质上是一种工程权衡:服务端维护 tracking_table(每 key 的客户端订阅列表),写入时通过 RESP3 Push 或 RESP2 重定向频道推送 invalidate 通知;客户端收到后驱逐本地副本,下次读取再从源端重新加载。

  • CLIENT TRACKING ON 启用流程、tracking_table 与 tracking_table_prefixes 双 rax 结构
  • lookupKey 中读命令登记的精确触发点,以及 signalModifiedKey 写时失效的调用链
  • Broadcasting 与 Tracking 的内存/网络/精度三维权衡,以及"K=100 万 / C=10 客户端"下的内存推导
  • RESP3 的 > Push 类型如何让单连接多路复用替代 RESP2 双连接 + Pub/Sub
  • NOLOOP 解决自写自收、OPTIN 让稳定热 key 不被临时数据挤走、OPTOUT 排除高频抖动 key
  • 命中率 h = 1 − r/W 的推导与延迟加速比 Speedup ≈ 1/(1−h) 的边际爆炸

32.1 客户端缓存解决的问题

客户端缓存不是泛泛的"再快一点",而是把稳定热 key 的读路径从网络请求变成本地查找。若一次 Redis 请求的大部分时间花在 RTT 上,继续优化命令内部逻辑收益有限;更有效的办法是让命中读不出进程。下面先看延迟分解,再看为什么"目录式失效"是 Redis 6 选择的解。

32.1.1 网络 RTT 在请求延迟中的占比

图 32-1-1 · 一次 Redis 同 AZ 请求的延迟分解
典型请求延迟分解 (Redis 7.2, 同 AZ, 100 字节 value):

  环节                  延迟           占比
  ──────────────────  ─────────  ────────
  Redis 命令执行         ~1 µs      0.5 %
  网络 RTT (同 AZ)       ~200 µs    99.0 %
  客户端反序列化         ~5 µs      0.5 %
  ──────────────────  ─────────  ────────
  总计                   ~206 µs    100 %

观察:
  Redis 服务端处理速度极快, 但网络穿越是最大开销
  对热 key 的高频读 (商品详情/配置/用户信息):
    每次都付 RTT 开销
    高 QPS 时网络带宽与连接成本叠加

解决方向:
  ✓ 把"最近的数据源"前移到客户端进程内
  ✓ 命中时延迟从 200 µs 降到 < 1 µs (本地 HashMap 查找)
  ✓ 不命中时回到原始路径

关键问题:
  Redis 中的 key 被修改后, 客户端如何感知?
    选项 1: 不感知, 用 TTL 兜底 → 脏读时长 = TTL
    选项 2: 客户端轮询版本号 → 退化为 RTT
    选项 3: 服务端推送失效 → Redis 6 选择此路径

32.1.2 Redis 选择目录式失效协议

图 32-1-2 · 4 种缓存一致性方案的取舍
图 32-1-3 · Redis 6 客户端缓存的三个核心承诺
Redis 6 客户端缓存的核心设计:

  目录式 (directory-based):
    服务端维护"哪些客户端缓存了哪些 key"
    写入时根据目录定向通知
    精确,但服务端有目录开销

  失效协议 (invalidation):
    通知中只含 key, 不含新 value
    客户端收到后驱逐本地副本
    下次读取时从服务端重新加载

  最终一致 (eventually consistent):
    通知是异步网络消息
    投递有微秒级延迟
    强一致需求场景禁用

服务端不替客户端管理:
  ✗ 本地缓存数据结构
  ✗ 本地 TTL 与淘汰策略
  ✗ 序列化格式

服务端只承诺:
  ✓ 失效通知至多丢一次 (断线时)
  ✓ 通知精确 (Tracking) 或前缀粗粒度 (BCAST)
为什么不推送新 value:推送 key 的失效通知是 O(key_len),而推送 value 是 O(value_size)。如果客户端已经"用完"该 key(例如展示页已渲染完毕),推送的新 value 就是浪费;如果客户端仍要用,它会立即触发一次新的 GET。Redis 把"是否还需要这个 key"的决策权交还给客户端,这也是无状态服务端的设计哲学。

32.2 Tracking 模式

Tracking 模式是 Redis 客户端缓存的默认形式。服务端为每个被读 key 维护一个 client_id 集合,写入时只通知集合中的客户端;通知粒度精确到 key,网络流量最小;代价是 tracking_table 的 per-key 内存随被读 key 数线性增长,需要 tracking-table-max-keys 上限 + 溢出驱逐。

32.2.1 启用与基本流程

# RESP3 协议下启用 (推荐, 单连接多路复用)
HELLO 3
CLIENT TRACKING ON

# RESP2 协议下需指定重定向目标 (双连接模式)
CLIENT TRACKING ON REDIRECT <client-id>

# 关闭 tracking
CLIENT TRACKING OFF

# 查看 tracking 全局状态
INFO tracking

32.2.2 服务端 tracking_table 结构

/* server.h - 简化字段 */
struct redisServer {
    rax *tracking_table;                   /* key → client_id_set */
    unsigned long tracking_table_max_keys; /* 默认 1,000,000 */
    rax *tracking_table_prefixes;          /* Broadcasting 前缀树 */
    unsigned long tracking_total_keys;
    unsigned long tracking_total_items;
};

struct client {
    uint64_t flags;                        /* CLIENT_TRACKING / BCAST / OPTIN / OPTOUT / NOLOOP */
    uint64_t client_tracking_redirection;  /* RESP2 重定向目标 */
    rax *client_tracking_prefixes;         /* BCAST 模式订阅的前缀 */
};

32.2.3 读命令登记触发点

/* db.c - lookupKeyReadWithFlags 简化路径 */
robj *lookupKey(redisDb *db, robj *key, int flags) {
    dictEntry *de = dictFind(db->dict, key->ptr);
    robj *val = de ? dictGetVal(de) : NULL;

    /* 关键: tracking 登记 */
    if (val && current_client_has_tracking()) {
        if (current_client->flags & CLIENT_TRACKING) {
            enableTracking(current_client, key);
        }
        /* BCAST 模式不在读时登记 */
    }
    return val;
}

/* tracking.c - enableTracking 简化 */
void enableTracking(client *c, robj *key) {
    /* 1. 检查表大小,溢出时驱逐 */
    if (server.tracking_total_keys >=
        server.tracking_table_max_keys) {
        evictTrackingTable();
    }

    /* 2. 加入 server.tracking_table */
    raxFind(server.tracking_table, key->ptr, sdslen(key->ptr));
    /* 若不存在创建 client_id_set, 加入当前 client_id */

    /* 3. 内存计数 */
    server.tracking_total_keys++;
    server.tracking_total_items++;
}
图 32-2-2 · 哪些命令会登记 tracking
登记规则:
  ✓ GET / MGET / HGET / HGETALL / LRANGE / SMEMBERS 等读命令
  ✓ 命令读取了不存在的 key 也会登记 (NULL value 也跟踪)
  ✓ 同一客户端重复读同 key 不重复登记 (set 去重)
  ✓ MULTI/EXEC 内的读也登记

  ✗ EXISTS 不登记 (不返回 value)
  ✗ TTL / TYPE 不登记
  ✗ 写命令不登记 (修改自身)
  ✗ DEBUG / CLUSTER / INFO 不登记

OPTIN 模式下:
  默认所有读都不登记
  仅当前一条命令是 CLIENT CACHING YES 时, 下条读命令登记
  适合"只缓存稳定热 key"场景

32.2.4 写命令触发失效

/* db.c - signalModifiedKey 简化 */
void signalModifiedKey(client *c, redisDb *db, robj *key) {
    touchWatchedKey(db, key);          /* WATCH 触发 */
    trackingInvalidateKey(c, key);     /* tracking 失效 */
    /* 其他: keyspace event 等 */
}

/* tracking.c - trackingInvalidateKey 简化 */
void trackingInvalidateKey(client *c, robj *key) {
    /* 1. 查找 server.tracking_table */
    raxNode *clients = raxFind(server.tracking_table,
                               key->ptr, sdslen(key->ptr));
    if (clients == raxNotFound) return;

    /* 2. 遍历所有持有副本的 client_id */
    for (client_id in clients) {
        client *target = lookupClientByID(client_id);

        /* NOLOOP: 自己写的不通知自己 */
        if ((target == c) &&
            (target->flags & CLIENT_TRACKING_NOLOOP))
            continue;

        sendTrackingMessage(target, key);
    }

    /* 3. BCAST 前缀匹配 */
    iterate_prefixes(server.tracking_table_prefixes, key);

    /* 4. 从 tracking_table 移除该 key */
    raxRemove(server.tracking_table, key->ptr, ...);
}
表 32-2-1 · 失效逻辑的四个关键性质
性质含义实现细节
时机写命令成功后立即触发signalModifiedKey 调用点
粒度整个 key,非字段HSET k f v 通知整个 k
去重同事务内多次写只通知一次事务/批量命令中合并失效
清理通知后从 tracking_table 移除客户端下次读取重新登记
失效后立即移除是关键优化:Redis 在通知发送后立即把该 key 从 tracking_table 移除——如果客户端不再读该 key,自然不再被跟踪,tracking_table 内存会自然回落。这也是"tracking_table 应等于活跃热 key 集合"的设计直觉:它不该单调增长。

32.3 Broadcasting 模式

Tracking 模式在 key 空间巨大时会让 tracking_table 内存吃紧。Broadcasting(BCAST)是一种嗅探式替代方案:客户端订阅前缀(如 user:),服务端只存一棵前缀树,写入时按前缀广播——任何 user:* 的写都通知所有订阅 user: 的客户端,无论它是否读过这个具体 key。

32.3.1 启用

# 订阅单个前缀
CLIENT TRACKING ON BCAST PREFIX user:

# 订阅多个前缀
CLIENT TRACKING ON BCAST PREFIX user: PREFIX product: PREFIX config:

# 空前缀 = 所有 key (危险, 不推荐)
CLIENT TRACKING ON BCAST PREFIX ""

# 与 NOLOOP 组合
CLIENT TRACKING ON BCAST PREFIX user: NOLOOP

32.3.4 Broadcasting 的内部实现

/* tracking.c - 简化 */
void trackingHandlePendingKeyInvalidations(void) {
    /* 在 signalModifiedKey 后调用 */

    /* 遍历所有 BCAST 前缀 */
    raxIterator iter;
    raxStart(&iter, server.tracking_table_prefixes);
    raxSeek(&iter, "^", NULL, 0);

    while (raxNext(&iter)) {
        bcastState *bs = iter.data;
        /* 检查 bs->keys (本次写命令影响的 key) 是否匹配前缀 */
        for (key in bs->keys) {
            if (key starts with bs->prefix) {
                for (client_id in bs->clients) {
                    sendTrackingMessage(client_id, key);
                }
            }
        }
    }
}
图 32-3-2 · Broadcasting 三大关键性质
Broadcasting 关键性质:

  服务端内存仅前缀树:
    若 N 个 BCAST 客户端订阅同前缀
    服务端只存 1 份前缀
    内存 = O(prefix 数 × prefix 长度)

  通知量:
    PREFIX user: + 写 user:1001 → 所有订阅 user: 的客户端收到
    无论该客户端是否曾读过 user:1001

  前缀匹配开销:
    每次写都遍历所有 BCAST 前缀
    前缀数过多 (> 100) 写性能下降
    实践中 BCAST 前缀通常 < 10

经验值:
  < 10 个 BCAST 前缀: 写性能几乎无损
  10-50 个: 单次写 +5-15% CPU
  > 100 个: 不可接受, 改 Tracking 或拆 Redis
不要用空前缀 BCAST:CLIENT TRACKING ON BCAST PREFIX "" 会让客户端订阅所有 key 的写入。在一个写 QPS 10k 的实例上,意味着该客户端每秒接收 10k 条 invalidate 消息——网络与客户端 CPU 都会被打爆。空前缀只在"全只读 + 偶发写"的特殊场景才有意义。

32.4 RESP3 Push 与 RESP2 重定向

客户端缓存协议层的核心矛盾是:命令-响应是同步的,失效通知是异步的——它们能否共用同一个 TCP 连接?RESP3 通过引入 Push 类型消息(> 前缀)给出了肯定回答;RESP2 没有 Push 能力,只能借助"双连接 + Pub/Sub 频道"的工程兜底。

32.4.1 RESP3 Push 类型消息

图 32-4-1 · RESP3 invalidate Push 的线缆格式
RESP3 Push (> 前缀) 协议:

  >2                    -- Push 类型, 2 个元素
  $10
  invalidate
  *1                    -- 数组, 1 个 key
  $11
  order:O1001

含义:
  这是一个 Push 消息 (异步推送, 不是命令响应)
  消息类型 = "invalidate"
  失效 key 列表 = ["order:O1001"]

客户端处理:
  从同一连接读取
  与命令响应区分: 命令响应是 +/-/$/:/* 前缀
  Push 是 > 前缀
  无需阻塞当前命令执行

批量失效示例 (一次通知多个 key):
  >2
  $10
  invalidate
  *3
  $11
  order:O1001
  $11
  order:O1002
  $11
  order:O1003

32.5 NOLOOP / OPTIN / OPTOUT

Redis 在 CLIENT TRACKING 上提供三个修饰符,把"通知粒度"从全有/全无细化到"按场景精挑"。它们各自解决一个具体的生产痛点:NOLOOP 解决"自写自收"、OPTIN 解决"临时数据污染 tracking_table"、OPTOUT 解决"高频抖动 key 撑爆 tracking"。

32.5.1 三个修饰符速查

表 32-5-1 · NOLOOP / OPTIN / OPTOUT 语义与用途
修饰符默认行为修改后用途
NOLOOP自己写也通知自己自己写不通知写后读模式避免冗余通知
OPTIN默认全跟踪默认不跟踪,仅 CLIENT CACHING YES 后下条命令精挑稳定热 key 缓存
OPTOUT默认全跟踪默认全跟踪,CLIENT CACHING NO 后下条不跟踪排除高频变动 key

32.5.2 NOLOOP 场景

图 32-5-1 · NOLOOP 关闭与开启时"自写自收"行为差异
不带 NOLOOP:
  client A: SET k v1
    → server: signalModifiedKey
    → server: tracking_table 中查 k
    → 若 A 自己也曾读过 k, A 也在订阅者列表
    → server 推送 invalidate 给 A

  问题:
    A 刚写, 自己当然知道值变了
    无需服务端再通知一次
    增加无意义网络流量

带 NOLOOP:
  client A: CLIENT TRACKING ON NOLOOP
  client A: SET k v1
    → server 跳过对 A 的通知
    → 其他订阅 k 的客户端仍收到通知

  收益:
    写后立即读自己写的值, 不会被自己的失效通知误删
    特别适合"写少读多但偶尔自写"的客户端

32.5.3 OPTIN 场景

# 默认不跟踪, 显式选择
CLIENT TRACKING ON OPTIN

# 不跟踪, 不进入 tracking_table
GET temp:abc

# 显式选择跟踪下一条命令
CLIENT CACHING YES
GET order:O1001    # 进入 tracking_table

# 又恢复默认 (不跟踪)
GET log:line:5     # 不跟踪
图 32-5-2 · OPTIN 让稳定热 key 不被临时数据挤走
OPTIN 适用场景:

  应用读取大量临时数据 + 少量稳定热数据
    临时数据: 用户搜索结果, 一次性聚合
    稳定数据: 配置, 商品基本信息, 用户档案

  问题: 默认全跟踪会让 tracking_table 充斥临时数据
    临时数据修改频率低 (几乎不修改)
    占用 tracking_table 容量
    挤掉真正需要缓存的稳定数据

  OPTIN 解决:
    应用层显式标记"这个 key 我要缓存"
    临时数据不进入 tracking_table
    服务端容量留给真正高价值的缓存对象

代码模板 (Java + lettuce):
  RedisCommands cmd = ...;
  cmd.clientTracking(TrackingArgs.Builder.enabled().optin());
  // 不缓存
  cmd.get("temp:abc");
  // 缓存下一条
  cmd.clientCaching(true);
  cmd.get("order:O1001");   // 进 tracking_table

图 32-5-3 · OPTOUT 排除高频抖动 key
OPTOUT 适用场景:

  大多数 key 适合缓存, 少数 key 高频变动:
    商品信息 (适合缓存) + 实时库存 (高频变动)
    用户档案 (适合) + 在线状态 (高频)

  默认 ON 跟踪所有, OPTOUT 标记不跟踪的高频 key
  避免高频 key 撑爆 tracking_table
  避免大量无效 invalidate 通知

经验法则:
  若高频 key 比例 < 5% → OPTOUT 更省事
  若稳定 key 比例 < 30% → OPTIN 更精确
  若两者各占一半 → 拆 Redis 实例

32.6 一致性与延迟数学

客户端缓存的工程价值最终要落在"数"上。本节给出四个核心公式:命中率与读写比、延迟加速比、tracking_table 内存预算、脏读概率。它们是容量规划、性能估算、风险评估的共同语言。

32.6.1 定理 32-6-1:延迟缩减

图 32-6-1 · 命中率 h 下的延迟加速比公式
设系统命中率 h ∈ [0, 1], 网络 RTT R, 本地查找延迟 L (L ≪ R)
每次查询期望延迟:
  E[latency] = h × L + (1 - h) × R

延迟加速比:
  Speedup = R / E[latency] = 1 / (1 - h + h × L/R)

当 L/R → 0 (典型值 L = 1 µs, R = 200 µs, L/R = 0.005):
  Speedup ≈ 1 / (1 - h)

数值:
  h = 0.5  → Speedup ≈ 2×
  h = 0.9  → Speedup ≈ 10×
  h = 0.95 → Speedup ≈ 20×
  h = 0.99 → Speedup ≈ 100×
  h = 0.999 → Speedup ≈ 1000×

观察:
  命中率提升的边际收益指数增长
  从 90% 到 99% 比从 50% 到 90% 收益更大
  命中率越高收益越大,但目标值要按业务收益和内存成本确定

32.6.2 定理 32-6-2:命中率与读写比

图 32-6-2 · 在 W 写间隔、r 读间隔下的稳态命中率
设某 key:
  W = 平均写入间隔 (秒)
  r = 平均读取间隔 (秒), r ≤ W

两次写入之间的读取次数 = W/r
其中第一次必为 miss (本地缓存被失效后清空)
后续 (W/r - 1) 次命中

命中率:
  h = (W/r - 1) / (W/r) = 1 - r/W

数值示例:
  W = 60 s, r = 0.1 s (每秒 10 读, 每分钟 1 写):
    h = 1 - 0.1/60 = 99.83%

  W = 60 s, r = 1 s (每秒 1 读, 每分钟 1 写):
    h = 1 - 1/60 = 98.33%

  W = 1 s, r = 0.1 s (每秒 10 读, 每秒 1 写):
    h = 1 - 0.1/1 = 90%

  W = 0.1 s, r = 0.1 s (读写比 1:1):
    h = 1 - 1 = 0% (理论, 实际更糟)

工程含义:
  读多写少的场景客户端缓存收益巨大
  读写比接近 1:1 时本地缓存基本无用
  适合: 配置, 商品基本信息, 用户档案
  不适合: 实时计数器, 高频变动状态

32.6.3 定理 32-6-3:tracking_table 内存预算

图 32-6-3 · 服务端 tracking_table 内存的三大因子
设:
  K = 被跟踪的不同 key 数
  C = 平均每 key 被多少客户端缓存
  L_k = key 平均长度 (字节)

tracking_table 内存:
  S = K × (L_k + sizeof(rax_node) + C × 8)
  ≈ K × (L_k + 64 + C × 8)

数值示例:
  K = 1,000,000
  L_k = 20 字节
  C = 10 (10 个 client 缓存同 key)

  S = 1,000,000 × (20 + 64 + 80) ≈ 164 MB

内存膨胀因素:
  ✗ K 增大: 被读 key 越多内存越大
  ✗ C 增大: 客户端越多每 key 集合越大
  ✗ 长 key: L_k 主导

控制方式:
  tracking-table-max-keys 限制 K (默认 1,000,000)
  超限时驱逐旧条目 (LRU 类)
  改 BCAST 可让 S 跌到 KB 级

32.6.4 定理 32-6-4:脏读概率

图 32-6-4 · 失效通知延迟 d 下的脏读概率
设:
  d = 失效通知到达延迟 (秒)
  W = key 写入间隔 (秒, Poisson 过程参数 λ = 1/W)

在收到失效通知前的读取窗口内, 写入概率:
  P(stale_read | read_during_d) = 1 - e^(-d/W) ≈ d/W
                                  (当 d ≪ W)

数值示例:
  d = 1 ms, W = 60 s:
    P(stale) ≈ 1/60000 ≈ 0.0017%

  d = 1 ms, W = 1 s:
    P(stale) ≈ 1/1000 = 0.1%

  d = 10 ms, W = 1 s:
    P(stale) ≈ 1%

工程含义:
  常态下脏读概率极低
  网络拥塞时 d 上升, 脏读概率上升
  强一致需求 (余额/库存) 不应依赖客户端缓存
  最终一致需求 (展示/搜索) 概率可忽略
"最终一致"≠"很快一致":在网络出现微突发(microburst)或 Redis 短时阻塞(例如 BGSAVE fork 期间)时,失效通知延迟 d 可以从 1 ms 跳到 100 ms。脏读概率随之放大 100 倍。任何"强校验"业务路径(余额、库存、风控)都必须从客户端缓存路径上摘出来,直接走 Redis 或 Lua 原子读改写。

图 32-7-1 · MESI 四状态在 Redis Tracking 中的对应
CPU MESI 协议 (Modified/Exclusive/Shared/Invalid):

  Modified:    本核独占 + 已修改, 主内存过时
    ⇔ Tracking: 客户端刚 SET 该 key (NOLOOP 场景)

  Exclusive:   本核独占 + 未修改
    ⇔ Tracking: 仅一个客户端读过该 key

  Shared:      多核持有相同干净副本
    ⇔ Tracking: 多个客户端缓存同 key

  Invalid:     已失效, 需重读
    ⇔ Tracking: 收到 invalidate 后驱逐本地

──────────────────────  ──────────────────
Read Hit (S/E/M 状态)   GET 命中本地缓存
Read Miss               GET 不命中, 查 Redis
Write (要变 M)          SET 命令
Bus Read (其他核读)     其他客户端 GET 同 key
Bus Invalidate (其他写) 其他客户端 SET → 推送 invalidate

32.7.3 目录式 vs 嗅探式的硬件映射

图 32-7-2 · Tracking ≈ 目录式;Broadcasting ≈ 嗅探式
Tracking 模式 ≈ 目录式缓存一致性 (Directory-based Coherence):
  服务端维护"目录": 哪些客户端持有哪些 key
  写入时定向通知, O(订阅者数)
  内存开销大, 网络流量小

  硬件实现: NUMA 系统 / 多 socket CPU

Broadcasting 模式 ≈ 嗅探式缓存一致性 (Snoopy Protocol):
  所有客户端订阅"总线" (前缀)
  写入广播到所有订阅者
  服务端无目录, 客户端按前缀过滤

  硬件实现: 单 socket 多核 (共享 bus)

工程映射:
  Tracking + Broadcasting 选型与硬件相同的考量:
    节点少 + 高一致 → 嗅探/Broadcasting
    节点多 + 跨域 → 目录/Tracking
  Redis 灵活提供两种, 由业务按场景选择
类比的价值在于看清取舍:Tracking 记录"谁读过什么",内存更精确但要维护目录;Broadcasting 按前缀广播,服务端状态更少但会产生虚假失效。Redis 6 两种都提供,让业务按 key 空间规模、写入频率和可接受的虚假失效来选择。

32.8 多部署模式

客户端缓存的"tracking_table 是单节点本地状态"在生产部署里会暴露所有"多节点假设"。下面按单节点 → 主从 → Cluster → Sentinel 四种部署逐个拆解,重点说清楚"什么时候必须清空本地缓存"。

32.8.1 四种部署的行为对照

表 32-8-1 · 不同部署模式下客户端缓存的行为与约束
部署行为关键约束
单节点标准失效表完整覆盖,最佳场景
主从复制读写分离需谨慎客户端缓存连接读写应同主节点;切换时清缓存
Cluster每分片独立 tracking按分片建立 tracking 连接;MOVED 后清相关 key

32.8.2 Cluster 模式的四大隐性挑战

图 32-8-1 · Cluster 客户端缓存的四类隐性挑战
Cluster 客户端缓存的隐性挑战:

  挑战 1: 多分片连接
    每个 master 独立维护 tracking_table
    客户端需对每个分片建立独立的 tracking 连接
    总连接数 = master 数 × N

  挑战 2: MOVED 重定向
    slot 迁移时, 旧节点的 tracking 关系不再同步新节点
    客户端持有的本地缓存可能脏读
    应清除该 slot 的所有 key 缓存, 在新节点重新登记

  挑战 3: ASK 重定向
    迁移期间临时重定向
    若客户端按 ASK 跑到新节点
    tracking 状态在两节点间不一致

  挑战 4: 节点失败
    某 master 失败后, 其 tracking_table 全部丢失
    failover 后新 master 从 0 开始
    客户端在该分片上的本地缓存全部不可信

最佳实践:
  ✓ 客户端按 slot 分区管理本地缓存
  ✓ 收到 MOVED/ASK 立即清相关 key 缓存
  ✓ 监听 cluster topology 变化, 清整片缓存
  ✓ 拓扑变化频繁的集群慎用客户端缓存

32.8.3 Sentinel / 主从切换

图 32-8-2 · failover 时序与本地缓存必须清空的根因
图 32-8-3 · 不清空本地缓存的脏读链路
切换后必须清空本地缓存的原因:

  时刻 T0: 客户端缓存了 k=v1 (从旧 Master 读)
  T1: 其他写入者 SET k v2 (在旧 Master)
  T1+: 旧 Master 推 invalidate 给客户端 (但可能在 failover 中丢失)
  T2: 旧 Master 崩溃
  T3: 新 Master 提升 (异步复制可能未同步 v2 → 新 Master 的 k 仍是 v1)

  情形 A: 客户端继续信任本地 v1
    实际可能: v1 (新 Master 同步落后) 或 v2 (新 Master 已同步)
    无法判断, 必然存在脏读风险

  情形 B: 客户端清空本地缓存, 重新读
    新 Master 返回 v1 或 v2 (取决于复制状态)
    至少与新 Master 的当前状态一致
    避免"读到比新 Master 更旧"的脏读

工程要点:
  failover 检测必须触发本地缓存清空
  代价: 短期 miss 率 100%, 性能下降
  收益: 一致性, 避免数据错误
典型风险案例: 如果 Sentinel failover 后客户端没有清空本地缓存,应用可能继续使用旧 Master 时期缓存的值;而新 Master 的 tracking_table 又没有这条客户端订阅记录,后续写入也未必能通知到它。修复思路很直接:在客户端库的重连 / 拓扑变更回调中调用 localCache.clear(),并把余额、库存这类强一致 key 排除在客户端缓存白名单外。

32.10.2 tracking_table 溢出的连锁反应

图 32-10-1 · 溢出 → 虚假失效 → 命中率崩塌的死亡螺旋
正常状态:
  tracking_total_keys < max_keys
  每个被读 key 对应一个 client_id_set
  失效通知精准

接近上限:
  新读取的 key 触发驱逐旧条目
  驱逐时发送虚假 invalidate (实际未修改)
  客户端清缓存 → miss → 重读 → 重新登记
  循环

完全溢出:
  每秒新增 key 数 > 驱逐速率
  大量 key 在循环
  每秒虚假 invalidate 通知 (~万级)
  客户端命中率 < 50%
  服务端 CPU 升高 (驱逐 + 通知开销)
  网络带宽消耗增加

  反向收益:
    本地缓存命中率太低
    实际比无缓存还慢

诊断:
  $ redis-cli INFO tracking | grep tracking_total_keys
  $ redis-cli CONFIG GET tracking-table-max-keys

  对比:
    tracking_total_keys = 1,823,456
    max = 1,000,000
    → 实际溢出 82.3%

修复路径:
  1. 提高 max_keys (CONFIG SET, 估算内存)
  2. 改 OPTIN, 限制跟踪 key 数
  3. 大 key 空间改 BCAST
  4. 客户端缩短本地 TTL 减少 tracking 压力

❌ 反模式 · 默认全跟踪 + 大量临时 key

# 大量读临时数据 (一次性聚合)
for log_id in range(1_000_000):
    r.get(f"log:line:{log_id}")

# 真正想缓存的稳定 key 被挤出
hot = r.get("config:feature-a")

tracking_table 被 100 万临时 key 撑爆,真正高价值的 config:* 反而在驱逐链路里频繁失效。命中率从 95% 跌到 30%,服务端 CPU 反向上升。

✅ 正确做法 · OPTIN + 白名单

r = redis.Redis(protocol=3)
r.execute_command("CLIENT", "TRACKING", "ON", "OPTIN", "NOLOOP")

# 临时数据: 不跟踪, 不污染 tracking_table
for log_id in range(1_000_000):
    r.get(f"log:line:{log_id}")

# 显式缓存稳定热 key
r.execute_command("CLIENT", "CACHING", "YES")
hot = r.get("config:feature-a")

OPTIN + 白名单让 tracking_table 只保留真正高价值的少量条目,内存与服务端 CPU 都回到可控水位,稳定 key 的命中率也更容易维持。

32.10.4 断线重连客户端模板

class CachedRedisClient:
    def __init__(self, redis_url):
        self.local_cache = {}
        self.redis = redis.Redis.from_url(redis_url, protocol=3)
        self.setup_tracking()

    def setup_tracking(self):
        self.redis.execute_command("HELLO", "3")
        self.redis.execute_command(
            "CLIENT", "TRACKING", "ON", "NOLOOP")
        self.redis.set_response_callback(
            "_invalidate_callback", self.on_invalidate)

    def on_invalidate(self, keys):
        for key in keys:
            self.local_cache.pop(key, None)

    def on_disconnected(self):
        # 关键: 断线必清空本地缓存
        size = len(self.local_cache)
        self.local_cache.clear()
        log.warning(
            f"redis disconnected, cleared {size} cache entries")

    def on_reconnected(self):
        # 重连后重新建立 tracking
        self.setup_tracking()
        log.info("redis reconnected, tracking re-enabled")

    def get(self, key):
        if key in self.local_cache:
            return self.local_cache[key]
        value = self.redis.get(key)
        if value is not None:
            self.local_cache[key] = value
        return value
客户端模板的 5 个回调点:on_disconnected 立即清空本地缓存;on_reconnected 重新 HELLO 3 + CLIENT TRACKING ON;on_invalidate 仅驱逐对应 key、不清整个缓存;本地缓存设置 LRU 上限(10 万条);本地 TTL 兜底(60-120 s),防止通知丢失永久脏。少做任何一个都会在生产里被某一次故障演练抓到。

32.11.2 sendTrackingMessage 关键路径

/* tracking.c - 失效通知发送 */
void sendTrackingMessage(client *c, char *keyname, size_t keylen, int proto) {
    int using_redirection = 0;
    if (c->client_tracking_redirection) {
        /* RESP2 重定向模式 */
        client *redir = lookupClientByID(c->client_tracking_redirection);
        if (!redir) {
            /* 重定向目标已断开,降级或断开本客户端 */
            c->flags |= CLIENT_CLOSE_AFTER_REPLY;
            return;
        }
        c = redir;
        using_redirection = 1;
    }

    if (c->resp == 2 && using_redirection) {
        /* RESP2: 通过 __redis__:invalidate 频道发送 */
        addReplyPubsubMessage(c, shared.invalidate_channel,
                             keyname, keylen);
    } else if (c->resp >= 3) {
        /* RESP3: Push 类型消息 */
        addReplyPushLen(c, 2);
        addReplyBulkCBuffer(c, "invalidate", 10);
        addReplyArrayLen(c, 1);
        addReplyBulkCBuffer(c, keyname, keylen);
    }
}

32.11.3 tracking_table 驱逐

/* tracking.c - tracking_table 驱逐 */
void evictTrackingTable(void) {
    /* 当 tracking_total_keys >= max_keys 时调用 */
    raxIterator iter;
    raxStart(&iter, server.tracking_table);
    raxSeek(&iter, "^", NULL, 0);

    /* 简单策略: 第一个 key (实质是任意一条) */
    if (raxNext(&iter)) {
        rax *clients = iter.data;
        /* 通知所有客户端该 key 已"失效" (实为驱逐) */
        for (client_id in clients) {
            sendTrackingMessage(client_id, iter.key, iter.key_len, 0);
        }
        raxRemove(server.tracking_table, iter.key, iter.key_len, NULL);
        server.tracking_total_keys--;
    }
    raxStop(&iter);
}

32.11.5 设计取舍

取舍 1:服务端辅助而非纯客户端方案

纯客户端方案(TTL/版本号轮询)要么脏读时长固定为 TTL(无视实际写频率),要么需要轮询验证(退化为 RTT 模式)。服务端辅助方案让"知道何时失效"的责任落在最适合的位置:Redis 自己,因为它知道每次写入的精确时刻。代价是服务端额外维护 tracking_table 内存,以及实现失效通知协议。

取舍 2:目录式 Tracking 作为默认而非纯嗅探式 BCAST

BCAST 模式所有匹配前缀的客户端都收到通知,包括从未读过该 key 的客户端,产生大量虚假流量。Tracking 模式精确投递,网络流量最小化。代价是 tracking_table 内存随被读 key 数线性增长,需要 max_keys 限制 + 溢出驱逐。Redis 同时提供两种,让业务按 key 空间规模选择。

取舍 3:RESP3 Push 而非延续 Pub/Sub 兜底

RESP3 引入 Push 类型是为客户端缓存量身打造的协议升级。RESP2 没有原生的服务端推送能力,只能借助 Pub/Sub 频道与客户端 ID 重定向,代价是必须双连接。RESP3 Push 让命令-响应与异步推送共用同一连接,客户端库实现单一多路复用即可。这是 Redis 6 在协议层的重大演进,远超客户端缓存本身的价值,为后续异步通知场景奠定基础。

取舍 4:NOLOOP / OPTIN / OPTOUT 作为修饰符而非新机制

三个修饰符不是核心机制的扩展,而是对实际生产场景观察后的精细化调优开关。同样的设计哲学在 Stream 的 XAUTOCLAIM 与 Pub/Sub 的 client-output-buffer-limit 中也能看到:Redis 优先提供原语,然后基于生产反馈持续添加可观测性与可控性接口,而不是把所有可能性预先做成"全功能命令"。
