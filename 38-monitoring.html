第 38 章 · 监控与故障排查

五维度监控矩阵 · INFO / SLOWLOG / LATENCY / MEMORY / CLIENT LIST · 命中率窗口差分 · redis_exporter · 告警 Runbook · 故障决策树

Redis 内存逼近上限、命中率下降、复制断链、慢命令阻塞、Pub/Sub 慢消费者、AOF 写失败等问题在事故前均可被指标捕捉。监控不是"看见",而是"指标触发告警 → 告警指向命令 → 命令定位根因 → 根因绑定动作"的完整证据链。任一环缺失,Grafana 仅是截图工具。

Redis 提供五类内建自省入口:INFO(全局计数器)、SLOWLOG(慢命令环形缓冲)、LATENCY(延迟事件采样)、MEMORY(分配器与 key 内存)、CLIENT LIST(客户端状态)。redis_exporter 将这些数据转换为 Prometheus 时序指标,Grafana 可视化,Alertmanager 触达运维。

  • 怎样把内存、延迟、连接、复制和持久化指标连成可执行证据链

  • 哪些 Redis 自省命令适合高频采集,哪些只能在告警后低频定位
  • 告警如何写成 runbook,让值班人员知道查什么、阈值是什么、下一步做什么

38.2.3 命中率必须用窗口差分

错误:瞬时累计

hit_rate = keyspace_hits / (keyspace_hits + keyspace_misses)
  • 历史高命中累计 → 掩盖最近下降
  • 长跑 Redis 累计值千亿级,新增几百万 miss 看不出变化

正确:窗口差分 + PromQL

delta_hits   = hits(t)   - hits(t - window)
delta_misses = misses(t) - misses(t - window)
hit_rate     = delta_hits / (delta_hits + delta_misses)
rate(redis_keyspace_hits_total[5m])
  / (rate(redis_keyspace_hits_total[5m])
     + rate(redis_keyspace_misses_total[5m]))
常见误判: Grafana 上把 hits / (hits + misses) 直接画成折线,长跑 Redis 累计值千亿级别,最近 1 小时新增几百万 miss 在图上看不出任何变化,等业务发现命中率掉了已经晚 30 分钟。必须用 rate() 窗口差分。

groups:

      rules:
            - alert: RedisMemoryHigh
              expr: redis_memory_used_bytes / redis_memory_max_bytes > 0.9
              for: 5m
              labels:
                  severity: critical
              annotations:
                  summary: "Redis memory usage above 90%"
                  runbook: "Check MEMORY STATS, bigkeys, client buffers, then scale or evict."
            - alert: RedisHitRateDrop
              expr: |
                  rate(redis_keyspace_hits_total[5m])
                      /
                  (rate(redis_keyspace_hits_total[5m])
                      + rate(redis_keyspace_misses_total[5m])) < 0.95
              for: 10m
              labels:
                  severity: warning

MONITOR 是核武器: 在生产实例上跑 MONITOR 会把每条命令格式化后转发给 MONITOR 客户端,主线程 CPU 与网络开销会显著上升,还可能泄露敏感参数。MONITOR 权限取决于 Redis 版本、ACL 与实际配置;生产环境应默认禁用或严格限权,只在受控诊断窗口使用。

38.11.3 设计取舍

取舍 1:事件采样而非全量直方图 — LATENCY 仅记录超过阈值的事件,不维护每请求耗时直方图。该选择开销几乎可忽略,代价是无法回答 "P99.99 是多少"。需要分位数时,需在客户端、代理或 Prometheus 层维护直方图。

取舍 2:INFO 全量字符串而非 JSON — INFO 输出为 RESP 字符串,每次调用拼接全部字段。该格式简单,代价是 exporter 解析需要正则。Redis 7.0 引入 RESP3 Map 类型,客户端可直接获得 key-value 结构。

取舍 3:SLOWLOG 环形缓冲而非外部存储 — 慢日志保存在内存环形缓冲(默认 128 条),Redis 重启即丢失。该设计避免了"监控本身需要持久化"的复杂性。生产应将 SLOWLOG 定期采集到外部时序库或日志系统。

取舍 4:CLIENT LIST 同步遍历 — CLIENT LIST 在主线程同步遍历 client 链表。该实现简单且数据准确(不存在抓取窗口的数据漂移),代价是高连接数下成为慢命令。频率应低于 1 次/分钟。
