第 3 章 · dict 哈希表

开链法 + 渐进式 rehash + SipHash-2-4 · 双表共存 · dictType 回调 · scan 反向二进制 · keyspace / expires / hashtable 编码

dict 是 Redis 中分布最广的数据结构。它同时承担:键空间(db->dict 与 db->expires)、Hash 类型的 hashtable 编码、Set 类型的 hashtable 编码、ZSet 的 member→score 索引、命令表(server.commands)、Pub/Sub 订阅关系(server.pubsub_channels)、cluster 节点表(server.cluster->nodes)等。一个 Redis 进程内常驻 3-N 个 dict 实例,所有 SET/GET/HSET/ZADD/SUBSCRIBE 等命令最终都会落到 dictFind 或 dictAddRaw。

dict 必须在单线程模型下提供:O(1) 平均查找、抗碰撞攻击、支持百万级 key 的扩容而不阻塞主线程。前两项决定算法选择(开链法哈希 + SipHash-2-4),第三项决定本章的核心机制——渐进式 rehash:把一次可能耗时数百毫秒的全表迁移,拆成"每次读写顺手搬一个 bucket"的小步,保证主线程任何单次操作的延迟可控。

本章按"动机 → 布局 → 7 种 CRUD 操作逐帧 → 算法对比 → 陷阱 → 源码"七层组织。§3.8 渐进式 rehash 是本章核心,展示如何把 O(n) 的搬迁分摊到每次操作。其他章节如 4-skiplist、6-listpack、8-rax 都会在自己的角色中调用 dict —— 学透 dict 等于打通了 Redis 一半的内核。

  • dict / dictht / dictEntry / dictType 四个结构在 64-bit 系统上的字节级布局
  • dictAdd 头插法、链路重连、ht_used 维护的 5 帧动画
  • dictFind 在 rehash 中的"先 ht[0] 后 ht[1]"双表查找语义
  • dictDelete 删除链头与链中节点的 prev 指针差异
  • dictReplace 的"先 Add 失败再 Find 更新"双路径与 valDestructor 释放时机
  • dictExpand 触发条件:负载因子 1 / BGSAVE 期间 force_resize_ratio=5
  • _dictRehashStep 把 O(n) 拆成 O(1) 步的核心实现 + serverCron 主动 rehash
  • dictScan 反向二进制迭代如何在 rehash 期间保证最终覆盖性
  • 开链 vs 开放寻址、哈希 vs 平衡树、SipHash vs MurmurHash 的取舍依据
  • 典型陷阱:KEYS 全表扫描、单 Hash 百万 field、fork 期 dict 修改触发 CoW

3.1 解决什么问题

在分析任何数据结构选型前,先把"业务到底要什么"列清楚。Redis 内部多处使用哈希表的场景虽然角色不同,但访问特征高度一致:按 key 的 O(1) 查找 + 动态扩容到百万级 + 抗用户构造的碰撞攻击。这三个硬约束,排除了几乎所有"只满足两条"的候选方案。

② 元素总数从数百到数十亿,需要动态扩容;
③ 单线程模型下任何 O(n) 操作都阻塞所有客户端;
④ 用户可控制 key 内容,需防止恶意构造哈希碰撞攻击(Hash-Flooding DoS)。

3.2 整体布局

dict 由 三层结构组成:外层 dict 容器持有"双 bucket 数组"(ht_table[0] 与 ht_table[1]);中层 bucket 数组每个槽位是 dictEntry*(开链法);最内层是 dictEntry 链节点(key/value/next)。双 ht 是渐进式 rehash 的物质基础:平时只用 ht_table[0],rehash 期间 ht_table[1] 出场,完成后两表交换、ht_table[1] 复位。

3.2.5 C 结构定义

typedef struct dictEntry {
    void *key;
    union {
        void *val;
        uint64_t u64;
        int64_t s64;
        double d;
    } v;
    struct dictEntry *next;
} dictEntry;

typedef struct dict {
    dictType *type;
    dictEntry **ht_table[2];
    unsigned long ht_used[2];
    long rehashidx;
    int16_t pauserehash;
    signed char ht_size_exp[2];
    void *metadata[];
} dict;

typedef struct dictType {
    uint64_t (*hashFunction)(const void *key);
    void *(*keyDup)(dict *d, const void *key);
    void *(*valDup)(dict *d, const void *obj);
    int (*keyCompare)(dict *d, const void *key1, const void *key2);
    void (*keyDestructor)(dict *d, void *key);
    void (*valDestructor)(dict *d, void *obj);
    int (*expandAllowed)(size_t moreMem, double usedRatio);
    size_t (*dictEntryMetadataBytes)(dict *d);
} dictType;
三个紧凑设计的妙处:
① ht_size_exp 用 1 byte 存 log2 而非 8 byte 存 size,省掉指数级浪费;
② dictEntry.v 是 union,expires dict 直接 s64 存毫秒时间戳省 8 byte 间接寻址;
③ metadata[] 柔性数组允许 dictType 声明"附加字节"(如 cluster 用它存 link),无需额外分配。

3.3 dictAdd 插入操作

插入是 dict 最常见的写操作,内部分两层:外层 dictAdd 串"找 bucket → 插链 → 设值",内层 dictAddRaw 负责真正的 entry 分配。dictAddRaw 在多个位置被复用(如 dictReplace),所以独立成函数。每次进入 dictAddRaw 都会顺手推进一步 rehash —— 这是渐进式 rehash 的核心节拍。

3.4 dictFind 查询操作

查询是 dict 最高频的操作:每个 GET / EXISTS / TYPE / TTL 命令都至少调用一次 dictFind。在未 rehash 时只查 ht[0],rehash 期间 必须先查 ht[0] 再查 ht[1] —— 因为新 entry 可能进了 ht[1],也可能旧 entry 还没搬走。双表查找的"先后"顺序很关键:先 ht[0] 命中可直接返回,没必要扫 ht[1]。

3.5 dictDelete 删除操作

删除涉及"找到目标 entry 并把它从链上摘下,然后释放内存"。关键差异在于"删除链头" vs "删除链中节点":链头要改 bucket 指针,链中节点要改前驱 prev->next。Redis 的实现把这两种情况合并到 dictGenericDelete,通过 prev == NULL 区分。

3.6 dictReplace 更新操作

dictReplace 是 SET 命令的内核:"已存在则更新,不存在则新增"。实现上是"先 dictAdd,失败则 dictFind + dictSetVal"的双路径。注意旧 value 必须由 valDestructor 释放(否则 redisObject 引用计数泄漏),Redis 通过 auxentry 局部变量保存旧值,先把新值挂上,再释放旧值,保证异常安全。

3.7 dictExpand 扩容触发

扩容由 _dictExpandIfNeeded 在每次 dictAddRaw 内被调用判定。核心阈值是负载因子(used / size):正常情况下 ≥ 1 立即扩容,但 BGSAVE / AOFRW 期间为减少 fork CoW 压力放宽到 5(dict_force_resize_ratio),只有触及 5 才被迫扩容。dictExpand 只分配新表 + 标记 rehashidx=0,真正的搬迁延后到后续操作。

3.8 _dictRehashStep 渐进式迁移(本章核心)

3.8.1 设计动机

百万 key 哈希表的一次性 rehash会成为单线程灾难:

100 ns
单次 dictEntry 迁移耗时
100 ms
1M entry × 100 ns 阻塞
≥ 100 ms
期间所有命令延迟
P99 SLA
直接破坏
渐进式 rehash 的核心思想:把这 100 ms 拆成 1M 次每次 100 ns,跟随每个客户端命令分摊。每次 dictAdd / dictFind / dictDelete 顺手搬 1 个 bucket,搬完整张表的总 CPU 时间不变(O(n)),但单次操作延迟稳定在 O(链长) ≈ O(1)。代价是 rehash 期间双表共存,内存占用约 1.5-2 倍。

渐进式 rehash 用"双表共存的暂时内存放大"换取"单次操作的延迟可控"。代价是 rehash 期间内存占用约为正常 2 倍——若内存预算紧张,扩容会推迟 fork 期持续时间,加剧 CoW 压力。

3.10.4 设计动机总结

Redis 选择"开链哈希 + 渐进 rehash + SipHash-2-4"的 5 个核心动机:
① O(1) 平均查找:百万级 keyspace 的查找延迟必须在 microsecond 量级,只有哈希表能达到;
② 单次操作延迟可控:渐进 rehash 把 O(n) 的扩容拆成 O(1) 步,主线程任何单次操作都不阻塞;
③ 抗碰撞攻击:用户可控制 key 内容,SipHash-2-4 作为加密强度伪随机函数防止主动碰撞构造;
④ 删除友好:开链法支持任意时刻删除,无 tombstone 负担,适合 expire/maxmemory 频繁淘汰场景;
⑤ 内存可控:每 entry 24 字节 + bucket 数组 8 字节,相比红黑树节省 30%+。

3.11.3 反例 1:KEYS 全量扫描 + 同步 DEL 大对象

会让线上扑街的写法

127.0.0.1:6379> KEYS *
(以百万 key,主线程阻塞数秒)

127.0.0.1:6379> DEL bigorder:O1
(数十万 dictEntry 同步释放,P99 突增)

# 现象:
# - KEYS * 扫遍 db->dict 两张 ht 全部 bucket 链
#   1M key 大概 30-100 ms 阻塞,期间所有客户端等待
# - DEL 大对象触发 valDestructor 链调用
#   每个 entry 释放 1-2 μs,10W field 累积 100-200 ms 主线程毛刺
# - 监控告警:slowlog get 10 全部是这两条命令

127.0.0.1:6379> SCAN 0 MATCH "order:*" COUNT 100
1) "192"
2) ...

127.0.0.1:6379> UNLINK bigorder:O1

# 配合:
127.0.0.1:6379> CONFIG SET lazyfree-lazy-user-del yes

# 收益:
# - SCAN 每次只扫 100 个 bucket,< 1 ms
# - UNLINK 主线程 O(1) 摘链,真正释放在 bio 后台线程
# - lazyfree-lazy-user-del 让 DEL 也走 lazy 路径
# - 整体 P99 降低 100-1000 倍

3.11.4 反例 2:单 Hash 塞百万 field

把 1M 用户属性塞进单个 Hash

# 错误:所有用户在一个 Hash
HSET user:profile uid_1 '{...}'
HSET user:profile uid_2 '{...}'
...
HSET user:profile uid_1000000 '{...}'

# 后果:
# - 单 Hash dict 1M field
# - rehash 期间双表共存 ~ 80 MB 仅元数据
# - HGET 时间稳定但 fork 时 CoW 整页拷贝
# - DEL user:profile 主线程阻塞数百 ms

按 hash 分片到 N 个 Hash

# 正解:用 uid hash 分片到 1000 个 Hash
shard = hash(uid) % 1000
HSET user:profile:{shard} uid_X '{...}'

# 收益:
# - 每个 Hash 1K field,rehash 内存占用低
# - DEL 单分片只影响 1K entry
# - 可以并行 SCAN 不同分片
# - cluster 模式下 {shard} 让分片落到同一节点
#   保证多 key 操作不跨槽

3.11.5 反例 3:fork 期间大量写触发 CoW

RDB / AOFRW fork 期间 dict 扩容的连锁反应:① 子进程通过 CoW 共享父进程内存;② 父进程任何 dict 写都可能触发扩容,扩容会修改整个 bucket 数组(几 MB 到几十 MB);③ 修改触发 CoW 把整页(4KB)拷贝到父进程私有空间;④ 监控指标 aof_last_cow_size / rdb_last_cow_size 飙升,RSS 远超 used_memory;⑤ 极端情况下 RSS 翻倍,触发 OOM-killer。
解决方案:错峰持久化(BGSAVE 在低峰期);关闭透明大页 THP(echo never > /sys/kernel/mm/transparent_hugepage/enabled);监控 aof_last_cow_size,> 1 GB 时告警;依靠 dict_force_resize_ratio=5 推迟扩容到 fork 完成。

int dictAdd(dict *d, void *key, void *val) {
    dictEntry *entry = dictAddRaw(d, key, NULL);
    if (!entry) return DICT_ERR;
    dictSetVal(d, entry, val);
    return DICT_OK;
}

int dictRehash(dict *d, int n) {
    int empty_visits = n * 10;
    if (!dictIsRehashing(d)) return 0;

    while (n-- && d->ht_used[0] != 0) {
        while (d->ht_table[0][d->rehashidx] == NULL) {
            d->rehashidx++;
            if (--empty_visits == 0) return 1;
        }
        dictEntry *de = d->ht_table[0][d->rehashidx];
        while (de) {
            dictEntry *nextde = de->next;
            uint64_t h = dictHashKey(d, de->key) &
                         DICTHT_SIZE_MASK(d->ht_size_exp[1]);
            de->next = d->ht_table[1][h];
            d->ht_table[1][h] = de;
            d->ht_used[0]--;
            d->ht_used[1]++;
            de = nextde;
        }
        d->ht_table[0][d->rehashidx] = NULL;
        d->rehashidx++;
    }

    if (d->ht_used[0] == 0) {
        zfree(d->ht_table[0]);
        d->ht_table[0] = d->ht_table[1];
        d->ht_used[0] = d->ht_used[1];
        d->ht_size_exp[0] = d->ht_size_exp[1];
        _dictReset(d, 1);
        d->rehashidx = -1;
        return 0;
    }
    return 1;
}

3.12.4 设计取舍

取舍 1:开链法 vs 开放寻址

开放寻址(线性探测、二次探测、双重哈希等)的内存连续性优于开链,cache friendly。但 Redis 选择开链:

  • 删除友好:expire/maxmemory 频繁淘汰,开放寻址需放 tombstone,长期累积退化
  • 所有权清晰:每 entry 独立 zmalloc,key/value 生命周期由 dictType 回调统一管理
  • rehash 灵活:开链可逐 bucket 迁移;开放寻址必须重建整张表,无法渐进

取舍 2:渐进 rehash vs 一次性 rehash

代价是 rehash 期间双表共存,内存占用约 1.5-2 倍。Redis 选择此设计的核心原因是单线程模型:任何主线程阻塞都直接传导到所有客户端。BGSAVE/AOFRW 期间通过 dict_force_resize_ratio 推迟扩容,在 fork CoW 与 rehash 内存峰值之间二次权衡。

取舍 3:SipHash-2-4 vs MurmurHash

SipHash-2-4 单次哈希约 50-80 ns,比 MurmurHash 慢约 30%。但 Redis 4.0 后强制启用 SipHash:

  • key 来自用户输入,可被构造碰撞攻击(如 Hash-Flooding DoS)
  • 加密强度伪随机性确保无法在不知道 secret key 的情况下预测哈希分布
  • 对单线程主路径而言,30% 的哈希计算成本是可接受的安全代价

取舍 4:size 必为 2 的幂

hash & (size - 1) 等价于 hash % size 但避免除法运算。代价是扩容只能整表翻倍(无法精细调整为 1.5 倍等中间值),内存利用率最差时为 50%(刚扩容后)。该取舍偏向 CPU 性能而非内存利用率 —— 对单线程的 Redis 主线程,每次 GET 省下的几个时钟周期累积成可观的吞吐提升。

用例 1:keyspace 千万级在线游戏
某 MMORPG 服务器把每个玩家的登录态、背包、属性等都存入 Redis,db->dict 长期维持 1500 万 key。关键观察:用 DEBUG HTSTATS 0 看到 ht[0] size = 16M,负载因子 0.93,链长直方图 P99 = 4。陷阱踩到:某次发布后客户端写入暴增,触发扩容到 32M;同时 RDB 在跑,fork 子进程 CoW 把 RSS 推到 28 GB(平时 16 GB),触发 OOM。修复:maxmemory 24gb + maxmemory-policy allkeys-lru + 错峰 RDB(凌晨 3 点);加监控 used_memory_rss / used_memory > 1.5 告警。
用例 2:Hash 编码降级导致内存翻倍
某用户中心用 Hash 存用户 profile,初期单 Hash < 100 field 时是 listpack 编码内存紧凑;后来加了 200+ 字段(原始数据 + 缓存 + 标签),触发 hash-max-listpack-entries=128 阈值,自动转 hashtable 编码。结果:每个 user Hash 内存从 4 KB → 12 KB,1000 万用户 Hash 总内存从 40 GB → 120 GB。修复:把不常用字段拆到单独 Hash(user:meta:{uid});热数据保留在 listpack(< 128 field);用 OBJECT ENCODING user:1 在线观察编码。
用例 3:DEBUG HTSTATS 帮助定位 SipHash secret 泄漏
某团队的 Redis 收到大量构造性 key(aaaa...、aaab... 长尾),怀疑遭遇 Hash-Flooding 攻击。诊断:用 DEBUG HTSTATS 0 看链长分布,P99 = 32(正常 < 5),最长链 256;同时 SLOWLOG 显示 GET 某些 key 耗时 1 ms+(本应 < 100 μs)。修复:重启 Redis(SipHash secret 在启动时随机生成,重启会换 secret 让所有"构造"失效);加 tcp-keepalive 60 检测异常连接;前置 WAF 限制 key 长度。
用例 4:不要这么用 — 用 Redis dict 当主存储
反例:某创业公司 MVP 阶段把所有业务数据(用户、订单、商品)都直接 SET 到 Redis,没有 MySQL 兜底。结果:① 重启时间随 keyspace 增长,1000 万 key 加载 RDB 需要 20 分钟;② 主从切换时 partial sync 失败,full sync 复制几十 GB 流量;③ 没有事务/外键/二级索引,业务复杂度全压在应用层。正解:Redis 是"内存中的快"加速 MySQL/PostgreSQL 等持久化存储;关键业务数据走数据库,Redis 只缓存热点数据 + 排行榜 + 计数器。
