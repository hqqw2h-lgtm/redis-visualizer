第 16 章 · 概率结构 Bloom Filter / Count-Min Sketch / TopK

概率换内存 · 假阳无假阴 · m=-N·ln(p)/(ln 2)² · k=ln 2·m/N · Scalable 几何级数 · Cuckoo 原生删除 · CMS d×w 矩阵取 min · HeavyKeeper 衰减 · RedisBloom 模块化

第 11 章的 HyperLogLog 估算集合的基数("有多少不同元素");本章关注另一类概率结构,回答三类不同问题:某元素是否在集合中(Bloom Filter)、某元素出现多少次(Count-Min Sketch)、集合中频次最高的 K 个元素是谁(TopK)。三者共同特点是用受控的近似换取常数级内存与查询复杂度,让 Redis 主线无法精确实现的高吞吐查询成为可能。

这些数据结构由 RedisBloom 模块(开源 Redis Stack 组件)提供,通过 Redis Modules API 注册 BF.* / CF.* / CMS.* / TOPK.* 命令族。本章描述算法原理、字节级位图布局、参数选择、可扩展变体,并给出 RedisBloom 核心源码片段。

  • Bloom Filter 三组件:位图 + 多 hash + SET/CHECK 语义,假阴无假阳有
  • 假阳率公式 p = (1 - e^(-kn/m))^k 推导与最优 k = ln 2 · m/n
  • 1e8 元素 1% 假阳率仅需 114 MiB,相比 Set 的 4–6 GiB 节省 35–50×
  • Scalable Bloom 几何级数累积假阳率有上界,牺牲查询性能换容量灵活
  • Cuckoo Filter 原生支持删除,以 ~25% 内存代价换取 BF 不具备的 DEL 能力
  • Count-Min Sketch 的 d × w 矩阵与"取 min over rows"过估上界保证
  • HeavyKeeper 算法:衰减系数 b 平衡长期热点与短期热点
  • RedisBloom 模块加载 + BF.* / CMS.* / TOPK.* 命令族

前置知识:第 1 章 SDS(Bloom 位图最终落在 SDS 之外的独立 buffer);第 3 章 dict(Cuckoo Filter 的 hash 表替代);第 11 章 HyperLogLog(对比另一类概率结构);第 21 章模块系统(理解 RedisBloom 的注册流程与自定义 type)。读者需要熟悉 redis-cli 与基础概率论(伯努利试验、几何级数)。

16.1 概率结构与精确结构的边界

Redis 主线提供精确的 Set / Hash / HSET HINCRBY 等结构,在元素规模较小时表现极佳;但当数据量进入千万级 / 亿级,精确结构的内存代价变得不可接受。概率结构的核心思想是接受受控的小概率错误(假阳率 1%、估算偏差 0.1%),换取独立于元素规模的常数级内存。

16.2 Bloom Filter 算法原理

16.2.1 三个组件:位图 + 多 hash + 集合语义

图 16-2-1 · Bloom Filter 三个组件 — 位图 + k 个 hash + SET/CHECK

Bloom Filter 由三个独立组件构成:

  1. 位图 bit[0..m-1],初始全 0,m 是位图长度。
  2. k 个 hash 函数 h_1, h_2, ..., h_k,把任意元素映射到 [0, m) 范围内的位置。
  3. SET / CHECK 语义:BF.ADD 把 k 个位置置 1;BF.EXISTS 检查 k 个位置是否全部为 1。

16.2.2 BF.ADD 与 BF.EXISTS 语义示例

图 16-2-2 · BF.ADD 与 BF.EXISTS 在 16 位位图上的演化

16.2.3 为什么 Bloom 不能删除

从图 16-2-2 可以直观看出 Bloom 不能删除的原因:bit[7] 同时被 apple 与 banana 标记。如果 BF.DEL bf "apple" 把三个 bit 全部清零,会让 banana 的 BF.EXISTS 误判为不存在(产生假阴),违反 Bloom 的核心契约。要支持删除,必须把每个 bit 替换为计数器(Counting Bloom)或换用 Cuckoo Filter 的指纹+踢出机制。详见 §16.7。

16.3 假阳率公式与最优 k

16.3.1 假阳率推导

图 16-3-1 · 假阳率公式从伯努利试验出发
设位图长度 m, hash 函数数量 k, 已插入元素数 n
  1. Step 1: 单次 hash 把某 bit 置 1 的概率 → P(单次置 1) = 1 / m
  2. Step 2: 单次 hash 后该 bit 仍为 0 的概率 → P(单次仍 0) = 1 - 1/m
  3. Step 3: 插入 n 个元素 (共 k·n 次 hash) 后该 bit 仍为 0 → P(bit=0) = (1 - 1/m)k·n ≈ e-kn/m
  4. Step 4: 该 bit 被置为 1 的概率 → P(bit=1) ≈ 1 - e-kn/m
  5. Step 5: 查询未插入元素 y, k 个位置全为 1 的概率 (假阳率) → p_fp = [1 - e-kn/m]k
化简 (令 r = m/n, bits-per-element): p_fp = (1 - e-k/r)k
对 k 求导取最优: k* = (m/n) · ln(2) ≈ 0.693 · m/n
最优 k 下的假阳率: p_fp* = (1/2)k* ≈ 0.6185m/n

16.3.3 反向工程:已知 N 与 p 求 m 与 k

图 16-3-2 · 工程反向计算公式

公式 (已知 N, p 求 m, k)

m = -N · ln(p) / (ln 2)²
k = (m / N) · ln(2)

示例 N=1e8, p=1%

m = -1e8 × ln(0.01) / (ln 2)²
  = -1e8 × (-4.6052) / 0.4805
  = 1e8 × 9.585
  = 9.585e8 bit
  ≈ 1.198e8 byte ≈ 114 MiB

k = (9.585e8 / 1e8) × 0.6931
  = 6.643 → 取 7

验证: m/n = 9.585,
p_fp* ≈ 0.6185^9.585 ≈ 0.0098 ≈ 1% ✓

16.5.1 BF.RESERVE 命令参数

图 16-5-1 · BF.RESERVE 参数语义

# 参数:
#   error_rate   目标假阳率 (0..1), 例如 p=0.01 表示 1%

#

# 返回:
#   OK    成功
#   ERR   key 已存在

# 典型调用:
BF.RESERVE users:bloom 0.001 1000000
# → 0.1% 假阳率, 容量 100 万, 默认 SCALING + EXPANSION 2

16.5.2 hash 实现技巧:双 hash 生成 k 个

图 16-5-2 · Kirsch-Mitzenmacher 双 hash 技巧 — 2 次 hash 等价 k 次

Kirsch & Mitzenmacher 在 2006 年的论文证明:只要 h_1 与 h_2 独立,组合 h_1 + i × h_2(i = 0..k-1)与 k 个真正独立 hash 在假阳率上渐近等价。这把 k 次 hash 计算降为 2 次,显著降低 CPU 开销。

RedisBloom 用 MurmurHash64A 做两次 base hash(seed 与 seed + 1),k 次 BF.ADD 内部 hash 计算开销为 O(1) 级别(即"两次 hash + k 次乘加")。

16.6 Scalable Bloom Filter

16.6.1 层次结构

图 16-6-1 · Scalable Bloom 的几何级数链 — capacity ×2, p ÷2

16.6.2 总假阳率上界推导

图 16-6-2 · 几何级数累积假阳率有上界
设第 i 层假阳率为 p_i, 总假阳率 (任意层假阳即视为整体假阳):

Ptotal = 1 − ∏ (1 − pi) ≤ Σ pi (Boole 不等式)

RedisBloom 几何级数策略 (r=0.5)

p_i = p × 0.5 × 0.5^i

p_0 = p × 0.5
p_1 = p × 0.25
p_2 = p × 0.125
...
p_n = p × 0.5^(n+1)

总假阳率上界

P_total ≤ Σ_{i=0..∞} p × 0.5 × 0.5^i
        = p × 0.5 × 2
        = p

★ 无论多少层, 总假阳率不超过初始目标 p
★ 这是 Scalable Bloom 的核心保证
关键性质:
  • 无论扩展多少层, 总假阳率始终 ≤ 设定的 p
  • 代价:每层 filter 的单层假阳率指数减小, 后续层的 m 与 k 都比前一层大, 第 n 层比第 0 层多用约 (n+1) 倍 bits-per-element
  • EXISTS 查询代价:O(layers × k_avg) 而非 O(k), 最坏情况扫所有 n+1 层
  • 缓解:层数通常 < 10(每层容量翻倍, 210 = 1024 倍初始容量)

图 16-7-2 · Cuckoo Filter — 指纹 + 双桶 + 踢出重定位

Cuckoo Filter 的核心在于双 hash + 异或对称性:从任一桶出发,异或指纹的 hash 即可定位另一个候选桶。这让"被踢出的元素"也能被高效重定位,而不需要重新计算原始元素。

16.10 TopK:HeavyKeeper 算法

16.10.1 与精确排序的区别

精确 Top K(全排序)

  • 对所有元素按频次排序, 取前 K
  • 内存:O(N)
  • 复杂度:O(N log N) 或 O(N)(用 heap)
  • 适用:N 较小, 内存够用

HeavyKeeper · 概率 + 衰减

  • 固定内存近似 Top K
  • 内存:O(K + d × w)
  • 复杂度:O(d) per insert / query
  • 适用:实时热门词 / 热点 IP / 最频繁请求
  • 数据流极大(N ≫ 内存), 无法精确
  • 业务允许 ~1% 排名误差

#           典型值 0.9 (平衡) / 0.7 (短期偏重)

# 典型配置:
TOPK.RESERVE hot 100 10000 7 0.9
# → 追踪 Top 100、宽度 10K、深度 7、衰减 0.9
# → 内存 ≈ 100 + 10000 × 7 × (指针 16B + counter 4B)
#         ≈ 1.4 MB

# 调参经验:
#   K 大 / 热点不稳定 → decay 偏小 (0.7)
#   K 小 / 长期热点   → decay 偏大 (0.95)
#   width × depth ≥ 10 × K 是经验下界

16.14.3 bloom_init / bloom_add 简化源码

/* sb.c — bloom_init 简化 */
void bloom_init(struct bloom *bloom, uint64_t entries,
                double error, uint64_t bf_seed) {
    bloom->entries = entries;
    bloom->error   = error;
    bloom->seed    = bf_seed;

    double num   = log(error);
    double denom = 0.480453013918201;          /* (ln 2)^2 */
    bloom->bits   = (uint64_t)(-(entries * num / denom));
    bloom->hashes = (uint8_t)(0.693147180559945 * (bloom->bits / entries));
                                               /* 0.693 = ln 2 */

    bloom->bytes = (bloom->bits + 7) / 8;
    bloom->bf    = (uint8_t *)RedisModule_Calloc(bloom->bytes, 1);
}

/* sb.c — bloom_add 简化(基于 Kirsch-Mitzenmacher 双 hash) */
int bloom_add(struct bloom *bloom, const void *buf, size_t len) {
    uint64_t a = MurmurHash64A(buf, len, bloom->seed);
    uint64_t b = MurmurHash64A(buf, len, bloom->seed + 1);

    int hits = 0;
    for (uint8_t i = 0; i < bloom->hashes; i++) {
        uint64_t x    = (a + i * b) % bloom->bits;
        uint64_t byte = x >> 3;                /* x / 8 */
        uint8_t  mask = 1 << (x & 7);         /* 1 << (x % 8) */
        if (bloom->bf[byte] & mask) hits++;
        bloom->bf[byte] |= mask;
    }
    return (hits == bloom->hashes);            /* 1 = 已存在,0 = 新增 */
}

/* sb.c — bloom_check 简化 */
int bloom_check(struct bloom *bloom, const void *buf, size_t len) {
    uint64_t a = MurmurHash64A(buf, len, bloom->seed);
    uint64_t b = MurmurHash64A(buf, len, bloom->seed + 1);

    for (uint8_t i = 0; i < bloom->hashes; i++) {
        uint64_t x    = (a + i * b) % bloom->bits;
        uint64_t byte = x >> 3;
        uint8_t  mask = 1 << (x & 7);
        if (!(bloom->bf[byte] & mask)) return 0;   /* 任一位为 0 立即返回 */
    }
    return 1;                                  /* 全部位为 1 → 可能存在 */
}

16.14.4 CMS_IncrBy 简化源码

/* cms.c — CMS 单元素累加 (简化) */
size_t CMS_IncrBy(CMSketch *cms, const char *item, size_t itemlen,
                  uint32_t value) {
    size_t min_count = UINT32_MAX;
    for (size_t i = 0; i < cms->depth; i++) {
        uint32_t hash = MurmurHash2(item, itemlen, i);     /* 每行独立 seed */
        size_t   col  = hash % cms->width;
        cms->array[i * cms->width + col] += value;
        if (cms->array[i * cms->width + col] < min_count)
            min_count = cms->array[i * cms->width + col];
    }
    cms->counter += value;
    return min_count;     /* 累加后该元素的当前估算值 */
}

16.15 设计取舍

16.15.1 取舍 1:Bloom 不支持删除

标准 Bloom 1 bit / position,删除需 Counting Bloom (4 bit / position) 或 Cuckoo Filter (12 bit / element)。RedisBloom 保留标准 Bloom 节省内存,删除需求转给 CF.*。这种"专注单一职责"的设计避免在主数据结构上增加复杂度,业务侧按需要选择。

16.15.2 取舍 2:Scalable Bloom 默认启用

不指定 NONSCALING 时自动扩展,牺牲查询性能换取容量灵活性。每次扩展查询需扫所有层,层数越多越慢。但层数指数增长(每层容量翻倍),即使到 10 层也已覆盖 1024× 初始容量,实际场景层数通常 ≤ 6,查询代价可控。

16.15.3 取舍 3:CMS 选 min 而非 mean

均值会被 hash 碰撞污染(被高估的行会拉高均值),min 至少有一个未被碰撞的行的概率高。理论保证更强(Cormode & Muthukrishnan 论文证明 min 是 (ε, δ)-估计的最紧上界)。代价是 min 总是过估,不会过低,业务上"用 CMS 排相对名次"比"用 CMS 求精确数值"更可靠。

16.15.4 取舍 4:TOPK 用 HeavyKeeper

相比 SpaceSaving / FrequentItems / Lossy Counting,HeavyKeeper 在 zipfian 分布(长尾)下精度更好,因为衰减替换偏向"近期被频繁访问的元素",符合大多数互联网场景的访问模式。代价是衰减系数 decay 需要业务调参,默认 0.9 适合多数场景,但热搜实时性要求高的业务需要调到 0.7–0.8。

16.15.5 取舍 5:Module 而非 Core

RedisBloom 作为模块而非主线代码,保持 Redis Core 的精简(主线 Core 仅处理基础类型与脚本),概率结构归类为"业务高级特性"。代价是开源 Redis 需要额外编译 + 加载(loadmodule),Redis Stack/企业版直接捆绑可省略此步。
