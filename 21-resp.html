第 21 章 · RESP 协议

REdis Serialization Protocol · 首字节判型 · 长度前缀 · 二进制安全 · RESP2 → RESP3 · Pipeline · 增量解析状态机

Redis 客户端与服务端之间通过一条 TCP 连接交换字节流,字节流的语法称为 REdis Serialization Protocol(RESP)。该协议的设计目标是:首字节判型、长度前缀、二进制安全、解析路径无需回溯。

RESP2 自 Redis 1.2 起作为唯一线缆协议,定义五种基本类型;RESP3 自 Redis 6.0 起作为可选升级协议,在 RESP2 基础上扩充至十五种首字节类型,补齐 Map、Set、Boolean、Null、Big number、Double、Verbatim string、Blob Error、Push、Attribute 等语义。客户端通过 HELLO 3 命令在握手阶段切换至 RESP3。

实现 RESP 解析的核心约束是:TCP 是流协议,单次 read() 可能读到半条命令或多条命令。Redis 在 client.querybuf 上维护增量解析状态机,通过 multibulklen 与 bulklen 两个计数器跟踪进度,半包等待事件循环下一轮,粘包则在同一轮内顺序处理。这一机制同时是 Pipeline 的物理基础。

  • RESP 与 HTTP/JSON/Protobuf 候选协议的对比依据(首字节判型、无 schema 协商)
  • RESP2 五种基本类型:Simple String +、Error -、Integer :、Bulk String $、Array *
  • 每种类型的字节级布局(逐字节 ASCII + CRLF 终止符)
  • processInputBuffer 状态机:multibulklen、bulklen 双计数器跟踪进度
  • 半包/粘包的字节级演变与"数据不足即退回事件循环"的不变量
  • RESP3 十种新增/独立类型:Map %、Set ~、Bool #、Null _、Big (、Double ,、Verbatim =、Blob Error !、Push >、Attribute |
  • Pipeline 的本质:客户端连续写、服务端无专属路径、RTT 摊薄 N→1
  • Cluster smart client 的按槽分组、MOVED/ASK 重定向、CROSSSLOT 限制
  • client-query-buffer-limit、proto-max-bulk-len、client-output-buffer-limit 三类配置
  • 10 类典型陷阱:Inline 误用、长度不匹配、批次过大、Push 错位、跨槽 MOVED 等

前置知识:第 19-20 章 ae 事件循环 + I/O 多路复用(知道 readQueryFromClient 是事件回调);Redis 命令模型(SET/GET/HGETALL/SUBSCRIBE);Cluster 场景可先了解第 27 章的 16384 slot 与第 15 章 CRC16 路由。

21.1 协议选型

图 21-1-1 · 命令请求-应答的整体路径

请求路径与应答路径共享同一条 TCP 连接,方向相反。请求侧以 Multibulk 数组承载命令,应答侧的具体类型由命令决定:PING 返回 Simple String,GET 返回 Bulk String 或 Null,HGETALL 在 RESP2 下返回扁平 Array、在 RESP3 下返回 Map。

RESP 的设计取舍是"对 Redis 命令模型足够"而非"通用 RPC"。命令集合频繁演进、客户端实现需要轻量、命令多为短小高频请求,这三条约束共同排除了 HTTP/JSON/Protobuf。RESP 的首字节判型与长度前缀在 O(1) 时间内确定下一步动作,无需回溯也无需 schema 协商。

图 21-2-2 · Simple String 字节布局

Simple String 适用于固定的短状态回复,典型值 +OK、+PONG、+QUEUED。该类型禁止 payload 中出现 CR 或 LF,因此不能承载用户输入。命令实现层使用 addReplyStatus() 写入。

图 21-2-3 · Error 字节布局与前缀分类

Error 与 Simple String 同构,仅首字节不同。客户端通过空格切分首词得到错误类型,后续文本为人类可读描述。Cluster 模式下的 MOVED / ASK 错误同时承载路由信息(slot 号 + 目标地址),客户端按错误类型解析。

图 21-2-4 · Bulk String 字节布局(二进制安全)
Bulk String 是 RESP 二进制安全的承载类型。payload 长度由 $ 后的十进制数确定,而非由内容中的分隔符确定,因此 payload 可包含 CR、LF、\0 等任意字节。proto-max-bulk-len 默认 512 MiB,超过即被拒绝。
图 21-2-5 · Array 字节布局(GET foo)

Array 是 RESP 唯一的复合类型,元素可以是任意基本类型或嵌套 Array。命令请求一律使用 Array of Bulk String,服务端在 processMultibulkBuffer() 中按此假设解析。

21.2.1 Inline 命令(仅供调试)

Inline 命令仅用于人工调试(nc/telnet 直接交互)。生产客户端一律使用 Multibulk:Multibulk 通过长度前缀承载任意字节,而 Inline 通过空格切分参数,空格与 CRLF 不可作为 value 内容。

21.3 增量解析状态机

typedef struct client {
    sds querybuf;        /* 累积已 read 但未消费的字节 */
    size_t qb_pos;       /* querybuf 已消费偏移 */
    int reqtype;         /* PROTO_REQ_INLINE 或 PROTO_REQ_MULTIBULK */
    int multibulklen;    /* 当前 Array 还剩多少 Bulk 待读 */
    long bulklen;        /* 当前 Bulk 的 payload 长度,-1 表示未读到 $len */
    int argc;            /* 已解析完成的参数个数 */
    robj **argv;         /* 已解析完成的参数对象数组 */
    /* ... 输出缓冲、订阅集等 ... */
} client;

querybuf 是 read() 的目标缓冲,qb_pos 跟踪解析光标。multibulklen 与 bulklen 是状态机的两个核心计数器:前者记录"当前 Array 还差几个 Bulk",后者记录"当前 Bulk 的 payload 还差几字节"。

图 21-3-1 · 解析状态机
状态机的关键不变量:任一状态转移失败(数据不足)时,直接退回事件循环,不修改已解析的字段。processInputBuffer() 在外层 while 循环中反复尝试解析,直到 querybuf 中无完整命令或解析返回错误。
图 21-3-2 · 半包到达的字节级演变
半包不会触发误执行的根因:bulklen 与 multibulklen 在每次状态转移时严格按"已读字节数 == 期望字节数"判定,任一未达即退回。querybuf 的剩余字节在下一轮 read() 后被追加,解析光标 qb_pos 接续之前的位置继续推进。

Texec 为命令在主线程的执行时间,通常 1-100 微秒。RTT 在跨机部署中通常 0.1-2 毫秒,跨地域可达数十毫秒。Pipeline 的收益与 RTT/Texec 比例成正比。

服务端无 Pipeline 专属代码路径。processInputBuffer() 在每次 socket 可读事件中被调用一次,内部循环消费 querybuf 中所有完整命令。Pipeline 的本质是:客户端连续写入若干完整 RESP 帧,服务端在同一次 read() 后看到多条完整命令并顺序执行。
图 21-5-2 · Pipeline / MULTI / Lua 语义对比

Pipeline 与 MULTI/EXEC 不互斥:可将 MULTI ... EXEC 整体放入 Pipeline,在保留事务语义的同时摊薄网络往返。Lua 脚本提供原子性与服务端计算能力,但脚本执行期间阻塞主线程,需控制时长在毫秒级。

int processMultibulkBuffer(client *c) {
    char *newline = NULL;
    int ok;
    long long ll;

    if (c->multibulklen == 0) {
        newline = strchr(c->querybuf+c->qb_pos, '\r');
        if (newline == NULL) {
            if (sdslen(c->querybuf)-c->qb_pos > PROTO_INLINE_MAX_SIZE) {
                addReplyError(c, "Protocol error: too big mbulk count string");
                setProtocolError("too big mbulk count string", c);
            }
            return C_ERR;
        }

        ok = string2ll(c->querybuf+1+c->qb_pos,
                       newline-(c->querybuf+1+c->qb_pos), &ll);
        if (!ok || ll > 1024*1024) {
            addReplyError(c, "Protocol error: invalid multibulk length");
            return C_ERR;
        }

        c->qb_pos = (newline-c->querybuf)+2;
        if (ll <= 0) return C_OK;

        c->multibulklen = ll;
        c->argv = zmalloc(sizeof(robj*)*c->multibulklen);
    }

    while (c->multibulklen) {
        if (c->bulklen == -1) {
            newline = strchr(c->querybuf+c->qb_pos, '\r');
            if (newline == NULL) break;
            if (c->querybuf[c->qb_pos] != '$') {
                addReplyErrorFormat(c,
                    "Protocol error: expected '$', got '%c'",
                    c->querybuf[c->qb_pos]);
                return C_ERR;
            }
            ok = string2ll(c->querybuf+c->qb_pos+1,
                           newline-(c->querybuf+c->qb_pos+1), &ll);
            if (!ok || ll < 0 || ll > server.proto_max_bulk_len) {
                addReplyError(c, "Protocol error: invalid bulk length");
                return C_ERR;
            }
            c->qb_pos = newline-c->querybuf+2;
            c->bulklen = ll;
        }

        if (sdslen(c->querybuf)-c->qb_pos < (size_t)(c->bulklen+2)) break;

        c->argv[c->argc++] = createStringObject(
            c->querybuf+c->qb_pos, c->bulklen);
        c->qb_pos += c->bulklen+2;
        c->bulklen = -1;
        c->multibulklen--;
    }

    if (c->multibulklen == 0) return C_OK;
    return C_ERR;
}

21.10.1 设计取舍

取舍 1 · 文本前缀 vs 二进制 tag: 文本前缀(*N\r\n)的字节开销略高于二进制 varint,但带来三处收益:协议可以手写调试(通过 nc/telnet),抓包可读,无字节序问题。Redis 选择文本前缀,代价是单命令额外几字节,在 Bulk String 长度通常远大于头部的场景下可忽略。
取舍 2 · 无连接级状态 vs 连接级模式: RESP2 的 Pub/Sub 让连接进入"特殊模式",订阅期间禁止发送其他命令。该设计简化了服务端实现但增加了客户端复杂度。RESP3 用 Push 类型解耦带外消息与正常应答,同一连接可同时订阅与执行命令。代价是客户端读循环必须在每次读取前判断首字节是否为 >。
取舍 3 · Pipeline 不引入服务端语义: Pipeline 完全是客户端行为,服务端无任何专属代码。该设计的收益是协议简洁、向后兼容,代价是客户端无法获得"批量原子"语义。需要原子性时,Redis 提供 MULTI/EXEC 与 Lua 脚本作为正交机制。
