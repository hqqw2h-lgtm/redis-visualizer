第 10 章 · 对象编码体系

11 种 encoding · 5 类对象状态机 · 单向不可逆升级 · 5–10× 内存差距 · O(1) ↔ O(N) 复杂度切换 · 阈值参数 · OBJECT ENCODING 诊断 · AOF reload 批量转换陷阱

Redis 在 type(对象类型,如 String/Hash/Set)之上引入第二维度 encoding(底层编码,如 int/embstr/raw/listpack/hashtable)。同一对象类型在不同数据规模下会切换底层编码:小集合使用紧凑编码(listpack/intset/embstr)节省 5–10 倍内存,大集合切换到通用编码(hashtable/skiplist+dict/raw)保证 O(1)/O(log N) 复杂度。

编码切换由写入路径上的判定逻辑触发,对客户端透明。本章讨论五大对象类型的编码状态机、阈值参数、切换成本与诊断手段。核心约束:编码升级不可逆,集合类一旦从紧凑编码切换为通用编码,即使元素回落到阈值以内,也不会自动降级。

  • redisObject 中 encoding 占 4 bit,11 种活跃编码常量,5 类对象的状态机
  • String 三态:int / embstr(≤ 44 B) / raw;阈值由 jemalloc 64 B size class 反推
  • Hash / Set / ZSet 在 7.x 用 listpack 替代 ziplist;每次写入检查 entries + 长度
  • 升级单向不可逆的动机:避免阈值附近反复 O(N) 转换
  • 9 个核心阈值参数 + 内存差距(1 万 Hash entry:250 KiB 紧凑 vs 1.6 MiB hashtable)
  • 陷阱:intset 混入字符串、AOF reload 批量升级、误以为 hashtable 会自动降级

前置知识:第 1 章 SDS 字符串结构(理解 sdshdr8 与 jemalloc size class);第 3 章 dict(dictEntry 24 B 与渐进 rehash);第 6 章 ziplist/listpack(连续 entry 序列布局);第 9 章 redisObject(16 B 总开销与 4 bit type/encoding 字段)。读者需要熟悉 redis-cli 与 OBJECT 调试命令。

10.1 redisObject 与 encoding 字段

每个 Redis 键对应一个 redisObject。对象类型(type)与底层编码(encoding)分别占 4 bit,各自独立演化。type 是对外语义,encoding 是底层实现——这种双层抽象让 Redis 在不动客户端的前提下,按数据规模在多种内部表示之间切换,达到内存与延迟的平衡。

10.1.1 redisObject 字段位宽

图 10-1-1 · redisObject 16 B 字段结构

redisObject 在 64 位系统下严格 16 字节。type 与 encoding 各占 4 比特(共 1 B);lru 占 24 比特(LFU 模式下高 16 位为访问时戳,低 8 位为计数);refcount 32 比特,值为 OBJ_SHARED_REFCOUNT(INT_MAX)时表示共享对象;ptr 在 int 编码下直接存储 long,其余编码下指向底层结构。

10.1.2 type 与 encoding 的命令层观察

图 10-1-2 · TYPE / OBJECT ENCODING 双字段返回路径

10.3.7 tryObjectEncoding 源码片段

/* src/object.c — String 编码选择主入口 (简化) */
robj *tryObjectEncoding(robj *o) {
    long value;
    sds s = o->ptr;
    size_t len;

    /* 已经是 INT/EMBSTR 不再处理 */
    if (!sdsEncodedObject(o)) return o;
    if (o->refcount > 1) return o;  /* 共享对象不动 */

    len = sdslen(s);

    /* 路径 1: 尝试转为 long 整数 (int 编码) */
    if (len <= 20 && string2l(s, len, &value)) {
        if ((server.maxmemory == 0 ||
             !(server.maxmemory_policy & MAXMEMORY_FLAG_NO_SHARED_INTEGERS)) &&
            value >= 0 && value < OBJ_SHARED_INTEGERS)
        {
            decrRefCount(o);
            incrRefCount(shared.integers[value]);
            return shared.integers[value];   /* 命中共享池 */
        }
        sdsfree(o->ptr);
        o->encoding = OBJ_ENCODING_INT;
        o->ptr = (void *)value;              /* 整数直存 ptr */
        return o;
    }

    /* 路径 2: 短字符串走 embstr */
    if (len <= OBJ_ENCODING_EMBSTR_SIZE_LIMIT) {  /* 44 */
        robj *emb = createEmbeddedStringObject(s, sdslen(s));
        decrRefCount(o);
        return emb;
    }

    /* 路径 3: 长字符串保持 raw, 顺便回收冗余空间 */
    trimStringObjectIfNeeded(o);
    return o;
}

10.4 List 编码:quicklist 唯一

10.4.4 list-compress-depth 压缩策略

图 10-4-3 · list-compress-depth=2 的 LZF 压缩范围
list-compress-depth = 2 时的节点压缩状态:

  head ── N1 ── N2 ── N3 ── N4 ── N5 ── N6 ── N7 ── tail
          │     │     │     │     │     │     │

压缩率参考(LZF 算法):
  纯文本(JSON 重复字段):    50% – 70%(节省 30%–50%)

  短英文消息:                30% – 50%

LZF 压缩在 listpack 节点级别工作,适用场景:大 List(消息队列、操作日志)且访问模式集中在首尾。list-compress-depth = 0(默认)关闭压缩,适合随机访问场景。

/* src/t_hash.c — Hash 编码升级判定 (简化) */
void hashTypeTryConversion(robj *o, robj **argv, int start, int end) {
    if (o->encoding != OBJ_ENCODING_LISTPACK) return;

    size_t sum = 0;
    for (int i = start; i <= end; i++) {
        if (sdsEncodedObject(argv[i])) {
            size_t len = sdslen(argv[i]->ptr);
            /* 单 field/value 长度超阈值 → 立即升级 */
            if (len > server.hash_max_listpack_value) {
                hashTypeConvert(o, OBJ_ENCODING_HT);
                return;
            }
            sum += len;
        }
    }

    /* listpack 整体不足以容纳新内容 → 升级 */
    if (lpSafeToAdd(o->ptr, sum) == 0)
        hashTypeConvert(o, OBJ_ENCODING_HT);

    /* field 总数超阈值 → 升级 */
    if (hashTypeLength(o) > server.hash_max_listpack_entries)
        hashTypeConvert(o, OBJ_ENCODING_HT);
}

10.8 Stream 编码:rax + listpack

Stream 是 Redis 5.0 引入的日志型数据类型,只有一种对象编码 OBJ_ENCODING_STREAM。底层是 rax(基数树)管理 entry ID(毫秒-序号)→ listpack 节点的映射,每个 listpack 节点存储同一时间窗口的若干 entry。其设计与 quicklist 类似(连续 listpack + 索引结构),但索引方式从双向链表换成 rax,以支持按 ID 范围查询。

     HSET → field_count = 130 → 升级 listpack→hashtable (O(N) 一次)
     HDEL × 5 → field_count = 125 → 保持 hashtable
     HSET → field_count = 130 → 保持 hashtable(无 O(N))

   设计权衡:

     - 偶尔反弹到阈值以下时降级,会被随后的写入再次升级

     - 降级路径需要 O(N) 重建 listpack,主线程阻塞

/* src/t_zset.c — ZSet 编码升级 (简化) */
void zsetConvert(robj *zobj, int encoding) {
    if (zobj->encoding == encoding) return;
    if (zobj->encoding != OBJ_ENCODING_LISTPACK) return;     /* skiplist→listpack 路径不存在 */
    serverAssert(encoding == OBJ_ENCODING_SKIPLIST);

    unsigned char *zl = zobj->ptr;
    unsigned char *eptr = lpSeek(zl, 0), *sptr;
    unsigned char *vstr;
    unsigned int vlen;
    long long vlong;
    double score;
    sds ele;

    zset *zs = zmalloc(sizeof(*zs));
    zs->dict = dictCreate(&zsetDictType);
    zs->zsl = zslCreate();

    while (eptr != NULL) {
        sptr  = lpNext(zl, eptr);
        vstr  = lpGetValue(eptr, &vlen, &vlong);
        score = zzlGetScore(sptr);
        ele   = (vstr == NULL) ? sdsfromlonglong(vlong)
                               : sdsnewlen((char *)vstr, vlen);

        /* 双结构同步插入:skiplist 排序 + dict 等值索引 */
        zskiplistNode *node = zslInsert(zs->zsl, score, ele);
        serverAssert(dictAdd(zs->dict, ele, &node->score) == DICT_OK);
        eptr = lpNext(zl, sptr);
    }
    zfree(zobj->ptr);
    zobj->ptr      = zs;
    zobj->encoding = OBJ_ENCODING_SKIPLIST;
}

性能交叉点:listpack 在 N ≤ 128 时由于 cache locality 优势(连续内存 + 无指针追踪),实际比 hashtable 更快。这是为何默认阈值就是 128——超过该值后 O(N) 扫描的开销才显著超过 hashtable 的指针追踪 + 对齐浪费。强制把阈值上调到 5000 会让 HGET 退化到 42 μs,P99 可达 100 μs+,完全失去内存收益的意义。

10.15 真实场景调优

10.15.1 场景 1:小型缓存(用户画像 / 会话)

场景:用户会话缓存,平均 30 个 field/会话,峰值 80 field

每个 field 都是短字符串(8–32 B),总会话数 5000 万。

调优方案:

  • CONFIG SET hash-max-listpack-entries 256(默认 128 不够,提高到 256 让 95% 会话留在 listpack)
  • CONFIG SET hash-max-listpack-value 128(默认 64 不够,提高到 128 容纳长 user_agent 字段)
  • 压测验证:HGET P99 在 N=80 时 1.5 μs(可接受),N=200 时 5 μs(不可接受,故 256 是安全上限)

收益:5000 万会话 × 80 entry 全部 listpack,内存从预估 28 GB 降到 4.5 GB(6.2× 节省),P99 仍 < 2 μs。

10.15.2 场景 2:大型频繁写(订单流水)

场景:订单事件流,每个订单 hash 含 200+ field,持续追加

field 数量增长无上界(状态机 + 操作日志混存),写入 QPS 5K。

调优方案:

  • CONFIG SET hash-max-listpack-entries 64(下调,让订单 hash 早点切到 hashtable,避免 P99 抖动)
  • CONFIG SET hash-max-listpack-value 32(下调,大 value 直接走 hashtable)
  • 新订单从第 65 个 field 开始就是 hashtable,O(1) 写入,P99 平稳在 0.9 μs

对比未调优:默认 128 阈值下,订单 hash 在 100–128 entry 区间反复 HSET,每次都是 O(N) listpack 扫描,P99 飙到 8 μs。下调阈值反而降低延迟,代价是单订单内存上升 30%(可接受,因订单数量级不大)。

10.15.3 场景 3:整数 ID Set(关注列表)

场景:用户关注列表,纯整数 user_id,平均 200 关注,长尾 50 000+

读多写少:每次刷 feed 都要遍历关注列表,日均 1 亿次 SMEMBERS。

调优方案:

  • CONFIG SET set-max-intset-entries 50000(默认 512 太低,大部分用户被推到 hashtable)
  • 提高到 50 000 后,90% 用户保持 intset:200 个 user_id 仅 1.6 KiB(intset),vs hashtable 16 KiB
  • 50 000 关注的大 V 用户:intset 占 400 KiB,hashtable 占 4 MiB,intset 仍优
  • 关键:intset 二分查找 O(log N) + 连续内存,SMEMBERS 性能反而比 hashtable 好(SMEMBERS 是顺序遍历,intset 完胜)

收益:5000 万用户 × 平均 200 关注,内存从 750 GB 降到 75 GB(10× 节省),SMEMBERS 延迟下降 40%。

10.17.3 AOF reload 批量转换详解

图 10-17-2 · AOF 重放触发的 N 次升级
AOF 历史命令序列 (节选 order:1001):
  HSET ... f1 v1 ... f128 v128    ← 此时 listpack 刚满
  HSET ... f129 v129               ← 触发 listpack → hashtable 升级 (O(128))

启动时按顺序重放:
  ① 前 128 次 HSET 在 listpack 上累积
  ② 第 129 次 HSET 触发 O(128) 转换
  ③ 后续 HSET 走 hashtable, O(1)

重放 1 万个类似 hash 的 AOF:
  10 000 × O(128) = 1.28 M 次 listpack 元素拷贝, 单核 ~ 数秒额外开销
  启动时间从 30 s → 35 s

缓解方案:
  ① 优先 RDB 恢复(序列化按当前编码直存,无重放)
  ② aof-use-rdb-preamble=yes(默认),AOF 头部是 RDB
  ③ 定期 BGREWRITEAOF, 让 AOF 本身是当前快照
转换瞬时阻塞:大 hash(N = 10 万)从 listpack 转 hashtable 时,主线程阻塞约 5 ms(实测,8 B field/value)。期间所有客户端命令排队。生产环境的 P99 监控会捕捉到这种 spike,典型现象是"原本 0.5 ms 的 P99 突然 5 ms",溯源时若发现是单 key 升级,需要在业务侧规划 key 拆分。
