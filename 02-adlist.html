第 2 章 · adlist 双端链表

Redis 内部通用链表 · 24 字节节点 · 双向 prev/next · 三回调泛型 · 安全遍历 · O(1) 已知节点删除

Redis 内部需要管理多种"对象集合":客户端连接列表(server.clients)、慢查询日志(server.slowlog)、监视器集合(server.monitors)、Pub/Sub 订阅关系等。这些集合的访问模式具有共同特征 — 头尾高频插删、顺序遍历、按已知节点删除、不需要按下标随机访问。

C 标准库不提供链表实现,引入 GLib 或 std::list 会增加外部依赖与 ABI 风险。adlist 是 Redis 自实现的通用双端链表,以 50 行核心代码 + 三个回调函数(dup / free / match) 实现"泛型 + 所有权可控"。Redis 3.2 之前用户 LIST 数据结构曾以 adlist 为底层,3.2 之后改为 quicklist。当前 adlist 仅用于 Redis 内部对象集合管理。

理解 adlist 的关键是看清三个层次:容器 list(48 字节,持有 head/tail/len + 三回调)、节点 listNode(24 字节,prev/next/value)、迭代器 listIter(16 字节,预存后继节点)。这三个结构的协同设计让 adlist 同时具备"O(1) 头尾插删、O(1) 已知节点删除、O(n) 顺序扫描、遍历中安全删除当前节点"四个核心能力。

  • adlist 在 Redis 内部 5 类集合(clients / slowlog / monitors / ready_keys / pubsub patterns)中的角色与"为什么不是用户 LIST 的底层"
  • list / listNode / listIter 三结构的字节级布局(48 / 24 / 16 字节)与三个回调函数的所有权语义
  • 头插 / 尾插 / 中间插入 / 删除 / 查询 / 复制 6 类 CRUD 操作的逐帧字节级动画(每个操作 5 帧)
  • listIter 的"预存后继"设计:为什么遍历中可以安全删除当前节点,但不能删后继
  • 三回调(dup/free/match)如何实现"泛型 + 所有权可控",忘了设回调的常见泄漏陷阱
  • 与 C 数组 / 单链表 / std::list / glib GList / boost intrusive_list 的五维矩阵对比
  • 关键源码:listDelNode / listNext 内联拆解与 7 个常见陷阱
  • "adlist + dict"组合(如 server.pubsub_channels):用 dict 解决 O(n) 查找,用 adlist 维护订阅顺序

前置知识:C 指针操作与结构体内存布局;malloc / free 的所有权语义;函数指针的回调模式;void * 泛型在 C 中的常用做法。建议先读完第 1 章 SDS,熟悉 Redis 的 zmalloc / zfree 内存分配封装。

2.1 解决什么问题

Redis server 内部维护多种"对象集合",其访问模式与用户数据结构(LIST/HASH 等)显著不同。这一节先拆解这些集合的共同特征,再分析为什么 C 标准库与 C++ STL 都不直接可用。

共同特征:
① 主要操作是头尾插删与顺序遍历,不需要按下标随机访问;
② 元素是较大结构体的指针(几 KB 起),节点本身的 24 字节开销不主导成本;
③ 调用方持有节点指针时需要 O(1) 删除,而非"按值搜索";
④ 元素生命周期与业务对象耦合(client 断开 → 释放 client 结构),需要"删除节点时联动释放 value"的能力。

常见误解:阅读 adlist 源码时,容易把它当作 Redis 用户 LIST 数据结构(LPUSH / RPUSH)的底层。事实上 Redis 3.2+ 用户 LIST 已改用 quicklist(quicklist 节点内嵌 listpack,详见第 7 章)。当前 adlist 仅用于"Redis 内部通用链表库",而非用户数据类型。OBJECT ENCODING mylist 永远返回 quicklist,验证这一事实。

2.2.5 C 结构定义

/* src/adlist.h */
typedef struct listNode {
    struct listNode *prev;
    struct listNode *next;
    void *value;
} listNode;

typedef struct listIter {
    listNode *next;
    int direction;
} listIter;

typedef struct list {
    listNode *head;
    listNode *tail;
    void *(*dup)(void *ptr);
    void (*free)(void *ptr);
    int (*match)(void *ptr, void *key);
    unsigned long len;
} list;
三回调的所有权语义:adlist 不知道 value 指向的对象有多大、是否需要释放、如何比较 — 这些信息由调用方通过三个函数指针注入。dup 用于 listDup(NULL = 浅拷贝),free 用于 listDelNode / listRelease(NULL = 不释放 value),match 用于 listSearchKey(NULL = 按指针比较)。这是 C 风格"用一份代码服务多种数据类型"的经典做法。

读图要点

  • (e) 收尾 length++:x 完成接入,链表长度从 2 → 3。

server.pubsub_channels 即采用 dict + adlist 组合:dict 按 channel 名 O(1) 查询订阅列表,adlist 在该 channel 上维护订阅顺序。

2.8.1 listIter 的"预存后继"设计

预存后继的关键:listNext 在返回当前节点之前,先把 iter.next 推进到当前节点的后继。这意味着调用方可以安全删除"刚拿到的当前节点",但不能删除"后继节点"或更远的节点(那些指针仍由当前迭代器持有引用)。

2.8.2 遍历删除场景 5 帧演示

listIter 的关键设计:进入循环体前已经把 iter.next 推进到下一节点,所以即使本次循环 listDelNode(current) 把 current 释放,iter.next 仍然有效。下面 5 帧演示删除 A、保留 B、删除 C、保留 D 的过程:

图 2-8-2 · 安全遍历删除 (L = [A,B,C,D] → 删 A,留 B,删 C,留 D)

读图要点

  • (a) 初始:iter.next 指向 L.head = A,表示"即将处理"。
  • (b) 预存 B:进入循环体前,iter.next = A.next = B(青色"预存"),这是 listIter 的核心 trick。
  • (c) 删 A:listDelNode(L, A) 把 A 释放,但 iter.next → B 仍然有效;L.head 同时被修正为 B(绿色)。
  • (d) 保留 B,删 C:取 B(奇数索引保留),预存 iter.next = C,然后取 C(偶数索引),删除 C。B → D 直连(绿色)。
  • (e) 保留 D:取 D,iter.next = NULL,循环退出。最终 L = (B, D)。
  • 不安全的反例:在循环里 listDelNode(L, ln->next) 会把 iter.next 指向的节点提前 free,下一轮 cur = iter.next 即悬挂引用(见下方反例对照代码)。

2.8.3 遍历过程中三种删除模式

表 2-8-1 · 遍历删除三种模式安全性
删除目标安全性原因
当前节点(刚返回的)安全iter.next 已预存后继,不依赖 cur
后继节点不安全iter.next 指向该节点,删除后 iter.next 成为悬挂指针

2.8.4 反例 / 正例对照

正确:删除"刚返回的当前节点"

listIter li;
listNode *ln;

listRewind(l, &li);
while ((ln = listNext(&li)) != NULL) {
    if (should_remove(ln->value)) {
        listDelNode(l, ln);
    }
}
/* 安全:listNext 已经把 iter.next 推进到 ln 的下一个,
 * 即使 ln 被 zfree 掉,下一轮 cur = iter.next 仍是合法节点 */

错误:删除"后继节点"导致 iter.next 悬挂

listIter li;

listRewind(l, &li);
while ((ln = listNext(&li)) != NULL) {
    if (ln->next && should_remove(ln->next->value)) {
        listDelNode(l, ln->next);  /* ★ 悬挂!*/
    }
}
/* 不安全:iter.next 已经预存为 ln.next,
 * 而 listDelNode 把它 zfree 了,

listDup 涉及两个独立的链表(src 与 dst),用上下两个 cluster 分别框起来。源 src 全程不变,dst 从空容器开始按尾插次序生长:

图 2-9-2 · listDup 深拷贝 4 帧 (src=[A,B] → dst=[A',B'])

读图要点

  • (a) 初始:dst = listCreate() 后是空容器(head=tail=NULL, len=0),src 是 [A, B]。两个 cluster 互不相连,dst 的回调 dup/free/match 已从 src 复制(实现见 §2.12 源码片段)。
  • (b) 复制 A:iter 在 src 的 A 位置(红色),A' = dup(A.value) 后通过 listAddNodeTail(dst, A') 接到 dst(绿色新增)。
  • (c) 复制 B:iter 推进到 src 的 B,新建 B' 后再次尾插。dst = (A', B') 完成。
  • (d) 退出:iter = NULL,循环结束。返回 dst,src 与 dst 是两个独立的 list 容器,互不影响。
  • dup 回调 = NULL 的浅拷贝路径:此时 A'.value == A.value(指针相同),src 与 dst 共享 value 内存;若用户后续 listRelease(src) 触发 free(A.value),dst 会读到悬挂指针(见 §2.11 陷阱)。

2.10.3 设计动机总结

Redis 选择 adlist 的 4 个理由:
① 纯 C 项目:不能用 C++ std::list / boost::intrusive_list,引入 C++ ABI 与运行时风险过高;
② 自管理依赖:不愿引入 glib GList 等外部库,降低发布复杂度与版本兼容压力;
③ O(1) 已知节点删除:Redis 内部集合的删除场景(client 断开、订阅退订、慢日志淘汰)几乎全部基于"调用方已持有节点指针"的前提,8 字节 prev 指针的开销在节点 24 字节中占 33%,但相对于 client 结构本身的几 KB 完全可忽略;
④ 三回调约定泛型所有权:50 行核心代码服务多种内部用途,代价是"忘记设回调导致泄漏"的运行时陷阱(详见 §2.11)。

Redis 是纯 C 实现 + 自管理依赖 + 高度关注内存与 ABI 稳定性的项目。adlist 是该约束下的最小可行设计:50 行核心代码、24 字节节点、三个回调函数实现"泛型 + 所有权可控"。其工程权衡为:用每节点 8 字节的 prev 指针换取已知节点的 O(1) 删除 — 而 Redis 内部集合的删除场景(client 断开、订阅退订、慢日志淘汰)几乎全部基于"调用方已持有节点指针"的前提。

2.11.3 反例 / 正例对照(陷阱 1:忘 free 回调)

未设 free 回调,value 泄漏

list *l = listCreate();
listAddNodeTail(l, sdsnew("client:1"));
listAddNodeTail(l, sdsnew("client:2"));
listDelNode(l, listFirst(l));
/* ★ 问题:l->free == NULL,
 * listDelNode 不会释放 sdsnew 分配的字符串,
 * "client:1" 永久泄漏 */

显式设 sdsfree 回调

list *l = listCreate();
listSetFreeMethod(l, (void (*)(void *))sdsfree);
listAddNodeTail(l, sdsnew("client:1"));
listAddNodeTail(l, sdsnew("client:2"));
listDelNode(l, listFirst(l));
/* ★ 正确:listDelNode 内 list->free(value)
 * 调用 sdsfree("client:1"),无泄漏 */
常见误用警告:Redis 源码里多处出现 listSetFreeMethod(list, decrRefCount) 等"挂回调"的写法。新建 list 后忘了挂回调是 PR 中最常见的内存泄漏点。Code Review 时应当统一检查"listCreate 之后是否立即 listSet*Method"。

2.12.3 关键源码片段

listDelNode — 4 个 if 分支处理 4 种边界

/* src/adlist.c */
void listDelNode(list *list, listNode *node) {
    if (node->prev) node->prev->next = node->next;
    else list->head = node->next;
    if (node->next) node->next->prev = node->prev;
    else list->tail = node->prev;
    if (list->free) list->free(node->value);
    zfree(node);
    list->len--;
}

listNext — 预存后继的迭代器核心

/* src/adlist.c */
listNode *listNext(listIter *iter) {
    listNode *current = iter->next;
    if (current != NULL) {
        if (iter->direction == AL_START_HEAD)
            iter->next = current->next;
        else
            iter->next = current->prev;
    }
    return current;
}
listNext 的关键 5 行:current = iter->next 拿到本次要返回的节点;iter->next = current->next 立刻把迭代器指针推进到后继 — 这是"安全删除当前节点"的根基。即使调用方拿到 current 后 listDelNode(l, current),迭代器的 next 字段已不再依赖 current 的内存。

2.12.4 设计取舍

取舍 1:void * 泛型 vs 类型专用链表

一份 adlist 代码服务于 client 列表、slowlog 队列、monitor 集合等多种内部用途。代价是所有权由 dup / free / match 三个回调约定。该选择减少了重复代码,但增加了"忘记设置回调导致泄漏"的运行时陷阱。

取舍 2:双向链表 vs 单向链表

每节点多用 8 字节(prev 指针),换取"已知节点的 O(1) 删除"。Redis 内部场景中 client 断开、订阅退订等操作几乎全是"已持有节点指针"的删除,该开销在节点 24 字节中占 33% — 但相对于 client 结构本身的几 KB 完全可忽略。

取舍 3:容器持有 head/tail/len vs 纯节点链

list 结构 48 字节多承担一个对象,但获得 O(1) 头尾访问与 O(1) 长度查询。在 connected_clients 大于 100k 的场景下,这两个 O(1) 查询是热路径。

取舍 4:不设置索引 vs 内嵌索引

adlist 不维护任何辅助索引,listSearchKey 与 listIndex 均为 O(n)。Redis 在需要快速查找的场景(如 pubsub_channels)采用 "dict + adlist" 组合而非给 adlist 增加索引 — 这避免了 adlist 自身复杂化,把权衡推给上层调用方。
