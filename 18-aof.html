第 18 章 · AOF 持久化

命令日志 · 三阶段管道 · always/everysec/no fsync · BGREWRITEAOF · Multi-Part AOF (BASE + INCR + manifest) · 混合持久化 · CoW 内存放大

RDB 通过周期性快照获取 keyspace 一致副本,但两次快照之间的写入仅存在于内存。AOF (Append Only File) 采用补偿性方案:每条成功执行的写命令以 RESP 协议格式追加到日志文件,重启时通过回放日志重建 keyspace。

AOF 的关键约束有三:其一,主线程命令执行路径上的额外开销要足够低;其二,fsync 策略必须可配置,以适配不同业务的丢失容忍度;其三,日志体积必须可控,避免无限膨胀消耗磁盘。Redis 通过分层管道(执行 → aof_buf → write → fsync)、三种 fsync 策略(always / everysec / no)、子进程重写(BGREWRITEAOF)三项机制同时满足三类约束。

Redis 7.0 进一步将单文件 AOF 拆为 BASE + INCR + MANIFEST 的多部分结构 (Multi-Part AOF),原子切换从单文件 rename 改为清单更新,简化重写期间的状态管理。

  • RDB 与 AOF 在丢失窗口、恢复速度、文件体积三个维度的取舍
  • "后写日志"语义为何与传统 WAL 方向相反,以及 aof_last_write_status 的错误保护机制
  • 三阶段管道(feedAppendOnlyFile → flushAppendOnlyFile → fsync)各自的物理含义与延迟量级
  • RESP 协议的 AOF 编码格式与典型 2-3 倍体积膨胀比
  • appendfsync always / everysec / no 三种策略的吞吐量基线(8k vs 100k vs 120k ops/s)
  • aof_delayed_fsync 指标作为 everysec 退化为主线程阻塞的早期信号
  • BGREWRITEAOF 子进程遍历 keyspace 生成最小等价命令的工作原理
  • Redis 7 Multi-Part AOF 通过 manifest 原子切换的实现机制
  • 混合持久化(aof-use-rdb-preamble yes)将 10M keys 恢复时间从 25 秒降至 5 秒
  • 主节点不开 AOF 时,必须明确由副本、RDB 或上游日志承担恢复窗口的原因
  • 重写期间 CoW 复制 + aof_rewrite_buf 内存峰值的估算公式

18.0 章首速览

本章按"持久化语义边界 → 写入管道 → 重写机制 → 文件结构 → 部署适配 → 观测与陷阱"六层组织。前两节定义 AOF 的命令日志语义与刷盘策略,中间两节解析重写如何控制日志体积,后两节聚焦生产观测信号与边界条件。

核心定位: AOF 是 RDB 的补偿机制,不是替代。RDB 解决"快速恢复 + 紧凑存储",AOF 解决"细粒度数据安全 + 约 1 秒级的丢失窗口"。当开启 AOF 时,Redis 5.0+ 默认使用的混合持久化把两者捏在一起:BASE 用 RDB 格式,INCR 用 AOF 格式,一份文件兼顾两边的优点。

18.1 AOF 与 RDB 的语义边界

18.1.1 RDB 与 AOF 的丢失窗口

图 18-1-1 · 崩溃恢复时间线 — RDB 与 AOF 的丢失窗口对比

RDB 的盲区是"两次快照之间的写入"。在默认 save 3600 1 / save 300 100 / save 60 10000 配置下,业务低峰期触发频率约 5-15 分钟一次,即崩溃可能丢失 5-15 分钟数据。该窗口对缓存场景可接受,对计数、库存、会话等场景不可接受。

AOF 通过"每条成功执行的写命令立即追加到日志"消除该盲区。最坏丢失窗口由 appendfsync 策略决定:always 时为单条命令未落盘,everysec 时为约 1 秒,no 时由内核脏页回写策略决定 (Linux 默认 dirty_expire_centisecs=3000,即 30 秒)。

18.1.2 AOF 的"后写日志"语义

图 18-1-2 · 命令执行与 AOF 追加的时序

Redis AOF 是后写日志(先 apply 后 log),与传统数据库的 WAL (Write-Ahead Log,先 log 后 apply) 方向相反。后写日志的工程价值在于:仅记录已成功执行的命令,避免将语法错误、类型错误、参数越界等失败命令写入恢复流。

错误状态保护: 代价是 apply 与 log 之间存在窗口——命令已修改内存但 aof_buf 写入失败时,需通过错误状态保护机制阻止后续写入,以避免内存与日志的不一致扩散。该机制由 aof_last_write_status 字段记录,write 失败后 Redis 拒绝所有写命令直至 AOF 恢复正常。

18.2 AOF 写入三阶段管道

18.2.1 命令到磁盘的三阶段路径

图 18-2-1 · 三阶段管道 — 命令到磁盘的完整路径

三阶段的物理含义不同:

  • 阶段 1 是用户态内存追加,纯内存拷贝,纳秒级;
  • 阶段 2 是用户态到内核态的数据传递(write() 系统调用),数据进入内核 page cache,微秒级;
  • 阶段 3 是内核到磁盘控制器的同步刷盘(fsync()),毫秒级。

aof_buf 是 SDS 类型的内存缓冲区,容量由 server.aof_buf 字段管理。每次事件循环的 beforeSleep() 阶段会调用 flushAppendOnlyFile(),将 aof_buf 内容通过 write() 写入 page cache 后清空。fsync 时机由策略决定。

18.2.2 aof_buf 与命令的关系

RESP 是文本协议,大多数命令可直接通过 cat 查看(value 含二进制字节时不可读)。该选择带来三项优势:体积虽大于二进制快照但可读、加载逻辑与命令处理路径同构、不受内部数据结构演进影响。

18.2.3 RESP 命令编码示例

SET name redis 的 RESP 编码字节流:

字段十六进制字节含义长度
*3\r\n0x2A 0x33 0x0D 0x0AArray, len=34 B
$3\r\n0x24 0x33 0x0D 0x0ABulk string, len=34 B
SET\r\n0x53 0x45 0x54 0x0D 0x0A命令名5 B
$4\r\n0x24 0x34 0x0D 0x0ABulk string, len=44 B
name\r\n0x6E 0x61 0x6D 0x65 0x0D 0x0A参数 16 B
$5\r\n0x24 0x35 0x0D 0x0ABulk string, len=54 B
redis\r\n0x72 0x65 0x64 0x69 0x73 0x0D 0x0A参数 27 B
合计34 B
体积膨胀比 ≈ 2-3 倍: 原始命令 SET name redis 仅 13 字节, RESP 编码 34 字节。这是文本协议为"可读 + 与命令处理路径同构"付出的代价。膨胀通过周期性 BGREWRITEAOF 控制 (§18.4)。

RESP 编码的体积膨胀比典型为 2-3 倍。该膨胀通过周期性 BGREWRITEAOF 控制(见 §18.4)。

18.3 三种 fsync 策略

18.3.1 appendfsync 策略对比

表 18-3-1 · 三种 fsync 策略全维度对比
策略fsync 频率执行者最大丢失主线程影响吞吐上限 (NVMe SSD)
always每次 flush主线程已 fsync 的命令通常不丢阻塞 ~0.1-1 ms~10,000 ops/s
everysec每秒一次bio_aof_fsync 线程约 1 秒,极端 2 秒几乎为零~100,000 ops/s
no不主动内核回写30 秒(dirty_expire_centisecs)零~120,000 ops/s

三种策略代表三种不同的安全-性能平衡点。always 在主线程上同步等待磁盘确认,吞吐受 fsync 延迟主导;everysec 将 fsync 卸载到后台线程,主线程仅做内存拷贝;no 完全交给操作系统,丢失窗口最大但开销最低。

18.3.2 always 策略的事件循环时序

图 18-3-2 · always 同步 fsync — 事件循环内的完整时序

always 模式下,每次事件循环的 flush 都触发同步 fsync。一个事件循环内的多条写命令可合并为一次 fsync (group commit),实际吞吐高于"单条命令一次 fsync"的简化估算,但吞吐仍受 fsync 延迟和磁盘 IOPS 主导。

18.3.3 everysec 策略的双线程协作

图 18-3-3 · everysec 后台 fsync 与 2 秒退化保护

everysec 是默认推荐策略。主线程仅承担 write() 开销(微秒级),fsync 由后台 bio 线程异步执行。当后台 fsync 因磁盘 I/O 饱和延迟超过 2 秒时,主线程下一次 flushAppendOnlyFile() 会阻塞等待,以防止 aof_buf 无限增长。

18.3.4 fsync 退化为阻塞的边界条件

✅ 正常 everysec 时序
时刻 (ms)事件
t=0write(aof_buf, 128B)
t=0.001submit BIO_AOF_FSYNC

→ 主线程不阻塞, 后台 fsync 1s 内完成
❌ 慢盘退化时序
时刻 (ms)事件
t=0write(aof_buf, 128B)
t=0.001submit BIO_AOF_FSYNC

诊断指标: aof_delayed_fsync 累计计数 · 告警阈值: 单位时间增量持续 > 0 (绝对值在长跑实例上会很大, 看不出近期退化)
核心观测信号: aof_delayed_fsync 是该退化的核心观测信号。该指标增长意味着主线程在 write() 路径上发生了阻塞等待,延迟会反馈到客户端 P99。监控应配置该指标的单位时间增量,而非绝对值(绝对值在长跑 Redis 上会累积到很大,看不出近期是否在退化)。

18.4 BGREWRITEAOF 重写机制

18.4.1 重写的目标与价值

❌ 重写前 (冗余 101 条命令)
SET counter 1
INCR counter            → counter = 2
INCR counter            → counter = 3
... (重复 97 次)
INCR counter            → counter = 100
SET name "old"
SET name "new"          ← 覆盖前一行
101 条命令, 大量冗余
✅ 重写后 (等价精简 2 条)
SET counter 100
SET name "new"
2 条命令, 与重写前完全等价

重写的核心思想是"用当前内存状态生成最小等价命令集"。对单 key 的多次 INCR / APPEND / LPUSH / SADD 等增量操作,重写后等价为单条 SET / 完整列表/集合的赋值。该过程不读旧 AOF 文件,而是直接遍历内存数据结构。

18.4.2 重写的整体流程 (Redis 7 Multi-Part)

图 18-4-2 · BGREWRITEAOF 流程(Redis 7 Multi-Part AOF)

Redis 7 Multi-Part AOF 的重写通过 manifest 文件原子切换,避免了 Redis 6 时代单文件原地 rename 的边界问题。子进程仅负责生成新 BASE,新写命令始终写入独立的 INCR 文件,不需要重写期间双写到旧文件。

18.4.4 子进程遍历内存的命令生成

内存中的 hash key user:42
user:42 = {
    name: "Alice",
    age:  30,
    role: "admin"
}
→ 子进程生成的等价 RESP 命令
*5\r\n
$4\r\nHSET\r\n
$7\r\nuser:42\r\n
$4\r\nname\r\n
$5\r\nAlice\r\n

*5\r\n
$4\r\nHSET\r\n
$7\r\nuser:42\r\n
$3\r\nage\r\n
$2\r\n30\r\n
...
大集合分批策略 (aof-rewrite-incremental-fsync):
  • list / set / hash / zset 每批最多 64 元素
  • 超过则分多条 LPUSH / SADD / HSET / ZADD
  • 避免单条命令的 RESP buffer 过大, 让 fsync 节奏更平滑

分批生成的目的有二:其一,RESP 协议的单个命令理论无大小限制,但子进程内存与磁盘 I/O 受限,过大的命令会导致重写延迟增加;其二,加载时单条命令的解析开销与体积线性相关,分批后加载更平滑。

18.4.5 自动重写触发条件

图 18-4-5 · 自动重写决策树

默认配置 auto-aof-rewrite-percentage 100 / auto-aof-rewrite-min-size 64mb 的语义是:文件至少 64 MiB 且相比上次重写后翻倍。该规则避免小文件频繁重写,同时控制大文件无限膨胀。

18.5 Multi-Part AOF 文件结构

18.5.1 appendonlydir 目录结构

manifest 是恢复入口的唯一来源。重启时 Redis 读取 manifest,按 seq 顺序加载 BASE 与 INCR 文件,先恢复基础状态再回放增量。该机制将"切换有效文件"从原地 rename 简化为追加 manifest 行 + fsync。

18.5.2 manifest 切换的原子性保障

图 18-5-2 · rename POSIX 原子切换

rename() 在同一文件系统内是 POSIX 原子操作,manifest 切换保证崩溃时要么读到旧 manifest(继续使用旧 BASE+INCR),要么读到新 manifest(使用新 BASE+INCR),不会读到中间状态。

18.5.3 加载流程

图 18-5-3 · 重启时的 AOF 加载

aof-load-truncated yes(默认开启)在断电导致末尾命令半写入时自动截断到最后完整命令并启动。该选项以"丢失最后几条未完成命令"为代价换取启动可用性。

18.6 混合持久化

18.6.1 混合持久化的文件结构

❌ 传统纯 AOF (aof-use-rdb-preamble no)
→ 10M keys 加载 ~25 秒
✅ 混合持久化 (aof-use-rdb-preamble yes)
→ 10M keys 加载 ~5 秒, 5× 加速

混合持久化由 aof-use-rdb-preamble yes 启用(开启 AOF 时 Redis 5.0+ 默认使用)。重写时子进程生成 RDB 格式的 BASE 文件而非 RESP,加载速度从"逐条解析回放"降为"二进制反序列化",10M keys 规模下恢复时间从约 25 秒降至约 5 秒。

18.7.2 主节点不开持久化的风险

图 18-7-2 · 空主节点重启污染副本 — 事故场景
严重事故场景: 主节点没有任何本地持久化时,崩溃重启会以空 keyspace 启动;如果副本仍把它当主节点重新同步,空快照会覆盖副本内存,后续 AOF 重写还会把空状态固化到磁盘。生产环境若主节点不开 AOF,必须确保有 RDB/副本提升/上游日志兜底,并在旧主恢复前先隔离或降级为副本。

18.7.3 Cluster 中 AOF 与 Slot 迁移

图 18-7-3 · Slot 迁移在两节点 AOF 的独立记录

Cluster 节点的 AOF 仅记录该节点的本地命令,不感知集群拓扑。Slot 迁移期间,源节点 AOF 记录 DEL,目标节点 AOF 记录 RESTORE,各自独立。重启时各节点按本地 AOF 恢复,集群拓扑由 cluster nodes 状态独立管理。

18.8 性能与观测

18.8.1 fsync 策略的吞吐量基线

always 的吞吐瓶颈是单次 fsync 延迟 × 命令数。NVMe SSD 单次 fsync 约 0.1 ms,理论上限 10,000 ops/s;HDD 约 5 ms,上限约 200 ops/s。多条命令在同一事件循环可合并为一次 fsync (group commit),实测高于该理论上限。

→ 60 GB 实例 BGREWRITEAOF 期间内存峰值可达 1.4×, 必须留足余量
真实事故: 某游戏公司大版本上线日 19:00,QPS 从 5k 飙到 30k,正巧触发 auto-aof-rewrite-percentage 100 阈值。重写过程持续 4 分钟,CoW 内存放大到 ~28 GB(原 60 GB 实例),系统总内存 95% 触发 OOM Killer 杀 Redis 进程。事后规则:其一,auto-aof-rewrite-min-size 从 64mb 调到 1gb 减少重写触发频率;其二,BGREWRITEAOF 改用 cron 在业务低峰 03:00 手动触发;其三,maxmemory 从 70% 降到 55% 留出 CoW 余量。

18.10.3 feedAppendOnlyFile 后写日志核心

/* aof.c: feedAppendOnlyFile - 后写日志的核心
 * 仅在命令成功执行后被调用 */
void feedAppendOnlyFile(int dictid, robj **argv, int argc) {
    sds buf = sdsempty();
    if (dictid != server.aof_selected_db) {
        /* DB 切换记录 */
        char seldb[64];
        snprintf(seldb, sizeof(seldb), "%d", dictid);
        buf = sdscatprintf(buf, "*2\r\n$6\r\nSELECT\r\n$%lu\r\n%s\r\n",
            (unsigned long)strlen(seldb), seldb);
        server.aof_selected_db = dictid;
    }
    /* 命令编码为 RESP */
    buf = catAppendOnlyGenericCommand(buf, argc, argv);
    /* 追加到 aof_buf */
    if (server.aof_state == AOF_ON ||
        server.aof_state == AOF_WAIT_REWRITE)
        server.aof_buf = sdscatlen(server.aof_buf, buf, sdslen(buf));
    sdsfree(buf);
}

18.10.4 flushAppendOnlyFile 三策略统一入口

/* aof.c: flushAppendOnlyFile - 三策略统一入口
 * beforeSleep 阶段调用 */
void flushAppendOnlyFile(int force) {
    ssize_t nwritten;
    if (sdslen(server.aof_buf) == 0) return;

    nwritten = aofWrite(server.aof_fd, server.aof_buf,
                        sdslen(server.aof_buf));
    if (nwritten != (ssize_t)sdslen(server.aof_buf)) {
        /* write 失败处理: 标记 last_write_status,拒绝后续写 */
        server.aof_last_write_status = C_ERR;
        return;
    }
    server.aof_current_size += nwritten;
    sdsclear(server.aof_buf);

    if (server.aof_fsync == AOF_FSYNC_ALWAYS) {
        redis_fsync(server.aof_fd);
        server.aof_fsync_offset = server.aof_current_size;
        server.aof_last_fsync = server.unixtime;
    } else if ((server.aof_fsync == AOF_FSYNC_EVERYSEC &&
                server.unixtime > server.aof_last_fsync)) {
        /* 提交后台 fsync */
        if (!sync_in_progress) aof_background_fsync(server.aof_fd);
    }
}

18.10.5 设计取舍

AOF 选择"后写日志"是为了保持命令日志的语义清洁:仅记录已生效的修改,避免恢复路径处理无效命令。代价是 apply-log 之间的窗口需要错误状态保护(aof_last_write_status),write 失败时 Redis 拒绝所有写命令直至 AOF 恢复。

appendfsync everysec 是默认策略的根本原因是吞吐与丢失窗口的非线性关系:从 always 到 everysec,丢失窗口从"已 fsync 命令通常不丢"放宽到"约 1 秒",但吞吐量常可提升一个数量级。具体 ops/s 取决于磁盘、内核和 value 大小,不要把示例基准当成通用容量承诺。

Redis 7 Multi-Part AOF 的设计动机是简化重写期间的状态管理。Redis 6 单文件方案需要重写期间双写到旧 AOF 与 rewrite_buf,切换瞬间将 rewrite_buf 拼接进新文件并 rename;该流程在主线程引入了"拼接 + rename"的不可中断段,且 aof_rewrite_buf 在写流量大时可达数百 MB。Multi-Part 方案将 BASE 与 INCR 物理分离,父进程始终只写 INCR,子进程仅写 BASE,切换从 rename 简化为更新 manifest,主线程影响显著降低。

18.11 本章要点

#要点
1AOF 是后写日志:仅记录成功执行的命令,与传统 WAL(预写日志)方向相反
2三阶段管道:命令执行 → aof_buf 追加 → write 至 page cache → fsync 落盘
3appendfsync 三策略:always(每命令 fsync,吞吐最低)、everysec(后台每秒,默认推荐)、no(交内核,丢失窗口由 OS 刷盘策略决定)
4aof_delayed_fsync 是 everysec 退化为主线程阻塞的早期信号,单位时间增量 > 0 立即告警
5BGREWRITEAOF 通过 fork + 子进程遍历 keyspace 生成最小等价命令,父进程继续接受新写并独立追加到 INCR
6Redis 7 Multi-Part AOF 将单文件拆为 BASE + INCR + manifest,切换从 rename 简化为 manifest 更新
7混合持久化(aof-use-rdb-preamble yes)通过 RDB 二进制 BASE 加 RESP 增量 INCR,恢复速度从 25 秒降至 5 秒(10M keys)
8主节点不开 AOF 时,必须明确 RDB、Replica 或上游事件流谁负责恢复;旧主故障重启前要先隔离,避免空数据集重新同步给副本
9备份单位是整个 appendonlydir 目录,仅备份单个 .aof 文件会遗漏 manifest 导致无法恢复
10重写期间 CoW 复制 + aof_rewrite_buf 可使内存峰值翻倍,生产环境建议业务低峰手动触发或上调 auto-aof-rewrite-min-size

章末总结

第 17 章 RDB + 第 18 章 AOF 共同构成 Redis 的持久化层:

  • RDB:周期性二进制快照,恢复快、文件小,但丢失窗口取决于 save 规则与最近一次成功快照;
  • AOF:命令日志,丢失窗口小(1 秒)、可读,但恢复慢、文件大;
  • 混合持久化:RDB 做 BASE + AOF 做 INCR,集两者优点。
