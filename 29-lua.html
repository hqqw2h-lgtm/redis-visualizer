第 29 章 · Lua 脚本与 Functions

EVAL / EVALSHA · Lua 5.1 沙箱 · KEYS/ARGV · 脚本缓存 · 效果复制 · Redis 7 Functions · BUSY 与 SCRIPT KILL · Cluster hash tag

Redis 单条命令的原子语义足以处理大量场景,但"读-判断-写"复合操作需要多次客户端往返,既增加延迟又引入竞态风险。Lua 脚本将这类操作打包到服务端执行,在单线程模型下天然获得原子性,且可基于前一命令结果进行条件分支,这是 MULTI/EXEC 无法做到的。Redis 7.0 引入的 Functions 进一步将脚本提升为持久化的一等公民,随 RDB/AOF 持久化、随复制传播,解决了 EVAL 脚本缓存的临时性问题。

Lua 脚本的工程价值有三:其一,执行期间不被插入 — 脚本内所有 redis.call 在主线程顺序执行;其二,条件分支 — 可基于读结果决定后续写操作,把"先校验再写"放到服务端完成;其三,网络优化 — 单次 RTT 完成原本需要多次往返的逻辑。也要记住:Lua 脚本出错后不会自动回滚已经执行的写命令,所以复杂脚本应先校验、再写入。代价同样明确:脚本在主线程执行,长脚本会阻塞其他客户端。

29.1 服务端脚本的必要性

"为什么需要服务端脚本"的答案不是"性能",而是"语义"。如果只是减少 RTT,Pipeline 与 MULTI/EXEC 都能做;但只要业务出现"读取一个值,基于结果决定是否写"的模式,客户端层就难以在不引入竞态的前提下完成。下面以 API 限流器为例,逐层展示客户端方案的语义缺口,以及 Lua 如何一举解决。

29.1.1 限流器场景的三种实现对比

图 29-1-1 · 同一限流需求的四种实现方式
业务需求: API 限流
  - 检查当前窗口请求数
  - 若超阈值返回拒绝
  - 否则计数 +1
  - 若是新窗口设置过期时间

实现方式 1: 客户端多次 RTT
  GET    rate:user:1001       → 87
  if 87 >= 100 return DENIED
  INCR   rate:user:1001       → 88
  if first-call: EXPIRE rate:user:1001 60

  问题:
    4 次 RTT,延迟高
    GET 与 INCR 之间可被其他客户端插入
    INCR 成功 EXPIRE 失败 → 永不过期 (内存泄漏)

实现方式 2: MULTI/EXEC
  MULTI
  GET rate:user:1001        ; 入队,无法读取结果
  INCR rate:user:1001       ; 无法基于 GET 结果判断
  EXPIRE rate:user:1001 60  ; 总是设置 (可能浪费)
  EXEC

  问题:
    无法基于 GET 结果决定是否 INCR
    INCR 与 EXPIRE 总是执行
    无法实现"超阈值拒绝"语义

实现方式 3: WATCH + MULTI/EXEC
  WATCH rate:user:1001
  v = GET rate:user:1001
  if v >= 100 → UNWATCH; return DENIED
  MULTI / INCR / EXPIRE / EXEC

  问题:
    高并发下 EXEC 频繁返回 nil (WATCH 冲突)
    需要客户端重试逻辑
    极端情况退化为单线程串行

29.1.2 Lua 方案 — 一次 RTT 完成全部

-- 限流器 Lua 脚本
local current = redis.call('GET', KEYS[1])
if current and tonumber(current) >= tonumber(ARGV[1]) then
    return 0  -- 已超限,拒绝
end

local result = redis.call('INCR', KEYS[1])
if result == 1 then
    redis.call('EXPIRE', KEYS[1], ARGV[2])
end
return 1  -- 放行
$ redis-cli EVAL \
  "$(cat ratelimit.lua)" \
  1 rate:user:1001 100 60
(integer) 1
图 29-1-2 · Lua 方案的优势与代价
Lua 方案的优势:

  ✓ 1 次 RTT,延迟最低
  ✓ 单线程原子,GET 到 INCR 期间无其他命令插入
  ✓ 可根据 GET 结果决策,无需 WATCH/重试
  ✓ INCR 与 EXPIRE 在同一原子执行,无中间状态

代价:
  ✗ 脚本执行期间,主线程被独占,其他客户端等待
  ✗ 长脚本会成为全局延迟源
  ✗ Cluster 模式下所有 key 必须同 slot

29.2 EVAL 命令与 Lua 沙箱

EVAL 是 Lua 脚本的入口命令,签名简单但语义至关重要:KEYS 与 ARGV 必须严格分离,这不是风格选择,而是 Cluster 路由的硬性要求。沙箱则是一组确定的"允许 / 禁止"清单,把 Lua 5.1 的能力裁剪到适合 Redis 服务端执行的最小集。

29.2.1 EVAL 命令语法

图 29-2-1 · EVAL 命令的参数结构
EVAL script numkeys [key [key ...]] [arg [arg ...]]

参数说明:
  script    Lua 5.1 脚本文本
  numkeys   后续参数中 key 的个数
  key ...   通过 KEYS[1..n] 在脚本中访问
  arg ...   通过 ARGV[1..n] 在脚本中访问

KEYS 与 ARGV 分离的设计意义:

  KEYS[]:
    脚本将访问的所有 Redis key
    必须在调用时显式声明
    Cluster 通过 KEYS 计算 slot 进行路由
    脚本内不应再通过其他方式构造 key (会导致 Cluster 路由失败)

  ARGV[]:
    非 key 类型的参数 (阈值、TTL、标志位)
    任意值,不影响 Cluster 路由

最佳实践:
  ✓ EVAL "..." 1 user:1001 100 60
    脚本访问 user:1001,阈值 100,TTL 60

  ✗ EVAL "redis.call('GET', 'user:' .. ARGV[1])" 0 1001
    动态构造 key 绕过 KEYS 声明
    Cluster 模式下无法正确路由

29.2.2 Lua 5.1 沙箱的能力边界

图 29-2-2 · Lua 沙箱的"允许 / 禁止"清单
图 29-2-3 · 沙箱设计的三大目的
沙箱的设计目的:

  目的 1: 安全
    禁止文件/网络 IO,脚本不能成为攻击向量
    禁止外部命令执行,脚本不能逃逸 Redis 进程

  目的 2: 确定性
    禁止外部时间/随机源,主从复制时结果一致
    全局变量保护,避免跨脚本状态污染

  目的 3: 可预测
    固定 Lua 5.1 版本,语言行为稳定
    沙箱在脚本启动时构造,执行后清理

29.2.3 全局变量保护

$ redis-cli EVAL "x = 1; return x" 0
(error) ERR Error compiling script (new function): user_script:1:
Script attempted to create global variable 'x'

$ redis-cli EVAL "local x = 1; return x" 0
(integer) 1
图 29-2-4 · 全局变量保护机制
全局变量保护机制:

  Lua 默认: x = 1 创建全局变量
  Redis 沙箱: 通过 metatable 拦截全局赋值,抛错

  保护原因:
    跨脚本状态泄露
    主从复制时全局状态不一致
    内存泄漏 (全局变量永不释放)

  正确写法:
    所有变量必须 local
    跨函数共享通过参数传递
    需要"全局"行为时用 Redis 数据结构存储
"忘写 local"是新手最常遇到的脚本错误:Lua 默认变量是全局,Redis 用 metatable 拦截全局赋值并直接抛错。生产脚本必须严格 local 化所有临时变量,这既是 Redis 的硬性要求,也是 Lua 编程的基本卫生。

29.3 EVALSHA 与脚本缓存

每次 EVAL 都重传完整脚本是带宽浪费。Redis 通过脚本缓存解决:首次 SCRIPT LOAD 把脚本编译成 Lua function,以 SHA1 为键存进 server.lua_scripts;后续仅传 40 字节 SHA1 即可复用。代价是缓存仅在内存,SCRIPT FLUSH、failover、重启都会丢失,客户端必须实现 NOSCRIPT 时的透明重载。

29.3.2 EVAL vs EVALSHA 的网络开销

图 29-3-2 · 50 K QPS 下两种方式的带宽对比
典型业务脚本: 500 字节
QPS: 50,000

EVAL (每次发送完整脚本):
  上行带宽 = 500 B × 50,000 = 25 MB/s
  下行带宽 = ~50 B × 50,000 = 2.5 MB/s
  总计 = 27.5 MB/s

EVALSHA (仅发送 SHA1):
  SHA1 字符串 40 字节
  上行带宽 = 40 B × 50,000 = 2 MB/s
  下行带宽 = ~50 B × 50,000 = 2.5 MB/s
  总计 = 4.5 MB/s

带宽节省: 23 MB/s (~84%)

延迟收益:
  小脚本 (< 1 KB): 提升 ~5%
  大脚本 (~10 KB): 提升 ~30-50%

29.3.3 NOSCRIPT 错误的处理模板

def execute_lua(redis_cli, script, sha1, keys, args):
    try:
        return redis_cli.evalsha(sha1, len(keys), *keys, *args)
    except redis.exceptions.NoScriptError:
        log.warning("NOSCRIPT, reloading")
        sha1 = redis_cli.script_load(script)
        return redis_cli.evalsha(sha1, len(keys), *keys, *args)
图 29-3-3 · NOSCRIPT 出现的三种场景
NOSCRIPT 出现的场景:

  场景 1: failover 后新主节点
    新主节点没有原主节点的脚本缓存
    若客户端未预加载脚本,第一次 EVALSHA 会因缓存缺失返回 NOSCRIPT

  场景 2: SCRIPT FLUSH 被执行
    运维清空了所有脚本缓存
    所有客户端的 EVALSHA 都失败

  场景 3: Redis 重启
    脚本缓存在内存中,不持久化
    重启后所有缓存丢失

正确处理:
  ✓ 客户端必须实现 SCRIPT LOAD + EVALSHA 重试
  ✓ 不依赖手动预加载,运行时动态恢复
  ✓ 监控 NOSCRIPT 频率,异常时排查 SCRIPT FLUSH 来源

  仅在脚本未执行任何写命令时可终止

  原因:

  lua-time-limit 配置 (默认 5000 ms)

  此时普通命令会被拒绝;只读脚本可尝试 SCRIPT KILL
  其他命令返回 BUSY

29.4 redis.call 与 redis.pcall

脚本内调用 Redis 命令有两个入口:redis.call 在命令失败时抛 Lua 异常,脚本立即终止;redis.pcall 在失败时返回错误表,脚本继续。前者类比 Java throw,后者类比 try-catch。理解二者差异是写出"正确的失败处理"脚本的前提。

29.4.1 错误处理的两种语义

-- redis.call 抛异常,脚本立即终止
redis.call('SET', 'k', 'v')
redis.call('LPUSH', 'k', 'x')  -- WRONGTYPE 错误
-- 脚本停止,客户端收到 ERR 信息
-- 已写入的 SET 不回滚

-- redis.pcall 返回错误表,脚本继续
redis.call('SET', 'k', 'v')
local result = redis.pcall('LPUSH', 'k', 'x')
-- result = {err = "WRONGTYPE ..."}
-- 脚本继续执行
if result.err then
    redis.log(redis.LOG_WARNING, result.err)
end
表 29-4-1 · redis.call vs redis.pcall
行为redis.callredis.pcall
命令成功返回结果返回结果
命令失败抛错,脚本终止返回 {err=...} 表,脚本继续
类比Java throwJava try-catch
推荐场景单一逻辑路径,失败应中止容错复杂脚本

图 29-4-1 · 类型转换的四个关键陷阱
类型转换的关键陷阱:

  陷阱 1: nil 表示
    Redis nil → Lua false (不是 nil)
    对 Redis nil 返回值,if value == nil 为 false
    正确: if not value or value == false

  陷阱 2: 数组下标
    Lua table 1-indexed,不是 0-indexed
    redis.call('LRANGE', k, 0, -1)[1] 是第一个元素

  陷阱 3: 状态回复
    return redis.call('SET', k, v)  →  {ok="OK"}
    客户端看到 OK 字符串,但 Lua 内部是 table

  陷阱 4: 大整数精度
    Lua 5.1 仅 double,2^53 以上整数会失精
    INCRBY 100000000000000000000 在 Lua 中变为 1e20
    大整数应保留为字符串

29.4.3 Lua number 精度陷阱实测

$ redis-cli EVAL "
  redis.call('SET', KEYS[1], '9007199254740993')
  local v = redis.call('GET', KEYS[1])
  return tonumber(v) + 0
" 1 bigint
(integer) 9007199254740992
图 29-4-2 · 9007199254740993 的精度损失
精度损失分析:

  9007199254740993 = 2^53 + 1
  Lua 5.1 double 精度: 2^53 (约 1.8e16)

  tonumber(v) → double:
    无法精确表示 2^53 + 1
    取最接近的 double 值: 2^53 (即 9007199254740992)

  低位丢失,数据错误

正确处理:
  ✓ 大整数保留 string 形式,不转 number
  ✓ INCRBY/DECRBY 命令在 Redis 端处理 64-bit 整数
  ✓ Redis 7+ 提供 redis.setresp(3),启用 RESP3 类型
  ✓ 业务避免在 Lua 中对 > 2^53 的整数做算术
"业务计数器超 2^53 静默错位"是排查难度极高的 bug:Lua 5.1 内置的 number 类型仅 double,精度上界 2^53(≈ 9.0×10¹⁵)。订单 ID、雪花 ID、纳秒时间戳都可能超过此线。一旦 tonumber(),低位无声丢失,业务下游唯一现象是"偶尔重复"。规则:大整数在 Lua 中应保持字符串形式,不要转 number 做算术。

29.5 原子性与确定性

Lua 脚本的执行隔离来自 Redis 的单线程模型:evalGenericCommand 调用 lua_pcall 时,主线程在执行 Lua VM,其他客户端命令只能等待。这保证脚本执行期间不被插队,但不代表"出错后回滚"。沙箱的确定性约束曾服务于脚本复制(Redis < 7);Redis 7 默认效果复制后,这部分约束有所放宽。

29.5.1 脚本原子性的形式化证明

图 29-5-1 · 脚本执行期间不被插队
结论: Lua 脚本执行期间不会与其他客户端命令交错

设脚本 S 执行命令序列 C_1, C_2, ..., C_n
对任意并发客户端 P:
  ∀ t: state(P, t) ∈ {state_before(S), state_after(S)}

证明:
  Redis 主线程在 evalGenericCommand 内执行 Lua VM
  Lua VM 调用 redis.call → luaRedisGenericCommand → call()
  call() 执行 Redis 命令,返回 Lua VM

  期间 ae 事件循环不返回 epoll_wait
  其他客户端命令在事件队列等待,不被处理

  ⊢ 脚本执行是原子调度单元
  ⊢ 多脚本的执行形成全序关系
  ⊢ 可把脚本看作一个连续执行片段,但失败不会自动回滚已执行写入  ∎

29.5.2 沙箱的确定性约束

图 29-5-2 · Redis Lua 沙箱的四类确定性约束
Redis Lua 沙箱的确定性约束:

  约束 1: 固定随机种子
    脚本启动时 math.randomseed(0)
    每次执行 random 序列相同

  约束 2: 排序后返回
    SMEMBERS / SUNION 等命令在脚本中自动排序
    消除集合无序带来的不确定性

  约束 3: 全局变量保护
    防止跨脚本状态泄露
    保证脚本仅依赖参数

  约束 4: 禁止外部 IO
    无 io/os 库
    无 socket/http
    脚本结果仅取决于 Redis 数据

确定性的目的:
  Redis 4 之前: 脚本复制
  主从都执行相同脚本,结果必须一致

  Redis 7+: 效果复制
  主节点执行脚本,从节点接收实际写命令
  脚本本身不在从节点执行
  确定性约束放宽,但仍保留基本沙箱

29.5.3 Lua 与 MULTI/EXEC 的关键差异

图 29-5-3 · Lua 与 MULTI/EXEC 在三个维度的对比

从工程视角看,MULTI/EXEC 适合"批量提交一组无依赖写"(批量删除、批量初始化),Lua 适合"一组有读写依赖的复合操作"(限流、库存扣减、分布式锁释放)。两者并非简单取代关系,而是 Redis 提供的两种多命令编排工具。

29.6 复制模式

脚本如何在 Master → Replica 之间传播,Redis 经历了从"脚本复制"到"效果复制"的演进。Redis 7.0 默认效果复制后,Lua 脚本可以使用 math.random、redis.call('TIME') 等非确定性能力,同时 Replica 不再需要 Lua 引擎参与。本节梳理两种模式的复制流形态、确定性约束以及 Redis 7 的兼容处理。

29.6.1 脚本复制 vs 效果复制

图 29-6-1 · 两种复制模式的数据流
图 29-6-2 · 两种模式的优劣对比
脚本复制 (Script Replication):
  Master 将 EVAL 命令本身写入复制流
  Replica 接收 EVAL 后重新执行

  优势:
    复制流体积小
    脚本一次复制,多次执行

  劣势:
    要求确定性 (math.random/TIME 不可用)
    Replica 也阻塞执行脚本
    脚本错误在主从都发生

效果复制 (Effects Replication):
  Master 执行脚本期间,redis.call 内的写命令
  逐个写入复制流 (作为 MULTI...EXEC 块)
  Replica 接收实际命令,不执行 Lua

  优势:
    Replica 不需要 Lua 引擎
    不需要确定性约束
    脚本可使用 random/TIME 等

  劣势:
    复制流体积大 (每个写命令独立)
    Master 与 Replica 执行成本不对称

Redis 7.0 默认效果复制
  redis.replicate_commands() 调用被忽略 (向后兼容)

29.6.2 效果复制的实际传播

  redis.call('SET', KEYS[1], 'a')
  redis.call('INCR', KEYS[2])
  redis.call('DEL', KEYS[3])
" 3 k1 k2 k3

# Master 复制流写入

# 不执行 Lua,只应用三条写命令

效果复制将脚本产生的实际写命令传播出去。Replica 看到的是 MULTI...EXEC 包装的命令序列,与主节点状态一致,但脚本本身不在 Replica 执行。这样可以避免副本重放脚本时再次遇到超时或非确定性结果;不过脚本仍要声明真实 key、控制耗时,并谨慎使用随机数、时间等非确定性输入。

效果复制的副作用 — 复制流变大:一段执行 1000 次 redis.call('INCR') 的脚本,脚本复制只传 1 条 EVAL,效果复制要传 1000 条 INCR。因此对于"内部多写"型脚本,Master → Replica 的复制带宽会显著上升。生产环境应监控 master_repl_offset 增速,大型 batch 任务考虑切到非脚本实现。

29.7 Redis 7.0 Functions

EVAL/EVALSHA 的核心痛点不在性能而在生命周期:脚本缓存仅在内存,重启 / failover / FLUSH 都会丢失,客户端必须处理 NOSCRIPT 并重载;同时脚本散落在客户端代码,难以版本管理。Redis 7.0 引入的 Functions 把脚本库纳入持久化与复制:FUNCTION LOAD 写入 RDB/AOF,随复制流传播。FCALL 不依赖 EVALSHA 的脚本缓存,但函数或库不存在时仍会返回错误。

Cluster 中 Lua 的工程考虑:

  ✓ 所有 KEYS 必须 hash 到同一 slot
  ✓ 用 {tag} 强制相关 key 同 slot
  ✓ 脚本内不动态构造 key (绕过 KEYS 检查)
  ✓ 测试: redis-cli CLUSTER KEYSLOT key1 / key2 应相等

  ✗ 不要在脚本内组合不同业务的 key
  ✗ 不要假设脚本可以访问任意 key

  hash tag 选择业务自然边界 (user_id, order_id)
  避免单一 tag 集中过多 key (热点)
  Lua 脚本与 hash tag 设计一起考虑
动态拼 key 是 Cluster 模式下脚本的头号杀手:写成 redis.call('GET', 'user:' .. ARGV[1]) 这种"在脚本里组装 key"的写法,在单节点下没事,迁到 Cluster 立刻 CROSSSLOT。Redis 用 KEYS 数组提前算 slot;只要绕过 KEYS,Cluster 就完全不知道脚本会摸哪些 key。规则:所有要访问的 key 都必须通过 KEYS 数组传入。

29.10.2 BUSY 状态的处理

图 29-10-1 · 触发条件 → BUSY 行为 → 三种恢复方法
触发条件:
  脚本执行超过 lua-time-limit (默认 5000 ms)
  Redis 状态变为 BUSY

BUSY 期间的行为:
  正在执行的脚本继续执行 (不中断)
  其他客户端命令返回:
    -BUSY Redis is busy running a script.
    You can only call SCRIPT KILL or SHUTDOWN NOSAVE.

恢复方法:

  方法 1: SCRIPT KILL
    仅当脚本未写入任何数据时可用
    脚本一旦执行 redis.call('SET'/'INCR'/...) 就不能 kill
    原因: 部分写入不能回滚,kill 会导致数据不一致

  方法 2: 故障切换 / 重启预案
    仅在确认等待不可接受时使用
    先评估已执行写入与持久化状态
    避免把 SHUTDOWN NOSAVE 当作常规恢复手段

  方法 3: 等待脚本完成
    最安全,但可能阻塞数分钟

预防:
  ✓ 监控 cmdstat_eval 的 usec_per_call
  ✓ 设置应用层超时,避免提交超长脚本
  ✓ Lua 内部避免无限循环
  ✓ Cluster 模式 lua-time-limit 应小于 cluster-node-timeout
    否则 BUSY 期间被误判为故障,触发 failover
SCRIPT KILL 可用
脚本仅做了读 / 计算,未发出任何写命令。Redis 此时安全终止脚本,客户端收到 ERR 后可重试。
SCRIPT KILL 不可用
脚本已经执行过 SET/INCR/DEL 等写命令。Redis 会拒绝 SCRIPT KILL,因为无法回滚已经生效的写入。常规选择是等待脚本结束;若必须恢复服务,按预先演练过的故障切换或重启流程处理。
Cluster 部署最关键的一条:让 lua-time-limit 明显小于 cluster-node-timeout,并监控脚本耗时。否则超长脚本可能让节点长时间无法响应 Cluster 心跳,造成误判、故障切换或更长时间的不可用窗口。

/* src/eval.c: EVAL 主入口 */
void evalGenericCommand(client *c, int evalsha) {
    sds funcdef;
    char funcname[43];

    /* 1. 解析 numkeys 与 KEYS/ARGV */
    long long numkeys;
    if (getLongLongFromObjectOrReply(c, c->argv[2], &numkeys, NULL)
        != C_OK) return;

    /* 2. EVAL: 编译并缓存脚本 */
    if (!evalsha) {
        funcdef = luaCreateFunction(c, c->argv[1]);
    } else {
        /* EVALSHA: 查找缓存 */
        funcdef = dictFetchValue(server.lua_scripts,
            (sds)c->argv[1]->ptr);
        if (funcdef == NULL) {
            addReplyErrorObject(c, shared.noscripterr);
            return;
        }
    }

    /* 3. 设置 Lua KEYS/ARGV 全局表 */
    luaSetGlobalArray(server.lua, "KEYS", c->argv + 3, numkeys);
    luaSetGlobalArray(server.lua, "ARGV", c->argv + 3 + numkeys,
                     c->argc - 3 - numkeys);

    /* 4. 调用 Lua 函数 */
    int err = lua_pcall(server.lua, 0, 1, -2);

    /* 5. 转换 Lua 返回值为 Redis reply */
    if (err == 0) {
        luaReplyToRedisReply(c, server.lua);
    } else {
        addReplyErrorFormat(c, "Error running script: %s",
                           lua_tostring(server.lua, -1));
    }
}

29.11.3 redis.call 的 C 实现

/* src/script_lua.c: redis.call 实现 */
int luaRedisGenericCommand(lua_State *lua, int raise_error) {
    int j, argc = lua_gettop(lua);
    /* 构造 Redis 命令对象 */
    robj **argv = zmalloc(sizeof(robj*) * argc);
    for (j = 0; j < argc; j++) {
        argv[j] = createStringObject(lua_tostring(lua, j+1),
            lua_strlen(lua, j+1));
    }

    /* 查找命令 */
    struct redisCommand *cmd = lookupCommand(argv, argc);
    if (cmd == NULL) {
        /* 错误处理 */
        return raise_error ? luaRaiseError(lua, "...") : 1;
    }

    /* 执行命令 (call 进入 Redis 命令路径) */
    client *c = scriptGetClient();
    c->argv = argv;
    c->argc = argc;
    c->cmd = cmd;
    call(c, CMD_CALL_NONE);

    /* 转换 Redis reply 为 Lua value */
    return redisProtocolToLuaType(lua, c->buf);
}

29.11.4 设计取舍

取舍 1:为什么是 Lua 5.1 而非 Python / JavaScript

Redis 选择 Lua 5.1 而非 Python/JavaScript 的根本原因是嵌入性与确定性。Lua 5.1 解释器仅 ~150 KB,可静态链接进 Redis 二进制,无外部依赖;Lua 语言简洁,沙箱可彻底控制能力边界;5.1 版本稳定,API 多年不变,兼容性可控。Python/JS 解释器体积大,运行时依赖多,沙箱隔离复杂。

取舍 2:EVAL 缓存的临时性

EVAL 的脚本缓存设计目标是"大多数场景脚本只需传输一次"。脚本编译为 Lua function,缓存在 server.lua_scripts 字典,key 为 SHA1。客户端预加载后,后续仅传 SHA1 (40 字节) 替代完整脚本 (可能 KB 级)。代价是缓存非持久化,failover/重启/FLUSH 后丢失,需要 NOSCRIPT 处理。

取舍 3:Functions 借鉴存储过程

Functions 设计目标是"脚本作为持久化业务逻辑"。Library 概念组织相关函数,FUNCTION LOAD 写入 RDB/AOF,随复制流自动传播。该设计借鉴了 SQL 数据库的存储过程,将"代码与数据共置"模式引入 Redis。代价是仅支持加载完整 library,不支持单函数热更新(必须 FUNCTION LOAD REPLACE 整个库)。

取舍 4:效果复制取代脚本复制

效果复制 (Effects Replication) 替代脚本复制的核心动机是简化。脚本复制要求脚本在主从上产生同样结果;效果复制则让 Replica 只应用主节点产生的实际写命令,不再重放 Lua。Redis 7.0 默认效果复制后,非确定性输入的限制减少了,但这不等于可以任意编写脚本:仍要正确声明 key、避免长时间阻塞,并明确错误处理。

关键事实

表 29-12-1 · Lua / Functions 关键事实速查
#要点
1Lua 脚本提供"读-判断-写"原子语义,严格强于 MULTI/EXEC,可基于读结果分支
2EVAL 命令通过 KEYS 与 ARGV 分离参数,KEYS 用于 Cluster 路由,ARGV 不参与路由

4EVALSHA 通过 SHA1 缓存避免重复传输脚本文本,客户端必须实现 NOSCRIPT 时的 SCRIPT LOAD 重试
5redis.call 失败抛异常脚本终止,redis.pcall 返回错误表脚本继续,前者用于快速失败,后者用于容错
6Lua nil ≠ Redis nil:Redis nil 转为 Lua false,需用 not value 判断;大整数 > 2^53 失精,保留 string
7脚本不回滚:部分 redis.call 已写入再抛错时,已写入数据保留,需先校验再写
8Redis 7.0 默认效果复制,放宽确定性约束;脚本复制模式仅旧版兼容
9Functions (Redis 7+) 持久化业务脚本,随 RDB/AOF 保存,随复制传播,FCALL 不返回 NOSCRIPT
10生产建议:脚本尽量保持毫秒级短执行;Cluster 中 lua-time-limit 应小于 cluster-node-timeout;监控 cmdstat_eval/fcall 的 usec_per_call

章末总结

Lua 脚本 + Functions 是 Redis 提供的"服务端可编程"能力。从 EVAL 的"临时脚本缓存"到 Functions 的"持久化函数库",Redis 用十年时间把脚本机制从一个"高级技巧"演化为"一等公民"。本章给出的判断顺序如下:

  • 第 28 章(MULTI/EXEC + WATCH)处理"批量提交、不需要基于读结果决策"的场景;
  • 本章 29.1 的限流案例展示了"必须基于读结果决策"的硬需求,只有 Lua 能优雅解决;
  • 本章 29.2-29.5 给出了写正确 Lua 必须懂的四件事:KEYS/ARGV 分离、沙箱边界、redis.call/pcall 语义、Lua ↔ Redis 类型映射;
  • 本章 29.7 把临时 EVALSHA 升级到持久化 Functions;Redis 7+ 中,需要统一部署和版本管理的脚本更适合 Functions;
  • 本章 29.8-29.10 收口部署 / 性能 / 陷阱三个工程视角:Cluster 必须 hash tag,脚本应保持短执行,lua-time-limit < cluster-node-timeout 是 Cluster 部署的关键约束。
