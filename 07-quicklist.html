第 7 章 · quicklist 快速列表

分块链表 · 双端 O(1) · 节点 listpack 紧凑 · LZF 冷压缩 · 节点分裂 · LINDEX 跨节点 · fill / compress 调参

Redis List 类型的元素规模跨度极大:消息队列可能数十万条,时间线可能数百万条,小型缓存可能仅几十条。单一编码无法在全场景下兼顾内存效率与操作延迟。纯 adlist 双向链表每节点 32-40 B 指针开销,百万级 List 内存浪费 3-5 倍;单一 ziplist/listpack 极致紧凑但中间插入 O(N) memmove,百万元素的搬迁是数 MB 级;动态数组头部插入 O(N),无法支持 O(1) LPUSH/LPOP。

quicklist(Redis 3.2 引入)用 "分而治之" 解决:外层是双向链表,每个节点内部存一个小型 listpack(默认目标 8 KB),两端节点(head/tail)保持未压缩,中间节点可选 LZF 压缩。LPUSH/RPUSH/LPOP/RPOP 在头/尾节点 O(1) 完成,节点超阈值时新建节点接入链表,不引发整列表搬迁;中间冷节点 LZF 压缩,百万元素 List 内存可降至 1/3;节点内 listpack 享受 cache locality,L1 内顺序扫描。

代价是 LINDEX/LINSERT/LRANGE 在中间位置仍需 O(N/K + K) 扫描(K 是节点平均元素数,默认 ~100),不是真正的 O(1) 随机访问。Redis 7.0 起进一步把节点内编码从 ziplist 改为 listpack,即使单节点内插入也彻底告别连锁更新。

  • quicklist "分而治之" 思想:把大 List 切成 N/K 个 8 KB 节点的设计动机
  • quicklist 容器与 quicklistNode 字节布局(48 B 容器 + 48 B 节点元数据 + listpack)
  • fill 与 compress 两个调参的取值表与含义

7.1 解决什么问题

quicklist 不是"另一个数据结构"——它是 Redis 对 LIST 类型在 3.2 版本上的系统级改造,目的是同时满足"双端 O(1)、内存紧凑、大 List 不卡主线程"三个看似矛盾的目标。这一节先把 LIST 在生产中的访问模式拆开看,再分析候选方案为什么都不够好。

7.1.1 业务需求拆解

场景主要命令元素规模访问模式期望复杂度

时间线 / 信息流LPUSH + LRANGE 0 N100k - 10M头部写,头部读 N 条O(1) 写, O(N) 读

三个不可让步的需求:
① 双端访问占 80%+ — 头/尾必须 O(1),否则消息队列直接不可用;
② 元素规模跨 4-6 个数量级 — 同一编码必须能扛 10 元素和 1000 万元素;
③ 元素本身可能是大字符串(JSON、日志条目),内存效率决定单实例容量上限。
中间随机访问场景少,但被滥用时(频繁 LINDEX 长列表中间)需要"诚实告知用户慢"。

7.2 quicklist 整体布局

quicklist 是 三层结构:容器 quicklist 持有头/尾指针与计数;每个 quicklistNode 是双向链表节点,持有 listpack 字节指针、计数、压缩状态;节点 entry 字段指向真正的 listpack 字节(或 LZF 压缩块)。

7.3 LPUSH 头节点未满(快路径)

7.4 LPUSH 头节点已满 → 新建节点

头节点装不下新值时,quicklist 不会"硬塞" — 它创建一个全新的 quicklistNode,节点内 listpack 只装这一个新值,然后通过 4 个指针修改把新节点接入链表头。不会引发整列表搬迁,这是 quicklist 相对单一 listpack 的核心优势。

7.6 压缩节点访问(解压-操作-重压)

当 compress > 0 时,中间节点处于 LZF 压缩状态,entry 字段指向压缩字节流,无法直接读写。任何访问中间节点的操作(LINDEX 落在中间、LRANGE 跨过中间、LINSERT 找到中间 pivot)都必须经过 解压 → 操作 → 重压 三步。

7.6.3 LZF 压缩特性

表 7-6-1 · LZF vs zlib 压缩特性对比
维度LZF对比 zlib / gzip

LZF 选择 速度优先,压缩比次要 的策略,适合"频繁访问中间节点"场景。zlib 压缩比稍高但速度差 10 倍,Redis 主线程不能承受 — 这是 antirez 在 quicklist 设计中明确的取舍。

什么时候才该开 compress? 经验法则:
① List 长度 > 1 万 + 业务确认是 FIFO 队列(中间节点真的不读);
② 对单 key 内存敏感(headroom < 30%);
③ P99 业务侧能接受 +20 μs 的中间访问延迟。
否则保持默认 compress=0(不压缩),用更多内存换稳定延迟。

7.8.3 quicklist 的 5 个核心设计动机

Redis 选择 quicklist + listpack(7.0+)替代历史方案的核心动机:
① 分而治之根除 ziplist O(N²) — 每节点 8 KB 上限,单节点内修改的 memmove 永远在 KB 级,不会爆炸;
② 保持 O(1) 双端访问 — 头/尾节点未压缩,LPUSH/RPUSH/LPOP/RPOP 直接命中;
③ 可选 LZF 压缩 — 中间冷数据 70% 压缩比,长队列内存效率显著;
④ listpack 替代 ziplist 节点 — Redis 7.0 起节点内编码从 ziplist 改为 listpack,即使单节点内插入也不会触发连锁更新(已被节点大小限制,但 listpack 自描述更优雅);
⑤ 双端队列优先 — LIST 主场景是消息队列,LINSERT/LINDEX 中间是次要,设计权重明确。

7.9.3 反/正例对照

把 List 当数组随机读

# 50 万条日志放进单 List
127.0.0.1:6379> RPUSH logs (50万条日志)
(integer) 500000

# 业务里随机抽样查看(典型滥用)
127.0.0.1:6379> for i in 0..1000:
                  LINDEX logs $((RANDOM % 500000))

# 现象:
# - 每次 LINDEX ~5 ms (跨节点 + lpSeek)
# - 1000 次 ≈ 5 秒,主线程被打满
# - QPS 直接腰斩
# - SLOWLOG 全是 LINDEX

用 ZSet 按时间戳索引日志

# 日志按时间戳放 ZSet
127.0.0.1:6379> ZADD logs 1700000001 "log1"
127.0.0.1:6379> ZADD logs 1700000002 "log2"

# 范围扫描 O(log N + M)
127.0.0.1:6379> ZRANGEBYSCORE logs $start $end LIMIT 0 100

# 现象:
# - 每次 ZRANGEBYSCORE ~50 μs (跳表 + 100 元素遍历)
# - 100 倍速 + 主线程不被打满
# - QPS 稳定
quicklist 不是 O(1) 的随机访问数据结构。Redis 文档把 LIST 类型描述为 "linked list",这只是逻辑层面;物理实现是 quicklist,LINDEX 中间元素仍然是 O(N/K + K)。如果业务需要 O(1) 索引,用 Hash(key 是索引值)或专门维护一个 ZSet。

7.10.3 关键源码片段:quicklistPushHead

/* src/quicklist.c — LPUSH 核心实现
 * 输入: quicklist *quicklist 容器, void *value 元素值, size_t sz 字节数
 * 返回: 1 = 创建了新头节点, 0 = 复用原头节点
 * ★ 这是 LIST 类型 LPUSH 命令的物理实现入口
 */
quicklist *quicklistPushHead(quicklist *quicklist, void *value, size_t sz) {
    quicklistNode *orig_head = quicklist->head;

    if (likely(_quicklistNodeAllowInsert(quicklist->head, quicklist->fill, sz))) {
        /* 快路径: 头节点能容纳新值
         * lpPrepend 在 listpack 头部 1 次 memmove 后插入
         * 节点元数据更新(count / sz)
         */
        quicklist->head->entry =
            lpPrepend(quicklist->head->entry, value, sz);
        quicklistNodeUpdateSz(quicklist->head);
    } else {
        /* 慢路径: 头节点已满,创建新节点
         * 1. zmalloc 新 quicklistNode 结构
         * 2. lpNew(0) 创建空 listpack
         * 3. lpPrepend 把唯一元素 value 写入
         * 4. _quicklistInsertNodeBefore 4 个指针修改接入链表
         */
        quicklistNode *node = quicklistCreateNode();
        node->entry = lpPrepend(lpNew(0), value, sz);
        quicklistNodeUpdateSz(node);
        _quicklistInsertNodeBefore(quicklist, quicklist->head, node);
    }

    quicklist->count++;
    quicklist->head->count++;
    return (orig_head != quicklist->head);
}

/* ★ 关键不变量:
 *   1. 无论快/慢路径,O(1) 摊销不变
 *   2. quicklist->count 一定在最后 +1,保证 LLEN 一致性
 *   3. likely() 提示 CPU 预测分支:绝大多数 LPUSH 走快路径
 *   4. 不会触发整列表 memmove(quicklist 相对单一 listpack 的核心收益)
 */

7.10.4 设计取舍

取舍 1:fill 默认 -2(8 KB)vs 其他

8 KB 是经验最优:

  • 小于 4 KB:节点过多,链表元数据开销升高,LZF 压缩收益降低
  • 大于 16 KB:单节点 listpack 操作 O(N_node) 主导,L1 cache(典型 32 KB / core)不够装一整个节点 + 程序工作集
  • 8 KB 刚好让"节点元数据 + listpack 工作区"贴合 L1,平衡内存碎片与 cache 友好性

取舍 2:compress 默认 0

默认不压缩的理由:压缩需要业务确认"中间节点是冷数据"。开启 compress 后所有中间访问额外付 ~20 μs 解压成本。Redis 选择 "默认安全" — 业务知道场景才显式开启。这与 Redis 在很多地方体现的 "少做事 / 默认稳定" 哲学一致(同样体现在 intset 不降级、hashtable 转换不可逆、ziplist prevlen 不缩容等设计上)。

取舍 3:不实现 O(log N) 中间访问

技术上可用 B-tree / Rope / 跳表索引实现 LINDEX O(log N),但:

  • 实现复杂度增加 3-5 倍
  • 需维护索引元数据(每节点 +24 字节)
  • LIST 主场景(双端队列)不需要中间访问

Redis 选择 "诚实告知" — LINDEX 长列表慢,文档与 SLOWLOG 提示用户改用其他类型(ZSet 用 score 索引、Hash 用 key 索引、Stream 用 ID 索引)。

取舍 4:listpack 替代 ziplist 节点(7.0)

quicklist 已用节点大小限制了连锁更新的影响范围,但 ziplist 的 prevlen 设计在节点内仍可能触发(虽然单节点内的连锁限制在 8 KB 内)。换成 listpack 后:

  • 节点内编辑也不会有 prevlen 级联
  • 代码统一(Hash / ZSet / List 全用 listpack)
  • 维护成本降低,bug 面收敛

7.12 真实用例

用例 1:消息队列(订单系统)
某电商订单系统用 LIST 做异步处理队列:LPUSH orders:queue $order_json 由 Web 端写入,BRPOP orders:queue 30 由 worker 消费。规模:峰值 5 万 QPS,日均消息 10 亿。编码:quicklist(自动)。性能:LPUSH P99 1.6 μs,BRPOP 阻塞机制 + worker 池 ≈ 2 ms 端到端。关键设计:① fill 默认 -2,单节点 8 KB ≈ 30 条订单,worker 拉取速度足够避免堆积;② 不开 compress,因为是 FIFO,头/尾访问密集,中间节点几乎不存在(队列长度稳定 < 1000);③ 监控 redis_quicklist_count,超过 10 万触发扩 worker 数量。
用例 2:时间线 / 朋友圈(社交应用)
用户朋友圈用 LIST 缓存最新 500 条:LPUSH timeline:user:42 $post_id + LTRIM timeline:user:42 0 499。规模:1000 万活跃用户,平均每用户 LIST 长度 200,总元素 20 亿。编码:quicklist。读模式:LRANGE timeline:user:42 0 19(取首屏 20 条),P99 30 μs(单节点内)。写模式:LPUSH + LTRIM 批量写,P99 5 μs。内存:~30 GB(平均每用户 3 KB);如开 compress=1 可降到 ~18 GB,但业务方测试发现 LRANGE P99 +30 μs 影响首屏渲染,放弃压缩。
用例 3:任务栈(后台 Job 调度)
CI/CD 任务调度用 LIST 做 LIFO 栈:LPUSH jobs:stack $task_id + LPOP jobs:stack。规模:中等(峰值 1 万元素)。编码:quicklist。性能:LPUSH/LPOP P99 1.5 μs,无瓶颈。陷阱:某次产品想加"任务优先级提升",写了 LREM + LPUSH 把指定 task 移到栈顶。结果:LREM 是 O(N) 扫描,N=1 万时 P99 跳到 5 ms,栈操作 SLA 直接破。修复:改用 ZSet,score 充当优先级,ZADD GT 提升优先级 O(log N)。
用例 4:日志缓冲(链路追踪)
分布式追踪系统每个 trace_id 用 LIST 存若干 span:RPUSH trace:$trace_id $span,30 秒后由 flush worker LRANGE 0 -1 取走刷盘。规模:峰值百万 trace/分钟,平均每 trace 15 个 span。编码:quicklist 单节点(15 个 span 远小于阈值)。性能:RPUSH P99 1.5 μs,LRANGE P99 8 μs。关键设计:trace_id key 加 30 秒 TTL,LIST 不需手动清理;LRANGE 全量取出后 DEL,quicklist 整个释放,内存稳定。
用例 5:不要这么用 — LIST 当随机访问数组
反例:某 ML 团队把 50 万条训练样本放 LIST,业务每次 LINDEX list $((RANDOM % 500000)) 抽样。结果:每次 LINDEX ~5 ms(跨节点 + lpSeek),抽样 1000 次 = 5 秒主线程驻留,QPS 从 5 万降到 100。正解:① 把样本改用 Hash 存(HSET samples $idx $sample,O(1) HGET);② 或在客户端缓存样本数组,Redis 只存索引;③ 或换专门的样本存储(比如 OSS / 分布式文件系统)。教训:LIST 不是数组,LINDEX 不是 O(1) 访问。
用例 6:LZF 压缩的真实收益(大长队列)
某监控系统把 24 小时的指标点用 LIST 缓存:RPUSH metrics:$host $point,1 主机一 LIST,长度 ~86400(每秒一点)。规模:1 万主机 × 86400 元素 = 8.6 亿元素。不开 compress:总内存 ~300 GB,需要 5 个实例;开 compress=2(头尾各 2 节点 RAW,中间 LZF):总内存 ~120 GB,2 个实例搞定。访问模式:99% 是尾部追加 + 头部读最近 1 小时(LRANGE 0 3599),都落在 RAW 节点,P99 不受影响。关键:业务确认中间节点(几小时前的数据)真的不读,才敢开 compress。
