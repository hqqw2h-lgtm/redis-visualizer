第 22 章 · jemalloc 内存分配器

jemalloc 5.x · arena / bin / slab · tcache 线程缓存 · size class 量化 · 三层碎片指标 · Active Defrag · MEMORY PURGE · THP 与 BGSAVE CoW

Redis 在 Linux 发行构建中通常链接 jemalloc 5.x 而非 glibc malloc 或 tcmalloc;实际以 INFO memory 为准。该选择影响 String embstr 44 字节阈值附近的内存浪费、内存碎片表现、MEMORY USAGE 与 INFO memory 的输出语义、Active Defrag 的可行性,以及 fork BGSAVE 时的 Copy-on-Write 行为。

本章描述 jemalloc 的核心架构(size class、arena、bin、slab、tcache),给出 Redis 关心的内存指标计算公式,并讨论 Active Defrag、MEMORY PURGE、Transparent Huge Pages 对生产 Redis 的具体影响。读完本章,你应当能够看着 INFO memory 的 12 个字段说出每一项的含义、为什么 RSS 不会立刻随 DEL 下降、以及哪些参数在哪些场景下应该调。

  • jemalloc 与 glibc / tcmalloc 三种分配器在 Redis 长跑场景下的实测对比

  • tcache 线程缓存怎样把锁竞争从"每分配一次"降到"每 ~50 次一次"
  • embstr 阈值 44 为什么是 Redis 实现常量,以及它与 size class 浪费的关系
  • used_memory / allocator_allocated / active / resident / RSS 的层次关系
  • 三种碎片(内部 / 外部 / 元数据)的来源、检测指标与处理手段
  • Active Defrag 的 je_get_defrag_hint 钩子与 cycle CPU 占用控制
  • MEMORY PURGE 与 dirty_decay_ms 在归还 RSS 上的分工
  • THP 把 fork CoW 粒度从 4 KiB 升到 2 MiB 后 BGSAVE 内存放大 50-200% 的根源
  • 常用 MALLOC_CONF 参数与 MEMORY MALLOC-STATS 的解读

22.1 为什么选 jemalloc

分配器选型是 Redis 工程上"看不见但影响巨大"的一环。glibc 默认的 ptmalloc2 在长期运行场景下可能碎片较高;tcmalloc 表现接近 jemalloc 但缺少 Redis Active Defrag 所需的同等 hint;jemalloc 兼具低碎片 + 主动整理 + 公开统计能力,是 Linux 发行构建中最常见的选择之一,实际以 INFO memory 的 mem_allocator 为准。

22.1.3 Redis 编译时怎么链接 jemalloc

图 22-1-2 · Redis 内置 jemalloc 的编译流程
  Redis 源码 deps/jemalloc 内置 jemalloc 5.x

  Makefile 顺序:
    1. ./configure --prefix=$REDIS_BUILD/deps/jemalloc \
                   --with-jemalloc-prefix=je_       /* 函数前缀 je_ */
                   --enable-cc-silence
    2. make CFLAGS=-DJEMALLOC_NO_DEMANGLE
    3. 链接 -ljemalloc

  Redis 通过 zmalloc / zfree 等 wrapper 调用 jemalloc:
    void *zmalloc(size_t size) {
        void *ptr = je_malloc(size + PREFIX_SIZE);
        update_zmalloc_stat_alloc(je_malloc_size(ptr));
        return ptr;
    }

  Linux 发行构建通常使用 jemalloc(以 INFO memory 的 mem_allocator 为准);
  Windows 原生不具备 Redis 生产模型依赖的 fork 语义;
  自行编译可切换 libc malloc 或 tcmalloc(--with-tcmalloc)。
为什么 Redis 生产构建常选 jemalloc:不是因为 jemalloc 总是最快,而是因为 Redis 与 jemalloc 集成了 je_get_defrag_hint 这类碎片整理 hint — 它告诉 Redis "这个指针所在区域可能值得迁移"。glibc 与 tcmalloc 缺少同等钩子,所以无法提供同样的 Active Defrag 能力。

22.2 arena / bin / slab 架构

jemalloc 的核心结构是三层:arena(并发隔离单位,每线程 hash 到一个) → bin(同一 size class 的所有 slab) → slab(一段连续内存,切成若干 region)。这套结构同时解决了三个问题:并发(arena)、size class 量化(bin)、批量分配(slab)。

22.2.2 单个 slab 的字节布局

图 22-2-2 · slab 内部布局(以 size class = 32 B 为例)
  一个 slab = 1 个 4 KiB OS 页(典型情况):

  ┌──────────────────────────────────────────────┐
  │ slab metadata header (~ 32 B)                │  bitmap, ref count
  ├────────────┬────────────┬────────────┬───────┤
  │ region_0   │ region_1   │ region_2   │  ...  │  每 region = 32 B
  │   32 B     │   32 B     │   32 B     │       │
  ├────────────┼────────────┼────────────┼───────┤
  │ region_127 │            │            │       │
  └────────────┴────────────┴────────────┴───────┘
   合计 4 KiB - 32 B header ≈ 4064 B / 32 B = 127 个 region

  分配流程:
    1. zmalloc(30 B) → jemalloc 上调到 32 B size class
    2. 在 bin_32 找当前 partial slab
    3. slab bitmap 找空闲 region
    4. bitmap 设位、返回 region 地址
    5. 若 slab 满 → 移到 full list,bin 选下一个 slab

  free 流程:
    1. zfree(ptr) 通过指针定位所属 slab(jemalloc 用 radix tree 维护)
    2. bitmap 清位
    3. 若 slab 完全空 → 归还 OS 或保留备用

22.3 tcache 线程缓存

arena 已经把锁竞争降到了"每 N 线程一把锁",但每次 malloc 仍要拿 arena 锁。jemalloc 在此之上加了 tcache:每线程一个本地缓存,缓存少量 free region;malloc 时优先从 tcache pop,完全无锁。tcache 空了才向 arena 批量补 ~50 个,把锁次数压到 1/50。这是 Redis 单线程模型下 jemalloc 锁开销"几乎不可见"的根本原因。

22.3.2 tcache 减少锁竞争原理

图 22-3-2 · 有/无 tcache 的锁竞争对比
  无 tcache 场景(每分配走 arena bin):

   线程 A:malloc → 获 arena_0 锁 → bin → ... → 释放锁
   线程 B:malloc → 等 arena_0 锁 → ...
   线程 C:malloc → 等 arena_0 锁 → ...

  有 tcache 场景:

   线程 A 自己的 tcache:[free₁, free₂, ..., free_n]
     malloc → 直接 pop 一个,无锁
   线程 B / C 同理,各自 tcache 独立

  锁竞争只在 tcache 空时(每 ~50 次分配 1 次):
    线程 A 取 arena_0 拿 50 个补充
    锁持有时间 << 单次 malloc

  Redis 主线程几乎全部分配走 tcache,
  jemalloc 锁开销基本不可见。

22.4 size class 三层

jemalloc 把分配请求按大小分到不同 size class。小对象通常走 slab/bin,大对象走 extent,更大的对象可能使用独占 extent 或 mmap。具体边界随 jemalloc 版本、页大小和构建选项变化;阅读 INFO memory 时要先分清"Redis 申请了多少"与"分配器/OS 保留了多少"。

图 22-4-2 · 不同请求大小在 jemalloc 与 glibc 下的实际分配

    glibc malloc:可能返回 32 字节(下一个 16B 对齐)

22.4.3 embstr 阈值 44 的推导

图 22-4-3 · embstr 阈值 44 是怎么从 jemalloc 64 字节 size class 算出来的
  String embstr:redisObject(16B) + sdshdr8(3B) + buf + \0(1B)
                = 20 + len 字节

  Redis 设计目标:embstr 整体落入单个 size class,
                   保证一次 malloc 命中 cache line

  关键 size class:
    32, 48, 64, 80, 96, 128 (small classes)

  如果选 size class = 64 B(jemalloc bin_64):
    embstr 上限 = 64 - 20 = 44 字节
    → 字符串内容长度 ≤ 44 时用 embstr
    → 长度 > 44 升级 raw

  embstr 阈值 44 是 Redis 源码硬编码的实现选择;
  它常见情况下正好落入 64B class,但命令行为不会随分配器变化。

  Redis 在 src/object.c:
    #define OBJ_ENCODING_EMBSTR_SIZE_LIMIT 44

  若换用其他分配器,阈值仍是 44;变化的是实际分配浪费率。
为什么要"恰好填满 size class":embstr 把 redisObject + SDS 头 + 字符串内容 + 末尾 \0 拼成一段连续内存,调用 一次 malloc 即可分配。44 字节阈值让常见 64B size class 下的浪费最小;它是实现细节,不改变 String 命令的逻辑语义。

22.5 mem_fragmentation_ratio 指标

"内存碎片"是个被误用的词。Redis 的 INFO memory 给出了三个层次的碎片指标,每一层对应不同的碎片来源。读懂三个指标的层次关系,才能正确诊断"为什么 RSS 比 used_memory 高"。

22.5.1 三层指标的层次

图 22-5-1 · 从 OS RSS 到 used_memory 的四层视角
  Redis 内存指标的三层视角:

  +─────────────────────────────────────+
  │ OS RSS                              │  ← top / ps 看到的物理内存
  +─────────────────────────────────────+
                ▼ 减去 OS 页对齐损失等
  +─────────────────────────────────────+
  │ allocator_resident                  │  ← jemalloc 报告的物理驻留
  +─────────────────────────────────────+
                ▼ 减去 jemalloc 内部碎片
  +─────────────────────────────────────+
  │ allocator_active                    │  ← jemalloc 已分配但未释放
  +─────────────────────────────────────+
                ▼ 减去 jemalloc 元数据
  +─────────────────────────────────────+
  │ allocator_allocated ≈ used_memory   │  ← Redis 申请字节,含对象开销
  +─────────────────────────────────────+

  比率含义:
    mem_fragmentation_ratio = RSS / used_memory
    allocator_frag_ratio    = allocator_active / used_memory
    allocator_rss_ratio     = allocator_resident / allocator_active
    allocator_frag_smallbins_ratio = small bins 内部 / 已分配

22.5.2 mem_fragmentation_ratio 直观读图

图 22-5-2 · mem_fragmentation_ratio 各取值范围的诊断结论
< 1.0 不是"好":它意味着 OS 已经把部分内存换出到 swap,Redis 的实际"驻留"小于"已分配",这是性能灾难。看到 mem_fragmentation_ratio < 1 应当立即检查 vm.swappiness、cgroup 限制、是否有其他进程抢内存。

22.6 allocator_frag_ratio:内部碎片

22.6.1 内部 vs 外部碎片

图 22-6-1 · 内部碎片(size class 量化) vs 外部碎片(slab 局部空闲)
  内部碎片(allocator_frag):
    分配 30 B → jemalloc 给 32 B size class
    多出来的 2 B 在该 region 存活期间用不上

    场景:
    - 写入大量 28~32 B 字符串(都被舍入到 32)
    - 写入大量 1.2 KB 数据(都被舍入到 1280)

    指标:allocator_frag_ratio
      = active / allocated
      理想 = 1.0,实际 1.05~1.10 是正常上界

  外部碎片(allocator_rss):
    slab 中删除 70% region 后剩 30% region
    slab 整体不能归还 OS,即使 70% 闲置

    场景:
    - 大量短生命周期 key 频繁创建/删除
    - 同 size class 的对象生命周期不一

    指标:allocator_rss_ratio
      = resident / active
      理想 = 1.0,实际 1.10~1.30 在长期运行后正常

22.6.2 slab 局部空闲示意

图 22-6-2 · 同一 bin 下 4 个 slab 的占用率分布
  bin_32 对应的 slab 序列(每 slab 127 个 32B region):

  slab_0:  [█████████████████████████████]   全满
  slab_1:  [█████████░░░░░░░░░░░░░░░░░░░░]   30% 用 70% 空
  slab_2:  [██░░░░░░░░░░░░░░░░░░░░░░░░░░░]   10% 用 90% 空
  slab_3:  [░░░░░░░░░░░░░░░░░░░░░░░░░░░░░]   全空 → 可归还 OS

  slab_1, slab_2 因为有部分 region 还在用,
  整 slab 不能归还 OS。

  解决方法:
    1. Active Defrag:在线迁移 region 到其他 slab,
       让 slab_1/slab_2 完全空,再归还 OS
    2. MEMORY PURGE:让 jemalloc 立即检查并归还空 slab
    3. 重启:全量内存归还(运维成本最高)

22.7 allocator_rss_ratio:外部碎片与归还

22.7.1 RSS 不归还 OS 的原因

图 22-7-1 · DEL 后 jemalloc 决定是否归还 OS 的决策树

jemalloc 默认延迟归还以避免频繁 mmap/munmap 系统调用。代价是短期内 RSS 不下降 — 这是 "DEL 完了 RSS 没动" 的常见原因,不是 bug。

22.7.2 dirty page decay 机制

图 22-7-2 · jemalloc 5.x dirty page 衰减时间线
  jemalloc 5.x 引入 dirty page decay 机制:

  时间线:
    t=0    DEL 大批 key,大量 region 释放
    t=0~10s  jemalloc 将可归还 page 放入 decay 队列
    t≈10s    decay 到期后 purge,通过 madvise/munmap 归还 OS
    t>10s    RSS 是否下降还取决于是否存在可完整归还的 extent/page

  decay 时间由 opt.dirty_decay_ms 控制:
    默认 10000 ms(10 秒)
    可调到 0 表示立即归还(代价是高频 syscall)

  Redis 编译时设置:
    deps/jemalloc/Makefile 中:
      JE_DECAY=10000
      ./configure --with-malloc-conf=dirty_decay_ms:10000

22.8 Active Defrag

Active Defrag 是 Redis 4.0 引入的"在线碎片整理"功能。它借助 jemalloc 的 defrag hint 找出迁移收益较高的对象,搬到更紧凑的位置,让原 slab/extent 更可能空出来。它解决的是"仍有活对象导致无法归还"的碎片;若只是 dirty page 尚未 purge,MEMORY PURGE 也可能降低 RSS。

22.8.2 Redis 怎么用 jemalloc 的 hint

图 22-8-2 · je_get_defrag_hint API 与 Redis 迁移逻辑
  Redis 的 jemalloc 构建可提供 je_get_defrag_hint:
    int je_get_defrag_hint(void *ptr);
    返回值:
      0 = 该 ptr 所在 slab 几乎满,迁移收益低
      1 = 该 ptr 所在 slab 较空,迁移可让 slab 释放

  Redis 利用此 hint:
    遍历 keyspace dict 找 hint = 1 的对象
    重新 zmalloc 分配 + memcpy 数据
    更新 dict / list / set / zset / hash 中的所有指针引用

  代价:
    - 每对象迁移有锁/拷贝开销
    - 指针更新需要遍历所有引用方
    - active-defrag-cycle-min ~ max 控制 CPU 占用比

  收益:
    在线整理碎片,RSS 显著下降
    无需重启

22.10 THP 与 fork CoW

Transparent Huge Pages(THP)是 Linux 内核自动把连续 4 KiB 页合并为 2 MiB 页的特性,目标是减少 TLB miss。但 Redis 在 BGSAVE 时走 fork(2) + Copy-on-Write,THP 把 CoW 粒度从 4 KiB 升到 2 MiB,直接导致 BGSAVE 期间 RSS 放大 50-200%。这是 Redis 文档反复强调"关闭 THP"的根本原因。

22.10.1 THP 怎么影响 fork CoW

图 22-10-1 · THP 启用前后 fork CoW 的内存放大对比
  Linux Transparent Huge Pages (THP):
    标准页大小 4 KiB
    THP 页大小 2 MiB(标准 huge page)

  THP 启用时,内核自动把连续 4 KiB 页合并为 2 MiB 页:
    优势:TLB miss 减少,大块内存访问性能高
    劣势:Copy-on-Write 粒度从 4 KiB 升到 2 MiB

  fork BGSAVE 场景:
    主进程 fork 出子进程
    Linux 用 CoW:fork 后内存共享,任一方写时复制对应页

  无 THP:
    主进程修改 1 字节 → 复制 4 KiB 页
    1 GB 内存修改 1% → 复制 ~10 MB

  有 THP:
    主进程修改 1 字节 → 复制 2 MiB 页
    1 GB 内存修改 1% → 复制 ~512 MB(50 倍放大)

  Redis 强烈建议关闭 THP:
    echo never > /sys/kernel/mm/transparent_hugepage/enabled

/* src/zmalloc.c — 简化 */
static __thread size_t used_memory_thread = 0;

void *zmalloc(size_t size) {
    void *ptr = je_malloc(size + PREFIX_SIZE);
    if (!ptr) zmalloc_oom_handler(size);

    /* 记录实际分配大小 */
    *((size_t*)ptr) = size;
    update_zmalloc_stat_alloc(zmalloc_size(ptr));
    return (char*)ptr + PREFIX_SIZE;
}

size_t zmalloc_size(void *ptr) {
    void *real = (char*)ptr - PREFIX_SIZE;
    return je_malloc_size(real);
    /* je_malloc_size 返回 jemalloc 实际分配大小
       (含 size class 量化的字节) */
}

void zfree(void *ptr) {
    if (ptr == NULL) return;
    void *real = (char*)ptr - PREFIX_SIZE;
    update_zmalloc_stat_free(je_malloc_size(real));
    je_free(real);
}

22.13.4 设计取舍

取舍 1:jemalloc 而非 tcmalloc / glibc

Redis 使用 jemalloc 构建时可获得 je_get_defrag_hint 这类整理 hint,这是 Active Defrag 的基础。glibc 与 tcmalloc 缺乏类似 hook,因此无法提供同等在线碎片整理能力。

取舍 2:tcache 默认启用

线程缓存以约 1 MB / 线程的内存换取无锁分配。Redis 主线程模型下 tcache 内存浪费仅 ~1 MB,远小于性能收益。

取舍 3:dirty_decay_ms 默认 10 秒

立即归还会引发频繁 syscall;永久保留浪费内存。10 秒在两者间平衡:短期波动不归还(避免抖动),长期过剩归还。

取舍 4:Active Defrag 默认关闭

整理消耗 CPU,大多数业务无明显碎片问题。仅在 mem_fragmentation_ratio 与 allocator 指标持续显示活对象碎片时启用,且通过 active-defrag-cycle-max 控制 CPU 占用上限。
