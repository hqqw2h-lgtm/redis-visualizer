第 34 章 · ACL 与 TLS

多用户访问控制 · 命令位图 O(1) 校验 · key/channel 模式 · TLS 1.2/1.3 握手 · mTLS · I/O threads 卸载 · 全 Cluster TLS

Redis 6.0 之前的安全模型由三个配置项构成:requirepass 全局密码、bind 网络绑定、protected-mode 默认拒外。该模型在身份识别、权限粒度、传输保密三个维度均不满足现代部署要求:所有客户端共享同一密码无法追溯、密码通过即获得全部命令权限、连接全程明文传输。

ACL(Access Control List)和 TLS 是 Redis 6.0 引入的两项安全基础设施。ACL 在命令执行前置入用户、命令位图、key 模式、channel 模式四级权限校验;TLS 在 socket 层之上嵌入 OpenSSL 握手与会话保护。两者覆盖的边界互补:TLS 保护"传输中数据是否可被读取或篡改",ACL 保护"通过传输的请求能否执行特定命令"。

  • Pre-ACL(requirepass)与 ACL 在身份、粒度、半径、审计、持久化五个维度的差异

  • 命令执行的四级权限校验决策树:认证 → 命令位图 → key 模式 → channel 模式
  • 命令位图为何能做到 O(1) 校验,以及为何 key / channel 仍为 O(P × len)
  • ACL SAVE / ACL LOAD / aclfile 持久化路径,以及"运行时改 + 不 SAVE"的典型陷阱
  • Redis 7.0 引入的 %R~ / %W~ 读写分离 key 模式,与命令规约的 readonly / write 标志如何联动
  • TLS 配置十二项:port 禁用、tls-port、证书、CA、auth-clients、protocols、ciphersuites、replication、cluster
  • tls-auth-clients no/optional/yes 三档对 mTLS 演进的影响
  • TLS 1.2(2 RTT)与 TLS 1.3(1 RTT)握手轮次差异及对短连接的影响

  • 七大典型陷阱:default 残留、仅 requirepass、未 SAVE、证书过期、Early Data 重放、Cluster TLS 混合、ACL 不同步

34.0 章首速览

核心结论: ACL 与 TLS 在 Redis 6.0 共同引入,但解决两个独立问题:ACL 解决"谁能执行什么",TLS 解决"线路上谁能看到与篡改"。生产部署必须两个都启用—— 仅 ACL 不防嗅探,仅 TLS 不防越权。每条命令在主线程内做四级校验,任何一级失败即写 ACL LOG;TLS 通过 connection 抽象层与明文路径共享 read/write 接口。
典型事故: 从 5.x 升到 6.x 后,如果只保留 requirepass 思路而不拆分 ACL 用户,旧客户端仍可能用共享密码获得过大权限。教训:先创建并验证专用用户和管理员用户,再禁用或收紧 default 用户,避免把自己锁在实例外。

34.1 Pre-ACL 与 ACL 模型对比

Redis 5.x 及更早版本只通过 requirepass 设置全局密码,任何认证客户端均获得完整命令集。该模型缺失身份识别、权限分层与审计能力。

图 34-1-1 · 两代安全模型的能力差异

Pre-ACL 与 ACL 的对比矩阵

维度Pre-ACL(共享密码)ACL(Redis 6.0+)
身份识别全部客户端共享同一密码独立用户 + 独立密码
权限粒度全有或全无命令级 + key 模式 + channel 模式
泄露影响半径100%(全部权限)单用户范围
审计能力无ACL LOG 记录拒绝事件

持久化写入 redis.confACL SAVE 写入 aclfile

图 34-1-2 · 权限粒度递进

ACL 采用扁平模型:用户与规则一对多,无角色继承层级。该选择的工程依据是命令热路径的权限检查必须保持近似 O(1),复杂的策略评估会拖慢主线程。组织层级权限管理由外部 IAM / 配置生成器承担。

34.2 ACL 规则语法

ACL 规则由若干以空格分隔的描述符构成,每条描述符以单字符前缀标识其语义。规则字符串作为 ACL SETUSER 命令的参数,或写入 aclfile 由启动时加载。

34.3 权限校验决策链

客户端发送命令后,Redis 在 processCommand 阶段进行四级校验:认证状态、命令位图、key 模式、channel 模式。任何一级失败即返回错误,并写入 ACL LOG。

图 34-3-1 · 四级权限校验决策树

图 34-3-2 · 命令位图的内存组织(O(1) 位运算)

位图存储在 aclSelector 结构体内。每个用户至少有一个 selector,新增 selector 通过 ACL SETUSER 的 (...) 语法附加,适合"读特定前缀 + 写另一前缀"等组合场景。

图 34-3-3 · 命令执行路径中的 ACL 拦截位置

/* src/server.c — processCommand 中的 ACL 拦截 */
if (ACLCheckAllPerm(c, &acl_errpos) != ACL_OK) {
    addACLLogEntry(c, ACL_DENIED_CMD, acl_errpos, c->cmd);
    rejectCommandFormat(c, "NOPERM this user has no permissions");
    return;
}

34.6 TLS 配置要素

明文 Redis 通信中,所有 RESP 帧(包括 AUTH 命令)以 ASCII 形式在 TCP 段中传输。在共享网络中可被嗅探或篡改。

图 34-6-1 · 明文与 TLS 通信路径对比

TLS 配置要素

配置项用途典型值
port 0禁用明文端口0
tls-port启用 TLS 监听端口6380
tls-cert-file服务端证书/etc/redis/tls/redis.crt
tls-key-file服务端私钥/etc/redis/tls/redis.key
tls-ca-cert-fileCA 证书(校验客户端)/etc/redis/tls/ca.crt
tls-auth-clients客户端证书校验级别optional / yes / no
tls-protocols允许 TLS 版本"TLSv1.2 TLSv1.3"
tls-ciphersuitesTLS 1.3 密码套件TLS_AES_256_GCM_SHA384...
tls-replication主从复制启用 TLSyes
tls-cluster集群总线启用 TLSyes
port 0 + tls-port 6380 组合后,明文客户端无法通过端口绕过 TLS。该约束是生产强制项,任何"双端口共存"配置均放大攻击面。

34.7 TLS 握手时序

TLS 1.2 与 TLS 1.3 的握手轮次差异直接影响新建连接的延迟。

图 34-7-1 · TLS 1.2 vs TLS 1.3 握手对比

握手版本能力对比

维度TLS 1.2TLS 1.3
握手轮次2 RTT1 RTT
密钥交换RSA / DHE / ECDHEECDHE / DHE 强制前向保密
密码套件含 RC4、CBC 等弱算法仅 AEAD(AES-GCM、ChaCha20-Poly1305)
0-RTT(Early Data)不支持协议支持,Redis 部署通常不应依赖
会话恢复Session ID / TicketPSK 模式
弃用算法SHA-1、MD5 等强制移除
0-RTT Early Data 由攻击者重放时,服务端无法区分原始请求与重放请求。Redis 自身不需要依赖 0-RTT;如果前置 TLS 终止器或代理启用了 Early Data,不要让 INCR、DEL 等非幂等写命令通过这条路径。

34.8 TLS 性能开销

TLS 引入握手延迟、对称加密 CPU 开销、记录层缓冲。在 Redis 单线程模型中,加密路径与命令执行竞争同一 CPU 周期。

典型环境下的吞吐对比

模式SET QPSGET QPSP99 延迟相对基准
无 TLS(基线)185,000198,0000.3 ms100%
TLS 1.2(AES-256-GCM)112,000125,0000.5 ms63%
TLS 1.3(AES-256-GCM)128,000142,0000.4 ms72%
TLS 1.3(ChaCha20-Poly1305)135,000148,0000.4 ms75%
TLS 1.3 + 长连接复用140,000152,0000.35 ms77%

测试条件:Redis 7.2、Xeon E-2278G 4 核、redis-benchmark -c 50 -n 1000000、同机房 0.1 ms RTT。AES-NI 硬件加速下 AES-GCM 略优;ARM 无硬件 AES 时 ChaCha20-Poly1305 反超。

图 34-8-1 · TLS 加密路径的 CPU 占用分布(100B payload)

吞吐下降的主要来源

来源占比估算缓解手段
对称加密 CPU5-10%使用 AES-NI 硬件加速或 ChaCha20

I/O threads 将 TLS 读写从主线程剥离到辅助线程,主线程仅负责命令执行。该模式在网络读写占比高的场景(高 QPS、大 payload)收益明显;在命令计算占比高(Lua、Sort)时收益有限。开启前需基于压测数据决定 io-threads 数量。

masteruser 指向主节点 ACL 中供副本使用的专用用户,至少要允许复制握手相关命令(如 +psync、+replconf、+ping)。masterauth 提供该用户的密码。

Redis ACL 与 AWS IAM 共享"白名单 + 模式匹配"模型。IAM 支持条件键(Condition)和策略组合,Redis ACL 不支持。该差异源于 Redis 的命令执行必须在主线程内完成权限决策,引入条件评估会拖慢热路径。

/* src/acl.h — aclSelector 与 user 结构 */
struct aclSelector {
    uint32_t flags;                                          // selector 全局标志
    uint64_t allowed_commands[USER_COMMAND_BITS_COUNT/64];   // 命令位图
    list *patterns;                                          // key 模式列表
    list *channels;                                          // channel 模式列表
    sds command_rules;                                       // 原始规则字符串
};

struct user {
    sds name;
    uint32_t flags;                  // on/off、nopass 等
    list *passwords;                 // 哈希密码列表
    list *selectors;                 // 多 selector(默认 1 个)
};
取舍 1:命令位图 vs 角色继承
ACL 选择固定大小的命令位图,每条命令的检查为单次位运算。相比 PostgreSQL 的角色继承图,该设计放弃了组织层级抽象,换取主线程权限检查的近似 O(1) 复杂度。组织层级权限管理由外部 IAM / 配置生成器承担。
取舍 2:OpenSSL 集成 vs 自实现 TLS
Redis 直接使用 OpenSSL,通过 connection 抽象层将明文与 TLS 读写统一。获得成熟密码学实现与持续安全更新,代价是 OpenSSL 升级时需要重新编译 Redis。connection 抽象层使得未来切换到 BoringSSL / wolfSSL 的工作量限定在适配层。
取舍 3:Cluster TLS 全有或全无
Cluster 不支持 TLS 与明文节点混合,源于 Gossip 协议在节点首次发现时无法协商加密能力。混合模式会引入"先尝试 TLS 失败再降级"的复杂逻辑,放大攻击面。强制全节点统一的代价是滚动升级需要预先在所有节点配置证书。
