第 11 章 · Postings 与跳跃迭代

.doc / .pos / .pay · docID delta · PackedInts · 128 doc/block · 多级 skip · advance · BlockMax impacts

第 10 章用 FST 把"找 term"变成 O(log N) 内存查找。本章接力下一棒:term 找到之后,如何高效地遍历它命中的几百万个 docID?如果裸存 int[9_500_000] 直接占 36 MiB,而且想"跳到 docID ≥ 50000"还得线性扫——这不是工程级能用的方案。Lucene 的答案是 delta 编码 + PackedInts + 128-doc block + 多级 skip data 四件套,把 1 千万 doc 的 postings 压到 ~5-6 MiB,同时让 advance(target) 在 1 亿 doc 上稳定保持 O(log freq) 复杂度。

具体一点:布尔求交的 leapfrog 算法本质就是 scorer.advance(target) 的高频反复调用——求交 5 个子句、Top 20 命中,平均每个 doc 要 4-5 次 advance 跳跃。如果 advance 是 O(N),整条查询路径就是 O(N²) 立刻爆炸。所以"advance 是 O(log freq)"不是性能优化,而是布尔求交存在的必要前提。本章把这条 O(log) 的链路拆开:每 128 docs 一个 block 的物理边界、PackedInts 紧凑整数的 SIMD 解码、三级 skip data(128/1024/8192)的金字塔结构、advance 从 L2→L1→L0→block-内线性扫的四级状态机,以及让 BM25 TopK 提速 30-60% 的 BlockMax impacts 上界跳过算法。读完之后,你就能从源码层面回答"为什么 Lucene 1 千万订单的 P95 是 5ms 而不是 5 秒"。

  • .doc 文件的物理布局:CodecHeader + 按 term 序串接的 postings 段 + CodecFooter,每段含若干 128-doc block + vint tail + skip data
  • docID delta 编码:相邻 docID 差值的统计平均远小于原值,1 千万高频 term 的 delta 95% < 256,bytes/doc 从 4 字节压到 0.5-0.8 字节
  • PackedInts 紧凑整数:每 block 自适应选 bitsPerValue(覆盖 max delta),128 个 long 用统一位宽打包,SIMD-friendly
  • 128 docs/block 的固定边界:为什么是 128(=2⁷,L1 cache 友好,SIMD 对齐),block 内禁止跨界访问
  • freq / positions / payloads / offsets 在 .doc/.pos/.pay 三文件的分流:按 IndexOptions 决定哪些写入,查询按需打开
  • 多级 skip data 的金字塔:level 0 每 128 docs(=1 block),level 1 每 1024 docs(=8 blocks),level 2 每 8192 docs(=64 blocks)
  • advance(target) 的四级状态机:L2 二分 → L1 二分 → L0 二分 → block 内线性扫,保证 O(log freq)
  • BlockMax impacts:每个 block 在 skip 元数据中嵌入 (maxFreq, minNorm) 上界,WAND 用它直接跳过低分块,P95 降 30-60%

前置知识:第 1 章的倒排索引概念(.doc / .pos / .pay 三文件分层)、第 4 章 leapfrog 求交算法(ConjunctionDISI.intersect 的短表 next + 长表 advance)、第 10 章 FST 词典(.tip + .tim 找到 term 之后才进入本章的 postings 段)。本章对 PostingsEnum 的物理形态做最深的剖面图,后续第 12 章讲 BKD-tree 做数值范围,与本章的 postings 完全独立。

11.1 倒排列表的需求:从逻辑到物理

"倒排索引"在数据结构教科书上的样子大概是 Map<Term, ArrayList<Integer>>——一个 HashMap,key 是 term,value 是 docID 列表。这个抽象在白板上很漂亮,但放到 1 千万 doc 的真实索引上立刻暴露三个致命缺陷:内存占用爆炸、advance 退化为线性扫描、序列化反序列化成本极高。Lucene 把这三个缺陷一次性解决,代价是引入了一个看似复杂的 delta + PackedInts + block + skip 四件套——本节先把"为什么需要这四件套"讲清楚,后面几节再逐个拆解实现。

"docCount marker = 0x80"是协议的隐式跳出条件。Reader 在读 block 时先读 1 byte:如果是 0x80(高位 1 + 低位 0),就知道这是满 block,接下来读 PFOR 编码的 128 个 delta;如果不是 0x80,Reader 切换到 vint tail 解码模式,逐 doc VInt 解码直到 docFreq 总数耗尽。这种"用第一个字节切换解码模式"的设计,既避免了在 term metadata 里额外存"block 数 / vint tail 长度",又让 reader 状态机简单。

.doc 文件的"字段邻接性"原则

注意一个细节:同一个 block 内,先写 128 个 delta,再写 128 个 freq——两个数组分别紧凑打包,而不是(delta_0, freq_0, delta_1, freq_1, ...)交错。原因:(1) 仅遍历 docID 时(BlockDocsEnum 不需要 freq),可以跳过整个 freq 块;(2) SIMD 解码 128 个 long 必须连续,交错会破坏向量加载;(3) bitsPerValue 对 delta 和 freq 通常不同,分开打包能各取最优位宽。这个设计是 PFOR-delta 高效解码的前提。

Lucene 99 的 ForDeltaUtil:解码 + 累加合一

在 Lucene 9.5+,ForDeltaUtil.decodeAndPrefixSum 把"PackedInts 解码 128 个 delta"和"prefix-sum 累加恢复绝对 docID"合并成一个 SIMD 流水线,使用 Java 21 Vector API。这避免了"先解码再累加"两次遍历 128 个 long 的成本,把单核解码吞吐从 80 M docs/s(VInt 时代)推到 350-500 M docs/s。

public final class ForDeltaUtil {
    private static final VectorSpecies<Long> SPECIES = LongVector.SPECIES_256;
    private static final int LANE = SPECIES.length();

    public void decodeAndPrefixSum(int bitsPerValue, DataInput in,
                                    long base, long[] longs)
            throws IOException {
        decode(bitsPerValue, in, longs);
        prefixSumOfBlock(base, longs);
    }

    private static void prefixSumOfBlock(long base, long[] longs) {
        for (int i = 0; i < longs.length; i += LANE) {
            LongVector v = LongVector.fromArray(SPECIES, longs, i);
            v = inLanePrefixSum(v);
            v = v.add(base);
            v.intoArray(longs, i);
            base = longs[i + LANE - 1];
        }
    }
}
SIMD prefix-sum 的实战收益 — 在 Elasticsearch 7.x(Lucene 8.x,纯标量解码)上,title:the 这种全索引扫描 1000 万 doc 的查询单核要 25-40ms;升级到 ES 8.x(Lucene 9.5+,SIMD prefix-sum)后,同样的查询 8-12ms,加速 ~3×。这个收益不需要任何应用层改动,纯靠 Lucene 内部用 Vector API 重写解码循环。但要求 JVM ≥ 21(早期 Vector API 在 17 是 Incubator,行为不稳定)。

11.4 PackedInts 紧凑整数:为什么不是固定 32-bit

PackedInts 是 Lucene 一切紧凑整数存储的底层抽象——不仅 .doc 用,DocValues 的 NUMERIC、SORTED 类型,FST 的 arc 输出,BKD-tree 的叶子节点统统用它。本节聚焦 PackedInts 在 postings 中的角色:为什么每 block 一个 bitsPerValue,而不是固定 32-bit 也不是 VInt。

PackedInts 的核心思想

11.8 nextDoc 算法:block 内顺序解码

advance 用于跳跃,nextDoc 用于顺序遍历。两者在 Lucene 内部走的代码路径有重要差异:nextDoc 大部分时间都在 block 内部增量,只在 block 边界触发 refillDocs;而 advance 通常需要 skip 跳级 + refillDocs 双开销。本节展开 nextDoc 的实现,并对比两者的复杂度模型。

BlockDocsEnum.nextDoc 的核心实现

@Override
public int nextDoc() throws IOException {
    if (docBufferUpto == BLOCK_SIZE) {
        refillDocs();
    }

    accum += docBuffer[docBufferUpto];
    freq = (int) freqBuffer[docBufferUpto];
    doc = (int) accum;
    docBufferUpto++;
    return doc;
}

private void refillDocs() throws IOException {
    final int left = docFreq - docUpto;
    if (left >= BLOCK_SIZE) {
        forDeltaUtil.decodeAndPrefixSum(docIn, accum, docBuffer);
        forUtil.decode(freqIn, freqBuffer);
        docBufferUpto = 0;
        docUpto += BLOCK_SIZE;
        accum = docBuffer[BLOCK_SIZE - 1];
    } else {
        readVIntBlock(docIn, docBuffer, freqBuffer, left);
        docBufferUpto = 0;
        docUpto += left;
    }
}

11.9 BlockMax impacts:让 TopK 跳过整块低分

11.11 物理大小估算:1 千万订单的 .doc 多大

设计搜索系统时一个反复出现的问题:"我有 1 千万订单 + 高频商品名,索引会多大?"——能从字段类型、df 分布、IndexOptions 推算出 .doc 文件的大致体积,是容量规划的基本功。本节给出三个真实场景的估算 + 推导过程。

容量推导通式

单 term postings 大小估算:

  full_blocks = floor(df / 128)
  vint_tail   = df - full_blocks × 128

  block_bytes = full_blocks × (
      1                                // bitsPerValue (delta) 头
    + ceil(128 × bitsPerValue_delta / 8)  // PackedInts delta
    + 1                                // exception count
    + exception_count × 1.5             // exception (idx + high_byte)
    + 1                                // bitsPerValue (freq) 头
    + ceil(128 × bitsPerValue_freq / 8)   // PackedInts freq
  )

  tail_bytes  = vint_tail × (avg_vint_delta + avg_vint_freq)
              ≈ vint_tail × 2.5

  skip_bytes  = (df / 128) × 12        // L0 entries
              + (df / 1024) × 16       // L1 entries
              + (df / 8192) × 20       // L2 entries

  total = block_bytes + tail_bytes + skip_bytes

参考量级(典型英文中频 term):
  bitsPerValue_delta ≈ 5-7
  bitsPerValue_freq  ≈ 3-4
  exception_count    ≈ 3-8 / block

enum 复用的最佳实践

try (DirectoryReader reader = DirectoryReader.open(dir)) {
    LeafReader leaf = reader.leaves().get(0).reader();
    Terms terms = leaf.terms("content");
    TermsEnum te = terms.iterator();

    PostingsEnum pe = null;
    while (te.next() != null) {
        pe = te.postings(pe, PostingsEnum.FREQS);
        while (pe.nextDoc() != DocIdSetIterator.NO_MORE_DOCS) {
            process(pe);
        }
    }
}

把 pe 作为参数传回 te.postings(pe, ...),Lucene 会复用同一对象,避免每个 term 分配新 enum——百万级 term 扫描场景下能减少 90% 的 GC 压力。

flag 必须与 IndexOptions 兼容。如果索引时字段配置 IndexOptions=DOCS_AND_FREQS(没有 positions),查询时 flag=POSITIONS 不会报错,但 nextPosition() 永远返回 -1 或抛异常。这种 mismatch 在 PhraseQuery 上尤其致命:看似查询能跑,实际所有 doc 都不命中。生产里务必让索引 IndexOptions 与查询 flag 配套规划——通常 IndexOptions 由字段建模决定,flag 由查询类型自动推导(如 PhraseQuery 自动用 POSITIONS)。
