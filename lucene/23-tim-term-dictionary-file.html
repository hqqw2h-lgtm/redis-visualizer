第 23 章 · .tim 词典数据文件

BlockTree · 共享前缀 · 25-48 term/block · sub block 嵌套 · singletonDocID · docStartFP delta · 跨文件指针 · 大词典 off-heap FST

.tim 文件保存 BlockTree term dictionary 的数据区。所有 term 按字典序排列后切分成定长 block,每个 block 含若干 term 与对应的 postings 指针(.doc/.pos/.pay)、统计信息(docFreq、totalTermFreq)。.tim 的索引由 .tip 提供:.tip 用 FST 把 term 的前缀映射到 .tim 中的 block 偏移。

本章用字节级视图拆解 .tim 的物理布局。每个 block 由 prefix + suffixes + term metadata 组成。.tim 是倒排索引的"目录"——查询 term 时先经 .tip 定位 block,再扫 block 中的 suffix,匹配后从 metadata 跳到 .doc/.pos/.pay 读 postings list。

  • BlockTree 切 block 的核心思想:term 排序 → 共享前缀 → suffix 列表
  • Term metadata 9 个字段:suffix / docFreq / totalTermFreq / docStartFP / singletonDocID / skipOffset / posStartFP / payStartFP / lastPosBlockOffset
  • 共享前缀的字节节省:典型 1.6-4.9× 压缩比,技术词典可达 5×
  • .tim 文件 4 段结构:Header / Field Block Sequences / FieldsMetadata / Footer
  • BlockTreeTermsWriter 的 PendingBlock 缓冲机制与 sub block 触发条件
  • seekExact 4 步流程:.tip FST lookup → .tim block 头 → 二分扫 suffix → 读 stats + metadata
  • singletonDocID 长尾词内联优化(docFreq=1 时省 1 次 I/O)
  • docStartFP delta encoding 节省 1.5× 字节
  • 高基数字段(user_id/session_id/URL)词典爆炸的避免方法:hash + Numeric
  • 大词典(>1M term)Lucene 9.x off-heap FST + vmtouch 预热

前置知识:第 1 章字段建模(StringField vs TextField);第 4 章 BM25 评分(理解 docFreq / totalTermFreq 的语义);第 11 章 Postings + SkipList(知道 .doc / .pos / .pay 三个文件的内容)。读者需要熟悉 VLong/VInt 编码与字典序排序。

23.1 BlockTree 算法

查询一个 term 的 postings,直觉做法是哈希:HashMap<String, PostingsList>。但磁盘上不能用哈希(随机 I/O 全是 page miss),且哈希无法做范围查询。Lucene 的答案是 BlockTree:term 排序 + 切 block + 前缀共享 + sub block 嵌套。这一节先讲核心思想。

23.1.1 BlockTree 核心思想

三个层级:
  1. 排序:所有 term 按字典序(byte 级 ascending)排好。
  2. 切 block:每 25-48 个 term 一组,组成 block。
  3. 共享前缀:block 内 term 都有相同前缀 → 前缀放在 .tip FST 节点上,block 内只存 suffix(去前缀的尾部)。
图 23-1-1 · BlockTree 切分思想
所有 term 按字典序排序:
  apple
  application
  applied
  apply
  banana
  banner
  band
  cherry
  ...

按定长 (默认 minBlockSize=25) 切 block:

  Block 1 (前缀 "appl"):
    apple        → suffix "e"
    applied      → suffix "ied"
    application  → suffix "ication"
    apply        → suffix "y"
    ...

  Block 2 (前缀 "ban"):
    banana       → suffix "ana"
    band         → suffix "d"
    banner       → suffix "ner"
    ...

  Block 3 (前缀 "cher"):
    cherry       → suffix "ry"
    ...

★ 每个 block 共享 prefix:
   prefix 写在 .tip FST 节点
   block 内只存 suffix (差量)

★ 如果某 prefix 下 term 数 > 48 (maxBlockSize):
   嵌套 sub block (BlockTree 多层)

23.1.4 sub block 嵌套

某前缀下 term 数 > 48 时,触发 sub block 嵌套。sub block 用更长前缀进一步切分,形成多层 BlockTree。

图 23-1-3 · sub block 嵌套示意(以前缀 "java" 为例)
前缀 "java" 下 100 个 term (超过 maxBlockSize=48):

  原本期望 1 个 block, 实际切成 sub block:

  parent block "java" (numEntries=0, isLeaf=false):
    包含 sub block 索引:
      → sub block "javac"
      → sub block "javad"
      → sub block "javag"
      → sub block "javan"
      → sub block "javar"
      → sub block "javas"

  sub block "javac" (prefix="javac"):
    suffix "lass"      → "javaclass"
    suffix "lassPath"  → "javaclassPath"
    suffix "ompiler"   → "javacompiler"
    ...

  sub block "javas" (prefix="javas"):
    suffix "cript"     → "javascript"
    suffix "criptCore" → "javascriptCore"
    ...

  .tip FST 多层:
    路径 "j" → "ja" → "jav" → "java" → parent block
    路径 "java" → "javac" → "javac" sub block
    路径 "java" → "javas" → "javas" sub block

  reader 从 root 走 FST, 遇到 sub block 标志继续向下走

23.2 Term metadata

每个 term 在 .tim block 中除了 suffix 字节,还有统计信息(docFreq / totalTermFreq)和 postings 指针(docStartFP / posStartFP / payStartFP)。这些信息让 reader 拿到 term 后能直接 seek 到 .doc / .pos / .pay 读 postings list。

字段类型含义何时存在

totalTermFreqVLong全部出现次数(- docFreq)所有 term

23.3 共享前缀的字节节省

block 内所有 term 共享前缀,只存 suffix。这是 BlockTree 节省空间的核心机制。前缀放在 .tip FST 节点上,所有走到该节点的 term 共用,block 自身只存差量。

23.3.1 字节节省演示

图 23-3-1 · 4 个 "appl*" term 的共享前缀节省
不共享前缀的存储 (基线):

  apple        : 5 byte
  application  : 11 byte
  applied      : 7 byte
  apply        : 5 byte

  小计:          28 byte
  + 长度字段 (VInt 1 byte) × 4: 4 byte
                              ─────
  合计:          32 byte

────────────────────────────────────────────────
共享 "appl" 前缀:

  prefix "appl" (在 .tip 中):  4 byte

  block 内 suffix:
    "e"        : 1 byte
    "ication"  : 7 byte
    "ied"      : 3 byte
    "y"        : 1 byte

  小计 suffix:                12 byte
  + 长度字段:                  4 byte
                              ─────
  block 内总:                 16 byte
  + prefix 在 .tip 中的占用:    4 byte
                              ─────
  合计:                       20 byte

★ 压缩比 32 / 20 = 1.6×

23.5.3 PendingBlock 缓冲机制

PendingBlock 在内存中累积 term,达到阈值切 block。这种机制让 prefix 可以贪心选择(选最长共同前缀),不需要一次扫全 term。

图 23-5-2 · PendingBlock 切分流程逐帧
帧 0 (初始):
   PendingBlock = []
   pendingCount = 0

帧 1 (添加 "apple"):
   PendingBlock = ["apple"]
   pendingCount = 1

帧 2 (添加 "applied"):
   PendingBlock = ["apple", "applied"]
   pendingCount = 2

帧 N (累积到 25 个):
   PendingBlock = ["apple", "applied", "application", "apply", ...]
   pendingCount = 25
   达到 minBlockSize:
     判断剩余 term 是否仍同 prefix
     若是 → 继续累积
     若否 → 切 block

帧 N+1 (切 block):
   扫描 PendingBlock 找最长共同前缀
     common prefix = "appl"
     suffix list  = ["e", "ied", "ication", "y", ...]
   写 block 到 .tim:
     numEntries = 25
     suffix list bytes
     term stats (docFreq, ttf)
     term metadata (docStartFP, ...)
   写 FST 节点到 .tip:
     "appl" -> blockFP
   reset PendingBlock = []

帧 N+2 (继续):
   添加 "banana", "banner", ...
   累积新 block ...

23.7 sub block 嵌套

23.7.1 多层嵌套结构

sub block 在 .tim 中是普通 block,但 parent block 的 metadata 区域含 sub block FP 索引,指向所有 sub block。

图 23-7-1 · 多层 BlockTree 嵌套(prefix="appl" 下 80 term)
prefix "appl" 下 term 数 80 (超过 maxBlockSize=48)
→ 切 sub block:

   sub block 1: prefix="apple"   suffix="...e, ...ed, ...es, ..."  (25 term)
   sub block 2: prefix="appli"   suffix="...cation, ...ed, ..."    (30 term)
   sub block 3: prefix="apply"   suffix=""                          (单 term)
   sub block 4: prefix="appro"   suffix="...ach, ...ve, ..."        (24 term)

.tim 中的物理布局:
   sub block 1 (numEntries=25, isLeaf=true)
   sub block 2 (numEntries=30, isLeaf=true)
   sub block 3 (numEntries=1,  isLeaf=true)
   sub block 4 (numEntries=24, isLeaf=true)
   parent block (numEntries=0, isLeaf=false, 含 sub block FP 索引)

.tip FST 也是分层的:
   "appl"  → parent block FP
   "apple" → sub block 1 FP
   "appli" → sub block 2 FP
   "apply" → sub block 3 FP
   "appro" → sub block 4 FP

23.9.2 docStartFP delta encoding

docStartFP 在 metadata 区域用 delta encoding(相邻 term 的差),减少字节数。第一个 term 是绝对值,后续 term 是 delta。

图 23-9-2 · docStartFP delta encoding 例子
term 顺序 + docStartFP:

  term 0:  docStartFP = 0x000100  (绝对 VLong)
  term 1:  docStartFP = 0x000180  → delta = 0x80
  term 2:  docStartFP = 0x000250  → delta = 0xD0
  term 3:  docStartFP = 0x000320  → delta = 0xD0
  ...

VLong 编码后:
  term 0: 3 byte (绝对值)
  term 1: 1 byte (delta=128)
  term 2: 2 byte (delta=208)
  term 3: 2 byte
  ...

平均 2 byte / term, vs 不 delta 平均 3-4 byte / term
节省 ~ 1.5×

23.14.2 反例 1:高基数字段词典爆

❌ 反例:user_id / session_id 直接 keyword

public Document indexEvent(Event event) {
    Document doc = new Document();
    doc.add(new StringField("user_id", event.userId, Field.Store.NO));
    doc.add(new StringField("session_id", event.sessionId, Field.Store.NO));
    doc.add(new StringField("url", event.url, Field.Store.NO));
    doc.add(new TextField("user_agent", event.userAgent, Field.Store.NO));
    return doc;
}

// 现象 (100M event, user_id 100M unique, session 50M unique):
// .tim 中 user_id 字段 5 GB
// .tim 中 session_id 字段 3 GB
// .tim 中 url 字段 2 GB
// .tim 总 12 GB
// .tip 中 FST 节点 200M 个, 内存 4 GB
// forceMerge 4 小时 OOM
// 单 segment open 30 秒

✅ 修复:hash + NumericDV + Points

public Document indexEvent(Event event) {
    Document doc = new Document();

    long uidHash = MurmurHash3.hash64(
        event.userId.getBytes(UTF_8));
    long sidHash = MurmurHash3.hash64(
        event.sessionId.getBytes(UTF_8));

    doc.add(new NumericDocValuesField("user_id_hash_dv", uidHash));
    doc.add(new LongPoint("user_id_hash", uidHash));

    doc.add(new NumericDocValuesField("session_id_hash_dv", sidHash));
    doc.add(new LongPoint("session_id_hash", sidHash));

    doc.add(new StoredField("user_id", event.userId));
    doc.add(new StoredField("session_id", event.sessionId));

    String urlHost = extractHost(event.url);
    doc.add(new StringField("url_host", urlHost, Field.Store.NO));
    doc.add(new StoredField("url", event.url));

    String[] uaTokens = parseUserAgent(event.userAgent);
    for (String t : uaTokens) {
        doc.add(new StringField("ua_token", t, Field.Store.NO));
    }
    return doc;
}

public Query findByUserId(String userId) {
    long hash = MurmurHash3.hash64(userId.getBytes(UTF_8));
    return LongPoint.newExactQuery("user_id_hash", hash);
}

// 收益:
// .tim 总下降到 100 MB (下降 120×)
// .tip 节点 1M, 内存 50 MB
// forceMerge 30 分钟
// segment open 1 秒
// 精确查 user_id 通过 hash 仍可

23.14.3 反例 2:未规范化 token 词典膨胀

❌ 反例:WhitespaceAnalyzer 不做规范化

public Document indexProduct(Product p) {
    Document doc = new Document();
    Analyzer simple = new WhitespaceAnalyzer();
    doc.add(new TextField("description", p.description,
        Field.Store.NO));
    return doc;
}

// 现象:
// "Apple", "apple", "APPLE", "apple,", "apple."
// 视为不同 token
// .tim 词典含大量大小写 / 标点变体
// match query "apple" 命中率低
// (只匹配小写无标点)
// 词典数膨胀 5-10×

✅ 修复:StandardAnalyzer + StopFilter

public Document indexProduct(Product p) {
    Document doc = new Document();

    StandardAnalyzer std = new StandardAnalyzer(
        StopFilter.makeStopSet(STOPWORDS));

    doc.add(new TextField("description", p.description,
        Field.Store.NO));

    return doc;
}

private static final List<String> STOPWORDS = Arrays.asList(
    "a", "an", "and", "are", "as", "at", "be", "but",
    "by", "for", "from", "has", "in", "is", "it", "its",
    "of", "on", "the", "to", "was"
);

public Query searchDescription(String query) {
    QueryParser parser = new QueryParser(
        "description", new StandardAnalyzer());
    return parser.parse(query);
}

// 收益:
// token 规范化:小写 + StopFilter + ASCIIFolding
// 词典数减少 5-10×
// match 命中率提升
// "Apple" / "apple" / "APPLE" 都匹配

23.14.4 反例 3:forceMerge 不及时

❌ 反例:从不 forceMerge

IndexWriterConfig cfg = new IndexWriterConfig(analyzer);
cfg.setMergePolicy(new TieredMergePolicy());
IndexWriter iw = new IndexWriter(dir, cfg);

// 长期运行后:
// segment 数 50+
// 单 query 走 50 个 leaf reader
// 每 leaf 一次 .tip + .tim seek
// 总开销 50 × 5 μs = 250 μs (热)
// 50 × 100 μs = 5 ms (冷)
// P99 高

✅ 修复:定期 forceMerge + 调优

policy.setMaxMergeAtOnce(10);
policy.setSegmentsPerTier(10);
policy.setMaxMergedSegmentMB(5 * 1024);
cfg.setMergePolicy(policy);

cfg.setRAMBufferSizeMB(256);
IndexWriter iw = new IndexWriter(dir, cfg);

@Scheduled(cron = "0 0 3 * * *")
public void nightlyForceMerge() throws IOException {
    iw.forceMerge(10, true);
}

@Scheduled(cron = "0 0 4 * * SUN")
public void weeklyForceMerge() throws IOException {
    iw.forceMerge(1, true);
}

// 收益:
// segment 数 < 30
// forceMerge 后 .tim 重新构建, 词典紧凑
// query P99 下降 80%

// BlockTreeTermsWriter.PendingBlock 关键逻辑(简化)
private void writeBlock(PendingBlock block) throws IOException {
    long startFP = termsOut.getFilePointer();

    int code = block.numEntries << 1;
    if (block.isLeaf) {
        code |= 1;
    }
    termsOut.writeVInt(code);

    for (PendingTerm term : block.terms) {
        BytesRef suffix = term.term.toBytesRef();
        suffix.bytes = Arrays.copyOfRange(suffix.bytes, prefixLen, suffix.length);
        suffix.length -= prefixLen;

        termsOut.writeVInt(suffix.length);
        termsOut.writeBytes(suffix.bytes, suffix.offset, suffix.length);
    }

    long lastDocStartFP = -1;
    for (PendingTerm term : block.terms) {
        BlockTermState ts = term.state;
        termsOut.writeVInt(ts.docFreq);
        if (writeFreqs) {
            termsOut.writeVLong(ts.totalTermFreq - ts.docFreq);
        }

        if (lastDocStartFP < 0) {
            termsOut.writeVLong(ts.docStartFP);
        } else {
            termsOut.writeVLong(ts.docStartFP - lastDocStartFP);  // delta!
        }
        lastDocStartFP = ts.docStartFP;

        if (writePositions) {
            termsOut.writeVLong(ts.posStartFP);
        }

        if (ts.docFreq == 1) {
            termsOut.writeVInt(ts.singletonDocID);                 // 内联!
        } else if (ts.docFreq > BLOCK_SIZE) {
            termsOut.writeVLong(ts.skipOffset);
        }
    }

    fst.add(prefix, block.fp);
}
