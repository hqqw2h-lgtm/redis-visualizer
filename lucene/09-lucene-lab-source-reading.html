第 9 章 · Lucene 源码阅读路径

最小 Lab · 12 个实验 · 问题驱动阅读 · 删除三种状态 · Snapshot · Rebuild · 阅读地图

前 8 章把 Lucene 当"黑盒框架"用,你已经知道倒排怎么建、BM25 怎么算、searchAfter 怎么翻页、segment 怎么合并。但当生产出现一个具体问题——比如"为什么 commit 卡 30 秒"或"forceMerge 后查询变慢了"——光靠 API 文档读不出根因,必须沿着调用栈钻进 Lucene 源码。这一章的任务就是把"读源码"这件事变成可重复的工程动作,而不是漫无目的的代码漫游。

方法论很简单:每一个源码路径都配一个可运行的最小 Lab。Lab 跑出来一个看得见的产物(文件、日志、计数),你回头去对照源码,源码再回到 Lab 验证假设——这是把"抽象代码"变成"可触摸现象"的唯一办法。本章给出 12 个 Lab + 1 个 Lucene 工程克隆向导 + 1 个"问题→源码入口"映射表。每个 Lab 200 行以内,跑一遍 5 分钟,产物清晰可观察。Lab 结束你会知道:写入慢去看 DocumentsWriter.flushControl;查询慢去看 BlockMaxConjunctionScorer;评分错去看 BM25Similarity.scorer。

  • Lucene 工程的 10 个核心模块结构,以及如何用 Gradle 构建并 Run 单个测试
  • 10 文档订单 Lab 的标准 pom.xml + 目录结构,作为后续所有实验的脚手架
  • 12 个 Lab 串联起 IndexWriter / IndexReader / IndexSearcher 三大入口的关键源码路径
  • Postings Dump 实验:用 TermsEnum + PostingsEnum 直接打印 .tim/.doc 解码结果
  • 删除的三种状态:Indexed → DeletedMarker (.liv) → Reclaimed (merge),以及 maxDoc/numDocs/磁盘的差异
  • SnapshotDeletionPolicy 的内部机制:为什么"快照"不复制文件而是阻止合并清理
  • forceMerge(1) 的真实代价(临时磁盘 IO 1-2 倍索引大小),以及 forceMergeDeletes() 的低代价替代
  • "问题 → 源码入口"映射表:12 类生产问题各自对应的 3-4 个关键类
  • Lucene 4.x → 9.x 的版本演进时间线(BKD-tree / BlockMax / HNSW 等大特性引入点)

前置知识:第 1-8 章的全部内容(本章是源码导览,不再展开理论)。读者要熟悉 IndexWriter / IndexSearcher 的 API,知道 segment 的物理形态。建议在 IDEA 里克隆 Lucene 9.10.0 工程并完成 Gradle Sync,边读源码边在 Lab 里设断点。

9.3 实验 1:写入与文件落盘

第一个 Lab 把 10 条订单写入,关闭 compound file 选项让所有文件直接落盘成单个 segment。这样能在 index/ 目录下看到第 1 章里画的那 13 种文件。

实验代码

public class OrderLab {
    public static void main(String[] args) throws Exception {
        Path indexPath = Paths.get("index/orders");
        try (Directory dir = FSDirectory.open(indexPath)) {
            IndexWriterConfig cfg = new IndexWriterConfig(new StandardAnalyzer());
            cfg.setOpenMode(OpenMode.CREATE);
            cfg.setUseCompoundFile(false);  // 关键: 关闭 .cfs 打包,让文件直接可见
            try (IndexWriter w = new IndexWriter(dir, cfg)) {
                List<OrderEvent> events = loadFromCsv("data/orders.csv");
                for (OrderEvent ev : events) {
                    w.addDocument(OrderToDocConverter.convert(ev));
                }
                w.commit();
            }
        }
    }
}

9.4 实验 2:Postings Dump

这个 Lab 直接打开 reader,用 TermsEnum 遍历某字段的所有 term,再用 PostingsEnum 解码每个 term 的倒排列表。看得见的产物是一份"term → docFreq + (docID, tf) 列表"的纯文本 dump。这相当于把 .tim/.tip/.doc 三个文件的内容反编码成人能读的形式。

实验代码

public class PostingsDumper {
    public static void main(String[] args) throws Exception {
        try (DirectoryReader reader = DirectoryReader.open(
                FSDirectory.open(Paths.get("index/orders")))) {
            for (LeafReaderContext leaf : reader.leaves()) {
                Terms terms = leaf.reader().terms("items");
                if (terms == null) continue;

                TermsEnum te = terms.iterator();
                BytesRef term;
                while ((term = te.next()) != null) {
                    System.out.println("term=" + term.utf8ToString()
                        + " df=" + te.docFreq()
                        + " ttf=" + te.totalTermFreq());
                    PostingsEnum pe = te.postings(null, PostingsEnum.FREQS);
                    int doc;
                    while ((doc = pe.nextDoc()) != DocIdSetIterator.NO_MORE_DOCS) {
                        System.out.println("  doc=" + doc + " freq=" + pe.freq());
                    }
                }
            }
        }
    }
}

9.8 实验 6:Explain

TopDocs top = searcher.search(q, 10);
for (ScoreDoc sd : top.scoreDocs) {
    Explanation expl = searcher.explain(q, sd.doc);
    System.out.println(expl);
    System.out.println("---");
}

explain 把每个 doc 的最终分数拆解为一棵 Explanation 树。它和 search 走的是相似但不同的代码路径——search 时为了性能,中间因子(idf、tfNorm、norm)不会保留;explain 时重新走一遍并把每个中间值都装进树。所以 explain 比 search 慢,但能看到 BM25 三因子的具体值。

9.9 实验 7:NRT 刷新

SearcherManager mgr = new SearcherManager(writer, true, true, new SearcherFactory());

// 写新 doc 但不 commit (NRT 不需要 commit 就可见)
w.addDocument(newDoc);

// 主动刷新(把内存 RAM 索引 flush 出来,产生一个新的临时 segment)
mgr.maybeRefresh();

// 取一个一致的 IndexSearcher 视图
IndexSearcher s = mgr.acquire();
try {
    TopDocs top = s.search(query, 10);
    // 此时 newDoc 已经能搜到
} finally {
    mgr.release(s);  // 引用计数 -1
}
表 9-9 · NRT 路径的源码
类作用
SearcherManagerNRT 的标准入口

NRT 不需要 commit。SearcherManager.maybeRefresh() 内部调用 writer.getReader(),把 RAM 中的 DWPT 数据 flush 成一个临时段(uncommitted),reader 直接打开包含这个临时段的 view。重启进程会丢这部分数据(因为没 commit),所以 NRT 适用"短延时可见"的搜索,持久化必须靠 commit。

CheckIndex 是 Lucene 自带的索引完整性诊断工具。生产里出现过 Bug、JVM 崩溃、或磁盘损坏后,先跑 CheckIndex 才能判断"是 Lucene 索引坏了"还是"是查询逻辑错了"。它会重新解码每个 segment 的所有结构(term、postings、stored、dv、point、norms),非常彻底但耗时——1 GB 索引大约 1-3 分钟。

9.14 删除的三种状态

9.5 节看过 updateDocument / deleteDocuments 会产生 .liv 文件,但很多人停在"删除完成"这个抽象。删除其实有三种独立的状态,每种对应不同的查询行为、磁盘占用和合并触发条件。这一节用一个 Lab 把三种状态都跑出来。

Lab:观察三种状态的实验

public class DeleteLifecycleLab {
    static void stats(String label, Directory dir) throws Exception {
        try (DirectoryReader r = DirectoryReader.open(dir)) {
            System.out.printf("%-20s maxDoc=%d numDocs=%d deleted=%d leaves=%d%n",
                label, r.maxDoc(), r.numDocs(),
                r.maxDoc() - r.numDocs(), r.leaves().size());
        }
    }

    public static void main(String[] args) throws Exception {
        Directory dir = FSDirectory.open(Path.of("target", "order-index"));

        // 阶段 1: 写 10 条订单
        try (IndexWriter w = new IndexWriter(dir,
                new IndexWriterConfig(new StandardAnalyzer()))) {
            for (int i = 0; i < 10; i++) {
                Document d = new Document();
                d.add(new StringField("orderId", "O-" + i, Field.Store.YES));
                w.addDocument(d);
            }
            w.commit();
        }
        stats("after 10 adds", dir);

        // 阶段 2: 删 2 条 (进 DeletedMarker 状态)
        try (IndexWriter w = new IndexWriter(dir,
                new IndexWriterConfig(new StandardAnalyzer()))) {
            w.deleteDocuments(new Term("orderId", "O-3"));
            w.deleteDocuments(new Term("orderId", "O-7"));
            w.commit();
        }
        stats("after delete 2", dir);

        // 阶段 3: forceMerge(1) (物理回收)
        try (IndexWriter w = new IndexWriter(dir,
                new IndexWriterConfig(new StandardAnalyzer()))) {
            w.forceMerge(1);
            w.commit();
        }
        stats("after forceMerge", dir);
    }
}
预期输出:

after delete 2       maxDoc=10  numDocs=8   deleted=2  leaves=1   ← maxDoc 不变!
after forceMerge     maxDoc=8   numDocs=8   deleted=0  leaves=1   ← 物理回收

9.15 Snapshot/Rebuild Lab:可回滚 vs 全量重建

线上索引不是数据库本体——OLTP DB 或事件日志才是源头,Lucene 索引可以丢了重建。基于这个原则,生产部署有两个守护机制:Snapshot(打快照,可回滚)、Rebuild(从源数据全量重建)。两者用途完全不同,选错会出大问题。

Snapshot vs Rebuild 的决策

Snapshot(时间点回滚)
原理:阻止合并清理某个 commit 引用的文件,文件仍在磁盘,reader 可打开
代价:磁盘暂时多占用旧 segment 大小
适用:试 boost 调优、改 Analyzer 参数、临时性试验回滚
不适用:跨 Lucene 大版本、改字段类型、Schema 迁移
恢复时间:几秒(直接打开旧 commit)
Rebuild(全量重建)
原理:从源数据(OLTP / 事件日志)新建到新目录,验证后 alias 切流
代价:重建时间 + 双倍磁盘临时占用
适用:升级 Lucene、改 Analyzer、改字段类型、Schema 迁移
不适用:无法回滚到旧数据(源数据已消失场景)
恢复时间:全量重建时间(分钟到小时)

Snapshot 的工作原理

Snapshot 不是文件复制。它告诉 IndexWriter:这个 commit 引用的文件不要删,即使后续合并产生新 commit。这样磁盘上同时存在两组段文件,reader 可以打开任一 commit 对应的视图。直到调用 release(snapshot) 之后,旧 commit 的文件才被允许在下一次合并时清理。

SnapshotDeletionPolicy snapshotPolicy = new SnapshotDeletionPolicy(
    new KeepOnlyLastCommitDeletionPolicy());

IndexWriterConfig conf = new IndexWriterConfig(new StandardAnalyzer())
    .setIndexDeletionPolicy(snapshotPolicy);

try (IndexWriter w = new IndexWriter(dir, conf)) {
    // ... 写入 ...
    w.commit();

    // 打快照
    IndexCommit snapshot = snapshotPolicy.snapshot();
    System.out.println("snapshot files: " + snapshot.getFileNames());

    // ... 继续写入,合并产生新 commit ...

    // 出问题: 用快照打开旧版本 reader
    try (DirectoryReader oldReader = DirectoryReader.open(snapshot)) {
        // 这个 reader 看到的是快照那一刻的索引状态
    }

    // 释放快照,旧 segment 允许被合并清理
    snapshotPolicy.release(snapshot);
}

Rebuild Lab:从事件源全量重建

public class RebuildLab {
    public static void main(String[] args) throws Exception {
        Path oldIndex = Path.of("target", "order-index-v1");
        Path newIndex = Path.of("target", "order-index-v2");

        // 全量重建到新目录
        try (IndexWriter w = new IndexWriter(FSDirectory.open(newIndex),
                new IndexWriterConfig(new StandardAnalyzer()))) {
            for (OrderEvent e : eventStore.scanAll()) {
                w.addDocument(toDoc(e));
            }
            w.forceMerge(1);
            w.commit();
        }

        // 抽样验证: 新旧 index 命中数应一致
        long oldCount = countDocs(oldIndex);
        long newCount = countDocs(newIndex);
        if (Math.abs(oldCount - newCount) > oldCount * 0.0001) {
            throw new IllegalStateException("count diff > 0.01%, abort switch");
        }

        // 通过验证后, 应用层 atomic 切 alias 指向 newIndex
        // (alias 是应用层概念, Lucene 本身没有 alias)
    }
}
alias 切换让索引重建从"30 分钟停服"降到"0 停服"。某次升级 Lucene 7→9,如果 in-place 升级要停服 35 分钟(rebuild 过程不能查询)。改 alias 切流方案:旧索引继续服务读流量,新索引在另一个目录后台重建,验证 OK 后 atomic 替换 alias 指向。用户感知到的停机时间 = 0;只有"alias 替换的几毫秒"内有竞态,可重试。
