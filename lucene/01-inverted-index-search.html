第 1 章 · 倒排索引与搜索执行链路

Term Dictionary · Postings · Skip Data · BlockTree FST · BM25 · TopK · NRT · LiveDocs

在 1 千万文档的订单库中查"哪些订单的标题包含 charger"——如果用 SQL LIKE '%charger%',数据库要扫 1 千万行。倒排索引把这个问题从 O(N) 降到 O(freq),常见 term 的 freq 在 10³-10⁵ 量级,扫描代价相比全表降低 100-1000 倍。这就是 Lucene 能在毫秒内从 TB 级语料里召回结果的根本原因。

但"倒排索引"四个字背后是一整套精密的工程:Analyzer 把句子切成 token,IndexWriter 把 token 攒成 RAM postings,Codec 把 postings 编码为 .tim/.tip/.doc/.pos/.pay 五个文件,BlockTree 用 FST 压缩前缀实现 O(log N) 词典查找,Skip Data 让 advance(target) 在 1 亿 doc 上仍是 O(log freq),BM25 用 idf/tf/dl 三个因子打出可解释的相关性分数,TopScoreDocCollector 维护小顶堆只保留 K 个候选,NRT 通过 SearcherManager 把"写入到可见"压到秒级,LiveDocs 让删除做到几乎零开销。本章把这条从 addDocument() 到 searcher.search() 的完整链路拆开,每一段都配图。

  • 写入链路:addDocument → DocumentsWriterPerThread → RAM buffer → flush → Codec → 五个文件落盘
  • 四种字段类型四条访问路径:TextField(全文)/StringField(精确)/Point(范围)/DocValues(排序)/StoredField(展示)
  • Postings 物理形态:docID delta + freq blocks(128 doc/block)+ PackedInts + 多级 Skip Data(O(log freq) advance)
  • BlockTree 词典:.tip 内存常驻 FST(共享前缀压缩 6-7×)+ .tim 磁盘 block(O(log N + B) seek)
  • 查询编译三层:Query(意图)→ Weight(已绑定 IndexSearcher 的 stats)→ Scorer(单 LeafReader 上的 DocIdSetIterator)
  • BooleanQuery 求交的 leapfrog 算法:短表 next + 长表 advance,总遍历次数 = O(min(freq))
  • BM25 三因子拆解:idf(稀有度)× tf(命中频次,被 k1 饱和)× 长度归一化(1-b+b·dl/avgdl)
  • NRT 的 maybeRefresh 时序、updateDocument 的 delete+add 语义、numDocs/maxDoc/liveDocs 三个量的关系

前置知识:基础的 Java I/O 与 NIO Channel,文件系统 mmap 的 page cache 概念,B-Tree 与跳跃表的常识。本章是后续所有章节的"地基"——第 10 章详解 FST 字节编码、第 11 章解剖 PostingsEnum 跳跃迭代、第 4 章展开 BM25 的数学推导,均建立在本章的整体框架之上。

1.1 写入入口与源码路径

所有数据流的起点都是一个 8 个字符的方法名:addDocument。在你的业务代码里,它看起来不过是一行 writer.addDocument(doc);但点进去,Lucene 要在内部完成线程绑定、字段分派、倒排建立、内存累积、阈值判断、segment 落盘等十几个动作。把这条调用链画清楚,后面所有"为什么写入慢""为什么 segment 多""为什么 commit 卡顿"的诊断都有了出发点。

从 addDocument 到 segment 文件:六层调用

图 1-1 · 单文档写入的核心调用链

IndexWriter.addDocument 在内部其实调用的是 updateDocuments(null, doc)——这暗示着所有"写入"在 Lucene 看来都是"先删后加",只是 add 模式的删除项为空。DocumentsWriter 维护一个 DWPT 池(DocumentsWriterPerThread),多线程并发时每个线程独占一个 DWPT,各自的 RAM postings 不共享内存,从根本上避免锁竞争——这是 Lucene 写入扩展性的设计基石。

为什么 add 走 update 接口? Lucene 的源码注释说得很直白:"unify the code path"。所有写入都走 DocumentsWriter.updateDocuments(deleteTerm, docs),deleteTerm 为 null 即纯添加,非空则删除匹配 term 的旧文档再加入新文档。这种统一让 IndexWriter 内部的并发控制、flush 触发、commit 协议只需要一份实现——updateDocument 不是 add 的特例,反过来 add 才是 update 的特例。

RAMBuffer 调参的两个极端 — RAMBuffer 设得太小(如 16 MiB),频繁 flush 会生成大量小 segment,merge 压力陡增,查询时打开过多 SegmentReader 让 P99 延迟抖动严重;设得太大(如 2 GiB),单次 flush 时间可达数秒,期间 IndexWriter 持有内部锁阻塞其他写入。生产经验值是 64-512 MiB,DWPT 数量与写线程数对齐(默认 8 个,可通过 setMaxThreadStates 调整)。
Elasticsearch 默认值的来源 — Elasticsearch 的 indices.memory.index_buffer_size 默认是 JVM heap 的 10%,且这个 buffer 在所有 shard 之间共享。一台 32 GiB heap 的节点上,所有 IndexWriter 共用约 3.2 GiB 的 RAM buffer。当某个 shard 的 indexing 速率突增,会"挤占"其他 shard 的 buffer,导致集体 flush 风暴——这是 ES 大批量导入时 segment 数量"莫名其妙"暴涨的根因。indices.memory.min_index_buffer_size(默认 48 MiB)能保证每个 shard 的最小份额。

1.2 字段建模与四条访问路径

同一个业务字段,例如订单的"金额",在搜索系统里通常需要走三条不同的路径:用户筛选 amount > 100 是范围过滤、按金额降序展示是排序、订单卡片上显示 ¥128.00 是返回展示。三条路径对应三种物理结构:BKD-tree、PackedInts 列存、LZ4 chunk。如果你只把 amount 写成一个 StoredField,排序时 Lucene 必须解压整个 chunk 拿到 amount——这就是"看起来能跑但生产不可用"的典型反例。

// ❌ 用 StoredField 想兼顾排序

    new SortField("amount", SortField.Type.LONG));
// IllegalStateException:

StoredField 不能排序——它是 LZ4 chunk,需要按 docID 顺序解压才能拿到值,代价比 docID 列读高 100 倍。

正例:三条路径三份数据

// ✅ 同一业务字段写三份
doc.add(new LongPoint("amountCents", 12800L));
doc.add(new NumericDocValuesField(
    "amountSort", 12800L));
doc.add(new StoredField(
    "amountText", "128.00"));

// 范围过滤走 BKD,排序走 DocValues,
// 展示走 LZ4 chunk——各条路径各自最优。
// 索引膨胀 ~30%,但查询性能稳定。

三份数据的总开销远小于"用错路径"带来的 P99 抖动。

"路径分离"是 Lucene 字段建模的第一原则 — 一个业务字段写三份并不浪费,因为三种结构各自做的事完全不同。BKD 做范围,DocValues 做排序,StoredField 做展示。同一份数据走错路径(比如用 StoredField 排序)会触发解压全 chunk,代价比正确路径高 2-3 个数量级。索引膨胀 30% 换来 P99 稳定可控,这是值得的。

1.3 Analyzer 与 TokenStream

Analyzer 是 Lucene 写入与查询的"共同语言"。写入时把原文切成 token,查询时同样的文本必须经过等价的 Analyzer 才能命中索引中的 term——一旦写入用 StandardAnalyzer(把 "USB-C" 切成 [usb, c])而查询用 WhitespaceAnalyzer(整体保留 "USB-C"),就会出现"明明有这个词却搜不到"的诡异现象。

实战:dump TokenStream

static void dumpTokens(Analyzer analyzer, String field, String text) throws IOException {
    try (TokenStream ts = analyzer.tokenStream(field, text)) {
        CharTermAttribute term = ts.addAttribute(CharTermAttribute.class);
        OffsetAttribute offset = ts.addAttribute(OffsetAttribute.class);
        PositionIncrementAttribute pos = ts.addAttribute(PositionIncrementAttribute.class);
        ts.reset();
        int position = 0;
        while (ts.incrementToken()) {
            position += pos.getPositionIncrement();
            System.out.printf("pos=%d term=%s offset=%d..%d%n",
                position, term, offset.startOffset(), offset.endOffset());
        }
        ts.end();
    }
}

1.4 倒排索引的物理结构

"倒排索引"在教科书上的定义很简单:Map<Term, List<DocID>>。但 Lucene 把这个 Map 拆成五个文件,每个文件按"何时需要读"分流——TermQuery 只读 .doc,PhraseQuery 才打开 .pos,高亮才用 .pay。文件分层就是"用最少的 page cache 服务最多的查询"。

逻辑模型:term → posting list

图 1-6 · 倒排索引的逻辑形态

每个 PostingList 是按 docID 升序排列的 (docID, freq, positions[], offsets[], payload) 元组序列。docID 升序是 leapfrog 求交、skip 跳跃迭代、delta 编码的共同前提——它不是某种"美化"约定,而是性能优化的必要条件。

为什么是 128? 不是 100、不是 256,Lucene 选了 128 这个看起来"非整数"的值。原因有三:(1) 128 = 2⁷,与 SIMD 指令的向量长度天然对齐;(2) PackedInts 解码时 128 个 long 正好用满 L1 cache 一行(8 byte × 128 = 1 KiB,远小于 32 KiB L1d);(3) skip 跳跃表的层级 128² = 16384、128³ = 2M,在常见 freq 范围内层级数刚好 2-3 层。这些"巧合"全是对齐工程经验的产物。

Apache Lucene PR #11365(2022):跳跃表分级数动态化 — 原本 skip 层级是固定的 ⌈log₁₂₈ freq⌉。但实际中,freq=10⁵ 的 term 用 2 层就够,freq=10⁹ 才需要 3 层。固定 3 层会让小 term 多读一次 skip。后来的版本里,Lucene 改成 "skipMultiplier=8 + 动态层数",让 skip 数据本身的开销降低 ~15%。这种"看起来很小"的优化在大型搜索集群里能省下数 TiB 的 segment 体积。

1.5 BlockTree 词典与 FST

每个字段有自己的 term dictionary。一个有 1 亿 term 的字段,如果用 HashMap 存,内存占用至少几十 GB——这显然不可行。BlockTree 用"内存中的 FST 索引 + 磁盘上的 block 数据"两层结构,把内存常驻部分压到 1-10 MiB,把磁盘部分用前缀共享 + LZ4 压到原始大小的 30%。

seekExact 的访问路径

整体复杂度 O(log N + B),N 为字段内 term 总数,B 为 block 内 term 数(默认 25-48)。.tip 的 FST 提供 O(log N) 的跳转,.tim 内 block 扫描复杂度 O(B)——FST 把"找前缀"的部分放在内存,block 把"找后缀"的部分放在磁盘,二者协作避免了任一方的极端情况。

1.6 查询执行模型

Lucene 的查询执行不是简单地"对 Query 求值",而是经过 Query → Weight → Scorer 三层编译。Query 是查询的"语法树"(只有意图),Weight 是绑定到具体 IndexSearcher 后的"已编译"形态(持有 idf/avgdl 等统计量),Scorer 是 Weight 在单个 LeafReader 上的执行器(通过 DocIdSetIterator 推进 docID 并产出 score)。这个三层模型让查询可以跨多 segment、跨多线程并行执行。

Leapfrog 求交算法的状态机

图 1-14 · 两个 PostingsEnum 的 leapfrog 求交
postings: wireless
[1, 7, 11, 19, 42, 99]
postings: charger(短表)
[7, 42, 100]
leapfrog 状态机:短表 next + 长表 advance
  1. 取较短的 postings(charger)的第一个 doc = 7
  2. 让 wireless.advance(7) → wireless 跳到 doc = 7 ✅ 命中
  3. 取 wireless 下一个 doc = 11
  4. 让 charger.advance(11) → charger 跳到 doc = 42
  5. 让 wireless.advance(42) → wireless 跳到 doc = 42 ✅ 命中
  6. 取 wireless 下一个 doc = 99
  7. 让 charger.advance(99) → charger 跳到 doc = 100,大于 99
  8. 让 wireless.advance(100) → wireless 到 NO_MORE_DOCS
命中 doc: 7, 42 · 总迭代 = 8 步 ≈ short.size × 2
"短表驱动"是 Lucene 求交的根本原则 — 两个 PostingsEnum 中,哪个表 freq 小就让它先 next,长表则用 advance 跳跃。这样总遍历次数被短表限制,即 O(min(freq)) 而非 O(sum(freq))。在 N 个子表的求交中,总代价由最稀疏的那个子表决定。所以"先用稀有 term 缩小候选集再求交"是查询优化的核心思路——这也解释了为什么把 status:PAID 这种高基数字段放在 FILTER 而不是 MUST 子句:filter 不计算 score,可以被 caching 加速,而且它的 cost 通常远小于全文 term。

1.7 BM25 评分公式

BM25 是 Lucene 5.0+ 默认相似度算法,替代了之前的 TF-IDF + Cosine。它解决了传统 TF-IDF 三个核心问题:tf 无上限(打 100 个相同词的文档分数离谱地高)、idf 不平滑(罕见词权重过大)、字段长度无归一化(长文档天然优势)。

BM25 公式分解

图 1-15 · BM25 公式与三个因子
BM25(q, d) = Σ idf(t) × (tf × (k1 + 1)) / (tf + k1 × (1 - b + b × dl / avgdl))
            t∈q

参数说明:
  tf      term 在当前文档字段的出现次数
  idf(t)  log(1 + (N - n + 0.5) / (n + 0.5))
  N       全索引文档总数
  n       含有 t 的文档数
  dl      当前字段长度(token 数)
  avgdl   字段平均长度
  k1      tf 饱和参数,Lucene 默认 1.2
  b       字段长度归一化参数,Lucene 默认 0.75

三个因子的语义

图 1-16 · idf · tf · dl 三因子的语义贡献
参数默认提高降低
k11.2tf 影响减弱(早饱和)tf 影响增强
b0.75长字段惩罚加大字段长度无关
idf 公式log(...+0.5)稀有词 boost 加大常见词价值上升

explain 输出解读

    new TermQuery(new Term("title", "charger")),
    top.scoreDocs[0].doc
);
System.out.println(explanation);
explain excerpt:
  1.8234 = weight(title:charger in 7) [BM25Similarity]
    0.8754 = idf, computed as log(1 + (N - n + 0.5) / (n + 0.5))
              from:
                4.0    = N (number of documents containing the term)
                3000.0 = n (number of documents containing the term)
                  → idf = log(1 + (3000 - 4 + 0.5) / (4 + 0.5))
    2.0833 = tfNorm, computed as (freq × (k1 + 1)) / (freq + k1 × (1 - b + b × dl / avgdl))
              from:
                2.0  = freq (term frequency)
                1.2  = k1
                0.75 = b

排查相关性问题先看 explain 拆分 — 用户反馈"为什么这个商品排第一?",生产排查的第一步是 searcher.explain(query, docId),它会把每个子句的 score 按 BM25 公式逐项展开。常见问题:(1) idf 太小说明该 term 在语料里太常见,考虑加 stop words;(2) tfNorm 太低说明字段过长,检查是不是把多个无关字段拼到一起了;(3) 整体 score 离群高,通常是 boost 配错了——这时把 boost 降到 1.0 看是不是天然高分,如果还是,那是 BM25 公式认为它确实匹配得好,业务规则需要重新讨论。

1.8 DocValues 与排序聚合

"按金额降序展示订单"——这个需求触发的不是倒排索引,而是 DocValues。DocValues 是 Lucene 4.0+ 引入的列式存储,按 docID 索引,每个 docID 直接对应一个值(NUMERIC、BINARY、SORTED 等)。排序时 Lucene 用 NumericDocValues.advanceExact(docID) 在 O(log P) 内拿到值(P 为 PackedInts 位宽),完全不需要解压 .fdt chunk。

排序聚合的访问路径

图 1-17 · 候选 docID → DocValues 列式访问 → 排序输出

DocValues vs StoredField 对比

用 StoredField 排序的代价

  • 访问方向:docID → 字段集合(行式)
  • 物理形态:LZ4 chunk(默认 16 KiB)
  • 排序时:必须先解压整个 chunk
  • 聚合时:必须解析整个 chunk 拿一字段
  • QPS:在 1 千万 doc 上 ~10 QPS(单线程)

用 DocValues 排序

  • 访问方向:docID → 单值(列式)
  • 物理形态:PackedInts(位宽 = ⌈log₂(maxVal-minVal)⌉)
  • 排序时:O(1) 列读 + 小顶堆
  • 聚合时:顺序读取列,SIMD 友好
  • QPS:在 1 千万 doc 上 ~1000 QPS(单线程)

1.9 Near Real-Time 搜索

"用户刚下单,马上就能搜到"——这个看似简单的需求,在不可变 segment 模型下并不简单。Lucene 的 NRT 通过 DirectoryReader.openIfChanged(reader, writer) 实现:writer 内部还没 commit 的 RAM postings 也能被 reader 看到,只要它们已经 flush 成新 segment。SearcherManager 把这套机制包装成线程安全的 reader 复用。

maybeRefresh 时序

图 1-19 · NRT refresh 的写入-可见时序

refresh 频率取舍

频率数据可见性reader reopen 成本segment 增长推荐场景
100 ms< 100 ms高极快不推荐
1 s< 1 s中快实时订单查询
5 s< 5 s低中客服后台搜索
30 s< 30 s极低慢离线分析
手动 commit由 commit 决定一次性高受 RAMBuffer 控制批量写入
 
SearcherManager manager = new SearcherManager(writer, null);

writer.addDocument(toDoc(order));
writer.flush();
manager.maybeRefresh();

IndexSearcher searcher = manager.acquire();
try {
    TopDocs docs = searcher.search(new TermQuery(new Term("title", "charger")), 10);
    System.out.println("hits=" + docs.totalHits.value);
} finally {
    manager.release(searcher);
}
nrt output:
  before refresh hits=0
  after maybeRefresh hits=1
不要每写一条就 refresh — 看上去"实时"很美,实际生产中每写每 refresh 会造成 reader 频繁 reopen,新 SegmentReader 的初始化(读 .tip 到内存、初始化 NormsProducer 等)在大索引上耗时几十毫秒。更糟糕的是触发的 flush 会生成大量小 segment,merge 跟不上时 segment 数膨胀到数百,查询打开过多文件 fd 造成系统不稳。生产中 refresh 间隔通常 1-5 秒,实在需要"立即可见"的场景考虑加业务层缓存,而不是缩短 refresh。

1.10 删除与更新

Lucene segment 是不可变的——一旦写入磁盘,不再修改一个字节。那"删除"怎么实现?答案是 LiveDocs:每个 segment 配一个 BitSet,标记哪些 docID 还活着。删除一个 doc 只是把 BitSet 对应位置 0,物理空间要等到 merge 时才回收。

updateDocument 的语义展开

图 1-20 · updateDocument = deleteDocuments(term) + addDocument

numDocs / maxDoc / liveDocs 三个量的关系

writer.addDocument(orderDoc("O1001", "PAID", "Wireless charger"));
writer.commit();

writer.updateDocument(new Term("orderId", "O1001"),
    orderDoc("O1001", "SHIPPED", "Wireless charger shipped"));
writer.commit();

try (DirectoryReader reader = DirectoryReader.open(directory)) {
    System.out.printf("numDocs=%d maxDoc=%d deleted=%d leaves=%d%n",
        reader.numDocs(),
        reader.maxDoc(),
        reader.maxDoc() - reader.numDocs(),
        reader.leaves().size());

    for (LeafReaderContext leaf : reader.leaves()) {
        Bits liveDocs = leaf.reader().getLiveDocs();
        for (int i = 0; i < leaf.reader().maxDoc(); i++) {
            System.out.printf("leaf=%d doc=%d live=%s%n",
                leaf.ord, i, liveDocs == null || liveDocs.get(i));
        }
    }
}
numDocs=1 maxDoc=2 deleted=1 leaves=2
leaf=0 doc=0 live=false
leaf=1 doc=0 live=true
maxDoc=2 而 numDocs=1 是 Lucene 不可变 segment 模型的直接证据。maxDoc 是物理 doc 总数(包括被标记删除的),numDocs 是活着的 doc 数。删除标记位于 LiveDocs 位图里,segment 内部的物理 doc 直到 merge 才被丢弃。这就是为什么"删了 100 万 doc 但磁盘没变小"——你必须等 merge 把这些 segment 合并掉,新 segment 里就没有这些 doc 了。forceMerge 可以立即触发这个回收,但代价是几十分钟的 CPU/IO 占用。
方案优势风险订单系统适用性

权限 filter 必须前置而非后置 — 一个常见的反模式:先 searcher.search(query, 100) 拿 TopK,再用业务代码过滤掉无权访问的订单。结果:用户能看到的真实结果可能不足 10 条,翻页时 page=2 size=20 会返回空——因为 TopK 已经被无权数据"挤占"。正确做法:在 Query 里加上 FILTER tenantId:T1 AND shopId:S1,让权限约束参与 leapfrog 求交,TopK 收集到的就已经全是有权数据。这一点在多租户系统里是合规底线——查询泄露其他租户数据是 SOC 2 / ISO 27001 都不能接受的。
