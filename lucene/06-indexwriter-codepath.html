第 6 章 · IndexWriter 写入路径源码深潜

addDocument · IndexingChain · PerField · TermsHash · DWPT · DeleteQueue · flushSegment · commit · rollback · tragic event

业务代码看到的是 writer.addDocument(doc) 一行 API,源码层面要走过 五层抽象、十几个组件、数十个文件、两阶段提交协议。从 IndexWriter 持有 write.lock 开始,到 DocumentsWriter 把请求路由到一个 DocumentsWriterPerThread(DWPT),再到 DefaultIndexingChain 把字段分派给 invert/stored/DV/points/vector 五种消费者,接着 TermsHash 把 token 写入 BytesRefHash + ByteBlockPool 的紧凑结构,最后 flushSegment 调用 Codec 五大写器把 RAM 物化为 14 个段文件,SegmentInfos 通过 rename pending_segments_N → segments_N 完成原子提交——任何一步出错都可能导致索引损坏或数据丢失。这一章把这条调用链一刀切开,逐层暴露每个组件的职责、并发模型、内存管理与异常恢复路径。

6.1 IndexWriter 的角色边界

IndexWriter 不是一个"写文件"的对象,而是一个事务外壳 + 一组共享状态的协调者。它对外只暴露 addDocument / updateDocument / deleteDocuments / commit / rollback / close 等几个接口,内部把"多线程并发"、"内存累积"、"flush 与 segment 落盘"、"两阶段提交"、"后台合并"等所有复杂度封装起来。理解 IndexWriter 的第一步,是看清它和周围组件的边界——DocumentsWriter 负责什么、SegmentMerger 负责什么、IndexCommit 又是什么角色,边界划清,源码就不再难读。

6.3 IndexingChain 与字段分派

IndexingChain 是字段的"分流器"。同一个 IndexableField 进来,根据它的 IndexableFieldType 配置(indexOptions / stored / docValuesType / pointDimensionCount / vectorDimension),会被分发到 5 种独立的物理路径——而且这 5 条路径互不阻塞,字段如果同时声明多种能力(例如 LongPoint + NumericDocValuesField),会同时进入多条路径。

processField 源码骨架

// org.apache.lucene.index.DefaultIndexingChain (简化版)
private int processField(int docID, IndexableField field, long fieldGen,
                          int fieldCount) throws IOException {
    String fieldName = field.name();
    IndexableFieldType ft = field.fieldType();

    PerField fp = getOrAddPerField(fieldName, ft);
    fp.fieldInfo.update(ft);

    if (ft.indexOptions() != IndexOptions.NONE) {
        if (fp.fieldGen != fieldGen) {
            fp.invert(field, true);
            fp.fieldGen = fieldGen;
        } else {
            fp.invert(field, false);
        }
    }

    if (ft.stored()) {
        storedFieldsConsumer.writeField(fp.fieldInfo, field);
    }

    DocValuesType dvType = ft.docValuesType();
    if (dvType != DocValuesType.NONE) {
        indexDocValue(fp, dvType, field);
    }

    if (ft.pointDimensionCount() != 0) {
        pointValuesWriter(fp, field);
    }

    if (ft.vectorDimension() != 0) {
        vectorValuesWriter(fp, field);
    }
    return fieldCount;
}
同一字段写多份不是浪费,是必要。订单的 amount 字段如果只写 StoredField,排序时 Lucene 必须解压整个 LZ4 chunk(默认 16 KiB)拿到 amount——代价比 DV 列读高 100 倍。正确做法是同时写 LongPoint("amount", v)(范围过滤走 BKD)+ NumericDocValuesField("amountSort", v)(排序走 DV 列)+ StoredField("amountText", text)(展示走 LZ4 chunk)。三份数据加起来索引膨胀 ~30%,但每条访问路径都最优,P99 稳定可控。

6.4 PerField 内存结构

每个字段在 IndexingChain 内部对应一个 PerField 对象。PerField 不是一个简单的数据结构,而是把"该字段在 RAM 中的所有状态"封装到一起的容器:倒排路径有 TermsHashPerField、stored 路径有 stored buffer、DV 路径有 DocValuesWriter、Points 路径有 BKD buffer。把 PerField 看作"字段在内存里的占位",理解 RAM 占用就有了着力点。

PerField 是怎么被生成的

// DefaultIndexingChain.getOrAddPerField 简化
PerField getOrAddPerField(String name, IndexableFieldType ft) {
    final int hashPos = name.hashCode() & hashMask;
    PerField fp = fieldHash[hashPos];
    while (fp != null && !fp.fieldInfo.name.equals(name)) {
        fp = fp.next;
    }
    if (fp == null) {
        FieldInfo fi = fieldInfos.add(name, ft);
        fp = new PerField(fi, indexCreatedVersionMajor,
                          invertState, termsHash, ...);
        fp.next = fieldHash[hashPos];
        fieldHash[hashPos] = fp;
        totalFieldCount++;
    }
    return fp;
}

注意 fieldHash 是每个 DWPT 私有的——不同 DWPT 各有一份 PerField 集合,这就是为什么 Lucene 的多线程写入无需锁:每个线程对自己 DWPT 的 PerField 独占读写。

6.5 invert 流程拆解

"invert" 是 IndexingChain 中最重的一条路径——CPU 占用 50% 以上,内存占用 60% 以上。它把"字段文本"转成"内存倒排表":Analyzer.tokenStream 切 token → BytesRefHash 把 term 哈希成 termID → ParallelPostingsArray 按 termID 索引状态 → ByteBlockPool 写入 postings 字节流(docID delta + freq + position + payload + offset)。

图 6-6 · 一个字段 invert 时的五步流水

BytesRefHash:term 去重的核心

每个字段(per-field)有自己的 BytesRefHash。它是一个开放寻址哈希表,key 是 term 字节,value 是单调递增的 termID(int)。当一个新 token 进来:

  1. 计算 BytesRef.hashCode()(MurmurHash3 变种)
  2. 用 hash & mask 找 bucket
  3. 若 bucket 空 → 新 termID = count++;把 term 字节写入 ByteBlockPool 拿到 byteStart;ids[bucket] = termID
  4. 若 bucket 占用 → 比较 term 字节是否相同;相同 → 复用 termID;不同 → 线性探测下一个 bucket
  5. 装载因子超过 0.5 → 双倍扩容 + rehash
// BytesRefHash.add(BytesRef bytes) 简化
public int add(BytesRef bytes) {
    final int length = bytes.length;
    int hashPos = findHash(bytes);
    int e = ids[hashPos];

    if (e == -1) {
        // 新 term:写入 ByteBlockPool 并分配 ID
        int len2 = 2 + length;
        if (len2 + pool.byteUpto > BYTE_BLOCK_SIZE) {
            pool.nextBuffer();
        }
        final byte[] buffer = pool.buffer;
        final int bufferUpto = pool.byteUpto;

        bytesStart[count] = bufferUpto + pool.byteOffset;
        // 长度前缀 (vshort) + 字节内容
        if (length < 128) {
            buffer[bufferUpto] = (byte) length;
            pool.byteUpto += length + 1;
            System.arraycopy(bytes.bytes, bytes.offset, buffer, bufferUpto + 1, length);
        } else {
            buffer[bufferUpto]     = (byte) (0x80 | (length & 0x7f));
            buffer[bufferUpto + 1] = (byte) ((length >> 7) & 0xff);
            pool.byteUpto += length + 2;
            System.arraycopy(bytes.bytes, bytes.offset, buffer, bufferUpto + 2, length);
        }
        e = count++;
        ids[hashPos] = e;

        if (count == hashHalfSize) {
            rehash(2 * hashSize, true);
        }
    }
    return e;
}

ParallelPostingsArray:termID 的状态机

有了 termID,接下来需要给每个 termID 维护当前状态:它最近一次出现在哪个 docID?最近一次的 position 是多少?当前 doc 内累计 freq 是多少?Lucene 把这些状态拆到几个并行的 int 数组里(SoA 而非 AoS),用 termID 作为下标:

数组类型含义更新时机
byteStarts[termID]intpostings slice 起始偏移新 term 时分配一次
lastDocIDs[termID]int该 term 最近写入的 docID每次 doc 切换
termFreqs[termID]int当前 doc 内累计 freq每次 token 命中
lastPositions[termID]int当前 doc 内最近 position每次 token 命中
lastOffsets[termID]int最近 offset(若开启)每次 token 命中

当一个 token charger 进来,termID=42,DefaultIndexingChain 会:

  1. 检查 lastDocIDs[42]:若 ≠ 当前 docID → 新 doc 出现 → 把上一个 doc 的 freq 写入 postings stream(用 vInt 编码)
  2. 更新 lastDocIDs[42] = 当前 docID,重置 termFreqs[42] = 1
  3. 把 (position, payload) 写入 .pos / .pay 对应的 slice
  4. 若 lastDocIDs[42] == 当前 docID → 同 doc 多次出现 → termFreqs[42]++
Elasticsearch thread_pool.write.size 与 DWPT 数量协同。ES 默认 thread_pool.write.size = # of allocated processors(节点 CPU 数,不超过 32)。每个 thread 写入时会从 IndexWriter 的 DWPT 池借出一个 DWPT,典型情况下"激活的 DWPT 数 ≈ 写线程数"。如果 thread_pool.write.queue_size 设得过大(默认 200),写突发时会同时激活几十个 DWPT,每个 RAM 占用都计入全局 256 MiB 限额——一旦超限,StallControl 立刻把所有写线程阻塞,业务侧表现就是"批量导入到一半突然卡住几秒"。生产经验是把 queue_size 调到 50-100,让背压更早生效,而不是等到 RAM 满了再 stall。

6.6 RAM postings 与 ByteBlockPool

"为什么 Lucene 的 RAM postings 不直接用 ArrayList<Integer>?"——这是任何认真读源码的人都会问的问题。答案在两个字:GC 压力。一个有 100 万 term × 平均 freq 100 的字段,如果每个 termID 都对应一个 ArrayList,就会有 1 亿个 Integer 装箱对象,Young GC 频率瞬间飙升。Lucene 用 ByteBlockPool(32 KiB 一块的字节池)+ IntBlockPool(8192 int 一块的 int 池)把所有变长缓冲共享到几十个大 byte[] 上,GC 看到的只是少量大对象,无装箱、无对象图遍历。

ByteBlockPool 的工作方式

图 6-7 · ByteBlockPool:slice + 链表 + 五级容量
ByteBlockPool 内部:

  ┌──────────────────────────────────────────────────────────┐
  │ buffers: byte[][]    每个元素是 32 KiB 的 byte[]          │
  │   buffers[0]:  [.....32 KiB.....]                         │
  │   buffers[1]:  [.....32 KiB.....]                         │
  │   buffers[2]:  [.....32 KiB.....]                         │
  │   ...按需扩展                                              │
  ├──────────────────────────────────────────────────────────┤
  │ Slice 分配(按 termID 隔离):                              │
  │   每个 term 在 pool 中拥有自己的链式 slice                │
  │   level 0: 5 byte    → level 1: 14 byte                   │
  │   level 1: 14 byte   → level 2: 20 byte                   │
  │   level 2: 20 byte   → level 3: 30 byte                   │
  │   level 3: 30 byte   → level 4: 40 byte                   │
  │   level 4: 40 byte   → level 5: 80 byte                   │
  │   每级末尾保存"下一段地址"指针(forward pointer)          │
  ├──────────────────────────────────────────────────────────┤
  │ 写满当前 slice → 分配下一级 slice → 写 forward pointer    │
  │ 读时跟随 forward pointer 跨段顺序读取                     │
  └──────────────────────────────────────────────────────────┘

为什么是 5 → 14 → 20 → 30 → 40 → 80?
  • 经验值,适配 freq 分布(大多 term freq 小)
  • 几何级数让"小 term 不浪费空间,大 term 不频繁切换"
  • level 5(80 byte)之后保持 80 一直延续,避免无限增长

ByteBlockPool 与 IntBlockPool 的分工

Pool每块容量存放内容访问模式

"DWPT 池让多线程写入零锁竞争 — 但代价是更多 segment"。Lucene 的写入吞吐能在 16 核机器上线性扩展到 14-15 倍,核心机制是:每个写线程绑定一个 DWPT,DWPT 内的所有 BytesRefHash / PostingsArray / ByteBlockPool 都是线程私有的,写入路径上完全没有锁。代价是 N 个 DWPT 同时 flush 时会产生 N 个独立的小 segment,merge 压力陡增。这就是为什么"加大写线程数"和"增加 segment 数"是同一枚硬币的两面——你必须配套调整 TieredMergePolicy 的合并目标段大小,否则查询时打开的 SegmentReader 数会爆炸。

6.7 DocumentsWriterPerThread 与多线程并发

DWPT 是 Lucene 写入扩展性的灵魂。DocumentsWriter 维护一个 DWPT 池,池里的每个 DWPT 都是一个"私有 segment 工厂"——它持有自己的 IndexingChain、自己的 BytesRefHash、自己的 ByteBlockPool,只在 flush 时和外界交互(通过 SegmentInfos.add)。多线程并发写入时,每个线程通过 ThreadState 绑定到一个 DWPT,完全无锁。

StallControl:全局背压机制

// DocumentsWriterStallControl 的核心
private boolean stalled;
private int numWaiting;
private final ReentrantLock lock = new ReentrantLock();
private final Condition stalledCondition = lock.newCondition();

public void waitIfStalled() {
    if (stalled) {
        lock.lock();
        try {
            while (stalled) {
                numWaiting++;
                try {
                    stalledCondition.await();
                } catch (InterruptedException e) {
                    Thread.currentThread().interrupt();
                    break;
                } finally {
                    numWaiting--;
                }
            }
        } finally {
            lock.unlock();
        }
    }
}

public void updateStalled(boolean stalled) {
    if (this.stalled != stalled) {
        lock.lock();
        try {
            this.stalled = stalled;
            if (!stalled) stalledCondition.signalAll();
        } finally {
            lock.unlock();
        }
    }
}

6.8 DeleteQueue 与 update 实现

"删除"在 Lucene 是一种延迟操作:writer.deleteDocuments(term) 不会立刻去 segment 文件里改任何东西,而是把删除项放进一个全局的 DocumentsWriterDeleteQueue,等到 flush 或 commit 时才广播到所有 in-flight DWPT 和已落盘的 segment。updateDocument(term, doc) 是 delete + add 的原子组合,但内部顺序是"先记录 delete term,后追加 doc",这样新 doc 不会被自己刚记录的 delete 误删。

DeleteQueue 的链表结构

DeleteSlice 的工作机制

每个 DWPT 持有一个 DeleteSlice,记录"我已经看到了 queue 的哪个节点"。当 DWPT 处理新文档时:

  1. 读 queue 当前 tail(volatile read)→ 与自己的 slice.tail 对比
  2. 若 queue 推进了 → 把新节点(delete term/query)累加到 DWPT 私有的 BufferedUpdates
  3. flush 该 DWPT 时,BufferedUpdates 跟着 segment 一起持久化为 .del 文件
  4. 所有 DWPT slice 都读过的节点 → queue head 推进 → 旧节点可被 GC

                              Iterable<? extends Iterable<? extends IndexableField>> docs)
        throws IOException {

    boolean success = false;

        long seqNo = docWriter.updateDocuments(docs, delNode);

// DocumentsWriter.updateDocuments
long updateDocuments(Iterable<? extends Iterable<? extends IndexableField>> docs,
                     DeleteQueue.Node<?> delNode) throws IOException {
    boolean hasEvents = preUpdate();

    final ThreadState perThread = perThreadPool.getAndLock();
    long seqNo;
    try {
        ensureInitialized(perThread);
        DWPT dwpt = perThread.dwpt;
        try {
            seqNo = dwpt.updateDocuments(docs, delNode, flushNotifications);
        } finally {
            if (dwpt.isAborted()) {
                flushControl.doOnAbort(dwpt);
            }
        }
    } finally {
        perThread.unlock();
    }
    if (postUpdate(null, hasEvents)) maybeFlush();
    return seqNo;
}

6.10 commit 的两阶段

commit 是 Lucene 持久化语义的"分水岭":commit 之前的所有 flush 都不算数,进程崩溃重启后会被清理。commit 之后写出的 segments_N 文件是新的提交点,DirectoryReader 看到的就是这个文件指向的 segment 集合。两阶段协议保证 commit 是原子的:要么完全成功,要么完全失败,绝无"半成品 commit"。

commit 时序

图 6-13 · commit 两阶段协议

两阶段的可恢复性

阶段动作失败现象重启后恢复
prepareCommitflush + applyDeletes + 收集元数据抛异常,IW 不进入 commit 状态下次 commit 重做

commit 与 flush 的语义差别

flush(进程崩溃丢失)

  • 把 RAM postings 物化为段文件
  • 新段文件已写入磁盘
  • 但 segments_N 还指向旧 segment
  • NRT reader 通过 IndexWriter 引用可见
  • 进程崩溃 → IndexFileDeleter 清理这些 orphan 段
  • 用 DirectoryReader.open(dir)(无 writer)看不到

commit(持久化)

  • 包含 prepareCommit 的所有动作
  • + 写 pending_segments_N.tmp
  • + fsync 临时文件
  • + 原子 rename → segments_N
  • + fsync 目录
  • 进程崩溃可恢复 → DirectoryReader.open(dir) 能读到
// IndexWriter.commit 简化
public final long commit() throws IOException {
    ensureOpen();
    return commitInternal(config.getMergePolicy());
}

private long commitInternal(MergePolicy mp) throws IOException {
    if (pendingCommit != null) {
        throw new IllegalStateException("prepareCommit was already called");
    }
    long seqNo = prepareCommitInternal(); // 第一阶段:收集 + 写 pending_*.tmp
    finishCommit();                        // 第二阶段:rename + fsync(dir)
    return seqNo;
}

private void finishCommit() throws IOException {
    boolean commitCompleted = false;
    try {
        synchronized (commitLock) {
            ensureOpen(false);
            if (tragedy.get() != null) {
                throw new IllegalStateException("commit failed: tragedy", tragedy.get());
            }
            if (pendingCommit != null) {
                pendingCommit.finishCommit(directory);   // rename pending → segments_N
                lastCommitChangeCount = pendingCommitChangeCount;
                segmentInfos.updateGeneration(pendingCommit);
                deleter.checkpoint(pendingCommit, true); // 引用计数 + 可能 delete 旧 segments_N
                pendingCommit = null;
                commitCompleted = true;
            }
        }
    } finally {
        if (!commitCompleted) {
            // 失败:清理 pending_segments_N.tmp
            deleter.decRef(filesToCommit);
        }
    }
}

6.11 rollback 的语义

rollback() 在 Lucene 里的语义比一般数据库的 rollback 要弱:它只能撤销"上次 commit 之后的所有 flushed segment + RAM 中未 flush 的内容",但对已经 commit 的数据无能为力。把它当作"放弃当前事务"而不是"撤销过去 5 分钟的写入",就不会用错。

rollback 影响范围

时间线:

  t=10   addDocument × 1k → DWPT-A RAM 累积 50 MiB

  t=30   addDocument × 2k → DWPT-A RAM 再 30 MiB
  t=40   flush()          → _2 落盘(尚未 commit)
  t=50   addDocument × 500 → DWPT-A 还在 RAM(_3 候选)

图 6-15 · rollback 内部时序
rollback 不能撤销已 commit 的数据。这是 Lucene 与 Postgres / MySQL 的本质区别——后者的 rollback 可以撤销整个事务,Lucene 的"事务"边界是两次 commit 之间。如果你 t=0 commit,t=10 commit,t=20 commit,t=30 想撤回 t=10 的写入,不可能。要做到这种"时间旅行",必须配合 SnapshotDeletionPolicy:每次 commit 时打一个 snapshot,需要回退时基于 snapshot 重新打开。生产里更常见的做法是在业务层做 soft delete + 版本号,而不是依赖 Lucene 的 rollback。

6.12 异常处理

"写到一半 OOM 了怎么办?" "磁盘满了 IndexWriter 还能用吗?"——这些问题的答案在 Lucene 的 tragic event 机制里。Lucene 把异常分成两类:可恢复异常(IllegalArgument、字段格式错)和 tragic event(OOM、IO 错、Codec 损坏)。前者拒绝当前 doc 但 IndexWriter 还能用;后者直接把 IndexWriter 标记为不可继续,后续所有 add/update 都抛 AlreadyClosedException,你必须 close 重开。

// IndexWriter.maybeCloseOnTragicEvent 简化
private void maybeCloseOnTragicEvent() throws IOException {
    if (this.tragedy.get() != null && shouldClose(false)) {
        rollbackInternal();   // 内部 rollback,不再持有 RAM
    }
}

private void onTragicEvent(Throwable tragedy, String location) throws IOException {
    if (this.tragedy.compareAndSet(null, tragedy)) {
        infoStream.message("IW", "hit tragic " + tragedy.getClass().getSimpleName() +
                                  " at " + location);
    }
    if (!(tragedy instanceof VirtualMachineError)) {
        // VM error 不能继续做 IO 操作
        maybeCloseOnTragicEvent();
    }
}

// 用户侧:add 时如何感知
try {
    writer.addDocument(doc);
} catch (AlreadyClosedException e) {
    Throwable cause = writer.getTragicException();
    log.error("IndexWriter is dead due to: ", cause);
    writer.close();              // close 不会再 flush
    writer = new IndexWriter(dir, cfg);  // 重新打开,自动 rollback 到 segments_N
}
OOM 后 IndexWriter 状态自检 — 不能继续 addDocument 必须 close + 重开。生产中遇到 OOM 后最常见的错误是"我捕获了 OOM,继续重试 addDocument",结果发现仍然抛 AlreadyClosedException——这就是 tragic event 标记的作用。正确做法是:① 立即停止所有 addDocument 调用;② 调 writer.close()(此时不会 flush,只清理 RAM 和写锁);③ 排查是不是 RAMBufferSize 设得过大、并发写线程太多、单文档字段太长(text 字段超过 1 MiB 单 doc);④ 调整后用 new IndexWriter(dir, cfg) 重新打开,Lucene 会自动 rollback 到上次 commit 的 segments_N,孤儿文件由 IndexFileDeleter 清理。生产事故复盘时记得查 writer.getTragicException() 拿到原始 Throwable。

6.15 本章小结

关注点核心 API / 类关键建议
角色边界IndexWriter · DocumentsWriter · DWPTIW 是事务外壳 / DW 是协调者 / DWPT 是私有 segment 工厂
调用栈addDocument → updateDocuments → DWPT → IndexingChain耗时 70-80% 在第 7 层(具体字段编码)
字段分派DefaultIndexingChain.processField5 条路径独立 / 同字段可走多条 / 不要用 StoredField 排序
PerField 内存BytesRefHash + ByteBlockPool + ParallelPostingsArray三槽协作 / GC 友好 / per-DWPT 私有
invert 流程FreqProxTermsWriterPerField.addtermID 隔离 / postings stream 链式 slice
RAM postingsByteBlockPool 32 KiB / 5-14-20-30-40-80 字节 slice共享池 + 几何级数避免内存 churn
DWPT 并发thread-sticky 池 + StallControl 背压无锁写 / 多 DWPT 并发线性扩展
DeleteQueueDocumentsWriterDeleteQueue + 每 DWPT slice异步广播 / update = delete + add 原子
flushSegment8 步 / 5 个 Codec 写器RAM → 14 个段文件 / 加入 SI(尚未 commit)
commit 协议prepareCommit + rename + fsync(dir)POSIX 原子 rename / 进程崩溃可恢复
rollbackwriter.rollback()仅撤销当前事务 / 不能撤销已 commit / 用 Snapshot 替代时间旅行
异常处理tragedy + AlreadyClosedExceptionOOM/IO 后必须 close + 重开 / 看 getTragicException
性能调参RAMBufferSize / maxThreadStates / mergePolicySSD 上 64-512 MiB / DWPT = 写线程数 / 关 CFS
源码路径8 个断点 / InfoStream + CheckIndex跟一次 add+commit 走完所有关键节点
