第 0 章 · 订单检索全景:从 Document 到 Segment 文件

字段四层路径 · 五种字段类型 · 四类底层结构 · 19 个 segment 文件 · 复合查询的文件访问轨迹

大多数 Lucene 教程把"全文搜索"等同于"倒排索引",但生产搜索服务远不止倒排——你既要 title 包含 charger 的全文召回,也要 status = PAID 的精确过滤,还要 amount > 100 的范围过滤、ORDER BY paidAt DESC 的列式排序、最后再返回完整的订单卡片。这五种语义不是同一种结构能优雅处理的——倒排只服务前两种,后三种各自需要 Points / DocValues / Stored Fields。

本章是全书的"地图"——不讨论任何文件后缀的字节编码,只讲清楚:业务字段如何映射到 Lucene 字段、字段如何流入四类底层结构(倒排 / Points / DocValues / Stored)、五种结构对应哪些段文件(共 19 个文件后缀)、一次复合查询如何按需打开这些文件。读完本章你能回答三个面试常问的问题:为什么同一个金额字段要写三份?segment 目录里那些 .tim/.tip/.doc/.fdt/.dvd 各是谁在用?复合 SQL 翻译成 Lucene Query 后,磁盘 I/O 路径是什么?

  • 四层访问路径:Analyzer 层(全文)/ 精确字段层(主键、状态)/ 数值结构层(范围、排序)/ 返回字段层(展示)
  • 五种 Lucene 字段类型:TextField / StringField / LongPoint / NumericDocValuesField / StoredField
  • 四类底层结构:倒排索引(.tim/.tip/.doc/.pos/.pay)、BKD-tree(.dim/.dii)、列式 DocValues(.dvd/.dvm)、行式 Stored(.fdt/.fdx/.fdm)
  • 19 个 segment 文件后缀的分工速查表 + 文件依赖图(segments_N → .si → .fnm → 各数据文件)
  • 复合查询的访问轨迹:title:charger AND status:PAID AND paidAt range ORDER BY paidAt → 哪些文件依次被读
  • flush vs commit 的区别(NRT 可见性、持久性边界)
  • "延迟取回"原则:倒排 + Points 阶段只产 docID,排序阶段只读 DocValues,Stored Fields 仅在 TopK 收尾时打开
  • 全书 35 章如何在本章地图上展开(§01-09 概念地图 / §10-18 核心算法 / §19-34 文件格式深潜)

0.1 四层访问路径与字段分流

所有"看似不同"的搜索请求,在 Lucene 内部最终都会被分解成五种语义之一:全文召回 · 精确过滤 · 范围过滤 · 排序聚合 · 返回展示。这五种语义在 Lucene 里对应四层访问路径——前两种共用"倒排"层,后三种各自有专属层。

三份数据 ≠ 数据冗余 — 三种结构各自做的事完全不同。BKD 答的是"范围内有哪些 doc",DocValues 答的是"docID 对应的值是多少",Stored 答的是"docID 的所有展示字段"。任何一种结构去做另两种的工作,都会带来数量级的开销。这就是 Lucene 字段建模的第一性原理:路径分离。

0.2 Document 字段建模

从一个订单 JSON 到 Lucene Document 的转换,在应用代码中显式完成——Lucene 不感知 JSON 结构,也不读取业务对象的注解。字段名、类型、是否分词、是否存储、是否参与 DocValues,均由调用方控制。

订单 JSON → Lucene Document

图 0-3 · 业务实体到 Document 的映射时序

完整建模代码

static Document toOrderDocument(OrderProjection o) {
    Document doc = new Document();

    // 身份字段:整 term 不分词
    doc.add(new StringField("orderId",  o.orderId(),  Field.Store.YES));
    doc.add(new StringField("tenantId", o.tenantId(), Field.Store.NO));
    doc.add(new StringField("shopId",   o.shopId(),   Field.Store.NO));
    doc.add(new StringField("status",   o.status().name(), Field.Store.YES));

    // 全文字段:分词写倒排
    doc.add(new TextField("titleText",   o.title(), Field.Store.YES));
    doc.add(new StringField("skuExact",  normalizeSku(o.sku()), Field.Store.YES));
    doc.add(new TextField("addressText", o.shippingAddress(), Field.Store.NO));

    // 时间:三向投影
    doc.add(new LongPoint("paidAtEpochMs",            o.paidAtEpochMs()));
    doc.add(new NumericDocValuesField("paidAtSort",   o.paidAtEpochMs()));
    doc.add(new StoredField("paidAtText",             o.paidAtIsoText()));

    // 金额:三向投影
    doc.add(new LongPoint("amountCents",            o.amountCents()));
    doc.add(new NumericDocValuesField("amountSort", o.amountCents()));
    doc.add(new StoredField("amountText",           o.amountText()));

    return doc;
}

0.4 倒排索引的访问语义

倒排索引的核心数据结构是 term → posting list。posting list 中每个元素包含 docID、term frequency,可选包含 position 与 payload。Lucene 在物理层将 posting 拆为三个文件,按需读取——TermQuery 只需 .doc,PhraseQuery 才打开 .pos,高亮才用 .pay。

操作平均最坏主要消耗

倒排适用

  • 选择度高的 term 检索:freq ≪ N 时,O(freq) 即可遍历
  • 多 term 求交:leapfrog 算法在最低 freq 上界内完成
  • 短语与位置:.pos 文件支持 PhraseQuery / SpanQuery

倒排不适用

  • 范围查询:term 数巨大时不适合(用 Points 替代)
  • 排序:无 docID → value 路径(用 DocValues 替代)
  • 取回展示:无完整字段值(用 Stored Fields 替代)
  • 删除:逻辑删除位图 .liv,物理移除依赖 merge

0.5 Stored Fields 的取回语义

Stored Fields 是搜索的"返回路径":TopK 排序完成后,Lucene 才用命中的 docID 去 .fdt 解压取出字段。这里的关键设计是 "延迟取回"——倒排 / Points / DocValues 阶段都不读 Stored Fields,大幅减少全 chunk 解压的 I/O。

0.6 DocValues 的列式语义

DocValues 在物理层是 docID 索引的列式数组。Lucene 通过 PackedInts 按位压缩,每 16384 个 doc 组成一个 block,block 内按 min-delta 编码。读取时通过 .dvm 中的 block 索引定位到块,再按 PackedInts 位宽解码出值——典型 1 个 long 仅占 3-5 bit,远小于 8 字节。

Elasticsearch 的 doc_values 默认开启原因 — ES 从 2.0 开始默认对所有非 analyzed 字段开启 doc_values,即使你只用全文搜索不排序。原因:fielddata(早期 ES 用堆内存做"伪 DocValues")在大字段上很容易触发 OOM,而 doc_values 走 mmap 让 OS page cache 接管。代价:索引大小膨胀 30-50%。但避免 OOM 的稳定性收益远超磁盘成本——这是大型搜索集群的"默认安全网"。

0.7 Points / BKD-tree 的范围语义

BKD-tree(Block KD-tree)是 Lucene 默认的多维点索引结构。每个内部节点描述一个超立方体 cell,叶节点存储具体 doc 的多维坐标。范围查询时按"整体外 / 整体内 / 相交"三态判断,前两种立即给出答案,只有第三种需要继续下钻。

codec 命名空间是版本兼容的关键 — Lucene 升级时,新版本可能引入新的 PostingsFormat / DocValuesFormat。文件名中的 _Lucene90_ 让旧 segment(用旧 codec)与新 segment(用新 codec)在同一目录共存——只要 SegmentInfos 记录了每个 segment 用什么 codec,DirectoryReader 就能调度到对应的 reader。这种"代际兼容"让 Lucene 能跨大版本无缝升级,不要求一次性 reindex。

查询执行的"延迟取回"时序

图 0-17 · 倒排 → DocValues → Stored 的访问顺序
"延迟取回"是 Lucene 查询性能的核心 — 倒排和 Points 阶段只产 docID 与候选评分,排序阶段只读 DocValues 列,Stored Fields 仅在 TopK 收尾时按命中 docID 解压。该顺序最小化磁盘 I/O——1 千万 doc 的复合查询能在 P99 < 50 ms 内完成。如果你把"延迟取回"破坏(例如在 collector 里就调 storedFields.document),P99 会立即抬到 500 ms+。

flush vs commit

图 0-19 · RAM → Flushed → Committed 的状态机
flush ≠ commit — flush 仅将内存数据写入 segment 文件;commit 才更新 segments_N 使新数据"对外可见"。两者的关键差异在持久性与可见性:flush 后崩溃,segment 文件存在但没被引用,下次启动时被当作垃圾清理;commit 后崩溃,新 segments_N 已落盘,数据保留。NRT(Near Real-Time)reader 能读到 flushed 但未 commit 的段,所以它可以做到"亚秒级可见",代价是这部分数据在崩溃时可能丢失——这是个明确的工程取舍。

0.14 设计取舍

Lucene 在多个维度上做了"看起来更繁琐但更稳健"的选择。把这些取舍并列摆出,有助于理解后续每一章的细节。

设计点Lucene 选择放弃的简单方案获得的能力付出的代价
Segment不可变 + merge单一可变索引读写隔离、原子 commit、缓存友好查询 fan-out + merge 成本
Analyzer字段级 TokenStream原文直接索引语言归一化、召回控制、高亮需要 token 回归测试
倒排term → doc/freq/pos 压缩列表扫描全部文档全文召回快更新非原地、依赖 LiveDocs
DocValuesdocID → value 列式StoredField 解析排序 / 聚合稳定写入多存一份
PointsBKD-tree 多维点字符串范围比较数值 / 时间 / 地理范围快不直接排序或展示
Stored FieldsTopK 后行式取回搜索阶段加载 JSON搜索阶段 I/O 小需明确 stored 边界
HNSW图近似最近邻暴力 KNN向量召回 < 10 ms召回率非 100%
不可变 segment 的级联效应 — 删除一个 doc 不会立即从 .doc / .fdt 中移除字节,而是在 .liv 中标记。物理移除发生在该 segment 参与 merge 时;merge 完成后旧 segment 整体丢弃,新 segment 不含已删除 doc。这种"延迟回收"的级联效应在生产中表现为:你 deleteAll() 后磁盘没立即变小、forceMerge(1) 才能真正回收空间、merge 期间 IO 繁忙是 Lucene 在"补还历史欠债"。理解这一点能让你不再对"为什么删了 100 万 doc 磁盘没变化"感到困惑。
