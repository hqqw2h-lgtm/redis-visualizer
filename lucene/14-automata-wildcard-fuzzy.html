14.1 模糊匹配的需求:从业务到自动机

用户在搜索框打 iphone 想找 iPhone 商品没问题——倒排索引在 .tim 里二分查 iphone 这个 term,毫秒级返回。可如果用户:① 打 iphon* 想覆盖 iphone-7 / iphone-8 等所有变体(WildcardQuery);② 打 iphne 漏字了想容错(FuzzyQuery);③ 在日志里搜 order-\d{6} 这种正则(RegexpQuery)——倒排索引立刻不够用,因为词典里根本没有这三个模式,需要的是"找出所有匹配模式的 term"。

为什么不能用倒排逐 term 暴力枚举?

假设业务字段 itemName 有 1000 万独立 term:
  apple, apple-7, apple-watch, banana, charger, charger-30w, ...

朴素方案:遍历整个 .tim,每个 term 用 java.util.regex 匹配模式
  for (term : terms) {
      if (Pattern.matches("char.*", term)) {
          collect(term);
      }
  }

代价分析:
  N = 10^7 个 term
  L = term 平均长度 ≈ 16 字节
  总比较次数 = N × L = 1.6 × 10^8 次字符比较
  耗时:~500 ms 单查询(假设 java.util.regex 优化版)

灾难场景:
  1. 高 QPS(1000 QPS)→ CPU 直接打满
  2. 索引增长 10×(到 1 亿 term)→ 单查询 5 秒
  3. 嵌套正则 → 回溯放大,某些模式可能数小时不返回

自动机求交的核心思路

把查询模式(char*)和词典(.tip 内的 FST)都看作有限状态机——前者描述接受的字符串集合(language),后者描述已知存在的字符串集合。两个状态机的"接受语言"取交集就是命中的 term 集合。Lucene 实现这个交集的方式不是先各自展开再求交(那样退化为暴力),而是双指针同步走:在每一步,让自动机告知"下一字符可走哪些"、FST 告知"下一字符在词典里有哪些",取双方都允许的转移继续走,任一不允许就立刻剪枝整棵子树。

表 14-1-2 · 暴力枚举 vs 自动机求交的代价对比
查询模式暴力 N×L自动机求交 候选 term加速比

剪枝越早越值钱。自动机与 FST 求交是逐字符走的——前缀越长、第一个字符的可选转移越窄,FST 子树就越早被完整剪掉。前缀 6 字符的 iphone* 在第一步就把 'a' 'b' 'c' 'd' 'e' 'f' 'g' 'h' 这 8 个根 arc 全砍了,剩下 'i' 子树里继续砍、再砍、再砍——剪枝越深效率越高。这也是为什么 leading wildcard *foo 是性能毒瘤——第一字符任意,FST 整棵树都要走一遍。

14.2 DFA / NFA 基础

有限自动机(Finite Automaton)是计算理论里最简单的状态机模型。给定一个字母表 Σ(对 Lucene 来说就是 Unicode 码点),自动机由 (状态集 Q, 转移函数 δ, 起始状态 q₀, 接受状态集 F) 四元组定义。它读入一个字符串,根据当前状态 + 当前字符决定下一状态,读完字符串后落在接受状态就接受该字符串,否则拒绝。所有正则语言(包括 Lucene 的 Wildcard / Regexp / Levenshtein 模式)都能用有限自动机精确表达。

DFA 与 NFA 的关键区别

NFA(非确定有限自动机)
转移函数:δ(q, c) → 状态集合(可多目的地)
ε 转移:允许不消耗字符就跳转
同输入同状态:可有多条转移
构造:从正则直接构建(Thompson 构造)
判定:O(L × N²),L=输入长,N=状态数
表达力:与 DFA 等价,可识别所有正则语言
用途:中间表示(便于构造)
DFA(确定有限自动机)
转移函数:δ(q, c) → 唯一状态
无 ε 转移:每一步必消耗一字符
同输入同状态:仅一条转移
构造:从 NFA 子集构造(可能膨胀)
判定:O(L) 线性扫一遍即可
表达力:与 NFA 等价
用途:Lucene 查询时必须是 DFA

第 2 步:子集构造,起始 D0 = ε-closure(s0)

  on 'b' → {s3, s5} = D2 (自循环)
  on 'c' → {s3, s5} = D2 (自循环)
  on 'd' → {s6}     = D3 ✓

NFA 7 状态 → DFA 4 状态(本例缩小,但下一节会看到指数膨胀的危险)
为什么必须是 DFA? Lucene 在 Terms.intersect 里逐字符走自动机,每一步必须知道唯一的下一状态——多状态会导致回溯,把 O(L) 退化到 O(L × N²)。所以 Operations.determinize() 是 CompiledAutomaton 构造时必经的一步。

14.3 Lucene Automata 与 Operations

Lucene 把自动机的所有数据结构与算法集中在 org.apache.lucene.util.automaton 包。Automaton 类是数据载体——本质是一个有向带标签图:每个节点是状态,每条边携带 (min, max, target) 三元组表示"输入字符落在 [min, max] 区间则转移到 target"。这种区间编码非常关键:一个 ?(任意单字符)在 Wildcard 里只占一条边 (0, MAX_UNICODE, target),而不是 10⁵ 条边。

14.4 WildcardQuery 编译

Wildcard 是最简单的"模糊查询",只有两个元字符:*(零或多任意字符)与 ?(单任意字符)。其他字符都是字面值,直接转换为 transition。\ 是转义符,可以让 \* 表示字面 *。

元字符的 NFA 翻译规则

表 14-4 · Wildcard 元字符 → NFA
字符含义NFA 翻译区间编码
字面 'c'必须为 c单转移到下一状态(c, c, next)
?任意单字符单转移,标签为 ANY(0, MAX_CODE, next)
*零或多任意self-loop + ε 转移self (0, MAX_CODE, self) + ε to next
\c转义,字面 c等同字面字符(c, c, next)

WildcardQuery "char*" 的 automaton 构造

WildcardQuery("items", "char*") 编译过程:

  字符级解析:'c''h''a''r' 字面字符,'*' = self-loop + ε

  NFA 构造(逐字符 concat):
    s0 ─c→ s1 ─h→ s2 ─a→ s3 ─r→ s4 ┄ε┄┄→ s5 (accept, ANY self-loop)

  determinize 后的 DFA:
    s0 ─c→ s1 ─h→ s2 ─a→ s3 ─r→ s4 ─ANY→ s4 (accept)

  接受语言示例:
    "char"      → s0→s1→s2→s3→s4(accept)             ✓
    "chargers"  → s0→s1→s2→s3→s4→s4→s4→s4→s4(accept) ✓
    "char-7"    → s0→s1→s2→s3→s4→s4→s4(accept)       ✓
    "ch"        → s0→s1→s2 不在 accept                ✗
    "chair"     → s0→s1→s2→s3 'i' 无转移              ✗

14.5 FuzzyQuery 与 Levenshtein 自动机

FuzzyQuery 解决用户输入打错字的容错搜索:用户搜 iphne(漏字"o"),期望也能命中 iphone。这里"匹配"不再是精确串等于,而是 编辑距离 ≤ k。Levenshtein 编辑距离允许三种操作:插入字符、删除字符、替换字符;Damerau-Levenshtein 额外允许字符交换。Lucene 把"target 与候选的编辑距离 ≤ k"这个语义编译成一个状态机——这就是 Levenshtein 自动机。

编辑距离的四种操作

表 14-5-1 · 编辑距离的基本操作

Levenshtein 自动机的状态定义

把状态定义为 (target 中的位置 i, 当前累积编辑距离 e)。从 (0, 0) 出发,每读入一个候选字符:① 与 target[i] 相等 → 走 (i+1, e);② 替换(假定该字符与 target[i] 不同)→ 走 (i+1, e+1);③ 插入(候选有多余字符)→ 走 (i, e+1);④ 删除(候选缺字符)→ 走 (i+1, e+1)。当 i = |target| 且 e ≤ k 时接受。

14.6 RegexpQuery

RegexpQuery 比 Wildcard 更强大,支持完整(但有限)的正则语法:字符类 [a-z]、量词 * + ?、选择 |、分组 (...) 等。Lucene 用 Thompson 构造法把正则编译为 NFA,再 determinize 为 DFA。但有意限制了不少 PCRE 元素——是不能支持,而是它们要么会让 NFA → DFA 不可控膨胀,要么超出"正则语言"范畴(必须用反向引用、回溯)。

"order-[0-9]+" 的编译过程

RegexpQuery 内部步骤:

  1. 词法解析 "order-[0-9]+":
     ['o', 'r', 'd', 'e', 'r', '-', char_class([0-9]), '+']

  2. Thompson 构造,逐元素拼 NFA:
     'o' 'r' 'd' 'e' 'r' '-' 都是字面字符
     [0-9] 是字符区间 transition
     + 是 1+ 量词,翻译为 (子NFA 至少走一次,再 self-loop)

  3. 拼接结果(NFA, 含 ε):
     s0 ─o→ s1 ─r→ s2 ─d→ s3 ─e→ s4 ─r→ s5 ─-→ s6
                                                  ┊
                                                  ε
                                                  ▼
                                             s7 ─[0-9]→ s7 (accept)

  4. determinize:
     ε 转移消除,s6 与 s7 合并(可空 ε 链)
     最终 DFA(7 状态):
       s0 ─o→ s1 ─r→ s2 ─d→ s3 ─e→ s4 ─r→ s5 ─-→ s7 ─[0-9]→ s7(accept)

  5. minimize:
     无可合并状态(每个状态语义不同)→ 7 状态保持

  6. CompiledAutomaton 包装,准备求交
嵌套量词 (a+)+ 是 RegexpQuery 头号陷阱。这种模式让 NFA 内部出现"多重 ε 入口指向同一状态",determinize 时子集大小指数膨胀。Lucene 加了 maxDeterminizedStates 上限(默认 10000)防止 OOM,超过即抛 TooComplexToDeterminizeException。生产代码必须在业务层校验:① 禁用 (...)+ 形式;② 模式长度不超过 100;③ 最好做正则白名单。"用户输入直接拼 RegexpQuery"是常见的安全漏洞,可被 ReDoS 攻击。

14.8 与 .tim 的协同:Terms.intersect 算法

所有前面的章节铺垫都在等这一节:CompiledAutomaton 与 .tip + .tim 词典的 FST双指针同步走。这是 Wildcard / Fuzzy / Regexp 性能的真正决定环节——前面所有编译都很便宜(毫秒级),求交才是主成本(几毫秒到几百毫秒,取决于剪枝率)。

求交的核心思路:双指针走

  s0 上 'c' 转移?  ❌

  s0 上 'i' 转移?  ✅ 走到 sA=s1
  → 进入 .tip 'i' 子树,sFST=ice 节点

    arc 'c' (ice 路径) → s1 上 'c' 转移? ❌ 跳过
    arc 'd' (idea)     → s1 上 'd' 转移? ❌ 跳过
    arc 'p' (iphone)   → s1 上 'p' 转移? ✅ 走 s2
      在 ip 子树:
        arc 'h' (iphone) → s2 上 'h' ? ✅ s3

                  arc '-' → s6 ANY ✅ s6
                    arc '7' → s6 ANY ✅ s6 → emit "iphone-7" ✓
                    arc '8' → s6 ANY ✅ s6 → emit "iphone-8" ✓
                    arc 'p' → s6 ANY ✅ s6 → emit "iphone-pro" ✓

Step 5:  arc 'p' (phone, paper)
  s0 上 'p' 转移?  ❌
  → 'p' 子树剪枝(phone, paper 全跳过)
收益统计:
访问的 .tim 子树:
  - 跳过: a/b/c/p 子树(假设各 100K-500K term)
  - 走入: i 子树 → ip 子树 → iphone 子树
  - 触发:仅 iphone 子树的 term 被列出

14.9 1024 term 阈值与 CONSTANT_SCORE_REWRITE

求交完毕只是第一步——Terms.intersect 返回的是命中 term 的迭代器(IntersectTermsEnum),还要决定怎么把这些 term 转成最终查询。MultiTermQuery(Wildcard / Fuzzy / Regexp 的共同父类)默认行为是把每个命中 term 包装成 TermQuery,再用 BooleanQuery(SHOULD) 取并集。但 BooleanQuery 有 maxClauseCount = 1024 的硬性上限——超过就抛 TooManyClauses。

import org.apache.lucene.search.WildcardQuery;
import org.apache.lucene.search.MultiTermQuery;
import org.apache.lucene.search.BooleanQuery;
import org.apache.lucene.index.Term;

WildcardQuery wq = new WildcardQuery(new Term("itemName", "i*"));

wq.setRewriteMethod(MultiTermQuery.CONSTANT_SCORE_BOOLEAN_REWRITE);
wq.setRewriteMethod(MultiTermQuery.CONSTANT_SCORE_REWRITE);
wq.setRewriteMethod(new MultiTermQuery.TopTermsScoringBooleanQueryRewrite(50));

BooleanQuery.setMaxClauseCount(8192);

TopDocs top = searcher.search(wq, 100);

1024 阈值的来源

为什么 BooleanQuery 默认 maxClauseCount=1024?

  1. 内存:每个 BooleanClause 包装一个 Query + 一个 Occur enum
     1024 clause ≈ 1024 × ~200 bytes = ~200 KB 单查询
     若 1 万,延迟到 100 万次构造,GC 压力显著

  2. 评分代价:BM25 + Disjunction 求并集是 leapfrog 多路归并
     每多一个 clause 就多一个 PostingsEnum 与堆操作
     1024 clause 已是 BooleanQuery 性能拐点

  3. 实际经验:99% 用户合理查询都 < 100 clause
     超过 1024 通常意味着 wildcard 或 fuzzy 失控
     抛异常比悄悄变慢更"诚实"

  4. 上下文限制:Lucene 历史上 maxClauseCount=1024 是全局 static
     8.0+ 改为可配置但默认仍 1024,以发现错误用法

  突破方案:CONSTANT_SCORE_REWRITE
  原理:
    - 不构造 BooleanQuery
    - 用 DocIdSetBuilder 直接收集所有命中 doc 的 docID
    - 包装为 ConstantScoreScorer,所有 doc 评分=1.0(或 boost)
    - 可处理百万级 term 命中,延迟与命中数线性相关
TooManyClauses 是 wildcard/fuzzy 最常见的生产异常。WildcardQuery("i*") 在 1000 万 term 索引上展开 ~10 万 term,默认 SCORING_BOOLEAN_REWRITE 立刻抛 TooManyClauses。修复:① 业务限制前缀长度 ≥ 3,把命中量降到 < 1000;② 或显式 setRewriteMethod(CONSTANT_SCORE_REWRITE),牺牲精确评分换大命中可行性;③ 不要简单地 setMaxClauseCount(1000000)——这只是把异常推迟到 OOM。

"leading wildcard" 指的是模式以 * 或 ? 开头,如 *phone、?phone、*phone*。这类模式让自动机的起始状态接受任意首字符,导致 .tim 求交时根 arc 一个都剪不掉,FST 整棵树都要走一遍——这是 Lucene 性能头号毒瘤。

ngram 索引方案

业务真需要"中缀匹配"(搜索栏所见即所得)时,改用 NGramTokenFilter:索引侧用 NGramTokenFilter(min=3, max=5) 生成所有 3-5 字符子串,查询侧用普通 TermQuery,完全绕开 wildcard。例:iphone-12-pro 切成 ["iph", "pho", "hon", "one", "ne-", ...]。

存储成本: ngram 数 ≈ |term| × (max - min + 1) ≈ 12 × 3 = 36 ngram/原 term
          1000 万 term → 3.6 亿 ngram, .tim 大 5-10×

查询代价: TermQuery("phone") → 直接 .tim 二分 → < 1 ms
          无自动机求交,无 1024 阈值问题

权衡:    + 查询极快  + 用户体验好(无字符限制)
          - 索引大 5-10×  - 适合数据 < 1 亿 doc  - 重建索引慢

适用:    商品名搜索栏 / 日志关键词高亮
某电商站内搜索的 leading wildcard 雪崩。某团队上线"通配符搜索"功能,允许用户输入任意 wildcard 模式直接走 WildcardQuery。上线 2 周后,周末高峰一个用户输入 * 触发 MatchAll,接着另一个输入 *phone* 全表扫,5 分钟内 P99 从 50ms 飙到 8s,集群 12 个节点 CPU 全部 100%,搜索整个不可用。事后修复:① 业务层校验最少 3 字符前缀;② * 单独提示"请输入更多字符";③ 监控 TooManyClauses 异常率,> 0.1% 立刻报警;④ 后续切换到 ngram 索引,wildcard query 完全弃用。

14.11 PrefixQuery 优化

PrefixQuery 是 WildcardQuery 的"语义清晰版":只接受字面前缀,拒绝任何元字符。其内部仍走 CompiledAutomaton + Terms.intersect 路径,但 API 上更安全——不会因为用户输入中夹杂 * ? 触发意外行为。

WildcardQuery vs PrefixQuery

WildcardQuery("foo*")
API:new WildcardQuery(Term)
支持元字符:* ? \
解析:扫描模式找元字符位置
自动机:与 PrefixQuery 物理同构
陷阱:*foo* 被允许,leading wildcard 慢
用户输入安全:不安全(用户可输入元字符)
推荐:仅在业务真需要 ? 或中缀时
PrefixQuery("foo")
API:new PrefixQuery(Term)
支持元字符:无,字面前缀
解析:整个字符串作为前缀
自动机:foo + ANY self-loop
陷阱:前缀过短(< 3)候选过多
用户输入安全:安全(任何字符当字面)
推荐:绝大多数"以...开头"场景的首选

PrefixQuery 的字节级流程

PrefixQuery("itemName", "order-") 的执行:

  1. 构造 PrefixQuery
       prefix = "order-"

  2. 编译为 CompiledAutomaton(隐式)
       PrefixQuery 内部不显式调用 Automata.makePrefix,
       而是 buildAutomaton(prefix) 直接走最优路径
     自动机:
       s0 ─o→ s1 ─r→ s2 ─d→ s3 ─e→ s4 ─r→ s5 ─-→ s6 ─ANY→ s6 (accept)

  3. terms.intersect(ca, null) 求交
       根 arcs:'a' 'b' 'c' 'd' 'e' 'f' 'g' 'h' 'i' 'j' 'k' 'l' 'm' 'n' 'o' 'p'...
       走到 'o' arc,其余全剪
       'o' 子树继续:'r' 'p' 'l' ... → 走 'r' 子树
       逐字符走完 "order-",剩下 ANY self-loop
       展开 "order-" 子树下所有 term

  4. rewrite 决策
       命中数 ≤ 1024 → BooleanQuery(SHOULD)
       命中数 > 1024 → ConstantScoreQuery(默认)

  5. 与 PostingsEnum 配合,获取每 term 的 doc 列表
       leapfrog 求并集
       BM25 评分(若 SCORING_BOOLEAN_REWRITE)

14.12 业务场景与取舍

三个真实场景的"该用哪个查询、怎么调参"建议。每个场景都给出技术选型 + 代码 + 性能预算。

场景 A:商品名拼写容错(电商搜索)

用户输入商品名,允许 1 个字符的拼写错误,例:输入 iphne → 期望命中 iphone。数据规模 itemName 字段 1000 万独立 term。选型:FuzzyQuery(maxEdits=1, prefixLength=1);不用 LD=2 是因为误命中过多——iphne 在 LD=2 下会误中 phone / iPhone-7 等无关 term。

String userInput = "iphne";
if (userInput.length() < 3) {
    return new TermQuery(new Term("itemName", userInput));
}
Query q = new FuzzyQuery(
    new Term("itemName", userInput),
    1,        // maxEdits:单字符容错足够
    1,        // prefixLength:首字符必须精确
    50,       // maxExpansions
    true);    // transpositions:允许字符交换

性能预算:自动机构造 < 1 ms + Terms.intersect 2-5 ms + TopK 评分 1-2 ms = 5-10 ms P95。

场景 B:订单号 / SKU 前缀搜索

运营搜索某客户的订单 / 某品类的 SKU,订单号格式 ORD-20260510-XXXXX,运营输入 ORD-2026 看本年订单。选型:PrefixQuery(语义清晰、安全);WildcardQuery 等价但允许元字符,有用户输入风险;FuzzyQuery 不适用——订单号必须精确。

String prefix = request.getParameter("orderPrefix");
if (prefix == null || prefix.length() < 4) {
    throw new IllegalArgumentException("至少 4 字符前缀");
}
PrefixQuery q = new PrefixQuery(new Term("orderId", prefix));
q.setRewriteMethod(MultiTermQuery.CONSTANT_SCORE_REWRITE);

性能预算:前缀 4 字符候选 ~万级,Terms.intersect 5-10 ms,CONSTANT_SCORE 无 1024 限制 = 10-30 ms P95。

场景 C:日志正则搜索

运维搜日志匹配错误码 error-[0-9]{4}-(timeout|conn),数据规模 logEntry 字段 5 亿 term。选型:RegexpQuery,但必须做模式校验防 ReDoS。

public Query buildLogRegex(String pattern) {
    if (pattern.length() > 100) {
        throw new IllegalArgumentException("正则过长");
    }
    if (pattern.matches(".*\\([^)]*[+*?][^)]*\\)[+*?].*")) {
        throw new IllegalArgumentException("禁用嵌套量词");
    }
    if (!pattern.matches("^[a-zA-Z0-9\\-_/\\[\\]\\(\\)\\|\\+\\*\\?\\.,]+$")) {
        throw new IllegalArgumentException("非白名单字符");
    }
    RegexpQuery rq = new RegexpQuery(new Term("logEntry", pattern));
    rq.setRewriteMethod(MultiTermQuery.CONSTANT_SCORE_REWRITE);
    return rq;
}

性能预算:自动机构造 1-3 ms + Terms.intersect 10-50 ms = 20-100 ms P95。
