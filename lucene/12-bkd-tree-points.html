第 12 章 · BKD-tree 与数值范围索引

K-D Tree · BKD 不可变 · IntersectVisitor · CELL_INSIDE/OUTSIDE/CROSSES · LongPoint · LatLonPoint · 距离查询

订单搜索里有一类查询倒排索引解不了:amountCents BETWEEN 100 AND 10000、paidAt BETWEEN epoch1 AND epoch2、距离北京 5km 内。这些多维范围查询用倒排会爆——每个具体值是一个独立的 term,1 千万种金额值就是 1 千万个 term,效率毁灭性下降。Lucene 给出的答案是 BKD-tree(K-Dimensional Block tree):把数值空间递归二分,叶子节点保存若干点的紧凑列表,内部节点保存维度切分线,查询时通过 IntersectVisitor 三态(INSIDE / OUTSIDE / CROSSES)递归剪枝——窄区间命中只访问几个叶子,1 ms 出结果。

BKD 与倒排是两套独立机制:倒排适合等值与文本,BKD 适合范围与多维。一个订单的 amountCents 既能用 LongPoint 范围查询,又能用 NumericDocValues 排序与脚本评分,但必须双写——Lucene 不会自动从 BKD 推 DocValues。这一章拆解 BKD 的构造算法、.dim 与 .dii 两个文件的字节级布局、查询执行流程、性能取舍,以及 8 个生产陷阱。对应 Lucene 9.x 的 Lucene90PointsFormat。

  • K-D tree 的核心思想:递归二分多维空间,每层选一个维度切中位数
  • BKD 与传统 K-D 的关键区别:不可变、bulk-load、叶子 1024 点、磁盘友好
  • BKDWriter 的 add → spill → sort → build 四阶段构造流程
  • .dim(叶子点数据)与 .dii(内部节点索引)的物理布局,docID 与维度值如何 delta + ZigZag 压缩
  • IntersectVisitor 的 3 种 cell 关系:CELL_INSIDE_QUERY / CELL_OUTSIDE_QUERY / CELL_CROSSES_QUERY
  • visit(docID) vs visit(docID, packedValue) 的代价差异:批量 vs 逐点
  • PointRangeQuery / newSetQuery / newDistanceQuery 三种查询的内部展开
  • LongPoint 与 NumericDocValues 必须双写的原因,以及缺一个就抛 IllegalStateException 的场景

前置知识:第 1 章字段建模(知道 LongPoint 是 4 种访问路径之一);第 4 章 BooleanQuery 求交;第 9 章源码阅读路径(本章会引用 BKDWriter / BKDReader 等具体类)。读者需要熟悉二分搜索与递归数据结构。

12.3 BKD 与传统 K-D 的关键区别

BKD-tree 由 Procopiuc 等人 2003 年提出,核心创新是把叶子做成 block(默认 1024 点),便于磁盘 IO 与 page cache。Lucene 的实现是不可变 BKD,只在 segment flush 时构造,不支持单点插入;所有 segment 上的 BKD 在合并时重建。

传统 K-D Tree
存储:全内存
构造:逐点插入(支持 add)
平衡:难保证(需要旋转)
叶子:单点
典型用例:内存中的 KNN 搜索
不适合:Lucene segment(需磁盘 + 不可变)
BKD Tree(Lucene 实现)
存储:磁盘友好(.dim + .dii)
构造:bulk-load(全量排序后构建)
平衡:始终平衡(中位数切分)
叶子:1024 点 block
典型用例:Lucene segment 上的 K-D
限制:不可变,合并时重建

不可变带来的代价与收益

表 12-3 · 不可变 BKD 的取舍

写入bulk-load 比逐点插入快 5-10×必须等 segment flush 才可查

12.4 构造算法

BKDWriter 的 4 阶段

图 12-4-1 · BKDWriter 的构造流程

BKDWriter 关键参数

表 12-4 · BKDWriter 参数
参数默认说明
maxPointsInLeafNode1024叶子最大点数
maxMBSortInHeap16内存排序上限(超出 spill 到磁盘)
维度数1-8业务决定;通常 1-2
bytesPerDim4 / 8每维字节(int=4 / long=8)

递归构造伪代码

buildBKD(points, depth, dimensions):
    if len(points) <= maxPointsInLeafNode:
        emit leaf node with points    # 1024 点直接当叶子
        return

    dim = depth % dimensions          # 维度轮转
    sort points by dim                # O(N log N) 但只在该维度
    mid = len(points) / 2
    splitValue = points[mid][dim]

    leftPoints  = points[0..mid-1]    # ≤ splitValue
    rightPoints = points[mid..len-1]  # >  splitValue

    leftChild  = buildBKD(leftPoints, depth+1, dimensions)
    rightChild = buildBKD(rightPoints, depth+1, dimensions)

    emit internal node:
        splitDim   = dim
        splitValue = splitValue
        left       = leftChild
        right      = rightChild

1 千万订单的实际开销

1000 万订单的 amountCents BKD 构造(参考量级):

  add 阶段:
    每文档 add 点 → in-memory PointArray
    1 字段 8 字节 + 4 字节 docID = 12 字节/点
    1000 万点 ≈ 120 MB

  spill 阶段(若超 maxMBSortInHeap=16):
    OfflinePointReader / OfflinePointWriter 用临时文件
    分片排序 + 多路归并

  build 阶段:
    全量按维度排序
    深度 = log2(N / maxPointsInLeafNode)
            = log2(10000000 / 1024)
            ≈ 14 层
    叶子数 ≈ 9766

  时间:
    内存场景: 5-15 s
    spill 场景: 30-90 s

  输出:
    .dim 大小 ≈ 100-150 MB(含压缩)
    .dii 大小 ≈ 200 KB(只是内部节点索引)
14 层 = 9766 叶子,不是直觉的"百万级节点"。BKD 是平衡树,深度恒定 log₂(N/1024) ≈ 14;每个叶子 1024 点是核心 IO 单位,叶子内部用 PFOR + ZigZag 压缩。整棵树的"有意义"节点数只有 ~10 K,内部节点信息全部装进 .dii 文件(约 200 KB),内存可全量加载。

.dii 文件结构(BKD 内部节点索引):

  ┌──────────────────────────────────────────┐
  │ Header                                    │
  │   magic + codec name + version            │
  ├──────────────────────────────────────────┤
  │ Per-Field Index                           │
  │   field A:                                 │
  │     numLeaves (VInt)                       │
  │     fieldStats                             │
  │     packed index data                      │
  │       internal nodes (split values + dim)  │
  │       leaf pointers (file offsets)         │
  │   field B:                                 │
  │     ...                                    │
  ├──────────────────────────────────────────┤
  │ DirOffset(8B)                             │
  ├──────────────────────────────────────────┤
  │ Footer (CRC32)                            │
  └──────────────────────────────────────────┘

内部节点的紧凑编码

每个内部节点:
    splitDim    (1 byte)         ← 切分的维度索引(1D 时省略)
    splitValue  (bytesPerDim)    ← 切分点(8B for LongPoint)
    leftCount   (VInt)            ← 子树叶子数
    leftOffset  (delta)           ← 左子树在 .dim 的相对偏移

  右子树 = 父节点偏移 + leftSize
  以此实现"前序遍历"的紧凑序列化
.dii 完全可 mmap 进内存。1 千万 doc 的索引,叶子约 9766 个,内部节点约 9765 个,每个内部节点编码 12-16 字节,总 .dii 大小 200-300 KB——比 page size 还小。Lucene 启动时一次性把整个 .dii 加载进 BKDReader,后续 BKD 树遍历完全在内存中,不触磁盘 IO。每次磁盘 IO 都是叶子访问(.dim 文件的 4-8 KiB block 顺序读),这是 BKD 性能的关键。

12.8 IntersectVisitor 接口

BKD 查询的核心接口是 IntersectVisitor。它通过三种 cell 关系(INSIDE / OUTSIDE / CROSSES)递归剪枝:整 cell 在查询范围内则批量收集,完全在外则剪枝,部分相交则递归子树或逐点判断。

三种 cell 关系(决策树)

图 12-8-1 · BKD cell 与查询范围的三种关系(手绘)
查询 [qMin=100, qMax=1000] vs cell [min, max]
① CELL_INSIDE
qMin=100  qMax=1000
   │         │
   ├─[200,800]──┤  cell
   │         │
cell 完全在 q 内
批量 visit(docID)
不做值比较
② CELL_OUTSIDE
qMin=100  qMax=1000
   │         │
   │         │  ├─[2000,5000]──┤
   │         │
cell 完全在 q 外
整子树剪枝
不递归不读叶子
③ CELL_CROSSES
qMin=100  qMax=1000
   │         │
   ├─[600,1500]─────┤  cell
   │         │
cell 部分重叠 q
递归子树 / 叶子内逐点
visit(docID, packedValue)
优化原理: ① INSIDE 的整 cell(可能 1024 点)用一次批量 visit,不需要值比较 — 跳过的工作量是 visit 总成本的 80% 以上;② OUTSIDE 让整子树根本不被打开;③ CROSSES 是真正的"硬开销",所有 BKD 调优都是为了把 INSIDE/OUTSIDE 的比例做大。

IntersectVisitor Java 接口

public abstract class IntersectVisitor {
    public abstract void visit(int docID) throws IOException;
    public abstract void visit(int docID, byte[] packedValue) throws IOException;
    public abstract Relation compare(byte[] minPackedValue, byte[] maxPackedValue);
}

public enum Relation {
    CELL_INSIDE_QUERY,
    CELL_OUTSIDE_QUERY,
    CELL_CROSSES_QUERY
}

12.9 PointRangeQuery 实现

Query q = LongPoint.newRangeQuery("amountCents", 1000L, 100000L);
TopDocs top = searcher.search(q, 100);
内部展开:
  1. createWeight 不收集统计量(范围查询不评分)
  2. scorer:
     - 通过 PointValues.intersect(visitor) 遍历 BKD-tree
     - visit 每个匹配 docID
     - 收集成 DocIdSet(FixedBitSet 或 RoaringDocIdSet)
  3. ConstantScoreScorer 包装为 docID iterator,score=1.0(常量)

命中比与延迟的关系

表 12-9 · Range Query 性能(1 千万 doc 量级)
查询范围命中比访问叶子数延迟
[极小, 极大]100%全部 ~9766200-500 ms

窄区间查询 1 ms 级,宽区间百 ms 级。某电商订单搜索最常见的"30 天内 100-1000 元订单"实测 P99 = 8ms,因为 30 天窗口在全数据中只占 5% × 0.5% = 0.025% 命中。但如果用户翻 1 年内全部订单,BKD 几乎全树扫,P99 飙到 380ms。这就是为什么订单搜索默认时间窗口要约束在 30-90 天:让 BKD 始终走窄区间路径。

// 矩形查询(LatLonPoint)

    minLat, maxLat, minLon, maxLon);

// 距离查询(中心点 + 半径)

    39.9, 116.4,    // 中心: 北京

2 维剪枝示例

2 维查询 LatLonPoint 城市矩形 [(39, 116), (40, 117)]:

  根节点 cell: 全球范围
  → CELL_CROSSES → 进入

  左子: lat < 0 cell
  → CELL_OUTSIDE → 剪枝

  右子: lat >= 0 cell
  → CELL_CROSSES → 进入

  右子的左子: 0 <= lat < 30
  → CELL_OUTSIDE → 剪枝

  右子的右子: 30 <= lat < 60
  → CELL_CROSSES → 进入

  ... 递归到叶子 ...

  最终仅访问几个叶子,大量剪枝

距离查询内部仍是 BKD 矩形查询(取最小外接矩形,MBR),再在 visitor 里加 Haversine 距离过滤。剪枝效率取决于半径与 cell 大小的比值——半径越小,MBR 越窄,剪枝越好。

12.13 与 DocValues 配合(必须双写)

这是 BKD 最大的"非显然"陷阱:LongPoint 只支持范围查询,不支持排序;排序需要 NumericDocValues;两者必须同时写。

Range + Sort 的执行流程

完整的范围查询 + 排序:

  Query q = LongPoint.newRangeQuery("paidAt", t1, t2);
  Sort sort = new Sort(new SortField("paidAt_dv",
      SortField.Type.LONG, true));

  执行流程:
    1. BKD intersect → DocIdSet 候选(范围内 doc)
    2. 候选过 LiveDocs 过滤(已删除)
    3. 用 NumericDocValues 取每候选的 paidAt_dv
    4. TopFieldCollector 维护小顶堆排序

  注意:
    - LongPoint 用于 range,DocValues 用于 sort
    - 两者必须双写
    - LongPoint 有 .dim,DocValues 有 .dvd

双写的字段建模

Document doc = new Document();
doc.add(new StoredField("orderId", orderId));

// LongPoint:支持范围查询 (写入 .dim/.dii)
doc.add(new LongPoint("amountCents", amountCents));
doc.add(new LongPoint("paidAt", paidAtMillis));

// NumericDocValues:支持排序与脚本 (写入 .dvd/.dvm)
doc.add(new NumericDocValuesField("amountCents", amountCents));
doc.add(new NumericDocValuesField("paidAt", paidAtMillis));

// 倒排:支持等值过滤(如 status, 写入 .tim/.tip/.doc)
doc.add(new StringField("status", status, Field.Store.NO));

writer.addDocument(doc);
漏掉 NumericDocValuesField 直接抛 IllegalStateException:searcher.search(query, n, sort) 时报 unexpected docvalues type NONE for field 'paidAt'。Lucene 不会从 BKD 自动推 DV——这俩是物理上独立的文件。生产里第一次跑 sort 抛此异常,几乎 100% 是字段建模忘了双写,而不是查询写错。

12.15 性能场景与量化

场景 1:订单金额范围查询

索引: 1 亿订单
字段: amountCents (LongPoint + NumericDocValues)
查询: amountCents BETWEEN 5000 AND 10000  (1% 命中, 100 万订单)
Top 100, sort by paidAt DESC

执行 breakdown(参考量级):
  1. BKD intersect:
     - 访问 ~100 个叶子(1% × 9766)
     - 收集 ~1 M 候选 DocIds
     - 耗时 30-50 ms

  2. LiveDocs 过滤:
     - 移除已删除文档
     - 耗时 5-10 ms

  3. NumericDocValues 取 paidAt:
     - 1 M 次随机访问 .dvd
     - 耗时 100-200 ms

  4. TopFieldCollector 排序:
     - 100 容量小顶堆
     - 耗时 50-100 ms

  Total: 200-360 ms

对比:无 NumericDocValues 直接报错

场景 2:LatLon 距离查询

索引: 1000 万门店
字段: location (LatLonPoint)
查询: 距离 (39.9, 116.4) 5 km 内

执行:
  1. 计算最小外接矩形(MBR):
     - lat ± 0.045°, lon ± 0.058°(约 5km)

  2. BKD intersect MBR:
     - 访问 ~50 个叶子
     - 收集 ~5000 候选(MBR 大于实际圆)
     - 耗时 5-10 ms

  3. visitor 过滤:
     - 对每个候选计算 Haversine 距离
     - 距离 > 5 km 的剔除
     - 耗时 2-5 ms

  4. 最终命中: ~3000 门店

  Total: 10-20 ms

场景 3:索引大小估算

1 亿订单 amountCents (LongPoint 1D, 8 bytes/point):

  原始数据: 100M × 8B = 800 MB
  BKD 索引大小:
    .dim ≈ 600-800 MB(含 docID + 压缩后值)
    .dii ≈ 2-5 MB(内部节点很少)

  对比:
    NumericDocValues .dvd ≈ 200-400 MB(列存压缩)
    倒排 (StringField 表示金额): 不可行(高基数)

  结论:
    范围查询建 LongPoint
    排序建 NumericDocValues
    高基数不要用倒排

12.16 反例 → 修复对照

反例 1:漏写 NumericDocValues

// 反例:只写 LongPoint
Document doc = new Document();
doc.add(new LongPoint("paidAt", paidAtMillis));
writer.addDocument(doc);

// 查询:要按 paidAt 排序
Query q = LongPoint.newRangeQuery("paidAt", t1, t2);
Sort sort = new Sort(new SortField("paidAt", SortField.Type.LONG, true));
TopDocs top = searcher.search(q, 100, sort);
// 抛: IllegalStateException: unexpected docvalues type NONE for field 'paidAt'

// 修复:同时写 LongPoint 和 NumericDocValues
Document fixed = new Document();
fixed.add(new LongPoint("paidAt", paidAtMillis));
fixed.add(new NumericDocValuesField("paidAt", paidAtMillis));
writer.addDocument(fixed);

反例 2:用 LongPoint 表达字符串区间

// 反例:把字符串"哈希成 long"再范围查询
long startHash = "ABC".hashCode();
long endHash = "XYZ".hashCode();
Query q = LongPoint.newRangeQuery("status", startHash, endHash);

// 后果:
// 1. hashCode 不保序,startHash 可能 > endHash
// 2. 不同字符串可能 hash 冲突
// 3. 查询语义完全错

// 修复:用 TermRangeQuery + StringField
Query fixed = TermRangeQuery.newStringRange(
    "status", "active", "pending", true, true);

反例 3:维度膨胀

// 反例:把 4 个独立字段塞成 4D Point
doc.add(new LongPoint("metrics",
    cpu, memory, disk, network));  // 4D LongPoint

// 后果:
// 1. .dim 大小 ~4× 单维
// 2. 范围查询 1 个维度也得遍历 4D BKD
// 3. CELL_CROSSES 比例显著提高
// 4. 查询延迟 5-10× 单维

// 修复:每字段独立 1D LongPoint + BooleanQuery
doc.add(new LongPoint("cpu", cpu));
doc.add(new LongPoint("memory", memory));
doc.add(new LongPoint("disk", disk));
doc.add(new LongPoint("network", network));

BooleanQuery.Builder bq = new BooleanQuery.Builder();
bq.add(LongPoint.newRangeQuery("cpu", 80, 100), Occur.MUST);
bq.add(LongPoint.newRangeQuery("memory", 80, 100), Occur.MUST);
// ... 多 BKD 并行剪枝,延迟显著低于 4D

反例 4:极宽范围未加 filter

// 反例:范围查询命中 99% 文档
Query q = LongPoint.newRangeQuery("amountCents", 1L, Long.MAX_VALUE);
// 然后期待返回 Top 100 by paidAt
// 后果:BKD 几乎全表扫,1 亿 doc 延迟 1-2 s

// 修复:加业务 filter 缩小范围
BooleanQuery.Builder bq = new BooleanQuery.Builder();
bq.add(new TermQuery(new Term("status", "completed")), Occur.MUST);  // 缩到 30%
bq.add(LongPoint.newRangeQuery("amountCents", 1000L, 100000L), Occur.MUST);  // 1%
bq.add(LongPoint.newRangeQuery("paidAt", t1, t2), Occur.MUST);  // 时间窗口
// 联合 filter 后命中 < 1%,延迟 < 100 ms
多 BKD 并行剪枝远优于多维 BKD。某监控系统把 (cpu, memory, disk, network) 4 维度合并成一个 4D LongPoint,查询"cpu > 80 AND memory > 80"的延迟从 200ms 直接飙到 1.8s。改成 4 个独立 1D LongPoint + BooleanQuery 后,延迟降到 35ms — BooleanQuery 让每个 BKD 独立剪枝,然后 leapfrog 求交,远比一棵 4D 树同时切 4 个维度高效。

自定义 IntersectVisitor 示例

PointValues pv = leafReader.getPointValues("amountCents");
pv.intersect(new IntersectVisitor() {
    DocIdSetBuilder builder = new DocIdSetBuilder(maxDoc);

    @Override
    public Relation compare(byte[] minPackedValue, byte[] maxPackedValue) {
        long min = LongPoint.decodeDimension(minPackedValue, 0);
        long max = LongPoint.decodeDimension(maxPackedValue, 0);
        if (min > queryMax || max < queryMin) return CELL_OUTSIDE_QUERY;
        if (min >= queryMin && max <= queryMax) return CELL_INSIDE_QUERY;
        return CELL_CROSSES_QUERY;
    }

    @Override
    public void visit(int docID) {
        builder.grow(1).add(docID);  // INSIDE 时:直接收集,无需值比较
    }

    @Override
    public void visit(int docID, byte[] packedValue) {
        long val = LongPoint.decodeDimension(packedValue, 0);
        if (val >= queryMin && val <= queryMax) {
            builder.grow(1).add(docID);  // CROSSES 叶子:逐点判断
        }
    }
});
