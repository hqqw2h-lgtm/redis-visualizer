第 17 章 · 压缩与存储字段编解码

Codec SPI · StoredFieldsFormat · 16 KB chunk · LZ4 · DEFLATE · BEST_SPEED · BEST_COMPRESSION · StoredFieldVisitor · .fdt / .fdx / .fdm · 整 chunk 解压代价

用户在订单列表里看到的 orderId / customerName / amountCents / items[] 这些"字面值",并不是从倒排索引里恢复出来的——倒排索引只保留 token,丢失了原始字符顺序、JSON 结构、数值精度。能把"完整原文"还给应用层的,是 Stored Fields:Lucene 把每个 doc 的 stored 字段串行化成字节流,按 16 KB chunk 聚合,用 LZ4 压缩后写入 .fdt,稀疏索引写入 .fdx,全局元数据写入 .fdm。一个 100 GB 的原始 stored 字段在这条管线后压缩到 28 GB,解压速度仍能达到 5 GB/s——这正是 "chunk + 字典压缩" 这个核心抽象的工程意义。

Stored Fields 还只是 Lucene 文件家族里的一员。.tim/.tip 是词典,.doc/.pos 是 postings,.dvd 是 DocValues,.dim 是 BKD,.fnm 是字段 schema,.si 是段元数据……每种文件格式版本演进的背后,是 Codec 抽象层在托底。Codec 通过 Java ServiceLoader SPI 注册,让 Codec.forName("Lucene99") 在运行期动态拼出一组 PostingsFormat / StoredFieldsFormat / DocValuesFormat ... 的实现——这种多态让 Lucene 能在保留向后兼容的前提下平滑演进文件格式。本章把"Stored Fields 的物理实现"和"Codec 抽象层"这两条主线一起拆开:从 chunk 的字节布局,到 LZ4/DEFLATE 算法,到 BEST_SPEED 与 BEST_COMPRESSION 选型决策,到 StoredFieldVisitor 选择性 fetch 的优化路径,到 Lucene90 → Lucene99 的演进细节,最后到 7 个生产陷阱。读完这一章,你能精确回答"为什么 Highlighter 慢"、"为什么 fetch 阶段比 query 阶段更重"、"为什么 mode 切换不会让旧段变小"。

  • Stored Fields 在 Lucene 文件家族中的角色:与 inverted index / DocValues / BKD 的边界,fetch 阶段必须读 stored,query 阶段不读
  • Codec 框架的 11 种 Format 接口:PostingsFormat / StoredFieldsFormat / DocValuesFormat / PointsFormat / KnnVectorsFormat ... 一个版本 Codec 是一组 Format 实现的笛卡尔积

17.1 Stored Fields 在搜索系统中的角色

电商订单搜索的典型流程是:用户输入"无线充电器",Lucene 在倒排索引上找到 50 个命中的 docID(按 BM25 排序),然后必须把每个 docID 翻译回原始的 {orderId: "ORDER-001", title: "无线充电器", price: 12500, items: [...]} 才能渲染列表。倒排索引能告诉你"哪些 doc 命中",但不能告诉你"它的 customerName 是什么"——后者必须从 Stored Fields 里取。这两类需求的物理实现完全不同,Lucene 把它们彻底隔离到不同的文件家族里。

Stored Fields 与其他索引数据的边界

图 17-1-1 · Lucene 文件家族:倒排 / DV / Stored / BKD 各司其职

这张图把 Lucene 一个段的 14 个文件按 语义角色 分了 5 组:倒排负责 query 阶段,DV 负责排序与聚合,Stored 负责 fetch 阶段,BKD 负责数值范围,KnnVectors 负责向量。Stored Fields 在查询阶段完全不被读取——它只在拿到 TopDocs 之后,需要给前端渲染时才被打开。这条边界划清,后面所有"为什么 fetch 比 query 慢"、"为什么不能用 stored 字段做过滤"的问题都迎刃而解。

展示字段(列表/详情页)orderId · title · customerName每次查询命中后20 - 200 字节

fetch 阶段的代价直觉

假设一次查询返回 50 个命中,每个 doc 平均 1 KB stored,那么 fetch 阶段需要读取 50 KB 字节并反序列化 50 个 doc——这听起来很轻量,但实际上 50 个 doc 可能分布在 50 个不同的 chunk 上(因为查询命中是按 BM25 score 排序的,与 docID 无关)。每个 chunk 解压一次约 50-200 μs,50 次 chunk 解压 = 2.5-10 ms,这经常是 P99 查询延迟的主要构成。

fetch 通常比 query 更慢 — 一次 BlockMax WAND 求 Top 50 在 1 千万 doc 上能做到 P99 = 5 ms;而紧接着的 fetch 阶段(读 50 个 chunk + 解压 + 反序列化)往往是 5-10 ms。这就是为什么生产搜索服务必须特别关注 stored fields 的 chunk 大小、fetch 策略、字段选择反序列化——这些都是本章后续小节的核心议题。

17.2 Codec 框架:SPI + 多态注册

Stored Fields 的物理实现并不是直接绑定在 IndexWriter 里的——它通过 StoredFieldsFormat 接口与 Lucene 的 Codec 抽象层 解耦。Codec 是 Lucene 文件格式的多态根类:每个 Codec 实例代表"一套完整的文件格式版本",由 11 种 Format 子接口组合而成。Lucene 9.10 的 Lucene99Codec 就是把 Lucene99PostingsFormat + Lucene90StoredFieldsFormat + Lucene90DocValuesFormat + ... 这一组实现"打包"成一个对象,通过 SPI 注册让运行期可以按名字查找。

Codec 抽象类源码骨架

// org.apache.lucene.codecs.Codec — Lucene 9.10
public abstract class Codec implements NamedSPILoader.NamedSPI {

    private static final NamedSPILoader<Codec> LOADER =
        new NamedSPILoader<>(Codec.class);

    public static Codec forName(String name) {
        if (LOADER == null) {
            throw new IllegalStateException(
                "You tried to lookup a Codec by name before all Codecs " +
                "could be initialized. ");
        }
        return LOADER.lookup(name);
    }

    public static Set<String> availableCodecs() {
        return LOADER.availableServices();
    }

    public static Codec getDefault() {
        return forName("Lucene99");
    }

    public abstract PostingsFormat postingsFormat();
    public abstract DocValuesFormat docValuesFormat();
    public abstract StoredFieldsFormat storedFieldsFormat();
    public abstract TermVectorsFormat termVectorsFormat();
    public abstract FieldInfosFormat fieldInfosFormat();
    public abstract SegmentInfoFormat segmentInfoFormat();
    public abstract NormsFormat normsFormat();
    public abstract LiveDocsFormat liveDocsFormat();
    public abstract CompoundFormat compoundFormat();
    public abstract PointsFormat pointsFormat();
    public abstract KnnVectorsFormat knnVectorsFormat();

    public abstract String getName();
}
Codec 是文件格式的"接口包"。Lucene 99 的代码里 stored fields 的物理实现叫 Lucene90StoredFieldsFormat——名字里的 90 不是过时,是"自 Lucene 9.0 起这个格式没改过"。一个 Codec 的版本号反映的是"Codec 这个组合"的版本,各 Format 单独的版本号独立演进。这种设计避免了"小改一处文件格式必须给整个 Codec 升大版本"的尴尬。

chunk 是 StoredFieldsFormat 的核心抽象——它独立于具体压缩算法、独立于具体字节布局。一旦你理解 "chunk = 一组 doc 的字节流被打包压缩",后面所有的细节都是这个核心抽象的边角实现:LZ4 vs DEFLATE 是"用什么压缩",16KB vs 60KB 是"多大的 chunk",.fdx 是"找到 chunk 的索引",StoredFieldVisitor 是"解压 chunk 后只反序列化哪些字段"。chunk 抽象不变,周边参数都是工程取舍。

17.4 chunk 抽象:Lucene 压缩的核心

"为什么 Lucene 不 per-doc 压缩(每个 doc 单独压一段)?为什么不全段压缩(整个段一次性压成一坨)?"——这两个极端都不可行,中间的工程平衡点就是 chunk。本节把 chunk 这个核心抽象的 why / how / 边界一次讲透。

为什么 chunk 必须是一组 doc 而不是单 doc

per-doc 压缩(naive)

  • 每个 doc 独立压成一个 LZ4 block
  • 压缩字典极短,绝大多数压不动
  • 典型 100-500 字节 doc 压缩比 ≈ 1.0×
  • 每 doc 一次 LZ4 header 开销(~10 B)
  • 读单 doc 极快(只解一个 block)
  • 结论:压缩失败,读快但文件不省

chunk 压缩(Lucene)

  • 累积 ≥ 16 KiB 字节后整 chunk 压
  • chunk 内 30-128 个 doc 字节被一起压
  • 压缩比 2-6×(text/JSON 内容相关)
  • 每 chunk 一次 LZ4 header,均摊 < 1 B/doc
  • 读单 doc 须解整 chunk(50-200 μs)
  • 结论:省空间,读 1 doc 代价可控

chunk 写入与切分时机

图 17-4-2 · StoredFields chunk 写入流水线

这条流水线的核心是 双触发条件:buffer 字节数到 16 KiB,或 doc 数到 128(BEST_SPEED 模式)。哪个先满足都立刻 flush 这一 chunk。前者保护"超大 doc 时不会无限膨胀 chunk",后者保护"超小 doc 时 chunk 不会塞太多 doc 导致单 doc 解压代价均摊不下来"。

fixed-block vs streaming 压缩

fixed-block(Lucene 选择)

  • chunk 边界与 doc 边界对齐
  • 每 chunk 独立解压,无前序依赖
  • 支持随机定位(配合 .fdx)
  • 压缩比稍低(字典每 chunk 重置)
  • 读单 doc:跳到 chunk + 解压一次
  • 适用:Lucene 这种"按 docID 随机访问"

streaming(gzip 文件那种)

  • 整个文件一次压缩字典从头到尾
  • 每点跳读必须从头解压到该点
  • 不支持随机定位
  • 压缩比高(字典最大化复用)
  • 读单 doc:O(目标位置) 解压
  • 适用:日志归档/全量备份
chunk 大小不能任意调大 — 把 chunkSize 从 16 KiB 调到 1 MiB 看似能提升压缩比,但读单 doc 必须解压 1 MiB——单次延迟从 50 μs 飙升到 3-5 ms。生产环境调大 chunkSize 之前,务必测量 fetch 阶段 P99 延迟,而不是只看磁盘占用。Lucene 默认 16 KB / 60 KB 是经过大量基准测试得到的工程经验值,不要轻易改动。

17.5 LZ4 与 DEFLATE 算法对比

chunk 的物理布局决定了"切多大",压缩算法决定了"怎么压"。Lucene 内置两种算法:LZ4(BEST_SPEED 默认)和 DEFLATE(BEST_COMPRESSION)。两者都属于 LZ77 字典压缩家族,但在压缩比与速度的权衡上做了截然不同的选择。本节把这两种算法的工作机制、压缩比、解压速度、生产取舍一次讲透。

LZ4 字典压缩思想

图 17-5-1 · LZ4 token 流编码示例

LZ4 的核心是 "找重复字符串,用 (距离, 长度) 替换"。压缩器用一个滑动哈希表(默认 64 KiB)记录最近字节的位置,扫描时尝试找到 ≥ 4 字节的匹配,匹配成功则输出一个 (offset, length) 对(typically 2-3 字节),否则把字面字节直接拷贝到输出。

  • Lucene 默认 chunk size:16 KB

DEFLATE(BEST_COMPRESSION)

  • 算法:LZ77 字典 + Huffman 编码
  • RFC 1951 标准
  • 滑动窗口 32 KiB
  • 压缩比:文本 4-6× / JSON 5-8×
  • 压缩速度:30 MB/s(level 6)
  • 解压速度:300 MB/s ★
  • 内存:压缩 ~256 KB / 解压 ~32 KB
  • Lucene chunk size:60 KB

// org.apache.lucene.codecs.lucene90.Lucene90StoredFieldsFormat
public class Lucene90StoredFieldsFormat extends StoredFieldsFormat {

    public enum Mode {
        BEST_SPEED,       // 16 KB chunk + LZ4
        BEST_COMPRESSION  // 60 KB chunk + DEFLATE level 6
    }

    private final Mode mode;

    public Lucene90StoredFieldsFormat() {
        this(Mode.BEST_SPEED);
    }

    public Lucene90StoredFieldsFormat(Mode mode) {
        this.mode = mode;
    }

    @Override
    public StoredFieldsReader fieldsReader(...) throws IOException {
        String previous = si.getAttribute(MODE_KEY);
        if (previous == null) {
            throw new IllegalStateException("Missing mode for segment: " + si.name);
        }
        Mode mode = Mode.valueOf(previous);
        return impl(mode).fieldsReader(directory, si, fn, context);
    }

    @Override
    public StoredFieldsWriter fieldsWriter(...) throws IOException {
        String previous = si.putAttribute(MODE_KEY, mode.name());
        if (previous != null && previous.equals(mode.name()) == false) {
            throw new IllegalStateException(
                "found existing value for " + MODE_KEY + " for segment: " +
                si.name + " old=" + previous + " new=" + mode.name());
        }
        return impl(mode).fieldsWriter(directory, si, context);
    }

    StoredFieldsFormat impl(Mode mode) {
        switch (mode) {
            case BEST_SPEED:
                return new Lucene90CompressingStoredFieldsFormat(
                    "Lucene90StoredFieldsFastData",
                    CompressionMode.FAST,
                    BEST_SPEED_BLOCK_LENGTH,    // 16 KB
                    BEST_SPEED_MAX_DOCS_PER_BLOCK,  // 128
                    10);
            case BEST_COMPRESSION:
                return new Lucene90CompressingStoredFieldsFormat(
                    "Lucene90StoredFieldsHighData",
                    CompressionMode.HIGH_COMPRESSION,
                    BEST_COMPRESSION_BLOCK_LENGTH,  // 60 KB
                    BEST_COMPRESSION_MAX_DOCS_PER_BLOCK,  // 512
                    10);
            default:
                throw new AssertionError();
        }
    }
}
chunk flush 是同步阻塞的 — 当 buffer 触发 chunkSize 时,LZ4 压缩 + 写 .fdt + 更新 .fdx 这三步都在调用 addDocument 的线程上同步执行。LZ4 压缩 16 KiB 大约 ~50 μs(@ 300 MB/s),所以写入吞吐的"尖峰"会在 chunk 边界出现。如果你测出 addDocument 的 P99 比 P50 高一个数量级,八成就是某些 doc 正好踩在了 chunk flush 上。这是正常现象,不需要"修",但要在容量规划里考虑——单 DWPT 写入吞吐受 LZ4 速度限制,生产用 16-32 个 DWPT 并行才能跑满磁盘带宽。

CompressingStoredFieldsReader 核心代码骨架

// org.apache.lucene.codecs.lucene90.compressing.Lucene90CompressingStoredFieldsReader
public final class Lucene90CompressingStoredFieldsReader extends StoredFieldsReader {

    private final FieldsIndex indexReader;       // .fdx
    private final IndexInput fieldsStream;       // .fdt
    private final Decompressor decompressor;
    private final BlockState state = new BlockState();   // 当前解压块缓存
    private final BytesRef bytes = new BytesRef();

    @Override
    public void document(int docID, StoredFieldVisitor visitor) throws IOException {
        if (state.contains(docID) == false) {
            // 缓存未命中:重新定位 + 解压
            fieldsStream.seek(indexReader.getStartPointer(docID));
            state.reset(fieldsStream);
            decompressor.decompress(fieldsStream,
                state.chunkSize,
                state.docOffset(docID),
                state.docLength(docID),
                bytes);
        }

        // 跳到 chunk 内 docID 偏移
        int docIdx = docID - state.docBase;
        int offset = state.endOffsetsInDoc[docIdx == 0 ? 0 : docIdx - 1];
        int length = state.endOffsetsInDoc[docIdx] - offset;

        ByteArrayDataInput in = new ByteArrayDataInput(
            bytes.bytes, bytes.offset + offset, length);

        int numFields = in.readVInt();
        for (int f = 0; f < numFields; f++) {
            int fieldNumber = in.readVInt();
            FieldInfo fieldInfo = fieldInfos.fieldInfo(fieldNumber);
            int bits = in.readByte() & 0xFF;
            switch (visitor.needsField(fieldInfo)) {
                case YES:
                    readField(in, visitor, fieldInfo, bits);
                    break;
                case NO:
                    skipField(in, bits);
                    break;
                case STOP:
                    return;
            }
        }
    }
}
读单 doc 必然解压整个 chunk — chunk 是 LZ4 压缩的最小单元,Lucene 没有"只解压 chunk 的某一段"的能力。即使你只想读 chunk 内第 30 个 doc,也必须把整个 chunk(典型 16 KiB,有 30-128 个 doc)全部解压到内存。Highlighter / 大列表回灌 / 详情页这类场景会被这个特性放大:fetch 50 hits 经常对应 50 次整 chunk 解压。这是不能消除的开销,但可以缓解(本章 §17.10 讲 hits 排序优化)。

chunk 缓存:Lucene 的小聪明

BlockState state 保留最近一次解压的 chunk 在内存中。如果下一次 document(docId) 请求的 docID 落在同一 chunk 内,直接复用解压缓冲——不再 I/O,不再解压,延迟从 60-600 μs 降到 5-50 μs(只剩反序列化)。这就是为什么 "hits 按 docID 升序处理"比"按 score 顺序处理"快得多——前者会大量命中 chunk 缓存。

17.10 StoredFieldVisitor:选择性 fetch

到这里我们已经理清了 stored fields 的写入与读取物理路径。现在要解决一个非常具体的生产问题:fetch 阶段的 CPU 浪费。reader.document(docId) 默认会把这个 doc 的所有 stored 字段都反序列化出来——但你的接口可能只需要其中 1-2 个字段。StoredFieldVisitor 是 Lucene 提供的"按需反序列化"API,典型场景能省 60-95% 的 fetch CPU。

默认 reader.document 的代价

// orderId       : "ORDER-001"          20 字节
// skuName       : "无线充电器 20W ..."  40 字节

// address       : "上海市浦东 ... "      80 字节

StoredFieldVisitor 选择性 fetch

// ✅ Visitor:只反序列化 orderId
public class FetchOnlyOrderId extends StoredFieldVisitor {

    public String orderId;

    @Override
    public Status needsField(FieldInfo fi) throws IOException {
        if (fi.name.equals("orderId")) {
            return Status.YES;       // 反序列化并回调
        }
        // 已拿到 orderId 即可提前结束,跳过后续所有字段(包括大 items_json)
        if (orderId != null) {
            return Status.STOP;
        }
        return Status.NO;            // 跳过,不反序列化
    }

    @Override
    public void stringField(FieldInfo fi, String value) throws IOException {
        if (fi.name.equals("orderId")) {
            this.orderId = value;
        }
    }

    // intField / longField / floatField / doubleField / binaryField 默认空实现
}

// 使用方式
FetchOnlyOrderId v = new FetchOnlyOrderId();
reader.storedFields().document(hit.doc, v);
String orderId = v.orderId;
// 单 doc fetch CPU:~ 5-15 μs(节省 70-90%)

Status 三种返回值的含义

Status含义何时用
YES反序列化并回调(stringField/intField/...)当前字段是需要的
NO跳过这个字段,只 skipBytes,不反序列化不需要的字段

STOP 是最被低估的优化 — 如果你的查询接口需要的字段都在 stored 字段顺序的前部(比如 orderId、tenantId、status 这种高频小字段),拿到后立即返回 STOP 可以跳过 chunk 内 doc 后部所有字段(包括 items_json 这种大字段)的字节流读取。Lucene 的 stored 字段在 chunk 里是按 schema 字段编号顺序串行的——前部字段拿到立即 STOP 等于直接 break 出 chunk 解压后的反序列化循环,典型再省 30-50% CPU。把高频回灌字段排在 schema 前部,是 stored fields 的隐藏调优技巧。

// ✅ 排序后顺序读,大幅命中 chunk cache
TopDocs top = searcher.search(q, 50);
ScoreDoc[] hits = top.scoreDocs.clone();
Arrays.sort(hits, (a, b) -> a.doc - b.doc);

Map<Integer, String> idMap = new HashMap<>();
for (ScoreDoc hit : hits) {
    FetchOnlyOrderId v = new FetchOnlyOrderId();
    reader.storedFields().document(hit.doc, v);
    idMap.put(hit.doc, v.orderId);
}
// 恢复 score 顺序

    .map(s -> idMap.get(s.doc))
    .collect(Collectors.toList());

真实场景:订单搜索 fetch 阶段降 75% — 在一个真实订单搜索服务里,query 路径 BlockMax WAND 把 P99 压到 5 ms,但接口总耗时 P99 仍然 18 ms——剩下的 13 ms 全在 fetch。原因是 hits 按 BM25 score 乱序,跨 chunk 跳读;每 hit 反序列化全部 12 个 stored 字段(包括 1.5 KB 的 items_json)。改造两步:① 改用 StoredFieldVisitor 只反序列化前端真正展示的 6 个字段,STOP 跳过 items_json;② fetch 之前把 hits 按 docID 排序,fetch 完再恢复 score 顺序。改造后 fetch 阶段 P99 从 13 ms 降到 3 ms,接口总 P99 从 18 ms 降到 8 ms。这两个优化对所有 fetch 重的搜索系统都通用。

17.11 Lucene90 → Lucene99 codec 演进

本章前面讲到"Lucene 99 的 stored fields 仍然是 Lucene90StoredFieldsFormat"——这不是巧合,而是 Lucene 文件格式版本演进的策略:每个 Format 单独打版本号,大版本号只跟一组 Format 的组合升。本节把 Lucene 9.0 → 9.10 这 10 个版本里 stored fields / codec 的实际变化整理出来,帮你理解"升级 Lucene 大版本到底变了什么"。

新 Lucene 永远能读旧段——这是 Lucene 设计的硬性承诺。机制是:每个段的 .si 文件持久化了写入时的 codec 名字(比如 "Lucene90"),段加载时调用 Codec.forName(siInfo.getCodec()) 按名字查表得到对应 codec 实现。Lucene 99 的 jar 里同时打包了 Lucene4xCodec / Lucene50Codec / ... / Lucene99Codec 全部历史 codec 实现,所以读旧段不需要"降版本"。

升级 Lucene 大版本的真实风险面 — "Lucene 9.0 → 9.10"听起来跨了 10 个小版本,实际上 stored fields / DocValues / Points / Norms 这 4 种 Format 的字节布局完全没变——已有索引升级 Lucene 后无需 rebuild。真正变了的是 PostingsFormat(impacts 编码)和 KnnVectorsFormat(向量量化)。所以升级测试的重点应该放在:① 倒排查询的 BlockMax 评分是否一致;② HNSW 向量召回是否一致(尤其量化参数变化)。stored / DV / points / norms 这 4 块基本是"无脑升"。

// 周期性扫所有段,把 stored fields 体积/压缩比导出为 metrics
public class StoredFieldsMetricsCollector {

    private final Meter compressionRatioGauge;
    private final Meter avgDocsPerChunkGauge;
    private final Meter totalStoredBytesGauge;

    public void collect(IndexReader reader) throws IOException {
        long totalChunks = 0;
        long totalDocs = 0;
        long totalUncompressed = 0;
        long totalCompressed = 0;

        for (LeafReaderContext leaf : reader.leaves()) {
            SegmentReader sr = (SegmentReader) FilterCodecReader.unwrap(leaf.reader());
            SegmentInfo info = sr.getSegmentInfo().info;
            Directory dir = info.dir;

            String fdmFile = info.name + "_Lucene90StoredFieldsFastMeta.fdm";
            try (IndexInput in = dir.openInput(fdmFile, IOContext.READ)) {
                CodecUtil.checkHeader(in,
                    "Lucene90StoredFieldsFastMeta",
                    VERSION_START, VERSION_CURRENT);
                long chunkSize = in.readVInt();
                long numChunks = in.readVLong();
                long numDocs = in.readVLong();
                long uncompressed = in.readVLong();
                long compressed = in.readVLong();

                totalChunks += numChunks;
                totalDocs += numDocs;
                totalUncompressed += uncompressed;
                totalCompressed += compressed;
            }
        }

        double ratio = (double) totalUncompressed / totalCompressed;
        double avgDocs = (double) totalDocs / totalChunks;

        compressionRatioGauge.mark((long) (ratio * 100));
        avgDocsPerChunkGauge.mark((long) (avgDocs * 100));
        totalStoredBytesGauge.mark(totalCompressed);

        log.info("stored.metrics ratio={} avgDocsPerChunk={} totalCompressed={}MB",
            ratio, avgDocs, totalCompressed / (1024 * 1024));
    }
}
真实场景:items_json 字段引发 chunk 膨胀 — 一个订单系统在某次产品迭代后,把订单明细 JSON(items_json)从 500 字节膨胀到平均 5 KiB(包含完整 SKU 元数据和图片 URL)。表现是:① stored 体积一周内翻倍;② chunk 平均 docs/chunk 从 60 降到 3-5(每个 doc 占满 chunk);③ fetch P99 从 8 ms 暴涨到 35 ms(几乎每个 doc 都触发 sliced)。修复路径:① 把 items_json 拆出来不放 stored,改放外部 KV(MongoDB / Redis);② 索引里只保留 itemSummary 这种小字段供搜索回灌。修复后压缩比恢复到 3.5×,fetch P99 回到 10 ms。启示:大 JSON 字段(> 1 KiB)放进 stored 一定要做容量规划,否则 chunk 抽象的优势会反过来变成代价。

陷阱 1:BEST_COMPRESSION 切换"看起来没生效"

// 检测哪些 doc 触发了 sliced
public void detectSlicedDocs(IndexReader reader) throws IOException {
    for (LeafReaderContext leaf : reader.leaves()) {
        SegmentReader sr = (SegmentReader) FilterCodecReader.unwrap(leaf.reader());

        // 用 StoredFieldVisitor 计每个 doc 反序列化字节数
        for (int docId = 0; docId < sr.maxDoc(); docId++) {
            ByteCountingVisitor v = new ByteCountingVisitor();
            sr.storedFields().document(docId, v);

            if (v.totalBytes > 16384) {  // 超过 BEST_SPEED chunkSize
                log.warn("doc={} stored={} bytes — sliced 风险",
                    docId, v.totalBytes);
            }
        }
    }
}
