第 30 章 · .fdm 存储字段元信息文件

.fdm Stored Fields Meta · chunkSize / docBase / numChunks / dirtyChunks · LZ4 vs DEFLATE · checksum · sparse compression mode

.fdm(Stored Fields Meta)是 stored fields 三件套的元信息文件。它和 .fdt(数据)、.fdx(索引)三位一体:.fdt 存压缩后的 stored fields chunk,.fdx 存"docID → chunk 偏移",而 .fdm 存的是对整个 stored fields codec 实例的描述:压缩算法(LZ4 / DEFLATE)、chunkSize、文档总数、chunk 总数、numDirtyChunks、numDirtyDocs、checksum 等。

.fdm 是这三件套里最小但最关键的文件:它通常只有 100-300 byte,却是打开 .fdt + .fdx 的钥匙。reader 启动时先读 .fdm 拿到 chunkSize / 压缩算法,才知道如何解 .fdx 的偏移、如何 LZ4-decompress .fdt 的 chunk。.fdm 也保存 numDirtyChunks / numDirtyDocs —— 这是 Lucene 9.x 用于估算是否需要 forceMerge 的关键统计:dirty chunk 是因 flush 而提前结束、未填满的 chunk,过多 dirty chunk 意味着压缩率低。

  • .fdm 文件大小特征:无论索引多大都 100-300 byte,因为只存"全局元信息"不存任何 doc
  • 字节布局:Header(Lucene90StoredFieldsMeta) + 7 个核心字段 + Footer
  • 7 个核心字段:chunkSize(默认 16 KB)、numDocs、numChunks、numDirtyChunks、numDirtyDocs、maxPointer、indexNumBytes
  • 压缩模式选择:Lucene 9.x 默认 LZ4 + Mode.BEST_SPEED,可切换 DEFLATE + Mode.BEST_COMPRESSION,两种模式的 chunkSize 不同(16 KB vs 60 KB)
  • dirty chunk 含义:flush 时 chunk 还没填满就被强制写出,会导致压缩率下降 5-10%
  • 为什么 .fdm 与 .fdt/.fdx 必须配套:仅有 .fdt+.fdx 不知道 chunkSize / 压缩算法,完全无法解码
  • checksum 校验:CodecUtil.checkFooter() 验证 .fdm 自身完整性
  • BEST_SPEED vs BEST_COMPRESSION 的权衡:索引体积差 30%,fetch 时延差 2-3x
  • forceMerge 决策依据:numDirtyDocs / numDocs > 10% 是常用的合并阈值
  • Merge 行为:多 segment 合并时 .fdm 完全重建,新 numChunks / numDirtyChunks 不沿用旧值

前置:第 17 章 storedFields LZ4 / DEFLATE 压缩;第 28 章 .fdt(数据本体);第 29 章 .fdx(索引);第 21 章 .si(segment-level metadata,与 .fdm 的 codec-level metadata 互补);第 03 章 segment merge(.fdm 在 merge 时重建)。

.fdm 是 stored fields 体系最小的文件,通常仅 30-50 字节。它保存 segment 级别的 stored fields 元信息:版本号、chunk 大小、单 chunk 最大 doc 数、chunk 总数、累计 dirty chunk 数与 dirty doc 数。这些参数让 reader 在打开 segment 时能立即校验编码参数是否一致,让 merge policy 能基于 dirty chunk 比率决策是否重压缩。

.fdm 在 Lucene 9.0 引入,把原本写在 .fdt / .fdx 头部的元信息分离到独立文件。分离的好处是 .fdt 与 .fdx 可以专注于"数据 + 索引",.fdm 单独承载 segment 级元数据,便于扩展新字段而不破坏 .fdt / .fdx 兼容性。

本章拆解 .fdm 的字节布局、各字段含义、merge policy 的 dirty chunk 判定逻辑,以及 .fdm 在 segment 升级与跨版本兼容中的角色。

模式 .fdt codecName .fdx codecName .fdm codecName

三个文件的 codec 名在打开 segment 时被 reader 校验,任何一个名称不匹配抛 IndexFormatTooNewException。同 segment 三个文件的 objectID 必须完全一致。

图 30-3-2:objectID 一致性校验
objectID 一致性流程:

  1. SegmentInfo.getId() 给出 segment 唯一 ID (16 byte)
  2. 写 .fdt 时 Header 写入此 ID
  3. 写 .fdx 时 Header 写入相同 ID
  4. 写 .fdm 时 Header 写入相同 ID
  5. 读取时 reader 比对三个 Header 的 objectID
  6. 任一不匹配抛 CorruptIndexException

防御场景:
  - 备份 / 恢复时混用不同 segment 的文件
  - 文件被错误覆盖
  - 跨 segment 误拷贝

图 30-4-2:dirty 比率阈值
dirty 判定与 merge 触发关系:

  dirtyRatio = numDirtyDocs / numDocs

  dirtyRatio < 1%       → 健康,无需特别 merge
  dirtyRatio 1% - 10%   → TieredMergePolicy 正常调度合并
  dirtyRatio > 10%      → 优先合并(deletes 类似)
  dirtyRatio > 25%      → forceMergeDeletes 候选

  实例(1M doc):
    dirtyDocs = 80     → 0.008%  健康
    dirtyDocs = 5000   → 0.5%   健康
    dirtyDocs = 50000  → 5%     普通调度
    dirtyDocs = 200000 → 20%    优先 merge

图 30-5-2:dirty chunk 的合并优化收益
未优化场景(7800 chunk + 20 dirty chunk):

  正常 chunk: 128 doc, 16 KiB → 4 KiB (压缩 4×)
  dirty chunk: 5 doc, 600 byte → 700 byte (无压缩,有 overhead)

  20 个 dirty chunk:
    占空间:14 KB
    含 100 个 doc
    平均每 doc 140 byte (vs 健康 chunk 32 byte)

合并后(merge 重新打 chunk):

  100 个 dirty doc + 邻近 doc → 重组为 1 个 chunk 128 doc
  压缩后 4 KiB (vs 原 14 KiB)
  节省 71%

  代价:一次性 I/O + CPU

forceMerge 触发场景:

  情况 1: 应用主动调用
    indexWriter.forceMerge(1)
    所有 segment 合并为 1 个,dirtyChunks 清零

  情况 2: forceMergeDeletes
    indexWriter.forceMergeDeletes()
    仅合并删除比率高的 segment

  情况 3: 周期性 maintenance
    cron: 每周一 03:00 forceMerge
    清理 dirty 与删除累积

  - .fdt 体积下降 30-50%(若 dirty 多)
  - .fdx 简化(chunk 数减少)

forceMerge 代价:
  - 一次性 CPU + I/O 极高
  - 期间 query latency 抖动
  - 大 segment merge 数小时

反例与正例对比

反例(从不监控 dirty,任由累积):

  无 dirty 监控 cron
  无定期 forceMergeDeletes
  RAM buffer 频繁 flush 导致部分 chunk 半满

  现象:
    1 年后 .fdt 体积是初始 3-5×
    fetch P99 上升
    磁盘频繁告警

正例(定期 maintenance):

  每周日凌晨 cron:
    1. 读 .fdm 计算 dirtyRatio
    2. dirtyRatio > 5% 的 segment 触发 forceMergeDeletes
    3. 业务峰值前完成

  收益:
    .fdt 体积稳定
    fetch P99 平稳
    无意外磁盘告警

30.12 三文件协作排障:.fdt + .fdx + .fdm 一起读

stored fields 不是单文件,而是 .fdt(数据)+ .fdx(索引)+ .fdm(元)三文件协作。任一文件损坏 / 缺失都会让 fetch 路径失败,但不同的失败症状指向不同的根因。本节把三个文件的失败模式映射到具体的故障类型,作为生产事故排查手册。

表 30-12-1:三文件分工速查
文件 大小 内容 没有它会怎样
_N.fdt 大 (整段 stored 内容压缩) 实际数据, chunk-level LZ4/DEFLATE 读不出任何 stored 字段, fetch 全空
_N.fdx 中 (~maxDoc * 4 bytes) docID -> chunk offset 映射 知道有数据但找不到 docID 对应的 chunk
_N.fdm 极小 (30-80 字节) chunkSize, maxDocsPerChunk, numChunks, dirty 统计 reader 启动时校验失败, 段无法 open

30.12.1 故障映射:症状 → 哪个文件出问题

表 30-12-2:三种典型故障的处置
故障 数据是否丢失 处置
.fdm 单独损坏 / 缺失 否, 数据在 .fdt 还在 用 CheckIndex 修复元数据;若失败则 forceMerge 把段并掉
.fdt 损坏 / 缺失 是, stored 字段数据完全丢失 该段 stored 不可恢复, 需从源数据 rebuild
.fdx 损坏 / 缺失 否, 但 .fdt 临时不可读 CheckIndex 重建 .fdx, 数据基于 .fdm 推算 chunk 位置
三文件 generation 不一致 否 旧 segments_N 引用旧 generation, 直接删孤儿文件

30.12.2 备份/复制时的三文件原子性

很多生产事故是因为运维 rsync / cp 时把 stored fields 三文件分批复制,中间被 rename / merge 干扰,落在目标侧的三文件 generation 不一致。任何复制必须把同一段的全部文件作为原子单元处理: 

# ❌ 错误: 分批复制导致 generation 不一致
cp src/_0.fdt dst/   # 此时源端可能 forceMerge, _0 已不存在
cp src/_0.fdx dst/   # 已经是新生成的
cp src/_0.fdm dst/   # 又是另一份

# ✅ 正确: 用 IndexWriter snapshot 拿一个一致视图, 再复制
SnapshotDeletionPolicy snap = new SnapshotDeletionPolicy(...);
IndexCommit commit = snap.snapshot();
// commit.getFileNames() 返回该 commit 所有引用文件
// 用这个清单做原子复制, 复制完释放 snapshot

30.12.3 升级跨大版本的 .fdm 兼容性

Lucene 版本 是否有独立 .fdm 升级注意
8.x 及以前 否 (元数据嵌在 .fdt 头) 9.x reader 期望 .fdm,直接读 8.x 索引会"no such file"
9.0 - 9.x 是 (Lucene90StoredFieldsFormat) 9.x reader 兼容 9.x 写入索引
10.x (将来) 可能格式变化 跨大版本一律走 IndexUpgrader 或 rebuild

升级路径:8.x → 9.x 必须 IndexUpgrader 或 rebuild,不能让 9.x reader 直接读 8.x 物理段。运维上把 IndexUpgrader 写进升级 runbook,跳过这一步是 90% "升级后 fetch 报错" 的根因。

30.13 本章要点

表 30-13-1:.fdm 关键事实
要点 关键依据 易错点
.fdm 是 stored fields 三件套之一 Lucene90StoredFieldsFormat 任一缺失 segment 不可读
仅含 segment 级元信息,文件 ~ 40 byte 字节布局 不存数据本身
5 个核心字段: chunkSize, maxDocs, numChunks, dirtyChunks, dirtyDocs finish() 写入 全部 VInt
引入版本 Lucene 9.0 Codec 演进 8.x 索引需升级
codecName 与 .fdt/.fdx 模式严格对应 Fast/High Meta 三件套必须同模式
objectID 必与 .fdt/.fdx 一致 Header 校验 文件混用导致 Corrupt
dirty 字段供 MergePolicy 决策 TieredMergePolicy 阈值典型 1-10%
forceMerge 后 dirty 归零 merge 重新切 chunk 代价高,需调度
加载时间 < 50 μs / segment 文件极小 不影响启动
写入在 finish() 阶段,与 .fdt/.fdx footer 同步 StoredFieldsWriter.finish 任一失败 commit 回滚
.fdt / .fdx / .fdm 必须作为原子单元复制 snapshot 拿一致视图 分批 cp 导致 generation 不一致
.fdm 损坏数据可恢复, .fdt 损坏数据丢失 CheckIndex 区分 不分清三文件的故障类型
8.x → 9.x 升级必须 IndexUpgrader / Rebuild 9.x reader 不兼容 8.x 物理段 直接读 8.x = "no such file"

本章小结

核心要点

  • .fdm 是 stored fields 三件套(.fdt / .fdx / .fdm)的元信息文件
  • 文件大小:无论索引多大都 100-300 byte,只存全局描述
  • 7 个核心字段:chunkSize / numDocs / numChunks / numDirtyChunks / numDirtyDocs / maxPointer / indexNumBytes
  • 压缩模式:LZ4 + BEST_SPEED(默认,chunkSize 16 KB) vs DEFLATE + BEST_COMPRESSION(chunkSize 60 KB)
  • dirty chunk:flush 时未填满的 chunk,过多会拉低压缩率
  • checksum:CodecUtil.checkFooter() 验证 .fdm 完整性
  • 三件套必须配套:仅 .fdt + .fdx 不知道 chunkSize / 压缩算法,无法解码
  • forceMerge 决策:numDirtyDocs / numDocs > 10% 是常用合并阈值
一句话总结:第 30 章核心是把"该文件在 codec 中的角色 / 字节布局 / 编码细节 / 写入读取路径 / 合并行为 / 性能特征 / 生产陷阱"七件事讲清楚。理解了本章,下次遇到该文件相关的 corruption / 性能 / 合并 issue 都能从字节级回答。
生产建议:① 监控该文件大小占比,占比异常需要分析;② CheckIndex 定期校验 checksum;③ 索引体积大时优先考虑 forceMerge 减少 segment 数量;④ 升级 Lucene 版本时关注 codec 名称变化(如 Lucene90 → Lucene99 → Lucene103);⑤ 不要直接动文件本体,所有写入都通过 IndexWriter 完成。
