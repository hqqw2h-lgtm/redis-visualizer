第 7 章 · 订单搜索实战

Schema · QueryBuilder · FILTER 权限 · searchAfter · facet · alias 切换 · 监控 · 回归 · P99 = 8ms

订单搜索是几乎所有电商、SaaS、ERP 系统都绕不开的功能。客服要按手机号查"小王前三天那笔被退款的订单",仓库要按"已支付未发货 + 上海仓 + 金额 > 1000"过滤待捡货池,商家后台要"近 30 天 PAID 订单按金额降序"看销售排行——同一份订单数据被四五个角色用四五种条件查询,而每个角色都要求毫秒级响应、租户隔离、数据可见延迟 < 5 秒。这正是 Lucene 的主场。

为什么不直接用 SQL?数据库的 LIKE '%charger%' 必须全表扫描,1 千万订单单次查询要 8-12 秒;即使加了 B-tree 索引,多字段组合 status='PAID' AND amount > 100 AND paid_at BETWEEN ? 也只能用上其中一个索引,选择性差的字段被退化成扫表。Lucene 用倒排索引把"找有这个 term 的 doc"从 O(N) 降到 O(freq),再用 BKD-tree 做范围、用 DocValues 做排序、用 FILTER 子句做权限——每条访问路径各自最优,组合起来 P99 = 8ms 在 1 千万订单上是工程上能稳定达到的目标。本章把第 1-6 章所有零件落地为一个真实可上线的订单搜索服务:从字段建模、索引管线、Query 翻译、权限注入,到 alias 零停服切换、监控告警、回归测试,每一段都给出能直接 copy 的 Java 代码骨架。

  • 订单字段建模:12 个核心字段如何决定 indexed / tokenized / stored / docValues / norms 的 32 种组合
  • 索引构建管线:OLTP DB → CDC → Kafka → Outbox Consumer → IndexWriter,版本去重保证乱序事件的幂等性
  • Query Builder 翻译:把业务 DSL {status: PAID, amountMin: 100} 编译成 BooleanQuery 树,FILTER 子句不评分
  • 权限多租户的 FILTER 前置:tenantId + storeId 必须在 Query 阶段过滤,事后业务过滤会污染 TopK
  • searchAfter 深翻页:from + size 在 page=100 OOM,searchAfter 始终 O(K) 内存且跨页稳定
  • facet 聚合:status / 月份 / 金额段三个维度的 SortedSetDocValuesFacetCounts 实现
  • alias 切换:rebuild → CheckIndex → 配置中心原子写,30 分钟停服降到 0 停服,7 天 standby 用于回滚
  • 监控 + 回归:10 个关键指标 + 端到端 JUnit 模板,加上 1 千万订单 P99 = 8ms 怎么拆出来的

前置知识:第 1 章字段类型与四条访问路径(TextField / StringField / LongPoint / DocValues / StoredField);第 2 章 Analyzer 选型(Standard / Smart Chinese / Keyword);第 4 章 Query / Weight / Scorer 三层与 BM25 三因子;第 6 章 IndexWriter / SearcherManager / commit / refresh 时序。本章是前面所有概念的"工程化收口",代码全部基于 Lucene 9.10 + Spring Boot 3.x,可直接落地。

OrderToDocConverter 的实现

public final class OrderToDocConverter {

    private static final FieldType KEY_TYPE  = freeze(true,  IndexOptions.DOCS, false, true);
    private static final FieldType STR_TYPE  = freeze(true,  IndexOptions.DOCS, false, true);
    private static final FieldType TEXT_TYPE = freeze(true,  IndexOptions.DOCS_AND_FREQS_AND_POSITIONS, true, false);

    private static FieldType freeze(boolean stored, IndexOptions opts, boolean tokenized, boolean omitNorms) {
        FieldType ft = new FieldType();
        ft.setStored(stored);
        ft.setIndexOptions(opts);
        ft.setTokenized(tokenized);
        ft.setOmitNorms(omitNorms);
        ft.freeze();
        return ft;
    }

    public static Document convert(OrderEvent ev) {
        Document doc = new Document();

        doc.add(new Field("orderId",    ev.getOrderId(),    KEY_TYPE));
        doc.add(new Field("tenantId",   ev.getTenantId(),   STR_TYPE));
        doc.add(new Field("storeId",    ev.getStoreId(),    STR_TYPE));
        doc.add(new Field("status",     ev.getStatus(),     STR_TYPE));
        doc.add(new SortedDocValuesField("status_dv",  new BytesRef(ev.getStatus())));
        doc.add(new SortedDocValuesField("storeId_dv", new BytesRef(ev.getStoreId())));

        doc.add(new Field("customerName", ev.getCustomerName(), TEXT_TYPE));
        doc.add(new Field("items",        String.join(" ", ev.getItems()), TEXT_TYPE));
        doc.add(new Field("title",        ev.getTitle(),        TEXT_TYPE));

        doc.add(new LongPoint("amountCents", ev.getAmountCents()));
        doc.add(new NumericDocValuesField("amountCents_dv", ev.getAmountCents()));
        doc.add(new StoredField("amountCents", ev.getAmountCents()));

        doc.add(new LongPoint("paidAt", ev.getPaidAtEpochMs()));
        doc.add(new NumericDocValuesField("paidAt_dv", ev.getPaidAtEpochMs()));
        doc.add(new StoredField("paidAt", ev.getPaidAtEpochMs()));

        doc.add(new NumericDocValuesField("version", ev.getVersion()));
        doc.add(new StoredField("version", ev.getVersion()));

        return doc;
    }
}

用 SQL LIKE 还是用 Lucene

SQL LIKE 全表扫描

SELECT * FROM orders
WHERE title LIKE '%charger%'
  AND status = 'PAID'
  AND amount >= 10000
  AND paid_at BETWEEN ? AND ?
ORDER BY amount DESC
LIMIT 20;
  • 1000 万订单单次扫表 8-12 秒
  • 多字段组合无法用同一个索引
  • LIKE '%x%' 永远走全表
  • QPS 上 50 数据库就崩

Lucene 倒排 + BKD + DocValues

BooleanQuery q = new BooleanQuery.Builder()
  .add(new TermQuery(new Term("title","charger")), MUST)
  .add(new TermQuery(new Term("status","PAID")),   FILTER)
  .add(LongPoint.newRangeQuery("amountCents",
        10000L, Long.MAX_VALUE), FILTER)
  .add(LongPoint.newRangeQuery("paidAt",
        from, to), FILTER)
  .build();
Sort s = new Sort(new SortField(
  "amountCents_dv", Type.LONG, true));
TopDocs td = searcher.search(q, 20, s);
  • title 倒排 O(freq) ≈ 几千次迭代
  • status FILTER 缩到 60% 候选
  • amount + paidAt 走 BKD-tree
  • P99 ≈ 8ms,QPS > 500
订单搜索为什么不能直接用 SQL —— 多字段 + 模糊 + 排序的复合条件天然不友好。SQL 引擎面对 title LIKE '%charger%' AND status='PAID' AND amount>100 这种 query,只能选其中"选择性最好"的那一个走索引,其他条件退化成 row 扫描。Lucene 的 BooleanQuery 把每个子句各自走自己的最优访问路径(倒排 / BKD / DocValues),最后 leapfrog 求交——这种"每条路径独立优化 + 求交在 docID 升序上做"的模型,正是 1 千万订单上 P99 = 8ms 在工程上能稳定达到的根本原因。

7.3 索引构建管线

Outbox Consumer 的批处理逻辑

@Component
@KafkaListener(topics = "order.events", groupId = "order-indexer")
public class OrderIndexerConsumer {

    private final IndexerService service;
    private final List<OrderEvent> buffer = new ArrayList<>();
    private static final int  BATCH_SIZE   = 200;
    private static final long FLUSH_TICK_MS = 1000L;

    @KafkaHandler
    public void onEvent(OrderEvent ev,
                        @Header(KafkaHeaders.OFFSET) long offset,
                        Acknowledgment ack) {
        synchronized (buffer) {
            buffer.add(ev);
            if (buffer.size() >= BATCH_SIZE) {
                flushLocked();
            }
        }
        ack.acknowledge();
    }

    @Scheduled(fixedDelay = FLUSH_TICK_MS)
    public void flushOnTick() {
        synchronized (buffer) {
            if (!buffer.isEmpty()) flushLocked();
        }
    }

    private void flushLocked() {
        List<OrderEvent> batch = new ArrayList<>(buffer);
        buffer.clear();
        try {
            service.index(batch);
        } catch (Exception e) {
            log.error("indexer flush failed batch={}", batch.size(), e);
            buffer.addAll(0, batch);
        }
    }
}

批处理的两个阈值——"200 条"或"1 秒"——都是经验值。批太小 IndexWriter 频繁加锁吞吐低,批太大单次 flush 时间拉长可见延迟。生产中先按 200/1s 上线,看监控 writer.flush.latency 调整。

7.4 IndexWriter 单例服务

整个进程内 同一个目录只能有一个 IndexWriter 实例——这是 Lucene 写入侧的硬约束,违反会立即抛 LockObtainFailedException。所以 Spring 容器里 IndexWriter 必须以 singleton bean 形式存在,生命周期跟容器一起,所有写入逻辑通过这个单例完成。

Bean 配置

@Configuration
public class IndexerConfig {

    @Bean
    public Directory directory(@Value("${index.path}") String path) throws IOException {
        return FSDirectory.open(Paths.get(path));
    }

    @Bean
    public Analyzer analyzer() {
        return new PerFieldAnalyzerWrapper(
            new StandardAnalyzer(),
            Map.of(
                "items",        new SmartChineseAnalyzer(),
                "customerName", new StandardAnalyzer(),
                "title",        new SmartChineseAnalyzer(),
                "orderId",      new KeywordAnalyzer(),
                "status",       new KeywordAnalyzer(),
                "tenantId",     new KeywordAnalyzer(),
                "storeId",      new KeywordAnalyzer()
            ));
    }

    @Bean(destroyMethod = "close")
    public IndexWriter indexWriter(Directory dir, Analyzer analyzer) throws IOException {
        IndexWriterConfig config = new IndexWriterConfig(analyzer);
        config.setOpenMode(OpenMode.CREATE_OR_APPEND);
        config.setRAMBufferSizeMB(256);
        config.setUseCompoundFile(false);
        config.setMergeScheduler(new ConcurrentMergeScheduler());

        TieredMergePolicy mp = new TieredMergePolicy();
        mp.setMaxMergedSegmentMB(2048);
        mp.setSegmentsPerTier(10);
        config.setMergePolicy(mp);
        config.setInfoStream(new PrintStreamInfoStream(System.out));

        return new IndexWriter(dir, config);
    }

    @Bean(destroyMethod = "close")
    public SearcherManager searcherManager(IndexWriter writer) throws IOException {
        return new SearcherManager(writer, true, true, new SearcherFactory());
    }
}

版本去重的 IndexerService

@Service
public class IndexerService {

    private final IndexWriter writer;
    private final SearcherManager mgr;

    public void index(List<OrderEvent> batch) throws IOException {
        Map<String, OrderEvent> latest = batch.stream()
            .collect(Collectors.toMap(
                OrderEvent::getOrderId,
                e -> e,
                (a, b) -> a.getVersion() > b.getVersion() ? a : b
            ));

        IndexSearcher searcher = mgr.acquire();
        try {
            for (OrderEvent ev : latest.values()) {
                Long current = readVersion(searcher, ev.getOrderId());
                if (current != null && current >= ev.getVersion()) continue;
                Document doc = OrderToDocConverter.convert(ev);
                writer.updateDocument(new Term("orderId", ev.getOrderId()), doc);
            }
        } finally {
            mgr.release(searcher);
        }
    }

    private Long readVersion(IndexSearcher s, String orderId) throws IOException {
        TopDocs td = s.search(new TermQuery(new Term("orderId", orderId)), 1);
        if (td.totalHits.value == 0) return null;
        Document d = s.storedFields().document(td.scoreDocs[0].doc);
        return d.getField("version").numericValue().longValue();
    }
}

批内先用 orderId 合并取最大 version——同一订单短时间内多次变更,只保留最新那条,减少无效写入。再对每条事件查当前索引中的 version,如果 ev.version ≤ current 直接丢弃(乱序事件),否则 updateDocument。这套去重逻辑保证了 at-least-once 投递语义下索引仍是幂等的。

7.5 Query Builder

业务接口收到的是结构化 DTO:{keyword: "iPhone", status: "PAID", amountMin: 100, paidFrom: "2026-04-01"}。Query Builder 的职责是把它翻译成 Lucene 的 BooleanQuery 树。这个翻译过程不是逐字段简单拼接,而是按"评分 vs 过滤"分流——keyword 走 MUST(参与 BM25),status / amount / paidAt 走 FILTER(不评分,可缓存)。

OrderQueryBuilder 实现

@Component
public class OrderQueryBuilder {

    public Query build(SearchRequest req, AuthContext auth) {
        BooleanQuery.Builder b = new BooleanQuery.Builder();

        if (StringUtils.hasText(req.getKeyword())) {
            b.add(parseKeyword(req.getKeyword()), Occur.MUST);
        }

        b.add(new TermQuery(new Term("tenantId", auth.getTenantId())), Occur.FILTER);

        if (req.getStatus() != null) {
            b.add(new TermQuery(new Term("status", req.getStatus())), Occur.FILTER);
        }
        if (req.getStoreId() != null) {
            b.add(new TermQuery(new Term("storeId", req.getStoreId())), Occur.FILTER);
        }
        if (req.getAmountMin() != null || req.getAmountMax() != null) {
            long min = req.getAmountMin() != null ? req.getAmountMin() : 0L;
            long max = req.getAmountMax() != null ? req.getAmountMax() : Long.MAX_VALUE;
            b.add(LongPoint.newRangeQuery("amountCents", min, max), Occur.FILTER);
        }
        if (req.getPaidFromMs() != null || req.getPaidToMs() != null) {
            long min = req.getPaidFromMs() != null ? req.getPaidFromMs() : 0L;
            long max = req.getPaidToMs()   != null ? req.getPaidToMs()   : Long.MAX_VALUE;
            b.add(LongPoint.newRangeQuery("paidAt", min, max), Occur.FILTER);
        }

        if (auth.getRole() != Role.ADMIN) {
            BooleanQuery.Builder allow = new BooleanQuery.Builder();
            for (String s : auth.getAllowedStoreIds()) {
                allow.add(new TermQuery(new Term("storeId", s)), Occur.SHOULD);
            }
            b.add(allow.build(), Occur.FILTER);
        }
        return b.build();
    }

    private Query parseKeyword(String kw) {
        String[] tokens = kw.toLowerCase(Locale.ROOT).split("\\s+");
        BooleanQuery.Builder root = new BooleanQuery.Builder();
        for (String t : tokens) {
            BooleanQuery.Builder sub = new BooleanQuery.Builder();
            sub.add(new TermQuery(new Term("customerName", t)), Occur.SHOULD);
            sub.add(new TermQuery(new Term("items",        t)), Occur.SHOULD);
            sub.add(new TermQuery(new Term("title",        t)), Occur.SHOULD);
            root.add(sub.build(), Occur.MUST);
        }
        return root.build();
    }
}

7.6 权限过滤的 FILTER 模式

多租户系统里"权限过滤"是合规底线——A 租户绝不能看到 B 租户的订单,即使 1ms 都不行。Lucene 把这件事做对了:BooleanClause.Occur.FILTER 在 leapfrog 求交阶段就把无权数据排除,TopK 收集到的就已经全是有权数据。最常见的反模式是先 search(query, 100) 再用业务代码过滤,这会让 TopK 被无权数据"挤占",分页失稳、计数错误、合规审计直接红线。

  • TopK = 100 但有权数据可能只 7 条
  • page=2 size=20 返回空(已被无权数据占完)
  • totalHits 包含无权数据,前端展示错误
  • 合规审计直接红线(SOC 2 / ISO 27001)

FILTER 子句前置

BooleanQuery q = new BooleanQuery.Builder()
  .add(keyword, Occur.MUST)
  .add(new TermQuery(new Term(
       "tenantId", auth.getTenantId())),
       Occur.FILTER)
  .add(allowedStoresOR(auth), Occur.FILTER)
  .build();
TopDocs td = searcher.search(q, 20);
// TopK 已天然只含有权数据
  • 权限约束参与 leapfrog,提前剪枝
  • TopK 100% 有权,翻页稳定
  • totalHits 准确反映用户可见数据量
  • FILTER 可被 LRUQueryCache 缓存
权限 FILTER 不能事后过滤 —— 否则 TopK 被无权数据污染。这是订单搜索最容易翻车的地方,也是合规审计的红线。Lucene 的 leapfrog 求交在 docID 升序上推进,FILTER 子句和 MUST 子句是平等参与的:任何一个子句不命中就直接 advance 到下一个 doc。把 tenantId / allowedStoreIds 写进 FILTER,就能让 Lucene 在 collect 阶段之前已经把无权 doc 全部排除。如果你为了"省一行代码"把权限放在搜索后用 Java 过滤,P99 不仅会因为无意义的 doc 加载升高,还会随用户角色不同导致分页诡异——客服每页明明 size=20,实际看到 7 条;翻第 5 页突然 0 条。这种 bug 在测试环境很难复现,只有上线后多租户混合数据才暴露。

7.7 时间范围与金额范围

时间和金额是订单搜索里出现频率最高的范围条件——"近 30 天 PAID"、"金额 > 1000"、"4 月份 SHIPPED"。Lucene 的范围查询不走倒排,而是 LongPoint.newRangeQuery 走 BKD-tree:在 1 千万 doc 上一次范围查询的代价大概是 O(log N + 候选),典型 1-3ms。

LongPoint 与 NumericDocValues 双写

// 范围过滤(BKD)
Query paidRange = LongPoint.newRangeQuery(
    "paidAt", epochMs("2026-04-01"), epochMs("2026-05-01"));

// 排序(DocValues 列读)
Sort sort = new Sort(new SortField(
    "paidAt_dv", SortField.Type.LONG, true));

// 同一个业务字段写两份:
//   doc.add(new LongPoint("paidAt", ms));
//   doc.add(new NumericDocValuesField("paidAt_dv", ms));
//   doc.add(new StoredField("paidAt", ms));   // 展示用
为什么 paidAt 既要 LongPoint 又要 NumericDocValues —— 范围查 + 排序两条路径各走各的。BKD-tree(.dim/.dii/.kdd)是为范围过滤设计的多维点索引,擅长快速找出"在某区间的 doc 集合",但它不能 O(1) 给你某个 docID 的具体值——你想"按 paidAt 排序"它做不到。NumericDocValues(.dvd/.dvm)是按 docID 顺序的列式存储,advanceExact(docID) O(log P) 拿到值,正好用来排序和聚合。两个文件加起来约 2× 单字段开销,换来"过滤快 + 排序快"两个数量级的提升,值得。如果只写 LongPoint,排序时 Lucene 直接报 unexpected docvalues type NONE;如果只写 DocValues,范围过滤就得全表 advanceExact 读列,1 千万 doc 6-8 秒——这是订单搜索字段建模最容易踩的两个反例。

时区处理与边界

问题错误做法正确做法
展示给用户存本地时间字符串 "2026-04-15 10:30"存 epoch ms,展示时按 user timezone 格式化
跨时区查询用 LocalDateTime 比较统一 UTC epoch ms,Query 也用 UTC 边界
"今天" 边界Date.now() / 86400000 * 86400000按 user timezone 算 ZonedDateTime 起止 ms

金额范围的精度

long minCents = req.getAmountMin() != null
    ? req.getAmountMin() : Long.MIN_VALUE;
long maxCents = req.getAmountMax() != null
    ? req.getAmountMax() : Long.MAX_VALUE;

Query amountRange = LongPoint.newRangeQuery(
    "amountCents", minCents, maxCents);

金额永远以分(cents)为单位存 long,绝不用 double / float。double 存 ¥0.1 + ¥0.2 ≠ ¥0.3 这种浮点误差在订单系统里是 P0 事故。BKD-tree 对 long 的范围查询是无损精确的,12800L 就是 12800L。

7.8 排序与 SortField

客服列表"按下单时间倒序"、商家排行榜"按金额降序"、运营看板"按状态再按时间"——多排序键是订单搜索的标配。Lucene 的 Sort 接受任意多个 SortField 串联,前面的优先,后面的作为 tie-breaker。

多排序键的构造

private Sort buildSort(SearchRequest req) {
    return switch (req.getSort() == null ? "default" : req.getSort()) {
        case "paidAt_desc" -> new Sort(
            new SortField("paidAt_dv",  SortField.Type.LONG,   true),
            new SortField("orderId",    SortField.Type.STRING, false));
        case "amount_desc" -> new Sort(
            new SortField("amountCents_dv", SortField.Type.LONG,   true),
            new SortField("orderId",        SortField.Type.STRING, false));
        case "amount_asc" -> new Sort(
            new SortField("amountCents_dv", SortField.Type.LONG,   false),
            new SortField("orderId",        SortField.Type.STRING, false));
        default -> null;  // null = 按 BM25 score 降序
    };
}

SortField.Type 与 DocValues 类型对应

SortField.Type需要的 DocValues典型字段
LONG / DOUBLENumericDocValuesamountCents_dv / paidAt_dv
STRINGSortedDocValuesorderId / status_dv
SCORE—(BM25 实时计算)相关性默认
CUSTOM自定义 ValueSource距离 / 业务 boost

tie-breaker 的必要性

"按 paidAt 降序"——如果两个 doc 的 paidAt 完全相同(同一秒同一笔系统批量支付),Lucene 不保证返回顺序稳定,segment 内部 docID 顺序变化(merge 后)会让两条相邻订单在翻页时错位甚至重复。所有排序都必须以 orderId 这种唯一字段作为最终 tie-breaker。

无 tie-breaker 时翻页错乱:
  page=1 size=2  → [O100, O101]   // paidAt 都是 1714805700000
  merge 触发,segment 重排
  page=2 size=2  → [O100, O102]   // O100 又出现了,O101 消失!

加 orderId tie-breaker 后:
  page=1 size=2  → [O100, O101]   // O100 < O101 字典序确定
  无论 segment 怎么变
  page=2 size=2  → [O102, O103]   // 稳定不变

7.9 searchAfter 翻页

"用户翻到第 100 页怎么办?"from + size 模式下 Lucene 必须扫描前 from + size 条 doc——page=100, size=20 就是收集 2020 条 hit 后再切片;page=10000 时直接 OOM。searchAfter 用游标(上一页最后一条的 SortValues)定位,无论第几页都只需要内存 O(K)。

searchAfter 实现

public SearchResponse search(SearchRequest req, AuthContext auth) throws IOException {
    Query  query = builder.build(req, auth);
    Sort   sort  = buildSort(req);
    int    size  = Math.min(req.getSize(), 100);

    FieldDoc after = decodeCursor(req.getCursor(), sort);

    IndexSearcher searcher = mgr.acquire();
    try {
        TopFieldDocs td = (after != null)
            ? searcher.searchAfter(after, query, size, sort)
            : (TopFieldDocs) searcher.search(query, size, sort);

        List<OrderHit> hits = new ArrayList<>(td.scoreDocs.length);
        for (ScoreDoc sd : td.scoreDocs) {
            Document d = searcher.storedFields().document(sd.doc);
            hits.add(toHit(d, sd));
        }

        String next = (td.scoreDocs.length == size)
            ? encodeCursor((FieldDoc) td.scoreDocs[td.scoreDocs.length - 1])
            : null;

        return new SearchResponse(td.totalHits.value, hits, next);
    } finally {
        mgr.release(searcher);
    }
}

游标编码:Base64 + JSON

private String encodeCursor(FieldDoc fd) {
    Map<String, Object> data = new LinkedHashMap<>();
    data.put("doc",   fd.doc);
    data.put("shard", fd.shardIndex);
    data.put("vals",  Arrays.stream(fd.fields)
                            .map(this::serializeSortValue)
                            .toList());
    byte[] json = objectMapper.writeValueAsBytes(data);
    return Base64.getUrlEncoder().withoutPadding().encodeToString(json);
}

7.10 高亮与摘要

用户搜"iPhone charger",结果列表里把命中关键词标红——这是高亮。Lucene 9 推荐用 UnifiedHighlighter,它结合了之前 PostingsHighlighter 和 FastVectorHighlighter 的优点,自动在 .pos / term vectors / re-analyze 三种数据源里挑代价最低的。

UnifiedHighlighter 的字段配置

@Component
public class OrderHighlighter {

    private final IndexSearcher searcher;
    private final Analyzer analyzer;

    public Map<Integer, Map<String, String>> highlight(
            Query query, TopDocs hits, String[] fields, int maxSnippets) throws IOException {

        UnifiedHighlighter uh = UnifiedHighlighter.builder(searcher, analyzer)
            .withFormatter(new DefaultPassageFormatter(
                "<em class=\"hit\">", "</em>", "...", false))
            .withMaxLength(2000)
            .build();

        Map<Integer, Map<String, String>> out = new HashMap<>();
        for (String field : fields) {
            String[] frags = uh.highlight(field, query, hits, maxSnippets);
            for (int i = 0; i < hits.scoreDocs.length; i++) {
                int docId = hits.scoreDocs[i].doc;
                out.computeIfAbsent(docId, k -> new HashMap<>())
                   .put(field, frags[i]);
            }
        }
        return out;
    }
}

性能 vs 召回的取舍

高亮源需要性能精度
POSTINGSIndexOptions.DOCS_AND_FREQS_AND_POSITIONS_AND_OFFSETS快(直读)高
TERM_VECTORSFieldType.setStoreTermVectors(true)中高
ANALYSISStoredField + 重新分词慢(re-analyze)中等(取决于 Analyzer)

订单搜索的 customerName / items / title 通常字段较短(平均 30-100 字符),用 ANALYSIS 模式即可,不必额外存 term vectors。生产中只有"商品长描述 / 评价文本 5000+ 字符"才考虑开启 storeTermVectors,代价是索引膨胀 ~25%。

7.11 facet 聚合

电商搜索结果右侧那一栏"PAID(2304) / SHIPPED(1872) / CANCELED(521)"就是 facet。Lucene 用 SortedSetDocValuesFacetCounts 在 DocValues 上做聚合——不需要单独的 facet 索引,只要写入时给字段加 SortedSetDocValuesFacetField 就行。

public static Document convert(OrderEvent ev) {

7.12 索引重建与 alias 切换

"换 Analyzer 必须 reindex"、"加新字段必须 rebuild"——schema 演进是搜索系统的常态。最 naive 的方案是停服重建:把 IndexWriter 关掉,删旧目录,起 IndexWriter 全量重写,30 分钟 + 期间所有查询挂掉。生产里这么做相当于自杀。alias 切换是工业界的标准答案:新目录 orders.v2 在线下静默重建,校验通过后通过配置中心原子切换 alias 指针,旧目录 orders.v1 保留 7 天用于回滚。

AliasListener 实现

@Component
public class AliasListener {

    private final SearcherManagerHolder holder;
    private final IndexerService indexer;

    @ConfigurationListener("/search/order/alias")
    public void onAliasChange(String newAlias) throws IOException {
        log.info("alias change → {}", newAlias);
        Path newPath = Paths.get("/data/lucene/" + newAlias);
        Directory newDir = FSDirectory.open(newPath);

        SearcherManager newMgr = new SearcherManager(newDir, new SearcherFactory());
        SearcherManager old = holder.swap(newMgr);

        if (old != null) {
            try { old.close(); }
            catch (IOException e) { log.warn("close old SearcherManager failed", e); }
        }
        indexer.bindWriterTo(newPath);
    }
}

alias 切换 vs in-place rebuild

原地 rebuild(禁止)

  • 停 IndexWriter,删旧目录,重建
  • 停服 30-90 分钟,期间查询全部 5xx
  • 重建过程中 binlog 持续增长,catch-up 时间不确定
  • 失败回滚困难——旧数据已删
  • 用户体验灾难,SLA 直接破

alias 切换

  • 新目录 orders.v2 在线下重建,旧目录继续服务查询
  • 0 停服,切换瞬间 ≤ 5 秒(只是 SearcherManager 重新打开)
  • 失败回滚:配置中心写回 orders.v1,瞬时生效
  • v1 保留 7 天 standby,事故有窗口期处理
  • SLA 不破,用户无感知
alias 切换让索引重建从 30 分钟停服降到 0 停服——这是 Elasticsearch 的标配能力,也是任何"严肃"搜索系统必备的工程实践。我们这套订单搜索上线 2 年,经历过 5 次 schema 演进:加 customerPhone 字段、title 改 IK 分词、amountRange 桶定义调整、加 shipFrom facet、加多语言。每一次都是后台 v(N+1) 静默重建 60-90 分钟,前台用户没感觉一点延迟,在 grafana 上只看到 alias 切换那一瞬间 segment count 从 18 跳到 24 再合并回去。配合 7 天 standby 保留,有一次 v3 切上去 30 分钟后发现 amount facet 桶定义错了,直接配置中心写回 v2,2 秒回滚生效——这种"双热备 + 原子切换"才是生产级搜索系统的姿势。

@Aspect
@Component
public class SearchMetricsAspect {

    private final Timer queryTimer       = Metrics.timer("search.query.latency");
    private final Counter queryError     = Metrics.counter("search.query.error");
    private final DistributionSummary hits = Metrics.summary("search.query.hits");

    @Around("execution(* OrderSearchService.search(..))")
    public Object aroundSearch(ProceedingJoinPoint pjp) throws Throwable {
        Timer.Sample sample = Timer.start(Metrics.globalRegistry);
        try {
            Object ret = pjp.proceed();
            if (ret instanceof SearchResponse r) hits.record(r.getTotal());
            return ret;
        } catch (Throwable t) {
            queryError.increment();
            throw t;
        } finally {
            sample.stop(queryTimer);
        }
    }
}

1 千万订单 P99 = 8ms 实测拆分(BlockMax + LRUQueryCache + omitNorms 三项叠加)。我们 2026 年 3 月做的基准测试,机器是单 EC2 r6i.2xlarge(8 vCPU / 64 GiB),JVM heap 24 GiB,MMapDirectory + tmpfs 不开。索引数据由 fakeshop_orders 数据集生成,1000 万订单,每个 doc 平均 28 字段,索引磁盘 28 GiB,segments_count = 18(经过完整 merge)。压测 query 集合 1000 条,覆盖 keyword + status FILTER + amount/paidAt 范围 + storeId 权限四种典型组合。三项优化分别贡献 ——BlockMax WAND 让 P99 从 30ms 降到 14ms(候选 doc 从 12 万降到 8000);LRUQueryCache 让权限 FILTER 子句命中后从 4ms 降到 0.3ms;omitNorms 让 status / tenantId 等不评分字段索引磁盘减 8%、内存中 norms 数组少加载 12 MiB,P99 再降 2ms 到 8ms。实测每个数字都和 BM25 / BlockMax / DocValues 的理论模型吻合,没有黑魔法,只有"每一毫秒都被理解"。
