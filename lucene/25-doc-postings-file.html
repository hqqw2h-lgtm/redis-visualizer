第 25 章 · .doc 倒排数据文件

PFOR-delta · 128 doc block · ForUtil SIMD · 多层 SkipList · Impacts & BlockMax WAND · per-term postings 字节级解剖

.doc 文件保存倒排索引中每个 term 的 docID 流与每文档词频(freq)。每个 term 对应一条 postings list,按 128 docID 一组打成 block,每组用 PFOR-delta 编码压缩;block 之间附带 SkipList 加速 advance(target)。.doc 是倒排索引的"数据"载体,与 .tim(term metadata)、.pos(positions)、.pay(payloads)共同构成 Lucene99PostingsFormat 的物理存储。

查询路径以 .tim 中保存的 docStartFP 跳入 .doc,顺序解码 docID 与 freq;短语查询、proximity scoring 还会同步消费 .pos。理解 .doc 的字节布局是理解 BM25 评分性能、BlockMax WAND 跳块、SIMD 解码等高级特性的前提。本章按"PFOR-delta 编码 → 字节布局 → SkipList → Impacts → 写入读取 → 性能与陷阱"的顺序拆解 .doc,并在每节配字节级网格图与帧级算法演示。

  • Delta 编码 + Frame of Reference + PFOR exception 三层叠加压缩思路
  • ForUtil 如何把 int[128] 按 bitsPerValue 打包到 long[],SIMD 如何加速 2-3×
  • .doc 文件 Header / per-term postings / Footer 的字节级布局,Lucene99 codec name 的 magic 序列
  • Doc Block 内部 docDeltas + freqs + impacts 三段结构,以及 tail VInt 段如何处理 < 128 的尾部
  • SkipList 多层结构(skipMultiplier=8)、字节布局、advance 帧级跳跃流程
  • Impacts 如何作为 (maxFreq, minNorm) Pareto 前沿驱动 BlockMax WAND 跳过整 block
  • 典型 docFreq 下的字节估算(50 → 100 byte;100 万 → 1.1 MB)和 advance 时延矩阵
  • 8 类生产陷阱:stop 词未过滤、误用 docID、JDK 旧无 SIMD、segment 过多、delete 累积膨胀

25.1 PFOR-delta 编码

25.1.1 Delta encoding 的基础

倒排表中 docID 单调递增,直接存储绝对值会浪费高位 bit。Delta encoding 把"绝对 docID"转换为"相邻差值",压缩后值域显著变小。

图 25-1-1 · Delta 编码原理
原始 docID(已排序):
  100, 105, 107, 130, 135, 140, 141, 200, 205, 207, ...

Delta 转换:
  100, 5, 2, 23, 5, 5, 1, 59, 5, 2, ...

观察:
  - 第一个 100 是绝对值
  - 后续是相邻差值
  - 差值通常 << 原值

为何更省 bit:
  原始 max=207 需 8 bit(2^8 = 256)
  Delta max=59 需 6 bit
  对每个 doc 节省 2 bit,千万 doc 节省 2.5 MB

  更进一步:整 block 用 max delta 的位数定长打包

25.1.2 Frame of Reference

Frame of Reference(FOR)对 block 内所有值用同一个 bit 宽度打包。block 内 max 决定 bitsPerValue,小值会被零填充。

图 25-1-2 · FOR 编码与浪费空间
Block 内 deltas:
  5, 2, 23, 5, 5, 1, 200, 5, 4, 6, ...(128 个值)

求 max(deltas) = 200 → 需 8 bit

FOR 编码:
  bitsPerValue = 8
  每值占 8 bit
  128 值 × 8 bit = 1024 bit = 128 byte

观察:
  大多数值小(< 32, 5 bit 足够)
  只有一个 200 推高了 bitsPerValue
  → 浪费 128 × 3 bit = 48 byte

PFOR 改进思路:
  把 200 当 exception 单独存
  剩余 127 值用 5 bit 打包
  127 × 5 bit + exception(2 byte) ≈ 82 byte
  节省 36%

25.1.3 PFOR 的异常处理

PFOR(Patched Frame of Reference)选择一个较小的 bitsPerValue,把超出该宽度的值放到 exception 列表中。解码时主流程读 packed 值,遇到 exception 位置则用 exception 列表覆盖。

图 25-1-3 · PFOR 编码逐帧演示
Block deltas(简化为 8 个值):
  index:  0   1   2   3   4   5   6   7
  value:  5   2  23   5   5 200   6   4

帧 0(选 bitsPerValue):
  candidate=5 bit → 容纳 ≤31,异常率 = 1/8 = 12.5%
  candidate=8 bit → 容纳 ≤255,异常率 = 0%
  PFOR 选 5 bit(节省 3 bit/值,容忍少量异常)

帧 1(主流程打包):
  packed[0..7] = 5, 2, 23, 5, 5, ?, 6, 4
  index=5 占位为低 5 bit (200 & 0x1F = 8)

帧 2(exception 写入):
  exception[0] = (index=5, value=200)
  exception_bytes = 2 byte(1 byte 索引 + 1 byte 高位 / 整值)

帧 3(总尺寸):
  packed: 8 × 5 bit = 40 bit = 5 byte
  exception: 2 byte
  metadata: 1 byte (numExceptions)
  合计:8 byte vs 全 8-bit 的 8 byte

观察:
  本例 PFOR 与 FOR 持平,小例无优势
  block size = 128 时 PFOR 才显出空间优势

25.2 ForUtil 实现细节

25.2.1 ForUtil 的核心方法

org.apache.lucene.codecs.lucene99.ForUtil 是 PFOR-delta 编码的核心实现,负责把 int[128] 按指定 bitsPerValue 打包到 long[],以及反向解包。

图 25-2-1 · ForUtil 编码流程
图 25-2-2 · bit-packing 字节示意
打包 8 个 5-bit 值到字节流:

输入:  0b00101 0b00010 0b10111 0b00101 0b00101 0b00001 0b00111 0b00001
       v0=5  v1=2  v2=23 v3=5  v4=5  v5=1  v6=7  v7=1

输出 byte 流:
  byte[0] = v0[4:0] | v1[2:0] << 5   = 0b01000101 = 0x45
  byte[1] = v1[4:3] | v2[5:0] << 2   = 0b10111100 = 0xBC
  byte[2] = v3[2:0] << 5 | v4[4:0]   = 0b10100101 = 0xA5
  ...

实际实现用 long 单位连续打包,避免按字节移位

8 值 × 5 bit = 40 bit → 5 byte(无浪费)
128 值 × 5 bit = 640 bit = 80 byte(无浪费)

25.2.2 SIMD 向量化

JDK 16+ 的 Vector API 让 ForUtil 可以用 SIMD 指令一次处理 4-8 个 long。Lucene99PostingsReader 在解码 doc block 时调用 ForUtil.decode(...),内部用 LongVector 平摊位移与按位运算。

图 25-2-3 · SIMD 解码加速

SIMD 解码 1 block(128 doc)耗时约 50-100 ns,相比标量解码(150-300 ns)快 2-3 倍。Lucene 9 标准 codec 默认开启 SIMD,JDK < 17 自动回退到标量路径。

25.3 跨 block 的累加恢复

25.3.1 Block 起点与累加

.doc 的每个 doc block 内部存的是 delta(相对 block 内前一 doc),整 block 末尾的累加值还需要再加上 block 起点 docID 才能得到绝对 docID。

图 25-3-1 · 跨 block 累加示意
docID 序列(绝对值):
  block 0: 100, 105, 107, 130, ..., 250
  block 1: 251, 255, 280, ..., 400

block 0 内部存储:
  base = 100(block 起点)
  deltas = 5, 2, 23, ...  (128 个)
  累加恢复: 100 → 105 → 107 → 130 → ...

block 1 内部存储:
  base = 251 (= 250 + 1, 也按 delta 编码)
  deltas = 4, 25, ...

reader 维护:
  long currentDoc = 0
  for each block:
    base = currentDoc + readVInt()  // block 起点 delta
    decode 128 deltas into int[128]
    for i in 0..128:
      currentDoc = base + prefix_sum(deltas[0..i])

25.3.2 tail VInt 段

图 25-3-2 · tail VInt 段
docFreq = 1000 时:
  完整 block: 1000 / 128 = 7 个(前 896 doc)
  tail VInt: 1000 - 896 = 104 个 doc

tail 编码:
  剩余 doc 用 VInt(无 block 头),逐对读:
    delta_0(VInt) freq_0(VInt)
    delta_1(VInt) freq_1(VInt)
    ...
    delta_103(VInt) freq_103(VInt)

为何 tail 不再 PFOR:
  PFOR 摊薄成本依赖 128 doc 一打包,tail < 128 时反而有元数据开销
  VInt 平均 1.2 byte/doc,tail 总开销 < 200 byte

  bufferUpto = 1
  return docBuffer[1]

25.7 SkipList 多层结构

25.7.1 SkipList 设计思路

倒排表中相邻 doc 必须顺序解码(PFOR 不支持随机访问 block 内某 doc),但 block 之间可以随机跳转。SkipList 在 block 之间建立稀疏索引,让 advance(target) 跳过多个 block 而不必逐 block 解码。

图 25-7-1 · SkipList 多层示意

              .    .    .    .    .    .    .    .    .    .    .    .    .    .    .
Block:   B0   B1   B2   B3   B4   B5   B6   B7   B8   B9   B10  B11  B12  B13  B14  B15
              |         |         |         |         |         |         |         |
Level 0:      *---------*---------*---------*---------*---------*---------*---------*
              (每 1 block 1 skip,实际默认 8 block 1 skip)

              |                   |                   |                   |
Level 1:      *-------------------*-------------------*-------------------*
              (每 8 block 1 skip)

              |                                       |
Level 2:      *---------------------------------------*
              (每 64 block 1 skip)

advance(target=950) 算法:
  从最高 level 开始
  Level 2:
    skipDoc[0] = 100, skipDoc[1] = 1100
    100 < 950 < 1100 → 下降到 Level 1
  Level 1:
    skipDoc[0] = 100, skipDoc[1] = 500, skipDoc[2] = 900, skipDoc[3] = 1300

  Level 0:
    从 skip[2] 起向后,skipDoc 序列:900, 1000, 1100, ...
    跳到 skip 对应 1000(1000 ≥ 950,跳到第一个 ≥)
    退一步到 900,seek 到 Block 9
  在 Block 9 内顺序 nextDoc 直到 docID ≥ 950

25.10 Impacts 与 BlockMax 元数据

25.10.1 Impacts 的含义

Impacts 是 Lucene 8 引入的 per-block 评分上界。每个 doc block 附带一组 (maxFreq, minNorm) 对(最多 8 个),让 BlockMax WAND 可以预估该 block 的 BM25 评分上界,从而决定是否解码该 block。

图 25-10-1 · Impacts 数据结构
每 block 附 Impacts 元数据(典型 6-12 byte):

  ┌──────────────────────────────────┐
  │ numImpacts(VInt) : 1-8           │
  ├──────────────────────────────────┤
  │ Impact[0]:                        │
  │   freqDelta(VInt) : 相对前一        │
  │   normValue(VLong)                │
  ├──────────────────────────────────┤
  │ Impact[1]:                        │
  │   freqDelta(VInt)                 │
  │   normValue(VLong)                │
  ├──────────────────────────────────┤
  │ ...                               │
  └──────────────────────────────────┘

含义:
  freq 升序,norm 降序 — 形成 Pareto 前沿
  reader 可计算 score = idf × tf(freq) / norm
  block 内 max score 不会超过这些 Impact 的最大组合

实例:
  numImpacts = 3
  Impact[0]: freq=2, norm=100  → score ≈ idf × 2 / 100 = 0.02 × idf
  Impact[1]: freq=5, norm=50   → score ≈ idf × 5 / 50  = 0.1 × idf
  Impact[2]: freq=10, norm=20  → score ≈ idf × 10 / 20 = 0.5 × idf
  
  block 上界 = max = 0.5 × idf

25.10.2 Impacts 与 BlockMax WAND

WAND(Weak AND)算法在多 term 查询中估算每个 doc 的潜在评分上界。BlockMax WAND 进一步在 block 粒度上跳过——若整 block 的最大评分仍 < Top-K 阈值,跳过整 block。

25.16.2 反例 → 修复对照(误用 docID)

❌ 反例
// 把 Lucene 的 docID 当业务主键
public class OrderCache {
  private Map<Integer, Order> cache = new HashMap<>();
  
  public void cache(int docID, Order order) {
    cache.put(docID, order);
  }
  
  public Order get(int docID) {
    return cache.get(docID);
    // merge 后 docID 重排,全部失效
  }
}

// 现象:
//   小 segment merge 完成后,
//   cache 命中率从 95% 降到 0%
//   后端突发流量打到数据库
✅ 修复
// 用 stored field 存业务主键
public class OrderCache {
  private Map<String, Order> cache = new HashMap<>();
  
  public void cache(IndexReader reader, int docID, Order order) {
    Document doc = reader.document(docID);
    String orderId = doc.get("orderId");
    cache.put(orderId, order);
  }
  
  public Order get(String orderId) {
    return cache.get(orderId);
    // merge 不影响业务主键
  }
}

// 收益:
//   merge 不影响 cache
//   cache 命中率稳定 95%+

25.17.2 BLOCK_SIZE 设计取舍

BLOCK_SIZE = 128 是 Lucene 9 的硬编码常量,不可调。该值是经过大量实测的折中:

  • 128 = SSE2/AVX2 友好的整数,SIMD 解码 1 block 一次完成

25.18.1 慢查询排查清单

实战清单: 慢查询(P99 > 100 ms)排查步骤
  1. 查 .doc 大小 — 单 segment .doc > 1 GB 触发 page miss 风险
  2. 查 docFreq 分布 — 用 IndexReader.docFreq(term) 找 top 10 term;任意 term docFreq > 10M 视为热点
  3. 查 segment 数量 — 超过 30 个建议 forceMerge;merge 后 .doc 大小通常下降 20%
  4. 查 codec 版本 — 必须 Lucene99PostingsFormat;旧版本无 Impacts,WAND 失效
  5. 查 JDK 与 SIMD — JDK 17+ 才启用 Vector API;启动日志查 "SIMD ForUtil decode"
  6. 查 page cache 命中率 — vmtouch 或 free -h 看可用内存;.doc 完整驻留 page cache 是理想状态
