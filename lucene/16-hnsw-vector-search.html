第 16 章 · HNSW 向量检索

Embedding · 多层小世界图 · 贪心下降 · efSearch / M · KnnFloatVectorField · Lucene99HnswVectorsFormat · 标量量化 · PQ · 召回 vs 延迟

用户在搜索框打"昨天那个上海客户的紧急订单"——倒排索引彻底失效。这种自然语言查询没有具体 term,只有语义,需要把句子先编码成 768 维 float 向量,再在 1 亿个订单向量里找最相似的 Top-100。Lucene 9.x 引入 KnnFloatVectorField 与 Lucene99HnswVectorsFormat,通过 HNSW(Hierarchical Navigable Small World) 多层图索引把这个查询从 O(N×dim) 暴搜降到 O(log N × dim),1 亿向量 1 ms 出 Top-100,recall@10 ≥ 0.95。这是 ANN(Approximate Nearest Neighbor)的工业标准。

HNSW 的核心是"分层小世界图":每个向量在图里是一个节点,节点连若干邻居形成图,把图拆成多层(顶层稀疏,底层全部节点),查询从顶层进入贪心找最近邻,逐层下降到底层取出 Top-K。本章拆解 HNSW 的层级结构、构造算法(随机分配 layer + heuristic 选邻居)、搜索算法(贪心 + 候选堆)、Lucene 实现(OnHeapHnswGraph + HnswGraphBuilder + HnswGraphSearcher)、.vec/.vem/.vex 三文件物理布局、标量量化(SQ7 内存省 4×)与 Product Quantization、段合并算法、与 filter 混合查询的退化与修复——全部对应 Lucene 9.x。

  • 为什么倒排不能解决向量检索:每个浮点是独立 term, 词典爆炸 + 无相似度概念
  • HNSW 多层图原理:Layer 0 全部节点, 上层节点数指数衰减(每层 1/e), 顶层 2-3 个

  • 邻居选择的 heuristic 多样性策略: 不让邻居都聚在 q 同方向, 避免局部极小

16.1 向量检索的痛点:为什么倒排不行

倒排索引是 Lucene 的灵魂,但它有一个根本前提: "匹配 = term 相等"。一个 term 出现在 doc 里就命中,否则不命中。这个二元判断对全文搜索完美——用户搜 "iphone", 你看 .tim 词典里有没有 "iphone" 这个 term, 然后用 PostingsEnum 取出 doc 列表。但向量检索是连续相似度: 用户的 query embedding 和每个 doc embedding 的余弦相似度是 0 到 1 之间的 float, 没有"相等不相等",只有"多接近"。

把 float 当 term 索引会怎样

假设把 768 维 float[] 强行拆成 768 个 term(每个维度一个值):

  doc 1: [0.12, -0.45, 0.78, ..., 0.03]
         → term: "dim0:0.12", "dim1:-0.45", "dim2:0.78", ..., "dim767:0.03"

  doc 2: [0.13, -0.44, 0.79, ..., 0.04]   (语义几乎一样)
         → term: "dim0:0.13", "dim1:-0.44", "dim2:0.79", ..., "dim767:0.04"

问题:
  1. 词典爆炸: float 精度无穷, 1 亿向量 × 768 维 = 768 亿独立 term
                .tim 文件直接 TB 级
  2. 没有相似度: doc 1 和 doc 2 在每个维度上都不完全相等
                查询无法找到"接近的"
  3. 距离不能用 BooleanQuery 表达
                cosine / L2 是连续函数, 不是 AND/OR 组合

倒排索引的世界里, 0.12 和 0.13 是完全不同的两个 term,
没有任何机制告诉它们"很接近"。

暴搜的不可行

退一步,直接对每个查询暴力计算与所有 doc 向量的距离呢?数学上无懈可击,工程上是灾难——O(N × dim) 浮点运算,1 亿 × 768 维 = 7.68 × 10¹⁰ FLOPs,在线查询不可接受。

16.2 HNSW 算法本质:分层小世界图

HNSW 是 Hierarchical Navigable Small World 的缩写, 由 Yu A. Malkov 和 D. A. Yashunin 在 2016 年的论文里提出。三个关键词: Hierarchical(分层) + Navigable(可导航) + Small World(小世界)。理解这三个词, 就理解了整个算法的灵魂。

Small World 图的核心性质

"小世界"是 Watts 和 Strogatz 1998 年的概念: 一个图, 每个节点连若干邻居, 满足三个性质——(1) 任意两点距离 O(log N)(短链), (2) 局部聚集(近邻点连在一起), (3) 少量长边(提供跨区域导航)。社交网络是典型: 你和地球上任何人之间平均只隔 6 个人(短链), 你的朋友们大多互相认识(局部聚集), 偶尔有个朋友在地球另一端(长边)。

规则图(全连接 / 网格)

  • 每节点连固定 K 个邻居(规则结构)
  • 导航需要 O(N) 步遍历(没有捷径)
  • 无长边, 跨区域要走完所有中间节点
  • 不适合 ANN: 贪心搜索容易陷入局部

小世界图(HNSW 用)

  • 每节点连 M 个邻居 + 部分远距离长边
  • 任意两点 O(log N) 步可达
  • 长边提供跨区域跳跃(关键)
  • 贪心搜索从远到近高效收敛

"分层"是为什么

单层小世界图已经能 O(log N) 找到最近邻, 但 HNSW 把它分成多层: Layer 0 包含全部 N 个节点, Layer 1 约 N/e 节点, Layer 2 约 N/e² 节点, ..., 顶层只有 2-3 个节点。顶层稀疏, 适合远距离粗略导航; 底层稠密, 适合近距离精细搜索。查询从顶层进入贪心找入口, 找到后下降到下一层, 在下一层从这个入口继续贪心, 一直到 Layer 0 取出 Top-K——这个"逐层精细化"是 HNSW 比单层 NSW 快的根本原因。

图 16-2-1 · HNSW 多层小世界图(手绘可视化)
"上稀下密" 是 HNSW 的几何精髓。顶层只有 2-3 个节点, 节点之间彼此长距离相连, 形成一个"区域索引"——查询从顶层进入, 第一步贪心就能定位到大致区域。每往下一层, 节点数 × e ≈ 2.7 倍密度, 同时引入更多近距离的邻居; 到 Layer 0 全部 N 节点都在, 邻居完全是局部的——精细化 Top-K。没有分层, 单层 NSW 也能找到最近邻, 但贪心搜索的步数会从 log(N) 退化到 √N 级。

"Navigable" 的含义

Navigable 在论文里是个数学术语: 给定一个图和起点, 贪心搜索(每步选当前节点的邻居中最接近 q 的)能在 polylogarithmic 步数内收敛到全局最近邻。Navigable 的关键是 邻居选择策略——选 M 个邻居时, 不能贪心地选最近的 M 个(那样邻居都聚在一团, 没有长边), 必须用 heuristic: 既要近, 又要互相分散, 保证有跨区域的长边。这点在 16.5 详细讲。

16.3 距离度量与相似度函数

HNSW 是距离不可知的——算法本身只需要"两个向量之间的距离"这个抽象操作, 不限定 L2 / cosine / dot。Lucene 提供了 4 种内置 VectorSimilarityFunction, 每种对应一个具体的数学函数和典型业务。

表 16-3-1 · Lucene VectorSimilarityFunction 4 种
常量数学公式范围语义典型业务
EUCLIDEAN‖a - b‖²(L2 平方)[0, +∞)距离, 越小越相似图像哈希
COSINE(a · b) / (‖a‖ × ‖b‖)[-1, 1]角度相似度, 越大越相似BERT / OpenAI embedding
DOT_PRODUCTa · b(-∞, +∞)需归一化向量, 等价 cosine性能优于 cosine
MAXIMUM_INNER_PRODUCTargmax(a · b)(-∞, +∞)不要求归一化的内积推荐系统(MIPS)

为什么 cosine 是文本检索的默认

BERT / Sentence-BERT / OpenAI text-embedding 等语义模型输出的向量, 模长(‖v‖)与语义无关——同一句话用不同长度的输入(全句 vs 关键词)会得到不同模长的向量, 但它们语义相同, 应该判定为高度相似。Cosine 只看角度, 自动消除模长差异。这是文本检索默认 COSINE 的根本原因。

cosine 公式拆解:

  cos(a, b) = (a · b) / (‖a‖ × ‖b‖)
            = Σ a_i × b_i / (sqrt(Σ a_i²) × sqrt(Σ b_i²))

  例子:
    a = [1, 0]                   ‖a‖ = 1
    b = [3, 0]                   ‖b‖ = 3        ← 模长不同
    cos(a, b) = (1×3 + 0×0) / (1 × 3) = 1.0    ← 角度 0°, 完全相似

  vs L2:
    ‖a - b‖² = (1-3)² + 0² = 4   ← L2 觉得它们很远

cosine 把 a 和 b 视为方向向量, 模长被归一化掉。
对于"语义相同, 表达不同"的文本场景, cosine 是正解。

归一化后的 dot 等价 cosine 但更快

若所有 doc 向量都先归一化(‖v‖ = 1), 那么 cos(a, b) = a · b——两者数学上等价, 但 dot 省掉了运行时的除法 + sqrt 计算, 典型快 2-3 倍。生产里推荐流程: 写入时 VectorUtil.l2normalize(v) 归一化, 索引设 DOT_PRODUCT, 查询时 query embedding 同样归一化。

归一化必须双侧。如果 doc 向量归一化了但 query 没归一化, dot 与 cosine 不等价, recall 会暴跌(典型从 0.95 → 0.6)。常见 bug: 训练时用 cosine loss(模型自动归一化输出), 但推理时调用方忘了对 query embedding 调 l2normalize——线上跑了几天发现召回莫名其妙差。修复: 在 encode pipeline 入口加强制归一化, 不依赖调用方自觉。

距离 → 相似度 → 评分

Lucene 内部把"距离 / 相似度"统一转换为 score(0-∞ 之间的浮点, 越大越好)。EUCLIDEAN 距离 d 转换成 1 / (1 + d)(距离越小, score 越接近 1); COSINE 直接 (1 + cos) / 2(把 [-1, 1] 线性映射到 [0, 1]); DOT 走 (1 + dot) / 2 但要求归一化。这套转换让所有 KNN query 与其他 BooleanQuery 拼接时 score 是可比的。

16.4 多层图与 log(N) 跳数

16.2 概念性提到了多层结构, 这一节给出具体数字。理解层级分配的概率分布与 log(N) 跳数的来源, 是后面调参 M 的基础。

层级随机分配

HNSW 给每个新节点随机抽取一个 layer, 抽法是指数分布: level = floor(-ln(uniform(0, 1)) × M_L), 其中 M_L = 1 / ln(M)(M 是每节点邻居数, 默认 16, 所以 M_L ≈ 0.36)。这个公式让节点数随层数指数衰减——Layer 0 全部, 上一层 1/e ≈ 37%, 再上 14%, 5%, 2%, ...

16.5 构造算法:插入与连边

HNSW 是增量构造的: 每个向量逐个 add 到图里, 不需要预知全部数据。这与 IVF(需要先 k-means 聚类)截然不同, 特别适合 Lucene 的 segment 模型(段是只追加的)。

Heuristic 邻居选择(关键)

从 efConstruction 个候选里选 M 个邻居, 简单做法是取距离最近的 M 个——但这导致邻居全聚在 q 同一方向, 没有跨区域长边, 图退化。Heuristic 策略: 按距离排序候选; 依次考虑每个候选 v, 当且仅当 v 离 q 比离任何已选邻居都近, 才把 v 加入选中列表; 直到选满 M。

Heuristic 选邻居伪代码:

  function selectNeighborsHeuristic(q, candidates, M):
      sort candidates by dist(q, c) ascending
      selected = []
      for c in candidates:
          # 条件: c 离 q 比离任何已选邻居都近
          if all(dist(q, c) < dist(c, s) for s in selected):
              selected.append(c)
              if len(selected) == M:
                  break
      return selected

  几何含义: 不要让邻居们都"挤"在 q 同一边
            选 c 的条件是 q 是 c 当前最近的"中心"
            这保证邻居分散在 q 周围不同方向
            形成跨区域长边, 维持 Navigable 性质
Heuristic 是 HNSW 比早期 NSW 强的关键。Malkov 在论文里明确指出: 简单选邻居(top M closest)在低维数据(D < 10)上工作得不错, 但在高维数据(D ≥ 100, BERT 是 768)上 recall 显著下降, 因为高维空间里"近邻"会聚在一团失去导航性。Heuristic 的"分散选择"本质上是模拟小世界图的"长边", 在高维空间里至关重要——这也是为什么 Lucene 默认用 heuristic, 不暴露简单策略的开关。

构造代价

每个 insert 调用 searchLayer(efConstruction) 平均 log(N) 次, 每次扩展 efConstruction 个候选(每个候选要算一次距离)。所以单次 insert ≈ O(efConstruction × M × log N) 次距离计算。1M 向量 768 维 + efConstruction=100 + M=16 → 总建图时间约 5-10 分钟(单线程, SSD)。

16.6 搜索算法:贪心下降 + 候选堆

构造完成后, 查询路径异常清晰: 顶层贪心下降到 Layer 0 入口, 然后在 Layer 0 用 efSearch 候选堆扩展, 取 Top-K。本节给出 searchLayer 的完整伪代码与 Lucene 实现。

searchLayer 伪代码

function searchLayer(q, ep, ef, layer):
    # ep: 入口节点 ord
    # ef: 候选堆大小(查询时 = efSearch, 构造时 = efConstruction)
    # layer: 当前层

    candidates = MinHeap()        # 按 dist(q, n) 升序, 弹出最近的
    results    = MaxHeap()        # 按 dist(q, n) 降序, 弹出最远的, 保留 Top-ef
    visited    = BitSet()

    candidates.push(ep, dist(q, ep))
    results.push(ep, dist(q, ep))
    visited.add(ep)

    while not candidates.empty():
        c = candidates.pop()        # 当前最近的候选
        # 终止条件: c 已经比 results 里最远的还远
        # → 继续探索不可能产生更好的结果
        if dist(q, c) > results.top():
            break

        for n in graph.neighbors(c, layer):
            if visited.contains(n):
                continue
            visited.add(n)
            d = dist(q, n)
            # n 比 results 最远的更近, 或 results 还没满
            if d < results.top() or len(results) < ef:
                candidates.push(n, d)
                results.push(n, d)
                if len(results) > ef:
                    results.popMax()    # 维持 ef 大小

    return results    # 返回 ef 个候选

16.10 标量量化 SQ7/SQ4: 内存 4× 缩减

HNSW 完美但有一个致命短板: 内存占用。1 亿个 768 维 float32 向量 = 1e8 × 768 × 4 = 307 GB, 加图结构 ~12 GB, 总计接近 320 GB——一台机器装不下。Lucene 9.9 引入 Lucene99HnswScalarQuantizedVectorsFormat, 把 float32 量化为 int7(SQ7)或 int4(SQ4), 内存直降 4-8 倍, recall 损失 < 1%。

SQ7 量化原理

SQ7(7-bit Scalar Quantization)算法:

  训练阶段(全局统计):
    扫描所有向量, 计算每个维度的 min, max
    或全局 quantile (1%, 99%) 防异常值

  量化(写入时):
    int7 = round((float - min) / (max - min) × 127)    // [0, 127]

  反量化(查询时):
    float ≈ int7 / 127 × (max - min) + min

  距离计算优化:
    cosine(a, b) ≈ Σ (q_a × scale + min) × (q_b × scale + min)
                 = scale² × Σ q_a × q_b + 常数项
    → 实际计算只需 byte × byte 内积, 比 float × float 快 4×

  内存收益:
    float32: 4 bytes/dim → byte: 1 byte/dim
    768 dim = 3072 → 768 bytes
    1e8 doc = 307 GB → 76 GB

SQ7 vs SQ4 vs FP32 对比

FP32 (无量化)

  • 4 byte / dim, 768 dim = 3072 bytes
  • 1e8 向量 = 307 GB
  • recall@10 baseline = 0.95
  • 查询 latency = 5 ms
  • 不适合 1e8+ 规模

SQ7 (1 byte / dim, 7 bit 实际)

  • 1 byte / dim, 768 dim = 768 bytes
  • 1e8 向量 = 76 GB
  • recall@10 = 0.94 (损失 ~1%)
  • 查询 latency = 3 ms(SIMD byte 内积更快)
  • 生产推荐 1e7-1e9 规模

FP32 vs SQ8 详细对比

FP32 全精度 ❌ (1e8+ 规模)

向量大小3072 B / 768D
1e8 内存~307 GB
SIMD 单步8 floats (AVX2)
距离 latency~0.4 µs / pair
recall@100.950
Page cache多机分片

SQ7 量化 ✓ (生产推荐)

向量大小768 B / 768D
1e8 内存~76 GB
SIMD 单步32 bytes (AVX2)

16.11 Product Quantization 进阶

SQ 是逐维度独立量化, 假设各维度分布相似——这个假设在低维(D < 100)成立, 在高维(D ≥ 768)就显得粗糙。Product Quantization (PQ) 把向量切成 m 个子向量, 每个子向量用 k-means 聚类, 用聚类中心 ID 替代子向量——这是 Faiss 的核心技术, Lucene 社区在路上。

距离计算: ADC (Asymmetric Distance Computation)

PQ 查询时的 ADC 距离计算:

  1. 离线: 训练 m × K 个聚类中心 c[i][j]
           其中 i ∈ [0, m), j ∈ [0, K)

  2. 写入: 每个 doc 向量 v_d 切成 m 子向量
           每子向量找最近聚类中心 ID
           v_d 压缩成 m bytes (每 byte = 中心 ID)

  3. 查询时:
     a. query 向量 q 切成 m 子向量 q[0..m-1]
     b. 对每个子空间 i, 预计算 q[i] 到 K 个中心的距离
        → 距离表 dist_table[i][j] = ‖q[i] - c[i][j]‖²
        总共 m × K 个 float (256 × 8 = 2KB), 完全装进 L1 cache
     c. 对每个 doc, 查 m 次表累加:
        dist(q, v_d) ≈ Σ_i dist_table[i][v_d_codes[i]]
        每 doc m 次表查询, 极快

  ADC 优势:
    - 距离计算只需 m 次查表 + 累加, 无浮点乘法
    - 距离表完全在 cache, 命中率 100%
    - 1e8 doc 的 Top-100 查询延迟 ~ 1 ms

合并代价

HNSW merge 是写入路径上最贵的操作。单段构造 1M 向量 ~5 分钟, 合并 10 段 1M 到 10M ~30 分钟——是构造的 6 倍。生产里要避免: (1) maxMergedSegmentMB 设小 (~5GB), 限制单次合并规模; (2) 业务低峰期触发 forceMerge(maxNumSegments=10), 不要在线 forceMerge 到 1; (3) 启用 numMergeWorkers 并行(Lucene 9.10+); (4) 用 SQ7 减小图规模, 间接加速 merge。没规划好 merge 的 HNSW 集群, 半夜段合并能把 GC / IO 全打爆。

16.13 内存与磁盘估算

HNSW 的内存模型必须精确算清楚——上线前算错, 线上 OOM 就晚了。本节给出三种典型规模下的具体数字。

内存估算公式

HNSW 总内存 = 向量数据 + 图结构 + JVM 开销

  向量数据 (heap or mmap):
    FP32: dim × 4 × N bytes
    SQ7:  dim × 1 × N bytes
    SQ4:  dim × 0.5 × N bytes (打包)

  图结构 (mmap 文件 .vex):
    Layer 0: 2 × M × 4 × N bytes (邻居 ord, 加倍 = 32 邻居)
    其他层:  M × 4 × N × Σ(1/e^l) ≈ M × 4 × N × 0.6
    总计:    M × 4 × N × 2.6

  JVM 开销:
    HnswGraphReader 缓存: ~100 MB
    OS page cache 份额:   建议 1.5× 向量数据大小

16.14 召回评估与 efSearch 调参

HNSW 是近似算法, 没有"召回率"数字就不能上线——你不知道返回的 Top-10 里有几个是真实最近邻。本节给出召回评估的标准流程与 efSearch 调参手册。

Brute Force 基线

评估必须有暴力 KNN 基线: 对同一组查询向量, 遍历所有 doc 算 cosine 取真实 Top-K, 然后比对 HNSW 返回的 Top-K, 计算 recall@K = |HNSW_topK ∩ exact_topK| / K。基线只在评估时跑, 不进生产查询路径。

import org.apache.lucene.index.LeafReaderContext;
import org.apache.lucene.index.IndexReader;
import org.apache.lucene.search.IndexSearcher;
import org.apache.lucene.search.KnnFloatVectorQuery;
import org.apache.lucene.search.TopDocs;
import org.apache.lucene.util.PriorityQueue;
import org.apache.lucene.index.FloatVectorValues;
import org.apache.lucene.search.DocIdSetIterator;
import java.util.*;

public class BruteForceKnnEvaluator {

    public static int[] exactTopK(IndexReader reader, String field,
                                   float[] query, int k) throws IOException {
        PriorityQueue<float[]> pq = new PriorityQueue<>(k) {
            @Override
            protected boolean lessThan(float[] a, float[] b) {
                return a[0] < b[0];
            }
        };

        for (LeafReaderContext leaf : reader.leaves()) {
            FloatVectorValues vv = leaf.reader().getFloatVectorValues(field);
            if (vv == null) continue;
            int doc;
            while ((doc = vv.nextDoc()) != DocIdSetIterator.NO_MORE_DOCS) {
                float score = cosine(query, vv.vectorValue());
                if (pq.size() < k) {
                    pq.add(new float[]{score, leaf.docBase + doc});
                } else if (score > pq.top()[0]) {
                    pq.top()[0] = score;
                    pq.top()[1] = leaf.docBase + doc;
                    pq.updateTop();
                }
            }
        }

        return Arrays.stream(pq.toArray(new float[0][]))
                     .sorted((a, b) -> Float.compare(b[0], a[0]))
                     .mapToInt(a -> (int) a[1])
                     .toArray();
    }

    public static double recallAtK(IndexReader r, String field,
                                    float[] q, int k) throws IOException {
        int[] exact = exactTopK(r, field, q, k);
        TopDocs hnsw = new IndexSearcher(r).search(
            new KnnFloatVectorQuery(field, q, k), k);
        Set<Integer> exactSet = new HashSet<>();
        for (int d : exact) exactSet.add(d);
        long hit = Arrays.stream(hnsw.scoreDocs)
                         .filter(s -> exactSet.contains(s.doc))
                         .count();
        return (double) hit / k;
    }

    private static float cosine(float[] a, float[] b) {
        float dot = 0, na = 0, nb = 0;
        for (int i = 0; i < a.length; i++) {
            dot += a[i] * b[i];
            na += a[i] * a[i];
            nb += b[i] * b[i];
        }
        return (float) (dot / (Math.sqrt(na) * Math.sqrt(nb)));
    }
}
