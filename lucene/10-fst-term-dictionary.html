第 10 章 · FST 与词典索引

Finite State Transducer · BlockTree · .tim · .tip · seekExact · Floor Block · TermsEnum · Util.toDot

"我有 1 千万个 unique term,要在毫秒级支持精确查找、前缀匹配、范围扫描,并且整个词典必须能塞进进程内存"——这是搜索引擎面对的现实约束。HashMap 在 1 千万 key 上要吃 600+ MiB 且不支持范围,TreeMap 是 250 MiB 但 GC 抖动严重,Trie 共享前缀后还要 200 MiB。Lucene 的答案是 FST(Finite State Transducer):同样 1 千万 term,FST 索引仅 30-80 MiB,可以 mmap 直接被 OS page cache 管理,GC 几乎为零。

FST 不是一个简单的 trie——它是 带输出的最小化自动机:每条转移边携带一个输出值(term → block 文件指针),不仅共享公共前缀,还共享 公共后缀。"charging / charing / charging" 这类词的"-ing"后缀只编码一次,结果是 1 千万英文 term 的 FST 比未压缩的 trie 小 60×。Lucene 把这个数据结构封装成 BlockTree 词典:.tip 文件常驻内存存 FST 索引(指向 block 入口),.tim 文件按字典序在磁盘上存放 25-48 个 term 一组的 block(term 后缀 + postings 指针)。一次 seekExact("charger") 只需 O(L + B):FST 沿 "char" 走 4 步定位 block 入口,在 25-48 个 term 的 block 内顺序扫"ger"后缀。本章把 FST 从最朴素的状态机一路画到 .tip 文件的字节布局,再回到 Lucene 在 Suggester / Completion / WildcardQuery 上的实战路径。

  • FST 的核心定义:为什么"带输出的最小化 DFA"是序列化友好的词典结构,以及 hash / trie / FST 三者的内存与查询权衡
  • FST 的增量构造算法:Lucene 为什么不能用纯递归(内存爆炸),而是用 frontier + hashConsing 的 streaming build
  • FST 的字节布局:Arc 的 flag bit / label / output / target 在 byte 数组里如何紧凑编码,direct addressing vs binary search vs linear scan 三种 arc lookup 模式

10.1 词典索引的需求:为什么不用 HashMap

最直观的"term → postings 指针"映射就是 HashMap<String, Long>。它在 Java 单元测试里跑得飞快,但放到一个有 1 千万 unique term 的搜索引擎里就垮了。理解这个垮的过程,就理解了为什么 Lucene 投入了十几年工程精力优化 FST——它不是"另一种实现",而是 唯一能同时满足"小到能装进内存 + 支持范围 + mmap 友好 + 段合并便宜"的数据结构。

所有查询都走 FST 的字典序自动机,代价与命中量线性,与 N 无关。

词典选型的核心不是 seekExact 速度,而是查询模式的覆盖度。倒排索引上 90% 的查询不是单 term 命中,而是 range / prefix / phrase / fuzzy / regex——这些查询都需要按字典序遍历或按前缀展开。HashMap 的 O(1) seekExact 在这些查询模式下完全派不上用场,反而因为不能按序遍历变成性能黑洞。FST 的"O(L) 单 term + O(L+K) 前缀 + 自动机求交"在所有模式下都是次线性的——这才是搜索引擎需要的。

10.2 FST 的概念:带输出的最小化 DFA

FST(Finite State Transducer,有限状态转换器)在形式语言里的定义是:一个 5 元组 (Q, Σ, Γ, δ, λ),其中 Q 是状态集合,Σ 是输入字母表,Γ 是输出字母表,δ 是状态转移函数 Q × Σ → Q,λ 是输出函数 Q × Σ → Γ*。说人话就是:它是一个普通的 DFA,但每条转移边 除了消费一个输入字符,还产生一段输出。Lucene 用 FST 表示 term → block 文件指针 的映射:输入是 term 的字符序列,输出是该 term 所在 block 在 .tim 文件里的字节偏移。

从 DFA 到 FST 的演进

图 10-2-1 · DFA / Trie / FST 的能力递进

关键的演进有三步:(1) 普通 DFA 只回答"接受 / 拒绝",不能存任何额外信息;(2) Trie 是带终结状态的 DFA,可以记"接受",但每条公共前缀只共享一次,公共后缀的"-ing"还是要重复编码 N 份;(3) 最小化 DFA 把等价状态合并,公共后缀也共享了——但它仍然是布尔语义;(4) FST 在每条 arc 上挂一个输出值 (long / BytesRef),沿着输入路径走一遍并把所有 arc 的 output 累加,得到的就是这个 term 的映射结果。

Outputs 抽象:FST 的多态接口

Lucene 的 FST 不写死输出类型——它把"如何累加 / 如何序列化"抽象成 Outputs<T> 接口。BlockTree 用 PositiveIntOutputs(累加 = 加法、序列化 = VLong);Suggester 用 PairOutputs(累加 = 配对加 + 序列化 = 串联);自定义场景可以提供任意 Outputs 实现。

public abstract class Outputs<T> {
    public abstract T common(T output1, T output2);

    public abstract T subtract(T output, T inc);

    public abstract T add(T prefix, T output);

    public abstract void write(T output, DataOutput out) throws IOException;

    public abstract T read(DataInput in) throws IOException;

    public abstract T getNoOutput();
}
FST 不是只有 Lucene 用 — Apache OpenNLP 的词典、Elasticsearch 的 Suggester、Solr 的 SortedTermsEnum、Vespa 的 attribute index、Tantivy(Rust 搜索引擎)的 term dict 都基于同一份 FST 算法。Apple 的 Siri 语音识别、Google 翻译的语言模型、ICU 的 BreakIterator 也都用 FST 做"输入序列 → 输出标签"的映射。这是一项跨越 30 年的技术——Mehryar Mohri 1997 年的论文《Finite-State Transducers in Language and Speech Processing》是它在 NLP 领域的奠基,Robert Muth 1995 年提出的增量构造算法是它在搜索领域的工程突破。

10.4 FST 的关键性质

10.3 节的例子展示的不是某种"巧妙写法",而是 FST 必须满足的四条数学性质。理解了这四条,才知道 Lucene 为什么不能用纯递归构造、为什么 term 必须按字典序输入、为什么最小化是"用 hashConsing 而不是后处理"。

四条数学性质

表 10-4-1 · FST 的形式化性质

什么叫"两个状态等价"

"等价"是 FST 最小化的核心定义。两个状态 s₁ 和 s₂ 等价当且仅当它们满足:

  1. finality 相同:要么都是 final 要么都不是;若都是 final 则它们的 final output 相同
  2. arc 集合相同:s₁ 和 s₂ 的所有出 arc 必须一一对应,即对每条 (label, target, output) 三元组都相等
  3. target 等价:对应 arc 的 target 状态本身也必须等价(递归定义)

这个定义看起来很抽象,但落地到 Lucene 的实现就是 NodeHash.fnv1a:把一个 node 的 (finality + arc 集合 + 每个 arc 的 target hashCode) 拼起来算 hash,用一个大 hash table 去重。这就是 hashConsing 的本质——把图最小化降级为 hash 查找,从 O(N²) 后处理变成 O(N) 增量。

为什么 term 必须按字典序输入

FST Builder 强制要求 term 按字典序输入 — 你必须在 build 之前就把 term 排序,否则 FSTCompiler.add(input, output) 会抛 IllegalArgumentException("inputs are added out of order")。原因是 FST 的增量最小化(见 §10.5)只能"冻结"已经处理完的子树——一旦 add 了 "charing",Builder 就认为以 "char" 开头的所有 term 已经全部加完,可以把这部分子树最小化并丢掉中间状态。如果之后再 add "cad",Builder 已经丢了 "ca" 的中间表示,无法插入。这是 streaming build 的代价:你只能在内存里维持一条 "frontier"(当前未冻结的路径),其他全部最小化。这个限制对 Lucene 不是问题,因为 .tim 文件本身就按 term 字典序写入,FST 是写入时同步构造的。

10.5 增量构造算法:为什么不能用纯递归

FST 最小化的"教科书算法"是 Hopcroft 1971 年的 partition refinement,O(N log N)。这个算法对 1 千万 term 来说看起来很高效,但它有一个致命问题:必须先把整个 trie 建好,然后再做最小化——这意味着内存里要先存 1 千万个 trie 节点,内存峰值 1-2 GiB。Lucene 不能接受这个开销,所以采用了 Robert Muth 1995 年提出的 增量最小化(incremental minimization)算法:在 streaming 加 term 的同时就做最小化,内存峰值只与当前未冻结的 frontier 路径长度成正比(几 KiB 而已)。

增量构造的核心思想:frontier + freeze

hashConsing 的实现:NodeHash

final class NodeHash<T> {
    private long[] table;
    private int mask;
    private final FST<T> fst;

    public long add(FSTCompiler.UnCompiledNode<T> nodeIn) throws IOException {
        long hash = hash(nodeIn);
        int pos = (int)(hash & mask);

        while (true) {
            long node = table[pos];
            if (node == 0) {
                long address = fst.addNode(this, nodeIn);
                table[pos] = address;
                return address;
            }
            if (nodesEqual(nodeIn, node)) {
                return node;
            }
            pos = (pos + 1) & mask;
        }
    }

    private long hash(FSTCompiler.UnCompiledNode<T> node) {
        long h = 0;
        for (int arcIdx = 0; arcIdx < node.numArcs; arcIdx++) {
            FSTCompiler.Arc<T> arc = node.arcs[arcIdx];
            h = 31 * h + arc.label;
            h = 31 * h + arc.target.hashCode();
            if (arc.output != null) h = 31 * h + arc.output.hashCode();
        }
        return h ^ (h >>> 32);
    }
}

10.7 BlockTree 与 .tip 协同

FST 存在内存里(.tip 文件 mmap),所以它必须很小——一千万 term 也只能允许几十 MiB 内存。但完整 term + postings 指针的元数据(docFreq、totalTermFreq、文件偏移)总共要几百 MiB 到几个 GiB,这部分必须留在磁盘(.tim 文件)。BlockTree 的设计就是把"内存索引层"和"磁盘数据层"切开:.tip 的 FST 只负责"找到 block 入口的 file pointer",.tim 的 block 负责"扫描 25-48 个 term 找到具体那一个的 metadata"。

10.8 seek 算法:findTargetArc 的三种路径

TermsEnum.seekExact("charger") 在底层就是 FST 走 7 步(c-h-a-r-g-e-r),每一步调用 findTargetArc(label, fromArc, toArc) 在当前节点的 arc 数组里找 label 匹配的那一条。这一节把 findTargetArc 的实现拆开——它根据节点存储模式分别调用 linear / binary / direct 三种路径。

FST.findTargetArc 的核心逻辑

public static <T> Arc<T> findTargetArc(int labelToMatch,
                                       Arc<T> follow,
                                       Arc<T> arc,
                                       BytesReader in,
                                       FST<T> fst) throws IOException {

    if (follow.isFinal() && labelToMatch == END_LABEL) {
        return arc.copyFrom(follow);
    }

    if (!targetHasArcs(follow)) {
        return null;
    }

    if (follow.bytesPerArc() != 0) {
        if (follow.nodeFlags() == ARCS_FOR_DIRECT_ADDRESSING) {
            return fst.findArcDirectAddressing(labelToMatch, follow, arc, in);
        }
        return fst.findArcBinarySearch(labelToMatch, follow, arc, in);
    }

    return fst.findArcLinearScan(labelToMatch, follow, arc, in);
}

seekCeil 的细节:不命中时如何"近似"

除了 seekExact,TermsEnum 还有 seekCeil(target):找到第一个 ≥ target 的 term。这个语义在 range query 和 prefix query 里都要用。FST 的实现是:沿 target 的字符走,如果走到某一步 label 不存在,找当前节点的下一个更大 label 的 arc,然后沿那条 arc 走到底(走最小路径)。

public SeekStatus seekCeil(BytesRef target) throws IOException {
    Arc<T> arc = fst.getFirstArc(scratchArc);
    BytesReader fstReader = fst.getBytesReader();

    for (int i = 0; i < target.length; i++) {
        int label = target.bytes[target.offset + i] & 0xFF;
        Arc<T> next = fst.findTargetArc(label, arc, scratch, fstReader);

        if (next == null) {
            Arc<T> ceil = fst.findTargetArcCeil(label, arc, scratch, fstReader);
            if (ceil == null) {
                return SeekStatus.END;
            }
            walkSmallestPath(ceil);
            return SeekStatus.NOT_FOUND;
        }

        arc = next;
    }

    return arc.isFinal() ? SeekStatus.FOUND : findNextFinal(arc);
}

10.9 复杂度推导:为什么是 O(log N + B) 而不是 O(log N)

很多教材把 BlockTree 的 seek 复杂度写作 "O(log N)",这其实是简化的——精确表达是 O(L + B) 或更严格的 O(L · log_d Σ + B),L 是 term 长度,B 是 block 内 term 数,d 是平均 fan-out,Σ 是字母表大小。这一节把这个公式逐项推导,让你能在面试或排错时一句话回答"为什么 FST 不是 O(log N)"。

为什么 L 与 log N 大致等价

从信息论角度,要区分 N 个 term,至少需要 log_Σ N 个字符(Σ 是字母表大小)。对英文 ASCII Σ=128,N=10⁷:log₁₂₈ 10⁷ ≈ 3.3——意味着平均长度只要 3.3 个字符就能区分一千万 term。实际中 term 平均长度 8-15 字符,远高于这个最低要求,所以 L 在实操中确实是 O(log N) 的同阶量,但是用 base = 字母表大小,所以 L 上限受字母表 entropy 决定。

BlockTree 阶段的复杂度

FST walk 走完后,落到 .tim 的 block 入口,然后 block 内顺序扫描。这部分是 O(B) 不可避免——除非把每个 term 都做成单独的 FST 节点(那 FST 就退化成 trie,内存爆炸)。Lucene 的 trade-off 是 B = 25-48,远小于 N,所以 O(B) 是常数级开销。

构造时间是 FST 唯一劣势

FST 构造比 HashMap 慢 8 倍——这是 hashConsing + 字节序列化的成本。但这个代价 Lucene 通过两个机制摊销:(1) FST build 是 segment flush 时一次性完成,后续 segment 不可变,FST 不会重建;(2) 段合并时 FST 是"顺序合并"(两个已排序 term 流的 zip merge),不是 rehash,代价 O(N) 而不是 O(N log N)。生产中 FST 构造时间占 indexing 总耗时 < 5%,完全可以接受。

Tantivy(Rust)和 Lucene 的 FST 之争 — Tantivy 是 Rust 写的搜索引擎,它的 FST 实现(基于 fst crate)与 Lucene 不同:它用了 byte-level direct addressing 更激进,在英文短词上比 Lucene FST 快 ~30%,但中文 UTF-8 多 byte 上慢 ~20%。Lucene 的"linear/binary/direct 三档自适应"在异构数据上更稳——这是 Lucene 选择多档存储模式的工程动机。两者的 seekExact 在 1 千万 term 上都是亚微秒级,但 Lucene 的最坏情况比 Tantivy 好,这是大型搜索集群更看重的指标。

10.12 FST 在自动补全的应用

FST 不只在词典索引上发光——它还是 Lucene Suggester / Completion 模块的底层引擎。Suggester 的需求是"用户输入前缀 'iph',返回 Top 10 含该前缀的候选 + 它们的权重"。这本质上是"前缀 → Top K 候选" 的映射,而 FST 用 PairOutputs 可以同时存 (term, weight),沿前缀走完之后用堆遍历最高权重的 K 条路径。

List<Result> lookup(BytesRef prefix, int topK) throws IOException {
    Arc<Pair<Long, BytesRef>> arc = fst.getFirstArc(scratchArc);
    BytesReader fstReader = fst.getBytesReader();
    Pair<Long, BytesRef> output = outputs.getNoOutput();

    for (int i = 0; i < prefix.length; i++) {
        int label = prefix.bytes[prefix.offset + i] & 0xFF;
        Arc<Pair<Long, BytesRef>> next = fst.findTargetArc(label, arc, scratchArc, fstReader);
        if (next == null) return Collections.emptyList();
        output = outputs.add(output, next.output());
        arc = next;
    }

    PriorityQueue<Result> topResults = new PriorityQueue<>(topK,
        Comparator.comparingLong(Result::weight).reversed());
    Util.TopNSearcher<Pair<Long, BytesRef>> searcher =
        new Util.TopNSearcher<>(fst, topK, topK + 1, weightComparator);
    searcher.addStartPaths(arc, output, true, prefix);
    Util.TopResults<Pair<Long, BytesRef>> results = searcher.search();

    return results.topN.stream()
        .map(r -> new Result(termFromIntsRef(r.input), r.output.output1))
        .toList();
}

10.13 范围扫描:TermsEnum.intersect / next 的 FST 路径

FST 的"按字典序遍历"能力让范围扫描和自动机求交都成为亚线性操作。这一节把 TermsEnum.next()(顺序往后)、TermsEnum.intersect(automaton)(自动机求交)的 FST 路径都拆开,展示为什么 WildcardQuery 不需要扫整个词典。

next() 的实现:沿当前 frontier 找下一个 final state

图 10-13-1 · TermsEnum.next() 在 FST 上的步进逻辑

这个遍历是深度优先 + 字典序:先走完当前子树最深处再向上回溯,回溯时找下一个 sibling。每次 next() 调用最多推进 O(L) 字符,平均 O(1)——因为大多数情况只在 leaf 附近做兄弟切换,不需要走全 L。1 千万 term 的全表遍历 next() 总代价 O(N · 1) = O(N),与 hash 表全表扫描同阶,但 FST 的 byte 顺序访问 cache 友好,实测快 ~3×。

这是 FST 最优雅的应用之一。WildcardQuery / RegexpQuery / FuzzyQuery 在 rewrite 时会构造一个 Automaton,然后调用 terms.intersect(automaton, startTerm)。底层做的是"两个自动机同步走":只展开"FST 和 Automaton 都接受"的状态序列,跳过不可能匹配的整个分支。

10.16 本章小结

FST 是 Lucene 在内存里压住"千万级 term 词典"的核心武器。它的本质是带输出的最小化 DFA——通过共享前缀 和 共享后缀,把 1 千万 term 的索引压到 35 MiB,可以 mmap 直接进 page cache,zero GC。配合 BlockTree 的 .tip + .tim 双层结构,seekExact 在 O(L+B) 内完成,而 range / prefix / wildcard / fuzzy 这些"非点查"模式得益于 FST 的字典序自动机性质,代价与命中量线性,与 N 无关。
