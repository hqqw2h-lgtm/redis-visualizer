第 26 章 · .pos 词项位置文件

postings 位置通道 · IndexOptions 触发 · 128-position PFOR-delta block · VInt tail · payload/offset 内联标记 · 短语/邻近/高亮专用

.pos 是 Lucene postings list 的位置数据通道。它保存每个词项 (term) 在文档内的所有出现位置 —— 位置以 token 序号表示,从 0 开始,按出现顺序单调递增,允许同位 token (synonym filter 输出 delta=0)。每文档的位置序列经 delta 编码后按 128 个一组打包成 PFOR-delta block,block 之间用 .doc 中的 SkipList 提供跳转锚点,尾部不足 128 的位置走 VInt tail。

.pos 仅在字段配置 IndexOptions ≥ DOCS_AND_FREQS_AND_POSITIONS 时生成。BM25 评分只读 .doc 即可;短语查询 (PhraseQuery)、跨距查询 (SpanQuery)、邻近查询 (proximity)、高亮 (highlight)、词项向量 (termVectors) 必须读 .pos。当字段同时启用 payloads 或 offsets 时,二者在 .pos 中以 1 位标记切换:存在 payload/offset 则跟随写出 payloadLength / offsetDelta,否则该位为 0。本章按"概念 → 字节布局 → 编码 → 写入 → 读取 → 性能 → 陷阱"组织,字节级 ASCII 网格图给出 header / per-term blocks / VInt tail / footer 的精确偏移与字段宽度。

  • IndexOptions 5 个等级如何决定 .pos 是否生成,以及 POSITIONS bit 的判定位置
  • .pos 与 .tim / .doc / .pay 4 文件的指针拓扑:posStartFP / lastPosBlockOffset / singletonDocID 三大入口
  • Header / Per-Term blocks / VInt Tail / Footer 4 段字节布局,前 64 byte 字节网格
  • Position Block 内部:1 byte bitsPerValue + packed deltas + payloadLengths + payloadBytes + offsets
  • PFOR-delta 在 .pos 与 .doc 上的 6 维差异,关键在允许 delta=0(同位 token)
  • 跨文档 position 累加器归零机制 —— .pos 没有 doc 分隔标记,完全依赖 .doc 的 freq 切分
  • VInt tail 中 code = (posDelta << 1) | hasPayload 的位拼装,以及 block 与 tail 在 payload 编码上的差异
  • Lucene90PostingsWriter.writePosBlock() 写入路径:max → bits → forUtil.encode → payload → offsets
  • Lucene90PostingsReader.refillPositions() 读取路径与 nextPosition() 状态机
  • 性能基线:热 block 解码 80-120 ns,冷 block 30-50 μs,短语查询 100 doc 命中 20-50 μs
  • 9 大陷阱:IndexOptions 漏开 / payload bit 误读 / 同位 token 误判 / 高基数 payload 体积爆炸

前置知识:第 25 章 .doc postings 文件(理解 docID + freq 流如何驱动 .pos 的解码切分);第 22 章 FieldInfos 与 IndexOptions 枚举(知道 5 个等级与 bit 序);第 23 章 .tim 词典(了解 term metadata 中 posStartFP 的来源);第 11 章 PFOR-delta 与 SkipList(本章 block 编码与跳跃机制的底层依赖)。读者需要熟悉 Lucene 的 codec 抽象与 CodecUtil 的 magic / footer 协议。

位置数据是短语查询的"坐标系":倒排索引告诉你"哪些 doc 含 term X",.pos 告诉你"X 在每个 doc 的第几个 token 位置"。短语查询本质是多 term 位置的几何对齐—— "quick brown" 命中 doc 7 当且仅当 quick 与 brown 至少存在一对位置 (p_q, p_b) 使 p_b - p_q == 1。没有 .pos,这个判定只能在原文重新分词,代价是 .pos 解码的 1000 倍以上。

26.1 文件用途与生命周期

本节回答 3 个问题:(1) .pos 何时生成,何时不生成?(2) 它在 segment 写入 / merge / 删除整个生命周期里如何流动?(3) 哪些查询能力依赖它?搞清这 3 点,才能在字段建模阶段决定要不要付出 30-50% 的索引体积来获得短语查询能力。

.pos 出现的必要条件是 IndexOptions 的位序中 POSITIONS bit 被置位。该判定在 Lucene90PostingsWriter.startTerm() 中执行,影响下游所有 block 的字段宽度。同字段在不同 doc 间的 IndexOptions 必须一致 —— 否则 CheckIndex 会发出警告,且短语查询会出现"部分 doc 命中、部分 doc 漏命中"的诡异现象。

定位 .pos 的常用做法是 Directory.listAll() 后过滤 .endsWith(".pos")。CompoundFile 模式下,实际数据被打包进 .cfs,需通过 CompoundFileDirectory 解出。下面是一段排查脚本:

# 列出非 compound segment 的 .pos
ls -lh /path/to/index/_*_Lucene90_0.pos

# compound 模式下 .pos 嵌入 .cfs,需用 lucene cli 拆出
java -cp lucene-core.jar org.apache.lucene.util.CommandLineUtil \
    cfs unpack /path/to/index/_0.cfs /tmp/unpacked/
ls -lh /tmp/unpacked/_0_Lucene90_0.pos
不要直接按字节偏移读 .pos:.pos 没有"按 docID 找位置"的全局索引,所有入口都是通过 .tim 中该 term 的 posStartFP 间接跳转。脱离 .tim 直接 seek(0x1234) 拿到的字节是无意义的(可能落在某个 packed block 中间)。读 .pos 必须从 .tip → .tim → .pos 三步走,或者直接用 PostingsEnum API。

26.3 与相邻文件的指针关系

.pos 不是孤立的文件,它通过 3 个关键指针 posStartFP / lastPosBlockOffset / singletonDocID 与 .tim 协同,通过 freq 与 .doc 协同,通过 hasPayload bit 与 .pay 协同。本节讲清这张拓扑图。

26.3.1 跨文件指针拓扑

26.3.2 .pos 入口指针表

表 26-3-1 · .pos 入口指针
字段来源类型含义
posStartFP.tim term metaVLong该 term 在 .pos 中的起始字节偏移
lastPosBlockOffset.tim term metaVLong最后一个完整 128-position block 的偏移

读取 .pos 不直接通过文件名定位,而是通过 .tim 中该 term 的 posStartFP 跳转。lastPosBlockOffset 用于在 advance 时快速识别"还有多少完整 block",剩余位置走 VInt tail 路径。singletonDocID 是一个重要优化:当某个 term 只在 1 个 doc 出现时,.tim 直接把那个 docID 内联,根本不需要 .doc 文件 —— 但 position 仍然要写到 .pos。

26.3.3 短语查询的跨文件路径

图 26-3-2 · 典型短语查询的跨文件路径

短语查询的 .pos 访问是"按 docID 步进的双指针扫描"。每文档内 .pos 解码 N 个位置 (N = freq),双方差值 == slop+1 即命中。这是 .pos 在线上的最高频访问模式 —— 几乎所有有 .pos 文件的索引,90% 的 .pos 字节读量都来自这条路径。

26.5 Per-Term Position Block 布局

per-term position block 是 .pos 的核心。每个 term 在 .pos 中占一段连续字节,起点由 .tim 的 posStartFP 指定,长度由 totalTermFreq 决定 —— 前 ⌊ttf/128⌋ 个 block 走 packed PFOR-delta,剩余 ttf%128 个位置走 VInt tail。

  │   payloadBytes         : raw bytes      │
  │                                          │

  │   offsetLength         : packed         │
  │                                          │
  ├────────────────────────────────────────┤
  │ Position Block 1                         │
  │   ...                                    │

26.5.4 VInt Tail 编码

图 26-5-3 · VInt Tail 编码
VInt Tail 编码(尾部不足 128 位置):

  假设 freq = 200,128 位置入 block 0,余 72 位置入 tail。

  per tail position:
    bit 0      : hasPayload (1 bit)
    bit 1+     : posDelta (剩余 bit)
    合并为 code = (posDelta << 1) | hasPayload
    写入:VInt(code)

    若 hasPayload:
      VInt(payloadLen) + raw bytes
    若启用 offsets:
      VInt(offsetStartDelta) + VInt(offsetLength)

  示例字节(2 个尾部位置,无 payload 与 offset):

  偏移   字节   含义
  ──────────────────────────────────────
  0x200  0x06   code = (3 << 1) | 0,posDelta = 3,无 payload
  0x201  0x14   code = (10 << 1) | 0,posDelta = 10,无 payload

  解码:
    pos[0] = lastPos + 3
    pos[1] = lastPos + 10
Block 与 Tail 在 payload 编码方式上不同:Block 模式下 payloadLength 与 payloadBytes 分两段独立打包(payloadLengths packed bits + payloadBytes 拼接),而 VInt tail 模式下每个 position 紧跟一个 VInt code(posDelta << 1 | hasPayload),hasPayload=1 时立刻跟 VInt(payloadLen) + bytes。reader 必须根据"是否在最后一个 packed block 之后"切换两种解析策略。这是 .pos 解析中最易写错的地方,也是为什么生产中要严格依赖 Lucene90PostingsReader 而不是手写解析。

26.6 Footer 与校验机制

Footer 是 Lucene 所有索引文件的统一收尾结构 —— 16 byte,包含 footer magic、algorithm ID、CRC32 校验值。本节展示 .pos 的 footer 字节布局与 CodecUtil.checkFooter 的工作流程。

26.6.1 Footer 字节布局

图 26-6-1 · Footer 字节布局
.pos 文件 Footer(末 16 字节):

  ┌──────────────────────────────────────┐
  │ magic        : 4 byte (0xC02893E8)    │
  │ algorithmID  : 4 byte (0 = CRC32C)    │
  │ checksum     : 8 byte (CRC32 LE)      │
  └──────────────────────────────────────┘

Footer 字节网格(以文件末尾 16 byte 为例):

偏移   ...   N-15 N-14 N-13 N-12 N-11 N-10 N-9  N-8  N-7  N-6  N-5  N-4  N-3  N-2  N-1
      ... |  C0 | 28 | 93 | E8 | 00 | 00 | 00 | 00 | A1 | F2 | 7C | 39 | 4B | 88 | 11 | 02 |
          |   FOOTER_MAGIC    | algorithmID = 0 | CRC32 over [0..N-16] little-endian      |

CRC32 计算范围:
  从文件起始 (offset 0) 到 algorithmID 末尾 (offset N-9),共 N - 8 字节
  即 magic + algorithmID 也参与校验

读取时校验:
  computed = CRC32C(file[0..N-9])
  stored   = readLong(file[N-8..N-1])
  if (computed != stored) throw CorruptIndexException

26.7 Position Delta 编码细节

本节展开 position delta 在 .pos 中的编码细节:为什么用 delta?为什么 delta 允许 0?如何与 .doc 的 docID delta 区分?跨 doc 时累加器如何归零?这些是 .pos 编码的核心机制。

26.7.1 position delta 累加链

图 26-7-1 · position delta 的累加链
原始 position 序列(同一 doc 内的 term 出现位置):
  [3, 17, 18, 25, 25, 31, 41, 53, ...]

注意:Lucene 允许同位置(position=25 出现两次,如 synonym filter)

转 delta:
  [3, 14, 1, 7, 0, 6, 10, 12, ...]
  ↑   ↑   ↑   ↑   ↑   ↑    ↑    ↑
  起点 17-3 18-17 25-18 同位 31-25 41-31 53-41

所有 delta ≥ 0,允许 0(同位 token)

存储:
  block 0 = 前 128 个 delta,PFOR-delta packed
  block 1 = 第 129-256 个 delta
  ...
  tail   = 剩余 < 128

解码:
  curPos = 0
  for each delta:
    curPos += delta
    emit curPos

26.7.4 跨文档 position 重置

图 26-7-3 · 跨文档的 position 重置
两个文档的 position 序列:

  doc 100: term "java" positions [2, 5, 11]
  doc 200: term "java" positions [0, 7, 9, 23]

.pos 中(连续存放):
  [2, 3, 6,    0, 7, 2, 14]
   └ doc 100 ┘  └─── doc 200 ───┘

  doc 边界由 .doc 中的 freq 提供:
    doc 100 freq=3 → 读 3 个 position
    doc 200 freq=4 → 读 4 个 position

  累加器在每文档开始时归零:
    decoder.position = 0
    for each freq:
      pos += readVInt() >> 1   (剥离 hasPayload bit)

.pos 没有任何 doc 分隔标记,完全依赖 .doc 提供的 freq 切分
delta=0 是 .pos 与 .doc 最本质的区别。.doc 的 docID 严格单调递增 → 可以用 delta - 1 节省 1 bit;.pos 必须支持同位 token(synonym filter / cjk filter / shingles 都会产生),所以 delta ≥ 0,不能预减 1。这也意味着不能在 .pos 解码时把 delta=0 当作终止符—— 早期(Lucene 4.x 之前)某些第三方 PostingsFormat 实现犯过这个错,导致同位 token 后续 position 全部丢失。

26.8 Payload 与 Offset 内联编码

Payload 是 token 维度的自定义字节(典型用途:为每个 token 附加权重、词性、向量等);Offset 是 token 在原文中的字符起止位置(用于高亮)。两者都可以选择性启用,启用方式由 FieldType.setStorePayloads(true) 与 IndexOptions.DOCS_AND_FREQS_AND_POSITIONS_AND_OFFSETS 控制。本节讲解它们在 .pos 中的内联编码。

26.8.1 hasPayload 标记位

图 26-8-1 · hasPayload 标记位
VInt tail 中的 position 编码(单个位置):

  bit 0   : hasPayload (1 = 跟着 payload 字节)
  bit 1+  : posDelta (剩余位)

  VInt code = (posDelta << 1) | hasPayload

  示例:
    posDelta=5, hasPayload=true   → code = (5<<1)|1 = 11 = 0x0B
    posDelta=5, hasPayload=false  → code = (5<<1)|0 = 10 = 0x0A
    posDelta=200, hasPayload=true → code = 401 = 0x91 0x03 (VInt 2 byte)

  紧跟编码顺序:
    VInt(code)
    if (hasPayload):
      VInt(payloadLen)
      raw payload bytes (length payloadLen)
    if (storeOffsets):
      VInt(offsetStartDelta)
      VInt(offsetLength)

26.8.2 不同启用组合的位置编码

表 26-8-1 · 不同启用组合的位置编码

NNVInt(posDelta)
YNVInt(code) + [VInt(plen) + bytes]
NYVInt(posDelta) + VInt(offsetDelta) + VInt(offsetLen)
YYVInt(code) + [VInt(plen) + bytes] + VInt(off) + VInt(len)

26.8.3 Block 模式下的 payload 处理

图 26-8-2 · 128-position block 中的 payload 处理
Block 模式下,payload 长度与字节流分离存放:

  Position Block 0 (128 positions):
  ┌────────────────────────────────────────┐
  │ posDeltaBits  : 1 byte                  │
  │ posDeltas     : packed bits             │
  ├────────────────────────────────────────┤
  │ payloadLengths: packed (1 byte/位置)    │
  │   每位 length(0-255)                     │
  ├────────────────────────────────────────┤
  │ payloadBytes  : raw 拼接               │
  │   total = sum(payloadLengths)           │
  └────────────────────────────────────────┘

读取顺序:
  1. 解 packed posDeltas → 128 个 delta
  2. 解 packed payloadLengths → 128 个 length
  3. 顺序消费 payloadBytes(按 length 切片)

Offset 同理,offsetStartDelta 与 offsetLength 各占一段 packed 区

Lucene90PostingsWriter 在每个 block 结束时分别 flush posDeltas / payloadLengths / payloadBytes / offsetDeltas / offsetLengths,字节布局上按此顺序追加。Block 模式与 VInt tail 模式的差别在于:block 把同类字段集中打包(更利于 SIMD bitpacking),tail 把每个 position 的所有字段紧密相邻(更利于稀疏访问)。两种模式各取所长。

BERT 向量 payload 的体积陷阱

某搜索团队尝试把 BERT token 向量(每 token 768 × float16 = 1536 byte)作为 payload 写入 .pos,期望在召回阶段同时拿到向量做 rerank。结果:1000 万 doc 的 .pos 文件膨胀到 2.3 TB,page cache 完全打不进去,P99 召回延迟从 30 ms 飙到 8 秒。正确做法:把向量改用 KnnVectorField 走 HNSW 索引(第 16 章),payload 仅存 8 byte 量化指针;.pos 体积回落到 80 GB,延迟回到 35 ms。生产规则:payload 平均长度应控制在 ≤ 8 byte,极限 ≤ 32 byte,超过就改用 BinaryDocValues 或 KnnVectorField。

private void writePosBlock() throws IOException {
    int max = 0;
    for (int i = 0; i < BLOCK_SIZE; i++) {
        max |= (int) posDeltaBuffer[i];
    }
    int bitsPerValue = max == 0 ? 0 : 32 - Integer.numberOfLeadingZeros(max);
    posOut.writeByte((byte) bitsPerValue);
    forUtil.encode(posDeltaBuffer, bitsPerValue, posOut);

    if (writePayloads) {
        for (int i = 0; i < BLOCK_SIZE; i++) {
            payloadLengthBuffer[i] = (long) payloadLengths[i];
        }
        forUtil.encode(payloadLengthBuffer, payloadBitsPerValue, posOut);
        posOut.writeBytes(payloadBytes, 0, payloadByteUpto);
    }

    if (writeOffsets) {
        forUtil.encode(offsetStartDeltaBuffer, offBitsPerValue, posOut);
        forUtil.encode(offsetLengthBuffer, offLenBitsPerValue, posOut);
    }

    posBufferUpto = 0;
    payloadByteUpto = 0;
}

26.10.4 nextPosition 代码骨架

public int nextPosition() throws IOException {
    if (posPendingCount == 0) {
        return -1;
    }

    if (posBufferUpto == BLOCK_SIZE) {
        if (posPendingCount >= BLOCK_SIZE) {
            posIn.readByte();
            forUtil.decode(posIn, posDeltaBuffer);
            if (indexHasPayloads) {
                forUtil.decode(posIn, payloadLengthBuffer);
                payloadByteUpto = 0;
                int bytes = (int) sumOf(payloadLengthBuffer);
                posIn.readBytes(payloadBytes, 0, bytes);
            }
            if (indexHasOffsets) {
                forUtil.decode(posIn, offsetStartDeltaBuffer);
                forUtil.decode(posIn, offsetLengthBuffer);
            }
        } else {
            refillPositionsTail();
        }
        posBufferUpto = 0;
    }

    position += (int) posDeltaBuffer[posBufferUpto++];
    posPendingCount--;
    return position;
}

✅ 正例: 明确不需要短语,降级 IndexOptions
