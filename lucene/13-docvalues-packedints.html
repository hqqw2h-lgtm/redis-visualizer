第 13 章 · DocValues 与 PackedInts 列存

5 种 DocValues 类型 · 6 种 NumericDV 编码 · PackedInts 紧凑位压缩 · DirectMonotonicReader · BLOCK_PFOR · sort/facet/function score 列存基础

排序、聚合、function score、collapse、join 等"按 doc 取值"场景需要快速访问字段值,这正是倒排索引(term → doc)不擅长的访问模式。Lucene 引入 DocValues 作为列存补充:.dvd 存数据,.dvm 存元数据,5 种类型(Numeric / Binary / Sorted / SortedSet / SortedNumeric)分别对应不同业务需求。底层依赖 PackedInts 实现紧凑位压缩,DirectMonotonicReader 实现 ordinal 加速。

DocValues 是 Lucene 处理"按 doc 取值"场景的标准方案。订单按 paidAt 排序、按 storeId 聚合 facet、按 amountCents 做 function score,都依赖 DocValues。本章拆解五种类型的物理结构、PackedInts 编码原理、Monotonic 表与磁盘/内存交互模式、写入与读取调用链、性能边界与生产陷阱。

  • DocValues 5 种类型语义:Numeric(单 long)/ Binary(单字节)/ Sorted(单字符串带 ord)/ SortedSet(多字符串)/ SortedNumeric(多 long)
  • NumericDV 6 种编码:CONST / TABLE_COMPRESSED / GCD_COMPRESSED / DELTA_COMPRESSED / MONOTONIC / BLOCK_PFOR 自动选择
  • PackedInts BitPacking 把 8 个 3-bit 值挤到 3 byte(压缩比 21×)
  • DirectMonotonicReader = base + slope·i + residual,1 ns 级随机访问
  • Direct Table 适合 unique < 256,bitsPerOrd 8 bit 即可
  • .dvd 文件 3 段:Header / 字段 Body 序列 / Footer;.dvm 是路标
  • SortedDV 三部分:per-doc ord / terms dictionary / terms address (MONOTONIC)
  • 1M doc NumericDV ≈ 1-3 MB,SortedSetDV 多值字段 10 MB
  • 读取 sort by 字段 P99 50-100 ms,facet 聚合 100-500 ms
  • 不同 segment 字段类型不一致 → merge 失败的常见陷阱

前置知识:第 1 章倒排索引(理解 term → doc 与 doc → value 的正反向);第 5 章订单搜索 sort/facet 业务场景;第 12 章 PackedInts 在 BKD 中的复用。本章不假设你看过 9.x 源码,会从 codec 接口讲起。

13.1.2 类型一旦定不可改

DocValues 字段类型在 segment 内不可变。如果同字段在不同 doc 用不同 DocValues 类型,Lucene 会抛 IllegalArgumentException。merge 时会校验所有 segment 的字段类型一致。

public Document buildOrderDoc(Order order) {
    Document doc = new Document();

    doc.add(new NumericDocValuesField("paidAt_dv", order.paidAt.toEpochMilli()));
    doc.add(new NumericDocValuesField("amountCents_dv", order.amountCents));

    doc.add(new SortedDocValuesField("status_dv", new BytesRef(order.status.name())));
    doc.add(new SortedDocValuesField("storeId_dv", new BytesRef(order.storeId)));

    for (String tag : order.tags) {
        doc.add(new SortedSetDocValuesField("tags_dv", new BytesRef(tag)));
    }

    for (long viewCount : order.viewCounts) {
        doc.add(new SortedNumericDocValuesField("view_counts_dv", viewCount));
    }

    doc.add(new NumericDocValuesField("version_dv", order.version));

    long uidHash = MurmurHash3.hash64(order.userId.getBytes(UTF_8));
    doc.add(new NumericDocValuesField("user_id_hash_dv", uidHash));

    return doc;
}
命名约定:DocValues 字段建议加 _dv 后缀以区分倒排字段,避免误用。同名字段同时建索引和 DV 时不会冲突,但混用容易混淆。status(倒排,filter/facet)+ status_dv(列存,sort/agg)是生产标配。

13.3.2 GCD_COMPRESSED 实例

很多时间戳字段实际只精确到秒(末尾三位 0),gcd = 1000,GCD 编码可节省约 10 bit/value。

图 13-3-2 · GCD_COMPRESSED 编码逐帧
原始数据 (timestamp_ms, 100 doc):
  values[0] = 1700000000000
  values[1] = 1700001000000
  values[2] = 1700001500000
  values[3] = 1700002000000
  ...

帧 1 (扫描求 gcd):
  gcd(values[0], values[1], ...) = 1000
  observation: 所有值都是 1000 的倍数

帧 2 (选编码):
  if gcd > 1: 选 GCD_COMPRESSED

帧 3 (编码):
  min = 1700000000000
  enc[i] = (values[i] - min) / gcd
    enc[0] = 0
    enc[1] = 1000
    enc[2] = 1500
    enc[3] = 2000

帧 4 (bit 计算):
  raw max - min = 1e8 (100M ms) → 27 bit
  enc max = 1e5 → 17 bit
  ★ 节省 10 bit / doc

帧 5 (解码):
  v = min + enc[doc] * gcd

13.4 PackedInts 编码原理

13.4.1 BitPacking 核心

PackedInts 是 Lucene 用于"已知值域定长打包"的核心工具。给定 bitsPerValue,把 long[] 紧凑写入 long[] 缓冲,平摊位移与按位运算。

图 13-4-1 · BitPacking 字节示意(8 个 3-bit 值)
输入:  [0, 1, 2, 3, 4, 5, 6, 7]
       (3 bit each)

打包到 byte 流:
  bit:    [000][001][010][011][100][101][110][111]
  连续:    000 001 010 011 100 101 110 111
  补齐:    24 bit = 3 byte
  byte 0: 0b00000101 = 0x05
  byte 1: 0b00111100 = 0x3C
  byte 2: 0b10101111 = 0xAF

总 3 byte 存 8 个 long (原本需 64 byte)
★ 压缩比 21×

13.5 Monotonic 编码

13.5.1 Monotonic Block 原理

Monotonic 编码针对"近似单调递增"序列(如时间戳、addresses),用 base + slope·i + residual 形式压缩。residual = value[i] - (base + slope·i),如果原始单调,residual 接近 0,bit 数极低。

图 13-5-1 · Monotonic Block 编码示意
单调递增 long 数组 (timestamp ms, 每秒 1 doc):
  values: [1700000000000, 1700000010000, 1700000022000, 1700000035000, 1700000048000, ...]

Step 1: 求 base + slope
  base  = values[0] = 1700000000000
  slope = (values[N-1] - values[0]) / (N-1) = 12500

Step 2: 计算 residual
  expected[i] = base + i * slope
    expected[0] = 1700000000000
    expected[1] = 1700000012500
    expected[2] = 1700000025000
    expected[3] = 1700000037500
    expected[4] = 1700000050000

  residual[i] = values[i] - expected[i]
    residual[0] = 0
    residual[1] = -2500
    residual[2] = -3000
    residual[3] = -2500
    residual[4] = -2000

Step 3: bit 计算
  原始 raw max = 1700000048000 → 41 bit
  residual max abs = 3000 → 12 bit (含符号)
  ★ 节省 29 bit / value

Step 4: 存储
  base    : 8 byte (long)
  slope   : 4 byte (float)
  bitsPerResidual : 1 byte
  residuals : packed (12 bit × N)

Step 5: 解码
  value[i] = base + i * slope + residual[i]
表 13-5-1 · Monotonic 与 BlockPacked 对比
编码适合场景压缩比解码代价
BlockPackedReader一般数值序列5-10×10-30 ns/值
MonotonicBlockReader近似单调50-100×10-30 ns/值
DirectMonotonicReader单调 + per-block50-100×5-10 ns/值
9.x 主力:Lucene 9.x 用 DirectMonotonicReader 替代旧 MonotonicBlockPackedReader,针对 DocValues 的 addresses 表(SortedSet 的 ord_addresses、Binary 的 length_addresses)优化,支持 O(1) 随机访问。这是 9.x 列存性能比 8.x 提升 2× 的关键。

13.6 Direct Table

Direct Table 把"unique 值集合"做成全局表,per-doc 只存 ord(到表的索引)。当 unique 值少(< 256)且重复率高时,Direct Table 比 BlockPacked 更优。

图 13-6-1 · TABLE_COMPRESSED 实例(orderStatus)
字段示例 (orderStatus, 100 doc):

unique values:
  [PENDING, PAID, SHIPPED, DELIVERED, CANCELLED, REFUNDED] (6 unique)

帧 1 (构建 table):
  table[0] = PENDING  (long encoding)
  table[1] = PAID
  table[2] = SHIPPED
  table[3] = DELIVERED
  table[4] = CANCELLED
  table[5] = REFUNDED

帧 2 (per-doc 存 ord):
  bitsPerOrd = ceil(log2(6)) = 3
  doc 0: ord=2 (SHIPPED)
  doc 1: ord=3 (DELIVERED)
  doc 2: ord=1 (PAID)
  ...

帧 3 (尺寸):
  table:  6 * 8 byte = 48 byte
  ords:   100 * 3 bit = 300 bit ≈ 38 byte
  合计:   86 byte

vs 不编码: 100 * 8 byte = 800 byte
★ 压缩比 9×

帧 4 (解码):
  ord = readBits(docID * 3, 3)
  value = table[ord]

13.9 SortedSet 与 SortedNumeric

SortedSetDocValues 每 doc 多个 ord,需要额外的 ord_addresses 表索引每 doc 的 ord 起止。

图 13-9-1 · SortedSet 字节布局(tags)
SortedSetDocValues 字段示例 (tags, 100 doc, 平均 3 tag/doc, 200 unique tag):

  ┌────────────────────────────────────┐
  │ docsWithField (DISI)                 │
  ├────────────────────────────────────┤
  │ ord_addresses                        │
  │   MonotonicReader                    │
  │   每 doc 起止 ord 索引                 │
  │   addr[0]=0, addr[1]=3, addr[2]=5,...│
  ├────────────────────────────────────┤
  │ ords[]                               │
  │   所有 doc 的 ord 拼接 + DELTA/BLOCK_PFOR│
  │   doc 0: [3, 5, 12]                  │
  │   doc 1: [7, 18]                     │
  │   doc 2: [2, 5]                      │
  │   ...                                │
  ├────────────────────────────────────┤
  │ terms dictionary                     │
  │   200 unique tag, 前缀压缩            │
  ├────────────────────────────────────┤
  │ terms address (MonotonicReader)      │
  └────────────────────────────────────┘

读取 doc=1 的所有 tag:
  start = readMonotonic(ord_addresses, 1) = 3
  end   = readMonotonic(ord_addresses, 2) = 5
  ords  = readOrds(start, end - start) = [7, 18]
  for ord in ords:
    termAddr = readMonotonic(termsAddr, ord)
    termEnd  = readMonotonic(termsAddr, ord + 1)
    tag = data.slice(termAddr, termEnd - termAddr)

SortedNumericDocValues 每 doc 多个 long,结构类似 SortedSet 但无 terms dictionary,直接存 PFOR 编码的 long。
