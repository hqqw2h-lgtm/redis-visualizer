第 20 章 · segments_N 提交点文件

commit point · SegmentInfos · 二阶段提交 · POSIX rename · IndexFileDeleter · SnapshotDeletionPolicy · IndexCommit 链表 · NRT

索引目录里几十上百个文件,Lucene 怎么知道"当前活着"的是哪些?答案藏在那个看起来不起眼的 segments_N 里。这个通常只有几 KiB 的小文件,却是整个索引的根节点——它列举了当前所有 segment 的名字、Codec、删除版本、文件清单,以及一个单调递增的 generation 号。读到 segments_42 的进程,看到的就是 commit 序号 42 时刻的索引快照;读到 segments_43 的进程,看到的是下一次 commit 之后的新视图——两个版本互不干扰,这就是 Lucene 的"伪 MVCC"。

把"当前状态"压缩成一个文件名带版本号的小文件,再靠 POSIX rename(2) 的原子语义保证发布,Lucene 用最朴素的文件系统语义实现了类似数据库 ACID 的提交协议——没有 redo log、没有 undo log、没有 LSN 协调。代价是:这个文件一旦损坏,整个索引就打不开;rename 不原子的文件系统(NFS / FAT32 / 对象存储)会让"半提交"成为可能,直接破坏 ACID。本章把 segments_N 完整拆开:从字节级的物理布局,到 prepareCommit + finishCommit 两阶段写入,到 IndexFileDeleter 的引用计数,到 SnapshotDeletionPolicy 的多版本保留,再到读路径上 FindSegmentsFile 如何从一堆 segments_* 中找到最新一份。每一个细节都是生产事故防线上的一个钉子。

  • segments_N 的物理结构:header(magic + objectID + 版本)+ generation + counter + 段列表 + userData + footer(校验和)
  • commit 的两阶段协议:prepareCommit 写 pending_segments_N + fsync,finishCommit 用 atomic rename 发布
  • N 的单调语义:每次 commit 严格 +1,生成 segment 编号的 counter 与 generation 一起持久化
  • IndexCommit 抽象:每个 segments_N 对应一个 IndexCommit,DeletionPolicy 决定保留几代
  • SnapshotDeletionPolicy:在线备份的核心机制,acquire/release 配对引用计数
  • 读路径定位:FindSegmentsFile 列目录 → 选最大 N → readCommit 解析字节流 → 校验 footer
  • NRT 不写 segments_N:in-memory SegmentInfos + reader pool,毫秒级可见但崩溃丢失
  • 反例:NFS 上 commit 半提交、误删 segments_N、高频 commit 拖垮吞吐、Snapshot 不释放磁盘膨胀

前置知识:第 3 章 Segment 不可变模型与 commit/flush 区别;第 6 章 IndexWriter 的代码路径(尤其 doFlush 与 commitInternal);第 19 章 write.lock 的排他写锁。理解 POSIX rename(2) 是同一文件系统下的原子操作、fsync(2) 与 fsync(directory) 的差异,会让本章的提交协议读起来豁然开朗。本章打下的提交模型是后续第 21-22 章理解 .si / .fnm 等单 segment 元数据文件的基础——segments_N 列举段、.si 描述段、两者一起构成索引完整元信息。

20.1 提交点的角色:索引的事务日志根

关系数据库用 redo log + WAL 来保证崩溃恢复,Lucene 没有这套机制——它把"当前已提交状态"压缩成一个单一文件 segments_N,文件名里的 N 就是"事务序号"。读 segments_42 看到的是 commit 序号 42 的快照,读 segments_43 看到的是下一次 commit 之后的新视图,两份元信息在磁盘上同时存在,直到 IndexFileDeleter 评估引用计数后才回收旧的。

segments_N 在索引目录中的位置

图 20-1 · 索引目录中 segments_N 的角色

这张图揭示一个关键事实:segments_N 不是 segment 文件本身。它只是一份"清单",列举出当前活跃的 _0 / _1 / _2 等 segment 各自包含哪些文件、删除版本是几代。删除一个 segments_N 不会删除 segment 主体文件——但读取 segments_N 失败,整个索引就打不开。

segments_N 是 Lucene 索引的 ACID 边界 — 三件事一起决定了 commit 的语义:(1) 原子性:rename 是 POSIX 原子操作,要么 segments_N+1 完整出现要么压根不存在,不可能写到一半;(2) 一致性:N 单调递增 + objectID(commit UUID)+ footer CRC32 三层校验,任何字节级损坏都会被 readCommit 检测;(3) 持久性:写完 pending 文件后必须 fsync(file),rename 后必须 fsync(directory),缺一不可。redo log 数据库做的所有事情,Lucene 用一个原子 rename + 一个文件名的版本号就替代了——前提是文件系统真的支持 atomic rename。

commit 与 flush 的语义对比

新手常把 flush 和 commit 当成同一回事——都是"把数据写出去"。在 Lucene 里它们是两个完全不同的动作:flush 只把内存里的 DWPT 落盘成 segment 主体文件,但不动 segments_N;commit 才会写新的 segments_N+1,让"已 flush 但未 commit 的 segment"正式进入读视图。

flush — 落盘但不可见

writer.addDocument(doc);
writer.flush();

// 发生什么:
// 1. RAM Buffer 中的 DWPT 写成 _N.si / .tim / .doc / ...
// 2. 文件落盘但 NOT fsync
// 3. segments_N 没有更新
// 4. NRT reader (DirectoryReader.open(writer))
//    能看到新 segment
// 5. 非 NRT reader (DirectoryReader.open(dir))
//    看不到新 segment
// 6. 进程崩溃 + 重启
//    → IndexWriter 启动忽略这些"裸 segment"
//    → 数据丢失

flush 只是"中间态",不能扛断电。生产代码不要把 flush 当作持久化保证。

commit — ACID 边界

writer.addDocument(doc);
writer.commit();

// 发生什么:
// 1. 触发隐式 flush (若有未落盘 doc)
// 2. 应用 pending 删除 (DeleteQueue)
// 3. 序列化 SegmentInfos 到字节流
// 4. 写 pending_segments_N+1
// 5. fsync(pending file)
// 6. rename → segments_N+1 (atomic)
// 7. fsync(directory)
// 8. IndexFileDeleter checkpoint
//    → 旧 segments_N 引用归零则删除
// 9. 进程崩溃 + 重启
//    → 读 segments_N+1 完整恢复

commit 是 Lucene 的"事务提交"——任意步骤崩溃,索引都能回到某个一致状态。

20.3 SegmentInfos 数据结构

SegmentInfos 是 segments_N 在内存中的对应数据结构。它本质是一个有序的 SegmentCommitInfo 列表 + 一组 commit 级别的元数据(generation / counter / userData / luceneVersion)。IndexWriter 在内存里维护一个"当前活跃"的 SegmentInfos,每次 flush 新 segment 就 add 进去,每次 merge 完成就替换其中几个旧 segment 为新 segment;commit 时把这份内存数据序列化成字节流写到 pending_segments 文件里。

SegmentInfos 类骨架

// org.apache.lucene.index.SegmentInfos (Lucene 9.x 简化版)
public final class SegmentInfos implements Cloneable, Iterable<SegmentCommitInfo> {
    public static final int VERSION_94 = 9;
    public static final int VERSION_CURRENT = VERSION_94;

    private long generation;          // segments_N 中的 N
    private long lastGeneration;      // 上一次 commit 的 N (用于 IndexFileDeleter)
    private int counter;              // 下一个 segment 编号
    private final int indexCreatedVersionMajor;  // 索引最初的 Lucene major 版本
    private Version luceneVersion;    // 当前 Lucene 版本
    private Version minSegmentLuceneVersion;     // 最旧 segment 的版本
    private byte[] id;                // 16 字节 commit UUID

    private final List<SegmentCommitInfo> segments = new ArrayList<>();
    private Map<String, String> userData = Collections.emptyMap();

    public void add(SegmentCommitInfo si) {
        segments.add(si);
    }

    public SegmentCommitInfo info(int i) {
        return segments.get(i);
    }

    public int size() {
        return segments.size();
    }

    public long getGeneration() {
        return generation;
    }

    public Map<String, String> getUserData() {
        return userData;
    }

    public void setUserData(Map<String, String> userData, boolean doIncrementVersion) {
        this.userData = userData == null ? Collections.emptyMap() : userData;
        if (doIncrementVersion) {
            changed();
        }
    }
}

SegmentCommitInfo:每个 segment 的元数据条目

SegmentInfos 列表里每个元素是一个 SegmentCommitInfo。它包装一个 SegmentInfo(指向 .si 文件,描述段的不变属性如 docCount / codec / files[])再加上"在当前 commit 时的可变状态"——删除版本、字段更新版本、DV 更新版本等。

// org.apache.lucene.index.SegmentCommitInfo (简化)
public class SegmentCommitInfo {
    public final SegmentInfo info;        // 不变部分,来自 .si 文件

    private long delGen;                  // LiveDocs 版本 (-1 = 无)
    private int delCount;                 // 已删除文档数

    private long fieldInfosGen;           // .fnm 版本
    private long docValuesGen;            // .dvm/.dvd 版本

    private int softDelCount;             // 软删除计数

    // 当前 commit 引用的 .fnm 文件集合
    private final Set<String> fieldInfosFiles = new HashSet<>();

    // 字段级 DV 更新文件:fieldNumber -> file set
    private final Map<Integer, Set<String>> dvUpdatesFiles = new HashMap<>();

    /** 当前 commit 引用的所有文件(给 IndexFileDeleter 引用计数用) */
    public Collection<String> files() throws IOException {
        Set<String> files = new HashSet<>(info.files());
        if (delGen != -1) {
            files.add(IndexFileNames.fileNameFromGeneration(
                info.name, IndexFileNames.LIVE_DOCS_EXTENSION, delGen));
        }
        files.addAll(fieldInfosFiles);
        for (Set<String> per : dvUpdatesFiles.values()) {
            files.addAll(per);
        }
        return files;
    }

    public boolean hasDeletions() {
        return delGen != -1;
    }

    public long getNextDelGen() {
        return delGen == -1 ? 1 : delGen + 1;
    }
}

20.4 commit 的二阶段:prepareCommit + finishCommit

Lucene 的 commit 在 API 层面是单一调用 writer.commit(),但内部分为两阶段:prepareCommit(准备阶段)把数据写到 pending 文件并 fsync,finishCommit(提交阶段)用原子 rename 把 pending 重命名为正式 segments_N。两阶段之间,如果有外部分布式协调器(如 ES 的两阶段提交),可以在 prepareCommit 之后、finishCommit 之前做最后决策——这是 Lucene 暴露 prepareCommit() 和 commit() 两个 API 的设计原因。

IndexWriter#commit 入口

// org.apache.lucene.index.IndexWriter (简化关键路径)
public final long commit() throws IOException {
    ensureOpen();
    return commitInternal(config.getMergePolicy());
}

private long commitInternal(MergePolicy mergePolicy) throws IOException {
    if (hasUncommittedChanges() == false && pendingCommit == null) {
        return -1;  // nothing to commit
    }
    // 阶段 1:prepareCommit (若没调用过)
    if (pendingCommit == null) {
        prepareCommitInternal(mergePolicy);
    }
    // 阶段 2:finishCommit
    return finishCommit();
}

public final long prepareCommit() throws IOException {
    ensureOpen();
    return prepareCommitInternal(config.getMergePolicy());
}

prepareCommit 实现

// 简化的 prepareCommitInternal
private void prepareCommitInternal(MergePolicy mergePolicy) throws IOException {
    synchronized (commitLock) {
        ensureOpen(false);

        // 1. flush 所有 RAM Buffer + 应用 pending 删除
        boolean anyChanges = docWriter.flushAllThreads();
        if (anyChanges) {
            applyAllDeletesAndUpdates();
        }

        SegmentInfos toCommit;
        synchronized (this) {
            // 2. clone 当前 SegmentInfos (避免 commit 期间被 merge 修改)
            toCommit = segmentInfos.clone();
            toCommit.setUserData(commitUserData, false);

            // 3. 增加 generation
            toCommit.changed();  // generation += 1
        }

        // 4. 写 pending_segments_N+1 + fsync
        try {
            directory.sync(toCommit.files(directory));  // fsync 所有 segment 文件
            toCommit.prepareCommit(directory);
            // ↑ 内部:createOutput("pending_segments_" + N+1)
            //   写 header + segments + footer
            //   close + fsync(pending file)

            pendingCommit = toCommit;
        } catch (Throwable t) {
            // 任何异常 → 清理 pending 文件
            deletePartialSegmentsFile(toCommit);
            throw t;
        }
    }
}

// 简化的 finishCommit
private long finishCommit() throws IOException {
    synchronized (commitLock) {
        if (pendingCommit == null) {
            return -1;
        }

        try {
            // 1. atomic rename: pending_segments_N -> segments_N
            pendingCommit.finishCommit(directory);
            // ↑ 内部:directory.rename(pending, finalName)
            //         directory.syncMetaData()  即 fsync(directory)

            synchronized (this) {
                // 2. 替换内存中的 SegmentInfos generation
                segmentInfos.updateGeneration(pendingCommit);

                // 3. IndexFileDeleter checkpoint:重算引用,删除孤儿文件
                deleter.checkpoint(pendingCommit, true);
            }
        } finally {
            notifyAll();
            pendingCommit = null;
        }

        return segmentInfos.getGeneration();
    }
}

两阶段提交的协调器场景

为什么 Lucene 暴露 prepareCommit() 和 commit() 两个 API?因为分布式协调器(如 ES 的 primary-replica 同步、跨索引事务)需要在"数据已落盘"和"对外可见"之间留一个决策窗口。典型用法:

// 多索引事务示例:跨两个 IndexWriter 的"伪事务"
IndexWriter writerA = ...;
IndexWriter writerB = ...;

try {
    // 阶段 1:都先 prepareCommit (数据落盘但未发布)
    writerA.prepareCommit();
    writerB.prepareCommit();

    // 阶段 2:协调器决策(向第三方注册中心确认)
    if (coordinator.shouldProceed()) {
        writerA.commit();   // finishCommit · 发布 segments_N+1
        writerB.commit();
    } else {
        writerA.rollback(); // 删除 pending_segments_N+1
        writerB.rollback();
    }
} catch (Exception e) {
    writerA.rollback();
    writerB.rollback();
    throw e;
}

注意这个"伪事务"不是真正的分布式事务——两次 finishCommit 之间仍可能崩溃,留下 A 已发布 / B 未发布的状态。要真正的 2PC,需要外部 transaction log + 重做协议。Lucene 提供的只是"在 commit 流程中切一刀"的机会。

20.5 atomic rename 与文件系统兼容

整个 commit 协议的 ACID 基础就一句话:同一文件系统下的 rename(2) 是原子操作。任意时刻其他进程读目录,要么看到旧名字 segments_42,要么看到新名字 segments_43,不会出现"名字一半写完"的中间态。一旦这个假设不成立(NFS、对象存储、FAT32),整个 commit 协议都失去 ACID 保证。

        new KeepOnlyLastCommitDeletionPolicy());

// - commit() 完成后,若该 commit 没被 snapshot,

SnapshotDeletionPolicy 完整用法

// 完整的"在线热备份"代码
SnapshotDeletionPolicy snapshotter = new SnapshotDeletionPolicy(
    new KeepOnlyLastCommitDeletionPolicy());

IndexWriterConfig config = new IndexWriterConfig(analyzer);
config.setIndexDeletionPolicy(snapshotter);

try (IndexWriter writer = new IndexWriter(dir, config)) {
    // 1. 业务正常写入
    for (Order o : orders) {
        writer.addDocument(toDoc(o));
    }
    writer.commit();  // segments_N 写入磁盘

    // 2. 申请快照(对当前 last commit 引用 +1)
    IndexCommit snapshot = snapshotter.snapshot();
    try {
        long generation = snapshot.getGeneration();
        log.info("Backup snapshot generation={} files={}",
                 generation, snapshot.getFileNames().size());

        // 3. 复制所有相关文件到备份目录
        Path backup = Paths.get("/backup/idx-gen-" + generation);
        Files.createDirectories(backup);

        for (String fileName : snapshot.getFileNames()) {
            Path src = ((FSDirectory) dir).getDirectory().resolve(fileName);
            Files.copy(src, backup.resolve(fileName),
                       StandardCopyOption.COPY_ATTRIBUTES);
        }

        // 4. 期间业务可继续写入(不阻塞)
        // 主线程或其他线程的 addDocument / commit 都正常
    } finally {
        // 5. 必须 release,否则被引用文件永不删除 → 磁盘膨胀
        snapshotter.release(snapshot);
    }
}
Elasticsearch / Solr 的 snapshot/restore API 底层就是 SnapshotDeletionPolicy — ES 的 POST /_snapshot/repo/snap1 实际上对每个 shard 都调用一次 SnapshotDeletionPolicy.snapshot,然后把 IndexCommit 引用的所有文件增量复制到 repository(S3 / HDFS / NFS / local FS)。增量是因为 segment 是不可变的:第一次快照复制全部 100 GB,第二次快照只需复制本次 commit 新增的 segment + 新的 segments_N(可能就几 GB)。restore 阶段反过来:从 repository 拉回所有引用文件,然后让新 IndexWriter 用这个目录启动。这套机制的核心就是 segments_N 提供的"commit 是文件清单"语义——只要有文件清单 + 不可变文件,增量备份就是天然支持的。Solr 的 ReplicationHandler / backup API 走的也是同一个底层。

// org.apache.lucene.index.SegmentInfos.FindSegmentsFile (简化)
public abstract class FindSegmentsFile<T> {
    private final Directory directory;

    public T run(IndexCommit commit) throws IOException {
        if (commit != null) {
            // 直接打开指定 commit
            return doBody(commit.getSegmentsFileName());
        }

        long lastGen = -1;
        long gen = -1;
        IOException exc = null;

        // 重试循环:最多 retry 次数次
        for (int attempt = 0; ; attempt++) {
            // 1. 列目录所有文件
            String[] files = directory.listAll();

            // 2. 找 generation 最大的 segments_*
            gen = getLastCommitGeneration(files);
            if (gen == -1) {
                throw new IndexNotFoundException(
                    "no segments_* file found in " + directory);
            }

            // 3. 尝试 readCommit
            String segmentFileName =
                IndexFileNames.fileNameFromGeneration(
                    IndexFileNames.SEGMENTS, "", gen);
            try {
                T result = doBody(segmentFileName);
                return result;
            } catch (IOException ioe) {
                exc = ioe;
                // 损坏 → 退到 N-1
                if (attempt >= MAX_RETRIES) {
                    throw exc;
                }
                // 等一下,可能是另一个进程正在 rename
                Thread.sleep(50);
            }
        }
    }

    public static long getLastCommitGeneration(String[] files) {
        long max = -1;
        for (String file : files) {
            if (file.startsWith(IndexFileNames.SEGMENTS) &&
                file.equals(IndexFileNames.OLD_SEGMENTS_GEN) == false &&
                file.startsWith("pending_") == false) {
                long gen = generationFromSegmentsFileName(file);
                if (gen > max) max = gen;
            }
        }
        return max;
    }

    protected abstract T doBody(String segmentFileName) throws IOException;
}

// org.apache.lucene.index.SegmentInfos#readCommit 简化
public static SegmentInfos readCommit(Directory directory, String segmentFileName)
        throws IOException {
    long generation = generationFromSegmentsFileName(segmentFileName);

    try (ChecksumIndexInput input = directory.openChecksumInput(segmentFileName)) {
        // 1. Codec header (magic + name + version)
        int format = CodecUtil.checkHeader(
            input, "segments", VERSION_CURRENT, VERSION_CURRENT);

        // 2. objectID 16B
        byte[] id = new byte[StringHelper.ID_LENGTH];
        input.readBytes(id, 0, id.length);

        // 3. Lucene version
        int luceneMajor = input.readVInt();
        int luceneMinor = input.readVInt();
        int luceneBugfix = input.readVInt();
        Version luceneVersion = Version.fromBits(luceneMajor, luceneMinor, luceneBugfix);

        int indexCreatedVersionMajor = input.readVInt();

        // 4. minVersion (optional)
        Version minVersion = null;
        if (input.readByte() != 0) {
            int minMajor = input.readVInt();
            int minMinor = input.readVInt();
            int minBugfix = input.readVInt();
            minVersion = Version.fromBits(minMajor, minMinor, minBugfix);
        }

        // 5. generation + counter
        long actualGeneration = input.readLong();
        if (actualGeneration != generation) {
            throw new CorruptIndexException(
                "filename N=" + generation + " but content N=" + actualGeneration,
                input);
        }
        long counter = input.readLong();

        // 6. numSegments + per-segment loop
        int numSegments = input.readInt();
        SegmentInfos sis = new SegmentInfos(indexCreatedVersionMajor);
        sis.id = id;
        sis.generation = generation;
        sis.luceneVersion = luceneVersion;
        sis.minSegmentLuceneVersion = minVersion;
        sis.counter = (int) counter;

        for (int i = 0; i < numSegments; i++) {
            String codecName = input.readString();
            Codec codec = Codec.forName(codecName);
            String segName = input.readString();
            byte[] segId = new byte[StringHelper.ID_LENGTH];
            input.readBytes(segId, 0, segId.length);
            SegmentInfo info = codec.segmentInfoFormat()
                .read(directory, segName, segId, IOContext.READ);

            long delGen = input.readLong();
            int delCount = input.readInt();
            long fieldInfosGen = input.readLong();
            long dvGen = input.readLong();
            int softDelCount = input.readInt();
            Set<String> fieldInfosFiles = input.readSetOfStrings();
            Map<Integer, Set<String>> dvUpdatesFiles = readDvUpdatesFiles(input);

            SegmentCommitInfo si = new SegmentCommitInfo(
                info, delCount, softDelCount, delGen, fieldInfosGen, dvGen,
                fieldInfosFiles, dvUpdatesFiles);
            sis.add(si);
        }

        // 7. userData
        sis.userData = input.readMapOfStrings();

        // 8. footer
        CodecUtil.checkFooter(input);

        return sis;
    }
}

恢复路径用 IndexCommit + DeletionPolicy 选择 N-1需 DeletionPolicy 保留多版本
readCommit 的"文件名 N == 内容 generation"校验是反误重命名防线 — 设想这种场景:运维误把 segments_42 复制成 segments_43 想"造一个新版本"。下次 IndexWriter 启动会读最大 N(43),解析时发现文件内 generation 字段是 42 → 立即抛 CorruptIndexException。这是 Lucene 设计中"防御性冗余"的典型——同一个值在文件名和内容里都存,任何一处被绕过都暴露。同样,objectID 也是这个目的:防止"两个不同时间产生的 _42 内容被混在一个目录"。如果你看到 CorruptIndexException: actual=X expected=Y 这类错误,99% 是有人手动改过文件名或者文件被截断。

20.9 NRT 与 segments_N

NRT(Near Real-Time)读取是 Lucene 的核心特性:用 SearcherManager 或 DirectoryReader.open(writer) 拿到的 reader,可以看到 IndexWriter 内存里"已 flush 但未 commit"的 segment——可见性达毫秒级。但 NRT reader 完全不写也不依赖 segments_N:它直接读 IndexWriter 内存里的 SegmentInfos。崩溃后这些"内存视图"全部丢失,只有 commit 过的数据才能恢复。

NRT 与 commit 的可见性对比

commit 模式 · 持久化但慢

// 生产代码常见反例:每条 commit
for (Doc d : docs) {
    writer.addDocument(d);
    writer.commit();  // 每条 fsync,吞吐崩塌
}

DirectoryReader reader = DirectoryReader.open(dir);

// 特征:
// - 可见性 = commit 频率(秒级)
// - 写吞吐 ≤ 1/fsync_latency(SSD ~200/s)
// - 崩溃丢失窗口 = commit 间隔
// - 写 segments_N+1 + fsync(file/dir)

把 commit 当 refresh 用,SSD 上吞吐降到 100-200 doc/s。

NRT 模式 · 毫秒级但易失

// 推荐:SearcherManager + 周期 commit
SearcherManager mgr = new SearcherManager(
    writer, true, true, new SearcherFactory());

// 高频 refresh(秒级)
scheduler.scheduleAtFixedRate(
    mgr::maybeRefresh, 1, 1, TimeUnit.SECONDS);

// 低频 commit(分钟级)
scheduler.scheduleAtFixedRate(
    () -> writer.commit(), 60, 60, TimeUnit.SECONDS);

// 特征:
// - refresh 不写 segments_N,不 fsync
// - 可见性 ~1s
// - 写吞吐 = 磁盘顺序写极限
// - 崩溃丢失 = 上次 commit 后的所有写入

refresh 与 commit 解耦,前者保证可见,后者保证持久。

NRT 的内部数据流

图 20-13 · NRT 不依赖 segments_N 的内存路径
NRT 不需要 segments_N 文件,但崩溃恢复仍依赖最新已 commit 的 segments_N — 进程重启时 IndexWriter 启动会读最大 N 的 segments_N,把它当作"上次崩溃前最后已知一致状态",然后清理目录里所有不在该 SegmentInfos 中的孤儿文件(包括所有 NRT-only 的裸 segment)。所以 NRT 的可见性"窗口"和 commit 频率"窗口"是两个独立旋钮:可见性 1s、持久性 60s 是常见配置——业务能秒级看到新数据,但断电会丢最近 60s 的写入。这个权衡在大多数搜索场景是可以接受的(被丢的数据通常可以从源系统重放)。
SearcherManager 的引用计数与 commit 解耦 — SearcherManager 内部维护一个 IndexSearcher 引用,maybeRefresh 时如果 IndexWriter 内存中的 SegmentInfos 已变(有新 flushed segment 或 NRT delete apply),就 reopen 一个新的 IndexSearcher,旧的引用计数降为 0 后异步关闭。整个过程不接触 segments_N 文件:不读、不写、不 fsync。这就是为什么 maybeRefresh 在 SSD 上能达到 1ms 级别,而 commit 至少 5-50ms。生产推荐组合:用户的"刚下单立刻能搜到"用 maybeRefresh(秒级)满足;"断电不丢数据"用 commit(分钟级)兜底;两者用同一个 IndexWriter 但绝不混用频率。

20.11 segments.gen 历史遗留

读早期(Lucene 4.x 之前)的索引代码或文档时,经常会看到 segments.gen 这个文件名。它在 Lucene 4.5 引入,4.10 之后逐步废弃,5.0 起完全移除。新版 Lucene 已经不写也不读这个文件,但理解它的历史能帮你看懂老代码注释和迁移文档。

segments.gen 与 segments_N 的对比

segments.gen · 已废弃

作用(Lucene 4.x 时代):
  - 文件名固定 segments.gen
  - 只存 generation 数值(8 bytes)
  - 用于在某些 NFS 实现下"快速"获取最新 N
  - 解决"列目录慢"的问题

实现:
  segments.gen 内容 = 当前 N 值
  读路径:
    1. 先读 segments.gen 拿 N
    2. 直接 open segments_N

问题:
  - segments.gen 与 segments_* 不原子一致
  - rename segments_N+1 完成后,segments.gen
    可能仍指向 N → 看到旧版本
  - NFS 上 segments.gen 反而成为不一致源
  - 增加了"两个文件需要同步"的复杂性

优化目标没达到,反而引入新的不一致风险。Lucene 4.10 标记 deprecated,5.0 移除。

segments_N · 当前唯一来源

现代 Lucene (5.0+) 实现:
  - 唯一的 commit 标识就是 segments_N 文件名
  - 读路径直接 directory.listAll() 找最大 N
  - 不再有 segments.gen 这个旁路文件

代码(SegmentInfos.FindSegmentsFile):
  files = directory.listAll();
  long maxGen = -1;
  for (String f : files) {
    if (f.startsWith("segments_") &&
        !f.startsWith("pending_")) {
      long gen = parseGen(f);
      if (gen > maxGen) maxGen = gen;
    }
  }

优势:
  - 单一来源 (single source of truth)
  - rename 是 atomic 的,不存在两文件不一致
  - listAll 在现代 FS 上 ≤ 1ms
  - 删除一份 segments.gen 的复杂代码

"少即是多"——用 atomic rename 替代两文件协同。这是 Lucene 设计哲学的体现。

20.12 反例与生产陷阱

把书本知识拉到生产现场,segments_N 涉及的反例几乎都集中在一个主题:对 commit 协议的语义理解错误。把 flush 当 commit、把 commit 当 refresh、把 NFS 当本地盘、把 segments_N 当临时文件——每一个都能让你的索引在凌晨 3 点出问题。

反例 1 · 高频 commit 拖垮吞吐

每条 doc commit 一次

// 反例代码
for (Doc d : docs) {
    writer.addDocument(d);
    writer.commit();  // 每条都 fsync
}

// 现象:
// - SSD 吞吐降到 100-200 doc/s
// - HDD 吞吐降到 10-50 doc/s
// - 远低于"裸写入"的 5万-50万 doc/s
// - 99% 时间花在 fsync,不在编码 doc

// 根因:
// - fsync 是同步阻塞,不能并行
// - 每次 fsync 至少 1 次磁盘 IO 往返
// - SSD 5ms / HDD 30ms 的硬下限

批量 + NRT refresh + 周期 commit

// 修复方案
SearcherManager mgr = new SearcherManager(
    writer, true, true, new SearcherFactory());

ScheduledExecutorService scheduler =
    Executors.newScheduledThreadPool(2);

// 高频 refresh(秒级,不写 segments_N)
scheduler.scheduleAtFixedRate(() -> {
    try { mgr.maybeRefresh(); }
    catch (IOException e) { log.warn("refresh", e); }
}, 1, 1, TimeUnit.SECONDS);

// 低频 commit(分钟级,写 segments_N)
scheduler.scheduleAtFixedRate(() -> {
    try { writer.commit(); }
    catch (IOException e) { log.warn("commit", e); }
}, 60, 60, TimeUnit.SECONDS);

// 业务侧只管批量写入
docs.parallelStream().forEach(d -> {
    try { writer.addDocument(d); } catch (IOException e) {}
});

反例 2 · 误删 segments_N 导致索引损毁

# 灾难现场
$ ls /var/lucene/idx/
segments_142  _0.si  _0.fnm  _0_Lucene99_0.tim  ...

# 运维误以为 segments_142 是临时文件
$ rm /var/lucene/idx/segments_142

# 重启应用
$ tail -f app.log
ERROR: org.apache.lucene.index.IndexNotFoundException:
  no segments* file found in MMapDirectory@/var/lucene/idx
  ...

修复路径(按优先级):

1. 立即用最近 SnapshotDeletionPolicy 快照恢复:
   - 停服
   - rsync /backup/idx-snapshot-X /var/lucene/idx-restore
   - 修改应用配置指向 idx-restore
   - 启动应用
   - 损失 = 快照后的写入数据

2. 若有 N-1 文件未删除(运气好):
   - 用 IndexCommit + DeletionPolicy 显式打开
   - rollback 到 N-1
   - 损失 = N-1 → N 之间的写入

3. 若无任何备份 + segment 文件完整:
   - 用 SegmentInfos.write(segmentInfos, dir, "segments_143")
     手动重建一个 segments_N
     (需熟悉所有 segment 的 delGen 等元数据)
   - 高风险,易出错

4. 全量重建:
   - 从源数据重新摄入
   - RTO 取决于数据量

预防:
- 索引目录权限严格控制 (chown app:app + chmod 750)
- 部署级监控:segments_* 文件数突变 → 告警
- 关键业务必开 SnapshotDeletionPolicy + 定期 rsync
- 备份至少保留 24 小时,周末/月末更长

反例 3 · NFS 上 commit 失败 / 半提交

现象:
- 应用部署在 NFS 共享挂载的索引目录
- IndexWriter.commit() 偶尔抛 IOException: rename failed
- 或读取时 segments_N 看到字段对不上的"半提交"快照
- CheckIndex 报 missing files

根因:
- NFS v3: rename 不保证原子,client cache 与 server
  状态可能不同步
- NFS v4 部分实现仍有 close-to-open 窗口
- 多客户端写同一索引时 NFS lock 不可靠
- fsync(directory) 在 NFS 上语义弱化

修复方案(按推荐度):
方案 1(强烈推荐):
  - 索引存本地 SSD/NVMe
  - 搜索通过 RPC 跨节点转发
  - 完全避免 NFS

方案 2(妥协):
  - NFS v4 + close-to-open consistency
  - mount 选项: noac,actimeo=0,nolock
  - 严格单进程写入(用业务侧锁,不依赖 NFS lock)
  - 可接受偶发 commit 失败 + 重试

方案 3(实验性,不推荐生产):
  - 自定义 LockFactory + Directory 适配层
  - 用 ZooKeeper / Etcd 协调写锁
  - 增加分布式协调复杂度

反例 4 · Snapshot 不释放导致磁盘膨胀

取了 snapshot 没 release

// 反例:释放路径漏掉
public void backupSync() throws IOException {
    IndexCommit snap = snapshotter.snapshot();
    // ... 同步复制文件 ...
    if (someCondition) {
        return;  // 提前返回,忘了 release
    }
    snapshotter.release(snap);
}

// 后果:
// - snap 引用计数永远 ≥ 1
// - 后续 commit 旧 segment 不能删
// - 磁盘持续膨胀
// - 几天后磁盘满 → ENOSPC → IndexWriter 死锁

try-finally 保证 release

public void backupSync() throws IOException {
    IndexCommit snap = snapshotter.snapshot();
    try {
        // 复制文件 / 上传到 S3 / etc.
        for (String name : snap.getFileNames()) {
            Files.copy(...);
        }
    } finally {
        snapshotter.release(snap);
    }
}

// 即使中间抛异常 / 提前 return,
// release 都被保证执行
// 引用计数及时归零
// 磁盘正常回收

反例 5 · OOM 后 IndexWriter 留下 pending 文件堆积

现象:
- 应用经常 OOM 重启
- 索引目录里出现多个 pending_segments_*
- 磁盘空间被占用但读不出来
- IndexWriter 启动时虽然能清理,但每次 OOM 都积累

文件示例:
  /var/lucene/idx/
    segments_42         (正常 commit)
    pending_segments_43 (上次 OOM 残留)
    pending_segments_44 (这次 OOM 残留)
    ...

修复:
1. IndexWriter 启动时会自动 cleanup,正常情况无需手动
   (调用 IndexFileDeleter#refresh 删除孤儿文件)
2. 如果 IW 启动也失败,手动删除所有 pending_segments_*:
   find /var/lucene/idx -name 'pending_segments_*' -delete
3. 根治 OOM:JVM 堆设置合理 + 监控 + 限流写入

注意:绝对不要在 IW 运行期间手动删 pending 文件
- 可能正在写入,删除会触发 IOException
- 中断 commit 流程,索引状态不可知

                             "segments_" + N+1)        // POSIX atomic
        -> directory.syncMetaData()                    // fsync(directory)
      -> segmentInfos.updateGeneration(pendingCommit)
      -> deleter.checkpoint(pendingCommit, true)       // 引用计数 + 删孤儿

// 读取 commit 完整调用链
DirectoryReader.open(directory)
  -> StandardDirectoryReader.open(directory, null, null)
    -> new FindSegmentsFile<DirectoryReader>(directory).run()
      -> directory.listAll()
      -> getLastCommitGeneration(files)               // 选最大 N
      -> doBody(segmentFileName)
        -> SegmentInfos sis = SegmentInfos.readCommit(directory, segmentFileName)
          -> ChecksumIndexInput input = directory.openChecksumInput(...)
          -> CodecUtil.checkHeader(input, "segments", ...)
          -> input.readBytes(id, 0, 16)
          -> input.readVInt() × 3                     // luceneVersion
          -> input.readLong()                         // generation
          -> for each segment: ... readSegmentInfo
          -> input.readMapOfStrings()                 // userData
          -> CodecUtil.checkFooter(input)             // CRC32 校验
        -> for each SegmentCommitInfo: openSegmentReader
        -> new StandardDirectoryReader(...)
