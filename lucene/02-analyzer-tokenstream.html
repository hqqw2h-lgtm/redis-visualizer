第 2 章 · Analyzer 与 TokenStream

CharFilter · Tokenizer · TokenFilter · AttributeSource · 复用槽 · TokenStream 生命周期 · SynonymGraph · PerFieldAnalyzerWrapper · reindex

在 1 千万订单的搜索系统里,用户搜 "iphone-15 pro"——能不能命中 "iPhone 15 Pro Max"?能不能区分 SKU "CN-2026-000045" 和文本 "CN 2026"?能不能让 "shipping" 同义匹配 "in transit"?这些问题的答案都不在 IndexWriter,不在 BM25,而在 Analyzer。Analyzer 是写入与查询路径之间的协议层,把原文转换为 token 序列,每个 token 携带 term、位置、原文偏移、类型——倒排索引、PhraseQuery、高亮统统建立在这个序列上。一旦索引侧 Analyzer 与查询侧 Analyzer 不一致,"明明有这个词却搜不到"就成了无法复现的灵异 bug。

但 Analyzer 不是一个黑盒。点开 Analyzer 子类的 createComponents 方法,看到的是一条从 Reader 到 TokenStream 的流水线:CharFilter 改写字符,Tokenizer 切出原始 token,TokenFilter 链依次改写、删除、扩展。token 在内部用 AttributeSource 这套"复用槽"机制传递,每次 incrementToken 不分配新对象——这是 Lucene 在 GC 上的一个核心优化。SynonymGraphFilter 让 "in transit" / "shipping" / "on the way" 三组同义词形成 token graph,PhraseQuery 在 graph 上做路径匹配。生产中订单字段一般要走 PerFieldAnalyzerWrapper,因为 orderId 不能分词、addressText 必须中文分词、fulfillmentNote 还要带业务同义词——一个全局 Analyzer 是必死的。本章把这些都拆开。

  • Analyzer 流水线四阶段:CharFilter / Tokenizer / TokenFilter / TokenStream 各自的输入输出契约
  • Tokenizer 选型矩阵:Standard / Keyword / Whitespace / SmartCN / IK / NGram 在 SKU 与中文地址上的真实形态
  • TokenFilter 三类操作:改写(LowerCase)/ 删除(StopFilter)/ 新增(SynonymGraph)对 position 与 offset 的影响
  • CharFilter 的 correctOffset 机制:HTML 实体改写后如何让高亮指向原文正确字节
  • AttributeSource 六种 Attribute:CharTerm / Offset / PositionIncrement / PositionLength / Type / Payload 的复用槽语义
  • TokenStream 五步生命周期:setReader → reset → incrementToken×N → end → close 与 try-with-resources 模板
  • SynonymGraphFilter 的 token graph:多 token 同义词的 PositionIncrement + PositionLength 如何让 PhraseQuery 走对路径
  • 索引/查询 Analyzer 一致性:写查必须共享 Analyzer 实例 + Analyzer 升级配合 reindex 的工程实践

前置知识:第 1 章涉及的倒排索引基础、字段建模四条访问路径、Reader 接口与字符流概念。第 4 章会展开 PhraseQuery / SpanQuery 在 token graph 上的路径匹配,第 11 章会展开 PostingsEnum 如何消费 TokenStream 写入的 term + position + offset——本章是这两条链路的共同上游入口。

2.1 Analyzer 流水线

Analyzer 不返回 token 列表,而是返回一个可迭代的 TokenStream。这个流式接口允许 Lucene 在不持有全部 token 的前提下完成倒排写入,内存占用与原文长度解耦——一份 1 MiB 的商品描述,Analyzer 内部只需要十几个字节的 attribute 槽,绝不会膨胀成几万个 Token 对象。

Analyzer.createComponents 的标准模板

public final class OrderNoteAnalyzer extends Analyzer {
    @Override
    protected TokenStreamComponents createComponents(String fieldName) {
        Tokenizer tokenizer = new StandardTokenizer();
        TokenStream stream = new LowerCaseFilter(tokenizer);
        stream = new StopFilter(stream, StopFilter.makeStopSet("the", "a", "to"));
        return new TokenStreamComponents(tokenizer, stream);
    }
}

TokenStreamComponents 包装"源端 Tokenizer"与"末端 TokenStream"两个引用。Tokenizer 是唯一直接读 Reader 的组件,所以 Analyzer 拿它来调 setReader;末端 TokenStream 是消费侧调 incrementToken 的入口。中间的 LowerCaseFilter / StopFilter 通过装饰器模式串接——每个 filter 持有上游引用,自己也是一个 TokenStream。

为什么 TokenStream 用流式而不是 List? — 设想一份 50 KB 的商品描述,如果 Analyzer 返回 List<Token>,就要分配几千个对象、占用几十 KiB 堆,大批量索引时 GC 压力陡增。流式模型让 Analyzer 始终只持有"当前 token"一个状态实例(就是后面要讲的 AttributeSource 复用槽),内存占用与原文长度解耦,GC 几乎不发生。这就是 Lucene 索引在 GB 级语料上仍能保持每秒万级写入吞吐的核心原因之一。

2.2 Tokenizer 选型

Tokenizer 决定第一刀切在哪——切完后的 token 形态再也改不回去。把订单号 CN-2026-000045 用 StandardTokenizer 切成 cn / 2026 / 000045,后续无论 LowerCaseFilter 还是 SynonymFilter 都救不回来"原始订单号整体匹配"这个语义——只能 reindex。

身份字段(orderId、SKU、tracking 号)优先 KeywordTokenizer 保持整体;同时另开一个分词字段(skuText)服务模糊召回。两个字段在 Lucene 中独立索引,查询时按业务语义选择其一——精确单号查 skuExact,关键词模糊召回查 skuText。代价是索引膨胀 ~30%,远低于"分错就要 reindex 几小时"的代价。

public static Analyzer skuExactAnalyzer() {
    return new Analyzer() {
        @Override
        protected TokenStreamComponents createComponents(String fieldName) {
            Tokenizer src = new KeywordTokenizer();
            TokenStream stream = new UpperCaseFilter(src);
            return new TokenStreamComponents(src, stream);
        }
    };
}

Elasticsearch keyword vs text 字段差异源于 Tokenizer 选择 — ES 把这两类需求"工程化"成两种字段类型:keyword 字段底层用 KeywordAnalyzer,token 整体保留,适合精确过滤、聚合、排序;text 字段底层用 StandardAnalyzer(或自定义),分词后用于全文召回。一个业务字段(如 product.name)在 mapping 里同时声明 keyword 子字段和 text 主字段,就是上面"双字段方案"在 ES 上的形态。订单号字段在 ES 里如果只声明 text,就会发现"按订单号过滤"性能极差——因为它走的是 term posting,不是 keyword 那种整体查找。

2.3 TokenFilter 链

TokenFilter 是装饰器。每个 filter 持有上游 TokenStream 引用,在 incrementToken() 中调用上游同名方法获取一个 token,然后改写、丢弃或扩展。多个 filter 串联形成处理链——这套结构让"加一个新 filter"零成本,也是 Lucene 分析器扩展的根基。

自定义同义词 filter 的最小骨架

public final class FulfillmentSynonymFilter extends TokenFilter {
    private final CharTermAttribute term = addAttribute(CharTermAttribute.class);
    private final PositionIncrementAttribute pos = addAttribute(PositionIncrementAttribute.class);
    private boolean emitUrgent;

    public FulfillmentSynonymFilter(TokenStream input) {
        super(input);
    }

    @Override
    public boolean incrementToken() throws IOException {
        if (emitUrgent) {
            clearAttributes();
            term.append("urgent");
            pos.setPositionIncrement(0);
            emitUrgent = false;
            return true;
        }
        if (!input.incrementToken()) {
            return false;
        }
        if ("same-day".contentEquals(term)) {
            emitUrgent = true;
        }
        return true;
    }

    @Override
    public void reset() throws IOException {
        super.reset();
        emitUrgent = false;
    }
}

这段 30 行代码演示了三个关键点。其一,同义词 token 的 PositionIncrement = 0,使其与原 token 共享位置,PhraseQuery 才能把它们当作同一个位置上的等价候选。其二,filter 在 reset() 中必须清空内部状态,否则 ReuseStrategy 复用时上一个文档的 emitUrgent 会泄漏到下一个文档。其三,生产环境推荐直接用 SynonymGraphFilter——它正确处理多 token 同义词的 graph 结构,自定义 filter 只在业务规则非常特殊时才写。

为什么 PositionIncrement = 0 是同义词的关键? — Lucene 的 PhraseQuery 在匹配 "wireless charger" 时,要求两个 term 在 相邻 position 出现。同义词如果填 PositionIncrement = 1,就被当作"下一个位置的新词"——查 "shipping pad" 命中不了 "in transit pad",因为 in / transit 各占一个 position,加起来是 4 个 token 的短语,而查询只期望 2 个 token 间隔 0。把 shipping 的 PositionIncrement 设为 0,告诉 Lucene"它和 in 共享位置 0",PhraseQuery 就能在 graph 上把 shipping 和 in transit 当作位置 0 上的等价路径,正确召回。

2.4 CharFilter 与偏移修正

CharFilter 在 Tokenizer 之前改写字符流。常见用途包括 HTML entity 解码、全角转半角、业务符号归一化(如 ① → 1)。但改写后字符位置与原文不再一致——Tokenizer 看到的"buyer"在第 4 位,原文里却在第 9 位(因为 &nbsp; 占了 6 字符)。如果不修正,搜索高亮就会指向错误的字节区间,展示出乱码。

static void dumpOffsets(Analyzer analyzer, String text) throws Exception {
    try (TokenStream stream = analyzer.tokenStream("note", new StringReader(text))) {
        CharTermAttribute term = stream.addAttribute(CharTermAttribute.class);
        OffsetAttribute offset = stream.addAttribute(OffsetAttribute.class);
        stream.reset();
        while (stream.incrementToken()) {
            System.out.printf("%s [%d,%d]%n",
                term, offset.startOffset(), offset.endOffset());
        }
        stream.end();
    }
}

自定义 CharFilter 必须正确维护 correction map — 继承 BaseCharFilter 后,每改写一处都要调 addOffCorrectMap(internalOffset, cumulativeDiff)。漏掉一次记录,后续所有 token 的 offset 会偏移到错误位置,高亮就会指向乱七八糟的字节区间。生产中 CharFilter 一旦上线,必须先用一批已知位置的 token 跑端到端 offset 校验测试,确认 startOffset / endOffset 与原文完全对齐——这一步成本很低但能避免上线后用户报"高亮错位"的 ticket。

2.5 AttributeSource 复用槽

TokenStream 内部不保存 token 列表。它持有的是一个 AttributeSource——一组"复用槽",每次 incrementToken() 把当前 token 的字段写入这些槽,消费侧从同一组槽读取。下一次 increment 来时槽内容被覆盖。这套机制让一千万 token 的索引过程只分配几十个 attribute 实例,GC 压力几乎为零。

"复用槽"是 Lucene 在 GC 上的核心优化 — 设想一份 1 万词的商品描述:如果每个 token 都是新对象,就要分配 1 万 × ≈ 5 个 attribute 字段 = 5 万对象。在百万级文档批量索引时,几亿对象的分配 / 回收会让 G1 GC 压力陡增,STW 频繁。复用槽方案下,每个 TokenStream 实例总共只有 5-6 个 attribute 对象,1 千万 token 全部"过水"也只是反复覆写这几个对象的内部字段。这就是为什么 Lucene 在常见 RAMBuffer 64-256 MiB 的设置下,G1 几乎看不到 Full GC——核心数据流压根没产生足够垃圾。

2.6 TokenStream 生命周期

TokenStream 不是"创建 → 用完 → 丢弃"的一次性对象,而是有完整 5 步生命周期的状态机。任何一步漏掉或顺序错乱,都会导致 token 丢失、Reader 泄漏或 IllegalStateException。生产代码用 try-with-resources 把这套契约固化成一个模板。

生命周期状态机

5 个生命周期方法

方法调用顺序职责出错后果

消费 TokenStream 的标准模板

try (TokenStream stream = analyzer.tokenStream(field, new StringReader(text))) {
    CharTermAttribute term = stream.addAttribute(CharTermAttribute.class);
    OffsetAttribute offset = stream.addAttribute(OffsetAttribute.class);
    stream.reset();
    while (stream.incrementToken()) {
        process(term, offset);
    }
    stream.end();
}

模板的关键有两点。第一,try-with-resources 自动调 close() 兜底 Reader 释放。第二,addAttribute 在 reset 之前调用——它注册的是状态槽(后续读写都指向该槽);reset 仅清空槽内容,不重新注册槽,如果在 reset 之后 add,你拿到的引用还是同一个槽,但代码语义上会让人误解为"重新注册"。固定写法是 attribute 在 reset 之前 add,reset 后只读不再 add。

错误示范(漏调 reset):
  TokenStream stream = ...;
  while (stream.incrementToken()) { ... }
  // ↑ IllegalStateException: TokenStream contract violation: reset()/close() call missing

错误示范(漏调 end):
  stream.reset();
  while (stream.incrementToken()) { ... }
  stream.close();
  // ↑ end() 没调,尾部 offset 信息丢失,高亮"最后一个词" 错位

2.7 ReuseStrategy

同一个 Analyzer 在一个线程里要分析几百万文档。如果每次 tokenStream() 都新建 Tokenizer + Filter 链,对象分配代价巨大。ReuseStrategy 把整套 TokenStreamComponents 缓存在 ThreadLocal 里,下次同字段直接复用——只需重新 setReader() 把新 Reader 接进去。

字段级复用的时序

图 2-9 · ReuseStrategy 的复用时序

三种复用策略

策略行为优势风险
GLOBAL_REUSE_STRATEGY所有字段共享一组组件对象最少字段差异导致状态污染
PER_FIELD_REUSE_STRATEGY每字段一组组件字段隔离清晰字段多时对象稍多
自定义按业务分组灵活维护成本高

订单搜索场景推荐 PER_FIELD_REUSE_STRATEGY。orderId / status / titleText / addressText 的 Analyzer 行为差异显著,字段级隔离避免状态串扰——titleText 上一个文档遗留的同义词内部状态绝不会泄漏到 addressText 的下一次分析。

状态泄漏的回归测试

@Test
void customFilterMustNotLeakStateAcrossOrders() throws Exception {
    Analyzer analyzer = new OrderFulfillmentAnalyzer();
    List first = AnalyzerAssert.terms(analyzer, "same-day delivery");
    List second = AnalyzerAssert.terms(analyzer, "standard delivery");

    assertEquals(List.of("same", "day", "delivery"), first);
    assertEquals(List.of("standard", "delivery"), second);
}
expected:
  first  = [same, day, delivery]
  second = [standard, delivery]

failure smell:
  second = [urgent, standard, delivery]
  cause  = 上一个文档的同义词状态未在 reset() 清空
自定义 TokenFilter 必须重写 reset() — Lucene 默认的 TokenFilter.reset() 只往上游传递,不清自定义字段。如果你在 filter 里加了 boolean emitUrgent、Queue<String> pendingTokens、State savedState 这种内部状态,必须在 reset() 里手动清零。一旦漏写,ReuseStrategy 会让上一文档的状态在下一文档复活,token 流形态突变,出现的 bug 是"间歇性多一个 token"——只在特定文档顺序下复现,日志里没法定位,堆栈里看不到。这种 bug 的修复成本是当初多写一行 reset 的 100 倍。

2.8 字段级 Analyzer 配置

订单系统的字段差异极大:orderId 必须不分词、addressText 必须中文分词、titleText 用英文 stem、fulfillmentNote 还要带业务同义词。一个全局 Analyzer 不可能同时满足这些需求。PerFieldAnalyzerWrapper 给每个字段绑一个独立 Analyzer,默认值兜底未配置的字段。

PerFieldAnalyzerWrapper 配置代码

public static Analyzer create() {
    Map fieldAnalyzers = new HashMap<>();
    fieldAnalyzers.put("orderId", new KeywordAnalyzer());
    fieldAnalyzers.put("trackingNo", new KeywordAnalyzer());
    fieldAnalyzers.put("status", new KeywordAnalyzer());
    fieldAnalyzers.put("skuExact", new KeywordAnalyzer());
    fieldAnalyzers.put("skuText", new StandardAnalyzer());
    fieldAnalyzers.put("titleText", new EnglishOrderTextAnalyzer());
    fieldAnalyzers.put("addressText", new SmartChineseAnalyzer());
    fieldAnalyzers.put("fulfillmentNote", new OrderFulfillmentAnalyzer());
    return new PerFieldAnalyzerWrapper(new StandardAnalyzer(), fieldAnalyzers);
}

身份字段与语义字段的分类

图 2-10 · 字段两大类对应的 Analyzer 分支

身份字段和语义字段的根本差异在于"分词意图":身份字段是给系统用的 ID,要的是字节级精确匹配;语义字段是给人看的内容,要的是语义召回。把它们混用一种 Analyzer,要么搜不到订单号,要么"商品描述"召回质量极差。

2.9 SynonymGraphFilter 与 token graph

同义词不是把词替换掉。在订单语境里,"in transit" / "shipping" / "on the way" 表达同一个状态,用户搜任意一个都应该全部命中。SynonymGraphFilter 把这种"多 token 同义词"组织成 token graph——每条同义路径占用相同的 position 区间,PhraseQuery / SpanQuery 在 graph 上做路径匹配,得到正确召回。

多 token 同义词的 graph 结构

图 2-11 · "in transit / shipping / on the way" 的 graph

三条路径从 pos 0 到 pos 2:in → transit(2 步,默认增量)、shipping(1 个 token 跨 2 步,PositionLength = 2)、on → the → way(3 步)。graph 的精髓是"同义路径并联",不是"线性插入新 token"——后者会破坏 PhraseQuery,因为查询里的"shipping"会被算成 1 个位置,与索引里的"in transit"(2 个位置)不匹配。

graph dump 代码

static void dumpGraph(Analyzer analyzer, String text) throws Exception {
    try (TokenStream stream = analyzer.tokenStream("titleText", new StringReader(text))) {
        CharTermAttribute term = stream.addAttribute(CharTermAttribute.class);
        PositionIncrementAttribute inc = stream.addAttribute(PositionIncrementAttribute.class);
        PositionLengthAttribute len = stream.addAttribute(PositionLengthAttribute.class);
        OffsetAttribute offset = stream.addAttribute(OffsetAttribute.class);

        int pos = -1;
        stream.reset();
        while (stream.incrementToken()) {
            pos += inc.getPositionIncrement();
            System.out.printf("term=%s pos=%d len=%d off=%d..%d%n",
                term, pos, len.getPositionLength(),
                offset.startOffset(), offset.endOffset());
        }
        stream.end();
    }
}
input:
  in transit

graph dump:
  term=in       pos=0 len=1 off=0..2
  term=shipping pos=0 len=2 off=0..10
  term=transit  pos=1 len=1 off=3..10

同义词放索引侧 vs 查询侧

维度索引侧查询侧
查询速度快(已展开)较慢(查询时扩展)
索引大小膨胀 1.5-3×不变

推荐场景稳定词库(品牌别名)频繁更新(热门搜索词)

实践经验:核心同义词(品牌别名、地址别名、固定业务术语)放索引侧,查询零扩展开销;高频更新的运营词、活动词放查询侧,改一次词典立即对所有历史数据生效。两侧混用是常见架构。

2.10 中文 Analyzer

中文订单地址 "上海市浦东新区张江路 88 号 3 单元 502 室" 用 StandardAnalyzer 切出来是逐字 token——上 / 海 / 市 / 浦 / 东 / 新 / 区——召回质量极差,因为查"浦东新区"在索引里找不到这串连续 token。中文必须用专门的中文 Analyzer,而不同方案在召回 / 精度 / 词典更新上的权衡差异显著。

中文地址的分词目标

原文:
  上海市浦东新区张江路 88 号 3 单元 502 室

期望 token:
  上海      上海市     浦东      浦东新区
  张江路    88 号       3 单元    502 室

候选方案:
  ① SmartChineseAnalyzer  HMM 分词
  ② IKAnalyzer             双向最大匹配 + 词典
  ③ AnsjAnalyzer           CRF 分词
  ④ NgramAnalyzer(2-3)     字符级 Ngram(兜底)

中文地址处理流水线

图 2-12 · 中文地址 Analyzer 的端到端流程

中文地址 Analyzer 的关键不在分词器本身,而在领域词典。"浦东新区""张江路""徐汇万科中心"等业务实体,如果不在词典中,分词器会拆为 "浦东 / 新区 / 张江 / 路 / 徐汇 / 万 / 科 / 中心",导致召回质量下降。生产实践是定期从订单数据中挖掘高频地址实体,加入自定义词典——这是一个常态化运营动作,不是一次性配置。

IK 自定义词典更新无法热加载的根本原因 — IKAnalyzer 启动时把 main.dic / stopword.dic / ext.dic 全部加载到 org.wltea.analyzer.dic.Dictionary 单例的内存树里。词典更新后即使你重新写文件,Dictionary 单例不会感知——它没有 inotify watch,也没有重载接口暴露给 Analyzer 重建。生产中要么重启 JVM,要么用 IKAnalyzer-fork(国内有人改造支持热加载,但与官方版本兼容性问题多),要么在应用层定期调度任务重新构造 IKAnalyzer 实例。这一点和 ES 的 ik 插件同源:ES ik 也支持远程词典 URL 周期拉取,但底层仍是"全部重载内存树",并非真正的增量更新。

2.11 索引/查询 Analyzer 一致性

这是一条铁律:索引侧用什么 Analyzer 切,查询侧就必须用相同的 Analyzer 切。一旦不一致,token 形态错位,查询的 term 在索引里根本不存在——结果就是"明明数据库里有这条记录,但搜索返回空"。这种 bug 在生产上的可观测性极差,日志里只看到"hits=0",定位往往要数小时。

不一致的失败模式

共享 Analyzer 实例的对照

反例:写查各自 new 一个 Analyzer

// 写入端

    new IndexWriterConfig(indexAnalyzer);

    w.addDocument(toDoc(order));

正例:依赖注入共享同一实例

@Bean
public Analyzer orderAnalyzer() {
    return OrderAnalyzers.create();
}

// 写入端注入
@Autowired Analyzer analyzer;
IndexWriterConfig wc =
    new IndexWriterConfig(analyzer);

// 查询端注入同一实例
@Autowired Analyzer analyzer;
QueryParser parser =
    new QueryParser("titleText", analyzer);
Query q = parser.parse("USB-C");
// 索引和查询走完全相同的 token pipeline
// hits 与索引里实际匹配的 doc 一一对应

通过 DI 强制只有一个 Analyzer 实例,从工程上根除"两端不同"的可能。

Analyzer analyzer = OrderAnalyzers.create();

IndexWriterConfig config = new IndexWriterConfig(analyzer);
try (IndexWriter writer = new IndexWriter(dir, config)) {
    writer.addDocument(toDoc(order));
}

QueryParser parser = new QueryParser("titleText", analyzer);
Query q = parser.parse("wireless charger");

IndexWriterConfig 与 QueryParser 共享同一个 Analyzer 实例,确保索引与查询走同一条 token pipeline。生产代码应通过依赖注入方式分发 Analyzer,杜绝在两处分别构造——构造点一旦分散,迟早有人改了一处忘了另一处。

2.12 回归测试

Analyzer 的输出形态是隐式契约。一旦改了某个 filter 的顺序或某条停用词,token 流就可能整体偏移,几百个查询的召回率统统掉一截——这种变化在代码 diff 里看不出,只能用回归测试守住。token dump 黄金文件是最有效的形式:把"这段输入应得到这串 token"写成断言,任何改动让结果变化都会立即暴露。

token dump 黄金文件

record TokenRow(String term, int position, int start, int end) {}

static List dumpRows(Analyzer analyzer, String field, String text) throws Exception {
    List rows = new ArrayList<>();
    try (TokenStream stream = analyzer.tokenStream(field, new StringReader(text))) {
        CharTermAttribute term = stream.addAttribute(CharTermAttribute.class);
        PositionIncrementAttribute inc = stream.addAttribute(PositionIncrementAttribute.class);
        OffsetAttribute offset = stream.addAttribute(OffsetAttribute.class);
        int pos = -1;
        stream.reset();
        while (stream.incrementToken()) {
            pos += inc.getPositionIncrement();
            rows.add(new TokenRow(term.toString(), pos,
                offset.startOffset(), offset.endOffset()));
        }
        stream.end();
    }
    return rows;
}

断言级测试

@Test
void skuTextTokensMustStayStable() throws Exception {
    Analyzer analyzer = OrderAnalyzers.create();
    List rows = dumpRows(analyzer, "skuText", "SKU-USB-C-20W");

    assertEquals(List.of(
        new TokenRow("sku", 0, 0, 3),
        new TokenRow("usb", 1, 4, 7),
        new TokenRow("c", 2, 8, 9),
        new TokenRow("20w", 3, 10, 13)
    ), rows);
}
test output:
  skuTextTokensMustStayStable PASSED
  rows = [sku@0(0..3), usb@1(4..7), c@2(8..9), 20w@3(10..13)]

2.13 升级路径与生产陷阱

Analyzer 升级是搜索系统最危险的变更之一——旧 segment 里的 token 是按旧 Analyzer 形态固化在 .tim 里的,新 Analyzer 不会"自动应用"到旧数据。生产中必须配合全量 reindex + alias 切换,这一点没有捷径。

Analyzer 升级流程

图 2-14 · 升级 Analyzer 的端到端流程

Analyzer 版本化字段

元字段用途
analyzer.version生成 token 的 Analyzer 版本号
analyzer.dict.version词典版本(IK / 同义词)
analyzer.git.shaAnalyzer 代码的 git commit
index.created.at索引构建时间戳

把 Analyzer 元信息写入 IndexWriterConfig.setUserData(),可通过 DirectoryReader.getIndexCommit().getUserData() 取回。重建索引前对比 userData 判断是否需要全量重建——避免"分析器没动却重建了索引"的资源浪费,也避免"分析器变了却没重建"的数据不一致。

索引重建的原子切换

重建流程:

  Step 1: 在 /data/orders-v2/ 离线构建新索引
  Step 2: CheckIndex 验证完整性
  Step 3: 对比关键查询的 NDCG@10
  Step 4: 原子切换 alias
          orders-current → orders-v2
  Step 5: 旧索引保留 24 小时
  Step 6: 确认无回滚需要 → 删除旧索引
升级 Analyzer 必须 reindex,没有"灰度部分文档"这种取巧 — 写过 ES 的同学常想"我能不能只对新文档用新 Analyzer,旧文档保持原样"。答案是不行。索引按 segment 组织,同一字段在所有 segment 中必须用同一种 Analyzer 才能保证 term 形态一致;一旦你让一个字段在不同 segment 里有不同 token 形态,查询时同一个 term 命中部分 segment、漏掉另一部分,翻页跳数、totalHits 偏低、score 离群——所有可观测信号都会"间歇性异常",这是搜索系统最难排查的故障类型。正确做法只有一个:全量 reindex + alias 原子切换 + 旧索引保留至少 24 小时回滚窗口。
