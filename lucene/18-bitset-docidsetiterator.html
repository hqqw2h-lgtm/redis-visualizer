第 11 章把 PostingsEnum 当作"docID 流"的物理形态讲清楚了——term 倒排里的 docID 集合可以用 PFOR + 多级 skip 高效遍历。但 Lucene 内部还有一类截然不同的 docID 集合:LiveDocs(段内删除标记)、Filter 缓存的 BitSet、FacetCollector 的命中标记、TwoPhaseIterator 的中间结果——它们的特点是"已经物化为内存集合,需要支持 O(1) 随机访问 + 顺序迭代 + 集合运算"。本章拆解这些场景下的物理结构:FixedBitSet 用 long[] 阵列实现稠密位图,SparseFixedBitSet 用双层 bucket 处理 < 5% 密度,RoaringDocIdSet 三种 container 自适应切换。所有这些容器都通过 BitSetIterator 包装成 DocIdSetIterator,与 PostingsEnum 一起被 ConjunctionDISI / DisjunctionDISI / TwoPhaseIterator 等求交求并算法消费。

具体一点:DocIdSetIterator(简称 DISI)是 Lucene 查询执行的"通用语言"。它只定义四个核心方法 nextDoc / advance / docID / cost,但所有 docID 流——无论来自倒排索引(PostingsEnum)、BitSet 包装(BitSetIterator)、求交结果(ConjunctionDISI)、求并结果(DisjunctionDISIPriorityQueue)、BKD 候选(PointInSetIterator)——都实现这套接口。这种统一让"求交求并"算法只需要按 DISI 协议工作,不关心底层来源。本章用大量的字节级布局图、leapfrog 时序图、状态机流程图,把"为什么 1000 万 docID 求交在毫秒内完成"这条路径拆开:从 long word 上的字级 AND 运算(O(N/64)),到 SparseFixedBitSet 的 12× 内存节省,到 conjunction 用 cost() 选驱动迭代器的启发式,到 LiveDocs 在每个候选 doc 上的 1-bit 过滤。读完之后你能从源码层面回答"为什么 Lucene 用 long[] 而不是 List<Integer>,以及为什么求交比逐 doc 检查快几个数量级"。

18.1 DocIdSetIterator 接口:查询执行的通用语言

Lucene 查询执行的核心抽象只有一个—— DocIdSetIterator。它定义了"按 docID 升序遍历 + 跳跃定位"的最小必要接口,所有 docID 流都实现这套接口。理解这个接口的契约和复杂度模型,是理解后续 BitSet 实现、求交求并算法、TwoPhaseIterator 的前提。本节先用一张类层级图把所有 DISI 实现串起来,再把接口契约逐字段拆开。

18.2 FixedBitSet:long[] 阵列上的字级位图

FixedBitSet 是密集 docID 集合的默认实现。它把"集合"映射到 long[] bits 数组,每个 long 64 个 bit,用 i >>> 6 算 word 索引、用 i & 63 算 bit 偏移。这种朴素到极致的布局是它在生产里跑得飞快的根本原因——CPU 指令 POPCNT / TZCNT / LZCNT 直接对 long 操作,JIT 把 hot loop 编成 SIMD,L1 cache 命中率几乎 100%。本节用字节布局图把 FixedBitSet 的物理形态讲清楚。

核心 API 实现

public final class FixedBitSet extends BitSet {
    private final long[] bits;
    private final int numBits;
    private final int numWords;

    public FixedBitSet(int numBits) {
        this.numBits = numBits;
        this.numWords = bits2words(numBits);
        this.bits = new long[numWords];
    }

    public boolean get(int index) {
        assert index >= 0 && index < numBits;
        int wordNum = index >> 6;
        long bitmask = 1L << (index & 0x3F);
        return (bits[wordNum] & bitmask) != 0;
    }

    public void set(int index) {
        assert index >= 0 && index < numBits;
        int wordNum = index >> 6;
        long bitmask = 1L << (index & 0x3F);
        bits[wordNum] |= bitmask;
    }

    public void clear(int index) {
        assert index >= 0 && index < numBits;
        int wordNum = index >> 6;
        long bitmask = 1L << (index & 0x3F);
        bits[wordNum] &= ~bitmask;
    }

    public int nextSetBit(int index) {
        int i = index >> 6;
        long word = bits[i] >> (index & 0x3F);
        if (word != 0) {
            return index + Long.numberOfTrailingZeros(word);
        }
        while (++i < numWords) {
            word = bits[i];
            if (word != 0) {
                return (i << 6) + Long.numberOfTrailingZeros(word);
            }
        }
        return DocIdSetIterator.NO_MORE_DOCS;
    }

    public int cardinality() {
        return (int) BitUtil.pop_array(bits, 0, numWords);
    }
}

18.3 SparseFixedBitSet:稀疏场景的 12× 节省

18.4 RoaringDocIdSet:三容器自适应

18.5 字级位运算:1000 万 docID 求交亚毫秒级

BooleanQuery 的 MUST / FILTER / MUST_NOT 子句在执行时会把多个 DocIdSet 求交、求并、求差。如果按"逐 docID 检查"的实现,1000 万 docID 求交需要 1000 万次 BitSet.get(),每次 O(1) 但常数巨大。Lucene 的实际实现是按 64-bit 字 (long word) 整批运算,把工作压缩到 ~150 万次 long 操作 (1000 万 / 64),配合 JIT 自动向量化为 SIMD,在 AVX-512 上做到亚毫秒。本节展示 FixedBitSet 的 AND/OR/ANDNOT 字级实现以及对应的 SIMD 加速。

三种集合运算的字级实现

public final class FixedBitSet extends BitSet {

    public void and(FixedBitSet other) {
        long[] thisBits  = this.bits;
        long[] otherBits = other.bits;
        int n = Math.min(thisBits.length, otherBits.length);
        for (int i = 0; i < n; i++) {
            thisBits[i] &= otherBits[i];
        }
        for (int i = n; i < thisBits.length; i++) {
            thisBits[i] = 0L;
        }
    }

    public void or(FixedBitSet other) {
        long[] thisBits  = this.bits;
        long[] otherBits = other.bits;
        int n = Math.min(thisBits.length, otherBits.length);
        for (int i = 0; i < n; i++) {
            thisBits[i] |= otherBits[i];
        }
    }

    public void andNot(FixedBitSet other) {
        long[] thisBits  = this.bits;
        long[] otherBits = other.bits;
        int n = Math.min(thisBits.length, otherBits.length);
        for (int i = 0; i < n; i++) {
            thisBits[i] &= ~otherBits[i];
        }
    }

    public int intersectionCount(FixedBitSet other) {
        return (int) BitUtil.pop_intersect(
            this.bits, other.bits, 0,
            Math.min(this.bits.length, other.bits.length));
    }
}

18.6 BitSetIterator:把 BitSet 包装成 DISI

BitSet 自身只支持 get(i) / set(i) / nextSetBit(i) 这种"按 docID 直接访问"的语义,不直接实现 DocIdSetIterator。BitSetIterator 把 BitSet 包装成 DISI,让 ConjunctionDISI / DisjunctionDISI 可以无差别地消费 BitSet。本节拆解 BitSetIterator 的实现,以及它的 nextDoc / advance 性能特征。

BitSetIterator 实现

public class BitSetIterator extends DocIdSetIterator {
    private final BitSet bits;
    private final int length;
    private final long cost;
    private int doc = -1;

    public BitSetIterator(BitSet bits, long cost) {
        this.bits = bits;
        this.length = bits.length();
        this.cost = cost;
    }

    @Override
    public int docID() { return doc; }

    @Override
    public int nextDoc() throws IOException {
        return advance(doc + 1);
    }

    @Override
    public int advance(int target) throws IOException {
        if (target >= length) {
            return doc = NO_MORE_DOCS;
        }
        return doc = bits.nextSetBit(target);
    }

    @Override
    public long cost() { return cost; }

    public BitSet getBitSet() { return bits; }
}

18.9 ConjunctionDISI:leapfrog 求交

BooleanQuery MUST / FILTER 子句的求交由 ConjunctionDISI 实现,它的核心算法是 leapfrog:cost 最小的子 DISI 当首发(lead),用 nextDoc 推进;其它子 DISI 用 advance 跟随。如果某个 follower 的 docID > lead 当前 doc,提升 lead 到该 docID 重启循环。这是 Lucene 求交的最关键算法,本节用时序图把它画清楚。

ConjunctionDISI 实现骨架

public final class ConjunctionDISI extends DocIdSetIterator {
    private final DocIdSetIterator lead;
    private final DocIdSetIterator[] others;

    public ConjunctionDISI(List<? extends DocIdSetIterator> iterators) {
        iterators.sort(Comparator.comparingLong(DocIdSetIterator::cost));
        this.lead = iterators.get(0);
        this.others = iterators.subList(1, iterators.size())
            .toArray(new DocIdSetIterator[0]);
    }

    @Override
    public int nextDoc() throws IOException {
        return doNext(lead.nextDoc());
    }

    @Override
    public int advance(int target) throws IOException {
        return doNext(lead.advance(target));
    }

    private int doNext(int doc) throws IOException {
        advanceHead: for (;;) {
            for (DocIdSetIterator other : others) {
                int next = other.advance(doc);
                if (next > doc) {
                    doc = lead.advance(next);
                    continue advanceHead;
                }
            }
            return doc;
        }
    }

    @Override
    public long cost() { return lead.cost(); }
}

18.10 DisjunctionDISI:最小堆求并

BooleanQuery SHOULD 子句的求并由 DisjunctionDISIPriorityQueue 实现:把所有子 DISI 放入按当前 docID 升序的最小堆,每次取堆顶作为输出,然后调 top.nextDoc() 重排堆。这种"K 路归并"的标准模式,把"求并"的复杂度控制在 O(N × log K)。本节展开堆操作的关键步骤。

实现骨架

public class DisjunctionDISIPriorityQueue {
    private final DisiWrapper[] heap;
    private int size;

    public DisjunctionDISIPriorityQueue(int maxSize) {
        this.heap = new DisiWrapper[maxSize];
    }

    public DisiWrapper top() { return heap[0]; }

    public DisiWrapper updateTop() {
        downHeap(0);
        return heap[0];
    }

    private void downHeap(int i) {
        DisiWrapper node = heap[i];
        int j = leftChild(i);
        while (j < size) {
            int rightSibling = j + 1;
            if (rightSibling < size
                && heap[rightSibling].doc < heap[j].doc) {
                j = rightSibling;
            }
            if (node.doc <= heap[j].doc) break;
            heap[i] = heap[j];
            i = j;
            j = leftChild(i);
        }
        heap[i] = node;
    }

    private static int leftChild(int i) { return 2 * i + 1; }
}

public int nextDoc() throws IOException {
    DisiWrapper top = pq.top();
    int currentDoc = top.doc;

    do {
        top.doc = top.iterator.nextDoc();
        top = pq.updateTop();
    } while (top.doc == currentDoc);

    return top.doc == NO_MORE_DOCS ? NO_MORE_DOCS : top.doc;
}

18.11 TwoPhaseIterator:cheap 候选 + expensive 确认

18.14 LiveDocs 在查询中:每候选 1 bit 过滤

查询时每个 DISI 推进的候选 doc 都要过一遍 LiveDocs。这个过滤通过 ConstantScoreScorer 或 FilteredDocIdSetIterator 的内层逻辑实现:Scorer.iterator().nextDoc() 拿到候选,Collector.collect() 之前先 liveDocs.get(doc)。如果是 false,跳过当前 doc 继续 nextDoc。本节展开这个机制以及它的性能影响。

"删了 30% 的 doc,P99 从 5ms 飙到 60ms"。某电商订单库的真实故障:每周末批量归档 1 周前的订单(15-20%),累积到第 3 周后 LiveDocs 删除率 30%+,P99 从 5ms 飙到 60ms。剖析发现 LRUQueryCache 命中率不变(filter 还是同样),但 BlockPostingsEnum.advance 次数翻 1.5 倍——因为 conjunction 推进时大量 candidate 是 deleted doc,需要继续 nextDoc。修复:把 cron job 加上 writer.forceMergeDeletes()(每周末归档后立即执行),P99 回到 6ms。LiveDocs 是查询路径上的"沉默税",删除率监控应该是搜索集群的标准告警项。
