第 15 章 · TopK、WAND 与 Collector

小顶堆 · minCompetitiveScore · ScoreMode · BlockMax · pivot 选择 · searchAfter · 早终止 · 跨 leaf MaxScoreAccumulator

绝大多数搜索查询只需要前 K 个高分结果(K 通常 10-100),完整扫描全部命中并排序极其浪费。Lucene 通过 TopK 小顶堆 + BlockMax WAND 算法,在 Top-K 已知最低分阈值 minCompetitiveScore 后,跳过整个低分块,典型加速 5-20×。本章拆解 TopK 小顶堆、BlockMax 上界、WAND 算法、Collector 体系、searchAfter 翻页与 ScoreMode 的配合。

ScoreMode 是 Lucene 8.x 引入的关键抽象。COMPLETE 表示需要完整 hit count,无法启用 WAND 跳过;TOP_SCORES 仅要求 TopK 准确,允许提前终止与跳过。生产中绝大多数前台搜索可用 TOP_SCORES,仅聚合 / 计数场景用 COMPLETE。本章给出选择决策、字节级 Impacts 布局、逐帧 WAND 算法演进、性能边界与陷阱。

  • TopK 为何用小顶堆:堆顶 = 最弱 K 候选,与新候选只比一次,O(log K) 维护
  • minCompetitiveScore 是 Collector → Scorer 的反馈信号,触发 BlockMax 跳过
  • ScoreMode 五种(COMPLETE / COMPLETE_NO_SCORES / TOP_SCORES / TOP_DOCS / TOP_DOCS_WITH_SCORES)与场景
  • BlockMax Impacts 在 .doc 文件中如何按 128 / 1024 / 8192 三级存储 (maxFreq, minNorm)
  • WAND pivot 选择:累加 maxScore 直到 ≥ minCompetitive,pivot 之前所有 doc 都不可能进 TopK
  • BlockMax WAND 与传统 WAND 的差异:块级上界 vs term 全局上界,跳过更激进
  • 自定义 Collector 必须显式声明 scoreMode(),否则 WAND 默认失效
  • MultiCollector 合并子 Collector 的 ScoreMode 取最严,加 facet 会让 WAND 全失效
  • searchAfter 用游标翻页,堆容量 O(K) 与翻页深度无关,from+size 深翻页一定 OOM
  • IndexSort + Early Termination:写入慢 10%,sort by 同字段查询加速 50-200×

前置知识:第 4 章 BM25 评分(知道 idf / tf / norm 三因子);第 11 章 PostingsEnum 与 SkipList(理解 advance / nextDoc 代价);第 14 章查询编译(认识 BooleanQuery / TermQuery 怎样组装出 Scorer)。读者需要熟悉优先队列、二分搜索与递归。

15.1 TopK 小顶堆

用户搜索"红色连衣裙",索引里命中可能有 30 万条,但 UI 只显示前 20 条。把 30 万 ScoreDoc 全排再切前 20,峰值内存 30 万 × 24 byte ≈ 7 MB,排序 O(N log N) ≈ 6 ms。这两个数字都不算特别大,但每秒几千 QPS 时,把 30 万次比较全做一次会把 CPU 烧光。小顶堆只维护当前 Top-K,堆顶 = K 个候选中分数最低的,新候选只跟堆顶比一次,大于则替换、小于则丢弃,O(log K) 维护。

15.1.1 小顶堆的关键不变量

三条不变量:
  1. 堆始终保持至多 K 个候选(初期 size < K 时直接插)。
  2. 堆顶 = 当前 K 个候选中最低分(小顶堆性质)。
  3. 新候选 score > 堆顶 → 弹堆顶 + 插新 + 下沉;否则丢弃。

  heap = [B:1.5, A:2.0, C:3.0]                        size=3   minComp=-∞

帧 4  doc=D score=0.5
  size<K → 直接插, D 上浮到顶 (最小)
  heap = [D:0.5, B:1.5, A:2.0, C:3.0]                 size=4   minComp=-∞

帧 5  doc=E score=2.5
  size<K → 直接插, 维持小顶
  heap = [D:0.5, B:1.5, A:2.0, C:3.0, E:2.5]          size=5=K
  ★ 第一次"填满 K", 设 minCompetitive = D:0.5  (堆顶)

帧 6  doc=F score=1.0
  size==K, F.score=1.0 > heap.peek=0.5 → 替换 + 下沉

  设 minCompetitive = F:1.0     ← 阈值上升

帧 7  doc=G score=3.5
  G.score=3.5 > heap.peek=1.0 → 替换
  heap = [B:1.5, E:2.5, A:2.0, C:3.0, G:3.5]
  设 minCompetitive = B:1.5     ← 阈值再升

帧 8  doc=H score=0.8
  H.score=0.8 < heap.peek=1.5 → 直接丢弃, 堆不变, 阈值不变

最终 Top-5 = [G:3.5, C:3.0, E:2.5, A:2.0, B:1.5]
最终 minCompetitive = 1.5

★ 关键观察: 阈值单调上升, 后续 score < 1.5 的 doc 完全可以跳过

15.2 minCompetitiveScore 反馈信号

Scorer.setMinCompetitiveScore(float) 是 ScoreMode.TOP_SCORES 下 Collector 通知 Scorer "当前堆顶最低分"的接口。Scorer 拿到该值后,在 advance 时若整 block 的 maxScore 上界 < minCompetitive,直接跳过整 block。这是 BlockMax WAND 的核心反馈机制。

表 15-2-1 · setMinCompetitiveScore 触发时机

15.2.3 跨 leaf 的 minCompetitive 共享

多 segment 索引中,每 segment 的 LeafCollector 独立维护堆,但 minCompetitive 跨 leaf 共享。Lucene 用 MaxScoreAccumulator 提供线程安全的全局最低分,各 leaf 都能从中读到当前最严的下界。

图 15-2-2 · MaxScoreAccumulator 跨 Leaf 协作
真实收益(Lucene Bench 数据):50 个 segment 的中型索引,单线程跑 5 term SHOULD 查询 K=20。无 MaxScoreAccumulator(各 leaf 独立 minComp):平均 42 ms;启用(9.x+):平均 28 ms,提升 33%。原因:第一个 leaf 已经把堆塞满到很高分数,后面 49 个 leaf 直接套用更严的上界,跳过的 block 多了 60%。

15.5 maxScore 上界估算

Impacts 给出了 (maxFreq, minNorm),但 Scorer 真正用的是 BM25 上界。BM25 公式对固定 term 而言,score 是 freq 与 docLen 的单调函数:freq ↑ → score ↑,docLen ↓(norm ↑) → score ↑。因此 maxScore = score(maxFreq, minNorm)。

15.5.1 BM25 上界公式

图 15-5-1 · BM25 上界推导
BM25 公式(回忆第 4 章):
   score = idf × (freq × (k1 + 1)) / (freq + k1 × (1 - b + b × docLen / avgDocLen))

对固定 term:
   idf, k1, b, avgDocLen 都是常量

score 是 (freq, docLen) 的函数:
   freq ↑   → score ↑   (单调递增)
   docLen ↓ → score ↑   (短文档加分,因为 fieldLength normalization)

因此最大 score 出现在 (maxFreq, minDocLen) 处:

   maxScore_block = idf × (maxFreq × (k1 + 1)) /
                    (maxFreq + k1 × (1 - b + b × minDocLen / avgDocLen))

minDocLen 通过 minNorm 反查 SmallFloat 表得到。

────────────────────────────────────────────────
具体例 (idf=1.5, k1=1.2, b=0.75, avgDocLen=120):

   block 0:  maxFreq=15, minDocLen=80
              maxScore = 1.5 × (15 × 2.2) / (15 + 1.2 × (0.25 + 0.75 × 80/120))
                       = 1.5 × 33    / (15 + 1.2 × 0.75)
                       = 49.5        / 15.9
                       = 3.11

   block 1:  maxFreq=5, minDocLen=200
              maxScore = 1.5 × (5 × 2.2) / (5 + 1.2 × (0.25 + 0.75 × 200/120))
                       = 16.5         / 7.0
                       = 2.36

   若 minCompetitive = 2.5:
     block 0 maxScore=3.11 ≥ 2.5  → 进入
     block 1 maxScore=2.36 < 2.5  → 跳过

15.6.2 advanceShallow vs advance 对比

❌ 仅用 advance

while (有候选) {
    int doc = scorer.advance(target);
    float s = scorer.score();
    if (s > minComp) {
        collect(doc, s);
    }
}
// 每个 doc 都解码 PFOR + 算 BM25
// 即使后面会丢弃, 仍付了解码成本
// 5 term SHOULD: 200 ms

✅ advanceShallow + advance 分离

while (有候选) {
    scorer.advanceShallow(target);             // 廉价
    Impacts imp = scorer.getImpacts();
    float maxBlock = computeMax(imp, sim);
    if (maxBlock < minComp) {
        // 整 block 跳过, 不解码
        target = blockEnd + 1;
        continue;
    }
    int doc = scorer.advance(target);          // 昂贵, 但只在可能竞争时
    float s = scorer.score();
    if (s > minComp) collect(doc, s);
}
// 5 term SHOULD: 20 ms (10× 加速)

15.7 WAND 算法

WAND(Weak AND)算法用于 N 子句 SHOULD 查询(OR)的 TopK 加速。核心思想:对每子句维护 maxScore 上界,按 docID 排序后,从最小 docID 累加上界,直到累加值 ≥ minCompetitive,该 docID 即 pivot。pivot 之前的所有 doc 不可能进 TopK,可跳过。

15.7.1 WAND 主循环逐帧推演

图 15-7-1 · WAND 算法 7 帧推演(3 子句 SHOULD)
输入: 3 个 SHOULD 子句, 各有 maxScoreBound:
  子句 1 (term="apple") : maxBound = 3.0,  当前 docID = 10
  子句 2 (term="orange"): maxBound = 2.0,  当前 docID = 25
  子句 3 (term="banana"): maxBound = 1.0,  当前 docID = 30

minCompetitive = 4.0  (堆顶, K=20 已填满)

────────────────────────────────────────────────
帧 0  按 docID 升序排列子句
   排序: [子句1:10, 子句2:25, 子句3:30]
   maxBound: [3.0, 2.0, 1.0]

帧 1  选 pivot
   累加: 3.0           → < 4.0  不够
   累加: 3.0 + 2.0 = 5.0 → ≥ 4.0  够了
   pivot index = 1 (子句2)
   pivot docID = 25

帧 2  检查所有 docID == pivot ?
   子句1.docID = 10 ≠ 25  → 推进 子句1.advance(25)
   子句1.advance(25) → 返回 30 (跳到 30)
   新状态: 子句1=30, 子句2=25, 子句3=30

帧 3  重新排序, 重新选 pivot
   排序: [子句2:25, 子句1:30, 子句3:30]
   累加: 2.0           → < 4.0  不够
   累加: 2.0 + 3.0 = 5.0 → ≥ 4.0  够了
   pivot index = 1 (子句1)
   pivot docID = 30

帧 4  推进 docID < pivot 的子句
   子句2.advance(30) → 返回 30 (匹配)
   新状态: 子句2=30, 子句1=30, 子句3=30
   ★ 全部 docID == pivot

帧 5  评分
   子句1.score() = 2.5
   子句2.score() = 1.8
   子句3.score() = 0.7
   sum = 5.0
   5.0 > 4.0 → 进 TopK
   collect(30, 5.0)
   堆顶更新, 新 minComp = 4.5

帧 6  推进所有子句到 31
   子句1.advance(31) → 50
   子句2.advance(31) → 80
   子句3.advance(31) → 60

帧 7  新 pivot 选择 (minComp=4.5)
   排序: [子句1:50, 子句3:60, 子句2:80]

public interface LeafCollector {
    void setScorer(Scorable scorer) throws IOException;
    void collect(int doc) throws IOException;

    default void setMinCompetitiveScore(float minScore) throws IOException {}

    default DocIdSetIterator competitiveIterator() throws IOException {
        return null;
    }
}
setScorer 时机非常关键:Lucene 在每 leaf 开始扫描前调一次 setScorer,Collector 必须在此把 Scorer 引用保存,后续 collect 才能拿到 score。同时,如果堆已经填满,必须在 setScorer 里立刻通知 Scorer 当前 minComp,否则第一个 block 错过 WAND 跳过。

15.11 自定义 Collector 编程

15.11.1 完整示例:带 deadline 的 TopK

public class TimedTopKCollector implements Collector {
    private final int k;
    private final PriorityQueue<ScoreDoc> heap;
    private final long startNs;
    private final long deadlineNs;
    private int collectedCount = 0;

    public TimedTopKCollector(int k, long timeoutMs) {
        this.k = k;
        this.heap = new PriorityQueue<>(k, (a, b) -> Float.compare(a.score, b.score));
        this.startNs = System.nanoTime();
        this.deadlineNs = startNs + TimeUnit.MILLISECONDS.toNanos(timeoutMs);
    }

    @Override
    public ScoreMode scoreMode() {
        return ScoreMode.TOP_SCORES;
    }

    @Override
    public LeafCollector getLeafCollector(LeafReaderContext ctx) {
        return new LeafCollector() {
            private Scorable scorer;
            private final int docBase = ctx.docBase;

            @Override
            public void setScorer(Scorable s) throws IOException {
                this.scorer = s;
                if (heap.size() == k) {
                    s.setMinCompetitiveScore(heap.peek().score);
                }
            }

            @Override
            public void collect(int doc) throws IOException {
                if (System.nanoTime() > deadlineNs) {
                    throw new CollectionTerminatedException();
                }

                float s = scorer.score();
                collectedCount++;

                if (heap.size() < k) {
                    heap.add(new ScoreDoc(doc + docBase, s));
                    if (heap.size() == k) {
                        scorer.setMinCompetitiveScore(heap.peek().score);
                    }
                } else if (heap.peek().score < s) {
                    heap.poll();
                    heap.add(new ScoreDoc(doc + docBase, s));
                    scorer.setMinCompetitiveScore(heap.peek().score);
                }
            }
        };
    }

    public TopDocs topDocs() {
        ScoreDoc[] arr = heap.toArray(new ScoreDoc[0]);
        Arrays.sort(arr, (a, b) -> Float.compare(b.score, a.score));
        return new TopDocs(
            new TotalHits(collectedCount, TotalHits.Relation.GREATER_THAN_OR_EQUAL_TO),
            arr);
    }
}

15.13 searchAfter 翻页

IndexSearcher.searchAfter(after, query, size) 是 Lucene 的稳定深翻页 API,基于游标(ScoreDoc cursor)而非 offset,内存与翻页深度无关。

15.13.1 searchAfter 实现机制

图 15-13-1 · searchAfter 游标翻页机制
searchAfter 翻页机制:

  Page 1: searcher.search(q, 20)
    返回 ScoreDoc[20]
    ScoreDoc[19] = (docID=42, score=2.3, shardIndex=0)

  Page 2: searcher.searchAfter(ScoreDoc[19], q, 20)
    内部: 把 ScoreDoc[19] 作为 minCompetitive 起始
    Collector 不收集 score < 2.3 的 doc
    或 score == 2.3 但 docID <= 42 的 doc

  Tie-breaker (核心!):
    score 相同时按 docID 升序
    docID 单调递增, 确保翻页稳定

  约束:
    1. 必须用同一 IndexReader (NRT 也行, 但保持一致)
    2. 排序键稳定 (含 tie-breaker)
    3. 索引中途不变 (commit 可能错位)
    4. shardIndex 必须设置 (多 shard 场景)

15.13.2 翻页代码

public List<TopDocs> paginate(IndexSearcher searcher, Query query, int pageSize, int pageCount)
        throws IOException {
    List<TopDocs> pages = new ArrayList<>();

    TopDocs page1 = searcher.search(query, pageSize);
    pages.add(page1);

    ScoreDoc cursor = page1.scoreDocs.length > 0
        ? page1.scoreDocs[page1.scoreDocs.length - 1]
        : null;

    for (int i = 1; i < pageCount && cursor != null; i++) {
        TopDocs nextPage = searcher.searchAfter(cursor, query, pageSize);
        if (nextPage.scoreDocs.length == 0) break;
        pages.add(nextPage);
        cursor = nextPage.scoreDocs[nextPage.scoreDocs.length - 1];
    }

    return pages;
}

15.15.2 IndexSort + Early Termination

如果索引在写入时配置 IndexWriterConfig.setIndexSort(sort),segment 内 doc 按 sort 字段排序。查询时 sort by 同字段可在前 K 命中后早终止,无须扫完整 segment。

15.19.2 反例 1:from+size 大翻页 → searchAfter

❌ 反例:from+size 实现深翻页

public TopDocs getPage(IndexSearcher searcher, Query query,
                       int page, int size) throws IOException {
    int from = (page - 1) * size;
    return searcher.search(query, from + size);
}

public TopDocs getPage100() throws IOException {
    return getPage(searcher, query, 100, 20);
}

public TopDocs getPage1000() throws IOException {
    return getPage(searcher, query, 1000, 20);
}

// 现象:
// getPage(100): 堆容量 = 2020, 延迟 150 ms
// getPage(1000): 堆容量 = 20020, 延迟 5000 ms
// getPage(10000): OOM
// 内存随 page 线性涨
// WAND 加速被堆维护开销吞噬

✅ 修复:searchAfter + cursor 控制器

public class CursorPaginator {
    private final IndexSearcher searcher;
    private final Query query;
    private final int pageSize;

    public TopDocs getFirstPage() throws IOException {
        return searcher.search(query, pageSize);
    }

    public TopDocs getNextPage(ScoreDoc cursor) throws IOException {
        return searcher.searchAfter(cursor, query, pageSize);
    }
}

@RestController
public class SearchController {
    @GetMapping("/search")
    public SearchResponse search(@RequestParam String q,
            @RequestParam(required=false) String cursor) {
        Query query = parseQuery(q);
        TopDocs topDocs;
        if (cursor == null) {
            topDocs = searcher.search(query, 20);
        } else {
            ScoreDoc cur = decodeCursor(cursor);
            topDocs = searcher.searchAfter(cur, query, 20);
        }
        String next = topDocs.scoreDocs.length == 20
            ? encodeCursor(topDocs.scoreDocs[19]) : null;
        return new SearchResponse(topDocs.scoreDocs, next);
    }
}

// 收益:
// page 100 / 1000 / 10000: 都是 20 ms
// 内存固定 O(pageSize)
// WAND 全程生效

15.19.3 反例 2:MultiCollector 失效 → 拆分查询

❌ 反例:同时收集 TopK + facet

public SearchResults search(Query query) throws IOException {
    TopScoreDocCollector topColl =
        TopScoreDocCollector.create(20, 1000);
    FacetsCollector facetsColl = new FacetsCollector();

    Collector multi = MultiCollector.wrap(topColl, facetsColl);

    searcher.search(query, multi);

    TopDocs topDocs = topColl.topDocs();
    Facets facets = new SortedSetDocValuesFacetCounts(state, facetsColl);

    return new SearchResults(topDocs, facets);
}

// 现象:
// FacetsCollector.scoreMode() = COMPLETE_NO_SCORES
// MultiCollector 取最严 → COMPLETE_NO_SCORES
// WAND 完全失效, TopK 退化为全扫
// 2 term SHOULD 查询 80 ms → 200 ms (慢 2.5×)

✅ 修复:并行执行两个独立查询

public SearchResults searchOptimized(Query query) {
    CompletableFuture<TopDocs> topF =
        CompletableFuture.supplyAsync(() -> {
            try {
                return searcher.search(query, 20);
            } catch (IOException e) {
                throw new UncheckedIOException(e);
            }
        }, executor);

    CompletableFuture<Facets> facetF =
        CompletableFuture.supplyAsync(() -> {
            try {
                FacetsCollector fc = new FacetsCollector();
                searcher.search(query, fc);
                return new SortedSetDocValuesFacetCounts(state, fc);
            } catch (IOException e) {
                throw new UncheckedIOException(e);
            }
        }, executor);

    TopDocs top = topF.join();
    Facets facets = facetF.join();
    return new SearchResults(top, facets);
}

// 收益:
// 主搜索 WAND 全程启用 → 30 ms
// facet 单独跑 (必须 COMPLETE), 并行 → ~80 ms
// 总时延 = max(30, 80) ≈ 80 ms (vs 串行 200 ms)
// 主搜索快 6×, 总查询快 2.5×

15.19.4 反例 3:自定义 Collector ScoreMode 错

❌ 反例:scoreMode 默认返回 COMPLETE

public class MyTopKCollector implements Collector {

        return ScoreMode.COMPLETE;   // ★ 错!
    }

    @Override
    public LeafCollector getLeafCollector(LeafReaderContext ctx) {
        return new LeafCollector() {
            @Override
            public void setScorer(Scorable s) {}
            @Override
            public void collect(int doc) {
                // 只取 TopK
            }
        };
    }
}

// 现象:
// ScoreMode = COMPLETE → WAND 不启用
// 实际只需 TopK, 但全扫
// 性能比 TopScoreDocCollector 差 5-10×

✅ 修复:声明 TOP_SCORES + setMinCompetitiveScore

public class MyTopKCollector implements Collector {
    private final int k;
    private final PriorityQueue<ScoreDoc> heap;

    @Override
    public ScoreMode scoreMode() {
        return ScoreMode.TOP_SCORES;  // ★ 关键!
    }

    @Override
    public LeafCollector getLeafCollector(LeafReaderContext ctx) {
        return new LeafCollector() {
            private Scorable scorer;

            @Override
            public void setScorer(Scorable s) throws IOException {
                this.scorer = s;
                if (heap.size() == k) {
                    s.setMinCompetitiveScore(heap.peek().score);
                }
            }

            @Override
            public void collect(int doc) throws IOException {
                float s = scorer.score();
                if (heap.size() < k) {
                    heap.add(new ScoreDoc(doc + ctx.docBase, s));
                    if (heap.size() == k) {
                        scorer.setMinCompetitiveScore(heap.peek().score);
                    }
                } else if (heap.peek().score < s) {
                    heap.poll();
                    heap.add(new ScoreDoc(doc + ctx.docBase, s));
                    scorer.setMinCompetitiveScore(heap.peek().score);
                }
            }
        };
    }
}

// 收益:
// ScoreMode = TOP_SCORES → WAND 启用
// 每次堆顶变化通知 Scorer
// 2 term SHOULD: 80 ms → 30 ms (2.7×)

public class TopKValidationTest {

    @Test
    public void testTopScoresEnablesWand() throws IOException {
        TopScoreDocCollector coll = TopScoreDocCollector.create(20, 1000);
        assertEquals(ScoreMode.TOP_SCORES, coll.scoreMode());
    }

    @Test
    public void testCompleteDisablesWand() throws IOException {
        TopScoreDocCollector topColl = TopScoreDocCollector.create(20, 1000);
        TotalHitCountCollector cntColl = new TotalHitCountCollector();
        Collector multi = MultiCollector.wrap(topColl, cntColl);
        assertEquals(ScoreMode.COMPLETE_NO_SCORES, multi.scoreMode());
    }

    @Test
    public void testSearchAfterStable() throws IOException {
        TopDocs page1 = searcher.search(query, 5);
        ScoreDoc last = page1.scoreDocs[4];
        TopDocs page2 = searcher.searchAfter(last, query, 5);
        TopDocs combined = combine(page1, page2);
        TopDocs page1to2 = searcher.search(query, 10);
        assertScoreDocsEqual(combined.scoreDocs, page1to2.scoreDocs);
    }

    @Test
    public void testTotalHitsRelation() throws IOException {
        TopDocs td = searcher.search(query, 20);
        if (td.totalHits.relation == TotalHits.Relation.GREATER_THAN_OR_EQUAL_TO) {
            int exact = searcher.count(query);
            assertTrue(exact >= td.totalHits.value);
        }
    }
}
