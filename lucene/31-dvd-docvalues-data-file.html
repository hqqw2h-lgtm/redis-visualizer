第 31 章 · .dvd DocValues Data 文件

列存数据通道 · 5 种 DocValuesType · BLOCK_SIZE=16384 · PackedInts · GCD/DELTA/TABLE/MONOTONIC · SORTED_SET 字典 + ords · 随机访问 RandomAccessInput · facet/sort/function score

.dvd 是 Lucene 的列存数据通道。当你写下 doc.add(new NumericDocValuesField("price", 99)),这条 long 最终落到 .dvd;当你写 SortedSetDocValuesField("tags", new BytesRef("nike")),字典字节与 ord 整数也落在 .dvd。.dvd 服务于排序、聚合、faceting、function score、collapse、join 等所有不依赖倒排的访问路径——这些场景都需要按 docID 拿"列值",而不是按 term 拿"行集合"。

本章按文件结构(Header → 5 种类型块 → Footer)拆解 .dvd,逐一展示字节布局、编码细节(GCD / DELTA / Table / Monotonic / Block PFOR)、写入路径(Lucene90DocValuesConsumer)、读取路径(RandomAccessInput + ord lookup)与性能特征。.dvd 由 5 种 DocValuesType 各自占用独立物理布局:NUMERIC(单值数字)、BINARY(单值字节序列)、SORTED(单值字符串带 ord)、SORTED_NUMERIC(多值数字)、SORTED_SET(多值字符串带 ord)。5 种类型对应 5 种完全不同的字节流,但共享同一份文件级 Header 与 Footer 框架。

  • 5 种 DocValuesType 各自的物理布局差异:NUMERIC 是 blocks[],BINARY 是 bytes + addresses,SORTED 是 ords + dict + termAddr,SORTED_SET 多一层 ord_addresses
  • NUMERIC 的 6 种编码模式:CONST / TABLE_COMPRESSED / GCD_COMPRESSED / DELTA_COMPRESSED / MONOTONIC / BLOCK_PFOR,以及 writer 如何根据 min/max/gcd/uniqueCount 自动选择
  • BLOCK_SIZE = 16384 为什么大于 .doc 的 128:批量扫描优化 vs 跳跃优化的取舍
  • BINARY 的 fixed-length 优化:minLen == maxLen 时省掉 addresses 表,直接 offset = base + docID * len 计算
  • SORTED 的三段结构:per-doc ord + terms dict + terms addr,以及 ord -> bytes 的两次 monotonic lookup
  • SORTED_SET 的四段结构:多了 ord_addresses[] 表,因为每个 doc 可能有多个 ord
  • 写入路径:DocValuesConsumer.addNumericField/addBinaryField/addSortedField/...,内部如何选择编码、写 docsWithField(IndexedDISI)、写 values
  • 读取路径:Lucene90DocValuesProducer + RandomAccessInput.readLong(pos) 实现 O(1) seek + block 解码
  • 三大应用:facet aggregation 用 SORTED_SET 的 ord 直接累加,sort 用 NUMERIC 的 advance,function score 用 NUMERIC 的随机访问
  • 9.x 紧凑字典 vs 8.x 老格式的字节占用差异:bitwidth 选择、jump table 引入、单值优化路径

前置知识:第 30 章 .dvm 元信息文件(本章假设你知道字段类型与编码模式由 .dvm 描述,.dvd 仅存数据);第 18 章 BitSet 与 IndexedDISI(理解 SPARSE/DENSE/ALL 三种 docID 集合编码);第 13 章 PackedInts(readBits、bitsPerValue、BlockPackedReader 是 .dvd 的底层数值编码工具)。

31.1 .dvd 角色与生命周期

.dvd 与倒排索引(.tim/.tip/.doc/.pos/.pay)是对偶的两条访问路径。倒排回答"哪些 doc 含 term X",DocValues 回答"doc Y 在字段 F 上的值是什么"。本节先把 .dvd 在体系中的位置、5 种类型的语义、生命周期讲清楚。

31.1.1 倒排 vs DocValues 的对偶

图 31-1-1 · 倒排索引与 DocValues 的访问路径对偶
列存的本质:倒排是"行 → 列"反转(每个 term 对应所有含它的 doc),DocValues 是不反转的列——每个字段是一个独立的列向量,按 docID 索引。倒排适合"找文档",列存适合"对找到的文档做计算"。两者都是必要的——单靠倒排无法做 ORDER BY price DESC(倒排只知道哪些 doc 有 price,不知道值的大小),单靠 DocValues 无法做 WHERE keyword MATCH 'apple'(列存全表扫太慢)。

31.1.4 .dvd 生命周期

图 31-1-2 · .dvd 文件生命周期
[写入阶段]
  1. Document 添加: doc.add(new NumericDocValuesField("price", 99))
  2. 内存 buffer: per-field 累积值 (DWPT 内 BufferedNumericDocValues)
  3. flush 时 Lucene90DocValuesConsumer 打开 .dvd
  4. 按字段顺序写入各类型块(选最优编码)
  5. finish() 写 Footer (magic + checksum)

[打开阶段]
  1. DirectoryReader.open 打开 segment
  2. Lucene90DocValuesProducer 打开 .dvm + .dvd
  3. 解析 .dvm 字段表 → 缓存到 Map<String, FieldEntry>
  4. .dvd 仅持有 IndexInput 句柄,不预加载

[查询阶段]
  1. Sort/Aggregation 触发 LeafReader.getSortedDocValues(field)
  2. Producer 按 .dvm 中 valuesOffset seek 到 .dvd
  3. 按 docID 解码 (block-by-block)
  4. 遵循 advance(target) 语义,不必顺序扫描

[合并阶段]
  1. SegmentMerger 创建新 .dvd
  2. 各 LeafReader 提供 DocValuesProducer
  3. 按 docID 顺序合并各字段
  4. SORTED / SORTED_SET 触发 ord 重映射

[删除阶段]
  1. .dvd 不保留删除信息(由 .liv 处理)
  2. 删除文档的 DocValues 仍占空间
  3. forceMerge 或 forceMergeDeletes 才真正回收

字段解读:
  magic        = 0x3FD76C17        (CodecUtil.CODEC_MAGIC,所有 Lucene 文件公用)
  codecName    = "Lucene90DocValuesData" (23 byte,与 .dvm 不同)
  version      = 0
  objectID     = SegmentInfo.getId() (16 byte,与 .dvm 严格一致)

31.3.3 IndexedDISI 编码

图 31-3-2 · IndexedDISI 自适应编码 4 模式

IndexedDISI(Indexed Doc ID Set Iterator)是 Lucene 用来记录"哪些 docID 有值"的位图编码,按 65536-doc block 划分,每 block 根据密度自动选择 EMPTY / SPARSE / DENSE / ALL 四种存储模式。详情见第 18 章。.dvd 中的 SPARSE 字段在 values 区前会先写一段 IndexedDISI,reader 通过 DISI 跳过没值的 doc。

电商场景:price vs discount 的密度差异

电商商品索引中 price 字段几乎所有 doc 都有(SKU 必填),DENSE 字段省 DISI bitmap。但 discount 仅大促期间约 10% 商品有,标记为 SPARSE,reader 在 numericDocValues.advance(docID) 时如果 DISI 命中说明该 doc 没 discount,直接跳到下一个有值的 doc。这避免了"无值 doc 占空间"的浪费——SPARSE 字段每 doc 仅 2 byte vs DENSE 每 doc ≥ 1 byte values。

31.5 BINARY 详解

BINARY 是单值 byte[],适合存 hash、bitmap、序列化的紧凑结构。它的核心优化是固定长度——当所有 doc 的 byte 长度相同时,可以省掉 addresses 表,直接用 offset = base + docID * len 计算位置。变长时用 MonotonicBlockPackedWriter 编码长度偏移表。

31.8.2 GCD_COMPRESSED 原理

图 31-8-2 · GCD_COMPRESSED 编码原理
GCD (Greatest Common Divisor) 编码:

原始数据(timestamp_ms, 100 doc, 精度到秒):
  values[0] = 1700000000000
  values[1] = 1700001000000
  values[2] = 1700001500000
  values[3] = 1700002000000
  ...

观察: 所有值都是 1000 的倍数(只精确到秒, 末尾 3 位 0)

GCD 计算:
  gcd = gcd(values[0], values[1], ..., values[99]) = 1000

编码:
  min = 1700000000000
  store: (v - min) / gcd
    enc[0] = (1700000000000 - 1700000000000) / 1000 = 0
    enc[1] = (1700001000000 - 1700000000000) / 1000 = 1000
    enc[2] = (1700001500000 - 1700000000000) / 1000 = 1500
    enc[3] = (1700002000000 - 1700000000000) / 1000 = 2000

bitsPerValue:
  原始 max - min = 100000000        → 27 bit
  GCD 编码后 max(enc) = 100000      → 17 bit
  节省 10 bit / doc

解码:
  v = min + enc[doc] * gcd

31.8.3 TABLE_COMPRESSED 原理

图 31-8-3 · TABLE_COMPRESSED 编码(uniqueValues ≤ 256)
字段示例(orderStatus, 100 doc):
  unique values: 6 个
    [PENDING, PAID, SHIPPED, DELIVERED, CANCELLED, REFUNDED]

table 构建:
  table[0] = PENDING
  table[1] = PAID
  table[2] = SHIPPED
  table[3] = DELIVERED
  table[4] = CANCELLED
  table[5] = REFUNDED

per-doc storage:
  每 doc 存 ord (3 bit, 因为 6 ≤ 8)
  doc 0: ord=2 (SHIPPED)
  doc 1: ord=3 (DELIVERED)
  doc 2: ord=1 (PAID)
  ...

总尺寸:
  table:  6 * 8 byte = 48 byte
  ords:   100 * 3 bit = 300 bit ≈ 38 byte
  合计:   86 byte

vs 不编码: 100 * 8 byte = 800 byte
压缩比:   9×

解码:
  ord = readBits(docID * 3, 3)
  value = table[ord]

31.8.5 9.x 紧凑 bitwidth vs 老格式

❌ 8.x 老格式: bitwidth 固定为 byte 对齐

字段 max - min = 100000  (需 17 bit)

8.x: bitwidth 必须是 8 / 16 / 24 / 32
     选 24 bit (向上取整到 byte 边界)

✅ 9.x 紧凑: 任意 bitwidth (1-64 bit)

每 doc 精确占用 17 bit

图 31-9-2 · BLOCK_PFOR chunk 字节布局
BLOCK_PFOR (block size = 16384):

  ┌──────────────────────────────────────┐
  │ Block 0 (doc 0..16383)              │
  │   Header                              │
  │     bitsPerValue : 1 byte (e.g. 18)  │
  │     numExceptions: 1 byte             │

  │ Block 2 (doc 32768..49151)          │
  │   ...                                 │
  └──────────────────────────────────────┘

Jump Table (跟在 blocks 末尾):
  blockOffsets[0] = .dvd 中 block 0 的字节偏移
  blockOffsets[1] = .dvd 中 block 1 的字节偏移
  blockOffsets[2] = .dvd 中 block 2 的字节偏移
  ...
  共 numBlocks 项 = ceil(numDocs / 16384)
  编码: MonotonicBlockPackedWriter

解码 docID = 50000:
  blockIdx = 50000 >> 14 = 3
  seek to blockOffsets[3]
  解码整个 block 3 (一次性 16384 个 long)
  indexInBlock = 50000 & 16383 = 752
  return decodedBlock[752]

性能:
  block 解码: 5-10 μs (CPU SIMD 友好的 unpack)
  缓存命中后顺序访问: 2 ns / doc

表 31-9-1 · BLOCK_PFOR 访问模式性能

public void addNumericField(FieldInfo field, DocValuesProducer values)
        throws IOException {

    long[] vals = collectAllValues(field, values);

    long min = Long.MAX_VALUE, max = Long.MIN_VALUE;
    long gcd = 0;
    LongHashSet uniques = new LongHashSet();
    for (long v : vals) {
        min = Math.min(min, v);
        max = Math.max(max, v);
        gcd = MathUtil.gcd(gcd, v - min);
        if (uniques.size() <= 256) {
            uniques.add(v);
        }
    }

    int format;
    if (min == max) {
        format = CONST;
    } else if (uniques.size() <= 256) {
        format = TABLE_COMPRESSED;
    } else if (gcd > 1 && gcd != Long.MAX_VALUE) {
        format = GCD_COMPRESSED;
    } else if (max - min < (1L << 48)) {
        format = DELTA_COMPRESSED;
    } else {
        format = BLOCK_PFOR;
    }

    writeNumericField(format, vals, min, max, gcd, uniques);

    writeFieldMetadata(field, format, min, max, gcd, /*offsets...*/);
}

public NumericDocValues getNumeric(FieldInfo field) throws IOException {
    NumericEntry entry = numerics.get(field.name);

    return new NumericDocValues() {
        long[] cachedBlock = null;
        int cachedBlockIdx = -1;
        int doc = -1;

        @Override
        public boolean advanceExact(int target) throws IOException {
            doc = target;
            int blockIdx = target >>> entry.blockShift;
            if (blockIdx != cachedBlockIdx) {
                long blockOffset = readJumpTable(entry, blockIdx);
                IndexInput in = data.slice("block",
                    entry.valuesOffset + blockOffset, entry.blockMaxLen);
                cachedBlock = decodeBlock(in, entry);
                cachedBlockIdx = blockIdx;
            }
            return true;
        }

        @Override
        public long longValue() {
            int indexInBlock = doc & ((1 << entry.blockShift) - 1);
            long raw = cachedBlock[indexInBlock];
            return entry.minValue + raw * entry.gcd;
        }

        @Override
        public int docID() { return doc; }
    };
}

31.12 Merge 与 ord 重映射

SORTED / SORTED_SET 字段 merge 时,各源 segment 的 ord 是独立编号的——同一个 "Nike" 在 segment A 是 ord=7,在 segment B 可能是 ord=12。merge 必须把所有源字典合并、重排,再生成新字典 + ord 重映射表,把每个 doc 的旧 ord 改成新 ord。这是 SORTED 字段 merge 的主要成本。

public OrdinalMap mergeOrdMap(SortedDocValues[] srcs) throws IOException {
    BytesRefHash mergedTerms = new BytesRefHash();
    for (SortedDocValues src : srcs) {
        for (int ord = 0; ord < src.getValueCount(); ord++) {
            mergedTerms.add(src.lookupOrd(ord));
        }
    }

    int[] sortedOrds = mergedTerms.sort();

    long[][] ordMaps = new long[srcs.length][];
    for (int i = 0; i < srcs.length; i++) {
        ordMaps[i] = new long[srcs[i].getValueCount()];
        for (int srcOrd = 0; srcOrd < srcs[i].getValueCount(); srcOrd++) {
            BytesRef term = srcs[i].lookupOrd(srcOrd);
            int mergedOrd = mergedTerms.find(term);
            ordMaps[i][srcOrd] = sortedOrds[mergedOrd];
        }
    }

    return new OrdinalMap(ordMaps, mergedTerms, sortedOrds);
}
大字典字段的 merge 成本

电商商品索引的 brand 字段约 5K unique,merge 1 GB segment 的 SORTED brand 字段约耗 30 秒(主要在 ord 重映射 + 字典前缀压缩)。如果错把 session_id(高基数 100M unique)用 SORTED,merge 一次可能 30 分钟 + OOM。这是 SORTED 字段最大的运维陷阱:基数必须低,高基数字段必须改用 hash + NUMERIC。

31.14.4 反例与正例

❌ 反例: 高基数字段用 SortedDocValues

doc.add(new SortedDocValuesField(
    "session_id",
    new BytesRef(uuid)));

现象:.dvd 文件 5 GB,forceMerge 30 分钟 OOM,sort by session_id 极慢。

✅ 正例: hash + 倒排

doc.add(new StringField("session_id", uuid, Store.NO));
doc.add(new NumericDocValuesField(
    "session_id_hash",
    MurmurHash3.hash64(uuid)));

收益:.dvd 8 MB(下降 600×),forceMerge 30 秒,精确查 session_id 走倒排,sort by hash 高效。

生产案例:从 SORTED 改 NUMERIC hash 解决 forceMerge OOM

某社交 feed 索引的 user_id 字段(50M unique UUID)误用 SortedDocValuesField。每次 forceMerge 触发字典合并,heap 飙到 32 GB OOM,merge 任务 30+ 分钟才崩。重构方案:user_id 走 StringField(倒排,精确等值查),DocValues 改用 NumericDocValuesField("user_id_hash", MurmurHash3.hash64(uid))。.dvd 字段从 5 GB 缩到 50 MB(64 bit hash, BLOCK_PFOR 后约 6 byte/doc),forceMerge 30 秒完成,sort by hash 替代了 sort by user_id(业务可接受 hash 顺序)。

不要在 collector 里全量加载 .dvd:写 SimpleCollector 时如果直接 numeric.advance(0) 后循环 numeric.longValue() 拉所有 doc 的值进 ArrayList,等于把整个 column 加载到 heap。1M doc * 8 byte = 8 MB 看起来不大,但 100 个字段 * 100 segment = 80 GB,直接 OOM。正确做法:用 iterator API 按需 advance,只在命中 doc 上取值。
