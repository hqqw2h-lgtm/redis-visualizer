第 33 章 · .dim Points 数据文件

BKD-tree · 叶 block 512 point · docID 4 编码模式 · Common Prefix 压缩 · IntersectVisitor 三态 · range 剪枝 · LatLonPoint · IpRange

.dim 是 Lucene 多维数值索引的叶节点数据文件,服务于 BKD-tree(Block KD-tree)的叶级存储。它支持地理坐标(2D)、IP 范围、时间戳范围、数值范围、向量近邻等场景,通过将点集递归二分到 256-512 point 的叶 block,既保持空间局部性,又能高效执行 range / contains / nearest 查询。

.dim 与 .dii(Points Index)配对工作:.dii 保存 BKD 内部节点(min/max envelope + 子节点指针),.dim 保存所有叶节点的实际 point 值与 docID 列表。reader 从 root 沿 .dii 下行,到叶节点时跳转到 .dim 读取该 block 的具体 point。本章拆解 .dim 的字节布局、BKD 叶 block 编码、common prefix 压缩、docID 编码、写入读取流程,以及在 1-8 维场景下的性能特征。

  • Points 适用 7 类场景:数值 range / IP 范围 / 时间戳 / 地理 2D / 向量 / IP v6 / 多维 hash
  • BKD-tree 与 B-tree 的差异:多维 split + 叶 block 集合 + envelope 剪枝
  • .dim 文件 3 段:Header / Leaf Blocks 序列 / Footer
  • 叶 block 4 部分:Block Header / docIDs / Common Prefix / Suffix Bytes
  • docID 4 种编码:SINGLE_VALUE / DELTA_VINT / BIT_24 / BIT_32 自动选择
  • Common Prefix 逐字节扫描提取最长共享前缀,典型压缩比 1.3-4×
  • BKD 写入复杂度 O(N log N),比 DocValues 慢 5× 但 range 查询快 50-100×
  • IntersectVisitor 三态:CELL_INSIDE_QUERY / CELL_CROSSES_QUERY / CELL_OUTSIDE_QUERY
  • 1% 选择性 range 查询从 100 ms 降到 1 ms(BKD 剪枝 99%)
  • Points 与 DocValues 不互替——同字段双索引才能同时满足 range filter + sort 场景

前置知识:第 12 章 BKD-tree 原理(理解递归二分、IntersectVisitor、叶 block 结构);第 1 章字段建模(LongPoint / LatLonPoint / IntPoint);第 13 章 DocValues(对比 .dvd 的 stream 优势)。读者需要熟悉 K-D tree、二分搜索与递归数据结构。

33.1.1 Points 适用场景

33.2 BKD-tree 概念

33.2.1 BKD-tree 结构

图 33-2-1 · BKD-tree 结构示意(2D 1024 point)
                           root
                      envelope [(0,0)-(180,90)]
                             /            \
                            /              \
                  dim=0 split @ 90    dim=0 split @ 90
                  left envelope        right envelope
                  [(0,0)-(90,90)]      [(90,0)-(180,90)]
                    /        \           /        \
            dim=1 split   dim=1 split  dim=1 split  dim=1 split
            @ 45 ...       @ 45         @ 45         @ 45
            ...
            leaf 0          leaf 1     leaf 2       leaf 3
            [256 point]    [256 point] [256 point]  [256 point]

每个内部节点:
  - dim:        在哪一维 split (round-robin 切换)
  - splitValue:  split 阈值
  - 子节点指针:   在 .dii 中
  - envelope:    子树覆盖的 min/max box

每个叶节点:
  - 256-512 point (.dim 中存)
  - per-point: docID + value (each dim)

33.2.2 BKD 关键参数

表 33-2-1 · BKD 关键参数
参数默认值含义
MAX_POINTS_IN_LEAF_NODE512叶 block 最大 point 数
MAX_DIMS8最大维度数
MAX_INDEX_DIMS8用于 split 的最大维度

splitDim 选择策略round-robin各维度循环 split

33.2.3 BKD vs B-tree 差异

B-tree

  • 1D 索引(单 key)
  • 叶节点 1 record
  • 仅按 key 二分比较
  • 难支持 K 维范围

BKD-tree

  • 多维 split,各维循环 split
  • 叶 block 256-512 point,更大 block 减少树高
  • envelope 加速剪枝,整子树跳过
  • 1-8 维 range / contains / nearest 都高效

33.3.4 典型 .dim 大小估算

图 33-3-3 · 1M doc LongPoint 的 .dim 大小
字段:LongPoint("timestamp", 1M doc, 1 维, 8 byte/dim)

BKD 结构:
  叶 block 数 = ceil(1M / 512) = 1953 个
  每叶 block 编码:
    Header                          : ~ 30 byte
    docIDs (delta + packed)         : 512 × 1.5 = 768 byte
    values (无 prefix 压缩时)       : 512 × 8   = 4096 byte
    经 prefix 压缩                  : ~ 1500 byte
                                    ─────────
    合计                            : ~ 2300 byte

.dim 总大小:
  Header + 1953 × 2300 + Footer
  = 50 + 4.5 MB + 16
  = 4.5 MB

────────────────────────────────────────
对比 numeric DocValues:
  .dvd: ~ 2.5 MB  (顺序 stream 友好)
  .dim: ~ 4.5 MB  (含树结构开销, 大 80%)

价值:
  .dvd 不能高效 range 查询 (扫全部 50-100 ms)
  .dim 支持 sub-millisecond range 查询 (1 ms)
  ★ 大 80% 磁盘 → 100× range 加速

33.6 Common Prefix 压缩

33.6.1 Common Prefix 原理

图 33-6-1 · LongPoint timestamp 字段的 Common Prefix 提取
block 内 256 个 timestamp, 假设值范围:
  最小:1700001000000 = 0x000018C66B7C3F00 (8 byte)
  最大:1700001999999 = 0x000018C66B8BCA01 (8 byte)

逐字节对比所有 point:
  byte 0:  全 0x00     ← 共享 ✓
  byte 1:  全 0x00     ← 共享 ✓
  byte 2:  全 0x00     ← 共享 ✓
  byte 3:  全 0x18     ← 共享 ✓
  byte 4:  全 0xC6     ← 共享 ✓
  byte 5:  全 0x6B     ← 共享 ✓
  byte 6:  从 0x7C 到 0x8B  ← 不共享 ✗
  byte 7:  从 0x3F 到 0x01  ← 不共享 ✗

commonPrefixLength[0] = 6 (前 6 byte 共享)
commonPrefixBytes     = 0x000018C66B (6 byte)

每 point suffix:
  byte 6, byte 7 → 2 byte/point

256 point 的存储:
  prefix:  6 byte (共享一次)
  suffix:  256 × 2 = 512 byte
                   ─────────
  合计:    518 byte

未压缩:
  256 × 8 = 2048 byte

★ 压缩比 2048 / 518 = 4×

33.11.3 反例 / 正例对比

❌ 反例:128 维向量用 BinaryPoint + BKD

doc.add(new BinaryPoint("vector", 128 dim float[]));
// 128 维 BKD, 叶 block 内 point 完全无 prefix 共享

// 现象:
// .dim 文件极大 (无压缩)
// KNN search 慢 (BKD 不擅长高维)
// OOM 风险
// 维度灾难: 高维下 BKD 几乎退化为线扫

✅ 修复:用 KnnFloatVectorField + HNSW

doc.add(new KnnFloatVectorField("vector",
    float[128], cosine));
// Lucene 9.0+ HNSW, 专为高维设计

// 收益:
// .vec 文件紧凑
// KNN search ms 级
// 完全 off-heap
// 召回率可调 (ef 参数)
// 详见第 16 章 HNSW 向量检索
临界点:维度 ≤ 8 用 BKD(.dim);维度 ≥ 16 用 HNSW(.vec);8-16 之间看数据分布。不要用 BinaryPoint 模拟向量索引,即使能跑通,KNN 性能差 100×,且没有召回率保证。
