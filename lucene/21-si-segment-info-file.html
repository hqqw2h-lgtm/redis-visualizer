第 21 章 · .si 段元信息文件

SegmentInfo · Lucene90SegmentInfoFormat · Diagnostics · attributes · objectID · IndexSort · SegmentCommitInfo · CheckIndex · 文件命名 · Snapshot 备份

运维误删了 _3.si 这一个 600 字节的小文件,整个 12 GiB 索引启动直接 CorruptIndexException;数据科学同学想知道某个 segment 是 flush 还是 merge 来的,只看 du -sh 看不出来——这两件事都指向同一个文件:每个 segment 目录下那个静悄悄的 .si。它体积小到几乎可以忽略,却是该 segment 的"身份档案":没有 .si 整个 segment 不可用;读懂 .si 就读懂了这个 segment 的诞生时刻、版本、文档总数、指向的所有数据文件名、甚至当时的 JVM 版本与操作系统。

一个 segment 由 14 个左右的物理文件组成,但 segments_N 不会去逐一引用它们——它只引用 .si。.si 才是 segment 的"目录入口":列出该 segment 拥有的所有文件名(.tim/.tip/.doc/.fdt/...)、记录 maxDoc、声明 isCompoundFile、保存 16 字节的 objectID、附上写入时刻的 diagnostics 元数据、可选携带 indexSort 排序键、再用 4+8 字节的 footer checksum 自我校验。它由 Lucene90SegmentInfoFormat 在 flush/merge 完成时写出,一旦写完就再不修改一个字节——所有"删除"、"字段更新"、"DocValues 更新"全部由 segments_N 上的 SegmentCommitInfo 用 generation 号代理完成。本章把 .si 从字节级布局到 Codec 注册、从 SegmentInfo Java 对象到 Lucene90SegmentInfoFormat 读写实现、从 Diagnostics 调试到 attributes 自定义、从单文件损坏到 Snapshot 备份策略,完整摊开。

  • .si 在 segment 文件群里的"目录入口"角色:segments_N 引用 .si,.si 列出所有数据文件
  • .si 字节级布局:header(magic+codec+version+objectID) → maxDoc → isCompound → diagnostics → files → attributes → indexSort? → footer
  • SegmentInfo 8 个核心字段:name / version / minVersion / maxDoc / isCompoundFile / id / files / diagnostics / attributes / indexSort
  • Codec → SegmentInfoFormat 的版本路由:Codec.forName("Lucene99").segmentInfoFormat() 返回 Lucene90SegmentInfoFormat
  • Diagnostics 12 个标准 key:source / lucene.version / java.version / os / timestamp / mergeMaxNumSegments / ... — 是 segment 来源审计的金矿
  • attributes 与 diagnostics 的边界:attributes 是 Codec 自定义可读取的 KV,diagnostics 是写一次永不读取的元数据
  • SegmentInfo 不可变 vs SegmentCommitInfo 可变:delGen / fieldInfosGen / dvGen / softDelCount 全部住在 SegmentCommitInfo 而非 SegmentInfo
  • 文件命名规则三段式:<segName>.<suffix> / <segName>.cfs / <segName>_<gen>.<suffix>
  • 启动路径:从最大 segments_N 找 segName → 用 segmentID 校验打开 .si → 读出 files 集 → 读其余 14 个文件
  • 故障恢复:CheckIndex -exorcise 是 .si 损坏的唯一武器,代价是该 segment 全部文档丢失

21.1 segment 文件群与 .si 入口

打开任意一个 Lucene 索引目录,你看到的不是"一个大文件",而是一群按 _0 / _1 / _2 / ... 编号分组的小文件——每个数字前缀就是一个 segment。一个 segment 通常由 12-16 个文件组成,扩展名各异(.si / .fnm / .tim / .tip / .doc / .pos / .pay / .fdt / .fdx / .fdm / .dvd / .dvm / .dim / .dii / .nvd / .nvm)。这堆文件靠什么"粘"成一个逻辑 segment?答案是 .si:它列出 segment 拥有的所有文件名,是这个 segment 的"目录入口"。

segments_N → .si → 14 个数据文件

图 21-1 · segments_N、.si、segment 文件群的三层引用关系

这张图揭示了一个反直觉的事实:segments_N 并不直接引用 .tim/.doc/.fdt 这些数据文件。它只引用 .si——剩下的所有文件名,都从 .si 的 files 集合里读取。换句话说,.si 是单 segment 的"清单文件",而 segments_N 是"清单的清单"。这种两层间接给了 Lucene 三个好处:① segment 内部文件命名变化(比如 9.x 引入 .vec/.vem 向量文件)不影响 segments_N 协议;② 单个 segment 可以被原子地"打包"成一个 .si——只要 .si 写完整,所有数据文件就被"挂上"了;③ SegmentCommitInfo 上的 generation 号(delGen / fieldInfosGen / dvGen)只描述"差异",原始文件清单永远来自 .si。

21.2 .si 字节级布局

把 _0.si 用 xxd 打开,你会看到一段紧凑的二进制——每个字节都有意义。Lucene90SegmentInfoFormat 把它分成七个段:header / Lucene 版本 / minVersion / 元数据(maxDoc + isCompound + diagnostics) / files / attributes / indexSort? / footer。下面是一张定制 byte map:

.si byte layout 详细图

图 21-2 · _0.si 字节级布局(物理顺序)
① Index Header (~30 B,固定起点)
magic(4 B) = 0x3FD76C17 — Lucene 通用魔数
codecName(VInt+UTF8) = "Lucene90SegmentInfo" — 19 字节字符串
version(int 4 B) = VERSION_CURRENT — 当前为 1
objectID(16 B) — 该 segment 的 UUID,与 segments_N 中 segmentID 必须一致
suffix(VInt+UTF8) = "" — 通常为空字符串
② Lucene Version (12 B)
major(int 4 B) · minor(int 4 B) · bugfix(int 4 B)
例:9 / 10 / 0 表示 Lucene 9.10.0
③ Min Version (1 B + 可选 12 B)
hasMinVersion(byte) = 0 或 1 — 是否有 minVersion
若 1:major / minor / bugfix 三个 int — 可读取该 segment 的最低 Lucene 版本
④ 段元数据 (变长,通常 50-200 B)
maxDoc(int 4 B) — 含已删除文档总数,单 segment 最大约 21 亿
isCompoundFile(byte) = 0x00 或 0x01
diagnostics: writeMapOfStrings(Map<String,String>) — 写入时元数据
⑤ Files Set (变长,主要体积来源)
writeSetOfStrings(Set<String>)
VInt(numFiles) + 每个文件名 VInt+UTF8
典型 13-16 个文件名,约 200-400 B
⑥ Attributes (变长,常为空)
writeMapOfStrings(Map<String,String>)
Codec 私有 KV,如 Lucene99HnswVectorsFormat.numConnections=16
⑦ IndexSort (可选,1 B + 可选若干字节)
numSortFields(VInt) — 0 表示无 sort
每个字段:fieldName + typeAsByte + reverseAsByte + missingValue
⑧ Footer (16 B,固定终点)
FOOTER_MAGIC(int 4 B) = 0xC02893E8
algorithmID(int 4 B) = 0
checksum(long 8 B) — header→indexSort 全部内容的 CRC32(XXHash64)

21.3 SegmentInfo 字段语义

字节级布局是物理表示;在 Java 内存里,这些字节会被反序列化为 org.apache.lucene.index.SegmentInfo 对象。这个类有 8 个核心字段(加上 directory / codec 两个运行时引用),它们一对一映射到 .si 的字节段。

SegmentInfo Java 类核心代码

package org.apache.lucene.index;

import java.util.Map;
import java.util.Set;
import org.apache.lucene.codecs.Codec;
import org.apache.lucene.search.Sort;
import org.apache.lucene.store.Directory;
import org.apache.lucene.util.Version;

public final class SegmentInfo {

    public static final int NO = -1;
    public static final int YES = 1;

    public final String name;
    private final int maxDoc;
    public final Directory dir;
    private boolean isCompoundFile;
    private final byte[] id;             // 16 字节 objectID
    private Codec codec;                  // 运行时绑定,不写入 .si
    private Map diagnostics;
    private Set setFiles;
    private final Map attributes;
    private final Sort indexSort;
    private Version version;
    private Version minVersion;

    public SegmentInfo(Directory dir, Version version, Version minVersion,
                       String name, int maxDoc, boolean isCompoundFile,
                       Codec codec, Map diagnostics,
                       byte[] id, Map attributes,
                       Sort indexSort) {
        this.dir = Objects.requireNonNull(dir);
        this.version = Objects.requireNonNull(version);
        this.minVersion = minVersion;
        this.name = Objects.requireNonNull(name);
        this.maxDoc = maxDoc;
        this.isCompoundFile = isCompoundFile;
        this.codec = codec;
        this.diagnostics = Map.copyOf(Objects.requireNonNull(diagnostics));
        this.id = id;
        if (id != null && id.length != StringHelper.ID_LENGTH) {
            throw new IllegalArgumentException("invalid id; got " + id.length);
        }
        this.attributes = Map.copyOf(Objects.requireNonNull(attributes));
        this.indexSort = indexSort;
    }

    public int maxDoc() { return maxDoc; }
    public Set files() { return setFiles; }
    public byte[] getId() { return id == null ? null : id.clone(); }
    public Sort getIndexSort() { return indexSort; }
    public Map getDiagnostics() { return diagnostics; }
    public Map getAttributes() { return attributes; }

    public void setFiles(Set files) {
        checkFileNames(files);
        this.setFiles = Set.copyOf(files);
    }

    private void checkFileNames(Set files) {
        for (String file : files) {
            if (!IndexFileNames.parseSegmentName(file).equals(name)) {
                throw new IllegalArgumentException(
                    "invalid file name: " + file + " for segment: " + name);
            }
        }
    }
}

21.4 Codec 版本注册与演进

SegmentInfo 字段是稳定的 API,但字节布局取决于 Codec 版本。Lucene 9.x 当前的 Codec 是 Lucene99Codec,它的 segmentInfoFormat() 返回 Lucene90SegmentInfoFormat(注意:Codec 名是 99,SegmentInfoFormat 名是 90,因为 .si 格式从 9.0 起就稳定了)。这种"Codec 版本号 ≠ Format 版本号"的设计让 Codec 可以演进新功能(如向量),而 .si 格式保持兼容。

Lucene86 SI vs Lucene90 SI 字节差异

Lucene86SegmentInfoFormat (Lucene 8.6-8.x)
  • codecName = "Lucene86SegmentInfo" (18 B)
  • VERSION_START=0,VERSION_CURRENT=0
  • 不保存向量字段(.vec/.vem 不存在)
  • files 集合典型 13 个文件
  • indexSort 编码:numFields VInt + 每字段 fieldName + provider + reverse + missingByte
  • attributes 用法少(几乎总为空 Map)
  • 典型 .si 大小 400-500 B
Lucene90SegmentInfoFormat (Lucene 9.0-9.10+)
  • codecName = "Lucene90SegmentInfo" (19 B)
  • VERSION_START=0,VERSION_CURRENT=1(8.x 写出的 .si 也能读)
  • 支持向量字段(.vec/.vem/.vex 进 files 集合)
  • files 集合典型 14-17 个文件
  • indexSort 编码兼容 8.6 + 增加 typeAsByte 扩展
  • attributes 用法增加(如 HnswVectorsFormat 的 numConnections)
  • 典型 .si 大小 500-700 B

"Codec 版本号 ≠ SegmentInfoFormat 版本号" 是有意设计 — 倒排格式、向量格式每个 minor 版本可能升级(Lucene90PostingsFormat → Lucene99PostingsFormat),但 .si 元信息格式从 9.0 至今保持稳定。这让 Lucene 可以在不动 .si 协议的情况下迭代其他文件格式——避免每次升级都要写 backward-codecs 兼容层。.si 由 8.6 升到 9.0 时确实改了 codec name(Lucene86SegmentInfo → Lucene90SegmentInfo),所以读旧索引时会触发 backward_codecs 包加载。生产中升级 Lucene 9.x → 9.y 几乎从不需要重写 .si。

21.5 SegmentInfoFormat.write

每次 flush 或 merge 完成,Lucene90SegmentInfoFormat.write 会被调用一次,生成 _N.si。整个写入是顺序追加——按 header → version → maxDoc → diagnostics → files → attributes → indexSort → footer 的物理顺序写入,不需要回头修改任何字节。这种"一次性写完"特性让 .si 天然适合 page cache,首次读取后近乎免费。

public final class Lucene90SegmentInfoFormat extends SegmentInfoFormat {

    public static final String SI_EXTENSION = "si";
    static final String CODEC_NAME = "Lucene90SegmentInfo";
    static final int VERSION_START = 0;
    static final int VERSION_CURRENT = 1;

    @Override
    public void write(Directory dir, SegmentInfo si, IOContext ioContext)
            throws IOException {
        final String fileName = IndexFileNames.segmentFileName(
            si.name, "", SI_EXTENSION);
        try (IndexOutput output = dir.createOutput(fileName, ioContext)) {
            CodecUtil.writeIndexHeader(output, CODEC_NAME, VERSION_CURRENT,
                si.getId(), "");
            Version version = si.getVersion();
            output.writeInt(version.major);
            output.writeInt(version.minor);
            output.writeInt(version.bugfix);

            Version minVersion = si.getMinVersion();
            if (minVersion != null) {
                output.writeByte((byte) 1);
                output.writeInt(minVersion.major);
                output.writeInt(minVersion.minor);
                output.writeInt(minVersion.bugfix);
            } else {
                output.writeByte((byte) 0);
            }

            output.writeInt(si.maxDoc());
            output.writeByte((byte) (si.getUseCompoundFile() ? SegmentInfo.YES : SegmentInfo.NO));
            output.writeMapOfStrings(si.getDiagnostics());

            Set files = si.files();
            for (String file : files) {
                if (!IndexFileNames.parseSegmentName(file).equals(si.name)) {
                    throw new IllegalArgumentException(
                        "invalid files: expected segment=" + si.name +
                        ", got=" + file);
                }
            }
            output.writeSetOfStrings(files);
            output.writeMapOfStrings(si.getAttributes());

            Sort indexSort = si.getIndexSort();
            int numSortFields = (indexSort == null) ? 0 : indexSort.getSort().length;
            output.writeVInt(numSortFields);
            if (indexSort != null) {
                for (SortField sortField : indexSort.getSort()) {
                    writeSortField(output, sortField);
                }
            }

            CodecUtil.writeFooter(output);
        }
    }
}

@Override
public SegmentInfo read(Directory dir, String segName, byte[] segmentID, IOContext ctx)
        throws IOException {
    final String fileName = IndexFileNames.segmentFileName(segName, "", SI_EXTENSION);
    try (ChecksumIndexInput input = dir.openChecksumInput(fileName, ctx)) {
        Throwable priorE = null;
        SegmentInfo si = null;
        try {
            int format = CodecUtil.checkIndexHeader(input, CODEC_NAME,
                VERSION_START, VERSION_CURRENT, segmentID, "");

            final Version version = Version.fromBits(
                input.readInt(), input.readInt(), input.readInt());

            byte hasMinVersion = input.readByte();
            final Version minVersion;
            if (hasMinVersion == 1) {
                minVersion = Version.fromBits(
                    input.readInt(), input.readInt(), input.readInt());
            } else if (hasMinVersion == 0) {
                minVersion = null;
            } else {
                throw new CorruptIndexException(
                    "Illegal boolean value " + hasMinVersion, input);
            }

            final int docCount = input.readInt();
            if (docCount < 0) {
                throw new CorruptIndexException("invalid docCount: " + docCount, input);
            }

            final boolean isCompoundFile = input.readByte() == SegmentInfo.YES;
            final Map diagnostics = input.readMapOfStrings();
            final Set files = input.readSetOfStrings();
            final Map attributes = input.readMapOfStrings();

            int numSortFields = input.readVInt();
            Sort indexSort;
            if (numSortFields > 0) {
                SortField[] sortFields = new SortField[numSortFields];
                for (int i = 0; i < numSortFields; i++) {
                    sortFields[i] = readSortField(input);
                }
                indexSort = new Sort(sortFields);
            } else if (numSortFields == 0) {
                indexSort = null;
            } else {
                throw new CorruptIndexException(
                    "invalid index sort field count: " + numSortFields, input);
            }

            si = new SegmentInfo(dir, version, minVersion, segName, docCount,
                isCompoundFile, null /*codec 由调用者设*/,
                diagnostics, segmentID, attributes, indexSort);
            si.setFiles(files);
        } catch (Throwable exception) {
            priorE = exception;
        } finally {
            CodecUtil.checkFooter(input, priorE);
        }
        return si;
    }
}

21.9 SegmentInfo vs SegmentCommitInfo

Lucene 用两个 Java 类描述一个 segment 的元信息:SegmentInfo(住在 .si)与 SegmentCommitInfo(住在 segments_N)。它们的边界精确反映了 Lucene 的"主体不可变 + 删除增量"设计——前者一旦写完永远不变,后者每次 commit 都重写。

两个类的边界对比

SegmentInfo (.si 文件 / 不可变)
  • 写入位置:_N.si
  • 生命周期:flush / merge 时写一次,永不修改
  • 字段:name / version / minVersion / maxDoc / isCompoundFile / id / files / diagnostics / attributes / indexSort
  • 含义:segment "本质" — 它有多少文档、由哪些文件组成、什么版本写的
  • 写入触发:SegmentInfoFormat.write
  • 大小:典型 600 B
  • 引用计数:IndexFileDeleter 跟踪 .si 与 files 集合
SegmentCommitInfo (segments_N 行 / 可变)
  • 写入位置:segments_N 中一行
  • 生命周期:每次 commit 重写整个 segments_N
  • 字段:info(指向 SegmentInfo) + delGen / fieldInfosGen / dvGen / softDelCount / nextWriteFieldInfosGen / nextWriteDocValuesGen / fieldInfosFiles / dvUpdatesFiles
  • 含义:segment "当前状态" — 删了多少、字段更新到第几代、DV 更新到第几代
  • 写入触发:SegmentInfos.commit
  • 大小:典型 60-150 B / segment
  • 引用计数:跟踪所有 gen 文件(.liv / _N_K.fnm / _N_K.dvd)

    public final SegmentInfo info;            // 指向 .si 的 SegmentInfo

    private long delGen;                       // .liv 当前代
    private int delCount;
    private long fieldInfosGen;                // .fnm 当前代
    private long docValuesGen;                 // .dvd 当前代
    private int softDelCount;
    private long nextWriteFieldInfosGen;
    private long nextWriteDocValuesGen;
    private final Set fieldInfosFiles;
    private final Map> dvUpdatesFiles;

    public Collection files() throws IOException {
        Collection files = new HashSet<>(info.files());

        // 加上当前 generation 的 .liv 文件
        if (delGen != -1) {
            files.add(IndexFileNames.fileNameFromGeneration(
                info.name, IndexFileNames.LIV_EXTENSION, delGen));
        }

        // 加上 .fnm 增量
        files.addAll(fieldInfosFiles);

        // 加上 DocValues 更新文件
        for (Set dvFiles : dvUpdatesFiles.values()) {
            files.addAll(dvFiles);
        }

        return files;
    }

    public long getDelGen()         { return delGen; }
    public int getDelCount()        { return delCount; }
    public long getFieldInfosGen()  { return fieldInfosGen; }
    public long getDocValuesGen()   { return docValuesGen; }
    public int getSoftDelCount()    { return softDelCount; }
}

命名规则代码 IndexFileNames

public final class IndexFileNames {

    public static final String SEGMENTS = "segments";
    public static final String PENDING_SEGMENTS = "pending_segments";
    public static final String LIV_EXTENSION = "liv";
    public static final String CFS_EXTENSION = "cfs";
    public static final String CFE_EXTENSION = "cfe";

    public static final char SEPARATOR = '_';

    public static String segmentFileName(String segmentName, String segmentSuffix,
                                          String ext) {
        if (ext.length() > 0 || segmentSuffix.length() > 0) {
            assert !ext.startsWith(".");
            StringBuilder sb = new StringBuilder(segmentName.length() + 2 +
                segmentSuffix.length() + ext.length());
            sb.append(segmentName);
            if (segmentSuffix.length() > 0) {
                sb.append(SEPARATOR).append(segmentSuffix);
            }
            if (ext.length() > 0) {
                sb.append('.').append(ext);
            }
            return sb.toString();
        }
        return segmentName;
    }

    public static String fileNameFromGeneration(String base, String ext, long gen) {
        if (gen == -1) {
            return null;
        } else if (gen == 0) {
            return segmentFileName(base, "", ext);
        } else {
            assert gen > 0;
            StringBuilder res = new StringBuilder()
                .append(base)
                .append(SEPARATOR)
                .append(Long.toString(gen, Character.MAX_RADIX));
            if (ext.length() > 0) {
                res.append('.').append(ext);
            }
            return res.toString();
        }
    }

    public static String parseSegmentName(String filename) {
        int idx = indexOfSegmentName(filename);
        return (idx == -1) ? filename : filename.substring(0, idx);
    }

    public static long parseGeneration(String filename) {
        int dotIndex = filename.indexOf('.');
        int separatorIndex = filename.indexOf(SEPARATOR, 1);
        if (separatorIndex == -1 || separatorIndex >= dotIndex) {
            return 0;
        }

21.15 Snapshot 与备份

.si 文件名一旦写出永不重写——这种"绝对不可变"特性让增量备份近乎免费:基于 mtime 或 inode 比较即可。但备份必须用 SnapshotDeletionPolicy 锁定 commit 期间引用的所有文件,否则备份过程中 merge 会把 .si 引用的数据文件提前删除,造成 dangling reference。

public class SnapshotJob {
    public static void main(String[] args) throws Exception {
        Path indexDir = Paths.get(args[0]);
        Path backupDir = Paths.get(args[1]);
        Files.createDirectories(backupDir);

        try (FSDirectory dir = FSDirectory.open(indexDir)) {
            IndexWriterConfig cfg = new IndexWriterConfig();
            cfg.setIndexDeletionPolicy(new SnapshotDeletionPolicy(
                new KeepOnlyLastCommitDeletionPolicy()));

            try (IndexWriter writer = new IndexWriter(dir, cfg)) {
                SnapshotDeletionPolicy sdp = (SnapshotDeletionPolicy)
                    writer.getConfig().getIndexDeletionPolicy();

                IndexCommit commit = sdp.snapshot();
                try {
                    System.out.printf("Snapshot taken: gen=%d, %d files%n",
                        commit.getGeneration(),
                        commit.getFileNames().size());

                    for (String filename : commit.getFileNames()) {
                        Path src = indexDir.resolve(filename);
                        Path dst = backupDir.resolve(filename);
                        Files.copy(src, dst, StandardCopyOption.REPLACE_EXISTING);
                        System.out.printf("  copied %s (%.2f KiB)%n",
                            filename, Files.size(src) / 1024.0);
                    }

                    // 写校验和
                    Files.write(backupDir.resolve("SHA256SUMS"),
                        commit.getFileNames().stream()
                            .map(f -> sha256(backupDir.resolve(f)) + "  " + f)
                            .collect(Collectors.toList()));
                } finally {
                    sdp.release(commit);
                }
            }
        }
    }

    static String sha256(Path p) {
        try (InputStream in = Files.newInputStream(p)) {
            MessageDigest md = MessageDigest.getInstance("SHA-256");
            byte[] buf = new byte[8192];
            int n;
            while ((n = in.read(buf)) > 0) md.update(buf, 0, n);
            return HexFormat.of().formatHex(md.digest());
        } catch (Exception e) {
            throw new RuntimeException(e);
        }
    }
}
