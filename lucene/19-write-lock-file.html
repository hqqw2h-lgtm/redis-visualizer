第 19 章 · write.lock 文件

独占写共享读 · LockFactory 体系 · NativeFSLock 字节布局 · obtainLock 调用链 · close 释放 · 崩溃残留 · NFS / 容器 / Kubernetes 陷阱

write.lock 是 Lucene 索引目录中最简单也最关键的一个文件。它通常只有 0 字节,文件本身没有内容,但它的"存在 + 文件锁状态"承担着一个不可妥协的语义边界:同一个目录在任意时刻只允许一个 IndexWriter 实例。一旦这条规则被打破——哪怕仅仅是几秒钟内有两个进程同时持有索引——segments_N 提交点就会被串改,倒排数据会出现"半截 segment",索引在 reader 视角下表现为损坏,只能整体重建。本章把 write.lock 拆开看:它在 NativeFSLockFactory / SimpleFSLockFactory / VerifyingLockFactory 三种工厂下分别长什么样、文件物理布局是什么、JVM 崩溃后会不会残留、LockObtainFailedException 的 6 种典型成因怎么排查。

这是一篇"文件级"小章节——内容相对短,但生产事故密度极高。订单系统迁移到 Kubernetes 时把 PVC 误配成 ReadWriteMany、容器从 emptyDir 拉起时发现索引不见了、运维 SSH 上去 rm /index/write.lock 然后造成双进程写入导致数据彻底损坏——这些故事都从 write.lock 这一个文件开始。理解了这一章,你会明白为什么 Lucene 官方文档反复警告"不要在 NFS 上运行 IndexWriter"、为什么 NativeFSLockFactory 是默认且唯一推荐的实现、为什么 kill -9 之后 NativeFSLock 不需要任何手动清理、为什么 SimpleFSLock 在 Lucene 9.x 已经被标记为不推荐。

  • 独占写、共享读语义:为什么 IndexReader 不需要锁,而 IndexWriter 必须持有 write.lock
  • LockFactory 五种实现:NativeFSLockFactory(默认)/ SimpleFSLockFactory(已不推荐)/ VerifyingLockFactory(测试装饰器)/ SingleInstanceLockFactory(RAM)/ NoLockFactory(严禁生产)
  • NativeFSLock 工作机制:FileChannel.tryLock() 委托给 fcntl(F_SETLK) / LockFileEx / flock,锁状态在 kernel,不在文件内容
  • write.lock 字节级布局:NativeFS 模式下文件大小 0 byte / VerifyingLock 30-50 byte 含 magic + pid + hostname + instanceID
  • obtainLock 调用栈:new IndexWriter(dir, cfg) → dir.obtainLock("write.lock") → LockFactory.obtainLock → FileChannel.tryLock
  • JVM 崩溃恢复行为差异:NativeFSLock 由 OS 自动释放(进程退出 → close fd → release lock),SimpleFSLock 必须手动 rm 残留文件
  • LockObtainFailedException 六大成因:其他 IW 持有 / 陈旧 SimpleFS 锁 / NFS 锁丢失 / 容器 tmpfs 重启 / Pod 调度 / 误删后重启
  • NFS 与 Kubernetes 部署清单:为什么必须用 ReadWriteOnce PVC、为什么 emptyDir 和 ReadWriteMany 都是死路

19.1 锁的语义与作用

Lucene 的并发模型一句话总结就是"写一读多"(single writer, multiple readers)。write.lock 实现的就是这条模型的物理边界——它独占写、不限制读。同一目录任意时刻只允许一个 IndexWriter,但允许任意多 IndexReader / SearcherManager / CheckIndex(只读模式)同时打开。读路径完全不和锁打交道,这是 Lucene 写一读多模型可以无限横向扩展读节点的根本原因。

进程边界:同进程多线程 vs 跨进程

写一读多的"写"指的是实例数,不是线程数。在同一个 JVM 进程内,一个 IndexWriter 实例是线程安全的——多线程可以并发调用 addDocument / updateDocument / deleteDocuments,内部通过 DocumentsWriterPerThreadPool 把请求分派到不同的 DWPT 实现无锁并发写入。但跨进程,即使是同一台机器的两个 JVM,也必须通过 write.lock 互斥。

"为什么读不需要锁" — Lucene 的 segment 是不可变的(immutable),一旦 flush 落盘,文件内容永不变更。新写入只产生新 segment,旧 segment 永远不会被原地覆盖,这是"读不需要任何锁"的物理基础。读端只需要原子性地读取 segments_N 这一个文件(它通过 atomic rename 切换),拿到当前 commit point 引用的所有 segment 名,然后 mmap 这些不可变文件就完事——没有任何写入会从下面抽走数据。这种"copy-on-write 风格的 segment 不可变性"配合 write.lock 的独占写,共同构成了 Lucene 高并发读的核心保证。

19.2 LockFactory 抽象

Lucene 把"如何取锁"抽象为 LockFactory,具体由五个实现分担不同环境下的语义需求。NativeFSLockFactory 是默认且唯一推荐的生产实现,其他四个要么是历史遗留(SimpleFSLockFactory)、要么是装饰器(VerifyingLockFactory)、要么是 RAM 专用(SingleInstanceLockFactory)、要么是危险测试用(NoLockFactory)。

// org.apache.lucene.store.LockFactory (Lucene 9.x 简化)
public abstract class LockFactory {

    public abstract Lock obtainLock(Directory dir, String lockName) throws IOException;
}

// org.apache.lucene.store.FSLockFactory
public abstract class FSLockFactory extends LockFactory {

    public static FSLockFactory getDefault() {
        return NativeFSLockFactory.INSTANCE;
    }

    @Override
    public final Lock obtainLock(Directory dir, String lockName) throws IOException {
        if (!(dir instanceof FSDirectory)) {
            throw new UnsupportedOperationException(
                "FSLockFactory can only be used with FSDirectory subclasses, got: " + dir);
        }
        return obtainFSLock((FSDirectory) dir, lockName);
    }

    protected abstract Lock obtainFSLock(FSDirectory dir, String lockName) throws IOException;
}

19.3 NativeFSLock 内部

NativeFSLockFactory 是默认且推荐的实现,它把"取锁"委托给操作系统级的文件锁。Java NIO 的 FileChannel.tryLock() 在不同平台下底层调用不同的系统调用——Linux 是 fcntl(F_SETLK) advisory 锁,Windows 是 LockFileEx mandatory 锁,macOS 是 flock 或 fcntl。三者对外的语义一致:同一文件同一时刻只能被一个 fd 持有写锁,fd 关闭(无论正常还是进程崩溃)kernel 自动释放。

关键源码

// org.apache.lucene.store.NativeFSLockFactory (Lucene 9.x 简化)
public final class NativeFSLockFactory extends FSLockFactory {

    public static final NativeFSLockFactory INSTANCE = new NativeFSLockFactory();

    private NativeFSLockFactory() {}

    @Override
    protected Lock obtainFSLock(FSDirectory dir, String lockName) throws IOException {
        Path lockDir = dir.getDirectory();
        Files.createDirectories(lockDir);
        Path lockFile = lockDir.resolve(lockName);

        Path realPath;
        try {
            realPath = lockFile.toRealPath();
        } catch (NoSuchFileException e) {
            Files.createFile(lockFile);
            realPath = lockFile.toRealPath();
        }

        FileChannel channel = FileChannel.open(realPath,
            StandardOpenOption.CREATE,
            StandardOpenOption.WRITE);

        boolean success = false;
        try {
            FileLock lock = channel.tryLock();
            if (lock == null) {
                throw new LockObtainFailedException(
                    "Lock held by another program: " + realPath);
            }
            NativeFSLock l = new NativeFSLock(realPath, channel, lock);
            success = true;
            return l;
        } finally {
            if (!success) {
                channel.close();
            }
        }
    }

    static final class NativeFSLock extends Lock {
        final Path path;
        final FileChannel channel;
        final FileLock lock;

        NativeFSLock(Path path, FileChannel channel, FileLock lock) {
            this.path = path;
            this.channel = channel;
            this.lock = lock;
        }

        @Override
        public void close() throws IOException {
            try (FileChannel c = channel; FileLock l = lock) {
                // try-with-resources 同时关 lock + channel
                // kernel 收到 close → 释放 advisory 锁
            }
        }

        @Override
        public void ensureValid() throws IOException {
            if (!lock.isValid()) {
                throw new AlreadyClosedException("Lock instance already released");
            }
        }
    }
}

"锁状态在 kernel,不在文件内容" — 这是 NativeFSLock 与 SimpleFSLock 最本质的区别。fcntl(F_SETLK) 调用的效果是 kernel 在内部数据结构里记录一条 (inode, pid, fd, lock_range) 元组,文件本身可以是 0 字节空文件。当进程退出(无论正常 close、kill -15、kill -9 或 segfault)kernel 都会回收该进程的所有 fd,顺带把锁元组移除。所以 ls -la write.lock 看到的是普通空文件,但 lsof write.lock 能查到当前持锁的进程 pid + fd——后者才是真相。这也是为什么 NativeFSLock 在 JVM 崩溃后无需任何手动清理。
Linux fcntl vs flock 的微妙差别 — 同一进程内 fcntl 锁是 per-(pid, file) 的,即使同一进程开两次 FileChannel 也不冲突;flock 是 per-fd 的,同一进程开两次依然冲突。Lucene 用 fcntl 的好处是单 JVM 内多次错误地 FSDirectory.open 同一目录,第一次取锁的 IW 不会被第二次的"自我冲突"误伤——但这其实并不是真正的保护:虽然没有冲突,两个 IW 实例依然会同时写、同时弄坏 segment 文件,所以单 JVM 单 IndexWriter 的纪律必须在应用层保证,不能依赖锁。

19.4 SimpleFSLock 与 NoLock

SimpleFSLockFactory 是早期 Lucene 的默认实现,它不使用 OS 锁,而是通过"文件存在与否"判定锁状态——obtainLock 时尝试 Files.createFile,失败说明已被持有;close 时 Files.delete。这种实现简单,但有一个致命缺陷:JVM 崩溃后文件残留,后续启动一律失败。Lucene 9.x 已经把它从默认中移除,生产环境不应再使用。

SimpleFSLock vs NativeFSLock

SimpleFSLockFactory

  • 仅靠 Files.exists() 判定锁
  • JVM 崩溃 → 文件残留
  • 下次启动:LockObtainFailedException
  • 必须 手动 rm /index/write.lock
  • NFS / SMB 一样不可靠(并发 createFile 不互斥)
  • 调试可见性差:ls -la 看到文件存在,但不知道谁持有
  • Lucene 9.x 已不推荐

NativeFSLockFactory

  • OS 文件锁(fcntl / LockFileEx / flock)
  • JVM 崩溃 → kernel 关 fd → 自动释放
  • 无需任何清理脚本
  • 跨进程互斥真实(同机器)
  • NFS / SMB 同样有问题(协议本身不一致)
  • lsof write.lock 直接看到持锁进程
  • Lucene 9.x 默认 + 唯一推荐

SimpleFSLock 简化实现

// org.apache.lucene.store.SimpleFSLockFactory (简化)
public final class SimpleFSLockFactory extends FSLockFactory {

    public static final SimpleFSLockFactory INSTANCE = new SimpleFSLockFactory();

    @Override
    protected Lock obtainFSLock(FSDirectory dir, String lockName) throws IOException {
        Path lockDir = dir.getDirectory();
        Files.createDirectories(lockDir);
        Path lockFile = lockDir.resolve(lockName);

        try {
            Files.createFile(lockFile);
            return new SimpleFSLock(lockFile);
        } catch (FileAlreadyExistsException e) {
            throw new LockObtainFailedException(
                "Lock held elsewhere: " + lockFile);
        }
    }

    static final class SimpleFSLock extends Lock {
        final Path path;
        SimpleFSLock(Path p) { this.path = p; }

        @Override
        public void close() throws IOException {
            Files.deleteIfExists(path);
        }

        @Override
        public void ensureValid() throws IOException {
            if (!Files.exists(path)) {
                throw new AlreadyClosedException("Lock file no longer exists");
            }
        }
    }
}
SimpleFSLock 崩溃残留是真实的生产事故源 — 老版本 Lucene 应用 / 早期 Solr 集群默认就是 SimpleFSLock。一旦机器被运维 kill -9、OOM Killer 杀死、或者主机断电,重启后服务无法启动,日志里只有冷冰冰的 LockObtainFailedException。新接手的运维很容易简单地"rm /index/write.lock 然后重启"——如果碰巧此时另一个进程还活着,就会变成双进程写入,索引彻底损坏,reader 报 CorruptIndexException,只能整体重建。这是为什么 Lucene 9.x 默认换成 NativeFSLock 的根本原因。

NoLockFactory:严禁生产

NoLockFactory 不取任何锁,任何 obtainLock 调用都立即返回一个 no-op Lock。这种实现仅用于明确不可能发生并发写入的纯测试场景(例如批量构建索引的 build 阶段,只有一个线程一个 JVM 一次性写完然后退出)。如果生产环境误用 NoLockFactory,两个 IndexWriter 同时启动会同时写,造成 segments_N 串改、.si 文件冲突、倒排数据混乱,索引彻底损坏。

// 严禁生产:NoLockFactory 用法
FSDirectory dir = FSDirectory.open(
    Path.of("/index/orders"),
    NoLockFactory.INSTANCE);  // ← 危险!

// 双 IndexWriter 同时打开 — 完全不报错
IndexWriter w1 = new IndexWriter(dir, new IndexWriterConfig(analyzer));
IndexWriter w2 = new IndexWriter(dir, new IndexWriterConfig(analyzer));

// 同时 add doc → segments_N 互相串改
w1.addDocument(doc1);
w2.addDocument(doc2);
w1.commit();  // 写 segments_43
w2.commit();  // 也写 segments_43,覆盖 w1 的版本 → 数据丢失

// 后续 DirectoryReader.open(dir) → CorruptIndexException

19.5 字节物理布局

19.6 IndexWriter 取锁路径

new IndexWriter(dir, config) 构造函数的第一件事就是调用 dir.obtainLock("write.lock")。如果取锁失败,后面所有的 SegmentInfos 读取、DocumentsWriter 初始化、IndexFileDeleter 引用计数全部不会发生,直接抛 LockObtainFailedException。这是 IndexWriter 一切状态的守门员。

构造时序:从 new 到持锁

图 19-7 · IndexWriter 构造与 obtainLock 时序

IndexWriter 构造函数的关键片段

// org.apache.lucene.index.IndexWriter (Lucene 9.x 简化)
public IndexWriter(Directory d, IndexWriterConfig conf) throws IOException {
    this.config = conf;
    this.directoryOrig = d;
    this.directory = new LockValidatingDirectoryWrapper(d, writeLock);

    // 第一步:取锁 — 失败立即抛,不构造任何后续状态
    writeLock = d.obtainLock(WRITE_LOCK_NAME);  // = "write.lock"

    boolean success = false;
    try {
        // 第二步:读取 segments_N
        if (create) {
            segmentInfos = new SegmentInfos(Version.LATEST.major);
        } else {
            segmentInfos = SegmentInfos.readLatestCommit(directoryOrig);
        }

        // 第三步:DocumentsWriter / DeleteQueue / FlushControl
        deleter = new IndexFileDeleter(...);
        docWriter = new DocumentsWriter(...);

        // 第四步:其他初始化
        bufferedUpdatesStream = new BufferedUpdatesStream(infoStream);
        success = true;
    } finally {
        if (!success) {
            // 失败 → 释放锁
            IOUtils.closeWhileHandlingException(writeLock);
            writeLock = null;
        }
    }
}

// Directory.obtainLock 默认实现
public Lock obtainLock(String name) throws IOException {
    return lockFactory.obtainLock(this, name);
}

LockFactory.obtainLock 简化版

// 等价的 NativeFSLockFactory.obtainFSLock 简化版
public Lock obtainLock(Directory dir, String name) throws IOException {
    Path lockFile = ((FSDirectory) dir).getDirectory().resolve(name);
    Files.createDirectories(lockFile.getParent());

    FileChannel ch = FileChannel.open(lockFile,
        StandardOpenOption.CREATE,
        StandardOpenOption.WRITE);

    try {
        FileLock fileLock = ch.tryLock();
        if (fileLock == null) {
            ch.close();
            throw new LockObtainFailedException(
                "Lock " + name + " held by another process: " + lockFile);
        }
        return new NativeFSLock(lockFile, ch, fileLock);
    } catch (IOException e) {
        ch.close();
        throw e;
    }
}

Elasticsearch 启动失败排查模板

ES 7.x / 8.x 启动时如果某个 shard 的 write.lock 拿不到,会在 elasticsearch.log 里看到 org.apache.lucene.store.LockObtainFailedException: Lock held by another program。最常见的三个真实成因:(1) 上一次进程异常退出,但用的是 SimpleFSLock(老配置升级时漏改),需要先 rm -f path/to/shard/index/write.lock 才能起来;(2) 同一节点上启动了第二个 ES 实例(运维误操作 systemctl start elasticsearch@node2),需要 jps + lsof 找到双进程;(3) Shard 在另一个节点已被分配,本节点 stale shard 复制还没清理,需要等 cluster state 同步或手动 reroute。生产排查流程永远是先 lsof,再判断是否敢删——不要"先 rm 再说"。

19.7 close 与锁释放

IndexWriter.close() 是锁释放的正路径。它的执行顺序是:先 doFlush 把内存中尚未落盘的 DWPT 全部写为 segment 文件,再 commitInternal 把 SegmentInfos 写入新的 segments_N+1(原子 rename),最后才调用 writeLock.close() 释放 OS 锁、关闭 FileChannel。无论中间任何一步抛异常,finally 块都会保证锁被释放。

close 后文件状态对比

LockFactorywrite.lock 文件OS 锁清理责任
NativeFSLock保留(0 字节)kernel 释放无 — 文件可重用
SimpleFSLock删除—SimpleFSLock 自删
VerifyingLock保留(诊断信息)kernel 释放无 — 留作下次诊断
"NativeFSLock 不删文件,只释锁" — 很多新手以为 writer.close() 后 write.lock 应该消失,看到文件依然在就开始怀疑。这是误解:NativeFSLock 的释放只是让 kernel 移除 (pid, fd, range) 元组,文件本身保留为 0 字节空文件。下次启动 IndexWriter 时复用这个空文件,再 fcntl(F_SETLK) 一次就完事。如果非要"看一眼是不是还在持有",用 lsof /index/write.lock 看进程列表是否为空——空就是空闲,有 java pid 就是某个 IW 还活着。

19.8 JVM 崩溃后残留

JVM 异常退出(kill -9 / OOM Killer / segfault / 主机断电)时,LockFactory 的两个实现行为完全不同。NativeFSLock 由 OS 自动释放——kernel 在进程退出时回收所有 fd,顺带移除关联的文件锁元组,下次启动 IndexWriter 直接成功。SimpleFSLock 永久残留——文件还在,下次启动 Files.createFile 抛 FileAlreadyExistsException,必须手动 rm。这是为什么生产环境必须用 NativeFSLock。

线上事故复盘:Pod 重启后双进程写

某搜索服务部署在 Kubernetes,使用 StatefulSet + PVC ReadWriteOnce。运维做版本升级时,旧 Pod 还在 graceful shutdown(writer.close() 正在 commit 海量删除标记,持续 90 秒),Kubernetes 默认 terminationGracePeriodSeconds=30 已经到期,直接 SIGKILL 旧 Pod。但 PVC 还没解除挂载,新 Pod 已经启动并尝试 obtainLock。NativeFSLock 通过 fcntl 检测到旧 fd 还存在(那个 fd 在 PVC 卷上),抛 LockObtainFailedException。修复:把 terminationGracePeriodSeconds 调到 180,并在容器 preStop hook 里调用 curl localhost:8080/admin/shutdown 触发 graceful close。这才彻底解决了"升级后偶发启动失败"。

19.10 NFS 与容器陷阱

"不要在 NFS 上运行 IndexWriter" 是 Lucene 官方文档反复强调的纪律。NFSv3 / v4 的 advisory 锁实现各客户端不一致,网络分区 + 时钟漂移 + 缓存不一致足以让 write.lock 形同虚设。同样的纪律延伸到容器化部署:emptyDir / tmpfs 重启就丢文件,ReadWriteMany PVC 等于多 Pod 并发写——这些都是写"半坏"索引的快车道。

// ✓ 修复
@Service
public class GoodIndexService implements DisposableBean {

    private final IndexWriter writer;
    private final Directory dir;

    public GoodIndexService(@Value("${index.path}") Path indexPath) throws IOException {
        this.dir = FSDirectory.open(indexPath);
        IndexWriterConfig cfg = new IndexWriterConfig(new StandardAnalyzer())
            .setRAMBufferSizeMB(256)
            .setOpenMode(IndexWriterConfig.OpenMode.CREATE_OR_APPEND);
        this.writer = new IndexWriter(dir, cfg);   // ← 取 write.lock 一次
    }

    public void index(Document doc) throws IOException {
        writer.addDocument(doc);   // ← IW 内部线程安全
    }

    @Scheduled(fixedDelay = 60_000)
    public void scheduledCommit() throws IOException {
        writer.commit();
    }

    @Override
    public void destroy() throws IOException {
        try { writer.commit(); } finally {
            writer.close();        // ← 释放 write.lock
            dir.close();
        }
    }
}

反例 2:单 JVM 多 IW vs 多 JVM 多 IW

多 JVM 同目录(灾难)

  • 双进程跨 JVM,跨进程锁互斥
  • 第二个 JVM 启动直接抛锁失败
  • 除非用 NoLockFactory 跳过(不要这样!)
  • 误用 NoLock + 双 JVM = segments_N 串改 + 索引彻底损坏
  • CheckIndex 抛大量异常,只能重建

单 JVM 多线程(正路)

  • 所有写线程共享 1 个 IndexWriter 实例
  • IW 内部 DocumentsWriterPerThreadPool 协调
  • 每线程绑定一个 DWPT,无锁并发写
  • write.lock 仅在 IW 实例创建 / 关闭时触发
  • 线性扩展到 CPU 核数

反例 3:误删 write.lock 的灾难

# ✗ 反例:遇到 LockObtainFailed 就直接 rm
$ ls -la /index/
... write.lock
$ rm /index/write.lock        # ← 危险!不先 lsof
$ systemctl restart search

# 现象:
#   原进程还活着 → rm 后原进程仍持有 NativeFSLock
#   新进程取锁成功(file 重建) → 双进程同时写 segments_N
#   两边的 commit 互相覆盖 → segments_N 损坏
#   reader 启动:CorruptIndexException
#   只能停服 + 全量 rebuild

# ✓ 修复:确认无进程后才操作
#!/bin/bash
INDEX_DIR=${1:-/index}
LOCK_FILE=$INDEX_DIR/write.lock

if [ ! -f "$LOCK_FILE" ]; then
    echo "write.lock 不存在,无需操作"
    exit 0
fi

PROC_INFO=$(lsof "$LOCK_FILE" 2>/dev/null)
if [ -n "$PROC_INFO" ]; then
    echo "ERROR: 锁仍被以下进程持有:"
    echo "$PROC_INFO"
    echo "拒绝删除。请先调查持有进程,确认无误再 kill 后再 rm。"
    exit 1
fi

echo "确认无进程持有锁,删除残留 write.lock"
rm "$LOCK_FILE"
echo "完成。可以重启 IndexWriter 服务。"

"write.lock 是一切写路径文件的前哨" — 这一章看似只讲了一个 0 byte 文件,但你会发现后续所有 Lucene 写路径文件——segments_N(下一章)、.si、.fnm、.fdt、.tim、.kdd 等等——的每一次 atomic rename + fsync 都发生在持有 write.lock 的窗口内。换句话说,只要 IW 还活着 + 还持锁,所有写文件就有"有且仅有一个写者"的硬保证。这是 Lucene "简单到只用一个 0 byte 锁文件 + 不可变 segment" 这套并发模型的灵魂。理解了 write.lock 这一章,后续 commit / SegmentInfos / IndexFileDeleter / NRT replication 才有了讨论的基础。
