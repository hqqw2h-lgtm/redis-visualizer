第 32 章 · .dvm DocValues 元信息文件

.dvm DocValues Meta · 字段表 · NUMERIC/BINARY/SORTED/SORTED_NUMERIC/SORTED_SET · 编码模式选择 · valuesOffset · DENSE/SPARSE DISI · ord_addresses

.dvm(DocValues Meta)是 DocValues 列存通道的字段表 + 编码描述:它逐字段记录"该列是 NUMERIC / BINARY / SORTED / SORTED_NUMERIC / SORTED_SET 中的哪种"、"用了哪种编码模式(DELTA / GCD / TABLE / MONOTONIC / BLOCK_PFOR)"、"在 .dvd 中的起始偏移与长度"、以及"docsWithField 的稠密/稀疏选择(DENSE / SPARSE / ALL)"。

.dvm 与 .dvd 是 1:1 配对的双文件:.dvd 是数据本体(几 MB-几 GB),.dvm 是读取它的目录(几 KB-几 MB,字段越多越大)。reader 打开 segment 时一次性把 .dvm 全部加载到内存,缓存为 Map<String, FieldEntry>;之后任何 getNumericDocValues(field) 调用都先查这张表,拿到 valuesOffset → seek 到 .dvd → block 解码。本章按"角色 → 字节布局 → 5 种 DocValuesType 的 FieldEntry → 编码模式选择 → 写入读取路径 → 合并 → 性能"组织,逐节拆解 .dvm。

  • .dvm 文件大小特征:每字段 60-300 byte,1000 个字段约 100-300 KB
  • 字节布局:Header(Lucene90DocValuesMetadata) + 字段表(numFields × FieldEntry) + IndexHeader 偏移 + Footer
  • FieldEntry 结构:fieldNumber + DocValuesType(1 byte) + 类型相关元信息(20-200 byte)
  • 5 种 DocValuesType 各自的 FieldEntry:NUMERIC(min/max/gcd/numericFormat)、BINARY(addressesOffset/length)、SORTED(ordsOffset/dictOffset)、SORTED_NUMERIC(addressesOffset)、SORTED_SET(ordsOffset/ordAddressesOffset/dictOffset)
  • NUMERIC 6 种编码模式:CONST / TABLE / GCD / DELTA / MONOTONIC / BLOCK_PFOR,writer 根据 min/max/gcd/uniqueCount 自动选择
  • DENSE / SPARSE / ALL 三种 docsWithField 编码:依据 cardinality 自动选择,DENSE 用 bitset,SPARSE 用 docID 列表
  • valuesOffset 的作用:reader 在 .dvd 中 O(1) seek 到该字段数据起点
  • 字典 dict + termAddr 表:SORTED / SORTED_SET 的字典字节与字典内偏移表均在 .dvd,但偏移指针在 .dvm
  • ord_addresses:仅 SORTED_SET 有,记录每 doc 的多 ord 子序列起止
  • 写入路径:Lucene90DocValuesConsumer 在 finish() 时把所有字段元信息一次性写到 .dvm
  • 读取路径:Lucene90DocValuesProducer 打开时全量加载 .dvm → 缓存 fieldEntries
  • 合并行为:.dvm 重建,字段顺序按合并后的 fieldInfos 排序

前置:第 31 章 .dvd(本章的对偶,DocValues 数据本体);第 22 章 .fnm(FieldInfos,字段表的来源);第 13 章 PackedInts / MONOTONIC(.dvm 中的偏移表与 .dvd 中的列编码都用 PackedInts);第 18 章 IndexedDISI(DENSE / SPARSE 编码);第 03 章 segment merge(.dvm 在 merge 时重建)。

.dvm 是 .dvd 的字段目录。它告诉 reader:每个 DocValues 字段的类型(NUMERIC/BINARY/SORTED/SORTED_SET/SORTED_NUMERIC)、编码模式(DELTA/GCD/TABLE/MONOTONIC/BLOCK_PFOR)、在 .dvd 中的偏移与长度、统计信息(min/max/gcd/uniqueCount)、以及解码所需的辅助参数(bitsPerValue/blockShift/jumpTableOffset)。

.dvm 是 reader 启动时全量加载到内存的一类小文件,典型大小 KB 到几 MB,加载后字段元信息常驻 Lucene90DocValuesProducer 内部 map,后续 getNumeric / getSorted 调用直接命中。

本章拆解 .dvm 字段记录的字节布局、五种类型的 metadata 字段差异、写入读取流程、与 .dvd 的偏移配对、以及版本演进与生产陷阱。

    Lucene90DocValuesConsumer.finish() 调用前
      → 各字段写完 .dvd 时已积累 metadata
    finish 阶段:

      所有字段写完后写 footer

  [打开阶段]
    DirectoryReader.open(directory)
      → SegmentReader 创建
      → Lucene90DocValuesProducer 构造
      → readMeta() 读 .dvm 全文 (一次性 O(N_fields))
      → 构建 fieldName -> FieldEntry map
      → 校验 .dvd objectID 与 .dvm 一致

  [查询阶段]
    LeafReader.getNumericDocValues("price")
      → producer.fields.get("price")
      → 拿到 FieldEntry,含 valuesOffset/length 等
      → 创建 NumericDocValues 实现,内部持有 indexInput slice

  [merge 阶段]
    新 .dvm 由 SegmentMerger 重新生成
    各字段 metadata 重新计算(min/max/gcd 可能变化)

图 32-6-2:SORTED 字典分块结构
Sorted 字典分块结构(termsDictBlockShift = 4 → 16 term/block):

  block 0 (term 0..15):
    block header:    blockOffset, blockLength
    sharedPrefixLen: 0
    term 0:           "Adidas\0"           (full)
    term 1:           share=2, suffix="ddas\0"  (delta)
    term 2:           share=1, suffix="merican Eagle\0"
    ...
    term 15:          share=3, suffix="ung\0"

  block 1 (term 16..31):
    sharedPrefixLen: 4 ("Sams" prefix)
    term 16:          "Samsung\0"
    term 17:          "Sandisk\0"
    ...

  termsIndex:
    block 0 起 -> blockOffset 0
    block 1 起 -> blockOffset 0x140
    block 2 起 -> blockOffset 0x280
    ...

读取 term ord = 23:
  blockIdx = 23 >> 4 = 1
  blockOffset = readMonotonic(termsIndex, 1) = 0x140
  seek to 0x140
  解码 block 1 直到 inBlockIdx = 23 & 15 = 7
  返回前缀 + suffix 拼接的完整 term

下一章预告

下一章讨论 .dim 文件,即 Points Data。Points 是 Lucene 用于多维数值索引的 BKD-tree 实现,支持地理坐标(2D)、IP 范围、时间戳范围、向量近邻等场景。下一章拆解 .dim 中 BKD 叶节点的字节布局与 leafBlock 编码。

本章小结

核心要点

  • .dvm 是 .dvd 的字段表 + 编码描述
  • 1:1 配对:.dvd 数据本体几 MB-几 GB,.dvm 字段表几 KB-几 MB
  • FieldEntry:每字段 fieldNumber + DocValuesType + 类型相关元信息 + valuesOffset
  • 5 种 DocValuesType 各自的 FieldEntry 结构差异 + 字典/ord_addresses 偏移指针
  • NUMERIC 6 种编码模式:CONST / TABLE / GCD / DELTA / MONOTONIC / BLOCK_PFOR
  • DENSE / SPARSE / ALL 三种 docsWithField 自动选择
  • reader 打开时全量加载 .dvm 到 Map<String, FieldEntry>
  • merge 时 .dvm 重建,字段顺序按合并后 fieldInfos 排序
一句话总结:第 32 章核心是把"该文件在 codec 中的角色 / 字节布局 / 编码细节 / 写入读取路径 / 合并行为 / 性能特征 / 生产陷阱"七件事讲清楚。理解了本章,下次遇到该文件相关的 corruption / 性能 / 合并 issue 都能从字节级回答。
生产建议:① 监控该文件大小占比,占比异常需要分析;② CheckIndex 定期校验 checksum;③ 索引体积大时优先考虑 forceMerge 减少 segment 数量;④ 升级 Lucene 版本时关注 codec 名称变化(如 Lucene90 → Lucene99 → Lucene103);⑤ 不要直接动文件本体,所有写入都通过 IndexWriter 完成。
