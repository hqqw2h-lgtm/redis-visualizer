第 22 章 · .fnm FieldInfos 文件

字段 schema · FieldInfo 16 属性 · IndexOptions 5 档 · DocValuesType 6 种 · Per-Field 字节布局 · GlobalFieldNumberMap · attributes per-field codec · softDelete · dvGen · Merge 合并策略 · Lucene94/99/100 演进

.fnm 是单个 segment 的字段 schema 表——它列举该 segment 出现过的每一个字段,记录字段名、字段编号、IndexOptions、DocValuesType、Points 维度、Vector 维度、attributes Map 等共 16 个属性。文件本身只有几百字节到几十 KB,但它承担着一条不可妥协的 contract:每个字段在 .fnm 中标记什么属性,就决定了该 segment 写入哪些下游 codec 文件。indexOptions != NONE 才会有 .tim/.tip/.doc/.pos,docValuesType != NONE 才会有 .dvd/.dvm,pointDimensionCount > 0 才会有 .dim/.dii/.kdd,vectorDimension > 0 才会有 .vec/.vex/.vem。理解 .fnm,等于理解 Lucene 字段模型与文件家族的路由表。

本章拆解 .fnm 物理字节布局、写读调用链、字段编号管理、DV 更新 generation、Merge 合并策略,以及 8 个生产陷阱(动态字段失控、dvType 不一致、字段名拼写差异、softDelete 漂移等)。对应 Lucene 9.4 的 Lucene94FieldInfosFormat,顺带对照 9.9 / 10.0 的微调。

  • FieldInfo 16 属性物理含义:name / number / 4 bit flag / IndexOptions / DocValuesType / dvGen / attributes / pointDim×3 / vectorDim×3

  • DocValuesType 六种 (NONE/NUMERIC/BINARY/SORTED/SORTED_SET/SORTED_NUMERIC) 与 .dvd 编码模式映射
  • 字段属性 → 文件后缀路由矩阵:每属性独立决定一组文件后缀是否生成

22.1 .fnm 在 segment 中的角色

每一个 segment 都有恰好一份 .fnm 文件(命名 _N.fnm,DocValues 更新后还会有 _N_K.fnm)。它在 Lucene 文件家族中扮演的角色,可以一句话概括为"字段下游 codec 文件的路由表 + 许可证":reader 拿着 .fnm 才知道该 segment 有哪些字段、每个字段是否产生倒排 / DocValues / Points / Vectors,以及对应文件名怎么拼。

FieldInfo 与 FieldType 的关系(对比)

FieldType — per-document 字段定义

  • 位置:org.apache.lucene.document.FieldType
  • 作用域:每个 Document.add(new XxxField(...)) 时携带
  • 生命周期:仅写入路径(IndexableField 实现)
  • 属性:与 FieldInfo 几乎一一对应
  • 不持久化(只是写入时的元数据携带者)
  • 常见类:TextField.TYPE_STORED / StringField.TYPE_NOT_STORED
  • 每个字段每次 add 都要带

FieldInfo — per-segment 字段元数据

  • 位置:org.apache.lucene.index.FieldInfo
  • 作用域:整个 segment 共享一份
  • 生命周期:写入累积 + 持久化到 .fnm + 读取重建
  • 属性:16 个,FieldType 的"merge 后版本"
  • 持久化在 .fnm 中
  • 由 FieldInfos.Builder 从 FieldType 累积
  • 一个 segment 一份,不可变

简单类比:FieldType 是"单个 doc 的字段意图"(我希望这个值这样索引),FieldInfo 是"整个 segment 的字段实情"(这个 segment 里这个字段最终就这样索引)。FieldInfos.Builder 在 flush 阶段把多个 doc 的 FieldType 合并成单一 FieldInfo——同字段不同 doc 的 IndexOptions 取最严、storeTermVector 取 OR、dvType 必须一致(否则抛 IllegalArgumentException)。

        this.name = Objects.requireNonNull(name);
        this.number = number;

    }

22.4 DocValuesType 六种

DocValuesType 是一个 6 元素枚举,决定该字段是否参与列存(DocValues)以及列存编码模式。DocValues 是 sort / aggregation / facet / function score 的物理基础,与倒排互不影响——同一字段可同时设 indexOptions = DOCS(走倒排)+ dvType = SORTED(走列存),称为"双写"。

"DocValuesType 在 segment 内不可改变,跨 segment merge 也不可冲突" — 一旦字段在某 segment 用 NUMERIC,该字段在同一 segment 后续 doc 必须继续用 NUMERIC(否则 FieldInfos.Builder 直接抛 IllegalArgumentException: cannot change DocValues type from NUMERIC to SORTED for field xxx)。跨 segment merge 时遇到字段类型不一致也是同样的报错(因为新 segment 只能有一种 dvType)。这是 Lucene 字段类型 schema-on-read 的硬约束——schema 在第一次写入时决定,不可变更,只能重建索引。这条约束极易在"灰度上线 mapping 改动"时踩中,详见 §22.17。

// org.apache.lucene.codecs.lucene94.Lucene94FieldInfosFormat (简化)
public final class Lucene94FieldInfosFormat extends FieldInfosFormat {
    static final String EXTENSION  = "fnm";
    static final String CODEC_NAME = "Lucene94FieldInfos";
    static final int FORMAT_START  = 0, FORMAT_CURRENT = 0;
    static final byte STORE_TERMVECTOR=0x01, OMIT_NORMS=0x02, STORE_PAYLOADS=0x04, SOFT_DELETES_FIELD=0x08;

    @Override
    public void write(Directory dir, SegmentInfo si, String segSuffix,
                      FieldInfos infos, IOContext ctx) throws IOException {
        String fileName = IndexFileNames.segmentFileName(si.name, segSuffix, EXTENSION);
        try (IndexOutput out = dir.createOutput(fileName, ctx)) {
            CodecUtil.writeIndexHeader(out, CODEC_NAME, FORMAT_CURRENT, si.getId(), segSuffix);
            out.writeVInt(infos.size());
            for (FieldInfo fi : infos) {
                out.writeString(fi.name);
                out.writeVInt(fi.number);
                byte bits = 0;
                if (fi.hasVectors())         bits |= STORE_TERMVECTOR;
                if (fi.omitsNorms())         bits |= OMIT_NORMS;
                if (fi.hasPayloads())        bits |= STORE_PAYLOADS;
                if (fi.isSoftDeletesField()) bits |= SOFT_DELETES_FIELD;
                out.writeByte(bits);
                out.writeByte((byte) fi.getIndexOptions().ordinal());
                out.writeByte((byte) fi.getDocValuesType().ordinal());
                out.writeLong(fi.getDocValuesGen());
                out.writeMapOfStrings(fi.attributes());
                out.writeVInt(fi.getPointDimensionCount());
                out.writeVInt(fi.getPointIndexDimensionCount());
                out.writeVInt(fi.getPointNumBytes());
                out.writeVInt(fi.getVectorDimension());
                out.writeByte((byte) fi.getVectorEncoding().ordinal());
                out.writeByte((byte) fi.getVectorSimilarityFunction().ordinal());
            }
            CodecUtil.writeFooter(out);
        }
    }
}
"FieldInfos.Builder 在 flush 阶段做字段属性合并" — 一个 doc 加 StringField("status", "PAID")(IndexOptions=DOCS),另一个 doc 同样字段加 TextField("status", ...)(IndexOptions=DOCS_AND_FREQS_AND_POSITIONS),Builder 会把两次的最严(DFP)记到 FieldInfo,然后整个 segment 都按 DFP 编码倒排数据。这是同 segment 内字段属性"最严生效"原则——但跨 segment merge 同样规则,SegmentMerger 也会做同一份合并。详见 §22.13。

// org.apache.lucene.codecs.lucene94.Lucene94FieldInfosFormat (read 部分简化)
@Override
public FieldInfos read(Directory dir, SegmentInfo si, String segSuffix, IOContext ctx)
        throws IOException {
    String fileName = IndexFileNames.segmentFileName(si.name, segSuffix, EXTENSION);
    try (ChecksumIndexInput in = dir.openChecksumInput(fileName)) {
        Throwable priorE = null;
        FieldInfo[] infos = null;
        try {
            CodecUtil.checkIndexHeader(in, CODEC_NAME, FORMAT_START, FORMAT_CURRENT,
                                       si.getId(), segSuffix);
            int numFields = in.readVInt();
            infos = new FieldInfo[numFields];
            for (int i = 0; i < numFields; i++) {
                String name = in.readString();
                int number = in.readVInt();
                byte bits = in.readByte();
                boolean termVec   = (bits & STORE_TERMVECTOR)   != 0;
                boolean omitNorms = (bits & OMIT_NORMS)         != 0;
                boolean payloads  = (bits & STORE_PAYLOADS)     != 0;
                boolean softDel   = (bits & SOFT_DELETES_FIELD) != 0;
                IndexOptions  iOpt  = IndexOptions.values()[in.readByte()];
                DocValuesType dvT   = DocValuesType.values()[in.readByte()];
                long          dvGen = in.readLong();
                Map<String,String> attrs = in.readMapOfStrings();
                int pDim = in.readVInt(), pIdx = in.readVInt(), pBytes = in.readVInt();
                int vDim = in.readVInt();
                VectorEncoding             vEnc = VectorEncoding.values()[in.readByte()];
                VectorSimilarityFunction  simF  = VectorSimilarityFunction.values()[in.readByte()];
                infos[i] = new FieldInfo(name, number, termVec, omitNorms, payloads,
                    iOpt, dvT, dvGen, attrs, pDim, pIdx, pBytes, vDim, vEnc, simF, softDel);
            }
        } catch (Throwable e) { priorE = e; }
        finally { CodecUtil.checkFooter(in, priorE); }
        return new FieldInfos(infos);
    }
}

22.9 字段编号管理(GlobalFieldNumberMap)

字段编号(fieldNumber)是性能优化:所有 codec 文件(.tim / .doc / .dvd / .dim / .vec)用 fieldNumber 索引字段,避免每次都用字段名做长字符串比较。.fnm 是 fieldNumber → fieldName 的映射表;GlobalFieldNumberMap 在 IndexWriter 实例级保证 fieldNumber 跨所有 segment 唯一。

fieldNumber vs fieldName 对比

fieldName · 业务身份

  • 类型:UTF-8 String,业务定义
  • 大小:8-32 byte 常见
  • 唯一域:全局(同 IndexWriter / 同索引)
  • 生命周期:跨 segment 一致
  • 用途:Java API terms("title") / getXxxDocValues("price")
  • 持久化在:.fnm 字段名 + Stored field 字段名
  • 缺点:每次比较 String 慢(O(L) L 为字符串长度)

fieldNumber · 性能优化编号

  • 类型:int,Lucene 内部分配
  • 大小:VInt 1-2 byte
  • 唯一域:IndexWriter 实例级(跨所有 segment)
  • 生命周期:在 GlobalFieldNumberMap 中常驻
  • 用途:Codec 文件内字段索引(.tim 按 fieldNumber 顺序)
  • 持久化在:.fnm 每条 FieldInfo 的 number
  • 优点:int 比较 O(1) + 编码省空间

public class MyCodec extends FilterCodec {

    @Override public KnnVectorsFormat knnVectorsFormat() {

            @Override public KnnVectorsFormat getKnnVectorsFormatForField(String field) {

//   PerFieldKnnVectorsFormat.suffix = "0"

22.11 softDelete 字段

softDeletesField 是 FieldInfo 的 bits bit-3。它的物理意义是:该字段是软删除的标记字段。reader 计算 LiveDocs 时会读 softDeletesField 字段的 NumericDocValues,值为 1 的 doc 视为已"软删除"——不真正物理删除,只是 query 不可见。

        dvGen = -1  (从未更新)
   b. reader 用基础 gen 找 DV 数据:
        _0_Lucene90_0.dvd  (gen 0)
        _0_Lucene90_0.dvm  (gen 0)
        在文件内按 fieldNumber 索引到 brand_dv 的 SORTED 块

关键:
  - 同一 segment 多次 DV 更新会有多份 .fnm + .dvd
  - 旧 gen 文件会在 IndexFileDeleter 找不到任何引用后被回收
  - reader 在打开时只看最新 gen 的 .fnm,字段 dvGen 决定每字段去哪个 .dvd 找数据
"dvGen 是 per-field generation,fieldInfosGen 是 per-segment generation" — 一个 segment 有一个 fieldInfosGen,但每个字段有自己的 dvGen。例如 segment _0 做了 3 次 DV 更新分别更新 amount / view_count / amount,fieldInfosGen 升到 3,但 amount 的 dvGen=3,view_count 的 dvGen=2,brand_dv 的 dvGen=-1(从未更新)。reader 读 _0_3.fnm 拿到所有字段的 dvGen,再分别去 _0_3_Lucene90_0.dvd / _0_2_Lucene90_0.dvd / _0_Lucene90_0.dvd 三个文件查值——这是 Lucene 增量 DV 更新省 IO 的核心机制。

22.13 Merge 字段属性合并策略

Merge 多 segment 时,SegmentMerger 必须把多个 .fnm 合并成一份。同名字段在不同 segment 可能属性不同,合并策略分 4 类:取最严(IndexOptions)、OR(boolean flag)、取最大(数值维度)、必须一致(类型相关)。

public Document indexEvent(Map<String,Object> e) {
    Document doc = new Document();
    for (var en : e.entrySet()) {
        if (en.getValue() instanceof String) {
            doc.add(new TextField(en.getKey(),

        } else if (en.getValue() instanceof Long) {

public Document indexEvent(Map<String,Object> e) {
    Document doc = new Document();

    // 显式固定字段
    if (e.containsKey("type")) {
      doc.add(new StringField("type",
        (String) e.get("type"), Field.Store.NO));
    }
    if (e.containsKey("user_id")) {
      doc.add(new StringField("user_id",
        (String) e.get("user_id"), Field.Store.NO));
    }
    if (e.containsKey("amount")) {
      doc.add(new NumericDocValuesField(
        "amount_dv", (Long) e.get("amount")));
    }
    if (e.containsKey("timestamp")) {
      doc.add(new NumericDocValuesField(
        "timestamp_dv", (Long) e.get("timestamp")));
    }

    // 动态字段聚合到 JSON
    String json = JsonUtils.toJson(e);
    doc.add(new BinaryDocValuesField("payload_dv",
        new BytesRef(json)));
    doc.add(new StoredField("payload", json));

    return doc;
}
// 字段数固定 < 50
// .fnm 加载 50 ms
// 启动 5 秒
真实场景:Elasticsearch dynamic mapping 误用到直写 Lucene — 团队 A 从 Elasticsearch 迁出,直接用 Lucene Java API 写索引,忘了 ES 的 dynamic mapping 是 Elasticsearch 上层的便捷功能(底层 Lucene 不知情)。结果 indexEvent 把每个事件 key 当字段直接 add,半年后字段数从 50 涨到 7000+,启动 IndexWriter 5 分钟、单次 commit 60+ 秒、merge 时间翻倍。修复方案是回到经典 Lucene 模式:固定 schema + JSON 聚合到 StoredField 或 BinaryDV,字段数立刻降回 30。这是从 ES 迁回 Lucene 直写最常见的踩坑。
