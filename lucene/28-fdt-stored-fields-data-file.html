第 28 章 · .fdt Stored Fields Data 文件

行存数据通道 · Field.Store.YES · chunk + LZ4/DEFLATE · 16 KiB/64 KiB · StoredFieldsWriter / StoredFieldVisitor · 与 .fdx/.fdm 三件套 · fetch 5-500 μs

.fdt 是 Lucene 的行存数据通道。当你写下 doc.add(new StringField("id", "ORD123", Field.Store.YES)),这串 "ORD123" 字节最终落到 .fdt;当你写 doc.add(new StoredField("payload", jsonBytes)),整个 JSON 字节流也落在 .fdt。.fdt 服务于 fetch 阶段——查询命中 docID 后,IndexSearcher 通过 storedFields().document(docID) 经 .fdx 跳转表定位 chunk,解压 chunk 后返回该 docID 对应的字段。

与列存的 .dvd 形成鲜明对比:.fdt 按 docID 顺序连续存放整行,以 chunk 为单位用 LZ4 (BEST_SPEED) 或 DEFLATE (BEST_COMPRESSION) 压缩。fetch 一个 doc 需要解压整 chunk(典型 16 KiB / 128 doc),然后按字段 token 流式解码。这种设计以 chunk 解压成本(5-10 μs)换 chunk 内随机访问能力——TopK 排序后的 K 个 doc fetch 几乎一定命中不同 chunk,所以 fetch 的延迟主要由 chunk 数决定,而非字段大小。

本章按文件结构(Header → 多个 Chunk Block → Dirty Chunks Trailer → Footer)拆解 .fdt,逐一展示字节布局、token 编码(typeBits + fieldNumber + value)、压缩选择(LZ4 vs DEFLATE)、写入路径(Lucene90CompressingStoredFieldsWriter)、读取路径(StoredFieldVisitor)与性能特征。

  • Field.Store.YES 在哪些场景必须使用,与 DocValues / Points / 倒排的关系——四种独立通道,正交不重叠
  • .fdt 的三段字节布局:Header(magic + codecName + objectID + chunkSize) → Chunk Blocks(docBase + numDocs + lengths + compressedData) → Footer(CRC32)
  • 单 chunk 内部的字段 token 编码:fieldNumberInfo (VInt) = (fieldNumber << 3) | typeBits,6 种 typeBits(STRING / BINARY / NUMERIC_INT / NUMERIC_FLOAT / NUMERIC_LONG / NUMERIC_DOUBLE)与各自的 value 编码
  • chunk 切分策略:bytes ≥ chunkSize 或 docs ≥ MAX_DOCS_PER_CHUNK 任一满足即 flush,BEST_SPEED 16 KiB/128 doc vs BEST_COMPRESSION 64 KiB/512 doc
  • BEST_SPEED 与 BEST_COMPRESSION 的权衡矩阵:LZ4 ~ 1 GB/s 解压 / 2-4× 压缩比 vs DEFLATE ~ 100 MB/s / 4-8× 压缩比
  • 超大文档触发的 sliced chunk:单 doc > chunkSize 时如何切分,以及失去单 doc 随机访问能力的代价
  • 写入路径:StoredFieldsWriter.startDocument / writeField / finishDocument / flush,内部如何累积 buffer、判断阈值、压缩、写 .fdt + .fdx 双流
  • 读取路径:StoredFieldsReader.visitDocument(docID, visitor) + .fdx 跳转表 + 解压 + 字段流式解码 + visitor 回调
  • StoredFieldVisitor 的 7 种回调(needsField / stringField / binaryField / intField / longField / floatField / doubleField)与 fieldsToLoad 白名单的优化空间
  • 性能基线:单 doc fetch 热路径 5-10 μs(LZ4),冷路径 100-500 μs(含 4 KiB I/O + 解压);跨 chunk 批量 fetch 1000 doc 5-10 ms vs 同 chunk 500 μs
  • 9 类生产陷阱:大字段误进 stored、用 stored 替代 DocValues 排序、单 doc > chunkSize 触发 slice、跨 chunk fetch、Visitor 全字段加载、压缩模式误用、三件套缺失等

28.1 用途与触发条件

.fdt 与倒排索引(.tim/.tip/.doc/.pos)、列存 DocValues(.dvd/.dvm)、Points(.dim/.dii)是四条独立的访问路径。倒排回答"哪些 doc 含 term X",DocValues 回答"doc Y 在字段 F 上的值",Points 回答"哪些 doc 的字段在范围 [a,b]",而 .fdt 回答"docID Y 当初保存了什么原始字段值"。本节先把 .fdt 的触发条件、字段类型映射讲清楚。

28.2 文件命名与定位

.fdt 与 .fdx(chunk 索引)、.fdm(元信息)三者构成 stored fields 体系。三者命名相同前缀 _<segment>_Lucene90_0,扩展名分别为 .fdt / .fdx / .fdm。任一文件缺失会导致 segment 无法打开。本节展示三件套在目录中的位置与命名拆解。

codecName 决定压缩算法:Lucene 通过 codecName 字符串区分 BEST_SPEED 与 BEST_COMPRESSION——reader 打开 .fdt 时,CodecUtil.checkIndexHeader 会用字符串匹配选定 LZ4Decompressor 或 DeflateDecompressor。这两种压缩模式使用不同的 codec 名而非同一 codec 加 flag,便于将来引入 zstd / brotli 等新算法时直接新增 codec 而不破坏向后兼容。

28.4 Chunk Block 字节布局

chunk 是 .fdt 的物理压缩单位。每个 chunk 内部独立,reader 可以从任意 chunk 开始解码而不依赖前后 chunk 状态。本节展示单 chunk 的字节布局、字段宽度、以及超大文档的 sliced chunk 处理。

28.4.1 单 Chunk 内部结构

图 28-4-1 · 单 Chunk 字节布局
单 Chunk 字节布局(典型 16 KiB 压缩前):

  │   chunk 中第一个文档的 docID        │
  ├────────────────────────────────────┤

28.4.4 Sliced Chunk(超大文档处理)

图 28-4-3 · Sliced Chunk:超大文档跨多 chunk
超大文档(单 doc > chunkSize)的 sliced chunk:

  Doc 500 stored fields 共 512 KiB(原始未压缩)
  chunkSize = 16 KiB

  写出布局:
    Slice 0: docBase=500, numDocs=1, isSliced=1, sliceID=0, totalSlices=32
      压缩 16 KiB 子块
    Slice 1: docBase=500, numDocs=1, isSliced=1, sliceID=1, totalSlices=32
      压缩 16 KiB 子块
    ...
    Slice 31: 末尾子块

  读取协议:
    .fdx 跳转表中,docID=500 对应起始 slice 偏移
    StoredFieldsReader 顺序读全部 32 个 slice
    解压并拼接成完整 doc 字段流

  注意:
    - 单 doc 跨多 chunk 不支持随机访问中间字段
    - 必须从 slice 0 开始顺序解压
sliced chunk 是性能陷阱:正常 chunk 一次解压可服务 1-128 doc,sliced chunk 一次只服务 1 doc 但要解压 N 个子块(N = ceil(docSize / chunkSize))。一个 512 KiB 的 doc 需要解压 32 次 → fetch 时延飙到 250-500 μs。生产中如果发现 .fdt 中有大量 sliced chunk(可通过 CheckIndex 输出 max chunk docs = 1 间接发现),应当拆分大字段(例如把 1 MB JSON 改为外存 + ID 引用),或干脆把字段从 stored 改为外部对象存储。

28.5 Stored Field Token 编码

解压后的字节流是按字段顺序排列的 token 流。每个字段由 fieldNumberInfo(VInt 同时编码 typeBits 与 fieldNumber)+ value bytes 组成。本节展示这套 token 编码,以及 6 种 typeBits 对应的 value 编码格式。

28.5.1 单字段在解压流中的编码

图 28-5-1 · 单 stored field 编码格式
解压后字节流中的单 stored field:

  ┌──────────────────┬──────────────────┐
  │ fieldNumberInfo  │ value bytes      │
  │ (VInt)            │ (按 typeBits 解析)│
  └──────────────────┴──────────────────┘

fieldNumberInfo 结构:
  bit 0-2 : typeBits (3 bit)
  bit 3+  : fieldNumber (剩余 bit)

typeBits 取值:
  000 (0)  : STRING       (UTF-8 byte 序列)
  001 (1)  : BINARY       (raw byte 序列)
  010 (2)  : NUMERIC_INT
  011 (3)  : NUMERIC_FLOAT
  100 (4)  : NUMERIC_LONG
  101 (5)  : NUMERIC_DOUBLE

各类型 value 编码:
  STRING        : VInt(byteLen) + UTF-8 bytes
  BINARY        : VInt(byteLen) + raw bytes
  NUMERIC_INT   : ZigZag VInt
  NUMERIC_FLOAT : 4 byte little-endian
  NUMERIC_LONG  : ZigZag VLong
  NUMERIC_DOUBLE: 8 byte little-endian

若 LZ4 压缩(单 doc 压缩比有限): ~ 24 byte
若批量 chunk 中(共 128 doc): 压缩比 3-5×
fieldNumber 与 typeBits 编码进同一 VInt:Lucene 把字段号和类型 tag 合并到一个 VInt((fieldNumber << 3) | typeBits)而不是两个独立 VInt,目的是节省 1 byte。绝大部分字段号 < 32(段内字段数有限),fieldNumberInfo 整体只占 1 byte;若分开两个 VInt 至少 2 byte。1M doc × 5 字段 × 1 byte 节省 = 5 MB——对于 100 MB 级 .fdt 是 5% 的省空间。

28.8.3 flush() 关键代码

private void flush(boolean force) throws IOException {
    int chunkDocs = numBufferedDocs;
    indexWriter.writeIndex(chunkDocs, fieldsStream.getFilePointer());

    final int[] lengths = new int[chunkDocs];
    int totalLen = 0;
    for (int i = 0; i < chunkDocs; i++) {
        lengths[i] = endOffsets[i] - startOffsets[i];
        totalLen += lengths[i];
    }
    fieldsStream.writeVInt(docBase);
    fieldsStream.writeVInt(chunkDocs);
    saveInts(numStoredFieldsInDoc, chunkDocs, fieldsStream);
    saveInts(lengths, chunkDocs, fieldsStream);

    compressor.compress(bufferedDocs.toArrayCopy(), 0, totalLen, fieldsStream);

    docBase += chunkDocs;
    numBufferedDocs = 0;
    bufferedDocs.reset();
}

28.9.3 DocumentStoredFieldVisitor 类示例

public class DocumentStoredFieldVisitor extends StoredFieldVisitor {
    private final Document doc = new Document();
    private final Set<String> fieldsToLoad;

    @Override
    public Status needsField(FieldInfo fieldInfo) {
        return fieldsToLoad == null || fieldsToLoad.contains(fieldInfo.name)
            ? Status.YES : Status.NO;
    }

    @Override
    public void stringField(FieldInfo fi, String value) {
        doc.add(new StoredField(fi.name, value));
    }

    @Override
    public void intField(FieldInfo fi, int value) {
        doc.add(new StoredField(fi.name, value));
    }
}
needsField 是最重要的优化点:每个字段开始前 reader 都会调 needsField(FieldInfo)。返回 Status.NO 时,reader 会读 VInt(length) 然后 seek(skip) 跳过该字段 value bytes,完全不解析。如果业务只需要 id 与 title 两个字段(其余 5 字段不要),把这两个字段名放进 fieldsToLoad 白名单可以省 60-70% 解析 CPU。典型生产代码经常踩这个坑——直接传 null(全部加载)而不实现白名单。

28.12.3 反例与正例

❌ 反例: 把整个 JSON 文档存进 stored

Document doc = new Document();
doc.add(new StringField("id", id, Store.YES));
doc.add(new StoredField("payload", jsonBytes));

现象:.fdt 占索引 80%,fetch 单 doc 1-5 ms (P99),page cache 利用率低。1M doc × 1 MB JSON = 1 TB 原始 → 300 GB 压缩后,内存装不下,每次 fetch 都打盘。

✅ 正例: stored 仅存 ID + 短摘要,大字段外存

Document doc = new Document();
doc.add(new StringField("id", id, Store.YES));
doc.add(new StoredField("title", title));
doc.add(new StoredField("summary", summary));

收益:.fdt 占索引 10-15%,fetch 单 doc 5-10 μs,实际大字段(jsonBytes)按需从 S3/HDFS 拉取(通过 id 二次查询)。1M doc × 250 byte stored ≈ 80 MB 压缩后,完全装得进 page cache。

生产案例:从 stored 大字段改外存解决磁盘耗尽

某新闻搜索索引把每条新闻的全文 HTML(平均 50 KB)塞进 StoredField("content", html),3M doc × 50 KB ≈ 150 GB 原始,LZ4 压缩 3× 后约 50 GB 占用磁盘的 70%。fetch 一篇文章正文 P99 1.5 ms。重构方案:.fdt 仅存 id + title + summary(共 ~ 300 byte),正文 HTML 存 S3,key 用 id。.fdt 缩到 800 MB(磁盘占比 8%),fetch .fdt P99 12 μs;正文需要时由前端发起异步 S3 GET(P99 30 ms,但延迟不阻塞搜索结果展示)。

用 stored 排序是反模式:绝对不要为了 sort 而在 stored 字段上做"先 fetch 再排序"——这是 O(N) fetch + O(N log K) 比较,N = 命中数。正确做法:为排序字段单独配 NumericDocValuesField("price", 99),sort by 直接走 .dvd 列存的 advance + 比较,N=1M 时 sort 80 ms vs fetch + sort 5-10 秒。stored 字段只负责 fetch 阶段的"展示数据",不参与 score / sort / facet。
