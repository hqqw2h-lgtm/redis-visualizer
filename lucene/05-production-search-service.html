第 5 章 · 生产搜索服务架构

CDC · IndexWriter · SearcherManager · Query Builder · 权限 Filter · searchAfter · RescoreQuery · FunctionScore · alias 切换 · Snapshot · 监控 · Playbook

前四章拆开了 Lucene 一颗"裸发动机"——倒排、Analyzer、Codec、segment merge、BM25。但真正能扛住客服 / 仓库 / 运营三方流量、跑在生产集群里 7×24 不熄火的,是把这颗发动机装进车架的架构。本章把发动机周围的离合、传动、悬挂、ABS 一并装上:从订单库的 CDC 事件流入、Indexer Worker 的幂等写入、SearcherManager 的 NRT 视图、Query Builder 的 DSL 翻译、权限 filter 的强制注入,到 alias 切换 + Snapshot 备份 + 监控告警 + 故障 playbook,组成一套真正能"明天就上线"的搜索服务。

这一层的工作不再是性能微调,而是边界与契约:订单库是事实来源 (system of record),Lucene 是读模型 (system of query)——两者必须解耦,中间用事件流连接,绝不能在交易事务里同步写索引。SearcherManager 必须以 1-5s 周期 maybeRefresh,绝不能"每写一条就刷一次"。权限 filter 必须前置进 Query,绝不能"先 search 再业务过滤"。alias 切换必须原子,绝不能"原地重建"。这些"绝不能"背后都是踩过的事故——把它们写在 wiki 顶部的工程师才能睡个好觉。

"事实来源"与"读模型"的分离 — Lucene 是 Read Model 而不是 Source of Truth。订单库写下后 commit,binlog 落盘——这才是事实成立的瞬间。Lucene 索引只是这个事实的派生视图,可以延迟、可以重建、可以丢失重做。一旦把 Lucene 当成"另一份主存储",写入失败要不要回滚订单事务、索引故障要不要拒绝下单、跨数据中心同步要不要分布式事务,这些问题立刻把架构拖进沼泽。把 Lucene 严格定义成"派生只读视图"是整套架构能简单的根本前提。

5.2 数据同步路径

从订单库到 Lucene 索引,数据有三种走法:同步双写、异步 MQ 直发、Outbox + CDC。三者最大的差异不在性能,而在故障语义——业务事务回滚了索引怎么办,MQ 投递成功但消费失败了怎么办,索引服务挂了交易要不要拒绝。生产里只有第三种"Outbox + CDC"能让所有这些问题都有确定答案。

三条路径的故障语义对比

反例:API 同步双写

@Transactional
public void payOrder(String id) {
    orderRepo.markPaid(id);
    luceneWriter.updateDocument(
        new Term("orderId", id),
        toDoc(orderRepo.find(id)));
    luceneWriter.commit();
}

问题 1:Lucene 写入失败导致整个交易回滚 — 用户付款失败。
问题 2:Lucene commit 慢(秒级)拖死交易事务,DB 锁持有过长。
问题 3:跨进程没有分布式事务,DB 提交但 Lucene 失败时数据漂移。

正例:Outbox + CDC 异步

@Transactional
public void payOrder(String id) {
    orderRepo.markPaid(id);
    outboxRepo.insert(OrderEvent.paid(id));
}

// 独立 Worker 消费 Kafka
@KafkaListener(topics = "order.events")
public void onEvent(OrderEvent ev) {
    if (isStale(ev)) return;
    writer.updateDocument(
        new Term("orderId", ev.orderId()),
        toDoc(ev));
}

业务事务只写两张本地表 — 原子保证。
索引故障不阻塞交易,事件堆积在 Kafka,恢复后追平。
重试 / 幂等 / 顺序由 Kafka + version 解决。

Outbox 表必须定期清理 — outbox 是"事件流水",不是"事件归档"。CDC 工具(Debezium)只读 binlog 而不读表,但 outbox 表一旦无限增长,主库性能会被慢慢拖死(行数、索引大小、备份时间全增长)。生产里通常配一个 daily job 删除 7 天前的 outbox 记录,或用 MySQL 8.0 的 partition by date drop partition。binlog 保留期必须 ≥ outbox 保留期 + 1 天,否则 CDC 中断后无法追上。

5.3 IndexWriter 的服务化封装

裸 IndexWriter 是个"半成品 API"——它要求调用者管理目录的打开关闭、单例性、commit 时机、异常恢复。把它包装成可注入的 Bean 之前,要解决四个问题:目录持有期、生命周期、异常处理、健康探针。

单例 + 优雅关闭

@Component
public class OrderIndexWriter {
    private static final Logger log = LoggerFactory.getLogger(OrderIndexWriter.class);

    private final FSDirectory directory;
    private final IndexWriter writer;
    private final AtomicBoolean closed = new AtomicBoolean(false);

    public OrderIndexWriter(LuceneProperties props) throws IOException {
        Path indexPath = Paths.get(props.indexPath());
        this.directory = FSDirectory.open(indexPath);

        IndexWriterConfig cfg = new IndexWriterConfig(new OrderAnalyzer())
            .setOpenMode(IndexWriterConfig.OpenMode.CREATE_OR_APPEND)
            .setRAMBufferSizeMB(256)
            .setMaxBufferedDocs(IndexWriterConfig.DISABLE_AUTO_FLUSH)
            .setUseCompoundFile(false)
            .setMergePolicy(new TieredMergePolicy());

        this.writer = new IndexWriter(directory, cfg);
        log.info("IndexWriter opened path={} segments={}",
            indexPath, writer.getSegmentCount());
    }

    public void update(String orderId, Document doc) throws IOException {
        if (closed.get()) throw new IllegalStateException("writer closed");
        writer.updateDocument(new Term("orderId", orderId), doc);
    }

    public IndexWriter raw() { return writer; }

    @PreDestroy
    public void close() throws IOException {
        if (closed.compareAndSet(false, true)) {
            log.info("closing IndexWriter, will commit pending changes");
            writer.commit();
            writer.close();
            directory.close();
        }
    }
}

同进程内只能有一个 IndexWriter

事实原因违反后果
目录上有 write.lockFSDirectory 的 NativeFSLockFactory 用 OS 文件锁第二个 IndexWriter 启动直接抛 LockObtainFailedException
单进程多实例无意义DWPT 池已经支持多线程并发写只是浪费锁竞争和内存
多进程写同目录禁止segment 文件名冲突,segments_N 元数据竞争索引立刻损坏
必须在进程关闭前 closewriter 持有文件句柄、未 commit 的 RAM buffer未 close 等于未 commit,数据丢失

关键参数与生产推荐

表 5-1 · IndexWriterConfig 关键参数
参数默认生产推荐影响

不要在请求线程里调 commit() — commit 把所有 DWPT 的 RAM buffer flush 到磁盘 + 写 segments_N + fsync,典型耗时 100ms 到几秒。HTTP 请求线程跑这个会让 P99 抖到秒级。生产里 commit 应该是定时任务(每 30s-5min 一次)或体积触发(累积 1GB 待 flush 时)。NRT 可见性靠 maybeRefresh,不靠 commit——commit 的目的是"重启后能恢复",不是"立即可见"。

5.4 SearcherManager 与 NRT

SearcherManager 是 Lucene NRT(Near Real-Time)的标准实现。它把"打开 reader、引用计数、定期刷新、安全关闭旧 reader"这些线程安全细节包了起来,业务代码只需要 acquire / release 两个动作。

SearcherManager 的标准封装

@Component
public class OrderSearcherManager implements DisposableBean {
    private final SearcherManager mgr;
    private final ScheduledExecutorService refresher;

    public OrderSearcherManager(OrderIndexWriter writerHolder) throws IOException {
        this.mgr = new SearcherManager(writerHolder.raw(), true, true,
            new SearcherFactory());
        this.refresher = Executors.newSingleThreadScheduledExecutor(
            r -> new Thread(r, "lucene-refresh"));
        refresher.scheduleAtFixedRate(this::refresh, 1, 1, TimeUnit.SECONDS);
    }

    private void refresh() {
        try {
            mgr.maybeRefresh();
        } catch (IOException e) {
            log.error("maybeRefresh failed, keep serving with old searcher", e);
        }
    }

    public <T> T withSearcher(SearcherTask<T> task) throws IOException {
        IndexSearcher s = mgr.acquire();
        try {
            return task.execute(s);
        } finally {
            mgr.release(s);
        }
    }

    @Override
    public void destroy() throws IOException {
        refresher.shutdownNow();
        mgr.close();
    }
}

interface SearcherTask<T> { T execute(IndexSearcher s) throws IOException; }

引用计数与"借出归还"

SearcherManager 内部对每个 IndexSearcher 维护一个 RefCount。acquire 让 RefCount +1,release 让 RefCount -1。当一个 searcher 被换下台(新 searcher 上线),它的 RefCount 已经 +1 后再减回 0 才会真正 close。这保证正在执行的查询不会因为 reader 切换而看到一半的视图。

Elasticsearch refresh_interval=30s 对吞吐的影响实测 — Elasticsearch 默认 refresh_interval=1s,在我们的订单 ES 集群上测过:把它从 1s 改到 30s,bulk 写入吞吐提高约 2.3×,segment 数量减少 5×,merge_time 降低 40%,代价是新数据 30s 才可见。这对"夜间批量导入历史订单"场景是显著收益。但白天交易时段 30s 太长——客服在订单刚发货后查不到会以为系统坏了。生产经验是:白天 1-5s,夜间批量写入时通过 API 临时调到 30s,写完再调回来。这个套路在 Lucene 直接用时同样适用——用 SearcherManager 提供的 setRefreshInterval 接口或外部调度即可。
release 必须放在 finally 里,而且只能调用一次 — release 没调,RefCount 永远不归零,旧 reader 永远不 close,文件句柄不释放,几小时后 fd 用尽 + 磁盘上文件无法删除。release 调两次,RefCount 提前归零,正在执行的查询用着已经 close 的 reader 抛 AlreadyClosedException。这两种 bug 在压力测试下才会暴露,生产环境一定要做代码审查的硬性 checkpoint。

5.5 Query Builder 抽象

业务代码不应该直接构造 BooleanQuery + TermQuery + LongPoint.newRangeQuery——那是 Lucene 内部 API,业务条件每加一个就要改 N 处。Query Builder 的职责是把"业务 DSL"翻译成 Lucene Query,并在中间强制注入安全约束(参数白名单、wildcard 防御、长度限制)。

Builder 的标准实现

public class OrderQueryBuilder {
    private static final Set<String> STATUS_WHITELIST =
        Set.of("CREATED", "PAID", "SHIPPED", "DELIVERED", "CANCELED");

    public Query build(SearchRequest req, AuthContext auth) {
        BooleanQuery.Builder b = new BooleanQuery.Builder();

        if (req.q() != null && !req.q().isBlank()) {
            b.add(parseKeyword(req.q()), BooleanClause.Occur.MUST);
        }

        if (req.status() != null) {
            require(STATUS_WHITELIST.contains(req.status()), "bad status");
            b.add(new TermQuery(new Term("status", req.status())),
                  BooleanClause.Occur.FILTER);
        }

        if (req.amountMin() != null || req.amountMax() != null) {
            long lo = req.amountMin() != null ? req.amountMin() : 0L;
            long hi = req.amountMax() != null ? req.amountMax() : Long.MAX_VALUE;
            b.add(LongPoint.newRangeQuery("amountCents", lo, hi),
                  BooleanClause.Occur.FILTER);
        }

        if (req.paidFrom() != null && req.paidTo() != null) {
            require(req.paidTo() - req.paidFrom() <= TimeUnit.DAYS.toMillis(90),
                "time window too wide");
            b.add(LongPoint.newRangeQuery("paidAt", req.paidFrom(), req.paidTo()),
                  BooleanClause.Occur.FILTER);
        }

        injectAuth(b, auth);
        return b.build();
    }

    private Query parseKeyword(String q) {
        String safe = QueryParserBase.escape(q.trim());
        try {
            return new QueryParser("title", new OrderAnalyzer()).parse(safe);
        } catch (ParseException e) {
            return new TermQuery(new Term("title", safe.toLowerCase()));
        }
    }
}
wildcard / regex 必须有最小前缀长度 — 用户能传 q=* 或 q=a* 这种宽 wildcard 是搜索服务的"性能炸弹"——后台展开几十万 term,内存炸、CPU 占满、其他查询全 P99 飙红。生产防御:WildcardQuery 的前缀必须 ≥ 3 字符,否则在 Builder 层拒绝(返回 400);RegexpQuery 默认禁用,只在白名单接口允许。这一点和 Elasticsearch 的 indices.query.bool.max_clause_count 是同一类防御。

5.6 权限过滤与多租户

多租户搜索服务里,"权限"不是查询的可选 filter,而是合规底线。租户 A 不能看到租户 B 的订单,商家不能看到其他商家的订单,客服只能看到自己分配的订单。这些规则必须在 Query 阶段强制注入,而不是在结果集上事后过滤。

权限注入代码

private void injectAuth(BooleanQuery.Builder b, AuthContext auth) {
    Objects.requireNonNull(auth.tenantId(), "tenantId required");
    b.add(new TermQuery(new Term("tenantId", auth.tenantId())),
          BooleanClause.Occur.FILTER);

    if (auth.role() != Role.TENANT_ADMIN) {
        Set<String> shops = auth.allowedShopIds();
        require(!shops.isEmpty(), "user has no shop scope");

        BooleanQuery.Builder shopFilter = new BooleanQuery.Builder();
        for (String s : shops) {
            shopFilter.add(new TermQuery(new Term("shopId", s)),
                           BooleanClause.Occur.SHOULD);
        }
        shopFilter.setMinimumNumberShouldMatch(1);
        b.add(shopFilter.build(), BooleanClause.Occur.FILTER);
    }

    if (auth.allowedWarehouseIds() != null
            && !auth.allowedWarehouseIds().isEmpty()) {
        BooleanQuery.Builder whFilter = new BooleanQuery.Builder();
        for (String w : auth.allowedWarehouseIds()) {
            whFilter.add(new TermQuery(new Term("warehouseId", w)),
                         BooleanClause.Occur.SHOULD);
        }
        whFilter.setMinimumNumberShouldMatch(1);
        b.add(whFilter.build(), BooleanClause.Occur.FILTER);
    }
}
为什么权限 filter 必须前置 — TopK 会被无权数据污染。一个常见的反模式:先 searcher.search(query, 100) 拿 TopK,再用业务代码过滤掉无权的订单。问题:TopK 收集的是按分数排序的前 100 条 — 不管有没有权限。租户 A 搜 "wireless charger",查到的 100 条里可能只有 3 条是租户 A 的,其他 97 条全是其他租户的高分订单。事后过滤后只剩 3 条结果——更糟糕的是分页:page=2 时还是从 TopK 第 100 条之后取,结果集稀疏到完全错乱。正确做法:把 FILTER tenantId:T1 AND shopId IN (...) 加进 Query,让权限约束参与 leapfrog 求交,TopK 收集到的就已经全是有权数据。这一点不仅是性能正确,更是多租户合规的硬性底线——SOC 2 / ISO 27001 审计员会专门检查搜索接口的权限过滤是不是在查询阶段。

5.7 排序与分页

"按金额降序、深翻页"是订单搜索几乎一定要支持的功能。但 from-size 翻页(from=1000 size=20)在 Lucene 里是 O(N) 反模式——TopK 堆要遍历到第 1000 条才能扔掉前面的。生产里只能用 searchAfter 游标。

from-size vs searchAfter 的代价对比

反例:from-size 深翻页

int from = (page - 1) * size;
TopDocs td = searcher.search(query,
    from + size,
    sort);

ScoreDoc[] hits = td.scoreDocs;
ScoreDoc[] page = Arrays.copyOfRange(
    hits, from, hits.length);

问题:Lucene 内部 TopK 堆是 from + size 大小。
page=100 size=20 → 堆大小 2020,要遍历几乎全部命中。
page=1000 时堆 20020,可能 1 千万 doc 全扫,P99 飙到秒级。
Elasticsearch 默认 index.max_result_window=10000 就是这个原因。

正例:searchAfter 游标

// 第一页:无 cursor
TopFieldDocs first = searcher.searchAfter(
    null, query, size, sort);

ScoreDoc lastHit = first.scoreDocs[size - 1];
String cursor = encode(lastHit);

// 第二页:用上一页最后一条
ScoreDoc after = decode(cursor);
TopFieldDocs next = searcher.searchAfter(
    after, query, size, sort);

每页堆大小恒定 = size,与翻页深度无关。
第 100 页和第 1 页代价几乎相同。
代价:不能"跳页",只能顺序翻。

SortField + searchAfter 实现

public PageResult<OrderHit> searchPage(SearchRequest req, AuthContext auth)
        throws IOException {
    Query query = builder.build(req, auth);
    Sort sort = new Sort(
        new SortField("amountSort", SortField.Type.LONG, true),
        new SortField(null, SortField.Type.DOC));

    return mgr.withSearcher(searcher -> {
        FieldDoc after = req.cursor() != null ? decode(req.cursor()) : null;
        TopFieldDocs td = searcher.searchAfter(after, query, req.size(), sort);

        List<OrderHit> hits = new ArrayList<>(td.scoreDocs.length);
        StoredFields sf = searcher.storedFields();
        for (ScoreDoc sd : td.scoreDocs) {
            Document d = sf.document(sd.doc, STORED_FIELDS);
            hits.add(toHit(d));
        }

        String nextCursor = td.scoreDocs.length == req.size()
            ? encode((FieldDoc) td.scoreDocs[td.scoreDocs.length - 1])
            : null;
        return new PageResult<>(hits, nextCursor, td.totalHits.value);
    });
}
deep paging 用 from-size 是反模式 — TopK 堆需要全量遍历到第 N 页。Elasticsearch 默认禁止 from+size > 10000(index.max_result_window),不是为了 ES 自己,是为了 Lucene 底层。Lucene 的 TopFieldCollector 维护一个堆大小为 from + size 的小顶堆,意味着即使你只要第 1000 页的 20 条,也得维护一个 20020 大小的堆——堆里每次 push 都是 O(log K),总代价是 O(N · log(from+size))。在 1 千万 doc 的索引里 P99 直接到秒级。生产里要么用 searchAfter,要么用 scroll(导出场景),要么硬性限制 page ≤ 100。

5.8 业务 boost 与 RescoreQuery

BM25 算的是"文本相关性"——它告诉你查询词与文档字段在语料统计上有多匹配。但生产里用户想要的不只是"相关",还有"对我有用":VIP 订单优先、最近下单的优先、客服正在跟进的优先、店铺评分高的优先。这些"业务相关性"无法编码进 BM25 公式,只能在 BM25 分数之上做第二阶段重排。Lucene 给出的标准答案就是 RescoreQuery(在 ES 里叫 rescore phase)——先用便宜的 BM25 召回 TopN,再用昂贵的业务函数对 TopN 内部重新排序。

两阶段检索:cheap recall + expensive rerank

图 5-8 · 两阶段检索的执行路径

两阶段架构的核心价值是把昂贵计算限定在 N 个文档上而不是全索引。BM25 求交是 O(min(freq)),在 1 千万 doc 索引上只扫稀有 term 命中的几万 doc;RescoreQuery 在阶段 2 只对 TopN(默认 N=200)做重计算,即使每个 doc 计算业务分数要花 50 μs,总代价也就 10ms 左右——在 P99 200ms 的预算里完全 hold 得住。

三种 boost 实现的层次

层次API计算时机代价典型用途
查询级 boostnew BoostQuery(query, factor)每次 advance/scoreO(全召回集)多字段权重 (title^2 vs body^1)
FunctionScoreFunctionScoreQuery + DoubleValuesSource每个候选 doc 计算一次O(全召回集)recency / 距离 / 简单数值因子

FunctionScoreQuery — 廉价业务因子直接合入 BM25

public Query buildFunctionScore(Query baseQuery, long now) {
    DoubleValuesSource recency = DoubleValuesSource.fromLongField("paidAtSort");
    DoubleValuesSource vip      = DoubleValuesSource.fromIntField("vipLevel");
    DoubleValuesSource shopScr  = DoubleValuesSource.fromDoubleField("shopScore");

    DoubleValuesSource recencyDecay = DoubleValuesSource.function(
        "recencyDecay", recency,
        v -> Math.exp(-(now - v) / (7.0 * 24 * 3600 * 1000)));

    DoubleValuesSource bizScore = DoubleValuesSource.function(
        "bizScore", List.of(vip, shopScr, recencyDecay),
        vals -> 1.0 + 0.5 * vals.get(0).doubleValue()
                    + 0.3 * vals.get(1).doubleValue()
                    + 0.2 * vals.get(2).doubleValue());

    return FunctionScoreQuery.boostByValue(baseQuery, bizScore);
}

FunctionScoreQuery 把 BM25 分数与业务因子相乘,公式形态固定为 finalScore = bm25 × bizScore(doc)。它适合"业务因子廉价、可向量化、能从 DocValues 直接读取"的场景——本例里 vipLevel / shopScore / paidAtSort 都是 NumericDocValuesField,O(1) 列读,几乎不增加单 doc 的打分代价。

RescoreQuery — 重计算限定在 TopN 内

public TopDocs searchWithRescore(IndexSearcher searcher,
                                  Query baseQuery, int topK)
        throws IOException {
    int topN = 200;
    TopDocs firstPass = searcher.search(baseQuery, topN);

    Query expensiveBiz = new BoostQuery(
        new MatchAllDocsQuery(),
        1.0f
    );
    Rescorer rescorer = new QueryRescorer(expensiveBiz) {
        @Override
        protected float combine(float firstPassScore, boolean secondPassMatches,
                                 float secondPassScore) {
            float biz = computeBusinessScore(firstPassScore, /* doc fields */);
            return 0.7f * firstPassScore + 0.3f * biz;
        }
    };

    return rescorer.rescore(searcher, firstPass, topK);
}

QueryRescorer 是 Lucene 提供的标准两阶段重排器。combine() 给你一个 hook 把 BM25 与业务分按权重混合。生产里 computeBusinessScore 经常是"读模型推理结果"或"调下游服务拿实时信号"——只在 TopN=200 上调用,而不是在全召回集上调用,几十毫秒的远程调用也能扛住。

BM25 的 explain 是"为什么相关",业务 boost 的 explain 是"为什么排第一" — 两者要分开 explain。生产 search API 必须暴露 ?explain=true 参数,返回时既给出 BM25 的 idf×tfNorm 拆分,也给出业务 boost 各因子的实际值与权重。运营反馈"为什么 X 商家的订单总排第一"时,你能立刻看到是 vipLevel=5 加了 0.5,还是 shopScore=4.8 加了 0.3,还是召回模型给了高分。没有这个能力,业务 boost 就是黑箱,没人能说清楚为什么这次结果排序变了——这种"调一调拉一拉"的状态在生产里是合规风险(电商平台尤其要注意搜索排序的可解释性)。
Elasticsearch 的 rescore phase 与 Lucene 一致 — ES 的 "rescore": { "window_size": 200, "query": {...} } 配置在 Lucene 层就是创建一个 QueryRescorer(window_size 即 TopN),底层 API 完全一样。区别只是 ES 把它包装成 JSON DSL,让"不懂 Lucene"的工程师也能配置。所以 ES 调 rescore 出问题时,直接看 Lucene 的 QueryRescorer.rescore 源码就能找到答案。Amazon Product Search 公开的论文里也提到他们用 LTR 模型在 rescore 阶段重排——核心架构和我们这里完全一致,差异只在业务因子复杂度。
不要把"过滤"伪装成 boost — 见过反例:把"禁售商品"用 BoostQuery(banQuery, -1000) 实现,期望负 boost 把它压到最后。问题是 BM25 分数本身就有 5-10 的浮动,-1000 看起来很大但在某些 doc 上仍可能为正(因为查询是 sum 不是 product)。结果是"被禁售的商品偶尔还会出现在搜索结果"——这是合规事故。正确做法:禁售要进 BooleanClause.MUST_NOT 或 FILTER 强制过滤,boost 只用于"排序偏好",绝不能承担"排除"语义。

SearcherManager 应对 alias 切换

@Component
public class AliasAwareSearcherManager {
    private volatile SearcherManager current;
    private volatile FSDirectory currentDir;

    @Scheduled(fixedDelay = 10_000)
    public void watchAlias() throws IOException {
        String desired = configCenter.get("orders.alias");
        if (!desired.equals(currentDir.getDirectory().getFileName().toString())) {
            log.info("alias change detected: {} -> {}",
                currentDir.getDirectory().getFileName(), desired);
            switchTo(desired);
        }
    }

    private synchronized void switchTo(String aliasTarget) throws IOException {
        FSDirectory newDir = FSDirectory.open(rootPath.resolve(aliasTarget));
        SearcherManager newMgr = new SearcherManager(newDir, new SearcherFactory());

        SearcherManager old = this.current;
        FSDirectory oldDir = this.currentDir;
        this.current = newMgr;
        this.currentDir = newDir;

        if (old != null) {
            try { old.close(); } catch (Exception ignored) {}
            try { oldDir.close(); } catch (Exception ignored) {}
        }
    }
}
为什么 alias 切换比原地重建索引安全 100 倍 — 我们曾经在一个生产事故里学到这个教训:Schema 升级团队在线上索引目录里跑全量 reindex,过程中 IndexWriter 持续写入,旧文档同时被删,新 segment 不断生成。Reindex 跑了 4 小时跑到 70% 时 OOM,索引已经处于"半新半旧"状态——一些订单只有新 schema 的字段,一些只有旧 schema,查询时 SortField 对部分文档拿不到 DocValues 直接抛 IllegalStateException。回滚?没法回滚,数据已经被覆盖。最后停服 6 小时从快照恢复。alias 切换的安全性来自不可变性:orders.v1 在整个 v2 构建过程中完全只读,失败再多次也不动 v1 一根毫毛;切换是配置层面 1 字节的修改,失败立刻可以切回 v1。这种"构建 / 切换分离"是所有大型搜索集群的标配。

5.10 备份与快照

Lucene 的 segment 是不可变的——这给备份带来天然优势:已经写入磁盘的 segment 永远不会被修改,只会被 merge 删除。SnapshotDeletionPolicy 就是利用这一点:它告诉 IndexWriter "这些 segment 不许删,我要拷走"。

@Component
public class LuceneMetrics {
    private final OrderIndexWriter writerHolder;
    private final OrderSearcherManager searcherMgr;

    public LuceneMetrics(MeterRegistry reg,
                         OrderIndexWriter w,
                         OrderSearcherManager sm) {
        this.writerHolder = w;
        this.searcherMgr = sm;

        Gauge.builder("lucene.segment.count", w, x -> x.raw().getSegmentCount())
            .register(reg);
        Gauge.builder("lucene.numdocs", w, x -> (double) x.raw().getDocStats().numDocs)
            .register(reg);
        Gauge.builder("lucene.maxdoc", w, x -> (double) x.raw().getDocStats().maxDoc)
            .register(reg);
        Gauge.builder("lucene.deletes.pct", w, this::deletesPct).register(reg);
        Gauge.builder("lucene.ram.bytes", w, x -> (double) x.raw().ramBytesUsed())
            .register(reg);
    }

    private double deletesPct(OrderIndexWriter w) {
        DocStats s = w.raw().getDocStats();
        if (s.maxDoc == 0) return 0;
        return 100.0 * (s.maxDoc - s.numDocs) / s.maxDoc;
    }
}

tmpfs 索引盘< 1 msRAM 大小断电全失仅缓存层 — 主索引禁用

单节点 vs 多 shard

图 5-13 · 多 shard 的扇出 + 合并
多 shard 的代价 — 别为了"以后扩展"提前分。多 shard 引入两个固定代价:(1) 每次查询 fan-out N 倍,网络 + 协调开销,P99 通常上升 30-50%;(2) 跨 shard 的全局 TopK 需要 over-fetch(每个 shard 取 size×N 条),内存 + 网络成本进一步放大;(3) idf 是 shard 局部的,跨 shard 的 BM25 可能轻微失真,需要 dfs_query_then_fetch 多一轮 RPC。单机 1 千万 doc 以下、单机 100GiB 索引以下,直接用单 shard MMapDirectory 是最优解。再大才考虑分片——即使 ES 也建议单 shard ≤ 50GB,而很多业务一辈子达不到这个量级。

5.14 与 Elasticsearch / OpenSearch 的关系

很多人在问"既然有 Elasticsearch,为什么还要直接用 Lucene"。答案:Lucene 是底层库,Elasticsearch 是分布式封装。两者的关系类似 Netty(底层 NIO)与 Spring Cloud(应用框架)——前者灵活精准,后者开箱即用。具体场景里直接用 Lucene 反而是更优解。

选型决策树

图 5-15 · Lucene 直用 vs ES 的决策树
何时直接用 Lucene 而非 ES — 单机 1 千万 doc 以下,业务深度定制评分。三类典型场景适合直接用 Lucene:(1) 单机量级:订单搜索、商品搜索、本地文档搜索这类"单业务系统内的搜索",1 千万级文档 / 100GiB 索引以下,单机 MMapDirectory 已经能跑出 P99 100ms 以下,部署个 ES 集群只是徒增 3 个节点 + ZK + Logstash 的运维负担。(2) 深度定制评分:需要把业务规则(如"VIP 客户订单加 boost"、"距离当前时间越近权重越大"、"用户行为反馈加权")无缝织入 BM25 时,Lucene 的 Similarity.scorer() 让你写 Java 代码就能拿到所有 stats;ES 只能用 painless 脚本,功能 + 性能都受限。(3) 嵌入式场景:桌面应用、小型 SaaS、IoT 设备本地搜索——根本没理由起一个 ES 集群。我们的订单搜索 1 千万 doc 单机 16GiB 内存跑了三年,P99 110ms,运维只有一个 cron 跑快照 + 一个监控仪表盘——这就是 Lucene 的甜蜜点。规模上来了再迁 ES,但过早分布式是最常见的过度设计。
