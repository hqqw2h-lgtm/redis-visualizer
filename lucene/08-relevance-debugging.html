第 8 章 · 相关性调优与排查方法论

Token Dump · Postings Dump · Rewrite · Explain · Boost · BM25 · MRR / NDCG / Recall · 黄金集 · 回归门禁 · FILTER vs SHOULD

相关性是搜索质量的核心命题。本章建立一套基于证据链的相关性排查与调优方法论:从用户症状出发,通过 token / postings / rewrite / explain 四层证据定位问题,通过离线评估(MRR / NDCG / Recall)与回归门禁判断是否可发布。

本章避免三类常见误区:不依赖直觉调 BM25 参数——所有调整必须有证据支持;单查询案例不驱动全局发布——必须有黄金集回归;相关性问题不全是 BM25 问题——Analyzer / 字段建模 / 同义词 / boost 任一环节出错都会影响结果。

  • 把症状分为 6 类:0 hits / 召回不全 / 排序错 / 重复结果 / 性能差 / 不稳定
  • 4 层证据收集:Token Dump → Postings Dump → Rewrite Dump → Explain Tree
  • 读懂 Explanation 树的 5 个关键信号:boost / idf / tf / dl/avgdl / freq
  • 用 Explain Diff 对比 Before/After,定量回答"为什么更好"
  • 掌握 Analyzer / Synonym / Boost 三大调优维度
  • 构造 ≥ 200 query 黄金集 + 双盲标注 + 月度刷新
  • 离线评估三指标:MRR(第一条命中重要)/ NDCG@K(整体排序)/ Recall@K(召回完整)
  • CI 回归门禁:NDCG -1% 警告、-3% 阻断,0 hits +1% 阻断
  • FILTER vs MUST vs SHOULD:业务硬条件走 FILTER,绝不进 scoring 区
  • 从症状走证据的诊断三段式:症状 → 第一证据 → 第二证据 → 修复

前置知识:第 4 章 Query/Scorer/BM25(理解 idf/tf/dl 模型);第 2 章 Analyzer / TokenStream(理解索引/查询侧分词差异);第 5 章生产搜索服务(知道生产指标埋点)。读者最好已实操过 BooleanQuery + BM25Similarity。

8.2 Token Dump

8.2.1 Token Dump 工具

public class TokenDumper {
    public static List<String> tokenize(Analyzer analyzer, String field, String text) throws IOException {
        List<String> tokens = new ArrayList<>();
        try (TokenStream ts = analyzer.tokenStream(field, text)) {
            CharTermAttribute cta = ts.addAttribute(CharTermAttribute.class);
            ts.reset();
            while (ts.incrementToken()) {
                tokens.add(cta.toString());
            }
            ts.end();
        }
        return tokens;
    }
}

8.3 Postings Dump

public class PostingsDumper {
    public static void dump(IndexReader reader, String field, String term) throws IOException {
        long total = 0;
        for (LeafReaderContext leaf : reader.leaves()) {
            Terms terms = leaf.reader().terms(field);
            if (terms == null) continue;
            TermsEnum te = terms.iterator();
            BytesRef tBr = new BytesRef(term);
            if (!te.seekExact(tBr)) continue;
            PostingsEnum pe = te.postings(null, PostingsEnum.FREQS);
            int doc;
            while ((doc = pe.nextDoc()) != DocIdSetIterator.NO_MORE_DOCS) {
                total++;
                if (total <= 10) {
                    System.out.println(field + ":" + term
                        + " doc=" + doc + " freq=" + pe.freq());
                }
            }
        }
        System.out.println("total docFreq=" + total);
    }
}
表 8-3-1 · Postings Dump 排查 checklist
检查项期望异常信号
docFreq> 0term 完全不存在
命中文档与预期一致命中错误的 doc
freq 分布合理极高 / 极低
多字段共存字段都有仅一个字段有

Explain Diff 的关键:Lucene Explanation 是结构化对象不是字符串,直接 diff 输出文本会被浮点小数位、 docFreq 段差异等噪声干扰。正确做法:把"调整前/调整后"两次 Explanation 都序列化为 JSON,然后按节点 path 比对每一项 value 变化。这样能定量回答:idf 涨了多少、boost 是否覆盖到 phrase 子树。

8.10 离线评估指标

8.10.1 三个常用指标

图 8-10-1 · MRR / NDCG / Recall 语义
MRR (Mean Reciprocal Rank):
  对每个查询, 记录第一个正确结果的位置 r
  score = 1/r
  MRR = avg(1/r) 跨所有查询

  rank=1 → 1.00
  rank=2 → 0.50
  rank=3 → 0.33
  rank=10 → 0.10

NDCG@K (Normalized DCG):
  DCG = Σ rel_i / log2(i+1)
  NDCG = DCG / IDCG  (理想排序的 DCG)
  范围 [0, 1], 越接近 1 越好

Recall@K:
  在前 K 个结果中, 相关结果占总相关结果的比例
  Recall@10 = 命中相关 / 总相关
表 8-10-1 · 指标选择
场景推荐指标理由
客服快速找单MRR第一条命中最重要
商家分析NDCG@10整体排序质量
召回评估Recall@K是否完整
综合NDCG + Recall平衡

8.12 评估代码骨架

public class RelevanceEvaluator {

    public EvalReport evaluate(IndexSearcher searcher,
                                List<EvalQuery> goldenSet,
                                int k) throws IOException {
        double totalMRR = 0;
        double totalNDCG = 0;
        double totalRecall = 0;

        for (EvalQuery eq : goldenSet) {
            TopDocs top = searcher.search(eq.getQuery(), k);
            Set<String> relevantIds = eq.getRelevantOrderIds();

            int firstRel = -1;
            double dcg = 0;
            int hitCount = 0;
            for (int i = 0; i < top.scoreDocs.length; i++) {
                String orderId = readOrderId(searcher, top.scoreDocs[i].doc);
                if (relevantIds.contains(orderId)) {
                    if (firstRel == -1) firstRel = i + 1;
                    hitCount++;
                    dcg += 1.0 / log2(i + 2);
                }
            }
            double idcg = idealDCG(Math.min(relevantIds.size(), k));
            double ndcg = idcg > 0 ? dcg / idcg : 0;
            double mrr  = firstRel > 0 ? 1.0 / firstRel : 0;
            double recall = relevantIds.isEmpty() ? 0
                : (double) hitCount / relevantIds.size();
            totalMRR += mrr;
            totalNDCG += ndcg;
            totalRecall += recall;
        }

        int n = goldenSet.size();
        return new EvalReport(totalMRR / n, totalNDCG / n, totalRecall / n);
    }
}

8.17 Filter 是门禁,不是加分项

订单履约里,租户、店铺、状态、城市、时间范围都是硬条件:不满足直接排除,满足也不应该让分数变高。这种条件应该走 BooleanClause.Occur.FILTER,而不是 MUST 或 SHOULD。

图 8-17-1 · 订单查询 FILTER 漏斗
表 8-17-1 · 三种放法对比
放法结果问题
MUST status:SHIPPED必须命中,还要进协调因子语义不清
SHOULD status:SHIPPED^5已发货更高分未发货也可能进结果,业务错误
FILTER status:SHIPPED只保留已发货,不参与评分语义最准,性能最好

Explanation 是结构化对象不是字符串,递归处理时不要 diff 文本:

public static void printExplanation(Explanation e, int depth) {
    String indent = "  ".repeat(depth);
    System.out.printf("%s%.4f :: %s :: match=%s%n",
        indent, e.getValue().doubleValue(), e.getDescription(), e.isMatch());
    for (Explanation child : e.getDetails()) {
        printExplanation(child, depth + 1);
    }
}

8.18.1 Explain Diff:同一订单前后查询逐项对比

调优 boost 或 Analyzer 之后,同一笔订单的 Explanation 应该有差异。把"调整前"和"调整后"的 Explain 树都序列化为 JSON,然后按 path 比对每个节点的 value 变化,就能定量回答"这次调整让 idf 涨了多少 / boost 是否覆盖到了 phrase 子树"。

直接 diff 文本不行——Lucene 输出的浮点小数位会变,description 里的 docFreq 也跟段不同而不同。必须先结构化再 diff。

8.19.1 调优流程的三个口诀

  • 不依赖直觉调 BM25 参数:所有调整都需要证据支持,黄金集 NDCG 不退化才允许发布。
  • 单查询案例不能驱动全局发布:1 个 query 调好了不代表 200 个都好;把这 1 个 case 加进黄金集,然后跑全集。
  • 相关性问题不全是 BM25 问题:Analyzer / 字段建模 / 同义词 / boost 任一环节出错都会让结果看起来像 BM25 不准。先排查前几环再调 BM25。
