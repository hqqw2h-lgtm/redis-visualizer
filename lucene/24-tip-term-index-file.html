第 24 章 · .tip Term Index 文件

BlockTreeTermsIndex · per-field FST · ceiling lookup · blockFP · 内存常驻 · mmap · floor block · IndexInput 拼装

第 10 章我们把 FST 当作"算法"讲完了——共享前缀、共享后缀、O(L) 查找。但在 Lucene 9.x 的真实磁盘格式里,FST 不是浮在内存里的 Java 对象图,而是一个固定字节布局的文件:.tip(Term Index)。它和 .tim 配对存在,两人的契约非常简单:.tip 给你一个字节偏移,.tim 在那个偏移处放着一个 block,block 里按字典序排着 25-48 个 term 的 metadata。

"为什么不能让 .tip 直接存 term → metadata 的全表?"——因为 1 千万 term × 50 byte metadata = 500 MB,这个量级根本塞不进可常驻内存的索引层。Lucene 的解法是切两层:.tip 只存"前缀 → block 入口"的极小 FST(平均 3-8 byte/term,1M term 才几十 MB),全量 mmap 进 page cache 永不下沉;.tim 存"block 入口 + 25-48 个 term 后缀 + metadata"的大块磁盘数据(平均 30-80 byte/term,几百 MB 到几个 GB),按需 page-fault 加载。一次 seekExact("charger") 就是先在 .tip 里 walk 4 步定位到 block 入口,再去 .tim 里把这个 block 读进来扫 28 个 suffix。两层加起来:亚毫秒延迟、几十 MB 内存、几个 GB 数据规模——这是 Lucene 词典的工程奇迹。

本章把 .tip 的"文件格式 + 加载契约 + 跨文件协议"展开:从 byte 0 的 magic number 开始,逐字段拆 header / per-field FST table / footer;讲清楚 fixed-arc / packed / bitmap 三种节点模式何时触发;把 FSTCompiler 在写 .tip 的同时如何分摊 NodeHash 节点共享讲透;再回到查询路径——BlockTreeTermsReader.terms(field) 给你的 FieldReader 是什么、SegmentTermsEnum 是怎么把 FST.lookupCeiling() 的输出和 .tim 的 IndexInput 拼起来的。最后讨论加载策略、Floor block、生产陷阱。

  • .tip 的文件级布局:Header + per-field FST 主体 + Field Index Table + Footer 各自占几字节,objectId 为什么必须与 .tim 一致
  • per-field FST:为什么每个字段有独立 FST,fieldNumber → fstOffset 的字段索引表如何在内存中定位字段
  • FST 节点的三种字节模式:fixed-arc array(numArcs ≥ 64,O(1) 直接索引)、packed(稀疏,O(numArcs))、bitmap(全 256 ASCII,O(1) bit 查询)
  • FST 输出的 8 byte 编码:bit 0-2 是 hasTerms / hasSubBlocks / isFloor 标志,bit 3-63 是 .tim 中 blockFP,沿路径累加得到最终偏移
  • Floor block 拆分逻辑:某个 prefix 下 term 数 > 48 时分裂为多个 floor 子块,FST 输出仅指向 floor head,floor head 内部再做二级 byte-aligned 索引
  • FSTCompiler + NodeHash 写入流程:reverse encoding 为什么让 root 在文件末尾、85% 节点共享率如何把 FST 大小从 200 MB 压到 30 MB
  • 加载契约:.tip 全量 mmap(默认推荐)、heap-resident byte[](< 100 MB 词典)、NIOFSDirectory(冷数据)三种策略的延迟 / 内存 / 启动时间权衡
  • IndexInput 拼装:BlockTreeTermsReader.open() 如何同时打开 .tim 与 .tip 两个 IndexInput,如何用 FieldReader 把它们绑成一个字段视图

前置知识:第 10 章 FST 与词典索引(BlockTree 的算法层)、第 23 章 .tim Terms 数据文件(.tim block 的内部布局)。本章是"算法 → 字节"的工程化补完——把第 10 章的 FST 抽象与第 23 章的 .tim block 通过 .tip 的字节协议粘起来。读者应熟悉 IndexInput / IndexOutput 抽象与 mmap 基础。

24.1 .tip 文件的角色

BlockTree 词典在磁盘上是两个文件而非一个:.tip(Term Index)在内存里、.tim(Term Data)在磁盘上。这种"内存索引层 + 磁盘数据层"的分工不是 Lucene 独有,B+Tree、LSM Tree、SSTable 都遵循类似设计——但 Lucene 通过 FST 把内存层压到了极致,1 千万 term 只占 30 MB。

注意 .tip 的"输出"就是 .tim 的"输入":FST 沿前缀走一遍累加的 long 值,被解码为 blockFP(高 61 bit)+ flags(低 3 bit),其中 blockFP 直接是 .tim 文件中 block 起始位置的字节偏移。这是 .tip 与 .tim 唯一的耦合:两个文件用 long 偏移做协议,objectId 校验匹配,不存在第三个 metadata 文件来"翻译"。

.tip 是路标,.tim 是数据。读 .tip 得到 8 byte 的 blockFP 和 3 bit flags,跨文件 seek 到 .tim @ blockFP 读 block,块内扫 25-48 个 suffix 找到目标 term 的完整 metadata(docStartFP / freq / docFreq / totalTermFreq / 前缀字节)。这种切分让"内存只装索引、磁盘装大数据"成为可能,也让 LSM 风格的段合并变成"两个已排序流的 zip merge",代价 O(N) 而非 O(N log N)。

24.2 .tip 与 .tim 的双层关系

第 10 章我们已经讲了 FST 的算法层——这一节专注于 .tip 的"FST 物化形态"在与 .tim 协作时的字节级契约。

FST 输出的 8 byte 编码

图 24-2-1 · FST 输出的位字段拆解
FST 累加 output(64-bit unsigned long)
   bit 63                                         bit 3 │ 2  1  0
   ┌───────────────────────────────────────────────────┬────────┐
   │           blockFP : 61 bit                          │  flags │
   │       (.tim 文件中 block 起始的字节偏移)            │  3 bit │
   └───────────────────────────────────────────────────┴────────┘

   flags 的 3 bit:
     bit 0 (0x01) = hasTerms      block 直接含 term metadata
     bit 1 (0x02) = hasSubBlocks  block 含子 block(BlockTree 嵌套)
     bit 2 (0x04) = isFloor       这个 block 是 floor head(需二次索引)

   解码示例:
     output = 0x000000000000254D
            = 0b 0010 0101 0100 1101
              = blockFP = 0x254D >> 3 = 0x4A9
              + flags    = 0x254D & 0x7 = 0b101
                         = hasTerms + isFloor

   blockFP 的语义:
     直接是 .tim 文件中 block 第一个 byte 的偏移
     reader.indexInput.seek(blockFP) 即可定位

.tip vs .tim 的访问模式对比

.tim · 大块磁盘数据,顺序扫描

特点:
  - per-block 25-48 个 term suffix
  - 块内必须顺序扫描(无内部索引)
  - 大小 200 MB - 80 GB
  - 必须 mmap,page-in 触发 IO

性能:
  - block 已 page in: ~ 200 ns
  - block 未 page in: ~ 100 µs (disk)
  - 一次扫描 28 个 suffix
  - 内存占 OS page cache 而非堆

.tim 的延迟由 page cache 命中率决定,冷查询主导 P99 抖动。

.tip · 小内存索引,随机寻址

特点:
  - per-field FST,共享前后缀
  - byte 数组上随机 walk arc
  - 大小 1 MB - 5 GB(典型 30 MB / 1M term)
  - 全量 mmap,常驻 page cache

性能:
  - FST walk: ~ 30 ns(L 长度的 4-8 步)
  - 累加 output: long 加法 4-8 次
  - 永不 page miss(常驻)
  - 内存占用极低

.tip 的延迟可预测,主要由 term 长度 L 决定,与 N 无关。

1 亿英文商品名词典的实测数据 — 某电商搜索集群:1 亿商品名(平均 token 数 12 / doc),unique term 1100 万,Lucene 9.7 默认配置。.tip 文件 = 38 MB(平均 3.5 byte/term,FST 节点共享率 87%);.tim 文件 = 520 MB(每 block 36 term,共 30 万 block);.tmd = 8 KB。query 路径上 P50 = 180 ns(.tip walk 30ns + .tim block scan 150ns,page in 状态);P99 = 620 ns(包含 1-2 次 .tim page fault)。整个词典子系统总内存占用 ~600 MB,JVM heap 占 0,全在 OS page cache。

Footer 校验区

.tip Footer(16 byte 固定):

  ┌──────────────────────────────────────────┐
  │ footerMagic : 4 byte (0xC02893E8)         │
  │ algorithmID : 4 byte (0x00000000 = CRC32) │
  │ checksum    : 8 byte (CRC32 of bytes 0..N-16) │
  └──────────────────────────────────────────┘

reader 打开时自动调 CodecUtil.checksumEntireFile():
  if (computed != stored)
      throw new CorruptIndexException("checksum mismatch")

footer 不参与 FST walk,只在 open / CheckIndex 时校验
objectID 是 .tip 与 .tim 配对的唯一锚点。同一 segment 的 .tip 和 .tim header 都写入同一份 16 byte SegmentInfo.id。reader 打开 .tip 时校验 objectID 与 .si 一致,打开 .tim 时同样校验,且必须 .tip 与 .tim 的 objectID 互相匹配。如果你手工把 _0.tip 与 _1.tim 拼起来(比如恢复时复制错文件),Lucene 会立即在 BlockTreeTermsReader.open 抛 CorruptIndexException("partner objectId mismatch")——这是设计上的强一致性约束,而不是运行时偶发检测。

Field Index Table:在内存中定位字段

reader 在打开 segment 时一次性读完 Field Index Table,在内存里建立 fieldNumber → fstOffset 的小映射(典型 100 字段 × 16 byte = 1.6 KB)。后续 terms("title") 调用时:

  1. 从 .fnm 查 "title" 字段的 fieldNumber(比如 5)
  2. 从内存中 fieldNumber → fstOffset 表查 fstOffset(比如 0x12FE34)
  3. 把 .tip 的 IndexInput slice 到 [fstOffset, fstOffset + numBytes],作为这个字段独立 FST 的视图
  4. 用这个 sliced IndexInput 创建 FieldReader,后续所有 walk arc 都在这个 slice 范围内
public final class FieldReader extends Terms {
    final FST index;
    private final long numTerms;
    private final long indexStartFP;
    private final long rootBlockFP;

    FieldReader(BlockTreeTermsReader parent, FieldInfo fieldInfo,
                long numTerms, BytesRef rootCode, long indexStartFP,
                IndexInput indexIn, ...) throws IOException {
        this.numTerms = numTerms;
        this.indexStartFP = indexStartFP;
        this.rootBlockFP = decodeBlockFP(rootCode);
        if (indexIn != null) {
            IndexInput clone = indexIn.clone();
            clone.seek(indexStartFP);
            this.index = new FST<>(FST.readMetadata(clone,
                ByteSequenceOutputs.getSingleton()),
                clone, new OffHeapFSTStore());
        } else {
            this.index = null;
        }
    }
}

24.5 FST 节点字节模式

FST 的 byte 数组里,每个节点根据 fan-out(出度,即 numArcs)选择不同的存储模式。FSTCompiler 在写入时自动挑模式;FST.findTargetArc 读取时根据节点头的 flag 选择对应解码路径。

三种节点模式的触发与代价

表 24-5-1 · FST 节点的三种存储模式
模式触发条件查找复杂度每 arc 字节典型场景
packed(变长稀疏)numArcs < 64O(numArcs) 顺序扫2-6 byte大多数中间节点
fixed-arc array(等长密集)numArcs ≥ 64O(1) 直接索引3-8 byte 等长root / 高 fan-out
arcs with bitmap(全 256 ASCII)label 范围密集 ≥ 50%O(1) bit 查询32 byte bitmap + arcroot 节点(英文 a-z)

fixed-arc 与 packed 节点对比

图 24-5-1 · fixed-arc(密集)vs packed(稀疏)字节布局
fixed-arc 节点(numArcs=128, bytesPerArc=6):

  Header:  nodeFlag(1) + numArcs(1-2) + bytesPerArc(1) + firstLabel(1) = 4 byte
  Arcs:    128 × 6 byte = 768 byte    每 arc 等长
  总:      772 byte

  查询 label='m' (ASCII 109):
    arcIdx = label - firstLabel = 109 - 97 = 12
    offset = 4 + 12 × 6 = 76
    直接读 byte[76..81] → O(1) 命中

  代价:短 output 的 arc 也占 6 byte → 对 numArcs < 64 反而比 packed 大

packed 节点(arcs=4: 'i','n','g','s'):

  Arc 0 (i): flags(1) + label(1) + output VLong(1) + target=NEXT     = 3 byte
  Arc 1 (n): flags(1) + label(1) + output VLong(2) + target VLong(1) = 5 byte
  Arc 2 (g): flags(1) + label(1) + output VLong(1) + target VLong(1) = 4 byte
  Arc 3 (s, LAST_ARC): flags(1) + label(1) + output(1) + target=NEXT = 3 byte
  总:       15 byte(同 4 arcs 用 fixed 模式占 28 byte,packed 省 46%)

  查询 label='g':顺序读 i → n → g 命中,O(numArcs)
  numArcs < 64 时,L1 cache 命中比二分快

三种模式的自动切换

图 24-5-3 · FSTCompiler 选择节点模式的决策树
三种模式自动切换是 FST 性能稳定的关键。一个英文 term 字典的 root 节点 fan-out 通常 26-52(a-z + A-Z),用 bitmap 或 fixed-arc 一次查表;字典深处节点 fan-out 1-4,packed 顺序扫反而比二分快(L1 cache 命中);中等深度节点 fan-out 5-15,packed + L2 命中。同一个 FST 在不同节点用不同模式,综合性能比"单一模式"快 2-3 倍——这种"按位置自适应"在 RocksDB / LSM 系列里也是同样思想(SSTable 的 index block 用 binary search,data block 用 linear scan)。

24.6 FST 写入流程

BlockTreeTermsWriter.finishTerm() 在写一个 term 时,先把 term 进 .tim block,等 block 满 25-48 个 term 后调用 flushBlock();flushBlock() 写完 .tim 字节,然后用 block 的 blockFP + flags 作为 output 调用 FSTCompiler.add(prefix, output)。所以 FST 是"以 block 为单位"递增构建的,而不是逐 term 添加。

NodeHash 与共享率

NodeHash 是 FST 写入期的核心去重器。它维护一个内存中的 (签名 → 已写偏移) 哈希表,签名计算覆盖该节点的"行为指纹":

private long hash(FSTCompiler.UnCompiledNode node) {
    long h = 0;
    h = 31 * h + (node.isFinal ? 1 : 0);
    if (node.isFinal && node.output != null) {
        h = 31 * h + node.output.hashCode();
    }
    for (int arcIdx = 0; arcIdx < node.numArcs; arcIdx++) {
        FSTCompiler.Arc arc = node.arcs[arcIdx];
        h = 31 * h + arc.label;
        h = 31 * h + arc.target.hashCode();
        if (arc.output != null) {
            h = 31 * h + arc.output.hashCode();
        }
        if (arc.nextFinalOutput != null) {
            h = 31 * h + arc.nextFinalOutput.hashCode();
        }
    }
    return h ^ (h >>> 32);
}

@Override
public boolean seekExact(BytesRef target) throws IOException {
    if (fr.index == null) {
        throw new IllegalStateException("terms index was not loaded");
    }
    FST.Arc arc = fr.index.getFirstArc(arcs[0]);
    BytesRef output = arc.output();
    int targetUpto = 0;

    while (targetUpto < target.length) {
        int label = target.bytes[target.offset + targetUpto] & 0xFF;
        FST.Arc nextArc = fr.index.findTargetArc(label,
            arc, getArc(1 + targetUpto), fstReader);
        if (nextArc == null) break;
        arc = nextArc;
        term.setByteAt(targetUpto, (byte) label);
        if (arc.output() != NO_OUTPUT) {
            output = fstOutputs.add(output, arc.output());
        }
        targetUpto++;
    }
    return scanToTermLeaf(target, false) == SeekStatus.FOUND;
}

Ceiling lookup 决策卡片

图 24-7-2 · seekCeil 的三种返回情况
seekCeil("charger") 的三种结果:

  ① FOUND — term 精确存在
     FST walk 完整路径 → 找到 final state
     output 直接是 charger 的 blockFP
     TE.term() == "charger"

  ② NOT_FOUND — 找到下一个更大 term
     比如词典里有 "charge" 和 "chargers" 但无 "charger"
     FST walk "charg" 后,'e' arc 之后无 'r' 直接到 final
     回溯找下一个最小路径 → "chargers"
     TE.term() == "chargers" (≥ target)

  ③ END — 词典里所有 term 都 < target
     比如查 "zzz" 而最大 term 是 "yes"
     FST walk 时遇到无更大 sibling
     TE.term() 为空,返回 END

逻辑伪代码:
  for each char in target:
    arc = findTargetArc(label, prevArc)
    if arc != null:
      output += arc.output
      continue
    else:
      ceil = findTargetArcCeil(label, prevArc)
      if ceil != null:
        walk minimum path from ceil
        return NOT_FOUND
      else:
        backtrack to parent and repeat
  if reached final:
    return FOUND

24.8 Floor block 双重索引

BlockTree 的 block 默认上限是 48 term。但某些 prefix 下天然就有几百几千 term(比如 "pro" 前缀在英文电商商品名里可能有 5000+ 个 product_xxx)——这种情况一个 block 装不下,Lucene 会把它水平切分成多个 floor block。

Floor block 的产生条件

图 24-8-1 · 何时分裂为 floor block

Floor block 字节布局

图 24-8-2 · .tim 中 floor block 的物理结构
.tip 中 prefix "pro" 节点 output:
  blockFP = 0x4000
  flags   = hasTerms (1) + isFloor (4) = 0b101

.tim @ 0x4000 (floor head):
  ┌────────────────────────────────────┐
  │ floor block header                   │
  │   isFloor flag        = true (在 .tip output 已编码)│
  │   numFollowedBlocks   = 4            │
  │   floorBranches       = ['d', 'h', 'p', 's']  │
  │     (各 sub-block 起始 lead byte)    │
  │   floorFollowedFPs    = [0x4080, 0x4140, 0x4200, 0x4280] │
  │     (各 sub-block 在 .tim 的偏移)    │
  ├────────────────────────────────────┤
  │ sub-block 0  (suffix < 'd' 的 term) │
  │ sub-block 1  ('d' ≤ suffix < 'h')   │
  │ sub-block 2  ('h' ≤ suffix < 'p')   │
  │ sub-block 3  ('p' ≤ suffix < 's')   │
  │ sub-block 4  ('s' ≤ suffix)         │
  └────────────────────────────────────┘

reader 决策路径:
  1. .tip FST output 解码 → blockFP=0x4000, flags 含 isFloor=1
  2. seek .tim @ 0x4000 → 读 floor head
  3. 按目标 suffix 首字符,二分 floorBranches
     比如 suffix = "ducer" → 首字符 'd' ≥ 'd' → 选 sub-block 1
  4. seek .tim @ 0x4140 → 读 sub-block 1
  5. 块内顺序扫 'ducer' suffix → 命中

总开销:
  - 1 次 FST walk (.tip)
  - 2 次 .tim block read(floor head + sub-block)
  - 比单 block 多 1 次 IO,但避免一个 1000+ term 的巨型 block

Block vs Floor block 对比

ToC1Block(单层,< 48 term)

触发: 该 prefix 下 term 数 ≤ 48
.tip FST output 标志: hasTerms=1, isFloor=0

结构:
  block_header
  suffixes[25-48]    ← 共享前缀压缩
  metadata[25-48]    ← TermState

读取: 1 次 .tim seek + 1 次块内扫
延迟: ~ 200 ns(page in)

适用: 大多数英文 term 前缀

简单单层结构,但 prefix 下 term 数过多时退化。

Floor block(多层,> 48 term)

触发: 该 prefix 下 term 数 49 - 5000+
.tip FST output 标志: hasTerms=1, isFloor=1

结构:
  floor_head:
    numFollowedBlocks
    floorBranches[]   ← lead byte 索引
    floorFollowedFPs[]
  sub_block_0..N

读取: 1 次 .tim head + 二分 + 1 次 sub-block
延迟: ~ 350 ns

适用: 高基数 prefix(如电商 product_*)

两层间接寻址,把巨型 block 切成可控 sub-block。

Floor block 是 Lucene 在"FST 共享"和"block 大小"之间的工程取舍。FST 共享前缀的天然倾向是把同 prefix 的所有 term 都聚到一个 block——但单 block 顺序扫的代价随 size 线性增长,1000 term 的 block 扫描比 50 term 慢 20×。Floor block 用"FST 出口指向 floor head + floor head 内部 byte-level 索引"两层间接,让单 block 永不超过 48 term,代价是多一次 .tim seek。Lucene 的默认 maxItemsInBlock=48 是经过反复 benchmark 校准的:小于这个值 floor 频繁触发(增加间接寻址),大于这个值 block 内扫描代价飙升。

mmap 与 page cache

MMapDirectory 通过 FileChannel.map() 把 .tip 映射到进程虚拟地址空间。访问触发 page fault → OS 从磁盘读 4 KB page 到 page cache → 后续访问命中 page cache 不再 syscall。这就是为什么 .tip 能"看起来在内存里但不占 JVM heap"——它占的是 OS 内核 page cache,从 JVM 视角是 native 区。

.tip 占的是 page cache 而非 JVM heap — 你 jconsole 看 -Xmx 再大,也看不到 .tip 内存。它体现在进程 RSS 和 OS page cache 上。这意味着:(1) 容量规划必须留足 page cache 余量,典型生产 4 vCPU / 16 GB 机器,JVM heap 6 GB,留 8 GB 给 page cache,.tip 占其中几十 MB;(2) 关 swap 必须,否则 page cache 被 swap out 后,.tip walk 会触发 disk read,P99 飙到 50 ms;(3) 不要让其他进程抢内存,大数据 ETL 服务和搜索服务不要混部。

public class SessionIndexer {
    public Document createDoc(String sessionId, String content) {
        Document doc = new Document();
        long hash = MurmurHash3.hash64(sessionId.getBytes(UTF_8));
        doc.add(new StringField("session_id_hash",
            Long.toString(hash, 36), Field.Store.NO));
        doc.add(new StoredField("session_id", sessionId));
        doc.add(new SortedDocValuesField("session_id_dv",
            new BytesRef(sessionId)));
        doc.add(new TextField("content", content, Field.Store.NO));
        return doc;
    }

    public Query buildQuery(String sessionId) {
        long hash = MurmurHash3.hash64(sessionId.getBytes(UTF_8));
        return new TermQuery(new Term("session_id_hash",
            Long.toString(hash, 36)));
    }
}
把 100M session_id FST 从 5 GB 压到 50 MB — 某 SaaS 公司的应用日志搜索集群,把 100M 唯一 session_id 当作 StringField 索引,.tip 膨胀到 5 GB,IndexWriter flush 时 OOM,merge 重建超时。改成 hash 字段后:hash 是 short alphabet(0-9 a-z 36 字符),平均 13 char,FST 共享率 70%,最终 .tip = 50 MB(下降 100×)。精确查询通过 TermQuery(session_id_hash, hash) 仍可命中(可接受 1/2^64 hash collision,业务可忽略),完整 ID 通过 StoredField 取回展示。这是高基数字段索引的标准修复模式。

                                    SegmentReadState state) throws IOException {

    this.indexIn = state.directory.openInput(
        IndexFileNames.segmentFileName(segmentName, state.segmentSuffix, "tip"),

            IndexFileNames.segmentFileName(segmentName, state.segmentSuffix, "tmd"))) {
        int numFields = metaIn.readVInt();
        for (int i = 0; i < numFields; i++) {

            BytesRef rootCode = readBytesRef(metaIn);
            long indexStartFP = metaIn.readVLong();
            FieldInfo fieldInfo = state.fieldInfos.fieldInfo(field);
            fields.put(fieldInfo.name,
                new FieldReader(this, fieldInfo, numTerms, rootCode,
                    indexStartFP, indexIn, ...));
        }
        CodecUtil.checkFooter(metaIn);
    }
}

//   .tip 中 session_id 字段 FST = 5 GB
//   IndexWriter flush 时 OOM
//   或写入到一定量后 NodeHash 堆爆
//   merge 时 FST 重建超时
// 修复:hash + DocValues + 倒排查询
public class SessionIndexer {
    public Document createDoc(String sessionId, String content) {
        Document doc = new Document();
        long hash = MurmurHash3.hash64(sessionId.getBytes(UTF_8));
        doc.add(new StringField("session_id_hash",
            Long.toString(hash, 36), Field.Store.NO));
        doc.add(new StoredField("session_id", sessionId));
        doc.add(new SortedDocValuesField("session_id_dv",
            new BytesRef(sessionId)));
        doc.add(new TextField("content", content, Field.Store.NO));
        return doc;
    }

    public Query buildQuery(String sessionId) {
        long hash = MurmurHash3.hash64(sessionId.getBytes(UTF_8));
        return new TermQuery(new Term("session_id_hash",
            Long.toString(hash, 36)));
    }
}

// 收益:
//   .tip 中 session_id_hash FST = 50 MB(下降 100×)
//   写入正常,无 OOM
//   精确查询仍可命中

反例 2:leading wildcard

// 反例:接受用户 leading wildcard "*foo"
public class SearchService {
    public List search(String pattern) {
        WildcardQuery q = new WildcardQuery(new Term("title", pattern));
        return searcher.search(q, 10).scoreDocs;
    }
}

// 现象:
//   "*foo" 扫描 100% term(1M term × 5 µs = 5 s)
//   单次查询 50 ms 上升到 5 s
//   并发 10 个 leading wildcard 查询直接耗尽 CPU
// 修复:拒绝 leading wildcard 或反向 token filter
public class SearchService {
    public List search(String pattern) {
        if (pattern.startsWith("*") || pattern.startsWith("?")) {
            throw new IllegalArgumentException(
                "leading wildcard not allowed");
        }
        WildcardQuery q = new WildcardQuery(new Term("title", pattern));
        return searcher.search(q, 10).scoreDocs;
    }

    public Document createDoc(String title) {
        Document doc = new Document();
        doc.add(new TextField("title", title, Field.Store.YES));
        doc.add(new TextField("title_reversed",
            new StringBuilder(title).reverse().toString(),
            Field.Store.NO));
        return doc;
    }

    public Query reversedQuery(String suffixPattern) {
        String reversed = new StringBuilder(
            suffixPattern.substring(1)).reverse().toString() + "*";
        return new WildcardQuery(new Term("title_reversed", reversed));
    }
}

// 收益:
//   leading wildcard 转 trailing wildcard,FST 可剪枝
//   查询时延 50 ms → 1 ms

24.17 本章小结

.tip 是 BlockTree 词典的"内存索引层":per-field FST 把 term 前缀映射成 .tim 中 block 的 64-bit 偏移,FST 沿前缀 walk 4-8 步即得到答案。它和 .tim 通过 8 byte output 编码(高 61 bit 是 blockFP,低 3 bit 是 hasTerms / hasSubBlocks / isFloor 三个标志)做最小协议;objectId 强校验保证两文件配对正确。1 千万 term 的 .tip 典型只占 30 MB,全量 mmap 进 page cache,亚毫秒 seek 不触磁盘——这是 Lucene 在亿级 term 词典上仍能"内存查询"的根源。
